    프로세스의 구조¶

    프로세스란¶

    프로세스는 실행 중인 프로그램이다. 디스크에 있는 바이너리 파일(/usr/bin/nginx 같은)을 실행하면, 커널이 메모리를 할당하고 PCB를 만들어 관리하기 시작한다. 그게 프로세스다.

    같은 프로그램을 두 번 실행하면 프로세스 두 개가 생긴다. 각각 독립된 메모리 공간과 PID를 가진다.

    # 같은 프로그램이지만 프로세스는 별개다
$ sleep 100 &
[1] 12345
$ sleep 100 &
[2] 12346
$ ps aux | grep sleep
user  12345  0.0  0.0  sleep 100
user  12346  0.0  0.0  sleep 100

    PCB (Process Control Block)¶

    커널이 프로세스 하나당 하나씩 관리하는 데이터 구조다. Linux에서는 task_struct라는 구조체가 PCB 역할을 한다. 컨텍스트 스위칭이 일어나면 현재 프로세스의 레지스터 값, 프로그램 카운터 등을 이 구조체에 저장하고, 다음 프로세스의 값을 꺼내서 복원한다.

    PCB에 들어가는 정보를 실제로 확인하는 방법이 있다.

    PID와 기본 정보¶

    # 현재 셸의 PID
$ echo $$
1234

# 프로세스 기본 정보
$ cat /proc/1234/status

State:  S (sleeping)
Tgid:   1234

    메모리 맵¶

    # 프로세스의 메모리 영역 배치 확인
$ cat /proc/1234/maps

0060a000-0060b000 rw-p  /usr/bin/bash   # 데이터 영역 (읽기/쓰기)

    r-xp처럼 퍼미션이 붙어 있다. 코드 영역은 쓰기가 안 되고, 스택/힙은 실행이 안 된다. 이게 메모리 보호의 기본이다.

    CPU 레지스터와 스케줄링 정보¶

    nr_voluntary_switches는 프로세스가 스스로 CPU를 양보한 횟수(I/O 대기 등), nr_involuntary_switches는 타임슬라이스가 끝나서 강제로 빼앗긴 횟수다. involuntary가 비정상적으로 높으면 CPU 경합이 심한 상황이다.

    프로세스 메모리 구조¶

    높은 주소

    스택과 힙이 서로를 향해 자라는 구조다. 이 둘이 만나면 메모리 부족이다.

    // 스택 오버플로우 재현

    char buf[4096];  // 스택에 4KB씩 할당
    recursive();     // 탈출 조건 없이 재귀
}
// 결과: Segmentation fault (core dumped)
// 기본 스택 크기는 8MB (ulimit -s로 확인)

    $ ulimit -s
8192  # 8MB

    프로세스 상태 전이¶

    교과서에는 New, Ready, Running, Waiting, Terminated 다섯 가지로 나오지만, Linux에서 실제로 보이는 상태는 다르다.

    Linux 프로세스 상태¶

    ps aux의 STAT 컬럼에 표시되는 값:

    상태 의미 언제 이 상태가 되나
    R Running / Runnable CPU에서 실행 중이거나 실행 대기 큐에 있음
    S Interruptible Sleep I/O 대기, sleep() 호출 등. 시그널로 깨울 수 있음
    D Uninterruptible Sleep 디스크 I/O 대기. kill -9로도 안 죽음
    T Stopped SIGSTOP이나 Ctrl+Z로 정지됨
    Z Zombie 종료됐지만 부모가 wait()를 안 호출함 
    $ ps aux | head -5
USER   PID  %CPU %MEM   STAT  COMMAND
root     1   0.0  0.1   Ss    /sbin/init
root     2   0.0  0.0   S     [kthreadd]
user  5678   0.0  0.2   S     /usr/bin/bash
user  5700  98.0  0.1   R     ./cpu_heavy

    STAT 컬럼 뒤에 붙는 접미사도 있다:

    • s: 세션 리더
    • +: 포그라운드 프로세스 그룹
    • l: 멀티스레드
    • <: 높은 우선순위 (nice 값이 음수)
    • N: 낮은 우선순위

    D 상태 프로세스 문제¶

    D 상태(Uninterruptible Sleep)는 NFS 마운트가 끊기거나 디스크가 응답하지 않을 때 자주 발생한다. 이 상태의 프로세스는 시그널을 받지 않기 때문에 kill -9로도 죽일 수 없다.

    # D 상태 프로세스 찾기
$ ps aux | awk '$8 ~ /D/ {print}'

# 어떤 I/O를 기다리는지 확인
$ cat /proc/<pid>/wchan
nfs_wait_bit_killable

    D 상태 프로세스가 쌓이면 시스템 load average가 올라간다. load average는 R + D 상태 프로세스의 합이기 때문이다.

    fork / exec / wait 시스템 콜¶

    프로세스 생성의 핵심이다. Unix 계열 OS에서 새 프로세스를 만드는 방법은 fork() 하나뿐이다 (init 프로세스 제외).

    fork()¶

    현재 프로세스를 복제한다. 호출하면 프로세스가 두 개가 된다.

    #include <stdio.h>
#include <unistd.h>
#include <sys/types.h>

int main() {
    printf("fork 전: PID=%d\n", getpid());

    pid_t pid = fork();

    if (pid < 0) {
        // fork 실패. 프로세스 수 제한에 걸렸거나 메모리 부족
        perror("fork failed");
        return 1;
    }

    if (pid == 0) {
        // 자식 프로세스: fork() 반환값이 0
        printf("자식: PID=%d, 부모 PID=%d\n", getpid(), getppid());
    } else {
        // 부모 프로세스: fork() 반환값이 자식의 PID
        printf("부모: PID=%d, 자식 PID=%d\n", getpid(), pid);
    }

    return 0;
}

    fork 전: PID=1000
부모: PID=1000, 자식 PID=1001
자식: PID=1001, 부모 PID=1000

    fork() 이후 부모와 자식 중 누가 먼저 실행될지는 보장되지 않는다. 출력 순서가 바뀔 수 있다.

    Copy-on-Write (COW): fork()가 부모의 메모리를 즉시 전부 복사하면 비용이 크다. 그래서 Linux는 COW를 쓴다. fork() 직후에는 부모와 자식이 같은 물리 페이지를 공유하고, 둘 중 하나가 메모리를 수정할 때 그 페이지만 복사한다.

    exec()¶

    현재 프로세스의 코드를 다른 프로그램으로 교체한다. fork()로 자식을 만든 뒤 exec()으로 다른 프로그램을 실행하는 게 일반적인 패턴이다.

    #include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();

    if (pid == 0) {
        // 자식: ls -la 실행
        // exec 성공하면 아래 printf는 실행되지 않는다.
        // exec은 현재 프로세스 이미지를 완전히 교체하기 때문이다.
        execl("/bin/ls", "ls", "-la", "/tmp", NULL);

        // exec이 실패한 경우에만 여기 도달
        perror("exec failed");
        _exit(127);  // exit()가 아니라 _exit() 사용
    }

    // 부모: 자식이 끝날 때까지 대기
    int status;
    waitpid(pid, &status, 0);

    if (WIFEXITED(status)) {
        printf("자식 종료 코드: %d\n", WEXITSTATUS(status));
    }

    return 0;
}

    exec 계열 함수가 여러 개 있다:

    함수 경로 지정 인자 전달 환경변수
    execl 전체 경로 가변 인자 상속
    execlp PATH 검색 가변 인자 상속
    execle 전체 경로 가변 인자 직접 지정
    execv 전체 경로 배열 상속
    execvp PATH 검색 배열 상속
    execve 전체 경로 배열 직접 지정

    실무에서 가장 많이 쓰는 건 execvp(PATH에서 찾아주고 인자를 배열로 전달)다.

    exec 후 _exit()을 써야 하는 이유¶

    자식 프로세스에서 exec이 실패했을 때 exit()가 아닌 _exit()를 써야 한다. exit()는 stdio 버퍼를 flush하고 atexit() 핸들러를 실행하는데, fork()로 복제된 자식에서 이걸 하면 부모의 버퍼 내용이 중복 출력되는 문제가 생긴다.

    wait() / waitpid()¶

    자식 프로세스의 종료를 기다리고 종료 상태를 수거한다. 이걸 호출하지 않으면 좀비 프로세스가 된다.

    #include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();

    if (pid == 0) {
        // 자식: 2초 후 종료
        sleep(2);
        return 42;
    }

    int status;

    // WNOHANG: 자식이 아직 안 끝났으면 바로 리턴 (논블로킹)
    pid_t result = waitpid(pid, &status, WNOHANG);
    if (result == 0) {
        printf("자식이 아직 실행 중\n");
    }

    // 블로킹 대기
    waitpid(pid, &status, 0);

    if (WIFEXITED(status)) {
        printf("정상 종료, 코드: %d\n", WEXITSTATUS(status));  // 42
    } else if (WIFSIGNALED(status)) {
        printf("시그널로 종료, 시그널: %d\n", WTERMSIG(status));
    }

    return 0;
}

    좀비 프로세스와 고아 프로세스¶

    서버 운영하다 보면 반드시 만나는 문제다.

    좀비 프로세스 (Zombie Process)¶

    자식 프로세스가 종료됐는데 부모가 wait()를 호출하지 않은 상태다. 프로세스 테이블에 종료 상태 정보만 남아 있고, 메모리나 CPU는 사용하지 않지만 PID를 점유한다.

    // 좀비 프로세스를 만드는 코드
#include <stdio.h>
#include <unistd.h>
#include <stdlib.h>

int main() {
    pid_t pid = fork();

    if (pid == 0) {
        printf("자식 종료\n");
        exit(0);  // 자식은 바로 종료
    }

    // 부모는 wait()를 호출하지 않고 계속 실행
    printf("부모가 wait() 안 하고 60초 대기\n");
    sleep(60);

    return 0;
}

    # 실행 후 다른 터미널에서 확인
$ ps aux | grep Z
user  1001  0.0  0.0  0  0 ?  Z  00:00  0:00 [zombie_test] <defunct>

    STAT이 Z이고 <defunct>로 표시된다. PID 하나를 점유하므로 좀비가 많아지면 새 프로세스를 못 만들 수 있다.

    # 시스템의 최대 PID 확인
$ cat /proc/sys/kernel/pid_max
32768

# 현재 좀비 프로세스 수
$ ps aux | awk '$8=="Z"' | wc -l

    좀비 프로세스 해결법¶

    1. 부모 프로세스에서 wait() 호출하기

    가장 기본적인 방법이다.

    2. SIGCHLD 시그널 핸들러 등록하기

    자식이 종료되면 커널이 부모에게 SIGCHLD 시그널을 보낸다. 이 시그널 핸들러에서 wait()를 호출하면 된다.

    #include <stdio.h>
#include <unistd.h>
#include <signal.h>
#include <sys/wait.h>
#include <errno.h>

void sigchld_handler(int sig) {
    int saved_errno = errno;  // errno 보존 필수

    // 종료된 자식이 여러 개일 수 있으므로 루프
    while (waitpid(-1, NULL, WNOHANG) > 0)
        ;

    errno = saved_errno;
}

int main() {
    struct sigaction sa;
    sa.sa_handler = sigchld_handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = SA_RESTART;  // 시스템 콜 자동 재시작
    sigaction(SIGCHLD, &sa, NULL);

    for (int i = 0; i < 10; i++) {
        if (fork() == 0) {
            sleep(1);
            _exit(0);
        }
    }

    // 부모는 다른 작업 수행
    sleep(30);
    return 0;
}

    핸들러에서 waitpid를 WNOHANG으로 루프 돌리는 이유: SIGCHLD 시그널은 합쳐질 수 있다. 자식 3개가 동시에 종료되어도 시그널은 1번만 올 수 있다. 그래서 한 번 핸들러가 호출되면 종료된 자식을 전부 수거해야 한다.

    3. SIGCHLD를 SIG_IGN으로 설정하기

    signal(SIGCHLD, SIG_IGN);  // 자식 종료 시 자동 수거

    POSIX에서 SIGCHLD를 SIG_IGN으로 설정하면 자식이 종료될 때 좀비가 되지 않고 바로 사라진다. 자식의 종료 상태가 필요 없을 때 쓴다.

    4. 이미 생긴 좀비 처리하기

    이미 좀비가 있으면 부모 프로세스를 죽이는 수밖에 없다. 좀비 자체에 kill을 보내도 이미 죽은 프로세스라 소용없다.

    # 좀비의 부모 PID 확인
$ ps -o pid,ppid,stat,cmd -p <zombie_pid>
  PID  PPID STAT CMD
 1001  1000 Z    [app] <defunct>

# 부모에게 SIGCHLD 보내기 (wait 호출을 유도)
$ kill -SIGCHLD 1000

# 그래도 안 되면 부모를 죽여야 함
$ kill 1000

    고아 프로세스 (Orphan Process)¶

    부모 프로세스가 자식보다 먼저 종료된 경우다. 이때 자식의 부모가 init(PID 1) 또는 systemd로 바뀐다.

    #include <stdio.h>
#include <unistd.h>
#include <stdlib.h>

int main() {
    pid_t pid = fork();

    if (pid == 0) {
        // 자식: 부모가 죽은 후 ppid 확인
        sleep(3);
        printf("자식: 내 부모 PID = %d\n", getppid());
        // 출력: 자식: 내 부모 PID = 1  (init이 양부모)
        exit(0);
    }

    // 부모: 즉시 종료
    printf("부모 종료\n");
    return 0;
}

    고아 프로세스 자체는 큰 문제가 아니다. init/systemd가 부모 역할을 하면서 종료 시 wait()를 호출해 좀비가 되는 것도 방지한다. 하지만 의도하지 않은 고아 프로세스가 계속 실행되면서 리소스를 먹는 경우가 있다.

    # 부모 PID가 1인 프로세스 찾기 (고아 프로세스 후보)
$ ps -eo pid,ppid,cmd | awk '$2==1'

    이 목록에서 시스템 데몬이 아닌 프로세스가 보이면 의도하지 않은 고아일 가능성이 있다.

    실무에서 자주 생기는 상황¶

    데몬 프로세스를 만들 때 의도적으로 고아를 만든다:

    // 전통적인 데몬 생성 패턴 (double fork)
pid_t pid = fork();
if (pid > 0) exit(0);      // 부모 종료 → 첫 번째 자식이 고아가 됨

setsid();                    // 새 세션 생성

pid = fork();                // 두 번째 fork
if (pid > 0) exit(0);       // 첫 번째 자식 종료

// 이제 이 프로세스는 세션 리더가 아니므로
// 터미널을 다시 할당받을 일이 없다

    요즘은 systemd가 이 처리를 해주기 때문에 직접 double fork를 하는 경우는 줄었다.

    셸 스크립트에서 백그라운드 프로세스 관리:

    #!/bin/bash

# 워커 프로세스 실행
./worker &
WORKER_PID=$!

# 트랩으로 정리
cleanup() {
    kill $WORKER_PID 2>/dev/null
    wait $WORKER_PID 2>/dev/null
}
trap cleanup EXIT SIGTERM SIGINT

# 메인 로직
wait $WORKER_PID

    trap을 안 걸면 스크립트가 죽었을 때 워커가 고아로 남는다.

    프로세스 디버깅 도구 정리¶

    # 프로세스 목록과 상태
ps aux                    # 전체 프로세스
ps -ef --forest           # 트리 형태로 부모-자식 관계 표시
pstree -p                 # 프로세스 트리 (PID 포함)

# 특정 프로세스 상세 정보
cat /proc/<pid>/status    # 상태 정보
cat /proc/<pid>/maps      # 메모리 맵
cat /proc/<pid>/fd        # 열린 파일
cat /proc/<pid>/cmdline   # 실행 명령어
cat /proc/<pid>/environ   # 환경변수

# 시스템 콜 추적
strace -p <pid>           # 실행 중인 프로세스의 시스템 콜 추적
strace -f ./app           # fork된 자식까지 추적 (-f 옵션)

# 리소스 사용량
/proc/<pid>/statm         # 메모리 사용량
/proc/<pid>/io            # I/O 통계

    strace -f는 fork/exec 문제를 디버깅할 때 쓴다. 자식 프로세스에서 exec이 왜 실패하는지, 어떤 파일을 못 열고 있는지 시스템 콜 수준에서 볼 수 있다.
