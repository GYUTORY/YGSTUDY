    파일 권한 및 소유권¶

    개요¶

    Linux 권한 시스템은 표면적으로는 단순하다. rwx 9비트, 소유자/그룹/기타. 그런데 실무에서 5년쯤 굴러보면 이 9비트만으로 설명되지 않는 상황이 너무 많다. 컨테이너 안에서 root로 만든 파일이 호스트에서는 누군지도 모르는 UID로 보이고, NFS 마운트한 디렉토리에서 chown 했더니 nobody로 바뀌고, chmod 777 한 번 박았다가 보안팀에서 티켓이 날아온다.

    이 문서는 권한 비트가 어떻게 계산되는지, 왜 그렇게 동작하는지, 그리고 운영 환경에서 권한 문제를 만났을 때 어디부터 의심해야 하는지를 다룬다. ACL의 mask나 Capabilities 같은 심화 주제는 파일 권한 심화 - ACL, Capabilities 문서에서 별도로 다룬다.

    rwx 9비트의 의미¶

    파일과 디렉토리에서 다르게 해석된다¶

    같은 r, w, x 글자라도 파일에 붙으면 의미가 다르고 디렉토리에 붙으면 또 다르다. 이걸 헷갈리면 디버깅이 안 된다.

    비트 파일 디렉토리
    r 파일 내용 읽기 디렉토리 안 항목 목록 보기 (ls)
    w 파일 내용 수정 디렉토리 안에 항목 생성/삭제/이름변경
    x 파일 실행 디렉토리 안 항목에 접근 (cd, 경로 traversal)

    특히 디렉토리의 x는 직관과 다르게 동작한다. x 없이 r만 있으면 파일 이름 목록은 볼 수 있지만 그 파일에 stat을 호출하지 못한다. 그래서 ls -l이 ?로 표시된다. 반대로 x만 있고 r이 없으면 파일 이름을 모르면 ls는 안 되지만 정확한 경로를 알면 접근은 된다. SFTP chroot 디렉토리나 일부 보안 설정에서 이런 조합을 의도적으로 쓴다.

    디렉토리 x 권한이 없을 때의 증상¶

    이게 운영하다 보면 자주 헷갈리는 부분이다. 같은 "권한 없음" 상황도 어느 단계에서 막혔는지에 따라 에러 메시지가 다르다.

    $ ls -ld /data
drw-r--r-- 2 root root 4096 Apr 27 10:00 /data

$ ls /data/file.txt
ls: cannot access '/data/file.txt': Permission denied

$ cat /data/file.txt
cat: /data/file.txt: Permission denied

    /data에 x가 없어서 그 안에 있는 파일을 stat하지 못한다. 파일 자체에 644 권한이 있어도 부모 디렉토리를 통과하지 못한다. 반대로 디렉토리에 r이 없고 x만 있으면 이런 식이 된다.

    $ ls -ld /data
d-wx--x--x 2 root root 4096 Apr 27 10:00 /data

$ ls /data
ls: cannot open directory '/data': Permission denied

$ cat /data/file.txt        # 정확한 이름을 알면 읽을 수 있다
hello

    여기서 더 헷갈리는 케이스는 중간 경로 어딘가에서 막혔을 때 나오는 No such file or directory다. 보통은 진짜 파일이 없을 때 나오는 에러지만, 중간 디렉토리에 x가 없어서 OS가 경로 해석을 끝까지 못 했을 때도 같은 메시지가 나오는 경우가 있다 (특히 setuid 컨텍스트 전환이 얽혔을 때). Permission denied가 아니라 No such file이 나오면 파일이 진짜 없는지부터 의심하기 마련이라 시간을 날리기 쉽다. 이럴 때 namei -l 한 번 돌려보면 어느 단계에서 막혔는지 보인다.

    $ namei -l /data/app/logs/app.log
f: /data/app/logs/app.log
 drwxr-xr-x root  root  /
 drwxr-xr-x root  root  data
 drw-r--r-- root  root  app          # 여기서 x가 빠졌다
 logs - Permission denied

    namei -l은 경로의 각 컴포넌트별로 권한을 보여준다. "분명 파일은 있는데 왜 안 되지" 싶을 때 첫 번째로 돌려야 하는 명령이다.

    숫자 표기법과 일반적인 조합¶

    8진수로 권한을 표기하는 건 rwx 각 비트를 421로 매핑한 결과다. r=4, w=2, x=1을 더한다.

    권한 8진수 용도
    rwxr-xr-x 755 실행 파일, 디렉토리 기본
    rw-r--r-- 644 일반 파일 기본
    rw------- 600 비밀 키, 민감 설정
    rw-rw---- 660 그룹 협업 파일
    rwxr-x--- 750 그룹 한정 실행
    rw-r----- 640 그룹 한정 읽기 (로그 등)
    rwxrwxrwx 777 거의 항상 잘못된 설정

    기본 8진수는 9비트지만, 실제로는 4자리로 적기도 한다. 맨 앞자리는 SetUID/SetGID/Sticky를 위한 특수 비트다. chmod 4755는 SetUID + 755를 의미한다.

    umask의 동작 원리¶

    umask는 신규 파일/디렉토리의 기본 권한을 결정한다. 운영에서 umask 022라는 값을 자주 보지만, 왜 022가 644가 되는지 정확히 설명할 수 있는 사람은 의외로 적다.

    AND NOT 연산이지 뺄셈이 아니다¶

    흔히 "기본 권한에서 umask를 뺀다"고 설명하지만 정확하게는 비트 연산이다. 신규 파일의 권한은 다음과 같이 계산된다.

    파일:     0666 AND (NOT umask)
디렉토리: 0777 AND (NOT umask)

    umask 022를 비트로 풀면 000 010 010이고, NOT을 씌우면 111 101 101이다. 파일 기본값 0666 = 110 110 110과 AND를 하면 110 100 100 = 0644가 나온다.

    뺄셈 모델은 umask가 0과 1만 있는 단순한 경우엔 결과가 같지만, 흔한 경우는 아니지만 이상한 umask를 만나면 차이가 드러난다. 예를 들어 umask 027이면 뺄셈 모델로는 666 - 027 = 640이 맞지만, umask 037이면 뺄셈으로 629라는 잘못된 값이 나온다. 실제로는 비트 연산이라서 666 AND NOT 037 = 640이다. 비트 단위로 "끄는" 마스크라고 이해하면 된다.

    왜 666/777부터 시작하는가¶

    신규 파일에 실행 권한을 주지 않는 것이 보안상의 기본 정책이기 때문이다. open(2) 시스템 콜로 새 파일을 만들 때 모드를 0666으로 요청해도, 커널은 의도적으로 실행 비트를 빼고 시작한다. 만약 0777부터 시작하면 사용자가 신경 쓰지 않은 사이 모든 파일에 실행 권한이 박혀버린다. 디렉토리는 traversal을 위해 x가 필요하니까 0777부터 시작한다.

    이 차이 때문에 같은 umask 022라도 파일은 644, 디렉토리는 755가 나온다. 디렉토리에 644가 나오면 그 안에 들어갈 수 없게 된다.

    운영 환경에서의 umask 값¶

    umask 파일 디렉토리 사용 사례
    022 644 755 서버 기본값, 가장 흔함
    027 640 750 보안 강화 (other 차단)
    002 664 775 그룹 협업 환경
    077 600 700 개인용 디렉토리, 강력한 격리

    /etc/login.defs나 /etc/profile, PAM 설정 (pam_umask)에서 시스템 전역 umask를 정한다. 사용자별로는 ~/.bashrc, ~/.profile에서 덮어쓴다. systemd 서비스는 unit 파일에 UMask=0027 같은 식으로 명시할 수 있다. 이걸 명시하지 않으면 systemd 기본값 0022로 동작하지, 사용자 셸의 umask는 무시된다는 점을 잊기 쉽다.

    chmod와 chown 사용 패턴¶

    chmod 숫자 모드와 문자 모드¶

    문자 모드는 기존 권한을 유지하면서 일부만 바꾸고 싶을 때 쓴다. 예를 들어 디렉토리에 그룹 쓰기만 추가하고 싶은데, 모든 파일의 정확한 현재 권한을 모를 때 숫자 모드로는 답이 안 나온다.

    chmod g+w /shared/team             # 그룹 쓰기만 추가, 나머지 유지
chmod -R u=rwX,g=rX,o= /opt/app    # 대문자 X는 디렉토리/실행파일에만 x 적용
chmod -R o-w /var/www              # other의 쓰기만 제거

    대문자 X를 잘 아는 사람이 의외로 적은데, 재귀 적용에서 매우 유용하다. 일반 파일에는 실행 비트를 안 붙이고 디렉토리와 이미 실행 가능한 파일에만 붙는다. 그래서 find -type f -exec chmod 644 + find -type d -exec chmod 755 두 번 돌리는 것을 한 줄로 끝낼 수 있다.

    chmod -R u=rwX,g=rX,o=rX /var/www/html

    chown은 -R 옵션을 쓸 때 심볼릭 링크 처리에 주의한다¶

    chown -R로 디렉토리 트리 소유자를 바꿀 때, 안에 심볼릭 링크가 있으면 링크 자체가 아니라 링크가 가리키는 대상의 소유자가 바뀐다. 트리 바깥의 파일까지 영향을 줄 수 있다.

    chown -R -h app:app /opt/app       # -h는 심볼릭 링크 자체의 소유자만 변경
chown -R --no-dereference app:app /opt/app

    -h나 --no-dereference를 붙여서 링크 대상이 아닌 링크 자체의 소유자만 바꾸는 게 안전하다. /opt/app/somelink -> /etc/passwd 같은 게 들어 있으면 큰일난다.

    특수 권한과 SetUID 보안 감사¶

    SetUID/SetGID/Sticky의 동작¶

    SetUID 비트(4xxx)가 설정된 실행 파일을 실행하면, 프로세스의 effective UID가 실행자가 아니라 파일 소유자가 된다. passwd 명령이 일반 사용자가 실행해도 root 권한으로 /etc/shadow를 수정할 수 있는 이유다.

    문제는 SetUID가 거의 모든 권한 상승 취약점의 진입점이라는 거다. SetUID 바이너리에 버그가 있으면 일반 사용자가 root가 된다. 그래서 운영 환경에서는 SetUID 바이너리를 정기적으로 감사한다.

    # 시스템 전체의 SetUID/SetGID 파일 찾기
find / -xdev \( -perm -4000 -o -perm -2000 \) -type f -print 2>/dev/null

# 각 파일의 권한과 함께 보기
find / -xdev -perm -4000 -type f -ls 2>/dev/null

    -xdev 옵션은 마운트 경계를 넘지 않게 한다. NFS나 다른 파일시스템까지 뒤지면 시간이 너무 걸린다. 결과를 baseline으로 저장해두고 주기적으로 diff를 떠서 새로 추가된 SetUID 파일이 있는지 본다. 정상적인 패키지 업데이트가 아니라면 의심해야 한다.

    스크립트(셔뱅이 있는 텍스트 파일)에 SetUID를 걸어도 Linux 커널은 무시한다. 보안상 이유로 셸 스크립트의 SetUID는 동작하지 않는다. 이걸 모르고 SetUID 스크립트를 만들어 놓고 "왜 안 되지" 하는 경우가 있다. 컴파일된 바이너리에서만 동작한다.

    Capabilities로 SetUID를 대체하기¶

    SetUID는 "전부 또는 전무"의 권한이라 너무 거칠다. ping이 raw socket을 열려고 root 권한 전체를 가져가는 건 과하다. Linux Capabilities는 root 권한을 40여 개의 세부 권한으로 쪼개서 필요한 것만 부여한다.

    # SetUID를 빼고 cap_net_raw만 부여
setcap cap_net_raw+ep /usr/bin/ping

# capability 확인
getcap /usr/bin/ping
# /usr/bin/ping = cap_net_raw+ep

    이러면 ping은 raw socket은 열 수 있지만, 파일을 마음대로 읽거나 다른 root 권한을 행사할 수는 없다. 새로 배포하는 도구는 가능하면 SetUID 대신 capabilities로 만든다. 자세한 capability 모델 (effective/permitted/inheritable, ambient)은 ACL_Capabilities 문서에서 다룬다.

    Sticky bit과 /tmp¶

    /tmp처럼 모두가 쓰기 가능한 디렉토리에 sticky bit(+t)를 걸면, 누구나 파일을 만들 수는 있지만 자기가 만든 파일만 삭제할 수 있게 된다. sticky bit이 없으면 누군가 /tmp 안의 다른 사람 파일을 지우거나 덮어쓸 수 있다. /tmp, /var/tmp, /dev/shm이 기본적으로 sticky bit이 걸려 있다 (drwxrwxrwt).

    chmod 777 안티패턴¶

    권한 문제로 막혔을 때 가장 빠른 해결책처럼 보이는 게 chmod 777이다. 이게 왜 안 되는 패턴인지 명확히 알아야 한다.

    왜 위험한가¶

    777은 모든 사용자에게 읽기/쓰기/실행을 모두 허용한다는 뜻이다. 멀티유저 시스템에서 이건 다음을 의미한다.

    • 다른 사용자가 그 파일을 마음대로 수정할 수 있다
    • 그 파일이 스크립트라면 다른 사용자가 코드를 바꾼 뒤 원래 사용자가 실행하게 만들 수 있다
    • 그 파일이 설정 파일이라면 인젝션의 진입점이 된다
    • 디렉토리에 적용하면 그 안의 파일을 누구나 삭제하거나 다른 파일로 교체할 수 있다

    특히 웹 디렉토리에 chmod -R 777 /var/www를 박는 건 자주 보는 사고 패턴이다. 웹 서버가 실행하는 사용자(www-data)뿐 아니라 그 호스트의 모든 사용자가 웹 컨텐츠를 수정할 수 있게 된다. 컨테이너 안이라도 마찬가지다. 보안팀이 가장 먼저 잡는 항목이다.

    진짜 문제는 무엇인가¶

    chmod 777로 해결되는 문제는 거의 항상 다음 중 하나다.

    • 프로세스가 잘못된 사용자로 실행되고 있다 (서비스 계정과 파일 소유자가 다름)
    • umask가 너무 빡빡해서 그룹 멤버가 못 읽는다
    • 디렉토리에 SetGID가 빠져서 새 파일이 엉뚱한 그룹으로 생성된다
    • 컨테이너 안 UID와 호스트 UID가 다르다 (다음 섹션 참고)

    777을 박기 전에 ls -l로 소유자/그룹/권한을 보고, id 명령으로 실행 사용자를 확인하고, namei -l로 경로를 따라가본다. 보통 5분 안에 진짜 원인이 나온다.

    # 잘못된 접근
chmod -R 777 /var/www/uploads

# 올바른 접근
chown -R www-data:www-data /var/www/uploads
chmod -R u=rwX,g=rX,o= /var/www/uploads
chmod g+s /var/www/uploads          # 새 파일도 그룹 상속

    컨테이너 환경의 UID/GID 문제¶

    컨테이너 시대로 넘어오면서 권한 문제의 양상이 완전히 바뀌었다. 컨테이너 안에서는 root지만 호스트에서는 일반 사용자, 호스트의 볼륨을 마운트했더니 안에서는 다른 사용자로 보이는 식의 문제가 일상이다.

    왜 UID/GID가 어긋나는가¶

    Linux 권한은 사용자 이름이 아니라 숫자 UID/GID로 저장된다. 호스트의 /etc/passwd와 컨테이너의 /etc/passwd는 별개다. 컨테이너 이미지의 app 사용자가 UID 1000일 수도 있고 1001일 수도 있다. 호스트의 app 사용자는 또 다른 UID다.

    볼륨을 마운트할 때 커널은 inode의 UID/GID 숫자만 본다. 호스트에서 만든 파일이 UID 1000 소유라면, 컨테이너 안에서 그 파일을 읽으려는 프로세스가 컨테이너 안 UID 1000으로 실행되어야 한다. 그게 컨테이너 안의 어떤 이름의 사용자인지는 상관없다.

    # 호스트에서
$ ls -ln /data
total 0
-rw-r--r-- 1 1000 1000 0 Apr 27 10:00 file.txt   # UID 1000

# 컨테이너 안에서
$ id
uid=1001(app) gid=1001(app)
$ cat /data/file.txt
cat: /data/file.txt: Permission denied            # UID가 안 맞음

    ls -ln은 이름이 아닌 숫자로 보여준다. UID/GID 문제를 디버깅할 때 첫 번째로 쓴다.

    해결 방법들¶

    가장 흔한 해결책은 컨테이너 안에서 호스트와 동일한 UID로 프로세스를 실행하는 것이다.

    docker run --user 1000:1000 -v /data:/data myimage

    Dockerfile에서 미리 같은 UID로 사용자를 만드는 방법도 있다.

    RUN groupadd -g 1000 app && useradd -u 1000 -g 1000 -m app
USER app

    볼륨에 init container 패턴으로 chown을 한 번 돌리는 방법도 자주 쓴다. 특히 PVC를 처음 마운트할 때 권한이 root:root로 시작하는 경우가 많다.

    initContainers:
- name: fix-permissions
  image: busybox
  command: ["sh", "-c", "chown -R 1000:1000 /data"]
  volumeMounts:
  - name: data
    mountPath: /data
  securityContext:
    runAsUser: 0

    Kubernetes에서는 fsGroup을 쓰면 마운트 시점에 자동으로 GID와 권한을 맞춰준다.

    spec:
  securityContext:
    runAsUser: 1000
    runAsGroup: 1000
    fsGroup: 1000
  containers:
  - name: app
    ...

    fsGroup: 1000이 설정되면 마운트되는 볼륨의 모든 파일에 GID 1000이 적용되고 그룹 쓰기 권한이 붙는다. 단점은 볼륨이 클 때 마운트 시점에 모든 파일을 chown하느라 파드 시작이 느려진다는 거다. fsGroupChangePolicy: OnRootMismatch를 쓰면 루트 디렉토리 GID만 검사해서 빠르게 처리한다.

    rootless 컨테이너와 user namespace¶

    rootless Docker나 Podman에서는 user namespace를 통해 컨테이너 안의 root(UID 0)가 호스트에서는 일반 사용자의 UID로 매핑된다. /etc/subuid, /etc/subgid에 설정된 범위로 매핑된다.

    $ cat /etc/subuid
alice:100000:65536

    이 설정은 alice 사용자에게 100000부터 65535개의 UID를 위임한다는 뜻이다. alice가 띄운 컨테이너의 root(0)는 호스트에서 100000, 컨테이너의 1은 100001 식으로 매핑된다. 컨테이너에서 만든 파일을 호스트에서 보면 100000 같은 이상한 UID로 보인다. 호스트에서 그 파일을 다루려면 같은 매핑을 알아야 한다.

    이 메커니즘을 잘 모르고 컨테이너 볼륨을 호스트에서 직접 수정하려다 권한 문제로 막히는 경우가 많다. podman unshare 명령으로 user namespace 안에 들어가서 작업하는 게 가장 깔끔하다.

    podman unshare chown -R 1000:1000 ~/data

    NFS 마운트의 권한 함정¶

    NFS 환경에서 권한이 이상하게 보이는 건 거의 NFS의 ID 매핑 동작 때문이다. 처음 보면 당황스럽지만 원리를 알면 단순하다.

    모든 nobody는 root_squash 때문이다¶

    기본 NFS 마운트에서 클라이언트가 root(UID 0)로 파일을 만들면, 서버는 이걸 익명 사용자(보통 nobody, UID 65534)로 매핑해서 저장한다. root 권한으로 NFS 서버를 마음대로 휘젓지 못하게 막는 안전장치다.

    # 클라이언트에서 root로 파일 생성
# server# touch /export/test.txt; ls -l
# -rw-r--r-- 1 root root 0 Apr 27 10:00 test.txt
client# touch /mnt/nfs/test2.txt
client# ls -l /mnt/nfs/test2.txt
-rw-r--r-- 1 nobody nobody 0 Apr 27 10:00 test2.txt

    서버에서 /etc/exports에 no_root_squash 옵션을 주면 root가 root로 보이게 되지만, 보안상 거의 권장되지 않는다. NFS 클라이언트 호스트가 침해되면 서버 파일시스템도 같이 털린다.

    UID 매핑이 안 맞을 때¶

    NFSv3에서는 사용자 이름이 아니라 UID 숫자로 권한을 판단한다. 클라이언트 A의 UID 1000과 서버의 UID 1000이 같은 사람이라는 보장이 없는데, NFS는 이걸 그대로 매칭한다. 회사 환경에서 LDAP 같은 중앙 인증을 쓰는 이유가 여기에 있다. UID/GID를 모든 호스트에서 일관되게 유지해야 한다.

    NFSv4에서는 idmapd를 통해 user@domain 형식으로 매핑할 수 있다. 다만 도메인 설정이 안 맞으면 모든 사용자가 nobody로 보인다. /etc/idmapd.conf의 Domain 설정과 nfs-idmapd 데몬 상태를 확인한다.

    # 클라이언트와 서버의 도메인이 같아야 한다
$ cat /etc/idmapd.conf
[General]
Domain = mycompany.local

    NFS에서 ACL이 안 먹는 경우¶

    NFSv3은 POSIX ACL을 표준으로 지원하지 않는다. 일부 구현이 사이드밴드로 지원하긴 하지만 호환성이 들쭉날쭉이다. NFSv4는 NFSv4 ACL이라는 다른 모델을 쓴다 (Windows ACL과 비슷). 리눅스 클라이언트는 이걸 POSIX ACL로 변환해서 보여주는데, 변환 과정에서 권한이 정확히 표현되지 않을 수 있다. NFS에서 ACL을 쓸 때는 양쪽 시스템에서 의도대로 보이는지 항상 검증한다.

    권한 디버깅 명령¶

    권한 문제가 터졌을 때 빠르게 원인을 좁히는 명령들이다.

    namei -l - 경로 컴포넌트별 권한¶

    가장 강력한 디버깅 도구다. 경로의 각 디렉토리마다 권한과 소유자를 보여준다.

    $ namei -l /var/lib/postgresql/data/pg_hba.conf
f: /var/lib/postgresql/data/pg_hba.conf
 drwxr-xr-x root        root        /
 drwxr-xr-x root        root        var
 drwxr-xr-x root        root        lib
 drwx------ postgres    postgres    postgresql        # 여기서 막힘
 data - Permission denied

    이걸로 거의 모든 "왜 접근이 안 되지" 문제의 원인이 보인다.

    getfacl - effective permission 확인¶

    ACL이 설정된 파일에서는 ls -l만으로는 부족하다. 끝에 +가 붙어 있으면 ACL이 있다는 뜻이고 getfacl로 봐야 한다.

    $ ls -l file.txt
-rw-rwxr--+ 1 alice alice 0 Apr 27 10:00 file.txt   # + 표시 주목

$ getfacl file.txt
# file: file.txt
# owner: alice
# group: alice
user::rw-
user:bob:rwx        #effective:rw-                   # ACL은 rwx인데 mask 때문에 rw-
group::r--
mask::rw-
other::r--

    #effective: 줄이 실제 적용 권한이다. ACL로 분명히 권한을 줬는데 안 먹을 때는 mask를 의심한다. 자세한 내용은 ACL_Capabilities 문서를 본다.

    sudo -u 로 실제 사용자처럼 시도해보기¶

    특정 서비스 계정의 권한 문제를 디버깅할 때, root로 보면 다 잘 보이니까 진짜 문제가 안 드러난다. 해당 사용자로 명령을 직접 실행해본다.

    sudo -u www-data ls /var/www/uploads
sudo -u www-data cat /etc/myapp.conf
sudo -u postgres psql -c "SELECT pg_read_file('/some/path')"

    서비스가 백그라운드에서 도는데 일부 파일만 못 읽고 있다면, sudo -u로 같은 작업을 셸에서 재현해서 좁혀들어간다.

    stat - 정밀 정보¶

    ls -l이 보여주지 않는 정보를 본다.

    $ stat /etc/passwd
  File: /etc/passwd
  Size: 2841        Blocks: 8          IO Block: 4096   regular file
Device: 802h/2050d  Inode: 2228355     Links: 1
Access: (0644/-rw-r--r--)  Uid: (    0/    root)   Gid: (    0/    root)
Context: system_u:object_r:passwd_file_t:s0          # SELinux 컨텍스트
Access: 2026-04-27 10:00:00.000000000 +0900
Modify: 2026-04-26 14:30:00.000000000 +0900
Change: 2026-04-26 14:30:00.000000000 +0900

    SELinux나 AppArmor가 활성화된 환경에서는 DAC(rwx) 외에 MAC도 작동한다. ls -l로는 권한이 멀쩡한데 접근이 막히면 getenforce, aa-status로 MAC 상태를 본다.

    lsattr - 확장 속성¶

    파일에 chattr +i로 immutable 비트가 걸려 있으면 root조차 수정/삭제하지 못한다. "왜 root인데 안 지워지지" 싶으면 lsattr로 확인한다.

    $ lsattr /etc/important.conf
----i---------e----- /etc/important.conf            # i 비트 = immutable
$ chattr -i /etc/important.conf                      # 해제

    운영 환경 권한 설정 패턴¶

    웹 컨텐츠 디렉토리¶

    chown -R www-data:www-data /var/www/html
find /var/www/html -type d -exec chmod 755 {} \;
find /var/www/html -type f -exec chmod 644 {} \;

# 업로드 디렉토리는 쓰기 필요, 실행은 차단
chown www-data:www-data /var/www/uploads
chmod 770 /var/www/uploads
chmod g+s /var/www/uploads

    업로드 디렉토리를 따로 분리하고 웹 서버 설정에서 그 디렉토리는 PHP/CGI 실행을 막아야 한다 (nginx의 location 블록에서 .php 처리 제외). 권한만으로는 업로드된 악성 스크립트의 실행을 완전히 막기 어렵다.

    로그 파일¶

    chown app:adm /var/log/myapp/
chmod 750 /var/log/myapp/
chown app:adm /var/log/myapp/app.log
chmod 640 /var/log/myapp/app.log

    adm 그룹은 로그 읽기 전용 그룹이다. 운영자를 adm 그룹에 넣으면 sudo 없이 로그를 읽을 수 있다. logrotate 설정에서 create 640 app adm 식으로 로테이션 후 권한도 명시한다.

    비밀 키와 설정¶

    chown root:root /etc/myapp/secret.key
chmod 400 /etc/myapp/secret.key

chown app:app /etc/myapp/app.conf
chmod 640 /etc/myapp/app.conf

    SSH 개인 키나 SSL 인증서 키, API 토큰이 들어간 설정은 400 또는 600이 기본이다. SSH는 키 권한이 너무 열려 있으면 아예 로딩을 거부한다.

    $ ssh -i /tmp/key user@host
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@         WARNING: UNPROTECTED PRIVATE KEY FILE!          @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
Permissions 0644 for '/tmp/key' are too open.

    공유 작업 디렉토리¶

    팀원 여러 명이 같이 쓰는 디렉토리는 SetGID + umask 002 조합이 핵심이다.

    groupadd developers
chown root:developers /shared/team
chmod 2770 /shared/team               # 2 = SetGID, 770 = 그룹 풀권한

# 팀원 셸 환경에서
umask 002

    SetGID 디렉토리에서 만들어진 파일은 부모 디렉토리의 그룹을 상속받는다. umask 002와 합쳐지면 같은 팀원끼리 서로 수정할 수 있는 환경이 된다. SetGID가 없으면 각자 만든 파일이 본인 그룹으로 생성되어 다른 팀원이 못 만지게 된다.

    권한과 보안 모델의 관계 정리¶

    지금까지 다룬 9비트 rwx, SetUID/GID/Sticky, umask, ACL은 전부 DAC(Discretionary Access Control)다. 파일 소유자의 재량으로 권한을 정한다는 모델이다. 이것 위에 다음 레이어들이 더 있다.

    • Capabilities: root 권한을 세분화한다. SetUID 대체 수단.
    • MAC (SELinux/AppArmor): DAC를 통과해도 별도 정책으로 막을 수 있다.
    • Namespaces: 컨테이너에서 UID/GID를 호스트와 다르게 매핑한다.
    • Seccomp: 시스템 콜 자체를 차단한다.

    운영에서 권한 문제를 만나면 어느 레이어에서 막혔는지 구분하는 게 첫 단계다. Permission denied만 보고는 알 수 없다. dmesg나 audit.log를 보면 SELinux/AppArmor가 막은 건지 DAC가 막은 건지 구분할 수 있다.

    # SELinux denial 확인
ausearch -m AVC -ts recent

# AppArmor denial 확인
dmesg | grep -i apparmor
journalctl -k | grep -i apparmor

    세부적인 capability 모델, ACL의 mask 동작, xattr/chattr, immutable 비트 활용은 파일 권한 심화 - ACL, Capabilities 문서에서 다룬다.
