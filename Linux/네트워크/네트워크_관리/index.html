    네트워크 관리¶

    개요¶

    서버 운영에서 네트워크 설정은 한 번 잘못 건드리면 SSH 세션이 끊겨서 콘솔 접속해야 하는 사고로 이어진다. 그래서 명령어를 외우는 것보다 어떤 설정이 어디에 저장되고, 재부팅 후에도 살아남는지를 정확히 아는 게 더 중요하다. 배포판마다 네트워크 관리 도구가 달라서 RHEL 계열에서 익숙했던 방식이 Ubuntu에서 안 통하는 일이 흔하다.

    명령어 한 줄로 IP를 바꾸는 건 누구나 할 수 있다. 진짜 문제는 그 IP가 재부팅 후에도 유지되는지, NetworkManager가 덮어쓰지 않는지, 다른 자동화 도구와 충돌하지 않는지를 따지는 부분이다. 이 문서는 그런 실무 시나리오를 중심으로 정리한다.

    네트워크 인터페이스 확인¶

    ip 명령어가 사실상 표준이다. ifconfig는 net-tools 패키지에 포함되어 있는데, 최신 배포판(Ubuntu 18.04+, RHEL 8+)에서는 기본 설치조차 안 된다. 신규 시스템에서는 ip만 사용한다고 생각하면 된다.

    ip addr show                 # 모든 인터페이스
ip addr show eth0
ip a                         # 짧은 별칭
ip -4 addr                   # IPv4만
ip -6 addr                   # IPv6만
ip -br a                     # 한 줄 요약 (스크립트에 유용)

    -br 옵션은 출력이 깔끔해서 자동화 스크립트에서 인터페이스 목록을 파싱할 때 자주 쓴다. ip -br a는 인터페이스명, 상태, IP를 한 줄로 보여줘서 수십 개의 인터페이스가 있는 호스트에서도 한눈에 본다.

    ip link show는 인터페이스의 L2 상태(UP/DOWN, MTU, MAC 주소)만 보여주고, IP 주소는 표시하지 않는다. 인터페이스가 정상적으로 활성화되어 있는지 먼저 확인할 때 유용하다.

    NetworkManager와 nmcli¶

    왜 NetworkManager인가¶

    CentOS 7, RHEL 8, 최신 Fedora, Ubuntu 데스크탑에서는 NetworkManager가 기본으로 동작한다. 서버에서도 RHEL 8부터는 NetworkManager가 표준이고, 이전의 network 서비스(initscripts)는 deprecated다. 그래서 ifcfg-* 파일을 직접 수정하는 방식보다 nmcli로 NetworkManager에 설정을 등록하는 방식이 권장된다.

    NetworkManager는 단순히 네트워크 설정 파일을 읽는 게 아니라, 자체 connection profile을 관리하면서 인터페이스 상태를 감시한다. 이 때문에 ip addr add로 임시로 IP를 추가해도 NetworkManager가 자기 정책에 맞게 다시 덮어쓰는 일이 발생한다. 운영 중에 IP가 사라졌다 나타났다 하는 현상의 대부분은 NetworkManager와 수동 설정이 충돌해서다.

    nmcli 기본 사용법¶

    nmcli는 NetworkManager의 CLI 클라이언트다. 명령 구조는 nmcli <object> <action> 형태인데, object는 connection, device, general 등이 있다.

    # 현재 활성 연결 확인
nmcli connection show
nmcli connection show --active

# 디바이스(인터페이스) 상태
nmcli device status

# 일반 상태 (NetworkManager가 인식하는 전체 상태)
nmcli general status

    connection(약어 con)은 NetworkManager가 관리하는 설정 프로파일이고, device는 실제 물리/가상 인터페이스다. 하나의 device에 여러 connection이 등록될 수 있고, 활성화되는 connection은 한 번에 하나다.

    정적 IP 설정¶

    NetworkManager 환경에서 정적 IP를 설정하는 표준 방식은 다음과 같다.

    # 새 connection 생성
nmcli connection add type ethernet con-name eth0-static ifname eth0 \
    ipv4.method manual \
    ipv4.addresses 192.168.1.10/24 \
    ipv4.gateway 192.168.1.1 \
    ipv4.dns "8.8.8.8 1.1.1.1"

# 활성화
nmcli connection up eth0-static

    기존 connection을 수정할 때는 nmcli connection modify를 사용한다.

    # IP 변경
nmcli connection modify eth0-static ipv4.addresses 192.168.1.20/24

# DNS 추가
nmcli connection modify eth0-static +ipv4.dns 9.9.9.9

# 자동 시작 설정
nmcli connection modify eth0-static connection.autoconnect yes

# 변경사항 적용 (재활성화)
nmcli connection down eth0-static && nmcli connection up eth0-static

    +와 - 접두사는 배열형 속성에서 항목을 추가/제거할 때 쓴다. ipv4.dns나 ipv4.addresses처럼 여러 값을 가질 수 있는 속성이 그렇다. 그냥 값을 지정하면 기존 값을 모두 덮어쓴다.

    운영 중에 자주 실수하는 부분이 nmcli con modify 후에 재활성화를 안 해서 변경이 적용되지 않는 경우다. modify는 디스크에 저장만 할 뿐, 동작 중인 connection에는 즉시 반영되지 않는다.

    connection 파일 위치¶

    nmcli로 등록한 설정은 키파일 또는 ifcfg 형식으로 저장된다. RHEL 9 / Fedora 36 이후로는 키파일 형식이 기본이다.

    /etc/NetworkManager/system-connections/        # 키파일 형식 (최신)
/etc/sysconfig/network-scripts/ifcfg-*         # ifcfg 형식 (RHEL 7~8)

    키파일은 INI 스타일이고 권한이 0600이어야 한다. 직접 편집하는 경우 nmcli connection reload로 NetworkManager에 다시 읽히게 해야 한다.

    /etc/sysconfig/network-scripts와 netplan¶

    배포판마다 네트워크 영구 설정 방식이 다르다. 이게 운영자를 가장 헷갈리게 하는 부분이다.

    RHEL 계열의 ifcfg¶

    RHEL 7과 CentOS 7까지는 /etc/sysconfig/network-scripts/ifcfg-eth0 같은 파일에 KEY=VALUE 형식으로 설정을 쓰고, network 서비스가 부팅 시 이를 읽어 적용했다.

    # /etc/sysconfig/network-scripts/ifcfg-eth0
TYPE=Ethernet
BOOTPROTO=static
NAME=eth0
DEVICE=eth0
ONBOOT=yes
IPADDR=192.168.1.10
PREFIX=24
GATEWAY=192.168.1.1
DNS1=8.8.8.8
DNS2=1.1.1.1

    RHEL 8부터는 NetworkManager가 이 파일을 읽어 처리하는 방식으로 바뀌었고, RHEL 9에서는 ifcfg 형식 자체가 deprecated되어 키파일 형식이 표준이다. 하지만 호환성 때문에 여전히 ifcfg 파일을 만들면 동작은 한다.

    ifcfg 파일을 직접 만들거나 수정한 경우 다음 명령으로 NetworkManager에 반영한다.

    nmcli connection reload
nmcli connection up eth0

    Ubuntu의 netplan¶

    Ubuntu 18.04부터는 netplan이 표준이다. /etc/netplan/*.yaml 파일에 YAML로 설정을 작성하면, netplan이 이를 NetworkManager 또는 systemd-networkd가 이해하는 형식으로 렌더링한다.

    # /etc/netplan/01-netcfg.yaml
network:
  version: 2
  renderer: networkd
  ethernets:
    eth0:
      dhcp4: no
      addresses:
        - 192.168.1.10/24
      routes:
        - to: default
          via: 192.168.1.1
      nameservers:
        addresses: [8.8.8.8, 1.1.1.1]

    renderer는 networkd(서버 기본) 또는 NetworkManager(데스크탑 기본)를 선택한다. 작성 후 다음 명령으로 적용한다.

    sudo netplan generate     # 렌더러용 설정 파일 생성
sudo netplan apply        # 적용

# 안전 적용 (변경 후 SSH가 끊기면 자동 롤백)
sudo netplan try

    netplan try는 실수로 잘못된 설정을 적용해 SSH가 끊기는 사고를 막아준다. 120초 내에 사용자가 확인하지 않으면 이전 설정으로 자동 복구한다. 원격 서버에서 IP를 변경할 때는 항상 try를 먼저 쓰는 습관을 들인다.

    ifcfg vs netplan 비교¶

    ifcfg는 인터페이스마다 파일 하나, KEY=VALUE 한 줄에 한 속성이라 단순하지만 조건부 설정이나 본드/브릿지 같은 복잡한 토폴로지를 표현하기 어렵다. netplan은 YAML이라 계층 구조가 자연스럽고, 본드/브릿지/VLAN을 한 파일에서 표현할 수 있다.

    대신 netplan은 추상화 레이어가 하나 더 끼어 있어서 디버깅이 어렵다. YAML에 적은 내용이 실제로 systemd-networkd의 어떤 파일로 렌더링되었는지 추적해야 할 때가 있다. 렌더링 결과는 /run/systemd/network/ 또는 /run/NetworkManager/system-connections/에서 확인한다.

    YAML 들여쓰기 실수로 설정 전체가 무시되는 경우도 자주 본다. netplan generate 시 에러가 안 나도 의도와 다른 결과가 나올 수 있어서 적용 전에 netplan get으로 파싱 결과를 확인하는 게 안전하다.

    영구 설정 방법별 차이¶

    리눅스에서 네트워크 설정을 영구적으로 적용하는 방법은 여러 개가 공존한다. 어떤 방식이 활성화되어 있는지 모르고 다른 방식으로 설정하면 부팅 후 의도와 다른 상태가 된다.

    NetworkManager¶

    가장 일반적이다. RHEL 8+, Fedora, Ubuntu 데스크탑에서 기본으로 동작한다. 인터페이스를 동적으로 감시하고, 케이블이 빠지면 down으로, 다시 꽂히면 up으로 자동 처리한다. 무선 환경에서 자동 전환이 필요하면 거의 필수다.

    서비스 상태 확인은 systemctl status NetworkManager로 한다. 설정은 nmcli 또는 connection 파일로 관리한다.

    systemd-networkd¶

    systemd가 직접 제공하는 네트워크 관리자다. NetworkManager보다 가볍고, 컨테이너나 미니멀 서버에서 선호된다. Ubuntu Server의 netplan 기본 렌더러도 이쪽이다.

    설정 파일은 /etc/systemd/network/*.network, *.netdev, *.link 형태로 인터페이스 종류별로 나뉜다.

    # /etc/systemd/network/10-eth0.network
[Match]
Name=eth0

[Network]
Address=192.168.1.10/24
Gateway=192.168.1.1
DNS=8.8.8.8

    활성화는 systemctl enable --now systemd-networkd로 한다. NetworkManager와 동시에 사용하면 두 데몬이 서로 인터페이스를 가져가려 해서 IP가 깜박이는 사고가 난다. 한쪽만 활성화해야 한다.

    rc.local (legacy)¶

    /etc/rc.local에 명령을 적어 부팅 시 실행하는 방식이다. 옛날에는 흔했지만 지금은 비추천이다. 이유는 다음과 같다.

    • 실행 시점이 모호하다. 다른 서비스가 시작된 뒤에 실행되어 의존성 문제가 생긴다.
    • 멱등성이 없다. 명령이 실패하면 그냥 넘어가고, 부팅 로그에서 추적이 어렵다.
    • 최신 systemd는 rc.local을 비활성화한 채로 배포한다. 활성화하려면 rc-local.service를 별도로 활성화해야 한다.

    급한 임시 설정 외에는 사용하지 않는다. 대신 systemd unit을 직접 작성하는 게 깔끔하다.

    어떤 방식을 써야 하는가¶

    운영 환경에서는 가급적 한 가지 방식만 쓴다. 혼용하면 어느 쪽이 IP를 설정했는지 추적이 어려워진다. 일반적인 선택 기준은 다음과 같다.

    • RHEL/CentOS 8 이상: NetworkManager + nmcli
    • Ubuntu Server 18.04 이상: netplan (renderer는 networkd)
    • 컨테이너 호스트, 미니멀 서버: systemd-networkd 직접 설정
    • 데스크탑/노트북: NetworkManager (자동 무선 전환 때문에)

    라우팅 관리¶

    기본 라우팅¶

    ip route show                 # 메인 테이블
ip route show table all       # 모든 테이블
ip route get 8.8.8.8          # 특정 목적지로 가는 경로

ip route add default via 192.168.1.1
ip route add 10.0.0.0/8 via 192.168.1.1
ip route add 10.0.0.0/8 via 192.168.1.1 metric 100
ip route del default

    ip route get은 디버깅에 자주 쓴다. 패킷이 실제로 어느 인터페이스로 어떤 게이트웨이를 거쳐 나가는지를 보여준다. 라우팅 테이블을 눈으로 읽고 추론하는 것보다 get으로 직접 물어보는 게 정확하다.

    정책 라우팅¶

    리눅스는 단일 라우팅 테이블이 아니라 여러 테이블을 가질 수 있고, 어떤 테이블을 사용할지를 룰(rule)로 결정한다. 이게 정책 라우팅(policy routing)이다.

    기본 룰은 다음과 같다.

    ip rule show
# 0:    from all lookup local
# 32766:    from all lookup main
# 32767:    from all lookup default

    local, main, default 세 개의 테이블이 기본으로 존재한다. 룰은 우선순위가 낮은 번호부터 평가된다. 정책 라우팅이 필요한 대표적인 시나리오는 인터페이스가 두 개 이상인 멀티홈 서버다.

    # 새 라우팅 테이블 정의
echo "100 isp1" >> /etc/iproute2/rt_tables

# eth1로 들어오는 트래픽은 isp1 테이블 사용
ip route add default via 10.0.0.1 dev eth1 table isp1
ip route add 10.0.0.0/24 dev eth1 src 10.0.0.10 table isp1

# 출발지 IP가 10.0.0.10인 패킷은 isp1 테이블로
ip rule add from 10.0.0.10 table isp1

    이렇게 하면 eth1로 들어온 요청에 대한 응답이 eth0의 기본 게이트웨이로 나가는 비대칭 라우팅 문제를 해결할 수 있다. 두 개의 ISP를 동시에 사용하거나, VPN 트래픽만 별도 게이트웨이로 보낼 때도 이 패턴을 쓴다.

    룰은 출발지 IP뿐 아니라 fwmark(iptables의 MARK 타겟이 붙인 표시)나 입력 인터페이스 같은 조건도 사용 가능하다. iptables의 mangle 테이블에서 패킷에 mark를 찍고, ip rule에서 그 mark를 보고 다른 라우팅 테이블로 보내는 패턴은 SD-WAN 같은 시스템의 기본 구조다.

    iptables와 영구 저장¶

    iptables 기본 사용¶

    iptables -L -n -v --line-numbers   # 현재 규칙 (숫자, 통계, 줄번호 포함)
iptables -S                        # 규칙을 명령어 형태로 출력
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -I INPUT 1 -p tcp --dport 22 -j ACCEPT  # 1번 위치에 삽입
iptables -D INPUT 3                 # 3번 규칙 삭제
iptables -F INPUT                   # INPUT 체인 비우기

    -A(append)와 -I(insert)의 차이는 운영에서 중요하다. 첫 번째 규칙이 DROP all이면 뒤에 어떤 ACCEPT를 append해도 매치되지 않는다. 그래서 새 규칙을 추가할 때 위치를 반드시 확인하고, 필요하면 -I로 앞쪽에 넣는다.

    --line-numbers는 규칙을 삭제할 때 필수다. -D로 삭제할 때는 동일한 매치 조건을 그대로 적거나, 줄번호를 지정해야 한다.

    iptables 영구 저장¶

    iptables 명령은 메모리상의 규칙만 변경한다. 재부팅하면 모두 사라진다. 영구화 방법은 배포판마다 다르다.

    RHEL/CentOS 7 (iptables-services)

    yum install iptables-services
service iptables save                          # /etc/sysconfig/iptables에 저장
systemctl enable iptables

    service iptables save가 호출되면 내부적으로 iptables-save > /etc/sysconfig/iptables가 실행되고, 부팅 시 iptables-restore < /etc/sysconfig/iptables가 자동으로 실행된다.

    Debian/Ubuntu (iptables-persistent)

    apt install iptables-persistent

# 규칙 저장
iptables-save > /etc/iptables/rules.v4
ip6tables-save > /etc/iptables/rules.v6

# 또는 패키지가 제공하는 명령 사용
netfilter-persistent save
netfilter-persistent reload

    netfilter-persistent는 /etc/iptables/rules.v4와 rules.v6를 부팅 시 자동으로 복원해준다. 패키지를 처음 설치할 때 현재 규칙을 저장할지 물어보므로 정신없이 Yes를 누르지 않도록 주의한다.

    RHEL 8 이후의 nftables

    RHEL 8부터는 백엔드가 nftables로 바뀌었다. 사용자 입장에서는 iptables 명령이 그대로 동작하지만, 내부적으로는 nftables 규칙으로 변환된다. iptables-save와 iptables-restore도 호환 모드로 동작한다. 새로 설계하는 시스템이라면 처음부터 nft 명령어 기반의 nftables 규칙으로 작성하는 게 미래 호환성 측면에서 낫다.

    iptables-save 출력 읽기¶

    *filter
:INPUT DROP [0:0]
:FORWARD DROP [0:0]
:OUTPUT ACCEPT [0:0]
-A INPUT -i lo -j ACCEPT
-A INPUT -m conntrack --ctstate RELATED,ESTABLISHED -j ACCEPT
-A INPUT -p tcp --dport 22 -j ACCEPT
COMMIT

    *filter는 테이블 이름, :CHAIN POLICY [pkt:byte]는 체인 기본 정책과 카운터, -A는 append된 규칙이다. 이 형식 그대로 iptables-restore에 들어가면 동일한 규칙이 복원된다. 백업/이관 시에는 이 출력만 보관하면 된다.

    firewalld와 zone¶

    zone 개념¶

    firewalld는 iptables/nftables 위에 한 단계 추상화를 얹어, "신뢰 수준"이라는 개념으로 트래픽을 분류한다. 각 zone은 미리 정의된 정책을 가지고, 인터페이스나 출발지 IP를 zone에 묶는 방식으로 동작한다.

    기본 제공되는 zone은 다음과 같다.

    firewall-cmd --get-zones
# block dmz drop external home internal nm-shared public trusted work

    각 zone의 의미는 단순하다.

    • drop: 들어오는 패킷을 모두 폐기, 응답도 없음
    • block: 들어오는 패킷 거부, ICMP host-prohibited 응답
    • public: 신뢰할 수 없는 네트워크. 명시적으로 허용한 서비스만 통과 (서버 기본값)
    • internal/work/home: 어느 정도 신뢰. 더 많은 서비스가 기본 허용
    • trusted: 모든 트래픽 허용. LAN 전용 인터페이스에 사용
    • dmz: 외부에 노출되지만 내부망과 분리된 호스트용

    zone 사용 예제¶

    # 기본 zone 확인
firewall-cmd --get-default-zone

# 활성 zone과 인터페이스 매핑
firewall-cmd --get-active-zones

# 특정 zone의 모든 설정
firewall-cmd --zone=public --list-all

# eth1을 internal zone으로
firewall-cmd --permanent --zone=internal --change-interface=eth1

# 출발지 IP 대역을 trusted로
firewall-cmd --permanent --zone=trusted --add-source=10.0.0.0/8

# public zone에 서비스 추가
firewall-cmd --permanent --zone=public --add-service=https
firewall-cmd --reload

    --permanent 없이 추가한 규칙은 런타임에만 반영되고 reload 시 사라진다. 영구 설정은 반드시 --permanent를 붙이고 마지막에 --reload로 런타임에도 반영한다.

    zone의 진짜 가치는 인터페이스가 여러 개일 때 드러난다. 외부망 인터페이스는 public, 내부 사설망 인터페이스는 internal로 묶어두면 패킷이 어느 인터페이스로 들어오느냐에 따라 자동으로 다른 정책이 적용된다. iptables로 같은 걸 하려면 -i eth0/-i eth1 매치를 모든 규칙에 일일이 적어야 한다.

    rich rule¶

    기본 service/port 단위로 표현이 안 되는 세밀한 규칙은 rich rule을 쓴다.

    # 192.168.10.0/24에서 오는 SSH만 허용, 나머지는 기본 정책
firewall-cmd --permanent --zone=public --add-rich-rule='
  rule family="ipv4" source address="192.168.10.0/24"
  service name="ssh" accept'

firewall-cmd --reload

    rich rule은 출발지/목적지 IP, 포트, 로깅, 레이트 리밋까지 표현 가능하다. 다만 문법이 길어서 가독성이 떨어진다. 룰이 많아지면 직접 nftables/iptables로 작성하는 게 차라리 명확하다.

    ss와 netstat 출력 해석¶

    ss 기본 사용¶

    ss(socket statistics)는 /proc/net/을 직접 읽지 않고 커널의 netlink 인터페이스를 통해 정보를 가져온다. 그래서 연결이 수만 개 있는 서버에서도 netstat보다 훨씬 빠르다.

    ss -tan                # TCP, 모든 상태, 숫자 표시
ss -tuln               # TCP/UDP 리스닝 포트만
ss -tulnp              # 프로세스 정보 포함 (root 권한 필요)
ss -tan state established
ss -tan state time-wait | wc -l    # TIME_WAIT 개수 세기
ss -i                  # TCP 내부 정보 (cwnd, rtt 등)
ss -s                  # 요약 통계

    ss -tan 출력 컬럼¶

    State      Recv-Q  Send-Q  Local Address:Port    Peer Address:Port
ESTAB      0       0       192.168.1.10:22       192.168.1.5:54321
LISTEN     0       128     0.0.0.0:80            0.0.0.0:*
    • State: TCP 연결 상태. ESTAB, LISTEN, TIME-WAIT, CLOSE-WAIT 등
    • Recv-Q: 수신 큐. LISTEN 상태에서는 현재 backlog의 사용량(완료된 연결 수)을 의미한다. 그 외 상태에서는 애플리케이션이 아직 read하지 않은 바이트 수다.
    • Send-Q: 송신 큐. LISTEN 상태에서는 backlog의 최대값을 의미한다. 그 외 상태에서는 상대방이 ack하지 않은 바이트 수다.
    • Local Address:Port: 로컬측 주소와 포트. 0.0.0.0은 모든 인터페이스, *은 와일드카드.
    • Peer Address:Port: 원격측 주소와 포트.

    LISTEN 상태에서 Send-Q가 backlog 최대값이라는 게 헷갈리는 부분이다. 예를 들어 Recv-Q: 100, Send-Q: 128이면 backlog 128 중 100이 사용 중이라는 뜻이다. 이 값이 가득 차면 새 연결이 거부되거나 SYN-ACK이 늦게 나가서 connection refused 또는 타임아웃이 발생한다. nginx나 Java 서버에서 backlog 튜닝이 필요한지를 판단할 때 이 값을 본다.

    ESTABLISHED 상태에서 Recv-Q가 큰 값이면 애플리케이션이 데이터를 읽지 못하고 있다는 뜻이다. CPU 포화나 GC pause로 read 호출이 지연될 때 자주 보인다. Send-Q가 큰 값이면 상대방이 ack를 안 보내거나 네트워크가 막혀 있다는 뜻이다.

    TCP 상태 해석¶

    상태별로 의미가 다르고, 운영에서 자주 보이는 패턴은 정해져 있다.

    • LISTEN: 서버가 연결을 기다리는 상태. listen() 호출 후.
    • SYN-SENT: 클라이언트가 SYN을 보내고 SYN-ACK를 기다리는 상태. 오래 머물면 방화벽이나 라우팅 문제다.
    • SYN-RECV: 서버가 SYN을 받고 SYN-ACK를 보낸 후 ACK를 기다리는 상태. 이 상태가 비정상적으로 많으면 SYN flood 공격이나 backlog 부족이다.
    • ESTABLISHED: 양쪽이 데이터 송수신 중인 정상 상태.
    • FIN-WAIT-1, FIN-WAIT-2: 우리 쪽이 먼저 close()를 호출하고 상대방의 응답을 기다리는 중. FIN-WAIT-2가 많이 쌓이면 상대방이 close를 안 하고 있다는 뜻이다.
    • CLOSE-WAIT: 상대방이 close()를 했고 우리가 close()를 호출해야 하는 상태. 이 상태가 쌓이면 애플리케이션이 소켓을 close 안 하는 버그다. 운영에서 가장 자주 만나는 문제 중 하나다.
    • TIME-WAIT: 우리가 마지막 ACK를 보낸 후 2MSL(보통 60초) 동안 대기하는 상태. 짧은 연결을 대량으로 만드는 클라이언트에서 흔히 나타난다.
    • LAST-ACK, CLOSING: 종료 과정의 일시적 상태. 보통 즉시 사라진다.

    CLOSE_WAIT vs TIME_WAIT 트러블슈팅¶

    CLOSE_WAIT가 쌓이면 애플리케이션 코드를 의심한다. 예외 발생 시 finally에서 socket close를 안 하거나, connection pool에서 idle 처리가 잘못된 경우다. 재시작하면 일시적으로 사라지지만 근본 원인을 안 잡으면 다시 쌓인다.

    TIME_WAIT가 수만 개 있어도 보통은 문제가 안 된다. 다만 동일한 소스 IP/포트로 한 목적지에 초당 수천 connection을 만드는 워크로드(예: 짧은 HTTP 요청 폭주)에서는 ephemeral port 고갈로 새 연결을 못 만드는 경우가 있다. 이때는 다음을 검토한다.

    # ephemeral 포트 범위 확장
sysctl -w net.ipv4.ip_local_port_range="10000 65535"

# TIME_WAIT 소켓 재사용 (안전)
sysctl -w net.ipv4.tcp_tw_reuse=1

    tcp_tw_recycle은 NAT 환경에서 문제를 일으켜서 커널 4.12부터 제거됐다. 옛날 문서에서 이걸 권하더라도 무시한다.

    netstat 비교¶

    netstat은 /proc/net/tcp, /proc/net/udp 같은 파일을 텍스트로 파싱한다. 연결이 많을 때 이 파일을 읽고 파싱하는 비용이 크다. 출력 형식은 ss와 비슷하지만 Recv-Q/Send-Q의 의미는 동일하다.

    netstat -tulnp         # TCP/UDP 리스닝, 숫자, 프로세스
netstat -anp           # 모든 연결
netstat -rn            # 라우팅 테이블 (= ip route)
netstat -i             # 인터페이스 통계 (= ip -s link)
netstat -s             # 프로토콜별 통계

    netstat -s가 보여주는 프로토콜별 통계(segments retransmitted, bad segments received, listen queue overflow 같은 카운터)는 디버깅에 유용하다. ss로는 같은 정보를 보기 어려워서 netstat -s만큼은 여전히 자주 쓴다.

    네트워크 테스트¶

    ping과 traceroute¶

    ping -c 4 8.8.8.8
ping -i 0.2 -c 100 8.8.8.8     # 0.2초 간격으로 100번
ping -s 1472 -M do 8.8.8.8     # MTU 테스트 (1472+28=1500)

traceroute -n 8.8.8.8           # 숫자 표시 (DNS 해석 안 함)
traceroute -T -p 443 example.com  # TCP 443으로 추적

    ping -M do는 패킷 단편화를 금지한다. 이 옵션과 함께 사이즈를 조정하면 경로의 MTU를 알 수 있다. VPN이나 IPsec 환경에서 통신은 되는데 큰 패킷만 끊기는 경우 PMTU 문제를 의심하고 이걸 쓴다.

    전통적인 traceroute는 UDP를 사용해서 방화벽에 막히는 경우가 많다. -T(TCP) 또는 -I(ICMP)로 변경 가능하고, 운영망에서는 TCP 443으로 테스트하는 게 가장 잘 통과한다.

    mtr¶

    mtr -r -c 100 -n 8.8.8.8        # 보고서 모드, 100회, 숫자 표시

    mtr은 ping과 traceroute를 합친 도구다. 각 hop별 패킷 손실률과 지연을 실시간으로 보여준다. "어딘가에서 패킷이 사라진다"는 모호한 문제를 진단할 때 mtr만한 게 없다. 중간 어떤 라우터에서 30% loss가 나오면 그 구간이 범인이다.

    nc¶

    nc -zv 192.168.1.1 80          # 포트 연결만 확인
nc -zv 192.168.1.1 80-90       # 포트 범위 스캔
nc -lp 8080                    # 8080에서 리스닝
echo "test" | nc -w 1 host 80  # 1초 타임아웃으로 데이터 전송

    방화벽 정책 검증할 때 nc가 가장 빠르다. telnet은 대화형이라 스크립트에 쓰기 불편하지만 nc는 -z로 검사만 하고 즉시 종료한다.

    DNS 확인¶

    dig example.com                 # 기본 A 레코드
dig +short example.com          # 결과만
dig MX example.com
dig @8.8.8.8 example.com        # 특정 DNS 서버 사용
dig +trace example.com          # 루트부터 따라가며 조회

nslookup example.com
host example.com

    dig +trace는 DNS 위임 체인을 추적해서 어느 단계에서 응답이 어떻게 오는지 보여준다. 새 도메인의 NS 위임이 제대로 되었는지, 캐시 DNS의 결과와 권한 DNS의 결과가 다른지 확인할 때 쓴다.

    /etc/resolv.conf는 시스템의 기본 DNS 설정인데, 최신 시스템에서는 systemd-resolved나 NetworkManager가 자동으로 관리한다. 직접 편집해도 다음 부팅이나 connection 활성화 시 덮어써질 수 있어서 nmcli의 ipv4.dns 또는 netplan의 nameservers로 영구 설정해야 한다.

    트러블슈팅 흐름¶

    연결이 안 될 때는 OSI 모델대로 아래에서 위로 올라가며 확인한다.

    # L1/L2: 인터페이스가 살아 있는가
ip link show eth0          # state UP인지

# L3: IP가 붙어 있고 라우팅이 있는가
ip addr show eth0
ip route show
ip route get 8.8.8.8       # 패킷이 어디로 나갈지

# L3 도달성
ping -c 3 192.168.1.1      # 게이트웨이
ping -c 3 8.8.8.8          # 외부

# L7: DNS
dig example.com

# L4: 포트가 열려 있는가
ss -tlnp | grep :80
nc -zv example.com 443

# 방화벽 확인
iptables -L -n -v
firewall-cmd --list-all

    이 순서대로 따라가면 대부분의 문제는 어디서 막혔는지 찾는다. 단계를 건너뛰고 추측하면 시간이 더 걸린다. 특히 "방화벽이 막는 것 같다"는 직감으로 iptables부터 보다가 사실은 IP가 안 붙어 있던 경우를 자주 본다.

    성능 문제는 별개다. 처리량이 떨어지거나 latency가 튀는 경우는 ss -i의 cwnd/rtt, sar -n DEV의 패킷/에러, ethtool -S eth0의 드라이버 카운터를 종합해서 봐야 한다. 이 부분은 별도의 튜닝 주제다.
