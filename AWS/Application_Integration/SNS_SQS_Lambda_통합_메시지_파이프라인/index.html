    개요¶

    통합 메시지 파이프라인이란?¶

    AWS SNS, SQS, Lambda를 통합하여 구축하는 메시지 파이프라인은 Event Driven Architecture(EDA)의 핵심 구성 요소입니다. 세 서비스를 조합하여 확장 가능하고, 신뢰성 높은 비동기 메시지 처리 시스템을 구축할 수 있습니다.

    각 서비스의 역할¶

    SNS (Simple Notification Service) - 이벤트 발행 및 팬아웃(Fan-out) 패턴 구현 - 여러 구독자에게 동시에 메시지 전달 - 푸시 기반 메시징

    SQS (Simple Queue Service) - 메시지 버퍼링 및 큐잉 - 메시지 순서 보장 (FIFO Queue) - 재시도 및 Dead Letter Queue 지원

    Lambda - 서버리스 이벤트 핸들러 - 자동 확장 및 비용 효율성 - 다양한 이벤트 소스 지원

    웹 클라이언트 통신 제한¶

    SNS와 SQS는 서버 간 통신을 위한 메시징 서비스입니다. 웹 브라우저나 모바일 앱 같은 클라이언트에서 직접 구독하거나 연결할 수 없습니다.

    SNS/SQS의 설계 목적과 제한

    SNS와 SQS는 마이크로서비스 간 비동기 통신을 위해 설계되었습니다. 서버 애플리케이션, Lambda 함수, 다른 AWS 서비스 같은 신뢰할 수 있는 엔드포인트와 통신하는 것이 목적입니다.

    SNS는 HTTP/HTTPS 엔드포인트, SQS 큐, Lambda 함수, Email, SMS, 모바일 푸시 알림 등의 구독자만 지원합니다. 이들은 모두 서버 측 리소스이거나 AWS가 관리하는 서비스입니다. 웹 브라우저는 이 목록에 포함되지 않습니다.

    SQS는 폴링 기반 메시지 수신 방식을 사용합니다. 클라이언트가 큐에 메시지가 있는지 주기적으로 확인하고, 있으면 가져오는 방식입니다. 이는 서버 애플리케이션이나 Lambda 함수가 수행하는 작업입니다. 웹 브라우저는 지속적인 폴링을 수행할 수 없고, 브라우저가 닫히면 연결이 끊어지기 때문에 SQS의 메시지 처리 모델과 맞지 않습니다.

    웹 클라이언트와 서버 간 통신의 차이

    웹 브라우저는 HTTP 요청-응답 모델을 기본으로 합니다. 클라이언트가 요청을 보내면 서버가 응답을 보내는 일방향 통신입니다. 실시간 양방향 통신이 필요하면 WebSocket 같은 별도 프로토콜이 필요합니다.

    SNS는 푸시 기반으로 동작합니다. 메시지가 발생하면 구독자들에게 즉시 전달합니다. 하지만 이 구독자들은 서버 엔드포인트여야 합니다. 웹 브라우저는 엔드포인트가 아니라 클라이언트입니다. 브라우저는 IP 주소가 동적으로 변할 수 있고, 방화벽이나 NAT 뒤에 있을 수 있으며, 연결이 불안정할 수 있습니다. 이런 특성 때문에 SNS가 직접 브라우저에 메시지를 푸시할 수 없습니다.

    웹 클라이언트가 실시간 메시지를 받아야 하는 경우

    웹 브라우저에서 실시간으로 메시지를 받아야 한다면 다른 방식을 사용해야 합니다. Socket.IO, WebSocket, Server-Sent Events(SSE) 같은 실시간 통신 프로토콜을 사용합니다.

    Socket.IO는 WebSocket을 기반으로 하되, 폴링 폴백을 제공하는 라이브러리입니다. WebSocket 연결이 실패하면 자동으로 HTTP 폴링으로 전환됩니다. 양방향 통신이 가능하고, 방과 같은 개념으로 여러 클라이언트를 그룹화할 수 있습니다.

    WebSocket은 TCP 기반의 양방향 통신 프로토콜입니다. 한 번 연결을 맺으면 서버와 클라이언트가 언제든지 메시지를 주고받을 수 있습니다. HTTP 업그레이드 요청으로 시작하며, 연결이 유지되는 동안 지속적으로 통신합니다.

    Server-Sent Events(SSE)는 서버에서 클라이언트로만 메시지를 보내는 단방향 통신입니다. HTTP 연결을 유지하면서 서버가 주기적으로 데이터를 푸시합니다. WebSocket보다 단순하지만 서버에서 클라이언트로만 통신이 가능합니다.

    웹 클라이언트 통신을 위한 아키텍처 패턴

    웹 클라이언트가 실시간 알림을 받아야 하는 경우, SNS → SQS → Lambda → WebSocket/Socket.IO 서버로 이어지는 파이프라인을 구성합니다.

    SNS가 이벤트를 발행하면 SQS 큐에 메시지가 쌓입니다. Lambda 함수가 SQS 메시지를 폴링해서 가져옵니다. Lambda 함수는 메시지를 처리한 뒤, WebSocket 연결을 유지하는 별도의 서버에 메시지를 전달합니다. 이 서버는 연결된 모든 클라이언트나 특정 클라이언트에게 메시지를 푸시합니다.

    이 방식의 장점은 SNS/SQS의 신뢰성과 확장성을 그대로 활용하면서, 웹 클라이언트와의 실시간 통신도 가능하다는 점입니다. SQS가 메시지를 버퍼링하기 때문에 Lambda 함수가 일시적으로 실패해도 메시지가 유실되지 않습니다.

    API Gateway WebSocket API를 활용한 방식

    API Gateway WebSocket API를 사용하면 Lambda 함수가 직접 클라이언트에게 메시지를 전송할 수 있습니다. 이 경우 SNS → SQS → Lambda → API Gateway WebSocket → 클라이언트 경로로 메시지가 전달됩니다.

    Lambda 함수가 SQS 메시지를 처리한 뒤, API Gateway Management API를 통해 특정 WebSocket 연결에 메시지를 전송합니다. 연결 ID를 관리하는 데이터베이스(DynamoDB 등)에 사용자와 연결 ID를 매핑해두고, Lambda 함수가 해당 연결 ID로 메시지를 보냅니다.

    이 방식은 별도의 WebSocket 서버를 운영할 필요가 없습니다. Lambda와 API Gateway만으로 구성할 수 있어 서버리스 아키텍처에 적합합니다. 다만 연결 ID 관리와 메시지 라우팅 로직을 직접 구현해야 합니다.

    브라우저에서 AWS SDK 사용의 한계

    AWS SDK를 브라우저에서 사용할 수 있지만, 이는 제한적입니다. SNS에 메시지를 발행하거나 SQS에 메시지를 보내는 용도로만 사용 가능합니다. 구독이나 폴링은 서버 측에서만 가능합니다.

    브라우저에서 SNS에 메시지를 발행하는 경우, AWS 자격 증명을 클라이언트에 노출해야 합니다. 이는 보안상 위험합니다. Cognito를 사용해 임시 자격 증명을 발급받아 사용할 수 있지만, 여전히 클라이언트 측에서 자격 증명을 관리해야 합니다.

    SQS 폴링을 브라우저에서 직접 수행하는 것은 권장되지 않습니다. 폴링은 지속적으로 수행되어야 하는데, 브라우저는 탭이 닫히거나 페이지를 떠나면 작업이 중단됩니다. 또한 폴링 간격을 짧게 하면 비용이 증가하고, 길게 하면 실시간성이 떨어집니다.

    실무에서 겪는 문제

    웹 클라이언트가 실시간 알림을 받아야 하는 기능을 구현할 때, SNS/SQS를 직접 사용하려고 시도하는 경우가 있습니다. 하지만 이는 불가능하므로 중간에 WebSocket 서버나 API Gateway WebSocket을 두어야 합니다.

    WebSocket 서버를 별도로 운영하는 경우, 서버의 확장성과 가용성을 고려해야 합니다. 여러 인스턴스로 확장할 때는 연결된 클라이언트 정보를 공유할 수 있는 저장소(Redis 등)가 필요합니다. 특정 사용자에게 메시지를 보내려면 해당 사용자가 어느 서버에 연결되어 있는지 알아야 하기 때문입니다.

    API Gateway WebSocket을 사용하는 경우, 연결 ID를 DynamoDB에 저장하고 관리해야 합니다. 사용자가 로그인하면 연결 ID를 저장하고, 로그아웃하거나 연결이 끊어지면 삭제해야 합니다. Lambda 함수가 메시지를 보낼 때는 DynamoDB에서 연결 ID를 조회해서 사용합니다.

    // src/aws/sns/sns.service.ts
import { Injectable, Logger } from '@nestjs/common';
import { ConfigService } from '@nestjs/config';
import { SNSClient, PublishCommand } from '@aws-sdk/client-sns';

@Injectable()
export class SnsService {
  private readonly logger = new Logger(SnsService.name);
  private readonly snsClient: SNSClient;

  constructor(private configService: ConfigService) {
    this.snsClient = new SNSClient({
      region: this.configService.get<string>('AWS_REGION'),
      credentials: {
        accessKeyId: this.configService.get<string>('AWS_ACCESS_KEY_ID'),
        secretAccessKey: this.configService.get<string>('AWS_SECRET_ACCESS_KEY'),
      },
    });
  }

  /**
   * SNS Topic에 메시지 발행
   */
  async publishMessage(
    topicArn: string,
    message: Record<string, any>,
    subject?: string,
  ): Promise<string> {
    try {
      const command = new PublishCommand({
        TopicArn: topicArn,
        Message: JSON.stringify(message),
        Subject: subject,
        MessageAttributes: {
          'event-type': {
            DataType: 'String',
            StringValue: message.eventType || 'unknown',
          },
          'timestamp': {
            DataType: 'String',
            StringValue: new Date().toISOString(),
          },
        },
      });

      const response = await this.snsClient.send(command);
      this.logger.log(`Message published to ${topicArn}: ${response.MessageId}`);

      return response.MessageId;
    } catch (error) {
      this.logger.error(`Failed to publish message to ${topicArn}:`, error);
      throw error;
    }
  }

  /**
   * 주문 생성 이벤트 발행
   */
  async publishOrderCreated(orderData: {
    orderId: string;
    userId: string;
    items: Array<{ productId: string; quantity: number }>;
    totalAmount: number;
  }): Promise<string> {
    const topicArn = this.configService.get<string>('ORDER_CREATED_TOPIC_ARN');

    return this.publishMessage(
      topicArn,
      {
        eventType: 'order.created',
        ...orderData,
      },
      'Order Created',
    );
  }

  /**
   * 사용자 등록 이벤트 발행
   */
  async publishUserRegistered(userData: {
    userId: string;
    email: string;
    name: string;
  }): Promise<string> {
    const topicArn = this.configService.get<string>('USER_REGISTERED_TOPIC_ARN');

    return this.publishMessage(
      topicArn,
      {
        eventType: 'user.registered',
        ...userData,
      },
      'User Registered',
    );
  }
}

    // src/aws/sqs/sqs.service.ts
import { Injectable, Logger } from '@nestjs/common';
import { ConfigService } from '@nestjs/config';
import {
  SQSClient,
  SendMessageCommand,
  ReceiveMessageCommand,
  DeleteMessageCommand,
  GetQueueAttributesCommand,
} from '@aws-sdk/client-sqs';

@Injectable()
export class SqsService {
  private readonly logger = new Logger(SqsService.name);
  private readonly sqsClient: SQSClient;

  constructor(private configService: ConfigService) {
    this.sqsClient = new SQSClient({
      region: this.configService.get<string>('AWS_REGION'),
      credentials: {
        accessKeyId: this.configService.get<string>('AWS_ACCESS_KEY_ID'),
        secretAccessKey: this.configService.get<string>('AWS_SECRET_ACCESS_KEY'),
      },
    });
  }

  /**
   * SQS Queue에 메시지 전송
   */
  async sendMessage(
    queueUrl: string,
    message: Record<string, any>,
    messageGroupId?: string, // FIFO Queue용
  ): Promise<string> {
    try {
      const command = new SendMessageCommand({
        QueueUrl: queueUrl,
        MessageBody: JSON.stringify(message),
        MessageAttributes: {
          'event-type': {
            DataType: 'String',
            StringValue: message.eventType || 'unknown',
          },
        },
        ...(messageGroupId && { MessageGroupId: messageGroupId }),
      });

      const response = await this.sqsClient.send(command);
      this.logger.log(`Message sent to ${queueUrl}: ${response.MessageId}`);

      return response.MessageId;
    } catch (error) {
      this.logger.error(`Failed to send message to ${queueUrl}:`, error);
      throw error;
    }
  }

  /**
   * SQS Queue에서 메시지 수신
   */
  async receiveMessages(
    queueUrl: string,
    maxNumberOfMessages: number = 10,
    waitTimeSeconds: number = 20, // Long Polling
  ): Promise<any[]> {
    try {
      const command = new ReceiveMessageCommand({
        QueueUrl: queueUrl,
        MaxNumberOfMessages: maxNumberOfMessages,
        WaitTimeSeconds: waitTimeSeconds,
        MessageAttributeNames: ['All'],
      });

      const response = await this.sqsClient.send(command);

      if (!response.Messages || response.Messages.length === 0) {
        return [];
      }

      return response.Messages.map((msg) => ({
        messageId: msg.MessageId,
        receiptHandle: msg.ReceiptHandle,
        body: JSON.parse(msg.Body || '{}'),
        attributes: msg.MessageAttributes,
      }));
    } catch (error) {
      this.logger.error(`Failed to receive messages from ${queueUrl}:`, error);
      throw error;
    }
  }

  /**
   * 처리 완료된 메시지 삭제
   */
  async deleteMessage(queueUrl: string, receiptHandle: string): Promise<void> {
    try {
      const command = new DeleteMessageCommand({
        QueueUrl: queueUrl,
        ReceiptHandle: receiptHandle,
      });

      await this.sqsClient.send(command);
      this.logger.log(`Message deleted from ${queueUrl}`);
    } catch (error) {
      this.logger.error(`Failed to delete message from ${queueUrl}:`, error);
      throw error;
    }
  }

  /**
   * Queue 상태 확인 (대기 중인 메시지 수 등)
   */
  async getQueueAttributes(queueUrl: string): Promise<Record<string, string>> {
    try {
      const command = new GetQueueAttributesCommand({
        QueueUrl: queueUrl,
        AttributeNames: ['ApproximateNumberOfMessages', 'ApproximateNumberOfMessagesNotVisible'],
      });

      const response = await this.sqsClient.send(command);
      return response.Attributes || {};
    } catch (error) {
      this.logger.error(`Failed to get queue attributes from ${queueUrl}:`, error);
      throw error;
    }
  }
}

    멱등성 보장¶

    // Lambda 함수에서 멱등성 보장
exports.handler = async (event) => {
  const dynamodb = new DynamoDBClient({});

  for (const record of event.Records) {
    const message = JSON.parse(record.body);
    const orderId = message.data.orderId;

    // 멱등성 키로 중복 처리 방지
    const idempotencyKey = `email-${orderId}`;

    try {
      // DynamoDB 조건부 쓰기로 중복 방지
      await dynamodb.putItem({
        TableName: 'processed-events',
        Item: {
          idempotencyKey: { S: idempotencyKey },
          processedAt: { S: new Date().toISOString() },
        },
        ConditionExpression: 'attribute_not_exists(idempotencyKey)',
      });

      // 이메일 발송 로직
      await sendEmail(message);

    } catch (error) {
      if (error.name === 'ConditionalCheckFailedException') {
        console.log(`Already processed: ${idempotencyKey}`);
        continue; // 이미 처리된 메시지는 스킵
      }
      throw error; // 다른 에러는 재시도
    }
  }
};

    // NestJS에서 구조화된 로깅
import { Logger } from '@nestjs/common';

@Injectable()
export class SnsService {
  private readonly logger = new Logger(SnsService.name);

  async publishMessage(topicArn: string, message: any) {
    const correlationId = this.generateCorrelationId();

    this.logger.log({
      message: 'Publishing message to SNS',
      topicArn,
      correlationId,
      eventType: message.eventType,
      timestamp: new Date().toISOString(),
    });

    try {
      const messageId = await this.publish(topicArn, message);

      this.logger.log({
        message: 'Message published successfully',
        topicArn,
        messageId,
        correlationId,
      });

      return messageId;
    } catch (error) {
      this.logger.error({
        message: 'Failed to publish message',
        topicArn,
        correlationId,
        error: error.message,
        stack: error.stack,
      });

      throw error;
    }
  }
}

    일반적인 문제 및 해결책¶

    문제 1: 메시지가 Lambda에 도달하지 않음¶

    증상: - SNS에 메시지 발행은 성공 - SQS에 메시지가 쌓임 - Lambda가 트리거되지 않음

    원인: - Lambda 이벤트 소스 매핑 설정 오류 - IAM 권한 부족 - SQS Queue와 Lambda 리전 불일치

    해결: 

    # Lambda 이벤트 소스 매핑 확인
aws lambda list-event-source-mappings --function-name send-email

# IAM 권한 확인
aws iam get-role-policy --role-name lambda-execution-role --policy-name sqs-access

# SQS Queue ARN 확인
aws sqs get-queue-attributes --queue-url <queue-url> --attribute-names QueueArn

    문제 2: 메시지가 DLQ로 계속 이동¶

    증상: - Lambda 함수가 계속 실패 - 메시지가 maxReceiveCount 후 DLQ로 이동

    원인: - Lambda 함수 내부 에러 - 타임아웃 설정 부족 - 메시지 형식 불일치

    해결: 

    // Lambda 함수에서 상세한 에러 로깅
exports.handler = async (event) => {
  console.log('Event:', JSON.stringify(event, null, 2));

  try {
    for (const record of event.Records) {
      const message = JSON.parse(record.body);
      console.log('Message:', JSON.stringify(message, null, 2));

      // 메시지 검증
      if (!message.data || !message.data.orderId) {
        throw new Error('Invalid message format');
      }

      await processMessage(message);
    }
  } catch (error) {
    console.error('Error details:', {
      message: error.message,
      stack: error.stack,
      event: JSON.stringify(event),
    });
    throw error;
  }
};

    문제 3: 메시지 순서가 보장되지 않음¶

    증상: - Standard Queue 사용 시 메시지 순서가 섞임

    해결: - FIFO Queue 사용 - MessageGroupId를 통한 그룹화

    // FIFO Queue에 순서 보장 메시지 전송
await sqsClient.send(new SendMessageCommand({
  QueueUrl: 'https://sqs.region.amazonaws.com/account/queue.fifo',
  MessageBody: JSON.stringify(message),
  MessageGroupId: orderId, // 같은 주문은 같은 그룹
  MessageDeduplicationId: `${orderId}-${timestamp}`, // 중복 제거
}));
