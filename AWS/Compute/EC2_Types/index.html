    AWS EC2 인스턴스 유형¶

    배경¶

    AWS EC2(Elastic Compute Cloud)는 클라우드에서 가상 서버를 제공하는 핵심 서비스입니다. 다양한 워크로드와 요구사항에 맞춰 최적화된 인스턴스 유형을 제공하여, 사용자는 애플리케이션의 특성에 맞는 적절한 컴퓨팅 리소스를 선택할 수 있습니다.

    EC2 인스턴스의 필요성¶

    • 확장성: 필요에 따라 리소스를 동적으로 조정
    • 비용 효율성: 사용한 만큼만 비용 지불
    • 다양성: 워크로드에 최적화된 다양한 인스턴스 유형
    • 관리 편의성: AWS가 하드웨어 관리 담당

    AWS의 컴퓨팅 서비스 구분¶

    EC2 (Elastic Compute Cloud) - 완전한 제어권을 가진 가상 서버 - 운영체제 선택 및 설정 가능 - 지속적으로 실행되는 애플리케이션에 적합

    Lambda - 서버리스 컴퓨팅 서비스 - 코드 실행에 필요한 인프라를 AWS가 자동 관리 - 이벤트 기반으로 실행 - 사용한 시간만큼만 비용 지불

    핵심¶

    인스턴스 유형 분류¶

    AWS EC2 인스턴스는 크게 5가지 카테고리로 분류됩니다:

    인스턴스 유형 특징 대표적인 용도
    범용 (General Purpose) CPU, 메모리, 네트워크 성능 균형 웹 서버, 애플리케이션 서버
    컴퓨팅 최적화 (Compute Optimized) 고성능 CPU 제공 고성능 연산 작업, 게임 서버
    메모리 최적화 (Memory Optimized) 높은 RAM 제공 대용량 인메모리 데이터베이스
    스토리지 최적화 (Storage Optimized) 고속 디스크 I/O 제공 빅데이터, 로그 분석
    가속 컴퓨팅 (Accelerated Computing) GPU 또는 FPGA 제공 AI, 머신러닝, 그래픽 렌더링

    1. 범용 (General Purpose) 인스턴스¶

    범용 인스턴스는 컴퓨팅, 메모리, 네트워킹 리소스를 균형있게 제공하는 유형입니다. 특정 리소스에 치우치지 않아 대부분의 일반적인 워크로드에 적합합니다.

    T 시리즈 (버스트 가능한 성능)¶

    특징: - 버스팅 메커니즘: 평상시에는 기본 CPU 성능으로 동작하다가 필요시 CPU 성능 급증 - CPU 크레딧 시스템: 유휴 시간에 크레딧을 축적하고, 부하가 높을 때 사용 - 가변적 워크로드에 최적: 트래픽이 일정하지 않은 애플리케이션에 적합 - 비용 효율성: 다른 인스턴스 대비 저렴한 비용

    사용 사례: - 개발 및 테스트 환경 - 소규모 웹 서버 - 마이크로서비스 - 코드 저장소

    주의사항: - 지속적으로 높은 CPU 사용률이 필요한 경우 부적합 - CPU 크레딧이 소진되면 기본 성능으로 제한 - 크레딧 모니터링 필요

    M 시리즈 (균형잡힌 성능)¶

    특징: - 균형잡힌 컴퓨팅/메모리 비율: CPU와 메모리가 1:4 비율로 제공 - 안정적인 성능: 버스팅 없이 지속적으로 일정한 성능 제공 - 중규모 이상의 애플리케이션: 프로덕션 환경에 적합

    사용 사례: - 데이터베이스 서버 - 애플리케이션 서버 - 엔터프라이즈 애플리케이션 - 백엔드 서버

    인스턴스 크기와 비용¶

    인스턴스 크기는 nano → micro → small → medium → large → xlarge → 2xlarge 순으로 증가하며, 각 단계마다 리소스와 비용이 약 2배씩 증가합니다.

    예시: T3 시리즈
t3.nano    (2 vCPU, 0.5GB RAM) → $3.8/월
t3.micro   (2 vCPU, 1GB RAM)   → $7.5/월  (2배)
t3.small   (2 vCPU, 2GB RAM)   → $15/월   (2배)
t3.medium  (2 vCPU, 4GB RAM)   → $30/월   (2배)
t3.large   (2 vCPU, 8GB RAM)   → $60/월   (2배)
t3.xlarge  (4 vCPU, 16GB RAM)  → $120/월  (2배)

    인스턴스 크기 선택 팁: - 큰 사이즈에서 시작하여 축소 권장: 초기에는 여유있게 시작하여 모니터링 후 적정 크기 파악 - 오버 프로비저닝보다 언더 프로비저닝이 더 위험: 성능 문제는 사용자 경험에 직접 영향 - CloudWatch 메트릭 활용: CPU, 메모리 사용률을 지속적으로 모니터링

    2. 컴퓨팅 최적화 (Compute Optimized) 인스턴스¶

    고성능 CPU를 제공하여 컴퓨팅 성능이 중요한 애플리케이션에 적합합니다.

    주요 시리즈¶

    • C 시리즈 (C6g, C5, C4): 고성능 연산 작업에 최적화

    사용 사례¶

    • 고성능 웹 서버
    • 게임 서버
    • 배치 처리 작업
    • 과학 계산

    3. 메모리 최적화 (Memory Optimized) 인스턴스¶

    RAM이 많아 대규모 데이터 처리 및 캐싱 작업에 최적화된 인스턴스입니다.

    주요 시리즈¶

    • R 시리즈 (R6g, R5, R4): 인메모리 데이터베이스에 적합
    • X 시리즈 (X1, X2): 고성능 SAP, 인메모리 애플리케이션에 적합

    사용 사례¶

    • Redis, Memcached 등 인메모리 데이터베이스
    • SAP HANA
    • 대용량 데이터 분석
    • 실시간 빅데이터 처리

    4. 스토리지 최적화 (Storage Optimized) 인스턴스¶

    고속 디스크 I/O 성능을 제공하여 빅데이터 및 로그 분석에 적합합니다.

    주요 시리즈¶

    • I 시리즈 (I3, I4): NVMe SSD를 통한 빠른 데이터 접근
    • D 시리즈 (D2): 대용량 데이터 저장 및 분석에 최적

    사용 사례¶

    • 빅데이터 워크로드
    • 로그 분석
    • 데이터 웨어하우스
    • 고성능 데이터베이스

    5. 가속 컴퓨팅 (Accelerated Computing) 인스턴스¶

    GPU 또는 FPGA를 사용하여 머신러닝, AI, 그래픽 처리 등에 최적화된 인스턴스입니다.

    주요 시리즈¶

    • P 시리즈 (P4, P3): GPU를 활용한 머신러닝 및 딥러닝 모델 학습
    • G 시리즈 (G5, G4): 그래픽 렌더링 및 동영상 처리에 최적

    사용 사례¶

    • 딥러닝 모델 학습
    • 그래픽 렌더링
    • 동영상 인코딩
    • 과학적 시뮬레이션

    예시¶

    실제 사용 사례별 인스턴스 선택¶

    웹 애플리케이션 서버¶

    • 권장 인스턴스: T3, M5 시리즈
    • 선택 이유: CPU, 메모리, 네트워크 성능이 균형 있게 제공되어 웹 요청 처리에 최적화
    • 구체적 사례:
    • 소규모 웹사이트: t3.micro (월 $7.5)
    • 중간 규모 웹사이트: t3.medium (월 $30)
    • 대규모 웹사이트: m5.large (월 $70)

    데이터베이스 서버¶

    • 권장 인스턴스: R5, R6g 시리즈
    • 선택 이유: 높은 메모리 용량으로 데이터 캐싱과 빠른 쿼리 처리 가능
    • 구체적 사례:
    • MySQL/PostgreSQL: r5.large (16GB RAM, 월 $90)
    • Redis/Memcached: r5.xlarge (32GB RAM, 월 $180)
    • SAP HANA: x1e.32xlarge (3.8TB RAM, 월 $26,688)

    빅데이터 분석¶

    • 권장 인스턴스: I3, D2 시리즈
    • 선택 이유: 고속 스토리지 I/O로 대용량 데이터 처리 최적화
    • 구체적 사례:
    • 로그 분석: i3.large (475GB NVMe SSD, 월 $156)
    • 데이터 웨어하우스: d2.xlarge (6TB HDD, 월 $690)

    머신러닝/AI¶

    • 권장 인스턴스: P3, P4, G4 시리즈
    • 선택 이유: GPU 가속으로 딥러닝 모델 학습 및 추론 성능 향상
    • 구체적 사례:
    • 모델 학습: p3.2xlarge (V100 GPU, 월 $3,060)
    • 추론 서비스: g4dn.xlarge (T4 GPU, 월 $526)
    • 대규모 학습: p4d.24xlarge (8x A100 GPU, 월 $32.77/시간)

    인스턴스 구매 옵션¶

    EC2 인스턴스는 사용 패턴과 예산에 따라 다양한 구매 옵션을 제공합니다.

    1. 온디맨드 (On-Demand) 인스턴스¶

    특징: - 선결제 없이 사용한 만큼만 비용 부과 - 시간 단위 또는 초 단위로 과금 - 약정 없이 언제든 시작/중지 가능

    적합한 경우: - 변동성이 높은 워크로드 - 단기 프로젝트 - 개발 및 테스트 환경 - 예측 불가능한 트래픽 패턴

    비용 예시: t3.medium 기준 $0.0416/시간

    2. 예약형 인스턴스 (Reserved Instances)¶

    특징: - 1년 또는 3년 약정으로 최대 60-75% 할인 - 특정 인스턴스 패밀리 및 가용영역 고정 - 선결제, 부분 선결제, 무선결제 옵션 제공

    적합한 경우: - 안정적인 장기 워크로드 - 24/7 운영이 필요한 프로덕션 서버 - 예측 가능한 사용 패턴 - 데이터베이스 서버

    비용 절감: 온디맨드 대비 최대 72% 할인

    3. Savings Plans¶

    특징: - 시간당 사용량($)에 대한 약정 - 인스턴스 패밀리, 크기, 리전 변경 가능 - 예약형보다 유연한 장기 워크로드 옵션

    적합한 경우: - 유연한 장기 워크로드 - 인스턴스 타입 변경이 필요한 경우 - 여러 리전에서 운영하는 경우

    비용 절감: 온디맨드 대비 최대 72% 할인

    4. 스팟 인스턴스 (Spot Instances)¶

    특징: - AWS의 미사용 EC2 용량을 저렴하게 제공 - 최대 90% 할인 - 언제든지 중단될 수 있음 (2분 전 알림) - 입찰 방식으로 가격 결정

    적합한 경우: - 중단 가능한 배치성 워크로드 - 데이터 분석 및 처리 - CI/CD 파이프라인 - 렌더링 작업

    스팟 인스턴스 활용 전략: - 인스턴스 다각화: 여러 인스턴스 타입과 가용영역 사용 - 체크포인팅 활용: 주기적으로 작업 상태 저장 - 스팟 인스턴스 수명 주기 관리: 종료 2분 전 알림을 받아 새 인스턴스 자동 시작 - 스팟 플릿 사용: 여러 스팟 인스턴스를 하나의 논리적 그룹으로 관리

    비용 절감: 온디맨드 대비 최대 90% 할인

    구매 옵션 비교¶

    옵션 할인율 약정 중단 가능 적합한 워크로드
    온디맨드 0% 없음 아니오 변동성 높은 워크로드
    예약형 최대 72% 1-3년 아니오 안정적 장기 워크로드
    Savings Plans 최대 72% 1-3년 아니오 유연한 장기 워크로드
    스팟 최대 90% 없음 예 중단 가능한 배치 작업

    인스턴스 연결 방법¶

    EC2 인스턴스에 접속하는 다양한 방법이 있으며, 보안 수준과 편의성에 따라 선택할 수 있습니다.

    1. SSH (Secure Shell)¶

    특징: - 전통적인 방식의 원격 접속 - 키 페어(공개키/개인키) 기반 인증 - 22번 포트 사용

    보안 권장사항: - Public IP 직접 노출은 권장하지 않음 - Private Subnet에 배치 후 Bastion Host 또는 VPN을 통한 접근 권장 - 보안 그룹에서 특정 IP만 허용

    # SSH 연결 예시
ssh -i "my-key.pem" ec2-user@ec2-xx-xxx-xxx-xxx.compute.amazonaws.com

    2. EC2 Instance Connect¶

    특징: - 브라우저 기반 SSH 연결 - 일회용 SSH 키 자동 생성 - AWS 콘솔에서 직접 접속 가능

    장점: - 별도의 키 관리 불필요 - 브라우저만 있으면 접속 가능 - IAM 정책으로 접근 제어

    단점: - 퍼블릭 서브넷에만 사용 가능 - 22번 포트가 열려있어야 함

    3. Session Manager (권장)¶

    특징: - 인터넷 연결 불필요 - 인스턴스에 Session Manager Agent만 설치하면 됨 - IAM 정책으로 세밀한 접근 제어 - 모든 세션 기록 자동 저장 (CloudWatch Logs, S3)

    장점: - 22번 포트를 열지 않아도 됨 (보안 강화) - Private Subnet의 인스턴스에도 접근 가능 - Bastion Host 불필요 - 감사 및 규정 준수에 유리

    설정 방법: 1. 인스턴스에 SSM Agent 설치 (Amazon Linux는 기본 설치) 2. 인스턴스에 IAM 역할 부여 (AmazonSSMManagedInstanceCore 정책) 3. AWS Systems Manager 콘솔에서 Session Manager로 접속

    # AWS CLI를 통한 Session Manager 접속
aws ssm start-session --target i-1234567890abcdef0

    연결 방법 비교¶

    방법 인터넷 필요 포트 보안 수준 감사 로그 권장도
    SSH 예 22 보통 수동 설정 필요 ⭐⭐
    Instance Connect 예 22 보통 IAM 기반 ⭐⭐⭐
    Session Manager 아니오 불필요 높음 자동 저장 ⭐⭐⭐⭐⭐

    Auto Scaling과 로드 밸런싱¶

    스케일링의 종류¶

    1. Scale Up/Down (수직 확장) - 인스턴스의 리소스 용량을 더 크게 또는 작게 조정 - 예: t3.medium → t3.large (Scale Up) 또는 t3.large → t3.medium (Scale Down) - 인스턴스 중지 및 타입 변경 필요 - 단일 인스턴스의 성능 한계 존재

    2. Scale Out/In (수평 확장) - 인스턴스의 개수를 추가(Scale Out) 또는 제거(Scale In) - 예: 인스턴스 3대 → 5대 (Scale Out) - 무중단 확장 가능 - 이론적으로 무한 확장 가능

    Auto Scaling 동작 방식¶

    기본 개념: - CloudWatch 메트릭 기반 자동 확장/축소 - 예: CPU 사용률 70% 초과 시 인스턴스 1대씩 추가 - Launch Template 또는 Launch Configuration 기반 인스턴스 복제

    Auto Scaling 구성 요소: 1. Launch Template (시작 템플릿) - 인스턴스 타입, AMI, 보안 그룹, 키 페어 등의 설정 템플릿 - Auto Scaling 시 이 템플릿을 기반으로 새 인스턴스 생성 - 버전 관리 가능

    1. Auto Scaling Group
    2. 최소/희망/최대 인스턴스 수 설정
    3. 스케일링 정책 정의

    4. 헬스 체크 설정

    5. Scaling Policy

    6. Target Tracking: 목표 값 유지 (예: CPU 50% 유지)
    7. Step Scaling: 단계별 조정 (예: CPU 70%면 1대, 90%면 2대 추가)
    8. Scheduled Scaling: 시간 기반 조정 (예: 평일 오전 9시에 인스턴스 증가)

    Auto Scaling과 로드 밸런서 통합: 

    사용자 요청
    ↓
로드 밸런서 (ALB/NLB)
    ↓
Auto Scaling Group
    ├─ 인스턴스 1 (Launch Template 기반)
    ├─ 인스턴스 2 (Launch Template 기반)
    └─ 인스턴스 3 (Launch Template 기반)

    동작 시나리오: 1. 로드 밸런서가 트래픽을 여러 인스턴스에 분산 2. CloudWatch가 평균 CPU 사용률 모니터링 3. CPU 70% 초과 시 Auto Scaling이 Launch Template 기반으로 새 인스턴스 생성 4. 새 인스턴스가 준비되면 로드 밸런서가 자동으로 트래픽 분산 시작 5. CPU 30% 미만 시 Auto Scaling이 불필요한 인스턴스 제거

    AWS 운영 지원 서비스¶

    AWS 빌드업 프로그램¶

    • 초기 AWS 도입 시 기술 지원
    • 아키텍처 설계 가이드
    • 베스트 프랙티스 제공

    MSP (Managed Service Provider)¶

    • AWS 환경의 지속적인 운영 지원
    • 24/7 모니터링 및 장애 대응
    • 비용 최적화 컨설팅
    • 보안 및 규정 준수 지원

    활용 시기: - 서비스가 절대 멈추면 안 되는 경우 - 내부에 AWS 전문 인력이 부족한 경우 - 복잡한 아키텍처 운영 시

    비용 최적화 전략¶

    Reserved Instances 활용¶

    • 1년 약정: 최대 40% 할인
    • 3년 약정: 최대 72% 할인
    • 전체 선불: 추가 할인
    • 적용 대상: 안정적인 워크로드, 24/7 운영 서비스

    Spot Instances 활용¶

    • 할인율: 최대 90% 할인
    • 적용 대상: 배치 작업, 개발/테스트 환경, 중단 가능한 워크로드
    • 주의사항: AWS가 언제든 인스턴스 종료 가능

    Auto Scaling과의 연계¶

    • 수평 확장: 트래픽 증가 시 인스턴스 자동 추가
    • 수직 확장: 인스턴스 타입 자동 변경
    • 비용 절약: 필요할 때만 리소스 사용

    운영 팁¶

    인스턴스 선택 가이드라인¶

    워크로드별 선택 기준¶

    CPU 집약적 워크로드 - 권장 시리즈: C 시리즈 (C5, C6g) - 특징: 고성능 CPU, 높은 네트워크 성능 - 적용 사례: 배치 처리, 게임 서버, 과학 계산, 고성능 웹 서버

    메모리 집약적 워크로드 - 권장 시리즈: R 시리즈 (R5, R6g), X 시리즈 (X1, X2) - 특징: 높은 메모리 대역폭, 대용량 RAM - 적용 사례: Redis, Memcached, SAP HANA, 대용량 데이터 분석

    스토리지 집약적 워크로드 - 권장 시리즈: I 시리즈 (I3, I4), D 시리즈 (D2) - 특징: 고속 NVMe SSD, 높은 I/O 성능 - 적용 사례: 빅데이터, 로그 분석, 데이터 웨어하우스, 고성능 데이터베이스

    GPU 가속 워크로드 - 권장 시리즈: P 시리즈 (P3, P4), G 시리즈 (G4, G5) - 특징: NVIDIA GPU, 높은 메모리 대역폭 - 적용 사례: 머신러닝, 딥러닝, 그래픽 렌더링, AI 모델 학습

    범용 워크로드 - 권장 시리즈: T 시리즈 (T3, T4), M 시리즈 (M5, M6g) - 특징: CPU, 메모리, 네트워크 성능 균형 - 적용 사례: 웹 서버, 애플리케이션 서버, 개발 환경, 마이크로서비스

    성능 모니터링 및 최적화¶

    CloudWatch를 활용한 모니터링¶

    • CPU 사용률: 80% 이상 지속 시 업그레이드 고려
    • 메모리 사용률: 80% 이상 지속 시 메모리 최적화 인스턴스 검토
    • 네트워크 I/O: 대역폭 한계 도달 시 더 큰 인스턴스 고려
    • 디스크 I/O: IOPS 한계 도달 시 스토리지 최적화 인스턴스 검토

    성능 최적화 전략¶

    • 수직 확장: 인스턴스 타입 업그레이드
    • 수평 확장: Auto Scaling을 통한 인스턴스 추가
    • 로드 밸런싱: 트래픽 분산으로 단일 인스턴스 부하 감소
    • 캐싱: ElastiCache를 통한 데이터베이스 부하 감소

    보안 고려사항¶

    네트워크 보안¶

    • 보안 그룹: 최소 권한 원칙 적용
    • VPC: 프라이빗 서브넷 활용
    • NACL: 서브넷 레벨 보안 강화
    • WAF: 웹 애플리케이션 방화벽 적용

    인스턴스 보안¶

    • IAM 역할: 인스턴스별 최소 권한 부여
    • 시스템 패치: 정기적인 보안 업데이트
    • 모니터링: CloudTrail, GuardDuty를 통한 이상 행위 탐지
    • 암호화: EBS 볼륨 암호화, 전송 중 데이터 암호화

    결론¶

    AWS EC2 인스턴스 유형 선택은 애플리케이션의 성능 요구사항, 비용 제약, 그리고 워크로드 특성을 종합적으로 고려해야 하는 중요한 결정입니다.

    각 인스턴스 시리즈는 특정 워크로드에 최적화되어 있으므로, 애플리케이션의 특성을 정확히 분석하여 적절한 인스턴스 유형을 선택하는 것이 핵심입니다. 또한 비용 최적화를 위해 Reserved Instances와 Spot Instances를 적절히 활용하고, CloudWatch를 통한 지속적인 모니터링으로 성능을 최적화할 수 있습니다.

    보안 설정과 함께 안정적이고 효율적인 클라우드 인프라를 구축하여 비즈니스 요구사항을 충족시킬 수 있습니다.
