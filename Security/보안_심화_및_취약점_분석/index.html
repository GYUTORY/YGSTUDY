    보안 심화 및 취약점 분석¶

    개요¶

    보안 취약점 분석과 대응 전략을 다룬다.

    OWASP Top 10 대응¶

    SQL Injection¶

    정의: SQL Injection은 사용자 입력을 SQL 쿼리에 직접 삽입해 데이터베이스를 조작하는 공격이다.

    공격 원리: 1. 사용자 입력이 SQL 쿼리 문자열에 직접 포함됨 2. 공격자가 악의적인 SQL 코드를 입력 3. 데이터베이스가 해당 코드를 실행 4. 데이터 유출, 수정, 삭제 발생

    공격 예시: 

    // 공격자가 userId에 입력: "1' OR '1'='1"
const query = `SELECT * FROM users WHERE id = ${userId}`;
// 실행되는 쿼리: SELECT * FROM users WHERE id = 1' OR '1'='1
// 결과: 모든 사용자 정보 유출

// 공격자가 userId에 입력: "1; DROP TABLE users; --"
const query = `DELETE FROM users WHERE id = ${userId}`;
// 실행되는 쿼리: DELETE FROM users WHERE id = 1; DROP TABLE users; --
// 결과: users 테이블 삭제

    방어 방법:

    취약한 코드: 

    const query = `SELECT * FROM users WHERE id = ${userId}`;
db.query(query); // SQL Injection 취약

    안전한 코드: 

    // Prepared Statement 사용
const query = 'SELECT * FROM users WHERE id = ?';
db.query(query, [userId]); // 안전

// ORM 사용 (권장)
const user = await User.findOne({ where: { id: userId } });

    실무 팁: - ORM을 사용하면 대부분 자동으로 방어된다 - 직접 쿼리를 작성할 때는 Prepared Statement를 사용해야 한다 - 입력값 검증도 함께 수행한다

    XSS (Cross-Site Scripting)¶

    정의: XSS는 웹 애플리케이션에 악의적인 JavaScript 코드를 삽입해 사용자 브라우저에서 실행하는 공격이다.

    공격 유형:

    1. Stored XSS (저장형): 악의적인 스크립트가 데이터베이스에 저장되고, 다른 사용자가 해당 페이지를 볼 때 실행된다.

    // 공격자가 댓글에 입력: <script>alert(document.cookie)</script>
// 데이터베이스에 저장됨
// 다른 사용자가 댓글을 볼 때 스크립트 실행

    2. Reflected XSS (반사형): URL 파라미터나 폼 입력에 포함된 스크립트가 즉시 실행된다.

    // 공격자가 URL에 입력: ?search=<script>alert('XSS')</script>
// 서버가 검색 결과에 그대로 포함
// 사용자 브라우저에서 스크립트 실행

    3. DOM-based XSS: 클라이언트 측 JavaScript가 DOM을 조작할 때 발생한다.

    // 취약한 코드
document.getElementById('output').innerHTML = userInput;
// 공격자가 userInput에 <img src=x onerror="alert('XSS')"> 입력

    공격 피해: - 쿠키 탈취 (세션 하이재킹) - 사용자 정보 유출 - 피싱 공격 - 악성 코드 실행

    방어 방법:

    입력 검증 및 이스케이핑: 

    function sanitizeInput(input) {
  return input
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#x27;')
    .replace(/\//g, '&#x2F;');
}

// 또는 라이브러리 사용 (권장)
const DOMPurify = require('isomorphic-dompurify');
const clean = DOMPurify.sanitize(userInput);

    Content Security Policy 설정: 

    app.use(helmet.contentSecurityPolicy({
  directives: {
    defaultSrc: ["'self'"],
    scriptSrc: ["'self'"], // 'unsafe-inline' 제거
    styleSrc: ["'self'", "'unsafe-inline'"]
  }
}));

    실무 팁: - innerHTML 대신 textContent 사용 - React, Vue 같은 프레임워크는 기본적으로 XSS 방어 - CSP 헤더를 설정해 스크립트 실행 제한

    CSRF (Cross-Site Request Forgery)¶

    정의: CSRF는 사용자가 의도하지 않은 요청을 다른 사이트에서 대신 보내는 공격이다. 사용자가 로그인한 상태에서 악의적인 사이트가 사용자의 권한으로 요청을 보낸다.

    공격 원리: 1. 사용자가 은행 사이트에 로그인 (세션 쿠키 저장) 2. 사용자가 악의적인 사이트 방문 3. 악의적인 사이트가 은행 사이트로 자동 요청 전송 4. 브라우저가 세션 쿠키를 자동으로 포함 5. 사용자 권한으로 요청 실행 (이체, 정보 변경 등)

    공격 예시: 

    <!-- 악의적인 사이트의 HTML -->
<img src="https://bank.com/transfer?to=attacker&amount=1000" />

<!-- 또는 자동 실행 스크립트 -->
<script>
  fetch('https://bank.com/transfer', {
    method: 'POST',
    body: JSON.stringify({ to: 'attacker', amount: 1000 }),
    credentials: 'include' // 쿠키 자동 포함
  });
</script>

    공격 조건: - 사용자가 대상 사이트에 로그인한 상태 - 쿠키 기반 인증 사용 - 상태 변경 요청 (GET이 아닌 POST, PUT, DELETE)

    방어 방법:

    1. CSRF 토큰 사용: 

    const csrf = require('csurf');
const csrfProtection = csrf({ cookie: true });

// 토큰 생성 (폼 렌더링 시)
app.get('/form', csrfProtection, (req, res) => {
  res.render('form', { csrfToken: req.csrfToken() });
});

// 토큰 검증 (요청 처리 시)
app.post('/transfer', csrfProtection, (req, res) => {
  // CSRF 토큰 검증 완료
  processTransfer(req.body);
});

    2. SameSite 쿠키 속성: 

    app.use(session({
  cookie: {
    sameSite: 'strict', // 또는 'lax'
    secure: true,
    httpOnly: true
  }
}));

    3. Referer 검증: 

    app.post('/transfer', (req, res) => {
  const referer = req.get('Referer');
  if (!referer || !referer.startsWith('https://yourdomain.com')) {
    return res.status(403).send('Invalid request origin');
  }
  processTransfer(req.body);
});

    실무 팁: - 상태 변경 요청에는 항상 CSRF 토큰 사용 - SameSite 쿠키 속성으로 추가 방어 - GET 요청은 상태를 변경하지 않도록 설계

    주요 보안 취약점 추가¶

    Directory Traversal (경로 탐색)¶

    정의: Directory Traversal은 ../ 같은 경로 조작을 통해 시스템 파일에 접근하는 공격이다.

    공격 예시: 

    // 취약한 코드
app.get('/file', (req, res) => {
  const filename = req.query.name;
  res.sendFile(`/uploads/${filename}`); // 위험
});

// 공격자가 입력: ?name=../../../etc/passwd
// 결과: /etc/passwd 파일 유출

    방어 방법: 

    const path = require('path');

app.get('/file', (req, res) => {
  const filename = req.query.name;
  const safePath = path.resolve('/uploads', filename);

  // 경로 검증
  if (!safePath.startsWith('/uploads')) {
    return res.status(403).send('Invalid path');
  }

  res.sendFile(safePath);
});

    Command Injection¶

    정의: Command Injection은 시스템 명령어를 실행하는 함수에 사용자 입력을 직접 전달해 악의적인 명령을 실행하는 공격이다.

    공격 예시: 

    // 취약한 코드
const { exec } = require('child_process');
const userInput = req.query.filename;
exec(`cat ${userInput}`, (error, stdout) => {
  // ...
});

// 공격자가 입력: file.txt; rm -rf /
// 결과: 시스템 파일 삭제

    방어 방법: 

    // 입력값 검증
const filename = req.query.filename;
if (!/^[a-zA-Z0-9._-]+$/.test(filename)) {
  return res.status(400).send('Invalid filename');
}

// 또는 exec 대신 execFile 사용
const { execFile } = require('child_process');
execFile('cat', [filename], (error, stdout) => {
  // ...
});

    XXE (XML External Entity)¶

    정의: XXE는 XML 파서가 외부 엔티티를 처리할 때 발생하는 취약점이다.

    공격 예시: 

    <!-- 공격자가 입력한 XML -->
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<foo>&xxe;</foo>

    방어 방법: 

    const libxml = require('libxmljs');

const xml = req.body;
const doc = libxml.parseXmlString(xml, {
  noent: false, // 외부 엔티티 비활성화
  noblanks: true
});

    SSRF (Server-Side Request Forgery)¶

    정의: SSRF는 서버가 사용자 입력으로 받은 URL에 직접 요청을 보낼 때 발생한다. 공격자가 내부 네트워크나 클라우드 메타데이터 서비스에 접근하는 데 쓰인다. OWASP Top 10 2021에서 신규 진입한 항목이다.

    왜 위험한가: 서버 입장에서 보내는 요청이기 때문에 내부 방화벽을 우회한다. AWS에서는 169.254.169.254 메타데이터 엔드포인트를 통해 IAM 크레덴셜을 탈취당하는 사고가 실제로 발생한다. 2019년 Capital One 해킹이 대표 사례다.

    공격 예시: 

    // 취약한 코드 - URL 프리뷰 기능
app.post('/preview', async (req, res) => {
  const { url } = req.body;
  const response = await fetch(url); // 사용자 입력 URL을 그대로 요청
  const html = await response.text();
  res.send(html);
});

// 공격자가 보내는 요청
// POST /preview
// { "url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/" }
// → AWS IAM 크레덴셜 유출

// 내부 서비스 스캔
// { "url": "http://10.0.0.1:8080/admin" }
// → 내부 관리자 페이지 접근

    방어 방법: 

    const { URL } = require('url');
const dns = require('dns').promises;
const ipaddr = require('ipaddr.js');

async function validateUrl(inputUrl) {
  const parsed = new URL(inputUrl);

  // 1. 프로토콜 제한 - http/https만 허용
  if (!['http:', 'https:'].includes(parsed.protocol)) {
    throw new Error('허용되지 않는 프로토콜');
  }

  // 2. DNS 조회 후 IP 대역 검증
  const addresses = await dns.resolve4(parsed.hostname);
  for (const addr of addresses) {
    const ip = ipaddr.parse(addr);
    const range = ip.range();
    // private, loopback, link-local 등 내부 대역 차단
    if (range !== 'unicast') {
      throw new Error(`내부 IP 대역 접근 차단: ${addr} (${range})`);
    }
  }

  // 3. 허용 도메인 화이트리스트 (가능한 경우)
  const allowedDomains = ['api.example.com', 'cdn.example.com'];
  if (!allowedDomains.includes(parsed.hostname)) {
    throw new Error('허용되지 않는 도메인');
  }

  return parsed.toString();
}

app.post('/preview', async (req, res) => {
  try {
    const safeUrl = await validateUrl(req.body.url);
    const response = await fetch(safeUrl);
    res.send(await response.text());
  } catch (err) {
    res.status(400).send(err.message);
  }
});

    실무에서 놓치는 부분: - DNS rebinding 공격: 첫 번째 DNS 조회에서는 외부 IP를 반환하고, 두 번째 조회에서 내부 IP를 반환하는 방식이다. DNS 조회 결과를 캐싱하거나, 실제 연결 시점에 IP를 다시 검증해야 한다. - URL 파서 우회: http://127.0.0.1 대신 http://0x7f000001, http://2130706433, http://127.1 같은 표현으로 우회할 수 있다. IP 파싱 라이브러리를 사용해서 정규화한 후 비교해야 한다. - 리다이렉트 체이닝: 외부 URL이 302로 내부 IP로 리다이렉트하는 경우가 있다. HTTP 클라이언트의 리다이렉트 추적을 비활성화하거나, 리다이렉트된 URL도 검증해야 한다.

    Insecure Deserialization (안전하지 않은 역직렬화)¶

    정의: 신뢰할 수 없는 데이터를 역직렬화할 때 공격자가 객체를 조작해서 임의 코드를 실행하거나 권한을 탈취하는 취약점이다. Java, PHP, Python, Node.js 가리지 않고 발생한다.

    왜 위험한가: 역직렬화는 바이트 스트림에서 객체를 복원하는 과정인데, 이 과정에서 생성자나 매직 메서드가 실행된다. 공격자가 직렬화된 데이터를 조작하면 서버에서 임의 코드가 실행된다. 2017년 Apache Struts 취약점(Equifax 사태)이 역직렬화 관련이었다.

    Node.js에서의 공격 예시: 

    // 취약한 코드 - node-serialize 사용
const serialize = require('node-serialize');

app.post('/session', (req, res) => {
  const sessionData = Buffer.from(req.cookies.session, 'base64').toString();
  const session = serialize.unserialize(sessionData); // 위험
  res.send(`Welcome ${session.username}`);
});

// 공격자가 보내는 쿠키 (base64 인코딩 전)
// {"username":"admin","role":"_$$ND_FUNC$$_function(){require('child_process').exec('cat /etc/passwd')}()"}
// → node-serialize의 IIFE 패턴으로 임의 코드 실행

    Java에서의 공격 원리: 

    // 취약한 코드
ObjectInputStream ois = new ObjectInputStream(request.getInputStream());
Object obj = ois.readObject(); // 사용자 입력을 바로 역직렬화

// 공격자가 Apache Commons Collections의 가젯 체인을 이용해
// Runtime.getRuntime().exec()를 호출하는 직렬화된 객체를 보낸다
// ysoserial 같은 도구로 페이로드를 자동 생성할 수 있다

    방어 방법: 

    // 1. 역직렬화 대신 JSON 사용
// 나쁜 예
const data = serialize.unserialize(userInput);

// 좋은 예 - JSON.parse는 함수를 실행하지 않는다
const data = JSON.parse(userInput);

// 2. 스키마 검증을 반드시 거친다
const Joi = require('joi');

const sessionSchema = Joi.object({
  username: Joi.string().alphanum().max(30).required(),
  role: Joi.string().valid('user', 'admin').required(),
  exp: Joi.number().integer().required()
});

app.post('/session', (req, res) => {
  let data;
  try {
    data = JSON.parse(Buffer.from(req.cookies.session, 'base64').toString());
  } catch (e) {
    return res.status(400).send('잘못된 세션 데이터');
  }

  const { error, value } = sessionSchema.validate(data);
  if (error) {
    return res.status(400).send('세션 검증 실패');
  }

  // 검증된 value만 사용
  res.send(`Welcome ${value.username}`);
});

// 3. Java의 경우 - 화이트리스트 기반 역직렬화 필터
// Java 9+ ObjectInputFilter 사용
ObjectInputFilter filter = ObjectInputFilter.Config.createFilter(
    "com.myapp.model.*;!*"  // 허용할 클래스만 명시
);
ois.setObjectInputFilter(filter);

    실무 주의사항: - node-serialize, js-yaml(3.x 이하), pickle(Python) 같은 라이브러리는 역직렬화 시 코드를 실행할 수 있다. js-yaml은 반드시 safeLoad(4.x에서는 기본값)를 사용한다. - 세션 데이터를 클라이언트 쿠키에 저장할 때 서명(HMAC)을 검증한 후에 파싱해야 한다. 서명 없이 파싱하면 조작된 데이터가 들어온다. - Redis나 Memcached에 직렬화된 객체를 저장하는 경우에도, 해당 캐시 서버가 노출되면 같은 문제가 발생한다.

    Broken Access Control — IDOR¶

    정의: IDOR(Insecure Direct Object Reference)는 사용자가 URL이나 파라미터의 ID 값을 바꿔서 다른 사용자의 리소스에 접근하는 취약점이다. OWASP Top 10 2021에서 1위를 차지했다.

    공격 예시: 

    // 취약한 코드 - 송장 조회
app.get('/api/invoices/:id', authenticate, async (req, res) => {
  const invoice = await Invoice.findById(req.params.id);
  if (!invoice) return res.status(404).send('Not found');
  res.json(invoice); // 소유자 검증 없이 반환
});

// 정상 요청: GET /api/invoices/1001 → 본인 송장
// 공격 요청: GET /api/invoices/1002 → 다른 사용자 송장 접근
// ID를 순차적으로 증가시키면서 전체 데이터를 긁을 수 있다

    방어 방법:

  return async (req, res, next) => {

    Race Condition¶

    정의: 두 개 이상의 요청이 동시에 같은 자원에 접근해서 의도하지 않은 결과가 발생하는 취약점이다. 결제, 포인트 차감, 쿠폰 사용 같은 금전 관련 로직에서 실제 사고가 난다.

    공격 시나리오 — 이중 결제: 

    // 취약한 코드 - 포인트로 상품 구매

  const user = await User.findById(req.user.id);
  const product = await Product.findById(req.body.productId);

  if (user.points < product.price) {
    return res.status(400).send('포인트 부족');
  }

  // 여기서 두 요청이 동시에 통과할 수 있다
  // 잔액: 1000, 가격: 800
  // 요청 A: 1000 >= 800 → 통과
  // 요청 B: 1000 >= 800 → 통과 (아직 차감 전)

  user.points -= product.price;
  await user.save();
  await Order.create({ userId: user.id, productId: product.id });

async function purchaseWithLock(userId, productId) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // 행 잠금 - 다른 트랜잭션은 이 행을 읽을 때까지 대기
    const userResult = await client.query(
      'SELECT points FROM users WHERE id = $1 FOR UPDATE',
      [userId]
    );
    const user = userResult.rows[0];

    const productResult = await client.query(
      'SELECT price FROM products WHERE id = $1',
      [productId]
    );
    const product = productResult.rows[0];

    if (user.points < product.price) {
      await client.query('ROLLBACK');
      throw new Error('포인트 부족');
    }

    await client.query(
      'UPDATE users SET points = points - $1 WHERE id = $2',
      [product.price, userId]
    );

    await client.query(
      'INSERT INTO orders (user_id, product_id) VALUES ($1, $2)',
      [userId, productId]
    );

    await client.query('COMMIT');
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}

async function useCoupon(userId, couponId) {

  } finally {

  }
}

    Prototype Pollution¶

    정의: JavaScript 객체의 프로토타입 체인을 오염시켜서 모든 객체의 동작을 변경하는 공격이다. __proto__나 constructor.prototype을 통해 Object.prototype에 속성을 주입한다.

    왜 JavaScript에서만 발생하는가: JavaScript의 모든 객체는 프로토타입 체인으로 연결되어 있다. 객체에 없는 속성을 참조하면 프로토타입을 타고 올라가서 찾는다. Object.prototype을 오염시키면 모든 객체에 영향을 준다.

    공격 원리: 

    // 프로토타입 체인 동작 방식
const obj = {};
console.log(obj.isAdmin); // undefined

// Object.prototype을 오염시키면
Object.prototype.isAdmin = true;
console.log(obj.isAdmin); // true
// 모든 객체에서 isAdmin이 true가 된다

const user = { name: '홍길동' };
console.log(user.isAdmin); // true — 정의하지 않았는데 true

    실제 공격 벡터 — 깊은 병합: 

    // lodash.merge, deepmerge 등에서 발생
function deepMerge(target, source) {
  for (const key in source) {
    if (typeof source[key] === 'object' && source[key] !== null) {
      if (!target[key]) target[key] = {};
      deepMerge(target[key], source[key]);
    } else {
      target[key] = source[key];
    }
  }
  return target;
}

    // 1. __proto__, constructor, prototype 키 차단
function safeMerge(target, source) {
  for (const key of Object.keys(source)) {  // for...in 대신 Object.keys
    if (key === '__proto__' || key === 'constructor' || key === 'prototype') {
      continue; // 위험한 키 무시
    }
    if (typeof source[key] === 'object' && source[key] !== null && !Array.isArray(source[key])) {
      if (!target[key]) target[key] = {};
      safeMerge(target[key], source[key]);
    } else {
      target[key] = source[key];
    }
  }
  return target;
}

// 2. Object.create(null)로 프로토타입 없는 객체 사용

// 4. JSON 파싱 시 키 검증
function parseJSON(input) {
  const parsed = JSON.parse(input);
  return stripDangerousKeys(parsed);
}

function stripDangerousKeys(obj) {
  if (typeof obj !== 'object' || obj === null) return obj;
  const clean = Array.isArray(obj) ? [] : Object.create(null);
  for (const key of Object.keys(obj)) {
    if (key === '__proto__' || key === 'constructor') continue;
    clean[key] = stripDangerousKeys(obj[key]);
  }
  return clean;
}

    class StaticSecurityAnalyzer {
  constructor() {
    this.rules = {
      hardcodedSecrets: /(password|secret|key|token)\s*=\s*['"][^'"]+['"]/gi,
      sqlInjection: /query\s*\(\s*['"][^'"]*\+/gi,
      xssVulnerability: /innerHTML\s*=\s*[^;]+/gi,
      weakCrypto: /md5|sha1|des\s*\(/gi
    };
  }

  analyzeCode(code) {
    const vulnerabilities = [];

    for (const [type, pattern] of Object.entries(this.rules)) {
      const matches = code.match(pattern);
      if (matches) {
        vulnerabilities.push({
          type,
          matches,
          severity: this.getSeverity(type)
        });
      }
    }

    return vulnerabilities;
  }
}

    실시간 위협 탐지¶

    의심스러운 요청 패턴을 실시간으로 감지한다.

    class ThreatDetector {
  constructor() {
    this.rateLimiter = new Map();
    this.suspiciousPatterns = [
      /\.\.\//g,  // Directory traversal
      /<script>/gi,  // XSS
      /union.*select/gi  // SQL Injection
    ];
  }

  async detectThreat(request) {
    const threats = [];

    // Rate limiting 체크
    if (this.isRateLimitExceeded(request.ip)) {
      threats.push({ type: 'RATE_LIMIT', severity: 'MEDIUM' });
    }

    // 의심스러운 패턴 체크
    const content = JSON.stringify(request.body) + request.url;
    for (const pattern of this.suspiciousPatterns) {
      if (pattern.test(content)) {
        threats.push({
          type: 'SUSPICIOUS_PATTERN',
          pattern: pattern.source,
          severity: 'HIGH'
        });
      }
    }

    return threats;
  }

  isRateLimitExceeded(ip) {
    const now = Date.now();
    const requests = this.rateLimiter.get(ip) || [];

    // 최근 1분간 요청 수
    const recentRequests = requests.filter(time => now - time < 60000);

    if (recentRequests.length > 100) {
      return true;
    }

    recentRequests.push(now);
    this.rateLimiter.set(ip, recentRequests);
    return false;
  }
}

    WAF 구현¶

    웹 애플리케이션 방화벽¶

    웹 공격을 차단하는 방화벽이다. AWS WAF, Cloudflare 등을 사용할 수 있다.

    class SimpleWAF {
  constructor() {
    this.blockedIPs = new Set();
    this.rules = this.loadRules();
  }

  loadRules() {
    return [
      { pattern: /\.\.\//g, action: 'BLOCK', reason: 'Directory Traversal' },
      { pattern: /<script>/gi, action: 'BLOCK', reason: 'XSS Attempt' },
      { pattern: /union.*select/gi, action: 'BLOCK', reason: 'SQL Injection' },
      { pattern: /eval\s*\(/gi, action: 'WARN', reason: 'Eval Usage' }
    ];
  }

  async inspect(request) {
    // IP 차단 확인
    if (this.blockedIPs.has(request.ip)) {
      return { action: 'BLOCK', reason: 'Blocked IP' };
    }

    // 규칙 검사
    const content = JSON.stringify(request.body) + request.url;
    for (const rule of this.rules) {
      if (rule.pattern.test(content)) {
        if (rule.action === 'BLOCK') {
          this.blockedIPs.add(request.ip);
        }
        return {
          action: rule.action,
          reason: rule.reason,
          pattern: rule.pattern.source
        };
      }
    }

    return { action: 'ALLOW' };
  }
}

    인증/인가 보안 강화¶

    JWT 보안¶

    JWT는 토큰 기반 인증에 사용한다. 토큰 탈취를 방지하기 위해 짧은 만료 시간과 refresh token을 사용한다.

    const jwt = require('jsonwebtoken');

class SecureJWTManager {
  constructor(secret, refreshSecret) {
    this.secret = secret;
    this.refreshSecret = refreshSecret;
    this.blacklist = new Set();
  }

  generateTokenPair(payload) {
    const accessToken = jwt.sign(payload, this.secret, {
      expiresIn: '15m',
      algorithm: 'HS256'
    });

    const refreshToken = jwt.sign(payload, this.refreshSecret, {
      expiresIn: '7d',
      algorithm: 'HS256'
    });

    return { accessToken, refreshToken };
  }

  verifyAccessToken(token) {
    if (this.blacklist.has(token)) {
      throw new Error('Token has been revoked');
    }

    try {
      return jwt.verify(token, this.secret);
    } catch (error) {
      throw new Error('Invalid token');
    }
  }

  revokeToken(token) {
    this.blacklist.add(token);
  }
}

    class OAuth2Server {
  async authorize(clientId, redirectUri, scope) {
    // 클라이언트 검증
    const client = await this.validateClient(clientId, redirectUri);

    // 인가 코드 생성
    const code = crypto.randomBytes(32).toString('hex');

      clientId,
      redirectUri,
      scope,
      expiresAt: Date.now() + 600000 // 10분
    });

    return code;
  }

  async token(code, clientId, clientSecret) {
    // 인가 코드 검증
    const authData = await this.validateAuthCode(code, clientId);

    // 클라이언트 인증
    await this.authenticateClient(clientId, clientSecret);

    // 액세스 토큰 발급
    const accessToken = jwt.sign(
      { clientId, scope: authData.scope },
      this.secret,
      { expiresIn: '1h' }
    );

    return {
      access_token: accessToken,
      token_type: 'Bearer',
      expires_in: 3600
    };
  }
}

    키 로테이션¶

    class KeyRotationManager {
  constructor() {
    this.keys = [];
    this.currentKeyIndex = 0;
  }

  rotateKey() {
    const newKey = crypto.randomBytes(32);
    this.keys.push({
      key: newKey,
      createdAt: Date.now(),
      id: crypto.randomBytes(8).toString('hex')
    });

    this.currentKeyIndex = this.keys.length - 1;

    // 오래된 키 제거 (30일 이상)
    this.keys = this.keys.filter(k => 
      Date.now() - k.createdAt < 30 * 24 * 60 * 60 * 1000
    );
  }

  getCurrentKey() {
    return this.keys[this.currentKeyIndex];
  }

  getKeyById(keyId) {
    return this.keys.find(k => k.id === keyId);
  }
}

    보안 사고 대응¶

    침입 탐지 및 대응¶

    보안 사고 발생 시 즉시 대응해야 한다. 자동화된 대응 시스템을 구축하는 게 좋다.

    class IncidentResponseSystem {
  constructor() {
    this.incidents = [];
    this.alertThreshold = {
      CRITICAL: 1,
      HIGH: 3,
      MEDIUM: 10
    };
  }

  async handleSecurityEvent(event) {
    // 사고 분류
    const severity = this.classifySeverity(event);

    // 자동 대응
    if (severity === 'CRITICAL') {
      await this.blockSource(event.source);
      await this.notifySecurityTeam(event);
      await this.isolateAffectedSystems(event);
    }

    // 로깅
    this.logIncident({
      ...event,
      severity,
      timestamp: Date.now()
    });
  }

  classifySeverity(event) {
    if (event.type === 'SQL_INJECTION' || event.type === 'RCE') {
      return 'CRITICAL';
    }
    if (event.type === 'XSS' || event.type === 'UNAUTHORIZED_ACCESS') {
      return 'HIGH';
    }
    return 'MEDIUM';
  }

  async blockSource(source) {
    // 방화벽에 IP 차단 규칙 추가
    await this.firewall.blockIP(source.ip);
  }
}
