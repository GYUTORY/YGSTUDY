    보안 심화 및 취약점 분석¶

    개요¶

    보안 취약점 분석과 대응 전략을 다룬다.

    OWASP Top 10 대응¶

    SQL Injection¶

    정의: SQL Injection은 사용자 입력을 SQL 쿼리에 직접 삽입해 데이터베이스를 조작하는 공격이다.

    공격 원리: 1. 사용자 입력이 SQL 쿼리 문자열에 직접 포함됨 2. 공격자가 악의적인 SQL 코드를 입력 3. 데이터베이스가 해당 코드를 실행 4. 데이터 유출, 수정, 삭제 발생

    공격 예시: 

    // 공격자가 userId에 입력: "1' OR '1'='1"
const query = `SELECT * FROM users WHERE id = ${userId}`;
// 실행되는 쿼리: SELECT * FROM users WHERE id = 1' OR '1'='1
// 결과: 모든 사용자 정보 유출

// 공격자가 userId에 입력: "1; DROP TABLE users; --"
const query = `DELETE FROM users WHERE id = ${userId}`;
// 실행되는 쿼리: DELETE FROM users WHERE id = 1; DROP TABLE users; --
// 결과: users 테이블 삭제

    방어 방법:

    취약한 코드: 

    const query = `SELECT * FROM users WHERE id = ${userId}`;
db.query(query); // SQL Injection 취약

    안전한 코드: 

    // Prepared Statement 사용
const query = 'SELECT * FROM users WHERE id = ?';
db.query(query, [userId]); // 안전

// ORM 사용 (권장)
const user = await User.findOne({ where: { id: userId } });

    실무 팁: - ORM을 사용하면 대부분 자동으로 방어된다 - 직접 쿼리를 작성할 때는 Prepared Statement를 사용해야 한다 - 입력값 검증도 함께 수행한다

    XSS (Cross-Site Scripting)¶

    정의: XSS는 웹 애플리케이션에 악의적인 JavaScript 코드를 삽입해 사용자 브라우저에서 실행하는 공격이다.

    공격 유형:

    1. Stored XSS (저장형): 악의적인 스크립트가 데이터베이스에 저장되고, 다른 사용자가 해당 페이지를 볼 때 실행된다.

    // 공격자가 댓글에 입력: <script>alert(document.cookie)</script>
// 데이터베이스에 저장됨
// 다른 사용자가 댓글을 볼 때 스크립트 실행

    2. Reflected XSS (반사형): URL 파라미터나 폼 입력에 포함된 스크립트가 즉시 실행된다.

    // 공격자가 URL에 입력: ?search=<script>alert('XSS')</script>
// 서버가 검색 결과에 그대로 포함
// 사용자 브라우저에서 스크립트 실행

    3. DOM-based XSS: 클라이언트 측 JavaScript가 DOM을 조작할 때 발생한다.

    // 취약한 코드
document.getElementById('output').innerHTML = userInput;
// 공격자가 userInput에 <img src=x onerror="alert('XSS')"> 입력

    공격 피해: - 쿠키 탈취 (세션 하이재킹) - 사용자 정보 유출 - 피싱 공격 - 악성 코드 실행

    방어 방법:

    입력 검증 및 이스케이핑: 

    function sanitizeInput(input) {
  return input
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#x27;')
    .replace(/\//g, '&#x2F;');
}

// 또는 라이브러리 사용 (권장)
const DOMPurify = require('isomorphic-dompurify');
const clean = DOMPurify.sanitize(userInput);

    Content Security Policy 설정: 

    app.use(helmet.contentSecurityPolicy({
  directives: {
    defaultSrc: ["'self'"],
    scriptSrc: ["'self'"], // 'unsafe-inline' 제거
    styleSrc: ["'self'", "'unsafe-inline'"]
  }
}));

    실무 팁: - innerHTML 대신 textContent 사용 - React, Vue 같은 프레임워크는 기본적으로 XSS 방어 - CSP 헤더를 설정해 스크립트 실행 제한

    CSRF (Cross-Site Request Forgery)¶

    정의: CSRF는 사용자가 의도하지 않은 요청을 다른 사이트에서 대신 보내는 공격이다. 사용자가 로그인한 상태에서 악의적인 사이트가 사용자의 권한으로 요청을 보낸다.

    공격 원리: 1. 사용자가 은행 사이트에 로그인 (세션 쿠키 저장) 2. 사용자가 악의적인 사이트 방문 3. 악의적인 사이트가 은행 사이트로 자동 요청 전송 4. 브라우저가 세션 쿠키를 자동으로 포함 5. 사용자 권한으로 요청 실행 (이체, 정보 변경 등)

    공격 예시: 

    <!-- 악의적인 사이트의 HTML -->
<img src="https://bank.com/transfer?to=attacker&amount=1000" />

<!-- 또는 자동 실행 스크립트 -->
<script>
  fetch('https://bank.com/transfer', {
    method: 'POST',
    body: JSON.stringify({ to: 'attacker', amount: 1000 }),
    credentials: 'include' // 쿠키 자동 포함
  });
</script>

    공격 조건: - 사용자가 대상 사이트에 로그인한 상태 - 쿠키 기반 인증 사용 - 상태 변경 요청 (GET이 아닌 POST, PUT, DELETE)

    방어 방법:

    1. CSRF 토큰 사용: 

    const csrf = require('csurf');
const csrfProtection = csrf({ cookie: true });

// 토큰 생성 (폼 렌더링 시)
app.get('/form', csrfProtection, (req, res) => {
  res.render('form', { csrfToken: req.csrfToken() });
});

// 토큰 검증 (요청 처리 시)
app.post('/transfer', csrfProtection, (req, res) => {
  // CSRF 토큰 검증 완료
  processTransfer(req.body);
});

    2. SameSite 쿠키 속성: 

    app.use(session({
  cookie: {
    sameSite: 'strict', // 또는 'lax'
    secure: true,
    httpOnly: true
  }
}));

    3. Referer 검증: 

    app.post('/transfer', (req, res) => {
  const referer = req.get('Referer');
  if (!referer || !referer.startsWith('https://yourdomain.com')) {
    return res.status(403).send('Invalid request origin');
  }
  processTransfer(req.body);
});

    실무 팁: - 상태 변경 요청에는 항상 CSRF 토큰 사용 - SameSite 쿠키 속성으로 추가 방어 - GET 요청은 상태를 변경하지 않도록 설계

    주요 보안 취약점 추가¶

    Directory Traversal (경로 탐색)¶

    정의: Directory Traversal은 ../ 같은 경로 조작을 통해 시스템 파일에 접근하는 공격이다.

    공격 예시: 

    // 취약한 코드
app.get('/file', (req, res) => {
  const filename = req.query.name;
  res.sendFile(`/uploads/${filename}`); // 위험
});

// 공격자가 입력: ?name=../../../etc/passwd
// 결과: /etc/passwd 파일 유출

    방어 방법: 

    const path = require('path');

app.get('/file', (req, res) => {
  const filename = req.query.name;
  const safePath = path.resolve('/uploads', filename);

  // 경로 검증
  if (!safePath.startsWith('/uploads')) {
    return res.status(403).send('Invalid path');
  }

  res.sendFile(safePath);
});

    Command Injection¶

    정의: Command Injection은 시스템 명령어를 실행하는 함수에 사용자 입력을 직접 전달해 악의적인 명령을 실행하는 공격이다.

    공격 예시: 

    // 취약한 코드
const { exec } = require('child_process');
const userInput = req.query.filename;
exec(`cat ${userInput}`, (error, stdout) => {
  // ...
});

// 공격자가 입력: file.txt; rm -rf /
// 결과: 시스템 파일 삭제

    방어 방법: 

    // 입력값 검증
const filename = req.query.filename;
if (!/^[a-zA-Z0-9._-]+$/.test(filename)) {
  return res.status(400).send('Invalid filename');
}

// 또는 exec 대신 execFile 사용
const { execFile } = require('child_process');
execFile('cat', [filename], (error, stdout) => {
  // ...
});

    XXE (XML External Entity)¶

    정의: XXE는 XML 파서가 외부 엔티티를 처리할 때 발생하는 취약점이다.

    공격 예시: 

    <!-- 공격자가 입력한 XML -->
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<foo>&xxe;</foo>

    방어 방법: 

    const libxml = require('libxmljs');

const xml = req.body;
const doc = libxml.parseXmlString(xml, {
  noent: false, // 외부 엔티티 비활성화
  noblanks: true
});

    SSRF (Server-Side Request Forgery)¶

    정의: SSRF는 서버가 사용자 입력으로 받은 URL에 직접 요청을 보낼 때 발생한다. 공격자가 내부 네트워크나 클라우드 메타데이터 서비스에 접근하는 데 쓰인다. OWASP Top 10 2021에서 신규 진입한 항목이다.

    왜 위험한가: 서버 입장에서 보내는 요청이기 때문에 내부 방화벽을 우회한다. AWS에서는 169.254.169.254 메타데이터 엔드포인트를 통해 IAM 크레덴셜을 탈취당하는 사고가 실제로 발생한다. 2019년 Capital One 해킹이 대표 사례다.

    공격 예시: 

    // 취약한 코드 - URL 프리뷰 기능
app.post('/preview', async (req, res) => {
  const { url } = req.body;
  const response = await fetch(url); // 사용자 입력 URL을 그대로 요청
  const html = await response.text();
  res.send(html);
});

// 공격자가 보내는 요청
// POST /preview
// { "url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/" }
// → AWS IAM 크레덴셜 유출

// 내부 서비스 스캔
// { "url": "http://10.0.0.1:8080/admin" }
// → 내부 관리자 페이지 접근

    방어 방법: 

    const { URL } = require('url');
const dns = require('dns').promises;
const ipaddr = require('ipaddr.js');

async function validateUrl(inputUrl) {
  const parsed = new URL(inputUrl);

  // 1. 프로토콜 제한 - http/https만 허용
  if (!['http:', 'https:'].includes(parsed.protocol)) {
    throw new Error('허용되지 않는 프로토콜');
  }

  // 2. DNS 조회 후 IP 대역 검증
  const addresses = await dns.resolve4(parsed.hostname);
  for (const addr of addresses) {
    const ip = ipaddr.parse(addr);
    const range = ip.range();
    // private, loopback, link-local 등 내부 대역 차단
    if (range !== 'unicast') {
      throw new Error(`내부 IP 대역 접근 차단: ${addr} (${range})`);
    }
  }

  // 3. 허용 도메인 화이트리스트 (가능한 경우)
  const allowedDomains = ['api.example.com', 'cdn.example.com'];
  if (!allowedDomains.includes(parsed.hostname)) {
    throw new Error('허용되지 않는 도메인');
  }

  return parsed.toString();
}

app.post('/preview', async (req, res) => {
  try {
    const safeUrl = await validateUrl(req.body.url);
    const response = await fetch(safeUrl);
    res.send(await response.text());
  } catch (err) {
    res.status(400).send(err.message);
  }
});

    실무에서 놓치는 부분: - DNS rebinding 공격: 첫 번째 DNS 조회에서는 외부 IP를 반환하고, 두 번째 조회에서 내부 IP를 반환하는 방식이다. DNS 조회 결과를 캐싱하거나, 실제 연결 시점에 IP를 다시 검증해야 한다. - URL 파서 우회: http://127.0.0.1 대신 http://0x7f000001, http://2130706433, http://127.1 같은 표현으로 우회할 수 있다. IP 파싱 라이브러리를 사용해서 정규화한 후 비교해야 한다. - 리다이렉트 체이닝: 외부 URL이 302로 내부 IP로 리다이렉트하는 경우가 있다. HTTP 클라이언트의 리다이렉트 추적을 비활성화하거나, 리다이렉트된 URL도 검증해야 한다.

    Insecure Deserialization (안전하지 않은 역직렬화)¶

    정의: 신뢰할 수 없는 데이터를 역직렬화할 때 공격자가 객체를 조작해서 임의 코드를 실행하거나 권한을 탈취하는 취약점이다. Java, PHP, Python, Node.js 가리지 않고 발생한다.

    왜 위험한가: 역직렬화는 바이트 스트림에서 객체를 복원하는 과정인데, 이 과정에서 생성자나 매직 메서드가 실행된다. 공격자가 직렬화된 데이터를 조작하면 서버에서 임의 코드가 실행된다. 2017년 Apache Struts 취약점(Equifax 사태)이 역직렬화 관련이었다.

    Node.js에서의 공격 예시: 

    // 취약한 코드 - node-serialize 사용
const serialize = require('node-serialize');

app.post('/session', (req, res) => {
  const sessionData = Buffer.from(req.cookies.session, 'base64').toString();
  const session = serialize.unserialize(sessionData); // 위험
  res.send(`Welcome ${session.username}`);
});

// 공격자가 보내는 쿠키 (base64 인코딩 전)
// {"username":"admin","role":"_$$ND_FUNC$$_function(){require('child_process').exec('cat /etc/passwd')}()"}
// → node-serialize의 IIFE 패턴으로 임의 코드 실행

    Java에서의 공격 원리: 

    // 취약한 코드
ObjectInputStream ois = new ObjectInputStream(request.getInputStream());
Object obj = ois.readObject(); // 사용자 입력을 바로 역직렬화

// 공격자가 Apache Commons Collections의 가젯 체인을 이용해
// Runtime.getRuntime().exec()를 호출하는 직렬화된 객체를 보낸다
// ysoserial 같은 도구로 페이로드를 자동 생성할 수 있다

    방어 방법: 

    // 1. 역직렬화 대신 JSON 사용
// 나쁜 예
const data = serialize.unserialize(userInput);

// 좋은 예 - JSON.parse는 함수를 실행하지 않는다
const data = JSON.parse(userInput);

// 2. 스키마 검증을 반드시 거친다
const Joi = require('joi');

const sessionSchema = Joi.object({
  username: Joi.string().alphanum().max(30).required(),
  role: Joi.string().valid('user', 'admin').required(),
  exp: Joi.number().integer().required()
});

app.post('/session', (req, res) => {
  let data;
  try {
    data = JSON.parse(Buffer.from(req.cookies.session, 'base64').toString());
  } catch (e) {
    return res.status(400).send('잘못된 세션 데이터');
  }

  const { error, value } = sessionSchema.validate(data);
  if (error) {
    return res.status(400).send('세션 검증 실패');
  }

  // 검증된 value만 사용

    정의: IDOR(Insecure Direct Object Reference)는 사용자가 URL이나 파라미터의 ID 값을 바꿔서 다른 사용자의 리소스에 접근하는 취약점이다. OWASP Top 10 2021에서 1위를 차지했다.

    정의: 두 개 이상의 요청이 동시에 같은 자원에 접근해서 의도하지 않은 결과가 발생하는 취약점이다. 결제, 포인트 차감, 쿠폰 사용 같은 금전 관련 로직에서 실제 사고가 난다.

    Prototype Pollution¶

    정의: JavaScript 객체의 프로토타입 체인을 오염시켜서 모든 객체의 동작을 변경하는 공격이다. __proto__나 constructor.prototype을 통해 Object.prototype에 속성을 주입한다.

    왜 JavaScript에서만 발생하는가: JavaScript의 모든 객체는 프로토타입 체인으로 연결되어 있다. 객체에 없는 속성을 참조하면 프로토타입을 타고 올라가서 찾는다. Object.prototype을 오염시키면 모든 객체에 영향을 준다.

    공격 원리:

const user = { name: '홍길동' };

function safeMerge(target, source) {
  for (const key of Object.keys(source)) {  // for...in 대신 Object.keys
    if (key === '__proto__' || key === 'constructor' || key === 'prototype') {
      continue; // 위험한 키 무시
    }
    if (typeof source[key] === 'object' && source[key] !== null && !Array.isArray(source[key])) {
      if (!target[key]) target[key] = {};
      safeMerge(target[key], source[key]);
    } else {
      target[key] = source[key];
    }
  }
  return target;

    추가 취약점¶

    Open Redirect¶

    사용자를 외부 악성 사이트로 리다이렉트하는 취약점이다. 로그인 후 returnUrl 파라미터로 원래 페이지로 돌려보내는 기능에서 자주 발생한다.

    공격 시나리오: 

    # 정상 URL
https://example.com/login?returnUrl=/dashboard

# 공격 URL — 피싱 사이트로 리다이렉트
https://example.com/login?returnUrl=https://evil.com/fake-login

# 우회 시도 — 슬래시 두 개
https://example.com/login?returnUrl=//evil.com

# 우회 시도 — URL 인코딩
https://example.com/login?returnUrl=https%3A%2F%2Fevil.com

    사용자 입장에서는 example.com 도메인의 링크를 클릭한 것이기 때문에 의심하지 않는다. 피싱 메일에서 이 취약점을 악용하는 사례가 많다.

    취약한 코드: 

    app.get('/login/callback', (req, res) => {
  const returnUrl = req.query.returnUrl || '/';
  res.redirect(returnUrl); // 검증 없이 리다이렉트

    방어 방법: 

    function isRelativeUrl(url) {
  // 슬래시로 시작하되, //로 시작하지 않아야 한다
  // //evil.com은 프로토콜 상대 URL로 해석되기 때문
  if (!url.startsWith('/') || url.startsWith('//')) {
    return false;
  }
  try {
    // URL 생성자로 파싱했을 때 origin이 바뀌면 외부 URL
    const parsed = new URL(url, 'https://example.com');
    return parsed.origin === 'https://example.com';
  } catch {
    return false;
  }
}

app.get('/login/callback', (req, res) => {
  const returnUrl = req.query.returnUrl || '/';
  if (!isRelativeUrl(returnUrl)) {
    return res.redirect('/'); // 검증 실패 시 홈으로
  }
  res.redirect(returnUrl);
});

    Server-Side Template Injection (SSTI)¶

    서버 측 템플릿 엔진에 사용자 입력이 템플릿 코드로 삽입되면 서버에서 임의 코드가 실행된다. Jinja2(Python), Pug/EJS(Node.js), Thymeleaf(Java) 등 템플릿 엔진을 사용하는 곳에서 발생한다.

    XSS와의 차이: XSS는 브라우저에서 JavaScript가 실행되는 것이고, SSTI는 서버에서 코드가 실행된다. 서버의 파일 시스템, 환경 변수, 데이터베이스에 직접 접근할 수 있어서 피해 범위가 다르다.

    공격 원리 — Python Jinja2: 

    # 취약한 코드 — 사용자 입력을 템플릿 문자열에 직접 삽입
from flask import Flask, request, render_template_string

app = Flask(__name__)

@app.route('/greet')
def greet():
    name = request.args.get('name', '')
    template = f'<h1>Hello {name}</h1>'  # 위험
    return render_template_string(template)

# 공격 입력: ?name={{7*7}} → <h1>Hello 49</h1> (템플릿 엔진이 실행)
# RCE 페이로드: ?name={{config.__class__.__init__.__globals__['os'].popen('id').read()}}
# → 서버에서 시스템 명령 실행

  res.render('page', { title: req.query.title }); // 변수로 전달
});

    핵심은 사용자 입력을 템플릿 문자열 자체에 포함하지 않는 것이다. 템플릿 파일은 고정하고, 데이터만 변수로 전달한다.

    HTTP Request Smuggling¶

    프론트엔드 서버(로드밸런서, 리버스 프록시)와 백엔드 서버가 HTTP 요청의 경계를 다르게 해석할 때 발생한다. 한 요청 안에 다른 요청을 밀어넣어서 접근 제어를 우회하거나 다른 사용자의 요청을 가로챈다.

    발생 조건: HTTP/1.1에서 요청 길이를 결정하는 헤더가 두 개 있다: Content-Length와 Transfer-Encoding: chunked. 두 헤더를 동시에 보내면 프록시와 백엔드가 서로 다른 헤더를 기준으로 요청을 파싱하는 경우가 생긴다.

    CL.TE 공격 (프록시는 Content-Length, 백엔드는 Transfer-Encoding을 사용): 

    POST / HTTP/1.1
Host: example.com
Content-Length: 13
Transfer-Encoding: chunked

0

GET /admin HTTP/1.1
Host: example.com

    프록시는 Content-Length: 13을 보고 0\r\n\r\nGET /adm까지를 하나의 요청으로 처리한다. 백엔드는 Transfer-Encoding: chunked를 보고 0\r\n\r\n에서 첫 번째 요청이 끝난 것으로 판단한다. 그 뒤의 GET /admin은 다음 요청의 시작으로 해석된다. 이 요청에 다른 사용자의 후속 요청 헤더가 붙을 수 있다.

    피해: - 접근 제어 우회: 프록시에서 /admin 접근을 차단해도 스머글링으로 우회 - 요청 하이재킹: 다른 사용자의 쿠키, 인증 정보 탈취 - 캐시 포이즌: CDN에 조작된 응답을 캐싱

    방어: - HTTP/2를 사용한다. HTTP/2는 바이너리 프레이밍을 사용해서 요청 경계가 명확하다. - 프록시와 백엔드가 같은 방식으로 요청을 파싱하도록 설정한다. - Transfer-Encoding과 Content-Length가 동시에 있는 요청을 거부하도록 설정한다. - AWS ALB, Cloudflare 같은 관리형 로드밸런서는 대부분 스머글링 방어가 내장되어 있다. 직접 Nginx를 프록시로 쓸 때 주의해야 한다.

    Timing Attack¶

    처리 시간의 차이를 관찰해서 정보를 추출하는 사이드채널 공격이다.

    가장 흔한 사례 — 문자열 비교: 

    // 취약한 코드 — 일반 문자열 비교
function verifyToken(input, secret) {
  return input === secret; // 첫 번째 다른 문자에서 즉시 false 반환
}

    === 비교는 첫 번째로 다른 바이트를 만나면 바로 false를 반환한다. 공격자가 토큰의 첫 번째 문자부터 하나씩 맞춰가면서 응답 시간 차이를 측정하면 전체 토큰을 알아낼 수 있다. 네트워크 지연 때문에 현실적으로 어려워 보이지만, 통계적 기법으로 수천 번 반복 측정하면 가능하다.

    방어 — 상수 시간 비교: 

    const crypto = require('crypto');

function verifyToken(input, secret) {
  // 길이가 다르면 해시 비교로 대체 (길이 자체도 타이밍 정보가 된다)
  const inputHash = crypto.createHmac('sha256', 'key').update(input).digest();
  const secretHash = crypto.createHmac('sha256', 'key').update(secret).digest();
  return crypto.timingSafeEqual(inputHash, secretHash);
}

    crypto.timingSafeEqual은 두 버퍼의 모든 바이트를 비교한 후에 결과를 반환한다. 어떤 위치에서 다르든 실행 시간이 동일하다.

    주의할 곳: - API 키, 웹훅 시크릿, HMAC 서명 검증에서 === 대신 timingSafeEqual 사용 - 비밀번호 비교는 bcrypt를 쓰면 내부적으로 상수 시간 비교를 수행한다 - 사용자 존재 여부 확인: "아이디가 없습니다" vs "비밀번호가 틀렸습니다" 같은 메시지도 타이밍 정보다. 사용자가 없는 경우에도 더미 해시를 비교해서 응답 시간을 맞춰야 한다.

    JWT 공격과 방어¶

    JWT 자체의 구조적 취약점을 이용하는 공격들이다. 토큰 탈취, 만료 시간 같은 일반적인 내용이 아니라 JWT 고유의 공격 벡터를 다룬다.

    none 알고리즘 공격¶

    JWT 헤더의 alg 필드를 none으로 변경하면 서명 없이 토큰을 위조할 수 있다.

    // JWT 구조: header.payload.signature
// 정상 토큰 헤더: {"alg":"HS256","typ":"JWT"}
// 공격 토큰 헤더: {"alg":"none","typ":"JWT"}

// 공격자가 만든 토큰 (서명 부분이 비어있음)
// eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJ1c2VyIjoiYWRtaW4ifQ.

// 취약한 검증 코드
const decoded = jwt.verify(token, secret); // 일부 라이브러리에서 none을 허용

    방어: 

    // 알고리즘을 명시적으로 지정한다
const decoded = jwt.verify(token, secret, { algorithms: ['HS256'] });
// none, HS384, RS256 등 다른 알고리즘은 전부 거부된다

    jsonwebtoken 라이브러리는 4.2.2 이후 none을 기본 차단하지만, algorithms 옵션을 명시하는 습관을 들여야 한다. 다른 언어의 JWT 라이브러리는 여전히 none을 허용하는 경우가 있다.

    Key Confusion (RS256 → HS256)¶

    RS256(비대칭)으로 서명된 토큰을 HS256(대칭)으로 바꿔서 공개키를 HMAC 비밀키로 사용하는 공격이다.

    공격 원리: 1. 서버가 RS256으로 토큰을 발급한다 (비밀키로 서명, 공개키로 검증) 2. 공격자가 서버의 공개키를 가져온다 (공개키는 누구나 접근 가능) 3. 헤더를 HS256으로 변경하고, 공개키를 HMAC 비밀키로 사용해서 서명한다 4. 서버의 검증 로직이 alg 헤더를 신뢰하면 HS256으로 검증을 시도한다 5. HS256 검증에 사용되는 키가 공개키(=공격자가 사용한 키)이므로 검증 통과

    // 취약한 코드 — 헤더의 alg를 신뢰
function verifyToken(token) {
  const header = JSON.parse(atob(token.split('.')[0]));
  if (header.alg === 'RS256') {
    return jwt.verify(token, publicKey, { algorithms: ['RS256'] });
  } else if (header.alg === 'HS256') {
    return jwt.verify(token, publicKey, { algorithms: ['HS256'] });
    // 공개키를 HMAC 시크릿으로 사용하게 된다
  }
}

// 안전한 코드 — 알고리즘 고정
function verifyToken(token) {
  return jwt.verify(token, publicKey, { algorithms: ['RS256'] });
  // RS256 외에는 무조건 거부
}

    kid (Key ID) Injection¶

    JWT 헤더의 kid 필드를 조작해서 서버가 공격자가 원하는 키를 사용하게 만드는 공격이다.

    // 정상 JWT 헤더
// {"alg":"HS256","typ":"JWT","kid":"key-2024-01"}

// 서버의 키 조회 로직이 DB에서 kid로 검색하는 경우
// 취약한 코드
async function getKey(kid) {
  const result = await db.query(`SELECT key_value FROM keys WHERE kid = '${kid}'`);
  return result.rows[0].key_value;
}

// 공격 1: SQL Injection via kid
// {"kid":"' UNION SELECT 'my-secret-key' --"}
// → 공격자가 알고 있는 키 값으로 서명 검증을 통과

// 공격 2: kid가 파일 경로인 경우
// {"kid":"/dev/null"} → 빈 문자열이 키가 됨
// {"kid":"../../etc/hostname"} → 알려진 값으로 서명 가능

    방어: 

    // kid에 대한 입력 검증
function getKey(kid) {
  if (!/^[a-zA-Z0-9_-]+$/.test(kid)) {
    throw new Error('잘못된 kid 형식');
  }
  // 파라미터 바인딩 사용
  const result = await db.query('SELECT key_value FROM keys WHERE kid = $1', [kid]);
  if (!result.rows[0]) {
    throw new Error('키를 찾을 수 없음');
  }
  return result.rows[0].key_value;
}

    jku/x5u 헤더 공격¶

    jku(JWK Set URL)나 x5u(X.509 URL) 헤더를 조작해서 공격자의 서버에 있는 키를 사용하게 만드는 공격이다.

    // 정상 헤더
// {"alg":"RS256","jku":"https://example.com/.well-known/jwks.json"}

// 공격 헤더 — 공격자 서버의 키셋을 참조
// {"alg":"RS256","jku":"https://evil.com/jwks.json"}
// 공격자가 자신의 키로 서명한 토큰이 검증을 통과한다

    방어: - jku, x5u 헤더를 아예 무시하고 서버에 등록된 키만 사용한다 - 사용해야 한다면 URL 화이트리스트를 적용한다

    OAuth 2.0 구현 시 취약점¶

    OAuth 2.0 스펙 자체는 보안을 고려하고 있지만, 구현 과정에서 놓치기 쉬운 부분이 있다.

    state 파라미터 누락 — CSRF¶

    state 파라미터 없이 OAuth를 구현하면 CSRF 공격에 노출된다.

    # 공격 시나리오
1. 공격자가 자신의 OAuth 인가 코드를 획득한다
2. 피해자에게 해당 코드가 포함된 콜백 URL을 클릭하게 만든다
   https://example.com/callback?code=공격자의_코드
3. 피해자의 계정이 공격자의 소셜 계정과 연결된다
4. 공격자가 소셜 로그인으로 피해자의 계정에 접근

    방어: 

    // 인가 요청 시 state 생성
app.get('/auth/google', (req, res) => {
  const state = crypto.randomBytes(32).toString('hex');
  // 세션에 저장해서 콜백에서 비교
  req.session.oauthState = state;

  const authUrl = new URL('https://accounts.google.com/o/oauth2/v2/auth');
  authUrl.searchParams.set('client_id', CLIENT_ID);
  authUrl.searchParams.set('redirect_uri', REDIRECT_URI);
  authUrl.searchParams.set('response_type', 'code');
  authUrl.searchParams.set('scope', 'openid email');
  authUrl.searchParams.set('state', state);
  res.redirect(authUrl.toString());
});

// 콜백에서 state 검증
app.get('/auth/google/callback', (req, res) => {
  if (req.query.state !== req.session.oauthState) {
    return res.status(403).send('state 불일치 — CSRF 의심');
  }
  delete req.session.oauthState;
  // 인가 코드로 토큰 교환 진행
});

    redirect_uri 검증 부실 — Open Redirect¶

    redirect_uri를 느슨하게 검증하면 인가 코드가 공격자에게 전달된다.

    # 등록된 redirect_uri: https://example.com/callback

# 공격 시도 — 서브디렉토리 조작
https://example.com/callback/../../../attacker.com

# 공격 시도 — 서브도메인 등록 후 사용
https://attacker.example.com/callback

# 공격 시도 — 쿼리스트링 추가
https://example.com/callback?next=https://evil.com

    방어: - redirect_uri는 정확히 일치(exact match)로 검증한다. 와일드카드나 부분 매칭을 허용하면 안 된다. - OAuth 제공자 설정에서 등록된 URI 목록을 가능한 한 좁게 관리한다.

    PKCE 미적용 — 인가 코드 탈취¶

    모바일 앱이나 SPA에서 Authorization Code Flow를 사용할 때 PKCE(Proof Key for Code Exchange)를 적용하지 않으면 인가 코드가 탈취될 수 있다. 모바일 환경에서 커스텀 URL 스키마를 등록한 악성 앱이 인가 코드를 가로채는 시나리오가 대표적이다.

    const crypto = require('crypto');

// 1. code_verifier 생성 (랜덤 문자열, 43~128자)
const codeVerifier = crypto.randomBytes(32)
  .toString('base64url');

// 2. code_challenge 생성 (SHA256 해시)
const codeChallenge = crypto.createHash('sha256')
  .update(codeVerifier)
  .digest('base64url');

// 3. 인가 요청에 code_challenge 포함
const authUrl = new URL('https://auth.example.com/authorize');
authUrl.searchParams.set('response_type', 'code');
authUrl.searchParams.set('client_id', CLIENT_ID);
authUrl.searchParams.set('code_challenge', codeChallenge);
authUrl.searchParams.set('code_challenge_method', 'S256');
// redirect_uri, scope, state 생략

// 4. 토큰 교환 시 code_verifier 전송
const tokenResponse = await fetch('https://auth.example.com/token', {
  method: 'POST',
  headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
  body: new URLSearchParams({
    grant_type: 'authorization_code',
    code: authorizationCode,
    code_verifier: codeVerifier, // 서버가 이걸 해시해서 challenge와 비교
    client_id: CLIENT_ID,
    redirect_uri: REDIRECT_URI
  })
});

    인가 코드를 탈취해도 code_verifier를 모르면 토큰을 교환할 수 없다. OAuth 2.1에서는 PKCE가 필수다.

    자동화된 보안 테스트¶

    정적 분석 (SAST) — CI 연동¶

    코드를 실행하지 않고 소스 코드 레벨에서 취약점을 찾는다.

    ESLint 보안 플러그인:

    프로젝트에 eslint-plugin-security를 설정한다.

    npm install --save-dev eslint-plugin-security

    // .eslintrc.json
{
  "plugins": ["security"],
  "extends": ["plugin:security/recommended"]
}

    이 플러그인은 eval() 사용, 정규식 ReDoS 위험, child_process.exec() 사용, 비순수 정규식 등을 감지한다. eslint-plugin-no-secrets를 추가하면 하드코딩된 시크릿도 잡는다.

    SonarQube CI 연동 (GitHub Actions):

    # .github/workflows/security.yml
name: Security Scan
on: [push, pull_request]

jobs:
  sonarqube:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # 전체 히스토리 필요 (blame 분석)
      - uses: sonarsource/sonarqube-scan-action@v3
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

    # sonar-project.properties
sonar.projectKey=my-project
sonar.sources=src
sonar.exclusions=**/node_modules/**,**/test/**
sonar.javascript.lcov.reportPaths=coverage/lcov.info

    SonarQube는 SQL Injection, XSS, 하드코딩된 비밀번호 등을 감지하고, 코드 품질과 보안을 동시에 관리한다. PR에 코멘트를 달아주는 기능이 있어서 코드 리뷰 시점에 잡을 수 있다.

    Semgrep (경량 SAST):

    SonarQube보다 설정이 간단하고, 커스텀 룰을 YAML로 작성할 수 있다.

    # .github/workflows/semgrep.yml
name: Semgrep
on: [push, pull_request]
jobs:
  semgrep:
    runs-on: ubuntu-latest
    container:
      image: semgrep/semgrep
    steps:
      - uses: actions/checkout@v4
      - run: semgrep scan --config auto --error
        # --config auto: 언어에 맞는 기본 룰셋 자동 적용
        # --error: 취약점 발견 시 exit code 1 (CI 실패)

    동적 분석 (DAST) — OWASP ZAP¶

    실행 중인 애플리케이션에 실제 공격 페이로드를 보내서 취약점을 찾는다.

    GitHub Actions에서 ZAP 실행:

    # .github/workflows/zap.yml
name: OWASP ZAP Scan
on:
  schedule:
    - cron: '0 3 * * 1'  # 매주 월요일 새벽 3시

jobs:
  zap:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # 애플리케이션 실행
      - run: docker compose up -d
      - run: sleep 10  # 기동 대기

      # ZAP Baseline Scan (패시브 스캔)
      - uses: zaproxy/action-baseline@v0.12.0
        with:
          target: 'http://localhost:3000'
          rules_file_name: 'zap-rules.tsv'
          fail_action: 'warn'

      # 결과 아티팩트 저장
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: zap-report
          path: report_html.html

    ZAP의 스캔 모드: - Baseline Scan: 패시브 스캔. 응답 헤더, 쿠키 설정 등을 검사한다. CI에 넣기 좋다. - Full Scan: 액티브 스캔 포함. SQL Injection, XSS 페이로드를 직접 보낸다. 스테이징 환경에서 실행한다. 프로덕션에서 돌리면 데이터가 오염될 수 있다.

    컨테이너 이미지 스캔 — Trivy:

    # .github/workflows/trivy.yml
name: Container Scan
on:
  push:
    paths: ['Dockerfile', 'package-lock.json']

jobs:
  trivy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build image
        run: docker build -t myapp:${{ github.sha }} .
      - uses: aquasecurity/trivy-action@0.28.0
        with:
          image-ref: 'myapp:${{ github.sha }}'
          format: 'sarif'
          output: 'trivy-results.sarif'
          severity: 'CRITICAL,HIGH'
          exit-code: '1'  # CRITICAL/HIGH 발견 시 CI 실패

    Trivy는 OS 패키지(Alpine apk, Debian apt)와 언어별 의존성(npm, pip, go.sum)의 CVE를 모두 검사한다. IaC(Terraform, CloudFormation) 설정 오류도 잡는다.

    AWS Secrets Manager¶

    AWS 환경에서는 Secrets Manager로 시크릿을 관리한다.

  const command = new GetSecretValueCommand({ SecretId: secretName });
  const response = await client.send(command);
  return JSON.parse(response.SecretString);
}

    자동 로테이션 설정:

    Secrets Manager는 Lambda를 사용해서 시크릿을 자동으로 로테이션한다. RDS의 경우 내장 로테이션 함수가 있다.

    실무에서 주의할 점: - 시크릿을 읽은 후 메모리에 캐싱한다. 매 요청마다 Secrets Manager를 호출하면 비용과 지연이 발생한다. 캐시 TTL은 로테이션 주기보다 짧게 설정한다. - IAM 역할로 접근을 제어한다. 특정 서비스만 특정 시크릿을 읽을 수 있도록 정책을 설정한다. - CloudTrail에서 시크릿 접근 로그를 모니터링한다. 비정상적인 접근 패턴이 보이면 알림을 설정한다.

    키 로테이션¶

    왜 키를 교체해야 하는가¶

    암호화 키나 서명 키는 시간이 지나면 유출 위험이 쌓인다. 퇴사자가 키를 알고 있을 수 있고, 어디선가 로그에 남았을 수 있다. 키를 교체하면 과거에 유출된 키의 영향 범위를 제한한다.

    JWT 서명 키 로테이션¶

    JWKS(JSON Web Key Set)를 사용하면 키 교체 시 기존 토큰이 무효화되지 않는다.

    const jose = require('jose');
const crypto = require('crypto');

// 키 저장소 (실제로는 DB나 KMS에 저장)
let keys = [];

async function rotateSigningKey() {
  const { publicKey, privateKey } = await jose.generateKeyPair('RS256');
  const kid = crypto.randomBytes(8).toString('hex');

  keys.push({ kid, publicKey, privateKey, createdAt: Date.now() });

  // 90일 지난 키 제거 (해당 키로 서명된 토큰은 이미 만료됨)
  keys = keys.filter(k => Date.now() - k.createdAt < 90 * 24 * 60 * 60 * 1000);
}

// 토큰 발급 — 항상 최신 키 사용
async function issueToken(payload) {
  const current = keys[keys.length - 1];
  return new jose.SignJWT(payload)
    .setProtectedHeader({ alg: 'RS256', kid: current.kid })
    .setExpirationTime('15m')
    .sign(current.privateKey);
}

// 토큰 검증 — kid로 키를 찾아서 검증
async function verifyToken(token) {
  const { kid } = jose.decodeProtectedHeader(token);
  const keyEntry = keys.find(k => k.kid === kid);
  if (!keyEntry) throw new Error('알 수 없는 키');
  return jose.jwtVerify(token, keyEntry.publicKey);
}

// JWKS 엔드포인트 — 공개키 목록 제공
app.get('/.well-known/jwks.json', (req, res) => {
  const jwks = keys.map(k => ({
    ...jose.exportJWK(k.publicKey),
    kid: k.kid,
    use: 'sig',
    alg: 'RS256'
  }));
  res.json({ keys: jwks });
});

    키 로테이션 주기는 서비스 특성에 따라 다르지만, 일반적으로 JWT 서명 키는 30~90일, API 키는 90일, DB 암호화 키는 1년 정도가 적정하다. AWS KMS를 사용하면 자동 로테이션을 1년 주기로 설정할 수 있다.

aws iam update-access-key --user-name compromised-user --access-key-id AKIA... --status Inactive

    2. 증거 보존: 

    # 메모리 덤프 (EC2 인스턴스)
# 인스턴스를 종료하면 메모리 증거가 사라진다 — 격리만 하고 종료하지 않는다

# EBS 스냅샷 생성 (디스크 포렌식용)
aws ec2 create-snapshot --volume-id vol-0123456789 --description "Incident-2024-001"

# 로그 보존 — CloudTrail, VPC Flow Logs, 애플리케이션 로그
# 로그 보존 기간을 사고 인지 전에 미리 충분히 설정해야 한다 (최소 90일)

    3. 영향 범위 파악: 

    # CloudTrail에서 탈취된 키로 수행된 작업 조회
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=AccessKeyId,AttributeValue=AKIA... \
  --start-time 2024-01-01T00:00:00Z

# 어떤 S3 버킷에 접근했는지 확인
aws s3api get-bucket-logging --bucket my-bucket

    4. 원인 제거와 복구: - 취약점이 확인되면 패치 적용 - 모든 시크릿(API 키, DB 비밀번호, JWT 서명 키) 교체 - 영향받은 사용자 비밀번호 강제 재설정 - 서비스 재배포 (클린 이미지에서)

    5. 사후 분석 (Post-mortem): - 타임라인 정리: 최초 침입 시점, 탐지 시점, 격리 시점, 복구 시점 - 탐지까지 걸린 시간(MTTD)과 복구까지 걸린 시간(MTTR) 측정 - 재발 방지 조치 도출

    사고 대응 절차를 사전에 문서화하고 정기적으로 시뮬레이션(Tabletop Exercise)을 하는 팀과 안 하는 팀의 대응 속도는 크게 다르다.
