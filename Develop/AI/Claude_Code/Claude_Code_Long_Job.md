# Claude Code 롱잡 운영

빌드 5분, 마이그레이션 8분, 통합 테스트 12분. 분 단위로 걸리는 작업을 Claude Code 세션 안에서 돌리려고 하면 첫 번째 벽이 Bash 도구의 2분 타임아웃이다. 두 번째 벽은 그걸 넘기려고 폴링을 돌리다 보면 컨텍스트가 토큰으로 가득 찬다는 점이다. 세 번째 벽은 백그라운드로 띄운 작업을 누가 정리할지 정해두지 않았다는 점이다.

이 문서는 단일 세션 안에서 분 단위 작업을 돌리는 운영 방식을 정리한다. 서브에이전트 백그라운드([Claude_Code_Agent.md](Claude_Code_Agent.md))나 cron 스케줄([Claude_Code_Routine.md](Claude_Code_Routine.md))과는 다른 영역이다. 여기서 다루는 건 "지금 이 세션에서 빌드를 띄워두고 다른 작업을 진행하다가 끝나면 결과를 회수한다"는 흐름이다.

---

## 1. 롱잡이 무엇이고 왜 따로 다뤄야 하나

### 1.1 짧은 잡과 롱잡의 경계

Bash 도구의 기본 타임아웃은 120초다. timeout 파라미터를 키워도 최대 600초(10분)까지가 끝이다. 600초를 넘기는 작업은 처음부터 백그라운드로 띄워야 한다. 600초 안쪽이라도 화면이 멈춘 동안 다른 작업을 진행하고 싶으면 백그라운드가 답이다.

실무에서 롱잡으로 분류해두는 작업들:

- 풀빌드 (`docker build`, `next build`, `vite build`로 전체 번들 생성)
- 통합 테스트와 e2e 테스트 (DB 컨테이너 띄우고 마이그레이션 돌리고 시나리오 실행)
- 데이터 마이그레이션 (수십만 건 이상 변환)
- 학습 / 파인튜닝
- npm install이 모노레포에서 도는 경우
- 외부 API를 페이지네이션으로 끝까지 긁어오는 스크립트

### 1.2 단일 세션 안에서 돌려야 하는 이유

cron 루틴이나 외부 스케줄러로 빼면 좋을 것 같지만, 빌드·테스트·마이그레이션은 보통 **방금 수정한 코드 컨텍스트가 살아있는 동안** 결과를 봐야 의미가 있다. 빌드가 깨지면 직전 변경에 원인이 있다. 다른 세션에서 돌리면 그 컨텍스트가 사라지고 다시 복원하는 데 시간이 든다.

서브에이전트로 띄우는 옵션도 있는데, 빌드처럼 **출력 자체가 크고 결과가 단순한 작업**은 서브에이전트가 비효율적이다. 서브에이전트는 자체 컨텍스트를 잡고 도구 호출도 따로 해서, 단순 빌드 한 번 돌리는 데 토큰 비용이 5~10배 뛴다. 이런 작업은 메인 세션이 직접 백그라운드 Bash로 띄우는 편이 깔끔하다.

## 2. run_in_background로 띄우는 첫 단계

### 2.1 Bash 도구의 백그라운드 모드

`run_in_background: true`를 넘기면 Bash 도구는 명령을 띄우자마자 핸들을 반환한다. 핸들에는 프로세스 ID와 출력 파일 경로가 들어있다.

```
Bash({
  command: "npm run build",
  description: "Production build",
  run_in_background: true
})
```

내부적으로는 자식 프로세스를 데몬화하고 stdout/stderr를 임시 파일로 리다이렉트한다. 메인 세션은 다음 도구 호출로 넘어가고, 빌드는 그동안 별도로 돈다.

### 2.2 띄우자마자 무엇을 할지 정해두기

백그라운드로 띄우는 순간 두 가지를 같이 정해야 한다.

먼저 **언제 결과를 확인할지**다. 끝날 때까지 기다릴지, 첫 에러 줄이 나오면 멈출지, 진행률이 50%를 넘으면 다음 단계로 갈지 결정한다.

그 다음은 **그동안 무엇을 할지**다. 백그라운드로 띄워놓고 메인 세션이 멍하니 기다리면 그건 그냥 동기 호출과 다를 게 없다. 띄워둔 사이에 다른 파일을 읽거나, 관련된 코드 변경을 미리 준비해야 백그라운드가 의미가 있다.

```
# 좋은 흐름
1. npm run build를 백그라운드로 띄움
2. 그동안 빌드 산출물을 사용하는 nginx 설정 파일을 확인하고 변경 검토
3. 빌드 완료 알림이 오면 결과 확인
4. 산출물 크기 확인 후 다음 단계 진행

# 나쁜 흐름
1. npm run build를 백그라운드로 띄움
2. BashOutput으로 매초 폴링
3. 폴링 결과를 읽고 또 폴링
```

### 2.3 Agent 도구의 백그라운드 모드와 묶기

`Agent` 도구도 `run_in_background: true`를 받는다. 이건 서브에이전트 자체를 백그라운드로 돌리는 옵션이라 성격이 다르다. 자세한 운영은 [Claude_Code_Agent.md](Claude_Code_Agent.md)에서 다루고, 여기서는 Bash 백그라운드와 묶어 쓰는 패턴 하나만 짚는다.

리서치형 에이전트와 빌드 작업을 동시에 돌릴 때 둘 다 백그라운드로 띄우면 메인 세션은 코드 작성에 집중할 수 있다.

```
1. Agent(Explore, "코드베이스에서 X 패턴 찾기", run_in_background: true) 시작
2. Bash("npm run test:integration", run_in_background: true) 시작
3. 메인 세션은 그동안 새 기능 구현
4. Agent 완료 알림 → 결과 반영
5. 테스트 완료 알림 → 실패 케이스 디버깅
```

이때 주의할 점은 둘 다 같은 파일을 건드리는 작업이면 충돌이 난다는 것이다. 서브에이전트가 worktree로 격리되지 않은 채 코드를 수정하는 동안 메인이 같은 파일을 편집하면 결과가 꼬인다.

## 3. 진행 상황을 추적하는 두 도구

### 3.1 BashOutput — 한 번 찍어보기

`BashOutput` 도구는 백그라운드 작업의 누적 출력을 한 번에 읽어온다. tail -f가 아니라 cat에 가깝다. 호출하는 시점까지 쌓인 출력을 모두 반환한다.

쓰는 시점:

- 작업이 끝났는지 확인하고 싶을 때
- 진행률이 출력에 찍히는 작업의 중간 상태를 보고 싶을 때
- 에러가 났는지 한 번 살피고 싶을 때

큰 문제는 호출할 때마다 누적 출력 전체가 컨텍스트에 들어간다는 것이다. 빌드 로그처럼 수만 줄이 나오는 작업이면 한 번 찍는 것만으로도 토큰 만 단위가 날아간다.

대처법은 두 가지다.

하나는 출력에 진행률 헤더를 찍는 명령으로 바꾸는 것이다. `npm run build --silent` 같은 식으로 노이즈를 줄이거나, 빌드 로그를 파일로 빼고 진행 상황만 stdout에 짧게 찍게 한다.

다른 하나는 마지막 N줄만 보고 싶다면 BashOutput의 출력을 그대로 받는 대신 다른 Bash 호출에서 `tail -100 <log-file>`로 따로 읽는 방법이다. 단, 이렇게 쓰려면 처음 띄울 때 출력을 명시적으로 파일로 빼야 한다.

```
Bash({
  command: "npm run build > /tmp/build.log 2>&1",
  run_in_background: true
})

# 나중에
Bash({ command: "tail -50 /tmp/build.log" })
```

### 3.2 Monitor — 알림으로 받기

`Monitor` 도구는 백그라운드 프로세스의 출력 파일을 따라가면서 새 줄이 추가될 때마다 알림 이벤트로 보낸다. 폴링하지 않고 푸시받는 방식이다.

쓰는 시점:

- 종료 시점이 가변적인데 끝났는지를 빠르게 알고 싶을 때
- 출력에 특정 패턴이 나타나면 다음 행동을 해야 할 때 (예: "Server listening" 같은 줄)

```
# Vite dev server를 띄우고 준비 완료 줄을 기다림
Bash({
  command: "npm run dev > /tmp/dev.log 2>&1",
  run_in_background: true
})

Monitor({
  shell_id: <handle>,
  until: "Local:.*localhost"
})
```

Monitor는 일치하는 줄이 나오면 멈춘다. 그래서 dev 서버 기동 대기 같은 패턴에 잘 맞는다.

빌드처럼 출력이 너무 많은 작업에 Monitor를 붙이면 모든 줄이 알림으로 와서 컨텍스트가 터진다. 이럴 땐 Monitor 대신 종료 후 BashOutput을 한 번 찍거나, `until` 패턴을 빌드 종료 시점의 특정 문자열로 좁힌다.

### 3.3 두 도구의 사용 구분

| 상황 | 선택 |
|---|---|
| 작업 종료까지 기다리기 | run_in_background로 띄우고 완료 알림 대기 |
| 특정 출력 줄을 트리거로 다음 행동 | Monitor with until |
| 중간 상태 한 번만 확인 | BashOutput 1회 |
| 진행률 추적 필요 | stdout은 짧게, log 파일은 별도, BashOutput으로 가끔 확인 |
| 에러 즉시 탐지 | Monitor with until: ERROR 패턴 |

## 4. 알림 기반 대기 — 폴링 안 하기

### 4.1 자동 완료 알림

run_in_background로 띄운 작업이 끝나면 하네스가 자동으로 알림을 보낸다. LLM은 "백그라운드 작업 X가 종료됨"이라는 시스템 메시지를 받고, 그 시점에 결과를 회수하면 된다.

이 메커니즘이 있어서 폴링은 필요 없다. 폴링하지 말라는 가이드는 단순한 권고가 아니라 비용 문제다.

### 4.2 sleep 안티패턴

빌드가 끝났는지 확인하려고 이런 명령을 쓰는 경우가 있다.

```bash
# 안티패턴
sleep 30 && BashOutput
```

이걸 5분짜리 빌드에 쓰면 30초마다 BashOutput을 찍으면서 토큰을 태운다. 그리고 sleep을 걸어둔 셸이 남아있는 동안 메인 세션은 다른 작업을 못한다.

비슷하게 안 좋은 패턴이 무한 루프 폴링이다.

```bash
# 무한 루프 폴링
until grep -q "BUILD SUCCESS" /tmp/build.log; do sleep 5; done
```

이걸 Bash 호출 하나에 넣으면 그 호출 자체가 길게 돌아서 다른 작업도 못한다. 매번 sleep + grep을 분리해서 돌리면 BashOutput과 같은 비용 문제가 생긴다.

### 4.3 정상 흐름

```
1. Bash run_in_background: true 호출 → 핸들 받음
2. 메인 세션은 다른 작업 진행 (코드 읽기, 다른 파일 편집)
3. 시스템 알림: "background job <handle> exited with code 0"
4. BashOutput 1회로 결과 회수
5. 결과 분석 후 다음 단계
```

이 사이클에서 폴링이 등장하면 흐름이 깨지고 있다는 신호다.

### 4.4 ScheduleWakeup — 자율 모드에서

자율 모드(/loop)나 명시적 대기가 필요한 경우 `ScheduleWakeup`을 쓴다. 이 도구는 일정 시간 후 LLM을 다시 깨운다. 폴링과 무엇이 다른가 하면, ScheduleWakeup은 깨어나는 시점까지 LLM이 컨텍스트를 사용하지 않는다는 점이다. 그 사이에 토큰을 안 쓴다.

ScheduleWakeup의 핵심 제약은 **프롬프트 캐시 5분 TTL**이다.

- 60~270초: 캐시가 살아있는 동안 깨어남. 빌드 짧은 거 확인할 때 쓴다.
- 300~3600초: 캐시 미스가 발생한다. 길게 비울 때 한 번에 쓴다.
- 정확히 300초 자는 건 가장 비효율이다. 캐시 미스를 사면서 길게 못 자기 때문.

빌드가 8분 걸린다고 할 때 60초마다 깨어나면 캐시를 8번 새로 채운다. 이때는 270초 자고 한 번 깨고, 다시 270초 자는 식으로 두세 번에 끊는 편이 낫다. 더 긴 작업이라면 1200~1800초 한 번에 자는 편이 합리적이다.

이건 Claude Code의 /loop 같은 자율 모드에서나 직접 쓸 일이 있다. 일반 대화 세션에서는 자동 알림이 와서 ScheduleWakeup을 직접 호출할 일은 거의 없다.

## 5. 분 단위 작업 유형별 처리

### 5.1 빌드

핵심은 출력을 메인 컨텍스트에 끌어들이지 않는 것이다.

```
Bash({
  command: "npm run build > /tmp/build-$(date +%s).log 2>&1",
  run_in_background: true,
  description: "Build with log to file"
})
```

종료 알림이 오면 종료 코드를 먼저 확인한다.

```
Bash({ command: "tail -30 /tmp/build-1715000000.log" })
```

성공이면 산출물 크기 한 줄만 받고 끝낸다. 실패면 마지막 100줄 정도를 보고 원인을 찾는다. 처음부터 전체 로그를 끌어오면 토큰만 낭비한다.

### 5.2 테스트

테스트는 빌드와 다르게 **부분 실패**가 의미 있다. 한 번에 다 돌리고 결과를 받기보다, 실패 케이스만 따로 추적할 수 있게 출력 포맷을 정해둔다.

```
Bash({
  command: "npm run test -- --reporter=json > /tmp/test.json 2>&1",
  run_in_background: true
})
```

JSON 리포트로 받으면 jq로 실패 케이스만 잘라낼 수 있다.

```
Bash({
  command: "jq '.testResults[] | select(.status==\"failed\") | .name' /tmp/test.json"
})
```

테스트가 30분 도는 e2e 같은 경우는 Monitor로 진행 상황을 보면 안 된다. 출력이 너무 많다. 종료까지 기다리고 실패 결과만 회수한다.

### 5.3 마이그레이션

데이터 마이그레이션은 빌드·테스트와 다른 점이 두 가지 있다. 중단되면 데이터가 어중간한 상태로 남는다. 그리고 진행률을 봐야 끝나는 시점을 가늠할 수 있다.

```
# 진행률을 짧게 stdout에 찍는 마이그레이션 스크립트
# 예: "Processed 10000/250000"
Bash({
  command: "node scripts/migrate-users.js > /tmp/migrate.log 2>&1",
  run_in_background: true
})

# 진행률 줄이 1만건 단위로 찍힌다고 할 때
Monitor({
  shell_id: <handle>,
  until: "Processed.*Done"
})
```

마이그레이션이 15분 넘게 도는데 진행률이 안 올라오면 자식 프로세스가 멈춘 게 아닌지 의심해야 한다. 이때는 BashOutput을 1회 찍어서 마지막 줄 시각이 언제인지 본다.

### 5.4 dev 서버 기동

dev 서버는 끝나지 않는 작업이다. 띄우고 준비 완료 줄을 기다리는 패턴이 정석이다.

```
Bash({
  command: "npm run dev > /tmp/dev.log 2>&1",
  run_in_background: true
})

Monitor({
  shell_id: <handle>,
  until: "ready in"
})
```

이후 메인 세션은 다른 작업으로 넘어가고, 세션 종료 직전에 KillBash로 정리한다.

## 6. 결과 회수와 백그라운드 작업 정리

### 6.1 결과 회수 패턴

알림이 와서 백그라운드 작업이 끝났다는 걸 알았을 때, 회수는 이 순서로 한다.

1. 종료 코드 확인 (알림 메시지에 보통 같이 옴)
2. 종료 코드가 0이면 결과 한 줄 요약만 가져오기
3. 0이 아니면 마지막 50~100줄 가져와서 원인 분석
4. 임시 로그 파일 삭제

전체 로그를 그냥 BashOutput으로 끌어오는 건 가능한 한 피한다. 메가 단위 로그가 메시지 히스토리에 남으면 이후 모든 턴에서 같은 토큰을 다시 보낸다.

### 6.2 KillBash로 명시적 종료

작업을 도중에 멈추거나 dev 서버 같은 끝나지 않는 작업을 정리할 때 `KillBash` 도구를 쓴다. 내부적으로는 SIGTERM을 보내고 일정 시간 후에도 종료되지 않으면 SIGKILL을 보낸다.

```
KillBash({ shell_id: <handle> })
```

여기서 자주 마주치는 함정이 있다. dev 서버나 빌드 도구는 자식 프로세스를 띄우는데, SIGTERM이 부모에게만 가고 자식은 좀비로 남는 경우가 있다. nginx, webpack-dev-server, vite, jest의 worker 프로세스가 이런 패턴이다.

```bash
# process group 단위로 종료되도록 setsid로 띄우기
Bash({
  command: "setsid npm run dev > /tmp/dev.log 2>&1 &",
  run_in_background: true
})
```

명령 자체에 trap을 걸어서 자식까지 같이 종료시키는 방법도 있다. 이런 처리를 안 해두면 세션 끝나고 좀비가 남아서 다음 세션 띄울 때 포트 충돌이 난다.

### 6.3 세션 종료 직전 정리

세션을 닫기 전에 살아있는 백그라운드 핸들을 확인하고, 필요한 것만 결과를 회수한 뒤 나머지는 KillBash로 정리한다.

```
# 살아있는 노드/뷰트 잔여 프로세스 확인
Bash({ command: "ps aux | grep -E 'npm|node|vite' | grep -v grep" })
```

세션이 강제 종료되어 정리가 안 됐다면 다음 세션에서 위 명령으로 잔여 프로세스를 찾아 직접 kill한다.

## 7. 장시간 세션에서 만나는 문제들

### 7.1 MCP 서버 끊김

세션이 한 시간 넘게 살아있으면 MCP 서버가 자식 프로세스 상태로 죽어있는 경우가 늘어난다. 메모리 누수, stdio 버퍼 막힘, 자식 프로세스 좀비화가 흔한 원인이다.

증상은 MCP 도구를 호출했을 때 `EPIPE`, 무응답, 타임아웃이다. 백그라운드 빌드와 직접 관련은 없지만 장시간 세션에서 함께 발생한다.

```
# MCP 상태 확인
/mcp

# 죽은 서버 재시작
claude mcp restart <server-name>
```

빌드 완료 알림은 MCP와 별도 채널이라 MCP가 죽어도 알림은 온다. 다만 MCP에 의존하는 후속 작업(예: GitHub MCP로 빌드 결과를 PR에 코멘트)이 있으면 그 단계가 막힌다.

### 7.2 출력 파일 적체

`/tmp` 아래에 만든 로그 파일들이 세션 도중 누적된다. 빌드 한 번에 50MB짜리 로그가 쌓이면 디스크가 빠르게 차고 읽기 속도도 떨어진다.

```bash
# 7일 지난 로그 정리
find /tmp -name "build-*.log" -mtime +7 -delete
```

세션 시작 시점에 한 번 정리하는 훅을 SessionStart에 걸어두는 방법도 있다.

### 7.3 좀비 프로세스 누적

KillBash로 종료했는데 ps로 보면 같은 이름의 프로세스가 남아있는 경우가 있다. 자식이 부모와 다른 process group에 있어서 SIGTERM이 닿지 않은 상태다.

```bash
# vite와 그 자식까지 강제 종료
pkill -9 -f vite

# 특정 포트를 잡고 있는 프로세스 종료
lsof -ti:5173 | xargs kill -9
```

이런 명령은 무관한 프로세스를 죽일 수 있어서 운영 환경에서는 조심해서 쓴다. 로컬 개발 환경에서만 쓴다.

### 7.4 백그라운드 작업이 끝났는데 알림이 안 옴

드물게 자식 프로세스가 종료됐는데 하네스가 알림을 안 보내는 경우가 있다. 부모 프로세스가 wait()를 호출하지 못한 좀비 상태이거나, 출력 파일에 후속 쓰기가 일어나서 종료 감지가 늦은 경우다.

```bash
# 핸들에 해당하는 PID가 살아있는지 확인
ps -p <pid>

# 안 살아있으면 BashOutput으로 결과 회수
```

이런 케이스는 핸들 상태가 stale이라는 뜻이다. BashOutput으로 마지막 출력을 가져오고, 핸들은 더 쓰지 않는다.

### 7.5 컨텍스트 압축이 백그라운드 핸들을 망각

장시간 세션에서 자동 압축이 일어나면 메시지 히스토리가 요약된다. 이때 백그라운드 작업의 핸들 정보가 요약에서 누락되면 LLM이 핸들을 잃고 결과를 못 가져온다.

대비책은 두 가지다.

하나는 백그라운드 핸들을 받자마자 그 정보를 메인 컨텍스트의 텍스트로 한 번 적어두는 것이다. "지금 빌드 핸들 X로 띄움" 같은 한 줄을 응답에 남기면 압축에서도 살아남을 가능성이 높아진다.

다른 하나는 진짜 긴 작업의 핸들과 PID를 외부 파일에 기록하는 것이다.

```bash
echo "build:$(date +%s):<pid>:/tmp/build.log" >> /tmp/active-jobs.txt
```

세션이 압축되거나 새로 시작되어도 이 파일을 보면 살아있는 작업이 뭔지 알 수 있다.

## 8. 정리

롱잡 운영의 핵심은 출력을 메인 컨텍스트에 끌어들이지 않는 것, 폴링 대신 알림을 기다리는 것, 세션 종료 전에 핸들을 정리하는 것이다.

출력은 파일로 빼고 결과는 종료 후 마지막 N줄만 본다. 매 폴링마다 누적 출력이 들어오는 패턴은 토큰을 빠르게 태운다. run_in_background는 끝나면 알림이 오기 때문에 sleep과 BashOutput을 반복하는 패턴은 알림 메커니즘이 있는데 굳이 안 쓰는 거다. 세션을 닫기 전에 KillBash로 핸들을 정리하고, process group 누락이 있으면 ps로 확인해서 직접 kill한다. 다음 세션에서 포트 충돌로 시간을 빼앗기지 않으려면 이 단계를 빼먹지 말아야 한다.
