# Claude Code 모델 선택

[Claude 문서](../Claude/Claude.md)에서 API 관점의 모델 비교를, [Claude Code 실전 팁](Claude_Code_Tips.md)에서 모델 선택을 간단히 다뤘다. 이 문서는 **Claude Code 에이전틱 실행 환경에서** 각 모델이 어떻게 다르게 동작하는지, 그리고 어떤 작업에 어떤 모델을 골라야 하는지에 집중한다.

현재 라인업은 Opus 4.7, Sonnet 4.6, Haiku 4.5다. Opus 4.7부터 1M 컨텍스트가 정식 지원되고, Fast 모드는 Opus 4.6·4.7에서만 동작한다.

---

## 1. 에이전틱 실행에서 모델별 도구 사용 패턴

API에서 Claude를 호출하면 한 번의 요청-응답이다. Claude Code는 다르다. 모델이 스스로 도구를 골라서 호출하고, 결과를 보고, 다음 도구를 고르는 과정을 반복한다. 이 자율적인 판단 과정에서 모델 간 차이가 확연하게 드러난다.

### Opus 4.7 — 넓게 탐색하고, 확인 후 작업한다

Opus 4.7은 코드를 수정하기 전에 주변 맥락을 먼저 파악하려 한다. "이 함수를 고쳐줘"라고 시키면, 해당 함수만 보는 게 아니라 호출하는 쪽, 테스트 코드, 관련 타입 정의까지 Grep과 Glob으로 먼저 훑는다.

```
사용자: "UserService.updateProfile에서 이메일 중복 체크 빠져있어. 추가해줘."

Opus 4.7의 도구 호출 순서:
1. Grep: "updateProfile" → 호출부 확인
2. Read: UserService.ts → 메서드 전체 코드 확인
3. Grep: "email" + "duplicate" → 기존 중복 체크 패턴 탐색
4. Read: UserRepository.ts → DB 접근 계층 확인
5. Read: user.service.spec.ts → 기존 테스트 패턴 확인
6. Edit: UserService.ts → 중복 체크 로직 추가
7. Edit: user.service.spec.ts → 테스트 추가
8. Bash: npm test → 검증
```

도구 호출이 많아서 비용과 시간이 든다. 하지만 "기존 패턴을 따라서" 수정하기 때문에, 코드 스타일이 튀지 않고 엣지 케이스를 놓치는 일이 적다. 4.6 대비 4.7은 사전 탐색 단계에서 중복 Read를 덜 하는 편이다. 같은 파일을 3번씩 읽어들이는 일이 줄었다.

### Sonnet 4.6 — 직선적으로 해결한다

Sonnet은 목표 지점으로 바로 간다. 같은 요청을 받으면 해당 파일을 읽고, 수정하고, 테스트를 돌린다. 중간에 주변 코드를 넓게 탐색하는 단계가 짧다.

```
Sonnet 4.6의 도구 호출 순서:
1. Read: UserService.ts
2. Edit: UserService.ts → 중복 체크 로직 추가
3. Bash: npm test → 검증
```

대부분의 일반적인 작업에서는 이 정도면 충분하다. 문제가 생기는 건 수정 범위가 여러 파일에 걸치거나, 기존 패턴과 다른 방식으로 구현해버릴 때다. Sonnet이 Repository에서 `findByEmail`이 이미 있는데도 직접 쿼리를 작성하는 식이다.

### Haiku 4.5 — 단순 작업에서만 쓴다

Haiku는 변수명 변경, 주석 추가, import 정리 같은 단순 작업에 적합하다. 멀티스텝 도구 호출이 필요한 작업에서는 중간에 맥락을 잃는 경우가 잦다. 앞에서 읽은 파일 내용을 다음 도구 호출에서 제대로 반영하지 못하거나, 도구 호출 자체를 잘못 구성하는 경우가 있다.

"파일 3개를 읽고, 공통 인터페이스를 추출해줘" 같은 작업을 Haiku에게 시키면, 첫 번째 파일의 구조를 세 번째 파일을 처리할 때 이미 잊어버리는 식이다.

## 2. Fast 모드의 정체

Claude Code에서 `/fast`로 토글하거나 설정에서 Fast 모드를 켤 수 있다. 이걸 켜면 "Sonnet으로 바뀌는 건가?" 하고 착각하기 쉽다. **모델은 바뀌지 않는다.** Opus를 쓰고 있었으면 여전히 Opus다.

Fast 모드는 **출력 생성 속도를 높이는 대신 일부 추론 깊이를 줄이는 설정**이다. 체감상 응답이 빨라지고, 간단한 작업에서는 결과 차이가 거의 없다. Fast 모드는 Opus 4.6과 4.7에서만 제공된다. Sonnet·Haiku에는 적용되지 않는다.

### 켜는 게 나을 때

- 단순한 파일 수정, 포맷팅, import 정리
- 명확한 지시가 있는 단일 파일 작업
- 대화형으로 빠르게 주고받아야 할 때

### 끄는 게 나을 때

- 여러 파일에 걸친 리팩토링
- 버그 원인 추적 (탐색 범위가 넓은 작업)
- 아키텍처 설계나 의사결정이 필요한 작업

기본 Opus에서 Fast 모드를 켜면 Sonnet 정도의 속도로 Opus 수준의 결과를 내는 느낌이다. 다만 복잡한 멀티스텝 작업에서는 Fast 모드가 탐색 깊이를 줄여서 Opus의 장점인 "넓게 보는 능력"이 약해진다.

## 3. 컨텍스트 윈도우와 1M 확장

Claude Code는 대화가 이어질수록 도구 호출 결과, 파일 내용, 명령 출력이 컨텍스트에 쌓인다. 모델별 컨텍스트 윈도우 크기가 세션의 수명을 결정한다.

| 모델 | 기본 윈도우 | 확장 | Claude Code 세션에서의 의미 |
|------|------------|------|---------------------------|
| Opus 4.7 | 200K | 1M | 대규모 코드베이스 탐색 가능. 파일 수십 개를 읽어도 여유 있음 |
| Sonnet 4.6 | 200K | — | 일반 작업에는 충분. 파일 20~30개 넘으면 자동 압축 시작 |
| Haiku 4.5 | 200K | — | 동일한 200K지만, 긴 컨텍스트에서 정보 활용 능력이 떨어짐 |

### 1M 컨텍스트가 실제로 필요한 시점

Opus 4.7은 1M 토큰까지 컨텍스트를 확장할 수 있다. 그런데 1M을 켰다고 항상 1M을 쓰는 건 아니다. 입력이 200K를 넘는 순간부터 1M 단가가 적용되니까, 굳이 켜놓을 필요가 없으면 기본 200K로 두는 게 낫다.

1M이 실제로 효과를 보는 경우는 다음과 같다.

- **모노레포 전체 흐름 추적**: 컨트롤러 → 서비스 → 리포지토리 → 미들웨어 → 설정까지 파일 30~50개를 읽으면서도 앞 내용을 유지해야 할 때. 200K로는 중간에 자동 압축이 발생하면서 앞부분 맥락이 날아간다.
- **장시간 디버깅 세션**: 같은 세션에서 가설 검증을 반복하는데, 앞쪽에서 본 로그·스택트레이스·코드를 계속 참조해야 할 때.
- **여러 파일에 걸친 리팩토링**: 30개 이상의 파일을 동시에 일관된 패턴으로 바꿔야 할 때. 압축이 발생하면 앞에서 정한 네이밍 규칙을 잊는다.

반대로 1M이 의미 없는 경우도 명확하다.

- **단일 파일 작업**: 파일 하나만 보면 끝나는 일에 1M을 켜놓으면 단가만 비싸진다.
- **세션을 짧게 끊어 쓰는 패턴**: `/clear`로 자주 초기화하는 사용자에게 1M은 의미가 없다.
- **명확한 지시 + 좁은 범위**: 컨텍스트가 200K를 넘을 일이 없는 작업.

`/cost`로 현재 세션 입력 토큰을 보고, 평소 100K 안팎이면 1M은 꺼두는 게 낫다. 200K를 자주 넘기는 패턴이라면 그때 1M을 고려한다.

### 컨텍스트 절약 습관

모델과 상관없이 세션을 오래 쓰면 컨텍스트가 차는 건 같다.

- 작업이 바뀌면 `/clear`로 초기화
- 탐색이 길어졌으면 `/compact`로 압축 후 계속
- "전체 프로젝트 분석" 같은 넓은 범위 요청은 피하고, 디렉토리나 모듈 단위로 쪼개서 요청

## 4. 작업 유형별 모델 선택

같은 코드베이스에서도 작업 성격에 따라 적정 모델이 다르다. 아래 분류는 5년차 백엔드 개발자가 일주일치 실무에서 마주치는 작업 유형을 기준으로 한다.

### 버그 수정

| 상황 | 권장 모델 | 이유 |
|------|----------|------|
| 에러 메시지 명확, 원인 위치 짐작됨 | Sonnet 4.6 | 직선적 수정이 빠름 |
| 재현 가능하지만 원인 모름 | Opus 4.7 (Fast off) | 호출 경로 추적이 필요 |
| 간헐적·재현 어려운 버그 | Opus 4.7 + Extended Thinking | 가설 수립과 검증 반복이 필요 |
| 타입 에러, lint 위반 | Haiku 4.5 또는 Sonnet 4.6 (Fast 없음) | 단순 수정 |

### 리팩토링

| 상황 | 권장 모델 | 이유 |
|------|----------|------|
| 단일 파일 내 메서드 추출·변수명 변경 | Sonnet 4.6 | 범위가 좁음 |
| 여러 파일에 걸친 패턴 변경 | Opus 4.7 | 호출부 일관성 유지가 핵심 |
| 인터페이스·시그니처 변경 | Opus 4.7 | 영향 범위 분석이 우선 |
| 단순 포맷팅·import 정리 | Haiku 4.5 | 판단 필요 없음 |

### 신규 기능 개발

| 상황 | 권장 모델 | 이유 |
|------|----------|------|
| 기존 패턴이 명확한 CRUD 추가 | Sonnet 4.6 | 패턴 복제로 충분 |
| 새 도메인·새 아키텍처 결정 필요 | Opus 4.7 (설계) → Sonnet 4.6 (구현) | 설계와 구현 분리 |
| 외부 API 연동, 인증 흐름 | Opus 4.7 | 엣지 케이스가 많음 |
| 마이그레이션 스크립트 | Opus 4.7 | 데이터 손실 위험 |

### 문서화

| 상황 | 권장 모델 | 이유 |
|------|----------|------|
| README·주석 추가 | Sonnet 4.6 (Fast 가능) | 빠른 처리 |
| 아키텍처 문서, ADR | Opus 4.7 | 의사결정 근거 정리 |
| 변경 로그·릴리스 노트 | Haiku 4.5 또는 Sonnet 4.6 | 단순 정리 |

### 대규모 탐색

| 상황 | 권장 모델 | 이유 |
|------|----------|------|
| "이 API의 요청 흐름 알려줘" | Opus 4.7 | 다단계 탐색 |
| "이 모듈에서 deprecated된 함수 다 찾아줘" | Sonnet 4.6 | 단일 패턴 검색 |
| 모노레포 전체 의존성 분석 | Opus 4.7 + 1M 컨텍스트 | 파일 수가 많음 |
| 특정 파일에서 패턴 검색 | Haiku 4.5 | Grep 호출 위주 |

## 5. Fast 모드 + 모델 의사결정 매트릭스

Fast 모드는 Opus에서만 동작한다는 점을 다시 확인하고 시작한다. Sonnet·Haiku에는 Fast 모드가 없으니, 이 매트릭스는 Opus 사용 시점의 보조 결정이다.

| 작업 성격 | 1차 선택 | Fast 모드 | 비고 |
|----------|---------|-----------|------|
| 단순 수정 (1~2 파일) | Sonnet 4.6 | — | Opus까지 갈 필요 없음 |
| 일반 기능 개발 | Sonnet 4.6 | — | 막히면 Opus로 전환 |
| 패턴 복제형 작업 | Sonnet 4.6 | — | 기존 코드 따라가기 |
| 넓은 탐색 필요 | Opus 4.7 | OFF | Fast 켜면 탐색 깊이가 줄어 손해 |
| 빠른 응답 + 정확도 둘 다 | Opus 4.7 | ON | 대화형 작업에서 체감 큼 |
| 설계·아키텍처 결정 | Opus 4.7 | OFF | 추론 깊이 보존 |
| 디버깅 (원인 추적) | Opus 4.7 | OFF | Fast는 가설 수립 단계에서 약해짐 |
| 대량 단순 작업 | Haiku 4.5 | — | 판단 필요 없는 일 |
| 검증·실행 | Haiku 4.5 | — | 테스트 돌리고 결과 정리 |

판단 기준을 한 줄로 요약하면: **속도가 답답한데 작업이 간단하면 Fast ON, 작업이 복잡하면 Fast OFF.** 비용 차이는 없으니 결과 품질만 기준으로 결정한다.

## 6. 모델 전환 트리거

Sonnet으로 시작했다가 Opus로 올려야 하는 순간이 있다. 반대로 Opus로 시작했는데 Sonnet으로 내려도 되는 경우도 있다. 아래 신호가 나타나면 전환을 고민한다.

### Sonnet → Opus 전환 신호

- **같은 파일을 3번 이상 읽는다**: 첫 읽기에서 맥락을 못 잡았다는 뜻이다. Sonnet의 한계.
- **테스트 실패 원인을 못 찾고 추측만 한다**: "이것 때문일 수도 있고, 저것 때문일 수도 있다"는 답이 반복되면 탐색이 부족하다.
- **수정 후 다른 곳이 깨진다**: 영향 범위 분석이 안 된 상태로 수정했다. 호출부를 못 본 것.
- **같은 버그를 2번 이상 시도해도 못 잡는다**: 첫 시도가 빗나갔으면 두 번째도 Sonnet으로 시도하지 말고 Opus로 올려라.
- **요구사항에 "기존 패턴 유지" 같은 제약이 명시되어 있다**: Sonnet은 패턴을 무시하고 새로 짜는 경향이 있다.

### Opus → Sonnet 다운그레이드 신호

- **설계가 끝났고, 구현만 남았다**: 설계 결과를 CLAUDE.md나 별도 파일에 저장하고 새 세션을 Sonnet으로 연다.
- **반복 작업이 시작됐다**: 같은 패턴으로 20개 파일을 똑같이 수정해야 한다면 Sonnet이 더 빠르고 저렴하다.
- **단순 명령 실행·결과 보고 단계**: 테스트 돌리고 결과 정리하는 일은 Sonnet도 충분하고, Haiku도 가능하다.
- **`/cost`가 $5를 넘었는데 작업이 50%도 안 끝났다**: 비용 통제가 필요한 시점. 남은 작업을 Sonnet으로 옮긴다.

### 세션 중 전환 방법

```bash
# 방법 1: 슬래시 명령
/model claude-opus-4-7

# 방법 2: Shift+Tab 단축키

# 방법 3: 시작 시 지정
claude --model claude-opus-4-7
```

세션 중에 모델을 바꾸면 컨텍스트는 유지된다. Sonnet으로 작업하다가 "이 부분은 복잡하니까 Opus로 바꿔야겠다" 싶으면 바로 전환하면 된다. 다만 Sonnet이 200K 가까이 컨텍스트를 채운 상태에서 Opus로 전환하면, 그 시점부터 Opus 단가로 누적 컨텍스트가 재전송된다. 비용 점프가 크다.

## 7. 세션 기준 비용 비교

API 토큰 단가표만 보면 "Opus가 5배 비싸네"로 끝나는데, Claude Code 세션에서의 실제 비용은 그보다 복잡하다. 같은 작업을 시켜도 모델마다 도구 호출 횟수와 탐색 범위가 다르기 때문이다.

### 모델별 토큰 단가 (2026-05 기준)

| 모델 | 입력 ($/1M) | 출력 ($/1M) | 캐시 읽기 ($/1M) | 캐시 쓰기 ($/1M) |
|------|------------|------------|----------------|----------------|
| Opus 4.7 (≤200K) | 15 | 75 | 1.5 | 18.75 |
| Opus 4.7 (>200K, 1M 모드) | 30 | 150 | 3.0 | 37.5 |
| Sonnet 4.6 | 3 | 15 | 0.3 | 3.75 |
| Haiku 4.5 | 1 | 5 | 0.1 | 1.25 |

1M 모드는 입력이 200K를 넘는 순간부터 적용되니까, 평소 작업에서는 200K 단가만 신경 쓰면 된다.

### 토큰 비용 계산 예시

세션 누적 입력 300K, 출력 30K로 끝난 작업을 가정한다.

**Sonnet 4.6:** 300K × $3 + 30K × $15 = $0.9 + $0.45 = **$1.35**

**Opus 4.7 (200K 모드, 캐시 없이):** 300K × $15 + 30K × $75 = $4.5 + $2.25 = **$6.75**

**Opus 4.7 (200K 모드, 캐시 70% 활용):** (90K × $15 + 210K × $1.5) + 30K × $75 = $1.35 + $0.315 + $2.25 = **$3.92**

**Opus 4.7 (1M 모드, 입력 500K):** 500K × $30 + 50K × $150 = $15 + $7.5 = **$22.5**

캐시 히트가 절반만 나도 Opus 비용이 40% 가까이 떨어진다. 같은 세션에서 같은 파일을 반복 참조하는 패턴이라면 캐시 효과가 크다. `/clear`를 너무 자주 누르면 캐시가 날아간다는 점은 주의한다.

### 30분 작업 세션 비용 비교 (체감 기준)

"NestJS 프로젝트에서 인증 모듈에 refresh token rotation 기능 추가"라는 중간 규모 작업을 각 모델로 수행했을 때의 패턴이다.

| 항목 | Opus 4.7 | Sonnet 4.6 | Haiku 4.5 |
|------|---------|-----------|-----------|
| 도구 호출 횟수 | 25~40회 | 10~20회 | 8~15회 |
| 읽는 파일 수 | 10~15개 | 4~8개 | 2~5개 |
| 입력 토큰 (누적) | 300K~500K | 100K~200K | 50K~100K |
| 출력 토큰 | 30K~50K | 15K~30K | 10K~20K |
| 대략 비용 (캐시 없음) | $7~14 | $0.6~1.5 | $0.1~0.3 |
| 대략 비용 (캐시 60%) | $4~8 | $0.4~1.0 | $0.07~0.2 |

Opus는 넓게 탐색하는 만큼 도구 호출이 많고, 매 호출마다 누적 컨텍스트가 재전송되니까 비용이 빠르게 늘어난다. Sonnet 대비 단가가 5배인데 도구 호출도 2배 이상이라, 총비용은 **5~10배 차이**가 나는 경우가 흔하다.

### Max 플랜 vs API 직접 과금

Claude Code의 과금 방식은 두 가지다.

**API 키 방식 (종량제)**

- Anthropic Console에서 API 키 발급 후 사용
- 토큰 단위 정확한 과금
- Opus를 많이 쓰면 비용이 빠르게 올라감
- `/cost`로 실시간 비용 확인 가능

**Max 플랜 (월 구독)**

- Pro ($20/월), Max ($100/월, $200/월) 등 구독 플랜
- 사용량 제한이 있지만 구간 내에서는 추가 비용 없음
- 무거운 작업을 하다 보면 하루 만에 사용량 제한에 걸리는 경우가 있음

**선택 기준:**

API 직접 과금이 맞는 경우:

- CI/CD 파이프라인에서 Claude Code를 돌릴 때
- 사용량 예측이 가능하고, 월 비용이 $100 이하일 때
- Haiku나 Sonnet 위주로 쓸 때 (단가가 낮아서 월 구독 대비 저렴)

Max 플랜이 맞는 경우:

- Opus 4.7을 주력으로 쓸 때 (API 과금이면 하루에 $50 넘길 수 있음)
- 비용 예측 없이 자유롭게 쓰고 싶을 때
- 팀원이 여럿이고 각자 API 키 관리가 번거로울 때

Max $200 플랜에서 Opus 4.7을 하루 종일 쓰면 보통 2~3일 안에 사용량 제한에 걸린다. 그 시점에서 Sonnet으로 전환하거나 다음 리셋까지 기다려야 한다. 작업 패턴에 따라 API 과금이 더 자유로울 수 있다.

## 8. 오케스트레이션에서 모델 조합

[Claude Code 오케스트레이션 문서](Claude_Code_Orchestration.md)에서 다단계 워크플로우를 다뤘다. 여기서는 각 단계에 어떤 모델을 배치하는지, 그 근거가 뭔지를 다룬다.

### 기본 조합: 설계 Opus, 구현 Sonnet, 검증 Haiku

```typescript
// 1단계: Opus 4.7이 설계한다
const planner = new ClaudeCode({ model: "claude-opus-4-7", maxTurns: 5 });
const plan = await planner.run({
  prompt: "auth 모듈을 리팩토링할 계획을 JSON으로 출력해.",
});

// 2단계: Sonnet 4.6이 구현한다
const builder = new ClaudeCode({ model: "claude-sonnet-4-6", maxTurns: 30 });
await builder.run({
  prompt: `다음 계획대로 코드를 수정해:\n${plan.output}`,
});

// 3단계: Haiku 4.5가 검증한다
const checker = new ClaudeCode({ model: "claude-haiku-4-5", maxTurns: 10 });
const review = await checker.run({
  prompt: "npm test를 실행하고, 실패한 테스트가 있으면 원인을 보고해.",
});
```

**왜 이렇게 나누는가:**

- **설계에 Opus 4.7을 쓰는 이유**: 리팩토링 범위를 잡을 때 "이 파일만 고치면 되나, 아니면 호출하는 쪽도 바꿔야 하나"를 판단하는 능력이 Sonnet보다 확실히 낫다. 넓게 탐색하는 Opus의 특성이 설계 단계에서 빛난다.
- **구현에 Sonnet 4.6을 쓰는 이유**: 계획이 구체적으로 나와 있으면 직선적으로 처리하는 Sonnet이 더 빠르고 저렴하다. 설계를 Opus가 해줬으니 Sonnet이 주변을 넓게 탐색할 필요가 없다.
- **검증에 Haiku 4.5를 쓰는 이유**: "테스트 돌리고 결과 보고"는 판단이 필요한 작업이 아니다. 명령 실행하고 출력 정리만 하면 되니까 가장 저렴한 모델이면 된다.

### 이 조합이 안 맞는 경우

**작업이 작을 때**: "함수 하나 수정하고 테스트 돌려줘" 같은 작업을 3단계로 나누면 오버헤드만 늘어난다. Sonnet 하나로 한 번에 처리하는 게 낫다.

**Haiku 검증이 실패하는 경우**: 테스트가 실패했을 때 "왜 실패했고 어떻게 고쳐야 하는지"까지 판단하게 하면 Haiku가 엉뚱한 수정을 시도하는 경우가 있다. 검증 단계에서 수정까지 해야 한다면 Sonnet 이상을 써야 한다.

**설계가 불필요한 경우**: 명확한 지시가 있는 작업("이 API에 페이지네이션 파라미터 추가해줘")은 설계 단계를 건너뛰고 바로 Sonnet으로 구현해도 된다.

### 비용 관점에서의 조합 효과

위 3단계 조합을 전부 Opus 4.7로 돌렸을 때와 비교하면:

| 방식 | 예상 비용 | 결과 품질 |
|------|-----------|-----------|
| 전부 Opus 4.7 | $10~16 | 높음 |
| Opus + Sonnet + Haiku 조합 | $3~5 | 거의 동일 |
| 전부 Sonnet | $1~2 | 설계 단계에서 누락 가능 |

조합을 쓰면 Opus만 쓸 때 대비 비용이 1/3 정도로 줄면서, 결과 품질은 큰 차이가 없다. 비용이 아깝다면 Opus를 설계에만 쓰고 나머지는 Sonnet으로 통일하는 것도 괜찮다.

## 9. 실전 모델 선택 체감

마지막으로 일주일치 실무에서 모델을 어떻게 섞어 썼는지 정리한다.

- **월요일 아침 코드리뷰**: Sonnet 4.6으로 시작. PR 5개를 가볍게 훑고 의견 정리.
- **화요일 신규 기능 설계**: Opus 4.7 (Fast OFF)로 도메인 모델 설계, 인터페이스 정의. 결과를 CLAUDE.md에 저장.
- **수요일 구현**: 새 세션을 Sonnet 4.6으로 열어 화요일 설계대로 구현. 막힌 부분 한 번은 Opus로 전환.
- **목요일 버그 트래킹**: 간헐적 버그. Opus 4.7 + Extended Thinking. 1M 컨텍스트는 켜지 않음 (입력 100K 이내).
- **금요일 정리**: Haiku 4.5로 변수명 통일, import 정렬, 주석 추가. Sonnet으로 릴리스 노트 초안 작성.

이 정도 패턴이면 한 주에 API 과금으로 $30~50 선이다. 같은 작업을 전부 Opus로 했으면 $200 가까이 나왔을 거다.
