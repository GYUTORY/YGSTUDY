# Claude Code Memory 시스템

## 1. Memory 시스템 개요

Claude Code는 대화가 끝나면 컨텍스트가 사라진다. 다음 대화에서 같은 설명을 반복하지 않으려면 메모리 시스템을 써야 한다.

메모리는 `~/.claude/projects/<프로젝트경로>/memory/` 디렉토리에 마크다운 파일로 저장된다. 각 파일은 YAML frontmatter로 메타데이터를 갖고, `MEMORY.md`가 인덱스 역할을 한다.

```
~/.claude/projects/-Users-dev-myproject/memory/
├── MEMORY.md              # 인덱스 파일
├── user_role.md           # 사용자 정보
├── feedback_testing.md    # 피드백
├── project_auth.md        # 프로젝트 상태
└── reference_linear.md    # 외부 참조
```

핵심 구조를 정리하면 이렇다:

- **MEMORY.md**: 메모리 파일들의 인덱스. 대화 시작 시 항상 로드된다. 200줄이 넘으면 잘리니까 한 줄에 하나씩 간결하게 적어야 한다.
- **개별 메모리 파일**: 실제 내용을 담는 파일. frontmatter에 name, description, type을 명시한다.
- **자동 로드**: MEMORY.md는 매 대화 시작 시 자동으로 컨텍스트에 포함된다. 개별 메모리 파일은 관련성이 있을 때 읽힌다.

## 2. Auto Memory 동작 방식

Claude Code의 메모리는 "auto memory"로 동작한다. 사용자가 명시적으로 "이거 기억해"라고 말하면 즉시 저장하고, 대화 중에 중요한 정보를 발견하면 자동으로 저장 여부를 판단한다.

### 2.1 저장 과정

메모리 저장은 두 단계로 이루어진다:

**1단계 - 메모리 파일 생성**

```markdown
---
name: 사용자 역할
description: 사용자의 직무와 기술 수준 — 응답 수준 조절에 사용
type: user
---

백엔드 5년차 개발자. Spring Boot, Kotlin 주력.
프론트엔드는 React 기본 수준만 알고 있다.
프론트 관련 설명 시 백엔드 개념에 대응시켜 설명하는 게 낫다.
```

**2단계 - MEMORY.md에 포인터 추가**

```markdown
- [사용자 역할](user_role.md) — 백엔드 5년차, Spring/Kotlin 주력
```

MEMORY.md에는 내용을 직접 쓰지 않는다. 항상 개별 파일을 만들고, 인덱스에는 한 줄짜리 링크만 남긴다.

### 2.2 저장하지 않는 것

메모리에 저장하면 안 되는 정보가 있다:

- **코드 패턴, 아키텍처, 파일 경로**: 코드를 직접 읽으면 되니까 메모리에 복제할 이유가 없다
- **Git 히스토리, 변경 이력**: `git log`가 정확하다
- **디버깅 해결법**: 수정 내역은 코드와 커밋 메시지에 있다
- **CLAUDE.md에 이미 적힌 내용**: 중복이다
- **현재 대화에서만 쓰이는 임시 정보**: 태스크나 플랜을 써야 한다

사용자가 PR 목록이나 활동 요약을 저장해달라고 하면, 그 중 "예상과 달랐던 것"이 뭔지 물어본다. 그 부분만 저장할 가치가 있다.

## 3. 메모리 타입별 활용

네 가지 타입이 있고, 각각 저장 시점과 활용 맥락이 다르다.

### 3.1 user — 사용자 정보

사용자가 누구인지, 어떤 기술 수준인지, 어떤 방식으로 일하는지 기록한다.

```markdown
---
name: 기술 수준
description: 사용자의 기술 스택과 수준 — 답변 깊이 조절용
type: user
---

Go 10년차. React는 이 프로젝트에서 처음 만졌다.
프론트 설명할 때 Go의 인터페이스, 채널 같은 개념에 비유하면 잘 받아들인다.
```

**저장 시점**: 사용자가 자기 역할, 기술 수준, 선호도를 언급할 때

**활용 예시**: 코드 설명 요청이 들어오면 이 메모리를 참고해서 설명 수준을 맞춘다. 시니어한테 기초 설명을 장황하게 하면 시간 낭비다.

### 3.2 feedback — 작업 방식 피드백

사용자가 "이렇게 하지 마", "이게 맞아" 같은 피드백을 줬을 때 기록한다. 수정 피드백만 저장하면 점점 소극적으로 변하니까, 잘 했을 때의 확인도 같이 저장한다.

```markdown
---
name: 테스트 방식
description: 테스트 작성 시 DB 모킹 금지 — 실DB 사용 필수
type: feedback
---

통합 테스트에서 DB를 모킹하지 않는다.

**Why:** 지난 분기에 모킹한 테스트가 통과했는데 프로덕션 마이그레이션이 깨진 적이 있다.
**How to apply:** 테스트 코드 작성 시 항상 실제 DB(testcontainers 등)를 사용한다.
```

**저장 시점**: 사용자가 작업 방식을 교정하거나 확인할 때. "그건 아니야"도, "맞아 그렇게 해"도 모두 해당한다.

**구조 규칙**: 규칙 → Why(이유) → How to apply(적용 방법)으로 작성한다. 이유가 있어야 엣지 케이스에서 판단할 수 있다.

### 3.3 project — 프로젝트 상태

코드나 Git에서 알 수 없는 프로젝트 맥락을 기록한다. 누가 무엇을 왜, 언제까지 하는지가 핵심이다.

```markdown
---
name: 머지 프리즈
description: 모바일 릴리스 커트로 인한 머지 제한 — 비핵심 PR 주의
type: project
---

2026-03-05부터 비핵심 머지 프리즈.
모바일 팀이 릴리스 브랜치를 만든다.

**Why:** 릴리스 브랜치 안정성 확보 목적
**How to apply:** 비핵심 PR 작업을 해당 일자 이후로 잡지 않도록 주의한다.
```

**저장 시점**: 상대 날짜("목요일까지")는 반드시 절대 날짜(2026-03-05)로 변환해서 저장한다. 나중에 읽었을 때 "목요일"이 언제인지 알 수 없기 때문이다.

**주의**: 프로젝트 메모리는 빠르게 낡는다. Why가 있어야 나중에 "이 메모리가 아직 유효한가?"를 판단할 수 있다.

### 3.4 reference — 외부 리소스 참조

외부 시스템의 위치와 용도를 기록한다.

```markdown
---
name: 파이프라인 버그 트래커
description: Linear INGEST 프로젝트 — 파이프라인 버그 추적 위치
type: reference
---

파이프라인 관련 버그는 Linear의 "INGEST" 프로젝트에서 관리한다.
관련 이슈를 언급하거나 확인할 때 이 프로젝트를 참조한다.
```

**저장 시점**: 사용자가 "~는 여기서 확인해", "~는 이 채널에 있어" 같이 외부 리소스 위치를 알려줄 때

## 4. MEMORY.md 인덱스 관리

MEMORY.md는 메모리 시스템의 진입점이다. 매 대화 시작 시 자동으로 로드되기 때문에 관리 방식이 중요하다.

### 4.1 기본 규칙

- **200줄 제한**: 넘으면 잘린다. 한 항목 한 줄로 쓴다.
- **frontmatter 없음**: MEMORY.md 자체는 인덱스지 메모리가 아니다.
- **내용 금지**: 실제 내용은 개별 파일에 쓰고, 여기에는 링크와 한 줄 설명만 남긴다.
- **150자 이내**: 각 항목은 `- [제목](파일명.md) — 한 줄 설명` 형태로 150자를 넘기지 않는다.

### 4.2 좋은 예시와 나쁜 예시

```markdown
# 나쁜 예시 — 내용을 직접 적음
- 사용자는 백엔드 5년차이고 Spring Boot를 주로 사용하며 React는 잘 모른다

# 좋은 예시 — 링크만 남김
- [사용자 역할](user_role.md) — 백엔드 5년차, Spring/Kotlin 주력
```

### 4.3 인덱스 정리 시점

메모리가 쌓이면 정리가 필요하다:

- 중복 메모리가 생겼을 때 → 하나로 합치고 인덱스에서 삭제
- 프로젝트 메모리의 마감일이 지났을 때 → 아직 유효한지 확인 후 삭제 또는 갱신
- 200줄에 근접했을 때 → 더 이상 관련 없는 항목부터 정리

## 5. 메모리 vs 태스크 vs 플랜

Claude Code에는 정보를 유지하는 세 가지 수단이 있다. 용도가 다르니까 섞어 쓰면 안 된다.

| 구분 | 메모리 | 태스크 | 플랜 |
|------|--------|--------|------|
| **범위** | 대화를 넘어 유지 | 현재 대화 내 | 현재 대화 내 |
| **용도** | 사용자/프로젝트 맥락 | 작업 진행 추적 | 구현 방향 합의 |
| **예시** | "DB 모킹 금지" | "1. 모델 수정 2. 마이그레이션 3. 테스트" | "인증 모듈을 JWT에서 세션으로 바꾸는 계획" |
| **저장 위치** | memory/ 디렉토리 | 대화 내 TodoWrite | 대화 내 Plan 모드 |

### 5.1 판단 기준

**메모리를 쓸 때**: 다음 대화에서도 알아야 하는 정보인가? "이 사용자는 한국어 문서를 선호한다" 같은 정보는 메모리다.

**태스크를 쓸 때**: 지금 하고 있는 작업의 진행 상황을 추적해야 하는가? "파일 5개 수정 중 3개 완료" 같은 건 태스크다. 대화가 끝나면 의미가 없다.

**플랜을 쓸 때**: 구현 방향에 대해 합의가 필요한가? "이 리팩토링을 어떤 순서로 할지" 같은 건 플랜이다. 진행 중에 방향이 바뀌면 플랜을 업데이트한다. 메모리에 저장하는 게 아니다.

### 5.2 흔한 실수

- 작업 단계를 메모리에 저장 → 다음 대화에서 이미 완료된 단계를 보게 된다. 태스크를 써야 한다.
- 구현 계획을 메모리에 저장 → 코드가 바뀌면 메모리와 실제가 달라진다. 플랜을 쓰고 대화 내에서 관리한다.
- 코드 구조를 메모리에 저장 → 리팩토링하면 틀린 정보가 된다. 코드를 직접 읽는 게 맞다.

## 6. Stale 메모리 대응

메모리는 작성 시점의 사실을 기록한 것이다. 시간이 지나면 틀릴 수 있다.

### 6.1 검증이 필요한 경우

메모리에 파일 경로가 적혀 있다면, 그 파일이 아직 존재하는지 확인한다. 함수명이나 플래그가 적혀 있다면 grep으로 찾아본다. "메모리에 X가 있다"와 "X가 지금 존재한다"는 다른 말이다.

```
# 메모리에 "utils/auth.ts의 validateToken 함수" 라고 적혀 있을 때
# 바로 사용자에게 안내하지 않고, 먼저 확인한다

grep -r "validateToken" src/utils/
# 파일이 이동했거나 함수명이 바뀌었을 수 있다
```

### 6.2 충돌 시 처리

메모리의 내용과 현재 코드/상태가 다를 때:

1. **현재 상태를 신뢰한다** — 메모리가 아니라 지금 보이는 것이 맞다
2. **메모리를 갱신하거나 삭제한다** — 틀린 메모리를 방치하면 다음에 또 혼란이 생긴다
3. **사용자에게 알린다** — "이전에 저장된 정보와 현재 상태가 다릅니다"라고 말한다

### 6.3 타입별 부패 속도

- **user**: 느리게 변한다. 사용자의 기술 수준이나 선호도는 수개월 단위로 바뀐다.
- **feedback**: 거의 안 변한다. "DB 모킹하지 마"는 조직의 결정이 바뀌지 않는 한 유효하다.
- **project**: 가장 빨리 변한다. 마감일, 담당자, 진행 상태는 매주 바뀔 수 있다.
- **reference**: 중간 수준. 외부 시스템의 URL이나 구조가 바뀔 수 있다.

프로젝트 타입 메모리를 읽을 때는 항상 "이게 아직 유효한가?"를 먼저 생각한다.

## 7. 팀 환경에서 메모리 관리

메모리는 기본적으로 개인별이다. `~/.claude/projects/` 경로가 사용자의 홈 디렉토리에 있기 때문에 팀원 간에 공유되지 않는다.

### 7.1 개인 메모리 vs 공유 설정

| 항목 | 개인 메모리 | CLAUDE.md |
|------|-----------|-----------|
| **위치** | `~/.claude/projects/...` | 프로젝트 루트 |
| **Git 추적** | 안 됨 | 됨 |
| **공유 범위** | 본인만 | 팀 전체 |
| **용도** | 개인 선호, 작업 맥락 | 프로젝트 규칙, 컨벤션 |

팀 전체가 알아야 하는 규칙은 CLAUDE.md에 적는다. 개인적인 선호나 맥락은 메모리에 저장한다.

### 7.2 실제 패턴

**규칙 분리**: "테스트에서 DB 모킹 금지"가 팀 규칙이라면 CLAUDE.md에 적는다. 본인만의 테스트 습관이라면 메모리에 저장한다. 같은 내용을 두 곳에 적으면 나중에 불일치가 생긴다.

**온보딩 활용**: 새 팀원이 Claude Code를 쓰기 시작하면, CLAUDE.md에 적힌 프로젝트 규칙은 자동으로 적용된다. 개인 메모리는 대화를 하면서 자연스럽게 쌓인다. 다른 사람의 메모리를 복사해오는 건 의미가 없다. 그 사람의 맥락이니까.

**프로젝트 메모리와 CLAUDE.md의 경계**: 프로젝트 상태(릴리스 일정, 담당자 변경 등)는 메모리에 적는다. 프로젝트 규칙(코드 스타일, 아키텍처 결정 등)은 CLAUDE.md에 적는다. 상태는 빠르게 변하고 개인별로 관심사가 다르기 때문이다.

### 7.3 CLAUDE.md에 넣을 것 / 메모리에 넣을 것

```
# CLAUDE.md에 적합한 내용
- 코드 컨벤션: "함수명은 camelCase"
- 아키텍처 결정: "인증은 모두 middleware에서 처리"
- 테스트 정책: "통합 테스트는 testcontainers 사용"

# 메모리에 적합한 내용
- "나는 Go는 잘 아는데 React는 처음이다"
- "이 사용자는 설명 끝에 요약 안 해도 된다"
- "3월 5일까지 머지 프리즈"
- "파이프라인 버그는 Linear INGEST에서 관리"
```

## 8. 실무에서 자주 겪는 문제

### 8.1 메모리가 너무 많아졌을 때

MEMORY.md가 200줄에 가까워지면 오래된 프로젝트 메모리부터 정리한다. 삭제 전에 해당 메모리가 아직 유효한지 확인하고, 불필요하면 개별 파일과 인덱스 항목을 모두 삭제한다.

### 8.2 같은 주제의 메모리가 여러 개일 때

합친다. 예를 들어 사용자 기술 수준에 대한 메모리가 3개 있으면, 최신 정보를 기준으로 하나로 합치고 나머지를 삭제한다.

### 8.3 메모리 기반 추천이 틀렸을 때

메모리에 "src/utils/auth.ts에 validateToken이 있다"고 적혀 있어서 안내했는데, 파일이 이동한 상태였다면:

1. 사용자에게 사과하고 현재 위치를 찾는다
2. 메모리를 갱신하거나, 코드 경로 같은 건 아예 메모리에서 삭제한다 (코드는 읽으면 되니까)

### 8.4 "기억해" 요청의 판단

사용자가 "이거 기억해"라고 하면 무조건 저장하되, 저장할 가치가 있는 형태로 가공한다.

- "이 PR 목록 기억해" → PR 목록 자체보다 "이 PR들을 왜 한꺼번에 올렸는지"가 메모리 대상이다
- "내 역할 기억해" → 그대로 user 타입으로 저장한다
- "이 파일 구조 기억해" → 파일 구조는 변하니까, 왜 이 구조인지(의도/제약)만 저장한다
