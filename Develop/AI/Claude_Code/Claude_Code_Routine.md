# Claude Code 개발 루틴

Claude Code를 써본 지 반년이 넘으면 기능 하나하나 외우는 단계는 끝난다. 그 다음은 **언제, 어느 주기로, 어떤 작업에 투입할지**가 문제다. 이 문서는 하루 일과 속에 Claude Code를 어떻게 배치하는지, 그리고 그 루틴이 어디서 깨지는지를 정리한 것이다.

기능 설명은 다른 문서에 있다. 여기서는 실제로 일과 안에서 돌아가는 흐름과, 흐름이 망가질 때 어떻게 손보는지를 다룬다.

---

## 1. 루틴을 따로 설계해야 하는 이유

Claude Code를 "필요할 때 켜서 쓰는 도구"로만 대하면 금방 한계가 온다. 가장 흔한 증상은 이 세 가지다.

첫째, **같은 작업에 매번 다른 프롬프트를 쓴다**. PR 리뷰를 어제는 `/review`로 하고, 오늘은 "diff 봐줘"로 하고, 내일은 Plan 에이전트에 통째로 넘긴다. 결과 품질이 들쑥날쑥하다.

둘째, **컨텍스트가 언제 터질지 예측이 안 된다**. 오전에 시작한 세션을 오후까지 끌고 가다가 갑자기 엉뚱한 답이 나오기 시작한다. 그제야 `/clear`를 친다.

셋째, **자동화를 켜놓고 까먹는다**. cron으로 스케줄링한 에이전트가 매일 새벽 3시에 돌고 있는데 알림이 쌓이기만 한다. 결과를 확인하지 않으면 자동화는 소음이다.

루틴을 설계한다는 건 이 세 문제를 **시간 축 위에 배치**한다는 뜻이다. "어느 시점에 어떤 작업을 어떤 방식으로 돌릴지"를 정해두면, 세션마다 매번 재발명하지 않아도 된다.

---

## 2. 아침 리뷰 — 하루 시작 15분

출근하고 IDE를 켠 직후 15분이 루틴의 핵심 구간이다. 이 시간에 밤사이 쌓인 변화를 훑지 않으면 오전 내내 흩어진 정보를 따라다니게 된다.

### 2.1 PR과 이슈 훑기

팀 규모가 어느 정도 있으면 PR이 하룻밤에 5~10개씩 쌓인다. 매번 열어서 읽기엔 피곤하고, 안 보면 리뷰가 밀린다. 이때 Claude Code에 GitHub MCP를 연결해두면 질문 한 줄로 끝난다.

```
"어제 이후 올라온 PR 중에 내가 리뷰어로 지정된 것만 추려줘.
각 PR마다 변경 범위랑 리스크 한 줄씩 요약해줘."
```

핵심은 **요약을 Claude에게 맡기되, 판단은 직접 하는 것**이다. "이 PR 머지해도 되겠어?"까지 물어보면 Claude가 판단을 흉내내는데, 그 판단이 맞을 확률은 생각보다 낮다. 요약만 받아서 직접 열어봐야 할 PR을 추리는 용도로만 써야 한다.

이슈도 같은 방식으로 훑는다.

```
"어제 open된 이슈 중에 Priority:High 라벨 붙은 것만 보여줘.
제목이랑 첫 3줄만 요약하고, 관련 있을 만한 기존 이슈 번호 있으면 같이 찾아줘."
```

### 2.2 내 브랜치 상태 점검

어제 작업하다 멈춘 브랜치가 있으면 오늘 다시 시작하기 전에 상태를 확인해야 한다. 중단했던 시점의 맥락을 복원하는 데 Auto Memory가 도움이 된다.

```
"어제 마지막으로 작업한 브랜치 어디까지였지?
메모리에 남겨둔 진행 상황 있으면 같이 보여줘."
```

세션 종료 시 메모리에 남겨두는 습관이 들어 있으면 이 질문 하나로 어제 컨텍스트가 복원된다. 안 남겨뒀으면 `git log --oneline -10`과 `git diff HEAD~3`로 복원해야 하는데, 시간이 두 배쯤 든다.

### 2.3 아침 리뷰를 세션 하나로 끝내지 마라

욕심을 부려서 "PR 리뷰 → 이슈 확인 → 어제 작업 이어서" 이걸 한 세션에 다 몰면, 오전이 끝나기 전에 컨텍스트가 탁해진다. PR 훑기, 이슈 훑기, 실제 코드 작업은 **각각 다른 세션**으로 분리하는 게 맞다. `/clear`를 아끼지 말아야 한다.

## 3. 커밋 전 점검 루틴

커밋을 찍기 직전이 Claude Code를 가장 잘 쓸 수 있는 구간이다. 변경사항이 명확하고, 범위가 좁고, 물어볼 게 구체적이기 때문이다.

### 3.1 점검 순서

실무에서 안착한 순서는 이렇다.

1. 스테이징 전에 diff 한 번 보기
2. `/simplify` 또는 "중복/불필요한 복잡도 있는지 봐줘"
3. `/review` 또는 "보안, 타입, 예외 처리 관점에서 리뷰해줘"
4. 수정 반영
5. 커밋 메시지는 직접 쓰거나 `/commit` 사용

Claude에게 커밋 메시지 생성까지 맡기면 편하긴 한데, **팀 컨벤션에 안 맞게 나오는 경우가 종종 있다**. Conventional Commits를 쓰는 팀에서 Claude가 scope를 생략한다거나, 줄바꿈 규칙을 무시한다거나. CLAUDE.md에 커밋 컨벤션을 박아두면 정확도가 올라가지만, 완벽하진 않다. 최종적으로는 사람이 한 번 보고 넘기는 게 맞다.

### 3.2 Hook으로 강제할 수 있는 것

루틴이라고 하면 사람이 매번 의식해서 실행하는 것처럼 들리지만, Hook을 쓰면 일부를 Claude Code 밖으로 빼서 강제할 수 있다.

`~/.claude/settings.json`에 커밋 직전 훅을 걸어두는 예시.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "if echo \"$CLAUDE_TOOL_INPUT\" | grep -q 'git commit'; then npm run lint && npm run test; fi"
          }
        ]
      }
    ]
  }
}
```

Claude가 `git commit`을 실행하려 할 때 린트와 테스트가 먼저 돈다. 실패하면 커밋이 막힌다. **루틴을 기억으로 유지하려 하지 말고, 훅으로 못 박는 편이 안전하다**는 게 핵심이다. 기억은 피곤한 날 새는데, 훅은 새지 않는다.

### 3.3 커밋 전에 메모리 갱신 여부 체크

커밋하기 직전에 스스로 던지는 질문 하나. "이번 작업에서 다음에 또 마주칠 만한 문제가 있었나?" 있으면 그 시점에 메모리에 남겨야 한다. 커밋하고 나서 `/clear`를 치면 컨텍스트가 날아가니까, 메모리 갱신은 커밋 전에 끝내는 게 안전하다.

## 4. 장시간 작업 자동화 — /loop과 schedule

30분, 한 시간 단위로 진행 상황을 확인해야 하는 작업이 있다. 테스트 스위트 실행, CI 상태 폴링, 느린 빌드 모니터링 같은 것들. 이걸 사람이 기다리면서 Claude Code에 매번 "지금 어때?" 물어보는 건 낭비다.

### 4.1 /loop — 세션 안에서 반복

`/loop`은 같은 프롬프트를 주기적으로 다시 실행시키는 기능이다. 예를 들어 5분마다 `/pr-babysit` 스킬을 돌리고 싶으면 이렇게 한다.

```
/loop 5m /pr-babysit
```

`/pr-babysit`은 미리 만들어둔 커스텀 스킬이다. 내부적으로 "내가 연 PR들 상태 확인하고, 리뷰 코멘트 달린 게 있으면 요약해줘" 같은 작업을 수행한다.

주기를 생략하면 Claude가 스스로 페이스를 조절한다.

```
/loop "빌드 상태 확인하고, 실패했으면 로그 요약해줘"
```

Claude가 다음 실행까지 몇 초를 기다릴지 판단한다. 캐시 TTL(5분)을 기준으로 짧게 돌 때는 60~270초, 길게 기다려도 될 때는 20~30분 사이를 고르는 식이다.

### 4.2 /loop의 함정

`/loop`을 켜놓고 자리를 뜨면 두 가지가 터진다.

첫째, **토큰 낭비**. 10분짜리 작업에 1분 루프를 걸면 10번 도는 동안 아무것도 안 바뀐 상태를 10번 확인한다. 각 루프마다 컨텍스트가 캐시에서 나갔다 들어오면서 비용이 누적된다. **루프 주기는 실제로 상태가 바뀔 만한 최소 시간**으로 잡아야 한다.

둘째, **알림 피로**. 루프가 돌 때마다 결과가 쌓이면 어느 순간부터 읽지 않는다. 읽지 않는 자동화는 그냥 돈 쓰는 거다. 루프를 걸 땐 "이 결과가 언제 의미 있는가?"를 같이 정해야 한다. "상태가 바뀌었을 때만 알려줘"를 프롬프트에 넣어두면 노이즈가 줄어든다.

```
/loop 3m "CI 상태 확인해줘. 직전 확인 때와 달라진 게 없으면 '변화 없음'만 출력하고 끝내라."
```

### 4.3 schedule — 세션 밖의 cron

`/schedule`은 Claude Code가 켜져 있지 않은 시간에도 원격 에이전트를 cron처럼 돌리는 기능이다. 새벽에 보안 스캔을 돌린다든가, 매일 아침 9시에 PR 대시보드를 만들어둔다든가 하는 용도.

```
/schedule create "매일 오전 9시에 어제 머지된 PR 요약해서 Slack으로 보내줘" 0 9 * * *
```

cron 표현식이 익숙하지 않으면 자연어로 설명해도 된다. "매일 아침 9시"를 자동으로 `0 9 * * *`로 바꿔준다.

### 4.4 schedule의 현실적 한계

scheduled 에이전트는 **결과를 받는 채널이 확실할 때만 가치가 있다**. Slack, 이메일, GitHub Issue 중 하나로 결과가 꽂히게 만들어두지 않으면, 에이전트가 돌아도 사람이 확인하지 않는다.

그리고 스케줄을 너무 많이 걸어두면 서로 간섭한다. 새벽 2시에 돌린 테스트 스위트가 새벽 3시에 시작하는 보안 스캔과 CI 리소스를 두고 경쟁한다거나, 결과 메시지가 한꺼번에 쏟아져서 정말 중요한 알림이 묻힌다거나. **현재 돌고 있는 스케줄을 주기적으로 확인하고 정리하는 시간**을 따로 잡아야 한다. 한 달에 한 번 `/schedule list`로 훑고, 쓰지 않는 걸 지운다.

## 5. 백그라운드 에이전트 운용

`run_in_background` 옵션을 쓰면 에이전트가 메인 세션과 별개로 돌아간다. 메인에서는 다른 작업을 하면서, 백그라운드에서는 탐색이나 리서치가 병렬로 진행된다.

### 5.1 언제 백그라운드로 돌리나

- 결과가 지금 당장 필요하지 않은 탐색 (예: "코드베이스 전체에서 deprecated API 쓰는 곳 찾아줘")
- 오래 걸리는 빌드/테스트 모니터링
- 여러 경로를 동시에 탐색하는 리서치 (각 경로를 별도 백그라운드 에이전트로)

### 5.2 언제 포그라운드로 돌리나

- 결과를 받아야 다음 작업을 결정할 수 있을 때
- 에이전트가 코드를 수정하는 경우 (수정 결과를 바로 검토해야 안전)

이 구분이 실전에서 가장 헷갈린다. "병렬이면 좋겠지"라는 이유로 전부 백그라운드로 돌리면, **결국 메인에서 대기만 하는 상태가 된다**. 결과가 필요한 시점에 결과가 없으면 백그라운드의 의미가 없다.

### 5.3 백그라운드 에이전트 회수 주기

백그라운드로 돌린 에이전트는 끝났을 때 알림이 오는 구조인데, 실수로 알림을 놓치면 방치된다. 그래서 **의식적으로 "아직 안 돌아온 에이전트 있나?" 질문을 던지는 구간**이 필요하다.

실제로 쓰는 패턴.

- 큰 세션 단위(`/clear` 직전)마다 점검: "백그라운드로 돌린 거 다 돌아왔나?"
- 하루 끝날 때 한 번: 남아 있는 백그라운드 작업 정리

이걸 안 하면 세션 종료 시점에 "그거 왜 안 왔지?" 하면서 결과물이 유실되는 경우가 있다.

## 6. 컨텍스트 오염 방지 — 리셋 주기

Claude Code 세션은 쓸수록 탁해진다. 읽은 파일, 실행한 명령, 중간에 바뀐 방향성 같은 게 다 컨텍스트에 쌓이기 때문이다. 언제 리셋해야 하는지 감을 익혀두지 않으면 품질이 천천히 떨어진다.

### 6.1 리셋 타이밍 판단 기준

경험적으로 이 신호가 오면 `/clear`가 맞다.

- **같은 설명을 두 번째 하게 될 때**: Claude가 아까 말한 걸 까먹고 되묻는다
- **엉뚱한 파일을 읽기 시작할 때**: 관련 없는 경로까지 탐색한다
- **답변이 이전 답변과 일관성이 깨질 때**: A라고 해놓고 몇 턴 뒤에 B라고 한다
- **작업 성격이 완전히 바뀔 때**: 구현 → 테스트 → 리뷰 같은 전환

이 중 세 번째가 가장 위험하다. 본인이 방향을 의심해야 하는데, Claude가 "방금까지 이렇게 말해놓고 왜 갑자기 다르게 말해?" 해도 그 시점엔 컨텍스트가 이미 오염돼 있다. 대화 품질이 이상하다 싶으면 일단 `/clear`부터 쳐본다.

### 6.2 세션 지속 시간의 현실적 한계

한 세션을 몇 시간씩 끌고 가는 게 가능하긴 한데, 품질 유지는 다른 문제다. 실무적으로 세 가지 구간으로 나뉜다.

- **0~30분**: 품질이 가장 좋은 구간. 복잡한 구현 작업을 여기에 배치
- **30분~1시간**: 사용 가능하지만 민감한 작업은 피해야 하는 구간
- **1시간 이후**: `/compact` 또는 `/clear`를 고려. 그냥 밀고 가면 품질이 떨어진다

이건 세션마다 다르긴 한데, "내 세션은 두 시간도 멀쩡하던데" 하는 경우는 대체로 작업이 단순했거나, 품질 저하를 본인이 눈치 못 챈 경우다.

### 6.3 /compact vs /clear

두 개의 쓰임이 다르다.

- `/compact`: 컨텍스트를 요약해서 유지. 이전 작업을 **이어서** 해야 할 때
- `/clear`: 컨텍스트 완전 초기화. 작업 성격이 **바뀔** 때

`/compact`는 요약 프롬프트를 같이 넣을 수 있다. 기본 요약은 정보가 빠지는 경우가 있어서, 직접 지시하는 게 낫다.

```
/compact "지금까지 수정한 파일 경로와 각 파일의 변경 의도를 중심으로 요약해줘.
다음 세션에서 테스트 작성하는 데 필요한 정보만 남겨줘."
```

## 7. 세션 종료 시 메모리 갱신

하루 끝에 Claude Code를 끄기 전에 해야 하는 루틴이 있다. **오늘 세션에서 다음에 또 마주칠 만한 정보를 메모리에 남기는 작업**이다.

### 7.1 무엇을 남기나

남길 만한 것과 아닌 것의 구분이 중요하다.

남긴다:
- 프로젝트의 **진행 상황**: "결제 모듈 리팩토링 3단계까지 진행. 다음은 테스트 마이그레이션"
- **피드백**: "이 프로젝트 테스트는 실제 DB를 쓰도록. 모킹 금지(작년 이슈 이후)"
- **외부 참조**: "성능 관련 이슈는 Linear의 PERF 프로젝트에서 추적"
- **비자명한 결정 사항**: "이 서비스는 Kafka 대신 SQS를 쓰기로 결정. 이유는 운영 부담"

안 남긴다:
- 오늘 수정한 코드 내용 (git log로 복원 가능)
- 디버깅 해결법 (코드에 이미 있다)
- 임시 작업 상태 (내일 `/clear`하면 어차피 날아감)
- 사용자 본인도 "이걸 왜 남겨뒀지?" 할 내용

### 7.2 갱신 프롬프트

종료 직전에 이 질문을 Claude에게 던진다.

```
"오늘 세션에서 다음 대화에 남길 만한 정보가 있나?
이미 메모리에 있는 것과 중복되지 않는 것만 골라줘."
```

Claude가 후보를 제시하면 검토하고 메모리에 넣는다. 매번 Claude가 똑똑하게 고르진 않으니까 사람이 걸러야 한다.

기존 메모리가 낡았으면 이때 같이 정리한다. 프로젝트 단계가 넘어갔는데 이전 단계 메모리가 그대로 남아 있으면 다음 세션에서 혼란을 준다.

## 8. Skill, Hook, CLAUDE.md, 루틴의 경계

이 넷은 역할이 겹쳐 보이는데, 실제로는 다르다. 혼동하면 같은 걸 네 군데에 적어두는 사태가 생긴다.

| 계층 | 실행 주체 | 성격 | 언제 쓰나 |
|------|-----------|------|-----------|
| **CLAUDE.md** | Claude가 읽음 | 컨텍스트 제공 | 프로젝트 상시 규칙, 컨벤션 |
| **Skill** | Claude가 호출 | 재사용 작업 묶음 | 특정 작업 패턴을 스킬로 캡슐화 |
| **Hook** | 런타임이 실행 | 자동 개입 | 사람이 까먹어도 실행돼야 하는 것 |
| **루틴** | 사람이 수행 | 시간 축 설계 | 언제 어떤 도구를 쓸지의 배치 |

### 8.1 뭘 어디에 넣어야 하나

판단 기준이 하나 있다. "이게 안 지켜져도 괜찮은가?"

- 안 지켜지면 안 되는 것 → **Hook**
- 지켜지면 좋은 것 → **CLAUDE.md** 또는 **Skill**
- 사람의 판단이 필요한 것 → **루틴**

예를 들어 "커밋 메시지에 이슈 번호 포함"은 팀 컨벤션이다. CLAUDE.md에 적으면 Claude가 맞춰서 메시지를 쓰려고 노력하긴 하지만, 가끔 빠뜨린다. **반드시** 포함되어야 한다면 Hook으로 검증해야 한다.

"PR 올리기 전 린트 + 테스트 + 리뷰"는 재사용 작업이다. Skill로 묶어두고 `/pr-check`로 부른다. 언제 부를지는 사람이 결정(= 루틴).

### 8.2 흔한 오배치

- CLAUDE.md에 긴 체크리스트를 몰아넣는다 → Claude가 일부만 수행한다. Hook이나 Skill로 분리해야 한다
- 모든 걸 Skill로 만든다 → 매번 어떤 Skill을 부를지 고민하는 데 시간을 쓴다. 자주 쓰는 것만 Skill로
- 루틴을 Skill로 착각한다 → "아침에 이걸 하는 루틴"은 Skill이 아니다. 시간 축 설계는 사람의 영역

## 9. 루틴이 깨지는 전형적 상황

루틴은 이상적으로 동작하지 않는다. 언젠가 반드시 깨진다. 깨질 때의 패턴을 알고 있으면 복구가 빠르다.

### 9.1 토큰 낭비로 조용히 비용이 쌓임

월말에 API 청구서를 보고 놀라는 경우. 원인은 대체로 세 가지다.

- `/loop`을 짧은 주기로 걸어놓고 잊어버림
- 컨텍스트 오염 상태에서 계속 질문을 던지면서 매 요청마다 오염된 전체 컨텍스트를 재전송
- 백그라운드 에이전트를 여러 개 띄워놓고 회수 안 함

대응은 주기적 점검이다. 주 1회 정도 "지금 돌고 있는 루프/스케줄/백그라운드 작업이 뭐가 있는지" 훑는다. 안 쓰는 건 즉시 중단한다.

### 9.2 자동화 폭주

스케줄을 너무 많이 걸거나, Hook이 재귀적으로 돌면서 루프가 생기는 경우. Hook이 커밋할 때 린트를 돌리고, 린트 실패 시 자동 수정을 하고, 자동 수정이 또 커밋을 유발하는 식.

Hook을 쓸 때는 **종료 조건을 명시적으로 넣어야 한다**. 재귀 방지 플래그를 세우거나, Hook 내부에서 Claude Code를 다시 호출하지 않게 제한한다.

### 9.3 알림 피로

자동화에서 오는 결과를 읽지 않기 시작한다. 이 시점엔 자동화 자체를 의심해야 한다. "이 알림은 진짜 내가 행동해야 할 때만 온다"가 되도록 조건을 조여야 한다.

- "매일 아침 PR 요약" → "리뷰어로 지정된 PR이 있을 때만 요약"
- "빌드 실패 알림" → "마스터 브랜치 빌드 실패일 때만"

### 9.4 루틴이 너무 무거워서 지켜지지 않음

단계가 너무 많으면 결국 건너뛰게 된다. 아침 리뷰에 15분 이상 쓰기 시작하면 곧 아침 리뷰를 안 하게 된다. 루틴은 **15분 안에 끝나는 것**을 기준으로 잡아야 현실적이다. 더 긴 게 필요하면 여러 날에 나눈다.

## 10. 실전 설정 예제

루틴을 구체적인 설정으로 내려받은 예시. 이걸 그대로 쓰기보다는, 본인 상황에 맞게 변형하는 기준으로 삼으면 된다.

### 10.1 settings.json — 커밋 전 훅

`~/.claude/settings.json`.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "case \"$CLAUDE_TOOL_INPUT\" in *\"git commit\"*) npm run lint && npm run test:unit || exit 2 ;; esac"
          }
        ]
      }
    ]
  }
}
```

`exit 2`는 Claude에게 피드백을 주면서 명령 실행을 막는다. 린트나 테스트가 실패하면 Claude가 이유를 알고 수정을 시도한다.

### 10.2 CLAUDE.md — 루틴 컨텍스트

프로젝트 루트 `CLAUDE.md`에 루틴 관련 규칙을 넣어둔 예시.

```markdown
# 개발 루틴 관련

## 세션 시작 시
- 사용자가 "어제 작업 이어서"라고 하면 메모리에서
  프로젝트 진행 상황 먼저 조회

## 커밋 전
- /simplify와 /review를 순서대로 실행
- 커밋 메시지는 Conventional Commits, 이슈 번호 필수
  예: `fix(order): calculateTotal 음수 처리 (#1234)`

## 세션 종료 시
- "세션 종료" 키워드가 나오면 다음을 수행:
  1. 오늘 작업 요약
  2. 메모리에 추가할 만한 내용 후보 제시
  3. 기존 메모리 중 낡은 것 정리 제안
```

### 10.3 커스텀 스킬 — /pr-babysit

`.claude/skills/pr-babysit.md`.

```markdown
---
description: "내가 연 PR들 상태 확인하고 리뷰 코멘트/CI 실패 요약"
user-invocable: true
---

# pr-babysit

## 수행 단계
1. `gh pr list --author @me --state open`으로 PR 목록 조회
2. 각 PR의 CI 상태를 `gh pr checks <번호>`로 확인
3. 각 PR의 리뷰 코멘트를 `gh api repos/.../pulls/<번호>/comments`로 조회
4. 다음 포맷으로 요약:
   - CI 실패한 PR: 어떤 잡이 어떤 이유로 실패했는지
   - 새 리뷰 코멘트: 어떤 파일에 어떤 지적이 달렸는지
   - 둘 다 없으면 "변화 없음"만 출력

## 주의
- CI가 pending이면 "아직 돌고 있음"으로 표시하고 넘어간다
- 리뷰 코멘트는 Claude가 이미 응답한 것은 제외
```

`/loop 10m /pr-babysit` 같은 식으로 돌리면 PR 관리 루틴이 반쯤 자동화된다.

### 10.4 스케줄 — 아침 요약

```
/schedule create "어제 머지된 PR 요약해서 Slack #dev-daily 채널로 보내줘. 
내가 리뷰한 PR은 제외." 0 8 * * 1-5
```

주중 아침 8시에만 돈다. 주말에 팀 채널이 조용해야 하는 환경이면 이렇게 범위를 조인다.

## 11. 루틴을 바꿀 때의 기준

한 번 정한 루틴을 절대 안 바꾸면 오히려 문제다. 팀 구성, 프로젝트 단계, 모델 업데이트에 따라 루틴도 변한다. 다만 바꿀 때 기준이 있다.

- **측정**: 현재 루틴의 어느 부분이 시간을 많이 먹는가, 어느 알림을 안 읽는가
- **제거 우선**: 새 단계를 추가하기 전에, 안 쓰는 단계를 먼저 뺀다
- **개별 교체**: 전체를 다시 짜지 말고, 한 단계씩 교체한다
- **2주 관찰**: 바뀐 단계를 2주 돌려보고 판단. 하루 써보고 "별로네" 하지 말 것

루틴은 설계보다 **유지**가 어렵다. 이 문서의 내용을 통째로 가져다 쓰기보다, 본인이 매일 하는 작업을 먼저 관찰하고 거기에 맞춰 조립하는 편이 낫다.
