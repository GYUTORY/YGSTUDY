# LLM 컨텍스트 윈도우

컨텍스트 윈도우는 모델이 한 번에 다루는 토큰 시퀀스의 길이다. 입력 프롬프트, 시스템 메시지, 이전 대화 이력, 도구 호출 결과, 생성 중인 출력이 전부 이 한정된 길이 안에 들어가야 한다. 2022년 GPT-3 시절 4K였던 길이가 2024년 들어 Claude 3.5의 200K, Gemini 1.5 Pro의 2M까지 늘어났고, 2025~2026년에는 1M~10M 컨텍스트가 흔해졌다.

길이가 늘어났다고 해서 그만큼 모델이 잘 활용한다는 뜻은 아니다. 실측 정확도는 길이가 늘어날수록 떨어지고, 입력 위치에 따라 회상률이 달라진다. 비용도 길이의 제곱에 가깝게 늘어난다. 이 문서는 컨텍스트 윈도우가 왜 늘릴 수 있게 됐는지, 어디서 깨지는지, 실무에서 어떻게 다뤄야 하는지를 다룬다.

## 컨텍스트 길이가 비싼 이유

Transformer의 셀프 어텐션은 시퀀스 길이 N에 대해 O(N²) 메모리와 연산을 쓴다. 4K에서 32K로 늘리면 어텐션 메모리만 64배 증가한다. 128K로 늘리면 1024배다. FlashAttention 같은 IO-aware 구현으로 활성화 메모리를 줄이고, KV 캐시를 GPU HBM에 효율적으로 배치해도 근본적인 O(N²)는 사라지지 않는다.

실제 추론 시 비용 구조는 두 단계로 나뉜다.

- **Prefill 단계**: 입력 전체에 대해 한 번에 forward pass를 돌려 KV 캐시를 만든다. 시퀀스 길이의 제곱에 비례하는 연산이 여기서 발생한다. 128K 입력은 prefill만 수 초가 걸린다.
- **Decode 단계**: 토큰을 하나씩 자기회귀로 생성한다. KV 캐시를 매번 메모리에서 읽어와야 하므로 메모리 대역폭에 묶인다. 컨텍스트가 길수록 KV 캐시가 커지고 토큰당 생성 시간이 늘어난다.

API 가격이 입력 토큰과 출력 토큰을 분리해서 매기는 이유가 여기에 있다. 입력은 prefill 비용을 반영하고 출력은 decode의 메모리 대역폭 비용을 반영한다. 보통 출력 가격이 입력의 3~5배다.

## 포지셔널 인코딩 — RoPE vs ALiBi

Transformer는 순서 정보를 가지고 있지 않다. 어텐션 자체가 set-to-set 연산이라 토큰의 위치를 어떻게든 주입해야 한다. 컨텍스트 확장 가능성은 포지셔널 인코딩 방식에 크게 좌우된다.

### Sinusoidal (원조)

GPT-2 시대의 절대 위치 인코딩. `position * sin/cos` 값을 임베딩에 더해서 위치 정보를 주입한다. 학습 시 본 길이를 넘어가면 그대로 무너진다. 길이 외삽(extrapolation)이 안 된다.

### RoPE (Rotary Position Embedding)

LLaMA, Qwen, DeepSeek 같은 현대 오픈 모델 대부분이 채택한 방식. 위치 정보를 쿼리와 키 벡터에 회전 행렬로 곱해서 주입한다. 두 토큰의 어텐션 점수가 절대 위치가 아니라 상대 위치(거리)에만 의존하도록 만든다.

회전 주파수는 `θ_i = base^(-2i/d)` 공식으로 정해진다. base 값(보통 10000)이 회전 속도를 결정한다. 학습 시 본 길이 밖의 위치는 회전 각도가 학습 분포 밖이라 어텐션 점수가 망가진다. 그래서 RoPE는 그대로 두면 외삽이 안 되지만, 회전 주파수를 조절하면 학습된 모델을 더 긴 길이로 확장할 수 있다. YaRN, NTK가 모두 이 방식이다.

### ALiBi (Attention with Linear Biases)

쿼리-키 거리에 비례하는 선형 페널티를 어텐션 점수에 직접 더한다. 멀리 있는 토큰일수록 페널티가 커진다. MosaicML의 MPT, BLOOM이 사용했다. 외삽이 자연스럽게 되지만 멀리 있는 토큰의 영향이 강하게 억제되기 때문에 진짜 롱 컨텍스트 작업(예: 100K 문서 안에서 정보 찾기)에서는 약하다는 평가가 많다. 그래서 2025년 기준 신규 모델 대부분은 RoPE를 쓴다.

### NoPE / 학습된 위치 임베딩

T5는 상대 위치 바이어스를 학습 가능한 파라미터로 뒀다. GPT-NeoX 일부 변형은 위치 인코딩을 아예 빼는(NoPE) 실험도 했는데, 작은 모델에서는 동작하지만 큰 모델에서는 RoPE만 못하다.

## 컨텍스트 확장 기법

8K로 학습한 모델을 128K로 늘리고 싶다. 처음부터 128K로 학습하면 비용이 폭증한다. 그래서 짧은 길이로 학습한 모델의 RoPE를 조절하고 약간의 파인튜닝만으로 긴 길이로 확장하는 기법이 발전했다.

### Position Interpolation (PI)

가장 단순한 방법. 학습 시 본 위치보다 작은 값으로 위치를 스케일링한다. 8K로 학습된 모델을 32K로 쓰려면 모든 위치를 1/4로 압축한다. 회전 각도가 학습 분포 안에 들어오므로 어텐션이 망가지지 않는다. 단점은 위치 분해능이 1/4로 떨어진다는 것이다. 가까운 토큰들의 구분이 흐려진다.

### NTK-aware Scaling

PI의 분해능 손실 문제를 보완한다. RoPE의 회전 주파수가 차원마다 다르다는 점을 이용해서, 고주파(짧은 거리 정보)는 그대로 두고 저주파(긴 거리 정보)만 스케일링한다. base 값을 조절하면 된다. 파인튜닝 없이도 어느 정도 동작하는 게 장점이다. LLaMA 1 시절 reddit에서 처음 제안돼서 빠르게 표준이 됐다.

base 조정 공식은 간단하다.

```python
def adjust_rope_base(original_base, original_max_len, target_max_len, dim):
    """NTK-aware scaling: base를 키워서 저주파만 늘린다"""
    scale = target_max_len / original_max_len
    # 차원 수 d, scale s에 대해 base를 s^(d/(d-2))로 키운다
    new_base = original_base * (scale ** (dim / (dim - 2)))
    return new_base

# LLaMA 2 7B를 4K에서 16K로 늘릴 때
new_base = adjust_rope_base(10000, 4096, 16384, 128)
# 약 39800 — 4배 길이 확장에 base는 약 4배
```

NTK는 파인튜닝 없이 동작하지만 길이를 너무 크게(예: 8배 이상) 늘리면 정확도가 빠르게 떨어진다.

### YaRN (Yet another RoPE extensioN)

NTK를 개량한 방법. NTK가 모든 차원을 일괄적으로 처리한 것과 달리, YaRN은 차원을 세 영역으로 나눠 처리한다.

- **고주파 차원**: 위치를 그대로 둔다(외삽). 짧은 거리 토큰 구분에 중요.
- **저주파 차원**: PI 방식으로 압축한다. 긴 거리 정보 표현용.
- **중간 영역**: 두 방식을 부드럽게 보간한다.

여기에 어텐션 온도(temperature) 보정을 추가했다. 컨텍스트가 길어지면 어텐션 분포가 평탄해지는데, 소프트맥스 전에 스케일 팩터를 곱해서 분포를 다시 날카롭게 만든다.

```python
def yarn_attention_scale(scale_factor):
    """YaRN의 온도 보정: 긴 컨텍스트에서 분포 평탄화를 보정"""
    import math
    return 0.1 * math.log(scale_factor) + 1.0

# 4배 확장 시 어텐션 점수에 1.139 곱
# 8배 확장 시 어텐션 점수에 1.208 곱
```

YaRN으로 LLaMA 2를 128K까지 확장한 사례가 검증됐고, Qwen 2.5의 1M 컨텍스트도 YaRN 변형을 쓴다. 4K → 128K(32배) 확장에 100M 토큰 정도의 추가 학습이면 된다. 처음부터 128K로 학습하는 것보다 훨씬 싸다.

### LongRoPE / 검색 기반 위치 보간

Microsoft가 제안한 LongRoPE는 진화 탐색으로 차원별 최적 스케일링 계수를 찾는다. 256K, 2M 컨텍스트에서 YaRN보다 perplexity가 낮다. 다만 탐색 비용이 들고 구현이 복잡해서 보편적으로 쓰이진 않는다.

## Lost-in-the-Middle 문제

긴 컨텍스트를 받을 수 있는 것과 그걸 잘 활용하는 것은 다르다. Stanford의 Liu et al. (2023)가 발표한 "Lost in the Middle" 논문이 결정타였다. 멀티 문서 QA 태스크에서 정답이 들어있는 문서의 위치를 바꿔가며 정확도를 측정했더니, **시작과 끝 위치에서는 정확도가 높았지만 중간 위치에서는 우연 수준까지 떨어졌다.** 사람이 긴 글을 읽을 때 가운데를 흘려 읽는 것과 비슷한 U자형 곡선이다.

원인은 여러 가지가 지목됐다.

- 학습 데이터의 분포. 인터넷 텍스트는 보통 첫머리와 끝에 결론·핵심이 몰린다. 모델이 이 분포를 학습한다.
- RoPE의 attention sink. BOS 토큰 같은 시작부 토큰이 비정상적으로 큰 어텐션을 받는다는 관찰이 있다(Xiao et al., StreamingLLM).
- Causal attention의 비대칭성. 뒤쪽 토큰은 앞을 다 볼 수 있지만 앞쪽 토큰은 자기 위치 이전만 본다. 정보 흐름이 비대칭.

이 문제는 모델 크기와 무관하게 나타나고, 컨텍스트 길이가 길수록 심해진다. GPT-4-32K, Claude 2 시절부터 일관되게 관측됐고 1M 모델에서도 사라지지 않았다.

실무에서 대응하는 방법은 보통 이렇다.

- **중요한 정보는 프롬프트의 처음 또는 끝에 배치한다.** 시스템 프롬프트는 맨 위에, 사용자 질문이나 지시는 맨 아래에 두는 패턴이 정착했다.
- **재정렬(reordering) 기법**. 검색 결과를 관련도 순으로 가운데에 몰아넣지 말고, 관련도 1위는 맨 끝, 2위는 맨 처음, 3위는 두 번째 끝… 이렇게 가장자리부터 채운다.
- **하이라이트 마커.** XML 태그나 `===` 같은 구분자로 중요 영역을 시각적으로 분리하면 회상률이 조금 올라간다.

## 1M 컨텍스트 실측 — 광고와 현실

2024년 Gemini 1.5 Pro가 1M, 이어서 2M 컨텍스트를 발표하면서 "needle in a haystack" 벤치마크가 화제가 됐다. 긴 텍스트 안에 특정 문장(바늘)을 숨기고 모델이 찾아내는지 보는 테스트다. Gemini는 1M에서 99% 이상 회상률을 보였다.

그런데 이게 진짜 실력을 반영하지 않는다는 비판이 곧 나왔다. needle in a haystack은 한 문장만 정확히 찾으면 되는 단순 회상 태스크다. 실제 업무에서는 여러 정보를 종합하거나, 미묘한 추론을 해야 한다.

후속 벤치마크들이 더 엄격하게 측정했다.

- **RULER (NVIDIA, 2024)**: 단순 검색뿐 아니라 다중 키 검색, 변수 추적, 집계 같은 13개 태스크로 측정. 대부분의 모델이 광고한 최대 길이의 절반 이하에서만 의미 있는 성능을 보였다. GPT-4 Turbo는 128K 광고지만 RULER 기준 실효 길이는 64K. Llama 3.1 70B는 128K 광고지만 실효 32K 수준.
- **LongBench, ∞Bench, BABILong**: 비슷한 결론. 모델이 "한 가지를 찾는" 작업에서는 잘하지만 "정보를 종합해서 추론"하는 작업에서는 32K~64K 이후 정확도가 빠르게 떨어진다.
- **Anthropic의 needle 변형 실험**: 텍스트와 무관한 문장을 바늘로 쓰면 회상이 잘 되지만, 텍스트와 의미적으로 비슷한 문장을 바늘로 넣으면 회상률이 급락한다. 모델이 "튀는 정보"를 잘 잡을 뿐 "비슷한 것 중에서 정확한 것을 고르는" 작업은 약하다.

내가 실무에서 직접 측정해본 패턴도 비슷하다. Claude 3.5 Sonnet에 200K 가까운 코드베이스를 넣고 "X 함수를 호출하는 모든 위치를 찾아라"라고 시키면 80% 정도는 잘 찾지만, 가운데 부분의 한두 개를 빼먹는다. 정확성이 중요하면 컨텍스트를 통째로 넣지 말고 검색·청킹으로 좁혀야 한다.

## 컨텍스트 캐싱 — 비용 구조의 본질

긴 컨텍스트의 prefill 비용을 매번 내는 건 비싸다. 같은 시스템 프롬프트나 문서를 반복해서 보낼 때 KV 캐시를 서버가 보관해주면 prefill을 건너뛸 수 있다. 이게 prompt caching이다. 2024~2025년 사이 세 주요 벤더가 모두 도입했고, 구조가 조금씩 다르다.

### Anthropic Prompt Caching

명시적 캐시 제어 방식. 메시지의 특정 블록에 `cache_control: {"type": "ephemeral"}`를 붙이면 그 지점까지의 prefix를 캐싱한다. 캐시는 기본 5분 TTL이고 같은 prefix로 요청이 들어오면 갱신된다. 1시간 TTL 옵션도 있다(별도 가격).

가격 구조(2026년 기준 Claude Opus 4.7):
- 캐시 쓰기(첫 요청): 입력 가격의 1.25배
- 캐시 읽기(히트): 입력 가격의 0.1배

손익 분기점은 단순하다. 같은 프롬프트를 두 번 이상 쓸 거면 거의 무조건 이득이다. 100K 시스템 프롬프트가 있고 동일 세션에서 10번 호출하면, 캐싱 없이는 100K × 10 = 1M 토큰 비용, 캐싱하면 100K × 1.25 + 100K × 0.1 × 9 = 215K 토큰 비용. 약 4.6배 절감.

```python
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "You are a code reviewer."
        },
        {
            "type": "text",
            "text": LARGE_CODEBASE_CONTEXT,  # 80K 토큰짜리 코드베이스
            "cache_control": {"type": "ephemeral"}
        }
    ],
    messages=[
        {"role": "user", "content": "이 PR의 보안 이슈를 검토해줘"}
    ]
)

# 응답에 cache_creation_input_tokens, cache_read_input_tokens가 따로 표시됨
print(response.usage)
```

주의할 점이 있다. 캐시는 prefix 매칭이라 한 글자라도 다르면 무효화된다. 시스템 프롬프트 안에 타임스탬프를 넣지 마라. 도구 정의를 동적으로 만들지 마라. 캐시 블록은 메시지 시퀀스에서 4개까지 둘 수 있다.

### OpenAI Prompt Caching

자동 방식. 1024 토큰 이상의 prefix가 일치하면 자동으로 캐시 히트가 잡힌다. 별도 마크업이 없다. TTL은 5~10분, 트래픽 적은 시간대엔 1시간까지 유지된다.

가격은 캐시 히트 시 입력 가격의 50% 할인. Anthropic처럼 명시적 쓰기 비용 가산은 없다. 대신 할인 폭이 작다(50% vs Anthropic의 90%).

자동이라 편하지만 캐시 히트 여부를 제어하기 어렵다. 프롬프트 구성 시 동적인 부분을 뒤로 몰아야 prefix 매칭이 잘 된다.

### Google Gemini Context Caching

명시적 방식이지만 Anthropic과 구조가 다르다. 캐시 객체를 미리 생성해두고 그 핸들을 API 호출에 전달한다. TTL을 직접 지정한다(분 단위, 시간 단위 가능).

가격:
- 캐시 저장: 시간당 토큰 단위 과금(스토리지 비용)
- 캐시 사용: 입력 가격의 25% (75% 할인)

장기 캐싱에 유리한 구조다. 100만 토큰 문서를 하루 종일 캐싱해두고 여러 사용자 질문을 받는 워크플로에 적합하다. 짧은 세션에서는 스토리지 비용이 오히려 손해가 될 수 있다.

```python
import google.generativeai as genai

# 캐시 객체 미리 생성
cache = genai.caching.CachedContent.create(
    model="gemini-1.5-pro-002",
    system_instruction="You are a legal document analyst.",
    contents=[LARGE_PDF_DOCUMENT],
    ttl="3600s"  # 1시간 보관
)

# 캐시를 참조해서 호출
model = genai.GenerativeModel.from_cached_content(cache)
response = model.generate_content("계약 위반 조항을 정리해줘")
```

### 어떤 캐싱이 언제 유리한가

세 벤더의 가격 구조가 다르니 워크로드에 맞춰 선택해야 한다.

- **짧은 세션·반복 호출(챗봇, 코드 에이전트)**: Anthropic. 5분 TTL이 충분하고 90% 할인이 크다.
- **자동·간편**: OpenAI. 명시적 제어 없이도 어느 정도 절감된다.
- **대용량 문서를 장시간 여러 사용자가 질의**: Gemini. 시간당 스토리지 비용을 내는 게 매 호출 prefill보다 싸다.

캐시 히트율을 모니터링해라. 의도와 달리 히트가 안 잡히는 경우가 자주 있다(타임스탬프, 동적 ID, 도구 정의 변경 등). Anthropic은 응답에 명시적으로 cache hit/miss 토큰을 분리해서 알려주니 로깅 필수다.

## 청킹 전략

문서가 컨텍스트 윈도우를 넘어가면 잘라야 한다. 청킹은 단순해 보이지만 실제로는 검색 품질을 좌우하는 핵심이다.

### 고정 길이 청킹

가장 단순. 토큰 단위 또는 문자 단위로 N개씩 자른다. 빠르지만 문장이나 코드 블록 가운데를 자르는 사고가 자주 난다. 일반 문서 RAG에서는 비추.

### 의미 단위 청킹 (Recursive Character Splitting)

LangChain의 RecursiveCharacterTextSplitter가 보편화시킨 방식. 우선순위 구분자 리스트(`\n\n`, `\n`, `. `, ` `)를 순서대로 시도하면서 청크가 목표 길이 이하로 떨어질 때까지 분할한다. 단락→문장→단어 순으로 자연스럽게 자른다.

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,        # 청크 목표 길이
    chunk_overlap=200,      # 청크 간 겹침
    separators=["\n\n", "\n", ". ", " ", ""]
)

chunks = splitter.split_text(long_document)
```

`chunk_overlap`은 청크 경계에서 문맥이 끊기는 걸 막는다. 200/1000 정도(20%) 겹침이 표준이다. 너무 많이 겹치면 중복 검색 결과가 생긴다.

### 시맨틱 청킹

임베딩으로 문장 간 의미적 거리를 측정해서 거리가 큰 지점에서 자른다. 주제가 바뀌는 지점을 자동으로 잡는다는 발상. LlamaIndex의 SemanticSplitterNodeParser가 대표적.

장점은 의미 단위 일관성, 단점은 임베딩 호출 비용과 청크 길이가 들쭉날쭉해진다는 점. 데이터셋이 작거나 비싼 도메인(법률, 의료)에서 검색 품질이 중요할 때 쓴다.

### 구조 인지 청킹

마크다운, 코드, HTML 같은 구조화된 문서는 그 구조를 살려야 한다.

- 마크다운: 헤딩 단위(`#`, `##`)로 자르고 헤딩 경로를 메타데이터로 보존
- 코드: AST 파싱으로 함수·클래스 단위로 자름(tree-sitter 기반)
- HTML: DOM 구조 따라 섹션 단위로 자름

코드 청킹은 특히 까다롭다. 함수 하나가 청크 길이를 넘기면 어떻게 처리할지, import문은 어디에 붙일지, 클래스 메서드의 컨텍스트(부모 클래스 시그니처)를 어떻게 같이 보낼지 결정해야 한다. tree-sitter로 AST를 만들어 노드 단위로 잘라내는 게 표준이 됐다.

### 청킹의 실무 함정

- **길이 단위가 제각각**: 토큰 vs 문자 vs 단어. 임베딩 모델은 토큰 기준이고 사용자가 보는 건 문자 기준이다. 통일해서 측정해라.
- **OOV 토큰 폭증**: 코드, JSON, base64 같은 데이터는 일반 텍스트보다 토큰화 효율이 훨씬 떨어진다. 같은 1000 청크 길이여도 코드는 50줄밖에 안 들어간다.
- **메타데이터 손실**: 청크만 잘라서 벡터 DB에 넣고 끝내지 마라. 파일 경로, 헤딩 경로, 섹션 번호, 원문 위치 같은 메타데이터를 같이 저장해야 검색 결과를 사람이 추적할 수 있다.
- **중복 청크**: 청킹 후 동일 내용 청크가 여러 개 생기는 경우가 많다(반복되는 헤더, 푸터). MinHash나 해시 기반 중복 제거가 필요하다.

## 요약 전략

대화 이력이 컨텍스트를 넘어갈 때 단순히 잘라내면 맥락을 잃는다. 요약으로 압축하는 방식이 일반적이다.

### Summary Buffer

LangChain의 ConversationSummaryBufferMemory 패턴. 최근 N개 메시지는 그대로 두고, 그 이전 메시지는 요약본으로 대체한다. 새 메시지가 들어와서 버퍼가 차면 가장 오래된 것부터 요약에 흡수시킨다.

문제는 요약 호출 비용이 들고, 요약 자체에 정보 손실이 있다는 점이다. 사실 관계는 잘 보존하지만 세부 뉘앙스나 사용자 선호 같은 건 잘 사라진다.

### Hierarchical Summary

대화를 여러 계층으로 요약한다. 메시지 → 토픽 요약 → 세션 요약 → 장기 메모리. 각 계층은 다른 입도(granularity)를 갖는다. 사용자가 "지난주에 우리 뭐 얘기했지?" 같은 질문을 하면 세션 요약을 참조하고, "방금 결정한 거 다시 말해줘"는 메시지 버퍼를 참조한다.

구현 비용이 크다. 보통 챗봇 제품에서는 마지막 5~10개 메시지 + 시스템에 사용자 프로파일 정도로 단순화한다.

### Entity / Fact Extraction

요약 대신 대화에서 사실(entity, fact)만 추출해 KV 스토어에 저장한다. "사용자 이름: Alex", "선호 언어: Python", "프로젝트: YGSTUDY". 새 메시지가 올 때 관련 사실만 컨텍스트에 주입한다.

장점은 정밀하고 갱신이 쉽다는 점, 단점은 자유 형식 정보(분위기, 톤, 의도)는 못 잡는다. 일반 챗봇보다 고객지원·CRM 시나리오에 적합하다.

## 슬라이딩 윈도우

스트리밍 대화나 무한 컨텍스트가 필요할 때 사용한다. KV 캐시의 일부만 보관하고 오래된 토큰의 KV는 버린다.

### StreamingLLM (Xiao et al., 2023)

처음 몇 개 토큰(attention sink)과 최근 N개 토큰만 캐시에 유지하는 구조. attention sink를 버리면 어텐션 분포가 무너진다는 관찰에서 출발했다. 이걸 보존하면 RoPE 모델도 무한 스트림에서 안정적으로 동작한다.

```python
# 단순화한 슬라이딩 윈도우 관리
def maintain_kv_cache(kv_cache, max_window, num_sinks=4):
    """attention sink + 최근 N개 보존"""
    seq_len = kv_cache.size(2)
    if seq_len <= max_window:
        return kv_cache
    
    sinks = kv_cache[:, :, :num_sinks, :]
    recent = kv_cache[:, :, -(max_window - num_sinks):, :]
    return torch.cat([sinks, recent], dim=2)
```

이 방식은 컨텍스트 자체를 확장하지 않는다. 모델이 "최근 N개 + 시작 몇 개"만 볼 수 있게 한다. 긴 문서 QA 같은 작업에는 부적합하고, 무한 채팅·로그 모니터링 같은 시나리오에 쓴다.

### Mistral의 Sliding Window Attention

Mistral 7B가 채택한 어텐션 패턴. 각 레이어에서 토큰이 직전 W개 토큰만 본다. 레이어를 거치면서 receptive field가 W × N(레이어 수)로 늘어난다. CNN의 receptive field와 비슷한 발상.

전체 어텐션보다 빠르고 KV 캐시도 W에 묶인다. 다만 명시적인 롱 컨텍스트 회상은 약하다. Mistral 후속 모델(Mixtral, Mistral Large)은 sliding window를 빼고 full attention으로 돌아왔다.

## 실무 의사결정

긴 컨텍스트를 다룰 때 매번 마주치는 질문들이다.

**Q: 모든 정보를 컨텍스트에 넣을까, RAG로 검색해서 넣을까?**

A: 정보가 자주 바뀌면 RAG. 매 호출마다 같은 정보를 쓰고 작업이 종합적 추론을 요구하면 long context + caching. RAG는 검색 정확도가 상한이지만 비용·지연이 안정적이고, long context는 모델 회상력에 의존하지만 종합적 추론에는 강하다. 코드베이스 분석은 보통 long context, 문서 QA는 보통 RAG가 유리하다.

**Q: 컨텍스트가 모델 한계의 몇 %까지 채우는 게 안전한가?**

A: 광고된 한계의 25~50%가 실무에서 통하는 한계선이다. 200K 모델이면 50K~100K, 1M 모델이면 250K~500K. 그 이상은 Lost-in-the-Middle과 종합 추론 능력 저하가 본격적으로 나타난다. RULER 같은 벤치마크 결과를 참고해라.

**Q: 청크 크기는 얼마가 적당한가?**

A: 일반 텍스트는 256~512 토큰 청크, 200 토큰 정도 overlap이 통념이다. 코드는 함수 단위로 자르되 1000~2000 토큰까지 허용. 검색 품질에 가장 큰 영향을 주는 변수라 실제 검색 메트릭으로 튜닝해야 한다.

**Q: 캐시 히트율은 어떻게 올리나?**

A: 프롬프트를 prefix-stable하게 짜라. 동적인 부분(타임스탬프, 사용자별 정보, 변하는 도구 결과)을 메시지 시퀀스 뒤쪽에 몰아라. 시스템 프롬프트와 정적 컨텍스트는 앞쪽에 고정. 캐시 히트율을 로깅하고 0%인 케이스를 추적하면 보통 사소한 동적 요소가 원인이다.

## 정리

컨텍스트 윈도우는 단순히 "몇 토큰까지 받아주냐"의 숫자가 아니다. 포지셔널 인코딩이 외삽 가능한지(RoPE + YaRN/NTK), 모델이 중간 위치 정보를 회상할 수 있는지(Lost-in-the-Middle), prefill 비용을 캐싱으로 분산할 수 있는지가 모두 얽혀 있다.

광고된 1M, 2M 같은 숫자에 휘둘리지 말고 본인 워크로드에서 실측해라. needle in a haystack은 통과해도 종합 추론은 못 하는 경우가 흔하다. 청킹·검색으로 컨텍스트를 좁히는 게 비용·정확도 측면에서 거의 항상 유리하다. 컨텍스트를 통째로 넣는 건 정말 그렇게 해야만 하는 작업에 한정해라.
