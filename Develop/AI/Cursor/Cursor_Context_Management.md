# Cursor Context 관리

## 1. 컨텍스트가 중요한 이유

Cursor에서 AI의 응답 품질은 프롬프트에 어떤 컨텍스트를 넣느냐에 따라 결정된다. 같은 질문이라도 관련 파일을 정확히 첨부하면 프로젝트 컨벤션에 맞는 코드가 나오고, 아무것도 첨부하지 않으면 제네릭한 코드가 나온다.

Cursor는 `@` 멘션 시스템으로 컨텍스트를 제어한다. 채팅, Composer, 인라인 편집 어디서든 `@`를 입력하면 파일, 폴더, 코드베이스, 문서, 웹, git 정보를 프롬프트에 첨부할 수 있다.

문제는 컨텍스트 윈도우에 한계가 있다는 점이다. 토큰을 다 쓰면 AI가 이전 대화를 잊어버리고, 파일을 너무 많이 첨부하면 정작 중요한 정보가 밀려난다. 컨텍스트를 어떻게 관리하느냐가 Cursor를 잘 쓰느냐 못 쓰느냐의 차이다.

## 2. 컨텍스트 태그 상세

### 2.1 @files

특정 파일을 프롬프트에 첨부한다. 가장 자주 쓰는 태그다.

```
"@SecurityConfig.java 여기서 CORS 설정이 왜 안 먹히는지 봐줘"
"@application.yml DB 커넥션 풀 설정 확인해줘"
```

파일 이름을 입력하면 자동완성 목록이 뜬다. 경로 전체를 칠 필요 없이 파일명만 쳐도 된다.

**주의할 점:**

- 파일 하나를 첨부하면 그 파일의 전체 내용이 토큰으로 잡힌다. 1000줄짜리 파일을 첨부하면 그만큼 컨텍스트 윈도우를 잡아먹는다.
- 같은 이름의 파일이 여러 개 있으면 경로까지 포함해서 선택해야 한다. `@src/main/resources/application.yml`처럼 경로를 명시한다.
- 열려 있는 탭의 파일은 자동으로 일부 컨텍스트에 포함되지만, 명시적으로 `@`로 첨부한 파일이 우선순위가 더 높다.

### 2.2 @folders

폴더 전체를 컨텍스트에 추가한다. 폴더 안의 파일 목록과 구조가 포함된다.

```
"@src/main/java/com/example/order/ 이 패키지의 구조를 분석해줘"
"@src/main/resources/ 설정 파일들 보고 프로파일 분리해줘"
```

폴더를 첨부하면 그 안의 모든 파일 내용이 포함되는 게 아니다. 파일 목록과 구조 정보가 들어가고, AI가 필요하다고 판단하면 개별 파일을 읽어온다. 파일 전체를 다 읽는 것보다 토큰 소비가 적다.

**사용하면 좋은 상황:**

- 프로젝트 구조를 파악시키고 싶을 때
- 특정 도메인 패키지의 전체적인 흐름을 물어볼 때
- 어떤 파일을 첨부해야 할지 모를 때 (폴더를 주고 AI가 찾게 한다)

### 2.3 @codebase

코드베이스 전체를 임베딩 인덱스로 검색한다. 특정 파일을 모르는 상태에서 코드베이스 전체를 대상으로 질문할 때 쓴다.

```
"@codebase Redis 캐시를 사용하는 곳이 어디야?"
"@codebase 예외 처리가 어떤 패턴으로 되어 있어?"
"@codebase 결제 관련 로직이 어느 패키지에 있어?"
```

`@codebase`를 쓰면 Cursor가 질문과 관련된 코드를 임베딩 유사도 기반으로 찾아서 프롬프트에 포함시킨다. 코드베이스가 크더라도 관련 부분만 추출하기 때문에 토큰을 전부 소비하지 않는다.

**동작 방식:**

1. 질문을 임베딩 벡터로 변환
2. 코드베이스 인덱스에서 유사도가 높은 코드 조각을 검색
3. 검색 결과 중 상위 항목을 프롬프트에 삽입
4. AI가 삽입된 코드 조각을 기반으로 응답

**한계:**

- 임베딩 기반이라 키워드가 정확히 일치하지 않으면 못 찾는 경우가 있다. "DB 커넥션"이라고 검색했는데 코드에서는 `HikariConfig`로만 되어 있으면 검색 정확도가 떨어진다.
- 방금 수정한 파일이 인덱스에 바로 반영되지 않을 수 있다. 인덱싱에 약간의 딜레이가 있다.
- 검색 결과가 불충분할 때가 있다. 이때는 `@codebase`를 버리고 직접 `@files`로 관련 파일을 첨부하는 게 낫다.

### 2.4 @docs

외부 라이브러리나 프레임워크의 공식 문서를 검색해서 프롬프트에 포함시킨다.

```
"@docs Spring Security 6.x에서 SecurityFilterChain 설정하는 방법"
"@docs React Query v5의 useQuery 사용법"
```

`Cursor Settings > Features > Docs`에서 자주 쓰는 문서를 미리 등록해둘 수 있다. URL을 입력하면 Cursor가 해당 사이트를 크롤링해서 인덱싱한다.

```
등록 예시:
- Spring Boot: https://docs.spring.io/spring-boot/reference/
- React: https://react.dev
- TypeScript: https://www.typescriptlang.org/docs/
```

**등록하지 않은 문서도 검색 가능하지만**, 등록된 문서가 검색 정확도가 높다. 팀에서 쓰는 프레임워크 문서는 미리 등록해두는 게 좋다.

**주의할 점:**

- `@docs`는 문서 사이트의 구조에 따라 크롤링 품질이 달라진다. SPA로 된 문서 사이트는 크롤링이 잘 안 되는 경우가 있다.
- 문서 버전을 명시하지 않으면 최신 버전 기준으로 답변한다. Spring Boot 2.x를 쓰고 있는데 3.x 기준 답변이 나올 수 있으니 버전을 함께 적어야 한다.

### 2.5 @web

실시간 웹 검색 결과를 프롬프트에 포함시킨다.

```
"@web Spring Boot 3.4 릴리즈 노트"
"@web JDK 24 새 기능 정리"
"@web GitHub Actions에서 Gradle 캐시 설정하는 방법"
```

`@docs`와의 차이는 `@docs`가 등록된 문서 사이트를 검색하는 반면, `@web`은 인터넷 전체를 검색한다는 점이다. 최신 정보나 블로그 글, Stack Overflow 답변 등을 참고할 때 쓴다.

**주의할 점:**

- 웹 검색 결과의 품질을 보장할 수 없다. 오래된 정보나 잘못된 정보가 섞여 들어올 수 있다.
- 검색 결과를 AI가 그대로 가져오는 게 아니라 요약해서 포함시키기 때문에, 원문을 직접 확인해야 하는 경우가 있다.

### 2.6 @git

git 히스토리를 컨텍스트에 포함시킨다.

```
"@git 최근 커밋에서 뭐가 바뀌었어?"
"@git 이 파일의 최근 변경 이력을 보여줘"
"@git PR #42에서 변경된 내용 설명해줘"
```

최근 커밋, diff, 브랜치 정보를 AI에게 전달할 수 있다. 코드 리뷰를 요청하거나, 최근 변경 사항의 영향 범위를 분석할 때 쓴다.

**사용 예시:**

```
"@git 최근 3개 커밋의 diff를 보고, 이 변경이 기존 테스트에 영향을 주는지 분석해줘"
"@git main 브랜치와 현재 브랜치의 차이점 정리해줘"
```

### 2.7 @definitions

심볼의 정의를 찾아서 컨텍스트에 포함시킨다. 코드에서 특정 클래스나 함수의 선언부를 빠르게 참조할 때 쓴다.

```
"@definitions OrderService 클래스의 메서드 목록 보여줘"
```

IDE의 Go to Definition과 비슷한 역할이다. 파일 전체를 첨부하지 않고 정의 부분만 가져오므로 토큰을 아낄 수 있다.

### 2.8 @notepads

`.cursor/notepads/` 폴더에 저장해둔 메모를 참조한다. 프로젝트별 컨벤션, API 스펙, 자주 쓰는 설정 등을 정리해두고 필요할 때 꺼내 쓴다.

```
"@notepads/api-convention 이 컨벤션에 맞춰서 응답 DTO 만들어줘"
"@notepads/db-schema 이 스키마 기반으로 Entity 클래스 생성해줘"
```

Rules와 다른 점은 Notepads는 명시적으로 호출해야 적용된다는 것이다. Rules는 조건에 따라 자동 적용되지만 Notepads는 `@`로 직접 부를 때만 포함된다.

## 3. 코드베이스 인덱싱

### 3.1 인덱싱 동작 원리

Cursor는 프로젝트를 열면 코드베이스 전체를 임베딩 인덱스로 변환한다. 이 인덱스가 `@codebase` 검색의 기반이 된다.

**인덱싱 과정:**

```
[1] 프로젝트의 모든 파일을 스캔
[2] 각 파일을 적절한 크기의 청크로 분할
[3] 각 청크를 임베딩 벡터로 변환
[4] 벡터 인덱스에 저장
[5] 파일이 변경되면 해당 파일의 인덱스만 재생성
```

인덱싱 상태는 우측 하단 상태바에서 확인할 수 있다. `Cursor Settings > Features > Codebase Indexing`에서 인덱싱을 관리한다.

### 3.2 인덱싱이 느린 경우

대규모 프로젝트에서 인덱싱이 오래 걸리는 원인은 대부분 불필요한 파일 때문이다. `node_modules`, `build`, `.gradle` 같은 디렉토리가 인덱싱 대상에 포함되면 시간이 수십 분 이상 걸릴 수 있다.

인덱싱 대상에서 제외해야 하는 디렉토리:

```
node_modules/
build/
dist/
.gradle/
target/
.next/
__pycache__/
venv/
.git/  (기본 제외)
```

이런 디렉토리를 제외하려면 `.cursorignore`를 설정한다.

### 3.3 인덱스 초기화

인덱스가 꼬인 것 같을 때(검색 결과가 이상하거나, 삭제한 파일이 계속 검색될 때) 인덱스를 재생성할 수 있다.

```
Cursor Settings > Features > Codebase Indexing > Resync Index
```

프로젝트 규모에 따라 몇 분에서 십수 분 걸린다. 인덱싱이 완료되기 전까지 `@codebase` 검색 결과가 불완전하다.

## 4. .cursorignore 설정

`.cursorignore`는 Cursor가 무시할 파일과 디렉토리를 지정하는 파일이다. 프로젝트 루트에 생성한다. `.gitignore`와 문법이 같다.

### 4.1 기본 설정

```gitignore
# .cursorignore

# 빌드 결과물
build/
dist/
target/
out/

# 의존성
node_modules/
.gradle/
.m2/

# 생성된 파일
*.min.js
*.min.css
*.map
*.lock

# 대용량 데이터
*.sql
*.csv
*.json  # 설정 파일 제외하려면 !로 예외 처리
!package.json
!tsconfig.json

# IDE 설정
.idea/
.vscode/

# 환경 변수
.env
.env.*
```

### 4.2 .gitignore와의 관계

Cursor는 `.gitignore`에 지정된 파일도 기본적으로 인덱싱에서 제외한다. `.cursorignore`는 `.gitignore`에 추가로 Cursor에서만 무시할 파일을 지정할 때 쓴다.

예를 들어 `*.lock` 파일은 git에는 포함하지만 Cursor 인덱싱에서는 제외하고 싶을 때 `.cursorignore`에 추가한다. lock 파일은 수천 줄짜리인 경우가 많아서 인덱싱하면 토큰만 낭비된다.

### 4.3 .cursorignore vs .cursorindexingignore

Cursor에는 두 종류의 ignore 파일이 있다.

| 파일 | 역할 |
|------|------|
| `.cursorignore` | 인덱싱 + AI 접근 모두 차단. AI가 이 파일을 읽거나 수정할 수 없다 |
| `.cursorindexingignore` | 인덱싱만 제외. AI가 `@files`로 직접 첨부하면 읽을 수 있다 |

보안상 민감한 파일(환경변수, 인증 키)은 `.cursorignore`에 넣어야 한다. 단순히 인덱싱 성능을 위해 제외하는 파일(lock 파일, 빌드 결과물)은 `.cursorindexingignore`에 넣는다.

```gitignore
# .cursorignore - AI가 절대 접근하면 안 되는 파일
.env
.env.*
**/credentials.json
**/secrets/

# .cursorindexingignore - 인덱싱만 제외
node_modules/
build/
*.lock
```

## 5. 대규모 프로젝트에서 컨텍스트 관리

### 5.1 모노레포 환경

수십만 줄 규모의 모노레포에서는 `@codebase`만으로 원하는 코드를 찾기 어렵다. 검색 범위가 너무 넓어서 관련 없는 코드가 섞여 들어온다.

**대응 방법:**

`@codebase` 대신 `@folders`로 범위를 좁히고, 그 안에서 `@files`로 정확한 파일을 지정한다.

```
# 나쁜 예 - 모노레포 전체에서 검색
"@codebase 결제 로직 어디 있어?"

# 나은 예 - 패키지 단위로 범위 제한
"@packages/payment/ 이 패키지에서 결제 승인 처리하는 파일이 뭐야?"
```

`.cursorignore`로 작업 중인 패키지 외의 코드를 제외하는 방법도 있다. 다만 이 방법은 패키지를 전환할 때마다 `.cursorignore`를 수정해야 하므로 번거롭다.

### 5.2 멀티 모듈 프로젝트

Spring Boot 멀티 모듈 프로젝트에서 흔한 상황이다. `core`, `api`, `batch`, `admin` 같은 모듈이 여러 개 있을 때, AI가 어느 모듈의 코드를 참조해야 하는지 헷갈려한다.

```
# 모듈을 명시하는 습관
"@modules/api/src/main/java/com/example/order/OrderController.java
이 컨트롤러에서 호출하는 서비스가 @modules/core/ 에 있는데,
트랜잭션 전파가 제대로 되는지 확인해줘"
```

프로젝트 구조를 `.cursor/rules/`에 명시해두면 AI가 모듈 간 관계를 이해하는 데 도움이 된다.

```markdown
<!-- .cursor/rules/project-modules.mdc -->
---
description: 멀티 모듈 구조 안내
alwaysApply: true
---

# 모듈 구조
- core: 도메인 엔티티, 리포지토리, 서비스 인터페이스
- api: REST API 컨트롤러, DTO, api 전용 서비스 구현체
- batch: 배치 잡, 스케줄러
- admin: 관리자 페이지 API
- common: 공통 유틸, 예외, 상수
```

### 5.3 프론트엔드 + 백엔드 풀스택 프로젝트

프론트엔드와 백엔드가 한 레포에 있을 때, 백엔드 작업 중에 프론트엔드 코드가 `@codebase` 검색에 섞여 들어오는 경우가 있다.

`.cursorindexingignore`로 현재 작업하지 않는 쪽을 제외하거나, Rules에서 현재 작업 영역을 명시한다.

```gitignore
# .cursorindexingignore
# 백엔드 작업 중일 때 프론트엔드 제외
frontend/
client/
```

### 5.4 레거시 코드가 많은 프로젝트

마이그레이션 중인 프로젝트에서 레거시 코드와 신규 코드가 공존하는 경우, AI가 레거시 패턴을 참고해서 코드를 생성하는 문제가 있다.

Rules에서 어떤 코드를 참조해야 하는지 명시한다.

```markdown
<!-- .cursor/rules/migration.mdc -->
---
description: 마이그레이션 가이드
alwaysApply: true
---

# 코드 참조 규칙
- 새 코드 작성 시 src/v2/ 패키지의 패턴을 따른다
- src/legacy/ 패키지의 코드는 참조하지 않는다
- 엔티티는 src/v2/domain/ 패키지의 구조를 따른다
```

## 6. 토큰 초과 시 대처법

### 6.1 토큰 초과 증상

컨텍스트 윈도우가 가득 차면 다음과 같은 증상이 나타난다.

- 이전에 수정한 파일을 원래대로 되돌려버린다
- 같은 코드를 중복 생성한다
- 프롬프트를 이해하지 못하고 엉뚱한 답변을 한다
- "이전에 말했듯이"라고 하면 "무슨 말씀인지 모르겠습니다"라고 응답한다

채팅 입력창 근처에 토큰 사용량 표시가 있다. 이 수치가 모델의 컨텍스트 윈도우 한계에 가까워지면 새 세션을 시작해야 한다.

### 6.2 세션 분리

하나의 세션에서 모든 작업을 하려고 하면 반드시 토큰이 넘친다. 작업 단위로 세션을 나누는 게 기본이다.

```
# 나쁜 예 - 하나의 세션에서 전부 처리
세션 1: "Entity 만들어줘" → "Service도 추가해줘" → "Controller도" 
       → "테스트도" → "API 문서도" → (토큰 초과)

# 나은 예 - 작업 단위로 세션 분리
세션 1: Entity + Repository 생성
세션 2: @Entity.java 참조해서 Service + Controller 생성
세션 3: @Service.java @Controller.java 참조해서 테스트 작성
```

새 세션을 열 때 이전 세션의 결과물을 `@files`로 첨부하면 연속성이 유지된다.

### 6.3 불필요한 컨텍스트 줄이기

토큰을 아끼는 습관:

**파일 첨부를 최소화한다.** 관련 파일 3~5개를 정확히 고르는 게, 폴더 전체를 던지는 것보다 낫다. "이것도 필요할 수 있으니까"라는 이유로 파일을 추가하면 토큰만 낭비된다.

**대화를 짧게 유지한다.** 5~10턴 이내로 하나의 작업을 끝내는 게 이상적이다. 대화가 길어지면 초반 컨텍스트가 밀려나면서 AI가 맥락을 잃는다.

**코드 전체를 붙여넣지 않는다.** "이 코드 리뷰해줘"라며 500줄을 붙여넣는 것보다 `@files`로 파일을 첨부하는 게 낫다. Cursor가 파일 내용을 더 효율적으로 처리한다.

### 6.4 Long Context Chat

Cursor에는 Long Context Chat 기능이 있다. 일반 채팅보다 더 긴 컨텍스트 윈도우를 사용하는 모드다. 대규모 파일을 분석하거나 여러 파일을 동시에 참조해야 할 때 쓴다.

`Cmd + L`로 채팅을 연 뒤, 모델 선택 드롭다운에서 Long Context 지원 모델을 선택한다. 토큰 한도가 더 높은 대신 응답 속도가 느리고, Fast 요청 크레딧 소비가 더 크다.

## 7. 컨텍스트 태그 조합 패턴

실무에서 자주 쓰는 태그 조합이다.

### 버그 수정

```
"@SecurityConfig.java @application.yml
CORS 프리플라이트 요청이 403으로 떨어지는데, 설정에 뭐가 빠졌는지 봐줘.
@git 최근에 이 파일들 변경한 커밋이 있으면 같이 확인해줘"
```

관련 파일을 명시하고, `@git`으로 최근 변경 사항도 함께 확인하는 패턴이다.

### 새 기능 구현

```
"@OrderController.java @OrderService.java
이 패턴을 참고해서 Product 도메인의 CRUD API를 만들어줘.
@docs Spring Data JPA에서 Specification 쓰는 방법도 참고해줘"
```

기존 코드를 참조 패턴으로 지정하고, 필요한 문서를 `@docs`로 함께 첨부한다.

### 코드 리뷰

```
"@git PR #42의 변경 내용을 리뷰해줘.
@codebase 변경된 부분이 다른 코드에 영향을 주는지도 확인해줘"
```

`@git`으로 변경 내용을 가져오고, `@codebase`로 영향 범위를 분석한다.

### 트러블슈팅

```
"@web 'Spring Boot HikariCP connection leak detected' 에러 해결법
@application.yml 현재 커넥션 풀 설정을 같이 봐줘"
```

에러 메시지를 `@web`으로 검색하고, 현재 설정 파일을 함께 첨부해서 구체적인 해결 방법을 얻는다.

## 8. 정리

| 상황 | 추천 태그 |
|------|-----------|
| 특정 파일의 코드를 보여주고 싶을 때 | `@files` |
| 프로젝트 구조를 파악시키고 싶을 때 | `@folders` |
| 어떤 파일인지 모르고 코드베이스에서 찾고 싶을 때 | `@codebase` |
| 라이브러리/프레임워크 사용법을 알고 싶을 때 | `@docs` |
| 최신 정보나 에러 해결법을 검색하고 싶을 때 | `@web` |
| 최근 변경 이력을 참고하고 싶을 때 | `@git` |
| 심볼 정의만 빠르게 참조하고 싶을 때 | `@definitions` |
| 프로젝트 컨벤션을 적용하고 싶을 때 | `@notepads` |

컨텍스트 관리의 핵심은 **필요한 정보만 정확히 첨부하는 것**이다. "많이 넣으면 잘 답변하겠지"는 틀린 생각이다. 토큰 한도 안에서 관련도 높은 정보를 선별해서 넣어야 AI가 정확한 답변을 한다.
