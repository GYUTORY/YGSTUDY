# GBrain

## 1. GBrain이란

GBrain은 한국의 AI 스타트업 마커(Marker)가 개발한 기업용 AI 지식 관리 플랫폼이다. 사내 문서, 매뉴얼, 위키, 데이터베이스 등을 LLM과 연결해서 조직 내부 지식을 검색하고 활용할 수 있게 해준다.

단순한 챗봇이 아니라, 기업이 보유한 비정형 데이터를 구조화하고 RAG(Retrieval-Augmented Generation) 파이프라인을 통해 사내 맥락에 맞는 답변을 생성하는 시스템이다.

### 1.1 GBrain이 해결하는 문제

기업 내부에서 흔히 겪는 상황이 있다.

- 신규 입사자가 온보딩 문서를 찾는데 Confluence, Notion, Google Drive, 사내 위키가 전부 따로 놀고 있다
- "이거 어디에 정리되어 있었지?" 하고 슬랙에서 물어보면, 아는 사람이 링크를 던져주는 식으로 지식이 공유된다
- 문서가 있어도 최신 버전인지 확인이 안 된다

GBrain은 이런 분산된 지식 소스를 하나로 모아서 자연어로 질문하면 관련 문서를 찾아 답변을 만들어준다.

### 1.2 포지셔닝

| 구분 | GBrain | 일반 ChatGPT/Claude | 사내 검색 도구(Elastic 등) |
|------|--------|---------------------|--------------------------|
| 데이터 소스 | 사내 문서 연동 | 공개 데이터 학습 | 사내 문서 인덱싱 |
| 답변 방식 | RAG 기반 생성형 답변 | 범용 답변 | 키워드 매칭 검색 결과 |
| 맥락 이해 | 조직 내부 맥락 반영 | 일반적 맥락 | 맥락 없음 |
| 접근 제어 | 문서별 권한 연동 | 없음 | 인덱스 단위 |

## 2. 아키텍처

### 2.1 전체 구조

GBrain은 크게 세 개 레이어로 구성된다.

```
[데이터 수집 레이어]
  - 커넥터: Confluence, Notion, Google Drive, SharePoint, S3 등
  - 크롤러: 사내 웹페이지, 위키
  - 직접 업로드: PDF, DOCX, PPTX, HWP

[처리 레이어]
  - 문서 파싱 및 청킹(Chunking)
  - 임베딩 생성 (벡터화)
  - 벡터 DB 저장 (유사도 검색용)
  - 메타데이터 인덱싱 (필터링용)

[서비스 레이어]
  - RAG 파이프라인: 검색 → 리랭킹 → 프롬프트 구성 → LLM 생성
  - API 서버: REST API 제공
  - 웹 UI: 채팅 인터페이스, 관리자 대시보드
```

### 2.2 RAG 파이프라인 동작 방식

사용자가 "신규 입사자 장비 신청은 어떻게 하나요?"라고 질문하면:

1. **쿼리 처리**: 질문을 벡터로 변환한다. 동시에 키워드 검색도 수행한다(하이브리드 검색).
2. **문서 검색**: 벡터 DB에서 유사한 문서 청크를 가져온다. 보통 상위 10~20개.
3. **리랭킹**: 가져온 청크를 질문과의 관련도로 재정렬한다. Cross-encoder 모델을 쓰는 경우가 많다.
4. **프롬프트 구성**: 상위 5개 정도의 청크를 LLM 프롬프트에 컨텍스트로 넣는다.
5. **답변 생성**: LLM이 컨텍스트를 기반으로 답변을 생성한다. 출처(source) 문서 링크도 함께 제공한다.

### 2.3 청킹(Chunking) 방식

문서를 LLM에 넣으려면 적절한 크기로 잘라야 한다. GBrain에서 사용하는 청킹 방식:

- **고정 크기 청킹**: 토큰 수 기준으로 일정 크기로 자른다. 단순하지만 문맥이 끊기는 경우가 있다.
- **의미 기반 청킹**: 문단, 섹션 단위로 자른다. 문맥 보존이 낫지만 크기가 불균일하다.
- **슬라이딩 윈도우**: 앞뒤 청크와 겹치는 부분을 두어 문맥 유실을 줄인다.

실무에서 청킹 크기를 잘못 설정하면 검색 품질이 크게 떨어진다. 보통 512~1024 토큰 사이에서 시작하고, 문서 특성에 따라 조정한다.

## 3. 주요 기능

### 3.1 멀티소스 커넥터

여러 데이터 소스를 연결할 수 있다. 현재 지원하는 주요 커넥터:

- **클라우드 스토리지**: Google Drive, OneDrive, SharePoint, S3
- **위키/문서**: Confluence, Notion, GitBook
- **커뮤니케이션**: Slack (채널 메시지 인덱싱)
- **직접 업로드**: PDF, DOCX, PPTX, HWP, TXT, Markdown

커넥터를 설정하면 주기적으로 동기화가 돌면서 새 문서나 변경된 문서를 자동으로 반영한다. 삭제된 문서도 인덱스에서 제거된다.

### 3.2 권한 관리

사내 문서에는 접근 권한이 있다. GBrain은 원본 소스의 권한 체계를 따른다.

```
예시: Confluence 연동 시

- A팀 스페이스 → A팀 멤버만 검색 가능
- 전사 공개 스페이스 → 모든 사용자 검색 가능
- 비공개 페이지 → 원본 권한자만 접근
```

이게 제대로 동작하지 않으면 보안 사고로 이어진다. 커넥터 설정 후 반드시 권한 동기화가 정상인지 확인해야 한다.

### 3.3 답변 품질 관리

- **출처 표시**: 모든 답변에 참조한 문서의 링크를 보여준다. 사용자가 원문을 직접 확인할 수 있다.
- **피드백 수집**: 답변에 대한 좋아요/싫어요를 수집해서 검색 품질 개선에 활용한다.
- **할루시네이션 방지**: 컨텍스트에 없는 내용을 답변하지 않도록 프롬프트에 제약 조건을 건다. 그래도 100% 방지는 안 된다.

### 3.4 관리자 대시보드

- 문서 인덱싱 현황: 총 문서 수, 마지막 동기화 시간, 실패한 문서 목록
- 사용 통계: 일별 질문 수, 답변 만족도, 자주 묻는 질문
- 검색 품질 모니터링: 답변 없음 비율, 낮은 유사도 질문 목록

## 4. 실제 사용법

### 4.1 초기 설정

GBrain을 도입할 때 가장 먼저 해야 할 일은 데이터 소스 연결이다.

```
1. 관리자 콘솔 접속
2. 데이터 소스 > 새 커넥터 추가
3. 인증 정보 입력 (OAuth, API Key 등)
4. 동기화 범위 설정 (특정 스페이스, 폴더 등)
5. 초기 인덱싱 실행
```

초기 인덱싱은 문서 양에 따라 수 시간에서 하루 이상 걸리는 경우도 있다. 수만 건의 문서가 있는 Confluence라면 인덱싱에 시간이 꽤 걸린다.

### 4.2 API 연동

자체 서비스에서 GBrain의 검색 기능을 사용하려면 API를 호출한다.

```python
import requests

GBRAIN_API_URL = "https://api.gbrain.ai/v1/search"
API_KEY = "your-api-key"

def ask_gbrain(question: str, workspace_id: str) -> dict:
    response = requests.post(
        GBRAIN_API_URL,
        headers={
            "Authorization": f"Bearer {API_KEY}",
            "Content-Type": "application/json"
        },
        json={
            "query": question,
            "workspace_id": workspace_id,
            "top_k": 5  # 상위 5개 문서 청크 사용
        }
    )
    return response.json()

result = ask_gbrain(
    question="배포 프로세스가 어떻게 되나요?",
    workspace_id="ws-abc123"
)

print(result["answer"])
for source in result["sources"]:
    print(f"- {source['title']}: {source['url']}")
```

### 4.3 Slack 봇 연동

GBrain을 Slack 봇으로 연결하면 채널에서 바로 질문할 수 있다.

```
@gbrain 서버 점검 일정이 언제야?

GBrain:
다음 서버 점검은 4월 15일 화요일 02:00~06:00으로 예정되어 있습니다.

출처: [서버 점검 공지 - Confluence](https://wiki.example.com/pages/123)
```

Slack 연동 시 주의할 점은 봇이 초대된 채널의 메시지만 인덱싱된다는 것이다. 프라이빗 채널에 봇을 추가하면 해당 채널 메시지도 검색 대상에 포함되니 보안 정책을 먼저 확인해야 한다.

## 5. 다른 AI 도구와의 차이점

### 5.1 ChatGPT Enterprise / Claude for Work와의 비교

| 항목 | GBrain | ChatGPT Enterprise | Claude for Work |
|------|--------|--------------------|-----------------| 
| 핵심 목적 | 사내 지식 검색·활용 | 범용 AI 어시스턴트 | 범용 AI 어시스턴트 |
| RAG 파이프라인 | 자체 구축, 커스텀 가능 | 파일 업로드 기반 | 프로젝트 기반 |
| 데이터 소스 연동 | 다양한 커넥터 제공 | 제한적 | 제한적 |
| 한국어 문서 처리 | 한국어 특화 (HWP 등) | 범용 | 범용 |
| 온프레미스 배포 | 가능 | 불가 | 불가 |
| 권한 관리 | 원본 소스 권한 연동 | 단순 공유 | 프로젝트 단위 |

GBrain의 가장 큰 차별점은 **사내 데이터 소스와의 깊은 연동**과 **한국 기업 환경에 맞는 지원**(HWP 파싱, 한국어 임베딩 모델 등)이다.

### 5.2 Glean, Guru 등 해외 유사 서비스와의 비교

Glean이나 Guru 같은 해외 enterprise search 서비스도 비슷한 기능을 제공한다. 차이점은:

- **한국어 처리 품질**: 한국어 형태소 분석, 조사 처리 등에서 GBrain이 낫다. 해외 서비스는 한국어에 대한 고려가 부족한 경우가 많다.
- **HWP 지원**: 한국 공공기관이나 대기업에서 아직 HWP를 쓰는 곳이 있다. GBrain은 HWP 파싱을 지원한다.
- **국내 SaaS 연동**: 네이버웍스(NAVER Works), 잔디(Jandi) 같은 국내 서비스 커넥터를 제공하는 경우가 있다.
- **데이터 주권**: 국내 클라우드나 온프레미스 배포가 가능해서 데이터가 해외로 나가지 않는다.

## 6. 도입 시 주의사항

### 6.1 데이터 품질이 곧 답변 품질이다

GBrain에 연결하는 문서의 품질이 낮으면 답변 품질도 낮다. "쓰레기가 들어가면 쓰레기가 나온다(Garbage In, Garbage Out)"는 RAG 시스템에서 특히 잘 맞는 말이다.

실제로 겪는 문제:

- 오래된 문서가 최신 문서보다 먼저 검색되는 경우. 날짜 기반 가중치를 조정해야 한다.
- 같은 내용이 여러 문서에 중복으로 있으면 어떤 게 정확한 건지 혼란이 생긴다.
- 문서 형식이 너무 다양하면 (표, 이미지, 코드블록 등) 파싱 품질이 떨어진다.

도입 전에 사내 문서 정리를 먼저 하는 게 맞다. 문서 정리 없이 AI만 붙이면 기대와 다른 결과가 나온다.

### 6.2 권한 동기화 문제

가장 민감한 부분이다. 원본 소스의 권한이 GBrain에 제대로 반영되지 않으면:

- 인사팀 문서가 개발팀에게 노출되는 사고가 발생할 수 있다
- 퇴사자 계정이 비활성화되지 않은 상태로 접근이 가능할 수 있다

주기적으로 권한 동기화 상태를 점검해야 한다. 특히 조직 개편이나 인사 이동이 있을 때 동기화가 정상 동작하는지 확인이 필요하다.

### 6.3 LLM 비용

RAG 파이프라인에서 LLM 호출은 비용이 발생한다. 질문 하나에 컨텍스트로 넣는 문서 양이 많으면 토큰 소비가 커진다.

비용을 관리하는 방법:

- 컨텍스트에 넣는 청크 수를 제한한다 (top_k 조정)
- 간단한 질문은 임베딩 유사도만으로 답변하고, LLM 호출을 건너뛴다
- 캐싱을 활용한다. 같은 질문이 반복되면 이전 답변을 재사용한다

### 6.4 사용자 교육

도구를 도입해도 사람들이 안 쓰면 의미가 없다. 실제로 도입 후 사용률이 낮은 경우를 보면:

- 검색 결과가 기대에 못 미쳐서 한두 번 써보고 안 쓰는 경우
- Slack이나 기존 검색에 익숙해서 새 도구를 쓸 동기가 없는 경우
- 어떤 질문을 할 수 있는지 모르는 경우

팀별로 실제 업무에서 자주 묻는 질문을 예시로 보여주고, 초기에 피드백을 받아서 검색 품질을 빠르게 개선하는 게 도입 성공의 핵심이다.

## 7. 트러블슈팅

### 7.1 검색 결과가 부정확한 경우

**증상**: 질문과 관련 없는 문서가 검색된다.

원인과 대응:

- **임베딩 모델 문제**: 한국어에 약한 임베딩 모델을 쓰고 있다면 한국어 특화 모델(예: KoSimCSE, multilingual-e5 등)로 교체한다.
- **청킹 크기 문제**: 청크가 너무 크면 관련 없는 내용이 섞여 들어간다. 반대로 너무 작으면 문맥이 부족하다. 512~1024 토큰 범위에서 조정한다.
- **메타데이터 필터 미설정**: 문서 유형, 작성일, 팀 등의 메타데이터로 필터링하면 검색 정확도가 올라간다.

### 7.2 동기화가 실패하는 경우

**증상**: 커넥터 상태가 에러로 표시되고 새 문서가 반영되지 않는다.

확인할 것:

- API 토큰이 만료되지 않았는지 확인한다. OAuth 토큰은 일정 기간 후 갱신이 필요하다.
- 원본 소스의 API rate limit에 걸리지 않았는지 확인한다. Confluence API는 동시 요청 수 제한이 있다.
- 네트워크 접근이 가능한지 확인한다. 온프레미스 배포인 경우 방화벽 규칙을 점검한다.

### 7.3 답변에서 할루시네이션이 발생하는 경우

**증상**: 문서에 없는 내용을 그럴듯하게 답변한다.

대응 방법:

- 시스템 프롬프트에 "제공된 컨텍스트에 없는 내용은 답변하지 마라"는 지시를 명시한다.
- temperature를 낮춘다. 0.0~0.3 범위로 설정하면 창의적 답변이 줄어든다.
- 검색된 문서의 유사도 점수가 임계값 이하인 경우 "관련 문서를 찾을 수 없습니다"로 응답하게 한다.

### 7.4 응답 속도가 느린 경우

**증상**: 질문 후 답변까지 10초 이상 걸린다.

병목 구간별 대응:

- **검색 단계**: 벡터 DB 인덱스가 최적화되어 있는지 확인한다. 문서 수가 많으면 ANN(Approximate Nearest Neighbor) 인덱스 파라미터를 조정한다.
- **리랭킹 단계**: Cross-encoder 리랭킹이 느리면 가벼운 모델로 바꾸거나 후보 수를 줄인다.
- **LLM 생성 단계**: 컨텍스트 길이를 줄이거나 스트리밍 응답을 적용한다.

### 7.5 HWP 파일 파싱 오류

**증상**: HWP 문서의 내용이 깨져서 인덱싱된다.

한글 파일 중 오래된 `.hwp` 형식은 바이너리 포맷이라 파싱이 까다롭다. `.hwpx`(OOXML 기반)는 상대적으로 파싱이 잘 된다.

대응:

- 가능하면 HWP를 PDF로 변환 후 업로드한다
- `.hwpx` 형식으로 재저장 후 연동한다
- 표나 특수 서식이 많은 HWP는 파싱 결과를 수동으로 확인한다
