# 데이터베이스 샤딩

## 샤딩이란

샤딩(Sharding)은 하나의 데이터베이스를 여러 개의 독립적인 데이터베이스 인스턴스로 나누는 수평 확장 기법이다. 각 인스턴스를 "샤드(Shard)"라 부르고, 전체 데이터의 일부만 담당한다.

```
전체 데이터: 1억 건의 사용자

샤드 분할 후:
├─ 샤드 1: 사용자 ID 1 ~ 2,500만
├─ 샤드 2: 사용자 ID 2,500만+1 ~ 5,000만
├─ 샤드 3: 사용자 ID 5,000만+1 ~ 7,500만
└─ 샤드 4: 사용자 ID 7,500만+1 ~ 1억
```

### 샤딩 vs 파티셔닝

혼동하는 경우가 많은데 핵심 차이는 물리적 분리 여부다.

- **파티셔닝**: 하나의 DB 서버 안에서 테이블을 논리적으로 나눈다. 물리적으로 같은 서버에 존재한다.
- **샤딩**: 데이터를 여러 독립적인 DB 서버로 물리적으로 분산한다. 각 샤드는 별도의 하드웨어에서 돌아간다.

파티셔닝만으로 단일 서버의 CPU, 메모리, 디스크 I/O 한계를 넘을 수 없다. 그 한계를 넘어야 할 때 샤딩이 필요하다.

## 샤딩 도입 전 대안 비교

샤딩은 복잡도를 크게 올린다. 먼저 다른 방법으로 해결 가능한지 확인해야 한다.

### Read Replica (읽기 복제본)

읽기 부하가 문제인 경우 가장 먼저 시도할 방법이다.

```
쓰기 → Primary DB (1대)
읽기 → Replica DB (N대)

MySQL 설정 예:
# Primary
[mysqld]
server-id=1
log-bin=mysql-bin
binlog-format=ROW

# Replica
[mysqld]
server-id=2
relay-log=relay-bin
read-only=ON
```

**한계**: 쓰기 부하는 분산되지 않는다. 쓰기가 초당 수천 건 이상이면 Primary가 병목이 된다. 복제 지연(replication lag)이 발생하면 읽기 결과가 오래된 데이터일 수 있다.

### 테이블 파티셔닝

단일 테이블이 너무 커서 쿼리가 느린 경우에 유용하다.

```sql
-- MySQL Range Partitioning
CREATE TABLE orders (
    id BIGINT AUTO_INCREMENT,
    user_id BIGINT,
    created_at DATETIME,
    amount DECIMAL(10,2),
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (YEAR(created_at)) (
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

**한계**: 서버 한 대의 자원(CPU, 메모리, 디스크) 제약을 벗어날 수 없다. 파티션 간 조인은 가능하지만 전체 파티션을 스캔하는 쿼리는 여전히 느리다.

### 캐싱 레이어

자주 읽히는 데이터를 메모리에 올려 DB 부하를 줄인다.

```java
// Redis 캐싱 예시
public User getUser(long userId) {
    String key = "user:" + userId;
    User cached = redisTemplate.opsForValue().get(key);
    if (cached != null) {
        return cached;
    }

    User user = userRepository.findById(userId);
    redisTemplate.opsForValue().set(key, user, Duration.ofMinutes(30));
    return user;
}
```

**한계**: 캐시 무효화(invalidation) 관리가 까다롭다. 쓰기가 많은 데이터는 캐시 히트율이 낮아서 효과가 없다. 데이터 자체의 크기가 서버 한계를 넘으면 캐싱으로 해결할 수 없다.

### 대안 비교 정리

| 방법 | 해결하는 문제 | 해결 못하는 문제 | 도입 난이도 |
|------|-------------|-----------------|-----------|
| Read Replica | 읽기 부하 | 쓰기 부하, 저장 용량 | 낮음 |
| 파티셔닝 | 단일 테이블 크기 | 서버 자원 한계 | 낮음 |
| 캐싱 | 반복 읽기 부하 | 쓰기 부하, 저장 용량 | 중간 |
| **샤딩** | 쓰기 부하, 저장 용량, 읽기 부하 | - | **높음** |

**샤딩을 검토해야 하는 시점**:
- 단일 서버 디스크 70% 이상이고 데이터가 계속 증가한다
- Read Replica를 써도 쓰기 부하로 Primary CPU가 80% 이상이다
- 인덱스 크기가 메모리를 초과해서 쿼리 응답이 느려졌다
- 글로벌 서비스 확장으로 지역별 데이터 격리가 필요하다

## 수평 샤딩과 수직 샤딩

### 수평 샤딩

같은 스키마의 테이블을 여러 DB에 행(Row) 단위로 나누는 방식이다. 모든 샤드의 테이블 구조가 동일하다.

```
주문 테이블 (1억 건) → 수평 샤딩:
├─ 샤드 1: 주문 ID 1~2,500만       (스키마 동일)
├─ 샤드 2: 주문 ID 2,500만+1~5,000만  (스키마 동일)
├─ 샤드 3: 주문 ID 5,000만+1~7,500만  (스키마 동일)
└─ 샤드 4: 주문 ID 7,500만+1~1억    (스키마 동일)
```

**장점**: 선형 확장 가능, 부하 분산, 장애 격리
**단점**: 크로스 샤드 조인이 어렵다, 리밸런싱이 복잡하다, 샤드 키 선택이 중요하다

### 수직 샤딩

테이블이나 도메인 단위로 각각 다른 DB에 저장하는 방식이다. 각 샤드가 다른 스키마를 가진다.

```
하나의 DB에 모든 테이블 → 수직 샤딩:
├─ 인증 DB: users_auth (id, email, password_hash, last_login)
├─ 프로필 DB: users_profile (id, name, bio, avatar_url)
├─ 주문 DB: orders (id, user_id, amount, status)
└─ 결제 DB: payments (id, order_id, card_info, txn_id)
```

**장점**: 도메인별 독립 확장, 보안 격리(결제 DB를 별도 네트워크에 배치), 팀별 독립 관리
**단점**: 크로스 DB 조인 불가(애플리케이션에서 조합해야 한다), 분산 트랜잭션이 필요하다

### 어떤 걸 선택해야 하나

| 상황 | 선택 |
|------|------|
| 단일 테이블 데이터가 너무 크다 | 수평 샤딩 |
| 도메인별로 부하 특성이 다르다 | 수직 샤딩 |
| 둘 다 해당한다 | 수직 샤딩 먼저 → 필요한 도메인에 수평 샤딩 추가 |

실무에서는 수직 샤딩(마이크로서비스별 DB 분리)을 먼저 하고, 특정 서비스의 데이터가 커지면 그 서비스 안에서 수평 샤딩을 추가하는 패턴이 일반적이다.

## 샤딩 키 선택

샤딩 키는 데이터가 어떤 샤드에 들어갈지 결정한다. 한번 정하면 바꾸기 매우 어려우니 신중하게 골라야 한다.

### 좋은 샤딩 키의 조건

1. **불변**: 한번 정해지면 변하지 않는 값 (user_id는 좋고, user_status는 나쁘다)
2. **균등 분포**: 모든 샤드에 데이터가 고르게 들어가야 한다
3. **쿼리 포함 빈도가 높음**: 대부분의 쿼리 WHERE절에 들어가는 값이어야 한다
4. **고유성**: 같은 키 값의 데이터가 같은 샤드에 모여야 관련 데이터를 한 번에 읽는다

### 샤딩 방식별 특성

#### 범위 기반 (Range-based)

```
user_id 기준:
샤드 A: 1 ~ 100만
샤드 B: 100만+1 ~ 200만
샤드 C: 200만+1 ~ 300만
```

범위 검색이 빠르다. "ID가 50만~60만인 사용자"를 조회하면 샤드 A만 보면 된다.
문제는 최근 가입자가 몰리면 마지막 샤드에 쓰기가 집중되는 핫스팟이 생긴다.

#### 해시 기반 (Hash-based)

```
hash(user_id) % 4:
0 → 샤드 A
1 → 샤드 B
2 → 샤드 C
3 → 샤드 D
```

데이터가 균등하게 분산된다. 하지만 범위 검색 시 모든 샤드를 뒤져야 한다.
가장 큰 문제는 샤드 수가 바뀌면 거의 모든 데이터가 재배치된다는 점이다. 이 문제를 해결하는 것이 Consistent Hashing이다.

#### 디렉토리 기반 (Directory-based)

```
매핑 테이블:
user:1~1000   → 샤드 0
user:1001~2000 → 샤드 1
user:premium  → 전용 샤드
```

가장 유연하지만, 매핑 테이블 자체가 SPOF(Single Point of Failure)가 될 수 있다. 매핑 테이블은 반드시 캐싱하고 복제본을 두어야 한다.

### 나쁜 샤딩 키 예시

| 샤딩 키 | 문제 |
|---------|------|
| `created_at` (생성일시) | 최근 데이터가 한 샤드에 몰린다 |
| `status` (주문 상태) | 값 종류가 적어서 균등 분산이 안 된다 |
| `country_code` | 한국 사용자가 90%면 한 샤드에 데이터가 몰린다 |
| `email` | 쿼리에서 user_id로 조회하는데 email로 샤딩하면 매번 전체 샤드 검색이 필요하다 |

## Consistent Hashing

해시 기반 샤딩에서 샤드를 추가하거나 제거할 때 데이터 이동을 최소화하는 핵심 기법이다.

### 기존 해시 샤딩의 문제

`hash(key) % N`으로 샤드를 결정하면, N(샤드 수)이 바뀔 때 거의 모든 키의 위치가 바뀐다.

```
샤드 3개일 때:        샤드 4개로 변경:
hash(A) % 3 = 0     hash(A) % 4 = 1  ← 이동
hash(B) % 3 = 1     hash(B) % 4 = 2  ← 이동
hash(C) % 3 = 2     hash(C) % 4 = 0  ← 이동
hash(D) % 3 = 0     hash(D) % 4 = 3  ← 이동
hash(E) % 3 = 1     hash(E) % 4 = 1  ← 유지 (운 좋게)

→ 대부분의 데이터가 재배치된다. 데이터가 수억 건이면 마이그레이션에 며칠이 걸린다.
```

### Consistent Hashing 원리

해시 값을 원형 링(0 ~ 2^32-1)에 배치한다. 샤드도 같은 링에 배치한다. 각 키는 시계 방향으로 가장 가까운 샤드에 할당된다.

```
해시 링 (0 ~ 2^32-1):

        0
       / \
     /     \
  샤드A     샤드B
   |    key1  |
   |  key2    |
   |          |
  샤드D     샤드C
     \     /
       \ /
      2^32

key1 → 시계 방향으로 가장 가까운 샤드B에 저장
key2 → 시계 방향으로 가장 가까운 샤드A에 저장
```

**샤드 추가 시**: 새 샤드(E)가 B와 C 사이에 들어가면, B~E 구간의 키만 E로 이동한다. 나머지 키는 그대로다.

```
샤드 E 추가 시 영향 범위:
├─ 샤드A 담당 키: 변동 없음
├─ 샤드B 담당 키: 일부가 샤드E로 이동
├─ 샤드C 담당 키: 변동 없음
└─ 샤드D 담당 키: 변동 없음

→ 전체 키의 약 1/N만 이동한다 (N = 샤드 수)
```

### Virtual Node (가상 노드)

실제 샤드 수가 적으면 링 위의 분포가 불균등해진다. 각 샤드를 여러 개의 가상 노드로 매핑하면 이 문제를 해결한다.

```
물리 샤드 3개, 가상 노드 각 4개:

샤드A → 가상노드 A-0, A-1, A-2, A-3
샤드B → 가상노드 B-0, B-1, B-2, B-3
샤드C → 가상노드 C-0, C-1, C-2, C-3

→ 링 위에 12개의 노드가 골고루 분포
→ 데이터 분산이 훨씬 균등해진다
```

가상 노드 수가 많을수록 분포가 균등해지지만, 라우팅 테이블 크기가 커진다. 실무에서는 물리 샤드당 100~200개의 가상 노드를 사용하는 경우가 많다.

### Java로 Consistent Hashing 구현

```java
import java.util.SortedMap;
import java.util.TreeMap;
import java.security.MessageDigest;
import java.nio.charset.StandardCharsets;

public class ConsistentHashRouter<T> {

    private final SortedMap<Long, T> ring = new TreeMap<>();
    private final int virtualNodeCount;

    public ConsistentHashRouter(int virtualNodeCount) {
        this.virtualNodeCount = virtualNodeCount;
    }

    public void addNode(T node) {
        for (int i = 0; i < virtualNodeCount; i++) {
            long hash = hash(node.toString() + "-vnode-" + i);
            ring.put(hash, node);
        }
    }

    public void removeNode(T node) {
        for (int i = 0; i < virtualNodeCount; i++) {
            long hash = hash(node.toString() + "-vnode-" + i);
            ring.remove(hash);
        }
    }

    public T getNode(String key) {
        if (ring.isEmpty()) {
            throw new IllegalStateException("샤드가 없다");
        }
        long hash = hash(key);
        // 시계 방향으로 가장 가까운 노드
        SortedMap<Long, T> tailMap = ring.tailMap(hash);
        long targetHash = tailMap.isEmpty() ? ring.firstKey() : tailMap.firstKey();
        return ring.get(targetHash);
    }

    private long hash(String key) {
        try {
            MessageDigest md = MessageDigest.getInstance("MD5");
            byte[] digest = md.digest(key.getBytes(StandardCharsets.UTF_8));
            return ((long)(digest[0] & 0xFF) << 24)
                 | ((long)(digest[1] & 0xFF) << 16)
                 | ((long)(digest[2] & 0xFF) << 8)
                 | ((long)(digest[3] & 0xFF));
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
}
```

```java
// 사용 예
ConsistentHashRouter<String> router = new ConsistentHashRouter<>(150);
router.addNode("shard-1");
router.addNode("shard-2");
router.addNode("shard-3");

String shard = router.getNode("user:12345");
// → "shard-2" (해시 값에 따라 결정)

// 샤드 추가 시 기존 키의 대부분은 그대로
router.addNode("shard-4");
// → 전체 키의 약 25%만 이동
```

## 쿼리 라우팅

### 단일 샤드 쿼리

가장 빠르고 권장하는 패턴이다. 쿼리에 샤드 키가 포함되어 있어서 하나의 샤드만 조회한다.

```
GET /users/1234
→ 샤드 키 추출: user_id = 1234
→ 샤드 결정: hash(1234) → shard-2
→ SELECT * FROM users WHERE id = 1234 (shard-2에서만 실행)
```

### 스캐터-게더 (Scatter-Gather)

샤드 키가 없는 쿼리는 모든 샤드에 보내고 결과를 모아야 한다.

```
전체 사용자 수 조회:
→ 모든 샤드에 병렬 요청
   shard-1: SELECT COUNT(*) → 250,000
   shard-2: SELECT COUNT(*) → 240,000
   shard-3: SELECT COUNT(*) → 260,000
   shard-4: SELECT COUNT(*) → 250,000
→ 애플리케이션에서 합산: 1,000,000

주의할 점:
- AVG는 단순 평균이 아니라 가중 평균으로 계산해야 한다
- ORDER BY + LIMIT는 각 샤드에서 정렬 후 머지 소트가 필요하다
- DISTINCT는 각 샤드 결과를 합친 후 다시 중복 제거해야 한다
```

### 크로스 샤드 조인 회피

크로스 샤드 조인은 가능하면 피해야 한다. 피할 수 없을 때의 대응 방법:

**1. 같은 샤드 키로 통일**: users와 orders를 모두 user_id로 샤딩하면 특정 사용자의 주문 조회가 단일 샤드에서 가능하다.

**2. 데이터 비정규화**: 자주 함께 조회되는 데이터는 중복 저장한다.

```sql
-- orders 테이블에 user_name도 저장
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    user_name VARCHAR(100),  -- 비정규화
    amount DECIMAL(10,2)
);
-- 조인 없이 주문과 사용자 이름을 한 번에 조회 가능
```

**3. 집계 테이블**: 자주 쓰는 집계 결과를 별도 테이블에 미리 계산해둔다.

## 실제 구현

### Java/Spring 환경에서 샤드 라우터 구현

Spring에서 `AbstractRoutingDataSource`를 사용해 샤드 라우터를 구현한다.

```java
// 1. 샤드 컨텍스트 — 현재 스레드가 어떤 샤드를 사용하는지 저장
public class ShardContext {

    public static void setShard(String shardId) {
        currentShard.set(shardId);

        return currentShard.get();
    }

    public static void clear() {
        currentShard.remove();
    }
}
```

```java
// 4. 샤드 결정 로직
@Component
public class ShardResolver {

    private static final int SHARD_COUNT = 4;

    public String resolve(long userId) {
        int shardIndex = (int)(userId % SHARD_COUNT);
        return "shard-" + shardIndex;
    }
}
```

```java
// 5. 서비스에서 사용
@Service
public class UserService {

    private final UserRepository userRepository;
    private final ShardResolver shardResolver;

    public User getUser(long userId) {
        try {
            ShardContext.setShard(shardResolver.resolve(userId));
            return userRepository.findById(userId);
        } finally {
            ShardContext.clear();  // 반드시 정리
        }
    }

    // 모든 샤드에서 집계 (Scatter-Gather)
    public long countAllUsers() {
        long total = 0;
        for (int i = 0; i < 4; i++) {
            try {
                ShardContext.setShard("shard-" + i);
                total += userRepository.count();
            } finally {
                ShardContext.clear();
            }
        }
        return total;
    }
}
```

이 방식의 문제는 샤드 결정 로직이 서비스 코드에 흩어진다는 점이다. AOP나 어노테이션으로 깔끔하게 분리할 수 있다.

```java
// 어노테이션 기반 샤드 결정
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Sharded {
    String keyExpression();  // SpEL로 샤드 키 추출
}

@Aspect
@Component
public class ShardingAspect {

    private final ShardResolver shardResolver;

    @Around("@annotation(sharded)")
    public Object routeToShard(ProceedingJoinPoint pjp, Sharded sharded) throws Throwable {
        // SpEL로 메서드 파라미터에서 샤드 키 추출
        Object[] args = pjp.getArgs();
        long shardKey = extractKey(sharded.keyExpression(), args);
        try {
            ShardContext.setShard(shardResolver.resolve(shardKey));
            return pjp.proceed();
        } finally {
            ShardContext.clear();
        }
    }
}

// 사용
@Sharded(keyExpression = "#userId")
public User getUser(long userId) {
    return userRepository.findById(userId);
}
```

### ShardingSphere 설정

직접 구현하는 대신 Apache ShardingSphere를 쓰면 애플리케이션 코드 변경 없이 샤딩을 적용할 수 있다.

```yaml
# application.yml — ShardingSphere-JDBC 설정
spring:
  shardingsphere:
    datasource:
      names: ds0,ds1,ds2,ds3
      ds0:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://shard0-host:3306/mydb
        username: app_user
        password: password
      ds1:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://shard1-host:3306/mydb
        username: app_user
        password: password
      ds2:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://shard2-host:3306/mydb
        username: app_user
        password: password
      ds3:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://shard3-host:3306/mydb
        username: app_user
        password: password
    rules:
      sharding:
        tables:
          users:
            actual-data-nodes: ds$->{0..3}.users
            table-strategy:
              standard:
                sharding-column: id
                sharding-algorithm-name: user-hash
          orders:
            actual-data-nodes: ds$->{0..3}.orders
            table-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: user-hash
        sharding-algorithms:
          user-hash:
            type: HASH_MOD
            props:
              sharding-count: 4
        binding-tables:
          - users,orders  # 같은 샤드 키로 바인딩 → 조인 가능
```

ShardingSphere는 SQL을 파싱해서 샤드 키를 추출하고, 자동으로 적절한 샤드로 라우팅한다. `binding-tables`로 users와 orders를 묶으면 같은 user_id의 데이터가 같은 샤드에 있다는 것을 알고, 크로스 샤드 조인을 피한다.

**주의사항**:
- ShardingSphere가 지원하지 않는 SQL 문법이 있다 (서브쿼리 일부, 특정 함수 등)
- 스캐터-게더 쿼리는 성능이 떨어진다 — WHERE에 샤드 키를 넣는 습관이 중요하다
- DDL(ALTER TABLE 등)은 모든 샤드에 전파해야 하므로 별도 관리가 필요하다

### Vitess 개요

Vitess는 YouTube에서 만든 MySQL 샤딩 미들웨어다. Kubernetes 위에서 돌아가는 것을 기본 전제로 설계되었다.

```yaml
# Vitess VSchema 설정 — 샤딩 규칙 정의
{
  "sharded": true,
  "vindexes": {
    "user_hash": {
      "type": "hash"
    }
  },
  "tables": {
    "users": {
      "column_vindexes": [
        {
          "column": "id",
          "name": "user_hash"
        }
      ]
    },
    "orders": {
      "column_vindexes": [
        {
          "column": "user_id",
          "name": "user_hash"
        }
      ]
    }
  }
}
```

**ShardingSphere vs Vitess**:

| 항목 | ShardingSphere | Vitess |
|------|---------------|--------|
| 배포 방식 | JDBC 드라이버 또는 프록시 | 프록시 (vtgate) |
| DB 지원 | MySQL, PostgreSQL, Oracle 등 | MySQL 전용 |
| 운영 환경 | 자유 | Kubernetes 권장 |
| 리샤딩 | 수동 | 온라인 리샤딩 내장 |
| 학습 곡선 | 중간 | 높음 |
| 적합한 경우 | Spring/Java 프로젝트 | 대규모 MySQL 클러스터 |

## MySQL vs PostgreSQL 샤딩

### MySQL 샤딩

MySQL 자체에는 샤딩 기능이 없다. 애플리케이션 레벨이나 미들웨어(ShardingSphere, Vitess, ProxySQL)를 써야 한다.

```sql
-- 각 샤드에 동일한 스키마 생성
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(255) UNIQUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_email (email)
) ENGINE=InnoDB;
```

**MySQL 샤딩 시 주의할 점**:

- `AUTO_INCREMENT`를 그대로 쓰면 샤드 간 ID가 충돌한다. 각 샤드에 다른 시작값과 증가값을 설정해야 한다.

```sql
-- 샤드 0: 0, 4, 8, 12, ...
SET @@auto_increment_offset = 1;
SET @@auto_increment_increment = 4;

-- 샤드 1: 1, 5, 9, 13, ...
SET @@auto_increment_offset = 2;
SET @@auto_increment_increment = 4;
```

실무에서는 Snowflake ID나 UUID를 쓰는 것이 낫다. auto_increment 오프셋 방식은 샤드 수가 바뀌면 설정을 다 바꿔야 한다.

- `UNIQUE` 제약 조건은 각 샤드 안에서만 보장된다. 전체 샤드에 걸쳐 유니크해야 하는 값(email 등)은 별도 테이블이나 캐시에서 중복 검사를 해야 한다.

- `FOREIGN KEY`는 같은 샤드 안에서만 동작한다. 크로스 샤드 참조 무결성은 애플리케이션에서 보장해야 한다.

### PostgreSQL 샤딩

PostgreSQL은 네이티브 파티셔닝을 지원하고, Citus 확장으로 분산 샤딩을 할 수 있다.

#### 네이티브 파티셔닝 (단일 서버)

```sql
-- 선언적 파티셔닝 (PostgreSQL 10+)
CREATE TABLE orders (
    id BIGINT,
    user_id BIGINT,
    amount DECIMAL(10,2),
    created_at TIMESTAMP
) PARTITION BY HASH (user_id);

CREATE TABLE orders_p0 PARTITION OF orders
    FOR VALUES WITH (MODULUS 4, REMAINDER 0);
CREATE TABLE orders_p1 PARTITION OF orders
    FOR VALUES WITH (MODULUS 4, REMAINDER 1);
CREATE TABLE orders_p2 PARTITION OF orders
    FOR VALUES WITH (MODULUS 4, REMAINDER 2);
CREATE TABLE orders_p3 PARTITION OF orders
    FOR VALUES WITH (MODULUS 4, REMAINDER 3);
```

이건 단일 서버 안에서의 파티셔닝이다. 서버 자원 한계를 넘으려면 Citus 같은 확장이 필요하다.

#### Citus를 사용한 분산 샤딩

```sql
-- Citus 확장 활성화
CREATE EXTENSION citus;

-- 워커 노드 등록
SELECT citus_add_node('worker-1', 5432);
SELECT citus_add_node('worker-2', 5432);
SELECT citus_add_node('worker-3', 5432);

-- 분산 테이블 생성
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(255)
);
SELECT create_distributed_table('users', 'id');

-- 같은 샤드 키로 코로케이션
CREATE TABLE orders (
    id BIGINT,
    user_id BIGINT REFERENCES users(id),
    amount DECIMAL(10,2)
);
SELECT create_distributed_table('orders', 'user_id', colocate_with => 'users');
```

Citus의 `colocate_with`는 ShardingSphere의 `binding-tables`와 같은 개념이다. 같은 user_id의 users, orders 데이터가 같은 워커 노드에 있어서 조인이 로컬에서 처리된다.

### MySQL vs PostgreSQL 샤딩 비교

| 항목 | MySQL | PostgreSQL |
|------|-------|------------|
| 네이티브 샤딩 | 없음 | 없음 (파티셔닝만 있음) |
| 네이티브 파티셔닝 | RANGE, LIST, HASH, KEY | RANGE, LIST, HASH |
| 분산 확장 | Vitess, ShardingSphere | Citus, ShardingSphere |
| 외래 키 (크로스 샤드) | 불가 | Citus에서 코로케이션 시 가능 |
| 온라인 리샤딩 | Vitess 지원 | Citus 지원 |
| 분산 쿼리 최적화 | 미들웨어 의존 | Citus가 쿼리 플래너 통합 |

PostgreSQL + Citus 조합이 분산 쿼리 처리에서 더 자연스럽다. Citus는 PostgreSQL 쿼리 플래너에 통합되어 있어서, 일반 SQL을 쓰면 자동으로 분산 실행 계획을 만든다. MySQL 쪽은 미들웨어가 SQL을 파싱하는 방식이라 지원하지 않는 SQL 패턴이 더 많다.

하지만 MySQL 생태계의 Vitess는 온라인 리샤딩이 성숙해있고, 대규모 운영 사례(YouTube, Slack, GitHub)가 많다.

## 샤드 리밸런싱

데이터가 늘어나거나 특정 샤드에 부하가 몰리면 리밸런싱이 필요하다.

### Consistent Hashing 기반 리밸런싱

Consistent Hashing을 쓰면 샤드 추가 시 전체 키의 약 1/N만 이동한다.

```
현재: 샤드 4개 (A, B, C, D)
추가: 샤드 E (C와 D 사이에 배치)

이동되는 데이터:
- C~E 구간의 키만 D에서 E로 이동
- 나머지 키는 그대로

Virtual Node 150개 기준:
- 총 키의 약 20% (1/5)만 이동
- 이동 대상 키를 사전에 정확히 계산 가능
```

### 온라인 리밸런싱 절차

서비스 중단 없이 리밸런싱하는 절차다. 순서가 중요하다.

```
1. 새 샤드 준비
   ├─ 서버 프로비저닝
   ├─ 스키마 생성
   └─ 연결 테스트

2. 이중 쓰기 시작 (Dual Write)
   ├─ 이동 대상 키의 쓰기를 기존 샤드 + 새 샤드에 동시 수행
   └─ 읽기는 기존 샤드에서만

3. 기존 데이터 복사 (Backfill)
   ├─ 이동 대상 기존 데이터를 배치로 새 샤드에 복사
   ├─ 배치 크기: 1,000~10,000건씩
   └─ 피크 시간대 피해서 진행

4. 데이터 검증
   └─ 기존 샤드와 새 샤드의 데이터 일치 확인

5. 읽기 전환 (점진적)
   ├─ 1% → 10% → 50% → 100%
   ├─ 각 단계에서 에러율, 응답 시간 확인
   └─ 문제 발생 시 즉시 롤백

6. 이중 쓰기 중단
   ├─ 쓰기도 새 샤드로 완전 전환
   └─ 기존 샤드에서 이동된 데이터 삭제 (1~2주 후)
```

### 오프라인 리밸런싱

서비스를 일시 중단하고 빠르게 재배치한다. 다운타임이 허용되는 소규모 서비스나 심야 유지보수 시간에 적합하다.

```
1. 유지보수 공지
2. 서비스 중지, 진행 중인 트랜잭션 완료 대기
3. 모든 샤드 백업
4. 새 샤딩 규칙으로 데이터 재배치
5. 데이터 무결성 검증
6. 라우팅 설정 변경
7. 서비스 재시작
```

수 시간 안에 끝낼 수 있고 구현이 단순하지만, 롤백이 어렵다.

## 실무 문제와 해결

### 핫스팟

특정 샤드에 트래픽이 몰리는 현상이다.

**원인**: 샤드 키 선택이 잘못되었거나, 특정 데이터(인기 상품, 유명인 계정)에 접근이 집중된다.

**대응**:
- 해시 기반 샤딩으로 전환하거나 복합 키(user_id + timestamp)를 사용한다
- 핫 데이터를 Redis 같은 캐시에서 처리한다
- 핫스팟 샤드에 Read Replica를 추가하여 읽기 부하를 분산한다
- 핫스팟 샤드를 두 개로 분할한다

### 분산 트랜잭션

여러 샤드에 걸친 트랜잭션이 필요한 경우다.

**Saga 패턴** — 실무에서 가장 많이 쓰는 방법이다.

```
포인트 전송: 사용자 A(shard-1) → 사용자 B(shard-2)

1단계: A 포인트 차감 (shard-1)
   성공 → 2단계로
   실패 → 전체 취소

2단계: B 포인트 증가 (shard-2)
   성공 → 완료
   실패 → 보상 트랜잭션: A 포인트 복구 (shard-1)
```

Saga 패턴은 최종 일관성(Eventual Consistency)만 보장한다. 1단계와 2단계 사이에 잠깐 불일치 상태가 생길 수 있다. 은행 이체처럼 즉시 일관성이 필요하면 2PC(Two-Phase Commit)를 써야 하지만, 2PC는 느리고 코디네이터 장애 시 블로킹이 발생한다.

**실무에서의 판단 기준**:
- 결제, 잔액 → 즉시 일관성 필요 → 같은 샤드에 배치하거나 2PC
- 프로필, 통계 → 최종 일관성으로 충분 → Saga 패턴
- 조회수, 로그 → 약한 일관성도 괜찮음 → 비동기 처리

### 글로벌 유니크 ID 생성

auto_increment는 샤드 간 충돌이 생긴다. 대안이 필요하다.

**Snowflake ID** — Twitter에서 만든 방식으로 가장 널리 쓰인다.

```
64bit 구조:
┌─ 1bit (부호) ─┬─ 41bit (타임스탬프) ─┬─ 10bit (머신 ID) ─┬─ 12bit (시퀀스) ─┐
│      0        │  현재 시간 - epoch   │   샤드/노드 구분  │  같은 밀리초 내  │
└───────────────┴─────────────────────┴──────────────────┴─────────────────┘

특성:
- 시간순 정렬 가능 (타임스탬프가 상위 비트)
- 충돌 없음 (머신 ID로 구분)
- 초당 약 4,096개 생성 가능 (머신당)
```

```java
public class SnowflakeIdGenerator {
    private final long epoch = 1609459200000L; // 2021-01-01
    private final long machineId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public SnowflakeIdGenerator(long machineId) {
        this.machineId = machineId & 0x3FF; // 10bit
    }

    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();
        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & 0xFFF; // 12bit
            if (sequence == 0) {
                timestamp = waitNextMillis(timestamp);
            }
        } else {
            sequence = 0;
        }
        lastTimestamp = timestamp;
        return ((timestamp - epoch) << 22) | (machineId << 12) | sequence;
    }

    private long waitNextMillis(long current) {
        long ts = System.currentTimeMillis();
        while (ts <= current) {
            ts = System.currentTimeMillis();
        }
        return ts;
    }
}
```

## 모니터링

샤딩 시스템은 모니터링 없이 운영할 수 없다.

### 핵심 메트릭

```
샤드별 수집 항목:
├─ 쿼리 응답 시간 (P50, P95, P99)
├─ 초당 쿼리 수 (QPS)
├─ 에러율
├─ 연결 풀 사용률
├─ 데이터 크기 (행 수, 디스크 사용량)
├─ CPU / 메모리 사용률
└─ 복제 지연 (Replica가 있는 경우)

불균형 감지 기준:
- 데이터 크기 차이 > 30% → 리밸런싱 검토
- QPS 차이 > 50% → 핫스팟 의심
- 응답 시간 차이 > 2배 → 성능 이슈 조사
```

### 알림 기준

```
즉시 대응:
- 샤드 연결 불가
- 에러율 > 1%
- 응답 시간 P99 > 1초

경고:
- 디스크 사용률 > 70%
- CPU 사용률 > 80%
- 데이터 불균형 > 30%
- 복제 지연 > 10초
```

## 실제 기업 사례

### Instagram

샤딩 키로 사용자 ID를 사용한다. 사용자의 게시물, 좋아요, 댓글, 팔로우 관계를 모두 같은 샤드에 저장해서 타임라인 조회가 단일 샤드에서 끝난다. 해시 기반 샤딩으로 균등 분산한다.

### Uber

도시(지역) 기반 샤딩을 사용한다. 서울 사용자의 데이터는 서울 데이터센터 샤드에, 뉴욕 사용자는 미국 동부 샤드에 저장한다. 네트워크 지연이 줄어들고 지역별 규제도 대응 가능하다. 단, 사용자가 다른 도시로 이동하는 경우의 처리가 복잡하다.

### Slack

워크스페이스 ID를 샤드 키로 사용한다. 대기업 고객은 전용 샤드를 할당받아 성능이 보장된다. 소규모 워크스페이스는 여러 개가 하나의 샤드를 공유한다.
