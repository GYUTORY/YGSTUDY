# 쿼리 옵티마이저와 실행 계획

## 쿼리 처리 흐름

SQL 문이 실행되면 아래 순서를 거친다.

```
  클라이언트
      │
      ▼
  ┌─────────┐
  │  파서    │  SQL 문법 검증, 파스 트리 생성
  └────┬────┘
       ▼
  ┌──────────────┐
  │  전처리기     │  테이블·컬럼 존재 여부, 권한 확인
  └──────┬───────┘
         ▼
  ┌──────────────────────┐      ┌─────────────────────┐
  │   옵티마이저          │ ◀── │  통계 정보           │
  │                      │      │  (행 수, 카디널리티,  │
  │  후보 실행 계획 생성  │      │   인덱스 분포)       │
  │  비용 계산 → 최적 선택│      └─────────────────────┘
  └──────┬───────────────┘
         ▼
  ┌──────────────┐
  │  실행 엔진    │  선택된 계획대로 데이터 접근
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │  스토리지 엔진│  InnoDB / MyISAM 등 실제 I/O
  └──────────────┘
```

옵티마이저는 같은 결과를 만드는 수백 가지 실행 방법 중 비용이 가장 낮은 것을 고른다. 비용 = 디스크 I/O + CPU 연산 + 메모리 사용의 가중합이다. 통계 정보가 오래되면 비용 산정이 틀어져 잘못된 계획이 선택된다.

---

## EXPLAIN 기본 사용법

```sql
-- 실행 계획 확인
EXPLAIN SELECT * FROM orders WHERE user_id = 1;

-- 세로 출력 — 컬럼이 많을 때 읽기 편하다
EXPLAIN SELECT * FROM orders WHERE user_id = 1\G

-- 실제 실행 후 실측 통계 확인 (MySQL 8.0.18+)
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 1;
```

`EXPLAIN`은 쿼리를 실행하지 않고 계획만 보여준다. `EXPLAIN ANALYZE`는 실제로 실행해서 예측 rows와 실측 rows를 비교할 수 있다. 운영 DB에서 `EXPLAIN ANALYZE`를 돌릴 때는 무거운 쿼리가 실제로 실행된다는 점에 주의해야 한다.

## EXPLAIN 출력 컬럼 해석

아래 쿼리의 실행 계획을 예로 든다.

```sql
EXPLAIN
SELECT o.id, o.total_amount, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'COMPLETED'
  AND o.created_at >= '2024-01-01'
ORDER BY o.created_at DESC
LIMIT 20;
```

```
+----+-------------+-------+--------+----------------------------+--------------------+---------+-----------------+------+----------+--------------------------------------+
| id | select_type | table | type   | possible_keys              | key                | key_len | ref             | rows | filtered | Extra                                |
+----+-------------+-------+--------+----------------------------+--------------------+---------+-----------------+------+----------+--------------------------------------+
|  1 | SIMPLE      | o     | range  | idx_status_created,idx_user| idx_status_created | 10      | NULL            | 5000 |   100.00 | Using index condition; Using filesort|
|  1 | SIMPLE      | u     | eq_ref | PRIMARY                    | PRIMARY            | 8       | mydb.o.user_id  |    1 |   100.00 | NULL                                 |
+----+-------------+-------+--------+----------------------------+--------------------+---------+-----------------+------+----------+--------------------------------------+
```

각 행이 하나의 테이블 접근을 나타낸다. 위에서 아래로 읽으면 실행 순서가 된다.

### type — 접근 방식

옵티마이저가 테이블에서 데이터를 어떻게 읽는지 보여준다. 위에서 아래로 갈수록 비용이 높다.

```
  성능 좋음
    │
    │  system ──── 행이 1개뿐인 테이블
    │  const  ──── PK나 UNIQUE 인덱스로 상수 조건 1행 조회
    │  eq_ref ──── JOIN 시 PK/UNIQUE로 정확히 1행 매칭
    │  ref    ──── 인덱스로 여러 행 조회 (등가 조건)
    │  range  ──── 인덱스 범위 스캔 (BETWEEN, >, <, IN)
    │  index  ──── 인덱스 전체를 처음부터 끝까지 스캔
    │  ALL    ──── 테이블 풀 스캔 — 인덱스 미사용
    │
  성능 나쁨
```

실무에서 가장 흔한 조합:

- **const + eq_ref**: PK 기반 단건 조회와 JOIN. 성능 문제가 거의 없다.
- **ref**: 일반 인덱스로 조회. 카디널리티가 적절하면 문제없다.
- **range**: 날짜 범위, 가격대 등. 범위가 너무 넓으면 풀 스캔보다 느릴 수 있다.
- **ALL**: 반드시 개선해야 하는 건 아니다. 수백 행짜리 작은 테이블이면 풀 스캔이 빠르다.

### key / possible_keys

```
possible_keys: idx_status_created, idx_user
key:           idx_status_created
```

`possible_keys`는 옵티마이저가 사용을 검토한 인덱스 목록이고, `key`는 실제 선택된 인덱스다. `key`가 NULL이면 인덱스를 쓰지 않았다는 뜻이다. possible_keys에 인덱스가 있는데 key가 NULL인 경우는 옵티마이저가 풀 스캔이 더 싸다고 판단한 것이므로, 인덱스 구조나 통계 정보를 점검해야 한다.

### rows

옵티마이저가 **예측하는** 검사할 행 수다. 결과 행 수가 아니다. `EXPLAIN ANALYZE`로 실측값과 비교하면 통계 정보의 정확도를 알 수 있다. 예측과 실측이 10배 이상 차이나면 `ANALYZE TABLE`로 통계를 갱신해야 한다.

### filtered

이전 테이블에서 넘어온 행 중 이 테이블의 조건을 통과하는 비율(%)이다. 100.00이면 조건 필터링 없이 전부 사용한다는 뜻이고, 낮으면 많은 행을 읽고 버린다는 뜻이다.

### Extra — 자주 보이는 값

| Extra 값 | 의미 | 개선 필요 여부 |
|---------|------|-------------|
| `Using index` | 커버링 인덱스 — 테이블 접근 없이 인덱스만으로 결과 반환 | 가장 빠른 형태 |
| `Using where` | 스토리지 엔진에서 읽은 후 서버 레이어에서 추가 필터링 | 보통 정상 |
| `Using index condition` | ICP(Index Condition Pushdown) — 인덱스 레벨에서 조건 필터링 | 좋은 상태 |
| `Using filesort` | 인덱스 순서가 아닌 별도 정렬 수행 | 대량 데이터면 개선 필요 |
| `Using temporary` | 임시 테이블 생성 (GROUP BY, DISTINCT, UNION 등) | 대량 데이터면 개선 필요 |
| `Using join buffer` | JOIN 시 인덱스 없이 Block Nested Loop 사용 | JOIN 컬럼에 인덱스 필요 |

## EXPLAIN 패턴별 대응법

실무에서 자주 마주치는 문제 패턴과 해결 방법을 정리한다.

### 패턴 1: Using temporary + Using filesort 동시 발생

GROUP BY와 ORDER BY가 다른 컬럼이거나, GROUP BY 컬럼에 인덱스가 없으면 발생한다.

```sql
-- 문제 쿼리
EXPLAIN
SELECT status, COUNT(*) AS cnt
FROM orders
WHERE created_at >= '2024-01-01'
GROUP BY status
ORDER BY cnt DESC;
```

```
type: range
key: idx_created_at
Extra: Using index condition; Using temporary; Using filesort
```

```
  실행 흐름:

  idx_created_at 범위 스캔
       │
       ▼
  WHERE 필터링으로 행 추출
       │
       ▼
  ┌──────────────────────┐
  │  임시 테이블 생성     │  ← Using temporary
  │  status별 COUNT 집계  │
  └──────┬───────────────┘
         ▼
  ┌──────────────────────┐
  │  cnt DESC 정렬       │  ← Using filesort
  └──────┬───────────────┘
         ▼
       결과 반환
```

**대응**: GROUP BY 기준 컬럼이 인덱스 선두에 있으면 임시 테이블 없이 처리된다. 다만 이 경우처럼 `ORDER BY cnt DESC`(집계 결과 기반 정렬)는 인덱스로 해결할 수 없으므로 filesort는 남는다. 중요한 건 임시 테이블의 크기를 줄이는 것이다.

```sql
-- status의 카디널리티가 낮으면 (3~5가지) 임시 테이블이 작아 문제 없다.
-- 카디널리티가 높은 컬럼으로 GROUP BY 한다면:
CREATE INDEX idx_orders_status_created ON orders (status, created_at);

-- status가 인덱스 선두 → GROUP BY에서 임시 테이블 불필요
-- created_at 범위 조건도 같은 인덱스에서 처리
```

### 패턴 2: 서브쿼리가 풀리지 않는 경우 (DEPENDENT SUBQUERY)

옵티마이저는 가능하면 서브쿼리를 JOIN으로 변환(서브쿼리 풀림, subquery unnesting)한다. 변환에 실패하면 `select_type`이 `DEPENDENT SUBQUERY`가 되고, 외부 쿼리의 행마다 서브쿼리가 반복 실행된다.

```sql
-- 문제 쿼리: 상관 서브쿼리
EXPLAIN
SELECT *
FROM orders o
WHERE o.total_amount > (
    SELECT AVG(total_amount)
    FROM orders
    WHERE user_id = o.user_id
);
```

```
+----+--------------------+--------+------+---------------+----------+------+------+
| id | select_type        | table  | type | possible_keys | key      | rows | Extra|
+----+--------------------+--------+------+---------------+----------+------+------+
|  1 | PRIMARY            | o      | ALL  | NULL          | NULL     | 500K | ...  |
|  2 | DEPENDENT SUBQUERY | orders | ref  | idx_user_id   | idx_user | 50   | ...  |
+----+--------------------+--------+------+---------------+----------+------+------+
```

외부 쿼리가 50만 행이면 서브쿼리가 50만 번 실행된다. idx_user_id로 매번 50행씩 읽는다 해도 총 2500만 행을 스캔하게 된다.

**대응**: 직접 JOIN으로 바꾸거나, 서브쿼리를 파생 테이블(derived table)로 분리한다.

```sql
-- 방법 1: JOIN + 파생 테이블
SELECT o.*
FROM orders o
JOIN (
    SELECT user_id, AVG(total_amount) AS avg_amount
    FROM orders
    GROUP BY user_id
) avg_t ON o.user_id = avg_t.user_id
WHERE o.total_amount > avg_t.avg_amount;

-- 방법 2: MySQL 8.0+ 윈도우 함수
SELECT *
FROM (
    SELECT *,
           AVG(total_amount) OVER (PARTITION BY user_id) AS avg_amount
    FROM orders
) t
WHERE total_amount > avg_amount;
```

변환 후 EXPLAIN에서 `DEPENDENT SUBQUERY`가 사라지고 `DERIVED` + `SIMPLE`로 바뀌면 성공이다.

### 패턴 3: Derived Table 물리화 (Materialization)

파생 테이블(FROM 절의 서브쿼리)을 옵티마이저가 병합(merge)하지 못하면 임시 테이블로 물리화한다. `select_type`이 `DERIVED`로 나온다.

```sql
EXPLAIN
SELECT d.user_id, d.order_count, u.name
FROM (
    SELECT user_id, COUNT(*) AS order_count
    FROM orders
    GROUP BY user_id
    HAVING COUNT(*) > 10
) d
JOIN users u ON d.user_id = u.id;
```

```
+----+-------------+------------+-------+...+------+------------------------------+
| id | select_type | table      | type  |...| rows | Extra                        |
+----+-------------+------------+-------+...+------+------------------------------+
|  1 | PRIMARY     | <derived2> | ALL   |...| 1000 | NULL                         |
|  1 | PRIMARY     | u          | eq_ref|...|    1 | NULL                         |
|  2 | DERIVED     | orders     | index |...| 500K | Using index                  |
+----+-------------+------------+-------+...+------+------------------------------+
```

```
  실행 흐름:

  orders 테이블 인덱스 풀 스캔 (50만 행)
       │
       ▼
  ┌──────────────────────────────────┐
  │  물리화: 임시 테이블 <derived2>   │
  │  user_id별 COUNT → HAVING 필터   │
  │  결과: ~1000행                    │
  └──────────────┬───────────────────┘
                 ▼
  <derived2> 풀 스캔 (1000행)
       │
       ▼
  users PK로 eq_ref JOIN
       │
       ▼
     결과 반환
```

GROUP BY + HAVING이 있으면 병합이 불가능하므로 물리화는 불가피하다. 이 경우 문제는 물리화 자체가 아니라 **물리화된 임시 테이블의 크기와 접근 방식**이다.

**대응**:

- `<derived2>`의 rows가 작으면(수천 행 이하) 성능 영향이 미미하다.
- 파생 테이블 결과가 크고, 외부에서 특정 조건으로 필터링한다면 MySQL 8.0+에서 파생 테이블에 자동 인덱스가 생성되는지 확인한다.
- GROUP BY 없이 단순 필터만 하는 파생 테이블이 물리화되면 `NO_MERGE` 힌트를 빼거나 쿼리를 단순화해서 병합을 유도한다.

```sql
-- 병합 가능한 형태로 변환 (GROUP BY 제거가 가능한 경우)
-- 예: 단순 필터만 있는 파생 테이블은 병합 가능
SELECT /*+ MERGE(d) */ *
FROM (SELECT * FROM orders WHERE status = 'COMPLETED') d
WHERE d.created_at >= '2024-01-01';

-- 위 쿼리는 병합되면 아래와 동일하게 실행된다:
SELECT * FROM orders
WHERE status = 'COMPLETED' AND created_at >= '2024-01-01';
```

### 패턴 4: 인덱스를 탔는데 느린 경우

type이 `ref`나 `range`로 잡혀서 인덱스를 사용하지만, rows가 전체 행의 30% 이상이면 풀 스캔보다 느릴 수 있다. 인덱스 범위 스캔은 인덱스 → 테이블 → 인덱스 → 테이블로 랜덤 I/O가 발생하기 때문이다.

```sql
-- status = 'ACTIVE'인 행이 전체의 80%인 경우
EXPLAIN
SELECT * FROM orders WHERE status = 'ACTIVE';
```

```
type: ref
key: idx_status
rows: 800000   ← 전체 100만 중 80만
```

옵티마이저가 풀 스캔을 선택하지 않은 것 자체가 통계가 틀린 신호다.

**대응**:

```sql
-- 1. 통계 갱신
ANALYZE TABLE orders;

-- 2. 갱신 후에도 잘못된 인덱스를 타면 힌트로 풀 스캔 유도
SELECT /*+ NO_INDEX(orders idx_status) */ *
FROM orders WHERE status = 'ACTIVE';

-- 3. 근본적으로는 카디널리티가 낮은 단일 컬럼 인덱스보다
--    다른 조건과 조합한 복합 인덱스가 낫다
CREATE INDEX idx_status_created ON orders (status, created_at);
-- status = 'ACTIVE' AND created_at >= '2024-01-01' 같이 범위를 좁히는 조건과 함께 사용
```

### 패턴 5: JOIN에서 Using join buffer (Block Nested Loop)

JOIN 대상 테이블의 연결 컬럼에 인덱스가 없으면 발생한다. MySQL 8.0.18+에서는 Hash Join으로 대체되지만, 그 이전 버전에서는 Block Nested Loop로 매우 느려진다.

```sql
EXPLAIN
SELECT o.*, p.name
FROM orders o
JOIN products p ON o.product_code = p.code;
-- products.code에 인덱스가 없는 경우
```

```
+----+--------+----------+------+...+------+-----------------------------------------------+
| id | table  | type     |...   |...| rows | Extra                                         |
+----+--------+----------+------+...+------+-----------------------------------------------+
|  1 | o      | ALL      |...   |...| 500K | NULL                                          |
|  1 | p      | ALL      |...   |...| 10K  | Using where; Using join buffer (hash join)    |
+----+--------+----------+------+...+------+-----------------------------------------------+
```

**대응**:

```sql
-- JOIN 대상 컬럼에 인덱스 생성
CREATE INDEX idx_products_code ON products (code);

-- 변경 후
-- p.type: eq_ref, key: idx_products_code, Extra: NULL
```

## 인덱스 추가 전후 비교

인덱스를 추가했을 때 실행 계획이 어떻게 바뀌는지 구체적으로 본다.

### 사례: 주문 조회 쿼리

```sql
SELECT id, total_amount
FROM orders
WHERE status = 'COMPLETED'
  AND created_at >= '2024-01-01'
ORDER BY created_at DESC
LIMIT 20;
```

**인덱스 없음:**

```
type:  ALL
rows:  1,000,000
Extra: Using where; Using filesort

  테이블 풀 스캔 (100만 행)
       │
       ▼
  WHERE 조건 필터링 → 5만 행 남음
       │
       ▼
  5만 행 정렬 (filesort)
       │
       ▼
  상위 20개 반환
```

**단일 컬럼 인덱스** `(status)`:

```sql
CREATE INDEX idx_status ON orders (status);
```

```
type:  ref
rows:  300,000
Extra: Using where; Using filesort

  idx_status로 'COMPLETED' 조회 (30만 행)
       │
       ▼
  created_at >= '2024-01-01' 필터링 → 5만 행
       │
       ▼
  5만 행 정렬 (filesort 여전히 발생)
       │
       ▼
  상위 20개 반환
```

**복합 인덱스** `(status, created_at)`:

```sql
CREATE INDEX idx_status_created ON orders (status, created_at);
```

```
type:  range
rows:  50,000
Extra: Using index condition

  idx_status_created에서
  status='COMPLETED' AND created_at>='2024-01-01' 범위 스캔
       │
       ▼
  인덱스 순서가 created_at ASC이므로
  역순으로 읽으면 ORDER BY DESC 처리 완료
       │
       ▼
  20개 읽으면 즉시 중단 (LIMIT)
```

세 번째 경우, 인덱스가 정렬 순서까지 커버하므로 filesort가 사라지고 LIMIT에 의해 실제 읽는 행도 20개에 가깝다. rows가 50,000으로 나오는 건 범위 전체의 예측값이고, LIMIT 덕분에 실제 스캔은 훨씬 적다.

## 통계 정보 관리

옵티마이저는 통계 정보를 기반으로 비용을 계산한다. 통계가 오래되면 잘못된 계획을 선택한다.

```sql
-- 테이블 통계 갱신
ANALYZE TABLE orders;

-- 현재 통계 확인
SELECT
    table_name,
    table_rows,
    data_length,
    index_length,
    update_time
FROM information_schema.tables
WHERE table_schema = 'mydb' AND table_name = 'orders';

-- 인덱스별 카디널리티 확인
SHOW INDEX FROM orders;
```

통계 갱신이 필요한 시점:

- 대량 INSERT/DELETE 이후 (`table_rows` 예측이 크게 빗나갈 때)
- `EXPLAIN`의 rows와 `EXPLAIN ANALYZE`의 actual rows가 10배 이상 차이날 때
- `OPTIMIZE TABLE` 실행 후 (테이블 재구성 시 통계도 초기화됨)

InnoDB는 기본적으로 테이블을 열 때 통계를 샘플링한다. `innodb_stats_persistent = ON`(MySQL 5.6+ 기본값)이면 `ANALYZE TABLE` 결과가 디스크에 저장되어 재시작 후에도 유지된다.

## 옵티마이저 힌트

옵티마이저 판단이 틀렸을 때 힌트로 실행 계획을 강제할 수 있다. 힌트는 임시 조치이며, 통계 갱신이나 인덱스 변경이 근본 해결책이다.

```sql
-- 특정 인덱스 강제 사용 (MySQL 전통 문법)
SELECT * FROM orders
FORCE INDEX (idx_status_created)
WHERE status = 'COMPLETED';

-- MySQL 8.0+ 힌트 문법 (주석 안에 작성)
SELECT /*+ INDEX(orders idx_status_created) */ *
FROM orders
WHERE status = 'COMPLETED';

-- JOIN 순서 고정
SELECT /*+ JOIN_ORDER(o, u) */ o.*, u.name
FROM orders o
JOIN users u ON o.user_id = u.id;

-- 서브쿼리 병합 방지 (파생 테이블 물리화 강제)
SELECT /*+ NO_MERGE(sub) */ *
FROM (SELECT * FROM orders WHERE status = 'COMPLETED') sub;

-- 서브쿼리를 세미조인으로 처리하지 않도록
SELECT /*+ NO_SEMIJOIN(@subq1) */ *
FROM orders
WHERE user_id IN (SELECT /*+ QB_NAME(subq1) */ id FROM users WHERE grade = 'VIP');
```

`FORCE INDEX`와 `/*+ INDEX() */`의 차이: FORCE INDEX는 옵티마이저가 풀 스캔보다 비용이 높다고 판단해도 해당 인덱스를 강제한다. `/*+ INDEX() */`는 비용이 너무 높으면 무시될 수 있다.

## 실무 팁

### EXPLAIN ANALYZE 읽는 법 (MySQL 8.0.18+)

```sql
EXPLAIN ANALYZE
SELECT o.id, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'COMPLETED'
LIMIT 10\G
```

```
-> Limit: 10 row(s)  (cost=2.35 rows=10) (actual time=0.15..0.28 rows=10 loops=1)
    -> Nested loop inner join  (cost=2.35 rows=10) (actual time=0.15..0.27 rows=10 loops=1)
        -> Index lookup on o using idx_status (status='COMPLETED')  (cost=1.10 rows=10) (actual time=0.12..0.15 rows=10 loops=1)
        -> Single-row index lookup on u using PRIMARY (id=o.user_id)  (cost=0.12 rows=1) (actual time=0.01..0.01 rows=1 loops=10)
```

안에서 밖으로 읽는다. 가장 안쪽이 먼저 실행되는 연산이다.

- **cost**: 옵티마이저 예측 비용
- **rows**: 예측 행 수
- **actual time**: 첫 행 반환 시간..마지막 행 반환 시간 (밀리초)
- **loops**: 해당 연산이 반복 실행된 횟수

`rows=10` 예측에 `actual rows=10`이면 통계가 정확하다는 뜻이다. `loops=10`은 외부 테이블에서 10행이 넘어와 10번 반복했다는 의미다.

### 인덱스가 무시되는 흔한 실수

```sql
-- 1. 인덱스 컬럼에 함수 적용
WHERE YEAR(created_at) = 2024
-- → WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'

-- 2. 묵시적 타입 변환 (user_id가 INT인데 문자열로 비교)
WHERE user_id = '123'
-- → WHERE user_id = 123

-- 3. LIKE 앞쪽 와일드카드
WHERE name LIKE '%kim%'
-- → Full-Text Index 사용 또는 별도 검색 엔진 도입

-- 4. 복합 인덱스 선두 컬럼 누락
-- 인덱스: (status, created_at)
WHERE created_at >= '2024-01-01'
-- status 조건 없이 created_at만 쓰면 이 인덱스를 타지 않는다

-- 5. OR 조건
WHERE status = 'COMPLETED' OR user_id = 1
-- 각각의 인덱스를 사용할 수 있지만 index_merge가 발생하며
-- 두 인덱스 결과를 합치는 비용이 추가된다
-- → UNION ALL로 분리하는 것이 나을 수 있다
```
