# 쿼리 옵티마이저와 실행 계획

## 개요

쿼리 옵티마이저는 SQL을 받아 **가장 효율적인 실행 계획을 선택**하는 데이터베이스 핵심 컴포넌트다. 같은 결과를 반환하더라도 실행 방법은 수백 가지가 될 수 있으며, 옵티마이저는 비용 기반으로 최적 경로를 선택한다.

```
쿼리 처리 흐름:
  SQL → 파서 → 전처리기 → 옵티마이저 → 실행 계획 → 스토리지 엔진
                                  ↑
                          통계 정보 활용
                          (인덱스, 행 수, 카디널리티)
```

---

## EXPLAIN 사용법

### 기본

```sql
-- 실행 계획 확인
EXPLAIN SELECT * FROM orders WHERE user_id = 1;

-- 더 상세한 정보 (파티션, 필터링 조건)
EXPLAIN SELECT * FROM orders WHERE user_id = 1\G

-- 실제 실행 후 통계 확인 (MySQL 8.0+)
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 1;
```

## EXPLAIN 출력 읽는 법

```sql
EXPLAIN
SELECT o.id, o.total_amount, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'COMPLETED'
  AND o.created_at >= '2024-01-01'
ORDER BY o.created_at DESC
LIMIT 20;
```

```
+----+-------------+-------+------------+--------+----------------------------+---------+---------+-------------------+------+----------+---------------------------------+
| id | select_type | table | partitions | type   | possible_keys              | key     | key_len | ref               | rows | filtered | Extra                           |
+----+-------------+-------+------------+--------+----------------------------+---------+---------+-------------------+------+----------+---------------------------------+
|  1 | SIMPLE      | o     | NULL       | range  | idx_status_created,idx_user | idx_status_created | 10 | NULL          | 5000 |   100.00 | Using index condition; Using filesort |
|  1 | SIMPLE      | u     | NULL       | eq_ref | PRIMARY                    | PRIMARY | 8       | mydb.o.user_id   |    1 |   100.00 | NULL                            |
+----+-------------+-------+------------+--------+----------------------------+---------+---------+-------------------+------+----------+---------------------------------+
```

### 핵심 컬럼 설명

#### type (접근 방식) — 성능 순서

| type | 설명 | 성능 |
|------|------|------|
| `system` | 행이 1개인 테이블 | 최고 |
| `const` | PK/Unique로 1행 조회 | 최고 |
| `eq_ref` | JOIN에서 PK/Unique 1행 | 매우 좋음 |
| `ref` | 인덱스로 여러 행 조회 | 좋음 |
| `range` | 인덱스 범위 스캔 | 보통 |
| `index` | 인덱스 풀 스캔 | 나쁨 |
| `ALL` | 테이블 풀 스캔 | 최악 |

```sql
-- const: PK로 단일 행 조회
EXPLAIN SELECT * FROM users WHERE id = 1;
-- type: const ✅

-- ref: 일반 인덱스로 조회
EXPLAIN SELECT * FROM orders WHERE user_id = 1;
-- type: ref ✅

-- ALL: 인덱스 없는 컬럼으로 조회
EXPLAIN SELECT * FROM orders WHERE memo = 'test';
-- type: ALL ❌ → 인덱스 필요
```

#### key / possible_keys

```sql
-- possible_keys: 사용 가능한 인덱스 목록
-- key: 실제 선택된 인덱스 (NULL이면 풀스캔)

-- key가 NULL이면 인덱스 미사용 → 개선 필요
```

#### rows

옵티마이저가 예측하는 검사할 행 수. 실제 결과 행 수가 아니라 **스캔 비용**이다. 값이 클수록 쿼리가 무겁다.

#### Extra

| Extra 값 | 의미 | 조치 |
|---------|------|------|
| `Using index` | 커버링 인덱스 사용 | 최적 ✅ |
| `Using where` | WHERE 조건 필터링 | 보통 |
| `Using filesort` | 정렬을 메모리/디스크로 처리 | 인덱스로 해결 가능 ⚠️ |
| `Using temporary` | 임시 테이블 생성 | 성능 저하 ⚠️ |
| `Using index condition` | ICP(Index Condition Pushdown) 사용 | 좋음 ✅ |

## 실행 계획 개선 예시

### 인덱스 추가 전후 비교

```sql
-- 문제 쿼리: status + created_at 필터 + ORDER BY
SELECT id, total_amount
FROM orders
WHERE status = 'COMPLETED'
  AND created_at >= '2024-01-01'
ORDER BY created_at DESC
LIMIT 20;
```

**인덱스 없는 경우:**
```
type: ALL
rows: 1,000,000
Extra: Using where; Using filesort
```

**복합 인덱스 추가 후:**
```sql
CREATE INDEX idx_orders_status_created ON orders (status, created_at);
```
```
type: range
rows: 5,000
Extra: Using index condition
```

### N+1 쿼리 탐지

```sql
-- 나쁜 패턴: 루프에서 쿼리 반복
-- Java/JPA에서 아래 같은 N+1이 발생하는지 EXPLAIN으로 확인
SELECT * FROM users LIMIT 100;
-- 이후 users 수만큼 반복:
SELECT * FROM orders WHERE user_id = ?;

-- 해결: JOIN으로 한 번에
EXPLAIN
SELECT u.*, o.id AS order_id, o.total_amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
LIMIT 100;
```

### GROUP BY + ORDER BY 최적화

```sql
-- Using temporary + Using filesort 발생하는 경우
EXPLAIN
SELECT user_id, COUNT(*) AS cnt
FROM orders
GROUP BY user_id
ORDER BY cnt DESC;
-- Extra: Using temporary; Using filesort ❌

-- 인덱스로 GROUP BY 최적화
CREATE INDEX idx_orders_user ON orders (user_id);

EXPLAIN
SELECT user_id, COUNT(*) AS cnt
FROM orders
GROUP BY user_id
ORDER BY cnt DESC;
-- Extra: NULL (인덱스 기반 GROUP BY) + Using filesort (cnt 정렬은 인덱스 불가)
```

## 통계 정보와 옵티마이저 힌트

### 통계 정보 갱신

```sql
-- 테이블 통계 수동 갱신 (오래된 통계로 잘못된 계획 선택 시)
ANALYZE TABLE orders;

-- 통계 확인
SELECT
    table_name,
    table_rows,
    data_length,
    index_length,
    update_time
FROM information_schema.tables
WHERE table_schema = 'mydb' AND table_name = 'orders';
```

### 옵티마이저 힌트

```sql
-- 특정 인덱스 강제 사용
SELECT * FROM orders
FORCE INDEX (idx_orders_status_created)
WHERE status = 'COMPLETED';

-- 인덱스 사용 금지
SELECT * FROM orders
IGNORE INDEX (idx_old_unused)
WHERE user_id = 1;

-- JOIN 순서 고정
SELECT /*+ STRAIGHT_JOIN */ o.*, u.name
FROM orders o
STRAIGHT_JOIN users u ON o.user_id = u.id;

-- MySQL 8.0+ 옵티마이저 힌트
SELECT /*+ INDEX(orders idx_orders_status_created) */ *
FROM orders
WHERE status = 'COMPLETED';

SELECT /*+ NO_MERGE(subquery) */ *
FROM (SELECT * FROM orders WHERE status = 'COMPLETED') subquery;
```

## 슬로우 쿼리 탐지 및 분석

```sql
-- 슬로우 쿼리 로그 활성화 (MySQL)
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 1초 이상 쿼리 기록
SET GLOBAL log_queries_not_using_indexes = ON;

-- 현재 실행 중인 쿼리 확인
SHOW PROCESSLIST;

-- 상세 정보
SELECT * FROM information_schema.processlist
WHERE command != 'Sleep'
ORDER BY time DESC;

-- Performance Schema — 쿼리별 누적 통계
SELECT
    digest_text,
    count_star AS exec_count,
    avg_timer_wait / 1e9 AS avg_ms,
    sum_rows_examined / count_star AS avg_rows_scanned
FROM performance_schema.events_statements_summary_by_digest
ORDER BY avg_timer_wait DESC
LIMIT 10;
```

## 인덱스 선택 기준 요약

```
인덱스 추가가 효과적인 경우:
  ✅ WHERE 조건에 자주 등장하는 컬럼
  ✅ JOIN ON 절에 사용되는 컬럼
  ✅ ORDER BY / GROUP BY 컬럼
  ✅ 카디널리티(구분값 수)가 높은 컬럼

인덱스가 도움 안 되는 경우:
  ❌ 카디널리티가 낮은 컬럼 (성별, 상태 2~3가지)
  ❌ LIKE '%keyword%' (앞에 % 있으면 인덱스 미사용)
  ❌ 인덱스 컬럼에 함수 적용: WHERE YEAR(created_at) = 2024
     → 대신: WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'
  ❌ 묵시적 타입 변환: WHERE user_id = '123' (user_id가 INT인 경우)
```
