# 데이터베이스 성능 튜닝

## 성능 문제는 어디서 시작되는가

운영 중인 서비스에서 "느리다"는 신고가 들어오면, 대부분 원인은 다음 중 하나다.

- 인덱스 없이 풀 테이블 스캔하는 쿼리
- 데이터가 수백만 건으로 늘었는데 초기 설계 그대로 운영
- 커넥션 풀이 모자라서 쿼리가 큐에 쌓임
- DB 서버 메모리 설정이 기본값 그대로

"쿼리가 느리다"고 바로 인덱스를 추가하면 안 된다. 먼저 어디가 병목인지 확인하는 게 순서다.

---

## 병목 진단 프로세스

### 1단계: Slow Query Log 활성화

MySQL에서 느린 쿼리를 잡으려면 먼저 로그를 켜야 한다.

```sql
-- MySQL slow query log 설정
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;        -- 1초 이상 쿼리 기록
SET GLOBAL log_queries_not_using_indexes = 'ON';  -- 인덱스 미사용 쿼리도 기록
```

PostgreSQL은 `postgresql.conf`에서 설정한다.

```ini
log_min_duration_statement = 1000    # 1초(ms) 이상 쿼리 기록
log_statement = 'none'               # 전체 로깅은 끄고 느린 쿼리만
```

### 2단계: 문제 쿼리 추출

MySQL의 `mysqldumpslow`로 느린 쿼리를 정렬해서 볼 수 있다.

```bash
# 실행 시간 기준 상위 10개
mysqldumpslow -s t -t 10 /var/log/mysql/slow-query.log

# 빈도 기준 상위 10개 — 1초짜리 쿼리가 분당 1000번 실행되면 그게 더 문제다
mysqldumpslow -s c -t 10 /var/log/mysql/slow-query.log
```

Percona의 `pt-query-digest`가 더 상세한 분석을 제공한다.

```bash
pt-query-digest /var/log/mysql/slow-query.log --limit=10
```

출력에서 봐야 할 것:

```
# Query 1: 245 QPS, 12.3x concurrency, ID 0xABC123
# Scores: V/M = 0.01
# Time range: 2026-03-27 00:00:00 to 2026-03-28 00:00:00
# Attribute    pct   total     min     max     avg     95%   stddev
# ============ === ======= ======= ======= ======= ======= =======
# Count         35   21000
# Exec time     68   1200s    10ms   850ms    57ms   120ms    45ms
# Rows sent      2   42000       2       2       2       2       0
# Rows examine  45   84.0M    4000    4200    4100    4100       50
```

`Rows examine`가 `Rows sent`보다 수천 배 크다면, 테이블을 불필요하게 많이 훑고 있다는 뜻이다. 이 쿼리가 인덱스 개선 대상이다.

PostgreSQL에서는 `pg_stat_statements`를 활용한다.

```sql
-- pg_stat_statements 확장 활성화 (postgresql.conf)
-- shared_preload_libraries = 'pg_stat_statements'

SELECT query,
       calls,
       total_exec_time / calls AS avg_time_ms,
       rows / calls AS avg_rows,
       shared_blks_hit + shared_blks_read AS total_blocks
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
```

## EXPLAIN / EXPLAIN ANALYZE 실행 계획 해석

### MySQL EXPLAIN 출력 읽기

```sql
EXPLAIN SELECT o.id, o.amount, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'shipped'
  AND o.order_date >= '2026-01-01'
ORDER BY o.order_date DESC
LIMIT 20;
```

출력 예시:

```
+----+-------------+-------+------+-------------------+-----------+---------+-------+--------+-----------------------------+
| id | select_type | table | type | possible_keys     | key       | key_len | ref   | rows   | Extra                       |
+----+-------------+-------+------+-------------------+-----------+---------+-------+--------+-----------------------------+
|  1 | SIMPLE      | o     | ref  | idx_status,idx_dt | idx_status| 62      | const |  15420 | Using where; Using filesort |
|  1 | SIMPLE      | c     | eq_ref| PRIMARY          | PRIMARY   | 4       | o.cid |      1 | NULL                        |
+----+-------------+-------+------+-------------------+-----------+---------+-------+--------+-----------------------------+
```

각 컬럼의 의미와 판단 기준:

**type (접근 방식)** — 성능에 가장 큰 영향을 미친다.

| type | 의미 | 판단 |
|------|------|------|
| `system`, `const` | PK나 UNIQUE로 1건 조회 | 가장 빠름. 손댈 필요 없음 |
| `eq_ref` | JOIN에서 PK/UNIQUE로 1건씩 매칭 | 정상 |
| `ref` | 비유니크 인덱스로 여러 건 조회 | 보통. rows가 크면 복합 인덱스 검토 |
| `range` | 인덱스 범위 스캔 (BETWEEN, >, <) | 범위가 넓으면 rows가 커짐 |
| `index` | 인덱스 풀 스캔 | 인덱스 전체를 읽음. 테이블 스캔보다는 낫지만 개선 여지 있음 |
| `ALL` | 풀 테이블 스캔 | 수만 건 이상이면 반드시 개선해야 함 |

**rows** — MySQL이 예상하는 검사 행 수. 실제와 다를 수 있다. `ANALYZE TABLE`을 실행하면 통계가 갱신되어 정확도가 올라간다.

**Extra** — 여기서 문제를 찾는 경우가 많다.

| Extra | 의미 | 조치 |
|-------|------|------|
| `Using filesort` | ORDER BY를 인덱스로 처리 못해서 별도 정렬 | 인덱스에 정렬 컬럼 추가 검토 |
| `Using temporary` | 임시 테이블 생성 (GROUP BY, DISTINCT 등) | 쿼리 구조 개선 필요 |
| `Using index` | 커버링 인덱스로 테이블 접근 없이 처리 | 이상적인 상태 |
| `Using where` | 인덱스로 필터링 후 추가 WHERE 조건 적용 | rows가 크면 인덱스에 조건 컬럼 추가 |
| `Using index condition` | ICP(Index Condition Pushdown) 적용됨 | MySQL 5.6+ 기본 동작, 정상 |

위 예시에서 `Using filesort`가 나왔다. `idx_status` 인덱스가 `status` 컬럼만 포함하기 때문에 `order_date` 정렬을 별도로 수행한다. 복합 인덱스로 바꾸면 해결된다.

```sql
-- filesort 제거를 위한 복합 인덱스
CREATE INDEX idx_status_date ON orders(status, order_date DESC);
```

이 인덱스를 사용하면 `status = 'shipped'` 필터와 `ORDER BY order_date DESC`를 인덱스 하나로 처리한다.

### PostgreSQL EXPLAIN ANALYZE 출력 읽기

PostgreSQL은 `EXPLAIN ANALYZE`를 쓰면 실제 실행 시간과 행 수까지 나온다. MySQL의 `EXPLAIN`은 예측값만 보여주는 것과 차이가 있다.

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.id, o.amount, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'shipped'
  AND o.order_date >= '2026-01-01';
```

```
Nested Loop  (cost=0.85..1245.30 rows=520 width=52)
             (actual time=0.045..12.340 rows=487 loops=1)
  Buffers: shared hit=1520 read=32
  ->  Index Scan using idx_status_date on orders o
        (cost=0.43..245.10 rows=520 width=24)
        (actual time=0.030..2.150 rows=487 loops=1)
        Index Cond: (status = 'shipped' AND order_date >= '2026-01-01')
        Buffers: shared hit=210 read=12
  ->  Index Scan using customers_pkey on customers c
        (cost=0.42..1.92 rows=1 width=32)
        (actual time=0.018..0.019 rows=1 loops=487)
        Index Cond: (id = o.customer_id)
        Buffers: shared hit=1310 read=20
Planning Time: 0.250 ms
Execution Time: 13.100 ms
```

읽는 순서: **안쪽(들여쓰기 깊은 쪽)부터 바깥으로**.

핵심 포인트:

- `actual time`: 첫 번째 숫자는 첫 행 반환 시간, 두 번째는 전체 완료 시간(ms)
- `rows`: 예측(cost 옆)과 실제(actual 옆)가 크게 다르면 통계가 오래된 것. `ANALYZE orders;` 실행
- `Buffers: shared hit` vs `shared read`: hit은 메모리 캐시, read는 디스크 접근. read 비율이 높으면 버퍼 풀(shared_buffers) 증설 검토
- `loops`: 해당 노드가 반복 실행된 횟수. loops=487이면 actual time에 487을 곱한 게 실제 총 시간

### EXPLAIN에서 놓치기 쉬운 것

**MySQL에서 type이 ref인데 rows가 수만 건인 경우** — 인덱스를 타긴 하지만 선택도(selectivity)가 낮아서 많은 행을 읽는다. `status` 같이 값 종류가 적은 컬럼 단독 인덱스가 대표적이다. 다른 조건 컬럼을 복합 인덱스에 추가해야 한다.

**PostgreSQL에서 Seq Scan이 나왔는데 인덱스가 있는 경우** — 테이블이 작거나(수천 건 이하), 조회 범위가 테이블의 상당 부분이면 옵티마이저가 의도적으로 Seq Scan을 선택한다. 이건 정상이다. 큰 테이블에서 Seq Scan이 나오면 문제다.

## 인덱스 설계

### 복합 인덱스 컬럼 순서

복합 인덱스는 왼쪽부터 순서대로 사용된다. 순서를 잘못 잡으면 인덱스가 있어도 못 쓴다.

```sql
-- 이 인덱스가 있을 때
CREATE INDEX idx_a_b_c ON orders(status, customer_id, order_date);

-- 사용됨
WHERE status = 'shipped'
WHERE status = 'shipped' AND customer_id = 100
WHERE status = 'shipped' AND customer_id = 100 AND order_date > '2026-01-01'

-- 사용 안됨 (첫 번째 컬럼 status 조건이 없음)
WHERE customer_id = 100
WHERE order_date > '2026-01-01'
```

컬럼 순서 정하는 기준:

1. **등호(=) 조건 컬럼을 앞에**, 범위(>, <, BETWEEN) 조건 컬럼을 뒤에 배치한다. 범위 조건 이후의 컬럼은 인덱스 스캔 범위를 줄이는 데 기여하지 못한다.
2. 등호 조건이 여러 개면 **카디널리티(값의 종류)가 높은 컬럼**을 앞에 놓는다. `customer_id`(수만 종류)가 `status`(3~5종류)보다 앞에 오는 게 낫다.

```sql
-- 나쁜 순서: status(카디널리티 낮음)가 앞
CREATE INDEX idx_bad ON orders(status, customer_id);

-- 좋은 순서: customer_id(카디널리티 높음)가 앞
CREATE INDEX idx_good ON orders(customer_id, status);
```

다만 한 가지 예외가 있다. 쿼리 패턴에서 `status`만 단독으로 조회하는 경우가 많다면, `status`를 앞에 두는 게 맞을 수 있다. 인덱스 설계는 쿼리 패턴에 맞춰야 한다.

### 커버링 인덱스

쿼리에 필요한 모든 컬럼이 인덱스에 포함되면, 테이블 데이터 페이지에 접근하지 않는다. I/O가 줄어서 체감 성능 차이가 크다.

```sql
-- 자주 실행되는 쿼리
SELECT name, email FROM users WHERE status = 'active' AND age > 25;

-- 커버링 인덱스: WHERE 조건 + SELECT 컬럼 모두 포함
CREATE INDEX idx_covering ON users(status, age, name, email);
```

EXPLAIN에서 `Using index`(MySQL) 또는 `Index Only Scan`(PostgreSQL)이 나오면 커버링 인덱스가 적용된 것이다.

주의할 점: 커버링 인덱스에 컬럼을 많이 넣으면 인덱스 크기가 커지고, INSERT/UPDATE 성능이 떨어진다. 자주 실행되는 핵심 쿼리에만 적용한다.

### 인덱스가 안 타는 흔한 실수

```sql
-- 1. 컬럼에 함수 적용
WHERE YEAR(created_at) = 2026      -- 인덱스 안 탐
WHERE created_at >= '2026-01-01'   -- 인덱스 탐
  AND created_at < '2027-01-01'

-- 2. 묵시적 타입 변환
WHERE phone = 01012345678          -- phone이 VARCHAR면 인덱스 안 탐
WHERE phone = '01012345678'        -- 인덱스 탐

-- 3. LIKE 앞쪽 와일드카드
WHERE name LIKE '%김%'             -- 인덱스 안 탐
WHERE name LIKE '김%'              -- 인덱스 탐

-- 4. OR 조건
WHERE status = 'A' OR city = 'Seoul'  -- 각각 인덱스 있어도 풀스캔 가능
-- 해결: UNION ALL로 분리
SELECT * FROM users WHERE status = 'A'
UNION ALL
SELECT * FROM users WHERE city = 'Seoul' AND status != 'A';
```

## Bulk Insert / Update 처리

### 대량 INSERT

한 건씩 INSERT하면 건마다 트랜잭션 커밋 + fsync가 발생한다. 10만 건이면 10만 번.

**배치 INSERT**

```sql
-- 느림: 건별 INSERT (기본 autocommit)
INSERT INTO logs (user_id, action, created_at) VALUES (1, 'login', NOW());
INSERT INTO logs (user_id, action, created_at) VALUES (2, 'login', NOW());
-- ... 10만 번 반복

-- 빠름: 멀티 로우 INSERT
INSERT INTO logs (user_id, action, created_at) VALUES
(1, 'login', NOW()),
(2, 'login', NOW()),
(3, 'login', NOW()),
-- ... 1000건씩 묶기
;
```

MySQL에서 멀티 로우 INSERT 시 한 번에 묶는 행 수는 `max_allowed_packet` 설정에 의존한다. 기본값이 4MB인 경우, 1000~5000건씩 나눠서 넣는 게 안전하다.

**트랜잭션으로 묶기**

```sql
-- autocommit 끄고 수동 트랜잭션
SET autocommit = 0;
START TRANSACTION;

INSERT INTO logs VALUES (...);
INSERT INTO logs VALUES (...);
-- 1000건마다 커밋
COMMIT;

START TRANSACTION;
-- 다음 1000건
...
COMMIT;

SET autocommit = 1;
```

트랜잭션을 너무 크게 잡으면(예: 100만 건을 한 트랜잭션) undo log가 비대해지고, 롤백 시 시간이 오래 걸린다. 1000~10000건 단위가 적당하다.

**LOAD DATA INFILE (MySQL)**

파일에서 직접 읽어 들이는 방식으로, INSERT 대비 10~20배 빠르다.

```sql
LOAD DATA INFILE '/tmp/users.csv'
INTO TABLE users
FIELDS TERMINATED BY ','
ENCLOSED BY '"'
LINES TERMINATED BY '\n'
IGNORE 1 ROWS
(name, email, age, status);
```

운영 환경에서 주의할 것:

- `secure_file_priv` 설정으로 허용된 디렉토리에서만 파일을 읽을 수 있다
- `LOAD DATA LOCAL INFILE`을 쓰면 클라이언트 측 파일을 읽지만, 보안 설정에서 막혀 있는 경우가 있다
- 기존 인덱스가 많으면 INSERT 속도가 느려진다. 대량 적재 시 인덱스를 잠시 비활성화하는 방법이 있다:

```sql
ALTER TABLE users DISABLE KEYS;
LOAD DATA INFILE '/tmp/users.csv' INTO TABLE users ...;
ALTER TABLE users ENABLE KEYS;
```

`DISABLE KEYS`는 non-unique 인덱스만 비활성화한다. 적재 후 한 번에 인덱스를 재구축하므로 건별로 인덱스를 갱신하는 것보다 빠르다.

**PostgreSQL COPY**

PostgreSQL에서는 `COPY` 명령이 동일한 역할을 한다.

```sql
COPY users (name, email, age, status)
FROM '/tmp/users.csv'
WITH (FORMAT csv, HEADER true);
```

### 대량 UPDATE

수백만 건을 한 번에 UPDATE하면 락이 오래 잡히고, replication lag이 생긴다.

```sql
-- 위험: 200만 건 한 번에 업데이트
UPDATE orders SET status = 'archived' WHERE order_date < '2025-01-01';

-- 안전: 1000건씩 나눠서 처리
UPDATE orders SET status = 'archived'
WHERE order_date < '2025-01-01' AND status != 'archived'
LIMIT 1000;

-- 영향받은 행이 0이 될 때까지 반복 실행
```

애플리케이션 코드에서 루프를 돌면서 처리하는 게 일반적이다.

```python
while True:
    cursor.execute("""
        UPDATE orders SET status = 'archived'
        WHERE order_date < '2025-01-01' AND status != 'archived'
        LIMIT 1000
    """)
    conn.commit()
    if cursor.rowcount == 0:
        break
    time.sleep(0.1)  # replication lag 방지를 위한 짧은 대기
```

## DB 서버 메모리 파라미터 튜닝

### MySQL (InnoDB)

**innodb_buffer_pool_size**

InnoDB의 핵심 설정이다. 테이블 데이터와 인덱스를 메모리에 캐싱하는 영역의 크기를 결정한다. 이 값이 작으면 디스크 I/O가 늘어나서 쿼리가 느려진다.

```ini
# my.cnf
[mysqld]
# 전용 DB 서버: 물리 메모리의 70~80%
# 예: 16GB 서버 → 12GB
innodb_buffer_pool_size = 12G

# 버퍼 풀을 여러 인스턴스로 분할 (경합 감소)
# 버퍼 풀이 1GB 이상이면 인스턴스 수를 늘린다
innodb_buffer_pool_instances = 8
```

현재 버퍼 풀 사용률 확인:

```sql
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_read_requests';  -- 메모리 읽기 요청
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_reads';           -- 디스크 읽기 (캐시 미스)

-- 히트율 계산: (read_requests - reads) / read_requests * 100
-- 99% 이상이 정상. 95% 이하면 버퍼 풀 증설 검토
```

**innodb_log_file_size**

redo log 파일 크기. 쓰기가 많은 워크로드에서 이 값이 작으면 체크포인트가 자주 발생해서 성능이 떨어진다.

```ini
# 쓰기 부하가 높은 서버
innodb_log_file_size = 1G
innodb_log_files_in_group = 2
```

**기타 주요 파라미터**

```ini
# 정렬, JOIN에 사용되는 세션별 버퍼 (기본값 256KB)
# 무작정 키우면 메모리 낭비. 대부분 기본값이면 된다
sort_buffer_size = 256K
join_buffer_size = 256K

# 쿼리 캐시 — MySQL 8.0에서 제거됨. 5.7 이하에서도 끄는 게 낫다
# 쓰기가 발생할 때마다 관련 캐시가 무효화되어 오히려 성능이 떨어진다
query_cache_type = 0

# 임시 테이블 크기 제한
tmp_table_size = 64M
max_heap_table_size = 64M
```

### PostgreSQL

**shared_buffers**

PostgreSQL이 직접 관리하는 공유 메모리 캐시. MySQL의 innodb_buffer_pool_size와 비슷한 역할이지만, PostgreSQL은 OS 파일 캐시도 같이 활용하기 때문에 메모리 전체를 할당하면 안 된다.

```ini
# postgresql.conf
# 전용 DB 서버: 물리 메모리의 25~40%
# 예: 16GB 서버 → 4~6GB
shared_buffers = 4GB
```

MySQL과 달리 25~40%만 할당하는 이유: PostgreSQL은 OS의 페이지 캐시에 크게 의존한다. shared_buffers를 너무 크게 잡으면 OS 캐시에 쓸 메모리가 줄어들어 오히려 성능이 나빠질 수 있다.

**work_mem**

정렬, 해시 JOIN 등에 사용하는 세션별 메모리. 한 쿼리 안에서 여러 정렬이 있으면 각각 work_mem만큼 사용한다.

```ini
# 기본값 4MB. 동시 접속이 100이면 최악의 경우 4MB * 100 * (쿼리당 정렬 수)
# 전체 메모리를 고려해서 설정
work_mem = 16MB
```

EXPLAIN ANALYZE에서 `Sort Method: external merge Disk`가 보이면 work_mem이 부족하다는 신호다. `Sort Method: quicksort Memory: xxxkB`가 나와야 정상이다.

**effective_cache_size**

실제 메모리를 할당하는 설정이 아니다. 옵티마이저에게 "OS 캐시 포함해서 이 정도 메모리를 쓸 수 있다"고 알려주는 힌트다. 이 값이 작으면 옵티마이저가 디스크 I/O 비용을 높게 잡아서 Seq Scan을 선호하게 된다.

```ini
# 물리 메모리의 50~75%
effective_cache_size = 12GB
```

## Prepared Statement와 쿼리 플랜 캐싱

### Prepared Statement의 두 가지 이점

1. **SQL Injection 방지** — 파라미터가 SQL 구문이 아닌 값으로 처리된다
2. **파싱/컴파일 비용 절감** — 같은 구조의 쿼리를 반복 실행할 때 파싱을 한 번만 한다

```sql
-- MySQL
PREPARE stmt FROM 'SELECT * FROM users WHERE id = ? AND status = ?';
SET @id = 123, @status = 'active';
EXECUTE stmt USING @id, @status;
DEALLOCATE PREPARE stmt;
```

### PostgreSQL의 플랜 캐싱

PostgreSQL에서 Prepared Statement를 처음 5회까지는 매번 파라미터 값에 맞는 custom plan을 생성한다. 6회째부터 generic plan(파라미터 값에 무관한 범용 플랜)의 비용을 비교해서, generic plan이 나쁘지 않으면 generic plan을 재사용한다.

문제가 되는 경우: 데이터 분포가 편향된 테이블에서 generic plan이 선택되면 특정 파라미터 값에서 성능이 나빠질 수 있다.

```sql
-- 강제로 custom plan만 사용하게 하려면 (PostgreSQL 12+)
SET plan_cache_mode = 'force_custom_plan';
```

### 애플리케이션에서의 사용

대부분의 ORM/드라이버가 Prepared Statement를 내부적으로 사용한다. 직접 관리해야 하는 경우:

```java
// Java JDBC — PreparedStatement 재사용
// Connection 풀에서 같은 쿼리를 반복 실행하면 서버 측 prepared statement가 재활용됨
try (PreparedStatement pstmt = conn.prepareStatement(
    "SELECT * FROM orders WHERE customer_id = ? AND status = ?")) {
    pstmt.setInt(1, customerId);
    pstmt.setString(2, "shipped");
    ResultSet rs = pstmt.executeQuery();
}
```

MySQL의 경우, JDBC URL에 `useServerPrepStmts=true`를 추가해야 서버 측 Prepared Statement가 활성화된다. 기본값은 클라이언트 측 에뮬레이션이라 파싱 절감 효과가 없다.

```
jdbc:mysql://localhost:3306/mydb?useServerPrepStmts=true&cachePrepStmts=true&prepStmtCacheSize=250
```

## 쿼리 최적화 기본 기법

### SELECT * 대신 필요한 컬럼만

```sql
-- 테이블에 TEXT, BLOB 컬럼이 있으면 네트워크 전송량 차이가 크다
SELECT * FROM articles WHERE category = 'tech';

-- 필요한 컬럼만 지정
SELECT id, title, author, created_at FROM articles WHERE category = 'tech';
```

### 서브쿼리를 JOIN으로 변환

```sql
-- IN 서브쿼리: MySQL 5.6 이전에서는 매 행마다 서브쿼리 재실행
SELECT * FROM orders
WHERE customer_id IN (SELECT id FROM customers WHERE city = 'Seoul');

-- JOIN으로 변환
SELECT o.* FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE c.city = 'Seoul';
```

MySQL 5.6+에서는 옵티마이저가 IN 서브쿼리를 내부적으로 semi-join으로 변환한다. 하지만 복잡한 서브쿼리에서는 변환이 안 되는 경우가 있으니 EXPLAIN으로 확인해야 한다.

### 페이지네이션 — OFFSET의 함정

```sql
-- OFFSET이 커지면 앞의 모든 행을 읽고 버린다
SELECT * FROM products ORDER BY id LIMIT 20 OFFSET 100000;
-- → 100020건을 읽고 100000건을 버림

-- 커서 기반 페이지네이션: 이전 페이지의 마지막 id를 기준으로
SELECT * FROM products WHERE id > 마지막_id ORDER BY id LIMIT 20;
-- → 20건만 읽음
```

## 연결 풀 관리

### 풀 크기 계산

```
적정 연결 수 = (CPU 코어 수 * 2) + 유효 디스크 수
```

4코어 서버에 SSD 1개면 적정 풀 크기는 약 10이다. 커넥션 수를 무작정 늘리면 컨텍스트 스위칭 비용이 증가해서 오히려 처리량이 떨어진다.

### HikariCP 설정 (Java)

```yaml
# application.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10
      minimum-idle: 5
      connection-timeout: 3000    # 3초 내 연결 못 얻으면 예외
      idle-timeout: 600000        # 유휴 연결 10분 후 제거
      max-lifetime: 1800000       # 30분마다 연결 갱신
      leak-detection-threshold: 60000  # 60초 이상 반납 안 된 연결 경고
```

`leak-detection-threshold`를 설정해두면 커넥션 누수를 조기에 발견할 수 있다. 로그에 연결 누수 경고가 나오면 해당 코드에서 `close()`를 빠뜨린 곳을 찾아야 한다.

## 파티셔닝

데이터가 수천만 건을 넘으면 인덱스만으로는 한계가 있다. 파티셔닝으로 쿼리가 접근하는 데이터 범위를 물리적으로 줄인다.

```sql
-- MySQL: 날짜 기반 Range 파티션
CREATE TABLE orders (
    id BIGINT AUTO_INCREMENT,
    order_date DATE NOT NULL,
    customer_id INT,
    amount DECIMAL(10,2),
    PRIMARY KEY (id, order_date)   -- 파티션 키가 PK에 포함되어야 함
) PARTITION BY RANGE (YEAR(order_date)) (
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026),
    PARTITION p2026 VALUES LESS THAN (2027),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

파티션 프루닝(partition pruning): WHERE 조건에 파티션 키가 포함되면 해당 파티션만 스캔한다. EXPLAIN에서 `partitions` 컬럼을 확인한다.

```sql
EXPLAIN SELECT * FROM orders WHERE order_date >= '2026-01-01';
-- partitions: p2026,p_future  ← 2개 파티션만 스캔
```

주의: 파티션 키가 WHERE 절에 없으면 모든 파티션을 스캔한다. 파티셔닝이 오히려 성능을 떨어뜨리는 경우가 이것이다.

## 실제 성능 개선 사례

### 사례 1: 주문 목록 조회 — 12초 → 45ms

**상황**: 관리자 페이지의 주문 목록 API 응답이 12초.

```sql
SELECT o.*, c.name, c.email
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.id
WHERE o.created_at >= '2026-01-01'
ORDER BY o.created_at DESC
LIMIT 50 OFFSET 5000;
```

**EXPLAIN 결과:**
```
type: ALL, rows: 3200000, Extra: Using where; Using temporary; Using filesort
```

풀 테이블 스캔 + 임시 테이블 + filesort. 세 가지가 동시에 나오면 최악이다.

**원인 분석:**
1. `created_at`에 인덱스가 없어서 320만 건 풀 스캔
2. OFFSET 5000으로 5050건을 읽고 5000건을 버림
3. LEFT JOIN 결과를 임시 테이블에 저장하고 정렬

**조치:**
```sql
-- 1. 복합 인덱스 추가
CREATE INDEX idx_created_at ON orders(created_at DESC);

-- 2. 커서 기반 페이지네이션으로 변경
SELECT o.id, o.amount, o.status, o.created_at, c.name, c.email
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.id
WHERE o.created_at < '2026-03-15 14:30:00'  -- 이전 페이지 마지막 값
ORDER BY o.created_at DESC
LIMIT 50;
```

**결과:** type: range, rows: 50, Extra: Using index condition — 12초 → 45ms

### 사례 2: 통계 배치 — 야간 배치 40분 → 3분

**상황**: 매일 새벽에 실행되는 일별 매출 집계 배치가 40분 소요.

```sql
-- 원래 쿼리: 서브쿼리 + GROUP BY
SELECT
    DATE(o.created_at) AS sale_date,
    c.region,
    COUNT(*) AS order_count,
    SUM(o.amount) AS total_amount
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.created_at >= '2026-03-01' AND o.created_at < '2026-03-28'
GROUP BY DATE(o.created_at), c.region;
```

**원인:**
- `DATE(o.created_at)` 함수 호출로 인덱스 사용 불가
- 27일치 주문 데이터 전체를 JOIN 후 GROUP BY

**조치:**

```sql
-- 1. 날짜 컬럼을 미리 저장 (테이블 스키마 변경)
ALTER TABLE orders ADD COLUMN order_date DATE GENERATED ALWAYS AS (DATE(created_at)) STORED;
CREATE INDEX idx_order_date_amount ON orders(order_date, amount);

-- 2. 생성 컬럼 사용
SELECT
    o.order_date AS sale_date,
    c.region,
    COUNT(*) AS order_count,
    SUM(o.amount) AS total_amount
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.order_date >= '2026-03-01' AND o.order_date < '2026-03-28'
GROUP BY o.order_date, c.region;
```

**결과:** 인덱스 range scan + 생성 컬럼으로 함수 호출 제거. 40분 → 3분.

### 사례 3: 대량 데이터 마이그레이션 — 6시간 → 25분

**상황**: 500만 건 사용자 테이블에 `phone_normalized` 컬럼 추가 후 데이터 채우기.

```sql
-- 원래: 한 번에 UPDATE
UPDATE users SET phone_normalized = REPLACE(REPLACE(phone, '-', ''), ' ', '');
-- → 6시간 소요, 그 동안 users 테이블 락
```

**조치:**

```python
# 1000건씩 배치 처리
batch_size = 1000
last_id = 0

while True:
    cursor.execute("""
        UPDATE users
        SET phone_normalized = REPLACE(REPLACE(phone, '-', ''), ' ', '')
        WHERE id > %s AND phone_normalized IS NULL
        ORDER BY id
        LIMIT %s
    """, (last_id, batch_size))

    conn.commit()
    affected = cursor.rowcount
    if affected == 0:
        break

    # 다음 배치의 시작점
    cursor.execute("SELECT MAX(id) FROM users WHERE id > %s ORDER BY id LIMIT %s", (last_id, batch_size))
    last_id = cursor.fetchone()[0]

    time.sleep(0.05)  # 서비스 영향 최소화
```

**결과:** 락 점유 시간이 건당 수 ms로 줄어서 서비스 중단 없이 마이그레이션 완료. 6시간 → 25분.
