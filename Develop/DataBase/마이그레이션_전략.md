# 데이터베이스 마이그레이션 전략

## 개요

DB 마이그레이션은 **스키마 변경을 코드처럼 버전 관리**하는 기법이다. 팀 협업 시 스키마 변경 이력이 추적되고, 배포 환경별로 일관된 스키마를 보장한다.

```
마이그레이션 없이 운영하면:
  - "이 컬럼 언제 추가됐지?" → 모름
  - 개발/스테이징/프로덕션 스키마 불일치
  - 배포 시 수동으로 SQL 실행 → 실수 발생
  - 롤백 시 어떤 SQL을 되돌려야 하는지 모름
```

---

## Flyway vs Liquibase

| 항목 | Flyway | Liquibase |
|------|--------|-----------|
| **설정 복잡도** | 단순 | 복잡 |
| **마이그레이션 형식** | SQL, Java | SQL, XML, YAML, JSON |
| **롤백** | 유료(Teams 이상) | 무료 지원 |
| **버전 관리** | 파일명 기반 | changeset ID 기반 |
| **Spring Boot 통합** | 자동 (starter 포함) | 자동 (starter 포함) |
| **학습 곡선** | 낮음 | 높음 |
| **권장 상황** | 단순 SQL 중심 | 멀티 DB, 롤백 필요 |

## Flyway 실무 설정

### 의존성

```xml
<!-- Maven -->
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-mysql</artifactId> <!-- MySQL 8+ -->
</dependency>
```

```gradle
// Gradle
implementation 'org.flywaydb:flyway-core'
implementation 'org.flywaydb:flyway-mysql'
```

### application.yaml

```yaml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration          # 마이그레이션 파일 위치
    baseline-on-migrate: true                  # 기존 DB에 처음 적용 시 필요
    baseline-version: 0                        # 기준 버전
    validate-on-migrate: true                  # 실행 전 체크섬 검증
    out-of-order: false                        # 순서 벗어난 마이그레이션 금지
    table: flyway_schema_history               # 이력 테이블명
    # 환경별 설정
    placeholders:
      schema: myapp
```

### 파일 네이밍 규칙

```
src/main/resources/db/migration/
  V1__create_users_table.sql          ← 버전 마이그레이션 (한 번만 실행)
  V2__add_email_column.sql
  V2.1__add_email_index.sql           ← 소수점 서브버전 가능
  V3__create_orders_table.sql
  R__refresh_user_view.sql            ← 반복 마이그레이션 (매번 실행)
  U2__undo_add_email_column.sql       ← 실행 취소 (Flyway Teams)

네이밍: V{버전}__{설명}.sql
  - V: 버전 마이그레이션 접두사
  - 버전: 숫자 (1, 2, 2.1, 2.1.1)
  - __: 언더스코어 2개 (필수)
  - 설명: 언더스코어로 단어 구분
```

### 마이그레이션 파일 예시

```sql
-- V1__create_users_table.sql
CREATE TABLE users (
    id         BIGINT       NOT NULL AUTO_INCREMENT,
    email      VARCHAR(255) NOT NULL,
    name       VARCHAR(100) NOT NULL,
    created_at DATETIME(6)  NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
    updated_at DATETIME(6)  NOT NULL DEFAULT CURRENT_TIMESTAMP(6) ON UPDATE CURRENT_TIMESTAMP(6),
    PRIMARY KEY (id),
    UNIQUE KEY uk_users_email (email)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

```sql
-- V2__create_orders_table.sql
CREATE TABLE orders (
    id           BIGINT         NOT NULL AUTO_INCREMENT,
    user_id      BIGINT         NOT NULL,
    total_amount DECIMAL(10, 2) NOT NULL,
    status       VARCHAR(20)    NOT NULL DEFAULT 'PENDING',
    created_at   DATETIME(6)    NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
    PRIMARY KEY (id),
    CONSTRAINT fk_orders_users FOREIGN KEY (user_id) REFERENCES users (id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

```sql
-- V3__add_phone_to_users.sql
ALTER TABLE users
    ADD COLUMN phone VARCHAR(20) NULL AFTER email;
```

```sql
-- V4__add_index_on_orders_status.sql
CREATE INDEX idx_orders_status_created
    ON orders (status, created_at DESC);
```

## 주의사항

### 적용 후 마이그레이션 파일 수정 금지

```
Flyway는 각 파일의 체크섬을 저장한다.
파일을 수정하면 체크섬 불일치로 배포 실패!

❌ V2__create_orders_table.sql 을 수정
✅ V5__alter_orders_add_column.sql 을 새로 만들어서 추가
```

### 데이터 마이그레이션

```sql
-- V5__migrate_status_data.sql
-- 스키마 변경 + 데이터 변환을 한 파일에서 처리
ALTER TABLE orders ADD COLUMN status_code INT NOT NULL DEFAULT 0;

UPDATE orders SET status_code = 1 WHERE status = 'PENDING';
UPDATE orders SET status_code = 2 WHERE status = 'COMPLETED';
UPDATE orders SET status_code = 3 WHERE status = 'CANCELLED';

ALTER TABLE orders DROP COLUMN status;
ALTER TABLE orders CHANGE COLUMN status_code status INT NOT NULL DEFAULT 0;
```

## 롤백 전략

Flyway 커뮤니티 버전은 자동 롤백 미지원. 실무에서는 다음 방법을 사용한다.

### 방법 1: 새 마이그레이션으로 되돌리기 (권장)

```sql
-- 실수로 V3에서 컬럼을 삭제한 경우
-- V6__restore_deleted_column.sql
ALTER TABLE users ADD COLUMN deleted_at DATETIME(6) NULL;
```

### 방법 2: 블루-그린 DB 마이그레이션

```
배포 시 두 버전이 동시에 동작해야 하는 경우:

Step 1: 새 컬럼 추가 (nullable, 기본값 있음)
  → 구버전 앱은 새 컬럼 무시, 신버전 앱은 새 컬럼 사용

Step 2: 신버전 배포 완료

Step 3: 구버전 컬럼에 NOT NULL 제약 추가 또는 삭제
```

```sql
-- Step 1: V7__add_new_column_nullable.sql (구버전 호환)
ALTER TABLE users ADD COLUMN full_name VARCHAR(200) NULL;

-- (신버전 배포 완료 후)
-- Step 2: V8__make_full_name_not_null.sql
UPDATE users SET full_name = CONCAT(first_name, ' ', last_name) WHERE full_name IS NULL;
ALTER TABLE users MODIFY COLUMN full_name VARCHAR(200) NOT NULL;
```

## 환경별 분리

```yaml
# application-local.yaml
spring:
  flyway:
    locations:
      - classpath:db/migration
      - classpath:db/testdata   # 테스트 데이터 (로컬 전용)

# application-prod.yaml
spring:
  flyway:
    locations:
      - classpath:db/migration  # 프로덕션은 마이그레이션만
    out-of-order: false
    validate-on-migrate: true
```

```
src/main/resources/
  db/
    migration/
      V1__create_users_table.sql
      V2__create_orders_table.sql
    testdata/
      V900__insert_test_users.sql   # 로컬/테스트 환경 초기 데이터
```

## CI/CD 파이프라인 통합

```yaml
# GitHub Actions
- name: Run DB Migration
  run: |
    ./mvnw flyway:migrate \
      -Dflyway.url=${{ secrets.DB_URL }} \
      -Dflyway.user=${{ secrets.DB_USER }} \
      -Dflyway.password=${{ secrets.DB_PASSWORD }}

- name: Validate Migration
  run: ./mvnw flyway:validate
```

## flyway_schema_history 테이블

Flyway가 자동 생성하는 이력 테이블이다.

```sql
SELECT installed_rank, version, description, type, script, checksum, success
FROM flyway_schema_history
ORDER BY installed_rank;

-- 결과 예시:
-- 1  | 1   | create users table  | SQL | V1__create_users_table.sql  | 12345678 | 1
-- 2  | 2   | create orders table | SQL | V2__create_orders_table.sql | 87654321 | 1
-- 3  | 3   | add phone to users  | SQL | V3__add_phone_to_users.sql  | 11223344 | 1
```

## 체크리스트

- [ ] 마이그레이션 파일은 **절대 수정 금지** — 새 파일 추가만
- [ ] 파일명 규칙 팀 내 통일 (V{숫자}__{동사}_{목적어}.sql)
- [ ] `validate-on-migrate: true` 설정 (프로덕션 필수)
- [ ] 대용량 테이블 ALTER는 pt-online-schema-change 또는 gh-ost 사용
- [ ] 마이그레이션 파일은 PR로 리뷰 후 머지
- [ ] 로컬/스테이징에서 먼저 실행해 검증
- [ ] 프로덕션 마이그레이션 전 DB 스냅샷 생성
