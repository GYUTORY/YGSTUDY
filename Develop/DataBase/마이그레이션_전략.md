# 데이터베이스 마이그레이션

## 개요

DB 마이그레이션은 **스키마 변경을 코드처럼 버전 관리**하는 기법이다. 팀 협업 시 스키마 변경 이력이 추적되고, 배포 환경별로 일관된 스키마를 보장한다.

```
마이그레이션 없이 운영하면:
  - "이 컬럼 언제 추가됐지?" → 모름
  - 개발/스테이징/프로덕션 스키마 불일치
  - 배포 시 수동으로 SQL 실행 → 실수 발생
  - 롤백 시 어떤 SQL을 되돌려야 하는지 모름
```

---

## Flyway vs Liquibase

| 항목 | Flyway | Liquibase |
|------|--------|-----------|
| **설정 복잡도** | 단순 | 복잡 |
| **마이그레이션 형식** | SQL, Java | SQL, XML, YAML, JSON |
| **롤백** | 유료(Teams 이상) | 무료 지원 |
| **버전 관리** | 파일명 기반 | changeset ID 기반 |
| **Spring Boot 통합** | 자동 (starter 포함) | 자동 (starter 포함) |
| **학습 곡선** | 낮음 | 높음 |
| **권장 상황** | 단순 SQL 중심 | 멀티 DB, 롤백 필요 |

## Flyway 실무 설정

### 의존성

```xml
<!-- Maven -->
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-mysql</artifactId> <!-- MySQL 8+ -->
</dependency>
```

```gradle
// Gradle
implementation 'org.flywaydb:flyway-core'
implementation 'org.flywaydb:flyway-mysql'
```

### application.yaml

```yaml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration          # 마이그레이션 파일 위치
    baseline-on-migrate: true                  # 기존 DB에 처음 적용 시 필요
    baseline-version: 0                        # 기준 버전
    validate-on-migrate: true                  # 실행 전 체크섬 검증
    out-of-order: false                        # 순서 벗어난 마이그레이션 금지
    table: flyway_schema_history               # 이력 테이블명
    # 환경별 설정
    placeholders:
      schema: myapp
```

### 파일 네이밍 규칙

```
src/main/resources/db/migration/
  V1__create_users_table.sql          ← 버전 마이그레이션 (한 번만 실행)
  V2__add_email_column.sql
  V2.1__add_email_index.sql           ← 소수점 서브버전 가능
  V3__create_orders_table.sql
  R__refresh_user_view.sql            ← 반복 마이그레이션 (매번 실행)
  U2__undo_add_email_column.sql       ← 실행 취소 (Flyway Teams)

네이밍: V{버전}__{설명}.sql
  - V: 버전 마이그레이션 접두사
  - 버전: 숫자 (1, 2, 2.1, 2.1.1)
  - __: 언더스코어 2개 (필수)
  - 설명: 언더스코어로 단어 구분
```

### 마이그레이션 파일 예시

```sql
-- V1__create_users_table.sql
CREATE TABLE users (
    id         BIGINT       NOT NULL AUTO_INCREMENT,
    email      VARCHAR(255) NOT NULL,
    name       VARCHAR(100) NOT NULL,
    created_at DATETIME(6)  NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
    updated_at DATETIME(6)  NOT NULL DEFAULT CURRENT_TIMESTAMP(6) ON UPDATE CURRENT_TIMESTAMP(6),
    PRIMARY KEY (id),
    UNIQUE KEY uk_users_email (email)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

```sql
-- V2__create_orders_table.sql
CREATE TABLE orders (
    id           BIGINT         NOT NULL AUTO_INCREMENT,
    user_id      BIGINT         NOT NULL,
    total_amount DECIMAL(10, 2) NOT NULL,
    status       VARCHAR(20)    NOT NULL DEFAULT 'PENDING',
    created_at   DATETIME(6)    NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
    PRIMARY KEY (id),
    CONSTRAINT fk_orders_users FOREIGN KEY (user_id) REFERENCES users (id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

```sql
-- V3__add_phone_to_users.sql
ALTER TABLE users
    ADD COLUMN phone VARCHAR(20) NULL AFTER email;
```

```sql
-- V4__add_index_on_orders_status.sql
CREATE INDEX idx_orders_status_created
    ON orders (status, created_at DESC);
```

## Liquibase 실무 설정

Flyway가 SQL 중심이라면, Liquibase는 changeset 단위로 변경을 관리한다. 롤백이 무료이고, 여러 DB 벤더를 동시에 지원해야 할 때 선택한다.

### 의존성

```xml
<!-- Maven -->
<dependency>
    <groupId>org.liquibase</groupId>
    <artifactId>liquibase-core</artifactId>
</dependency>
```

```gradle
// Gradle
implementation 'org.liquibase:liquibase-core'
```

### application.yaml

```yaml
spring:
  liquibase:
    enabled: true
    change-log: classpath:db/changelog/db.changelog-master.yaml
    default-schema: myapp
    # 프로덕션에서는 반드시 false
    drop-first: false
```

### changelog 구조

Liquibase는 마스터 changelog에서 개별 파일을 include하는 방식이다.

```
src/main/resources/db/changelog/
  db.changelog-master.yaml           ← 마스터 파일
  changes/
    001-create-users.yaml
    002-create-orders.yaml
    003-add-phone-to-users.yaml
```

```yaml
# db.changelog-master.yaml
databaseChangeLog:
  - includeAll:
      path: db/changelog/changes/
      relativeToChangelogFile: false
```

### changeset 작성

```yaml
# 001-create-users.yaml
databaseChangeLog:
  - changeSet:
      id: 001-create-users
      author: kimdev
      changes:
        - createTable:
            tableName: users
            columns:
              - column:
                  name: id
                  type: BIGINT
                  autoIncrement: true
                  constraints:
                    primaryKey: true
                    nullable: false
              - column:
                  name: email
                  type: VARCHAR(255)
                  constraints:
                    nullable: false
                    unique: true
              - column:
                  name: name
                  type: VARCHAR(100)
                  constraints:
                    nullable: false
      rollback:
        - dropTable:
            tableName: users
```

`rollback` 블록을 명시하면 `liquibase rollback` 명령으로 되돌릴 수 있다. Flyway와 가장 큰 차이점이다.

### SQL 형식도 지원한다

XML/YAML이 싫으면 SQL로 써도 된다.

```sql
-- changes/003-add-phone-to-users.sql
-- changeset kimdev:003-add-phone
ALTER TABLE users ADD COLUMN phone VARCHAR(20) NULL;
-- rollback ALTER TABLE users DROP COLUMN phone;
```

주석으로 `changeset`과 `rollback`을 지정하는 형태다. `-- changeset {author}:{id}` 형식을 지켜야 한다.

### Liquibase 실무 주의사항

**changeset ID는 한번 정하면 변경하면 안 된다.** Liquibase는 `DATABASECHANGELOG` 테이블에 author + id 조합으로 실행 여부를 판단한다. ID를 바꾸면 같은 changeset이 다시 실행된다.

**precondition을 활용한다.** 테이블이나 컬럼 존재 여부를 체크해서 이미 적용된 환경에서 에러가 나지 않게 한다.

```yaml
- changeSet:
    id: 003-add-phone
    author: kimdev
    preConditions:
      - onFail: MARK_RAN
      - not:
          - columnExists:
              tableName: users
              columnName: phone
    changes:
      - addColumn:
          tableName: users
          columns:
            - column:
                name: phone
                type: VARCHAR(20)
```

`onFail: MARK_RAN`은 조건 불충족 시 실행은 건너뛰되 실행한 것으로 기록한다. 수동으로 스키마를 변경한 환경과 자동 마이그레이션 환경이 섞여 있을 때 쓴다.

## 대용량 테이블 스키마 변경

MySQL에서 `ALTER TABLE`은 내부적으로 테이블 전체를 복사한다(MySQL 5.6 이전). MySQL 8.0의 Instant DDL이 일부 해결하지만, 컬럼 삭제, 타입 변경, 인덱스 추가 등은 여전히 테이블 복사가 발생한다. 수천만 행 이상 테이블에서 `ALTER TABLE`을 그냥 실행하면 수십 분~수 시간 동안 테이블 락이 걸린다.

### pt-online-schema-change (Percona Toolkit)

Percona에서 만든 도구로, 테이블 락 없이 스키마를 변경한다.

**동작 원리:**

1. 원본 테이블과 동일한 구조의 빈 테이블(`_tablename_new`)을 생성한다
2. 빈 테이블에 원하는 `ALTER`를 적용한다
3. 원본 테이블에 트리거를 설치해서, 복사 중 발생하는 INSERT/UPDATE/DELETE를 새 테이블에도 반영한다
4. 원본 데이터를 chunk 단위로 새 테이블에 복사한다
5. 복사 완료 후 원본과 새 테이블의 이름을 RENAME으로 교체한다
6. 트리거와 구 테이블을 삭제한다

```bash
# 설치
apt-get install percona-toolkit
# 또는
brew install percona-toolkit

# 실행 예시: users 테이블에 컬럼 추가
pt-online-schema-change \
  --alter "ADD COLUMN phone VARCHAR(20) NULL" \
  --host=127.0.0.1 \
  --port=3306 \
  --user=root \
  --password=secret \
  D=myapp,t=users \
  --execute

# dry-run으로 먼저 확인
pt-online-schema-change \
  --alter "ADD COLUMN phone VARCHAR(20) NULL" \
  D=myapp,t=users \
  --dry-run
```

**주의할 점:**

- 원본 테이블에 트리거가 이미 있으면 사용 불가. MySQL은 테이블당 동일 이벤트의 트리거를 하나만 허용한다.
- FK(외래키)가 걸린 테이블은 `--alter-foreign-keys-method` 옵션이 필요하다. 자식 테이블의 FK도 같이 바꿔야 해서 시간이 더 걸린다.
- `--chunk-size`로 한번에 복사할 행 수를 제어한다. 기본값은 1000이고, 슬레이브 지연이 커지면 자동으로 줄인다.

### gh-ost (GitHub Online Schema Transmogrifier)

GitHub에서 만든 도구다. pt-online-schema-change와 목적은 같지만, **트리거를 사용하지 않는다**는 점이 다르다.

**동작 원리:**

1. 원본과 동일한 ghost 테이블을 생성하고 ALTER를 적용한다
2. 원본 데이터를 chunk 단위로 ghost 테이블에 복사한다
3. **binlog를 감시**해서 복사 중 발생하는 변경사항을 ghost 테이블에 반영한다
4. 복사 완료 후 테이블 이름을 교체한다

트리거 대신 binlog를 쓰기 때문에, 원본 테이블에 이미 트리거가 있어도 사용 가능하다.

```bash
# 설치
wget https://github.com/github/gh-ost/releases/latest/download/gh-ost-binary-linux-amd64.tar.gz

# 실행 예시
gh-ost \
  --host=127.0.0.1 \
  --port=3306 \
  --user=root \
  --password=secret \
  --database=myapp \
  --table=users \
  --alter="ADD COLUMN phone VARCHAR(20) NULL" \
  --execute

# --test-on-replica 옵션으로 슬레이브에서 먼저 테스트
gh-ost \
  --host=replica-host \
  --database=myapp \
  --table=users \
  --alter="ADD COLUMN phone VARCHAR(20) NULL" \
  --test-on-replica
```

**gh-ost의 장점:**

- 트리거 불필요. binlog 기반이라 원본 테이블에 부하를 덜 준다.
- 실행 중 `--throttle-flag-file`이나 소켓을 통해 일시 중지/재개가 가능하다. pt-osc는 중단하면 처음부터 다시 해야 한다.
- 슬레이브 지연 감지 시 자동 throttle.

**gh-ost의 제약:**

- `binlog_format=ROW`가 필수다. MIXED나 STATEMENT에서는 동작하지 않는다.
- FK가 있는 테이블은 지원하지 않는다. FK가 있으면 pt-osc를 써야 한다.
- binlog를 읽어야 해서 replication 관련 권한(`REPLICATION SLAVE`, `REPLICATION CLIENT`)이 필요하다.

### pt-osc vs gh-ost 선택 기준

| 상황 | 선택 |
|------|------|
| 테이블에 트리거가 이미 있음 | gh-ost |
| 테이블에 FK(외래키)가 있음 | pt-osc |
| 작업 중 일시중지/재개가 필요 | gh-ost |
| binlog_format이 ROW가 아님 | pt-osc |
| 그 외 일반적인 경우 | gh-ost (부하가 적음) |

## Zero-Downtime 마이그레이션 패턴

프로덕션 배포 시 앱의 구버전과 신버전이 동시에 동작하는 구간이 있다. 이 구간에서 스키마 변경이 호환되지 않으면 서비스 장애가 난다.

### Expand/Contract 패턴

스키마 변경을 두 단계로 나눈다.

**Expand (확장):** 기존 스키마와 호환되는 상태로 새 스키마를 추가한다. 구버전 앱은 새 컬럼/테이블을 무시하고, 신버전 앱은 새 구조를 사용한다.

**Contract (축소):** 구버전 앱이 완전히 내려간 후, 더 이상 필요 없는 구 컬럼/테이블을 정리한다.

예시: `users.name` 컬럼을 `first_name` + `last_name`으로 분리하는 경우

```sql
-- Phase 1: Expand (구버전 배포 중에 실행)
-- V10__expand_add_name_columns.sql
ALTER TABLE users ADD COLUMN first_name VARCHAR(100) NULL;
ALTER TABLE users ADD COLUMN last_name VARCHAR(100) NULL;

-- 기존 데이터 변환 (백그라운드 배치로 처리해도 됨)
UPDATE users
SET first_name = SUBSTRING_INDEX(name, ' ', 1),
    last_name  = SUBSTRING_INDEX(name, ' ', -1)
WHERE first_name IS NULL;
```

이 시점에서 구버전 앱은 `name` 컬럼을 읽고, 신버전 앱은 `first_name` + `last_name`을 읽고 쓴다. 신버전 앱은 `name` 컬럼도 같이 업데이트해줘야 한다(dual-write).

```sql
-- Phase 2: Contract (구버전 앱이 완전히 내려간 후)
-- V15__contract_drop_name_column.sql
ALTER TABLE users DROP COLUMN name;
ALTER TABLE users MODIFY COLUMN first_name VARCHAR(100) NOT NULL;
ALTER TABLE users MODIFY COLUMN last_name VARCHAR(100) NOT NULL;
```

Expand와 Contract 사이에 최소 1번의 배포 사이클이 있어야 한다. 실무에서는 1~2주 간격을 두는 경우가 많다.

### MySQL Online DDL

MySQL 5.6부터 `ALGORITHM`과 `LOCK` 옵션으로 온라인 DDL을 제어할 수 있다.

```sql
-- INPLACE: 테이블 복사 없이 변경 (가능한 경우)
ALTER TABLE users ADD COLUMN phone VARCHAR(20) NULL, ALGORITHM=INPLACE, LOCK=NONE;

-- INSTANT: MySQL 8.0.12+, 메타데이터만 변경 (가장 빠름)
ALTER TABLE users ADD COLUMN phone VARCHAR(20) NULL, ALGORITHM=INSTANT;
```

**ALGORITHM 종류:**

- `INSTANT`: 메타데이터만 변경. 밀리초 단위로 끝난다. 컬럼 추가(테이블 끝에만), 기본값 변경, ENUM 값 추가 등 제한적으로 지원한다.
- `INPLACE`: 테이블 전체 복사는 하지 않지만, 내부적으로 인덱스 재구성 등은 수행한다.
- `COPY`: 테이블 전체를 복사한다. 가장 느리고 공간도 2배 필요하다.

**LOCK 종류:**

- `NONE`: DDL 중에도 읽기/쓰기 모두 가능
- `SHARED`: 읽기만 가능, 쓰기 차단
- `EXCLUSIVE`: 읽기/쓰기 모두 차단

```
실무 팁:
  MySQL이 ALGORITHM=INSTANT를 지원하지 않는 변경이면 자동으로 INPLACE → COPY로 fallback한다.
  명시적으로 ALGORITHM=INSTANT를 지정하면 지원 안 되는 경우 에러가 나서,
  의도치 않게 COPY 알고리즘이 실행되는 것을 막을 수 있다.
```

### 컬럼 삭제/이름 변경 시 안전한 순서

컬럼을 바로 삭제하거나 이름을 바꾸면 구버전 앱에서 에러가 난다.

**컬럼 삭제:**

```
1. 신버전 앱 코드에서 해당 컬럼 참조를 모두 제거
2. 신버전 배포 완료
3. 다음 마이그레이션에서 컬럼 삭제
```

**컬럼 이름 변경:**

```
1. 새 이름의 컬럼 추가 (expand)
2. 신버전 앱에서 양쪽 컬럼 모두 기록 (dual-write)
3. 기존 데이터 복사
4. 신버전 앱에서 새 컬럼만 읽도록 변경
5. 구 컬럼 삭제 (contract)
```

직접 `RENAME COLUMN`을 쓰면 안 된다. 구버전 앱이 구 이름으로 쿼리하는 순간 에러가 터진다.

## Node.js ORM 마이그레이션

### Prisma Migrate

Prisma는 `schema.prisma` 파일을 수정하면 마이그레이션 SQL을 자동 생성한다.

```prisma
// schema.prisma
datasource db {
  provider = "mysql"
  url      = env("DATABASE_URL")
}

generator client {
  provider = "prisma-client-js"
}

model User {
  id        Int      @id @default(autoincrement())
  email     String   @unique @db.VarChar(255)
  name      String   @db.VarChar(100)
  phone     String?  @db.VarChar(20)    // 새로 추가
  orders    Order[]
  createdAt DateTime @default(now()) @map("created_at") @db.DateTime(6)
  updatedAt DateTime @updatedAt @map("updated_at") @db.DateTime(6)

  @@map("users")
}

model Order {
  id          Int      @id @default(autoincrement())
  userId      Int      @map("user_id")
  user        User     @relation(fields: [userId], references: [id])
  totalAmount Decimal  @map("total_amount") @db.Decimal(10, 2)
  status      String   @default("PENDING") @db.VarChar(20)
  createdAt   DateTime @default(now()) @map("created_at") @db.DateTime(6)

  @@map("orders")
}
```

```bash
# 스키마 변경 후 마이그레이션 생성
npx prisma migrate dev --name add_phone_to_users

# 생성되는 파일 구조:
# prisma/migrations/
#   20260329120000_add_phone_to_users/
#     migration.sql

# 프로덕션 적용
npx prisma migrate deploy

# 마이그레이션 상태 확인
npx prisma migrate status
```

생성된 SQL 파일:

```sql
-- prisma/migrations/20260329120000_add_phone_to_users/migration.sql
ALTER TABLE `users` ADD COLUMN `phone` VARCHAR(20) NULL;
```

**Prisma Migrate 실무 주의사항:**

`prisma migrate dev`는 개발용이다. 스키마 차이를 감지해서 마이그레이션을 생성하고 바로 적용한다. 프로덕션에서는 반드시 `prisma migrate deploy`를 쓴다. deploy는 아직 적용되지 않은 마이그레이션 파일만 순서대로 실행한다.

자동 생성된 SQL이 마음에 안 들면 직접 수정해도 된다. 다만 수정 후 `prisma migrate dev`를 다시 실행하면 drift가 감지되면서 경고가 나온다. 수정이 필요하면 생성 직후, deploy 전에 하는 게 좋다.

Prisma는 `_prisma_migrations` 테이블로 이력을 관리한다. Flyway의 `flyway_schema_history`와 같은 역할이다.

```bash
# 마이그레이션이 꼬였을 때 특정 마이그레이션을 "적용된 것으로" 표시
npx prisma migrate resolve --applied 20260329120000_add_phone_to_users

# 롤백은 자동 지원하지 않는다. 직접 롤백 SQL을 작성해서 실행해야 한다.
```

### TypeORM Migrations

TypeORM은 엔티티 변경을 감지해서 마이그레이션을 생성한다.

```typescript
// src/entity/User.ts
import { Entity, PrimaryGeneratedColumn, Column, OneToMany, CreateDateColumn, UpdateDateColumn } from 'typeorm';
import { Order } from './Order';

@Entity('users')
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ type: 'varchar', length: 255, unique: true })
  email: string;

  @Column({ type: 'varchar', length: 100 })
  name: string;

  @Column({ type: 'varchar', length: 20, nullable: true })
  phone: string;  // 새로 추가

  @OneToMany(() => Order, order => order.user)
  orders: Order[];

  @CreateDateColumn({ name: 'created_at', type: 'datetime', precision: 6 })
  createdAt: Date;

  @UpdateDateColumn({ name: 'updated_at', type: 'datetime', precision: 6 })
  updatedAt: Date;
}
```

```typescript
// data-source.ts
import { DataSource } from 'typeorm';

export const AppDataSource = new DataSource({
  type: 'mysql',
  host: process.env.DB_HOST,
  port: 3306,
  username: process.env.DB_USER,
  password: process.env.DB_PASSWORD,
  database: 'myapp',
  entities: ['src/entity/**/*.ts'],
  migrations: ['src/migration/**/*.ts'],
  // 프로덕션에서 synchronize: true는 절대 쓰지 않는다
  synchronize: false,
});
```

```bash
# 엔티티 변경 감지 → 마이그레이션 생성
npx typeorm migration:generate src/migration/AddPhoneToUsers -d data-source.ts

# 빈 마이그레이션 생성 (직접 작성)
npx typeorm migration:create src/migration/AddPhoneToUsers

# 실행
npx typeorm migration:run -d data-source.ts

# 마지막 마이그레이션 되돌리기
npx typeorm migration:revert -d data-source.ts
```

자동 생성된 마이그레이션:

```typescript
// src/migration/1711700400000-AddPhoneToUsers.ts
import { MigrationInterface, QueryRunner } from 'typeorm';

export class AddPhoneToUsers1711700400000 implements MigrationInterface {
  public async up(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(
      `ALTER TABLE \`users\` ADD \`phone\` varchar(20) NULL`
    );
  }

  public async down(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(
      `ALTER TABLE \`users\` DROP COLUMN \`phone\``
    );
  }
}
```

**TypeORM 실무 주의사항:**

`synchronize: true`는 개발 환경에서만 쓴다. 이 옵션은 앱 시작 시 엔티티와 DB 스키마를 비교해서 자동으로 맞추는데, 컬럼을 삭제하면 데이터도 날아간다. 프로덕션에서 이걸 켜놓고 엔티티 필드 하나 지웠다가 데이터가 사라진 사고가 실제로 있다.

`migration:generate`가 만드는 SQL을 반드시 확인한다. 가끔 불필요한 ALTER가 포함되거나, 의도와 다른 순서로 SQL이 생성되는 경우가 있다.

TypeORM은 `migrations` 테이블에 실행 이력을 저장한다. `migration:revert`는 가장 마지막 마이그레이션만 되돌린다. 여러 개를 되돌리려면 반복 실행해야 한다.

## 마이크로서비스 환경의 스키마 버전 관리

모놀리스에서는 마이그레이션 도구 하나로 전체 스키마를 관리하면 된다. 마이크로서비스에서는 서비스마다 DB가 다르고, 서비스 간 의존 관계가 있어서 복잡해진다.

### 서비스별 독립 마이그레이션

각 서비스는 자기 DB의 마이그레이션만 담당한다. 마이그레이션 파일은 해당 서비스의 리포지토리에 포함한다.

```
order-service/
  src/main/resources/db/migration/
    V1__create_orders_table.sql
    V2__add_shipping_info.sql

user-service/
  src/main/resources/db/migration/
    V1__create_users_table.sql
    V2__add_phone_column.sql
```

버전 번호는 서비스별로 독립적이다. order-service의 V1과 user-service의 V1은 아무 관계가 없다.

배포도 독립적으로 한다. order-service를 배포할 때 user-service의 마이그레이션은 신경 쓸 필요 없다.

### 공유 DB를 분리할 때의 마이그레이션 순서

모놀리스에서 마이크로서비스로 전환할 때, 하나의 DB를 여러 서비스용으로 나누는 작업이 필요하다. 순서를 잘못 잡으면 서비스 장애가 난다.

```
전제: monolith_db에 users, orders, products 테이블이 있고,
     user-service, order-service, product-service로 분리한다.

1단계: 새 DB 생성
  - user_db, order_db, product_db 생성

2단계: 이중 쓰기 (dual-write) 설정
  - 앱에서 monolith_db와 새 DB 양쪽에 쓰기
  - 읽기는 여전히 monolith_db에서

3단계: 기존 데이터 복사
  - monolith_db의 데이터를 새 DB로 일괄 복사
  - 복사 후 데이터 정합성 확인

4단계: 읽기 전환
  - 읽기를 새 DB로 전환
  - monolith_db는 여전히 쓰기용으로 유지 (롤백 대비)

5단계: 쓰기 전환
  - monolith_db로의 쓰기 중단
  - 새 DB만 사용

6단계: monolith_db 정리
  - 분리된 테이블 삭제
```

**서비스 간 JOIN이 필요한 경우:**

공유 DB 시절에는 `users JOIN orders`가 가능했지만, DB를 분리하면 안 된다. 이 부분을 먼저 해결해야 한다.

```
해결 방법:
  - API 호출로 대체: order-service가 user-service API를 호출해서 유저 정보를 가져온다
  - 데이터 복제: 자주 참조하는 데이터(예: user_name)를 orders 테이블에 비정규화한다
  - 이벤트 기반 동기화: user-service가 유저 변경 이벤트를 발행하고,
    order-service가 구독해서 로컬 캐시를 갱신한다
```

DB 분리 전에 크로스 서비스 JOIN을 모두 제거해야 한다. JOIN이 남아있는 상태에서 DB를 분리하면 쿼리가 실패한다.

### 스키마 호환성 계약

서비스 A가 서비스 B의 데이터를 이벤트로 수신하는 경우, B의 스키마가 바뀌면 A도 영향을 받는다. 이벤트 스키마에 버전을 붙여서 관리한다.

```json
// 이벤트 페이로드에 스키마 버전 포함
{
  "event_type": "user.updated",
  "schema_version": "2",
  "data": {
    "user_id": 12345,
    "first_name": "Kim",
    "last_name": "Dev"
  }
}
```

소비자(서비스 A)는 `schema_version`을 보고 파싱 로직을 분기한다. 생산자(서비스 B)는 새 스키마 버전을 발행할 때 일정 기간 동안 구 버전도 함께 발행한다(deprecated period).

## 마이그레이션 실패 시 트러블슈팅

### 체크섬 불일치 복구 (Flyway)

이미 적용된 마이그레이션 파일을 누군가 수정하면 체크섬이 달라지면서 Flyway가 시작을 거부한다.

```
Flyway 에러:
Migration checksum mismatch for migration version 3
-> Applied to database : 1234567890
-> Resolved locally    : 9876543210
```

**원인 파악이 먼저다.** 정말 파일 내용이 바뀌었는지, 아니면 줄바꿈 문자(CRLF/LF) 차이인지 확인한다.

```bash
# git에서 원본 확인
git diff HEAD -- src/main/resources/db/migration/V3__add_phone_to_users.sql

# 줄바꿈 문자 확인
file src/main/resources/db/migration/V3__add_phone_to_users.sql
```

**줄바꿈 문제인 경우:** `.gitattributes`에서 SQL 파일의 줄바꿈을 고정한다.

```
# .gitattributes
*.sql text eol=lf
```

**실제로 파일이 변경된 경우:** 이미 프로덕션에 적용된 상태이면 파일을 원래대로 돌려놓는 게 가장 안전하다. 원본을 복구할 수 없는 상황이면 이력 테이블의 체크섬을 직접 수정한다.

```sql
-- 주의: 프로덕션에서 이 쿼리를 실행하기 전에 반드시 백업
-- 현재 체크섬 확인
SELECT version, checksum, script FROM flyway_schema_history WHERE version = '3';

-- 체크섬 갱신 (로컬 파일의 체크섬으로)
-- Flyway repair 명령이 이 작업을 자동으로 해준다
```

```bash
# repair 명령: 이력 테이블의 체크섬을 현재 파일 기준으로 갱신
./mvnw flyway:repair

# Spring Boot에서
java -jar app.jar --spring.flyway.repair-on-migrate=true  # 위험. 일회성으로만 사용
```

`flyway:repair`는 체크섬을 맞춰줄 뿐, SQL 내용의 차이를 되돌려주지 않는다. DB에 실제로 적용된 SQL과 파일의 SQL이 다른 상태가 되므로, 반드시 현재 DB 스키마가 기대한 상태인지 확인해야 한다.

### 중간 실패 상태 대응

마이그레이션 파일 안에 SQL이 여러 줄인데, 중간에 하나가 실패하면 어떻게 되는가.

**MySQL:** DDL은 암묵적으로 커밋된다(auto-commit). `CREATE TABLE` 후 `ALTER TABLE`이 실패하면, CREATE는 이미 적용된 상태다. Flyway는 이 마이그레이션을 `failed`로 기록한다.

**PostgreSQL:** DDL도 트랜잭션 안에서 실행된다. 하나가 실패하면 전체가 롤백된다. 깔끔하다.

```sql
-- flyway_schema_history에서 실패한 마이그레이션 확인
SELECT version, description, success FROM flyway_schema_history WHERE success = 0;
```

MySQL에서 중간 실패가 발생하면:

```
1. DB에 실제로 어디까지 적용됐는지 확인한다
   - SHOW CREATE TABLE로 현재 상태 확인
   - 실패한 SQL의 에러 메시지 확인

2. 수동으로 나머지를 적용하거나, 이미 적용된 부분을 되돌린다

3. flyway_schema_history에서 실패 기록을 정리한다
   DELETE FROM flyway_schema_history WHERE version = '5' AND success = 0;

4. 마이그레이션 파일을 수정하거나 (아직 다른 환경에 적용 전이면)
   새 파일로 나머지를 처리한다

5. 다시 실행한다
```

이런 상황을 예방하려면 **하나의 마이그레이션 파일에 하나의 DDL만 넣는 게 좋다.** 파일이 많아지더라도 중간 실패를 다루기가 훨씬 수월하다.

### 프로덕션 롤백 실제 절차

마이그레이션을 프로덕션에 적용했는데 문제가 생긴 경우.

**컬럼 추가만 한 경우 (가장 단순):**

신규 컬럼은 앱이 모르면 무시하므로, 구버전 앱으로 롤백하면 끝이다. 컬럼은 나중에 정리하면 된다.

**컬럼 삭제/타입 변경을 한 경우 (위험):**

앱만 롤백하면 안 된다. 구버전 앱이 삭제된 컬럼을 참조하면 에러가 난다.

```
프로덕션 롤백 순서:

1. 서비스 트래픽 차단 (점검 모드 또는 로드밸런서에서 제거)

2. 마이그레이션 전 DB 스냅샷이 있는지 확인
   - 있으면: 스냅샷으로 복원이 가장 빠르고 확실하다
   - 없으면: 역방향 마이그레이션 SQL을 수동 작성해서 실행

3. 역방향 마이그레이션 실행
   - 삭제된 컬럼 복원, 타입 되돌리기 등
   - 데이터 손실 여부 확인 (컬럼을 삭제했으면 데이터는 이미 없다)

4. flyway_schema_history에서 해당 마이그레이션 삭제
   DELETE FROM flyway_schema_history
   WHERE version = '{롤백 대상 버전}';

5. 구버전 앱 배포

6. 정상 동작 확인 후 트래픽 복원
```

**데이터가 손실된 경우 (컬럼 삭제 후 데이터 복구):**

```
컬럼 삭제 시 데이터는 사라진다. 복구하려면:
  - 마이그레이션 전 스냅샷에서 해당 컬럼 데이터를 추출
  - 추출한 데이터를 현재 DB에 INSERT/UPDATE

이게 불가능하면 binlog에서 복구를 시도한다:
  mysqlbinlog --start-datetime="2026-03-29 10:00:00" binlog.000123 > recovery.sql
  이 작업은 DBA와 함께 진행해야 한다.
```

그래서 프로덕션 마이그레이션 전에는 반드시 DB 스냅샷을 생성한다. 롤백이 필요할 때 스냅샷이 없으면 선택지가 크게 줄어든다.

## flyway_schema_history 테이블

Flyway가 자동 생성하는 이력 테이블이다.

```sql
SELECT installed_rank, version, description, type, script, checksum, success
FROM flyway_schema_history
ORDER BY installed_rank;

-- 결과 예시:
-- 1  | 1   | create users table  | SQL | V1__create_users_table.sql  | 12345678 | 1
-- 2  | 2   | create orders table | SQL | V2__create_orders_table.sql | 87654321 | 1
-- 3  | 3   | add phone to users  | SQL | V3__add_phone_to_users.sql  | 11223344 | 1
```
