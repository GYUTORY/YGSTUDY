# EMQX 부하 테스트

[EMQX.md](EMQX.md)에서 리스너 설정, 인증, 클러스터, Rule Engine을 다뤘다. 그 문서는 "어떻게 운영하느냐"고, 이 문서는 "정말 버티느냐를 어떻게 확인하느냐"다. max_connections를 1,024,000으로 박아놨다고 노드 한 대가 백만 커넥션을 받는 게 아니다. 그 숫자는 상한일 뿐이고, 실제로 몇 개까지 붙는지는 부하를 걸어보기 전엔 모른다.

문제는 부하 테스트를 처음 돌려보면 EMQX가 아니라 테스트 클라이언트 쪽이 먼저 죽는다는 것이다. 커넥션 5만 개쯤에서 더 이상 안 붙길래 브로커 한계인 줄 알았는데 알고 보니 클라이언트 머신의 소스 포트가 동난 거였다. 이 문서는 그런 함정을 먼저 정리하고, 그다음 노드 한 대 용량을 측정해서 클러스터 노드 수를 산정하는 순서로 간다.

## emqtt_bench

EMQX 진영에서 공식으로 쓰는 부하 도구가 emqtt_bench다. Erlang으로 짜여 있어서 한 프로세스가 커넥션 수만 개를 들고도 가볍다. JMeter MQTT 플러그인이나 직접 짠 파이썬 스크립트로 동시 커넥션 수만 개를 만들려면 클라이언트 쪽이 먼저 터지는데, emqtt_bench는 그 부담이 작다.

설치는 릴리스 바이너리를 받거나 소스를 빌드한다.

```bash
# 릴리스 바이너리 (Linux x86_64)
wget https://github.com/emqx/emqtt-bench/releases/download/0.4.30/emqtt-bench-0.4.30-ubuntu22.04-amd64.tar.gz
tar zxf emqtt-bench-0.4.30-ubuntu22.04-amd64.tar.gz
cd emqtt-bench && ./bin/emqtt_bench --help
```

Erlang/OTP가 깔린 환경이면 소스에서 빌드해도 된다.

```bash
git clone https://github.com/emqx/emqtt-bench
cd emqtt-bench && make
```

서브커맨드는 세 개를 주로 쓴다. `conn`은 커넥션만 맺고 유지한다(동시 접속 용량 측정용). `pub`은 메시지를 발행한다. `sub`은 구독한다. 처리량을 보려면 pub과 sub을 따로 띄워서 한쪽이 쏘고 한쪽이 받게 한다.

## 동시 커넥션 부하

가장 먼저 보는 건 노드 한 대가 커넥션을 몇 개까지 받느냐다. `conn`으로 커넥션만 쌓는다.

```bash
./bin/emqtt_bench conn \
  -h 10.0.0.10 \
  -p 1883 \
  -c 100000 \
  -i 1 \
  --prefix bench
```

- `-c 100000`: 동시 커넥션 10만 개
- `-i 1`: 커넥션을 1ms 간격으로 맺는다(establish 간격)
- `--prefix bench`: clientid 접두사. 여러 클라이언트 머신을 쓸 때 충돌을 피한다

`-i`를 0으로 두면 10만 개를 최대한 빨리 한꺼번에 맺으려 든다. 이러면 establish rate가 클라이언트 머신 CPU에 막혀서 의미 없는 측정이 된다. 처음엔 1ms 정도로 두고 establish가 안정적으로 올라가는지 본다. 커넥션 establish 자체의 한계를 보려면 그때 `-i`를 줄인다.

여러 머신에서 같은 브로커로 쏠 때는 clientid가 겹치면 안 된다. MQTT는 같은 clientid로 두 번째 접속이 오면 첫 번째를 끊는다(takeover). 부하 테스트 중에 커넥션 수가 올라가다 말고 진동하면 clientid 충돌로 서로 쫓아내는 중일 수 있다. `--prefix`를 머신마다 다르게 준다.

커넥션을 유지한 채로 keepalive PINGREQ만 오가는 상태가 IoT 디바이스의 평상시 모습이다. 대부분의 디바이스는 1초에 한 번 데이터를 쏘지 않는다. 몇 분에 한 번 상태를 보내고 나머지 시간은 그냥 붙어만 있다. 그래서 "동시 커넥션 100만"과 "초당 메시지 100만"은 완전히 다른 부하고, 용량 산정도 따로 해야 한다.

## publish / subscribe 처리량

커넥션이 붙는 것과 메시지가 흐르는 건 별개다. 처리량은 sub을 먼저 띄워 받을 쪽을 만들고, pub으로 쏜다.

받는 쪽(구독자 1,000개가 같은 토픽 패턴 구독):

```bash
./bin/emqtt_bench sub \
  -h 10.0.0.10 \
  -p 1883 \
  -c 1000 \
  -t "bench/%i" \
  -q 1
```

쏘는 쪽(발행자 1,000개가 각자 토픽에 초당 메시지):

```bash
./bin/emqtt_bench pub \
  -h 10.0.0.10 \
  -p 1883 \
  -c 1000 \
  -t "bench/%i" \
  -q 1 \
  -I 1000 \
  -s 256
```

- `-t "bench/%i"`: `%i`가 클라이언트 인덱스로 치환된다. pub의 `bench/0`을 sub의 `bench/0`이 받는다
- `-I 1000`: 메시지 발행 간격 1,000ms(클라이언트당 초당 1건)
- `-s 256`: 페이로드 256바이트
- `-q 1`: QoS 1

여기서 토픽 매칭을 맞추는 게 핵심이다. pub과 sub의 `-c`와 `-t %i`가 짝이 맞아야 발행한 메시지를 구독자가 실제로 받는다. sub 쪽 콘솔에 찍히는 수신 rate가 pub 쪽 발행 rate를 따라오는지 본다. 발행은 초당 10만인데 수신이 8만이면 어딘가에서 메시지가 밀리거나 버려지고 있다는 신호다. QoS 0이면 브로커가 부하 시 그냥 버리고, QoS 1이면 in-flight에 쌓이면서 지연이 늘어난다.

QoS를 바꿔가며 측정하면 차이가 크게 난다. QoS 0은 fire-and-forget이라 처리량이 제일 높다. QoS 1은 PUBACK이 오가서 왕복이 늘고, QoS 2는 4-way 핸드셰이크라 같은 메시지 수에도 패킷이 네 배다. 실제 서비스에서 쓸 QoS로 테스트해야 의미가 있다. QoS 0으로 측정한 처리량을 보고 용량을 잡았다가 운영에서 QoS 1로 돌리면 숫자가 안 맞는다. QoS별 비용 차이는 [EMQX.md의 QoS 2 메시지 적체](EMQX.md#qos-2-메시지-적체) 항목과 같은 맥락이다.

## 커넥션 폭주 재현

운영에서 제일 무서운 건 평상시 부하가 아니라 동시 재접속이다. 통신사 기지국이 잠깐 끊겼다 복구되면, 그 아래 붙어 있던 디바이스 수십만 대가 거의 동시에 재접속을 시도한다. 평상시엔 잘 돌던 브로커가 이 순간 accept 큐가 밀리고 인증 백엔드가 터진다. 이걸 테스트에서 재현해야 한다.

방법은 establish 간격 `-i`를 극단적으로 줄여서 짧은 시간에 커넥션을 몰아넣는 것이다.

```bash
# 평상시: 1ms 간격으로 천천히
./bin/emqtt_bench conn -h 10.0.0.10 -c 200000 -i 1

# 폭주: 간격 없이 최대 속도로 20만 개를 던진다
./bin/emqtt_bench conn -h 10.0.0.10 -c 200000 -i 0
```

폭주 시나리오에서 봐야 할 건 두 가지다. 첫째, establish rate가 어디서 천장을 치는가. 리스너의 `acceptors` 수가 부족하면 여기서 막힌다([EMQX.md 리스너 설정](EMQX.md#리스너-설정)의 acceptors 항목). 둘째, 인증 백엔드가 버티는가. DB 인증을 쓰면 재접속 폭주 동안 인증 쿼리가 초당 수만 번 날아간다. 인덱스 안 걸린 쿼리 하나가 이 순간 DB를 멈춰 세운다.

폭주 테스트는 한 클라이언트 머신으로는 부족한 경우가 많다. 클라이언트 쪽 establish 능력이 먼저 한계라서 브로커를 충분히 못 때린다. 여러 머신에서 동시에 `-i 0`으로 쏴야 진짜 폭주가 재현된다. 머신 5대를 같은 시각에 띄우려면 간단하게 SSH로 동시 실행하거나 `pdsh` 같은 걸 쓴다.

```bash
# 5대 머신에 동시에 던지기 (각 머신 prefix 다르게)
for i in 1 2 3 4 5; do
  ssh bench$i "cd emqtt-bench && ./bin/emqtt_bench conn \
    -h 10.0.0.10 -c 200000 -i 0 --prefix m$i" &
done
```

JWT 인증으로 바꾸면 폭주 내성이 확 좋아진다. JWT는 서명만 검증하고 DB를 안 보니까 재접속 폭주 동안에도 외부 의존이 없다. DB 인증으로 폭주를 못 버티면 JWT 전환을 검토할 단계다.

## 클라이언트 쪽이 먼저 걸린다

여기가 부하 테스트에서 제일 시간 잡아먹는 부분이다. 브로커가 아니라 부하를 거는 쪽 OS 한계에 먼저 막힌다. 측정값이 이상하면 브로커보다 클라이언트를 먼저 의심한다.

### 소스 포트 고갈

이게 가장 먼저, 가장 자주 걸린다. 클라이언트 머신이 브로커로 TCP 커넥션을 하나 맺을 때마다 로컬 소스 포트를 하나 쓴다. 소스 포트 범위는 기본적으로 `net.ipv4.ip_local_port_range`로 정해지고 보통 32768~60999, 즉 약 2만8천 개다. 한 클라이언트 머신에서 한 브로커 IP로 맺을 수 있는 커넥션이 이 범위에 묶인다. 커넥션이 2만8천 개쯤에서 더 안 붙으면 거의 100% 이거다.

```bash
# 현재 범위 확인
cat /proc/sys/net/ipv4/ip_local_port_range
# 32768   60999

# 범위를 넓힌다 (1025~65535, 약 6만4천 개)
sysctl -w net.ipv4.ip_local_port_range="1025 65535"
```

범위를 최대로 넓혀도 한 머신에서 한 브로커 IP로는 6만 개가 천장이다. 출발지 IP 하나 + 목적지 IP/포트 하나 조합에서 구분 가능한 소스 포트가 그게 전부라서다. 그 이상을 한 머신에서 만들려면 클라이언트에 IP를 여러 개 붙이거나(`-S` 옵션으로 소스 IP 지정), 브로커 리스너 포트를 여러 개 두거나, 클라이언트 머신 자체를 여러 대로 늘려야 한다. 백만 커넥션 테스트를 하려면 클라이언트가 최소 20~30대는 있어야 하는 이유가 이거다.

```bash
# 클라이언트에 보조 IP를 여러 개 붙이고 라운드로빈으로 쓰기
./bin/emqtt_bench conn -h 10.0.0.10 -c 100000 \
  --ifaddr 192.168.1.10,192.168.1.11,192.168.1.12
```

### file descriptor 한계

소켓 하나가 fd 하나다. 이건 브로커 쪽([EMQX.md의 file descriptor 한계](EMQX.md#file-descriptor-한계))뿐 아니라 클라이언트 쪽에도 똑같이 적용된다. emqtt_bench를 돌리는 머신의 `ulimit -n`이 1024면 커넥션 1,000개 남짓에서 `Too many open files`가 뜬다.

```bash
# 현재 한계
ulimit -n

# 셸 세션에서 올리기 (root거나 hard limit 안에서)
ulimit -n 1048576

# 영구 설정: /etc/security/limits.conf
# *  soft  nofile  1048576
# *  hard  nofile  1048576
```

systemd로 부하 도구를 띄우면 셸 ulimit이 안 먹으니 unit의 `LimitNOFILE`을 따로 올린다. 커널 전역 한계 `fs.file-max`도 충분히 큰지 본다.

### TIME_WAIT 누적

커넥션을 맺고 끊기를 반복하는 테스트(폭주 시나리오를 여러 번 돌릴 때)는 닫힌 소켓이 TIME_WAIT 상태로 남는다. TIME_WAIT 소켓도 소스 포트를 잡고 있어서, 짧은 시간에 많이 쌓이면 포트가 또 동난다. 새 테스트를 돌리려는데 자꾸 connection 실패가 나면 `ss -s`로 TIME_WAIT 수를 본다.

```bash
ss -s
# TCP:   58000 (estab 200, closed 57000, timewait 56800 ...)

# TIME_WAIT 소켓 재사용 허용 (클라이언트 쪽에서만 신중히)
sysctl -w net.ipv4.tcp_tw_reuse=1
```

`tcp_tw_reuse=1`은 부하 클라이언트 머신에서 켜는 건 괜찮지만, 실제 서비스 서버에서 NAT 뒤 클라이언트가 섞인 환경에 켜면 문제가 생길 수 있으니 테스트 머신에서만 쓴다.

### somaxconn (브로커 쪽)

이건 브로커 쪽 커널 파라미터인데 폭주 테스트와 직결된다. `net.core.somaxconn`은 TCP accept 큐 길이다. establish가 한꺼번에 몰리면 아직 accept되지 않은 커넥션이 이 큐에 쌓이는데, 기본값 128~4096이면 폭주 시 넘쳐서 커넥션이 거부된다.

```bash
# 브로커 노드에서
sysctl -w net.core.somaxconn=65535
sysctl -w net.ipv4.tcp_max_syn_backlog=65535
```

폭주 테스트에서 establish rate가 천장을 치는데 EMQX의 acceptors는 여유가 있다면 somaxconn을 의심한다. 커널 큐에서 막히면 EMQX는 아예 커넥션을 못 보고, dashboard에는 establish가 멈춘 것처럼 보인다.

## TLS 핸드셰이크 영향

평문(1883)으로 측정한 처리량을 TLS(8883)에 그대로 적용하면 안 된다. TLS 핸드셰이크는 비대칭 암호 연산이 들어가서 establish 비용이 평문보다 한 자릿수 이상 비싸다. 평문은 establish가 초당 수만인데 TLS는 같은 노드에서 초당 수천으로 떨어지는 경우가 흔하다.

```bash
# TLS 커넥션 부하
./bin/emqtt_bench conn \
  -h 10.0.0.10 \
  -p 8883 \
  -c 50000 \
  -i 1 \
  --ssl
```

여기서 두 가지를 구분해야 한다. 핸드셰이크 비용(establish 시점에만 든다)과 유지 비용(붙어 있는 동안 든다)이다. 한 번 핸드셰이크가 끝나면 그 뒤 메시지 송수신은 대칭 암호라 평문 대비 오버헤드가 작다. 그래서 커넥션을 천천히 맺어 오래 유지하는 패턴이면 TLS가 동시 커넥션 총량에 주는 영향은 생각보다 작다. 반대로 디바이스가 자주 끊고 재접속하는 패턴이면 매번 핸드셰이크가 일어나서 establish CPU가 노드 용량을 결정한다.

재접속 폭주 + TLS 조합이 최악이다. 수십만 디바이스가 동시에 TLS 핸드셰이크를 하면 CPU가 핸드셰이크 연산에 다 묶여서 establish rate가 바닥을 친다. 이 조합을 쓸 거면 반드시 그 시나리오 그대로 테스트한다. 평문 폭주는 잘 버티는데 TLS 폭주에서 무너지는 노드를 자주 본다.

완화 방법으로 TLS 세션 재사용(session resumption)을 켜면 재접속 시 full handshake를 건너뛴다. 다만 디바이스 클라이언트가 세션 티켓을 저장하고 재사용하도록 구현돼 있어야 효과가 있다. 펌웨어가 매번 새 핸드셰이크를 하면 서버에서 재사용을 켜도 소용없다. 부하 테스트할 때 디바이스의 실제 재접속 동작을 흉내 내야 하는 이유다.

## 측정 중 봐야 할 지표

부하를 거는 동안 클라이언트 콘솔 숫자만 보면 안 된다. 브로커 쪽 dashboard와 Prometheus를 같이 띄워놓고 본다.

dashboard(`http://브로커:18083`) Overview에서 실시간으로 보는 것:

- Connections: 현재 커넥션 수. 목표치까지 선형으로 오르다 어디서 꺾이는지 본다
- Connection establish rate: 초당 신규 커넥션. 폭주 테스트의 핵심 지표다. 여기가 천장을 치는 지점이 노드의 establish 한계다
- Messages in/out: 초당 수신/발행 메시지. pub 부하의 처리량 한계를 본다. in은 받는데 out이 안 따라오면 라우팅이나 구독자 쪽이 병목이다
- Dropped: 버려진 메시지. 0이 아니면 어딘가 과부하다

Prometheus 엔드포인트를 켜서 Grafana로 보면 더 깊이 본다. EMQX.md에서 말한 `/api/v5/prometheus/stats`를 스크레이프한다.

```bash
curl http://10.0.0.10:18083/api/v5/prometheus/stats | grep -E 'emqx_connections_count|emqx_messages_received'
```

부하 중 특히 보는 지표:

- `emqx_connections_count`: 노드별 커넥션 수. 클러스터면 노드 간 분배가 고른지 본다
- `emqx_messages_received` / `emqx_messages_sent`: 메시지 in/out 누적. rate로 환산해서 처리량을 본다
- VM 메모리: Erlang VM 총 메모리. 커넥션 수에 비례해 오르다 갑자기 튀면 세션 큐나 in-flight가 쌓이는 거다([EMQX.md 메모리 사용량 모니터링](EMQX.md#메모리-사용량-모니터링))
- GC / reductions: Erlang VM의 GC 빈도와 reduction 수. 부하가 커지면 GC가 잦아지고, GC가 CPU를 많이 먹기 시작하면 그 노드는 한계에 가깝다. 처리량은 아직 여유 있어 보이는데 응답 지연이 늘면 GC를 의심한다
- CPU per core: Erlang 스케줄러가 코어를 고르게 쓰는지. 특정 코어만 100%면 acceptor나 특정 프로세스에 부하가 쏠린 거다

지표를 볼 때 한 가지 숫자가 아니라 같이 움직이는 패턴을 본다. 커넥션은 계속 느는데 establish rate가 떨어지고 메모리가 가파르게 오르면, 그 노드는 곧 한계다. 그 직전 커넥션 수가 노드 한 대의 실용 용량이다.

## 노드 한 대 용량 측정과 클러스터 산정

부하 테스트의 최종 목적은 "노드 몇 대가 필요한가"에 답하는 거다. 순서는 이렇다.

먼저 노드 한 대에 커넥션을 천천히 쌓으면서 메모리/CPU/establish가 꺾이는 지점을 찾는다. 예를 들어 16코어 32GB 노드에서 커넥션 80만 개까지는 메모리가 선형으로 오르다가 그 근처에서 GC가 잦아지고 establish가 떨어진다고 하자. 그러면 이 노드의 실용 용량은 안전 마진을 빼고 60만 개쯤으로 잡는다. 천장이 80만이라고 80만으로 운영하면 재접속 폭주 한 번에 넘어간다. 평상시 용량은 천장의 70% 안쪽으로 잡는 게 안전하다.

커넥션 용량과 메시지 처리량을 따로 측정해야 한다. 같은 노드가 "커넥션 60만 유지"는 되는데 "초당 메시지 30만 처리"는 안 될 수 있다. 둘은 다른 자원을 쓴다. 커넥션은 메모리와 fd, 메시지는 CPU와 네트워크다. 실제 디바이스의 메시지 빈도(몇 분에 한 번인지, 초당 몇 건인지)를 알아야 어느 쪽이 먼저 한계인지 안다. 디바이스 100만 대인데 각자 5분에 한 번 보내면 평균 초당 약 3,300건이라 처리량은 여유고 커넥션 유지가 관건이다. 반대로 디바이스 1만 대가 각자 초당 10건이면 커넥션은 적은데 처리량이 관건이다.

용량이 나오면 디바이스 총수를 노드당 실용 용량으로 나눈다. 디바이스 300만 대, 노드당 60만이면 커넥션 노드 5대가 최소다. 여기에 노드 하나 죽어도 버티게 +1, 폭주 시 잠깐의 추가 부하를 위해 여유를 더한다. core-replicant 구조에서 이 커넥션 노드들은 replicant로 두고, core는 별도로 3~5대를 둔다([EMQX.md core-replicant 아키텍처](EMQX.md#core-replicant-아키텍처)). 커넥션 용량을 늘릴 땐 replicant만 추가하면 되니까, 위에서 잰 "노드 한 대 용량"이 곧 replicant 한 대 용량이고, 그걸로 replicant 수를 산정한다.

마지막으로 산정한 노드 수로 클러스터를 실제로 띄워서 전체 부하를 다시 건다. 노드 한 대 측정값을 단순히 곱하면 안 되는 이유가 있다. 클러스터는 노드 간 라우팅 테이블 동기화 비용이 추가된다. 구독이 많고 토픽이 노드 간에 흩어져 있으면 메시지가 노드를 건너 라우팅되면서 단일 노드 측정값보다 처리량이 떨어진다. 그래서 산정한 구성 그대로 한 번은 전체 부하 테스트를 돌려서 실제로 목표 커넥션과 처리량이 나오는지 확인하고, 안 나오면 노드를 더하거나 토픽 설계를 손본다.

## 테스트를 돌리는 순서

정리하면 부하 테스트는 대체로 이 순서로 한다.

클라이언트 머신부터 튜닝한다. ulimit, ip_local_port_range, tcp_tw_reuse를 올리지 않으면 브로커를 측정하기도 전에 클라이언트가 먼저 죽어서 시간만 버린다. 그다음 브로커 노드의 somaxconn, fd, acceptors를 올린다. 이게 안 돼 있으면 측정값이 커널 한계에 막힌 가짜 숫자다.

준비가 되면 단일 노드에 커넥션을 천천히 쌓아 용량 천장을 찾고, 따로 처리량 천장을 찾는다. 그다음 establish 간격을 0으로 줄여 재접속 폭주를 재현하고, 실제 쓸 QoS와 TLS 설정 그대로 다시 측정한다. 마지막으로 산정한 노드 수로 클러스터를 띄워 전체 부하로 검증한다.

이 과정에서 dashboard와 Grafana를 계속 띄워놓고 establish rate, 메시지 in/out, 메모리, GC를 같이 본다. 클라이언트 콘솔 숫자만 믿으면 클라이언트 병목을 브로커 한계로 오해한다.
