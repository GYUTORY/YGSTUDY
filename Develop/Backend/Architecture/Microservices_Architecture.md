# 마이크로서비스 아키텍처

## 시작하기 전에

MSA가 유행했던 시기에 운영팀이 가장 많이 들었던 말이 "왜 우리는 모놀리스인가요"였다. 5년 정도 지나고 나니 분위기가 바뀌었다. 분산 모놀리스로 굳어버린 시스템을 다시 합치자는 PR이 올라온다. MSA는 만능이 아니다. 모놀리스의 한계를 분명히 겪고, 조직이 서비스를 따로 운영할 준비가 됐을 때 비로소 의미가 있다.

이 문서는 MSA의 정의를 늘어놓기보다, 서비스를 나누는 기준과 실패하기 쉬운 지점을 정리한다. 작은 팀이 MSA에 손댔다가 한 달 만에 빌드 시스템과 인프라 운영에 시간을 다 쓰고 코드를 거의 못 짠 사례, 결제팀과 주문팀이 같은 DB를 공유하다가 결국 한쪽이 다른 쪽의 트랜잭션 락에 30분씩 묶이던 사례, API Gateway 한 개에 라우팅 룰 2,000개가 박혀서 배포가 두려워진 사례를 떠올리며 읽으면 된다.

## MSA가 해결하는 문제

MSA는 코드를 잘게 쪼개기 위해 만든 게 아니다. 조직을 잘게 쪼개기 위해 만든 거다. Conway's Law를 뒤집어 적용한 결과물이라고 보는 게 정확하다. 한 팀이 한 서비스의 코드 변경부터 배포·운영까지 책임지면 의사결정 속도가 빨라진다. 다른 팀과 PR을 두고 다툴 일이 줄고, 자기 서비스만 잘 굴리면 된다.

문제는 이 조직 구조 자체가 비싼 전제라는 점이다. 10명이 한 모놀리스를 만들고 있다면, 그 10명을 3개 팀으로 쪼개도 각 팀은 3~4명짜리 작은 팀이다. 각 팀이 별도 인프라, 별도 배포 파이프라인, 별도 모니터링을 가져야 한다면 운영 부담이 코드 작성 시간을 잡아먹는다. 일반적으로 백엔드 인원이 15~20명을 넘기기 전에는 MSA의 운영 비용이 이득을 넘어선다.

기술적 효용도 있다. 서비스마다 다른 언어, 다른 DB를 쓸 수 있고, 트래픽이 몰리는 서비스만 따로 스케일아웃할 수 있다. 다만 이 효용은 첫 번째 효용에 비하면 부차적이다. 트래픽이 정말로 한 서비스에만 몰리는지, 언어를 정말로 바꿔야 하는지 따져보면 "그렇진 않다"인 경우가 많다.

## 서비스 경계를 어디서 끊을 것인가

MSA에서 가장 어려운 결정이다. 잘못 끊으면 두 서비스가 영원히 같이 배포되거나, 한 도메인 작업을 위해 5개 서비스를 동시에 수정해야 한다. 경계가 흐릿한 채로 시작한 MSA는 거의 분산 모놀리스로 끝난다.

### DDD의 Bounded Context와 일치시킨다

기준이 모호할 때 가장 신뢰할 만한 출발점은 도메인 주도 설계의 Bounded Context다. 한 컨텍스트 안에서는 같은 용어가 같은 의미를 갖는다. "주문"이라는 단어가 주문 컨텍스트와 정산 컨텍스트에서 다른 의미를 가진다면, 두 컨텍스트는 서로 다른 서비스 후보다.

쇼핑몰을 예로 들면 대략 이런 분리가 자연스럽다.

- Catalog (상품 카탈로그, 가격 정보)
- Cart (장바구니)
- Order (주문 생성, 주문 상태)
- Payment (결제 수단, 결제 처리)
- Shipping (배송 처리, 배송 추적)
- Inventory (재고 차감, 재고 조회)
- User (회원, 권한)

이 분리가 절대 기준은 아니다. 회사마다 핵심 도메인이 다르다. 어떤 회사는 Catalog와 Inventory가 사실상 한 팀이 보는 한 도메인일 수 있고, 어떤 회사는 Order 안에 Payment와 Shipping까지 들어있는 형태가 자연스러울 수 있다. 도메인 전문가와 이야기하면서 용어가 갈리는 지점을 찾는 게 먼저다.

### 서비스 크기를 잡는 실용적 기준

"마이크로"라는 이름 때문에 작을수록 좋다고 오해하는 사람이 많다. 작은 게 미덕인 게 아니다. 실용적 기준은 이렇다.

- 한 팀(3~8명)이 온전히 책임질 수 있는가
- 데이터베이스를 독립적으로 가질 수 있는가 (공유 DB가 필요하면 경계가 잘못 잡힌 신호다)
- 다른 서비스와 무관하게 배포할 수 있는가
- 장애가 났을 때 다른 서비스가 살아있을 의미가 있는가

이 네 가지가 모두 맞으면 적절한 크기다. 한 팀이 서비스를 5개씩 들고 있다면 너무 쪼갠 거다. 반대로 한 서비스를 3개 팀이 같이 만지고 있다면 덜 쪼갠 거다.

서비스 크기를 코드 라인 수나 엔드포인트 수로 재는 건 의미가 없다. 100줄짜리 서비스도 있을 수 있고, 5만 줄짜리 서비스도 있을 수 있다. 도메인의 복잡도가 결정한다.

### 자주 보는 경계 실패

가장 흔한 실패는 기술 계층으로 서비스를 나누는 거다. "Controller 서비스", "DB 서비스", "캐시 서비스" 같은 분리는 분산 모놀리스의 지름길이다. 한 요청을 처리하려고 모든 서비스를 거쳐야 하고, 결국 하나만 죽어도 전체가 죽는다. 경계는 반드시 비즈니스 도메인 기준이어야 한다.

두 번째 실패는 CRUD 단위 분리다. "User 생성 서비스", "User 조회 서비스", "User 수정 서비스"로 나누면 같은 데이터를 만지는 코드가 세 군데에 흩어진다. 한 도메인의 모든 동작은 한 서비스 안에 두는 게 기본이다.

세 번째는 도메인이 충분히 안정화되기 전에 쪼개는 거다. 신규 서비스 도메인은 아직 경계를 모른다. 모놀리스로 시작해서 6개월~1년 굴려본 다음에 경계가 보이면 그때 쪼개는 게 낫다.

## 서비스 간 통신

서비스를 쪼개고 나면 통신 방식을 정해야 한다. 동기 호출과 비동기 메시징, 둘 다 쓸 자리가 있다. 어느 한쪽으로만 통일하면 반드시 어색해지는 지점이 생긴다.

### 동기 통신: REST / gRPC

요청자가 응답을 기다려야 하는 경우다. 사용자가 화면에서 클릭하고 결과를 보는 흐름은 대부분 동기다. 주문 화면에서 "재고 있어요?"를 물어보는 호출이 대표적이다.

REST(HTTP/JSON)는 디버깅이 쉽고 어디서나 통한다. gRPC는 스키마가 명확하고 빠르다. 사내 서비스 간이면 gRPC, 외부 노출이면 REST가 일반적이다.

```
[Client] ──HTTP──> [API Gateway] ──gRPC──> [Order Service] ──gRPC──> [Inventory Service]
                                                │
                                                └──gRPC──> [User Service]
```

동기 호출은 직관적이지만 한 가지 함정이 있다. 호출 체인이 길어질수록 전체 응답시간이 늘고, 어느 한 서비스라도 죽으면 전체 흐름이 깨진다. Order가 Inventory를 부르고, Inventory가 Warehouse를 부르고, Warehouse가 Stock을 부르는 식이면 Stock 하나의 P99 지연이 전체를 무너뜨린다. 동기 호출 체인은 깊을수록 위험하다. 보통 2~3홉을 넘으면 설계가 잘못된 신호다.

타임아웃, 리트라이, 서킷 브레이커가 반드시 필요하다. 타임아웃 없이 호출하다가 한 서비스가 느려지면 호출하는 쪽 스레드 풀이 다 막혀서 같이 죽는다. 이건 MSA 운영 첫 달에 한 번씩은 겪는 사고다.

### 비동기 통신: 메시지 큐 / 이벤트

응답이 즉시 필요 없는 작업은 메시지로 던지는 게 낫다. 주문이 들어오면 "주문 생성됨" 이벤트만 발행하고, 결제·재고·배송 서비스가 각자 그 이벤트를 받아서 처리한다. 발행자는 누가 받는지 알 필요 없다.

```
[Order Service] ──publish "OrderCreated"──> [Kafka]
                                              │
                          ┌───────────────────┼───────────────────┐
                          ↓                   ↓                   ↓
                  [Payment Service]   [Inventory Service]   [Shipping Service]
```

이벤트 기반은 서비스 간 결합도를 낮춘다. 새로운 서비스가 생겨도 발행자 쪽 코드를 바꿀 필요가 없다. 결제 서비스가 잠시 다운돼도 이벤트는 큐에 쌓여있다가 복구되면 처리된다.

대신 디버깅이 어렵다. 한 흐름이 5개 서비스를 거치고 그 사이에 큐가 끼어있으면, 어디서 흐름이 끊겼는지 추적하기가 까다롭다. 분산 트레이싱(OpenTelemetry, Jaeger, Zipkin)이 사실상 필수다.

중복 메시지 처리도 골치 아픈 문제다. Kafka든 RabbitMQ든 대부분 "at-least-once" 전달을 보장한다. 같은 메시지가 두 번 들어올 수 있다는 뜻이다. 컨슈머는 멱등(idempotent)하게 짜야 한다. 이벤트 ID를 받아서 이미 처리한 이벤트면 skip하는 식이다.

```
def handle_order_created(event):
    if processed_events.exists(event.id):
        return
    process(event)
    processed_events.save(event.id)
```

### 동기와 비동기를 섞어 쓰기

실무에서는 한 시스템 안에 두 방식이 같이 굴러간다. 사용자가 직접 결과를 보는 흐름은 동기, 백그라운드 처리는 비동기로 두는 게 일반적이다.

예를 들어 주문 생성은 동기로 처리한다. "주문이 접수되었습니다" 화면을 즉시 보여줘야 하기 때문이다. 다만 주문 생성 안에서 결제, 재고, 배송을 모두 동기로 부르면 응답시간이 폭발한다. 주문 자체는 동기로 처리하되, 결제·재고·배송은 "OrderCreated" 이벤트를 발행하고 각자 처리하게 한다.

## 데이터 소유권과 DB-per-service

MSA에서 가장 자주 깨지는 원칙이 데이터 분리다. 코드는 쪼갰는데 DB는 그대로 공유하는 경우가 많다. 이건 MSA가 아니다. 분산 모놀리스다.

### 왜 DB를 공유하면 안 되는가

A 서비스와 B 서비스가 같은 테이블을 쓴다고 가정하자. A가 컬럼 하나를 추가하고 싶다. B의 동의가 필요하다. B가 안 쓰는 컬럼인데도 마이그레이션을 같이 배포해야 한다. 결국 두 서비스는 같이 배포된다. 독립 배포가 안 되니 MSA의 핵심 이점이 사라진다.

성능 문제도 있다. B가 큰 쿼리를 돌려서 락을 걸면 A의 트랜잭션이 줄줄이 대기한다. 한쪽이 다른 쪽의 문제를 그대로 받는다.

### DB-per-service 원칙

각 서비스는 자기 DB를 가진다. 다른 서비스의 DB에 직접 접근하지 않는다. 데이터가 필요하면 그 서비스의 API를 호출하거나 이벤트로 받는다.

```
[Order Service]      [Payment Service]    [Inventory Service]
       │                     │                     │
       ↓                     ↓                     ↓
   order_db            payment_db            inventory_db
```

DB는 물리적으로 다를 수도 있고, 같은 DB 서버 안의 다른 스키마일 수도 있다. 시작 단계에서는 같은 서버의 다른 스키마로 두고, 트래픽이 커지면 분리하는 게 현실적이다. 처음부터 DB 인스턴스를 서비스마다 따로 띄우는 건 운영 부담이 크다.

### 데이터 중복은 받아들인다

한 서비스에서 다른 서비스의 데이터가 필요할 때, 매번 API 호출로 가져오면 성능이 안 나온다. 자주 쓰는 데이터는 로컬에 복제해서 쓴다.

예를 들어 주문 서비스가 사용자 이름을 필요로 한다. 매번 User 서비스를 호출하기는 부담스럽다. User 서비스가 "UserUpdated" 이벤트를 발행하고, 주문 서비스가 그 이벤트를 받아서 자기 DB에 사용자 이름을 캐싱해두는 방식이 일반적이다.

이걸 데이터 중복으로 보면 어색하지만, MSA에서는 데이터 일관성을 부분적으로 포기하고 가용성을 얻는 거다. 사용자 이름이 바뀌고 주문 서비스에 반영되기까지 약간의 지연이 있을 수 있다. 받아들일 수 있는 수준의 일관성 약화다(eventual consistency).

### 데이터 조회의 어려움

여러 서비스에 흩어진 데이터를 한 화면에 보여줘야 할 때가 문제다. 모놀리스라면 JOIN 한 번이면 끝나는 쿼리가 MSA에서는 여러 서비스 호출로 흩어진다.

세 가지 접근이 있다.

첫째, API Composition. 호출하는 쪽이 여러 서비스를 부르고 결과를 합친다. 간단하지만 호출 수가 많고 응답시간이 늘어난다.

둘째, CQRS + Materialized View. 조회 전용 DB를 만들고, 각 서비스의 이벤트를 받아서 이 DB에 미리 합쳐둔다. 조회 성능은 빠르지만 인프라가 늘어난다.

셋째, BFF에서 합치기. API Gateway나 BFF 계층에서 여러 서비스를 호출해 합쳐서 응답한다. 클라이언트 입장에서는 단일 호출처럼 보인다.

어느 쪽이든 모놀리스의 JOIN보다는 비싸다. MSA로 가면 조회 패턴부터 다시 설계해야 한다.

## 분산 트랜잭션을 피하는 방법

여러 서비스가 한 트랜잭션에 묶이면 일이 복잡해진다. 2PC(Two-Phase Commit) 같은 분산 트랜잭션 프로토콜이 있긴 하다. 다만 실무에서는 거의 안 쓴다. 한 서비스가 락을 걸고 다른 서비스의 응답을 기다리는 구조라 가용성이 크게 떨어지고, MSA가 추구하는 독립성과 정면으로 충돌한다.

### Saga 패턴

대신 Saga 패턴을 쓴다. 큰 트랜잭션을 여러 개의 로컬 트랜잭션으로 쪼개고, 각 단계가 실패하면 앞서 성공한 단계를 보상 트랜잭션으로 되돌린다.

주문 생성을 예로 보면 이렇다.

```
1. Order 서비스: 주문 생성 (PENDING)
2. Payment 서비스: 결제 처리
3. Inventory 서비스: 재고 차감
4. Shipping 서비스: 배송 예약
5. Order 서비스: 주문 확정 (CONFIRMED)
```

3번에서 재고가 부족하면 결제를 환불(보상 트랜잭션)하고 주문을 취소한다. 모든 단계가 보상 트랜잭션을 갖고 있어야 한다.

Saga는 두 가지 방식이 있다.

Choreography는 각 서비스가 이벤트만 듣고 자기 일을 한다. 중앙 조율자가 없다. 단순하지만 흐름 전체를 한 곳에서 보기 힘들다. 서비스가 늘어나면 흐름 추적이 어려워진다.

Orchestration은 중앙 조율자(Saga Orchestrator)가 각 단계를 호출한다. 흐름이 한 곳에 모여 있어서 이해하기 쉽다. 다만 조율자가 또 하나의 서비스가 된다.

3~4개 서비스 정도면 Choreography로도 견딘다. 5개 이상이 한 흐름에 묶이면 Orchestration이 낫다.

### Saga 운영의 어려움

Saga는 트랜잭션을 흉내 내지만 진짜 트랜잭션은 아니다. 격리성(Isolation)이 없다. 중간 상태가 외부에 보인다. 결제는 됐는데 재고 차감이 아직 안 된 짧은 순간이 존재한다. 이 사이에 다른 트랜잭션이 끼어들면 이상한 상태가 만들어진다.

보상 트랜잭션도 항상 가능한 게 아니다. 이메일을 이미 보내버렸으면 "안 보낸 척"할 수 없다. 외부 시스템 호출이 들어간 단계는 보상이 어렵다. 보상 불가능한 단계는 마지막에 두는 게 원칙이다.

장애 복구도 까다롭다. Saga 중간에 서비스가 죽으면 어디까지 진행됐는지 알아야 다시 시작할 수 있다. Saga 상태를 영속화하는 저장소가 필요하다.

## 서비스 디스커버리

서비스 인스턴스가 여러 개 떠 있고, 그 인스턴스들이 오토스케일링으로 늘었다 줄었다 한다. 어느 인스턴스를 호출할지 어떻게 알 수 있나. 이게 서비스 디스커버리 문제다.

### 클라이언트 사이드 vs 서버 사이드

클라이언트 사이드 디스커버리는 호출하는 쪽이 서비스 레지스트리(Eureka, Consul 등)에 물어보고 직접 인스턴스를 고른다. Netflix가 OSS로 풀던 시절의 방식이다.

서버 사이드 디스커버리는 호출하는 쪽이 그냥 고정된 주소를 부른다. 그 주소가 로드밸런서나 프록시고, 로드밸런서가 뒤에 있는 인스턴스로 분배한다.

쿠버네티스가 표준이 된 지금은 거의 다 서버 사이드다. Service라는 추상화가 디스커버리와 로드밸런싱을 같이 처리한다. `http://order-service`로 호출하면 쿠버네티스가 알아서 라우팅한다. 별도의 레지스트리 인프라가 필요 없다.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: order-service
spec:
  selector:
    app: order
  ports:
    - port: 8080
```

쿠버네티스를 안 쓰는 환경이면 Consul이나 Nomad, 혹은 클라우드 제공자의 서비스 디스커버리(AWS Cloud Map 등)를 쓴다.

### 서비스 메시

서비스가 많아지면 서비스 간 호출 자체를 인프라로 추상화하고 싶어진다. 트래픽 라우팅, 리트라이, 서킷 브레이커, mTLS, 트레이싱 같은 게 매 서비스마다 코드로 들어가면 부담이 크다.

서비스 메시(Istio, Linkerd 등)는 각 서비스 옆에 사이드카 프록시를 띄워서 이걸 다 인프라 레벨에서 처리한다. 코드는 그냥 평범한 HTTP 호출만 한다. 프록시가 알아서 mTLS도 걸고, 리트라이도 하고, 트레이싱도 남긴다.

서비스 메시는 강력하지만 운영 복잡도가 또 한 단계 올라간다. 서비스 10개 미만이면 도입 부담이 효용보다 크다. 보통 30~50개 서비스를 넘기 시작할 때 진지하게 검토한다.

## API Gateway와 BFF

클라이언트가 서비스 인스턴스를 직접 부르는 건 좋지 않다. 클라이언트가 모든 서비스의 위치를 알아야 하고, 인증·로깅·rate limiting을 각 서비스마다 따로 구현해야 한다.

### API Gateway가 하는 일

API Gateway는 외부 트래픽의 단일 진입점이다.

- 라우팅: 경로별로 적절한 서비스로 보낸다 (`/orders/*` → Order 서비스).
- 인증/인가: JWT 검증, OAuth, API Key 처리.
- Rate Limiting: 사용자당, IP당 요청 제한.
- 로깅과 모니터링: 모든 트래픽이 한 곳을 지나가니까 관찰하기 쉽다.
- 응답 변환: 내부 gRPC를 외부 REST로 바꾸거나, 여러 서비스 응답을 합치기도 한다.

Kong, Nginx, AWS API Gateway, Spring Cloud Gateway, Envoy 같은 게 흔히 쓰인다.

### Gateway가 비대해지는 함정

API Gateway가 편하다고 비즈니스 로직까지 거기에 넣기 시작하면 또 다른 모놀리스가 된다. Gateway에 라우팅 룰이 2,000개 쌓이고, 응답 변환 로직이 복잡해지고, 결국 Gateway를 만지려면 모든 팀이 모여야 한다. Gateway 배포가 두려워지는 순간 설계가 잘못된 거다.

Gateway는 얇아야 한다. 비즈니스 로직은 서비스에 두고, Gateway는 라우팅과 횡단 관심사(인증·로깅·rate limit)만 본다.

### BFF: 클라이언트별 게이트웨이

웹과 모바일이 필요한 데이터가 다른 경우가 많다. 웹은 한 화면에 정보를 풍부하게 보여주고, 모바일은 데이터 사용량을 줄이기 위해 필요한 것만 보여준다. 같은 API로 양쪽을 다 만족시키려면 어색해진다.

BFF(Backend for Frontend)는 클라이언트마다 전용 게이트웨이를 두는 패턴이다.

```
[Web Client] ──> [Web BFF] ──┐
                              ├──> [Order Service]
[Mobile Client] ──> [Mobile BFF] ──> [Product Service]
                              │
[Admin Client] ──> [Admin BFF] ──> [User Service]
```

각 BFF가 자기 클라이언트가 필요한 데이터만 추려서 합쳐 보내준다. 웹 BFF는 한 화면에 필요한 5개 서비스 호출을 합쳐서 한 번에 응답하고, 모바일 BFF는 필요한 필드만 추려서 응답한다.

BFF의 함정도 있다. BFF가 너무 많아지면 유지비가 커진다. 클라이언트가 2~3종이고 차이가 크면 도입할 만하지만, 클라이언트가 비슷비슷한데도 BFF를 굳이 나누는 건 과하다.

## 흔히 겪는 실패 패턴

MSA 도입에서 반복되는 실수가 있다. 이걸 피하기만 해도 절반은 성공이다.

### 분산 모놀리스

가장 흔한 실패다. 코드는 여러 서비스로 쪼개졌는데 실제로는 한 덩어리처럼 움직인다. 한 기능을 만들려면 5개 서비스를 동시에 수정하고 동시에 배포해야 한다.

원인은 보통 경계 설정 실패다. 도메인이 아니라 기술 계층이나 CRUD 단위로 나눴거나, 한 도메인을 두 서비스가 같이 소유하고 있다. 서비스 간 결합도가 모놀리스 안의 클래스 간 결합도보다 높다.

증상이 보이면 빨리 인정하고 합치는 게 낫다. 같이 변하는 서비스는 같이 있어야 한다. 분산된 채로 두면 운영 비용만 늘어난다.

### 공유 DB

설명한 대로다. 서비스를 쪼개도 DB를 같이 쓰면 MSA가 아니다. DB 분리 없이 시작한 MSA는 무조건 분산 모놀리스가 된다.

마이그레이션이 부담스러워서 미루는 경우가 많은데, 시작 시점에 분리하는 것보다 나중에 분리하는 게 훨씬 비싸다. MSA를 시작할 거면 DB 분리부터 한다.

### 동기 호출 체인

서비스 A가 B를 부르고, B가 C를 부르고, C가 D를 부르고... 이런 체인이 길어지면 응답시간이 폭발하고 장애 전파가 빨라진다. D 하나가 느려지면 A의 응답시간이 같이 늘어난다. D가 죽으면 A가 사용자에게 에러를 던진다.

체인이 2~3홉을 넘기 시작하면 설계를 다시 봐야 한다. 비동기로 바꿀 수 있는 부분이 있는지, 한 서비스에 책임을 모을 수 있는지 검토한다.

### 인프라 준비 없이 시작

MSA는 인프라가 받쳐줘야 굴러간다. CI/CD 파이프라인이 서비스마다 따로 돌아야 하고, 로그가 한 곳에 모여야 하고, 분산 트레이싱이 있어야 하고, 모니터링이 서비스별로 분리돼 있어야 한다.

이걸 다 갖추지 않고 서비스만 쪼개면 운영팀이 고통받는다. 장애가 나도 어디가 문제인지 찾는 데 시간이 다 간다. 인프라 준비 → 모놀리스에서 한 서비스 분리 → 운영 검증 → 다음 서비스 분리 순서가 안전하다.

### 너무 작은 서비스

"마이크로"라는 말에 현혹돼서 함수 단위로 서비스를 만드는 경우가 있다. 그러면 서비스 수가 빠르게 100개를 넘어가고, 서비스 간 통신이 시스템의 대부분을 차지한다.

서비스 하나가 너무 작으면 합친다. 한 팀이 5개 이상의 서비스를 들고 있고, 그 서비스들이 항상 같이 배포된다면 합치는 게 낫다.

### 모든 팀이 다른 기술 스택

언어를 자유롭게 선택할 수 있다는 게 MSA의 장점으로 자주 인용된다. 다만 실제로 그렇게 했다가는 운영팀과 인프라팀이 무너진다. Go, Java, Python, Node, Rust 서비스가 한 회사 안에 있으면 각 언어의 라이브러리·보안 패치·관찰 도구를 모두 관리해야 한다.

기본 스택을 1~2개로 정하고, 특별한 이유가 있을 때만 다른 언어를 쓰는 게 낫다. 자유는 비싼 비용을 동반한다.

## 모놀리스에서 MSA로 이행

기존 모놀리스를 MSA로 옮기는 작업은 빅뱅으로 하면 거의 실패한다. 점진적으로 가야 한다.

### Strangler Fig 패턴

식물이 큰 나무를 천천히 감싸서 결국 그 자리를 차지하는 데서 따온 이름이다. 모놀리스를 둘러싸고 새 서비스를 하나씩 추가해서 점진적으로 트래픽을 옮긴다.

```
처음:
[Client] ──> [Monolith]

중간:
[Client] ──> [Gateway] ──> [Monolith]
                  │
                  └──> [New Service A]

나중:
[Client] ──> [Gateway] ──┬──> [Service A]
                          ├──> [Service B]
                          └──> [Service C (이전 모놀리스)]
```

API Gateway를 앞에 두고, 새로 떼어낸 기능은 새 서비스로 보내고 나머지는 모놀리스로 보낸다. 떼어낼 기능을 하나씩 늘려가면서 모놀리스를 줄여나간다.

### 무엇부터 떼어낼 것인가

가장 안 변하는 부분, 가장 잘 격리된 부분부터 떼어내는 게 안전하다. 인증, 알림 발송, 파일 업로드 같은 서비스가 후보다. 반대로 도메인의 핵심에 가까운 부분(주문, 결제)은 의존성이 많아서 어렵다.

처음 1~2개 서비스를 떼어내는 동안 인프라가 만들어진다. CI/CD, 로깅, 모니터링이 자리잡으면 다음 서비스를 떼어내는 비용이 줄어든다.

### 멈출 줄도 알아야 한다

이행 도중에 "여기까지가 적당한 지점"이라고 판단되면 멈춰도 된다. 모놀리스 + 몇 개의 마이크로서비스 형태로 안정적으로 굴러가는 회사도 많다. 핵심은 비즈니스가 요구하는 만큼 분리하는 거지, 모든 걸 다 쪼개는 게 아니다.

## 정리하며

MSA는 도구이지 목표가 아니다. 모놀리스가 한계에 부딪혔을 때, 조직이 서비스를 독립적으로 운영할 준비가 됐을 때, 그때 도입하는 게 맞다. 유행이라서, 이력서에 쓰고 싶어서, 마이크로서비스가 모던해 보여서 도입한 MSA는 운영 부담만 늘리고 끝난다.

쪼갰다면 끝까지 쪼개야 한다. 코드만 쪼개고 DB는 공유하는 어정쩡한 상태가 가장 나쁘다. 서비스 경계를 도메인 기준으로 잡고, DB를 분리하고, 동기 호출 체인을 줄이고, 비동기 메시징을 잘 쓰는 게 기본이다. 그러고 나서야 API Gateway, 서비스 디스커버리, Saga 같은 패턴이 의미를 갖는다.

처음부터 완벽한 MSA를 만들려고 하지 않는다. 모놀리스로 시작해서 경계가 보이면 떼어내는 식으로 가는 게 현실적이다. 5년 뒤에 후회하지 않을 결정은 보통 그쪽이다.
