# 이벤트 기반 아키텍처

## 시작하기 전에

MSA를 도입한 팀이 곧이어 마주치는 게 EDA다. 동기 REST 호출로 묶여있던 서비스들이 늘어나면 호출 그래프가 거미줄처럼 꼬이고, 결제 한 번 처리하려고 7개 서비스가 동기로 줄줄이 호출되는 상황이 만들어진다. 그러다 한 서비스가 200ms씩 느려지면 전체 응답시간이 1.5초가 된다. 이 시점에서 누군가 "이거 Kafka 쓰면 되지 않나요"라고 말하면서 EDA가 시작된다.

문제는 EDA가 동기 호출의 문제를 해결하는 동시에 새로운 문제를 잔뜩 가져온다는 거다. 메시지 중복, 순서 보장, 스키마 깨짐, 소비자 lag, 멱등성 처리, 컨슈머 재처리. 이 문서는 EDA를 도입할 때 부딪히는 실제 문제들을 정리한다. Kafka를 깔고 토픽 5개 만든 다음에 발견하는 함정들이다.

## 이벤트, 메시지, 커맨드

용어부터 정리한다. 이 셋을 섞어 쓰면 시스템 설계 회의에서 30분씩 헛돈다.

**커맨드(Command)**는 특정 수신자에게 어떤 행동을 하라고 지시하는 거다. "결제를 처리해라(`ProcessPayment`)"가 커맨드다. 보내는 쪽은 받는 쪽이 누구인지 알고, 받는 쪽은 그 명령을 거부할 수도 있다. 일반적으로 1:1 통신이고, 실패할 수 있다는 전제가 깔린다.

**이벤트(Event)**는 이미 일어난 사실을 알리는 거다. "결제가 완료되었다(`PaymentCompleted`)"가 이벤트다. 보내는 쪽은 누가 받든 관심이 없고, 받는 쪽이 0명일 수도 있다. 과거형으로 표현되고, 거부할 수 없다(이미 일어난 일이다). 1:N 통신이 자연스럽다.

**메시지(Message)**는 이 둘을 포함하는 상위 개념이다. 큐나 토픽을 통해 전달되는 모든 데이터 단위를 가리킨다. RabbitMQ는 메시지 큐이고 Kafka는 이벤트 스트림이지만, 둘 다 메시지를 다룬다고 표현해도 틀리지 않는다.

이 구분이 왜 중요하냐면, 커맨드를 이벤트처럼 다루거나 이벤트를 커맨드처럼 다루면 결합도가 어긋난다. 예를 들어 주문 서비스가 `OrderCreated` 이벤트를 발행했는데, 알림 서비스가 이걸 받아서 "주문자에게 이메일 전송"이라는 작업을 한다고 치자. 여기까지는 정상이다. 그런데 누군가 주문 서비스에서 `SendEmail`이라는 이벤트를 발행하면 안 된다. 이건 이벤트가 아니라 알림 서비스에 보내는 커맨드다. 주문 서비스는 "이메일을 보내라"를 모르고, "주문이 만들어졌다"만 알아야 한다. 이메일 보내는 정책은 알림 서비스의 책임이다.

```kotlin
data class OrderCreated(
    val orderId: String,
    val userId: String,
    val totalAmount: BigDecimal,
    val createdAt: Instant
)

data class SendEmailCommand(
    val to: String,
    val template: String,
    val params: Map<String, String>
)
```

위 두 클래스의 차이는 단순한 네이밍이 아니다. `OrderCreated`는 발행자 입장에서 "내가 해야 할 일"이 끝났다는 선언이고, `SendEmailCommand`는 누군가에게 "네가 이걸 해라"라고 시키는 명령이다. 이걸 혼동하면 발행자가 모든 소비자의 책임을 떠안게 된다.

## Pub/Sub 토폴로지

EDA의 기본 구조는 발행자(Publisher)와 구독자(Subscriber) 사이에 브로커가 있는 형태다. 구현 방식에 따라 토폴로지가 달라지는데, 실무에서 자주 보는 형태는 셋 정도다.

가장 단순한 건 **토픽 기반 팬아웃**이다. 발행자가 토픽에 메시지를 넣으면 구독자들이 자기 컨슈머 그룹으로 메시지를 가져간다. Kafka가 이 모델을 쓴다. 토픽 하나에 컨슈머 그룹 N개가 붙으면 각 그룹은 메시지를 한 부씩 받는다. 그룹 내부에서는 파티션 단위로 분배된다.

```
[Producer] → [Topic: order-events]
                 ├→ [Consumer Group: notification-service] → 인스턴스 3개
                 ├→ [Consumer Group: analytics-service] → 인스턴스 5개
                 └→ [Consumer Group: inventory-service] → 인스턴스 2개
```

각 컨슈머 그룹은 독립적으로 offset을 관리한다. 알림 서비스가 한 시간 다운돼도 분석 서비스의 소비에는 영향이 없다. 이게 EDA의 핵심 장점이다.

두 번째는 **익스체인지 기반 라우팅**이다. RabbitMQ의 모델이다. 발행자가 익스체인지에 메시지를 보내면, 익스체인지가 라우팅 키나 헤더를 보고 적절한 큐로 보낸다. 큐 하나당 하나의 소비자 그룹이 붙는다. 라우팅 룰이 복잡한 경우에 유리하지만, 라우팅 룰이 한곳에 집중돼서 운영 부담이 생긴다.

세 번째는 **이벤트 스트림 + 멀티플렉싱**이다. Kafka 토픽 하나에 여러 도메인 이벤트를 섞어 넣고, 메시지 헤더의 타입 필드로 구분한다. 토픽 수가 줄어들어 관리는 편한데, 컨슈머가 자기와 무관한 메시지도 받아서 필터링해야 한다. 이벤트 타입이 명확히 분리되지 않는 도메인에 쓴다.

토폴로지 선택에서 자주 실수하는 게 토픽을 너무 잘게 쪼개는 거다. `order-created`, `order-paid`, `order-shipped`, `order-cancelled` 토픽을 따로 만들면 컨슈머가 모든 토픽을 구독해야 한다. 주문 도메인은 보통 `order-events` 하나에 모으고, 메시지 타입으로 구분하는 게 운영하기 편하다. 토픽을 분리하는 기준은 도메인 경계지, 이벤트 타입 단위가 아니다.

## 이벤트 스토밍으로 도메인 이벤트 도출

EDA 설계의 시작은 "어떤 이벤트가 있어야 하는가"를 정하는 거다. 책상에서 "결제 완료 이벤트가 필요할 것 같다"라고 결정하면 절반은 빠뜨리고 절반은 불필요한 이벤트가 생긴다. 이걸 체계적으로 뽑는 방법이 이벤트 스토밍이다.

진행 방식은 단순하다. 큰 벽이나 화이트보드에 도메인 전문가와 개발자를 모은다. 주황색 포스트잇에 "일어난 사실"을 과거형으로 적는다. 시간 순서대로 왼쪽에서 오른쪽으로 붙인다. 한 시간쯤 하면 벽이 주황색으로 도배되고, 그게 도메인 이벤트의 1차 목록이다.

```
주문서_작성됨 → 장바구니_확정됨 → 결제수단_선택됨 → 결제_요청됨 → 
결제_승인됨 → 주문_생성됨 → 재고_차감됨 → 출고_요청됨 → 운송장_발급됨 → 
배송_시작됨 → 배송_완료됨
```

이 목록을 그대로 코드로 옮기지는 않는다. 이벤트 스토밍의 가치는 다른 데 있다. 이벤트 목록을 보면 도메인의 흐름이 한눈에 들어오고, 어디서 트랜잭션 경계가 끊기는지, 어디서 컨텍스트가 바뀌는지가 드러난다. 예를 들어 위 흐름에서 `결제_승인됨`과 `주문_생성됨` 사이에 시간차가 있을 수 있다는 게 보이면, 둘을 같은 트랜잭션에 묶으면 안 된다는 결론이 나온다.

또 하나 얻는 건 핫스팟이다. 한 이벤트에 5개 이상의 컨슈머가 붙는다면 그 이벤트는 도메인 경계의 핵심이다. `결제_승인됨` 이벤트에 회계, 알림, 포인트, 분석, 추천 시스템이 모두 붙는다면, 이 이벤트의 스키마는 매우 신중하게 설계해야 한다. 한 번 정의하면 바꾸기 어렵다.

이벤트 스토밍에서 자주 나오는 함정이 "기술적 이벤트"를 도메인 이벤트처럼 적는 거다. `데이터베이스_업데이트됨`, `API_호출됨` 같은 건 도메인 이벤트가 아니다. 도메인 이벤트는 비즈니스적으로 의미 있는 사실이어야 한다. 도메인 전문가가 봤을 때 "이건 우리 비즈니스에서 중요한 일이야"라고 말할 수 있어야 한다.

## 이벤트 스키마 진화

이벤트 스키마는 한 번 정해도 시간이 지나면 반드시 바뀐다. 새 필드가 추가되고, 의미가 바뀌고, 필드가 사라진다. 문제는 이벤트가 발행되는 순간 누가 그 이벤트를 소비하고 있는지 발행자는 모른다는 거다. 스키마를 바꾸면 어떤 컨슈머가 깨질지 알 수 없다.

이 문제를 다루는 표준 도구가 스키마 레지스트리다. Confluent Schema Registry가 대표적이고, AWS Glue Schema Registry, Apicurio도 같은 역할을 한다. 작동 방식은 간단하다. 메시지 자체에는 스키마 전체를 넣지 않고 스키마 ID만 넣는다. 컨슈머는 ID로 레지스트리에 물어서 실제 스키마를 받는다. 이렇게 하면 메시지 크기가 줄고, 스키마 변경 이력이 한곳에 모인다.

Avro는 스키마 레지스트리와 가장 자연스럽게 어울리는 직렬화 포맷이다. JSON처럼 사람이 읽을 수 있는 IDL로 스키마를 정의하고, 바이너리로 직렬화한다.

```json
{
  "type": "record",
  "name": "OrderCreated",
  "namespace": "com.shop.order.events",
  "fields": [
    {"name": "orderId", "type": "string"},
    {"name": "userId", "type": "string"},
    {"name": "totalAmount", "type": {"type": "bytes", "logicalType": "decimal", "precision": 12, "scale": 2}},
    {"name": "createdAt", "type": {"type": "long", "logicalType": "timestamp-millis"}},
    {"name": "couponCode", "type": ["null", "string"], "default": null}
  ]
}
```

`couponCode`가 nullable이고 default가 `null`인 게 중요하다. 새 필드를 추가할 때 default를 주지 않으면 구버전 메시지를 신버전 컨슈머가 읽을 수 없다. Avro의 스키마 호환성 규칙이 이런 식이다.

스키마 진화에는 세 가지 호환성 모드가 있다.

**Backward 호환성**은 신버전 컨슈머가 구버전 메시지를 읽을 수 있는 거다. 필드를 추가할 때는 default 값을 줘야 하고, 필드를 삭제할 때는 default가 있던 필드만 지울 수 있다. 컨슈머를 먼저 배포하고 프로듀서를 나중에 배포할 때 쓴다.

**Forward 호환성**은 구버전 컨슈머가 신버전 메시지를 읽을 수 있는 거다. 필드를 삭제할 때는 default 값이 있어야 하고, 추가할 때는 default와 관계없이 가능하다. 프로듀서를 먼저 배포하고 컨슈머를 나중에 배포할 때 쓴다.

**Full 호환성**은 위 두 가지를 모두 만족하는 거다. 필드 추가/삭제 모두에 default가 필요하다. 대규모 시스템에서는 어느 쪽을 먼저 배포할지 통제하기 어렵기 때문에 Full 호환성을 기본으로 쓰는 경우가 많다.

실무에서 가장 자주 일어나는 사고가 필드의 의미를 바꾸는 거다. `amount` 필드가 원래 원화 단위였는데 다국가 결제를 지원하면서 통화별 단위로 바뀌는 식이다. 이건 스키마 호환성으로는 잡히지 않는다. 같은 타입이고 같은 이름이니까 레지스트리는 통과시킨다. 의미가 바뀌는 변경은 새 필드를 추가하거나 새 토픽을 만드는 게 안전하다. `amount`를 그대로 두고 `amountInCurrency`를 새로 추가하는 식이다.

## at-least-once와 멱등성

EDA에서 가장 자주 부딪히는 함정이 메시지 중복이다. Kafka도 RabbitMQ도 기본적으로 at-least-once 보장을 한다. 즉 메시지는 한 번 이상 전달되지만, 정확히 한 번이라는 보장은 없다. 컨슈머가 메시지를 처리하고 offset commit 직전에 죽으면, 재시작 후 같은 메시지를 다시 받는다.

exactly-once를 진짜로 보장하려면 Kafka의 transactional producer/consumer를 써야 하는데, 처리량이 떨어지고 구성이 복잡하다. 대부분의 실무에서는 at-least-once를 받아들이고 컨슈머가 멱등성을 보장하는 방식으로 간다. 이게 표준이다.

멱등성을 만드는 가장 단순한 방법은 이벤트 ID를 저장하는 거다.

```kotlin
@Transactional
fun handle(event: PaymentCompleted) {
    val exists = processedEventRepository.existsByEventId(event.eventId)
    if (exists) {
        log.info("이미 처리된 이벤트 무시: ${event.eventId}")
        return
    }
    
    // 실제 비즈니스 처리
    orderRepository.markAsPaid(event.orderId)
    pointService.grant(event.userId, event.totalAmount)
    
    processedEventRepository.save(ProcessedEvent(event.eventId, Instant.now()))
}
```

`processed_event` 테이블에 이벤트 ID를 unique key로 박아두고, 트랜잭션 안에서 비즈니스 처리와 이벤트 ID 저장을 같이 한다. 이러면 같은 이벤트를 두 번 받아도 두 번째는 unique 제약조건에 걸려 무시된다.

이 방식의 함정은 `processed_event` 테이블이 계속 커진다는 거다. 하루 1억 건의 이벤트를 처리하는 시스템이라면 한 달이면 30억 row가 쌓인다. 보통은 일정 기간(예: 7일)이 지난 row를 배치로 삭제한다. 7일이 지난 메시지가 다시 들어올 가능성은 거의 없기 때문이다. Kafka의 retention과 맞춰서 잡으면 된다.

더 가벼운 방법은 비즈니스 자체를 멱등하게 만드는 거다. "포인트 +100"은 두 번 실행하면 +200이 되지만, "포인트 잔액을 1500으로 설정"은 두 번 실행해도 결과가 같다. 가능하면 상태 변화(delta)가 아니라 최종 상태(absolute)를 다루는 게 안전하다. 다만 비즈니스 의미에 따라 이게 안 되는 경우도 많다. 입금 처리는 절대 멱등하게 만들 수 없으니, 이런 경우는 이벤트 ID 추적이 답이다.

## 이벤트 순서 보장

"같은 주문의 이벤트는 순서대로 처리되어야 한다." 이 요구사항을 처음 들으면 당연하게 들리지만, 분산 시스템에서는 비용이 크다.

Kafka는 파티션 단위로만 순서를 보장한다. 같은 파티션 안의 메시지는 발행 순서대로 소비되지만, 다른 파티션 사이에서는 순서를 보장하지 않는다. 그러면 같은 주문의 이벤트가 같은 파티션으로 가도록 만들어야 한다. 이게 파티션 키다.

```kotlin
producer.send(ProducerRecord(
    "order-events",
    orderId,  // 파티션 키
    event
))
```

같은 `orderId`를 가진 메시지는 항상 같은 파티션으로 간다. 그 파티션을 소비하는 컨슈머 인스턴스는 하나만 있을 수 있고, 그 인스턴스는 메시지를 발행 순서대로 받는다.

여기서 함정이 두 개 있다. 첫째, 컨슈머가 한 파티션의 메시지를 여러 스레드로 병렬 처리하면 순서가 깨진다. 메시지를 받는 건 단일 스레드여도 비즈니스 처리를 스레드 풀로 넘기면 의미가 없다. 같은 파티션의 메시지는 순차적으로 처리해야 한다. 처리량이 부족하면 파티션 수를 늘리는 방향으로 가야지, 한 파티션을 멀티스레드로 풀면 안 된다.

둘째, 재시도가 순서를 깬다. 컨슈머가 메시지를 받아 처리하다가 실패해서 다른 큐(DLQ)로 보내고 다음 메시지를 처리한다면, 같은 주문의 두 번째 메시지가 첫 번째 메시지보다 먼저 성공해버린다. 이걸 막으려면 실패 시 처리를 중단하고 같은 메시지를 계속 재시도하거나, 같은 주문의 후속 메시지도 함께 차단해야 한다. 둘 다 처리량을 깎는다.

순서가 정말로 필요한지부터 따져봐야 한다. 대부분의 비즈니스 이벤트는 사실 순서보다는 최종 일관성으로 충분하다. "결제 승인" 이벤트가 "주문 생성" 이벤트보다 늦게 도착해도, 컨슈머가 "주문이 없으면 잠깐 기다린 뒤 재시도"하는 식으로 처리하면 된다. 진짜로 순서가 깨지면 안 되는 케이스는 상태 머신을 따라가는 경우(주문 상태가 `생성→결제→배송→완료`로만 가야 하는 경우) 정도다.

상태 머신의 경우에도 이벤트에 버전 번호나 타임스탬프를 박아두고 컨슈머가 검증하는 게 안전하다. 메시지 순서에만 의존하지 말고 메시지 내용 자체로도 순서를 판단할 수 있게 만든다.

```kotlin
@Transactional
fun handle(event: OrderStateChanged) {
    val order = orderRepository.findById(event.orderId)
    if (order.version >= event.version) {
        log.info("오래된 이벤트 무시: 현재 v${order.version}, 이벤트 v${event.version}")
        return
    }
    order.applyState(event.newState, event.version)
}
```

## 컨슈머 그룹 설계

Kafka의 컨슈머 그룹은 EDA의 운영을 좌우하는 핵심 개념이다. 같은 그룹 안의 컨슈머 인스턴스들이 토픽의 파티션을 나눠 가진다. 토픽 파티션이 10개고 컨슈머 그룹에 인스턴스가 3개면, 각 인스턴스가 3~4개 파티션을 맡는다. 인스턴스가 11개면 1개는 놀게 된다(파티션 수가 병렬 처리의 상한이다).

컨슈머 그룹을 설계할 때 결정해야 할 게 몇 가지 있다.

**그룹의 단위**는 "독립적으로 소비할 단위"다. 일반적으로 서비스 하나가 그룹 하나다. `notification-service`가 컨슈머 그룹 하나를 갖는다. 같은 서비스 안에서 처리 로직이 완전히 분리되어야 한다면(예: 이메일 발송과 푸시 발송이 완전히 다른 처리 경로) 그룹을 둘로 나눌 수도 있지만, 일반적으로는 한 서비스 한 그룹이 깔끔하다.

**파티션 수**는 그룹의 최대 병렬도를 정한다. 운영 들어가서 파티션 수를 늘리기는 가능하지만, 늘리는 순간 같은 키의 메시지가 다른 파티션으로 갈 수 있어 순서 보장이 깨진다. 처음에 넉넉하게 잡는 게 낫다. 일반적으로 예상 컨슈머 인스턴스 수의 2~4배로 시작한다. 인스턴스가 5개 예상되면 파티션을 12~20개 정도로 잡는다.

**리밸런싱**은 컨슈머가 추가되거나 죽을 때 일어난다. 파티션이 재분배되는 동안 그룹 전체가 잠깐 멈춘다. 이걸 STW(stop-the-world)라고 부른다. 인스턴스가 자주 바뀌는 환경(쿠버네티스에서 잦은 재시작, autoscaling)에서는 리밸런싱이 자주 일어나 lag이 커진다. Kafka 2.4+ 부터 Cooperative Rebalancing이 도입되어 이 문제가 완화되긴 했다.

**lag 모니터링**은 컨슈머 그룹 운영의 핵심이다. 컨슈머 그룹의 offset과 토픽의 가장 최신 offset의 차이가 lag이다. lag이 계속 커진다는 건 발행 속도를 소비 속도가 못 따라간다는 뜻이다. 일시적으로 커지는 건 정상인데(피크 트래픽), 지속적으로 커진다면 컨슈머 인스턴스를 늘리거나 처리 로직을 가볍게 만들어야 한다. lag 모니터링 없이 EDA를 운영하는 건 계기판 없이 비행기 모는 거랑 같다.

```kotlin
@Component
class ConsumerLagMetric(private val adminClient: AdminClient) {
    
    @Scheduled(fixedRate = 30000)
    fun report() {
        val groupOffsets = adminClient.listConsumerGroupOffsets("order-processor")
            .partitionsToOffsetAndMetadata().get()
        
        val topicEndOffsets = adminClient.listOffsets(
            groupOffsets.keys.associateWith { OffsetSpec.latest() }
        ).all().get()
        
        groupOffsets.forEach { (partition, groupOffset) ->
            val endOffset = topicEndOffsets[partition]?.offset() ?: 0
            val lag = endOffset - groupOffset.offset()
            meterRegistry.gauge("kafka.consumer.lag", 
                listOf(Tag.of("partition", partition.partition().toString())), 
                lag.toDouble())
        }
    }
}
```

이 lag 메트릭을 Grafana에 띄워두고, 일정 임계치를 넘으면 알람을 받는다. 임계치는 트래픽 패턴에 따라 다르지만, 평소 lag의 10배 정도를 알람 기준으로 잡는다.

## 페이로드 크기와 fat vs thin 이벤트

이벤트 안에 얼마나 많은 정보를 담을지가 EDA 설계의 마지막 큰 결정이다. 양극단을 fat event와 thin event라고 부른다.

**Thin event**는 식별자만 담는 방식이다.

```json
{
  "eventType": "OrderCreated",
  "orderId": "ord_123",
  "occurredAt": "2026-05-11T10:30:00Z"
}
```

컨슈머가 `orderId`를 받아서 주문 서비스에 다시 조회 API를 호출해 상세 정보를 가져온다. 이벤트 페이로드는 작고 깔끔하지만, 컨슈머가 늘어날수록 주문 서비스에 호출이 N배 늘어난다. 컨슈머가 10개면 한 이벤트당 10번의 조회 호출이 일어난다.

**Fat event**는 컨슈머가 필요로 할 만한 모든 정보를 이벤트에 담는다.

```json
{
  "eventType": "OrderCreated",
  "orderId": "ord_123",
  "occurredAt": "2026-05-11T10:30:00Z",
  "userId": "user_456",
  "userName": "김철수",
  "userEmail": "kim@example.com",
  "items": [
    {"productId": "p_1", "productName": "노트북", "quantity": 1, "unitPrice": 1500000}
  ],
  "totalAmount": 1500000,
  "shippingAddress": {...},
  "paymentMethod": "card"
}
```

컨슈머가 이벤트만 보고 처리를 끝낼 수 있으니 조회 호출이 없다. 발행 서비스가 다운돼 있어도 컨슈머는 정상 동작한다. 대신 페이로드가 크다. Kafka의 기본 메시지 크기 제한이 1MB인데, 이걸 넘어가는 이벤트도 나온다.

선택의 기준은 단순하다. 컨슈머가 자주 추가 조회를 해야 한다면 fat으로 가고, 페이로드가 너무 커진다면 thin으로 간다. 보통은 중간 어딘가에 있다. 자주 쓰는 정보(userId, totalAmount, 상태)는 이벤트에 담고, 잘 안 쓰거나 큰 정보(전체 상품 리스트, 배송 추적 이력)는 조회하게 만든다.

Fat event에서 문제가 되는 게 데이터의 시점이다. 이벤트에 담긴 사용자 이름이 발행 시점의 이름이라는 게 명확해야 한다. 컨슈머가 이걸 마치 현재 사용자 이름인 양 처리하면, 사용자가 이름을 바꾼 후에 도착한 알림에 옛날 이름이 박힌다. 이벤트는 본질적으로 과거의 스냅샷이다. 현재 상태가 필요하면 조회 API를 호출해야 한다.

페이로드 크기가 1MB를 넘는 경우는 별도 처리가 필요하다. 본문은 S3 같은 객체 스토리지에 저장하고, 이벤트에는 그 URL을 담는다. 이걸 claim check 패턴이라 부른다. 이미지 처리, 동영상 메타데이터, 큰 PDF 같은 케이스에서 자주 쓴다. 이벤트 스트림은 메시지 큐가 아니라 라우팅 신호로 쓰고, 실제 데이터는 다른 곳에서 가져오는 구조다.

## 마지막으로

EDA는 시스템을 느슨하게 결합시키지만, 그 대가로 추적이 어려워진다. 한 주문이 어디까지 처리됐는지 알려면 5개 서비스의 로그를 다 봐야 한다. 분산 추적(Distributed Tracing) 없이 EDA를 운영하는 건 권장하기 어렵다. 이벤트마다 traceId를 박아두고, OpenTelemetry로 한곳에 모으는 작업을 같이 해야 한다.

또 하나, EDA로 가면 디버깅 패러다임이 바뀐다. 동기 시스템에서는 stack trace 하나 보면 어디서 뭐가 잘못됐는지 보이지만, EDA에서는 "어떤 이벤트가 들어왔고, 컨슈머가 어떻게 반응했고, 그 결과 어떤 이벤트가 다시 나갔는지"를 시간 순으로 재구성해야 한다. 이벤트 로그를 잘 남기고, 토픽에 흐른 메시지를 일정 기간 보관하는 게 운영의 기본이다. Kafka의 retention을 짧게 잡으면 사고 났을 때 원인 추적이 막힌다. 일반적으로 7~14일 정도는 보관한다.

EDA는 정말로 비동기 처리가 필요할 때 도입하는 거다. 동기 호출 몇 개를 비동기로 바꾸려고 Kafka를 까는 건 과한 결정이다. 서비스 간 결합도가 실제로 문제가 되고 있는지, 비동기 처리가 비즈니스 요구사항인지 먼저 따져봐야 한다.
