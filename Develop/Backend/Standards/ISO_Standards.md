# ISO 표준 — 백엔드 개발자가 실제로 마주치는 것들

ISO 표준이라고 하면 문서만 두꺼운 관료적인 규격이라고 생각하기 쉽다. 실제로 대부분의 ISO 표준은 개발자가 직접 읽을 일이 없다. 하지만 몇 가지는 코드를 짜면서, 서비스를 운영하면서, 감사를 받으면서 반복적으로 만나게 된다.

여기서는 백엔드 개발 실무에서 빈도 높게 마주치는 네 가지 ISO 표준을 다룬다.

---

## ISO 8601 — 날짜와 시간 포맷

### 왜 계속 만나는가

API 설계, DB 저장, 로그 파싱, 외부 시스템 연동 — 날짜/시간을 다루는 모든 곳에서 ISO 8601이 등장한다. REST API 응답에 `"2026-04-06T15:30:00+09:00"` 같은 문자열이 들어가는 게 이 표준 때문이다.

기본 형식은 이렇다:

```
날짜: 2026-04-06
시간: 15:30:00
날짜+시간: 2026-04-06T15:30:00
UTC: 2026-04-06T06:30:00Z
오프셋: 2026-04-06T15:30:00+09:00
```

`T`는 날짜와 시간의 구분자, `Z`는 UTC를 뜻한다. `+09:00`은 UTC 대비 오프셋이다.

### 타임존 처리에서 자주 틀리는 것

**오프셋과 타임존은 다르다.** `+09:00`은 오프셋이지 `Asia/Seoul`이 아니다. 한국은 서머타임이 없어서 항상 `+09:00`이지만, 미국 동부는 EST(`-05:00`)와 EDT(`-04:00`)를 오간다. 오프셋만 저장하면 서머타임 전환 시점에 시간 계산이 틀어진다.

```java
// 틀리기 쉬운 방식 — 오프셋만 저장
OffsetDateTime odt = OffsetDateTime.parse("2026-03-08T02:30:00-05:00");
// 이 시간이 EST인지 EDT인지 알 수 없다

// 타임존 ID를 사용해야 한다
ZonedDateTime zdt = ZonedDateTime.of(
    LocalDateTime.of(2026, 3, 8, 2, 30),
    ZoneId.of("America/New_York")
);
// 서머타임 전환 시 자동으로 처리된다
```

**DB에는 UTC로 저장한다.** 이건 거의 예외 없는 규칙이다. `TIMESTAMP WITH TIME ZONE` 컬럼을 써도 PostgreSQL은 내부적으로 UTC로 변환해서 저장한다. MySQL의 `DATETIME`은 타임존 정보가 없으므로 애플리케이션 레벨에서 UTC 변환을 해야 한다.

```sql
-- PostgreSQL: timestamptz는 UTC로 저장, 조회 시 세션 타임존으로 변환
CREATE TABLE events (
    id BIGSERIAL PRIMARY KEY,
    event_time TIMESTAMPTZ NOT NULL,  -- UTC로 저장됨
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 저장
INSERT INTO events (event_time) VALUES ('2026-04-06T15:30:00+09:00');
-- 내부 저장값: 2026-04-06 06:30:00 UTC
```

### 파싱에서 겪는 문제들

외부 시스템과 연동하면 ISO 8601을 따른다고 하면서 미묘하게 다른 포맷을 보내는 경우가 많다.

```java
// 실제로 받아본 형식들
"2026-04-06T15:30:00Z"          // 정상
"2026-04-06T15:30:00.000Z"      // 밀리초 포함 — 정상
"2026-04-06T15:30:00.000+0000"  // 콜론 없는 오프셋 — ISO 8601에서는 허용하지만 파서에 따라 실패
"2026-04-06 15:30:00"           // T 없이 공백 — ISO 8601 위반이지만 흔함
"20260406T153000Z"              // 기본 형식(basic format) — ISO 8601 허용, 파서에 따라 실패
```

Java의 `DateTimeFormatter.ISO_OFFSET_DATE_TIME`은 `+0000` 형식(콜론 없는 오프셋)을 파싱하지 못한다. 외부 API와 연동할 때 이 차이 때문에 장애가 나는 경우가 있다.

```java
// 콜론 없는 오프셋을 처리하려면 커스텀 포매터가 필요하다
DateTimeFormatter flexible = new DateTimeFormatterBuilder()
    .append(DateTimeFormatter.ISO_LOCAL_DATE_TIME)
    .optionalStart()
    .appendOffset("+HHMMss", "Z")  // +0900, +09:00 모두 처리
    .optionalEnd()
    .toFormatter();

OffsetDateTime parsed = OffsetDateTime.parse("2026-04-06T15:30:00+0900", flexible);
```

Python은 상대적으로 관대한 편이다:

```python
from datetime import datetime, timezone

# Python 3.7+의 fromisoformat은 제한적이었다
# Python 3.11부터 Z 접미사를 처리한다
dt = datetime.fromisoformat("2026-04-06T15:30:00+09:00")

# 3.11 미만에서 Z를 처리하려면 직접 바꿔야 했다
raw = "2026-04-06T15:30:00Z"
dt = datetime.fromisoformat(raw.replace("Z", "+00:00"))

# UTC로 변환
utc_dt = dt.astimezone(timezone.utc)
```

### 로그에서의 ISO 8601

로그 파싱 시 타임스탬프 포맷이 통일되지 않으면 집계가 안 된다. 서비스별로 포맷이 다르면 ELK나 Datadog에서 시간 기준 정렬이 깨진다.

로그 타임스탬프는 UTC ISO 8601로 통일하는 게 맞다. Logback 설정 예시:

```xml
<pattern>%d{yyyy-MM-dd'T'HH:mm:ss.SSS'Z', UTC} [%thread] %-5level %logger - %msg%n</pattern>
```

이렇게 하면 모든 서비스의 로그가 같은 시간 기준으로 정렬된다.

## ISO 27001 — 정보보안 관리

### 개발자가 만나는 시점

회사가 ISO 27001 인증을 받거나 갱신할 때 개발자도 감사 대응에 참여하게 된다. 인증 심사가 아니더라도 금융권이나 공공기관과 일하면 "ISO 27001 인증 보유" 요구사항이 RFP에 들어가는 경우가 많다.

ISO 27001은 정보보안 관리 체계(ISMS)에 대한 표준이다. 조직이 정보 자산을 어떻게 보호하는지에 대한 프로세스와 통제 항목을 정의한다.

### 감사 때 개발자에게 요구하는 것들

감사관이 개발자에게 직접 묻거나 증거를 요구하는 항목들이 있다.

**접근 제어 — 누가 무엇에 접근하는가**

- 프로덕션 DB 접근 권한은 누구에게 있는가
- SSH 키 관리는 어떻게 하는가
- 퇴사자의 접근 권한 회수 절차는 있는가
- 관리자 계정은 개인별로 분리되어 있는가 (공용 계정 사용 여부)

실무에서는 이런 질문이 온다: "프로덕션 서버에 접속할 수 있는 사람의 목록을 보여주세요." AWS IAM 정책이 정리되어 있지 않으면 이 질문에 답하는 데 며칠이 걸린다.

**변경 관리 — 코드가 어떻게 프로덕션에 반영되는가**

- 코드 리뷰 프로세스가 있는가 (PR 승인 기록)
- 배포 파이프라인에 승인 단계가 있는가
- 롤백 절차가 문서화되어 있는가
- 핫픽스 시에도 같은 프로세스를 따르는가

Git 히스토리와 CI/CD 파이프라인 로그가 증거가 된다. PR 없이 main에 직접 푸시하는 습관이 있으면 감사 때 문제가 된다.

**로깅과 모니터링**

- 보안 관련 이벤트(로그인 실패, 권한 변경)를 로깅하는가
- 로그 보존 기간은 어떻게 되는가
- 로그 위변조 방지 대책이 있는가

로그를 S3에 보관한다면 버저닝이 켜져 있는지, 삭제 권한은 제한되어 있는지 확인해야 한다.

**암호화**

- 전송 중 데이터 암호화(TLS)
- 저장 데이터 암호화(at-rest encryption)
- 암호화 키 관리 방법

"DB 암호화 하고 있습니다"라고 말하려면 실제로 어떤 알고리즘을 쓰는지, 키는 어디에 저장되어 있는지 알아야 한다. KMS를 쓰고 있다면 키 로테이션 주기도 확인 대상이다.

### 개발자가 평소에 신경 쓸 것

감사 때 급하게 준비하는 건 티가 난다. 평소에 해두면 좋은 것들:

- PR 기반 코드 리뷰를 프로세스로 유지한다
- 프로덕션 접근 권한을 최소 권한 원칙으로 관리한다
- 비밀값(API 키, DB 패스워드)을 코드에 넣지 않는다 — Vault나 AWS Secrets Manager 사용
- 보안 이벤트 로깅을 빠뜨리지 않는다

## ISO 25010 — 소프트웨어 품질 모델

### 어디서 만나는가

비기능 요구사항을 정의할 때 "성능, 보안, 유지보수성 같은 거요"라고 두루뭉술하게 말하는 경우가 많다. ISO 25010은 소프트웨어 품질을 8가지 특성으로 나눠서 체계적으로 정리한 모델이다.

요구사항 문서를 작성하거나 아키텍처 리뷰를 할 때, 빠뜨린 품질 속성이 없는지 점검하는 용도로 쓴다.

### 8가지 품질 특성

| 특성 | 의미 | 실무에서 다루는 예 |
|------|------|-------------------|
| 기능 적합성 | 요구된 기능을 정확히 수행하는가 | API가 스펙대로 동작하는지 |
| 성능 효율성 | 자원 대비 성능이 적절한가 | 응답 시간, 처리량, 메모리 사용 |
| 호환성 | 다른 시스템과 공존/연동 가능한가 | API 버전 호환, 데이터 포맷 호환 |
| 사용성 | 사용자가 쉽게 쓸 수 있는가 | API 설계의 일관성, 에러 메시지 품질 |
| 신뢰성 | 장애 없이 안정적으로 동작하는가 | 가용성, 장애 복구, 데이터 정합성 |
| 보안성 | 정보와 기능이 적절히 보호되는가 | 인증, 인가, 데이터 암호화 |
| 유지보수성 | 수정/확장이 쉬운가 | 코드 모듈화, 테스트 용이성 |
| 이식성 | 다른 환경으로 옮기기 쉬운가 | 컨테이너화, 환경 설정 분리 |

### 비기능 요구사항 정의에 활용하는 방법

프로젝트 초기에 비기능 요구사항을 정의할 때, 이 8가지를 하나씩 검토하면 빠뜨리는 항목이 줄어든다.

예를 들어 주문 처리 서비스를 설계한다면:

- **성능 효율성**: 주문 API 응답 시간 200ms 이내, 초당 1000건 처리
- **신뢰성**: 결제 트랜잭션 실패 시 자동 재시도, 99.9% 가용성
- **보안성**: PCI DSS 준수, 카드 정보 토큰화
- **유지보수성**: 결제 수단 추가 시 기존 코드 변경 최소화

이렇게 구체적인 수치와 조건을 붙여야 한다. "성능이 좋아야 합니다" 같은 요구사항은 쓸모가 없다.

이 모델 자체가 코드에 직접 영향을 주지는 않지만, 아키텍처 결정의 근거를 만들 때 유용하다. "왜 캐시를 도입하는가?" → "성능 효율성 요구사항(응답 시간 200ms 이내) 충족을 위해"처럼 결정의 이유를 명확하게 할 수 있다.

## ISO/IEC 12207 — 소프트웨어 생명주기 프로세스

### 어디서 만나는가

SI 프로젝트, 정부/공공 프로젝트, 대기업 외주 개발에서 "소프트웨어 개발 생명주기를 준수하라"는 요구사항이 나올 때 이 표준이 근거가 된다. 직접 읽을 일은 거의 없지만, 프로세스의 뼈대를 이해하면 왜 그런 산출물을 요구하는지 알 수 있다.

### 구조

ISO/IEC 12207은 소프트웨어 생명주기에 관련된 프로세스를 크게 세 그룹으로 나눈다:

**기본 프로세스**
- 획득(Acquisition): 발주자가 소프트웨어를 구매/확보하는 과정
- 공급(Supply): 수주자가 소프트웨어를 제공하는 과정
- 개발(Development): 요구사항 분석 → 설계 → 구현 → 테스트
- 운영(Operation): 배포 후 서비스 운영
- 유지보수(Maintenance): 결함 수정, 기능 개선

**지원 프로세스**
- 문서화, 형상관리, 품질보증, 검증, 확인, 감사, 문제해결

**조직 프로세스**
- 관리, 인프라, 개선, 교육훈련

### 개발자에게 실제로 영향을 주는 부분

대부분의 개발자는 이 표준을 직접 참조하지 않지만, 이 표준에서 파생된 요구사항을 일상적으로 만난다:

**형상관리(Configuration Management)**

Git으로 코드를 관리하는 것 자체가 형상관리다. 다만 ISO/IEC 12207 맥락에서는 단순히 Git을 쓰는 것 이상을 요구한다.

- 형상 항목 식별: 코드뿐 아니라 설정 파일, 빌드 스크립트, 문서도 관리 대상
- 변경 통제: 변경 요청 → 영향 분석 → 승인 → 반영의 프로세스
- 형상 상태 기록: 언제 누가 무엇을 왜 바꿨는지 추적 가능해야 함

Git 커밋 메시지를 제대로 쓰고, PR에 변경 이유를 기록하는 것이 이 요구사항을 충족하는 가장 현실적인 방법이다.

**검증과 확인(Verification & Validation)**

- 검증(Verification): 각 단계의 산출물이 해당 단계의 요구사항을 만족하는가 → 코드 리뷰, 단위 테스트
- 확인(Validation): 최종 산출물이 사용자의 원래 요구를 만족하는가 → 인수 테스트, UAT

"테스트를 왜 해야 하는가"에 대한 공식적인 근거가 여기에 있다. 공공 프로젝트에서 테스트 계획서와 결과서를 산출물로 요구하는 이유다.

**문서화**

공공 프로젝트에서 요구하는 산출물 목록(요구사항 정의서, 설계서, 테스트 계획서, 사용자 매뉴얼 등)이 이 표준에 근거한다. 산출물을 "왜 만들어야 하는가"가 궁금하면 이 표준이 답이다.

## 정리

| 표준 | 만나는 상황 | 핵심 |
|------|------------|------|
| ISO 8601 | API 설계, DB 저장, 로그 | 날짜/시간은 UTC ISO 8601로 통일한다 |
| ISO 27001 | 보안 감사, 인증 | 접근 제어, 변경 관리, 로깅을 평소에 유지한다 |
| ISO 25010 | 비기능 요구사항 정의 | 8가지 품질 특성으로 빠뜨림 없이 점검한다 |
| ISO/IEC 12207 | SI/공공 프로젝트 | 프로세스와 산출물 요구의 근거가 되는 표준이다 |

일상적으로 가장 자주 신경 쓰는 건 ISO 8601이다. 나머지는 프로젝트 성격에 따라 만나는 빈도가 다르지만, 한 번쯤은 마주치게 되니 어떤 맥락에서 나오는 표준인지 알아두면 당황하지 않는다.
