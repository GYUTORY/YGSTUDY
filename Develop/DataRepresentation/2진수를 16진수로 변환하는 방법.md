# 2진수를 16진수로 변환하는 방법

## 왜 두 진수를 같이 다루는가

컴퓨터 내부 표현은 어차피 2진수다. 그런데 사람이 11111111을 보고 "아 0xFF지" 하는 것보다, 0xFF를 보고 "아 1바이트 다 켜진 거지" 하는 게 훨씬 빠르다. 즉 16진수는 사람을 위한 "2진수 압축 표기"다.

실무에서 16진수를 보게 되는 상황은 대체로 정해져 있다. 메모리 덤프, 네트워크 패킷, 파일 매직 넘버, 색상값, 비트 플래그, 에러 코드, 해시값. 이런 데이터는 한 번 익숙해지면 굳이 10진수로 바꾸지 않고 그냥 16진수 상태에서 비트 패턴을 머릿속에서 본다. 그게 가능해지는 게 이 문서의 목표다.

진법별 정리:

| 진수 | 사용 문자 | 한 자리당 비트 수 | 주 용도 |
|------|-----------|-------------------|---------|
| 2진수  | 0, 1       | 1비트 | CPU 레지스터, 비트 연산 |
| 8진수  | 0~7        | 3비트 | Unix 파일 권한(chmod 755) |
| 10진수 | 0~9        | log₂10 ≈ 3.32비트 | 사람의 일상 |
| 16진수 | 0~9, A~F   | 4비트 | 메모리, 색상, 디버깅 출력 |

8진수가 안 쓰이는 이유는 단순하다. 8비트(1바이트) 정렬과 안 맞는다. 1바이트는 16진수 두 자리(2 × 4비트 = 8비트)로 정확히 떨어지지만, 8진수로는 2자리 + α가 되어 가독성이 떨어진다. 그래서 옛날 PDP-11 같은 12비트, 18비트, 36비트 머신 시절 잔재로 chmod 정도에만 남았다.

## 4자리 묶기 원리

16진수 한 자리는 0부터 15까지 16가지 값을 표현한다. 2⁴ = 16이므로 2진수 4비트와 정확히 1:1 대응한다. 이 단순한 사실 때문에 변환은 진짜 "묶고 매핑"하면 끝난다. 10진수처럼 나눗셈을 반복할 필요가 없다.

기본 매핑은 외우는 수밖에 없다. 변환표를 굳이 다시 그리지 않고, 머릿속에 박아두면 좋은 형태로 정리한다.

| 2진수 | 16진수 | 2진수 | 16진수 |
|-------|--------|-------|--------|
| 0000  | 0      | 1000  | 8      |
| 0001  | 1      | 1001  | 9      |
| 0010  | 2      | 1010  | A (10) |
| 0011  | 3      | 1011  | B (11) |
| 0100  | 4      | 1100  | C (12) |
| 0101  | 5      | 1101  | D (13) |
| 0110  | 6      | 1110  | E (14) |
| 0111  | 7      | 1111  | F (15) |

### 자주 헷갈리는 패턴 암기 요령

이 표를 통째로 외우는 것보다, 자주 마주치는 비트 패턴 몇 개를 통으로 외우는 게 실전에서 빠르다.

- `1111` = `F` = 4비트 다 켜짐
- `11111111` = `0xFF` = 1바이트 다 켜짐, 부호 있는 byte로는 -1
- `10000000` = `0x80` = 최상위 비트만 켜짐, 부호 있는 byte로는 -128
- `01111111` = `0x7F` = 최상위 비트만 꺼짐, 부호 있는 byte 최댓값 +127
- `1010` = `0xA` = 짝수 자리 패턴 (격자무늬)
- `0101` = `0x5` = 홀수 자리 패턴 (반대 격자)
- `1100` = `0xC`, `0011` = `0x3` = 상하반/하반만

`0xA`와 `0x5`는 서로 비트 반전(NOT) 관계다. `0xA ^ 0x5 = 0xF`. `0xFF`와 `0x00`도 마찬가지. 이런 관계를 알면 마스크값을 만들거나 디버깅할 때 패턴이 한눈에 들어온다.

특히 헷갈리기 쉬운 게 `1010`이다. 10진수 10이 아니라 16진수로 `A`다. 처음에는 "10일까 A일까" 망설이게 되는데, 4비트 묶음에서 10진수 10 이상은 반드시 알파벳이라는 점을 기억하면 된다.

## 변환 실전

### 2진수에서 16진수로

오른쪽부터 4비트씩 끊고 매핑한다. 자릿수가 4의 배수가 아니면 왼쪽에 0을 채운다.

```
입력:    1101011011
패딩:    0011 0101 1011    (왼쪽에 00 추가)
매핑:     3    5    B
결과:   0x35B
```

### 16진수에서 2진수로 (역방향)

이쪽이 오히려 쉽다. 한 자리씩 4비트로 펼치기만 하면 된다. 패딩도 필요 없다.

```
입력:   0xCAFE
펼침:   1100 1010 1111 1110
결과:   1100101011111110
```

실무에서 16진수 → 2진수 변환은 비트 마스크 만들 때 자주 한다. 예를 들어 "하위 12비트만 남기는 마스크"를 만들고 싶다면 `0xFFF`라고 쓰면 끝난다. 굳이 `0b111111111111`이라고 쓸 일이 없다. 16진수 한 자리가 곧 4비트라는 감각이 있으면 마스크 길이를 자릿수만 보고 즉시 안다. `0xFFFF`는 16비트, `0xFFFFFFFF`는 32비트, `0xFFFFFFFFFFFFFFFF`는 64비트.

## 비트 마스크와 시프트의 실무 활용

16진수 표기가 가장 빛나는 순간이 비트 연산이다. 패킹된 데이터에서 특정 필드를 꺼내거나 합치는 작업이 자주 나오는데, 10진수로는 코드 의도가 안 보인다.

### RGB 색상에서 채널 분리

웹에서 색상을 `#RRGGBB` 형태로 쓰는데, 이걸 정수 한 개에 패킹해 저장하면 24비트(상위 8비트는 보통 알파나 0)다. 빨강 채널만 빼고 싶으면 시프트와 마스크를 같이 쓴다.

```javascript
const color = 0xFF8040;  // R=FF, G=80, B=40

const r = (color >> 16) & 0xFF;  // 0xFF (255)
const g = (color >> 8) & 0xFF;   // 0x80 (128)
const b = color & 0xFF;          // 0x40 (64)
```

`>> 16`은 16비트(=16진수 4자리) 오른쪽으로 밀어 R 채널을 최하위로 가져오는 동작이고, `& 0xFF`는 하위 8비트만 살리는 마스크다. 굳이 R, G, B 변수를 따로 두지 않고 정수 하나로 색을 다루면 메모리도 덜 쓰고 비교도 빠르다. 캔버스나 이미지 픽셀 버퍼는 거의 다 이 형태다.

반대로 합칠 때는 시프트와 OR을 쓴다.

```javascript
const r = 0xFF, g = 0x80, b = 0x40;
const color = (r << 16) | (g << 8) | b;  // 0xFF8040
```

알파 채널까지 합치면 32비트가 되고, 엔디언 차이 때문에 ARGB냐 RGBA냐 BGRA냐로 갈린다. iOS Core Graphics는 ARGB, OpenGL 텍스처는 보통 RGBA, Windows GDI는 BGRA를 쓰는데, 같은 정수 `0xFF8040FF`라도 어떻게 해석하느냐에 따라 색이 달라진다. 이미지 라이브러리에서 색이 이상하게 나올 때 십중팔구 이 부분이 원인이다.

### IP 주소를 32비트 정수로 패킹

IPv4 주소는 점 4개로 끊지만 내부적으로는 32비트 정수다. DB에 IP를 저장할 때 문자열 `"192.168.1.10"`보다 정수 한 개로 저장하면 인덱스도 효율적이고 범위 검색이 단순해진다.

```python
def ip_to_int(ip: str) -> int:
    a, b, c, d = map(int, ip.split('.'))
    return (a << 24) | (b << 16) | (c << 8) | d

def int_to_ip(n: int) -> str:
    return f"{(n >> 24) & 0xFF}.{(n >> 16) & 0xFF}.{(n >> 8) & 0xFF}.{n & 0xFF}"

n = ip_to_int("192.168.1.10")
print(hex(n))           # 0xc0a8010a
print(int_to_ip(n))     # 192.168.1.10
```

`192.168.1.10` = `0xC0A8010A`. 이걸 4바이트로 끊으면 `C0` `A8` `01` `0A` = 192, 168, 1, 10이다. 16진수로 보면 패킹 결과가 옥텟별로 깔끔하게 떨어진다.

서브넷 마스크도 같은 식으로 다룬다. `/24`는 상위 24비트가 1인 마스크, 즉 `0xFFFFFF00`. `(ip & mask) == (network & mask)` 한 줄로 같은 서브넷인지 판단할 수 있다. 문자열로 쪼개서 비교하는 코드보다 훨씬 빠르고 짧다.

### 비트 플래그

권한 같은 걸 enum 비트로 만들 때도 16진수가 자연스럽다.

```java
public static final int READ    = 0x01;  // 0001
public static final int WRITE   = 0x02;  // 0010
public static final int EXECUTE = 0x04;  // 0100
public static final int DELETE  = 0x08;  // 1000

int permissions = READ | WRITE;          // 0x03
boolean canWrite = (permissions & WRITE) != 0;
```

플래그가 8개를 넘기면 `0x10`, `0x20`, `0x40`, `0x80`으로 자연스럽게 이어진다. 1, 2, 4, 8, 16, 32, 64, 128로 외우는 것보다 16진수 패턴이 머리에 더 잘 들어온다.

## 부호 있는 정수와 음수 변환

여기서 학부 수준 변환과 실무 코드의 차이가 갈린다. `0xFF`가 항상 255인 게 아니다.

### byte b = (byte)0xFF가 -1이 되는 이유

자바에서 byte는 8비트 부호 있는 정수다. 표현 범위는 -128 ~ +127이고, 음수는 2의 보수로 표현한다. `0xFF`는 비트 패턴 `11111111`인데, 부호 있는 8비트로 해석하면 -1이다. (최상위 비트가 1이면 음수, 나머지 비트를 반전 후 1을 더한 값에 마이너스를 붙임. `~11111111 + 1 = 00000000 + 1 = 1`, 그래서 `-1`)

```java
byte b = (byte) 0xFF;
System.out.println(b);          // -1
System.out.println(b & 0xFF);   // 255
```

`b & 0xFF`라는 패턴은 자바 코드에서 정말 자주 보인다. byte를 0~255 범위의 unsigned 값으로 다루고 싶을 때 쓴다. byte를 그냥 int로 캐스팅하면 부호 확장(sign extension)이 일어나서 `0xFF`가 `0xFFFFFFFF`(-1)가 되어버린다. `& 0xFF`로 마스킹해야 상위 비트를 잘라내고 0~255로 만들 수 있다.

```java
byte[] data = {(byte) 0xCA, (byte) 0xFE};
int hi = data[0];                  // -54 (틀린 해석)
int hi_correct = data[0] & 0xFF;   // 202 (맞는 해석)

// 두 바이트를 unsigned 16비트로 합치기
int value = ((data[0] & 0xFF) << 8) | (data[1] & 0xFF);
System.out.println(Integer.toHexString(value));  // cafe
```

이 마스킹을 빼먹으면 바이트 배열을 다루는 코드가 무작위로 음수를 뱉기 시작한다. 자바 InputStream의 `read()`가 -1을 EOF로 쓰는 게 아니라 굳이 int 반환형을 쓰는 이유도 부호 확장 회피와 EOF 구분 때문이다.

C에서도 `signed char`와 `unsigned char` 차이로 같은 문제를 겪는다. ASCII 범위(0~127)만 다룰 때는 표시가 안 나다가, 한글이나 바이너리 데이터가 들어오면 음수가 튀어나오면서 깨진다. C 표준에서 `char`의 부호성은 구현 정의(implementation-defined)라서 플랫폼마다 다르다. ARM은 기본 unsigned, x86은 기본 signed인 경우가 많다. 바이너리 다루는 코드는 무조건 `uint8_t`를 써야 한다.

### 2의 보수 한눈에 읽기

음수의 16진수 표기를 보고 "이게 얼마지" 빠르게 가늠하는 요령이 있다. 32비트 부호 있는 정수에서 `0x80000000`이 가장 작은 음수(`Integer.MIN_VALUE`, -2,147,483,648)이고 `0xFFFFFFFF`가 -1이다. 즉 `0xFFFF...` 패턴으로 시작하면 작은 음수, `0x80...` ~ `0xC0...` 정도면 큰 음수다.

| 16진수 | 32비트 signed | 의미 |
|--------|---------------|------|
| `0x00000000` | 0          | 0 |
| `0x7FFFFFFF` | 2147483647 | INT_MAX |
| `0x80000000` | -2147483648 | INT_MIN |
| `0xFFFFFFFF` | -1         | 모든 비트 켜짐 |
| `0xFFFFFFFE` | -2         | -1보다 1 작음 |

`0x7FFFFFFF + 1 = 0x80000000`인데, signed로 해석하면 `INT_MAX + 1 = INT_MIN`이라는 정수 오버플로우 사고가 난다. 게임 엔진에서 점수를 int에 누적하다가 음수로 뒤집히는 그 버그다. C/C++의 signed overflow는 정의되지 않은 동작(undefined behavior)이라 컴파일러 최적화에 따라 버그가 더 이상하게 발현된다.

## JavaScript의 BigInt 함정

JavaScript의 Number는 IEEE 754 배정밀도(double)다. 정수 표현은 가수부 52비트 + 암묵적 1비트 = 53비트까지만 정확하다. 즉 `Number.MAX_SAFE_INTEGER`는 `2^53 - 1` = `9007199254740991`이다. 이걸 16진수로 보면 `0x1FFFFFFFFFFFFF`(13자리, 약 53비트).

64비트 정수를 다루는 시스템(타임스탬프 ns, Twitter Snowflake ID, 데이터베이스 BIGINT 등)에서 이 한계가 자주 터진다.

```javascript
// 64비트 16진수 리터럴 - 정밀도 손실
const id = 0x1234567890ABCDEF;
console.log(id);              // 1311768467463790000  (마지막 자리 깨짐)
console.log(id.toString(16)); // "1234567890abcd00"   (DEF가 사라짐)

// BigInt로 처리
const idBig = 0x1234567890ABCDEFn;  // 끝에 n
console.log(idBig);              // 1311768467463790063n
console.log(idBig.toString(16)); // "1234567890abcdef"
```

리터럴 끝에 `n`을 붙이면 BigInt가 된다. JSON에는 BigInt 타입이 없어서 BigInt를 그냥 `JSON.stringify`하면 예외가 난다. API 응답으로 64비트 ID를 받을 때 서버가 숫자로 직렬화하면 클라이언트에서 정밀도가 깨지므로, Snowflake ID 같은 건 보통 문자열로 직렬화한다. Twitter API가 `id`와 `id_str` 두 필드를 같이 주는 게 이 이유다.

비트 연산자는 더 까다롭다. JavaScript의 `&`, `|`, `<<`, `>>`는 피연산자를 32비트 정수로 변환한 뒤 동작한다. 따라서 `Number`로 32비트를 넘는 비트 연산을 하려고 하면 의도와 다른 결과가 나온다.

```javascript
const big = 0x1FFFFFFFF;       // 33비트 값
console.log(big & 0xFFFFFFFF); // -1  (32비트로 잘리면서 부호 확장)

const bigInt = 0x1FFFFFFFFn;
console.log(bigInt & 0xFFFFFFFFn); // 4294967295n  (정상)
```

64비트 마스크가 필요한 코드는 처음부터 BigInt로 통일해야 한다. Number와 BigInt는 자동 변환되지 않아서 섞으면 `TypeError`가 나는데, 이게 오히려 안전장치 역할을 한다.

## hexdump와 디버거 출력 읽기

바이너리 파일을 들여다볼 때는 보통 `xxd`나 `od` 같은 도구를 쓴다. 출력 형식만 알면 디스크에 어떤 바이트가 있는지 바로 본다.

`xxd` 출력 예시:

```
$ xxd image.png | head -3
00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452  .PNG........IHDR
00000010: 0000 0100 0000 0100 0802 0000 0090 7753  ..............wS
00000020: de00 0000 0c49 4441 5478 da63 6000 0200  .....IDATx.c`...
```

세 영역으로 나뉜다. 왼쪽은 파일 오프셋(16진수), 가운데는 16바이트씩의 16진수 덤프, 오른쪽은 같은 데이터의 ASCII 표현이다. 출력 가능한 ASCII가 아니면 점(`.`)으로 표시한다. PNG 파일의 매직 넘버 `89 50 4E 47 0D 0A 1A 0A`이 첫 8바이트로 보이고, ASCII 영역에서 `.PNG`라고 보이는 게 그 부분이다(`50 4E 47` = `PNG`).

`od` 출력은 옵션에 따라 다르다. 16진수 + ASCII 조합은 `od -A x -t x1z -v`로 본다.

```
$ od -A x -t x1z -v image.png | head -3
000000 89 50 4e 47 0d 0a 1a 0a 00 00 00 0d 49 48 44 52  >.PNG........IHDR<
000010 00 00 01 00 00 00 01 00 08 02 00 00 00 90 77 53  >..............wS<
```

읽을 때 핵심은 두 가지. 첫째, 좌측 오프셋도 16진수다. `00000010`은 16바이트 위치, `00000020`은 32바이트 위치다. 둘째, 가운데 16진수 영역에서 두 자리가 1바이트다. PNG 청크 길이 같은 4바이트 빅엔디언 정수를 읽으려면 4쌍을 묶어서 보면 된다. 위 출력에서 `00 00 00 0d`는 `0x0000000D` = 13, IHDR 청크의 데이터 길이가 13바이트라는 뜻이다.

엔디언이 다르면 같은 바이트열도 다르게 해석된다. x86은 리틀엔디언이라 메모리에 `0xCAFEBABE`를 4바이트 정수로 저장하면 디스크/덤프에는 `BE BA FE CA` 순으로 보인다. 자바 클래스 파일 매직 넘버가 `CA FE BA BE`로 보이는 건 자바 클래스 파일이 빅엔디언이기 때문이다. 네트워크 프로토콜도 빅엔디언(network byte order)이라 패킷 캡처에서는 항상 빅엔디언으로 본다.

GDB나 LLDB 같은 디버거에서도 `x/16xb $rsp` 같은 명령으로 메모리를 16진수 바이트 단위로 본다. `x` 명령은 examine, `/16xb`는 16개를 16진수(`x`)의 바이트(`b`) 형식으로라는 뜻이다. 디버깅하다가 메모리 주소 `0x7FFE...`처럼 큰 값을 보면 보통 스택 영역이고, `0x5555...`나 `0x4000...`은 코드/힙 쪽이다(ASLR 때문에 매번 바뀌긴 하지만 대략적인 영역 감각은 유지된다).

## 16진수 표기 컨벤션

같은 16진수 값이라도 언어/시스템마다 prefix가 다르다. 코드를 옮겨 쓸 때 prefix만 바꾸면 되는 경우가 많다.

| 표기 | 사용 환경 | 예시 |
|------|-----------|------|
| `0x...` | C, C++, Java, Python, JavaScript, Go, Rust 등 | `0xFF`, `0xCAFEBABE` |
| `#...`  | CSS, HTML 색상 | `#FF8040`, `#000` |
| `&H...` 또는 `&h...` | Visual Basic, BASIC 계열 | `&HFF`, `&hCAFE` |
| `$...`  | 6502/68000 어셈블리, Pascal, PowerShell | `$FF`, `$CAFE` |
| `0h...` | TI 어셈블러 일부 | `0hFF` |
| `...h`  | Intel 어셈블리 (suffix) | `0FFh`, `0CAFEh` |
| `16#...#` | Ada, VHDL | `16#FF#` |
| `x'...'` | SQL(일부 DB), DB2 | `x'FF'` |

Intel 어셈블리에서 `0FFh`처럼 앞에 `0`을 붙이는 건, 첫 글자가 알파벳이면 식별자로 오인되기 때문이다. `FFh`라고 쓰면 라벨 이름처럼 보여서 어셈블러가 거부한다. 그래서 항상 숫자(0~9)로 시작하도록 `0FFh`라고 쓴다.

CSS에서 `#FFF`는 `#FFFFFF`의 단축형이다. 각 자리를 두 번씩 반복해서 6자리로 펼친 것과 같다. `#ABC`는 `#AABBCC`. 이걸 모르면 `#F00`이 빨강이라는 게 갑자기 안 외워진다.

Python 3에서는 2진수 `0b...`, 8진수 `0o...`, 16진수 `0x...`로 prefix가 통일됐다. Python 2의 `0755`(8진수) 같은 옛날 표기는 사라졌는데, `0755`가 7백5십5인지 8진수 493인지 헷갈리던 사고를 막기 위함이다.

## 언어별 변환 코드

원리를 알면 코드는 짧다. 표준 라이브러리에 다 있다.

```javascript
// JavaScript
const hex = (0b1101011011).toString(16);    // "35b"
const bin = parseInt("35B", 16).toString(2); // "1101011011"
```

```python
# Python
hex_val = hex(0b1101011011)        # "0x35b"
bin_val = bin(0x35B)               # "0b1101011011"

# format()으로 prefix 없이 패딩
f"{0xCAFE:016b}"                   # "1100101011111110" (16자리 0 패딩)
f"{0b1101011011:04X}"              # "035B" (대문자, 4자리)
```

```java
// Java
String hex = Integer.toHexString(0b1101011011);     // "35b"
int bin   = Integer.parseInt("35B", 16);            // 859
String padded = String.format("%08X", 0xCA);        // "000000CA"

// Long은 명시적으로 unsigned 출력
String unsignedHex = Long.toHexString(-1L);         // "ffffffffffffffff"
```

```go
// Go
hex := fmt.Sprintf("%x", 0b1101011011)              // "35b"
hex2 := fmt.Sprintf("%08X", 0xCA)                   // "000000CA"
n, _ := strconv.ParseInt("35B", 16, 32)             // 859
```

서식 지정자 패턴은 거의 모든 언어가 비슷하다. `%x`는 소문자, `%X`는 대문자, `%08X`는 8자리 0 패딩 대문자. C printf 시절부터 굳어진 관례라 새 언어가 나와도 거의 그대로 따라간다. Rust의 `{:08X}`, Swift의 `String(format:)`도 같은 형식이다.

큰 수에 대한 라이브러리 함수는 정밀도 한계를 넘으면 조용히 잘못된 결과를 뱉는 게 아니라 다른 자료형으로 명시적 전환을 요구한다. JavaScript의 `parseInt`는 53비트 넘으면 정밀도가 깨지지만 예외는 안 던진다. 그래서 큰 16진수 문자열을 다룰 때는 BigInt 생성자를 직접 써야 한다.

```javascript
// 32비트 넘는 16진수 안전하게 파싱
const big = BigInt("0xCAFEBABEDEADBEEF");
console.log(big.toString(16));  // "cafebabedeadbeef"
```

## 자주 만나는 사고와 디버깅

byte 캐스팅 문제는 앞서 다뤘으니, 이외에 자주 보는 함정 몇 가지.

**서로 다른 endian의 바이트 스왑.** 네트워크에서 받은 4바이트를 그냥 `int`로 메모리 복사하면 x86에서는 바이트 순서가 뒤집힌다. `htonl`, `ntohl`, Java의 `ByteBuffer.order(ByteOrder.BIG_ENDIAN)` 같은 변환을 명시적으로 해야 한다. 디버깅할 때 hexdump 출력과 메모리 값을 비교하면 한쪽이 거꾸로 보여서 금방 눈에 띈다.

**문자열 인코딩 디버깅.** 한글이 깨졌을 때 hexdump를 보면 진단이 빠르다. UTF-8의 한글 한 글자는 3바이트(예: '가' = `EA B0 80`), CP949는 2바이트(`B0 A1`). 깨진 문자의 바이트 패턴을 보면 어느 인코딩으로 잘못 해석됐는지 추정할 수 있다. UTF-8 한글 3바이트 중 첫 바이트가 `EA`, `EB`, `EC`, `ED`로 시작하는 게 단서다(범위가 좁아서 식별이 쉽다).

**해시값 비교 실수.** `md5`, `sha256` 같은 출력은 16진수 문자열이라 대소문자 차이로 비교가 어긋날 수 있다. `"ABC123" == "abc123"`은 false다. 비교 전에 한쪽으로 정규화해야 한다. 또 16진수 문자열 길이로 알고리즘을 추정할 수 있다(MD5=32, SHA-1=40, SHA-256=64).

**0과 0x00 혼동.** 보호 모드 OS에서 메모리 주소 `0x00000000`을 역참조하면 segfault다. NullPointerException 트레이스에서 `0x0` 근처 주소가 나오면 거의 null 역참조다. 반면 `0xDEADBEEF`, `0xBAADF00D`, `0xFEEDFACE` 같은 값은 디버깅을 돕기 위해 메모리 초기화 패턴으로 일부러 쓴다(uninitialized memory marker). 디버거에서 이런 값이 보이면 "이 메모리는 아직 초기화 안 됐다"는 신호다.

## 실무 감각 정리

2진수/16진수 변환 자체는 4비트 묶기 한 줄로 끝난다. 진짜 어려운 건 같은 비트 패턴을 컨텍스트별로 다르게 해석하는 부분이다. `0xFF`가 unsigned면 255, signed byte면 -1, 마스크면 "하위 8비트 추출", flag면 "8개 옵션 다 켜짐", 색상이면 "최대 강도 채널"이다.

이 감각이 붙으면 디버거 출력이나 패킷 덤프를 보고 비트 단위로 뭐가 잘못됐는지 추론하는 속도가 달라진다. 코드 한 줄에 마법의 상수 `0xFF`가 보일 때 "왜 굳이 16진수로 썼지"가 아니라 "아 1바이트 마스크 또는 unsigned 변환이구나"가 자동으로 떠올라야 한다.
