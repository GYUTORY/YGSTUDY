# 엔디언과 2의 보수, IEEE 754 부동소수점

컴퓨터가 숫자를 저장하는 방식은 세 가지로 나뉜다.

- **엔디언**: 멀티바이트 데이터를 메모리에 어떤 순서로 배치하는가
- **2의 보수**: 음수 정수를 어떻게 표현하는가
- **IEEE 754**: 소수점이 있는 숫자를 어떻게 표현하는가

이 세 가지를 모르면 네트워크 패킷 파싱, 바이너리 파일 읽기, 금액 계산에서 버그가 난다. 원인을 찾기도 어렵다.

---

## 엔디언 (Endianness)

### 빅엔디언 vs 리틀엔디언

32비트 정수 `0x12345678`을 메모리 주소 0x00부터 저장한다고 하자.

| 주소 | 0x00 | 0x01 | 0x02 | 0x03 |
|------|------|------|------|------|
| 빅엔디언 | 12 | 34 | 56 | 78 |
| 리틀엔디언 | 78 | 56 | 34 | 12 |

빅엔디언은 사람이 읽는 순서 그대로다. 리틀엔디언은 LSB(최하위 바이트)가 먼저 온다.

- **빅엔디언 사용처**: 네트워크 프로토콜(TCP/IP), JVM, SPARC, PowerPC
- **리틀엔디언 사용처**: x86/x64(Intel, AMD), ARM(대부분), Windows/Linux/macOS

네트워크로 데이터를 보낼 때는 무조건 빅엔디언(네트워크 바이트 오더)으로 변환해야 한다. 안 하면 리틀엔디언 머신에서 보낸 `0x12345678`이 상대방에게 `0x78563412`로 읽힌다.

### 엔디언 감지 — 현재 시스템이 뭘 쓰는지 확인

**Python**

```python
import sys

print(sys.byteorder)  # 'little' 또는 'big'
```

**Java**

```java
import java.nio.ByteOrder;

public class EndianCheck {
    public static void main(String[] args) {
        System.out.println(ByteOrder.nativeOrder());
        // LITTLE_ENDIAN 또는 BIG_ENDIAN
    }
}
```

**JavaScript (Node.js)**

```javascript
const os = require('os');
console.log(os.endianness()); // 'LE' 또는 'BE'
```

**C**

```c
#include <stdio.h>
#include <stdint.h>

int main() {
    uint32_t val = 0x01;
    uint8_t *byte = (uint8_t *)&val;

    if (*byte == 0x01) {
        printf("Little-endian\n");
    } else {
        printf("Big-endian\n");
    }
    return 0;
}
```

원리는 간단하다. 멀티바이트 값을 넣고 첫 번째 바이트가 뭔지 보면 된다.

### 바이트 순서 변환 — 실무에서 자주 쓰는 패턴

**Java ByteBuffer**

```java
import java.nio.ByteBuffer;
import java.nio.ByteOrder;

// 리틀엔디언 바이너리 파일에서 읽은 4바이트
byte[] rawBytes = {0x78, 0x56, 0x34, 0x12};

ByteBuffer buffer = ByteBuffer.wrap(rawBytes);
buffer.order(ByteOrder.LITTLE_ENDIAN);
int value = buffer.getInt();  // 0x12345678

// 네트워크 전송용 빅엔디언으로 변환
ByteBuffer netBuffer = ByteBuffer.allocate(4);
netBuffer.order(ByteOrder.BIG_ENDIAN);
netBuffer.putInt(value);
byte[] networkBytes = netBuffer.array();  // {0x12, 0x34, 0x56, 0x78}
```

**Python struct 모듈**

```python
import struct

# 리틀엔디언으로 패킹
packed = struct.pack('<I', 0x12345678)
print(packed.hex())  # 78563412

# 빅엔디언으로 패킹
packed = struct.pack('>I', 0x12345678)
print(packed.hex())  # 12345678

# 바이너리 파일에서 리틀엔디언 4바이트 읽기
with open('data.bin', 'rb') as f:
    raw = f.read(4)
    value = struct.unpack('<I', raw)[0]
```

**주의할 점**: 바이너리 프로토콜이나 파일 포맷을 다룰 때, 명세서에 엔디언이 명시되어 있다. PNG는 빅엔디언, BMP는 리틀엔디언이다. 명세를 안 읽고 파싱하면 값이 뒤집혀서 나온다.

## 2의 보수 (Two's Complement)

### 왜 2의 보수를 쓰는가

음수를 표현하는 방법은 여러 가지다.

| 방식 | +5 (8비트) | -5 (8비트) | 0 표현 | 문제 |
|------|-----------|-----------|--------|------|
| 부호-크기 | 0000 0101 | 1000 0101 | +0, -0 두 개 | 덧셈/뺄셈 회로가 따로 필요 |
| 1의 보수 | 0000 0101 | 1111 1010 | +0, -0 두 개 | end-around carry 처리 필요 |
| 2의 보수 | 0000 0101 | 1111 1011 | 0 하나 | 없음 |

2의 보수에서는 덧셈기 하나로 덧셈과 뺄셈을 모두 처리한다. 하드웨어가 단순해진다. 0도 하나뿐이다. 그래서 현대 CPU는 전부 2의 보수를 쓴다.

### 변환 방법

양수에서 음수로 바꾸는 법: **비트 반전 후 1을 더한다.**

```
 5 = 0000 0101
     ↓ 비트 반전
     1111 1010
     ↓ +1
-5 = 1111 1011
```

수식으로는 `-x = ~x + 1`이다.

### 표현 범위

n비트 2의 보수 정수의 범위: **-2^(n-1) ~ 2^(n-1) - 1**

| 타입 | 비트 | 최솟값 | 최댓값 |
|------|------|--------|--------|
| byte (Java) | 8 | -128 | 127 |
| short | 16 | -32,768 | 32,767 |
| int | 32 | -2,147,483,648 | 2,147,483,647 |
| long | 64 | -9,223,372,036,854,775,808 | 9,223,372,036,854,775,807 |

음수가 양수보다 하나 많다. `Math.abs(Integer.MIN_VALUE)`가 음수를 반환하는 이유다.

### 정수 오버플로우 — 실제로 터지는 경우

**Java에서 오버플로우 확인**

```java
public class OverflowDemo {
    public static void main(String[] args) {
        int max = Integer.MAX_VALUE;  // 2147483647
        System.out.println(max + 1);  // -2147483648 (오버플로우)

        // 배열 크기 계산에서 오버플로우
        int length = 1_000_000;
        int elementSize = 4000;
        int totalSize = length * elementSize;  // 오버플로우 발생
        System.out.println(totalSize);  // -294967296 (음수!)

        // 안전한 방법: Math.multiplyExact (Java 8+)
        try {
            int safe = Math.multiplyExact(length, elementSize);
        } catch (ArithmeticException e) {
            System.out.println("오버플로우 감지: " + e.getMessage());
        }

        // 또는 long으로 먼저 캐스팅
        long safeTotalSize = (long) length * elementSize;
        System.out.println(safeTotalSize);  // 4000000000
    }
}
```

**Python은 오버플로우가 없다**

```python
# Python int는 크기 제한이 없다 (arbitrary precision)
x = 2**63
print(x + 1)  # 9223372036854775809 — 정상 동작

# 하지만 numpy는 C 타입이라 오버플로우가 발생한다
import numpy as np
a = np.int32(2147483647)
print(a + np.int32(1))  # -2147483648
```

**부호 확장 — 타입 변환 시 주의할 점**

```java
byte b = -1;          // 1111 1111
int i = b;            // 1111 1111 1111 1111 1111 1111 1111 1111 = -1 (부호 확장)
int unsigned = b & 0xFF;  // 0000 0000 0000 0000 0000 0000 1111 1111 = 255

System.out.println(i);        // -1
System.out.println(unsigned);  // 255
```

`byte`를 `int`로 캐스팅하면 부호 비트가 상위 비트로 채워진다. 바이너리 데이터를 다룰 때 `& 0xFF`로 마스킹하지 않으면 음수 값이 튀어나온다. Java에서 바이트 배열 파싱할 때 가장 흔한 실수다.

## IEEE 754 부동소수점

### 비트 구조

부동소수점은 세 부분으로 나뉜다: **부호(Sign)**, **지수(Exponent)**, **가수(Mantissa/Significand)**.

```
단정밀도 (float, 32비트)
┌───┬──────────┬───────────────────────┐
│ S │ Exponent │       Mantissa        │
│1b │   8bit   │        23bit          │
└───┴──────────┴───────────────────────┘

배정밀도 (double, 64비트)
┌───┬─────────────┬────────────────────────────────────────────────────┐
│ S │  Exponent   │                    Mantissa                       │
│1b │   11bit     │                     52bit                         │
└───┴─────────────┴────────────────────────────────────────────────────┘
```

값 계산: `(-1)^S × 1.Mantissa × 2^(Exponent - Bias)`

- float의 Bias: 127
- double의 Bias: 1023

가수 앞에 "숨은 1"(hidden bit)이 붙는다. 23비트 가수 필드지만 실질 정밀도는 24비트다.

### 특수값

| 지수 | 가수 | 의미 |
|------|------|------|
| 0 | 0 | ±0 |
| 0 | ≠ 0 | 비정규화 수 (denormalized) |
| 모두 1 | 0 | ±∞ |
| 모두 1 | ≠ 0 | NaN |

```java
System.out.println(1.0 / 0.0);    // Infinity
System.out.println(-1.0 / 0.0);   // -Infinity
System.out.println(0.0 / 0.0);    // NaN
System.out.println(Double.NaN == Double.NaN);  // false (NaN은 자기 자신과도 같지 않다)
```

NaN 비교는 항상 false다. `Double.isNaN()`을 써야 한다.

### 0.1 + 0.2 문제 — 왜 0.3이 아닌가

0.1은 이진수로 `0.0001100110011...`의 무한 순환소수다. 유한 비트에 담으면 잘린다.

**Java**

```java
System.out.println(0.1 + 0.2);             // 0.30000000000000004
System.out.println(0.1 + 0.2 == 0.3);      // false
```

**Python**

```python
print(0.1 + 0.2)           # 0.30000000000000004
print(0.1 + 0.2 == 0.3)    # False
```

**JavaScript**

```javascript
console.log(0.1 + 0.2);           // 0.30000000000000004
console.log(0.1 + 0.2 === 0.3);   // false
```

모든 언어에서 동일하게 발생한다. IEEE 754를 쓰는 한 피할 수 없다.

### 비교는 epsilon으로

```java
double a = 0.1 + 0.2;
double b = 0.3;
double epsilon = 1e-10;

// 잘못된 비교
if (a == b) { /* 절대 여기 안 온다 */ }

// 올바른 비교
if (Math.abs(a - b) < epsilon) {
    System.out.println("같다고 본다");
}
```

```python
import math

a = 0.1 + 0.2
b = 0.3

# Python 3.5+
print(math.isclose(a, b))  # True (기본 상대 허용 오차 1e-9)

# 허용 오차 직접 지정
print(math.isclose(a, b, rel_tol=1e-9, abs_tol=1e-12))
```

### 금액 계산 — float/double 쓰면 안 된다

```java
// 하면 안 되는 코드
double price = 0.10;
double quantity = 3;
System.out.println(price * quantity);  // 0.30000000000000004

// BigDecimal 사용
import java.math.BigDecimal;
import java.math.RoundingMode;

BigDecimal priceDecimal = new BigDecimal("0.10");  // 문자열로 생성해야 한다
BigDecimal qty = new BigDecimal("3");
BigDecimal total = priceDecimal.multiply(qty);
System.out.println(total);  // 0.30

// 주의: new BigDecimal(0.1)은 쓰면 안 된다
System.out.println(new BigDecimal(0.1));
// 0.1000000000000000055511151231257827021181583404541015625
// double 0.1의 실제 값이 그대로 들어간다

// 나눗셈은 RoundingMode 필수
BigDecimal result = new BigDecimal("10").divide(
    new BigDecimal("3"), 2, RoundingMode.HALF_UP
);
System.out.println(result);  // 3.33
```

**Python에서 Decimal**

```python
from decimal import Decimal, ROUND_HALF_UP

# 문자열로 생성
price = Decimal('0.10')
total = price * 3
print(total)  # 0.30

# float로 생성하면 안 된다
bad = Decimal(0.1)
print(bad)  # 0.1000000000000000055511151231257827021181583404541015625

# 반올림
result = Decimal('10') / Decimal('3')
print(result.quantize(Decimal('0.01'), rounding=ROUND_HALF_UP))  # 3.33
```

### 결합법칙이 성립하지 않는다

```python
a = 1e20
b = -1e20
c = 1.0

print((a + b) + c)  # 1.0
print(a + (b + c))  # 0.0
```

큰 수와 작은 수를 더하면 작은 수가 씹힌다. `a + (b + c)`에서 `b + c`는 `-1e20 + 1.0`인데, 1.0이 `-1e20`에 비해 너무 작아서 사라진다. 과학 계산에서 합산 순서를 바꾸면 결과가 달라지는 이유다.

### 부동소수점 비트 들여다보기

```java
long bits = Double.doubleToLongBits(0.1);
System.out.println(Long.toBinaryString(bits));
// 11111110111001100110011001100110011001100110011001100110011010

int floatBits = Float.floatToIntBits(0.1f);
System.out.printf("부호: %d%n", (floatBits >> 31) & 1);
System.out.printf("지수: %d (실제: %d)%n",
    (floatBits >> 23) & 0xFF,
    ((floatBits >> 23) & 0xFF) - 127);
System.out.printf("가수: %s%n",
    Integer.toBinaryString((floatBits & 0x7FFFFF) | 0x800000));
```

```python
import struct

# float의 바이트 표현 확인
packed = struct.pack('!f', 0.1)  # 빅엔디언 float
print(' '.join(f'{b:08b}' for b in packed))
# 00111101 11001100 11001100 11001101
```

디버깅할 때 값이 이상하면 비트 레벨에서 확인하는 게 빠르다.

## 세 개념이 엮이는 순간

네트워크로 float 값을 전송하는 상황을 생각해 보자.

1. float `3.14`의 IEEE 754 비트 패턴: `0x4048F5C3`
2. 이 4바이트를 리틀엔디언 머신 메모리에서 읽으면: `C3 F5 48 40`
3. 네트워크로 보내려면 빅엔디언으로 변환: `40 48 F5 C3`
4. 수신 측에서 자기 시스템의 엔디언에 맞게 다시 변환

```java
import java.nio.ByteBuffer;
import java.nio.ByteOrder;

float value = 3.14f;

// 네트워크 전송용 (빅엔디언)
ByteBuffer sendBuf = ByteBuffer.allocate(4);
sendBuf.order(ByteOrder.BIG_ENDIAN);
sendBuf.putFloat(value);
byte[] networkData = sendBuf.array();

// 수신 측에서 복원
ByteBuffer recvBuf = ByteBuffer.wrap(networkData);
recvBuf.order(ByteOrder.BIG_ENDIAN);
float received = recvBuf.getFloat();  // 3.14
```

정수 필드에 2의 보수로 저장된 음수 값이 들어 있고, 엔디언이 다른 시스템에서 읽으면 부호 비트 위치까지 바뀐다. 바이너리 프로토콜을 설계하거나 파싱할 때 세 개념 모두 알아야 하는 이유다.
