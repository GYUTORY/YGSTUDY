# Binary와 Bit의 차이

## 같은 말처럼 쓰지만 다르다

실무에서 "8비트 정수"와 "8자리 이진수"를 같은 말로 쓰는 경우가 흔하다. 결과적으로 같은 비트 패턴을 가리키긴 해도, 두 표현이 가리키는 개념은 서로 다르다. 이 차이를 흐릿하게 두면 네트워크 대역폭 계산에서 8배 오차가 나거나, 패킷 헤더 파싱에서 필드 경계를 잘못 자르는 일이 생긴다.

핵심만 먼저 짚으면, **Bit는 정보의 단위(unit)**이고 **Binary는 수를 표현하는 체계(system)**다. 비트는 "얼마만큼의 정보인가"를 재는 자(尺)이고, 이진수는 "그 정보를 어떻게 적을 것인가"의 표기법이다. 자와 표기법은 같이 다니지만 같은 것이 아니다.

## 개념 비교

| 항목 | Bit | Binary |
|---|---|---|
| 정의 | 정보의 최소 단위 (0 또는 1 한 자리) | 0과 1만으로 수를 적는 기수법 (밑이 2) |
| 영문 풀이 | Binary digIT | Base-2 numeral system |
| 단위로서의 의미 | 1비트 = 1자리의 정보량 | 단위가 아님 — 표기 체계 |
| 셈하는 대상 | 저장 용량, 전송 속도, 엔트로피 | 어떤 정수를 어떻게 표기할지 |
| 측정 | "이 채널은 100Mbps다" | "0xFF는 이진수로 11111111이다" |
| 관련 표준 | IEC 80000-13 (정보량 단위) | 위치 기수법 (밑 2) |
| 표기 | b (소문자), 8b = 1B | 0b... 또는 ...₂ |

비트는 "온도 단위(℃)"에 가깝고, 이진수는 "온도를 표시하는 숫자판 형식"에 가깝다. 25℃를 25라고 적든 0b11001이라고 적든 정보량(=비트 수가 아니라 의미)은 같다. 다만 어떤 표기법으로 적었는가가 다를 뿐이다.

엄밀히 말하면 정보 이론에서의 비트는 "확률 1/2 사건이 가지는 정보량"으로 정의되며, 항상 정수일 필요도 없다. 어떤 사건의 발생 확률이 0.7이면 그 사건의 정보량은 약 0.51비트다. 다만 컴퓨터 시스템에서는 저장과 전송이 0/1 두 상태로 떨어지기 때문에 정수 비트로 다루는 일이 압도적으로 많다.

## "8비트 정수"와 "8자리 이진수"의 차이

8비트 정수는 **0~255의 값을 담는 저장 공간 8칸**을 가리킨다. 즉, 자료형의 너비를 말한다. 반면 8자리 이진수는 **수 하나를 0/1 8개로 적은 표기**다. 다음 두 문장을 비교해 보면 차이가 분명해진다.

- "이 필드는 8비트 정수다." → 항상 0~255 (또는 부호 있으면 -128~127) 범위 안에서만 값을 가진다는 뜻. 9를 담아도 8비트를 차지한다.
- "이 수의 8자리 이진수 표현은 00001001이다." → 9라는 값 자체를 8자리짜리 이진수로 적어놨다는 뜻.

같은 9라는 값이라도 16비트 정수에 담으면 16자리 이진수(`0000000000001001`)가 된다. 표기 자릿수와 자료형 너비는 별개다. 이걸 헷갈리면 비트마스크를 만들 때 상위 비트를 의도치 않게 0으로 깎아내는 버그가 생긴다.

## bps와 Bps, 8배 차이의 단골 버그

네트워크/스토리지 단위 표기에서 b(소문자)는 비트, B(대문자)는 바이트다. 둘은 8배 차이가 난다.

- 100Mbps 회선 = 100,000,000 비트/초 = **약 12.5MB/s** 다운로드 (헤더 오버헤드 빼면 더 적음)
- 100MB/s 디스크 쓰기 = 800Mbps

운영팀과 대화하다 "회선이 1Gbps인데 왜 백업이 1GB/s가 안 나오냐"는 질문을 받는다. 답은 단위가 다르기 때문이다. 1Gbps는 이론상 125MB/s이고, 실제로는 TCP/IP 오버헤드와 스위치 부하로 100~115MB/s 정도가 한계다.

비슷한 자리에서 자주 보는 단위들:

- MTU 1500바이트, 점보 프레임 9000바이트 → 항상 **B(byte)** 단위
- 회선 속도 1Gbps, 10Gbps → 항상 **b(bit)** 단위
- iperf3 결과의 `Bitrate` 컬럼 → 기본은 비트(bps)지만 `-f M`이면 바이트(MB/s)로 바뀜

`bit rate`라고 쓰여 있으면 비트 단위, `throughput`이라고만 쓰여 있으면 도구 기본값을 확인해야 한다. iperf3와 dd, scp가 같은 전송에 대해 서로 다른 숫자를 뱉어내는 일은 단위 차이에서 오는 게 대부분이다.

## 코드에서의 비트 연산과 이진수 리터럴

비트 연산자는 "비트 단위로 1과 0을 다루는 연산"이고, 이진수 리터럴은 "정수를 이진수 표기로 적는 방법"이다. 둘은 같이 쓰일 뿐 별개의 기능이다.

### JavaScript

```javascript
// 이진수 리터럴 — 표기법
const a = 0b1010;        // 10
const b = 0b0011;        // 3

// 비트 연산자 — 비트 단위 연산
console.log(a & b);      // 0b0010 = 2  (AND)
console.log(a | b);      // 0b1011 = 11 (OR)
console.log(a ^ b);      // 0b1001 = 9  (XOR)
console.log(a << 2);     // 0b101000 = 40 (왼쪽 시프트)
console.log(a >> 1);     // 0b0101 = 5  (오른쪽 시프트, 부호 유지)
console.log(a >>> 1);    // 0b0101 = 5  (부호 없는 시프트)

// 출력 시 이진수 표기로 다시 적기
console.log((a & b).toString(2).padStart(4, '0')); // "0010"
```

JS에서 한 가지 함정은 비트 연산이 항상 32비트 부호 있는 정수로 수행된다는 점이다. `(1 << 31)`은 `-2147483648`이 되고, `(1 << 32)`는 0이 아니라 1이 된다(시프트 양이 32로 나눠진다). 큰 값에 비트 연산을 쓰려면 `BigInt`로 감싸고 `1n << 33n` 식으로 적어야 한다.

### Java

```java
public class BitDemo {
    public static void main(String[] args) {
        int a = 0b1010;          // 10, 이진수 리터럴
        int b = 0b0011;          // 3
        
        System.out.println(a & b);       // 2
        System.out.println(a | b);       // 11
        System.out.println(a ^ b);       // 9
        System.out.println(a << 2);      // 40
        System.out.println(a >> 1);      // 5
        System.out.println(a >>> 1);     // 5 (부호 없는 시프트)
        
        // 표기로 되돌리기
        System.out.println(Integer.toBinaryString(a & b)); // "10"
        
        // byte는 signed 8비트라서 0xFF가 -1이 된다
        byte c = (byte) 0xFF;
        System.out.println(c);                    // -1
        System.out.println(c & 0xFF);             // 255 (int로 승격 후 마스킹)
    }
}
```

Java에서 `byte`가 부호 있는 8비트라는 점은 바이너리 프로토콜 파싱에서 매번 발목을 잡는다. `byte b = (byte) 0x80` 한 줄로는 128이 아니라 -128이 들어가고, `b & 0xFF` 패턴으로 풀어줘야 한다. C# `byte`(unsigned)나 Go `byte`(=uint8)와 다르다.

### Python

```python
a = 0b1010       # 10
b = 0b0011       # 3

print(a & b)     # 2
print(a | b)     # 3 ... 가 아니라 11. 다시 계산: 1010 | 0011 = 1011 = 11
print(a ^ b)     # 9
print(a << 2)    # 40
print(a >> 1)    # 5

# 표기로 되돌리기
print(bin(a & b))           # '0b10'
print(format(a & b, '04b')) # '0010'

# 파이썬 int는 임의 정밀도라 시프트 오버플로가 없다
print(1 << 100)  # 1267650600228229401496703205376
```

Python의 `int`는 자릿수 제한이 없다. JS/Java처럼 `1 << 31`에서 부호가 뒤집히는 일이 없는 대신, 외부 시스템(C 라이브러리, 네트워크 패킷)과 비트를 주고받을 때는 `struct.pack`/`int.to_bytes(length, byteorder)`로 명시적인 너비를 잡아줘야 한다. 그렇지 않으면 8바이트짜리 필드에 9바이트가 흘러가는 일이 생긴다.

## "binary"라는 단어의 또 다른 의미 — 인코딩 라벨

라이브러리 API에서 `'binary'`라는 문자열을 만나면 이게 진짜 이진수와는 별 상관 없는 문맥일 때가 많다.

```javascript
const buf = Buffer.from('hello', 'utf8');
console.log(buf.toString('binary'));  // "hello"
console.log(buf.toString('hex'));     // "68656c6c6f"
```

Node.js Buffer의 `'binary'`는 ISO-8859-1(Latin-1) 인코딩의 별칭이다. 각 바이트(0~255)를 그대로 같은 코드포인트의 유니코드 문자로 매핑한다. 그래서 ASCII 범위 안의 문자는 그대로 나오지만, 한글이나 이모지를 거치면 깨진다.

```javascript
const buf = Buffer.from('안녕', 'utf8');
console.log(buf.length);              // 6 (UTF-8 기준)
console.log(buf.toString('binary'));  // "ì\x95\x88ë\x85\x95" — 깨짐
console.log(buf.toString('utf8'));    // "안녕"
```

Node 공식 문서도 `'binary'`는 deprecated이며 `'latin1'`을 쓰라고 안내한다. 코드 리뷰에서 `toString('binary')`를 마주치면 "이걸 정말 Latin-1로 다루려는 건지, 아니면 hex/base64로 바꿔야 하는 건지" 물어봐야 한다. 옛날 코드일수록 본인은 "이진 데이터를 문자열로 보존"한다고 믿었지만 사실은 인코딩 손상을 만들고 있는 경우가 많다.

비슷한 혼동이 다른 언어/라이브러리에도 있다.

- Python `open(path, 'rb')`의 `b`는 binary mode — 디코딩 없이 바이트로 읽으라는 뜻이지 "이진수"와는 무관
- HTTP `Content-Transfer-Encoding: binary` — 변환 없이 그대로 전송하라는 의미
- MySQL `BINARY` 타입 — 고정 길이 바이너리 문자열, 비트 시퀀스가 아님

이 자리에서의 binary는 거의 다 **"텍스트가 아닌 바이트 그대로"**라는 뜻이고, 이진수 표기와는 다른 층위의 단어다.

## 비트 단위 필드를 가진 패킷 파싱

네트워크 헤더는 8의 배수가 아닌 비트 폭으로 필드를 잘게 쪼개 둔다. IPv4 헤더의 첫 바이트가 대표적이다.

```
 0                   1
 0 1 2 3 4 5 6 7 8 ...
+-+-+-+-+-+-+-+-+-+-...
|Version|  IHL  | TOS ...
+-+-+-+-+-+-+-+-+-+-...
```

Version 4비트 + IHL(Internet Header Length) 4비트가 한 바이트(8비트) 안에 같이 들어 있다. 패킷 캡처에서 첫 바이트가 `0x45`로 보이는 게 바로 이 조합이다 — `4`가 IPv4 버전이고 `5`가 IHL=5(=20바이트 헤더)다.

이걸 코드로 풀려면 비트 시프트와 마스킹이 필요하다.

```python
first_byte = 0x45                    # 패킷 첫 바이트
version = (first_byte >> 4) & 0x0F   # 상위 4비트 -> 4
ihl     = first_byte & 0x0F          # 하위 4비트 -> 5
header_len = ihl * 4                  # 20바이트
```

여기서 `& 0x0F`를 빼먹으면 무슨 일이 생기는가? 보통의 IPv4 패킷은 상위 4비트가 `0100`이라 빼먹어도 결과가 같아 보인다. 그러다 IPv6 트래픽이 섞여 들어오면 첫 바이트가 `0x60` 같은 값이 되고, 상위 비트를 마스킹하지 않은 코드는 `ihl=6`으로 오해해서 헤더 길이를 24바이트로 계산하고 그 다음 페이로드 오프셋이 통째로 어긋난다. tcpdump 결과는 정상인데 직접 짠 파서만 이상한 값을 뱉는 케이스가 이 패턴이다.

비슷하게 TCP 플래그 바이트도 1비트씩 8개의 플래그(URG, ACK, PSH, RST, SYN, FIN 등)가 모여 있다. `flags & 0x02`로 SYN을 검사하고 `flags & 0x10`으로 ACK를 검사한다. 이런 코드를 쓰면서 "8비트 정수 안에 비트 8개가 들어 있다"는 사실과 "이 정수를 0b... 표기로 적으면 어떻게 보이는가"를 분리해서 생각해야 한다. 정수의 너비(8비트)와 그 정수의 이진수 표기(8자리)가 우연히 일치할 뿐, 16비트 필드 안의 4비트 부분 필드를 다룰 때는 표기 자릿수가 16자리라는 점을 잊으면 마스크를 잘못 만든다.

## 정리

Bit은 정보를 세는 단위, Binary는 수를 적는 표기법이다. 같은 비트 패턴을 가리키더라도, 단위로 말할 때(대역폭, 자료형 너비)와 표기로 말할 때(리터럴, 헤더 덤프) 머릿속에서 구분해야 한다.

실무에서 두 개념을 분리해 두면 다음과 같은 함정에 덜 빠진다.

- bps와 Bps의 8배 차이를 자동으로 의식하게 된다
- 자료형 너비(8비트 정수)와 표기 자릿수(8자리 이진수)가 다른 차원임을 안다
- API의 `'binary'`가 인코딩 라벨일 가능성을 먼저 의심한다
- 비트 폭이 8의 배수가 아닌 패킷 필드를 마스킹/시프트로 정확히 잘라낸다

비트와 이진수가 자주 같이 쓰이는 건 사실이지만, "비슷한 말"로 묶어 두면 위 네 가지 자리에서 매번 같은 종류의 버그를 다시 만든다.
