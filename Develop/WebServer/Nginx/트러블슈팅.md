# Nginx 트러블슈팅

Nginx 운영 중 반복적으로 마주치는 에러들과 설정 실수를 정리했다. 대부분 설정 한 줄 차이로 발생하는 문제들이라, 원인을 모르면 몇 시간씩 삽질하게 된다.

---

## 자주 발생하는 에러

### 502 Bad Gateway

upstream 서버와 연결 자체가 안 될 때 발생한다.

**주요 원인:**

1. upstream 서버가 죽어있다

```bash
# upstream 프로세스 확인
ps aux | grep gunicorn
systemctl status your-app

# 포트 리스닝 확인
ss -tlnp | grep 8000
```

2. 소켓 파일 경로가 틀렸거나 권한이 없다

```nginx
# 소켓 파일이 실제로 존재하는지 확인
upstream backend {
    server unix:/run/gunicorn/gunicorn.sock;
}
```

```bash
ls -la /run/gunicorn/gunicorn.sock
# srw-rw-rw- 이어야 nginx가 접근 가능
# nginx 워커가 해당 소켓에 접근할 수 있는 유저/그룹인지 확인
```

3. SELinux가 차단하고 있다 (CentOS/RHEL 계열)

```bash
# SELinux 로그 확인
ausearch -m avc -ts recent | grep nginx

# httpd가 네트워크 연결하도록 허용
setsebool -P httpd_can_network_connect 1
```

이 세 번째 원인을 모르면 CentOS에서 하루 종일 헤맨다. 로그에도 별다른 내용이 안 나오고, 같은 설정이 Ubuntu에서는 되는데 CentOS에서만 안 되는 상황이 생긴다.

### 504 Gateway Timeout

upstream 서버가 응답은 하는데 너무 느릴 때 발생한다.

```nginx
location /api/ {
    proxy_pass http://backend;

    # 기본값이 60초인데, 무거운 API는 이걸 넘긴다
    proxy_connect_timeout 10s;   # 연결 수립 대기
    proxy_send_timeout 60s;      # 요청 전송 대기
    proxy_read_timeout 120s;     # 응답 수신 대기
}
```

주의할 점은 timeout 값을 무작정 늘리는 게 답이 아니라는 것이다. API 자체가 왜 느린지를 먼저 봐야 한다. timeout을 300초로 늘려놓으면 느린 요청이 커넥션을 계속 물고 있어서 다른 요청까지 영향을 받는다.

**upstream에서 keepalive를 쓸 때 빠지는 함정:**

```nginx
upstream backend {
    server 127.0.0.1:8000;
    keepalive 32;
}

location /api/ {
    proxy_pass http://backend;

    # keepalive를 쓰려면 이 두 줄이 반드시 있어야 한다
    proxy_http_version 1.1;
    proxy_set_header Connection "";
}
```

`keepalive`를 설정해놓고 `proxy_http_version`과 `Connection` 헤더를 안 넣으면, keepalive가 동작하지 않아서 매 요청마다 새 연결을 맺는다. 부하가 높을 때 504가 간헐적으로 발생하는 원인이 된다.

### 413 Request Entity Too Large

파일 업로드 시 가장 흔하게 보는 에러다.

```nginx
# http, server, location 블록 어디든 넣을 수 있다
client_max_body_size 50m;
```

기본값이 1MB라서, 이미지 업로드만 해도 바로 걸린다.

주의: Nginx 뒤에 또 다른 리버스 프록시(AWS ALB 등)가 있으면 거기서도 제한이 걸릴 수 있다. Nginx 설정만 바꾸고 "왜 안 되지?" 하는 경우가 많다.

```nginx
# 대용량 파일 업로드가 필요한 경로만 따로 설정
location /upload {
    client_max_body_size 500m;
    client_body_timeout 120s;
    client_body_buffer_size 128k;

    proxy_pass http://backend;
    proxy_request_buffering off;  # 큰 파일은 버퍼링 끄는 게 낫다
}
```

`proxy_request_buffering off`를 안 하면 Nginx가 파일 전체를 메모리(또는 임시 파일)에 다 받은 뒤에 upstream으로 보낸다. 대용량 파일이면 디스크를 잡아먹는다.

### 403 Forbidden

**원인 1: 파일 권한**

```bash
# Nginx 워커 프로세스의 유저 확인
grep "user" /etc/nginx/nginx.conf
# user nginx; 또는 user www-data;

# 해당 유저가 파일을 읽을 수 있는지 확인
namei -l /var/www/html/index.html
```

`namei` 명령이 핵심이다. 파일 자체 권한은 맞는데 상위 디렉토리에 실행 권한(x)이 없어서 403이 나는 경우가 많다.

```bash
# 전형적인 해결
chmod 755 /var/www
chmod 755 /var/www/html
chmod 644 /var/www/html/index.html
```

**원인 2: index 파일이 없고 autoindex도 꺼져있다**

```nginx
location / {
    root /var/www/html;
    # index.html이 없으면 403 반환
    index index.html index.htm;
}
```

디렉토리에 `index.html`이 없으면 Nginx는 디렉토리 목록을 보여주는 대신 403을 반환한다. `autoindex on;`으로 목록을 보여줄 수도 있지만, 프로덕션에서는 보안상 켜면 안 된다.

## 설정에서 자주 실수하는 패턴

### proxy_pass 슬래시 함정

이게 Nginx에서 가장 흔한 실수다. 슬래시 하나 차이로 동작이 완전히 달라진다.

```nginx
# 케이스 1: 슬래시 없음
location /api/ {
    proxy_pass http://backend;
}
# /api/users → http://backend/api/users (경로 그대로 전달)

# 케이스 2: 슬래시 있음
location /api/ {
    proxy_pass http://backend/;
}
# /api/users → http://backend/users (/api/ 부분이 잘린다)

# 케이스 3: 경로 포함
location /api/ {
    proxy_pass http://backend/v2/;
}
# /api/users → http://backend/v2/users (/api/가 /v2/로 치환)
```

규칙은 단순하다: `proxy_pass`에 URI(슬래시 포함 경로)가 붙으면 `location`에 매칭된 부분이 치환된다. URI가 없으면 원래 경로 그대로 전달된다.

```nginx
# 흔한 실수 예시
location /app {
    proxy_pass http://backend/;
}
# /app → http://backend/
# /application → http://backend/lication  ← 의도하지 않은 결과

# location 끝에 슬래시를 붙여야 한다
location /app/ {
    proxy_pass http://backend/;
}
```

### add_header 상속 규칙

이건 모르면 보안 헤더가 빠지는 사고가 난다.

```nginx
server {
    add_header X-Frame-Options "SAMEORIGIN";
    add_header X-Content-Type-Options "nosniff";

    location / {
        proxy_pass http://backend;
        # 여기서는 위의 헤더가 상속된다
    }

    location /api/ {
        add_header X-Request-Id $request_id;
        proxy_pass http://backend;
        # X-Frame-Options, X-Content-Type-Options 전부 사라진다
        # add_header를 하나라도 쓰면 상위 블록의 add_header가 전부 무시된다
    }
}
```

**하위 블록에서 `add_header`를 하나라도 쓰면, 상위 블록의 `add_header`는 전부 상속되지 않는다.** 이건 Nginx 공식 문서에도 나와있지만, 실제로 겪기 전까지는 인식하기 어렵다.

해결 방법은 두 가지다.

```nginx
# 방법 1: 하위 블록에서 필요한 헤더를 전부 다시 선언
location /api/ {
    add_header X-Frame-Options "SAMEORIGIN";
    add_header X-Content-Type-Options "nosniff";
    add_header X-Request-Id $request_id;
    proxy_pass http://backend;
}

# 방법 2: include로 공통 헤더를 관리
# /etc/nginx/snippets/security-headers.conf
# add_header X-Frame-Options "SAMEORIGIN";
# add_header X-Content-Type-Options "nosniff";

location /api/ {
    include snippets/security-headers.conf;
    add_header X-Request-Id $request_id;
    proxy_pass http://backend;
}
```

방법 2가 관리하기 편하다. 헤더가 여러 개면 include로 빼는 게 실수를 줄인다.

### if is evil

Nginx의 `if`는 프로그래밍 언어의 `if`와 다르게 동작한다. `location` 안에서 `if`를 쓰면 예상치 못한 동작이 생긴다.

```nginx
# 위험한 패턴
location / {
    set $deny 0;

    if ($request_uri ~* "\.php$") {
        set $deny 1;
    }

    if ($deny = 1) {
        return 403;
    }

    # 이 위치의 다른 지시자들이 if 블록 안에서는 무시될 수 있다
    proxy_pass http://backend;
}
```

`if`는 내부적으로 임시 location을 만들어서 처리한다. 그래서 `if` 블록 안에서 `proxy_pass`나 `fastcgi_pass`가 사라지는 현상이 생긴다.

**안전하게 if를 쓸 수 있는 경우:**

```nginx
# return과 rewrite는 if 안에서 써도 된다
if ($request_method = POST) {
    return 405;
}

if ($http_user_agent ~* "badbot") {
    return 403;
}

# 변수 설정도 된다
set $backend "default";
if ($arg_version = "v2") {
    set $backend "v2";
}
```

**if 대신 쓸 수 있는 방법:**

```nginx
# map으로 변수 매핑 (http 블록에서 선언)
map $request_uri $is_blocked {
    default         0;
    ~*\.php$        1;
    ~*\.asp$        1;
}

server {
    if ($is_blocked) {
        return 403;
    }
}

# 조건부 라우팅은 별도 location으로 분리
location ~* \.php$ {
    return 403;
}

location / {
    proxy_pass http://backend;
}
```

`map`은 http 블록에서만 선언할 수 있다. server나 location 안에서는 못 쓴다.

## 설정 디버깅 순서

Nginx 설정이 의도대로 동작하지 않을 때, 아래 순서로 확인한다.

### 1. 설정 파일 문법 검사

```bash
nginx -t
# nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
# nginx: configuration file /etc/nginx/nginx.conf test is successful
```

이게 통과한다고 설정이 맞다는 뜻은 아니다. 문법만 검사하는 거다.

### 2. 실제 적용된 설정 확인

```bash
# 어떤 설정 파일들이 로드되는지 확인
nginx -T  # 대문자 T: 전체 설정을 stdout에 출력
nginx -T | grep -A5 "location /api"
```

`include`로 파일을 여러 개 나눠놓으면, 실제로 어떤 설정이 적용되는지 알기 어렵다. `-T` 옵션으로 합쳐진 전체 설정을 볼 수 있다.

### 3. 에러 로그 레벨 올리기

```nginx
# 디버깅할 때만 debug 레벨로 올린다
error_log /var/log/nginx/error.log debug;

# 특정 클라이언트 IP에서만 debug 로그를 남기려면
events {
    debug_connection 192.168.1.100;
}
```

debug 레벨은 로그 양이 엄청나다. 프로덕션에서 켜면 디스크가 금방 찬다. 반드시 특정 IP로 제한하거나, 확인 후 바로 끈다.

### 4. 요청이 어느 location에 매칭되는지 확인

```nginx
# 각 location에 고유한 헤더를 추가해서 확인
location / {
    add_header X-Debug-Location "root";
    proxy_pass http://backend;
}

location /api/ {
    add_header X-Debug-Location "api";
    proxy_pass http://backend;
}
```

```bash
curl -I http://localhost/api/users
# X-Debug-Location: api 가 나오는지 확인
```

### 5. upstream 연결 직접 테스트

```bash
# Nginx를 거치지 않고 upstream에 직접 요청
curl http://127.0.0.1:8000/api/users

# 소켓 파일로 직접 요청
curl --unix-socket /run/gunicorn/gunicorn.sock http://localhost/api/users
```

Nginx 문제인지 upstream 문제인지 분리하는 게 첫 번째다.

### 6. reload vs restart

```bash
# reload: 새 설정으로 워커 프로세스를 교체 (기존 연결 유지)
nginx -s reload

# restart: 프로세스 완전 재시작 (기존 연결 끊김)
systemctl restart nginx
```

설정 변경 후에는 `reload`를 쓴다. `restart`는 기존 연결을 끊기 때문에 프로덕션에서는 피해야 한다. 간혹 `reload`로 반영이 안 되는 것 같다고 `restart`를 하는 경우가 있는데, `reload`가 실패하면 에러 로그를 확인하는 게 맞다.

## 기타 흔한 문제

### worker_connections 부족

```nginx
events {
    worker_connections 1024;  # 기본값
}
```

동시 접속이 많으면 에러 로그에 `worker_connections are not enough`가 찍힌다.

```bash
# 현재 연결 수 확인
ss -s
# 또는
cat /proc/sys/fs/file-nr
```

`worker_connections`를 늘릴 때는 OS의 파일 디스크립터 제한도 함께 올려야 한다.

```bash
# /etc/security/limits.conf
# nginx    soft    nofile    65536
# nginx    hard    nofile    65536

# 또는 systemd 서비스 파일에서
# [Service]
# LimitNOFILE=65536
```

### 버퍼 관련 경고

에러 로그에 이런 메시지가 반복되면:

```
upstream sent too big header while reading response header from upstream
```

```nginx
proxy_buffer_size 8k;          # 응답 헤더용 버퍼 (기본 4k)
proxy_buffers 8 8k;            # 응답 본문용 버퍼
proxy_busy_buffers_size 16k;
```

세션이나 쿠키가 큰 애플리케이션에서 자주 발생한다. `proxy_buffer_size`를 8k나 16k로 올리면 해결된다.
