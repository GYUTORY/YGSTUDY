# AWS SNS (Simple Notification Service) 완전 가이드

## 서비스 개요

AWS SNS(Simple Notification Service)는 완전 관리형 메시지 알림 서비스입니다. 애플리케이션과 서비스 간의 메시지 전달을 위한 중앙 집중식 허브 역할을 합니다. 퍼블리셔-구독자(Pub-Sub) 패턴을 기반으로 하여, 한 곳에서 메시지를 발행하면 여러 구독자에게 동시에 전달할 수 있는 구조를 제공합니다.

### 핵심 아키텍처 개념

**퍼블리셔(Publisher)**

메시지를 생성하고 SNS 주제에 발행하는 주체입니다. 애플리케이션, 서비스, 또는 AWS 서비스가 될 수 있습니다. 메시지 발행 시 구독자에 대한 직접적인 지식이 필요하지 않습니다. 주제에 메시지를 보내기만 하면 SNS가 모든 구독자에게 전달합니다.

실무에서 주문 서비스가 주문 완료 이벤트를 발행하면, 재고 서비스, 결제 서비스, 배송 서비스가 각각 구독하고 있습니다. 주문 서비스는 어떤 서비스가 구독하고 있는지 알 필요가 없습니다. 새로운 서비스가 추가되어도 주문 서비스 코드를 수정할 필요가 없습니다.

**주제(Topic)**

메시지의 중앙 집중식 엔드포인트입니다. 고유한 ARN(Amazon Resource Name)을 가집니다. 메시지를 받아서 모든 구독자에게 전달하는 역할을 합니다.

주제는 논리적으로 관련된 메시지들을 그룹화합니다. 예를 들어 "주문-이벤트" 주제에는 주문 생성, 주문 취소, 주문 완료 같은 이벤트들이 발행됩니다. 각 이벤트 타입은 메시지 속성으로 구분할 수 있습니다.

**구독자(Subscriber)**

주제를 구독하여 메시지를 수신하는 엔드포인트입니다. 이메일, SMS, 모바일 푸시, HTTP(S), Lambda, SQS 등 다양한 형태를 지원합니다.

같은 주제에 여러 구독자가 등록되어 있으면, 메시지가 발행될 때 모든 구독자에게 동시에 전달됩니다. 구독자 중 하나가 실패해도 다른 구독자에게는 영향을 주지 않습니다.

## 핵심 특징과 장점

### 푸시 기반 메시징 시스템

SNS는 푸시 기반 메시징을 채택합니다. 구독자가 메시지를 폴링할 필요 없이 실시간으로 알림을 받을 수 있습니다. 시스템 리소스를 절약하고 응답성을 향상시킵니다.

폴링 방식과 비교하면, 폴링은 클라이언트가 주기적으로 서버에 요청을 보내서 메시지가 있는지 확인합니다. 메시지가 없어도 계속 요청을 보내야 하므로 불필요한 네트워크 트래픽과 서버 부하가 발생합니다. 푸시 방식은 메시지가 발생했을 때만 전송하므로 효율적입니다.

SNS는 메시지가 발행되면 즉시 모든 구독자에게 전달을 시도합니다. 구독자가 HTTP 엔드포인트인 경우 POST 요청으로 메시지를 전달합니다. Lambda 함수인 경우 Lambda를 트리거합니다. SQS 큐인 경우 큐에 메시지를 추가합니다.

### 높은 확장성과 가용성

SNS는 자동으로 확장되어 트래픽 증가에 대응합니다. 초당 수백만 건의 메시지를 처리할 수 있습니다. AWS의 글로벌 인프라를 활용하여 고가용성을 보장합니다. 99.9% 이상의 가용성 SLA를 제공합니다.

트래픽이 갑자기 증가해도 별도의 설정 없이 자동으로 처리량이 증가합니다. 예를 들어 블랙프라이데이 같은 이벤트로 주문이 급증해도 SNS는 자동으로 확장되어 모든 메시지를 처리합니다.

가용성 측면에서 SNS는 여러 가용 영역에 걸쳐 운영됩니다. 한 가용 영역에 장애가 발생해도 다른 가용 영역에서 서비스를 계속 제공합니다.

### 다양한 전송 프로토콜 지원

**이메일**

SMTP를 통한 이메일 알림을 전송합니다. 단순 텍스트 이메일과 HTML 이메일을 모두 지원합니다. 구독자로 이메일 주소를 등록하면, 메시지가 발행될 때 해당 주소로 이메일이 전송됩니다.

이메일 구독은 확인 과정이 필요합니다. SNS가 이메일 주소로 확인 링크를 보내고, 사용자가 링크를 클릭해야 구독이 활성화됩니다. 이는 스팸 방지를 위한 조치입니다.

**SMS**

전 세계 모바일 기기로 문자 메시지를 전송합니다. 국가별로 전송 비용이 다르고, 일일 전송 한도가 있습니다. SMS는 비용이 높기 때문에 중요한 알림에만 사용하는 것이 좋습니다.

SMS 구독도 확인 과정이 필요합니다. 사용자가 받은 확인 코드를 입력해야 구독이 활성화됩니다.

**모바일 푸시**

APNS(Apple Push Notification Service), FCM(Google Firebase Cloud Messaging), ADM(Amazon Device Messaging)을 지원합니다. iOS, Android, Kindle 기기로 푸시 알림을 전송할 수 있습니다.

모바일 푸시를 사용하려면 각 플랫폼의 인증서나 키를 SNS에 등록해야 합니다. APNS의 경우 인증서나 키를 업로드하고, FCM의 경우 서버 키를 등록합니다.

**HTTP/HTTPS**

RESTful API 엔드포인트로 메시지를 전달합니다. 구독자로 HTTP/HTTPS URL을 등록하면, 메시지가 발행될 때 해당 URL로 POST 요청을 보냅니다.

HTTP 엔드포인트는 200 OK 응답을 반환해야 구독이 유지됩니다. 계속 실패하면 구독이 비활성화됩니다. 엔드포인트가 일시적으로 사용 불가능한 경우를 대비해 재시도 메커니즘이 있습니다.

**AWS 서비스**

Lambda, SQS, Kinesis Data Firehose 등과 연동할 수 있습니다. Lambda 함수를 구독자로 등록하면 메시지가 발행될 때 Lambda가 트리거됩니다. SQS 큐를 구독자로 등록하면 메시지가 큐에 추가됩니다.

### 메시지 필터링 기능

구독자별로 메시지 필터 정책을 설정하여 필요한 메시지만 수신할 수 있습니다. 불필요한 트래픽을 줄이고 비용을 절약하는 데 도움이 됩니다.

예를 들어 "주문-이벤트" 주제에 여러 구독자가 있습니다. 재고 서비스는 주문 생성 이벤트만 필요하고, 배송 서비스는 주문 완료 이벤트만 필요합니다. 각 구독자에 필터 정책을 설정하면 필요한 메시지만 받을 수 있습니다.

필터 정책은 메시지 속성을 기반으로 동작합니다. 메시지에 포함된 속성 값에 따라 필터링합니다. 여러 조건을 AND 또는 OR로 조합할 수 있습니다.

## 메시지 전달 보장과 재시도 정책

### At-Least-Once 전달 보장

SNS는 메시지가 최소 한 번은 전달되도록 보장합니다. 네트워크 오류나 일시적인 장애로 인해 메시지 전달이 실패할 경우, 자동으로 재시도합니다.

At-Least-Once는 메시지가 한 번 이상 전달될 수 있다는 의미입니다. 네트워크 오류로 인해 재시도하는 과정에서 동일한 메시지가 여러 번 전달될 수 있습니다. 구독자는 중복 메시지를 처리할 수 있어야 합니다.

메시지에 고유 ID를 포함하거나, 구독자 측에서 중복 제거 로직을 구현해야 합니다. SQS를 구독자로 사용하는 경우 SQS의 중복 제거 기능을 활용할 수 있습니다.

### 재시도 정책

**표준 주제**

지수 백오프를 사용한 재시도를 수행합니다. 첫 번째 재시도는 짧은 간격으로 시도하고, 실패할수록 재시도 간격이 길어집니다. 최대 재시도 횟수에 도달하면 메시지 전달을 포기합니다.

재시도 간격은 구독자 타입에 따라 다릅니다. HTTP 엔드포인트의 경우 즉시 재시도하고, 실패하면 점진적으로 간격을 늘립니다. Lambda의 경우 Lambda의 재시도 정책을 따릅니다.

**FIFO 주제**

순서가 보장되는 메시지 전달을 제공합니다. 메시지 그룹 ID를 기준으로 순서를 보장합니다. 같은 메시지 그룹 ID를 가진 메시지들은 순서대로 전달됩니다.

FIFO 주제는 표준 주제보다 처리량이 낮습니다. 초당 3,000건의 메시지를 처리할 수 있습니다. 순서가 중요한 경우에만 사용하는 것이 좋습니다.

**Dead Letter Queue**

최대 재시도 횟수를 초과해도 전달되지 않은 메시지를 처리하기 위해 Dead Letter Queue(DLQ)를 설정할 수 있습니다. SQS 큐를 DLQ로 지정하면, 실패한 메시지가 해당 큐로 전달됩니다.

DLQ에 쌓인 메시지를 모니터링하고, 주기적으로 확인해서 문제를 해결해야 합니다. DLQ에 메시지가 계속 쌓이면 구독자 측에 문제가 있다는 신호입니다.

## 보안과 접근 제어

### IAM 기반 권한 관리

IAM을 통해 세밀한 권한 제어가 가능합니다. 주제별, 구독자별 접근 권한을 설정할 수 있습니다. 교차 계정 액세스도 지원합니다.

주제에 대한 Publish 권한을 특정 IAM 역할이나 사용자에게만 부여할 수 있습니다. 이렇게 하면 인증된 사용자만 메시지를 발행할 수 있습니다.

구독자에 대한 Subscribe 권한도 제어할 수 있습니다. 특정 주제에 구독을 추가하거나 제거하는 권한을 제한할 수 있습니다.

### 암호화

**전송 중 암호화**

TLS/SSL을 통해 전송 중 암호화를 제공합니다. HTTP 엔드포인트로 메시지를 전달할 때 HTTPS를 사용하면 메시지가 암호화되어 전송됩니다.

**저장 시 암호화**

KMS 키를 사용하여 저장 시 암호화를 설정할 수 있습니다. 주제를 생성할 때 KMS 키를 지정하면, 메시지가 암호화되어 저장됩니다.

**엔드포인트 인증**

HTTP 엔드포인트의 경우 메시지 서명을 통해 메시지 무결성을 보장합니다. SNS가 전송하는 모든 메시지에는 서명이 포함되어 있습니다. 구독자는 이 서명을 검증하여 메시지가 SNS에서 온 것인지 확인할 수 있습니다.

서명 검증을 하지 않으면 악의적인 사용자가 가짜 메시지를 보낼 수 있습니다. 반드시 서명을 검증해야 합니다.

## 비용 구조와 최적화

### 사용량 기반 과금

메시지 발행 건수에 따라 과금됩니다. 전송 프로토콜별로 요금이 다릅니다. 데이터 전송 비용도 별도로 적용됩니다.

표준 주제의 경우 메시지 발행은 무료입니다. 구독자에게 메시지를 전달할 때만 비용이 발생합니다. HTTP 엔드포인트로 전달하는 경우 전달 건수당 요금이 부과됩니다. Lambda로 전달하는 경우 Lambda 실행 비용만 발생하고 SNS 전달 비용은 없습니다.

SMS는 비용이 높습니다. 국가별로 요금이 다르고, 일일 전송 한도가 있습니다. 대량으로 SMS를 보내면 예상치 못한 비용이 발생할 수 있습니다.

### 비용 최적화

**메시지 필터링**

필터링을 통해 불필요한 전송을 방지하면 비용을 절약할 수 있습니다. 구독자가 필요 없는 메시지를 받지 않도록 필터 정책을 설정합니다.

**메시지 크기**

메시지 크기를 최소화하면 전송 비용을 줄일 수 있습니다. 큰 데이터는 S3에 저장하고 참조 링크만 메시지에 포함합니다.

**리전 선택**

리전별로 가격이 다를 수 있습니다. 비용을 고려하여 적절한 리전을 선택합니다.

## 실제 사용 사례

### 이커머스 주문 알림 시스템

온라인 쇼핑몰에서 주문이 발생하면 SNS를 통해 다양한 알림을 동시에 전송할 수 있습니다. 주문 확인 이메일, 배송 상태 SMS, 모바일 앱 푸시 알림을 하나의 주제에서 관리합니다.

주문 서비스는 주문 완료 이벤트를 SNS 주제에 발행합니다. 이메일 서비스는 주문 확인 이메일을 보내고, SMS 서비스는 배송 시작 알림을 보내고, 모바일 앱은 푸시 알림을 보냅니다. 각 서비스는 독립적으로 동작하며, 하나가 실패해도 다른 서비스에는 영향을 주지 않습니다.

### 모니터링 및 알림 시스템

시스템 장애나 임계값 초과 시 개발팀, 운영팀, 관리자에게 즉시 알림을 전송합니다. CloudWatch와 연동하여 자동화된 모니터링 시스템을 구축할 수 있습니다.

CloudWatch 알람이 임계값을 초과하면 SNS 주제에 알림을 발행합니다. 개발팀은 Slack으로 알림을 받고, 운영팀은 이메일로 받고, 관리자는 SMS로 받습니다. 심각도에 따라 다른 알림 채널을 사용할 수 있습니다.

### 마이크로서비스 간 이벤트 전파

마이크로서비스 아키텍처에서 서비스 간 느슨한 결합을 위해 SNS를 활용합니다. 주문 서비스에서 주문 완료 이벤트를 발행하면, 재고 관리, 결제, 배송 서비스가 각각 필요한 작업을 수행할 수 있습니다.

주문 서비스는 다른 서비스에 대해 알 필요가 없습니다. 새로운 서비스가 추가되어도 주문 서비스 코드를 수정할 필요가 없습니다. 이벤트를 구독하는 서비스만 추가하면 됩니다.

### 사용자 참여도 향상

앱 사용자에게 개인화된 푸시 알림을 전송하여 사용자 참여도를 높입니다. 사용자 행동 패턴에 따라 맞춤형 메시지를 전송할 수 있습니다.

사용자가 특정 상품을 조회하면 해당 상품의 할인 알림을 보냅니다. 장바구니에 담긴 상품이 있으면 구매 유도 메시지를 보냅니다. 메시지 필터링을 통해 사용자별로 다른 메시지를 전송할 수 있습니다.

## SNS와 다른 AWS 서비스 연동

### SNS + SQS 패턴

SNS에서 SQS로 메시지를 전달하는 패턴은 매우 일반적입니다. 메시지 버퍼링, 배치 처리, 재시도 로직 구현에 유용합니다.

SNS가 푸시 기반이라면 SQS는 풀 기반입니다. SNS가 메시지를 SQS 큐에 추가하면, 애플리케이션이 큐에서 메시지를 가져옵니다. 애플리케이션이 일시적으로 사용 불가능해도 메시지가 큐에 쌓여 있어 나중에 처리할 수 있습니다.

SQS를 사용하면 메시지 처리 속도를 제어할 수 있습니다. 트래픽이 급증해도 큐에 메시지가 쌓여 있어 순차적으로 처리할 수 있습니다.

### SNS + Lambda 패턴

Lambda 함수를 SNS 구독자로 설정하여 서버리스 이벤트 처리 시스템을 구축할 수 있습니다. 메시지 수신 시 Lambda 함수가 자동으로 실행되어 비즈니스 로직을 처리합니다.

Lambda는 자동으로 확장되어 트래픽에 대응합니다. 메시지가 많으면 Lambda 인스턴스가 자동으로 증가합니다. 비용은 실행 시간과 메모리 사용량에 따라 결정됩니다.

Lambda 함수가 실패하면 SNS가 재시도합니다. 최대 재시도 횟수를 초과하면 DLQ로 메시지를 보낼 수 있습니다.

### SNS + CloudWatch 패턴

CloudWatch 알람이 임계값을 초과하면 SNS를 통해 알림을 전송합니다. 시스템 모니터링과 알림을 자동화할 수 있습니다.

CloudWatch 메트릭이 임계값을 초과하면 알람이 트리거됩니다. 알람은 SNS 주제에 메시지를 발행하고, 구독자들이 알림을 받습니다.

여러 알람을 하나의 SNS 주제로 연결하면 중앙에서 알림을 관리할 수 있습니다. 알림 채널을 변경하거나 추가할 때 SNS 구독자만 수정하면 됩니다.

## 운영 시 주의사항

### 메시지 설계 원칙

**메시지 크기 최소화**

메시지 크기를 최소화하여 전송 비용을 절약합니다. 큰 데이터는 S3에 저장하고 참조 링크만 메시지에 포함합니다. SNS는 단일 메시지 최대 크기가 256KB입니다.

**구조화된 메시지 형식**

JSON 형식을 사용하여 파싱 효율성을 향상시킵니다. 메시지 구조를 일관되게 유지하면 구독자가 쉽게 파싱할 수 있습니다.

**메시지 버전 관리**

메시지에 버전 정보를 포함하여 하위 호환성을 유지합니다. 메시지 구조가 변경되어도 구독자가 버전에 따라 다르게 처리할 수 있습니다.

### 모니터링과 로깅

**CloudWatch 메트릭**

메시지 전송 성공률을 모니터링합니다. 실패한 메시지 수, 전송 지연 시간 등을 추적합니다.

**Dead Letter Queue**

실패한 메시지에 대한 DLQ를 설정합니다. DLQ에 메시지가 쌓이면 구독자 측에 문제가 있다는 신호입니다. 주기적으로 확인해서 문제를 해결해야 합니다.

**구독자별 전송 지연 시간**

구독자별로 전송 지연 시간을 추적합니다. 특정 구독자로의 전송이 느리면 해당 구독자의 문제일 수 있습니다.

### 보안 고려사항

**민감한 정보 처리**

민감한 정보는 메시지에 포함하지 않고 참조 ID만 전송합니다. 예를 들어 사용자 비밀번호를 메시지에 포함하지 않고, 사용자 ID만 포함합니다.

**HTTPS 엔드포인트 사용**

HTTP 엔드포인트로 메시지를 전달할 때는 HTTPS를 사용하여 전송 중 암호화를 보장합니다.

**주제 정책**

주제 정책을 통해 접근 권한을 세밀하게 제어합니다. 특정 IAM 역할이나 사용자만 메시지를 발행할 수 있도록 제한합니다.

**서명 검증**

HTTP 엔드포인트의 경우 메시지 서명을 반드시 검증해야 합니다. 서명을 검증하지 않으면 악의적인 사용자가 가짜 메시지를 보낼 수 있습니다.

### 성능 최적화

**메시지 필터링**

필터링을 통해 불필요한 전송을 방지합니다. 구독자가 필요 없는 메시지를 받지 않도록 필터 정책을 설정합니다.

**SQS와 연동**

배치 처리 가능한 경우 SQS와 연동하여 효율성을 향상시킵니다. SQS에서 메시지를 배치로 가져와서 처리하면 비용을 절약할 수 있습니다.

**리전 선택**

리전별 가격 차이를 고려하여 최적 리전을 선택합니다. 지연 시간도 고려해야 합니다.

## 주의사항과 제한사항

### 메시지 크기 제한

단일 메시지 최대 크기는 256KB입니다. 큰 데이터는 S3에 저장하고 참조 링크만 전송하는 것이 좋습니다.

메시지에 큰 데이터를 포함하면 전송 비용이 증가하고, 전송 시간이 길어집니다. 구독자가 처리하는 데 시간이 오래 걸릴 수 있습니다.

### 전송 프로토콜별 제한

**SMS**

일일 전송 한도가 있고, 국가별 제한이 있습니다. 비용이 높기 때문에 중요한 알림에만 사용하는 것이 좋습니다.

SMS를 대량으로 보내면 예상치 못한 비용이 발생할 수 있습니다. 비용 알람을 설정하여 예산을 초과하지 않도록 주의해야 합니다.

**이메일**

스팸 필터링으로 인한 전송 실패 가능성이 있습니다. 이메일 서비스 제공자가 스팸으로 판단하면 전송이 실패할 수 있습니다.

발신자 이메일 주소를 신뢰할 수 있는 도메인으로 설정하고, SPF, DKIM 레코드를 설정하여 스팸으로 분류되지 않도록 해야 합니다.

**모바일 푸시**

플랫폼별 인증서 관리가 필요합니다. APNS의 경우 인증서나 키를 업로드하고, FCM의 경우 서버 키를 등록해야 합니다.

인증서가 만료되면 푸시 알림이 전송되지 않습니다. 인증서 만료일을 모니터링하고 갱신해야 합니다.

### 비용 관리

메시지 발행 건수에 따라 과금되므로 예상치 못한 비용이 발생할 수 있습니다. 사용량을 모니터링하고 비용 알람을 설정하여 예산을 초과하지 않도록 주의해야 합니다.

필터링을 통해 불필요한 전송을 방지하면 비용을 절약할 수 있습니다. 구독자가 필요 없는 메시지를 받지 않도록 필터 정책을 설정합니다.

### 웹 클라이언트 직접 연결 불가

SNS는 웹 브라우저에서 직접 구독할 수 없습니다. HTTP/HTTPS 엔드포인트로만 구독할 수 있고, 이는 서버 측 엔드포인트여야 합니다.

웹 브라우저에서 실시간으로 메시지를 받아야 하는 경우, WebSocket이나 Socket.IO 같은 실시간 통신 프로토콜을 사용해야 합니다. SNS → SQS → Lambda → WebSocket 서버 경로로 메시지를 전달하거나, API Gateway WebSocket API를 사용할 수 있습니다.
