# AWS Elastic Beanstalk

## 개요

AWS Elastic Beanstalk는 애플리케이션을 빠르게 배포하고 관리할 수 있는 PaaS(Platform as a Service)다. 코드만 업로드하면 AWS가 인프라를 자동으로 프로비저닝하고 관리한다. EC2, 로드 밸런서, Auto Scaling, 모니터링을 개발자가 직접 설정할 필요가 없다.

### 왜 Elastic Beanstalk가 필요할까

처음 AWS에서 웹 애플리케이션을 배포하려고 할 때 많은 것을 배워야 한다.

**전통적인 배포의 복잡함:**

**1일차: EC2 인스턴스 설정**
- AWS 콘솔에 로그인한다
- EC2 서비스로 이동한다
- 인스턴스 타입을 선택한다. t3.medium? t3.large?
- AMI를 선택한다. Amazon Linux? Ubuntu?
- 보안 그룹을 만든다. 어떤 포트를 열어야 할까?
- 키 페어를 생성한다
- 인스턴스를 시작한다

**2일차: 애플리케이션 설치**
- SSH로 인스턴스에 접속한다
- Node.js를 설치한다
- Nginx를 설치한다
- 애플리케이션 코드를 업로드한다
- 환경 변수를 설정한다
- 애플리케이션을 시작한다

**3일차: 로드 밸런서 설정**
- ALB를 생성한다
- 타겟 그룹을 만든다
- Health Check를 설정한다
- 인스턴스를 타겟 그룹에 등록한다
- 도메인을 ALB에 연결한다

**4일차: Auto Scaling 설정**
- Launch Template을 만든다
- Auto Scaling Group을 만든다
- Scaling Policy를 설정한다
- 테스트한다

**5일차: 모니터링 설정**
- CloudWatch 대시보드를 만든다
- 알람을 설정한다
- 로그를 CloudWatch Logs로 전송한다

5일이 지났다. 아직 애플리케이션 개발은 시작하지도 못했다. 인프라 설정에만 일주일이 걸렸다.

**Elastic Beanstalk의 해결:**

**1시간 만에 배포:**
1. Elastic Beanstalk 콘솔에 간다
2. "Create Application" 클릭
3. 플랫폼 선택: Node.js
4. 코드 업로드: my-app.zip
5. "Create" 클릭

10분 후, 애플리케이션이 실행 중이다. URL이 생성되었다. `http://my-app.us-west-2.elasticbeanstalk.com`

**AWS가 자동으로 한 것:**
- EC2 인스턴스를 생성했다
- Node.js를 설치했다
- Nginx를 설정했다
- 로드 밸런서를 생성했다
- Auto Scaling을 구성했다
- CloudWatch 모니터링을 설정했다
- Health Check를 구성했다

모든 것이 자동이다. 개발자는 코드에만 집중할 수 있다.

### PaaS의 의미

PaaS는 Platform as a Service의 약자다. 플랫폼을 서비스로 제공한다는 의미다.

**클라우드 서비스 계층:**

**IaaS (Infrastructure as a Service):**
예: EC2, VPC, S3
- AWS가 제공: 서버, 네트워크, 스토리지
- 개발자가 관리: OS, 런타임, 애플리케이션, 데이터

**PaaS (Platform as a Service):**
예: Elastic Beanstalk, Heroku
- AWS가 제공: 서버, 네트워크, OS, 런타임
- 개발자가 관리: 애플리케이션, 데이터

**SaaS (Software as a Service):**
예: Gmail, Salesforce
- AWS가 제공: 모든 것
- 개발자가 관리: 사용만 함

Elastic Beanstalk는 PaaS다. 인프라와 플랫폼을 AWS가 관리한다. 개발자는 애플리케이션만 관리한다.

### 누구를 위한 서비스인가

**스타트업:**
빠르게 제품을 출시하고 싶다. 인프라에 시간을 쓰고 싶지 않다. Elastic Beanstalk가 완벽하다. 몇 시간 안에 배포할 수 있다.

**소규모 팀:**
DevOps 엔지니어가 없다. 개발자 2-3명이 전부다. Elastic Beanstalk를 사용하면 인프라 전문가 없이도 운영할 수 있다.

**프로토타입:**
아이디어를 빠르게 검증하고 싶다. Elastic Beanstalk로 빠르게 배포하고, 고객 피드백을 받고, 빠르게 수정한다.

**전통적인 웹 애플리케이션:**
모놀리식 애플리케이션이나 간단한 마이크로서비스. 복잡한 아키텍처가 필요 없다.

## 다른 서비스와 비교

선택하기 전에 차이점을 이해해야 한다.

### Elastic Beanstalk vs EC2

**EC2를 직접 사용:**

**장점:**
- 완전한 제어. 모든 것을 원하는 대로 설정할 수 있다
- 유연성. 특수한 요구사항도 충족할 수 있다
- 커스터마이징. 모든 소프트웨어를 설치할 수 있다

**단점:**
- 복잡하다. 많은 것을 배워야 한다
- 시간이 오래 걸린다. 설정에 며칠이 걸린다
- 유지보수 부담. 패치, 업데이트, 모니터링을 직접 해야 한다

**Elastic Beanstalk:**

**장점:**
- 간단하다. 몇 번의 클릭으로 배포한다
- 빠르다. 10분이면 실행된다
- 자동 관리. 패치, 업데이트, 모니터링을 AWS가 한다

**단점:**
- 제한적이다. 모든 것을 원하는 대로 할 수 없다
- AWS 종속. 다른 클라우드로 이전이 어렵다

**언제 EC2를 선택할까:**
- 특수한 하드웨어가 필요한 경우 (GPU, 고성능 네트워크)
- 매우 세밀한 제어가 필요한 경우
- 기존 시스템을 그대로 마이그레이션하는 경우

**언제 Elastic Beanstalk를 선택할까:**
- 빠르게 시작하고 싶은 경우
- 인프라 관리를 최소화하고 싶은 경우
- 표준 웹 애플리케이션인 경우

### Elastic Beanstalk vs ECS/EKS

**ECS/EKS:**

**장점:**
- 컨테이너 기반. Docker 이미지를 사용한다
- 마이크로서비스에 적합. 여러 서비스를 독립적으로 관리한다
- 유연성. 복잡한 아키텍처를 구성할 수 있다

**단점:**
- 복잡하다. 컨테이너 오케스트레이션을 배워야 한다
- 설정이 많다. Task Definition, Service, Cluster 등
- 시간이 걸린다. 익숙해지는 데 시간이 필요하다

**Elastic Beanstalk:**

**장점:**
- 간단하다. 컨테이너를 몰라도 된다
- 빠르다. 바로 시작할 수 있다
- 전통적 방식. 익숙한 배포 방식이다

**단점:**
- 모놀리식에 적합. 마이크로서비스는 관리가 어렵다
- 제한적. 복잡한 아키텍처는 구현하기 어렵다

**언제 ECS/EKS를 선택할까:**
- 마이크로서비스 아키텍처인 경우
- 컨테이너를 이미 사용하는 경우
- 복잡한 배포 전략이 필요한 경우 (Canary, Blue-Green)

**언제 Elastic Beanstalk를 선택할까:**
- 모놀리식 애플리케이션인 경우
- 간단한 웹사이트나 API인 경우
- 빠르게 시작하고 싶은 경우

### Elastic Beanstalk vs Lambda

**Lambda:**

**장점:**
- 서버리스. 서버 관리가 전혀 없다
- 저렴. 실행 시간만 과금된다
- 자동 스케일링. 무제한으로 확장된다

**단점:**
- 15분 제한. 장시간 실행이 안 된다
- 상태 비저장. 메모리에 데이터를 유지할 수 없다
- 콜드 스타트. 첫 요청이 느릴 수 있다

**Elastic Beanstalk:**

**장점:**
- 시간 제한 없음. 24시간 실행 가능하다
- 상태 유지 가능. 메모리, 파일 시스템을 사용할 수 있다
- 전통적 애플리케이션. 기존 코드를 그대로 사용한다

**단점:**
- 서버 관리 필요. 최소한의 관리는 필요하다
- 비용. 인스턴스가 계속 실행되면 비용이 나간다

**언제 Lambda를 선택할까:**
- 이벤트 기반 처리인 경우
- 간헐적으로 실행되는 경우
- 단순한 함수인 경우

**언제 Elastic Beanstalk를 선택할까:**
- 전통적인 웹 애플리케이션인 경우
- 24시간 실행되는 서비스인 경우
- 기존 프레임워크를 사용하는 경우 (Express, Django, Spring)

## 핵심 개념

Elastic Beanstalk의 구조를 이해하면 사용하기 쉬워진다.

### Application

Application은 최상위 컨테이너다. 프로젝트 하나가 Application 하나다.

**예시:**
온라인 쇼핑몰을 만든다고 하자.
- Application 이름: my-shopping-mall
- 이 Application 안에 여러 환경이 있다

**Application의 역할:**
- 환경들을 그룹화한다
- 버전들을 관리한다
- 전체 설정을 관리한다

**실제 사용:**
Application은 한 번 만들고 계속 사용한다. 거의 건드리지 않는다. 실제 작업은 Environment에서 한다.

### Environment

Environment는 애플리케이션이 실행되는 환경이다. 실제 리소스가 여기에 생성된다.

**환경 분리:**

**개발 환경 (Development):**
- 개발자들이 사용한다
- 테스트와 실험을 한다
- 인스턴스 1대면 충분하다
- 비용을 최소화한다

**스테이징 환경 (Staging):**
- 프로덕션과 동일한 구성이다
- 최종 테스트를 한다
- 고객에게 공개하기 전 검증한다

**프로덕션 환경 (Production):**
- 실제 고객이 사용한다
- 고가용성이 필수다
- 여러 인스턴스를 실행한다
- 로드 밸런서를 사용한다

**환경 타입:**

**Web Server Environment:**
HTTP 요청을 처리한다. 일반적인 웹 애플리케이션이다.

**구성:**
- 로드 밸런서 (ALB)
- Auto Scaling Group
- EC2 인스턴스 여러 대
- CloudWatch 모니터링

**사용 사례:**
- 웹사이트
- REST API
- GraphQL API
- 모바일 앱 백엔드

**Worker Environment:**
백그라운드 작업을 처리한다. SQS 큐에서 메시지를 가져와 처리한다.

**구성:**
- SQS 큐
- EC2 인스턴스
- 데몬 프로세스 (메시지를 계속 폴링)

**사용 사례:**
- 이미지 처리
- 이메일 발송
- 데이터 분석
- 정기적인 배치 작업

**실무 팁:**
대부분은 Web Server Environment를 사용한다. 무거운 작업이 있으면 Worker Environment를 추가로 만든다. 예를 들어, 사용자 요청은 Web Server가 받고, 이미지 리사이징은 Worker가 처리한다.

### Application Version

Application Version은 배포 가능한 코드의 버전이다.

**버전 생명주기:**

**1. 코드 변경:**
개발자가 코드를 수정한다. 버그를 고치거나 새 기능을 추가한다.

**2. 패키징:**
코드를 ZIP 파일로 묶는다.

```bash
# Node.js 예시
zip -r my-app-v1.2.3.zip . -x "*.git*" "node_modules/*"
```

**3. 업로드:**
Elastic Beanstalk에 업로드한다. 자동으로 S3에 저장된다.

```bash
eb deploy
```

또는 콘솔에서 업로드한다.

**4. 버전 생성:**
Elastic Beanstalk가 새 Application Version을 만든다.
- Label: v1.2.3
- S3 경로: s3://elasticbeanstalk-us-west-2-123456789012/my-app/my-app-v1.2.3.zip
- 생성 시간: 2026-01-18 10:30:00

**5. 배포:**
Environment에 이 버전을 배포한다.

**버전 관리:**

**모든 버전 유지:**
Elastic Beanstalk는 모든 버전을 S3에 저장한다. 언제든 이전 버전으로 돌아갈 수 있다.

**예시:**
- v1.2.1: 1주일 전 배포. 안정적이었다
- v1.2.2: 3일 전 배포. 버그가 있었다
- v1.2.3: 오늘 배포. 버그 수정

v1.2.3에 문제가 생기면 즉시 v1.2.1로 롤백한다. 클릭 한 번이면 된다.

**수명 주기 정책:**
오래된 버전은 자동으로 삭제한다. S3 비용을 절감한다.

**설정 예시:**
- 최대 200개 버전 유지
- 180일 이상된 버전 삭제
- 현재 사용 중인 버전은 삭제 안 함

### Platform

Platform은 애플리케이션이 실행되는 운영체제와 런타임 환경이다.

**지원 플랫폼:**

**Node.js:**
- 여러 버전 지원: 14, 16, 18, 20
- Amazon Linux 2
- Nginx 프록시
- npm, yarn 지원

**Python:**
- 여러 버전 지원: 3.8, 3.9, 3.10, 3.11
- Amazon Linux 2
- Apache/Nginx 프록시
- pip, virtualenv 지원

**Java:**
- Java 8, 11, 17
- Tomcat
- Maven/Gradle 지원

**기타:**
- .NET Core
- PHP
- Ruby
- Go
- Docker (커스텀 플랫폼)

**플랫폼 선택:**
애플리케이션의 언어와 프레임워크에 맞는 플랫폼을 선택한다.

**예시:**
- Express.js 앱 → Node.js 플랫폼
- Django 앱 → Python 플랫폼
- Spring Boot 앱 → Java 플랫폼

**플랫폼 업데이트:**
AWS가 주기적으로 플랫폼을 업데이트한다. 보안 패치, 버그 수정이 포함된다.

**Managed Updates:**
자동 업데이트를 활성화하면 AWS가 자동으로 플랫폼을 업데이트한다. 다운타임 없이 진행된다.

**설정:**
- 업데이트 시간: 매주 일요일 오전 2시
- 다운타임: 없음 (롤링 업데이트)
- 실패 시: 자동 롤백

## 배포 방식 상세

배포 방식에 따라 다운타임, 속도, 안정성이 달라진다.

### All at Once

가장 간단하고 빠른 방식이다. 모든 인스턴스를 동시에 업데이트한다.

**동작 과정:**

**상황:**
인스턴스 3대가 v1.0을 실행 중이다. v1.1로 업데이트하려고 한다.

**1. 배포 시작:**
Elastic Beanstalk가 모든 인스턴스에 새 버전을 배포한다.

**2. 서비스 중단:**
3대 모두 애플리케이션을 중지한다. 사용자는 접속할 수 없다.

**3. 업데이트:**
새 버전(v1.1)을 설치한다. 의존성을 설치한다. 애플리케이션을 시작한다.

**4. 서비스 재개:**
3대 모두 준비되면 다시 트래픽을 받는다.

**소요 시간:**
- 애플리케이션 중지: 10초
- 새 버전 설치: 60초
- 애플리케이션 시작: 30초
- 총: 약 100초 다운타임

**장점:**
- 가장 빠르다. 단순하다
- 추가 리소스가 필요 없다
- 추가 비용이 없다

**단점:**
- 다운타임이 발생한다
- 사용자가 영향을 받는다
- 롤백이 느리다 (다시 All at Once 배포)

**사용 사례:**
- 개발 환경
- 테스트 환경
- 다운타임이 허용되는 경우
- 트래픽이 거의 없는 시간대 (새벽)

**실무 팁:**
프로덕션에서는 사용하지 않는다. 개발 환경에서 빠르게 테스트할 때만 사용한다.

### Rolling

인스턴스를 배치 단위로 순차적으로 업데이트한다. 다운타임이 없다.

**동작 과정:**

**상황:**
인스턴스 4대가 v1.0을 실행 중이다. Batch Size는 2다.

**1단계: 첫 번째 배치 (인스턴스 1, 2):**
1. 인스턴스 1, 2를 로드 밸런서에서 제거한다
2. 트래픽은 인스턴스 3, 4가 처리한다
3. 인스턴스 1, 2에 v1.1을 배포한다
4. Health Check를 통과하면 로드 밸런서에 추가한다

**2단계: 두 번째 배치 (인스턴스 3, 4):**
1. 인스턴스 3, 4를 로드 밸런서에서 제거한다
2. 트래픽은 인스턴스 1, 2가 처리한다 (이미 v1.1)
3. 인스턴스 3, 4에 v1.1을 배포한다
4. Health Check를 통과하면 로드 밸런서에 추가한다

**완료:**
모든 인스턴스가 v1.1을 실행한다.

**Batch Size 선택:**

**작은 배치 (1개 또는 25%):**
- 안전하다. 문제가 생겨도 영향이 적다
- 느리다. 시간이 오래 걸린다

**큰 배치 (50% 또는 100%):**
- 빠르다. 시간이 단축된다
- 위험하다. 문제가 생기면 많은 인스턴스가 영향을 받는다

**장점:**
- 다운타임이 없다
- 추가 비용이 없다
- 간단하다

**단점:**
- 배포 중 용량이 감소한다
- 인스턴스 4대 중 2대가 업데이트되면, 2대만 트래픽을 처리한다
- 트래픽이 많으면 성능 저하가 발생할 수 있다
- 두 버전이 동시에 실행된다 (v1.0과 v1.1)

**사용 사례:**
- 프로덕션 환경
- 비용을 최소화하고 싶은 경우
- 트래픽이 많지 않은 경우

**실무 팁:**
Batch Size를 25%로 설정한다. 안전하면서도 합리적인 속도다.

### Rolling with Additional Batch

Rolling과 유사하지만, 용량을 유지한다.

**동작 과정:**

**상황:**
인스턴스 4대가 v1.0을 실행 중이다. Batch Size는 2다.

**1단계: 추가 인스턴스 생성:**
1. 새 인스턴스 2대를 시작한다 (인스턴스 5, 6)
2. v1.1을 배포한다
3. Health Check를 통과하면 로드 밸런서에 추가한다
4. 이제 6대가 실행 중이다 (4대는 v1.0, 2대는 v1.1)

**2단계: 첫 번째 기존 배치 업데이트:**
1. 인스턴스 1, 2를 로드 밸런서에서 제거한다
2. v1.1을 배포한다
3. 로드 밸런서에 추가한다
4. 이제 4대가 v1.1, 2대가 v1.0

**3단계: 두 번째 기존 배치 업데이트:**
1. 인스턴스 3, 4를 로드 밸런서에서 제거한다
2. v1.1을 배포한다
3. 로드 밸런서에 추가한다
4. 이제 6대 모두 v1.1

**4단계: 추가 인스턴스 제거:**
1. 인스턴스 5, 6을 종료한다
2. 다시 4대로 돌아간다

**장점:**
- 다운타임이 없다
- 용량이 유지된다. 항상 4대 이상이 실행된다
- 성능 영향이 없다

**단점:**
- 추가 비용이 발생한다. 배포 중 6대가 실행된다
- 배포 시간이 길다
- 조금 복잡하다

**사용 사례:**
- 프로덕션 환경
- 트래픽이 많은 경우
- 성능을 유지해야 하는 경우
- 비용보다 안정성이 중요한 경우

**실무 팁:**
트래픽이 많은 프로덕션 환경에서는 이 방식을 사용한다. 비용이 조금 들지만, 사용자 경험이 유지된다.

### Immutable

가장 안전한 방식이다. 완전히 새로운 인스턴스를 만들고 트래픽을 전환한다.

**동작 과정:**

**상황:**
인스턴스 4대가 v1.0을 실행 중이다.

**1단계: 새 Auto Scaling Group 생성:**
1. 완전히 새로운 Auto Scaling Group을 만든다
2. 새 Launch Template을 사용한다
3. 1개의 인스턴스를 시작한다 (테스트용)
4. v1.1을 배포한다

**2단계: Health Check:**
1. 새 인스턴스가 Health Check를 통과하는지 확인한다
2. 실패하면? 즉시 중단하고 롤백한다. 기존 4대는 영향 없다
3. 성공하면? 계속 진행한다

**3단계: 전체 확장:**
1. 새 Auto Scaling Group을 4대로 확장한다
2. 모든 새 인스턴스가 Health Check를 통과한다

**4단계: 트래픽 전환:**
1. 로드 밸런서가 새 인스턴스로 트래픽을 보낸다
2. 기존 인스턴스(v1.0)는 더 이상 트래픽을 받지 않는다

**5단계: 정리:**
1. 기존 Auto Scaling Group을 삭제한다
2. 기존 인스턴스 4대를 종료한다

**완료:**
새 인스턴스 4대가 v1.1을 실행한다.

**장점:**
- 가장 안전하다
- 롤백이 즉시 가능하다. 기존 인스턴스가 아직 실행 중이다
- 새 버전이 완전히 검증된 후 트래픽을 받는다
- 두 버전이 섞이지 않는다

**단점:**
- 비용이 가장 높다. 8대가 동시에 실행된다 (배포 중)
- 배포 시간이 가장 길다
- 리소스 할당량을 고려해야 한다

**사용 사례:**
- 중요한 프로덕션 배포
- 큰 변경사항이 있는 경우
- 롤백 가능성이 높은 경우
- 안정성이 최우선인 경우

**실무 팁:**
중요한 업데이트는 Immutable로 배포한다. 마이너 버전 업그레이드, 큰 기능 추가, 프레임워크 업그레이드 등. 비용이 들지만 안전하다.

### Blue/Green

완전히 새로운 환경을 만들고 DNS를 전환한다.

**동작 과정:**

**상황:**
프로덕션 환경(Blue)이 v1.0을 실행 중이다. URL: `production.elasticbeanstalk.com`

**1단계: Green 환경 생성:**
1. 새 환경을 만든다. 이름: production-green
2. v1.1을 배포한다
3. 동일한 구성을 사용한다 (인스턴스 타입, Auto Scaling 설정 등)
4. URL: `production-green.elasticbeanstalk.com`

**2단계: 테스트:**
1. production-green URL로 접속한다
2. 충분히 테스트한다
3. 부하 테스트를 수행한다
4. 문제가 없는지 확인한다

**3단계: DNS 전환 (Swap):**
1. Elastic Beanstalk 콘솔에서 "Swap Environment URLs" 클릭
2. Blue와 Green의 URL을 교환한다
3. production.elasticbeanstalk.com → Green 환경
4. production-green.elasticbeanstalk.com → Blue 환경

**4단계: 모니터링:**
1. 프로덕션 트래픽이 Green 환경으로 간다
2. 메트릭을 모니터링한다
3. 에러가 발생하는지 확인한다

**5단계: 정리 또는 롤백:**

**문제가 없으면:**
- Blue 환경(v1.0)을 유지하거나 삭제한다
- 보통 며칠 후 삭제한다

**문제가 있으면:**
- 다시 URL을 교환한다 (롤백)
- 몇 초 만에 이전 버전으로 돌아간다

**장점:**
- 즉시 롤백 가능하다. DNS 전환만 하면 된다
- 새 환경을 충분히 테스트할 수 있다
- 프로덕션 환경을 건드리지 않는다
- 다운타임이 거의 없다 (DNS 전파 시간만)

**단점:**
- 비용이 가장 높다. 두 환경을 동시에 실행한다
- DNS 전파 시간이 필요하다 (몇 분)
- 데이터베이스 마이그레이션이 복잡하다
- 설정 관리가 복잡하다

**사용 사례:**
- 메이저 버전 업그레이드
- 대규모 아키텍처 변경
- 충분한 테스트가 필요한 경우
- 즉시 롤백이 필요한 경우

**데이터베이스 문제:**
Blue와 Green이 같은 데이터베이스를 사용한다. 스키마 변경이 있으면 어떻게 할까?

**해결 방법:**
1. 하위 호환성을 유지한다
2. 먼저 데이터베이스를 변경한다 (새 컬럼 추가)
3. Blue(v1.0)는 새 컬럼을 무시한다
4. Green(v1.1)은 새 컬럼을 사용한다
5. 롤백해도 Blue가 정상 작동한다

**실무 팁:**
Blue/Green은 비용이 많이 든다. 정말 중요한 배포에만 사용한다. 일반 배포는 Rolling with Additional Batch나 Immutable로 충분하다.
