# AWS OpenSearch (관리형 검색·로그 분석)

## 개요

OpenSearch는 Elasticsearch를 AWS가 관리형으로 돌려주는 서비스다. 2021년에 Elastic이 라이선스를 바꾸면서 AWS가 Elasticsearch 7.10을 포크해 OpenSearch라는 이름으로 갈라져 나왔다. 그래서 7.10 이전 Elasticsearch를 써봤으면 API도 쿼리 DSL도 그대로 익숙하다. Kibana는 OpenSearch Dashboards로 이름이 바뀌었지만 화면은 거의 같다.

쓰는 상황은 둘로 나뉜다. 하나는 검색이다. 상품 검색, 로그 전문 검색처럼 RDB의 LIKE로는 감당 안 되는 텍스트 매칭을 역색인으로 처리한다. 다른 하나는 로그 분석이다. 애플리케이션 로그, ALB 로그, VPC 플로우 로그를 한곳에 모아 Dashboards로 들여다본다. 실무에서 OpenSearch를 도입하는 절반 이상이 후자, 즉 자체 ELK 스택을 걷어내고 관리형으로 옮기는 경우다.

ELK를 직접 운영해본 사람은 안다. Elasticsearch 노드 죽으면 새벽에 깨서 복구하고, 디스크 차면 인덱스 지우고, 버전 올릴 때마다 클러스터 전체를 조마조마하게 본다. OpenSearch는 이 운영 부담 중 노드 교체·패치·스냅샷 같은 인프라 레벨을 AWS가 가져간다. 대신 샤드 설계, 인덱스 수명주기, 쿼리 튜닝은 여전히 우리 몫이다. 관리형이라고 다 알아서 해주는 게 아니라, 인프라만 떼어간 거라고 보면 정확하다.

여기서 가장 흔한 오해를 짚고 가자. "관리형이니까 알아서 안 죽겠지"라는 생각이 클러스터를 죽인다. 샤드를 잘못 설계하면 AWS가 관리해주든 말든 클러스터는 똑같이 불안정해진다. AWS가 막아주는 건 하드웨어 고장이지 설계 실패가 아니다.

## 도메인 구성

OpenSearch에서 클러스터 하나를 도메인(domain)이라 부른다. 도메인을 만들 때 정하는 것이 노드 구성, 스토리지, 네트워크(VPC 여부), 접근 정책이다. 한 번 만들고 나면 노드 타입이나 개수는 나중에 바꿀 수 있지만, VPC 안에 넣을지 퍼블릭으로 둘지는 생성 시점에 정해야 하니 신중해야 한다.

### 노드 타입

노드는 역할에 따라 세 종류로 나뉜다.

데이터 노드는 실제 인덱스 데이터를 들고 있고 검색·색인 요청을 처리한다. CPU, 메모리, 디스크가 전부 여기서 소모된다. 클러스터 용량을 늘린다는 건 보통 데이터 노드를 늘리거나 키운다는 뜻이다.

마스터 노드(전용 마스터)는 클러스터 상태를 관리한다. 어떤 샤드가 어느 노드에 있는지, 노드가 살아있는지를 추적한다. 데이터를 들고 있지 않아서 검색 부하와 무관하게 클러스터의 두뇌 역할만 한다. 데이터 노드 수가 적은 개발 환경에서는 전용 마스터 없이 데이터 노드가 마스터를 겸하게 둬도 된다. 하지만 운영 환경이라면 전용 마스터를 반드시 둬야 한다. 데이터 노드가 색인 부하로 버벅이는 순간 마스터 역할까지 같이 흔들려서, 클러스터 상태 관리가 멈추고 split brain이 나기 때문이다.

전용 마스터는 홀수로 둔다. 보통 3개다. split brain을 막는 쿼럼이 과반수 투표라, 짝수면 동수가 나올 수 있어서 의미가 없다. 2개를 두는 건 1개보다 나쁘다. AWS 콘솔에서도 전용 마스터는 3개나 5개만 권장한다.

UltraWarm·Cold 노드는 오래된 인덱스를 싼 스토리지로 내리는 계층이다. 데이터 노드는 비싼 인스턴스 스토리지(EBS 또는 NVMe)를 쓰는데, 한 달 지난 로그를 여기 계속 둘 이유가 없다. UltraWarm은 S3를 백킹으로 쓰면서 검색은 되되 색인은 안 되는 읽기 전용 계층이고, Cold는 더 싸지만 검색하려면 다시 살려야(thaw) 한다. 로그 보존 기간이 길면 이 계층 설계가 비용을 크게 가른다.

### 가용 영역 배치

데이터 노드를 2개 이상의 AZ에 나눠 배치하는 옵션(Zone Awareness)을 켜야 한다. AZ 하나가 통째로 죽어도 다른 AZ의 복제 샤드로 버틴다. 3-AZ를 켜면 전용 마스터 3개가 각 AZ에 하나씩 들어가서 AZ 장애에도 쿼럼이 유지된다. 단일 AZ에 다 몰아넣으면 그 AZ 장애 한 번에 클러스터가 통째로 내려간다. 관리형이라도 AZ 장애는 우리가 배치로 막아야 한다.

```
운영 환경 기본 구성 예
- 전용 마스터: 3개 (m6g.large.search 정도), 3-AZ 분산
- 데이터 노드: 6개 (r6g.large.search), 2 또는 3-AZ 분산
- 복제본: replica 1 (AZ 장애 대비)
```

마스터는 메모리보다 안정성이 중요해서 적당한 인스턴스면 되고, 데이터 노드는 검색·색인 부하에 따라 메모리 많은 r 계열을 쓰는 경우가 많다.

## 샤드 설계 — 클러스터를 죽이는 1순위 원인

OpenSearch 운영에서 사고의 대부분은 샤드 설계 실패에서 나온다. 처음 쓰는 사람이 가장 안 와닿는 부분이고, 한 번 잘못 잡으면 클러스터를 새로 만들기 전까지 못 고치는 경우가 많아서 제일 먼저 이해해야 한다.

### 샤드가 뭔지부터

인덱스 하나는 내부적으로 여러 개의 샤드(shard)로 쪼개진다. 샤드는 Lucene 인덱스 하나에 대응하는 실제 데이터 조각이고, 이게 데이터 노드들에 흩어져서 색인과 검색이 병렬로 돌아간다. 프라이머리 샤드(primary)가 원본이고, 레플리카 샤드(replica)가 복제본이다.

핵심은 프라이머리 샤드 수를 인덱스 생성 시점에 정하면 나중에 못 바꾼다는 점이다. 레플리카는 운영 중에 늘렸다 줄였다 할 수 있지만 프라이머리는 못 바꾼다. 바꾸려면 reindex로 데이터를 새 인덱스에 통째로 복사해야 한다. 이 제약 때문에 초기 설계가 그대로 운영 내내 따라온다.

### 샤드를 너무 많이 만드는 실패

가장 흔한 사고가 샤드 과다(oversharding)다. "병렬 처리되니까 샤드 많으면 빠르겠지"라고 생각해서 인덱스마다 샤드를 잔뜩 잡는다. 로그를 날짜별 인덱스로 쌓으면서 인덱스마다 프라이머리 5개씩 잡았다고 하자. 하루 하나씩 1년이면 인덱스 365개에 프라이머리만 1825개, 레플리카까지 3650개다.

샤드 하나하나가 메모리(힙)와 파일 핸들, 클러스터 상태 메타데이터를 잡아먹는다. 샤드가 수만 개가 되면 마스터 노드가 클러스터 상태를 관리하다 힙이 터진다. 증상이 고약하다. 검색은 멀쩡한데 새 인덱스 생성이 느려지거나, 노드 재시작 시 샤드 할당이 몇십 분씩 걸리거나, 클러스터 상태가 yellow에서 안 빠진다. AWS 가이드는 노드당 샤드 수를 힙 1GB당 25개 이하로 권장한다. 데이터 노드 힙이 16GB면 노드당 400개 정도가 상한이라는 뜻이다.

작은 로그 인덱스에 샤드를 여러 개 잡는 것도 낭비다. 하루 로그가 2GB인데 프라이머리 5개로 쪼개면 샤드 하나가 400MB짜리가 된다. 샤드는 보통 10~50GB가 적정이라, 이렇게 작게 쪼개면 오버헤드만 늘고 병렬 이득은 없다. 작은 인덱스는 프라이머리 1개면 충분하다.

### 샤드를 너무 적게 만드는 실패

반대 실패도 있다. 하루에 500GB 들어오는 로그를 프라이머리 1개로 받으면 샤드 하나가 500GB로 부푼다. 거대 샤드는 검색이 느리고, 노드 간 재배치(리밸런싱)가 일어날 때 통째로 옮겨야 해서 네트워크와 디스크를 오래 점유한다. 노드 하나 빠질 때 복구가 한세월 걸린다.

### 적정선 잡기

샤드 크기를 10~50GB 사이에 들어오게 프라이머리 수를 역산한다. 하루 인입량이 300GB고 샤드를 30GB로 맞추고 싶으면 프라이머리 10개다.

```json
PUT /logs-app-2026.05.27
{
  "settings": {
    "index.number_of_shards": 10,
    "index.number_of_replicas": 1
  }
}
```

매일 인입량이 들쭉날쭉하면 이걸 수동으로 매번 계산하는 건 비현실적이다. 그래서 로그는 날짜 기반 인덱스 대신 데이터 스트림 + 롤오버를 쓴다. 인덱스가 일정 크기나 일정 나이에 도달하면 자동으로 새 인덱스로 굴려서(rollover), 샤드 크기를 일정하게 유지한다. 인덱스 템플릿에 프라이머리 수를 박아두고 롤오버 조건을 걸어두면, 트래픽이 늘어도 인덱스 개수만 늘 뿐 샤드 크기는 안정적으로 유지된다.

```json
PUT _index_template/logs-app-template
{
  "index_patterns": ["logs-app-*"],
  "template": {
    "settings": {
      "index.number_of_shards": 3,
      "index.number_of_replicas": 1
    }
  }
}
```

롤오버 조건(예: 50GB 또는 1일)은 뒤에 나오는 ISM 정책으로 건다.

## ISM으로 인덱스 수명주기 관리

로그를 매일 쌓다 보면 디스크가 찬다. 오래된 로그는 검색 빈도가 낮으니 싼 계층으로 내리거나 지워야 한다. 이걸 손으로 하면 빠뜨려서 사고 난다. ISM(Index State Management)이 이 수명주기를 자동화한다. Elasticsearch의 ILM에 대응하는 OpenSearch 기능이다.

ISM은 인덱스를 상태(state)로 보고, 조건이 충족되면 다음 상태로 전이(transition)시킨다. hot → warm → cold → delete 식의 흐름을 정책으로 정의한다.

```json
PUT _plugins/_ism/policies/logs-policy
{
  "policy": {
    "description": "앱 로그 30일 보존 후 삭제",
    "default_state": "hot",
    "states": [
      {
        "name": "hot",
        "actions": [
          { "rollover": { "min_size": "50gb", "min_index_age": "1d" } }
        ],
        "transitions": [
          { "state_name": "warm", "conditions": { "min_index_age": "3d" } }
        ]
      },
      {
        "name": "warm",
        "actions": [
          { "warm_migration": {} },
          { "replica_count": { "number_of_replicas": 0 } }
        ],
        "transitions": [
          { "state_name": "delete", "conditions": { "min_index_age": "30d" } }
        ]
      },
      {
        "name": "delete",
        "actions": [{ "delete": {} }]
      }
    ],
    "ism_template": [
      { "index_patterns": ["logs-app-*"], "priority": 100 }
    ]
  }
}
```

`ism_template`을 정의하면 패턴에 맞는 인덱스가 생길 때 정책이 자동으로 붙는다. 이게 없으면 인덱스마다 수동으로 정책을 붙여야 해서 빠뜨리기 쉽다.

운영에서 ISM 관련해 겪는 문제 몇 가지가 있다.

롤오버는 alias나 데이터 스트림 위에서만 동작한다. 그냥 만든 인덱스에 rollover 액션을 걸면 안 돈다. 인덱스를 alias에 연결하고 그 alias로 색인해야 롤오버가 새 인덱스를 만들고 alias를 자동으로 옮긴다. 이걸 모르고 롤오버가 안 도는 걸로 한참 헤매는 경우가 많다.

ISM은 즉시 실행이 아니라 주기적으로 인덱스를 훑어 조건을 평가한다. 기본 주기가 30~48분 정도라, 조건을 충족해도 바로 전이되지 않는다. "30일 지났는데 왜 안 지워지지" 하고 보면 다음 평가 주기를 기다리는 중인 경우가 흔하다. 급하면 `_plugins/_ism/explain`으로 현재 상태와 다음 평가 시점을 확인한다.

warm_migration은 UltraWarm 노드가 있는 도메인에서만 쓸 수 있다. UltraWarm을 안 켰으면 이 액션이 실패하면서 인덱스가 그 상태에서 멈춘다. 정책을 짜기 전에 도메인에 해당 계층이 있는지부터 확인해야 한다.

## Dashboards 접근 제어

OpenSearch Dashboards(예전 Kibana)는 클러스터에 그대로 노출하면 큰일 난다. 데이터 전체를 조회·삭제할 수 있는 콘솔이라 인증을 반드시 걸어야 한다. 접근 제어는 크게 네트워크 레벨과 애플리케이션 레벨로 나뉜다.

### 네트워크 레벨

도메인을 VPC 안에 두면 외부에서 직접 못 들어온다. 퍼블릭 엔드포인트로 만들면 인터넷에 노출되니, 운영 도메인은 VPC 안에 넣는 게 기본이다. VPC 도메인은 사내망이나 VPN, 또는 프록시를 거쳐야 Dashboards에 접근한다. 다만 VPC 안에 넣으면 외부 SaaS(예: 외부 모니터링 도구)에서 직접 못 붙으니, 접근 경로를 미리 정리하고 만들어야 한다. 이건 생성 시점에 정하는 거라 나중에 바꾸기 번거롭다.

### Fine-Grained Access Control

OpenSearch에는 FGAC(Fine-Grained Access Control)라는 세분화 접근 제어가 있다. 이걸 켜면 인덱스 단위, 문서 단위, 필드 단위로 권한을 나눈다. 내부 사용자 DB를 두고 마스터 유저로 로그인하거나, IAM 역할과 매핑하는 방식이 있다.

실무에서 자주 쓰는 구성은 이렇다. 운영팀은 모든 인덱스를 보고, 개발팀은 자기 서비스 로그 인덱스만 본다. 보안상 민감한 필드(예: 사용자 이메일)는 특정 역할만 볼 수 있게 필드 레벨로 가린다. 역할(role)을 만들고 인덱스 패턴별 권한을 부여한 뒤, 사용자나 백엔드 역할을 그 role에 매핑한다.

Dashboards 로그인은 Cognito와 붙이는 경우가 많다. Cognito 사용자 풀로 로그인 화면을 띄우고, 사내 IdP(SAML)와 연동하면 회사 계정으로 SSO가 된다. 직접 SAML을 OpenSearch에 붙이는 것도 된다. 어느 쪽이든 Dashboards에 인증 없이 들어가지지 않게 만드는 게 출발점이다.

주의할 점은 FGAC를 켜면 HTTPS가 강제되고, 클러스터 내부 통신 암호화(node-to-node encryption)도 같이 요구된다는 거다. 그리고 FGAC는 도메인 생성 후에 켜기 까다로운 옵션이라, 운영 도메인은 처음 만들 때 켜두는 편이 낫다.

## 로그 수집 파이프라인

OpenSearch는 데이터를 받아서 색인하는 쪽이고, 로그를 모아 보내주는 수집기가 따로 필요하다. 보내는 도구가 여럿이라 상황에 맞게 고른다.

### Logstash / Data Prepper

Logstash는 ELK 시절부터 쓰던 수집·변환 파이프라인이다. input → filter → output 구조로, 로그를 받아 파싱·가공한 뒤 OpenSearch로 보낸다. grok으로 비정형 로그를 필드로 쪼개는 변환이 강점이다. 다만 JVM 기반이라 무겁고 메모리를 많이 먹는다. OpenSearch 진영에서는 Logstash 대체로 Data Prepper를 미는데, 같은 변환을 더 가볍게 한다.

```
# Logstash output을 OpenSearch로 보내는 설정 예
output {
  opensearch {
    hosts => ["https://search-mydomain.region.es.amazonaws.com:443"]
    auth_type => {
      type => "aws_iam"
      region => "ap-northeast-2"
    }
    index => "logs-app-%{+YYYY.MM.dd}"
  }
}
```

### Fluent Bit

컨테이너 환경, 특히 EKS에서는 Fluent Bit이 사실상 표준이다. C로 짜여서 가볍고, 노드마다 DaemonSet으로 띄워서 컨테이너 stdout 로그를 긁어 OpenSearch로 보낸다. EKS에서 애플리케이션 로그를 OpenSearch로 모을 때 거의 이걸 쓴다.

```ini
[OUTPUT]
    Name            opensearch
    Match           kube.*
    Host            search-mydomain.region.es.amazonaws.com
    Port            443
    TLS             On
    AWS_Auth        On
    AWS_Region      ap-northeast-2
    Index           logs-app
    Suppress_Type_Name On
```

`Suppress_Type_Name On`을 빼먹으면 OpenSearch 2.x에서 색인이 실패한다. 옛 Elasticsearch의 타입 개념이 사라졌는데 Fluent Bit이 `_type`을 붙여 보내서 거부당한다. 버전 올린 뒤 로그가 안 들어오면 이걸 먼저 의심한다.

### Kinesis Data Firehose

Firehose는 관리형 수집 스트림이다. 수집기를 직접 운영하지 않고 AWS에 맡기고 싶을 때 쓴다. CloudWatch Logs 구독, 애플리케이션 PutRecord, 또는 다른 AWS 서비스에서 Firehose로 흘려보내면 버퍼링해서 OpenSearch로 배치 색인한다. 색인 실패한 레코드를 S3에 백업하는 기능이 있어서 데이터 유실을 막는다.

Firehose의 장점은 버퍼링과 재시도, 백압(backpressure) 처리를 AWS가 해준다는 점이다. 트래픽이 튀어도 Firehose가 버퍼에 쌓았다가 흘려보내서 OpenSearch가 색인 폭주로 죽는 걸 어느 정도 막는다. 단점은 변환 능력이 약해서 복잡한 파싱은 Lambda 변환을 끼워야 한다는 점, 그리고 버퍼 크기·간격만큼 지연이 생긴다는 점이다. 실시간성이 중요하면 버퍼 간격을 줄이되, 너무 줄이면 작은 배치가 자주 들어와서 색인 효율이 떨어진다.

### 어느 걸 고를까

수집기 선택은 환경으로 갈린다. EKS 컨테이너 로그면 Fluent Bit, 비정형 로그를 무겁게 파싱해야 하면 Logstash나 Data Prepper, 수집기 운영 자체를 피하고 싶고 AWS 서비스 로그가 주력이면 Firehose다. 여러 개를 섞어 쓰는 경우도 흔하다. 컨테이너 로그는 Fluent Bit으로, ALB·CloudFront 로그는 Firehose로 보내는 식이다.

색인 부하 관리는 어느 수집기를 쓰든 신경 써야 한다. 수집기가 동시에 대량으로 밀어넣으면 OpenSearch의 색인 큐(bulk thread pool)가 차서 `429 Too Many Requests`(es_rejected_execution_exception)가 뜬다. 이때 수집기가 재시도를 안 하면 로그가 유실된다. bulk 크기를 적당히 키우고(레코드 수천 개 단위), 수집기 측 재시도와 백오프를 켜둬야 한다.

## 버전 업그레이드와 블루/그린 배포

매니지드의 가장 큰 함정이 여기 있다. OpenSearch는 설정 변경 대부분을 블루/그린 배포로 처리한다. 노드 타입 변경, 버전 업그레이드, 일부 설정 변경을 하면 AWS가 새 환경(그린)을 통째로 띄워서 데이터를 복사하고, 다 끝나면 트래픽을 그쪽으로 넘긴 뒤 옛 환경(블루)을 내린다.

문제는 이게 데이터를 통째로 복사하는 작업이라 시간이 오래 걸리고, 그동안 리소스를 두 배로 쓴다는 점이다. 데이터가 수 TB면 블루/그린 한 번에 몇 시간이 걸린다. 그동안 클러스터는 떠 있지만 성능이 떨어진다. 새 노드로 샤드를 복사하느라 디스크·네트워크가 바쁘고, 색인·검색 지연이 평소보다 길어진다. 그래서 설정 변경은 트래픽 적은 시간대에 잡아야 한다.

블루/그린 도중에는 다른 설정 변경을 못 한다. 한 번 시작하면 끝날 때까지 도메인이 "처리 중" 상태로 잠긴다. 급하게 노드를 더 붙이고 싶어도 진행 중인 배포가 끝나야 한다. 그래서 변경을 여러 개 몰아서 한 번에 하는 게 나을 때도 있다.

용량이 부족하면 블루/그린이 실패한다. 그린 환경을 띄울 디스크·샤드 여유가 없으면 배포가 중간에 막힌다. 디스크 사용률이 높은 상태에서 버전 업그레이드를 걸면 그린 쪽에서 샤드 할당이 안 돼서 진행이 멈춘다. 업그레이드 전에 디스크 여유를 확보하는 게 순서다.

### 버전 업그레이드 자체의 주의점

OpenSearch는 한 단계씩 올리는 게 안전하다. 여러 메이저 버전을 건너뛰면 호환성 문제가 한꺼번에 터진다. AWS가 지원하는 업그레이드 경로가 정해져 있어서, 멀리 떨어진 버전으로는 한 번에 못 가고 중간 버전을 거쳐야 하는 경우가 있다.

업그레이드 전에 스냅샷을 반드시 찍는다. AWS가 자동 스냅샷을 만들지만, 수동으로 S3에 하나 더 떠두는 게 안전하다. 업그레이드는 되돌릴 수 없어서, 올린 뒤 문제가 생기면 새 도메인을 만들어 스냅샷에서 복원하는 수밖에 없다.

업그레이드로 깨지는 건 보통 클라이언트 쪽이다. 7.x에서 매핑 타입(`_type`)이 사라지고, 일부 쿼리 DSL 문법이 바뀌고, 클라이언트 라이브러리 호환 버전이 달라진다. 앱에서 쓰는 OpenSearch 클라이언트가 새 버전을 지원하는지, 색인 요청에 `_type`이 박혀 있지 않은지를 업그레이드 전에 확인한다. 스테이징 도메인에 먼저 올려서 앱이 정상 동작하는지 보고, 운영에 올리는 순서가 맞다.

플러그인과 인덱스 호환성도 본다. 아주 오래된 버전에서 만든 인덱스는 최신 버전이 못 읽는 경우가 있다. 이런 인덱스는 업그레이드 전에 reindex로 새로 만들어둬야 한다. 안 그러면 업그레이드 후에 그 인덱스가 안 열린다.

## 운영에서 자주 겪는 문제

클러스터 상태가 red면 데이터 유실 가능성이 있다. yellow는 레플리카가 할당 안 된 상태라 검색·색인은 되지만, red는 프라이머리 샤드가 할당 안 된 상태라 일부 데이터를 못 읽는다. red가 뜨면 `_cluster/allocation/explain`으로 왜 샤드 할당이 안 되는지부터 본다. 보통 디스크 부족이 원인이다.

디스크 워터마크를 넘으면 색인이 멈춘다. OpenSearch는 디스크 사용률이 일정 선(기본 85%)을 넘으면 그 노드에 새 샤드를 안 만들고, 더 높은 선(기본 95%, flood stage)을 넘으면 인덱스를 읽기 전용으로 잠근다. 디스크가 차서 색인이 갑자기 막히면 인덱스가 read-only로 잠긴 거다. 디스크를 비우거나 늘린 뒤 `index.blocks.read_only_allow_delete`를 풀어야 색인이 재개된다. 로그성 클러스터는 ISM으로 오래된 인덱스를 제때 지워서 이 상황 자체를 안 만드는 게 맞다.

JVM 힙 압력(JVMMemoryPressure)을 봐야 한다. 힙이 75%를 자주 넘으면 GC가 잦아지고, 더 차면 노드가 응답을 못 한다. 원인은 대개 샤드 과다나 무거운 집계 쿼리다. CloudWatch에서 JVMMemoryPressure, CPUUtilization, FreeStorageSpace, ClusterStatus를 기본 알람으로 걸어둔다.

검색이 느려지면 샤드 수와 쿼리를 같이 본다. 샤드가 너무 많으면 쿼리 하나가 수백 개 샤드에 흩어져서 코디네이션 오버헤드가 커지고, 무거운 집계나 와일드카드 검색은 그 자체로 느리다. 느린 쿼리는 slow log를 켜서 어떤 쿼리가 시간을 먹는지 잡아낸다.

매니지드라 직접 못 만지는 설정이 있다. `elasticsearch.yml`을 직접 수정하거나, 일부 클러스터 레벨 설정(`_cluster/settings`)을 바꾸는 게 막혀 있다. 자체 호스팅 ELK에서 하던 튜닝이 OpenSearch에서 안 먹히는 경우가 있어서, 옮기기 전에 필요한 설정이 매니지드에서 허용되는지 확인해야 한다.
