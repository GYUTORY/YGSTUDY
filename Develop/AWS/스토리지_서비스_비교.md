# AWS 스토리지 서비스 비교

## 클라우드 스토리지란?

클라우드 스토리지는 **인터넷을 통해 데이터를 안전하게 저장하고 언제 어디서나 접근할 수 있는 유연한 저장 공간**입니다. 전통적인 하드디스크나 SSD와 달리, 물리적 하드웨어를 직접 관리할 필요 없이 클라우드 제공자가 모든 인프라를 관리합니다.

### 클라우드 스토리지 서비스의 특징

**1. 용량 쉽게 확장 가능**
- 필요에 따라 즉시 용량 증설
- 하드웨어 추가 구매 불필요
- 무제한에 가까운 확장성

**2. 언제든 접근 가능**
- 인터넷만 연결되어 있으면 전세계 어디서나 접근
- 여러 디바이스에서 동시 접근 가능
- RESTful API를 통한 프로그래밍 방식 접근

**3. 여러 데이터 센터에 데이터 복제**
- 자동으로 여러 가용영역에 데이터 복제
- 하드웨어 장애에도 데이터 안전하게 보호
- 99.999999999% (11 nines) 내구성

**4. 데이터 암호화**
- 전송 중 암호화 (TLS/SSL)
- 저장 시 암호화 (AES-256)
- 키 관리 서비스 통합

**5. 사용한 만큼만 비용 지불**
- 선불 비용 없음
- 실제 사용량 기반 과금
- 용량 계획 불필요

## AWS 클라우드 스토리지 유형

### 1. S3 (Simple Storage Service) - 객체 스토리지

#### 특징

**데이터 저장 방식:**
- 데이터를 하나의 **객체(Object)**로 저장
- 파일과 메타데이터를 함께 저장
- 계층 구조 없이 평면적으로 저장 (폴더는 논리적 구조)

**확장성:**
- 무제한에 가까운 확장성
- 단일 객체 최대 5TB
- 버킷당 객체 수 제한 없음

**접근 방식:**
- RESTful API를 통해 데이터 접근
- HTTP/HTTPS 프로토콜 사용
- SDK 및 CLI 도구 제공

**비용:**
- **가장 저렴한 스토리지 옵션**
- 스토리지 클래스별 차등 요금
- 요청 횟수 기반 추가 비용

#### 스토리지 클래스

| 클래스 | 용도 | 검색 시간 | 비용 |
|--------|------|----------|------|
| **Standard** | 자주 접근하는 데이터 | 즉시 | 높음 |
| **Intelligent-Tiering** | 접근 패턴 불규칙 | 즉시 | 자동 최적화 |
| **Standard-IA** | 자주 접근하지 않음 | 즉시 | 중간 |
| **One Zone-IA** | 재생성 가능 데이터 | 즉시 | 낮음 |
| **Glacier** | 장기 보관 | 분~시간 | 매우 낮음 |
| **Glacier Deep Archive** | 규정 준수 보관 | 12시간 | 최저 |

#### 보안

- **버킷 정책**: 버킷 레벨 접근 제어
- **ACL (Access Control List)**: 객체별 세밀한 제어
- **IAM 정책**: 사용자/역할 기반 접근 제어
- **암호화**: 서버측 암호화 (SSE-S3, SSE-KMS, SSE-C)

#### 사용 사례

**1. 웹 콘텐츠 저장/배포**
- 이미지, 비디오, CSS, JavaScript 파일 호스팅
- CloudFront와 연계한 CDN 구성
- 정적 웹사이트 호스팅

**2. 대용량 데이터 저장**
- 빅데이터 분석을 위한 데이터 레이크
- 로그 파일 중앙 집중 저장
- 센서 데이터, IoT 데이터 저장

**3. 백업 및 아카이브**
- 데이터베이스 백업
- 재해 복구 (DR) 솔루션
- 규정 준수를 위한 장기 보관

**4. 정적 웹사이트 호스팅**
- HTML, CSS, JavaScript 직접 호스팅
- 서버 없이 웹사이트 운영
- Route 53과 연계한 커스텀 도메인

#### 비용 (us-east-1 기준)

```
스토리지 비용:
- Standard: $0.023/GB/월
- Standard-IA: $0.0125/GB/월
- Glacier: $0.004/GB/월

요청 비용:
- GET 요청: $0.0004/1,000건
- PUT 요청: $0.005/1,000건

데이터 전송:
- 인바운드: 무료
- 아웃바운드: $0.09/GB (첫 1TB)
```

#### S3 접근 시 비용 청구

**질문:** S3에 접근이 거부되어도 비용이 청구되나요?

**답변:**
- **인증 실패로 거부**: 비용 청구 안 됨
- **권한 부족으로 거부**: API 요청이 처리되었으므로 **비용 청구됨**
- 403 Forbidden 응답도 하나의 요청으로 간주

### 2. EBS (Elastic Block Store) - 블록 스토리지

#### 특징

**데이터 저장 방식:**
- 데이터를 일정 크기의 **블록(Block)**으로 저장
- 파일 시스템 위에서 동작
- 직접 파티셔닝 및 포맷 가능

**성능:**
- **낮은 지연시간, 높은 성능**
- EC2 인스턴스에 직접 연결
- 네트워크를 통한 블록 레벨 접근

**제한사항:**
- **단일 EC2 인스턴스에만 연결 가능** (Multi-Attach 제외)
- 동일 가용영역 내에서만 사용
- 최대 용량 제한 있음

#### EBS 볼륨 타입

**1. gp2, gp3 (범용 SSD)**

**특징:**
- 중소규모 데이터베이스에 적합
- 개발 및 테스트 환경
- 최대 16,000 IOPS 제공
- 비용 효율적

**사용 사례:**
- 웹 서버 루트 볼륨
- 소규모 데이터베이스
- 개발/테스트 환경

**비용:** $0.10/GB/월 (gp3)

**2. io1, io2 (프로비저닝 IOPS SSD)**

**특징:**
- I/O 집약적 워크로드용 고성능 SSD
- 최대 64,000 IOPS 제공
- 일관된 성능 보장
- 대규모 데이터베이스 및 실시간 분석

**사용 사례:**
- 대규모 프로덕션 데이터베이스 (Oracle, MySQL, PostgreSQL)
- NoSQL 데이터베이스 (MongoDB, Cassandra)
- 실시간 분석 애플리케이션

**비용:** $0.125/GB/월 (io2) + IOPS당 요금

**3. st1 (Throughput Optimized HDD)**

**특징:**
- 처리량 집약적 워크로드에 최적화
- 순차 읽기/쓰기 성능 우수
- 빅데이터, 로그 처리에 적합
- 저렴한 비용

**사용 사례:**
- 빅데이터 처리
- 로그 수집 및 분석
- 데이터 웨어하우스

**비용:** $0.045/GB/월

**4. sc1 (Cold HDD)**

**특징:**
- 가장 저렴한 HDD 옵션
- 자주 접근하지 않는 데이터
- 순차 접근 패턴

**사용 사례:**
- 콜드 데이터 저장
- 아카이브 백업
- 비용 최적화가 중요한 경우

**비용:** $0.015/GB/월

#### 사용 사례

**1. 데이터베이스 스토리지**
- MySQL, PostgreSQL, Oracle 등의 데이터 파일
- 트랜잭션 로그 저장
- 높은 IOPS와 낮은 지연시간 필요

**2. 파일 시스템**
- EC2 인스턴스의 루트 볼륨
- 애플리케이션 데이터 저장
- 사용자 파일 시스템

**3. 운영체제 볼륨**
- EC2 인스턴스 부팅 디스크
- 시스템 파일 저장
- 스냅샷을 통한 백업

**4. 빅데이터 분석을 위한 임시 스토리지**
- Hadoop, Spark 작업 데이터
- ETL 처리 중간 데이터
- 일시적 데이터 저장

### 3. EFS (Elastic File System) - 파일 스토리지

#### 특징

**데이터 저장 방식:**
- 디렉토리 구조로 파일을 저장
- 전통적인 파일 시스템과 동일한 방식
- NFS (Network File System) 프로토콜 사용

**공유 가능:**
- **여러 EC2 인스턴스에서 동시 접근 가능**
- 수천 개의 인스턴스 동시 마운트
- 읽기/쓰기 동시 수행

**탄력성:**
- 자동으로 확장 및 축소
- 페타바이트 규모까지 확장
- 용량 계획 불필요

**성능:**
- 범용 모드: 낮은 지연시간
- Max I/O 모드: 높은 처리량
- 버스팅 및 프로비저닝 처리량 모드

#### 스토리지 클래스

**Standard:**
- 자주 접근하는 데이터
- 높은 가용성
- 비용: $0.30/GB/월

**Infrequent Access (IA):**
- 자주 접근하지 않는 데이터
- 자동 계층화 가능
- 비용: $0.025/GB/월

#### 사용 사례

**1. 콘텐츠 관리 시스템 (CMS)**
- 워드프레스, 드루팔 등의 공유 스토리지
- 미디어 파일 중앙 집중 관리
- 여러 웹 서버에서 동시 접근

**2. 웹 서버 파일 공유**
- 정적 파일 공유 (이미지, CSS, JavaScript)
- 세션 파일 공유
- 로그 파일 중앙 집중

**3. 빅데이터 분석 및 머신러닝 워크로드**
- 학습 데이터셋 공유
- 모델 파일 공유
- 여러 컴퓨팅 노드에서 동시 접근

**4. 미디어 및 엔터테인먼트 워크플로우**
- 비디오 편집 프로젝트 공유
- 렌더링 팜의 공유 스토리지
- 협업 작업 환경

## AWS 스토리지 서비스 비교

### 주요 특성 비교

| 특성 | S3 | EBS | EFS |
|------|----|----|-----|
| **저장 방식** | 객체 스토리지 | 블록 스토리지 | 파일 스토리지 |
| **접근 방식** | HTTP/HTTPS API | 블록 디바이스 | NFS 프로토콜 |
| **동시 접근** | 무제한 | 단일 인스턴스 | 수천 개 인스턴스 |
| **확장성** | 무제한 | 최대 64TB | 페타바이트 |
| **지연시간** | 수십~수백 ms | 한 자릿수 ms | 한 자릿수 ms |
| **내구성** | 99.999999999% | 99.8-99.9% | 99.999999999% |
| **비용** | **가장 저렴** | 중간 | 높음 |
| **가용영역** | 리전 전체 | 단일 AZ | 리전 전체 |

### 속도 및 성능 비교

**처리 속도:**
```
읽기/쓰기 속도 (순위):
1위: EBS (io2) - 최대 64,000 IOPS
2위: EFS - 범용 모드
3위: EBS (gp3) - 최대 16,000 IOPS
4위: S3 - HTTP 기반 접근
```

**지연시간:**
```
지연시간 (낮은 순):
1위: EBS - 한 자릿수 밀리초
2위: EFS - 한 자릿수 밀리초
3위: S3 - 수십 밀리초
```

**처리량:**
```
대용량 데이터 처리:
1위: S3 - 병렬 처리 가능
2위: EBS (st1) - 순차 처리 최적화
3위: EFS - 다중 클라이언트 처리
```

### 비용 효율성 비교

**스토리지 비용 (us-east-1 기준):**
```
1TB 저장 시 월 비용:
- S3 Standard: $23
- EBS gp3: $100
- EBS io2: $125
- EFS Standard: $300
- EFS IA: $25
```

**비용 최적화 팁:**
- **장기 보관**: S3 Glacier 또는 EFS IA
- **고성능 필요**: EBS io2
- **공유 필요**: EFS
- **웹 콘텐츠**: S3 + CloudFront

### 파일 다운로드가 잦은 경우

**질문:** 30MB 파일을 올리고 다운로드가 잦을 때 어떤 것이 더 유용한가?

**답변:**

**1. S3 + CloudFront (권장)**
- S3에 원본 저장
- CloudFront로 CDN 구성
- 빠른 다운로드 속도
- 비용 효율적
- 전세계 배포에 최적

**2. EBS**
- EC2 인스턴스를 통한 다운로드 필요
- 단일 인스턴스 병목 가능성
- 추가 네트워크 비용
- 권장하지 않음

**3. EFS**
- 여러 EC2에서 동시 제공 가능
- 높은 비용
- 단순 파일 제공에는 과한 선택

**결론:**
- **다운로드 빈도가 높은 경우**: S3 + CloudFront
- **EC2 내부 사용**: EBS
- **여러 서버에서 공유**: EFS

## Private Subnet 접근 방법

Private Subnet의 리소스에 접근하는 방법은 보안 수준과 편의성에 따라 선택할 수 있습니다.

### 1. Bastion Host (Jump Server)

**구성:**
```
인터넷
  ↓
Public Subnet
  └─ Bastion Host (보안 강화된 EC2)
      ↓
Private Subnet
  └─ 대상 EC2 인스턴스
```

**특징:**
- Public Subnet에 배치된 중계 서버
- SSH 접속을 위한 진입점
- 보안 그룹으로 특정 IP만 허용

### 2. VPN 연결

**구성:**
```
온프레미스 네트워크
  ↓ VPN Tunnel
AWS VPC Private Subnet
  └─ 대상 리소스
```

**특징:**
- 암호화된 터널 연결
- 온프레미스와 AWS 간 안전한 연결
- VPN Gateway 필요

### 3. AWS Direct Connect

**구성:**
```
온프레미스 데이터센터
  ↓ 전용 회선
AWS Direct Connect Location
  ↓
AWS VPC Private Subnet
```

**특징:**
- 전용 네트워크 연결
- 높은 대역폭과 낮은 지연시간
- 안정적인 연결
- 높은 비용

### 4. Session Manager (권장)

**구성:**
```
사용자
  ↓ AWS Console/CLI
AWS Systems Manager
  ↓ 인터넷 불필요
Private Subnet EC2
```

**특징:**
- 인터넷 연결 불필요
- 22번 포트를 열 필요 없음
- IAM 정책 기반 접근 제어
- 세션 로그 자동 기록

### 보안 그룹 설정

Private Subnet의 리소스는 **보안 그룹에 접근 허용 인스턴스를 설정**하여 제어합니다.

**예시: 웹 서버와 DB 서버**

```
Public Subnet (ALB)
  ↓ 보안 그룹: ALB → Web Server 허용
Private Subnet (Web Server)
  ↓ 보안 그룹: Web Server → DB 허용
Private Subnet (DB Server)
```

**보안 그룹 규칙:**
```
DB 서버 보안 그룹:
- Inbound: 
  - MySQL 포트 3306
  - Source: Web Server 보안 그룹 ID
  
Web Server 보안 그룹:
- Inbound:
  - HTTP 포트 80
  - Source: ALB 보안 그룹 ID
```

## 실무 활용 팁

### 1. 스토리지 선택 체크리스트

```
데이터 특성 분석:
├─ 자주 접근하는가?
│  ├─ 예 → Standard 클래스 선택
│  └─ 아니오 → IA 또는 Glacier 고려
│
├─ 여러 서버에서 공유 필요?
│  ├─ 예 → EFS 또는 S3
│  └─ 아니오 → EBS
│
├─ 성능이 중요한가?
│  ├─ 예 → EBS (io2)
│  └─ 아니오 → S3 또는 EFS
│
└─ 비용이 가장 중요한가?
   ├─ 예 → S3 (Glacier)
   └─ 아니오 → 성능과 비용 균형 고려
```

### 2. 하이브리드 접근

실무에서는 여러 스토리지를 조합하여 사용합니다.

**일반적인 웹 애플리케이션 구성:**
```
S3
├─ 정적 파일 (이미지, CSS, JS)
└─ 사용자 업로드 파일

EBS (gp3)
├─ EC2 루트 볼륨
├─ 애플리케이션 코드
└─ 임시 데이터

EBS (io2)
└─ 데이터베이스 볼륨

EFS
└─ 공유 파일 시스템 (세션, 로그)
```

### 3. 수명 주기 정책 활용

**S3 수명 주기:**
```
0일: S3 Standard (업로드)
  ↓
30일: S3 Standard-IA (자동 이동)
  ↓
90일: S3 Glacier (자동 이동)
  ↓
365일: S3 Glacier Deep Archive (자동 이동)
```

**비용 절감 효과:**
- Standard 대비 90% 이상 절감 가능
- 자동화로 관리 부담 최소화
