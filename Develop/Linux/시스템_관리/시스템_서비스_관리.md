# 시스템 서비스 관리

## 이 문서가 다루는 범위

같은 디렉토리에 `Systemd_Deep_Dive.md`가 있어서, 이 문서는 그 앞단에 해당하는 내용을 담는다. SysV init에서 systemd로 어떻게 바뀌었는지, runlevel과 target은 어떻게 대응되는지, unit 파일을 처음 작성할 때 알아야 하는 옵션 차이는 무엇인지, 운영 중 서비스가 안 뜰 때 어디부터 보면 되는지를 다룬다. cgroup v2, sd_notify, socket activation 같은 깊이 있는 주제는 Deep Dive 쪽으로 미룬다.

5년 정도 RHEL/Ubuntu 양쪽을 만지면서 느낀 건, 신입 엔지니어가 "왜 `service`랑 `systemctl`이 둘 다 있냐", "왜 `enable`했는데 안 뜨냐", "재시작 루프가 도는데 왜 자동으로 멈추냐" 같은 데서 자주 막힌다는 점이다. 이 글은 그런 지점을 풀어준다.

## SysV init과 systemd

### 옛날 방식

RHEL 6, CentOS 6, 오래된 임베디드 시스템에서 아직도 볼 수 있는 SysV init은 `/etc/init.d/` 아래에 쉘 스크립트를 두고 `start`, `stop`, `restart`, `status` 인자를 받는 식이었다. 부팅하면 `/etc/rcN.d/SXXservice` 같은 심볼릭 링크를 알파벳 순서로 실행한다. 여기서 N은 runlevel이다.

치명적 한계가 두 가지 있었다. 첫째, **순차 실행**이라 부팅이 느렸다. nfs가 30초 걸리면 그 뒤에 있는 모든 서비스가 30초씩 밀린다. 둘째, **의존성 표현이 빈약**했다. 시작 순서를 숫자로 끼워 넣는 방식이라 "DB가 정말 ready 됐을 때 앱을 시작" 같은 의미를 표현할 수 없었다. 그냥 "DB 다음 번호를 부여해서 나중에 시작되길 빌었던" 것이다.

upstart도 잠깐 있었지만(Ubuntu 9.10 ~ 14.04, RHEL 6의 일부) 결국 systemd로 수렴했다. 실무에서는 SysV 스크립트를 작성할 일은 거의 없고, 다만 옛날 서드파티 어플라이언스나 임베디드를 만질 때 `/etc/init.d/` 스크립트를 읽고 수정해야 할 때가 있다.

### systemd가 바꾼 것

systemd가 가져온 본질적인 변화는 세 가지다.

- **선언적 unit 파일**: 쉘 스크립트가 아니라 `.ini` 형식의 선언이다. 시작 명령, 의존성, 재시작 정책, 리소스 한도를 모두 한 파일에 적는다.
- **병렬 부팅**: 의존성 그래프를 먼저 만들고, 의존이 풀리는 대로 동시에 시작한다. 노트북에서 부팅이 5초 안에 끝나는 게 이 덕분이다.
- **단일화된 관리 인터페이스**: 서비스, 마운트, 타이머(cron 대체), 소켓, 디바이스, 슬라이스를 모두 같은 `systemctl`로 다룬다.

`service nginx restart`를 치면 systemd가 깔린 시스템에서는 내부적으로 `systemctl restart nginx`로 변환된다. 즉 `service` 명령은 호환성을 위한 래퍼고, 새로운 운영 스크립트를 짤 때는 `systemctl`만 쓰면 된다.

## runlevel과 target

### 매핑

SysV의 runlevel은 `0~6` 숫자였다. systemd는 그 자리를 **target**이 차지하지만 의미를 더 일반화했다. target은 단순히 "이 시점까지 어떤 unit이 떠야 하는가"라는 동기화 지점이고, 그래프 상의 노드일 뿐이다.

| runlevel | systemd target | 의미 |
|----------|----------------|------|
| 0 | `poweroff.target` | 종료 |
| 1 | `rescue.target` | 단일 사용자 모드 (root 셸, 네트워크 없음) |
| 2 | `multi-user.target` | 멀티 유저 (Debian 계열에서 NFS 없는 멀티유저였음) |
| 3 | `multi-user.target` | 멀티 유저, 네트워크, CLI |
| 4 | `multi-user.target` | 사용자 정의 |
| 5 | `graphical.target` | 멀티 유저 + GUI |
| 6 | `reboot.target` | 재부팅 |

호환을 위해 `/lib/systemd/system/runlevel3.target` 같은 심볼릭 링크가 `multi-user.target`을 가리키도록 깔려 있다. 그래서 옛날 자동화 스크립트가 `init 3`을 호출해도 동작한다.

### 현재 target 확인하고 바꾸기

```bash
systemctl get-default                    # 부팅 기본 target
systemctl set-default multi-user.target  # GUI 끄기 (서버에서 자주)
systemctl isolate rescue.target          # 즉시 rescue로 전환 (위험, 콘솔에서만)
```

서버를 GUI 없이 쓰겠다면 `multi-user.target`을 default로 잡는다. `graphical.target`은 `multi-user.target`을 포함(Wants)하기 때문에, GUI 모드에서도 모든 서비스가 같이 떠있다.

`isolate`는 현재 target과 그 하위만 남기고 다 내린다. SSH로 들어와서 `isolate rescue.target`을 치면 본인 세션이 끊겨서 다시 못 들어오는 사고가 생긴다. 콘솔 외에서는 쓰지 않는 게 안전하다.

## unit 파일 기본기

### 위치와 우선순위

unit 파일이 놓이는 디렉토리는 여러 개이고 우선순위가 있다. 같은 이름이면 위에 적힌 곳이 이긴다.

```
/etc/systemd/system/        # 관리자가 직접 만든 것 (가장 우선)
/run/systemd/system/        # 런타임에 생성된 것 (재부팅하면 사라짐)
/usr/lib/systemd/system/    # 패키지가 깔아둔 것 (배포판 기본)
/lib/systemd/system/        # 위와 같음 (Debian 계열에서 흔히 사용)
```

패키지로 설치한 nginx의 unit 파일은 `/usr/lib/systemd/system/nginx.service`다. 절대 거기서 직접 수정하지 않는다. 패키지 업데이트가 들어오면 덮어써지기 때문이다. 수정할 일이 있다면 drop-in으로 한다(아래에서 다룬다).

### 최소한의 service 파일

```ini
# /etc/systemd/system/myapp.service
[Unit]
Description=My App
Documentation=https://wiki.example.com/myapp
After=network-online.target postgresql.service
Wants=network-online.target

[Service]
Type=simple
User=myapp
Group=myapp
WorkingDirectory=/opt/myapp
EnvironmentFile=/etc/myapp/env
ExecStart=/opt/myapp/bin/server
Restart=on-failure
RestartSec=5s

[Install]
WantedBy=multi-user.target
```

신입이 가장 자주 빠뜨리는 부분은 `User=`다. 안 적으면 root로 돈다. 웹 앱을 root로 돌리는 건 두말할 것 없이 사고의 시작이다.

`After=`와 `Wants=`를 같이 쓰는 패턴(`network-online.target`)을 기억해 두면 좋다. `After=`만 있으면 시작 순서만 잡고 의존성은 안 만들기 때문에, network-online이 실제로 안 떠 있으면 의존이 없다고 판단해 그냥 출발한다. `Wants=`로 의존을 명시해야 systemd가 network-online을 활성화시킨다.

### `WantedBy`의 의미

`WantedBy=multi-user.target`은 "이 서비스를 enable하면 `multi-user.target.wants/` 디렉토리에 심볼릭 링크를 만들어라"라는 뜻이다. 결과적으로 부팅 시 multi-user.target이 도달할 때 이 서비스도 시작된다. `enable`/`disable`은 사실 이 심볼릭 링크를 만들고 지우는 것이다.

```bash
systemctl enable myapp
# 내부적으로:
# ln -s /etc/systemd/system/myapp.service /etc/systemd/system/multi-user.target.wants/myapp.service
```

이걸 알면 `is-enabled`가 왜 `enabled`/`disabled`/`masked`/`static` 같은 다양한 상태를 가지는지 이해된다. `static`은 `[Install]` 섹션이 아예 없어서 enable 자체가 의미 없는 unit(다른 unit이 의존성으로 끌고 가는 unit)을 가리킨다.

## Restart 정책의 실제 동작

### 옵션 차이

`Restart=` 옵션은 단순히 "재시작 할까 말까"가 아니라 종료 사유에 따라 다르게 동작한다. 종료 사유는 정상 종료(exit 0), 비정상 종료(exit !=0), 시그널, watchdog 타임아웃, abort 등으로 나뉜다.

| 옵션 | 정상 종료 | 비정상 종료 | 신호 | watchdog |
|------|-----------|-------------|------|----------|
| `no` | x | x | x | x |
| `on-success` | o | x | x | x |
| `on-failure` | x | o | o(SIGHUP/INT/TERM/PIPE 제외) | o |
| `on-abnormal` | x | x | o | o |
| `on-watchdog` | x | x | x | o |
| `on-abort` | x | x | SIGABRT만 | x |
| `always` | o | o | o | o |

웹 서버나 API처럼 24시간 떠 있어야 하는 서비스는 `on-failure`나 `always`를 쓴다. `always`는 `systemctl stop`으로 정상 종료시켜도 다시 살아나려고 하기 때문에 막연히 좋은 선택이 아니다. 운영 중에 잠깐 내려야 할 때 헷갈린다. 보통은 `on-failure`가 정답이다. 정상 종료는 의도된 종료고, 비정상이나 신호는 사고이므로 살리고 싶은 게 일반적이다.

배치 작업 같은 일회성 스크립트는 `Restart=no`(또는 `Type=oneshot` + Restart 미지정)로 둔다. 안 그러면 작업이 끝나자마자 다시 도는 사고가 생긴다.

### 재시작 루프 방지

앱이 시작 직후 죽는 버그가 있으면 `Restart=always` 때문에 무한히 재시작한다. CPU도 먹고 로그도 폭주한다. systemd는 이걸 막기 위해 burst 한도를 갖고 있다.

```ini
[Unit]
StartLimitIntervalSec=60
StartLimitBurst=3

[Service]
Restart=on-failure
RestartSec=5s
```

위 설정은 "60초 안에 3번 이상 시작하면 더 이상 재시작하지 않고 failed 상태로 둔다"는 뜻이다. 한도에 걸리면 `systemctl status`에 `start-limit-hit`이 찍힌다. 이때는 `systemctl reset-failed myapp` 후에 다시 `start`해야 한다.

`RestartSec=`를 짧게 잡으면 burst에 더 빨리 걸린다. 5~10초 정도가 무난하다. 0초로 두면 서비스가 죽고 살아나는 게 거의 동시라 로그를 분석하기도 어렵다.

## 의존성 관리: After / Requires / Wants / BindsTo

이 네 개의 차이를 헷갈리는 사람이 많다. 핵심은 **순서**와 **의존**이 별개라는 것이다.

- `After=`/`Before=`는 **시작 순서**만 정한다. 의존을 만들지 않는다.
- `Requires=`/`Wants=`/`Requisite=`/`BindsTo=`는 **의존**을 만든다. 순서는 따로 안 정해진다.

따라서 보통은 `After=`와 `Requires=`(또는 `Wants=`)를 짝으로 쓴다.

### Wants vs Requires vs BindsTo

| 옵션 | 대상 unit이 시작 안 되면? | 대상 unit이 나중에 죽으면? |
|------|---------------------------|----------------------------|
| `Wants=db.service` | 그래도 자기는 시작 (best-effort) | 자기는 그대로 운영 |
| `Requires=db.service` | 자기도 시작 안 함 (활성화 실패) | 자기는 그대로 운영 |
| `BindsTo=db.service` | 자기도 시작 안 함 | 자기도 같이 멈춤 |
| `Requisite=db.service` | 자기도 시작 안 함 (대상이 이미 떠 있어야 함, 시작 안 시킴) | 영향 없음 |

차이 감을 잡으려면 이런 시나리오를 생각해 보면 된다. PostgreSQL이 아직 안 뜬 상태에서 앱을 시작하려고 한다.

- `Wants=postgresql.service`: postgresql이 시작 시도되고, 실패해도 앱은 떠 보려 한다(아마 connection refused로 실패). 외부 시스템이라 실패 가능성이 있는 의존에 적합.
- `Requires=postgresql.service`: postgresql 시작이 실패하면 앱도 활성화 실패로 처리된다. 핵심 의존에 쓴다. 다만 **postgresql이 나중에 재시작되면 앱은 영향이 없다.** 의외로 자주 헷갈리는 부분이다.
- `BindsTo=postgresql.service`: postgresql이 죽으면 앱도 같이 죽인다. 컨테이너처럼 라이프사이클이 묶여야 할 때 쓴다. 가상 인터페이스가 죽으면 그 위 서비스도 같이 내리는 식.

`Requisite=`는 거의 안 쓴다. 명시적으로 "이미 떠 있는지 확인만 해라"는 의미인데, 부팅 그래프에서는 실용성이 낮다.

### After만 적고 Requires를 빼먹는 사고

```ini
# 잘못된 패턴 — postgresql이 안 떠 있어도 그냥 시작
[Unit]
After=postgresql.service
```

`After=`만 있고 `Requires=`/`Wants=`가 없으면, 앱과 postgresql이 둘 다 부팅 그래프에 들어와 있을 때만 순서가 의미 있다. postgresql이 enable되어 있지 않거나 실패한 상태면 systemd는 그냥 앱부터 시작한다. 시작 순간에 DB 커넥션이 안 잡혀서 앱이 죽는다.

반대로 `Requires=postgresql.service`만 있고 `After=`가 없으면, **시작 순서가 보장되지 않아서** 둘이 동시에 시작되고 거의 항상 앱이 먼저 떠서 connection refused가 난다. 의존과 순서를 같이 적는 게 안전하다.

## drop-in으로 unit 수정하기

패키지로 깔린 nginx의 unit 파일을 직접 수정하지 말라고 했다. 그럼 어떻게 바꾸나? drop-in을 쓴다.

```bash
systemctl edit nginx
```

이 명령은 `/etc/systemd/system/nginx.service.d/override.conf` 파일을 에디터로 띄운다. 여기에 적은 값이 원본 unit에 **덧씌워진다**. 패키지 업데이트가 들어와도 drop-in은 그대로 남아 있다.

```ini
# /etc/systemd/system/nginx.service.d/override.conf
[Service]
LimitNOFILE=131072
Environment="WORKER_CONNECTIONS=8192"
```

drop-in은 디렉토리 안의 모든 `.conf` 파일을 알파벳 순으로 읽는다. 그래서 여러 파일로 쪼개도 된다. 자동화 도구(Ansible, Puppet)가 `10-resources.conf`, `20-env.conf` 식으로 나눠서 떨어뜨리는 패턴이 흔하다.

### 주의할 동작

값이 단순히 추가가 아니라 **덮어쓰기**가 되는 옵션이 있다. 대표적으로 `ExecStart=`는 원본 값을 비우고 덧붙여야 한다.

```ini
[Service]
ExecStart=
ExecStart=/usr/sbin/nginx -g 'daemon off;'
```

빈 `ExecStart=` 한 줄이 "원본을 지운다"는 의미다. 안 적고 `ExecStart=...`만 적으면 systemd가 ExecStart가 두 번 정의됐다고 거부한다. 다른 list 타입 옵션(`Environment=`, `LimitX=`도 일부)도 비슷한 패턴이다.

drop-in을 다 쓴 뒤에는 항상 `systemctl daemon-reload`가 필요하다. `systemctl edit`은 자동으로 호출해 주지만, 수동으로 파일을 만든 경우엔 직접 쳐야 한다.

### 적용된 unit 전체 보기

원본 unit + drop-in이 합쳐진 결과물은 이렇게 본다.

```bash
systemctl cat nginx          # 원본 + drop-in 모두 표시
systemctl show nginx         # 적용된 모든 속성을 key=value로
```

`systemctl cat`은 디버깅에 가장 자주 쓴다. "내가 적은 옵션이 정말 들어갔나" 확인할 때 첫 번째로 보는 명령이다.

## masked unit

`mask`는 disable보다 한 단계 강한 비활성화다. unit 파일을 `/dev/null`로 심볼릭 링크해서 어떤 의존성도 이 unit을 끌어올 수 없게 만든다.

```bash
systemctl mask snapd          # 의존성으로도 시작 못 하게
systemctl unmask snapd        # 풀기
```

언제 쓰나? 패키지가 깔려 있는데 절대 시작되면 안 되는 경우다. 예를 들어 클라우드 이미지에 `snapd`나 `apparmor`처럼 기본 설치된 서비스 중 우리 환경에서 충돌을 일으키는 것이 있을 때, 단순히 `disable`하면 다른 unit의 `Wants=`/`Requires=`로 끌려와 시작될 수 있다. mask하면 systemd가 "이 unit은 존재하지 않는 것"처럼 다룬다.

`is-enabled`로 확인했을 때 `masked`로 나오면 `disable`로는 풀리지 않는다. 반드시 `unmask`를 써야 한다.

## journalctl 연동

systemd로 돌리는 모든 서비스는 stdout/stderr가 자동으로 journal에 흘러간다. 앱에서 별도 로그 라이브러리를 안 써도 `print` 정도만 해도 `journalctl -u myapp`으로 다 보인다. 이게 systemd의 큰 장점이다.

자주 쓰는 패턴 몇 가지.

```bash
journalctl -u myapp                        # 전체
journalctl -u myapp -n 100 --no-pager      # 마지막 100줄
journalctl -u myapp -f                     # tail -f
journalctl -u myapp --since "10 min ago"   # 시간 필터
journalctl -u myapp -p err                 # ERROR 이상만
journalctl -u myapp -o json-pretty         # 구조화된 출력
journalctl -u myapp _PID=1234              # 특정 PID
journalctl _SYSTEMD_UNIT=myapp.service     # unit 명시
```

`-p`(priority)는 syslog 레벨을 따른다: `emerg(0)`, `alert(1)`, `crit(2)`, `err(3)`, `warning(4)`, `notice(5)`, `info(6)`, `debug(7)`. 운영 중 사고를 분석할 때 `-p err`로 좁히면 노이즈가 많이 줄어든다.

journal은 기본적으로 **휘발성**(`/run/log/journal`)일 수도 있다. `/var/log/journal/` 디렉토리가 있으면 영속, 없으면 휘발이다. 영속화하려면 디렉토리만 만들면 된다.

```bash
sudo mkdir -p /var/log/journal
sudo systemd-tmpfiles --create --prefix /var/log/journal
sudo systemctl restart systemd-journald
```

journal 디스크 용량은 `/etc/systemd/journald.conf`의 `SystemMaxUse=`로 제한한다. 안 막으면 `/var/log`이 가득 차서 디스크 풀이 나는 사고가 가끔 있다. 1G~4G 정도가 일반적인 서버 설정이다.

## 트러블슈팅

### 첫 단계: 상태와 로그

서비스가 안 뜬다는 보고를 받으면 거의 항상 이 두 명령으로 시작한다.

```bash
systemctl status myapp -l --no-pager
journalctl -u myapp -n 200 --no-pager
```

`status`는 마지막 종료 코드, signal, 마지막 몇 줄의 로그를 한 번에 보여준다. `Active: failed (Result: exit-code)` 같은 줄과 `Process: 12345 ExecStart=... (code=exited, status=1/FAILURE)`를 같이 본다. exit code만 봐도 환경 변수 누락(127), 권한(126), 일반 에러(1) 정도는 구분이 된다.

### 부팅이 느릴 때: systemd-analyze

서버 부팅이 이상하게 느릴 때 어디가 발목 잡는지 본다.

```bash
systemd-analyze                       # 전체 부팅 시간
systemd-analyze blame                 # 가장 오래 걸린 unit 순서대로
systemd-analyze critical-chain        # 부팅 critical path
systemd-analyze critical-chain myapp.service
```

`blame`은 단순히 unit별 활성화 시간을 정렬한다. 1위라고 해서 진짜 원인이 아닐 수 있다는 점을 기억해야 한다. 어떤 unit은 다른 unit이 끝나기를 기다리느라 시간이 길게 보일 수 있기 때문이다.

`critical-chain`은 그래서 더 정확한 도구다. "default.target에 도달하기까지 시간을 잡아먹은 의존 사슬"을 보여준다. 운영에서 부팅 5분 걸리는 클러스터 노드를 디버깅한 적이 있는데, blame 1위는 mongodb였지만 critical-chain은 cloud-init이 NTP 동기화를 기다리는 게 진짜 원인이라고 가리켰다. blame만 보고 mongodb 튜닝했으면 한참 헤맸을 것이다.

```bash
systemd-analyze plot > boot.svg       # 부팅 타임라인 SVG로
```

`plot`은 모든 unit의 시작/종료를 막대 그래프로 그려서 시각화한다. 보고서 만들 때 유용하다.

### unit 파일 검증

문법 오류는 `verify`로 잡는다.

```bash
systemd-analyze verify /etc/systemd/system/myapp.service
```

`verify`는 unit 파일을 파싱해 보고, 의존하는 unit이 존재하는지, deprecated 옵션을 쓰는지, ExecStart의 절대 경로가 실제로 존재하는지 같은 걸 검사한다. 배포 자동화 파이프라인에서 unit 파일을 만들면 적어도 이걸 한 번 돌리고 넘긴다.

### 자주 만나는 증상과 원인

- `systemctl start`는 성공인데 바로 죽는다 → `Type=` 잘못 잡았을 가능성. fork하는 데몬에 `Type=simple`을 두면 자식 PID를 추적 못 해서 죽었다고 판단한다. 또는 `ExecStart`가 daemonize 옵션을 줘서 부모가 즉시 종료되는데 `Type=forking`이 아닐 때.
- `enable`했는데 부팅 시 안 뜬다 → unit에 `[Install]` 섹션이 없거나 `WantedBy=`의 target이 부팅 시 도달하지 않는 target일 수 있다. `systemctl is-enabled`가 `static`이라면 `[Install]`이 없는 것.
- 환경 변수가 안 먹힌다 → `EnvironmentFile=`의 파일 권한, BOM, CRLF를 의심한다. Windows에서 만든 파일을 그대로 올리면 `\r`이 값에 섞여서 이상하게 동작한다.
- `daemon-reload` 후에도 옛 설정이 남는다 → 서비스 자체는 `restart`해야 새 unit이 적용된다. `daemon-reload`는 systemd가 unit 파일을 다시 읽게만 한다.
- `start-limit-hit` → 위에서 다룬 burst 한도. `reset-failed` 후 재시작하거나 `StartLimitBurst`를 조정한다.

### `failed` 상태 정리

운영 중 한 번 실패한 unit은 `systemctl --failed`에 계속 잡힌다. 원인을 해결한 뒤에는 상태를 명시적으로 리셋해 줘야 깔끔하다.

```bash
systemctl --failed                   # 실패한 unit 목록
systemctl reset-failed               # 전체 리셋
systemctl reset-failed myapp         # 특정 unit
```

모니터링이 `--failed`를 카운트해서 알람 보내는 시스템이라면 이걸 안 정리하면 알람이 계속 울린다.

## 정리

옛날 SysV의 `/etc/init.d/` 스크립트와 runlevel은 systemd의 unit과 target으로 거의 1:1로 매핑된다. 다만 systemd는 의존성을 명시적으로 표현할 수 있고 병렬로 시작한다는 점에서 본질적으로 더 강력한 모델이다.

unit 파일을 작성할 때 가장 자주 실수하는 곳은 `User=` 누락, `After=`만 적고 `Requires=`/`Wants=`를 빠뜨리는 것, `Restart=always`로 의도와 다르게 살리는 것이다. drop-in으로 패키지 unit을 안전하게 수정하고, `systemctl cat`으로 합쳐진 결과를 확인하는 습관을 들이면 운영 중 사고가 많이 줄어든다.

서비스가 안 뜨면 `systemctl status` → `journalctl -u`가 처음, 부팅이 느리면 `systemd-analyze critical-chain`이 정답이다. 더 깊이 들어가는 cgroup, sd_notify, socket activation, watchdog 같은 주제는 같은 디렉토리의 `Systemd_Deep_Dive.md`에서 다룬다.
