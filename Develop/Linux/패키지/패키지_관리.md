# 패키지 관리

## 개요

apt, yum, dnf 같은 패키지 관리자는 단순히 바이너리를 푸는 도구가 아니다. 의존성 그래프를 풀고, 저장소 메타데이터를 관리하고, GPG 서명을 검증하고, 트랜잭션 단위로 롤백 가능한 상태를 유지한다. 운영 서버에서 한 번이라도 의존성 지옥(dependency hell)에 빠져본 적이 있다면 이 도구들이 얼마나 많은 일을 대신해 주는지 체감한다.

이 문서는 명령어 레퍼런스가 아니다. 실무에서 자주 마주치는 충돌 시나리오, 저장소 우선순위 설정, 캐시 손상 복구, 자체 패키지 빌드까지 한 번씩은 꼭 겪게 되는 상황을 다룬다.

## 패키지 관리자의 동작 원리

### 메타데이터와 의존성 해결

apt든 dnf든 동작 흐름은 비슷하다. 저장소(repository)의 메타데이터를 가져와 로컬에 캐시하고, 사용자가 요청한 패키지의 의존성 트리를 SAT(Boolean satisfiability) 풀이에 가까운 알고리즘으로 해결한다. 이 과정에서 충돌(conflict), 대체(replace), 제공(provide), 권장(recommend) 같은 관계를 모두 고려한다.

dnf는 libsolv라는 SAT 솔버를 사용한다. yum이 의존성 해결에 시간이 오래 걸리고 가끔 답을 못 찾던 문제를 dnf가 개선한 핵심이 이 솔버다. apt는 자체 의존성 해결기를 쓰는데, `apt-get -s install` 같은 시뮬레이션 옵션으로 실제 변경 전에 결과를 미리 볼 수 있다.

```bash
# 의존성 해결 결과만 미리 보기 (apt)
apt-get -s install nginx

# dnf의 의존성 트리 출력
dnf repoquery --requires nginx
dnf repoquery --whatrequires libssl

# 패키지가 무엇을 제공하는지
dnf repoquery --provides nginx
```

운영 환경에서 패키지를 설치하기 전에 시뮬레이션 한 번 돌려보는 습관이 있으면 사고가 줄어든다. 특히 `apt upgrade`가 갑자기 50개 패키지를 제거하겠다고 할 때 `-s`로 먼저 확인하지 않고 yes를 누르면 그날 야근 확정이다.

### 트랜잭션과 히스토리

dnf와 yum은 모든 작업을 트랜잭션으로 기록한다. 잘못 설치했다면 그 트랜잭션만 롤백할 수 있다.

```bash
# 트랜잭션 히스토리
dnf history list
dnf history info 42

# 특정 트랜잭션 롤백
dnf history undo 42
dnf history rollback 40   # 40번 시점 상태로
```

apt는 기본적으로 트랜잭션 단위 롤백이 약하다. `/var/log/apt/history.log`에 기록은 남지만 이걸로 자동 롤백은 안 된다. 운영 서버에서는 변경 전 상태를 별도로 백업해두거나 이미지 스냅샷으로 대비하는 편이 낫다.

## 의존성 충돌 실전 사례

### httpd-tools 버전 미스매치

CentOS 7에서 EPEL과 IUS, Remi 같은 서드파티 저장소를 같이 쓰면 `httpd-tools` 같은 공통 의존성에서 충돌이 자주 난다. 예를 들어 메인 저장소의 httpd 2.4.6과 IUS의 httpd24u-2.4.46이 동시에 설치 후보로 올라오면 dnf는 어느 쪽을 골라야 할지 모른다.

```text
Error: Transaction check error:
  package httpd-tools-2.4.6-99.el7.x86_64 conflicts with
  httpd24u-tools provided by httpd24u-tools-2.4.46-1.el7.ius.x86_64
```

이 상황에서 흔히 하는 실수가 `--allowerasing`을 붙여서 강제로 진행하는 것이다. 그러면 httpd-tools를 지우고 httpd24u-tools를 설치하는데, 다른 패키지가 httpd-tools에 의존하고 있으면 그것까지 줄줄이 제거된다. 의존성 그래프가 일직선이 아니라 거미줄이라 한쪽을 건드리면 어디까지 영향이 가는지 미리 보고 결정해야 한다.

올바른 접근은 저장소 우선순위를 설정하거나, 한 저장소 안에서만 패키지를 받도록 명시하는 것이다.

```bash
# IUS 저장소만 사용해서 설치
dnf install --enablerepo=ius --disablerepo=base,updates httpd24u

# 또는 충돌하는 메인 패키지를 먼저 명시적으로 제거
dnf remove httpd httpd-tools mod_ssl
dnf install --enablerepo=ius httpd24u httpd24u-tools mod24_ssl
```

### libssl 충돌

OpenSSL은 ABI(Application Binary Interface)가 메이저 버전마다 깨진다. libssl1.0.0과 libssl1.1, libssl3가 한 시스템에 공존할 수는 있지만, 특정 패키지가 libssl1.0.0에 링크되어 있고 그 라이브러리가 사라지면 그 패키지는 죽는다.

Ubuntu 20.04에서 22.04로 업그레이드할 때 libssl1.1 → libssl3 전환이 일어난다. 자체 빌드한 사내 패키지가 libssl1.1에 동적 링크되어 있으면 업그레이드 후 그 바이너리가 안 뜬다. `ldd`로 확인하면 `libssl.so.1.1 => not found`가 뜬다.

```bash
ldd /usr/local/bin/myapp | grep -i ssl
# libssl.so.1.1 => not found

# libssl1.1 패키지가 있는지 확인
apt list --installed | grep libssl
```

해결책은 두 가지다. libssl1.1을 별도 저장소에서 받아 공존시키거나, 사내 패키지를 libssl3로 재빌드한다. 운영 환경이면 후자가 정답이지만, 급할 때는 옛 버전 deb를 직접 설치해서 시간을 번다.

```bash
# 옛 libssl1.1 deb를 임시로 설치 (Ubuntu 22.04에서 20.04 패키지 가져오기)
wget http://archive.ubuntu.com/ubuntu/pool/main/o/openssl/libssl1.1_1.1.1f-1ubuntu2.19_amd64.deb
sudo dpkg -i libssl1.1_1.1.1f-1ubuntu2.19_amd64.deb
```

이건 임시방편이다. 보안 패치가 안 들어오므로 빨리 재빌드해서 정상 상태로 돌려놔야 한다.

### Held packages

Debian 계열에서 `apt upgrade`를 했는데 "The following packages have been kept back" 메시지가 뜨면 의존성 변화가 너무 커서 apt가 보수적으로 업그레이드를 보류한 상태다. 보통 새 패키지를 설치하거나 기존 패키지를 제거해야 진행되는 경우다.

```text
The following packages have been kept back:
  linux-generic linux-headers-generic linux-image-generic
```

`apt full-upgrade`(과거 dist-upgrade)를 쓰면 제거가 필요한 변경까지 진행한다. 하지만 운영 서버에서 무턱대고 full-upgrade는 위험하다. `apt-get -s full-upgrade`로 어떤 패키지가 제거될지 먼저 확인한다.

명시적으로 hold를 걸어둔 패키지도 kept back으로 나타난다. 이걸 풀려면 명시적으로 unhold를 해야 한다.

```bash
# hold 상태 확인
apt-mark showhold

# hold 해제
sudo apt-mark unhold linux-image-generic
```

## 저장소 우선순위 관리

### yum/dnf priorities

여러 저장소가 같은 패키지를 제공할 때 어디서 받을지 결정하는 메커니즘이 필요하다. yum/dnf에서는 `priorities` 플러그인 또는 저장소 설정의 `priority=` 값을 사용한다.

```bash
# priorities 플러그인 설치 (yum)
yum install yum-plugin-priorities

# dnf는 기본 내장
```

각 저장소 파일에 priority를 명시한다. 숫자가 낮을수록 우선순위가 높다(1이 최상위, 99가 최하위). 메인 저장소는 1, EPEL은 10, 서드파티 사내 저장소는 5처럼 정책을 정해둔다.

```ini
# /etc/yum.repos.d/CentOS-Base.repo
[base]
name=CentOS-$releasever - Base
mirrorlist=http://mirrorlist.centos.org/?release=$releasever&arch=$basearch&repo=os
gpgcheck=1
gpgkey=file:///etc/pki/rpm-gpg/RPM-GPG-KEY-CentOS-7
priority=1

# /etc/yum.repos.d/epel.repo
[epel]
name=Extra Packages for Enterprise Linux 7 - $basearch
baseurl=http://download.fedoraproject.org/pub/epel/7/$basearch/
gpgcheck=1
gpgkey=file:///etc/pki/rpm-gpg/RPM-GPG-KEY-EPEL-7
priority=10
```

이 설정의 효과는 같은 패키지가 base와 epel 양쪽에 있으면 항상 base에서 가져온다는 것이다. EPEL이 base보다 새 버전을 가지고 있어도 base가 우선이다. 이게 운영 환경에서 원하는 동작이다. 메인 저장소만 신뢰하고, 서드파티는 메인에 없는 패키지만 보충하는 구조다.

### apt pinning

apt에서 같은 일을 하려면 `/etc/apt/preferences.d/` 아래에 핀(pin) 설정을 둔다.

```ini
# /etc/apt/preferences.d/01-main-priority
Package: *
Pin: release o=Ubuntu
Pin-Priority: 900

# /etc/apt/preferences.d/02-thirdparty-priority
Package: *
Pin: origin "repo.example.com"
Pin-Priority: 500
```

apt의 Pin-Priority는 숫자 의미가 yum과 반대다. 숫자가 클수록 우선순위가 높다. 1000을 넘으면 다운그레이드까지 허용한다. 보통 900-990을 메인, 500을 서드파티에 둔다.

특정 패키지만 다른 저장소에서 받고 싶을 때도 핀을 쓴다. 예를 들어 backports에서 nginx만 받고 나머지는 stable에서 받는 식이다.

```ini
# /etc/apt/preferences.d/nginx-from-backports
Package: nginx nginx-*
Pin: release a=focal-backports
Pin-Priority: 600

Package: *
Pin: release a=focal-backports
Pin-Priority: 100
```

핀 설정이 의도대로 동작하는지 확인할 때는 `apt-cache policy`를 본다.

```bash
apt-cache policy nginx
# nginx:
#   Installed: 1.18.0-0ubuntu1.4
#   Candidate: 1.20.1-1~focal
#   Version table:
#      1.20.1-1~focal 600
#         500 http://archive.ubuntu.com/ubuntu focal-backports/main amd64 Packages
#      1.18.0-0ubuntu1.4 500
#         500 http://archive.ubuntu.com/ubuntu focal-updates/main amd64 Packages
```

Candidate가 무엇으로 잡혀있는지 확인하면 핀이 제대로 먹었는지 알 수 있다.

## dnf module stream

RHEL 8/CentOS 8에서 도입된 모듈러리티(modularity)는 같은 패키지의 여러 메이저 버전을 골라 쓸 수 있게 해주는 메커니즘이다. PostgreSQL을 예로 들면 같은 RHEL 8 위에서 PostgreSQL 10, 12, 13을 모듈 스트림으로 골라 설치할 수 있다.

```bash
# 사용 가능한 모듈 목록
dnf module list
dnf module list postgresql

# 특정 스트림 활성화
dnf module enable postgresql:13

# 활성 스트림에서 설치
dnf module install postgresql:13/server

# 스트림 전환
dnf module reset postgresql
dnf module enable postgresql:15
```

실무에서 골치 아픈 건 한 번 활성화한 스트림을 바꿀 때다. PostgreSQL 10 스트림으로 설치한 상태에서 13으로 올리려면 단순히 enable만 바꿔서는 안 된다. 데이터 디렉토리는 그대로 두고 패키지만 갈아끼우면 PostgreSQL 13이 10의 데이터 형식을 못 읽는다.

```bash
# 모듈 전환 시 일반적인 흐름
systemctl stop postgresql
pg_dumpall > /backup/dump.sql

dnf module reset postgresql
dnf module install postgresql:13/server

postgresql-setup --initdb
systemctl start postgresql
psql < /backup/dump.sql
```

모듈 스트림은 RHEL 9에서 점차 줄어드는 추세다. RHEL 9에서는 AppStream 저장소에 직접 여러 버전을 넣는 방식으로 바뀌고 있다. 그래도 RHEL 8을 운영하는 동안에는 모듈 개념을 알아둬야 한다.

## apt-key deprecation 대응

Ubuntu 22.04와 Debian 11부터 `apt-key`는 deprecated다. 22.04에서는 경고만 뜨지만 24.04부터는 동작하지 않는다. 새 방식은 GPG 키를 `/etc/apt/keyrings/` 또는 `/etc/apt/trusted.gpg.d/`에 직접 두고 sources.list 항목에서 `signed-by=`로 명시하는 것이다.

옛 방식과 새 방식의 차이는 신뢰 범위에 있다. 옛 방식은 한 번 등록한 키가 모든 저장소를 서명할 수 있는 권한을 가졌다. 그래서 어떤 서드파티 키가 시스템에 들어오면 그 키 보유자가 우분투 메인 저장소까지 위장할 수 있는 위험이 있었다. 새 방식은 키와 저장소를 1:1로 묶어서 그 키는 그 저장소의 패키지만 검증할 수 있다.

새 방식 설정 흐름이다.

```bash
# 1. keyrings 디렉토리 준비
sudo install -d -m 0755 /etc/apt/keyrings

# 2. GPG 키를 받아서 dearmor 후 저장
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod 0644 /etc/apt/keyrings/docker.gpg

# 3. sources.list에 signed-by 명시
echo "deb [arch=amd64 signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu jammy stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list

# 4. 메타데이터 업데이트
sudo apt update
```

기존에 `apt-key add`로 등록했던 키들을 정리하려면 다음을 본다.

```bash
# 등록된 키 목록
apt-key list
# Warning: apt-key is deprecated...

# 옛 키 제거
sudo apt-key del <key_id>
```

`/etc/apt/trusted.gpg`와 `/etc/apt/trusted.gpg.d/` 둘 다 옛 시스템 위치다. 새로 추가하는 건 `/etc/apt/keyrings/`로 가고, sources에서 명시적으로 참조하는 게 권장된다.

## 패키지 holding과 versionlock

운영 서버에서 특정 패키지를 절대 자동 업그레이드하지 않게 막아야 할 때가 있다. PostgreSQL이나 커널처럼 업그레이드 후 재시작이나 데이터 마이그레이션이 필요한 패키지는 변경 시점을 사람이 통제해야 한다.

### dnf versionlock

```bash
# 플러그인 설치
dnf install python3-dnf-plugin-versionlock

# 패키지 잠금
dnf versionlock add postgresql-server
dnf versionlock add kernel-3.10.0-1160.el7

# 잠금 목록
dnf versionlock list

# 잠금 해제
dnf versionlock delete postgresql-server
dnf versionlock clear   # 전체 해제
```

versionlock은 `/etc/dnf/plugins/versionlock.list`에 NEVRA(Name-Epoch-Version-Release-Architecture) 형식으로 저장된다. 이 파일을 직접 편집해도 된다. 형상관리 도구(Ansible, Salt)로 일관되게 관리할 때 유용하다.

```text
# /etc/dnf/plugins/versionlock.list
postgresql-server-13.4-1.el8.x86_64
kernel-4.18.0-348.el8.x86_64
```

### apt-mark hold

apt 쪽은 `apt-mark hold`로 같은 일을 한다.

```bash
sudo apt-mark hold postgresql-13
sudo apt-mark hold linux-image-generic linux-headers-generic

# hold 목록
apt-mark showhold

# hold 해제
sudo apt-mark unhold postgresql-13
```

hold가 걸린 패키지는 `apt upgrade` 시 kept back으로 나타난다. dpkg 레벨에서는 selection 상태가 `hold`로 바뀐다.

```bash
dpkg --get-selections | grep hold
# postgresql-13                                    hold
```

hold만 걸어두면 끝이 아니다. CI/CD 파이프라인의 `apt-get install -y` 같은 자동화 스크립트가 의존성 변경을 요구하면 hold가 걸린 패키지 때문에 install 자체가 실패할 수 있다. 이런 상황을 어떻게 처리할지 운영 정책으로 미리 정해둬야 한다.

## 캐시 손상 복구

### dpkg 중단 후 복구

apt 작업 중에 시스템이 갑자기 꺼지거나, 디스크가 가득 차서 dpkg가 중간에 멈추면 패키지 데이터베이스가 일관성 없는 상태로 남는다. 다음 apt 명령에서 이런 메시지가 뜬다.

```text
E: dpkg was interrupted, you must manually run 'sudo dpkg --configure -a'
to correct the problem.
```

지시대로 하면 된다.

```bash
sudo dpkg --configure -a
```

이 명령은 모든 미완료 패키지의 설정 단계를 다시 실행한다. 보통 여기서 끝나지만, 의존성 단계에서 막히면 추가 작업이 필요하다.

```bash
# 의존성 깨진 패키지 강제 수정
sudo apt --fix-broken install

# 그래도 안 되면 문제 패키지를 강제로 재설치
sudo apt install --reinstall <패키지>

# 최후의 수단: 강제 제거
sudo dpkg --remove --force-remove-reinstreq <패키지>
```

`--force-*` 옵션은 dpkg의 안전장치를 끄는 거라 신중해야 한다. 시스템 핵심 패키지에 함부로 쓰면 부팅 안 되는 상태로 빠질 수 있다. 가능하면 그 전에 `dpkg --configure -a`와 `apt --fix-broken install`로 해결한다.

### rpmdb 손상 복구

RPM 데이터베이스(`/var/lib/rpm/`)도 비정상 종료로 손상될 수 있다. 증상은 `rpm -qa`가 멈추거나, dnf가 "rpmdb open failed" 같은 메시지를 뱉는다.

```bash
# rpmdb 잠금 파일 정리
rm -f /var/lib/rpm/__db*

# 재빌드
rpm --rebuilddb
```

`rebuilddb`는 기존 데이터를 다시 인덱싱하는 작업이다. 데이터 자체가 손상된 게 아니라 인덱스가 깨진 정도면 이걸로 복구된다.

데이터 자체가 깨졌다면 백업에서 복원하거나 시스템 재설치가 필요하다. 그래서 운영 서버에서는 `/var/lib/rpm/`을 정기적으로 백업해두는 편이 좋다.

```bash
# 백업
tar czf /backup/rpmdb-$(date +%F).tar.gz /var/lib/rpm/

# 복원
systemctl stop dnf-makecache.timer
tar xzf /backup/rpmdb-2026-04-01.tar.gz -C /
rpm --rebuilddb
```

## 위험 옵션 사용 시 주의점

### --skip-broken

`dnf install --skip-broken` 또는 `yum install --skip-broken`은 의존성을 풀 수 없는 패키지를 건너뛰고 나머지만 설치한다. 편해 보이지만 위험하다. 어떤 패키지가 왜 못 풀리는지 알지 못한 채 일부만 설치되면 시스템 상태가 불완전해진다.

```bash
# 위험: 무엇이 빠졌는지 모르고 진행
dnf install --skip-broken httpd php mysql

# 권장: 시뮬레이션 먼저
dnf install --downloadonly httpd php mysql
# 또는 의존성 출력
dnf repoquery --requires httpd
```

`--skip-broken`은 일회성 시스템(컨테이너 빌드 같은)에서 빠르게 진행할 때만 쓰고, 운영 서버에서는 의존성 충돌의 원인을 먼저 찾는다.

### --allowerasing

dnf의 `--allowerasing`은 충돌하는 기존 패키지를 자동으로 지우고 새 패키지를 설치한다. 이름 그대로 "지우는 걸 허용"한다는 옵션이다.

```bash
dnf install --allowerasing httpd24u
# Removed:
#   httpd-2.4.6-99.el7.x86_64
#   httpd-tools-2.4.6-99.el7.x86_64
#   mod_ssl-2.4.6-99.el7.x86_64
```

여기서 함정은 제거되는 게 충돌 패키지뿐만 아니라 그것에 의존하는 모든 패키지라는 점이다. 메일 서버, 웹 애플리케이션 같은 게 줄줄이 사라질 수 있다.

```bash
# 무엇이 지워질지 미리 보기
dnf install --allowerasing --assumeno httpd24u
# (실제 변경 없이 어떤 패키지가 영향받는지 출력)
```

운영 서버에서는 `--allowerasing` 대신 충돌 패키지를 명시적으로 제거하고 새로 설치하는 흐름이 안전하다. 그래야 무엇이 사라지는지 한 번 더 확인하게 된다.

### apt --force-yes / dpkg --force-*

apt의 `--force-yes`(현재는 `--allow-downgrades`, `--allow-remove-essential`, `--allow-change-held-packages` 등으로 분리됨)와 dpkg의 `--force-*` 옵션도 같은 류다. 시스템이 보내는 안전 신호를 끄는 옵션이다.

```bash
# essential 패키지 제거 시도 (위험)
sudo apt remove --allow-remove-essential bash
# 절대 하지 말 것. bash 없이는 시스템이 거의 못 돈다.

# hold 무시하고 변경
sudo apt install --allow-change-held-packages postgresql-13
```

이런 옵션은 사람이 그 결과를 명확히 이해하고 책임질 수 있을 때만 쓴다.

## 자체 패키지 빌드

### RPM 빌드

사내에서 만든 바이너리나 설정을 패키지로 배포하려면 직접 RPM/DEB를 빌드하게 된다. 그냥 tar.gz로 배포하지 않는 이유는 의존성 관리, 버전 추적, 롤백 가능성 때문이다.

RPM 빌드의 기본 흐름은 다음과 같다.

```bash
# 빌드 도구 설치
dnf install rpm-build rpmdevtools

# 빌드 트리 생성
rpmdev-setuptree
# ~/rpmbuild/{BUILD,RPMS,SOURCES,SPECS,SRPMS} 생성

# spec 파일 작성
vi ~/rpmbuild/SPECS/myapp.spec

# 소스 tarball을 SOURCES에 두고 빌드
cp myapp-1.0.0.tar.gz ~/rpmbuild/SOURCES/
rpmbuild -bb ~/rpmbuild/SPECS/myapp.spec
# RPMS/x86_64/myapp-1.0.0-1.el8.x86_64.rpm 생성
```

spec 파일의 핵심 섹션이다.

```spec
Name:           myapp
Version:        1.0.0
Release:        1%{?dist}
Summary:        My internal application

License:        Proprietary
URL:            https://example.com/myapp
Source0:        %{name}-%{version}.tar.gz

BuildRequires:  gcc make
Requires:       libssl >= 1.1, jq

%description
사내 데이터 처리 도구.

%prep
%setup -q

%build
make

%install
rm -rf %{buildroot}
make install DESTDIR=%{buildroot}

%files
%{_bindir}/myapp
%{_sysconfdir}/myapp/config.yaml

%changelog
* Mon Apr 28 2026 김개발 <dev@example.com> - 1.0.0-1
- 초기 패키징
- TLS 1.3 지원 추가
```

### changelog 작성 시 주의점

`%changelog`는 단순한 메모가 아니다. RPM의 일부로 패키지에 박힌다. 사용자는 `rpm -q --changelog myapp`으로 이걸 본다. 운영 환경에서 어떤 버전이 무엇을 바꿨는지 추적하는 1차 자료다.

규칙이 몇 개 있다.

날짜 형식은 정확히 `* 요일 월 일 연도 이름 <이메일> - 버전-릴리스` 패턴이다. 한 칸이라도 틀리면 rpmlint가 경고를 뱉고, 일부 빌드 시스템에서는 실패한다.

```text
* Mon Apr 28 2026 김개발 <dev@example.com> - 1.0.0-1
- 변경 내용 1
- 변경 내용 2
```

최신 변경이 위로 가야 한다. 파일 맨 위가 가장 최근 항목이다. 시간순이 아니라 역순이라는 점을 헷갈리지 않는다.

같은 날짜에 여러 릴리스가 있어도 변경 항목을 합치지 말고 별도 항목으로 둔다. 패키지 사용자가 어떤 변경이 어느 릴리스에 들어갔는지 정확히 추적할 수 있어야 한다.

릴리스 번호는 같은 버전 안에서 패키징을 다시 했을 때 올린다. 코드는 그대로인데 spec만 고쳤거나 의존성이 바뀐 경우다. 코드가 바뀌면 Version 자체를 올린다.

### DEB 빌드

Debian 계열의 패키지 빌드는 `debuild` 또는 `dpkg-buildpackage`로 한다.

```bash
# 빌드 도구
sudo apt install build-essential devscripts debhelper

# 작업 디렉토리에 debian/ 디렉토리 생성
cd myapp-1.0.0/
dh_make --createorig -s -y -e dev@example.com

# debian/ 안의 파일들을 편집
# - control: 패키지 메타데이터, 의존성
# - rules: 빌드 규칙
# - changelog: 변경 이력 (형식 엄격)
# - install: 설치할 파일 목록

# 빌드
debuild -us -uc
# 또는
dpkg-buildpackage -us -uc -b
```

`debian/changelog`는 RPM의 `%changelog`와 비슷하지만 형식이 더 엄격하다. `dch`(Debian changelog) 명령으로 안전하게 편집한다.

```text
myapp (1.0.0-1) unstable; urgency=medium

  * 초기 패키징
  * TLS 1.3 지원 추가

 -- 김개발 <dev@example.com>  Mon, 28 Apr 2026 10:00:00 +0900
```

서명자 줄 앞의 공백이 정확히 한 칸, 마지막 줄의 날짜는 RFC 2822 형식이어야 한다. `dch -i`(새 버전 항목 추가)나 `dch -e`(현재 항목 편집)를 쓰면 형식 오류를 피할 수 있다.

### 사내 저장소

빌드한 패키지는 보통 사내 저장소에 올려서 배포한다. RPM은 `createrepo_c`, DEB는 `reprepro` 같은 도구로 저장소를 만든다.

```bash
# RPM 저장소 생성
mkdir -p /var/www/repo/x86_64
cp *.rpm /var/www/repo/x86_64/
createrepo_c /var/www/repo/x86_64/

# 메타데이터 갱신 (패키지 추가 후마다)
createrepo_c --update /var/www/repo/x86_64/

# DEB 저장소 (reprepro)
reprepro -b /var/www/debrepo includedeb focal myapp_1.0.0-1_amd64.deb
```

저장소 자체에 GPG 서명을 거는 것이 보안상 권장된다. 클라이언트 쪽에서 키를 등록하지 않으면 설치가 거부되도록 `gpgcheck=1`을 강제하는 게 운영 정책으로 좋다.

## 의존성 추적과 디버깅

### 누가 이 라이브러리를 요구하는가

특정 라이브러리를 업그레이드하거나 제거하기 전에 어떤 패키지가 그것에 의존하는지 알아야 한다.

```bash
# RHEL/CentOS
dnf repoquery --whatrequires libssl
rpm -q --whatrequires openssl-libs

# Debian/Ubuntu
apt-cache rdepends libssl1.1
apt rdepends --installed libssl1.1
```

운영 서버에서 `apt remove libssl1.1`을 했더니 ssh 클라이언트, sudo, systemd까지 같이 지워지려 해서 식겁한 경험이 있다면 사전 조회의 중요성을 안다.

### 파일 단위 추적

특정 파일이 어느 패키지에서 왔는지, 그 파일이 변경되었는지 확인할 수 있다.

```bash
# 파일이 속한 패키지
rpm -qf /etc/nginx/nginx.conf
dpkg -S /etc/nginx/nginx.conf

# 패키지의 파일 변경 여부 (RPM)
rpm -V nginx
# S.5....T.  c /etc/nginx/nginx.conf
# S=size, 5=md5, T=mtime 변경됨, c=설정 파일

# 변경된 모든 파일 검사
rpm -Va | grep -v '^.\{8\}c'
```

`rpm -V`는 침해 사고 대응 시 시스템 파일이 변조되었는지 확인하는 1차 도구로도 쓴다. 정상적인 변경(설정 파일 등)과 비정상적인 변경(바이너리 변조)을 구분해서 본다.

## 마무리

패키지 관리자는 평소엔 잘 동작한다. 문제가 터지는 건 의존성이 꼬였을 때, 저장소 우선순위가 의도와 다를 때, 캐시가 손상됐을 때, 자체 빌드한 패키지의 changelog 형식이 틀렸을 때다. 이런 상황을 한두 번 겪고 나면 `dnf install -y`를 무심코 누르기 전에 한 번 멈춰서 시뮬레이션을 돌려보는 습관이 생긴다.

운영 환경의 패키지 관리에서 중요한 건 변경의 추적 가능성과 롤백 가능성이다. 어떤 패키지가 언제 누구에 의해 어떤 이유로 설치/제거됐는지, 문제가 생겼을 때 직전 상태로 돌아갈 수 있는지가 핵심이다. dnf history, /var/log/apt/history.log, 사내 저장소의 빌드 기록, 형상관리 도구의 변경 이력이 같이 맞물려 돌아갈 때 운영이 안정된다.
