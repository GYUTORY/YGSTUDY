# 패키지 관리 심화

## 개요

기본 명령어를 넘어서 의존성이 꼬였을 때, 자체 저장소를 운영해야 할 때, 패키지를 직접 빌드해서 배포해야 할 때 필요한 내부 동작을 정리한다. RHEL 8 이후 DNF가 libsolv를 쓰면서 의존성 해결 방식이 완전히 바뀌었고, APT도 11.x 이후 solver3 옵션이 들어오면서 옛날 방식만 알고 있으면 디버깅이 안 되는 경우가 많다.

5년쯤 패키지 관리하다 보면 결국 부딪히는 문제가 몇 가지 있다. "왜 이 패키지가 다른 패키지를 끌고 오는지", "왜 충돌이 발생하는지", "왜 같은 저장소를 추가했는데 우선순위가 다르게 동작하는지". 이런 건 명령어 옵션을 외운다고 풀리지 않고, 메타데이터 구조와 솔버 동작을 이해해야 답이 나온다.

## APT, YUM, DNF 내부 동작 차이

### YUM의 한계

YUM은 Python으로 작성된 깊이우선 탐색(DFS) 기반 depsolver를 사용했다. 패키지 A가 B를 요구하면 B를 후보로 잡고, B가 또 C를 요구하면 C를 잡는 식으로 트리를 따라 내려간다. 문제는 다음 두 가지다.

첫째, 백트래킹이 제한적이다. 어떤 가지에서 충돌이 생겨도 위로 올라가서 다른 후보를 시도하는 능력이 약하다. 그래서 "could not find any package to install"이라는 모호한 에러가 자주 떴다.

둘째, 순서에 따라 결과가 달라진다. 같은 트랜잭션이라도 패키지를 어떤 순서로 처리하느냐에 따라 다른 해결책이 나오거나 실패한다. CentOS 7 시절 `yum update`가 한 번 실패하고 다시 돌리면 성공하는 현상이 흔했던 이유다.

```bash
# YUM 시절 흔하던 에러 패턴
yum install httpd
# Error: Package: httpd-2.4.6-97.el7.x86_64 (base)
#        Requires: httpd-tools = 2.4.6-97.el7
#        Installed: httpd-tools-2.4.6-95.el7.x86_64
#        Available: httpd-tools-2.4.6-97.el7.x86_64
```

### DNF와 libsolv

DNF는 SUSE에서 만든 libsolv를 솔버로 가져왔다. libsolv는 패키지 의존성 문제를 SAT(Boolean Satisfiability) 문제로 변환해서 푼다. 각 패키지의 설치 여부를 변수로 두고, 의존성과 충돌을 절(clause)로 표현한 다음 SAT solver에 넘긴다.

이 방식의 장점은 명확하다. 해결책이 존재하면 반드시 찾고, 존재하지 않으면 어떤 제약 조건 때문에 실패했는지 정확히 알려준다. `dnf install`이 실패할 때 "best update candidate" 같은 메시지와 함께 충돌 그래프를 보여주는 게 그 결과물이다.

```bash
dnf install some-package
# Problem: package some-package-1.0-1.el8.x86_64 requires libfoo.so.2()(64bit), 
# but none of the providers can be installed
#  - cannot install both libfoo-2.0-1.el8.x86_64 and libfoo-1.5-2.el8.x86_64
#  - package libbar-3.0 requires libfoo-1.5
```

libsolv의 동작을 더 자세히 보고 싶으면 `--debugsolver` 옵션을 쓴다. `/var/cache/dnf/debugdata/`에 솔버 입력과 결정 과정이 떨어진다.

```bash
dnf --debugsolver install httpd
ls /var/cache/dnf/debugdata/
# solver.result  solver.solv  solver.last
```

### APT의 pkgProblemResolver

APT는 자체 구현한 pkgProblemResolver를 쓴다. 휴리스틱 기반이고, 점수(score)를 매겨서 패키지를 유지할지 제거할지 결정한다. 핵심 로직은 `apt-pkg/algorithms.cc`의 `Resolve()` 메서드에 있다.

점수 계산은 대략 이런 식이다. essential 패키지는 +5000점, priority가 required면 +4000, important면 +2000. 이미 설치된 패키지는 가산점을 받고, 의존성을 깨면 감점된다. 점수가 낮은 패키지가 제거 대상이 된다.

이 방식은 빠르지만 최적해를 보장하지 않는다. 그래서 가끔 `apt`가 "이 의존성을 풀려면 이런 시스템 패키지를 제거해야 한다"는 무서운 제안을 하는 경우가 있다. Ubuntu 22.04 이후로는 APT 2.5+에서 solver3라는 옵션이 들어왔는데, 이것도 SAT 기반이다.

```bash
# APT가 어떻게 의존성을 풀려고 하는지 보기
apt -o Debug::pkgProblemResolver=true install package-name 2>&1 | head -50

# solver3 사용 (Debian 12+)
apt --solver 3 install package-name
```

### 솔버 정책 차이

DNF는 기본적으로 "best" 정책을 쓴다. 가능한 한 최신 버전을 설치하려 하고, 안 되면 실패한다. APT는 "안 되면 다른 거라도 설치"하는 경향이 강하다. 이 차이가 운영에 미치는 영향이 크다.

```bash
# DNF: best 정책 끄기 (구버전이라도 설치)
dnf install --nobest package-name

# DNF: 의존성 충돌 시 약화된 의존성도 무시
dnf install --skip-broken package-name

# APT: 의존성 무시 (위험)
apt install --no-install-recommends package-name
dpkg -i --force-depends package.deb
```

`--force-depends` 같은 건 정말 마지막 수단이다. 의존성이 깨진 상태로 설치되고 나중에 다른 트랜잭션이 다 망가진다.

## 의존성 해결 메커니즘

### Provides, Requires, Conflicts, Obsoletes

RPM과 DEB 모두 비슷한 메타데이터를 가진다. 이름이 조금 다를 뿐이다.

| RPM | DEB | 의미 |
|-----|-----|------|
| Requires | Depends | 반드시 필요 |
| Recommends | Recommends | 권장 (기본 설치) |
| Suggests | Suggests | 제안 (수동 설치) |
| Provides | Provides | 가상 패키지 제공 |
| Conflicts | Conflicts | 동시 설치 불가 |
| Obsoletes | Replaces/Breaks | 대체/충돌 |

Provides가 중요하다. `httpd`라는 이름의 패키지가 `webserver`를 Provides하면, 다른 패키지가 `Requires: webserver`라고 쓸 때 httpd가 후보가 된다. nginx도 `Provides: webserver`라고 선언하면 둘 중 하나만 있으면 된다.

실제 RPM에서 보면 이렇다.

```bash
rpm -q --provides httpd
# httpd = 2.4.37-43.module+el8.6.0+13889+f87e1bd9
# httpd-mmn = 20120211x8664
# webserver

rpm -q --requires httpd | head -5
# /bin/sh
# /etc/mime.types
# httpd-filesystem
# httpd-tools = 2.4.37-43.module+el8.6.0+13889+f87e1bd9
# system-logos-httpd
```

DEB에서도 비슷하다.

```bash
apt-cache show nginx | grep -E '^(Depends|Provides|Conflicts):'
# Depends: nginx-core (<< 1.18.0-7~) | nginx-full (<< 1.18.0-7~) | ...
# Provides: httpd, httpd-cgi
# Conflicts: nginx-common (<< 1.18.0)
```

### 버전 비교 규칙

여기가 함정이 많다. RPM과 DEB의 버전 비교 규칙이 다르고, 둘 다 직관과 어긋나는 부분이 있다.

RPM 버전 형식: `[epoch:]version-release[.dist]`. 비교는 epoch → version → release 순서로 한다. epoch가 없으면 0으로 친다.

```bash
# RPM 버전 비교 직접 해보기
rpmdev-vercmp 1.0-1.el8 1.0-2.el8
# 1.0-1.el8 < 1.0-2.el8

rpmdev-vercmp 1.10 1.9
# 1.10 > 1.9  (숫자로 비교)

rpmdev-vercmp 1.0~rc1 1.0
# 1.0~rc1 < 1.0  (~ 는 pre-release)
```

특히 `~`(틸드)가 중요하다. `1.0~rc1`은 `1.0`보다 작다. 알파/베타/RC 버전을 표현할 때 쓴다. 옛날에는 `1.0.rc1`로 쓰면 `1.0`보다 크게 비교돼서 문제가 됐다.

DEB도 비슷하지만 더 복잡하다. 버전 형식은 `[epoch:]upstream_version[-debian_revision]`. dpkg의 비교 알고리즘은 숫자 부분과 비숫자 부분을 번갈아 비교한다.

```bash
dpkg --compare-versions "1.0-1" lt "1.0-2" && echo "true"
# true

dpkg --compare-versions "1.0~rc1" lt "1.0" && echo "true"
# true

# epoch는 모든 것을 이긴다
dpkg --compare-versions "1:0.1" gt "999" && echo "true"
# true
```

epoch는 패키지 버전 체계가 바뀌었을 때(예: 날짜 기반 → 시맨틱 버전) 강제로 새 버전을 더 크게 만들 때 쓴다. 한 번 epoch를 올리면 되돌리기 어려우니 신중해야 한다.

### 충돌 해결

충돌은 보통 세 가지 경로로 풀린다.

첫째, Obsoletes로 자동 해결. RHEL 7 → 8 업그레이드 때 `mariadb-libs`가 `mysql-libs`를 Obsoletes로 가지고 있어서 자동으로 교체된다.

둘째, alternatives 시스템. `/usr/bin/java`처럼 여러 패키지가 같은 파일을 제공할 때, 심볼릭 링크로 우회한다.

```bash
update-alternatives --list java
# /usr/lib/jvm/java-11-openjdk-amd64/bin/java
# /usr/lib/jvm/java-17-openjdk-amd64/bin/java

update-alternatives --config java
```

셋째, 수동 개입. 충돌이 풀리지 않으면 사용자가 한쪽을 명시적으로 제거해야 한다. `dnf swap`이나 `apt install A- B+`(A 제거하고 B 설치) 같은 문법을 쓴다.

```bash
dnf swap firewalld iptables-services
apt install nginx- apache2+
```

## 저장소 메타데이터 구조

### YUM/DNF 저장소 (repomd)

`createrepo_c`로 만드는 RPM 저장소는 `repodata/` 디렉토리 아래에 메타데이터를 둔다.

```
repo/
├── Packages/
│   ├── httpd-2.4.37-43.el8.x86_64.rpm
│   └── ...
└── repodata/
    ├── repomd.xml
    ├── repomd.xml.asc           # GPG 서명
    ├── <hash>-primary.xml.gz    # 패키지 메타데이터
    ├── <hash>-filelists.xml.gz  # 파일 목록
    ├── <hash>-other.xml.gz      # 변경 로그
    └── <hash>-comps.xml         # 그룹 정보 (optional)
```

`repomd.xml`이 진입점이다. 다른 메타데이터 파일들의 경로, 체크섬, 타임스탬프가 들어 있다. DNF는 먼저 이걸 받아서 캐시된 것과 비교하고, 바뀐 파일만 다시 받는다.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<repomd xmlns="http://linux.duke.edu/metadata/repo">
  <revision>1697123456</revision>
  <data type="primary">
    <checksum type="sha256">abc...</checksum>
    <location href="repodata/abc-primary.xml.gz"/>
    <timestamp>1697123456</timestamp>
    <size>123456</size>
  </data>
  <data type="filelists">...</data>
  <data type="other">...</data>
</repomd>
```

`primary.xml.gz`에 패키지 이름, 버전, Provides, Requires, Conflicts가 다 들어 있다. SAT solver는 이걸 입력으로 받는다. `filelists.xml.gz`는 패키지 안의 모든 파일 경로가 들어 있는데, 크기가 커서 기본적으로 안 받는다. `dnf provides /path/to/file` 같은 명령을 쓸 때만 받아온다.

DNF 4부터는 `repomd.xml.zck`(zchunk 압축) 형식도 지원한다. 메타데이터의 변경된 부분만 받을 수 있어서 대역폭을 아낀다.

### APT 저장소

APT 저장소는 `dists/<distribution>/` 아래에 트리를 만든다. 구조가 좀 더 복잡하다.

```
repo/
├── pool/
│   └── main/n/nginx/nginx_1.18.0-1_amd64.deb
└── dists/
    └── stable/
        ├── Release           # 메타데이터 인덱스
        ├── Release.gpg       # 분리된 GPG 서명
        ├── InRelease         # 인라인 GPG 서명된 Release
        └── main/
            ├── binary-amd64/
            │   ├── Packages.gz   # 패키지 목록
            │   └── Release       # 컴포넌트 메타데이터
            └── source/
                └── Sources.gz
```

`Release` 파일이 진입점이다. 컴포넌트(main, contrib, non-free), 아키텍처(amd64, arm64), 각 `Packages` 파일의 체크섬이 들어 있다.

```
Origin: MyRepo
Label: MyRepo
Suite: stable
Codename: bookworm
Date: Thu, 17 Apr 2026 10:00:00 UTC
Architectures: amd64 arm64
Components: main contrib
SHA256:
 abc123...  1234 main/binary-amd64/Packages
 def456...  2345 main/binary-amd64/Packages.gz
```

`InRelease`는 같은 내용을 GPG로 인라인 서명한 파일이다. APT 1.1부터는 이걸 우선적으로 받는다. `Release` + `Release.gpg` 조합은 두 번 다운로드해야 해서 효율이 떨어진다.

`Packages.gz`에는 각 패키지의 메타데이터(이름, 버전, Depends, Description, Filename 등)가 들어 있다. APT는 이걸 받아서 캐시 DB(`/var/lib/apt/lists/`)에 저장한다.

```bash
ls /var/lib/apt/lists/
# deb.debian.org_debian_dists_bookworm_InRelease
# deb.debian.org_debian_dists_bookworm_main_binary-amd64_Packages.lz4
```

## 자체 저장소 구성

### YUM/DNF 저장소: createrepo_c

`createrepo_c`는 C로 다시 작성된 createrepo다. Python 버전보다 10배 이상 빠르다. RPM 파일들을 디렉토리에 넣고 명령 한 번이면 메타데이터가 생성된다.

```bash
# 패키지 디렉토리 준비
mkdir -p /srv/repo/Packages
cp *.rpm /srv/repo/Packages/

# 메타데이터 생성
createrepo_c /srv/repo

# 패키지 추가/제거 후 업데이트 (전체 재생성보다 빠름)
createrepo_c --update /srv/repo
```

`--workers N` 옵션으로 병렬 처리할 수 있다. 패키지가 수천 개면 차이가 크다. `--deltas`를 쓰면 deltarpm도 생성하는데, 요즘은 거의 안 쓴다.

저장소를 nginx로 서빙할 때는 디렉토리 인덱싱을 켜야 한다.

```nginx
location /repo {
    root /srv;
    autoindex on;
    types {
        application/x-rpm rpm;
    }
}
```

클라이언트에서는 `/etc/yum.repos.d/myrepo.repo` 파일을 만든다.

```ini
[myrepo]
name=My Repository
baseurl=https://repo.example.com/repo
enabled=1
gpgcheck=1
gpgkey=https://repo.example.com/RPM-GPG-KEY-myrepo
priority=10
```

여러 호스트로 분산하려면 `mirrorlist`를 쓴다. 미러 URL 목록이 담긴 파일을 가리키면 DNF가 알아서 라운드로빈한다.

### APT 저장소: reprepro vs aptly

reprepro는 가벼운 도구다. 단일 distribution을 관리하는 데 적합하다. 설정이 직관적이고, 디렉토리 구조도 표준적이다.

```bash
mkdir -p /srv/apt/{conf,db,dists,pool}

cat > /srv/apt/conf/distributions <<EOF
Origin: MyRepo
Label: MyRepo
Codename: bookworm
Architectures: amd64 arm64 source
Components: main
Description: My APT repository
SignWith: ABCD1234
EOF

# 패키지 추가
reprepro -b /srv/apt includedeb bookworm package_1.0-1_amd64.deb

# 패키지 제거
reprepro -b /srv/apt remove bookworm package
```

aptly는 더 강력하다. snapshot, mirror, publish 같은 개념이 있어서 운영 환경에서 쓸 만하다. 특정 시점의 저장소 상태를 snapshot으로 잡아두고, 문제가 생기면 이전 snapshot으로 publish를 바꿔치기할 수 있다.

```bash
# upstream 미러
aptly mirror create -architectures=amd64 debian-stable \
    https://deb.debian.org/debian bookworm main

aptly mirror update debian-stable

# snapshot 생성
aptly snapshot create stable-2026-04-17 from mirror debian-stable

# 자체 저장소로 publish
aptly publish snapshot stable-2026-04-17

# 다음 달에 업데이트
aptly mirror update debian-stable
aptly snapshot create stable-2026-05-17 from mirror debian-stable
aptly publish switch bookworm stable-2026-05-17
```

운영 환경에서 패키지 회귀가 발생했을 때 `aptly publish switch`로 즉시 롤백할 수 있다는 게 큰 장점이다. reprepro는 이게 안 돼서 백업에서 복구해야 한다.

## RPM 패키지 빌드

### spec 파일 구조

RPM 빌드의 핵심은 spec 파일이다. 패키지의 모든 정보가 여기에 들어간다.

```spec
Name:           myapp
Version:        1.0.0
Release:        1%{?dist}
Summary:        My application

License:        MIT
URL:            https://example.com/myapp
Source0:        %{name}-%{version}.tar.gz

BuildRequires:  gcc, make, cmake >= 3.10
Requires:       systemd, openssl-libs >= 1.1.1

%description
Long description of my application.
Multiple lines are fine.

%prep
%autosetup

%build
%cmake .
%cmake_build

%install
%cmake_install
install -D -m 0644 %{name}.service \
    %{buildroot}%{_unitdir}/%{name}.service

%files
%license LICENSE
%doc README.md
%{_bindir}/%{name}
%{_unitdir}/%{name}.service
%config(noreplace) %{_sysconfdir}/%{name}.conf

%post
%systemd_post %{name}.service

%preun
%systemd_preun %{name}.service

%postun
%systemd_postun_with_restart %{name}.service

%changelog
* Thu Apr 17 2026 Author <author@example.com> - 1.0.0-1
- Initial release
```

매크로(`%{_bindir}`, `%{_unitdir}` 등)는 `/usr/lib/rpm/macros`에 정의돼 있다. `rpm --eval '%{_bindir}'`로 확인할 수 있다. 매크로를 쓰면 배포판에 따라 경로가 자동으로 바뀐다.

`%config(noreplace)`가 중요하다. 사용자가 수정한 설정 파일을 패키지 업데이트 시 덮어쓰지 않고, `.rpmnew`로 새 버전을 옆에 둔다. 이걸 안 쓰면 사용자 설정이 날아간다.

### rpmbuild 실행

빌드 환경 구조:

```
~/rpmbuild/
├── BUILD/      # 압축 풀린 소스, 빌드 결과
├── BUILDROOT/  # %install 단계의 가상 루트
├── RPMS/       # 완성된 바이너리 RPM
├── SOURCES/    # 원본 tarball, 패치
├── SPECS/      # spec 파일
└── SRPMS/      # 소스 RPM
```

```bash
# 빌드 환경 생성
rpmdev-setuptree

# 소스와 spec 배치
cp myapp-1.0.0.tar.gz ~/rpmbuild/SOURCES/
cp myapp.spec ~/rpmbuild/SPECS/

# 빌드 의존성 자동 설치
sudo dnf builddep ~/rpmbuild/SPECS/myapp.spec

# 바이너리 + 소스 RPM 빌드
rpmbuild -ba ~/rpmbuild/SPECS/myapp.spec

# 결과
ls ~/rpmbuild/RPMS/x86_64/
# myapp-1.0.0-1.el8.x86_64.rpm
# myapp-debuginfo-1.0.0-1.el8.x86_64.rpm
# myapp-debugsource-1.0.0-1.el8.x86_64.rpm
```

운영 환경 빌드는 mock을 쓴다. 깨끗한 chroot에서 빌드하기 때문에 빌드 머신의 환경 오염을 막을 수 있다.

```bash
mock -r rocky-8-x86_64 --rebuild myapp-1.0.0-1.src.rpm
```

`%check` 섹션을 추가하면 빌드 후 자동으로 테스트가 돌아간다. CI 파이프라인에 RPM 빌드를 넣을 때 필수다.

```spec
%check
make test
%{__python3} -m pytest tests/
```

## DEB 패키지 빌드

### debian/ 디렉토리 구조

DEB 빌드는 RPM과 철학이 다르다. 단일 spec 파일이 아니라 `debian/` 디렉토리 아래의 여러 파일로 구성된다.

```
myapp-1.0.0/
├── src/
├── Makefile
└── debian/
    ├── changelog       # 변경 이력 (필수)
    ├── control         # 패키지 메타데이터 (필수)
    ├── copyright       # 라이선스 (필수)
    ├── rules           # 빌드 스크립트 (필수, Makefile)
    ├── compat          # debhelper 호환성 레벨 (선택)
    ├── install         # 설치할 파일 매핑
    ├── postinst        # 설치 후 스크립트
    ├── prerm           # 제거 전 스크립트
    └── source/
        └── format      # quilt 또는 native
```

`control` 파일이 RPM의 spec 헤더에 해당한다.

```
Source: myapp
Section: utils
Priority: optional
Maintainer: Author <author@example.com>
Build-Depends: debhelper-compat (= 13), cmake (>= 3.10), libssl-dev
Standards-Version: 4.6.0

Package: myapp
Architecture: any
Depends: ${shlibs:Depends}, ${misc:Depends}, libssl3
Description: My application
 Long description here.
 Each continuation line starts with a space.
```

`${shlibs:Depends}`가 흥미롭다. 빌드된 바이너리가 링크하는 공유 라이브러리를 자동으로 찾아서 의존성에 추가한다. 수동으로 의존성을 다 적을 필요가 없다.

`rules` 파일은 사실상 Makefile이다. debhelper의 `dh` 명령이 거의 모든 걸 자동화해준다.

```makefile
#!/usr/bin/make -f
%:
	dh $@ --with systemd

override_dh_auto_configure:
	dh_auto_configure -- -DCMAKE_BUILD_TYPE=Release
```

### dpkg-buildpackage

```bash
# 빌드 의존성 설치
sudo apt build-dep .
# 또는 mk-build-deps 사용
sudo mk-build-deps -i

# 빌드
dpkg-buildpackage -us -uc -b
# -us: 소스 서명 안 함
# -uc: changes 파일 서명 안 함  
# -b: 바이너리만 빌드

# 결과는 상위 디렉토리에 생성됨
ls ../
# myapp_1.0.0-1_amd64.deb
# myapp_1.0.0-1_amd64.changes
# myapp_1.0.0-1_amd64.buildinfo
```

운영 환경에서는 sbuild나 pbuilder로 깨끗한 chroot에서 빌드한다. mock의 데비안 버전이라고 보면 된다.

```bash
sudo sbuild-createchroot --include=eatmydata,ccache \
    bookworm /srv/chroot/bookworm-amd64 http://deb.debian.org/debian

sbuild -d bookworm myapp_1.0.0-1.dsc
```

`debian/changelog`는 단순한 변경 이력이 아니라 패키지 버전을 결정한다. 새 버전을 만들 때 `dch -i` 명령을 쓰면 된다.

```bash
dch -i  # 새 항목 추가, 에디터 열림
dch -r  # 릴리스 준비 (UNRELEASED → 배포명)
```

## GPG 서명

### 키 생성

운영 저장소를 운영하려면 패키지 서명용 GPG 키가 필수다. 사용자 키와 분리하고, 만료일을 길게 잡는다.

```bash
# 키 생성 (대화형)
gpg --full-generate-key
# 종류: RSA and RSA
# 크기: 4096
# 만료: 5y (5년) 또는 0 (무기한, 비추천)
# 이름/이메일: 저장소용 식별자 (예: MyRepo Signing Key <repo@example.com>)

# 또는 배치 모드
cat > keygen.batch <<EOF
%echo Generating signing key
Key-Type: RSA
Key-Length: 4096
Subkey-Type: RSA
Subkey-Length: 4096
Name-Real: MyRepo Signing Key
Name-Email: repo@example.com
Expire-Date: 5y
%no-protection
%commit
EOF

gpg --batch --generate-key keygen.batch

# 공개 키 export
gpg --armor --export repo@example.com > RPM-GPG-KEY-myrepo

# fingerprint 확인
gpg --fingerprint repo@example.com
```

키를 안전하게 보관하는 게 핵심이다. HSM이나 YubiKey를 쓰는 게 좋고, 최소한 빌드 서버와 분리된 서명 전용 호스트에 두어야 한다. 키가 유출되면 저장소 전체를 못 믿게 된다.

### RPM 서명

RPM은 패키지 자체에 서명을 넣는다. `~/.rpmmacros`에 키 ID를 설정해두고 `rpm --addsign`을 쓴다.

```bash
cat > ~/.rpmmacros <<EOF
%_signature gpg
%_gpg_name repo@example.com
%_gpg_path /home/builder/.gnupg
EOF

# 단일 패키지 서명
rpm --addsign myapp-1.0.0-1.el8.x86_64.rpm

# 여러 패키지 일괄 서명
rpm --addsign /srv/repo/Packages/*.rpm

# 서명 확인
rpm --checksig myapp-1.0.0-1.el8.x86_64.rpm
# myapp-1.0.0-1.el8.x86_64.rpm: digests signatures OK

rpm -qpi myapp-1.0.0-1.el8.x86_64.rpm | grep Signature
# Signature   : RSA/SHA256, Thu Apr 17 10:00:00 2026, Key ID abc123def456
```

저장소 메타데이터도 서명해야 한다. 이건 `repomd.xml`에 GPG signature를 분리 파일로 붙이는 방식이다.

```bash
gpg --detach-sign --armor /srv/repo/repodata/repomd.xml
# 결과: /srv/repo/repodata/repomd.xml.asc
```

`.repo` 파일에 `repo_gpgcheck=1`을 넣으면 클라이언트가 메타데이터 서명도 검증한다.

```ini
[myrepo]
gpgcheck=1         # 패키지 서명 검증
repo_gpgcheck=1    # 메타데이터 서명 검증
gpgkey=https://repo.example.com/RPM-GPG-KEY-myrepo
```

### DEB 서명

DEB는 패키지 파일 자체가 아니라 `Release` 파일을 서명한다. 패키지의 무결성은 `Packages` 파일 안의 SHA256 해시로 보장하고, `Release`는 `Packages`의 해시를 가지고 있고, `Release`가 서명돼 있다. 체인 구조다.

```bash
# Release.gpg: 분리 서명 (구식, 호환성용)
gpg --default-key repo@example.com -abs \
    -o /srv/apt/dists/bookworm/Release.gpg \
    /srv/apt/dists/bookworm/Release

# InRelease: 인라인 서명 (권장)
gpg --default-key repo@example.com --clearsign \
    -o /srv/apt/dists/bookworm/InRelease \
    /srv/apt/dists/bookworm/Release
```

reprepro와 aptly는 `SignWith`만 설정하면 자동으로 서명한다. 수동으로 할 일이 거의 없다.

개별 .deb 파일도 서명할 수 있는데(`debsigs`), 거의 안 쓴다. 표준 APT 도구가 .deb 자체 서명을 검증하지 않기 때문이다.

소스 패키지 업로드(빌드 후 나오는 .changes 파일)는 `debsign`으로 서명한다.

```bash
debsign -k repo@example.com myapp_1.0.0-1_amd64.changes
```

### 클라이언트 키 등록

클라이언트가 저장소를 신뢰하려면 공개 키를 등록해야 한다.

RPM 계열:

```bash
sudo rpm --import https://repo.example.com/RPM-GPG-KEY-myrepo
rpm -q gpg-pubkey --qf '%{NAME}-%{VERSION}-%{RELEASE}\t%{SUMMARY}\n'
```

APT 계열은 옛날에는 `apt-key add`를 썼는데, 이게 deprecated 됐다. Debian 11 / Ubuntu 22.04부터는 `/etc/apt/trusted.gpg.d/`나 `/etc/apt/keyrings/`에 키링 파일을 두고 sources.list에서 명시적으로 참조해야 한다.

```bash
# 키 다운로드해서 dearmor (binary 형식으로 변환)
curl -fsSL https://repo.example.com/key.asc | \
    sudo gpg --dearmor -o /etc/apt/keyrings/myrepo.gpg

# sources.list 항목에 signed-by 명시
echo "deb [signed-by=/etc/apt/keyrings/myrepo.gpg] https://repo.example.com/apt bookworm main" | \
    sudo tee /etc/apt/sources.list.d/myrepo.list
```

`signed-by`가 없으면 키링에 등록된 모든 키가 모든 저장소에 신뢰되는데, 이건 보안상 문제다. 한 저장소의 키가 유출되면 다른 저장소까지 위조 가능해진다.

## 우선순위와 핀 고정

### dnf priorities

같은 패키지를 여러 저장소가 제공할 때 어디서 가져올지 결정한다. `dnf-plugin-priorities` 플러그인이 필요했지만 DNF 4부터는 기본 내장이다.

```ini
[base]
name=CentOS Base
baseurl=...
priority=99      # 낮을수록 우선

[epel]
name=EPEL
baseurl=...
priority=10      # 더 우선

[mycompany]
name=Internal
baseurl=...
priority=1       # 최우선
```

priority가 같으면 버전이 높은 쪽이 이긴다. 그래서 internal 저장소를 priority=1로 두면, EPEL에 더 새 버전이 있어도 internal의 구버전이 설치된다. 운영 환경에서 안정성 보장에 유용하다.

특정 패키지만 다른 저장소에서 가져오고 싶으면 `excludepkgs`를 쓴다.

```ini
[epel]
excludepkgs=nginx*    # nginx는 EPEL에서 가져오지 않음
```

또는 `includepkgs`로 화이트리스트.

```ini
[mongodb]
includepkgs=mongodb-org*    # 이 저장소에서는 이것만
```

### apt pinning

APT는 더 세밀한 우선순위 제어가 가능하다. `/etc/apt/preferences.d/` 아래에 핀 파일을 둔다.

```
# /etc/apt/preferences.d/backports
Package: *
Pin: release a=bookworm-backports
Pin-Priority: 100
```

priority 값의 의미:

- 1000+ : 다운그레이드해서라도 설치
- 990   : 기본 릴리스 (설치된 버전 유지)
- 500   : 일반 (높은 버전 우선)
- 100   : 설치된 경우만 업데이트
- 0     : 설치 안 함
- 음수  : 절대 설치 안 함

backports를 100으로 두면 자동 설치되지 않고, 명시적으로 `apt install -t bookworm-backports package`를 해야 한다.

특정 패키지만 특정 저장소에서:

```
# /etc/apt/preferences.d/nginx
Package: nginx*
Pin: origin nginx.org
Pin-Priority: 700
```

버전 고정:

```
Package: kubelet
Pin: version 1.28.*
Pin-Priority: 1000
```

이걸로 마이너 버전 자동 업그레이드를 막을 수 있다. Kubernetes 클러스터 운영할 때 필수다.

### apt-mark hold

더 간단한 방법은 hold다. `apt upgrade`나 `apt-get upgrade`가 해당 패키지를 건드리지 않는다.

```bash
sudo apt-mark hold kubelet kubeadm kubectl
apt-mark showhold

sudo apt-mark unhold kubelet
```

dnf의 versionlock 플러그인도 비슷한 역할을 한다.

```bash
sudo dnf install python3-dnf-plugin-versionlock
sudo dnf versionlock add kubelet
sudo dnf versionlock list
```

## DNF history와 트랜잭션 디버깅

### 트랜잭션 이력

DNF는 모든 트랜잭션을 SQLite DB(`/var/lib/dnf/history.sqlite`)에 기록한다. 무엇을, 언제, 누가, 어떻게 설치/제거/업데이트했는지 다 들어 있다.

```bash
dnf history list
# ID  | Command line              | Date and time   | Action(s) | Altered
# ---------------------------------------------------------------
# 12  | install httpd             | 2026-04-17 10:30 | Install   | 5
# 11  | update                    | 2026-04-15 03:00 | Update    | 23

dnf history info 12
# Transaction ID : 12
# Begin time     : Thu 17 Apr 2026 10:30:15
# Begin rpmdb    : 412:abc...
# End time       : Thu 17 Apr 2026 10:31:02
# User           : root
# Command Line   : install httpd
# Packages Altered:
#     Install httpd-2.4.37-43.el8.x86_64 @AppStream
#     Install httpd-tools-2.4.37-43.el8.x86_64 @AppStream
#     ...
```

### Rollback과 Undo

`dnf history undo`는 특정 트랜잭션을 되돌린다. 설치된 건 제거하고, 제거된 건 다시 설치한다.

```bash
dnf history undo 12
# httpd 설치를 되돌림 → httpd와 함께 설치된 의존성 제거
```

`dnf history rollback`은 특정 시점 이후의 모든 트랜잭션을 되돌린다.

```bash
dnf history rollback 11
# 트랜잭션 11 직후 상태로 복귀 (12, 13, 14... 모두 undo)
```

다만 한계가 있다. 패키지가 저장소에서 사라졌으면(예: 마이너 버전 업데이트로 옛 버전이 제거됨) 롤백이 실패한다. 메이저 업그레이드는 사실상 롤백 불가능하다.

`dnf history redo`로 과거 트랜잭션을 다시 실행할 수도 있다. 동일한 패키지 세트를 다른 호스트에 복제할 때 쓸 만하다.

APT에는 동등한 기능이 없다. `dpkg.log`와 `apt history.log`로 이력은 볼 수 있지만 자동 롤백은 안 된다. 수동으로 이전 버전을 찾아서 다운그레이드해야 한다.

```bash
# APT에서 트랜잭션 이력 보기
less /var/log/apt/history.log
zless /var/log/apt/history.log.1.gz

# 특정 패키지 다운그레이드
apt install nginx=1.18.0-6.1+deb11u3
```

## 트랜잭션 실패 디버깅

### RPM 트랜잭션이 깨졌을 때

가장 흔한 증상은 `rpmdb open failed`. 트랜잭션 도중 강제 종료되면 RPM DB가 손상된다.

```bash
# DB 무결성 검사
rpm --verify --all 2>&1 | head

# DB 재구성 (RHEL 8 이전, BDB)
rpm --rebuilddb

# RHEL 9 이후는 sqlite 백엔드
dnf reinstall rpm
```

특정 패키지 파일이 깨졌는지 확인:

```bash
rpm -V httpd
# S.5....T.  c /etc/httpd/conf/httpd.conf
# .M.......    /usr/sbin/httpd
```

각 글자의 의미: S(크기), M(모드), 5(MD5), D(디바이스), L(링크), U(사용자), G(그룹), T(시간), P(능력). `c`는 config 파일이라는 뜻.

트랜잭션이 중간에 실패해서 일부만 적용된 경우:

```bash
# 미완료 트랜잭션 확인
dnf history list | head
# Transaction is incomplete

# 강제로 정리
dnf history rollback last
# 또는
dnf check
package-cleanup --problems
```

### APT 트랜잭션이 깨졌을 때

`dpkg`가 중간에 죽으면 패키지가 `iU`(설치 시도) 또는 `iF`(설정 실패) 상태로 남는다.

```bash
# 상태 확인
dpkg -l | grep -E '^[a-z][A-Z]'

# 설정 시도
sudo dpkg --configure -a

# 깨진 의존성 자동 해결
sudo apt --fix-broken install
sudo apt -f install
```

`sources.list`에 잘못된 저장소가 있어서 `apt update`가 실패하는 경우:

```bash
# 어떤 저장소가 문제인지 확인
sudo apt update 2>&1 | grep -E '^(Err|W:)'

# 특정 항목만 무시하고 update
sudo apt -o Dir::Etc::SourceList=/dev/null \
        -o Dir::Etc::SourceParts=/etc/apt/sources.list.d update
```

`dpkg --force-*` 옵션은 정말 위험하다. 알고 쓰면 트러블슈팅에 유용하지만, 모르고 쓰면 시스템이 더 망가진다.

```bash
# 의존성 무시하고 강제 설치 (위험)
dpkg -i --force-depends package.deb

# 같은 버전 재설치 (보통 안전)
dpkg -i --force-confnew package.deb   # 새 설정 파일 강제 사용
dpkg -i --force-confold package.deb   # 기존 설정 파일 유지
```

### 캐시 문제

저장소에서 패키지가 업데이트됐는데 클라이언트가 옛 메타데이터를 보는 경우.

```bash
# DNF 캐시 강제 갱신
dnf clean all
dnf makecache

# APT 캐시 강제 갱신
apt clean
apt update --no-cache  # 또는
rm -rf /var/lib/apt/lists/*
apt update
```

CDN 캐시가 문제인 경우도 있다. 특히 자체 저장소를 CloudFront 같은 CDN 뒤에 두면, 메타데이터는 짧은 TTL로 캐시하고 패키지 파일은 길게 캐시하는 식으로 분리해야 한다. `repomd.xml`이 1시간 캐시되면 클라이언트들이 새 패키지를 못 본다.

## 마무리

패키지 관리는 평소엔 명령어 몇 개면 충분하지만, 한번 꼬이면 끝없이 파고들어야 하는 영역이다. 솔버가 어떻게 결정을 내리는지, 메타데이터가 어떻게 구성되는지, 서명이 어디서 검증되는지 이해하고 있어야 문제 상황에서 헤매지 않는다.

자체 저장소를 운영한다면 처음부터 GPG 서명을 적용하고, snapshot 기반 운영(aptly나 별도 디렉토리 복사)을 도입하는 게 나중에 문제가 생겼을 때 살길이다. priority/pinning은 처음엔 복잡해 보이지만, 저장소가 두 개 이상 추가되는 순간부터 명시적으로 관리하지 않으면 어떤 패키지가 어디서 왔는지 추적이 안 된다.
