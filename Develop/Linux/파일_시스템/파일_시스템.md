# 파일 시스템

## 개요

리눅스에서 파일 시스템을 다룬다는 것은 단순히 `df`로 용량을 보거나 `mkfs`로 포맷하는 것보다 한참 아래 계층까지 내려간다. 운영 중인 서버에서 디스크가 차거나 inode가 고갈되는 상황을 한 번이라도 겪어봤다면, 명령어 사용법보다 그 안에서 실제로 무슨 일이 일어나는지를 이해하는 쪽이 훨씬 빠른 해결로 이어진다는 걸 안다.

이 문서는 파일 시스템 타입별 비교나 명령어 카탈로그가 아니라, VFS 계층부터 inode·디렉토리 엔트리·저널링·마운트 옵션이 실제 어떻게 맞물려 동작하는지, 그리고 운영 중에 자주 마주치는 문제들이 왜 발생하고 어떻게 풀어야 하는지에 초점을 둔다.

## VFS 계층 구조

리눅스에서 어떤 애플리케이션이 `open("/var/log/app.log")`를 호출하면, 커널은 곧장 ext4나 xfs 코드를 실행하지 않는다. 중간에 VFS(Virtual File System)라는 추상화 계층이 끼어 있다. VFS는 모든 파일 시스템이 따라야 할 공통 인터페이스를 정의하고, 실제 파일 시스템 구현체(ext4, xfs, btrfs, tmpfs, nfs 등)는 그 인터페이스를 자기 방식으로 구현한다.

VFS가 다루는 핵심 객체는 네 가지다.

- `super_block`: 마운트된 파일 시스템 하나당 하나. 파일 시스템 전체 메타데이터(블록 크기, 마운트 옵션, 사용 중인 inode 수 등)를 담는다.
- `inode`: 파일 또는 디렉토리 하나의 메타데이터(소유자, 권한, 크기, 타임스탬프, 데이터 블록 위치). 파일 이름은 inode에 들어 있지 않다.
- `dentry`: 디렉토리 엔트리. 파일 이름과 inode를 연결하는 캐시 항목. 경로 탐색 성능의 핵심.
- `file`: 열려 있는 파일 하나. 같은 inode를 두 프로세스가 열면 file 객체는 두 개가 만들어진다.

```
프로세스 → open() → VFS → ext4_file_open() → 디스크 블록 I/O
                    │
                    └─ dentry 캐시 (경로 → inode 매핑)
                    └─ inode 캐시 (메타데이터)
                    └─ page 캐시 (실제 파일 데이터)
```

이 구조 덕분에 `cp /mnt/ext4/a /mnt/xfs/b` 같은 작업이 자연스럽게 동작한다. VFS 입장에서는 양쪽 다 똑같은 인터페이스를 가진 파일일 뿐이다.

실무에서 이 구조가 의미 있는 지점은 캐시다. `cat large.log` 한 번 실행하고 나면 두 번째부터는 디스크를 거의 안 친다. 페이지 캐시가 메모리에 데이터를 들고 있기 때문이다. `free -h`에서 `buff/cache` 영역이 큰 것은 정상이고, 메모리가 모자라면 커널이 알아서 회수한다. 모니터링 알람이 "메모리 95% 사용 중"이라고 울려도 대부분 캐시라서 문제가 아니다.

```bash
sync; echo 3 > /proc/sys/vm/drop_caches
```

캐시를 강제로 비우는 명령인데, 운영 환경에서 함부로 쓸 일은 없다. 벤치마크나 디버깅 목적으로 콜드 상태를 만들 때만 쓴다.

## inode, 슈퍼블록, 블록 그룹

ext4를 포맷하면 디스크가 통째로 하나의 큰 영역이 되는 게 아니라 블록 그룹(block group) 단위로 잘게 쪼개진다. 보통 128MB짜리 그룹이 디스크 끝까지 늘어선다. 각 블록 그룹은 슈퍼블록 사본, 그룹 디스크립터, 블록 비트맵, inode 비트맵, inode 테이블, 데이터 블록을 포함한다.

```
[Group 0] [Group 1] [Group 2] ...
   │
   ├─ Superblock (백업본)
   ├─ Group Descriptor
   ├─ Block Bitmap (이 그룹에서 어떤 블록이 사용 중인가)
   ├─ Inode Bitmap (이 그룹에서 어떤 inode가 사용 중인가)
   ├─ Inode Table (실제 inode 구조체 배열)
   └─ Data Blocks
```

이렇게 잘라놓는 이유는 단순하다. 메타데이터와 데이터가 가까이 붙어 있어야 디스크 헤드 이동(SSD라면 페이지 점프)이 줄어든다. 같은 디렉토리에 있는 파일들은 가능하면 같은 블록 그룹에 배치하려고 한다.

inode 자체는 파일 이름을 가지고 있지 않다. inode 안에는 권한, 소유자, 크기, 타임스탬프, 그리고 실제 데이터가 어느 블록에 있는지를 가리키는 포인터들이 들어 있다. ext4 기본 inode 크기는 256바이트다. 디렉토리는 inode 하나에 "이름 → inode 번호" 매핑 테이블을 데이터 블록으로 가진다.

```bash
stat /etc/passwd
```

이 명령으로 보이는 Inode, Links, Access/Modify/Change 시간이 전부 inode에 들어 있는 정보다. 파일 이름은 그 파일이 속한 디렉토리의 데이터 블록에 저장된다. 그래서 같은 inode를 여러 디렉토리 엔트리가 가리킬 수 있고, 그게 하드 링크다.

inode 수는 포맷 시점에 결정된다. ext4는 기본적으로 16KB당 1개의 inode를 만든다. 한 번 정해지면 나중에 늘릴 수 없다. xfs는 동적 할당이라 이 제약이 없다. 작은 파일을 수백만 개 다루는 워크로드(메일 스풀, 캐시 디렉토리, 이미지 썸네일)라면 ext4 기본값으로 포맷했을 때 inode가 먼저 떨어지는 경우가 많다.

```bash
mkfs.ext4 -N 10000000 /dev/sdb1
mkfs.ext4 -i 4096 /dev/sdb1
```

`-N`은 inode 개수를 직접 지정하고 `-i`는 inode 하나당 바이트 비율을 정한다. 작은 파일이 많이 쌓일 디스크라면 포맷할 때 미리 결정해야 한다.

## 디렉토리 엔트리 동작

`/var/log/app.log`를 여는 과정을 따라가 보면 VFS와 디렉토리 엔트리가 어떻게 협력하는지 보인다.

1. 루트 inode를 찾는다. (슈퍼블록에서 바로 가져옴)
2. 루트 디렉토리 데이터 블록을 읽어 `var`의 inode 번호를 찾는다.
3. `var` 디렉토리 데이터 블록을 읽어 `log`의 inode 번호를 찾는다.
4. `log` 디렉토리 데이터 블록을 읽어 `app.log`의 inode 번호를 찾는다.
5. 그 inode를 읽어 데이터 블록 위치를 얻는다.

매번 이 짓을 하면 너무 느려서 커널이 dentry 캐시로 결과를 들고 있는다. 자주 접근하는 경로는 디스크를 거의 안 친다.

디렉토리 엔트리가 단순히 배열이면 파일 수가 많을 때 검색이 O(n)이 된다. ext4는 HTree(해시 트리)를 써서 디렉토리 안에서 이름 검색이 O(log n)이 되도록 한다. xfs는 B+ tree를 쓴다. 그럼에도 한 디렉토리에 파일이 수십만 개 쌓이면 `ls`가 느려지고, 특히 `ls -l`은 모든 파일의 inode를 추가로 읽어야 해서 더 느리다.

```bash
\ls -U      # 정렬 없이 그냥 디렉토리 엔트리 순서대로
find . -maxdepth 1 -type f | wc -l
```

파일이 너무 많은 디렉토리를 다룰 때는 `ls`보다 `find`가 빠르고, 정렬을 끄면 더 빠르다. `\ls`는 alias 우회용이다.

지운 파일이 디렉토리 엔트리에서만 빠지고 inode는 즉시 회수되지 않을 수 있다. 누군가 그 파일을 열어두고 있다면 더더욱 그렇다. 이게 뒤에 나올 "rm 했는데 공간이 안 빈다" 문제의 근본 원인이다.

## 하드 링크와 심볼릭 링크

링크는 파일 시스템에서 헷갈리기 쉬운 개념인데, inode를 이해하면 단번에 정리된다.

### 하드 링크

`ln source target`은 새로운 디렉토리 엔트리를 만들어 같은 inode를 가리키게 한다. 디스크 상의 파일은 하나뿐이고 이름만 두 개다.

```bash
echo "hello" > /tmp/a
ln /tmp/a /tmp/b
ls -li /tmp/a /tmp/b
# 같은 inode 번호, Links 카운트가 2
```

`stat` 결과의 `Links` 필드가 inode를 가리키는 디렉토리 엔트리 수다. `rm /tmp/a`를 해도 `/tmp/b`가 남아 있으면 inode는 살아 있다. 데이터 블록은 마지막 하드 링크가 사라지고 그 inode를 열고 있는 프로세스가 없을 때 회수된다.

제약이 있다. 같은 파일 시스템 안에서만 만들 수 있고, 디렉토리에는 일반적으로 만들 수 없다(루프 방지).

```bash
ln /mnt/ext4_disk/a /mnt/xfs_disk/b
# Invalid cross-device link
```

### 심볼릭 링크

`ln -s source target`은 별도의 작은 파일을 만들고 그 안에 원본 경로를 텍스트로 저장한다. inode가 다르고, 데이터 블록에 경로 문자열이 들어 있다.

```bash
ln -s /tmp/a /tmp/c
ls -li /tmp/a /tmp/c
# inode 번호 다름, /tmp/c는 lrwxrwxrwx 권한
readlink /tmp/c
# /tmp/a
```

심볼릭 링크는 파일 시스템을 넘어갈 수 있고, 디렉토리도 가리킬 수 있고, 존재하지 않는 경로도 가리킬 수 있다. 마지막 특성이 dangling link 문제로 이어진다.

```bash
rm /tmp/a
ls -l /tmp/c
# 링크는 그대로 남지만
cat /tmp/c
# cat: /tmp/c: No such file or directory
```

원본이 사라져도 심볼릭 링크는 남는다. 정기적으로 백업 디렉토리를 정리하는 스크립트가 원본을 지우면서 링크를 안 지우는 경우, dangling link가 누적된다. `find /path -xtype l`로 깨진 심볼릭 링크를 찾을 수 있다.

심볼릭 링크는 상대 경로로도 만들 수 있다. 컨테이너 이미지나 패키징에서 자주 쓰는 트릭이다.

```bash
cd /opt
ln -s /opt/app-1.2.3 current
# 배포 후 새 버전으로 바꿀 때
ln -sfn /opt/app-1.2.4 current
```

`-f`는 기존 링크 덮어쓰기, `-n`은 `current`가 디렉토리를 가리키는 심볼릭 링크일 때 그 안으로 들어가지 않고 링크 자체를 갈아끼우게 한다. `-n` 없이 하면 `/opt/current/app-1.2.4`가 만들어져버려서 사고가 난다. 무중단 배포에서 자주 사용하는 패턴이다.

### 둘 중 무엇을 쓸 것인가

- 같은 파일 시스템 안에서 동일 파일을 여러 위치에서 참조해야 하고, 누가 원본인지 신경쓰고 싶지 않다 → 하드 링크
- 파일 시스템을 넘어가거나, 디렉토리를 가리키거나, 원본 경로를 의미적으로 표현하고 싶다 → 심볼릭 링크
- 운영 도구로 추적하기 쉬운 쪽 → 심볼릭 링크 (`ls -l`에서 즉시 보인다)

대부분의 경우 심볼릭 링크가 답이다. 하드 링크는 백업 도구(`rsync --link-dest`)처럼 동일 파일을 디스크 공간 없이 여러 스냅샷에 공유시키는 특수 용도가 아니면 잘 안 쓴다.

## 저널링 모드 비교

ext4와 xfs는 둘 다 저널링을 한다. 갑자기 전원이 나가도 파일 시스템이 깨지지 않게 하는 메커니즘이다. 저널 영역에 변경 사항을 먼저 기록하고, 나중에 실제 위치에 적용한다. 만약 적용 도중에 죽으면 다음 부팅에서 저널을 재생해 일관성을 복구한다.

ext4는 세 가지 저널링 모드가 있다.

| 모드 | 저널에 기록 | 안정성 | 성능 |
|------|-------------|--------|------|
| `data=journal` | 메타데이터 + 데이터 | 가장 강함 | 가장 느림 (모든 쓰기가 두 번) |
| `data=ordered` | 메타데이터만, 단 데이터를 먼저 디스크에 쓴 후 메타데이터를 저널에 기록 | 강함 | 기본값 |
| `data=writeback` | 메타데이터만, 데이터/메타데이터 순서 보장 없음 | 약함 | 가장 빠름 |

`data=journal`은 데이터까지 저널에 두 번 쓴다. 쓰기 성능이 절반 가까이 떨어지지만, 어떤 상황에서도 데이터가 파일 끝에 잘려나가거나 0으로 채워지지 않는다. 데이터베이스 WAL처럼 이미 자체적으로 동기화 보장을 하는 워크로드에는 오히려 손해다.

`data=ordered`가 기본값이다. 메타데이터를 저널에 기록하기 전에 관련 데이터 블록을 반드시 디스크에 쓴다. 그래서 크래시 후에도 메타데이터가 가리키는 위치에 옛 데이터가 남거나 0이 채워지는 일은 없다. 거의 모든 일반 워크로드에 충분하다.

`data=writeback`은 메타데이터만 저널에 보호하고 데이터 블록은 언제 써질지 보장하지 않는다. 크래시 시 파일 끝에 쓰레기 데이터가 노출될 수 있다. 임시 파일이나 캐시처럼 잃어도 되는 데이터가 대부분인 디스크에서 성능을 더 짜낼 때 쓰지만, 실수로 영구 데이터 디스크에 적용하면 사고로 이어진다.

```bash
mount -o remount,data=ordered /dev/sda1 /
tune2fs -o journal_data_writeback /dev/sdb1
```

mount 옵션으로는 `data=` 모드를 변경할 수 있는데 ext4에서는 마운트된 상태에서 변경이 제한된다. `tune2fs -o`로 기본값을 바꾸고 재마운트해야 적용되는 경우가 많다.

xfs는 메타데이터 저널만 한다. ext4의 `data=writeback`과 비슷한 모델인데, 로그 구조가 더 효율적이고 대용량에서 안정적이다. 데이터베이스 서버에서 xfs를 선호하는 이유 중 하나가 이 점이다.

## 마운트 옵션 심화

`/etc/fstab`의 마운트 옵션은 운영 성능과 안정성에 직결된다. 기본값으로 두면 대부분 문제없지만, 워크로드에 맞춰 손봐야 할 때가 있다.

### atime 관련

파일을 읽기만 해도 atime(접근 시간)이 갱신되도록 되어 있는 게 POSIX 기본 동작이다. 매번 메타데이터 쓰기가 발생해서 읽기 워크로드가 많은 시스템에서 불필요한 I/O가 된다.

- `strictatime`: 읽을 때마다 atime 업데이트. 기본 동작이지만 거의 안 쓴다.
- `relatime`: atime이 mtime이나 ctime보다 옛날이거나 24시간 이상 지났을 때만 갱신. 대부분의 배포판 기본값이다.
- `noatime`: atime을 아예 안 건드린다. 가장 성능에 유리하다.
- `lazytime`: atime과 mtime을 메모리에만 갱신하고 디스크 쓰기는 미루거나 합친다. 아주 빈번하게 쓰이는 파일에 효과적이다.

```bash
# /etc/fstab
UUID=xxx  /var/lib/mysql  ext4  defaults,noatime  0 2
UUID=yyy  /var/log        ext4  defaults,lazytime 0 2
```

데이터베이스 디렉토리에는 `noatime`을 거의 항상 건다. 메일이나 로그 디렉토리처럼 atime이 의미 있는 일부 워크로드(메일 보관 기간 판단 등)가 아니라면 안 켤 이유가 없다.

### barrier

저널이 디스크에 실제로 기록된 후에 데이터를 쓰도록 보장하는 옵션이다. 기본 켜져 있다. 끄면 성능이 빨라지지만 갑작스러운 전원 차단 시 파일 시스템 손상 위험이 커진다.

```bash
# 절대 운영 환경에서 끄지 말 것
mount -o remount,nobarrier /dev/sda1 /
```

UPS가 있고 배터리 백업된 디스크 컨트롤러를 쓴다면 켜고 끄는 게 의미가 있을 수 있는데, 클라우드 가상 머신이나 일반 SSD에서는 그냥 두는 게 맞다.

### discard

SSD에서 삭제된 블록을 펌웨어에 알리는 TRIM 명령을 자동으로 보낼지 결정한다. 켜면 SSD 수명에 도움되지만, 삭제할 때마다 추가 명령이 나가서 워크로드에 따라 지연이 생긴다.

```bash
# /etc/fstab
UUID=xxx  /  ext4  defaults,discard  0 1

# 또는 주기적으로 실행 (권장)
fstrim -v /
```

대부분 배포판이 `fstrim.timer`를 통해 주 1회 일괄 TRIM을 돌리도록 설정되어 있다. `discard` 마운트 옵션은 켜지 않고 주기적 fstrim만 돌리는 게 일반적인 운영 방식이다.

### nofail

부팅 시 해당 마운트가 실패해도 부팅을 계속하게 한다. 외장 디스크나 NFS, 클라우드 추가 볼륨처럼 항상 붙어 있다고 보장 못 하는 마운트에 쓴다.

```bash
# /etc/fstab
//nas/share  /mnt/nas  cifs  credentials=/etc/cifs.cred,nofail,_netdev  0 0
```

`nofail` 없이 NFS/CIFS를 fstab에 박아두면 네트워크가 늦게 올라올 때 부팅이 emergency mode로 떨어진다. 클라우드에서 EBS 추가 볼륨을 fstab에 등록할 때도 마찬가지로 `nofail`을 같이 쓰는 게 안전하다. `_netdev`는 네트워크가 준비된 후에 마운트하라는 힌트다.

## bind mount와 overlay fs

같은 디렉토리를 다른 위치에도 노출시키거나, 여러 디렉토리를 겹쳐 보이게 하는 트릭은 컨테이너 시대에 특히 자주 쓰인다.

### bind mount

디렉토리(또는 파일)를 다른 경로에 마운트한다. 심볼릭 링크와 비슷해 보이지만 다르다. 심볼릭 링크는 경로 해석 단계에서 풀리지만 bind mount는 VFS가 동일한 inode를 그대로 노출한다.

```bash
mount --bind /var/log /chroot/var/log
mount --bind /etc/resolv.conf /chroot/etc/resolv.conf
```

chroot 환경이나 컨테이너 런타임이 호스트의 일부 파일을 격리된 환경 안으로 끌어들일 때 핵심 메커니즘이다. 도커가 볼륨을 마운트하는 것도 결국 bind mount다.

심볼릭 링크와의 결정적 차이: bind mount된 디렉토리에서 `..`가 다르게 보인다. 심볼릭 링크는 원본 경로의 부모로 가지만 bind mount는 마운트 지점의 부모로 간다. chroot 탈출을 막는 데 중요한 차이다.

`/etc/fstab`에도 적을 수 있다.

```bash
/var/log  /chroot/var/log  none  bind  0 0
```

읽기 전용 bind도 만들 수 있는데, 한 단계 더 필요하다.

```bash
mount --bind /etc /mnt/etc
mount -o remount,ro,bind /mnt/etc
```

`bind` 마운트 후 `remount,ro,bind`로 두 번 마운트해야 실제 읽기 전용이 된다. 한 줄로 `mount -o ro,bind`를 시도하면 ext4의 ro 옵션은 무시되고 원본의 rw 상태가 그대로 노출된다. 이걸 모르면 보안 격리가 안 된다.

### overlay fs

여러 디렉토리를 겹쳐서 하나의 파일 시스템처럼 보이게 한다. 도커 이미지 레이어, 라이브 CD, A/B 부트가 전부 이걸로 동작한다.

```
upperdir (쓰기 가능)
   ↑
lowerdir (읽기 전용, 여러 개 가능)
   ↑
merged (사용자가 보는 통합 뷰)
```

읽기는 위에서 아래로 찾고, 쓰기는 항상 upperdir에 일어난다. lowerdir에 있던 파일을 수정하려고 하면 copy-up이 일어나 upperdir로 복사된 후 거기서 수정된다.

```bash
mkdir -p /tmp/overlay/{lower,upper,work,merged}
mount -t overlay overlay \
  -o lowerdir=/tmp/overlay/lower,upperdir=/tmp/overlay/upper,workdir=/tmp/overlay/work \
  /tmp/overlay/merged
```

`workdir`은 overlay fs가 내부적으로 원자적 연산을 위해 쓰는 작업 영역인데, upperdir과 같은 파일 시스템 위에 있어야 한다. 직접 손대지 않는다.

도커가 이미지 빌드할 때 각 레이어를 lowerdir로 쌓고, 컨테이너 실행할 때 새 upperdir을 얹는 구조라고 보면 된다. 컨테이너에서 파일 하나 수정하면 그 파일만 upperdir로 복사되고, 컨테이너를 지우면 upperdir이 사라져 원본은 그대로다.

운영 환경에서 직접 overlay를 마운트할 일은 많지 않지만, 컨테이너 디스크가 차오를 때 어떤 레이어가 얼마나 잡아먹고 있는지 추적하려면 이 구조를 알아야 한다. `/var/lib/docker/overlay2/` 아래를 들여다볼 일이 생긴다.

## 트러블슈팅: rm 했는데 df 공간이 안 빈다

운영 중 가장 자주 마주치는 상황이다. 큰 로그 파일을 `rm`으로 지웠는데 `df`로 보면 여유 공간이 그대로다.

원인은 inode 회수 메커니즘이다. inode가 실제로 회수되어 데이터 블록이 풀리려면 두 조건이 모두 충족되어야 한다.

1. 파일을 가리키는 모든 디렉토리 엔트리가 사라짐 (Links == 0)
2. 그 inode를 열고 있는 파일 디스크립터가 하나도 없음

`rm`은 1번만 처리한다. 어떤 프로세스가 그 파일을 열어둔 채로 있으면 2번이 안 충족돼서 inode가 살아 있고, 디스크 공간도 그대로다. 로그 파일을 지웠는데 그걸 쓰던 데몬(예: 애플리케이션, nginx, journald)이 여전히 fd를 들고 있다면 정확히 이 상황이 된다.

확인하는 방법은 `lsof`다.

```bash
lsof | grep deleted
```

또는 더 정확히는,

```bash
lsof +L1
```

`+L1`은 링크 카운트가 1 미만(즉, 0)인 열린 파일을 보여준다. 출력에서 `(deleted)` 표시가 붙은 파일들이 범인이다.

해결 방법은 두 가지다.

- 그 fd를 잡고 있는 프로세스를 재시작한다. `systemctl restart <service>` 또는 `kill -HUP <pid>`로 로그를 다시 열게 한다.
- 재시작이 곤란하면 `/proc/<pid>/fd/<n>`을 통해 파일 내용을 비운다. fd 자체는 유지되므로 프로세스는 영향받지 않는다.

```bash
# 예: pid 1234가 /var/log/app.log를 fd 7로 열고 있고 그 파일을 rm한 상태
: > /proc/1234/fd/7
```

이 방법은 데이터를 0바이트로 만들어 디스크 공간을 즉시 회수한다. 단, 프로세스가 그 fd에 계속 쓰면 새 데이터가 다시 쌓이므로 임시방편이다. 결국엔 제대로 된 로그 로테이션이 답이다.

## 트러블슈팅: No space left on device인데 df는 여유있음

`df -h`로 보면 디스크 사용량이 80%인데 파일 생성에서 `No space left on device`가 나는 황당한 상황이 있다. 원인은 보통 두 가지 중 하나다.

### inode 고갈

`df -i`로 확인한다. inode 사용량이 100%면 답이 나온다.

```bash
df -i
# Filesystem    Inodes   IUsed    IFree IUse% Mounted on
# /dev/sda1     655360  655360        0  100% /
```

작은 파일이 너무 많이 쌓였다는 뜻이다. 어디에 쌓였는지 찾으려면,

```bash
for d in /*; do echo "$d: $(find $d 2>/dev/null | wc -l)"; done | sort -t: -k2 -n
```

또는 디렉토리별로 빠르게 카운트하는 도구가 있다면 더 좋다. 자주 범인이 되는 곳은 `/var/spool`, `/var/cache`, 메일 스풀, 세션 파일 디렉토리다.

ext4는 inode 수가 포맷 시 고정이라 늘릴 방법이 없다. 데이터를 백업하고 재포맷할 때 `mkfs.ext4 -N <count>` 또는 `-i <bytes_per_inode>`로 더 많이 만들거나, xfs로 옮기는 게 근본 대책이다.

### 디렉토리 엔트리 한계

ext4의 한 디렉토리에 들어갈 수 있는 파일 수는 dir_index를 켰을 때도 실용적인 한계가 있다. 한 디렉토리에 수백만 개를 넣으려고 하면 엔트리 블록 한계에 부딪힌다. 이때도 `No space left`가 날 수 있다.

해결은 디렉토리를 쪼개는 것이다. 해시 기반으로 `cache/00/`, `cache/01/`, ... 같이 두 글자씩 prefix로 분배하는 식이 흔하다.

### 예약 공간

ext4는 기본 5%를 root 전용으로 예약한다. 일반 사용자 입장에서는 95%까지 쓰면 디스크 가득이다. `df`는 이걸 반영한 사용 가능 공간을 보여주지만, 일부 모니터링 도구나 애플리케이션이 다르게 계산할 수 있다.

```bash
tune2fs -m 1 /dev/sdb1
```

데이터 디스크라면 예약을 1%나 0%로 줄여도 된다. 루트 파티션은 5%를 유지하는 게 안전하다. 디스크가 가득 차도 root가 들어와서 정리할 수 있는 여유가 필요하다.

## 디스크 풀 대응 시 안전한 정리 순서

디스크가 100% 차서 서비스가 멈췄다는 알람이 울리면, 무턱대고 큰 파일부터 지우면 안 된다. 잘못하면 서비스가 더 나빠지거나 복구가 어려워진다.

순서를 지킨다.

1. **먼저 어떤 디렉토리가 디스크를 잡고 있는지 파악한다.**

   ```bash
   du -h --max-depth=1 / 2>/dev/null | sort -h | tail -20
   ```

   루트부터 내려가면서 큰 디렉토리를 찾는다. 보통 `/var/log`, `/var/lib`, `/var/cache`, `/tmp`, `/home`이 후보다.

2. **삭제 안전한 대상부터 손댄다.** 캐시, 임시 파일, 오래된 로그가 우선이다.

   ```bash
   journalctl --vacuum-size=500M
   apt clean        # 또는 yum clean all
   find /tmp -mtime +7 -delete
   ```

3. **로그 파일은 함부로 rm 하지 말고 truncate한다.** rm을 하면 앞서 설명한 fd 점유 문제로 공간이 안 빈다.

   ```bash
   truncate -s 0 /var/log/app.log
   # 또는
   : > /var/log/app.log
   ```

   `: > file`은 셸이 파일을 열고 0바이트로 덮어쓰는 동작이다. 데몬이 같은 파일을 append 모드로 들고 있어도 같은 inode를 공유하므로 즉시 공간이 회수된다.

4. **애플리케이션 데이터는 마지막에, 그것도 백업과 확인 후에.** 데이터베이스 WAL이나 임시 테이블 파일을 건드리면 서비스가 망가진다.

5. **공간이 좀 확보되면 로그 로테이션 설정을 점검한다.** 같은 일이 반복되지 않도록 `logrotate` 설정을 손본다.

`logrotate`의 `copytruncate` 옵션을 알아둔다. 일반 모드는 파일을 rename한 뒤 새 파일을 만드는데, fd로 열어놓은 데몬이 SIGHUP을 안 받으면 새 파일에 안 쓰고 옛 fd에 계속 쓴다(앞서 본 deleted 파일 문제). `copytruncate`는 파일을 복사한 후 원본을 truncate해서 fd를 안전하게 유지한다. 단, 복사와 truncate 사이에 일부 로그가 유실될 수 있다.

```
/var/log/app/*.log {
    daily
    rotate 7
    compress
    copytruncate
    missingok
    notifempty
}
```

데몬이 `SIGHUP`이나 `SIGUSR1`로 로그를 다시 여는 것을 지원하면 `postrotate` 스크립트로 시그널을 보내는 쪽이 더 깔끔하지만, 그게 안 되는 애플리케이션에는 `copytruncate`가 현실적이다.

## /var/log truncation 시 주의점

`> file`과 `truncate file`을 헷갈리지 말아야 한다. 둘 다 파일을 0바이트로 만들지만 동작이 다르다.

- `: > /var/log/app.log` 또는 `> /var/log/app.log`: 셸이 새로 파일을 열고 O_TRUNC로 자른다. inode는 유지되고 다른 프로세스가 가진 fd도 그대로 유효하다. append 모드로 열린 fd는 다음 쓰기에서 파일 끝(현재 위치)을 다시 계산한다.
- `truncate -s 0 /var/log/app.log`: truncate 시스템 콜을 직접 호출. 똑같이 inode 유지.
- `rm /var/log/app.log; touch /var/log/app.log`: inode가 바뀐다. 데몬이 옛 fd를 들고 있으면 그 fd는 보이지 않는 deleted 파일을 가리킨다. 새로 만든 파일은 비어 있고, 옛 데몬이 쓰는 데이터는 디스크에 보이지 않는 곳으로 사라진다. 절대 이렇게 하면 안 된다.

```bash
# 안전 (inode 유지, fd 유효)
: > /var/log/app.log

# 안전 (동일)
truncate -s 0 /var/log/app.log

# 위험 (inode 바뀜, fd 무효화)
rm /var/log/app.log
touch /var/log/app.log
```

또 하나 헷갈리는 함정이 있다. append 모드(`O_APPEND`)로 열린 fd는 매 쓰기마다 파일 끝으로 자동 이동한다. 그래서 `: > file`로 자르면 다음 쓰기는 0바이트 위치부터 다시 채워진다. 반면 append가 아닌 일반 쓰기 모드 fd는 자기 내부 오프셋을 유지한다. 이 fd가 오프셋 5GB에 있는 상태에서 파일을 0바이트로 자르면, 다음 쓰기에서 5GB 위치에 쓰려고 한다. 사이는 sparse hole로 채워지고, `du`는 작게 나오지만 `ls -l`로는 5GB로 보인다. 일부 도구가 이 sparse 상태에서 오작동한다.

거의 모든 로그 데몬은 append 모드로 열기 때문에 실무에서는 큰 문제가 되지 않지만, 자체 제작한 도구나 비표준 로깅에서는 한 번씩 사고를 본다.

## inode 정보 확인 명령

이론보다 실제로 inode 상태를 들여다볼 때 자주 쓰는 명령들이다.

```bash
stat /etc/passwd                    # inode 번호, Links 카운트, 시간들
ls -li                              # 디렉토리 안 파일들의 inode 번호
df -i                               # 파일 시스템별 inode 사용량
find /path -inum 12345              # 특정 inode 번호로 파일 찾기
debugfs -R "stat <12345>" /dev/sda1 # 디스크에서 직접 inode 정보 (ext4)
```

`debugfs`는 ext4의 디스크 구조를 직접 들여다볼 수 있는 도구다. 디렉토리 엔트리, inode 테이블, 블록 비트맵을 검사할 수 있다. 일반 운영에서 쓸 일은 없지만, 손상된 파일 시스템을 분석할 때 유용하다.

xfs는 `xfs_db`가 비슷한 역할을 한다.

## 마무리

파일 시스템은 명령어 몇 개로 끝나는 주제가 아니다. VFS가 있고, inode와 디렉토리 엔트리가 있고, 저널이 돌고, 캐시가 있고, fd가 inode를 잡고 있다. 운영 중에 마주치는 대부분의 디스크 문제는 이 구조를 모르면 명령어를 외워도 풀리지 않는다.

특히 "rm 했는데 공간이 안 빈다", "df는 여유있는데 No space left다", "로그 truncate 후에도 데몬이 옛 위치에 쓴다" 같은 상황은 구조를 이해하면 즉시 진단된다. 모르면 재부팅이 답이라고 생각하게 되는데, 운영 중인 서비스에 재부팅은 거의 항상 마지막 수단이다.

평소에 inode 사용량 모니터링을 디스크 사용량과 함께 걸어두고, 로그 로테이션 설정에서 fd 처리가 어떻게 되는지 확인해두면 새벽에 알람으로 깨는 횟수가 줄어든다.
