# 사용자 및 권한 관리

## 개요

리눅스에서 사용자 관리는 단순히 `useradd`로 계정을 만드는 작업이 아니다. 어떤 UID로 만들어지는지, 그 UID가 컨테이너 환경에서 호스트와 어떻게 매핑되는지, sudo 권한이 어디까지 위임되어 있는지, PAM 모듈이 인증 흐름에 어떻게 개입하는지 같은 것들이 실제 운영에서는 더 중요하다.

5년 정도 운영 일을 하다 보면 결국 같은 사고를 반복하게 된다. `usermod -aG` 대신 `-G`를 써서 누군가가 docker 그룹에서 빠지는 사고, NOPASSWD로 묶어둔 명령에 vim이 들어 있어 root 쉘이 따이는 사고, 컨테이너 안의 UID 1000과 호스트의 UID 1000이 다른 사람이라 마운트한 볼륨 권한이 박살나는 사고. 이 문서는 그런 실수들을 다시 반복하지 않기 위한 정리다.

## 사용자 정보 파일 구조

### /etc/passwd

`cat /etc/passwd`로 읽을 수 있는 텍스트 파일이다. 콜론으로 구분된 7개 필드를 가진다.

```text
deploy:x:1001:1001:Deploy User:/home/deploy:/bin/bash
```

| 필드 | 값 | 설명 |
|------|------|------|
| 1 | deploy | 로그인 이름 |
| 2 | x | 비밀번호 자리 (실제 해시는 shadow에) |
| 3 | 1001 | UID |
| 4 | 1001 | 기본 GID |
| 5 | Deploy User | GECOS 필드 (이름, 사무실, 전화번호 등 자유 메모) |
| 6 | /home/deploy | 홈 디렉토리 |
| 7 | /bin/bash | 로그인 쉘 |

두 번째 필드의 `x`는 "비밀번호 해시를 shadow 파일에서 읽어라"는 의미다. 옛날 시스템에서는 여기에 직접 해시가 들어 있었지만, passwd는 모든 사용자가 읽을 수 있어서(644) 해시가 노출되는 문제가 있었다. 그래서 shadow 파일로 분리됐다.

### /etc/shadow

비밀번호 해시와 만료 정책을 저장한다. 권한이 `0640` 또는 `0000`이라 root만 읽을 수 있다.

```text
deploy:$y$j9T$F.x...$abc...:19500:0:99999:7:::
```

9개 필드를 가진다.

| 필드 | 의미 |
|------|------|
| 1 | 사용자명 |
| 2 | 암호화된 비밀번호 해시 |
| 3 | 마지막 비밀번호 변경일 (1970-01-01 이후 일수) |
| 4 | 비밀번호 변경 후 다시 변경 가능한 최소 일수 |
| 5 | 비밀번호 만료까지 최대 일수 (PASS_MAX_DAYS) |
| 6 | 만료 경고 시작 일수 |
| 7 | 만료 후 비활성화까지 유예 일수 |
| 8 | 계정 만료일 (epoch days) |
| 9 | 예약 필드 |

두 번째 필드의 해시 포맷은 `$id$salt$hash` 구조이고, `$id$`로 알고리즘을 구분한다.

| 식별자 | 알고리즘 | 비고 |
|------|------|------|
| `$1$` | MD5 | 1990년대 방식. 더 이상 쓰면 안 된다 |
| `$2a$`, `$2b$`, `$2y$` | bcrypt | OpenBSD 계열 |
| `$5$` | SHA-256 crypt | glibc 표준 |
| `$6$` | SHA-512 crypt | RHEL/CentOS 7~8 기본 |
| `$y$` | yescrypt | Debian 11+, Fedora 35+ 기본 |
| `*` 또는 `!` | 잠금 상태 | `passwd -l`이 만든 흔적 |
| 빈 문자열 | 비밀번호 없음 | SSH 키 전용 계정에서 의도적일 수 있음 |

운영 중에 `$1$`로 시작하는 해시를 발견하면 오래된 시스템에서 마이그레이션된 흔적이다. 이때는 `passwd -e`로 다음 로그인 때 강제 변경을 유도해 새 알고리즘으로 다시 저장되도록 한다. `/etc/login.defs`의 `ENCRYPT_METHOD`를 확인하면 새로 만들어지는 비밀번호의 알고리즘을 알 수 있다.

### /etc/group과 /etc/gshadow

group 파일은 그룹명, 비밀번호 자리, GID, 멤버 목록 4개 필드다.

```text
docker:x:998:deploy,jenkins
```

멤버 목록에 사용자 이름이 콤마로 나열된다. 여기에 들어간 사용자는 그 그룹을 보조 그룹으로 가진다. 기본 그룹(passwd 4번째 필드의 GID)은 group 파일의 멤버 목록에는 보통 없다는 점이 헷갈린다. `id deploy`로 보면 둘 다 보인다.

gshadow는 그룹 비밀번호 해시와 그룹 관리자(그룹에 사용자를 넣고 뺄 수 있는 사람)를 저장한다. 실무에서는 그룹 비밀번호를 거의 쓰지 않아서 신경 쓸 일이 적다.

## 시스템 계정 vs 일반 계정

UID로 구분된다. `/etc/login.defs`의 `UID_MIN`과 `UID_MAX`가 일반 계정 범위를 정한다. 대부분의 배포판에서 `UID_MIN=1000`, `SYS_UID_MIN=100`, `SYS_UID_MAX=999`이다.

```bash
grep -E '^UID_(MIN|MAX)|^SYS_UID' /etc/login.defs
```

UID 0은 root, 1~99는 데몬용 예약 영역, 100~999는 패키지가 만든 시스템 계정, 1000 이상이 일반 계정이다. 시스템 계정은 보통 로그인이 필요 없으므로 쉘을 `nologin`이나 `false`로 막는다.

```bash
useradd -r -s /usr/sbin/nologin -M -d /var/lib/myapp myapp
```

`-r`은 시스템 계정으로 만들어 UID를 SYS_UID_MIN~SYS_UID_MAX 범위에서 할당한다. `-M`은 홈 디렉토리를 만들지 않는다(`-m`의 반대). `-s /usr/sbin/nologin`을 주면 이 사용자로 ssh 로그인이나 `su -` 시도가 들어와도 "This account is currently not available." 메시지만 출력하고 끊어진다.

`/sbin/nologin`과 `/bin/false`의 차이가 있다. nologin은 메시지를 출력하고 종료 코드 1로 종료한다. false는 아무것도 출력하지 않고 종료 코드 1로 종료한다. cron이 false 쉘 사용자로 작업을 실행하려고 하면 실패하므로, 서비스 계정은 nologin이 더 적합하다. 일부 서비스(예: git over ssh)는 `git-shell` 같은 제한된 쉘을 쓰기도 한다.

서비스 계정을 systemd 유닛에서 쓰려면 `systemd-sysusers`를 활용하는 편이 깔끔하다. `/usr/lib/sysusers.d/myapp.conf`에 다음처럼 정의한다.

```text
u myapp - "My App Service" /var/lib/myapp /usr/sbin/nologin
g myapp -
```

`systemd-sysusers`를 실행하면 UID/GID가 없으면 만들고 있으면 건너뛰는 식으로 멱등적으로 처리한다. 패키지 설치 스크립트에서 useradd를 직접 호출하던 옛날 방식보다 안정적이다.

## useradd, usermod, userdel

### useradd vs adduser

이름이 비슷해서 헷갈리는데 동작은 꽤 다르다. 이게 배포판마다 또 다르다.

| 환경 | useradd | adduser |
|------|------|------|
| Debian/Ubuntu | low-level 명령. 홈 디렉토리, 비밀번호, GECOS 모두 따로 설정 | Perl 스크립트. 대화형으로 묻고 useradd를 호출하는 wrapper |
| RHEL/CentOS/Rocky | 표준 명령 | useradd로 가는 심볼릭 링크 |
| Alpine | busybox 버전 | busybox 버전 (다른 동작) |

Ubuntu에서 `adduser deploy`라고 하면 비밀번호를 묻고 풀네임도 묻고 홈 디렉토리도 알아서 만들어준다. 하지만 같은 명령을 RHEL에서 치면 홈 디렉토리가 없는 계정이 만들어지고 비밀번호도 설정 안 된다(그냥 useradd가 실행되니까). 스크립트에서 양쪽 호환을 원한다면 무조건 `useradd`를 쓰는 게 맞다.

### useradd 동작에 영향을 주는 파일

`useradd`가 어떻게 동작할지는 두 파일이 결정한다.

`/etc/default/useradd`는 `useradd -D`로 보거나 직접 편집한다.

```text
GROUP=100
HOME=/home
INACTIVE=-1
EXPIRE=
SHELL=/bin/sh
SKEL=/etc/skel
CREATE_MAIL_SPOOL=yes
```

여기서 SHELL이 `/bin/sh`로 되어 있는 배포판이 많다. 새 사용자에게 bash를 주고 싶다면 `useradd -s /bin/bash`로 매번 명시하거나 이 파일을 고친다. SKEL은 새 홈 디렉토리에 복사할 템플릿이다. `/etc/skel/.bashrc`를 수정하면 이후 만들어지는 사용자들의 기본 .bashrc가 바뀐다. 이미 만들어진 사용자에게는 영향을 주지 않는다.

`/etc/login.defs`는 더 광범위한 정책을 결정한다.

```text
PASS_MAX_DAYS   90
PASS_MIN_DAYS   1
PASS_MIN_LEN    8
PASS_WARN_AGE   7
UID_MIN         1000
UID_MAX         60000
GID_MIN         1000
GID_MAX         60000
ENCRYPT_METHOD  YESCRYPT
USERGROUPS_ENAB yes
```

`USERGROUPS_ENAB yes`가 켜져 있으면 사용자를 만들 때 같은 이름의 그룹을 자동으로 만든다(UPG, User Private Group 정책). 이 값이 `no`면 모든 새 사용자가 GROUP 변수의 GID(보통 100, users)를 기본 그룹으로 갖는다.

### usermod -aG 사고 사례

운영하다가 한 번씩 누가 docker 그룹에서 사라지는 일이 생긴다. 보통 이 명령 때문이다.

```bash
# 잘못된 사용 - 기존 그룹이 모두 날아간다
usermod -G docker deploy

# 올바른 사용 - 기존 그룹 유지하면서 docker 추가
usermod -aG docker deploy
```

`-G`만 쓰면 보조 그룹 목록을 통째로 교체한다. deploy 사용자가 원래 `wheel`, `sudo`, `adm` 그룹에 있었다면 이 명령 한 줄로 다 빠지고 docker만 남는다. sudo 권한이 wheel을 통해 부여되던 환경이면 deploy는 더 이상 sudo도 못 쓰는 상태가 된다. `-aG`(append)를 반드시 써야 한다.

복구는 다시 `-aG`로 빠진 그룹을 추가하면 되지만, 그 사이에 `groups deploy` 출력을 누가 보고 "이 사용자는 docker만 쓰는구나"라고 판단해서 다른 권한을 줘버리면 일이 커진다. Ansible이나 Chef로 그룹을 관리할 때는 `append: yes` 옵션을 빠뜨리지 말아야 한다.

### newgrp와 세션 재로그인

`usermod -aG docker deploy`로 그룹에 추가했는데 deploy 사용자로 `docker ps`를 치면 여전히 권한 거부가 나는 경우가 있다. 그룹 정보는 로그인할 때 커널의 process credential에 박히기 때문에, 이미 떠 있는 쉘에는 변경 사항이 반영되지 않는다.

세 가지 방법이 있다.

```bash
# 1. 완전히 로그아웃 후 재로그인 (가장 확실하다)
exit
ssh deploy@host

# 2. newgrp로 새 그룹을 활성화한 서브쉘 생성
newgrp docker
docker ps

# 3. sg로 일회성 명령 실행
sg docker -c "docker ps"
```

`newgrp`는 새로운 쉘 프로세스를 띄우면서 그 쉘의 기본 그룹을 지정한 그룹으로 바꾼다. 부모 쉘에는 영향이 없으므로 `exit`로 빠져나오면 원래대로 돌아온다.

## sudoers 심화

### 기본 문법

`/etc/sudoers`의 각 줄은 다음 구조다.

```text
USER  HOST = (RUNAS_USER:RUNAS_GROUP)  TAGS: COMMANDS
```

흔히 보는 `deploy ALL=(ALL:ALL) ALL`을 분해하면

- `deploy`: 누가 (실제 사용자)
- `ALL`: 어떤 호스트에서 (호스트 이름 매칭)
- `(ALL:ALL)`: 누구로/어떤 그룹으로 실행할 수 있나
- `ALL`: 어떤 명령어를

`(ALL:ALL)`의 두 번째 ALL은 `sudo -g <group> command` 형태로 임의의 그룹으로 명령을 실행할 수 있게 한다는 뜻이다. 보통은 `(ALL)`만 써도 충분하다. `(root)`로 명시하면 root로만 실행 가능하고 `sudo -u other`는 막힌다.

### Defaults 지시어

전역 설정은 `Defaults`로 지정한다.

```text
Defaults    env_reset
Defaults    secure_path="/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"
Defaults    timestamp_timeout=15
Defaults    !requiretty
Defaults    logfile=/var/log/sudo.log
Defaults    log_input, log_output
Defaults    iolog_dir=/var/log/sudo-io/%{user}
Defaults:deploy    !requiretty
```

각 옵션의 의미.

| 옵션 | 동작 |
|------|------|
| `env_reset` | sudo 실행 시 환경 변수를 거의 비우고 안전한 것만 통과시킨다. PATH, USER, HOME 등은 새로 설정 |
| `secure_path` | env_reset이 켜져 있을 때 sudo가 사용할 PATH를 지정. `/usr/local/bin`이 빠져 있으면 거기 설치한 명령은 sudo로 못 부른다 |
| `timestamp_timeout` | sudo 인증 후 비밀번호 재입력 없이 쓸 수 있는 시간(분). 0이면 매번 입력, -1이면 세션 동안 무제한 |
| `requiretty` | tty가 있어야만 sudo 사용 허용. 일부 자동화 스크립트가 tty 없이 sudo를 부를 때 `!requiretty`로 꺼야 동작 |
| `logfile`, `log_input`, `log_output` | sudo 실행 로그와 입출력을 파일에 남김. `iolog_dir`은 입출력 로그 저장 디렉토리 |

`Defaults:deploy`처럼 `:사용자명`을 붙이면 특정 사용자에게만 적용한다. `Defaults>root`는 RUNAS 사용자가 root일 때만, `Defaults!CMND_ALIAS`는 특정 명령에만 적용 같은 변형도 있다.

### Alias로 정리

규모가 커지면 사용자/명령/호스트를 묶어서 관리한다.

```text
User_Alias  ADMINS = alice, bob, carol
User_Alias  DEVS = %developers, dave

Cmnd_Alias  SERVICE_CTL = /usr/bin/systemctl start *, \
                          /usr/bin/systemctl stop *, \
                          /usr/bin/systemctl restart *
Cmnd_Alias  LOG_VIEW = /usr/bin/journalctl, /usr/bin/tail /var/log/*

Host_Alias  PROD = prod-web-01, prod-web-02, prod-db-01

ADMINS  ALL = (ALL:ALL) ALL
DEVS    PROD = (deploy) NOPASSWD: SERVICE_CTL, LOG_VIEW
```

위 설정은 ADMINS는 어디서든 모든 권한, DEVS는 PROD 호스트에서만 deploy 사용자로 systemctl/journalctl을 비밀번호 없이 실행할 수 있다는 뜻이다.

### NOPASSWD의 보안 위험

`NOPASSWD`는 비밀번호 입력 없이 sudo를 허용한다. CI/CD나 자동화에 필요할 때가 있지만 잘못 쓰면 위험하다.

```text
# 위험 - 모든 명령에 NOPASSWD
deploy  ALL=(ALL) NOPASSWD: ALL

# 위험 - 인터프리터에 NOPASSWD
deploy  ALL=(ALL) NOPASSWD: /usr/bin/vim, /usr/bin/less
```

vim에 NOPASSWD를 주면 deploy 사용자가 `sudo vim /tmp/x`를 실행한 다음 vim 안에서 `:!bash`를 치면 root 쉘이 열린다. less도 마찬가지로 `!command`로 외부 명령 실행이 가능하다. find의 `-exec`, awk의 `system()`, perl의 `system`도 같은 문제다.

이런 명령들의 목록과 우회 방법은 GTFOBins(`gtfobins.github.io`)에 정리되어 있다. NOPASSWD를 주기 전에 이 사이트에서 해당 명령을 검색해보면 어떤 우회가 가능한지 나온다.

자주 쓰이는 위험 명령.

| 명령 | 우회 방법 |
|------|------|
| vim, vi, nano, emacs | 에디터 내 쉘 명령 (`:!bash`, `!sh`) |
| less, more, man | `!command`로 외부 명령 |
| find | `find . -exec /bin/sh \;` |
| awk, perl, python, ruby | 인터프리터 내 system() 호출 |
| tar | `--checkpoint-action=exec=/bin/sh` |
| systemctl | `systemctl edit`이 EDITOR 호출 |
| git | `git -p` 또는 `git -c core.pager=...`로 페이저 우회 |

NOPASSWD를 줄 때는 단일 동사 명령(`/bin/systemctl restart nginx`처럼 인자까지 고정)으로 한정해야 한다.

### /etc/sudoers.d/ 분리 관리

`/etc/sudoers`를 직접 편집하지 말고 `/etc/sudoers.d/` 디렉토리에 파일로 분리하는 게 일반적이다. `/etc/sudoers`의 마지막 줄에 `@includedir /etc/sudoers.d`(옛 문법은 `#includedir`)가 있어서 그 디렉토리의 모든 파일을 읽는다.

```bash
visudo -f /etc/sudoers.d/deploy
```

```text
# /etc/sudoers.d/deploy
deploy  ALL=(root) NOPASSWD: /usr/bin/systemctl restart myapp
```

분리하는 이유는

- 패키지가 자기 정책을 깔끔히 추가/제거할 수 있다 (`/etc/sudoers.d/jenkins` 같은 파일)
- Ansible 같은 도구가 main 파일을 안 건드리고 새 파일만 떨궈서 관리 가능
- 사고로 메인 sudoers를 망가뜨릴 위험을 줄임

주의: `sudoers.d`의 파일명에 점(`.`)이나 물결(`~`)이 들어가면 sudo가 무시한다. `deploy.bak`이나 `deploy.swp` 같은 파일은 읽히지 않는다는 뜻이지만, vim의 `~` 백업 파일도 마찬가지로 무시되는 게 의도된 동작이다. 파일 권한은 `0440`이어야 한다.

## sudo 권한 검증과 로그 분석

### sudo -l

특정 사용자에게 어떤 sudo 권한이 있는지 확인하려면 그 사용자로 `sudo -l`을 실행한다.

```bash
sudo -l -U deploy
```

```text
User deploy may run the following commands on web-01:
    (root) NOPASSWD: /usr/bin/systemctl restart nginx
    (root) /usr/bin/journalctl
```

권한 위임을 점검할 때 가장 먼저 봐야 할 명령이다. sudoers 파일을 직접 읽지 않아도 효과적인 권한이 무엇인지 알 수 있다.

### 로그 위치

배포판에 따라 다르다.

| 배포판 | 위치 |
|------|------|
| Debian/Ubuntu | `/var/log/auth.log` |
| RHEL/CentOS/Rocky | `/var/log/secure` |
| systemd journal 사용 시 | `journalctl _COMM=sudo` |

```bash
# 최근 sudo 사용 내역
sudo journalctl _COMM=sudo --since "1 hour ago"

# 특정 사용자
sudo journalctl _COMM=sudo | grep "USER=deploy"

# 실패한 sudo (TTY 입력 잘못/비밀번호 틀림)
sudo journalctl _COMM=sudo | grep -E "incorrect password|not in the sudoers"
```

journalctl로 보는 게 편한데, 로그 라인은 보통 다음과 같은 형태다.

```text
sudo: deploy : TTY=pts/0 ; PWD=/home/deploy ; USER=root ; COMMAND=/usr/bin/systemctl restart nginx
```

USER 필드가 RUNAS 사용자(누구로 실행했는지)이고 맨 앞이 실제 호출자다. 사고 조사할 때 누가 언제 어떤 명령을 root로 실행했는지 추적하는 시작점이다.

`Defaults log_input, log_output`이 켜져 있으면 `/var/log/sudo-io/`에 입력 키 시퀀스와 출력 화면이 통째로 저장된다. `sudoreplay`로 재생할 수 있다.

```bash
sudo sudoreplay deploy/00/00/01
```

침해 사고 분석할 때 유용하지만 디스크 사용량이 많아서 모든 호스트에 켜기는 부담스럽다. 보통 점프 호스트나 핵심 서버에만 적용한다.

## 비밀번호 만료 정책

### chage

shadow 파일의 비밀번호 정책 필드를 chage로 관리한다.

```bash
chage -l deploy
```

```text
Last password change                                    : Mar 15, 2026
Password expires                                        : Jun 13, 2026
Password inactive                                       : never
Account expires                                         : never
Minimum number of days between password change          : 0
Maximum number of days between password change          : 90
Number of days of warning before password expires       : 7
```

설정은 다음처럼 한다.

```bash
# 비밀번호 90일 만료, 만료 7일 전 경고
chage -M 90 -W 7 deploy

# 다음 로그인 시 강제 변경 (생일 카드 같은 임시 비밀번호 발급 후)
chage -d 0 deploy

# 계정 자체를 2026-12-31에 만료시킴
chage -E 2026-12-31 deploy

# 만료 정책 제거 (절대 만료 안 됨)
chage -M -1 deploy
```

`chage -d 0`은 비밀번호 마지막 변경일을 1970-01-01로 만들어서 시스템이 "한참 전부터 만료된 상태"로 인식하게 한다. 사용자가 다음 로그인 시 강제로 새 비밀번호를 만들어야 한다. 임시 비밀번호 발급 후 실무에서 자주 쓰는 패턴이다.

`/etc/login.defs`의 정책은 새로 만들어지는 사용자의 기본값이지 기존 사용자에게 자동 적용되는 게 아니다.

```text
PASS_MAX_DAYS   90
PASS_MIN_DAYS   1
PASS_WARN_AGE   7
```

위 값은 새로 만든 사용자가 자동으로 90일 만료, 1일 최소 변경 간격, 7일 전 경고를 갖게 된다는 뜻이다. 이미 있는 사용자에게 일괄 적용하려면 `chage`를 사용자별로 돌려야 한다.

```bash
# 일반 사용자 전체에 정책 적용
awk -F: '$3>=1000 && $3<60000 {print $1}' /etc/passwd | \
  while read u; do chage -M 90 -W 7 "$u"; done
```

## PAM 기본 개념

PAM(Pluggable Authentication Modules)은 인증/세션/계정/비밀번호 관련 동작을 모듈 체인으로 처리하는 프레임워크다. sshd, login, sudo 같은 서비스가 인증을 직접 구현하지 않고 PAM에 위임한다. PAM 설정 파일은 `/etc/pam.d/` 아래 서비스 이름으로 있다.

```bash
ls /etc/pam.d/
# common-auth  common-password  login  sshd  sudo  systemd-user
```

각 파일의 한 줄은 다음 구조다.

```text
TYPE    CONTROL         MODULE                 ARGS
auth    required        pam_unix.so            try_first_pass
auth    requisite       pam_faillock.so        preauth
session required        pam_limits.so
```

TYPE은 `auth`(자격 증명 확인), `account`(계정 유효성), `password`(비밀번호 변경), `session`(로그인 전후 작업) 4종이다. CONTROL은 그 모듈이 실패했을 때 처리(`required`, `requisite`, `sufficient`, `optional`, 또는 `[success=1 default=ignore]` 같은 세밀한 제어)를 정한다.

### pam_faillock으로 잠금 정책

여러 번 비밀번호를 틀리면 계정을 잠그는 정책은 옛날에는 `pam_tally2`로 했지만 RHEL 8/Debian 11 이후로는 `pam_faillock`으로 대체됐다.

`/etc/pam.d/common-auth`(Debian) 또는 `/etc/pam.d/system-auth`(RHEL)에 추가한다.

```text
auth    required    pam_faillock.so preauth silent deny=5 unlock_time=900
auth    [success=1 default=bad]  pam_unix.so
auth    [default=die]    pam_faillock.so authfail deny=5 unlock_time=900
auth    sufficient    pam_faillock.so authsucc
```

`deny=5`는 5회 실패 시 잠금, `unlock_time=900`은 900초(15분) 후 자동 해제. 잠금 상태 확인은 다음과 같다.

```bash
faillock --user deploy
faillock --user deploy --reset    # 수동 해제
```

RHEL은 `authselect`로 PAM 설정을 관리하는 추세라 `/etc/pam.d/`를 직접 고치면 다음 authselect 적용 때 덮어써진다. `authselect select sssd with-faillock` 같은 식으로 프로필을 선택해야 한다.

### pam_limits

`/etc/security/limits.conf` 또는 `/etc/security/limits.d/*.conf`의 리소스 제한이 적용되는 시점이 PAM 세션 단계다. `/etc/pam.d/login`이나 `/etc/pam.d/sshd`에 `session required pam_limits.so`가 있어야 한다. 도커 컨테이너 안에서 `ulimit -n` 값이 `limits.conf`와 다를 때 보통 PAM 모듈이 호출되지 않는 비대화형 환경이라 그렇다.

## 컨테이너와 UID/GID 매핑

### 호스트와 컨테이너 UID 충돌

컨테이너에서 가장 자주 마주치는 권한 문제다. 호스트에 `deploy:1001` 사용자가 있고, 컨테이너 안에는 `appuser:1000` 사용자가 있다고 하자.

```bash
docker run -v /home/deploy/data:/data myimage
```

컨테이너 안의 `appuser`(UID 1000)가 `/data`에 쓰기를 시도하면 호스트의 `/home/deploy/data`(소유자 UID 1001)에 권한이 없어서 실패한다. 반대로 컨테이너에서 root(UID 0)로 파일을 만들면 호스트에서 봤을 때도 root 소유라 deploy가 못 지운다.

해결 방법은 몇 가지가 있다.

첫째, UID를 명시적으로 맞춘다.

```bash
docker run --user 1001:1001 -v /home/deploy/data:/data myimage
```

이미지에 그 UID 사용자가 없어도 동작은 한다(쉘 프롬프트가 `I have no name!`으로 보일 뿐). 다만 이미지 내부에서 그 UID에 쓰기 권한이 있는 디렉토리만 사용 가능해야 한다.

둘째, 빌드 시 UID를 인자로 받는다.

```dockerfile
ARG UID=1001
ARG GID=1001
RUN groupadd -g $GID app && useradd -u $UID -g $GID -m app
USER app
```

```bash
docker build --build-arg UID=$(id -u) --build-arg GID=$(id -g) -t myimage .
```

개발 환경에서 호스트 사용자와 일치시킬 때 쓰는 패턴이다. 운영에서는 모든 호스트가 같은 UID를 갖도록 표준화하는 편이 낫다.

### rootless 컨테이너와 subuid/subgid

Podman이나 rootless Docker는 사용자 네임스페이스를 사용해 컨테이너 내부의 root(UID 0)를 호스트의 일반 사용자 UID에 매핑한다. 이 매핑 정보가 `/etc/subuid`와 `/etc/subgid`에 있다.

```text
# /etc/subuid
deploy:100000:65536
```

이 줄은 "deploy 사용자가 컨테이너 내부 UID 0~65535를 호스트 UID 100000~165535에 매핑할 수 있다"는 뜻이다. 즉 rootless 컨테이너 안에서 root로 파일을 만들면 호스트에서는 100000번 UID 소유로 보인다.

`useradd`는 기본적으로 `/etc/subuid`와 `/etc/subgid`에 65536개씩 자동 할당한다(`useradd.conf`의 `SUB_UID_COUNT`, `SUB_GID_COUNT`). 시스템 계정으로 만든 경우 자동 할당 안 되므로 수동 추가가 필요하다.

```bash
usermod --add-subuids 100000-165535 --add-subgids 100000-165535 deploy
```

확인은 `cat /etc/subuid /etc/subgid`. rootless 컨테이너가 권한 오류를 내면 거의 이 매핑이 빠졌거나 범위가 겹치는 경우다.

## visudo 문법 오류 복구

`visudo`는 저장 직전에 문법 검사를 한다. 문법 오류가 있으면 다시 편집할 기회를 준다.

```text
>>> /etc/sudoers.tmp: syntax error near line 30 <<<
What now?
```

`e`로 다시 편집, `x`로 변경 사항 버리고 종료, `Q`로 강제 저장(절대 누르지 말 것). 이게 visudo를 써야 하는 이유다. 직접 `vim /etc/sudoers`로 편집하면 검사 없이 저장되어, 저장 후 sudo 자체가 동작하지 않는 사태가 발생할 수 있다.

문법 오류로 sudo가 망가졌을 때 복구 방법.

### 1. 다른 root 세션이 살아 있을 때

가장 운이 좋은 경우다. 이미 열려 있는 root 쉘에서 `visudo -c`로 검증하고 수정한다.

```bash
visudo -c    # 문법 검사만
visudo       # 편집 후 검증
```

### 2. pkexec 사용

PolicyKit이 설치된 시스템에서 sudo 없이 root 권한을 얻는 또 다른 경로다. sudo가 망가져도 pkexec는 살아 있다.

```bash
pkexec visudo
```

비밀번호를 물으면 입력. 데스크탑 환경에서는 GUI 다이얼로그가 뜨고, 서버라면 콘솔에서 입력받는다. 단 호출 사용자가 `wheel` 그룹(또는 PolicyKit 정책상 권한을 가진 그룹)에 있어야 한다.

### 3. 단일 사용자 모드 (single user mode)

위 방법이 모두 안 되면 재부팅 후 단일 사용자 모드로 들어간다.

GRUB 부트 메뉴에서 `e`를 눌러 커널 라인 편집, `linux ...` 줄 끝에 `single` 또는 `init=/bin/bash`를 추가, `Ctrl-X`로 부팅. 루트 파티션이 읽기 전용으로 마운트되어 있을 수 있으므로 다음을 먼저 해야 한다.

```bash
mount -o remount,rw /
visudo
mount -o remount,ro /
exec /sbin/init
```

systemd 환경에서는 `systemd.unit=rescue.target`이나 `systemd.unit=emergency.target`으로 부팅한다. emergency가 더 최소 환경이다.

### 4. 라이브 USB로 마운트

물리/가상 콘솔 접근도 안 되는 원격 서버는 실질적으로 복구가 어렵다. 클라우드 환경이라면 인스턴스의 디스크를 분리해서 다른 인스턴스에 붙인 다음 `/etc/sudoers`를 직접 편집하는 게 마지막 수단이다. 그래서 처음부터 visudo를 쓰는 게 중요하다.

## 권한 비트 (chmod, chown) 핵심

### chmod 표기법

숫자 표기와 기호 표기 둘 다 쓴다.

```bash
chmod 755 script.sh        # rwxr-xr-x
chmod 644 file             # rw-r--r--
chmod 600 ~/.ssh/id_rsa    # rw-------

chmod u+x script.sh        # 소유자에 실행 권한 추가
chmod g-w file             # 그룹의 쓰기 권한 제거
chmod o= file              # other 권한 모두 제거
chmod a+r file             # 모든 사용자에 읽기 권한
```

특수 비트.

| 비트 | 숫자 | 의미 |
|------|------|------|
| setuid | 4000 | 실행 시 파일 소유자 권한으로 동작 (`/usr/bin/passwd`가 대표) |
| setgid | 2000 | 실행 시 그룹 권한으로 동작, 디렉토리에 걸면 그 안에 만들어진 파일이 부모 디렉토리 그룹을 상속 |
| sticky | 1000 | 디렉토리에 걸면 자기 소유 파일만 삭제 가능 (`/tmp`가 대표) |

```bash
chmod 4755 /usr/local/bin/myapp    # setuid
chmod 2775 /var/shared             # setgid + 그룹 쓰기
chmod 1777 /tmp                    # sticky
```

### chown

소유자와 그룹을 변경한다. root만 가능(소유자는 자기 그룹은 바꿀 수 있다).

```bash
chown deploy file
chown deploy:deploy file
chown -R deploy:deploy /var/lib/myapp
chown --reference=template.conf newfile    # 다른 파일과 같은 소유자/그룹으로
```

`-R`은 재귀이고, 심볼릭 링크 처리는 `-L`(링크 따라감), `-P`(링크 자체)로 제어한다. 운영 디렉토리에 `chown -R`을 칠 때는 와일드카드를 잘못 잡거나 디렉토리를 잘못 지정해서 `/`를 통째로 바꾸는 사고가 자주 난다. 명령 실행 전에 `pwd`와 `ls`를 한 번 더 확인하는 습관이 필요하다.

### umask

새로 만들어지는 파일/디렉토리의 기본 권한에서 빼는 마스크다.

```bash
umask        # 현재 값 (보통 0022)
umask 0077   # 그룹/other 권한 모두 제거
```

umask 0022면 새 파일은 `666 - 022 = 644`, 새 디렉토리는 `777 - 022 = 755`로 만들어진다. 멀티 유저 서버에서 다른 사용자가 내 파일을 못 읽게 하려면 셸 시작 파일에 `umask 0077`을 넣는다. 시스템 전역은 `/etc/login.defs`의 `UMASK` 값.

## SSH 키 기반 인증

비밀번호 기반 ssh는 무차별 대입에 취약하므로 운영 서버는 보통 키 기반으로만 둔다.

```bash
ssh-keygen -t ed25519 -C "deploy@laptop"
ssh-copy-id -i ~/.ssh/id_ed25519.pub deploy@server
```

`ssh-copy-id`는 원격의 `~/.ssh/authorized_keys`에 공개키를 추가하고 권한도 적절히(700, 600) 맞춰준다. 수동으로 해야 한다면 권한이 중요하다.

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
```

`~/.ssh`나 `authorized_keys`의 권한이 너무 열려 있으면 sshd가 무시한다(StrictModes). `/var/log/auth.log`에 "Authentication refused: bad ownership or modes"라고 찍힌다. 이때 위 권한을 다시 잡으면 풀린다.

`/etc/ssh/sshd_config`에서 `PasswordAuthentication no`로 설정하고 `PermitRootLogin prohibit-password`(키만 허용)로 두는 게 표준이다. 변경 후 `systemctl reload sshd`. 절대로 본인 키 등록 전에 PasswordAuthentication을 끄면 안 된다. 동시에 두 개의 ssh 세션을 열어두고 한 쪽에서 설정 바꾸고 다른 쪽으로 검증하는 패턴이 안전하다.
