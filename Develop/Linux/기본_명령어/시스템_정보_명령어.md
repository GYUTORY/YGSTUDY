# 시스템 정보 명령어

## 개요

운영 서버에 SSH로 접속하면 가장 먼저 확인하는 게 "여기가 어디고, 나는 누구이며, 지금 몇 시인가" 세 가지다. 호스트가 잘못됐는데 명령어 실행했다가 운영 DB에 영향을 준 사고는 신입 시절 누구나 한 번쯤 겪는다. SSH 프롬프트가 바뀌었어도 항상 `hostname`과 `whoami`로 한 번 더 확인하는 습관이 있어야 한다.

여기서 다루는 명령어들은 단독으로는 단순하지만 셸 스크립트에서 OS 분기 처리, 로그 파일 이름에 타임스탬프 박기, 부팅 직후 상태 진단 같은 곳에서 거의 매일 쓰인다. 단순해 보여도 `who am i`와 `whoami`처럼 미묘한 차이가 있는 명령어들이 있고, 이걸 모르고 쓰다가 sudo 환경에서 의도와 다르게 동작하는 경우가 있다.

## uname - 커널과 아키텍처 정보

`uname`은 커널 정보를 보여준다. 시스템이 리눅스인지 macOS인지, 어떤 아키텍처(x86_64, aarch64, armv7l)인지 확인할 때 쓴다. 셸 스크립트에서 OS 분기 처리할 때 가장 먼저 떠올리는 게 이 명령어다.

```bash
# 옵션 없이 쓰면 커널 이름만 나온다
uname
# Linux

# -a (all): 가능한 모든 정보
uname -a
# Linux web-prod-01 5.15.0-91-generic #101-Ubuntu SMP Tue Nov 14 13:30:08 UTC 2023 x86_64 x86_64 x86_64 GNU/Linux

# -s: 커널 이름 (Linux, Darwin, FreeBSD)
uname -s
# Linux

# -r: 커널 릴리스 버전
uname -r
# 5.15.0-91-generic

# -m: 머신 아키텍처
uname -m
# x86_64

# -n: 호스트네임 (hostname과 동일)
uname -n
```

실무에서 가장 많이 쓰는 게 `uname -s`와 `uname -m`이다. macOS와 Linux 모두에서 동작해야 하는 스크립트를 짤 때 OS별로 다른 처리를 해야 하는 경우가 있다.

### uname으로 OS 분기 처리

```bash
#!/bin/bash
# 셸 스크립트에서 OS 분기

OS=$(uname -s)

case "$OS" in
    Linux*)
        STAT_CMD="stat -c %Y"  # GNU stat
        ;;
    Darwin*)
        STAT_CMD="stat -f %m"  # BSD stat (macOS)
        ;;
    *)
        echo "지원하지 않는 OS: $OS" >&2
        exit 1
        ;;
esac

# 사용
mtime=$($STAT_CMD /tmp/somefile)
```

이게 왜 필요한가 하면, GNU 계열(Linux)과 BSD 계열(macOS)의 `stat`, `sed`, `date`, `xargs` 같은 명령어들이 옵션 체계가 다르다. 같은 작업인데 옵션이 안 통해서 스크립트가 깨지는 일이 흔하다. macOS에서 잘 돌던 스크립트가 운영 서버(Linux)에 올라가서 망가지는 패턴이다.

### 아키텍처 확인

ARM 서버가 늘어나면서 `uname -m`도 더 자주 쓰이게 됐다. AWS Graviton, Apple Silicon, 라즈베리파이 등을 지원해야 하는 빌드 스크립트에서는 아키텍처별로 바이너리를 받아야 한다.

```bash
ARCH=$(uname -m)

case "$ARCH" in
    x86_64)  ARCH_TAG="amd64" ;;
    aarch64) ARCH_TAG="arm64" ;;
    armv7l)  ARCH_TAG="armv7" ;;
    *)
        echo "Unsupported arch: $ARCH" >&2
        exit 1
        ;;
esac

curl -L "https://example.com/release/myapp-${ARCH_TAG}.tar.gz" -o myapp.tar.gz
```

여기서 함정이 있는데, macOS Apple Silicon에서 `uname -m`은 `arm64`를 반환하지만 Linux에서는 같은 ARM64 아키텍처가 `aarch64`로 나온다. 같은 칩 계열인데 이름이 다르다. 크로스 플랫폼 스크립트를 짤 때 이 두 값을 모두 처리해야 한다.

## hostname - 호스트네임 확인과 변경

```bash
# 현재 호스트네임
hostname
# web-prod-01

# FQDN (Fully Qualified Domain Name)
hostname -f
# web-prod-01.example.com

# IP 주소 (DNS 역방향 조회 결과)
hostname -i
# 192.168.1.10

# 도메인만
hostname -d
# example.com
```

`hostname -i`는 의외로 신뢰할 수 없다. `/etc/hosts`에 어떻게 적혀 있느냐에 따라 결과가 달라지고, 멀티 NIC 환경에서는 의도와 다른 IP가 나온다. 실제 IP 확인은 `ip addr show`나 `ip route get 1.1.1.1`을 쓰는 게 정확하다.

호스트네임 변경은 일시적/영구적 두 가지 방식이 있는데, 잘못 건드리면 sudo 사용 시 `sudo: unable to resolve host` 같은 경고가 뜨면서 매번 1~2초씩 느려진다.

```bash
# 일시 변경 (재부팅하면 사라짐)
sudo hostname new-host-name

# 영구 변경 (systemd 기반 배포판)
sudo hostnamectl set-hostname new-host-name

# /etc/hosts도 같이 수정해야 한다
sudo vim /etc/hosts
# 127.0.1.1   new-host-name
```

운영에서 호스트네임을 바꿀 일은 거의 없지만, 이미지로 떠진 VM을 복제해서 쓸 때 모두 같은 이름이라 충돌이 나는 경우가 있다. 이때 `hostnamectl set-hostname`과 `/etc/hosts`를 같이 수정해야 한다.

## whoami와 id - 내가 누구인가

```bash
whoami
# deploy

id
# uid=1001(deploy) gid=1001(deploy) groups=1001(deploy),27(sudo),114(docker)

# 특정 사용자 정보
id postgres
# uid=999(postgres) gid=999(postgres) groups=999(postgres)

# 숫자 ID만
id -u           # 자기 UID
id -g           # 자기 primary GID
id -G           # 모든 그룹 GID 목록
id -un          # 사용자 이름 (whoami와 동일)
id -gn          # primary 그룹 이름
```

`whoami`는 단순히 사용자명만 알려주지만, `id`는 그룹 정보까지 다 보여준다. 권한 문제 디버깅할 때 `id`가 훨씬 유용하다. 예를 들어 도커 소켓에 접근이 안 될 때, 사용자가 `docker` 그룹에 들어 있는지 `id`로 먼저 확인한다.

```bash
# Docker 권한 디버깅
id
# uid=1001(deploy) gid=1001(deploy) groups=1001(deploy)
# docker 그룹이 없네 → 그래서 docker 명령어가 안 먹는다

sudo usermod -aG docker deploy
# 그룹 추가 후에는 재로그인 또는 newgrp docker 필요
```

여기서 자주 부딪히는 함정: `usermod -aG`로 그룹을 추가해도 현재 셸 세션에는 반영이 안 된다. 새로 로그인하거나 `newgrp docker`를 실행해야 한다. `id`로 확인했을 때 그룹이 있는데도 권한이 없다면 셸 세션을 다시 띄우면 해결된다.

### root 권한 체크 패턴

스크립트에서 root 권한이 필요한지 확인할 때 `whoami`보다 `id -u`로 비교하는 게 정석이다.

```bash
#!/bin/bash
# root 권한 체크

if [ "$(id -u)" -ne 0 ]; then
    echo "이 스크립트는 root 권한이 필요합니다" >&2
    exit 1
fi
```

`whoami` 결과로 `"root"` 문자열을 비교하는 코드도 봤는데, 컨테이너나 다른 환경에서 root 사용자명이 다른 경우가 거의 없긴 해도 UID 0으로 비교하는 게 표준이다.

### who am i와 whoami의 차이

비슷해 보이는 두 명령어인데 동작이 완전히 다르다.

```bash
# whoami: 현재 effective user
whoami
# deploy

# who am i: 로그인 세션의 원본 사용자
who am i
# deploy   pts/0   2026-04-30 09:15 (192.168.1.5)
```

이 차이가 결정적으로 드러나는 게 `sudo su -`를 한 다음이다.

```bash
deploy@host:~$ whoami
deploy

deploy@host:~$ sudo su -

root@host:~# whoami
root
# 현재는 root로 동작 중

root@host:~# who am i
deploy   pts/0   2026-04-30 09:15 (192.168.1.5)
# 원래 deploy로 로그인했었다는 사실은 남아있다
```

`who am i`는 utmp 기반이라 원본 로그인 세션 정보를 보여준다. 누가 root로 권한 상승을 했는지 추적할 때 유용하다. 감사 로그를 남기는 스크립트에서 "현재 작업 권한"과 "실제로 누가 시작한 작업인지"를 구분해서 기록할 때 둘을 같이 쓴다.

```bash
ORIG_USER=$(who am i | awk '{print $1}')
CURR_USER=$(whoami)

logger -t myapp "권한 상승: ${ORIG_USER} → ${CURR_USER}"
```

다만 `who am i`는 cron이나 시스템 서비스처럼 utmp에 등록되지 않은 환경에서는 빈 결과를 반환한다. 그래서 항상 결과가 있다고 가정하면 안 된다.

## who와 w - 누가 접속해 있나

`who`는 현재 로그인된 사용자 목록을 보여준다. `w`는 같은 정보에 더해서 그 사용자가 무엇을 하고 있는지까지 알려준다.

```bash
who
# deploy   pts/0   2026-04-30 09:15 (192.168.1.5)
# admin    pts/1   2026-04-30 10:42 (10.0.0.3)
# admin    pts/2   2026-04-30 11:01 (10.0.0.3)

w
#  11:23:45 up 14 days,  3:21,  3 users,  load average: 0.15, 0.10, 0.08
# USER     TTY      FROM             LOGIN@   IDLE   JCPU   PCPU WHAT
# deploy   pts/0    192.168.1.5      09:15    1:45m  0.05s  0.05s -bash
# admin    pts/1    10.0.0.3         10:42    0.00s  0.10s  0.10s w
# admin    pts/2    10.0.0.3         11:01   12:30   2:34   2:30  vim app.conf
```

`w`의 첫 줄은 `uptime`과 동일하다. 그 아래 항목 중 자주 보는 건 `IDLE`(마지막 입력 후 경과시간), `WHAT`(현재 실행 중인 명령어)이다. 운영 중인 서버에 들어갔는데 다른 사람이 `vim`으로 설정 파일을 열어두고 있다면 동시 편집을 피해야 한다.

```bash
# 누가 어떤 작업을 하는지 확인
w admin
# admin     pts/1    10.0.0.3       10:42    0.00s  ...
# admin     pts/2    10.0.0.3       11:01   12:30s  ... vim app.conf
```

운영 사고로 다 같이 서버에 몰릴 때 `w`로 누가 어떤 작업을 하는지 보고 역할 분담을 한다. 한 사람이 로그를 보는 동안 다른 사람이 같은 명령어를 또 치는 건 시간 낭비고, 더 나아가 의도치 않은 변경을 만들 수 있다.

## last - 로그인 이력과 부팅 이력

`last`는 `/var/log/wtmp`를 읽어서 로그인/로그아웃 이력을 보여준다. 누가 언제 들어왔는지, 시스템이 언제 부팅됐는지 다 나온다.

```bash
# 최근 로그인 이력
last
# deploy   pts/0    192.168.1.5      Tue Apr 30 09:15   still logged in
# admin    pts/2    10.0.0.3         Tue Apr 30 11:01 - 11:45  (00:44)
# reboot   system boot 5.15.0-91     Tue Apr 16 08:02   still running
# ...

# 최근 N개만
last -n 10

# 특정 사용자
last deploy

# 부팅 이력만
last reboot
# reboot   system boot 5.15.0-91     Tue Apr 16 08:02   still running
# reboot   system boot 5.15.0-89     Mon Mar 18 14:30 - 08:01 (28+17:31)
# reboot   system boot 5.15.0-87     Sat Feb 10 02:11 - 14:29 (36+12:18)
```

`last reboot`는 시스템이 언제 재부팅됐는지 파악할 때 가장 직관적이다. 운영 사고 직후 "혹시 서버가 갑자기 재부팅된 거 아니냐"는 질문에 답할 때 이걸 본다.

```bash
# 현재 부팅 시점만
last reboot -n 1
# reboot   system boot 5.15.0-91     Tue Apr 16 08:02   still running

# 비정상 종료 이력
last -x | grep shutdown
last -x | grep crash
```

`-x` 옵션은 시스템 종료, 런레벨 변경 같은 이벤트도 보여준다. 갑자기 죽은 서버를 분석할 때 정상 종료(`shutdown`)인지 비정상 종료(crash)인지 구분에 쓴다.

## date - 시간 출력과 포맷팅

`date`는 단순한 시간 출력 명령어처럼 보이지만, 포맷 문자열이 핵심이다. 로그 파일 이름, 백업 파일 이름, 스크립트 내부 타임스탬프 등에서 거의 무조건 쓴다.

```bash
# 기본 출력
date
# Tue Apr 30 11:23:45 KST 2026

# UTC 시간
date -u
# Tue Apr 30 02:23:45 UTC 2026

# ISO 8601
date -Iseconds
# 2026-04-30T11:23:45+09:00

# Unix 타임스탬프
date +%s
# 1777594025

# Unix 타임스탬프를 사람이 읽는 형식으로
date -d @1777594025
# Tue Apr 30 11:23:45 KST 2026
```

### 자주 쓰는 포맷 문자열

```bash
# 백업 파일 이름에 박는 패턴
date +%Y%m%d_%H%M%S
# 20260430_112345

# 로그 디렉토리 분리
date +%Y/%m/%d
# 2026/04/30

# RFC 3339
date +%Y-%m-%dT%H:%M:%S%z
# 2026-04-30T11:23:45+0900

# 사람이 읽기 좋은 한국 포맷
date "+%Y년 %m월 %d일 %H시 %M분"
# 2026년 04월 30일 11시 23분
```

실무에서 `+%Y%m%d_%H%M%S`는 거의 외워서 쓴다. 백업 스크립트에서 가장 많이 쓰는 패턴이다.

```bash
#!/bin/bash
# DB 백업 파일에 타임스탬프 박기

TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="/backup/db_${TIMESTAMP}.sql.gz"

pg_dump mydb | gzip > "$BACKUP_FILE"
echo "백업 완료: $BACKUP_FILE"
# 백업 완료: /backup/db_20260430_112345.sql.gz
```

### 날짜 계산

`date -d`(GNU 전용, BSD에서는 `date -v`)로 상대적 시간 계산이 가능하다.

```bash
# 어제 (GNU date)
date -d "yesterday" +%Y-%m-%d
# 2026-04-29

# 7일 전
date -d "7 days ago" +%Y-%m-%d
# 2026-04-23

# 다음달 첫째날
date -d "next month" +%Y-%m-01
# 2026-05-01

# 특정 시점
date -d "2026-01-01 10:30:00" +%s
# 1767230400
```

```bash
# macOS (BSD date)는 -v 옵션
date -v-1d +%Y-%m-%d        # 어제
date -v-7d +%Y-%m-%d        # 7일 전
date -v-1m +%Y-%m-%d        # 한 달 전
```

이게 GNU와 BSD가 완전히 다른 대표적인 명령어다. macOS에서 짠 스크립트를 운영(Linux)에 올리면 `date -d` 부분이 깨지고, 반대로 운영에서 짠 스크립트를 macOS에서 돌려도 깨진다. 크로스 플랫폼 스크립트라면 `uname -s`로 분기 처리해야 한다.

### 시스템 시간 변경

운영 서버에서 시간이 틀어지면 큰 문제다. 인증서 만료, JWT 토큰, 로그 정렬, DB 트랜잭션 순서까지 영향을 받는다. 직접 `date -s`로 바꾸는 건 권장하지 않고, NTP를 통해 동기화하는 게 정석이다.

```bash
# 직접 변경 (root 권한, 비권장)
sudo date -s "2026-04-30 11:23:45"

# systemd 기반 시스템에서 시간 동기화 상태 확인
timedatectl
#                Local time: Tue 2026-04-30 11:23:45 KST
#            Universal time: Tue 2026-04-30 02:23:45 UTC
#                  RTC time: Tue 2026-04-30 02:23:45
#                 Time zone: Asia/Seoul (KST, +0900)
# System clock synchronized: yes
#               NTP service: active

# 타임존 변경
sudo timedatectl set-timezone Asia/Seoul
```

NTP가 꺼져 있거나 동기화가 안 된 서버는 시간이 점점 틀어지는데, 보통 클라우드 인스턴스에서는 호스트와 자동 동기화되지만 온프레미스 서버에서는 직접 chrony나 systemd-timesyncd를 띄워야 한다.

## uptime - 부팅 후 경과 시간과 부하

```bash
uptime
#  11:23:45 up 14 days,  3:21,  3 users,  load average: 0.15, 0.10, 0.08
```

읽는 법은 이렇다.

- `11:23:45`: 현재 시각
- `up 14 days, 3:21`: 부팅 후 14일 3시간 21분 경과
- `3 users`: 현재 로그인된 사용자 수
- `load average: 0.15, 0.10, 0.08`: 최근 1분, 5분, 15분 평균 부하

load average는 CPU 코어 수와 비교해서 봐야 한다. 4코어 서버에서 load average가 1.0이면 25% 가량 사용 중이고, 4.0을 넘기면 작업 큐에 대기가 쌓이고 있다는 뜻이다. CPU 코어 수는 `nproc` 또는 `cat /proc/cpuinfo | grep -c processor`로 확인한다.

```bash
# 부팅 시점만 출력
uptime -s
# 2026-04-16 08:02:24

# 사람이 읽기 좋은 형식
uptime -p
# up 14 days, 3 hours, 21 minutes
```

`uptime -s`는 부팅 시점이 정확히 언제였는지 알 때 유용하다. `last reboot`보다 직관적이다. 자동화 스크립트에서 부팅 시점이 N시간 이내인지 확인할 때 이걸 쓴다.

```bash
#!/bin/bash
# 최근 1시간 이내 부팅됐는지 체크

BOOT_EPOCH=$(date -d "$(uptime -s)" +%s)
NOW=$(date +%s)
ELAPSED=$((NOW - BOOT_EPOCH))

if [ $ELAPSED -lt 3600 ]; then
    echo "최근 1시간 이내 부팅됨 — 워밍업 대기 필요"
    sleep 60
fi
```

부팅 직후에는 캐시가 비어 있고, JIT 워밍업이 안 된 상태라 일부 서비스의 응답이 느리다. 헬스체크 스크립트에서 부팅 직후에는 임계치를 완화하는 패턴으로 쓴다.

## cal - 달력

단순한 명령어지만 가끔 유용하다. 며칠 후의 요일을 알아볼 때, 분기 마감일 계산할 때 쓴다.

```bash
# 이번 달
cal

# 이번 년 전체
cal -y

# 특정 년월
cal 5 2026

# 3개월 (전월, 이번 달, 다음 달)
cal -3
```

운영자 입장에서 자주 보는 건 다음 주 토요일이 며칠인지 같은 거다. 점검 일정을 잡을 때 쓴다. 별로 자주 쓰진 않지만 외우고 있으면 가끔 빠르게 답할 수 있다.

## OS 배포판 식별 - lsb_release와 /etc/os-release

`uname -s`로는 Linux인지 macOS인지만 알 수 있고, 어떤 배포판(Ubuntu, CentOS, Debian)인지는 알 수 없다. 패키지 매니저를 분기 처리하거나 배포판별로 다른 라이브러리를 설치해야 할 때 배포판 식별이 필요하다.

### lsb_release

```bash
# 모든 정보
lsb_release -a
# Distributor ID: Ubuntu
# Description:    Ubuntu 22.04.3 LTS
# Release:        22.04
# Codename:       jammy

# 배포판 ID만
lsb_release -i
# Distributor ID: Ubuntu

# 버전만 (스크립트에서 자주 쓰는 형태)
lsb_release -rs
# 22.04

lsb_release -is
# Ubuntu

lsb_release -cs
# jammy
```

`-s` 옵션은 헤더 없이 값만 출력한다. 스크립트에서 변수로 받을 때 깔끔하다.

문제는 `lsb_release`가 미니멀 이미지나 일부 배포판에는 기본 설치가 안 되어 있다는 점이다. Alpine, 일부 Docker 이미지, 최소 설치 CentOS 등에서는 `command not found`가 뜬다. 그래서 더 안정적인 방법은 `/etc/os-release`를 읽는 것이다.

### /etc/os-release

거의 모든 현대 리눅스 배포판이 `/etc/os-release` 파일을 가지고 있다. systemd 표준이라 의존할 만하다.

```bash
cat /etc/os-release
# PRETTY_NAME="Ubuntu 22.04.3 LTS"
# NAME="Ubuntu"
# VERSION_ID="22.04"
# VERSION="22.04.3 LTS (Jammy Jellyfish)"
# VERSION_CODENAME=jammy
# ID=ubuntu
# ID_LIKE=debian
# HOME_URL="https://www.ubuntu.com/"
# ...
```

`/etc/os-release`는 셸 스크립트 형식이라 `source`로 읽으면 변수로 바로 쓸 수 있다.

```bash
#!/bin/bash
# OS 정보를 변수로 로드

. /etc/os-release

echo "OS: $NAME"
echo "Version: $VERSION_ID"
echo "ID: $ID"

# 패키지 매니저 분기
case "$ID" in
    ubuntu|debian)
        sudo apt update && sudo apt install -y nginx
        ;;
    centos|rhel|rocky|almalinux)
        sudo yum install -y nginx
        ;;
    fedora)
        sudo dnf install -y nginx
        ;;
    alpine)
        sudo apk add nginx
        ;;
    *)
        echo "지원하지 않는 배포판: $ID" >&2
        exit 1
        ;;
esac
```

`ID_LIKE`도 활용도가 높다. Ubuntu는 `ID_LIKE=debian`, Rocky Linux는 `ID_LIKE="rhel centos fedora"`처럼 부모 배포판을 명시한다. 직접 매칭이 안 될 때 `ID_LIKE`로 폴백할 수 있다.

```bash
. /etc/os-release

if [ "$ID" = "debian" ] || [[ " $ID_LIKE " == *" debian "* ]]; then
    PKG_MANAGER="apt"
elif [ "$ID" = "rhel" ] || [[ " $ID_LIKE " == *" rhel "* ]]; then
    PKG_MANAGER="yum"
fi
```

### 종합 OS 식별 함수

크로스 플랫폼 스크립트를 자주 짠다면 OS 식별 함수를 하나 만들어두면 편하다.

```bash
#!/bin/bash

detect_os() {
    local os arch distro version

    os=$(uname -s)
    arch=$(uname -m)

    case "$os" in
        Linux)
            if [ -f /etc/os-release ]; then
                . /etc/os-release
                distro="$ID"
                version="$VERSION_ID"
            elif command -v lsb_release >/dev/null 2>&1; then
                distro=$(lsb_release -is | tr '[:upper:]' '[:lower:]')
                version=$(lsb_release -rs)
            else
                distro="unknown"
                version="unknown"
            fi
            ;;
        Darwin)
            distro="macos"
            version=$(sw_vers -productVersion)
            ;;
        *)
            distro="unknown"
            version="unknown"
            ;;
    esac

    echo "${os}/${distro}/${version}/${arch}"
}

detect_os
# Linux/ubuntu/22.04/x86_64
# 또는
# Darwin/macos/14.4/arm64
```

`/etc/os-release`를 우선 읽고, 없으면 `lsb_release`로 폴백하고, 둘 다 없으면 `unknown`으로 처리한다. macOS는 `sw_vers`라는 별도 명령어가 있다.

## 커널 버전과 부팅 파라미터

```bash
# 커널 버전
uname -r
# 5.15.0-91-generic

# 빌드 정보까지
cat /proc/version
# Linux version 5.15.0-91-generic (buildd@lcy02-amd64-085) (gcc (Ubuntu 11.4.0-1ubuntu1~22.04) ...

# 부팅 시 커널 파라미터
cat /proc/cmdline
# BOOT_IMAGE=/boot/vmlinuz-5.15.0-91-generic root=UUID=... ro quiet
```

`/proc/cmdline`은 잘 안 보지만, 커널 파라미터가 의도대로 들어갔는지 확인할 때 쓴다. 예를 들어 hugepages 설정, IOMMU 활성화 같은 것들이 부팅 시점에 적용되는데, 정상적으로 적용됐는지 여기서 확인할 수 있다.

## 정리

이 명령어들은 단순해 보여도 운영 환경에서 매일 쓰인다. 가장 중요한 건 SSH 접속 직후 `hostname`, `whoami`, `date`로 환경을 한 번 확인하는 습관이다. 잘못된 서버에서 명령어 실행해서 사고를 내는 것보다 1초 더 쓰는 게 훨씬 낫다.

스크립트 작성 시에는 `uname -s`로 OS 분기, `/etc/os-release`로 배포판 분기를 함께 쓰는 게 표준이다. macOS와 Linux의 명령어 차이(GNU vs BSD)는 매번 부딪히는 함정이라 처음부터 분기 처리를 염두에 두고 짜야 나중에 고생을 덜 한다. `date` 포맷 문자열 `+%Y%m%d_%H%M%S`는 거의 외워서 쓰는 패턴이고, `who am i`와 `whoami`의 차이는 권한 상승 환경에서 디버깅할 때 의외로 자주 나오는 포인트다.
