# 매뉴얼 및 도움말 찾기

## 개요

실무에서 명령어 옵션을 모두 외우는 사람은 없다. `tar`만 해도 옵션이 100개가 넘고, `find`는 표현식이 따로 있는 수준이라 그때그때 찾아 쓰는 게 정상이다. 문제는 어디서 어떻게 찾느냐다. 신입 시절에는 무조건 구글링부터 하는 경우가 많은데, 실제로는 로컬에 있는 매뉴얼이 가장 정확하고 빠르다. 운영 서버에서 인터넷이 막혀 있는 경우도 흔하기 때문에 `man`과 `--help`만 잘 다뤄도 큰 문제는 거의 다 해결된다.

문서 시스템은 크게 세 갈래로 나뉜다. 전통적인 `man` 페이지, GNU 프로젝트가 밀던 `info`, 그리고 명령어 자체가 제공하는 `--help`다. 여기에 키워드로 명령어를 거꾸로 찾는 `apropos`, 명령어의 실체를 확인하는 `type`/`which`, 그리고 요즘 많이 쓰이는 `tldr` 같은 외부 도구가 더해진다. 각 도구가 어떤 상황에서 빛을 발하는지 알아야 시간을 낭비하지 않는다.

## man 페이지의 구조

`man`은 manual의 줄임말이고, 1971년 유닉스 초창기부터 있던 시스템이다. 모든 매뉴얼은 섹션으로 구분되어 있는데, 이게 처음 보면 헷갈린다. `man passwd`를 입력했을 때 명령어 매뉴얼이 나오는지 설정 파일 매뉴얼이 나오는지가 섹션에 따라 다르다.

### 섹션 번호 의미

| 섹션 | 내용 | 예시 |
|------|------|------|
| 1 | 일반 사용자 명령어 | `ls`, `cp`, `grep` |
| 2 | 시스템 콜 | `open`, `read`, `fork` |
| 3 | 라이브러리 함수 | `printf`, `malloc` |
| 4 | 디바이스 파일 | `tty`, `null` |
| 5 | 파일 포맷, 설정 파일 | `passwd`, `fstab`, `crontab` |
| 6 | 게임 | `fortune` |
| 7 | 기타 (프로토콜, 규약) | `signal`, `regex` |
| 8 | 시스템 관리자 명령어 | `mount`, `iptables` |
| 9 | 커널 루틴 (비표준) | - |

여기서 자주 부딪히는 게 같은 이름이 여러 섹션에 있는 경우다. `passwd`가 대표적인데, 섹션 1은 비밀번호 변경 명령어, 섹션 5는 `/etc/passwd` 파일 포맷이다. 그냥 `man passwd`를 치면 1번 섹션이 먼저 나오고, 파일 포맷이 보고 싶으면 `man 5 passwd`로 섹션을 명시해야 한다.

```bash
# 섹션 1 (명령어)
man passwd

# 섹션 5 (파일 포맷)
man 5 passwd

# 모든 섹션에 있는 같은 이름의 매뉴얼 찾기
man -a passwd
# Enter로 다음 섹션 이동, q로 종료
```

`man -a`는 같은 이름의 매뉴얼을 차례대로 다 보여준다. 평소엔 잘 안 쓰지만 디버깅할 때 유용하다. 시스템 콜과 라이브러리 함수가 같은 이름인 경우가 많아서 C 코드 디버깅 시 `man 2 read`와 `man 3 read`를 구분해야 할 때가 있다.

### man 페이지 내부 탐색

`man`은 기본적으로 `less` 페이저로 열린다. 이걸 모르면 매뉴얼을 띄워놓고 마우스로 스크롤하다가 시간을 다 쓰게 된다.

```bash
# man 페이지 안에서 사용하는 키
/keyword     # 키워드 검색 (앞으로)
?keyword     # 키워드 검색 (뒤로)
n            # 다음 검색 결과
N            # 이전 검색 결과
g            # 문서 처음으로
G            # 문서 끝으로
h            # 도움말
q            # 종료
```

긴 매뉴얼에서 특정 옵션을 빨리 찾으려면 `/--옵션이름`으로 검색하는 게 가장 빠르다. 예를 들어 `man tar`를 띄우고 `/--exclude`를 치면 해당 옵션 설명 부분으로 바로 이동한다.

### man 페이지 구조 이해

man 페이지는 정형화된 섹션으로 구성된다.

- NAME: 명령어 이름과 한 줄 설명
- SYNOPSIS: 사용법 요약
- DESCRIPTION: 상세 설명
- OPTIONS: 옵션 목록
- EXAMPLES: 예시 (없는 경우도 많음)
- FILES: 관련 파일
- SEE ALSO: 관련 매뉴얼
- BUGS: 알려진 버그

실무에서 가장 먼저 보는 건 SYNOPSIS다. 대괄호 `[]`로 감싼 건 선택, 파이프 `|`는 OR, `...`은 반복 가능을 뜻한다. 이 표기법만 익혀도 90%는 읽힌다.

```
SYNOPSIS
       grep [OPTIONS] PATTERN [FILE...]
       grep [OPTIONS] -e PATTERN ... [FILE...]
       grep [OPTIONS] -f FILE ... [FILE...]
```

여기서 `[FILE...]`는 파일 인자가 0개 이상이라는 뜻이고, `-e PATTERN ...`은 `-e`와 패턴이 여러 번 올 수 있다는 의미다.

## info 시스템

`info`는 GNU 프로젝트가 `man`을 대체하려고 만든 시스템이다. 결과적으로는 대체에 실패했고, 지금은 GNU 도구들(`coreutils`, `bash`, `gcc` 등)의 상세 매뉴얼을 보는 용도로 주로 쓰인다. `man ls`보다 `info ls`가 훨씬 자세한 경우가 많다.

```bash
# info 진입
info ls
info bash
info coreutils
```

문제는 `info`가 하이퍼텍스트 기반이라 탐색이 직관적이지 않다는 점이다. 노드(node) 단위로 구성되어 있고, 키 바인딩이 Emacs 스타일이라 처음 보면 당황스럽다.

```bash
# info 안에서
n            # 다음 노드
p            # 이전 노드
u            # 상위 노드
Tab          # 다음 링크
Enter        # 링크 따라가기
l            # 이전에 보던 노드로 (브라우저 뒤로가기)
q            # 종료
s keyword    # 키워드 검색
```

평소에는 `man`만 써도 되지만, GNU 도구의 고급 기능을 알아야 할 때는 `info`가 답이다. 예를 들어 `info '(coreutils) ls invocation'` 같은 식으로 특정 노드를 직접 지정해서 들어갈 수도 있다.

GNU의 `make`, `gcc`, `gdb` 같은 도구들은 사실상 info 매뉴얼이 공식 문서다. 회사에서 빌드 시스템이나 컴파일러 옵션 디버깅할 때 한 번씩 마주친다.

## --help 옵션

대부분의 명령어는 `--help` 또는 `-h` 옵션을 지원한다. 이게 가장 빠른 방법이다. `man`을 띄우는 것보다 시작 속도가 비교가 안 될 정도로 빠르고, 옵션 요약만 필요할 때는 `--help`면 충분하다.

```bash
ls --help
grep --help
tar --help
```

다만 일관성이 없다. BSD 계열 도구들(macOS의 기본 도구 다수)은 `--help`를 지원하지 않고 `-h`만 받거나, 둘 다 지원하지 않고 `usage`만 출력하는 경우도 있다.

```bash
# Linux GNU
df --help

# macOS BSD
df -h        # 이건 --human-readable, --help가 아님
df --help    # illegal option 에러
```

macOS에서 GNU 도구를 쓰려면 `brew install coreutils`로 설치하고 `gls`, `gdf` 같은 g 접두사로 호출하거나, PATH 앞에 GNU 도구 경로를 추가해야 한다. 이걸 모르고 macOS에서 작성한 셸 스크립트가 리눅스 서버에서 동작 안 하는 경우가 자주 있다.

`busybox` 환경(알파인 리눅스, 도커 슬림 이미지)에서는 더 심하다. 명령어들이 기능 축소판이라 `--help`도 짧고 옵션도 다르다. 컨테이너 디버깅할 때 한 번쯤 겪는다.

## 명령어를 거꾸로 찾기: apropos, whatis

옵션은 알겠는데 명령어 이름이 기억 안 날 때가 있다. "디스크 사용량 보는 명령어가 뭐였더라" 같은 상황이다. 이때 쓰는 게 `apropos`와 `whatis`다.

### apropos

`apropos`는 NAME 섹션에서 키워드를 검색해서 관련 매뉴얼을 다 찾아준다. `man -k`와 같다.

```bash
apropos disk
# df (1)               - report file system disk space usage
# du (1)               - estimate file space usage
# fdisk (8)            - manipulation of disk partition tables
# ...

man -k disk    # apropos와 동일
```

키워드를 여러 개 쓰면 OR 검색이 된다. AND로 좁히려면 정규식이나 grep을 써야 한다.

```bash
apropos disk | grep -i partition
```

신입 시절에 가장 도움 된 명령어 중 하나다. 명령어 이름은 모르는데 뭘 하고 싶은지는 아는 상황에서 검색 엔진보다 빠르다.

### whatis

`whatis`는 매뉴얼의 한 줄 요약(NAME 섹션)만 출력한다. 명령어가 정확히 뭘 하는 건지 빠르게 확인할 때 쓴다.

```bash
whatis ls
# ls (1)               - list directory contents

whatis ls cp mv
# ls (1)               - list directory contents
# cp (1)               - copy files and directories
# mv (1)               - move (rename) files
```

`apropos`와 달리 정확한 이름 매칭이라 키워드 검색은 안 된다.

### mandb

`apropos`와 `whatis`는 매뉴얼 데이터베이스를 미리 만들어둬야 동작한다. 새로 설치한 시스템이나 패키지 추가 후에는 데이터베이스를 갱신해야 한다.

```bash
sudo mandb
# Processing manual pages under /usr/share/man...
# ...
```

대부분의 배포판은 `cron`으로 주기적으로 실행하지만, 컨테이너 환경에서는 직접 실행해야 할 때가 있다. `apropos`가 결과를 안 내놓으면 `mandb`부터 의심하면 된다.

## 명령어의 실체 확인: type, which, command -v

이건 도움말 시스템은 아니지만 도움말을 찾기 전에 거치는 단계다. 어떤 명령어를 실행했을 때 도대체 어떤 파일이 실행되는지 확인하는 작업이다. 이걸 안 해서 디버깅이 꼬이는 경우가 흔하다.

### 셋의 차이

```bash
# which: PATH에서 실행 파일을 찾음 (외부 명령어만)
which ls
# /usr/bin/ls

# type: 셸이 명령어를 어떻게 해석하는지 알려줌 (내장, 함수, alias 포함)
type ls
# ls is aliased to `ls --color=auto'

type cd
# cd is a shell builtin

# command -v: 스크립트에서 안전하게 쓸 수 있는 POSIX 표준
command -v ls
# alias ls='ls --color=auto'
```

`which`는 alias나 셸 빌트인을 인식 못 한다. 그래서 `which cd`를 치면 결과가 안 나오거나 `/usr/bin/cd`가 나오는데, 실제로 셸에서 `cd`를 입력하면 빌트인이 실행된다. 이걸 모르면 "왜 `cd`에는 `--help`가 없지?" 같은 데서 막힌다.

```bash
# cd는 빌트인이라 별도 매뉴얼이 없음
man cd
# No manual entry for cd

# 빌트인 도움말은 help 명령어로
help cd

# 또는 bash 매뉴얼에서 SHELL BUILTIN COMMANDS 섹션
man bash
# /SHELL BUILTIN COMMANDS
```

### 빌트인 명령어 도움말

`cd`, `pwd`, `echo`, `export`, `alias`, `kill`, `set`, `unset` 같은 셸 빌트인은 `man`이 안 통한다. 별도 실행 파일이 없으니 매뉴얼도 없다. 대신 셸이 자체 도움말을 제공한다.

```bash
# bash/sh 빌트인 도움말
help
help cd
help export

# zsh
run-help cd

# 셸 종류와 무관하게 bash 매뉴얼 보기
man bash
```

`echo` 같은 건 빌트인과 외부 명령어가 둘 다 있어서 동작이 미묘하게 다른 경우가 있다. `/bin/echo --help`는 GNU echo의 도움말을 보여주지만, 셸에서 그냥 `echo --help`를 치면 빌트인이 실행되어 `--help`가 그대로 출력된다.

```bash
echo --help
# --help

/bin/echo --help
# Usage: /bin/echo [SHORT-OPTION]... [STRING]...
# ...
```

이건 실제로 셸 스크립트 디버깅할 때 자주 만나는 함정이다.

## sudo의 매뉴얼

`sudo`는 자주 쓰는데 매뉴얼이 의외로 분산되어 있다. 명령어 자체는 `man sudo`지만, 설정 파일은 다른 매뉴얼이다.

```bash
man sudo            # sudo 명령어 사용법
man sudoers         # /etc/sudoers 설정 파일 (섹션 5)
man visudo          # sudoers 편집기
man sudo.conf       # sudo 자체 설정
```

운영하다 보면 `sudoers` 매뉴얼을 가장 많이 본다. `NOPASSWD`, `Cmnd_Alias`, `Defaults` 같은 키워드를 검색해서 권한 위임 정책을 짤 때 참고한다.

```bash
# sudoers 안에서 자주 보는 부분
man 5 sudoers
# /SUDOERS FILE FORMAT
# /User_Spec
# /NOPASSWD
```

sudo로 명령어 실행 시 매뉴얼 보는 것도 알아두면 좋다. 일반 사용자가 못 보는 매뉴얼(예: 일부 시스템 관리 도구)은 `sudo man <명령어>`로 봐야 할 때가 있다. 거의 없는 경우지만 가끔 만난다.

## /usr/share/doc 활용

패키지 설치 시 함께 들어오는 문서가 `/usr/share/doc/<패키지명>/` 아래에 깔린다. man 페이지에 없는 README, CHANGELOG, 예제 설정 파일 같은 게 여기 있다.

```bash
ls /usr/share/doc/ | head
# adduser
# apt
# bash
# ...

ls /usr/share/doc/nginx/
# changelog.Debian.gz
# copyright
# examples
# README.md
```

특히 `examples` 디렉토리에 실제 동작하는 설정 파일이 있는 경우가 많다. nginx, postfix, rsyslog 같은 데몬 설정할 때 처음부터 짜는 게 아니라 여기 있는 예제부터 시작하면 빠르다.

```bash
# nginx 설정 예제
ls /usr/share/doc/nginx-doc/examples/

# 압축된 changelog
zless /usr/share/doc/nginx/changelog.Debian.gz
```

데비안/우분투는 라이센스 정보가 `copyright` 파일에 있고, 레드햇 계열은 `LICENSE` 파일이다. 라이센스 감사할 때 한 번씩 본다.

## 외부 도구: tldr, cheat

전통적인 매뉴얼 시스템의 약점은 예제가 부족하다는 것이다. `man tar`를 봐도 옵션 설명만 나오고 "어떤 상황에 어떻게 쓰는지"는 거의 안 나온다. 이걸 보완하려고 만들어진 게 `tldr`과 `cheat`다.

### tldr

`tldr`은 커뮤니티가 작성한 짧은 예제 모음이다. 옵션 설명이 아니라 자주 쓰이는 명령어 패턴을 보여준다.

```bash
# 설치 (npm 또는 pip)
npm install -g tldr
# 또는
pip install tldr

# 사용
tldr tar
# Archive a directory:
#   tar cf target.tar source_directory
# Extract a tar file:
#   tar xf source.tar
# ...
```

`man tar`로 30분 헤맬 걸 `tldr tar`로 1분에 끝낼 때가 많다. 다만 예제가 너무 단순한 경우도 있어서, 복잡한 옵션 조합이 필요하면 결국 `man`으로 돌아가야 한다.

오프라인 캐시도 지원해서 인터넷 안 되는 환경에서도 쓸 수 있다.

```bash
tldr --update    # 캐시 갱신
```

### cheat

`cheat`은 `tldr`과 비슷한데 자체 치트시트를 추가할 수 있다는 게 다르다. 팀에서 자주 쓰는 명령어 패턴을 공유하는 용도로 좋다.

```bash
cheat tar
cheat -e tar     # 직접 편집
```

회사 내에서 자주 쓰는 운영 명령어를 cheat로 정리해두면 신입 온보딩 자료로도 쓸 수 있다.

## 실무에서 옵션 찾는 흐름

5년차쯤 되면 자기만의 패턴이 생긴다. 보통 이런 순서로 찾는다.

```mermaid
flowchart TD
    A[옵션 기억 안 남] --> B{명령어 이름 기억나나?}
    B -->|아니오| C[apropos 키워드]
    B -->|예| D{대충 옵션 형태 기억나나?}
    C --> E[man 명령어]
    D -->|예| F[명령어 --help &#124; grep 옵션]
    D -->|아니오| G[tldr 명령어]
    G --> H{원하는 거 찾았나?}
    H -->|예| I[실행]
    H -->|아니오| E
    F --> I
    E --> J[/--옵션이름 검색]
    J --> I
```

핵심은 `--help | grep`이다. man 페이지를 열어서 페이저로 검색하는 것보다 `--help` 출력에 `grep`을 거는 게 빠를 때가 많다.

```bash
# tar에서 exclude 관련 옵션만 보기
tar --help | grep -A 1 exclude

# rsync에서 delete 관련 옵션
rsync --help | grep -i delete

# find에서 mtime 관련 옵션
find --help 2>&1 | grep -A 2 mtime
```

`find` 같은 일부 명령어는 `--help`를 지원 안 하거나 출력이 stderr로 나가는 경우가 있어서 `2>&1`로 합쳐줘야 한다.

## 자주 쓰는 매뉴얼 단축 패턴

운영 중에 매뉴얼 자주 보는 명령어들은 단축 패턴을 외워두면 좋다.

```bash
# bash 빌트인 전체 목록
help

# 시그널 번호 확인
man 7 signal
kill -l

# 정규식 매뉴얼
man 7 regex

# 시스템 콜 에러 코드
man 3 errno

# 파일 권한 비트 의미
man 2 stat | grep -A 30 "S_IFMT"

# 환경 변수 매뉴얼
man 7 environ
```

특히 `man 7 signal`은 코어덤프 분석할 때, `man 3 errno`는 시스템 콜 에러 디버깅할 때 자주 본다.

## 매뉴얼 출력 저장과 검색

긴 매뉴얼을 텍스트로 저장해두고 grep을 거는 패턴도 자주 쓴다.

```bash
# man 페이지를 텍스트 파일로 저장
man tar | col -b > tar-manual.txt

# PDF로 변환 (Linux)
man -t tar | ps2pdf - tar-manual.pdf

# HTML로 변환
man --html=cat tar > tar-manual.html
```

`col -b`는 백스페이스 문자(굵은 글씨용)를 제거한다. 이걸 안 하면 텍스트 파일에 이상한 제어 문자가 섞여서 grep 결과가 깨진다.

운영 서버에서 인터넷 안 되는 환경이라면 자주 쓰는 매뉴얼을 미리 텍스트로 뽑아서 따로 보관하는 것도 한 방법이다.

## 자주 만나는 문제

### 매뉴얼이 없다고 나올 때

```bash
man tar
# No manual entry for tar
```

대부분 매뉴얼 패키지가 설치 안 된 경우다. 도커 슬림 이미지나 최소 설치 시스템에서 흔하다.

```bash
# 데비안/우분투
apt install man-db manpages

# 레드햇/CentOS
yum install man-pages

# 알파인
apk add man-pages mandoc
```

알파인은 기본적으로 `man`이 없다. `mandoc`이나 `man-db`를 따로 설치해야 한다.

### 한국어 매뉴얼

영어 매뉴얼이 부담스러울 때 한국어 매뉴얼을 깔 수 있긴 한데 추천하지 않는다. 번역 품질이 들쭉날쭉하고 업데이트가 느려서 옵션이 누락된 경우가 많다. 영어 매뉴얼이 어려우면 차라리 `tldr`로 예제 위주로 보는 게 낫다.

```bash
# 한국어 매뉴얼 (선택사항)
apt install manpages-ko
```

### 매뉴얼이 짧거나 부실할 때

`busybox`나 임베디드 환경에서 `--help`를 쳤는데 옵션 한 줄만 나오는 경우가 있다. 이때는 GNU 버전의 매뉴얼을 다른 시스템에서 보는 게 빠르다.

온라인 매뉴얼 사이트들도 알아두면 좋다.
- `man7.org`: 리눅스 man 페이지 공식 미러
- `manned.org`: 여러 배포판 매뉴얼 비교
- `explainshell.com`: 명령어를 입력하면 옵션을 분해해서 설명

운영 서버에서는 안 되지만 로컬에서 빠르게 확인할 때 유용하다.
