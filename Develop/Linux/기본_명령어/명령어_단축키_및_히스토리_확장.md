# 명령어 단축키 및 히스토리 확장

서버에 들어가서 명령어를 칠 때 마우스가 없다는 사실을 의식하지 않고 손이 움직이는 수준이 되어야 작업 속도가 의미 있게 빨라진다. 매번 화살표 키로 한 글자씩 이동하고 백스페이스로 지우는 방식으로는 긴 명령어를 다룰 때 시간이 너무 많이 든다. 여기서 정리하는 단축키와 히스토리 확장은 거의 모든 Linux 환경에서 동일하게 동작하므로, 한 번 익혀두면 어떤 서버에 들어가든 그대로 쓸 수 있다.

## Readline 라이브러리와 편집 모드

Bash가 명령어 입력을 받을 때 직접 키보드 입력을 처리하지 않는다. GNU Readline이라는 라이브러리가 입력 편집을 담당한다. Bash뿐 아니라 psql, gdb, mysql 같은 대화형 도구들도 대부분 Readline을 쓰기 때문에 한 번 익힌 단축키가 다른 도구에서도 그대로 통한다.

Readline은 두 가지 키 바인딩 모드를 지원한다. 기본은 emacs 모드고, vi 모드로 바꾸려면 다음과 같이 한다.

```bash
set -o vi      # vi 모드로 전환
set -o emacs   # emacs 모드로 전환 (기본값)

# 영구 적용은 ~/.inputrc에
echo "set editing-mode vi" >> ~/.inputrc
```

vi 모드를 쓰면 명령어 라인 자체를 vi처럼 편집할 수 있다. ESC를 누르면 노멀 모드로 전환되고 `dd`, `cw`, `b`, `w` 같은 vi 명령이 그대로 동작한다. 평소 vim을 쓰는 사람이면 자연스럽지만, 모드 전환을 매번 해야 해서 emacs 모드보다 호흡이 끊긴다는 단점이 있다. 실무에서는 emacs 모드로 두고 자주 쓰는 단축키만 손에 익히는 쪽이 일반적이다.

현재 모드는 `set -o | grep -E 'vi|emacs'`로 확인할 수 있다. 두 모드가 동시에 켜질 수도 있는데, 이 경우 마지막에 켠 쪽이 우선한다.

## 커서 이동 단축키

emacs 모드 기준으로 자주 쓰는 커서 이동 단축키는 다음과 같다.

| 단축키 | 동작 |
|---|---|
| `Ctrl+A` | 줄 맨 앞으로 이동 |
| `Ctrl+E` | 줄 맨 뒤로 이동 |
| `Ctrl+B` | 한 글자 뒤로 |
| `Ctrl+F` | 한 글자 앞으로 |
| `Alt+B` | 한 단어 뒤로 |
| `Alt+F` | 한 단어 앞으로 |
| `Ctrl+XX` | 현재 위치와 줄 맨 앞을 토글 |

긴 명령어 중간에서 옵션 하나를 고치고 싶을 때 화살표 키 대신 `Alt+B`/`Alt+F`로 단어 단위 이동을 하면 훨씬 빠르다. 단, macOS 터미널에서는 `Alt` 키가 메타 키로 인식되지 않을 때가 있다. iTerm2면 Profiles → Keys → General에서 Left Option Key를 Esc+로 바꿔야 동작한다. 기본 터미널은 Preferences → Profiles → Keyboard에서 "Use Option as Meta key"를 체크해야 한다.

`Alt+B`가 안 먹히는 환경에서는 `Esc` 후 `B`를 누르는 방법으로 대체할 수 있다. Readline이 메타 키를 ESC 시퀀스로도 받기 때문이다.

## 삭제 단축키

삭제는 백스페이스로 한 글자씩 지우는 게 아니라 단위별로 지워야 한다.

| 단축키 | 동작 |
|---|---|
| `Ctrl+W` | 커서 앞쪽 한 단어 삭제 |
| `Ctrl+U` | 커서부터 줄 맨 앞까지 삭제 |
| `Ctrl+K` | 커서부터 줄 맨 뒤까지 삭제 |
| `Ctrl+H` | 커서 앞 한 글자 삭제 (Backspace와 동일) |
| `Ctrl+D` | 커서 위치 글자 삭제 (입력이 비어 있으면 셸 종료) |
| `Alt+D` | 커서 앞쪽 한 단어 삭제 (Ctrl+W와 단어 경계 정의가 다름) |
| `Ctrl+Y` | 마지막에 삭제한 내용 붙여넣기 (yank) |

`Ctrl+W`와 `Alt+D`는 단어 경계 처리가 다르다. `Ctrl+W`는 공백 단위로 끊어서 지우고, `Alt+D`는 영숫자 경계로 끊는다. `/var/log/nginx/access.log`에서 마지막 단어를 지운다고 할 때 `Ctrl+W`는 전체 경로를 한 번에 지우지만 `Alt+D`는 `log`만 지운다. 파일 경로를 다룰 일이 많은 셸 환경에서는 보통 `Ctrl+W`가 더 직관적이다.

`Ctrl+U`로 줄을 통째로 지웠다가 `Ctrl+Y`로 다시 붙여넣을 수도 있다. 비밀번호를 입력하다가 잘못 친 경우 `Ctrl+U`로 한 번에 지우고 다시 입력하는 패턴을 자주 쓴다. 화면에 비밀번호가 노출되지 않도록 처리하는 prompt에서도 잘 동작한다.

`Ctrl+D`는 입력이 비어 있을 때 EOF로 해석되어 현재 셸을 종료시킨다. SSH 세션에서 잘못 누르면 연결이 끊어지므로, 이를 막고 싶으면 `set -o ignoreeof`를 설정하거나 `IGNOREEOF=10` 같은 환경변수를 둔다.

## 히스토리 검색

가장 많이 쓰는 건 `Ctrl+R` 역검색이다. 누르면 프롬프트가 `(reverse-i-search)\`':`로 바뀌고 입력하는 문자열이 포함된 가장 최근 히스토리를 찾아준다.

```
(reverse-i-search)`docker': docker logs -f --tail 100 api-server
```

여기서 한 가지 흔히 놓치는 점이 있다. 매치된 결과가 만족스럽지 않으면 `Ctrl+R`을 한 번 더 눌러야 그 다음으로 오래된 매치로 넘어간다. Enter를 치면 그 명령이 바로 실행되므로, 명령을 실행하지 않고 편집만 하려면 화살표 키나 `Ctrl+A`/`Ctrl+E`를 눌러서 검색 모드를 빠져나와야 한다. 검색을 취소하고 원래 입력 상태로 돌아가려면 `Ctrl+G`를 누른다.

순방향 검색은 `Ctrl+S`인데 기본적으로 터미널이 이 키를 출력 정지(XOFF)로 가로채기 때문에 동작하지 않는 경우가 많다. 풀려면 `~/.bashrc`나 `~/.bash_profile`에 다음을 넣는다.

```bash
stty -ixon
```

이걸 풀면 `Ctrl+R`로 너무 멀리 갔을 때 `Ctrl+S`로 돌아올 수 있다.

`Ctrl+R`로 검색하다 보면 비슷한 명령이 여러 개 떠서 원하는 걸 찾기 힘들 때가 있다. 이럴 때는 검색 문자열을 더 길게 쳐서 후보를 좁히는 게 빠르다. 한 번에 찾으려고 하지 말고 키워드 두세 개를 조합한다고 생각하면 된다.

## 히스토리 확장

히스토리 확장은 `!`로 시작하는 특수 표현으로, 과거 명령이나 그 일부를 현재 명령줄로 끌어다 쓰는 기능이다. 입력 편집과 별개로 셸이 명령을 실행하기 직전에 치환을 수행한다.

### 자주 쓰는 확장 표현

| 표현 | 의미 |
|---|---|
| `!!` | 바로 직전 명령 전체 |
| `!$` | 직전 명령의 마지막 인자 |
| `!*` | 직전 명령의 모든 인자 |
| `!^` | 직전 명령의 첫 번째 인자 |
| `!:n` | 직전 명령의 n번째 인자 (0은 명령어 자체) |
| `!n` | 히스토리 n번째 명령 |
| `!-n` | 현재 기준 n번째 이전 명령 |
| `!string` | string으로 시작하는 가장 최근 명령 |
| `!?string?` | string을 포함하는 가장 최근 명령 |
| `^old^new` | 직전 명령에서 old를 new로 한 번 치환 후 실행 |
| `!!:gs/old/new/` | 직전 명령에서 old를 new로 모두 치환 후 실행 |

가장 많이 쓰이는 패턴은 `sudo !!`이다. 권한이 없어서 명령이 실패했을 때 다시 칠 필요 없이 바로 sudo를 붙여 재실행한다.

```bash
$ systemctl restart nginx
Failed to restart nginx.service: Access denied
$ sudo !!
sudo systemctl restart nginx
```

`!$`도 굉장히 자주 쓴다. 디렉토리를 만들고 바로 들어가거나, 파일을 옮기고 그 위치를 확인하는 패턴이다.

```bash
$ mkdir -p /var/log/myapp/2026
$ cd !$
cd /var/log/myapp/2026

$ mv config.yaml /etc/myapp/
$ ls !$
ls /etc/myapp/
```

`^old^new`는 오타를 고칠 때 쓴다. 직전 명령에서 한 번만 치환되므로 같은 단어가 여러 번 나오면 `!!:gs/old/new/`를 써야 한다.

```bash
$ git brnach -a
git: 'brnach' is not a git command.
$ ^brnach^branch
git branch -a
```

### 확장 결과 미리 보기

확장이 어떻게 풀리는지 먼저 보고 싶으면 `:p` 수정자를 붙인다. 명령은 실행하지 않고 풀린 형태만 히스토리에 추가된다.

```bash
$ !ssh:p
ssh deploy@10.0.1.5
```

확인 후 그대로 실행하려면 `!!`를 누르면 된다. 운영 서버에서 잘못 풀리면 큰일나는 명령(rm, drop, kill 등)을 다룰 때는 `:p`로 한 번 확인하는 습관을 들이는 게 안전하다.

`shopt -s histverify`를 켜놓으면 모든 히스토리 확장이 자동으로 한 번 표시되고 Enter를 한 번 더 눌러야 실행된다. 운영 환경 셸에서 켜두는 사람이 많다.

```bash
shopt -s histverify
```

## history 명령과 fc

`history` 명령은 히스토리 파일과 메모리상의 히스토리를 다룬다.

```bash
history          # 전체 히스토리 출력
history 20       # 최근 20개만 출력
history -c       # 메모리상의 히스토리 비움 (파일은 유지)
history -d 100   # 100번 항목 삭제
history -w       # 메모리상의 히스토리를 파일에 기록
history -r       # 파일에서 메모리로 히스토리 다시 읽음
history -a       # 현재 세션에서 추가된 항목만 파일에 추가
history -n       # 파일에서 아직 안 읽은 항목을 메모리로
```

`-w`와 `-a`의 차이는 잘못 쓰면 히스토리가 덮어쓰기될 수 있어 알아둬야 한다. `-w`는 메모리 내용으로 파일을 통째로 덮어쓴다. 여러 셸 세션이 동시에 떠 있을 때 한 세션에서 `-w`를 호출하면 다른 세션에서 만든 히스토리가 사라진다. `-a`는 현재 세션에서 추가된 항목만 파일 끝에 덧붙이므로 안전하다.

`fc`(fix command)는 히스토리에 있는 명령을 에디터로 열어서 편집한 뒤 실행한다.

```bash
fc           # 직전 명령을 $EDITOR로 열기
fc -l        # 최근 히스토리 출력 (history와 비슷)
fc -l -10    # 최근 10개
fc 100 110   # 100번부터 110번까지 편집
fc -s old=new   # 직전 명령에서 old를 new로 치환 후 실행 (^old^new와 동일)
```

여러 줄로 된 명령을 고치거나, 파이프라인이 길어서 인라인 편집이 힘들 때 `fc`로 vim에서 편집하는 게 편하다.

## 히스토리 환경변수

히스토리 동작은 환경변수로 제어한다. 운영 서버 표준 설정으로 자주 쓰는 값들이다.

```bash
# ~/.bashrc 또는 /etc/bashrc

HISTSIZE=10000              # 메모리에 유지할 명령어 수
HISTFILESIZE=20000          # 히스토리 파일에 저장할 명령어 수
HISTCONTROL=ignoredups:erasedups   # 중복 제거
HISTIGNORE="ls:ll:cd:pwd:exit:history"   # 기록하지 않을 명령
HISTTIMEFORMAT="%Y-%m-%d %H:%M:%S "      # 타임스탬프 포함
```

### HISTSIZE / HISTFILESIZE

`HISTSIZE`는 현재 셸 세션이 메모리에 들고 있는 항목 수, `HISTFILESIZE`는 디스크 파일(`~/.bash_history`)이 보관할 수 있는 최대 항목 수다. 두 값을 다르게 잡으면 셸 종료 시점에 잘림이 발생할 수 있다. 보통 `HISTFILESIZE`를 `HISTSIZE`의 2배 정도로 잡거나 둘 다 같은 값을 쓴다.

너무 크게 잡으면 셸 시작 시 히스토리 로딩에 시간이 걸린다. 1만~5만 정도가 일반적이고, 10만을 넘기면 노이즈만 늘어난다.

### HISTCONTROL

값은 콜론으로 구분된 옵션들이다.

- `ignorespace`: 공백으로 시작하는 명령은 기록 안 함
- `ignoredups`: 직전 명령과 같으면 기록 안 함
- `ignoreboth`: 위 두 개 동시 적용
- `erasedups`: 새로 추가될 때 같은 명령을 히스토리에서 모두 지우고 새로 추가

`ignorespace`는 비밀번호를 명령줄에 넣어야 할 때 유용하다. 명령 앞에 공백 한 칸 넣으면 히스토리에 안 남는다.

```bash
$  mysql -u admin -pSecret123    # 맨 앞 공백 → 기록 안 됨
```

다만 명령이 프로세스 목록(`ps -ef`)에는 그대로 보이므로 진짜 민감한 값은 환경변수나 설정 파일을 통해 전달해야 한다. 히스토리만 가린다고 안전한 게 아니다.

### HISTIGNORE

콜론으로 구분된 패턴 목록이다. 와일드카드도 쓸 수 있다.

```bash
HISTIGNORE="ls:ll:cd:pwd:exit:history:clear:bg:fg:jobs"
HISTIGNORE="$HISTIGNORE:[ ]*"   # 공백으로 시작하는 명령
HISTIGNORE="$HISTIGNORE:&"      # 직전 명령과 동일
```

너무 공격적으로 잡으면 정작 필요할 때 히스토리를 못 찾으니 짧고 자주 치는 명령만 골라 넣는 게 낫다. `cd /some/important/dir`까지 무시되면 곤란하므로 인자가 없는 형태만 무시하는 식으로 패턴을 짤 때도 있다.

### HISTTIMEFORMAT

이 변수를 설정하면 `history` 출력에 타임스탬프가 붙고, `~/.bash_history` 파일에도 각 명령 앞에 `#` 주석으로 epoch 시간이 기록된다.

```
#1714435200
systemctl restart nginx
#1714435210
journalctl -u nginx -n 50
```

운영 서버에서 사고 분석할 때 누가 언제 무슨 명령을 쳤는지 추적하려면 필수다. 한번 켜놓고 운영하다가 끄면 그 시점부터 기록된 명령은 타임스탬프 없이 들어가서 파일이 섞이므로, 처음부터 켜놓는 게 좋다.

## 여러 셸 세션 간 히스토리 공유

기본 동작은 셸이 종료될 때만 히스토리를 파일에 쓴다. tmux로 여러 창을 띄워 작업하는 환경에서는 한 창에서 친 명령을 다른 창에서 `Ctrl+R`로 못 찾는 경우가 생긴다. 이걸 즉시 공유하려면 `PROMPT_COMMAND`에 history -a/-c/-r을 엮는다.

```bash
shopt -s histappend                            # 종료 시 덮어쓰지 않고 추가
PROMPT_COMMAND="history -a; history -c; history -r; $PROMPT_COMMAND"
```

각 줄의 의미는 이렇다.

- `histappend`: 셸 종료 시 `~/.bash_history`를 덮어쓰지 않고 append. 여러 세션이 떠 있는 상황에서 안전하다.
- `history -a`: 현재 세션에서 친 명령을 즉시 파일에 추가
- `history -c`: 메모리 히스토리 비움
- `history -r`: 파일에서 다시 읽어오기

이 조합을 매 프롬프트마다 실행하면 다른 창에서 친 명령이 즉시 보인다. 단점은 히스토리 번호(`!100` 같은 표현)가 매번 바뀌어서 번호 기반 참조가 안 된다는 점이다. `!!`이나 `!string`은 정상 동작한다.

또 하나 주의할 점은 `Ctrl+R`로 검색했을 때 직전에 친 명령이 바로 안 잡힐 수 있다는 점이다. 다른 창에서 친 명령들이 먼저 들어와 있어서 검색 순서가 바뀐다. 이게 거슬리면 공유는 끄고 필요할 때만 `history -a`/`-r`을 수동으로 호출하는 방식으로 쓴다.

zsh를 쓴다면 `setopt SHARE_HISTORY` 한 줄이면 된다. Bash는 기본 동작이 다르기 때문에 위 조합을 직접 짜야 한다.

## 자동 완성과 단축

`Tab`은 파일 이름과 명령어 자동 완성을 한다. 두 번 누르면 후보 목록을 보여준다. `bash-completion` 패키지가 설치돼 있으면 `git`, `systemctl`, `docker` 같은 명령의 서브커맨드와 옵션까지 완성된다. RHEL/CentOS 계열에서는 기본 설치가 안 돼 있어 직접 설치해야 한다.

```bash
# Debian/Ubuntu
apt install bash-completion

# RHEL/CentOS/Rocky
dnf install bash-completion
```

설치 후 `source /etc/profile.d/bash_completion.sh` 또는 새 셸을 열어야 적용된다.

`Alt+.`은 직전 명령의 마지막 인자를 현재 위치에 삽입한다. `!$`와 비슷하지만 즉시 텍스트로 풀려서 보이고, 한 번 더 누르면 그 전 명령의 마지막 인자로 넘어간다.

```bash
$ vim /etc/nginx/conf.d/api.conf
$ ls -la <Alt+.>
$ ls -la /etc/nginx/conf.d/api.conf
```

`!$`와 차이는 `!$`는 실행 직전 치환이고, `Alt+.`은 입력 시점에 풀려서 즉시 보인다는 점이다. 풀린 결과를 보면서 편집할 거면 `Alt+.`이 낫고, 그냥 직전 인자를 그대로 쓸 거면 `!$`가 키 입력이 적다.

## 실무에서 자주 쓰는 조합

```bash
# 명령 결과를 보고 sudo로 다시 실행
$ apt update
E: Could not open lock file - open (13: Permission denied)
$ sudo !!

# 만든 디렉토리로 바로 진입
$ mkdir -p /data/backups/$(date +%Y%m%d)
$ cd !$

# 직전 파일을 다른 명령으로
$ tail -n 100 /var/log/syslog
$ grep -i error !$

# 명령 일부만 바꿔서 재실행
$ kubectl get pods -n staging
$ ^staging^production
kubectl get pods -n production

# 인자만 빼고 명령 바꾸기
$ ls -la /etc/nginx/conf.d/
$ vim !*

# fc로 긴 명령 편집
$ docker run --rm -it --network host -v /data:/data -e DEBUG=1 myimage
$ fc      # vim 열려서 편집 후 :wq 하면 실행
```

이 패턴들은 별것 아닌 것 같지만 하루에 수십 번 반복되는 동작들이다. 화살표 키로 직전 명령 띄워서 마우스로 끌고 다니던 사람이 이 단축키들을 손에 익히면 작업 시간이 체감상 절반으로 준다. 처음에는 의식해서 써야 하지만 일주일 정도 강제로 쓰면 손이 먼저 움직인다.
