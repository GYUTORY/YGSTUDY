# 작업 스케줄링

## 개요

리눅스에서 작업을 자동으로 실행하는 방법은 크게 세 가지다. 반복 작업은 cron, 일회성 작업은 at, systemd 환경이라면 systemd timer를 쓴다. 데스크톱이나 노트북처럼 항상 켜져 있지 않은 시스템에서는 anacron이 추가로 필요하다.

실무에서 cron을 처음 다루면 거의 모두 같은 함정에 빠진다. 쉘에서는 잘 돌아가던 스크립트가 cron에 올리면 동작하지 않는다. 환경 변수 차이, 경로 차이, MAILTO 미설정으로 묻힌 에러 메시지, 동시에 두 번 도는 잡, 새벽 3시에 잠깐 죽었다 살아난 서버에서 누락된 작업까지. 이 문서는 그 함정들을 중심으로 정리한다.

## cron 동작 모델

`cron` 데몬은 1분마다 깨어나서 등록된 crontab을 읽고 시간이 맞는 명령을 실행한다. 실행 환경은 사용자가 로그인했을 때와는 완전히 다르다. 이게 가장 큰 함정이다.

cron이 명령을 실행할 때 만들어주는 환경은 대략 다음과 같다.

```
HOME=<사용자 홈>
LOGNAME=<사용자명>
PATH=/usr/bin:/bin
SHELL=/bin/sh
```

`PATH`가 `/usr/bin:/bin`만 잡혀 있다는 점이 핵심이다. `aws`, `kubectl`, `nvm`으로 깐 `node`, `pyenv`로 깐 `python`은 전부 다른 경로에 있어서 호출이 안 된다. `~/.bashrc`, `~/.profile`, `/etc/profile.d/*` 같은 것도 읽지 않는다.

### crontab 편집

```bash
crontab -e                       # 현재 사용자 crontab 편집
crontab -l                       # 목록 확인
crontab -r                       # 전체 삭제 (확인 없이 지워지므로 주의)
crontab -u deploy -e             # 다른 사용자 crontab 편집 (root 권한)
```

`crontab -r`은 확인 프롬프트 없이 그대로 지운다. 작업 등록 후에는 백업해두는 습관이 필요하다.

```bash
crontab -l > ~/crontab.bak.$(date +%F)
```

### 시간 필드 형식

```
* * * * * command
│ │ │ │ │
│ │ │ │ └── 요일 (0-7, 0과 7은 일요일)
│ │ │ └──── 월 (1-12)
│ │ └────── 일 (1-31)
│ └──────── 시 (0-23)
└────────── 분 (0-59)
```

자주 쓰는 표현이다.

```bash
0 2 * * * /opt/app/backup.sh           # 매일 02:00
0 */6 * * * /opt/app/check.sh          # 6시간마다
*/5 * * * * /opt/app/health.sh         # 5분마다
0 9-17 * * 1-5 /opt/app/work.sh        # 평일 09-17시 매시 정각
0 0 1,15 * * /opt/app/biweekly.sh      # 매월 1일과 15일 자정
```

`일`과 `요일`을 동시에 지정하면 OR로 동작한다. 두 조건 중 하나라도 맞으면 실행된다는 점을 모르고 짜면 의도와 다르게 너무 자주 실행된다.

```bash
# 잘못된 의도: "매월 13일이면서 금요일에만 실행"
0 0 13 * 5 /path/script.sh
# 실제 동작: "매월 13일" 또는 "매주 금요일" 둘 중 하나라도 맞으면 실행
```

이 패턴은 cron 자체로 풀 수 없어서 스크립트 내부에서 날짜 체크를 다시 해야 한다.

## cron 환경 변수 함정

쉘에서 잘 도는 스크립트가 cron에서만 죽는 가장 흔한 원인이다.

### PATH 문제

cron의 `PATH`는 `/usr/bin:/bin`이다. `/usr/local/bin`도 안 들어 있다. Homebrew, asdf, nvm, pyenv로 설치한 도구는 전부 호출 실패한다.

문제가 발생하는 전형적인 모습이다.

```bash
# crontab
*/10 * * * * docker ps | grep -c Up > /tmp/count.txt
```

쉘에서는 잘 도는데 cron 로그에는 `docker: command not found`만 남는다. `which docker`로 확인하면 `/usr/bin/docker`인 경우는 그나마 운이 좋고, 보통은 `/usr/local/bin/docker`나 `/snap/bin/docker`에 있다.

해결 방법은 세 가지다.

첫째, crontab 상단에서 PATH를 명시적으로 설정한다. 가장 흔한 방식이다.

```bash
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
MAILTO=ops@example.com

*/10 * * * * docker ps | grep -c Up > /tmp/count.txt
```

둘째, 명령을 절대 경로로 쓴다.

```bash
*/10 * * * * /usr/local/bin/docker ps | grep -c Up > /tmp/count.txt
```

셋째, 스크립트 안에서 환경을 직접 구성한다. 이게 가장 안전하다. crontab은 한 줄만 두고 스크립트가 자기 환경을 책임진다.

```bash
#!/bin/bash
# /opt/app/check.sh
set -euo pipefail
export PATH=/usr/local/bin:/usr/bin:/bin
source /etc/profile.d/myapp.sh         # 필요한 환경 변수 로드

docker ps | grep -c Up > /tmp/count.txt
```

### 로그인 쉘과의 차이

`cron`은 로그인 쉘이 아니라서 `~/.bashrc`, `~/.bash_profile`, `~/.profile`을 읽지 않는다. 이 안에 `nvm`, `rbenv`, `pyenv` 초기화가 들어 있다면 cron에서는 적용되지 않는다.

`nvm`의 `node`를 cron에서 호출하려면 다음과 같이 명시적으로 환경을 잡아야 한다.

```bash
0 * * * * bash -lc 'source ~/.nvm/nvm.sh && nvm use 20 && node /opt/app/job.js'
```

`bash -l`은 로그인 쉘로 실행해서 프로파일을 읽도록 한다. 다만 사용자 홈의 dotfile에 의존하기 때문에 프로덕션에서는 추천하지 않는다. 차라리 systemd unit으로 옮기고 `Environment=` 지시자로 환경을 박는 편이 명확하다.

### 그 외 자주 빠뜨리는 변수

- `LANG`, `LC_ALL`: 로케일 미설정으로 한글 출력이 깨지거나 정렬이 다르게 동작한다.
- `TZ`: 시스템 타임존과 다른 시간대로 잡을 때 필요하다.
- `HOME`: 일부 프로그램(예: aws cli, git)이 설정 파일을 못 찾는다.

```bash
SHELL=/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin
LANG=ko_KR.UTF-8
TZ=Asia/Seoul
HOME=/home/deploy
MAILTO=ops@example.com
```

## MAILTO와 로깅

cron은 명령의 표준 출력과 표준 에러를 잡아서 메일로 보낸다. `MAILTO`가 비어 있으면(`MAILTO=""`) 메일을 보내지 않고, 설정되지 않으면 사용자의 로컬 메일함으로 들어간다.

문제는 두 가지다. 첫째, 로컬 메일이 어디로 가는지 모르고 그냥 묻혀버린다. 둘째, 메일이 잘 가더라도 정상 출력까지 매분 메일이 와서 결국 전체를 무시하게 된다.

### 출력을 어떻게 처리할지가 핵심이다

가장 단순한 패턴은 모든 출력을 파일로 보내는 것이다.

```bash
0 2 * * * /opt/app/backup.sh >> /var/log/backup.log 2>&1
```

`2>&1`이 `>> /var/log/backup.log` 뒤에 와야 한다. 순서를 바꾸면 stderr가 리다이렉트되지 않는다.

```bash
# 잘못된 순서: stderr는 여전히 메일로 간다
0 2 * * * /opt/app/backup.sh 2>&1 >> /var/log/backup.log

# 올바른 순서
0 2 * * * /opt/app/backup.sh >> /var/log/backup.log 2>&1
```

정상 시에는 조용하고 에러 시에만 메일을 받고 싶다면 `chronic`(moreutils 패키지)을 쓴다.

```bash
0 2 * * * chronic /opt/app/backup.sh
```

`chronic`은 명령이 0을 리턴하면 모든 출력을 버리고, 0이 아니면 모아둔 출력을 그대로 표준 출력으로 흘려준다. cron이 그걸 메일로 보내준다. 평소엔 메일이 안 오다가 실패할 때만 정확한 에러가 도착한다.

### MAILTO 분리

작업별로 알림 받는 사람이 다를 때는 crontab을 섹션으로 나눠서 MAILTO를 바꾼다.

```bash
PATH=/usr/local/bin:/usr/bin:/bin

MAILTO=db-team@example.com
0 3 * * * /opt/db/backup.sh

MAILTO=app-team@example.com
*/5 * * * * /opt/app/health.sh

MAILTO=""
*/1 * * * * /opt/app/heartbeat.sh        # 출력 없음, 메일 안 보냄
```

MAILTO 지정은 그 아래 모든 잡에 적용된다. 다음 MAILTO 지정이 나오기 전까지다.

### cron 실행 로그

cron 데몬 자체의 로그는 어떤 잡이 언제 실행됐는지를 보여준다. 잡의 출력은 따로 처리해야 한다.

```bash
# Debian/Ubuntu
grep CRON /var/log/syslog
journalctl -u cron --since today

# RHEL/CentOS/Rocky
tail -f /var/log/cron
journalctl -u crond --since "1 hour ago"
```

`Aug  5 03:00:01 host CRON[12345]: (deploy) CMD (/opt/app/backup.sh)` 같은 줄이 찍힌다. 실행됐다는 사실만 보여주고 명령의 결과는 없다.

## /etc/cron.d vs /etc/cron.daily

시스템 cron은 두 가지 형태로 나뉜다. 헷갈리기 쉬워서 정리한다.

### /etc/cron.d/

crontab 형식의 파일을 이 디렉토리에 두면 cron이 읽는다. 단, 사용자 crontab과 달리 사용자 필드가 추가된다.

```bash
# /etc/cron.d/myapp
SHELL=/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin
MAILTO=ops@example.com

# 분 시 일 월 요일 사용자 명령
0 2 * * * deploy /opt/app/backup.sh >> /var/log/backup.log 2>&1
```

패키지가 설치 시 자기 cron 잡을 두기에 적합하다. 파일명 규칙이 있다. 점(`.`)이나 특수문자가 들어 있으면 무시된다. `myapp.cron`은 안 되고 `myapp`이나 `myapp_cron`처럼 써야 한다.

`run-parts --report --test /etc/cron.d`로 미리 어떤 파일이 인식되는지 확인할 수 있다.

### /etc/cron.{hourly,daily,weekly,monthly}/

이 디렉토리들은 일반 실행 파일을 둔다. crontab 형식이 아니다. `/etc/crontab`에 등록된 다음 줄이 시간이 되면 디렉토리 안의 파일들을 `run-parts`로 실행한다.

```bash
# /etc/crontab (RHEL 계열)
17 *  * * * root run-parts /etc/cron.hourly
25 6  * * * root run-parts /etc/cron.daily
47 6  * * 7 root run-parts /etc/cron.weekly
52 6  1 * * root run-parts /etc/cron.monthly
```

`run-parts`는 실행 가능 권한이 있고 파일명이 규칙(`[a-zA-Z0-9_-]+`)에 맞는 파일만 실행한다. `.sh` 확장자가 붙어 있으면 실행되지 않는다. 자주 하는 실수다.

```bash
# 실행 안 됨
cp backup.sh /etc/cron.daily/backup.sh
chmod +x /etc/cron.daily/backup.sh

# 실행됨
cp backup.sh /etc/cron.daily/backup
chmod +x /etc/cron.daily/backup
```

또 시간이 정해져 있어서 정확한 분 단위 제어가 안 된다. 그게 필요하면 `/etc/cron.d/`나 사용자 crontab을 쓴다.

## anacron과 차이

서버는 보통 24시간 켜져 있어서 cron으로 충분하지만, 노트북이나 가끔 꺼지는 시스템에서는 cron 잡이 통째로 누락된다. 새벽 2시에 백업하기로 해놨는데 노트북이 꺼져 있으면 그날 백업은 그냥 사라진다. anacron은 이 문제를 해결한다.

anacron은 잡을 실행할 때마다 마지막 실행 시각을 기록해둔다. 다음에 시스템이 부팅되면 "이 잡이 N일 동안 실행 안 됐네"를 확인하고 밀린 잡을 실행한다.

```bash
# /etc/anacrontab
SHELL=/bin/sh
PATH=/sbin:/bin:/usr/sbin:/usr/bin
MAILTO=root
RANDOM_DELAY=45

# 주기(일) 지연(분) 잡이름      명령
1       5       cron.daily      run-parts /etc/cron.daily
7       25      cron.weekly     run-parts /etc/cron.weekly
@monthly 45     cron.monthly    run-parts /etc/cron.monthly
```

각 필드 의미다.

- `주기`: 며칠마다 실행할지
- `지연`: 부팅 후 몇 분 뒤에 실행할지 (부팅 직후 부하 분산)
- `잡 이름`: 마지막 실행 시각을 기록할 식별자
- `명령`: 실제 실행할 명령

배포판에 따라 cron의 daily/weekly/monthly 잡이 anacron으로 처리된다. 시스템이 그 시간에 켜져 있지 않더라도 다음 부팅 후에 누락분이 실행된다는 뜻이다.

cron과 anacron의 차이를 정리하면 다음과 같다.

| 항목 | cron | anacron |
|------|------|---------|
| 실행 단위 | 분 | 일 |
| 누락 작업 처리 | 안 함 | 부팅 시 실행 |
| 적합한 환경 | 24시간 켜진 서버 | 가끔 꺼지는 시스템 |
| 시간 정확도 | 지정 시각에 정확 | 지연(RANDOM_DELAY) 적용 |

서버에서도 anacron 사고가 일어나는 경우가 있다. 점검 때문에 새벽에 5분 정도 꺼졌다가 켰는데 그날 cron.daily가 누락되는 식이다. 이런 누락이 곤란한 작업은 systemd timer의 `Persistent=true`로 처리하는 게 깔끔하다.

## systemd timer

systemd가 들어간 시스템에서는 cron보다 systemd timer가 더 강력하다. 로그가 journal로 통합되고, 의존성 관리가 되고, 누락 처리(`Persistent=true`)도 된다.

타이머는 `.timer` 파일과 `.service` 파일 한 쌍으로 동작한다. `.timer`가 시간을 보고, 시간이 되면 같은 이름의 `.service`를 실행한다.

### 서비스 파일

```ini
# /etc/systemd/system/backup.service
[Unit]
Description=Daily database backup
After=network-online.target
Wants=network-online.target

[Service]
Type=oneshot
User=deploy
Group=deploy
WorkingDirectory=/opt/app
Environment=PATH=/usr/local/bin:/usr/bin:/bin
Environment=BACKUP_DIR=/var/backups
EnvironmentFile=-/etc/default/backup
ExecStart=/opt/app/backup.sh
StandardOutput=journal
StandardError=journal
TimeoutStartSec=30min
```

`Type=oneshot`은 한 번 실행하고 끝나는 작업에 쓴다. `EnvironmentFile=-/etc/default/backup`의 앞 `-`는 파일이 없어도 에러 내지 말라는 뜻이다.

`TimeoutStartSec`을 명시하지 않으면 기본 90초에서 죽는다. 백업처럼 오래 걸리는 작업은 반드시 늘려야 한다.

### 타이머 파일

```ini
# /etc/systemd/system/backup.timer
[Unit]
Description=Run backup daily at 03:00
Requires=backup.service

[Timer]
OnCalendar=*-*-* 03:00:00
Persistent=true
RandomizedDelaySec=5min
Unit=backup.service

[Install]
WantedBy=timers.target
```

`Persistent=true`가 핵심이다. 시스템이 03:00에 꺼져 있었다면 부팅 후 즉시 실행해서 누락을 막는다. anacron 역할이다.

`RandomizedDelaySec`은 여러 대의 서버에서 동시에 같은 시각에 잡이 실행되어 외부 시스템(API, 백업 스토리지 등)에 부하가 몰리는 걸 분산시킨다.

### OnCalendar 표현식

cron보다 표현이 풍부하고 가독성이 좋다.

```ini
OnCalendar=hourly                        # 매시 정각
OnCalendar=daily                         # 매일 00:00
OnCalendar=weekly                        # 매주 월요일 00:00
OnCalendar=monthly                       # 매월 1일 00:00

OnCalendar=*-*-* 03:00:00                # 매일 03:00
OnCalendar=Mon..Fri 09:00                # 평일 09:00
OnCalendar=Mon,Wed,Fri *-*-* 14:30       # 월/수/금 14:30
OnCalendar=*-*-01,15 02:00               # 매월 1일과 15일 02:00
OnCalendar=*-*-* *:0/15:00               # 15분마다
OnCalendar=*-*-* 09..17:00:00            # 09시부터 17시까지 매시 정각
```

표현식이 의도대로 해석되는지는 `systemd-analyze calendar`로 검증한다.

```bash
$ systemd-analyze calendar "Mon..Fri 09:00"
  Original form: Mon..Fri 09:00
Normalized form: Mon..Fri *-*-* 09:00:00
    Next elapse: Mon 2026-05-04 09:00:00 KST
       From now: 3 days left
```

다음 실행 시각까지 보여줘서 cron 표현식 디버깅보다 훨씬 편하다.

### 타이머 활성화와 확인

```bash
systemctl daemon-reload
systemctl enable --now backup.timer

systemctl list-timers --all              # 모든 타이머와 다음 실행 시각
systemctl status backup.timer            # 타이머 상태
journalctl -u backup.service -n 100      # 서비스 실행 로그
journalctl -u backup.service --since "1 hour ago"
```

`list-timers` 출력에서 `LAST`와 `PASSED` 컬럼이 마지막 실행 시각이다. `NEXT`와 `LEFT`가 다음 실행이다. 의도한 대로 잡혔는지 한눈에 보인다.

### 타이머의 트리거 방식

`OnCalendar` 외에도 다양한 트리거가 있다.

```ini
[Timer]
OnBootSec=10min                          # 부팅 후 10분
OnUnitActiveSec=1h                       # 마지막 실행 후 1시간
OnUnitInactiveSec=30min                  # 비활성 상태 30분 경과
```

부팅 직후 한 번 실행하고 그 후 1시간마다 돌리는 워밍업 잡 같은 패턴은 cron으로는 어렵지만 timer로는 자연스럽다.

```ini
[Timer]
OnBootSec=10min
OnUnitActiveSec=1h
```

## at 명령

at은 일회성 작업 전용이다. cron은 반복용이라 "오늘 23시 한 번만 돌리고 끝"인 작업에는 과한 도구다.

```bash
echo "/opt/app/migrate.sh" | at 23:00
echo "shutdown -h now" | at now + 30 minutes
at -f /opt/app/cleanup.sh tomorrow
```

대화식으로도 쓸 수 있다.

```bash
$ at 14:30
warning: commands will be executed using /bin/sh
at> /opt/app/deploy.sh
at> echo "deploy done" | mail -s "deploy" ops@example.com
at> <Ctrl+D>
job 5 at Thu Apr 30 14:30:00 2026
```

대기 중인 작업 관리 명령이다.

```bash
atq                                       # 대기 중인 작업 목록
at -c 5                                   # 작업 5번의 내용 보기
atrm 5                                    # 작업 5번 삭제
```

at도 cron처럼 환경 변수가 빈약하다. 작업 등록 시점의 환경을 일부 기록해두긴 하지만 의존하지 않는 게 안전하다. 스크립트 안에서 PATH를 다시 잡는다.

`atd` 데몬이 떠 있어야 동작한다. 설치만 하고 활성화 안 한 경우가 있다.

```bash
systemctl enable --now atd
```

at은 사용자 권한 제어가 까다롭다. `/etc/at.allow`가 있으면 거기 있는 사용자만 사용 가능, 없고 `/etc/at.deny`가 있으면 거기 없는 사용자만 가능하다. 둘 다 없으면 root만 쓸 수 있다(배포판에 따라 다름).

운영 환경에서는 at보다 systemd의 transient timer를 더 권장한다.

```bash
systemd-run --on-active=30min /opt/app/cleanup.sh
systemd-run --on-calendar="2026-05-01 03:00" /opt/app/migrate.sh
```

journal에 로그가 남고, 실패 시 추적이 쉽다.

## 잡 중복 실행 방지 (flock)

스케줄러는 이전 실행이 끝났는지 신경 쓰지 않는다. 5분마다 도는 잡이 있는데 한 번 실행이 7분 걸리면, 두 번째 인스턴스가 첫 번째와 동시에 돈다. DB 락이 걸리거나 같은 파일을 쓰다가 깨지거나 이중 처리가 발생한다.

PID 파일로 직접 잠그는 패턴이 흔하지만 함정이 많다. 프로세스가 죽으면서 PID 파일이 안 지워지면 다음 실행이 못 돈다. 프로세스가 살아있는지 확인하는 코드도 race condition이 있다. `flock`이 정답이다.

`flock`은 파일 디스크립터 단위로 커널 레벨 락을 건다. 프로세스가 어떻게 죽든 커널이 락을 풀어준다. 별도 정리가 필요 없다.

### 가장 단순한 패턴

```bash
*/5 * * * * /usr/bin/flock -n /var/lock/myjob.lock /opt/app/job.sh
```

`-n`은 락을 못 잡으면 즉시 실패하라는 뜻이다. 이전 실행이 아직 돌고 있다면 이번 실행은 그냥 종료된다. 락 파일은 미리 만들 필요 없고 `flock`이 알아서 생성한다.

타임아웃을 두고 싶다면 `-w`를 쓴다.

```bash
*/5 * * * * /usr/bin/flock -w 60 /var/lock/myjob.lock /opt/app/job.sh
```

60초 안에 락을 잡으면 실행, 못 잡으면 실패한다.

### 스크립트 안에서 락 걸기

스크립트 자기 자신에 대해서 락을 거는 패턴이다. 스크립트 파일 자체를 락 대상으로 삼는다.

```bash
#!/bin/bash
exec 200>/var/lock/$(basename "$0").lock
flock -n 200 || { echo "이미 실행 중"; exit 1; }

# 실제 작업
do_backup
```

`exec 200>...`은 파일 디스크립터 200을 락 파일로 연다. `flock -n 200`이 그 디스크립터에 락을 건다. 스크립트가 끝나면 디스크립터가 닫히면서 락이 자동으로 풀린다.

### flock의 한계

flock은 같은 파일시스템 안에서만 동작한다. NFS는 동작이 보장되지 않는다(NFS 버전과 마운트 옵션에 따라 다름). 여러 호스트가 공유 스토리지를 통해 락을 걸려 한다면 flock 말고 다른 방법(Redis SETNX, etcd, 데이터베이스 락)을 써야 한다. 분산 환경 얘기는 다음 절에서 다룬다.

## 실패 알림

cron의 기본 알림은 메일이지만, 요즘 메일을 실시간으로 보는 사람은 거의 없다. 실패는 슬랙이나 모니터링 시스템으로 흘려야 즉시 알 수 있다.

### 단순한 패턴

스크립트 끝에서 종료 코드를 체크해서 알림을 보낸다.

```bash
#!/bin/bash
set -euo pipefail

trap 'notify_failure $LINENO $?' ERR

notify_failure() {
    local line=$1
    local code=$2
    curl -sS -X POST "$SLACK_WEBHOOK" \
        -H 'Content-Type: application/json' \
        -d "{\"text\": \"backup.sh 실패 (line=$line, exit=$code, host=$(hostname))\"}"
}

/opt/app/backup.sh
```

`trap '...' ERR`로 실패 시점을 잡는다. `set -e`와 함께 써야 ERR 트랩이 의도대로 동작한다.

### healthchecks.io 패턴

cron 잡이 실행됐는지를 외부에서 모니터링하는 패턴이다. healthchecks.io 같은 서비스가 대표적이고, 자체 운영도 어렵지 않다.

```bash
#!/bin/bash
HC_URL="https://hc-ping.com/<uuid>"

# 시작 신호
curl -fsS -m 10 --retry 3 "$HC_URL/start" > /dev/null

# 작업 실행
/opt/app/backup.sh
EXIT_CODE=$?

# 종료 신호 (exit 코드 포함)
curl -fsS -m 10 --retry 3 "$HC_URL/$EXIT_CODE" > /dev/null

exit $EXIT_CODE
```

이 방식의 장점은 잡이 "실행되지 않은" 경우도 잡아낸다는 점이다. cron 데몬이 죽었거나, crontab이 통째로 사라졌거나, 서버가 멈췄다면 healthchecks 쪽에서 시간이 지나도 ping이 안 와서 알람이 발생한다. 스크립트 안에서 알림 보내는 방식은 잡이 아예 실행 안 됐을 때 당연히 알림도 안 온다.

### systemd timer + OnFailure

systemd timer를 쓴다면 OnFailure 지시자로 알림 서비스를 트리거할 수 있다.

```ini
# /etc/systemd/system/backup.service
[Service]
Type=oneshot
ExecStart=/opt/app/backup.sh
OnFailure=notify-failure@%n.service
```

```ini
# /etc/systemd/system/notify-failure@.service
[Unit]
Description=Notify failure of %i

[Service]
Type=oneshot
ExecStart=/opt/scripts/slack-notify.sh %i
```

`%i`로 실패한 유닛 이름이 전달된다. 실패한 모든 유닛에 같은 알림을 붙일 수 있다.

## 분산 환경에서의 cron 충돌

여러 대의 동일한 서버에서 같은 cron 잡이 돌면 한 작업이 N번 실행된다. 백업이 5번 동시에 돌면서 같은 파일을 덮어쓰거나, 결제 정산이 중복으로 일어나거나, API 호출 한도를 초과한다. 오토스케일링으로 인스턴스 수가 변하는 환경에서는 더 심각하다.

해결 방법 몇 가지를 비교한다.

### 전용 스케줄러 노드

가장 단순하다. 동일한 애플리케이션 서버 N대 중 한 대에만 cron을 등록하거나, 아예 cron 전용 인스턴스 하나를 둔다.

문제는 그 한 대가 죽으면 잡이 통째로 안 돈다는 점이다. 단순한 만큼 단일 장애점이다. 새벽 잡 정도는 이걸로 충분한 경우가 많다.

### 분산 락

여러 노드가 같은 잡을 시작하려 할 때 첫 번째만 잡이 돌고 나머지는 종료한다. Redis의 `SET NX` 명령이 가장 흔하다.

```bash
#!/bin/bash
LOCK_KEY="cron:backup"
LOCK_TTL=3600                            # 1시간

# 락 획득 시도
if [ "$(redis-cli SET "$LOCK_KEY" "$(hostname)" NX EX $LOCK_TTL)" != "OK" ]; then
    echo "다른 노드가 잡을 실행 중"
    exit 0
fi

trap "redis-cli DEL $LOCK_KEY" EXIT

# 실제 작업
/opt/app/backup.sh
```

TTL이 핵심이다. 노드가 죽어서 락을 못 풀어도 일정 시간 지나면 풀린다. 잡 실행 시간보다 충분히 길게 잡되, 너무 길면 다음 실행이 또 막힌다. 잡이 TTL보다 오래 걸릴 수도 있다면 주기적으로 TTL을 갱신해야 한다(Redlock 알고리즘).

### 분산 잡 스케줄러 도입

규모가 커지면 cron 자체를 버리고 전용 스케줄러를 쓴다.

- Kubernetes의 `CronJob`: 클러스터 차원에서 한 번만 실행 보장. 누락 처리(`startingDeadlineSeconds`)와 동시성 정책(`concurrencyPolicy`) 제공.
- Nomad의 periodic job
- 애플리케이션 레벨 스케줄러: Quartz(Java), Celery beat(Python), Sidekiq cron(Ruby) 등

Kubernetes CronJob 예시다.

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: backup
spec:
  schedule: "0 2 * * *"
  startingDeadlineSeconds: 300            # 시작이 5분 이상 늦어지면 스킵
  concurrencyPolicy: Forbid               # 이전 실행이 돌고 있으면 새로 실행 안 함
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 5
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: backup
            image: myapp/backup:latest
            command: ["/opt/app/backup.sh"]
```

`concurrencyPolicy`의 세 옵션이 핵심이다.

- `Allow`: 동시 실행 허용 (기본값)
- `Forbid`: 이전 실행이 안 끝났으면 새 실행 스킵
- `Replace`: 이전 실행을 죽이고 새 실행

`startingDeadlineSeconds`는 누락 작업의 처리 한계 시간이다. 이 값을 너무 짧게 잡으면 컨트롤러가 잠깐 죽었다 살아났을 때 그 사이 잡이 영구 누락된다. 너무 길게 잡으면 누적된 잡이 한꺼번에 터진다.

### 시간대와 클럭 드리프트

분산 환경에서는 노드들의 시간이 미세하게 다르다. NTP가 동기화하지만 1초 단위 차이는 흔하다. cron이 정각에 일제히 트리거되는 잡은 어떤 노드에서 먼저 시작될지 모른다. flock이나 분산 락이 충돌을 막아주지만, 락 경합 자체에서 발생하는 부하가 있을 수 있다.

`RandomizedDelaySec`(systemd) 또는 잡 시작 부분에서 의도적인 랜덤 sleep을 넣어서 분산시키는 방법이 자주 쓰인다.

```bash
sleep $((RANDOM % 300))                  # 0-300초 랜덤 대기
/opt/app/job.sh
```

## 트러블슈팅

cron 잡이 안 돌 때 점검 순서다.

```bash
# 1. cron 데몬이 살아있는지
systemctl status cron       # Debian/Ubuntu
systemctl status crond      # RHEL 계열

# 2. crontab이 정상 등록됐는지
crontab -l

# 3. 실행 로그
journalctl -u cron --since "1 hour ago"
grep CRON /var/log/syslog | tail -20

# 4. 명령을 cron 환경에서 직접 흉내내서 실행
env -i HOME="$HOME" PATH=/usr/bin:/bin /bin/sh -c '/opt/app/script.sh'
```

마지막 명령이 핵심이다. `env -i`로 환경을 비우고 cron이 주는 최소 환경만 만든 후 실행한다. 쉘에서는 잘 도는데 cron에서 안 도는 문제가 거의 다 이걸로 재현된다.

### 출력이 사라질 때

cron 잡의 stdout/stderr이 어디로도 안 가는 경우가 있다. MAILTO=""로 메일 비활성화 + 리다이렉트 미설정 조합이다. 잡이 실패해도 단서가 안 남는다. 일단 모든 잡을 파일로 리다이렉트하고 운영 안정화 후 chronic 같은 도구로 정리한다.

### 시간대 문제

cron은 시스템 시간대를 따른다. UTC로 운영되는 서버에 한국 시간 기준으로 02:00을 잡으면 실제로는 11:00에 돈다. `timedatectl`로 시스템 시간대를 확인하거나, cron 라인에 TZ를 명시한다.

```bash
CRON_TZ=Asia/Seoul                       # 일부 cron 구현체만 지원

0 2 * * * /opt/app/job.sh
```

`CRON_TZ`는 cronie(RHEL 계열) 등 일부 구현에서만 동작한다. 호환성이 걱정되면 스크립트 안에서 `TZ=Asia/Seoul`을 설정하고 실행한다.

systemd timer는 `OnCalendar`에 시간대를 직접 지정할 수 있다.

```ini
OnCalendar=Asia/Seoul *-*-* 02:00:00
```

## 정리

cron의 함정은 크게 세 종류다. 환경 변수가 빈약해서 명령을 못 찾는 문제, 출력이 메일로 가다가 무시되어 에러를 모르는 문제, 동시 실행과 누락 실행의 처리 부재. 각각 PATH 명시·MAILTO 관리·flock과 healthchecks·systemd timer의 Persistent로 대응한다.

규모가 작으면 cron + flock + healthchecks 조합이 가성비가 좋다. 누락 처리가 중요하면 systemd timer로 옮긴다. 분산 환경이라면 Kubernetes CronJob이나 애플리케이션 레벨 스케줄러를 검토한다. 도구 선택보다 잡이 실제로 돌고 있는지 외부에서 검증하는 체계가 더 중요하다.
