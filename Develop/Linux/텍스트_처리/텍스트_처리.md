# 텍스트 처리

## 개요

텍스트 파일을 검색, 필터링, 변환하는 명령어. 로그 분석과 데이터 처리에 필수다.

grep, sed, awk 세 도구가 핵심이고, 나머지(cut, sort, uniq 등)는 파이프로 조합해서 쓴다. 각 도구의 역할이 겹치는 부분이 있는데, 어떤 상황에서 뭘 써야 하는지가 실무에서 제일 중요하다.

---

## grep

패턴으로 텍스트를 검색한다.

### 기본 사용

```bash
grep "pattern" file.txt
grep -i "pattern" file.txt          # 대소문자 무시
grep -v "pattern" file.txt          # 매치되지 않는 줄
grep -n "pattern" file.txt          # 줄 번호 포함
grep -c "pattern" file.txt          # 매치 개수만
grep -l "pattern" *.txt             # 파일명만 출력
grep -r "pattern" /path             # 재귀적 검색
```

### 정규표현식

```bash
grep "^pattern" file.txt            # 줄 시작
grep "pattern$" file.txt            # 줄 끝
grep "p.ttern" file.txt             # .은 임의 문자
grep "p[aeiou]ttern" file.txt       # 문자 클래스
grep "p[0-9]ttern" file.txt         # 숫자
grep -E "pattern1|pattern2" file.txt # 확장 정규표현식
```

`-E` 옵션으로 확장 정규표현식을 사용한다. `egrep`과 동일하다.

### 컨텍스트 옵션

```bash
grep -A 3 "pattern" file.txt       # 매치 후 3줄
grep -B 3 "pattern" file.txt       # 매치 전 3줄
grep -C 3 "pattern" file.txt       # 매치 전후 3줄
grep -w "pattern" file.txt         # 단어 단위 매치
grep -f patterns.txt file.txt      # 패턴 파일 사용
```

`-A`, `-B`, `-C`는 로그 분석 시 에러 전후 맥락을 확인할 때 쓴다. 예를 들어 Exception이 터졌을 때 그 위 5줄에 있는 요청 정보를 같이 보려면 `-B 5`를 쓴다.

### grep 실무 팁

**grep vs fgrep vs egrep**

- `grep -F` (= `fgrep`): 정규표현식 해석 없이 고정 문자열로 검색한다. `.`이나 `*`가 포함된 문자열을 검색할 때 이스케이프 없이 쓸 수 있다.
- `grep -E` (= `egrep`): `+`, `?`, `|`, `()` 같은 확장 정규표현식을 이스케이프 없이 쓸 수 있다.

```bash
# URL에서 마침표를 리터럴로 검색하려면
grep -F "api.example.com" access.log     # 정규표현식 해석 안 함
grep "api\.example\.com" access.log      # 이스케이프 필요
```

**--color와 --include로 범위 좁히기**

```bash
# 특정 확장자만 검색
grep -rn --include="*.java" "NullPointerException" /app/src

# 특정 디렉토리 제외
grep -rn --exclude-dir=node_modules "TODO" .

# 바이너리 파일 무시
grep -rI "pattern" /path    # -I는 바이너리 파일을 건너뜀
```

대규모 프로젝트에서 `grep -r`을 쓰면 바이너리 파일까지 뒤져서 느려진다. `--include`나 `-I` 옵션을 습관적으로 붙인다.

## sed

텍스트 스트림을 편집한다.

### 치환

```bash
sed 's/old/new/' file.txt          # 첫 번째 매치만 치환
sed 's/old/new/g' file.txt         # 모든 매치 치환
sed 's/old/new/2' file.txt         # 2번째 매치만 치환
sed -i 's/old/new/g' file.txt      # 파일 직접 수정
sed -i.bak 's/old/new/g' file.txt  # 백업 파일 생성
```

`-i` 옵션은 파일을 직접 수정한다. 백업을 만들거나 테스트 후 사용한다.

macOS의 sed는 BSD 기반이라 `-i` 뒤에 빈 문자열을 명시해야 한다: `sed -i '' 's/old/new/g' file.txt`. Linux(GNU sed)에서는 그냥 `-i`만 써도 된다. 이 차이 때문에 스크립트가 다른 OS에서 깨지는 경우가 많다.

### 삭제

```bash
sed '5d' file.txt                  # 5번째 줄 삭제
sed '1,5d' file.txt                # 1-5번째 줄 삭제
sed '/pattern/d' file.txt          # 패턴 매치 줄 삭제
sed '/^$/d' file.txt               # 빈 줄 삭제
```

### 추가/삽입

```bash
sed '5a\new line' file.txt         # 5번째 줄 뒤에 추가
sed '5i\new line' file.txt         # 5번째 줄 앞에 삽입
sed '$a\new line' file.txt         # 마지막 줄에 추가
```

### 인쇄

```bash
sed -n '5p' file.txt               # 5번째 줄만 출력
sed -n '1,5p' file.txt             # 1-5번째 줄 출력
sed -n '/pattern/p' file.txt       # 패턴 매치 줄만 출력
```

`-n` 옵션과 `p` 명령어를 조합하여 특정 줄만 출력한다.

### 다중 명령어

```bash
sed -e 's/old1/new1/' -e 's/old2/new2/' file.txt
sed 's/old1/new1/; s/old2/new2/' file.txt
```

### Pattern Space와 Hold Space

sed를 레퍼런스 수준으로만 쓰면 단순 치환/삭제까지만 가능하다. 여러 줄에 걸친 처리가 필요하면 sed의 내부 동작을 이해해야 한다.

sed는 입력을 한 줄씩 읽어서 처리하는데, 내부에 두 개의 버퍼를 사용한다.

```
┌─────────────────────────────────────────────────┐
│                 sed 처리 흐름                      │
├─────────────────────────────────────────────────┤
│                                                 │
│   입력 스트림                                     │
│       │                                         │
│       ▼                                         │
│   ┌──────────────┐                              │
│   │ Pattern Space │ ← 현재 처리 중인 줄            │
│   │  (작업 공간)   │                              │
│   └──────┬───────┘                              │
│          │  h/H: pattern → hold                 │
│          │  g/G: hold → pattern                 │
│          │  x  : 서로 교환                        │
│          ▼                                      │
│   ┌──────────────┐                              │
│   │  Hold Space  │ ← 임시 저장소                  │
│   │  (임시 보관)   │                              │
│   └──────────────┘                              │
│          │                                      │
│          ▼                                      │
│   출력 스트림                                     │
│                                                 │
└─────────────────────────────────────────────────┘
```

- **Pattern Space**: 현재 처리 중인 줄이 들어가는 작업 공간. sed의 모든 명령(`s`, `d`, `p` 등)은 이 공간에서 실행된다.
- **Hold Space**: 나중에 쓰려고 잠시 보관하는 임시 저장소. 초기값은 빈 줄(`\n`)이다.

**버퍼 조작 명령어**

| 명령 | 동작 |
|------|------|
| `h` | Pattern Space → Hold Space (덮어쓰기) |
| `H` | Pattern Space → Hold Space (줄바꿈 후 추가) |
| `g` | Hold Space → Pattern Space (덮어쓰기) |
| `G` | Hold Space → Pattern Space (줄바꿈 후 추가) |
| `x` | Pattern Space ↔ Hold Space 교환 |
| `N` | 다음 줄을 Pattern Space에 줄바꿈으로 추가 |

**실제 사용 예: 줄 순서 뒤집기**

`tac` 명령이 없는 환경에서 파일 내용을 역순으로 출력하는 경우:

```bash
sed -n '1!G;h;$p' file.txt
```

한 줄씩 풀면 이렇다:

1. `1!G` — 첫 줄이 아니면 Hold Space 내용을 Pattern Space 뒤에 추가
2. `h` — 현재 Pattern Space를 Hold Space에 저장
3. `$p` — 마지막 줄이면 출력

매 줄마다 Hold Space에 쌓아두고, 마지막에 한 번에 출력하는 구조다.

**실제 사용 예: 연속된 빈 줄을 하나로 압축**

```bash
sed '/^$/{ N; /^\n$/d }' file.txt
```

빈 줄을 만나면 다음 줄(`N`)을 가져와서, 그것도 빈 줄이면 삭제한다. `cat -s`와 같은 동작이다.

**실제 사용 예: 특정 패턴 사이의 블록 추출**

설정 파일에서 특정 섹션만 뽑아야 할 때:

```bash
# [database] 섹션부터 다음 섹션 시작 전까지 추출
sed -n '/^\[database\]/,/^\[/{ /^\[database\]/p; /^\[/!p }' config.ini
```

## awk

텍스트를 필드 단위로 처리한다. sed가 줄 단위 변환에 특화돼 있다면, awk는 구조화된 데이터의 필드 조작에 특화돼 있다.

### 기본 사용

```bash
awk '{print $1}' file.txt          # 첫 번째 필드 출력
awk '{print $1, $3}' file.txt      # 1, 3번째 필드 출력
awk '{print $NF}' file.txt         # 마지막 필드 출력
awk -F: '{print $1}' file.txt      # 구분자 지정 (:)
```

기본 구분자는 공백이다. `-F`로 구분자를 지정한다.

### 패턴 매칭

```bash
awk '/pattern/ {print}' file.txt
awk '$1 == "value" {print}' file.txt
awk '$3 > 100 {print}' file.txt
```

### BEGIN/END

```bash
awk 'BEGIN {FS=":"} {print $1}' file.txt
awk '{sum+=$1} END {print sum}' file.txt
```

`BEGIN`에서 초기화하고, `END`에서 결과를 출력한다.

### 내장 변수

| 변수 | 의미 |
|------|------|
| `NR` | 현재 줄 번호 (전체 기준) |
| `FNR` | 현재 파일 내 줄 번호 |
| `NF` | 현재 줄의 필드 수 |
| `$0` | 현재 줄 전체 |
| `FS` | 입력 필드 구분자 |
| `OFS` | 출력 필드 구분자 |
| `RS` | 입력 레코드 구분자 |
| `ORS` | 출력 레코드 구분자 |
| `FILENAME` | 현재 처리 중인 파일명 |

### 배열(Associative Array)

awk의 배열은 자바의 HashMap, 파이썬의 dict와 같다. 인덱스가 문자열이다.

```bash
# IP별 요청 횟수 집계
awk '{count[$1]++} END {for (ip in count) print ip, count[ip]}' access.log
```

이 한 줄이 하는 일:

1. 매 줄마다 첫 번째 필드(IP)를 키로 count 배열에 1을 더한다
2. 파일 끝에서 배열을 순회하며 IP와 횟수를 출력한다

**배열 활용 예제**

```bash
# HTTP 상태 코드별 집계
awk '{status[$9]++} END {for (s in status) print s, status[s]}' access.log

# 시간대별 요청 분포 (access.log의 시간 필드에서 시(hour)만 추출)
awk -F'[/ :]' '{hour=$7; traffic[hour]++} END {for (h in traffic) print h"시", traffic[h]"건"}' access.log

# 여러 필드를 키로 사용 — SUBSEP으로 결합
awk '{key = $1 SUBSEP $9; count[key]++} END {
    for (k in count) {
        split(k, arr, SUBSEP)
        print arr[1], arr[2], count[k]
    }
}' access.log
```

**배열에서 요소 삭제**

```bash
# delete로 특정 키 삭제
awk '{count[$1]++} END {
    delete count["127.0.0.1"]
    for (ip in count) print ip, count[ip]
}' access.log
```

**배열 존재 여부 확인**

```bash
# in 연산자로 확인
awk '{
    if ($1 in blocked) next
    print
}' blocked_ips.txt access.log
```

주의: `if (arr[key])` 이렇게 쓰면 키가 없을 때 빈 값으로 자동 생성된다. 존재 여부만 확인하려면 반드시 `in` 연산자를 쓴다.

### 사용자 정의 함수

awk 스크립트가 길어지면 함수로 분리한다.

```bash
awk '
function bytes_to_human(bytes) {
    if (bytes >= 1073741824) return sprintf("%.1fG", bytes/1073741824)
    if (bytes >= 1048576) return sprintf("%.1fM", bytes/1048576)
    if (bytes >= 1024) return sprintf("%.1fK", bytes/1024)
    return bytes "B"
}

{
    size[$1] += $10
}

END {
    for (ip in size) print ip, bytes_to_human(size[ip])
}
' access.log
```

**여러 함수를 조합한 로그 분석**

```bash
awk '
function parse_timestamp(ts) {
    # [10/Apr/2026:13:55:36 형태에서 시간만 추출
    split(ts, parts, ":")
    return parts[2]
}

function percentile(arr, n, pct,    i) {
    # 단순 근사 — 정렬된 배열에서 pct 위치값 반환
    i = int(n * pct / 100) + 1
    if (i > n) i = n
    return arr[i]
}

{
    hour = parse_timestamp($4)
    resp_time = $NF + 0    # 문자열을 숫자로 변환
    hourly_count[hour]++
    hourly_total[hour] += resp_time
}

END {
    for (h in hourly_count) {
        avg = hourly_total[h] / hourly_count[h]
        printf "%s시 — 요청 %d건, 평균 응답 %.3f초\n", h, hourly_count[h], avg
    }
}
' access.log
```

### printf로 포맷 출력

`print` 대신 `printf`를 쓰면 출력 형식을 정밀하게 제어한다.

```bash
# 고정 너비 테이블 형태 출력
awk 'BEGIN {printf "%-15s %10s %10s\n", "IP", "요청수", "전송량"}
{
    count[$1]++; bytes[$1]+=$10
}
END {
    for (ip in count)
        printf "%-15s %10d %10d\n", ip, count[ip], bytes[ip]
}' access.log
```

## grep, sed, awk 선택 기준

세 도구는 역할이 겹치는 부분이 있다. 실무에서 어떤 걸 쓸지 고르는 기준이다.

```
┌─────────────────────────────────────────────────────────┐
│              작업 복잡도에 따른 도구 선택                    │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  단순 검색/필터링          필드 추출/치환        집계/변환   │
│  ──────────────          ─────────────       ─────────  │
│  grep                    sed, cut            awk        │
│                                                         │
│  "이 패턴이 있는 줄?"     "이 부분을 바꿔라"     "계산해라" │
│                                                         │
│  ◄──────────────────────────────────────────────────►   │
│  빠름                                             느림   │
│  단순                                             복잡   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 성능 차이

같은 작업을 세 도구로 할 수 있을 때, 속도 차이가 있다.

**단순 패턴 검색** — 1GB 로그 파일 기준 대략적인 처리 시간:

```bash
# 가장 빠름 — grep은 패턴 검색에 최적화돼 있다
grep "ERROR" large.log                    # ~1초

# 약간 느림 — sed는 줄 단위 처리 오버헤드가 있다
sed -n '/ERROR/p' large.log               # ~2-3초

# 가장 느림 — awk는 필드 파싱까지 한다
awk '/ERROR/' large.log                   # ~3-5초
```

수치는 환경에 따라 다르지만, 순서는 일관적이다. grep > sed > awk 순으로 빠르다.

**고정 문자열 검색이면 fgrep이 압도적으로 빠르다**

```bash
grep -F "specific.error.code" large.log   # 정규표현식 엔진을 안 쓰므로 빠름
```

### 언제 뭘 쓰는가

| 상황 | 도구 | 이유 |
|------|------|------|
| "이 문자열이 있는 줄을 찾아라" | `grep` | 검색에 특화, 가장 빠름 |
| "이 패턴을 다른 문자열로 바꿔라" | `sed` | 스트림 치환이 본업 |
| "3번째 필드가 100 이상인 줄만" | `awk` | 필드 단위 조건 처리 |
| "필드별 합계/평균을 구해라" | `awk` | 연산과 집계 기능 |
| "여러 줄에 걸친 패턴 변환" | `sed` | Hold Space로 멀티라인 처리 |
| "구분자 기반 단순 필드 추출" | `cut` | 가장 단순하고 빠름 |
| "복잡한 리포트 생성" | `awk` | 변수, 배열, 함수 지원 |

원칙: **할 수 있는 가장 단순한 도구를 쓴다.** grep으로 되는 걸 awk로 하면 속도도 느리고 읽기도 어렵다.

## 파이프 조합 — 실제 서버 로그 파싱

실무에서는 단일 도구로 끝나는 경우가 드물다. 파이프로 여러 도구를 조합한다.

### Nginx access.log 분석

Nginx의 기본 로그 포맷:

```
192.168.1.100 - - [10/Apr/2026:13:55:36 +0900] "GET /api/users HTTP/1.1" 200 1234 "-" "Mozilla/5.0"
```

**상위 10개 IP 추출**

```bash
awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -10
```

파이프 각 단계:

1. `awk '{print $1}'` — IP 필드만 추출
2. `sort` — 정렬 (uniq가 동작하려면 정렬 필수)
3. `uniq -c` — 중복 카운트
4. `sort -rn` — 카운트 기준 내림차순 정렬
5. `head -10` — 상위 10개만

**특정 시간대의 5xx 에러만 추출**

```bash
grep "10/Apr/2026:14:" access.log | awk '$9 >= 500 {print $7, $9}'
```

grep으로 시간대를 먼저 필터링하고, awk로 상태 코드가 500 이상인 줄에서 URI와 상태 코드를 뽑는다.

**분 단위 요청 추이 확인**

```bash
awk -F'[/ :]' '{print $8":"$9}' access.log | sort | uniq -c
```

시간:분 단위로 묶어서 요청 수를 세면, 트래픽 급증 시점을 찾을 수 있다.

### 애플리케이션 로그 분석

Java/Spring 로그 형태:

```
2026-04-10 13:55:36.123 ERROR [http-nio-8080-exec-1] c.e.UserService - User not found: id=12345
2026-04-10 13:55:36.456  WARN [http-nio-8080-exec-2] c.e.PaymentService - Timeout after 3000ms
```

**에러별 발생 빈도**

```bash
grep "ERROR" app.log | sed 's/.*] //' | sort | uniq -c | sort -rn | head -20
```

1. `grep "ERROR"` — 에러 줄만 필터
2. `sed 's/.*] //'` — 타임스탬프와 스레드 정보를 잘라내고 메시지만 남김
3. `sort | uniq -c | sort -rn` — 빈도순 정렬

**시간대별 에러 건수**

```bash
grep "ERROR" app.log | awk '{print substr($2, 1, 2)"시"}' | sort | uniq -c
```

**특정 사용자 ID의 요청 추적**

```bash
grep "id=12345" app.log | awk '{print $1, $2, $4, $6}'
```

### 시스템 로그 분석

**SSH 접속 실패 IP 추출 (brute force 탐지)**

```bash
grep "Failed password" /var/log/auth.log | \
    awk '{print $(NF-3)}' | \
    sort | uniq -c | sort -rn | head -10
```

10회 이상 실패한 IP만 보려면:

```bash
grep "Failed password" /var/log/auth.log | \
    awk '{print $(NF-3)}' | \
    sort | uniq -c | sort -rn | \
    awk '$1 >= 10'
```

**디스크 사용량 모니터링**

```bash
df -h | awk 'NR>1 {
    gsub(/%/, "", $5)
    if ($5+0 > 80) printf "경고: %s 사용률 %s%%\n", $6, $5
}'
```

### JSON 로그 처리

최근 애플리케이션은 JSON 형태로 로그를 남기는 경우가 많다. `jq`가 있으면 쓰는 게 맞지만, 없는 환경에서 급하게 처리할 때:

```bash
# {"level":"ERROR","message":"timeout","service":"payment"} 형태
grep '"level":"ERROR"' app.json.log | \
    sed 's/.*"message":"\([^"]*\)".*/\1/' | \
    sort | uniq -c | sort -rn
```

이건 임시방편이다. JSON을 정규표현식으로 파싱하면 이스케이프된 따옴표나 중첩 구조에서 깨진다. 가능하면 `jq`를 설치한다.

## 정규표현식 디버깅 — 흔한 실수와 해결법

grep, sed, awk 모두 정규표현식을 쓰는데, 같은 패턴이 도구마다 다르게 동작하는 경우가 있다.

### BRE vs ERE

```
┌───────────────────────────────────────────────────────┐
│        정규표현식 방언(dialect) 차이                      │
├───────────────┬───────────────┬───────────────────────┤
│   메타문자      │  BRE (기본)    │  ERE (확장)            │
├───────────────┼───────────────┼───────────────────────┤
│  그룹핑        │  \( \)        │  ( )                  │
│  OR           │  \|           │  |                    │
│  1회 이상      │  \+           │  +                    │
│  0/1회        │  \?           │  ?                    │
│  반복 횟수     │  \{n,m\}      │  {n,m}                │
├───────────────┴───────────────┴───────────────────────┤
│  grep   → BRE 기본,  grep -E → ERE                    │
│  sed    → BRE 기본,  sed -E → ERE                     │
│  awk    → ERE 기본 (항상)                               │
└───────────────────────────────────────────────────────┘
```

이걸 모르면 이런 실수가 나온다:

```bash
# 실수: grep에서 +를 이스케이프 안 함 → 매치 안 됨
grep "[0-9]+" file.txt              # BRE에서 +는 리터럴

# 수정 1: 이스케이프
grep "[0-9]\+" file.txt

# 수정 2: ERE 사용
grep -E "[0-9]+" file.txt
```

### 흔한 실수 모음

**1. 탐욕적 매칭(Greedy Matching)**

```bash
# 의도: 첫 번째 따옴표 쌍만 추출
echo '"hello" and "world"' | grep -oE '".*"'
# 결과: "hello" and "world"  ← 전체가 매치됨

# 수정: 비탐욕적 매칭은 grep에 없다. 문자 클래스로 우회
echo '"hello" and "world"' | grep -oE '"[^"]*"'
# 결과: "hello"
#        "world"
```

`.*`은 가능한 한 많이 매치한다(탐욕적). `[^"]*`로 "따옴표가 아닌 문자"만 매치하게 바꾼다. 이건 grep/sed/awk 공통으로 적용되는 패턴이다. PCRE(`grep -P`)를 쓸 수 있으면 `.*?` 비탐욕적 매칭이 가능하지만, 모든 시스템에 있는 건 아니다.

**2. 특수문자 이스케이프 누락**

```bash
# 의도: IP 주소 192.168.1.1 검색
grep "192.168.1.1" file.txt
# 문제: .이 임의 문자와 매치되어 192x168y1z1도 매치됨

# 수정: 마침표 이스케이프
grep "192\.168\.1\.1" file.txt

# 또는 고정 문자열 검색
grep -F "192.168.1.1" file.txt
```

**3. 앵커(^, $) 위치 실수**

```bash
# 의도: error로 시작하는 줄
grep "^error" file.txt    # 맞음

# 실수: 괄호 안의 ^는 부정(negation) 의미
grep "[^error]" file.txt  # e, r, o가 아닌 문자가 있는 줄 ← 전혀 다른 의미
```

**4. sed에서 구분자 충돌**

```bash
# 의도: 경로 치환
sed 's/\/usr\/local\/bin/\/opt\/bin/g' file.txt   # 읽기 어려움

# 수정: 다른 구분자 사용 (어떤 문자든 가능)
sed 's|/usr/local/bin|/opt/bin|g' file.txt         # 깔끔함
sed 's#/usr/local/bin#/opt/bin#g' file.txt         # 이것도 됨
```

슬래시가 포함된 경로를 치환할 때 `|`나 `#`을 구분자로 쓰면 이스케이프 지옥에서 벗어난다.

**5. awk에서 정규표현식 매칭 연산자**

```bash
# == 은 문자열 완전 일치
awk '$1 == "error" {print}' file.txt    # $1이 정확히 "error"인 줄

# ~ 은 정규표현식 매칭
awk '$1 ~ /error/ {print}' file.txt     # $1에 "error"가 포함된 줄

# !~ 은 정규표현식 불일치
awk '$1 !~ /error/ {print}' file.txt    # $1에 "error"가 없는 줄
```

이 차이를 모르면 `==`로 부분 매칭을 기대하다가 결과가 안 나온다.

### 정규표현식 테스트 방법

복잡한 패턴을 한 번에 작성하면 디버깅이 어렵다. 단계적으로 확인한다.

```bash
# 1단계: 간단한 패턴으로 대상 줄을 좁힌다
grep "ERROR" app.log | head -5

# 2단계: 패턴을 구체화한다
grep -E "ERROR.*timeout" app.log | head -5

# 3단계: -o로 매치된 부분만 확인한다 (의도대로 매치되는지)
grep -oE "[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+" access.log | head -5

# 4단계: 최종 파이프라인에 넣는다
grep -oE "[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+" access.log | sort | uniq -c | sort -rn
```

`-o` 옵션은 매치된 부분만 출력한다. 패턴이 의도대로 동작하는지 확인할 때 필수다.

## 기타 텍스트 처리 도구

### cut

필드를 추출한다.

```bash
cut -d: -f1 /etc/passwd            # 구분자로 필드 추출
cut -f1,3 file.txt                 # 1, 3번째 필드
cut -c1-10 file.txt                # 1-10번째 문자
cut -d',' -f2 file.csv
```

간단한 필드 추출에는 `cut`이 빠르다. 복잡한 처리는 `awk`를 사용한다.

### sort

텍스트를 정렬한다.

```bash
sort file.txt
sort -n file.txt                   # 숫자 정렬
sort -r file.txt                   # 역순 정렬
sort -u file.txt                   # 중복 제거
sort -k2 file.txt                  # 2번째 필드로 정렬
sort -t: -k3 -n /etc/passwd
```

`-n` 옵션 없이 숫자를 정렬하면 문자열로 정렬된다. 9가 10보다 뒤에 오는 걸 보고 당황하는 경우가 많다.

### uniq

중복된 줄을 제거한다.

```bash
uniq file.txt                      # 연속된 중복만 제거
uniq -c file.txt                   # 개수 포함
uniq -d file.txt                   # 중복된 줄만
uniq -u file.txt                   # 중복되지 않은 줄만
sort file.txt | uniq               # 모든 중복 제거
```

`uniq`는 연속된 중복만 제거한다. 파일 전체의 중복을 없애려면 반드시 `sort`를 먼저 거쳐야 한다. 이걸 모르고 `uniq`만 쓰면 중복이 그대로 남는다.

### wc

줄, 단어, 바이트 수를 센다.

```bash
wc file.txt                        # 줄, 단어, 바이트
wc -l file.txt                     # 줄 수만
wc -w file.txt                     # 단어 수만
wc -c file.txt                     # 바이트 수만
wc -m file.txt                     # 문자 수
```

### head / tail

파일의 앞부분/뒷부분을 출력한다.

```bash
head file.txt                      # 처음 10줄
head -n 20 file.txt
tail file.txt                      # 마지막 10줄
tail -n 20 file.txt
tail -f file.txt                   # 실시간 모니터링
tail -F file.txt                   # 파일 재생성 시에도 모니터링
```

`tail -f`와 `tail -F`의 차이: `-f`는 파일 디스크립터를 따라가고, `-F`는 파일명을 따라간다. 로그 로테이션으로 파일이 새로 생성되면 `-f`는 끊기고 `-F`는 새 파일을 따라간다. 운영 환경에서는 `-F`를 쓴다.

### tr

문자를 변환하거나 삭제한다.

```bash
tr 'a-z' 'A-Z' < file.txt          # 소문자를 대문자로
tr -d '\n' < file.txt               # 줄바꿈 삭제
tr -s ' ' < file.txt               # 연속된 공백을 하나로
tr -d '0-9' < file.txt             # 숫자 삭제
```

### paste / join

```bash
# paste: 파일을 열 단위로 병합
paste file1.txt file2.txt
paste -d: file1.txt file2.txt       # 구분자 지정

# join: 두 파일을 공통 필드로 결합 (두 파일 모두 정렬 필수)
join file1.txt file2.txt
join -1 2 -2 1 file1.txt file2.txt  # 필드 지정
```

## 실무 파이프라인 레시피

자주 쓰는 조합을 정리한다.

### 로그에서 느린 요청 찾기

```bash
# 응답 시간이 3초 이상인 요청의 URI와 응답 시간
awk '$NF > 3.0 {printf "%s %s %.2fs\n", $4, $7, $NF}' access.log | sort -t' ' -k3 -rn | head -20
```

### CSV 데이터 처리

```bash
# CSV에서 3번째 컬럼 합계 (헤더 스킵)
tail -n +2 data.csv | awk -F',' '{sum += $3} END {print sum}'

# 특정 조건 필터링 후 새 CSV 생성
awk -F',' 'NR==1 || $4 > 1000' data.csv > filtered.csv
```

### 설정 파일 일괄 수정

```bash
# 모든 .conf 파일에서 옛날 서버 주소를 새 주소로 변경
find /etc/app/ -name "*.conf" -exec sed -i 's/old-server\.example\.com/new-server\.example\.com/g' {} +
```

### 실시간 로그 모니터링 + 필터링

```bash
# 에러만 실시간 감시
tail -F /var/log/app.log | grep --line-buffered "ERROR"

# 에러 발생 시 IP와 URI 함께 표시
tail -F access.log | awk '$9 >= 500 {print strftime("%H:%M:%S"), $1, $7, $9}'
```

`--line-buffered`가 없으면 grep이 버퍼링해서 출력이 지연된다. 파이프 중간에 grep을 쓸 때는 항상 붙인다.
