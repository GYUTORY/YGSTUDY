# MVP (Minimum Viable Product)

MVP라는 말이 너무 흔해져서 의미가 닳았다. 누군가는 "기능 다 빼고 빨리 출시하는 거"라고 말하고, 누군가는 "데모용 화면 몇 개 띄우는 거"라고 말한다. 둘 다 틀렸다. MVP는 *학습을 위한 최소 제품*이다. 가설을 검증하기 위해 만드는 것이지, 시간을 줄이려고 만드는 것이 아니다.

이 문서는 백엔드 개발자가 MVP를 실제로 짤 때 겪는 문제를 정리한다. 어디까지 빼도 되는지, 무엇은 절대 빼면 안 되는지, 그리고 짠 코드가 버려질 수도 있다는 전제로 어떻게 만들어야 하는지에 관한 이야기다.

---

## 1. MVP의 정의

MVP는 Eric Ries의 *The Lean Startup*에서 나온 용어다. 원문 정의는 이렇다.

> "The minimum viable product is that version of a new product which allows a team to collect the maximum amount of validated learning about customers with the least effort."

핵심은 두 가지다.

- **검증된 학습(validated learning)을 얻는다**
- **노력은 최소화한다**

여기서 "최소"는 기능의 수가 아니라 학습 비용이다. 기능을 10개 만들든 1개 만들든, 가설 검증에 필요한 만큼이면 그게 MVP다. 기능을 줄였다고 MVP가 되는 건 아니다. 가설이 없으면 MVP도 없다.

### MVP가 검증하는 가설

가설은 보통 세 가지 중 하나다.

1. **문제 가설** — 사용자에게 정말 이 문제가 있는가
2. **솔루션 가설** — 이 방식으로 그 문제가 해결되는가
3. **시장 가설** — 사람들이 돈을 낼 만큼 가치 있다고 느끼는가

대부분의 실패한 MVP는 가설을 명시하지 않고 시작한다. "일단 만들어보고 반응을 본다"는 말은 가설이 아니라 기도다.

---

## 2. MVP가 아닌 것

MVP라는 단어가 남용되는 이유는 비슷한 다른 개념들과 섞여서다. 셋을 명확히 구분해야 한다.

### 프로토타입 (Prototype)

내부에서 보여주기 위한 것. 사용자가 쓰지 않는다. 클릭하면 동작하는 척만 해도 된다. Figma 프로토타입, 와이어프레임, 종이에 그린 화면 모두 프로토타입이다.

목적은 *우리끼리 합의*다. 디자이너와 PM, 개발자가 같은 그림을 보기 위해 만든다.

### PoC (Proof of Concept)

기술적으로 가능한지를 확인하는 것. 사용자가 쓰지 않는다. 한 번 돌려보고 버린다. "이 API로 정말 1초 안에 응답이 오나?", "이 라이브러리가 우리 데이터 크기를 감당하나?" 같은 질문에 답하기 위해 만든다.

PoC는 코드 품질을 신경 쓰지 않는다. 하드코딩된 값, 인증 우회, 에러 무시 다 허용된다. 어차피 검증만 끝나면 폐기한다.

### MVP

실제 사용자가 사용한다. 그 사용 데이터로 가설을 검증한다. 프로토타입이나 PoC와 결정적으로 다른 점은 *외부 사용자가 만진다*는 것이다.

|구분|사용자|목적|코드 수명|품질 기준|
|---|---|---|---|---|
|프로토타입|내부|시각적 합의|즉시 폐기|동작 흉내만|
|PoC|개발자|기술 검증|검증 후 폐기|동작만 하면 됨|
|MVP|실제 외부 사용자|가설 검증|살아남거나 버려짐|사용자가 망가뜨리지 않을 정도|

이 차이가 중요한 이유는 MVP에는 *최소한의 운영 부담*이 따라오기 때문이다. 사용자가 데이터를 입력하는 순간, 그 데이터를 잃어버리면 안 된다. 사용자가 로그인하는 순간, 비밀번호를 평문으로 저장하면 안 된다. PoC라면 괜찮지만 MVP라면 사고다.

## 3. MVP 범위를 산정하는 법

범위 산정은 MVP에서 가장 어렵다. 빼야 할 것을 빼지 못해서 6개월짜리가 되거나, 빼면 안 될 것을 빼서 사용자가 떠난다.

### 가설을 먼저 적는다

문서든 노트든, 한 문장으로 적는다.

> "30~40대 직장인은 점심시간에 식당 대기열 정보를 보면 3분 안에 식당을 선택할 것이다."

이 가설을 검증하는 데 필요한 최소 기능을 역산한다.

- 대기열 정보를 어디서 어떻게 받을 것인가
- 사용자는 어떻게 식당을 본다
- "선택했다"를 어떻게 측정한다

여기서 회원가입, 결제, 리뷰, 친구 초대, 알림 같은 것들은 *지금 검증할 가설과 무관*하다. 다 뺀다.

### 검증에 직접 기여하지 않는 기능은 다 뺀다

다음은 자주 들어가는데 거의 매번 뺄 수 있는 것들이다.

- 회원가입/로그인 (검증 단계에서는 익명 또는 매직링크로 충분한 경우가 많다)
- 비밀번호 재설정
- 프로필 편집
- 알림 설정 페이지
- 다국어 지원
- 다크모드
- 관리자 페이지 (DB 직접 조회로 대체)
- 통계 대시보드 (외부 도구로 대체)
- 모바일 앱 (반응형 웹으로 시작)

이걸 다 빼면 "그게 무슨 서비스냐"는 말이 나온다. 맞다. 무슨 서비스인지 *가설을 검증하기 전까지는 모른다*. 정답이 정해진 다음에 살을 붙인다.

### "있으면 좋겠다"는 무조건 뺀다

MVP 회의에서 "이건 있어야 하지 않을까?"라는 말이 나오면 일단 뺀다. "이게 없으면 가설을 검증할 수 없다"가 입증되어야만 넣는다.

기준이 모호해 보이지만 실제로는 단순하다. 그 기능이 없어도 사용자가 핵심 가설에 답하는 행동을 할 수 있느냐만 본다. 답할 수 있으면 뺀다.

## 4. 흔히 저지르는 실수

### 4.1 과도한 기능 추가

가장 흔한 실수다. "이왕 만드는 김에 X도 같이"가 누적되면 MVP가 1.0이 된다. 출시는 늦어지고, 만든 기능 중 절반은 아무도 안 쓴다.

이걸 막는 가장 단순한 방법은 *기능 추가 1건당 출시일 1주일 연기*를 명시적으로 계산해서 보여주는 것이다. 추상적인 "느려질 거예요"는 통하지 않는다.

### 4.2 기술 부채 무시

반대 극단도 있다. "어차피 MVP니까 대충 짜고 나중에 다시 짜자"는 태도. 이게 위험한 이유는 *MVP가 성공하면 다시 짤 시간이 없다*는 데 있다.

성공한 MVP는 사용자가 들어오기 시작한다. 사용자가 들어오면 신규 기능 요구가 쏟아진다. 이 시점에 "코드가 너무 더러워서 새로 짜야 한다"고 말할 수 있는 팀은 거의 없다. 결국 더러운 코드 위에 더러운 코드를 쌓는다. 6개월 뒤에는 누구도 손을 못 댄다.

그래서 MVP의 코드 품질 기준은 "최소한 사람이 읽을 수 있어야 한다"이다. 자세히는 6장에서 다룬다.

### 4.3 측정 지표가 없음

가설 없이 만든 MVP보다 더 흔한 게 *지표 없이 만든 MVP*다. 일주일 뒤에 "잘 되고 있나요?"라고 물으면 답할 수 없다. "사용자가 몇 명이에요"는 답이 아니다.

가설마다 측정 가능한 숫자가 있어야 한다.

- 가설: "사용자는 식당을 3분 안에 선택한다"
- 지표: 첫 진입부터 식당 상세 페이지 클릭까지의 시간 중앙값

이 지표를 *MVP 코드 안에 넣고 출시*해야 한다. 나중에 붙이려고 하면 데이터가 없다.

### 4.4 사용자가 사용하지 않는 MVP

가장 슬픈 실수다. 6주 만들었는데 아무도 안 쓴다. 이걸로는 가설을 검증할 수 없다. "사용자가 없어서 가설을 검증 못 했다"는 것 자체가 결과이긴 하지만, 보통은 *유통 채널을 고민하지 않은* 탓이다.

MVP를 시작할 때 "어떻게 첫 100명을 데려올 것인가"를 코드보다 먼저 정해야 한다. 친구·지인부터, 커뮤니티 직접 글, 오프라인 모임, 작은 인플루언서 등. 이 계획이 없으면 MVP는 출시되자마자 사막에 떨어진다.

## 5. 빠른 검증을 위한 아키텍처 선택

MVP의 아키텍처 결정 원칙은 단순하다. *직접 만드는 것을 최소화한다*. 직접 만든 만큼 검증이 늦어진다.

### 5.1 모놀리스 우선

MSA부터 시작하지 않는다. MSA는 *조직과 트래픽이 커진 다음의 답*이다. MVP 단계에서 MSA를 택하면 다음과 같은 문제가 생긴다.

- 서비스 간 통신 코드를 짜야 한다
- 로컬 개발 환경이 무거워진다
- 배포 파이프라인이 N개가 된다
- 데이터 일관성이 문제가 된다
- 디버깅이 어려워진다

검증되지 않은 서비스에 이런 비용을 쓸 이유가 없다. 단일 프로세스, 단일 DB, 단일 레포로 시작한다. 잘 쪼개고 싶다면 *내부 모듈 경계*만 잘 그어두면 된다. 나중에 트래픽이 늘면 그때 분리한다.

```
src/
├── auth/        # 모듈 경계는 분명히
├── restaurant/
├── order/
└── shared/
```

같은 프로세스 안이지만 서로 직접 import하지 않고 인터페이스로만 호출하면, 나중에 분리할 때 비용이 작다. 처음부터 분리하지는 않는다.

### 5.2 매니지드 서비스 활용

직접 운영하지 않는다. MVP 단계의 시간은 운영보다 검증에 써야 한다.

|영역|직접 운영|매니지드 (권장)|
|---|---|---|
|DB|PostgreSQL on EC2|RDS, Supabase, Neon|
|캐시|Redis on EC2|ElastiCache, Upstash|
|메시지 큐|RabbitMQ on EC2|SQS, Cloudflare Queues|
|파일 저장|NFS, MinIO|S3, R2|
|로그|ELK 직접 구축|Datadog, Logtail, CloudWatch|
|배포|직접 K8s|Fly.io, Render, Vercel, Railway|

비용이 비싸 보이지만, 엔지니어 한 명의 한 달 인건비가 매니지드 서비스 1년치보다 비싸다. 검증 단계에서 직접 운영은 손해다.

### 5.3 인증/결제는 SaaS로

인증을 직접 만들지 않는다. 보안 사고가 났을 때의 손해가 절약한 시간보다 훨씬 크다.

- 인증: Auth0, Clerk, Supabase Auth, Firebase Auth, AWS Cognito
- 결제: Stripe, Toss Payments, PortOne

이걸 직접 구현하면 비밀번호 해싱, 세션 관리, OAuth 콜백, 리프레시 토큰, 이메일 검증, 비밀번호 재설정, 2FA를 다 짜야 한다. 한 달이 사라진다. 그 한 달은 가설 검증에 써야 한다.

결제는 더 심각하다. PCI DSS, 환불 처리, 영수증 발행, 부분 환불, 결제 실패 재시도 같은 걸 직접 구현하면 안 된다.

### 5.4 단순한 백엔드 구성 예시

```
[Client (반응형 웹)]
        ↓
[Cloudflare / CDN]
        ↓
[Fly.io 또는 Render에 배포된 모놀리스 서버]
        ↓                  ↓                ↓
   [Managed DB]      [Auth SaaS]     [Payment SaaS]
   (RDS/Supabase)    (Clerk 등)      (Stripe 등)
        ↓
   [S3 (파일)]
   [Datadog (로그/지표)]
```

여기서 직접 짜야 하는 코드는 *비즈니스 로직 한 덩어리*뿐이다. 검증에 필요한 시간이 가장 짧다.

## 6. MVP 단계의 코드 품질 기준

"MVP니까 막 짜도 된다"는 흔한 오해다. 다시 말하지만, 성공한 MVP는 다시 짤 시간을 주지 않는다. 그렇다고 클린 아키텍처를 다 적용하면 만들지 못한다. 그 중간 기준이 필요하다.

### 적용한다

- **변수·함수 이름은 의미 있게.** `data`, `result`, `tmp` 같은 이름 금지. 6개월 뒤의 나는 남이다.
- **함수는 한 가지 일.** 100줄 함수는 디버깅 불가다.
- **모듈 경계.** 도메인 단위로 폴더를 나눈다. import 화살표가 한 방향으로 흐르게.
- **에러 처리.** try/catch를 빠뜨려서 500이 나면 사용자가 떠난다.
- **DB 트랜잭션.** 돈이나 핵심 데이터를 다루는 곳은 트랜잭션을 꼭 건다.
- **타입.** TypeScript, Python type hints, Java/Kotlin 등 정적 타입을 쓴다.

### 적용하지 않는다

- **과한 추상화.** 인터페이스 3개를 거쳐서 구현체에 도달하는 구조 금지. 인터페이스는 *두 번째 구현체가 생긴 다음에* 만든다.
- **레이어 분리.** Repository, Service, Controller를 무리하게 다 만들지 않는다. 단일 클래스 안에서 처리해도 된다.
- **DDD 전술.** Aggregate, Value Object, Domain Event는 도메인이 안정된 뒤의 이야기다.
- **이벤트 소싱.** MVP에서 쓰면 거의 항상 후회한다.
- **마이크로프론트엔드, BFF.** 검증 끝난 뒤에 고민한다.
- **테스트 100% 커버리지.** 핵심 비즈니스 로직과 결제 흐름만 테스트한다. CRUD까지 다 쓰면 시간이 모자란다.

### 코드 예시: MVP 수준의 적당한 구조

지나치게 분리한 예시.

```typescript
// 과한 분리 - MVP 단계에 불필요
interface IRestaurantRepository {
  findById(id: string): Promise<Restaurant>;
}
class RestaurantRepository implements IRestaurantRepository { ... }

interface IRestaurantService {
  getRestaurant(id: string): Promise<RestaurantDto>;
}
class RestaurantService implements IRestaurantService {
  constructor(private repo: IRestaurantRepository) {}
  ...
}

class RestaurantMapper { ... }
class RestaurantController { ... }
```

MVP 단계에 적당한 예시.

```typescript
// restaurant/restaurant.ts
export async function getRestaurant(id: string) {
  const row = await db
    .selectFrom("restaurant")
    .selectAll()
    .where("id", "=", id)
    .executeTakeFirst();

  if (!row) {
    throw new NotFoundError(`restaurant ${id} not found`);
  }
  return row;
}

// app.ts
app.get("/restaurants/:id", async (req, res) => {
  const restaurant = await getRestaurant(req.params.id);
  res.json(restaurant);
});
```

구조는 단순하지만, 함수 이름·에러 처리·타입은 살아 있다. 나중에 사용처가 늘어나면 그때 클래스로 끌어올린다. *그 시점이 오지 않으면 끌어올릴 필요도 없다*.

## 7. 백엔드 관점에서 빼도 되는 것과 절대 빼면 안 되는 것

여기가 MVP에서 가장 자주 망하는 영역이다. 빠르게 만든다는 핑계로 보안과 운영을 빼면 사용자가 떠난다.

### 빼도 되는 것

- **세련된 UI** — 디자이너 없이 Tailwind 기본 스타일로 충분하다
- **다양한 결제 수단** — 신용카드 하나만
- **다국어** — 한국어만
- **소셜 로그인 5종** — 구글 하나만
- **추천 알고리즘** — 인기순 정렬로 시작
- **검색 엔진** — DB LIKE 쿼리로 시작
- **알림** — 이메일 한 종류
- **관리자 페이지** — DB 도구로 직접 조회
- **A/B 테스트 프레임워크** — 일단 단일 버전만
- **CI/CD 정교화** — `main` 푸시 시 배포 한 줄
- **자동 스케일링** — 단일 인스턴스로 시작

### 절대 빼면 안 되는 것

#### 7.1 인증과 인가

비밀번호 평문 저장, JWT 검증 누락, 사용자 ID를 클라이언트가 보내게 두기 같은 실수가 MVP 코드에서 자주 발견된다. 사용자 한 명이 다른 사람의 데이터를 보면 *그 순간 서비스는 끝난다*. SNS에 올라가고 신뢰가 사라진다.

직접 짜지 말고 매니지드 인증을 쓰면 거의 막아진다. 자기 데이터만 접근하게 하는 인가 검사만 꼼꼼히 한다.

```typescript
app.get("/orders/:id", requireAuth, async (req, res) => {
  const order = await db
    .selectFrom("order")
    .selectAll()
    .where("id", "=", req.params.id)
    .where("user_id", "=", req.user.id)
    .executeTakeFirst();

  if (!order) {
    return res.status(404).json({ error: "not found" });
  }
  res.json(order);
});
```

`where("user_id", "=", req.user.id)` 한 줄을 빠뜨리면 IDOR이 된다. MVP라도 이건 빼면 안 된다.

#### 7.2 로깅

로깅 없이 MVP를 출시하면 *사용자가 무슨 일을 겪었는지 알 수 없다*. "안 돼요"라는 메시지가 와도 어디서 막혔는지 모른다.

최소한 다음은 남긴다.

- 모든 HTTP 요청의 method, path, status, latency, user_id
- 모든 서버 에러 (stack trace 포함)
- 결제 흐름의 모든 단계

도구는 직접 구축하지 않는다. Datadog, Logtail, Better Stack 같은 SaaS를 붙인다.

```typescript
// 최소 로깅 미들웨어 예시
app.use((req, res, next) => {
  const start = Date.now();
  res.on("finish", () => {
    logger.info({
      method: req.method,
      path: req.path,
      status: res.statusCode,
      latency_ms: Date.now() - start,
      user_id: req.user?.id ?? null,
    });
  });
  next();
});
```

#### 7.3 백업

DB 백업이 없으면 *하루 만에 망할 수 있다*. 매니지드 DB(RDS, Supabase, Neon)는 자동 백업이 기본이지만, 켜져 있는지 확인한다. 켜져 있다고 끝이 아니라, *복원 한 번을 직접 해봐야 한다*. 한 번도 복원해보지 않은 백업은 백업이 아니다.

#### 7.4 결제 멱등성

결제 API를 호출하는데 멱등키(idempotency key)를 안 붙이면 중복 결제가 일어난다. Stripe, Toss 모두 멱등키를 지원한다. MVP라도 이건 무조건 붙인다.

```typescript
const idempotencyKey = `order-${order.id}-attempt-${attempt}`;
await stripe.paymentIntents.create(
  { amount, currency: "krw" },
  { idempotencyKey }
);
```

#### 7.5 비밀 정보 관리

`.env` 파일을 깃에 올리면 안 된다. 한 번 올린 비밀은 영원히 외부에 있는 것으로 간주한다. 처음부터 `.gitignore`에 넣고, Fly.io secrets / Vercel env / AWS Secrets Manager 같은 곳에 저장한다.

## 8. 검증 지표 설계

지표 없는 MVP는 항해 일지 없는 배다. 어디로 가는지 모른다.

### 지표는 가설마다 하나씩

MVP에서 추적하는 지표는 *많지 않다*. 가설마다 1~2개면 충분하다. 너무 많으면 어느 것도 안 본다.

|가설|지표|성공 기준|
|---|---|---|
|사용자가 식당을 3분 안에 선택한다|첫 진입~상세 클릭 중앙값|3분 이하|
|사용자가 결제까지 간다|식당 상세~결제 완료 전환율|10% 이상|
|사용자가 다시 온다|7일 재방문율|30% 이상|

성공 기준을 *출시 전에* 정한다. 출시한 후에 정하면 나오는 숫자에 맞춰서 기준을 조정하게 된다.

### 측정 도구

직접 만들지 않는다. Mixpanel, Amplitude, PostHog 같은 도구를 쓴다. PostHog는 자체 호스팅도 되고 무료 플랜이 있어서 MVP에 적합하다.

이벤트는 *사용자 행동 단위*로 보낸다.

```typescript
posthog.capture(userId, "restaurant_detail_viewed", {
  restaurant_id: restaurant.id,
  source: "list",
});

posthog.capture(userId, "order_completed", {
  order_id: order.id,
  amount: order.amount,
});
```

이름 규칙을 처음부터 정해두면 좋다. `domain_object_action_pasttense` 같은 일관된 패턴.

### 정성 지표도 본다

숫자만 보면 *왜 그런지*를 모른다. 사용자 5~10명에게 직접 인터뷰하거나 Hotjar 같은 도구로 세션 녹화를 본다. MVP 단계에서는 정량 지표보다 정성 관찰이 훨씬 정보량이 많다.

## 9. MVP에서 본 제품으로 전환하는 시점

전환 시점은 명확한 신호가 있다.

### 신호 1: 가설이 검증됐다

지표가 성공 기준을 충족했다. "사람들이 정말 우리 서비스를 쓴다"가 데이터로 확인됐다. 이제 만든 게 헛수고가 아니라는 게 분명하다.

### 신호 2: 운영 부담이 커진다

DB 쿼리가 느려진다. 로그가 쌓여서 검색이 안 된다. 배포 한 번에 30분이 걸린다. 장애가 잦아진다. MVP 코드가 한계에 도달했다는 신호다.

### 신호 3: 신규 기능 추가가 점점 어려워진다

"이거 하나 붙이는 데 일주일 걸려요"라는 말이 나오기 시작한다. 코드 구조의 빚이 이자를 받아가는 시점이다.

이 세 신호가 동시에 오면 본 제품으로 전환할 때다. *세 신호 중 두 개라도 분명하면 시작해야 한다*. 너무 늦으면 신규 기능을 못 내고, 너무 빠르면 검증되지 않은 가설 위에 큰 집을 짓는다.

### 전환 작업의 우선순위

전환을 다시 짠다고 *전부 갈아엎지는 않는다*. 가장 아픈 곳부터 고친다.

1. 운영 자동화: 배포, 모니터링, 알림
2. 성능 병목: 느린 쿼리, N+1, 캐싱
3. 코드 구조: 가장 자주 손대는 모듈부터 리팩터링
4. 그 외 영역은 *문제가 생길 때까지 손대지 않는다*

스타트업이 가장 자주 망가지는 패턴 중 하나가 "이제 본격적으로 시작하니까 처음부터 다시 짜자"다. 검증된 코드를 버리고 처음부터 다시 짜는 동안 사용자가 다 떠난다.

## 10. 버려질 수 있다는 전제로 코드 짜기

MVP의 진짜 어려움은 여기에 있다. 코드를 *살아남을 수도 있고 버려질 수도 있다*는 전제로 짜야 한다는 점이다. 두 극단 모두 비용을 발생시킨다.

### 살아남을 가능성 대비: 사람이 읽을 수 있게

위에서 말한 코드 품질 기준이 여기 해당한다. 6개월 뒤에 내가 다시 봐도 이해되는 정도. 다른 개발자가 합류했을 때 한 시간 안에 흐름을 파악할 정도.

### 버려질 가능성 대비: 결합도를 낮춰서

특정 라이브러리·프레임워크·DB에 깊게 묶이지 않게 한다. 도메인 로직과 인프라 코드를 섞지 않는다. 도메인 로직이 살아남으면 인프라만 바꿔 끼울 수 있다.

이게 클린 아키텍처의 핵심 아이디어와 비슷하지만, MVP 단계에서는 *형식*까지 따라가지 않는다. *원칙*만 가져온다. Repository 인터페이스를 일일이 만들지 않더라도, 비즈니스 로직 함수가 DB 라이브러리를 직접 모르게는 만든다.

```typescript
// 도메인 로직 - DB 라이브러리를 모름
export function calculateOrderTotal(items: OrderItem[]): number {
  return items.reduce((sum, item) => sum + item.price * item.quantity, 0);
}

// 인프라 - 도메인을 알지만 분리되어 있음
export async function createOrder(userId: string, items: OrderItem[]) {
  const total = calculateOrderTotal(items);
  return db
    .insertInto("order")
    .values({ user_id: userId, total, status: "pending" })
    .returningAll()
    .executeTakeFirstOrThrow();
}
```

`calculateOrderTotal`은 순수 함수다. 어떤 인프라로 이주해도 그대로 살아남는다. MVP에서 이 정도의 분리만 지켜도 본 제품 전환 시 도메인 로직 대부분을 가져갈 수 있다.

### 데이터는 코드보다 중요하다

코드는 버릴 수 있어도 *데이터는 못 버린다*. 사용자가 만든 데이터가 사라지면 사용자도 사라진다.

그래서 MVP 단계에서도 DB 스키마는 신중하게 짠다. 컬럼 이름, 타입, NULL 허용 여부 같은 결정은 *나중에 바꾸기 어렵다*. 마이그레이션 도구(Prisma, Drizzle, Flyway, Liquibase 등)를 처음부터 쓴다. `ALTER TABLE`을 손으로 치지 않는다.

이건 비용이 거의 들지 않는데 효과가 크다. MVP에서 거의 유일하게 "처음부터 잘 해야 하는" 영역이다.

## 11. 정리

MVP는 *적게 만드는 기술*이 아니라 *빨리 배우는 기술*이다. 가설을 적고, 가설을 검증할 최소 범위를 정하고, 매니지드 서비스로 직접 만드는 영역을 줄이고, 인증·로깅·백업·결제 멱등성처럼 빼면 안 되는 영역은 지킨다. 짠 코드는 살아남을 수도 있고 버려질 수도 있다는 전제로, 사람이 읽을 수 있는 수준에서 멈춘다. 지표로 가설을 검증하고, 검증되면 본 제품으로 옮긴다.

MVP가 끝났을 때 결과는 둘 중 하나다. 가설이 맞아서 본 제품으로 옮겨가거나, 가설이 틀려서 코드를 버리고 다음 가설을 시작한다. *두 경우 다 성공*이다. 가설 검증 없이 큰 제품을 만든 다음에 망하는 것보다 백 배 낫다.
