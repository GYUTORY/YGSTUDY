# MSA 데이터 관리 패턴

## 개요

MSA에서 가장 먼저 부딪히는 문제는 데이터다. 모놀리스에서는 하나의 DB에 조인 한 번이면 끝나던 조회가, 서비스가 분리되는 순간부터 전부 깨진다. 서비스마다 DB를 따로 가져야 한다는 건 알겠는데, 그렇게 하면 기존에 잘 되던 것들이 안 된다.

이 문서는 MSA에서 데이터를 어떻게 나누고, 나눈 데이터를 어떻게 조회하고 동기화하는지를 다룬다. 이벤트 소싱이나 CQRS의 내부 동작 원리는 별도 문서에서 다루고, 여기서는 데이터 아키텍처 관점에서 각 패턴의 사용 시점과 실무에서 겪는 문제에 집중한다.

## Database per Service 패턴

### 기본 원칙

서비스마다 자체 데이터베이스를 가진다. 다른 서비스의 DB에 직접 접근하지 않는다.

**주문 서비스와 상품 서비스 분리 예시:**

```
주문 서비스 → order_db (orders, order_items 테이블)
상품 서비스 → product_db (products, categories 테이블)
```

주문 서비스가 상품 정보를 알아야 하면 상품 서비스 API를 호출한다. `product_db`에 직접 쿼리하지 않는다.

### 왜 DB를 분리해야 하는가

DB를 분리하지 않으면 서비스를 분리하는 의미가 없다.

**서비스 독립 배포가 불가능해진다:**
- 상품 서비스에서 `products` 테이블 스키마를 바꾸면 주문 서비스도 같이 수정해야 한다
- 스키마 변경 때마다 관련된 모든 서비스를 동시에 배포해야 한다
- 배포 일정이 서비스 간에 결합된다

**장애가 전파된다:**
- 한 서비스의 쿼리가 DB에 부하를 주면 다른 서비스도 영향을 받는다
- 특정 서비스의 잘못된 쿼리가 공유 DB에 락을 걸면 전체가 멈춘다

**스케일링이 제한된다:**
- 주문 서비스만 트래픽이 많아서 DB를 스케일업하고 싶어도, 공유 DB를 쓰면 전체를 스케일업해야 한다
- 서비스마다 적합한 DB를 선택할 수 없다. 주문은 RDB, 검색은 Elasticsearch가 맞을 수 있는데, 공유 DB면 선택지가 없다

### DB 분리의 범위

DB를 분리하는 방식은 여러 가지다. 상황에 따라 적절한 수준을 선택한다.

**1. 완전 분리 (별도 DB 인스턴스):**

```
주문 서비스 → PostgreSQL (order-db-instance)
상품 서비스 → PostgreSQL (product-db-instance)
검색 서비스 → Elasticsearch (search-cluster)
```

- 가장 확실한 분리다
- 운영 비용이 높다. DB 인스턴스마다 모니터링, 백업, 보안 설정이 필요하다
- 서비스 수가 적을 때 적합하다

**2. 스키마 분리 (같은 인스턴스, 다른 스키마):**

```sql
-- 같은 PostgreSQL 인스턴스
CREATE SCHEMA order_service;
CREATE SCHEMA product_service;

-- 주문 서비스는 order_service 스키마만 사용
CREATE TABLE order_service.orders (...);

-- 상품 서비스는 product_service 스키마만 사용
CREATE TABLE product_service.products (...);
```

- 운영 비용이 낮다
- DB 수준의 접근 제어로 스키마 간 접근을 차단할 수 있다
- 물리적 리소스는 공유하므로 부하 격리가 안 된다
- 서비스 수가 많거나 비용을 아껴야 할 때 현실적인 선택이다

**3. 테이블 분리 (같은 스키마, 네이밍 규칙으로 구분):**

```sql
-- 같은 스키마 안에서 접두사로 구분
order_orders, order_order_items
product_products, product_categories
```

- 분리 수준이 가장 낮다
- 결국 다른 서비스 테이블에 조인을 걸 유혹이 생긴다
- 마이그레이션 중간 단계로만 사용하고, 최종 목표는 아니다

### Polyglot Persistence

서비스마다 데이터 특성에 맞는 DB를 선택할 수 있다.

```
주문 서비스   → PostgreSQL (트랜잭션 필요)
상품 카탈로그 → MongoDB (스키마가 자주 바뀜)
검색 서비스   → Elasticsearch (전문 검색)
세션 관리     → Redis (빠른 읽기/쓰기)
추천 서비스   → Neo4j (관계 그래프)
```

현실적으로 전부 다른 DB를 쓰는 경우는 드물다. 대부분 주력 DB 하나(주로 PostgreSQL이나 MySQL)를 쓰고, 특수한 요구사항이 있는 서비스만 다른 DB를 도입한다. DB 종류가 늘어날수록 운영 부담이 커지기 때문이다.

## 공유 DB 안티패턴

### 공유 DB가 문제인 이유

여러 서비스가 하나의 DB를 공유하면서 같은 테이블을 직접 읽고 쓰는 구조다.

```
주문 서비스 ──┐
              ├──→ 공유 DB (orders, products, users, payments)
상품 서비스 ──┤
              │
결제 서비스 ──┘
```

**실제로 겪는 문제들:**

**1. 스키마 변경 공포:**

상품 서비스에서 `products` 테이블에 컬럼을 추가한다고 하자.

```sql
ALTER TABLE products ADD COLUMN weight DECIMAL(10,2);
```

이 변경이 다른 서비스에 영향을 주는지 확인하려면 모든 서비스 코드를 뒤져야 한다. `SELECT *`를 쓰는 서비스가 있으면 매핑이 깨질 수 있다. 결국 "이 테이블 건드리지 마세요"라는 암묵적 규칙이 생긴다.

**2. 숨은 결합:**

주문 서비스에서 상품 가격을 조회할 때 `products` 테이블을 직접 조인한다.

```sql
SELECT o.id, o.quantity, p.price
FROM orders o
JOIN products p ON o.product_id = p.id
WHERE o.user_id = ?;
```

상품 서비스에서 가격 계산 로직이 바뀌면(할인율 적용, 통화 변환 등) 주문 서비스의 이 쿼리는 여전히 원래 가격을 가져온다. 비즈니스 로직이 두 곳에서 따로 돌아가는 셈이다.

**3. 데드락과 성능 문제:**

서비스 A가 `orders` 테이블에 락을 걸고, 서비스 B가 같은 테이블에 접근하면 데드락이 발생할 수 있다. 문제는 서로 다른 서비스이기 때문에 어떤 서비스가 락을 잡고 있는지 파악하기 어렵다는 것이다.

### 공유 DB가 허용되는 경우

모든 상황에서 DB를 분리해야 하는 건 아니다.

**읽기 전용 공유:**
- 레거시 시스템에서 새 서비스가 기존 데이터를 읽기만 할 때
- CDC(Change Data Capture)를 도입하기 전 과도기적으로 사용한다
- 읽기 전용 레플리카를 제공하는 것이 낫다

**동일 팀이 관리하는 서비스:**
- 같은 팀이 두 서비스를 모두 관리하고, 스키마 변경을 함께 제어할 수 있을 때
- 이 경우에도 서비스가 성장하면 결국 분리해야 한다

**분리 비용이 너무 클 때:**
- 마이그레이션에 필요한 리소스가 없을 때
- 당장 문제가 없는데 원칙만으로 분리하는 건 낭비다
- 단, 기술 부채로 인식하고 로드맵에는 넣어 두어야 한다

## 서비스 간 데이터 조회 - API Composition

### 문제 상황

모놀리스에서 한 번의 조인으로 가져오던 데이터가 서비스 분리 후에는 여러 서비스에 흩어져 있다.

**주문 상세 조회 예시:**

모놀리스에서는 이렇게 한 번에 가져왔다.

```sql
SELECT o.id, o.status, o.created_at,
       u.name AS user_name, u.email,
       p.name AS product_name, p.price,
       pay.status AS payment_status
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id
JOIN payments pay ON pay.order_id = o.id
WHERE o.id = ?;
```

서비스 분리 후에는 주문, 사용자, 상품, 결제가 전부 다른 서비스에 있다.

### API Composition 패턴

API Composer가 여러 서비스를 호출해서 결과를 조합한다.

```java
@Service
public class OrderDetailComposer {

    private final OrderServiceClient orderClient;
    private final UserServiceClient userClient;
    private final ProductServiceClient productClient;
    private final PaymentServiceClient paymentClient;

    public OrderDetailResponse getOrderDetail(Long orderId) {
        // 1. 주문 정보 조회
        OrderDto order = orderClient.getOrder(orderId);

        // 2. 나머지 서비스 병렬 호출
        CompletableFuture<UserDto> userFuture =
            CompletableFuture.supplyAsync(() -> userClient.getUser(order.getUserId()));
        CompletableFuture<ProductDto> productFuture =
            CompletableFuture.supplyAsync(() -> productClient.getProduct(order.getProductId()));
        CompletableFuture<PaymentDto> paymentFuture =
            CompletableFuture.supplyAsync(() -> paymentClient.getPaymentByOrder(orderId));

        CompletableFuture.allOf(userFuture, productFuture, paymentFuture).join();

        // 3. 결과 조합
        return OrderDetailResponse.builder()
            .orderId(order.getId())
            .status(order.getStatus())
            .userName(userFuture.join().getName())
            .productName(productFuture.join().getName())
            .price(productFuture.join().getPrice())
            .paymentStatus(paymentFuture.join().getStatus())
            .build();
    }
}
```

### API Composition의 한계

**응답 시간이 길어진다:**
- 순차 호출이면 각 서비스 응답 시간의 합이 된다
- 병렬 호출해도 가장 느린 서비스의 응답 시간이 전체 응답 시간이다
- 서비스 하나가 느려지면 전체 API가 느려진다

**부분 실패 처리:**

```java
public OrderDetailResponse getOrderDetail(Long orderId) {
    OrderDto order = orderClient.getOrder(orderId);
    
    UserDto user;
    try {
        user = userClient.getUser(order.getUserId());
    } catch (Exception e) {
        // 사용자 서비스 장애 시 어떻게 할 것인가?
        // 1. 전체 실패 처리 → 사용자 경험 나빠짐
        // 2. 부분 데이터 반환 → 화면에 "알 수 없음" 표시
        // 3. 캐시된 데이터 반환 → 오래된 데이터일 수 있음
        user = UserDto.unknown();
    }
    // ...
}
```

부분 실패 시 어떻게 처리할지는 비즈니스 요구사항에 따라 다르다. 정답은 없지만, 모든 경우를 미리 정의해 두어야 한다.

**페이징과 정렬:**

API Composition으로 페이징을 구현하면 문제가 생긴다.

```
"최근 주문 10건을 상품 가격 순으로 정렬해서 조회"

1. 주문 서비스에서 주문을 가져온다 → 몇 건을 가져와야 하는가?
2. 상품 서비스에서 가격을 가져온다
3. 메모리에서 정렬한다
4. 상위 10건을 반환한다
```

정렬 기준이 다른 서비스에 있으면 전체 데이터를 메모리로 가져와서 정렬해야 한다. 데이터가 많으면 성능 문제가 심각해진다.

### API Composition을 쓰기 적합한 경우

- 단건 조회 (주문 상세 페이지 등)
- 호출 대상 서비스가 2~3개 이내
- 정렬이나 필터링이 단일 서비스 데이터 기준
- 실시간 데이터가 중요한 경우 (최신 상태를 항상 보여줘야 할 때)

## 서비스 간 데이터 조회 - CQRS Read Model

### API Composition이 안 되는 경우

리스트 조회에 여러 서비스 데이터가 필요하고, 정렬/필터/페이징까지 해야 하면 API Composition으로는 감당이 안 된다.

```
"전체 주문 목록을 상품명으로 검색하고, 결제 금액 순으로 정렬해서, 페이지당 20건씩 조회"
```

이걸 API Composition으로 하면 전체 주문을 다 가져와서 메모리에서 처리해야 한다. 현실적이지 않다.

### CQRS Read Model 구축

읽기 전용 저장소를 별도로 만들고, 각 서비스의 이벤트를 받아서 비정규화된 뷰를 유지한다.

```
주문 서비스 ──(OrderCreated)──→ 이벤트 버스 ──→ 조회 서비스 ──→ read_db
상품 서비스 ──(ProductUpdated)─→ 이벤트 버스 ──→ 조회 서비스 ──→ read_db
결제 서비스 ──(PaymentCompleted)→ 이벤트 버스 ──→ 조회 서비스 ──→ read_db
```

**Read Model 테이블:**

```sql
CREATE TABLE order_summary (
    order_id      BIGINT PRIMARY KEY,
    user_id       BIGINT,
    user_name     VARCHAR(100),
    product_id    BIGINT,
    product_name  VARCHAR(200),
    product_price DECIMAL(10,2),
    quantity      INT,
    total_amount  DECIMAL(10,2),
    payment_status VARCHAR(20),
    order_status  VARCHAR(20),
    created_at    TIMESTAMP,
    updated_at    TIMESTAMP
);

-- 검색과 정렬을 위한 인덱스
CREATE INDEX idx_order_summary_product_name ON order_summary(product_name);
CREATE INDEX idx_order_summary_total_amount ON order_summary(total_amount);
CREATE INDEX idx_order_summary_created_at ON order_summary(created_at);
```

이 테이블에는 여러 서비스의 데이터가 한 곳에 들어가 있다. 정규화 원칙을 깨지만, 읽기 성능을 위한 의도적인 선택이다.

**이벤트 핸들러:**

```java
@Service
public class OrderSummaryProjection {

    private final OrderSummaryRepository repository;

    @EventHandler
    public void on(OrderCreatedEvent event) {
        OrderSummary summary = new OrderSummary();
        summary.setOrderId(event.getOrderId());
        summary.setUserId(event.getUserId());
        summary.setProductId(event.getProductId());
        summary.setQuantity(event.getQuantity());
        summary.setOrderStatus("CREATED");
        summary.setCreatedAt(event.getCreatedAt());
        repository.save(summary);
    }

    @EventHandler
    public void on(ProductInfoEnrichedEvent event) {
        // 상품 서비스에서 상품 정보가 변경되었을 때
        repository.findByProductId(event.getProductId())
            .forEach(summary -> {
                summary.setProductName(event.getProductName());
                summary.setProductPrice(event.getPrice());
                summary.setTotalAmount(
                    event.getPrice().multiply(BigDecimal.valueOf(summary.getQuantity()))
                );
                summary.setUpdatedAt(Instant.now());
                repository.save(summary);
            });
    }

    @EventHandler
    public void on(PaymentCompletedEvent event) {
        repository.findById(event.getOrderId())
            .ifPresent(summary -> {
                summary.setPaymentStatus("COMPLETED");
                summary.setUpdatedAt(Instant.now());
                repository.save(summary);
            });
    }
}
```

### Read Model의 일관성 문제

Read Model은 이벤트가 도착해야 갱신된다. 이벤트 처리에 지연이 생기면 조회 결과가 실제 상태와 다를 수 있다.

**Eventual Consistency:**
- 주문을 생성하고 바로 주문 목록을 조회하면 방금 생성한 주문이 안 보일 수 있다
- 상품 가격을 변경해도 주문 목록에 반영되기까지 시간이 걸린다

**실무에서의 대응:**

```java
// 주문 생성 직후 상세 조회 → Write DB에서 직접 읽는다
@GetMapping("/orders/{id}")
public OrderDetailResponse getOrder(@PathVariable Long id,
                                     @RequestParam(required = false) Boolean realtime) {
    if (Boolean.TRUE.equals(realtime)) {
        // Write 모델에서 직접 조회 + API Composition
        return orderDetailComposer.getOrderDetail(id);
    }
    // Read Model에서 조회
    return orderSummaryService.getOrderSummary(id);
}
```

- 단건 조회는 API Composition으로 실시간 데이터를 보여준다
- 리스트 조회는 Read Model을 쓰되, 약간의 지연을 허용한다
- 사용자에게 "처리 중"이라는 상태를 보여주는 것도 방법이다

### Read Model을 쓰기 적합한 경우

- 리스트 조회에 여러 서비스 데이터가 필요할 때
- 복잡한 검색, 정렬, 필터링이 필요할 때
- 조회 빈도가 높고 약간의 지연을 허용할 수 있을 때
- 데이터 양이 많아서 API Composition의 메모리 부담이 클 때

## 데이터 동기화 방법

서비스 간 데이터를 동기화하는 방식은 크게 세 가지다.

### 이벤트 기반 동기화

서비스가 데이터를 변경하면 이벤트를 발행하고, 관심 있는 서비스가 이벤트를 구독해서 자체 저장소를 갱신한다.

```java
// 상품 서비스 - 가격 변경 시 이벤트 발행
@Service
public class ProductService {

    private final ProductRepository repository;
    private final EventPublisher eventPublisher;

    @Transactional
    public void updatePrice(Long productId, BigDecimal newPrice) {
        Product product = repository.findById(productId)
            .orElseThrow();
        BigDecimal oldPrice = product.getPrice();
        product.setPrice(newPrice);
        repository.save(product);

        eventPublisher.publish(new ProductPriceChangedEvent(
            productId, oldPrice, newPrice, Instant.now()
        ));
    }
}

// 주문 서비스 - 이벤트 수신하여 로컬 캐시 갱신
@Component
public class ProductEventHandler {

    private final LocalProductCacheRepository cacheRepository;

    @KafkaListener(topics = "product-events")
    public void handleProductPriceChanged(ProductPriceChangedEvent event) {
        cacheRepository.updatePrice(event.getProductId(), event.getNewPrice());
    }
}
```

**주의할 점:**
- 이벤트 순서가 보장되어야 한다. 같은 상품의 가격이 1000 → 2000 → 1500으로 바뀌었는데, 이벤트가 1000 → 1500 → 2000 순서로 도착하면 최종 상태가 틀어진다
- Kafka를 쓸 때 같은 `productId`를 파티션 키로 사용하면 동일 상품에 대한 이벤트 순서를 보장할 수 있다

### CDC (Change Data Capture)

DB의 변경 로그(binlog, WAL)를 읽어서 다른 서비스로 전파한다.

```
상품 서비스 DB ──(binlog)──→ Debezium ──→ Kafka ──→ 주문 서비스
```

**Debezium 설정 예시:**

```json
{
  "name": "product-connector",
  "config": {
    "connector.class": "io.debezium.connector.mysql.MySqlConnector",
    "database.hostname": "product-db",
    "database.port": "3306",
    "database.user": "debezium",
    "database.password": "****",
    "database.server.id": "1",
    "database.include.list": "product_service",
    "table.include.list": "product_service.products",
    "topic.prefix": "product-cdc"
  }
}
```

**CDC의 장점:**
- 애플리케이션 코드를 수정하지 않아도 된다. DB 변경이 자동으로 이벤트가 된다
- 이벤트 발행을 빼먹을 일이 없다
- 레거시 시스템에서 이벤트를 뽑아낼 때 유용하다

**CDC의 단점:**
- DB 스키마 수준의 변경 이벤트만 나온다. 비즈니스 의미를 담기 어렵다
- `price` 컬럼이 바뀌었다는 건 알지만, "할인 적용"인지 "정가 변경"인지는 알 수 없다
- Debezium 같은 커넥터를 운영해야 한다. 커넥터가 죽으면 동기화가 멈춘다

### 이벤트 기반 vs CDC 선택 기준

**이벤트 기반이 맞는 경우:**
- 비즈니스 의미가 중요한 경우 (가격 변경 이유를 알아야 할 때)
- 신규 개발이라 코드에서 이벤트 발행을 넣을 수 있을 때
- 도메인 이벤트를 중심으로 시스템을 설계하고 싶을 때

**CDC가 맞는 경우:**
- 레거시 시스템에서 데이터 변경을 감지해야 할 때
- 애플리케이션 코드를 수정할 수 없을 때
- 빠르게 데이터 동기화를 구축해야 할 때

### 로컬 데이터 복제

자주 참조하는 다른 서비스의 데이터를 로컬에 복제해 둔다.

```java
// 주문 서비스 내부에 상품 정보 로컬 테이블
@Entity
@Table(name = "local_product_cache")
public class LocalProductCache {
    @Id
    private Long productId;
    private String productName;
    private BigDecimal price;
    private Instant lastUpdated;
}
```

주문을 생성할 때 상품 서비스 API를 호출하지 않고 로컬 캐시에서 상품 정보를 가져온다. 상품 정보가 변경되면 이벤트로 캐시를 갱신한다.

**주의할 점:**
- 캐시와 원본 사이에 불일치가 생길 수 있다
- 캐시 갱신이 실패하면 오래된 데이터를 쓰게 된다
- 어떤 데이터를 복제할지 범위를 명확히 정해야 한다. 전부 복제하면 공유 DB와 다를 바 없다
- 복제 데이터의 주인(owner)은 원래 서비스다. 로컬에서 수정하면 안 된다

## 조인이 안 되는 상황에서의 대응법

### 문제 인식

서비스를 분리하면 DB 조인이 안 된다. 이건 MSA의 근본적인 트레이드오프다. "조인이 안 되니까 서비스를 합치자"가 아니라, 조인 없이 같은 결과를 내는 방법을 찾아야 한다.

### 대응 1: 데이터 비정규화

조인 대신 필요한 데이터를 미리 복사해 둔다.

```sql
-- 모놀리스: 주문 조회 시 상품명을 조인으로 가져옴
SELECT o.id, p.name FROM orders o JOIN products p ON o.product_id = p.id;

-- MSA: 주문 테이블에 상품명을 저장해 둠
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    product_id BIGINT,
    product_name VARCHAR(200),  -- 주문 시점의 상품명
    product_price DECIMAL(10,2), -- 주문 시점의 가격
    quantity INT,
    status VARCHAR(20)
);
```

주문 생성 시점에 상품명과 가격을 같이 저장한다. 나중에 상품명이 바뀌어도 주문 당시의 정보는 유지된다. 오히려 비즈니스적으로 맞는 경우가 많다. 주문 영수증에는 주문 당시의 가격이 나와야 한다.

**적합한 경우:**
- 특정 시점의 스냅샷이 필요한 데이터 (주문 시점 가격, 배송 시점 주소)
- 변경 빈도가 낮은 데이터 (상품 카테고리명, 사용자 이름)
- 조회 빈도가 높고 실시간 정확성이 중요하지 않은 데이터

### 대응 2: 식별자만 저장하고 필요할 때 조회

```java
// 주문 테이블에는 product_id만 저장
// 화면에서 상품명이 필요하면 그때 상품 서비스를 호출

@GetMapping("/orders/{id}")
public OrderResponse getOrder(@PathVariable Long id) {
    Order order = orderRepository.findById(id).orElseThrow();
    ProductDto product = productClient.getProduct(order.getProductId());
    
    return new OrderResponse(order, product.getName(), product.getPrice());
}
```

단건 조회에서는 이 방식이 단순하다. 리스트 조회에서는 N+1 문제가 생기므로 batch API를 만들어야 한다.

```java
// N+1 방지: 상품 ID 목록을 한 번에 조회
@GetMapping("/orders")
public List<OrderResponse> getOrders(Pageable pageable) {
    Page<Order> orders = orderRepository.findAll(pageable);
    
    Set<Long> productIds = orders.stream()
        .map(Order::getProductId)
        .collect(Collectors.toSet());
    
    // 한 번의 호출로 여러 상품 정보를 가져온다
    Map<Long, ProductDto> products = productClient.getProductsByIds(productIds);
    
    return orders.stream()
        .map(order -> new OrderResponse(
            order,
            products.get(order.getProductId()).getName(),
            products.get(order.getProductId()).getPrice()
        ))
        .collect(Collectors.toList());
}
```

### 대응 3: 전용 Read Model 구축

앞서 설명한 CQRS Read Model 방식이다. 조인이 필요한 조회가 복잡하고 빈번하다면 별도의 비정규화된 읽기 전용 저장소를 만드는 게 맞다.

### 어떤 방식을 선택할 것인가

| 상황 | 방식 |
|------|------|
| 시점 데이터가 필요 (주문 시점 가격) | 비정규화 |
| 단건 조회, 서비스 2~3개 | API Composition |
| 리스트 조회, 실시간 필요 | Batch API 호출 |
| 리스트 조회, 복잡한 검색/정렬 | CQRS Read Model |
| 조회 빈도가 매우 높음 | Read Model + 캐싱 |

정답은 하나가 아니다. 같은 서비스 안에서도 API에 따라 다른 방식을 쓸 수 있다.

## 레거시 공유 DB 분리 과정

### 현실적인 문제

운영 중인 모놀리스에서 "내일부터 서비스별로 DB를 나누겠다"는 불가능하다. 수십~수백 개의 테이블이 서로 조인으로 엮여 있고, 어떤 코드가 어떤 테이블을 읽는지도 정확히 파악하기 어렵다.

### 단계 1: 테이블 의존성 파악

먼저 어떤 서비스(또는 모듈)가 어떤 테이블을 쓰는지 매핑한다.

```sql
-- MySQL: 최근 쿼리 로그에서 테이블 접근 패턴 분석
SELECT 
    db, 
    LEFT(sql_text, 200) AS query_sample,
    COUNT(*) AS count
FROM performance_schema.events_statements_summary_by_digest
WHERE schema_name = 'monolith_db'
GROUP BY db, LEFT(sql_text, 200)
ORDER BY count DESC;
```

코드 분석도 병행한다. 어떤 모듈이 어떤 테이블에 접근하는지 정리한다.

```
주문 모듈: orders, order_items, products(읽기), users(읽기)
상품 모듈: products, categories
사용자 모듈: users, addresses
결제 모듈: payments, orders(읽기)
```

여기서 "읽기"로 표시된 부분이 문제다. 이 의존성을 끊어야 서비스를 분리할 수 있다.

### 단계 2: 논리적 경계 설정

테이블을 소유 서비스별로 그룹핑한다. 각 테이블의 주인을 정한다.

```
orders, order_items → 주문 서비스 소유
products, categories → 상품 서비스 소유
users, addresses → 사용자 서비스 소유
payments → 결제 서비스 소유
```

소유권 원칙: 해당 테이블에 쓰기(INSERT, UPDATE, DELETE)를 하는 서비스가 소유자다. 여러 서비스가 쓰기를 하면 테이블을 분리하거나 소유권을 재정의해야 한다.

### 단계 3: 직접 접근을 API로 전환

다른 서비스 소유 테이블에 직접 조인하는 코드를 API 호출로 바꾼다.

```java
// Before: 직접 조인
@Query("SELECT o FROM Order o JOIN User u ON o.userId = u.id WHERE u.email = :email")
List<Order> findByUserEmail(String email);

// After: API 호출로 변환
public List<Order> findByUserEmail(String email) {
    // 1. 사용자 서비스에서 userId 조회
    UserDto user = userServiceClient.getUserByEmail(email);
    
    // 2. 자체 테이블에서 주문 조회
    return orderRepository.findByUserId(user.getId());
}
```

이 작업이 가장 시간이 많이 걸린다. 조인이 수십 개인 쿼리도 있고, 서브쿼리 안에 다른 서비스 테이블이 들어가 있는 경우도 있다.

### 단계 4: 뷰(View)를 활용한 점진적 분리

한 번에 모든 코드를 바꿀 수 없으면 DB 뷰를 중간 단계로 활용한다.

```sql
-- 상품 테이블을 상품 서비스 DB로 옮긴 후
-- 기존 공유 DB에는 뷰를 남겨둔다
CREATE VIEW products AS
SELECT * FROM dblink('product_db_connection', 'SELECT * FROM products')
AS t(id BIGINT, name VARCHAR, price DECIMAL, ...);
```

기존 코드는 수정하지 않아도 뷰를 통해 데이터를 읽을 수 있다. 이 뷰를 거쳐가는 쿼리를 하나씩 찾아서 API 호출로 전환하고, 전부 전환되면 뷰를 제거한다.

**뷰 방식의 한계:**
- 쓰기는 안 된다. 읽기 전용이다
- dblink, FDW(Foreign Data Wrapper) 성능이 직접 조인보다 느리다
- 조인을 포함하는 뷰는 더 느려진다
- 트랜잭션 격리가 보장되지 않는다

### 단계 5: 물리적 분리

모든 직접 접근이 API로 전환되면 테이블을 물리적으로 분리한다.

```
1. 새 DB 인스턴스 생성
2. 테이블 마이그레이션 (데이터 복사)
3. 새 서비스가 새 DB를 바라보도록 설정 변경
4. 기존 DB에서 테이블 제거 (또는 뷰로 대체)
5. 정상 동작 확인 후 뷰 제거
```

### 분리 과정에서 흔히 겪는 문제

**1. 외래 키 제약 조건:**

```sql
-- 기존: 주문 테이블에 사용자 외래 키
ALTER TABLE orders ADD CONSTRAINT fk_user 
    FOREIGN KEY (user_id) REFERENCES users(id);
```

DB를 분리하면 외래 키가 동작하지 않는다. 데이터 무결성을 애플리케이션 레벨에서 관리해야 한다.

```java
// 주문 생성 전에 사용자 존재 여부를 API로 확인
public Order createOrder(CreateOrderRequest request) {
    // 사용자가 존재하는지 확인
    boolean userExists = userServiceClient.exists(request.getUserId());
    if (!userExists) {
        throw new IllegalArgumentException("존재하지 않는 사용자");
    }
    return orderRepository.save(request.toEntity());
}
```

이 방식은 TOCTOU(Time of Check to Time of Use) 문제가 있다. 확인 시점과 저장 시점 사이에 사용자가 삭제될 수 있다. 실무에서는 이 정도 불일치는 보상 처리로 대응한다.

**2. 분산 트랜잭션:**

모놀리스에서 하나의 트랜잭션으로 처리하던 것이 여러 서비스에 걸치게 된다. Saga 패턴으로 대체해야 한다. 이 부분은 별도 문서에서 상세히 다룬다.

**3. 리포팅 쿼리:**

관리자 대시보드나 BI 리포트에서 여러 테이블을 조인하는 복잡한 쿼리를 쓰고 있다면, 서비스 분리 후에 이 쿼리들이 전부 깨진다.

```
대응 방법:
- 리포팅 전용 데이터 웨어하우스를 구축한다
- 각 서비스의 데이터를 ETL로 데이터 웨어하우스에 적재한다
- 리포팅 쿼리는 데이터 웨어하우스에서 실행한다
```

리포팅 데이터는 실시간일 필요가 없는 경우가 대부분이므로, 야간 배치로 동기화해도 충분하다.

**4. 마이그레이션 중 이중 쓰기:**

분리 과정에서 기존 DB와 새 DB에 동시에 쓰는 기간이 생길 수 있다. 이중 쓰기는 데이터 불일치의 원인이 된다.

```
잘못된 방법:
1. 새 DB에 쓰기
2. 기존 DB에 쓰기 → 이 단계가 실패하면 불일치 발생

올바른 방법:
1. 기존 DB에만 쓰기 (변경 없음)
2. CDC로 기존 DB → 새 DB로 실시간 복제
3. 새 DB가 따라잡으면 쓰기를 새 DB로 전환
4. 기존 DB의 테이블 제거
```

CDC를 활용하면 이중 쓰기 없이 안전하게 마이그레이션할 수 있다.

### 분리 순서 결정

모든 테이블을 동시에 분리하지 않는다. 의존성이 적은 테이블부터 분리한다.

```
1차: 독립적인 테이블 (다른 서비스가 읽기만 하는 테이블)
     예: 상품 카탈로그, 코드 테이블
     
2차: 핵심 도메인 테이블 (외래 키로 많이 참조되는 테이블)
     예: 사용자, 주문
```
