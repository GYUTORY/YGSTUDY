# 서비스 메시와 사이드카 패턴

## 서비스 메시가 해결하는 문제

MSA에서 서비스 간 통신은 단순히 HTTP 호출 하나로 끝나지 않는다. 재시도, 타임아웃, 서킷 브레이커, 로드 밸런싱, 인증, 암호화, 트래픽 관측 같은 것들이 전부 필요하다. 이걸 각 서비스 코드에 넣으면 어떻게 되는지 보자.

```java
// 서비스 코드에 통신 관련 로직이 섞인 경우
@Service
public class OrderService {
    public PaymentResponse callPayment(OrderRequest req) {
        // 재시도 로직
        // 서킷 브레이커
        // 타임아웃
        // mTLS 인증서 로드
        // 메트릭 수집
        // 실제 비즈니스 로직
    }
}
```

서비스가 10개면 이 코드가 10곳에 들어간다. Java 서비스는 Resilience4j로 해결했는데, Go 서비스가 추가되면 Go 버전도 구현해야 한다. Python 서비스까지 오면 같은 로직을 세 번째 작성하게 된다. 라이브러리 버전이 다르면 재시도 동작이 서비스마다 달라지는 문제도 생긴다.

서비스 메시는 이 통신 관련 로직을 애플리케이션 코드에서 빼낸다. 네트워크 레벨에서 처리하기 때문에 서비스가 어떤 언어로 작성되었든 상관없다. 서비스 코드에는 비즈니스 로직만 남는다.

## 사이드카 패턴

### 개념

사이드카 패턴은 오토바이 옆에 붙은 사이드카에서 이름을 따왔다. 메인 컨테이너(애플리케이션) 옆에 보조 컨테이너(프록시)를 붙여서 동작시키는 방식이다.

Kubernetes에서는 하나의 Pod 안에 두 개의 컨테이너가 뜬다. 같은 네트워크 네임스페이스를 공유하기 때문에 localhost로 통신한다.

```
Pod
┌─────────────────────────────────────────┐
│  ┌──────────────┐  ┌──────────────────┐ │
│  │ order-service │  │  envoy-proxy     │ │
│  │ (port 8080)  │←→│  (port 15001)    │ │
│  │              │  │                  │ │
│  │ 비즈니스     │  │  mTLS, 재시도,    │ │
│  │ 로직만 처리   │  │  메트릭, 트래픽   │ │
│  └──────────────┘  └──────────────────┘ │
└─────────────────────────────────────────┘
```

order-service가 payment-service를 호출하면 이런 흐름이 된다:

1. order-service가 `http://payment-service:8080/api/pay`로 요청을 보낸다
2. iptables 규칙에 의해 요청이 로컬 envoy-proxy로 리다이렉트된다
3. envoy-proxy가 mTLS로 암호화하고, 재시도 정책을 적용하고, 메트릭을 기록한다
4. payment-service Pod의 envoy-proxy가 요청을 받아서 복호화한다
5. payment-service 컨테이너에 요청이 도달한다

서비스 입장에서는 그냥 HTTP 호출을 한 것뿐인데, 사이에서 프록시가 모든 걸 처리한 것이다.

### iptables 리다이렉션

사이드카 패턴에서 가장 헷갈리는 부분이 "서비스 코드를 안 건드렸는데 어떻게 프록시를 거치게 되는가"다. 답은 iptables다.

Istio의 경우 init 컨테이너(`istio-init`)가 Pod 시작 시 iptables 규칙을 설정한다. 포트 15001과 15006으로 인바운드/아웃바운드 트래픽을 리다이렉트한다.

```bash
# istio-init이 설정하는 iptables 규칙 (단순화)
iptables -t nat -A OUTPUT -p tcp -j ISTIO_OUTPUT
iptables -t nat -A ISTIO_OUTPUT -m owner --uid-owner 1337 -j RETURN  # envoy 자신의 트래픽은 제외
iptables -t nat -A ISTIO_OUTPUT -j REDIRECT --to-port 15001          # 나머지는 envoy로
```

envoy 프로세스의 UID(1337)에서 나가는 트래픽은 리다이렉트하지 않는다. 이걸 안 하면 envoy가 보낸 요청이 다시 envoy로 돌아오는 루프에 빠진다.

이 방식의 단점은 iptables 규칙이 잘못되면 Pod 내부 통신이 전부 먹통이 된다는 점이다. 사이드카 프록시가 죽으면 서비스 자체는 살아있어도 트래픽이 흐르지 않는다. 운영 중 사이드카 프록시의 리소스 부족으로 OOM이 터지면 이 상황이 발생할 수 있다.

## Istio

### 아키텍처

Istio는 데이터 플레인과 컨트롤 플레인으로 나뉜다.

**데이터 플레인**: 각 Pod에 주입되는 Envoy 프록시. 실제 트래픽을 처리한다.

**컨트롤 플레인**: istiod라는 단일 바이너리. 이전에는 Pilot, Citadel, Galley로 분리되어 있었는데 1.5 버전부터 하나로 합쳐졌다. istiod가 하는 일은 세 가지다:

- 서비스 디스커버리와 라우팅 규칙을 Envoy 설정으로 변환해서 배포한다
- 인증서 발급과 갱신을 관리한다 (mTLS)
- 사용자가 정의한 정책(VirtualService, DestinationRule 등)을 적용한다

```
┌──────────────────────────────────────────┐
│              컨트롤 플레인                 │
│  ┌──────────────────────────────────┐    │
│  │            istiod                │    │
│  │  - xDS API로 Envoy 설정 배포     │    │
│  │  - 인증서 관리 (CA)              │    │
│  │  - 서비스 디스커버리              │    │
│  └──────────────────────────────────┘    │
└──────────────────────────────────────────┘
                    ↕ xDS (gRPC)
┌──────────────────────────────────────────┐
│              데이터 플레인                 │
│  ┌────────┐  ┌────────┐  ┌────────┐     │
│  │Pod A   │  │Pod B   │  │Pod C   │     │
│  │app+envoy│  │app+envoy│  │app+envoy│     │
│  └────────┘  └────────┘  └────────┘     │
└──────────────────────────────────────────┘
```

### 설치와 사이드카 주입

```bash
# Istio 설치 (istioctl 사용)
istioctl install --set profile=demo -y

# 특정 네임스페이스에 자동 사이드카 주입 활성화
kubectl label namespace default istio-injection=enabled
```

`istio-injection=enabled` 라벨이 있는 네임스페이스에 Pod가 생성되면 자동으로 Envoy 사이드카가 주입된다. 기존에 돌고 있는 Pod에는 적용되지 않으니 `kubectl rollout restart deployment`로 재시작해야 한다.

주입을 원하지 않는 특정 Pod는 어노테이션으로 제외할 수 있다:

```yaml
metadata:
  annotations:
    sidecar.istio.io/inject: "false"
```

### mTLS 자동화

서비스 메시를 도입하는 가장 큰 이유 중 하나가 mTLS다. 서비스 간 통신을 암호화하고, 서로의 신원을 인증서로 검증한다. 코드를 한 줄도 안 건드리고 이걸 적용할 수 있다.

Istio가 mTLS를 처리하는 과정:

1. istiod가 각 서비스에 대해 X.509 인증서를 발급한다
2. 인증서를 Envoy 사이드카에 마운트한다
3. Envoy끼리 TLS 핸드셰이크로 상호 인증을 수행한다
4. 인증서가 만료되기 전에 istiod가 자동으로 갱신한다

```yaml
# 네임스페이스 전체에 mTLS STRICT 모드 적용
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: default
  namespace: production
spec:
  mtls:
    mode: STRICT
```

**mTLS 모드:**

| 모드 | 동작 |
|------|------|
| PERMISSIVE | mTLS와 평문 트래픽 모두 허용. 마이그레이션 과정에서 사용 |
| STRICT | mTLS만 허용. 사이드카가 없는 서비스에서 오는 요청은 거부 |
| DISABLE | mTLS 비활성화 |

처음 도입할 때는 PERMISSIVE로 시작한다. 모든 서비스에 사이드카가 주입된 걸 확인한 뒤에 STRICT로 전환한다. 한 번에 STRICT로 바꾸면 사이드카가 없는 서비스의 트래픽이 전부 끊긴다.

특정 서비스만 mTLS에서 제외해야 하는 경우도 있다. 외부에서 들어오는 웹훅이나, 메시 밖에 있는 레거시 서비스에서 호출이 오는 경우다.

```yaml
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: webhook-receiver
  namespace: production
spec:
  selector:
    matchLabels:
      app: webhook-receiver
  mtls:
    mode: PERMISSIVE
  portLevelMtls:
    8443:
      mode: STRICT    # 내부 통신은 mTLS
    8080:
      mode: DISABLE   # 외부 웹훅 수신 포트는 평문 허용
```

### 트래픽 제어

Istio의 트래픽 제어는 VirtualService와 DestinationRule 두 리소스로 한다.

**VirtualService**: 트래픽을 어디로 보낼지 결정한다. HTTP 라우팅 규칙, 가중치 기반 분배, 헤더 매칭 등을 설정한다.

**DestinationRule**: 도착지의 연결 정책을 정한다. 로드 밸런싱 방식, 커넥션 풀, 서킷 브레이커 등을 설정한다.

#### 카나리 배포

```yaml
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: payment-service
spec:
  hosts:
    - payment-service
  http:
    - route:
        - destination:
            host: payment-service
            subset: stable
          weight: 90
        - destination:
            host: payment-service
            subset: canary
          weight: 10
---
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: payment-service
spec:
  host: payment-service
  subsets:
    - name: stable
      labels:
        version: v1
    - name: canary
      labels:
        version: v2
```

v2에 10%만 보내면서 에러율과 응답 시간을 관찰한다. 문제 없으면 비율을 점진적으로 올린다. 배포 패턴 문서에서 다뤘던 Istio VirtualService 트래픽 분배가 이 설정이다.

#### 헤더 기반 라우팅

특정 사용자나 QA 팀만 새 버전으로 보내는 것도 가능하다.

```yaml
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: payment-service
spec:
  hosts:
    - payment-service
  http:
    - match:
        - headers:
            x-test-user:
              exact: "true"
      route:
        - destination:
            host: payment-service
            subset: canary
    - route:
        - destination:
            host: payment-service
            subset: stable
```

#### 서킷 브레이커

DestinationRule에서 서킷 브레이커를 설정한다. 애플리케이션 코드에 Resilience4j 같은 라이브러리를 넣지 않아도 된다.

```yaml
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: payment-service
spec:
  host: payment-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        h2UpgradePolicy: DEFAULT
        http1MaxPendingRequests: 50
        http2MaxRequests: 100
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
```

`outlierDetection`이 서킷 브레이커 역할이다. 5xx 에러가 연속 5번 발생하면 해당 인스턴스를 30초간 로드 밸런싱 풀에서 제외한다. `maxEjectionPercent: 50`은 전체 인스턴스의 절반 이상을 제외하지 않겠다는 설정이다. 이게 없으면 일시적 장애로 인스턴스 전부가 제외되는 상황이 생길 수 있다.

### 관측성 통합

Istio가 주입한 Envoy 프록시는 모든 트래픽 정보를 수집한다. 별도의 SDK 없이 다음 메트릭이 자동 생성된다:

- `istio_requests_total`: 요청 수 (소스, 목적지, 응답 코드별)
- `istio_request_duration_milliseconds`: 요청 소요 시간
- `istio_request_bytes` / `istio_response_bytes`: 요청/응답 크기

Prometheus가 이 메트릭을 수집하고, Grafana에서 서비스 간 트래픽 현황을 볼 수 있다. 분산 추적 문서에서 다룬 OpenTelemetry와 겹치는 부분이 있는데, Istio의 메트릭은 프록시 레벨에서 수집하는 것이라 애플리케이션 내부 로직의 Span까지는 잡지 못한다. 비즈니스 로직 수준의 트레이싱이 필요하면 OpenTelemetry SDK를 함께 사용해야 한다.

**Kiali**: Istio 전용 관측 도구다. 서비스 간 트래픽 흐름을 실시간 그래프로 보여준다. 어떤 서비스가 어떤 서비스를 호출하는지, 에러율은 얼마인지, mTLS가 적용되어 있는지를 한눈에 확인할 수 있다.

```bash
# Kiali 설치 (Istio demo 프로파일에는 포함됨)
kubectl apply -f https://raw.githubusercontent.com/istio/istio/release-1.24/samples/addons/kiali.yaml
kubectl port-forward svc/kiali -n istio-system 20001:20001
```

## Envoy

### Envoy의 위치

Envoy는 Istio의 데이터 플레인이다. Istio 없이 단독으로 사용하는 경우도 있다. CNCF 졸업 프로젝트이고, C++로 작성되어 성능이 좋다.

Envoy의 핵심 기능:

- L4/L7 프록시: TCP, HTTP/1.1, HTTP/2, gRPC 지원
- 동적 설정: xDS API로 런타임에 설정을 변경할 수 있다. 재시작 불필요
- 관측성: 내장 통계, 분산 추적 지원
- 확장성: Wasm(WebAssembly)으로 커스텀 필터를 만들 수 있다

### xDS API

Envoy의 설정을 동적으로 업데이트하는 프로토콜이다. "x"는 다양한 디스커버리 서비스를 의미한다.

| API | 역할 |
|-----|------|
| LDS (Listener Discovery) | 리스너 설정 (포트, 프로토콜) |
| RDS (Route Discovery) | 라우팅 규칙 |
| CDS (Cluster Discovery) | 업스트림 클러스터 (목적지 서비스) |
| EDS (Endpoint Discovery) | 클러스터 내 엔드포인트 (IP:Port) |
| SDS (Secret Discovery) | TLS 인증서 |

istiod는 Kubernetes의 Service, Deployment, Istio CRD(VirtualService, DestinationRule 등) 변경을 감지하고, xDS API를 통해 각 Envoy에 업데이트된 설정을 푸시한다. VirtualService를 수정하면 몇 초 안에 모든 Envoy에 반영된다.

### Envoy 단독 사용

서비스 메시 전체를 도입하기 부담스러우면 Envoy를 API Gateway나 사이드카 프록시로 단독 사용할 수 있다.

```yaml
# envoy.yaml (간단한 리버스 프록시 설정)
static_resources:
  listeners:
    - name: listener_0
      address:
        socket_address:
          address: 0.0.0.0
          port_value: 8080
      filter_chains:
        - filters:
            - name: envoy.filters.network.http_connection_manager
              typed_config:
                "@type": type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3.HttpConnectionManager
                stat_prefix: ingress_http
                route_config:
                  virtual_hosts:
                    - name: backend
                      domains: ["*"]
                      routes:
                        - match:
                            prefix: "/api/orders"
                          route:
                            cluster: order_service
                            timeout: 5s
                            retry_policy:
                              retry_on: "5xx,connect-failure"
                              num_retries: 3
                http_filters:
                  - name: envoy.filters.http.router
                    typed_config:
                      "@type": type.googleapis.com/envoy.extensions.filters.http.router.v3.Router
  clusters:
    - name: order_service
      type: STRICT_DNS
      lb_policy: ROUND_ROBIN
      load_assignment:
        cluster_name: order_service
        endpoints:
          - lb_endpoints:
              - endpoint:
                  address:
                    socket_address:
                      address: order-service
                      port_value: 8080
```

Envoy 설정은 YAML이 깊어서 처음 보면 복잡하다. Istio를 쓰면 VirtualService, DestinationRule 같은 상위 추상화로 이 설정을 자동 생성해준다. 설정 복잡도를 줄이는 게 Istio를 쓰는 이유 중 하나다.

## Linkerd

### Istio와 다른 점

Linkerd는 Istio보다 가볍다. 구조가 단순하고 리소스 사용량이 적다. Rust로 작성된 자체 프록시(linkerd2-proxy)를 사용한다.

| 항목 | Istio | Linkerd |
|------|-------|---------|
| 데이터 플레인 프록시 | Envoy (C++) | linkerd2-proxy (Rust) |
| 프록시 메모리 사용량 | Pod당 약 50~100MB | Pod당 약 10~20MB |
| 설정 복잡도 | 높음 (CRD 많음) | 낮음 (가능한 한 자동) |
| 기능 범위 | 넓음 (Ingress, Egress, VM 등) | 좁음 (핵심 기능에 집중) |
| mTLS | 설정 필요 (PeerAuthentication) | 기본 활성화 |
| 멀티 클러스터 | 지원 | 지원 |
| CNCF 등급 | Graduated | Graduated |

### 설치와 사용

```bash
# Linkerd CLI 설치
curl --proto '=https' --tlsv1.2 -sSfL https://run.linkerd.io/install | sh

# 클러스터 사전 검사
linkerd check --pre

# 컨트롤 플레인 설치
linkerd install --crds | kubectl apply -f -
linkerd install | kubectl apply -f -

# 특정 네임스페이스에 메시 적용
kubectl annotate namespace default linkerd.io/inject=enabled
kubectl rollout restart deployment -n default
```

Linkerd는 설치 후 별도 설정 없이 mTLS가 활성화된다. Istio처럼 PeerAuthentication을 따로 만들 필요가 없다.

### Linkerd를 선택하는 경우

- 서비스 수가 적고, 서비스 메시의 핵심 기능(mTLS, 관측성, 재시도)만 필요할 때
- 사이드카 프록시의 리소스 오버헤드를 최소화해야 할 때
- Envoy의 복잡한 설정을 다루고 싶지 않을 때
- 운영 인력이 적어서 단순한 시스템을 선호할 때

반대로 트래픽 미러링, 세밀한 라우팅 규칙, Wasm 확장, VM 워크로드 통합이 필요하면 Istio가 맞다.

## Ambient Mesh (사이드카 없는 서비스 메시)

### 사이드카 패턴의 한계

사이드카 패턴은 잘 동작하지만 몇 가지 운영 이슈가 있다.

**리소스 오버헤드**: 서비스마다 프록시가 하나씩 붙으니 Pod 수만큼 프록시가 뜬다. 서비스 100개면 프록시도 100개다. Pod당 50~100MB 메모리를 잡으면 프록시만으로 5~10GB를 쓰는 셈이다.

**sidecar race condition**: Pod가 시작될 때 애플리케이션 컨테이너가 사이드카보다 먼저 뜨면, 사이드카가 준비되기 전에 외부 호출을 시도해서 실패한다. `holdApplicationUntilProxyStarts` 설정으로 해결할 수 있지만, Pod 시작 시간이 늘어난다.

**업그레이드 부담**: Istio를 업그레이드하면 모든 Pod의 사이드카를 재시작해야 한다. 서비스 수백 개를 순차적으로 재시작하는 건 시간이 오래 걸린다.

### Istio Ambient Mesh

Istio 1.18부터 도입된 Ambient Mesh는 사이드카 없이 서비스 메시를 구현한다. 노드당 하나의 ztunnel(제로 트러스트 터널) 데몬셋이 L4 처리(mTLS, TCP 라우팅)를 담당하고, L7 처리(HTTP 라우팅, 재시도 등)가 필요한 서비스만 waypoint proxy를 거친다.

```
기존 사이드카 방식:
  Pod마다 envoy → 리소스 N배

Ambient Mesh:
  노드당 ztunnel 1개 (L4)
  필요한 서비스만 waypoint proxy (L7)
```

```bash
# Ambient 모드로 Istio 설치
istioctl install --set profile=ambient

# 네임스페이스에 Ambient 메시 적용
kubectl label namespace default istio.io/dataplane-mode=ambient
```

사이드카 주입이 아니라 라벨 하나로 메시에 편입된다. 기존 Pod를 재시작할 필요도 없다.

아직 일부 기능이 사이드카 모드보다 제한적이다. 새로 도입하는 환경이라면 검토해볼 만하지만, 이미 사이드카 방식으로 운영 중이라면 안정화될 때까지 지켜보는 게 현실적이다.

## 서비스 메시 도입 시 주의사항

### 도입 시점

서비스가 5개 미만이면 서비스 메시가 과하다. 라이브러리 레벨(Resilience4j, gRPC interceptor 등)로 충분하다. 서비스가 10개를 넘어가고, 언어가 2개 이상 섞이고, mTLS나 세밀한 트래픽 제어가 필요해지는 시점에 도입을 검토한다.

### 디버깅 복잡도 증가

사이드카가 끼어들면서 네트워크 문제가 생겼을 때 원인 파악이 복잡해진다. "서비스 문제인지, 프록시 문제인지, 메시 설정 문제인지" 세 가지를 다 봐야 한다.

```bash
# Envoy 프록시 상태 확인
istioctl proxy-status

# 특정 Pod의 Envoy 설정 덤프
istioctl proxy-config all <pod-name> -o json

# Envoy 액세스 로그 확인
kubectl logs <pod-name> -c istio-proxy
```

`proxy-config` 명령으로 Envoy에 실제 적용된 라우팅 규칙, 클러스터 정보를 확인할 수 있다. VirtualService를 적용했는데 트래픽이 예상대로 안 흐를 때 이 명령으로 확인한다.

### 리소스 계획

사이드카 프록시의 리소스를 미리 계획해야 한다. 기본값으로 시작하면 트래픽이 몰릴 때 프록시가 CPU나 메모리 부족으로 느려지거나 죽는다.

```yaml
# 사이드카 리소스 설정
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  template:
    metadata:
      annotations:
        sidecar.istio.io/proxyCPU: "100m"
        sidecar.istio.io/proxyMemory: "128Mi"
        sidecar.istio.io/proxyCPULimit: "500m"
        sidecar.istio.io/proxyMemoryLimit: "512Mi"
```

트래픽이 많은 서비스는 프록시 리소스를 넉넉하게 잡는다. 트래픽이 적은 서비스는 줄인다. 전체 서비스에 동일하게 설정하면 리소스가 낭비되거나 부족해진다.

### 점진적 도입

한 번에 전체 클러스터에 적용하지 않는다.

1. 비프로덕션 환경에 먼저 적용해서 운영 경험을 쌓는다
2. 프로덕션에서 트래픽이 적은 네임스페이스부터 시작한다
3. mTLS는 PERMISSIVE로 시작하고, 모든 서비스에 사이드카가 주입된 걸 확인한 뒤 STRICT로 전환한다
4. 트래픽 제어(VirtualService, DestinationRule)는 기존 동작을 그대로 재현하는 것부터 시작한다. 처음부터 복잡한 라우팅 규칙을 넣으면 문제가 생겼을 때 원인 특정이 어렵다
