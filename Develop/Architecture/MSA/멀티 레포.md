# 모노레포 vs 멀티레포

서비스를 여러 개로 쪼개면 코드를 어디에 둘지 정해야 한다. 모든 서비스를 한 저장소에 넣을지(모노레포), 서비스마다 저장소를 따로 둘지(멀티레포). 이 결정은 코드 한 줄 바꿀 때 PR을 몇 개 올려야 하는지, 공통 라이브러리를 고쳤을 때 배포 순서를 어떻게 잡을지, 빌드가 몇 분 걸리는지까지 매일 영향을 준다.

둘 중 무엇이 정답이라고 말하기는 어렵다. 팀 수, 서비스 간 의존 정도, 빌드 인프라에 얼마나 투자할 수 있는지에 따라 다르다. 여기서는 추상적인 장단점 나열 대신, 실제로 운영하면서 부딪히는 지점을 중심으로 둘을 비교한다.

```
멀티레포                          모노레포
─────────                        ─────────
user-service/    (repo)          monorepo/
payment-service/ (repo)          ├── apps/
product-service/ (repo)          │   ├── user-service/
shared-utils/    (repo)          │   ├── payment-service/
                                 │   └── product-service/
각 저장소가 독립                 └── packages/
공유는 패키지 배포로                 └── shared-utils/
                                 한 저장소, 공유는 직접 참조
```

## 멀티레포에서 공통 코드를 다루는 문제

멀티레포의 핵심 마찰은 공통 코드다. 로깅, 인증 미들웨어, DTO 타입처럼 여러 서비스가 같이 쓰는 코드를 어딘가 한 곳에 두고 나눠 써야 하는데, 저장소가 분리돼 있으니 직접 import할 방법이 없다. 그래서 공통 코드를 패키지로 만들어 사내 레지스트리에 배포하고, 각 서비스가 버전을 명시해서 끌어다 쓴다.

### 버전 드리프트

패키지로 나누는 순간 버전 드리프트가 생긴다. `@company/shared-utils`를 1.2.0으로 올렸는데, user-service는 이미 1.5.0을 쓰고, payment-service는 아직 1.1.0에 묶여 있고, 새로 만든 order-service만 1.2.0을 쓰는 식이다. 어느 서비스가 어떤 버전을 쓰는지 한눈에 안 보인다.

```json
// user-service/package.json
{ "dependencies": { "@company/shared-utils": "^1.5.0" } }

// payment-service/package.json
{ "dependencies": { "@company/shared-utils": "1.1.0" } }

// order-service/package.json
{ "dependencies": { "@company/shared-utils": "^1.2.0" } }
```

문제는 공통 라이브러리에 보안 패치를 넣었을 때 드러난다. 1.6.0으로 올리고 나면 N개 서비스 저장소에 각각 PR을 올려 버전을 올려야 한다. 서비스가 30개면 PR이 30개다. 각 PR은 해당 서비스 팀의 리뷰를 받아야 하고, 머지 타이밍도 제각각이라 패치가 전체에 반영되기까지 몇 주가 걸리기도 한다.

`^1.5.0` 같은 캐럿 범위를 쓰면 `npm install` 시점에 최신 마이너를 자동으로 당겨오니 패치 전파는 빨라지지만, 빌드 재현성이 깨진다. 어제 통과한 빌드가 오늘 깨질 수 있다. 그래서 실무에서는 lockfile을 커밋하고 버전을 정확히 고정(pinning)하는 쪽으로 간다. 대신 전파 비용을 감수한다.

### 다이아몬드 의존성

버전이 갈리면 다이아몬드 의존성으로 번진다. order-service가 `payment-client@2.0`과 `shipping-client@1.0`을 동시에 의존하는데, 둘이 각각 `shared-types`의 다른 메이저 버전을 요구하는 상황이다.

```
       order-service
        /         \
payment-client   shipping-client
   (shared-types@2)  (shared-types@1)
        \         /
        shared-types  ← 2와 1을 동시에 요구
```

npm/yarn은 node_modules에 두 버전을 중첩 설치해서 일단 돌아가게 만든다. 그런데 `shared-types`에 정의된 `Money` 타입 객체가 payment-client를 거쳐 shipping-client로 넘어가면, 런타임에 같은 모양인데 다른 클래스 인스턴스라 `instanceof` 검사가 깨지거나 타입 단언이 어긋난다. 추적하기 까다로운 버그다. 결국 두 클라이언트의 `shared-types` 버전을 맞추려고 또 cross-repo PR을 돌려야 한다.

### 사내 레지스트리 운영

공통 패키지를 npm 공개 레지스트리에 올릴 수는 없으니 사내 레지스트리를 둔다. Verdaccio를 직접 띄우거나, GitHub Packages, AWS CodeArtifact, JFrog Artifactory 같은 호스팅을 쓴다.

```
# .npmrc - 사내 스코프만 사설 레지스트리로 라우팅
@company:registry=https://npm.company.internal/
//npm.company.internal/:_authToken=${NPM_TOKEN}
registry=https://registry.npmjs.org/
```

운영하면서 부딪히는 건 인증 토큰 관리다. CI에서 `NPM_TOKEN`을 시크릿으로 주입해야 하고, 토큰이 만료되면 모든 서비스 빌드가 한꺼번에 멈춘다. 로컬 개발자 환경에도 같은 설정이 필요해서 온보딩 문서에 `.npmrc` 설정이 빠지면 신규 입사자가 첫날 빌드를 못 돌린다. 사설 패키지 배포 권한을 누가 갖는지, 버전 태깅 규칙을 어떻게 강제할지도 정해둬야 아무나 1.0.0을 덮어쓰는 사고를 막는다.

## 모노레포에서 공통 코드를 다루는 방법

모노레포는 같은 저장소 안에 공통 코드가 있으니 패키지 배포 단계가 사라진다. workspace 기능으로 로컬 패키지를 심볼릭 링크처럼 직접 참조한다.

```json
// 루트 package.json
{
  "workspaces": ["apps/*", "packages/*"]
}

// apps/order-service/package.json
{ "dependencies": { "@company/shared-types": "workspace:*" } }
```

`workspace:*`는 레지스트리가 아니라 같은 저장소의 `packages/shared-types`를 직접 가리킨다. shared-types를 고치면 order-service가 즉시 그 변경을 본다. 버전 드리프트가 구조적으로 생기지 않는다. 모든 서비스가 항상 같은 코드를 본다.

대신 다른 문제가 생긴다. shared-types를 고치면 그것을 쓰는 모든 서비스가 동시에 영향을 받는다. 멀티레포라면 버전을 안 올린 서비스는 영향이 없는데, 모노레포는 그런 격리가 없다. 그래서 "바뀐 것만 다시 빌드·테스트"하는 도구가 필수가 된다. 이게 다음 절의 빌드 도구 이야기로 이어진다.

## 모노레포 빌드 도구

서비스가 수십 개인 모노레포에서 매번 전체를 빌드·테스트하면 CI가 30분, 1시간씩 걸린다. 핵심은 변경된 코드와 그 영향을 받는 패키지만 골라서 작업하는 것(affected)과, 한 번 빌드한 결과를 재사용하는 캐시다.

### Turborepo

설정이 가볍다. 의존 그래프를 보고 영향받은 패키지만 빌드하며, 이미 빌드한 결과는 해시 기반으로 캐싱해서 건너뛴다.

```json
// turbo.json
{
  "$schema": "https://turbo.build/schema.json",
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**"]
    },
    "test": {
      "dependsOn": ["build"],
      "outputs": ["coverage/**"]
    },
    "lint": {}
  }
}
```

`dependsOn: ["^build"]`은 이 패키지를 빌드하기 전에 의존하는 패키지를 먼저 빌드하라는 뜻이다. 캐럿(`^`)이 의존성 방향을 가리킨다. CI에서는 변경 범위를 기준으로 affected만 돌린다.

```bash
# main과 비교해 바뀐 패키지와 그 영향권만 빌드
turbo run build --filter='...[origin/main]'

# 테스트도 동일하게, 캐시 히트하면 즉시 통과
turbo run test --filter='...[origin/main]'
```

`shared-types`만 바꿨다면 그것을 의존하는 서비스들만 다시 빌드되고, 무관한 서비스는 캐시에서 바로 통과한다. 처음 Turborepo를 붙이면 CI 시간이 절반 이하로 줄어드는 경우가 흔하다.

### 원격 캐시

로컬 캐시는 내 머신에만 남는다. CI 러너는 매번 새 환경이라 캐시가 비어 있어서 효과를 못 본다. 원격 캐시를 붙이면 누군가(또는 다른 CI 잡)가 이미 빌드한 결과를 공유한다. A가 빌드한 패키지를 B가 받아 쓰고, CI가 만든 캐시를 로컬에서도 당겨 쓴다.

```bash
# 원격 캐시 연결 (Vercel 호스팅 또는 자체 호스팅 서버)
turbo login
turbo link

# 자체 호스팅 시 turbo.json 또는 환경변수로 엔드포인트 지정
# TURBO_API, TURBO_TOKEN, TURBO_TEAM
```

원격 캐시가 제대로 돌면 "PR 올렸는데 내가 안 건드린 패키지 테스트가 5초 만에 통과"하는 경험을 한다. 다만 캐시 키 계산에 환경변수나 파일 해시가 빠지면 잘못된 캐시 히트(stale cache)가 나서 옛날 결과를 가져오는 사고가 생긴다. `globalEnv`, `inputs`를 정확히 선언해야 한다.

### Nx

Turborepo와 비슷한 affected·캐시 기능에 더해 의존 그래프 시각화, 코드 생성기(generator), 플러그인 생태계가 붙는다. 그래프를 그려보면 어떤 서비스가 어떤 라이브러리에 묶여 있는지 한눈에 보인다.

```bash
# 변경 영향권 그래프를 브라우저로 확인
nx affected:graph --base=origin/main

# 영향받은 프로젝트만 테스트
nx affected --target=test --base=origin/main
```

설정과 개념(project.json, target, executor)이 Turborepo보다 많아서 초기 학습 비용이 있다. 대신 규모가 커지고 코드 생성·일관성 강제가 중요해지면 그만큼 값을 한다.

### Bazel

Google이 만든 빌드 시스템이다. JS/TS뿐 아니라 Go, Java, C++, Python을 한 저장소에서 같은 방식으로 빌드한다. 언어가 섞인 대규모 모노레포(서비스 수백~수천 개)에서 쓰는 이유는 입력이 똑같으면 출력도 똑같다는 것을 강하게 보장하는 hermetic 빌드와, 그에 기반한 정확한 캐시·원격 실행 때문이다.

```python
# BUILD.bazel - 모든 의존성을 명시적으로 선언
ts_project(
    name = "shared_types",
    srcs = glob(["src/**/*.ts"]),
    deps = ["//packages/common:common"],
)

ts_project(
    name = "order_service",
    srcs = glob(["apps/order/**/*.ts"]),
    deps = [":shared_types"],
)
```

진입장벽이 높다. 모든 의존성을 `BUILD` 파일에 손으로 선언해야 하고, npm 같은 기존 생태계를 Bazel 규칙으로 감싸는 작업이 만만치 않다. 빌드 그래프를 한번 제대로 세워두면 캐시 적중률과 빌드 정확성이 다른 도구와 차이가 나지만, 그 상태에 이르기까지 전담 인력이 붙어야 한다. 서비스 몇 개 수준이면 Bazel은 과하다. Turborepo나 Nx로 충분하다.

## CI/CD 파이프라인

### 멀티레포: 서비스마다 파이프라인 복제

멀티레포는 저장소마다 독립적인 워크플로를 둔다. 각 저장소에 push되면 그 서비스만 빌드·배포한다. 구조가 단순하고 격리가 명확하다.

```yaml
# 각 서비스 저장소의 .github/workflows/deploy.yml (거의 동일하게 복제됨)
name: Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm test
      - run: npm run build
      - run: npm run deploy
```

문제는 이 워크플로가 30개 저장소에 거의 똑같이 복사돼 있다는 점이다. Node 버전을 18에서 20으로 올리거나, 배포 스텝에 보안 스캔을 추가하려면 30개 저장소를 각각 고쳐야 한다. 공통 워크플로를 reusable workflow로 빼서 호출하는 식으로 줄일 수 있지만, 그 호출 코드 자체는 여전히 각 저장소에 들어간다. 파이프라인 표준화가 조직 차원의 숙제로 남는다.

### 모노레포: 바뀐 패키지만 빌드·배포

모노레포는 워크플로가 하나지만, 한 저장소가 모든 서비스를 담고 있어서 "무엇이 바뀌었는지" 판단하는 단계가 핵심이 된다. 무관한 서비스까지 매번 배포할 수는 없다. 변경 경로를 필터링해서 영향받은 서비스만 골라낸다.

```yaml
# .github/workflows/deploy.yml - paths-filter로 바뀐 서비스만 감지
name: Deploy
on:
  push:
    branches: [main]
jobs:
  detect:
    runs-on: ubuntu-latest
    outputs:
      user: ${{ steps.filter.outputs.user }}
      payment: ${{ steps.filter.outputs.payment }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            user:
              - 'apps/user-service/**'
              - 'packages/shared-types/**'
            payment:
              - 'apps/payment-service/**'
              - 'packages/shared-types/**'

  deploy-user:
    needs: detect
    if: needs.detect.outputs.user == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run deploy --workspace=user-service

  deploy-payment:
    needs: detect
    if: needs.detect.outputs.payment == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run deploy --workspace=payment-service
```

`shared-types`를 두 서비스 필터에 모두 넣은 이유는, 공통 패키지가 바뀌면 그것을 쓰는 서비스를 전부 다시 배포해야 하기 때문이다. paths-filter를 직접 관리하면 의존 관계가 바뀔 때마다 필터를 손으로 갱신해야 한다. 그래서 Turborepo의 `--filter='...[origin/main]'`이나 Nx의 `affected`로 의존 그래프 기반 감지를 맡기는 쪽이 누락이 적다. paths-filter는 단순한 케이스에, affected는 의존 관계가 복잡할 때 쓴다.

## 원자적 커밋과 cross-repo 변경

### 모노레포: 한 커밋으로 여러 서비스 변경

`shared-types`의 `User` 타입에 필드를 추가하면서 그것을 쓰는 세 서비스를 같이 고쳐야 한다고 하자. 모노레포에서는 한 PR, 한 커밋에 전부 담긴다.

```bash
# 한 커밋에 공통 타입 + 영향받는 서비스 전부
git add packages/shared-types/ apps/user-service/ apps/order-service/ apps/payment-service/
git commit -m "feat: User에 phoneVerified 추가 및 사용처 반영"
```

이 커밋은 항상 일관된 상태다. 어느 시점에 체크아웃해도 타입 정의와 사용처가 맞물려 있다. 리뷰어도 변경 전체를 한 PR에서 본다. 리팩터링 범위가 서비스를 넘나들 때 모노레포의 가장 큰 장점이 여기서 나온다.

### 멀티레포: PR 순서와 배포 순서

같은 변경을 멀티레포에서 하면 cross-repo 문제가 된다. shared-types 저장소에 PR을 올려 머지하고, 패키지를 1.6.0으로 배포한 뒤, 세 서비스 저장소에 각각 PR을 올려 1.6.0을 당겨 쓰도록 고친다. PR이 최소 네 개고 순서가 정해져 있다.

```
1. shared-types PR 머지 → 1.6.0 배포   (먼저)
2. user-service PR: shared-types ^1.6.0 으로 올림
3. order-service PR: 동일
4. payment-service PR: 동일
```

순서가 꼬이면 깨진다. shared-types를 아직 배포 안 했는데 서비스 PR을 먼저 머지하면 그 서비스 빌드가 없는 버전을 찾다가 실패한다. 반대로 하위 호환을 깨는 변경(breaking change)을 shared-types에 넣고 바로 배포하면, 아직 안 올린 서비스들이 다음 `npm install` 때 깨진다.

배포 순서도 의존 방향을 따라야 한다. 이벤트 스키마를 바꿀 때 보통 컨슈머(받는 쪽)를 먼저 신·구 양쪽 다 처리하도록 배포하고, 그다음 프로듀서(보내는 쪽)를 바꾼다. 이 순서를 사람이 기억해서 챙겨야 하고, 여러 팀이 얽히면 배포 일정 조율 회의가 따로 생긴다. 모노레포의 원자적 커밋이 없애주는 비용이 멀티레포에서는 운영 절차로 되돌아온다.

대신 멀티레포의 격리가 장점이 되는 경우도 있다. payment-service만 핫픽스를 배포하고 싶을 때, 다른 서비스 변경에 얽매이지 않고 그 저장소만 독립적으로 내보낸다. 모노레포에서는 같은 메인 브랜치에 섞인 다른 변경과 분리하려면 추가 작업이 든다.

## 권한 분리와 CODEOWNERS

멀티레포는 권한 분리가 저장소 단위로 자연스럽다. payment-service 저장소에 결제 팀만 write 권한을 주면 다른 팀은 코드를 못 건드린다. 보안·규제 때문에 결제 코드를 격리해야 하는 조직에서 멀티레포를 택하는 이유 중 하나다.

모노레포는 모두가 한 저장소에 있으니 디렉터리 단위로 책임을 나눈다. GitHub의 CODEOWNERS가 그 역할을 한다.

```
# .github/CODEOWNERS
/packages/shared-types/   @company/platform-team
/apps/payment-service/    @company/payment-team
/apps/user-service/       @company/identity-team
*.bazel                   @company/build-infra
/.github/workflows/       @company/devops
```

해당 경로를 건드리는 PR은 지정된 팀의 리뷰 승인을 받아야 머지된다. 다만 CODEOWNERS는 리뷰 강제일 뿐 write 권한 차단은 아니다. 누군가 payment-service를 고치는 PR을 올리는 것 자체는 막지 못한다. 진짜 접근 차단이 필요하면 모노레포로는 한계가 있고, 그 코드는 별도 저장소로 빼는 편이 맞다. 권한 격리가 비즈니스 요구사항이면 멀티레포 쪽으로 기운다.

공통 패키지에 CODEOWNERS를 거는 건 양날의 검이다. `shared-types`를 platform-team이 소유하면 품질은 지키지만, 모든 서비스 팀의 타입 변경 PR이 platform-team 리뷰에서 병목이 된다. 소유권 경계를 너무 촘촘히 잡으면 리뷰 대기가 길어진다.

## 두 방식 사이 마이그레이션

조직이 커지면서 멀티레포를 모노레포로 합치거나, 비대해진 모노레포에서 일부를 떼어내는 일이 생긴다. 이때 가장 신경 쓰는 건 git history 보존이다. 단순 복사로 옮기면 커밋 이력, blame, 작성자 정보가 전부 날아간다. 장애 원인을 찾을 때 "이 줄을 왜 이렇게 짰지"를 git blame으로 추적하는데, 이력이 끊기면 그게 막힌다.

### 멀티레포 → 모노레포: git subtree로 합치기

여러 저장소를 history를 살린 채 한 모노레포의 하위 디렉터리로 가져온다.

```bash
# 모노레포 쪽에서 각 서비스를 subtree로 흡수
git remote add user-service git@github.com:company/user-service.git
git fetch user-service
git subtree add --prefix=apps/user-service user-service main

git remote add payment-service git@github.com:company/payment-service.git
git fetch payment-service
git subtree add --prefix=apps/payment-service payment-service main
```

subtree로 가져오면 원래 저장소의 커밋들이 모노레포 이력에 합쳐진다. 다만 파일 경로가 `apps/user-service/` 아래로 바뀌어서, 옮기기 전 커밋의 경로와 옮긴 후 경로가 달라 blame이 경계에서 한 번 끊기는 건 감수해야 한다.

### 경로를 재배치하며 옮기기: git filter-repo

원래 저장소가 루트에 코드를 두고 있었다면, 모노레포에 넣기 전에 미리 하위 경로로 옮겨두는 편이 깔끔하다. `git filter-repo`로 전체 이력의 경로를 다시 쓴다.

```bash
# user-service 저장소 복제본에서 모든 이력을 apps/user-service/ 아래로 이동
git clone git@github.com:company/user-service.git
cd user-service
git filter-repo --to-subdirectory-filter apps/user-service

# 그 다음 모노레포에서 이 정리된 저장소를 가져오면 경로가 처음부터 맞는다
```

`git filter-repo`는 `git filter-branch`보다 빠르고 안전하다. filter-branch는 큰 저장소에서 몇 시간씩 걸리고 함정이 많아서 공식 문서도 filter-repo를 권한다. 별도 설치가 필요한 외부 도구다.

### 모노레포 → 멀티레포: 일부만 떼어내기

특정 디렉터리만 이력째 새 저장소로 분리할 때도 filter-repo를 쓴다.

```bash
git clone <monorepo> payment-service
cd payment-service
git filter-repo --path apps/payment-service/ --path-rename apps/payment-service/:
# apps/payment-service/만 남기고 루트로 끌어올린 새 저장소가 된다
```

마이그레이션은 한 번에 끝나지 않는 경우가 많다. 합치는 동안 양쪽에서 동시에 커밋이 들어오면 분기 시점 이후의 변경을 다시 맞춰야 한다. 보통 코드 동결(freeze) 구간을 잡고, 그 시간에 옮기고 검증한 뒤 새 저장소로 전환한다. CI 설정, 배포 파이프라인, import 경로, 사내 레지스트리 의존을 workspace 참조로 바꾸는 작업이 따라오니 디렉터리 옮기는 것보다 주변 정리가 더 오래 걸린다.

## 무엇을 고를까

선택을 가르는 실제 기준은 다음 쪽에 가깝다.

서비스끼리 공통 코드를 자주 같이 고치고, 한 변경이 여러 서비스를 넘나드는 일이 잦다면 모노레포가 그 비용을 크게 줄인다. 단, 모노레포는 affected 빌드와 원격 캐시 같은 도구 투자가 전제다. 그게 없으면 저장소가 커질수록 CI가 느려지면서 오히려 발목을 잡는다.

팀이 완전히 독립적으로 움직이고, 서비스마다 기술 스택이 다르거나, 보안·규제로 코드 접근을 물리적으로 격리해야 한다면 멀티레포가 맞는다. 대신 공통 라이브러리 버전 전파와 cross-repo 변경 순서를 다루는 운영 절차를 받아들여야 한다.

둘을 섞는 경우도 흔하다. 같이 변하는 서비스 묶음은 모노레포로 두고, 독립성이나 격리가 중요한 서비스는 별도 저장소로 빼는 식이다. 처음부터 전부 한쪽으로 몰지 말고, 함께 변경되는 빈도를 보고 경계를 긋는 게 현실적이다.
