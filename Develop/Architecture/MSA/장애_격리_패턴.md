# 장애 격리 패턴

## 왜 장애 격리가 필요한가

모놀리식에서는 한 모듈이 느려지면 해당 요청만 느려진다. 스레드 풀을 공유하니까 전체가 느려질 수는 있지만, 서비스 자체가 죽지는 않는다.

MSA에서는 다르다. 서비스 A가 서비스 B를 HTTP로 호출하는데 B가 응답을 안 주면, A의 스레드가 B를 기다리면서 묶인다. 요청이 계속 들어오면 A의 스레드 풀이 고갈되고, A를 호출하는 다른 서비스도 연쇄적으로 죽는다.

이게 **장애 전파(Cascading Failure)**다. 결제 서비스 하나가 느려져서 주문, 상품, 장바구니까지 전부 먹통이 되는 상황이 실제로 생긴다.

장애 격리 패턴은 이 연쇄 반응을 끊는 것이 목적이다.

---

## 장애 전파 시나리오

실제 운영에서 겪는 전파 흐름을 먼저 이해해야 한다.

### 시나리오 1: 느린 서비스가 전체를 죽이는 경우

```
사용자 → API Gateway → 주문 서비스 → 결제 서비스 (응답 지연)
                         ↓
                   스레드 대기 → 스레드 풀 고갈 → 주문 서비스 응답 불가
                         ↓
                   API Gateway 타임아웃 → 사용자 전체 서비스 장애
```

결제 서비스의 DB 커넥션이 부족해서 응답이 5초씩 걸리기 시작한다. 주문 서비스는 기본 타임아웃이 30초라 스레드가 30초 동안 묶인다. 초당 100건 요청이면 30초 만에 3,000개 스레드가 결제 대기 상태에 빠진다. 주문 서비스 자체가 응답 불가가 되고, 상품 조회 같은 결제와 관련 없는 기능도 먹통이 된다.

### 시나리오 2: 재시도가 장애를 키우는 경우

```
주문 서비스 → 결제 서비스 (5xx 응답)
     ↓
재시도 3회 → 원래 트래픽의 4배 → 결제 서비스 완전히 다운
     ↓
복구 시도 → 재시도 트래픽이 계속 쏟아짐 → 복구 불가
```

서비스가 간헐적으로 500을 반환할 때, 클라이언트 3곳이 각각 3회 재시도하면 트래픽이 12배가 된다. 이미 힘든 서비스에 트래픽을 더 쏟아붓는 격이라 복구가 안 된다. 이런 상황을 **retry storm**이라고 한다.

### 시나리오 3: 하나의 외부 API가 전체를 끌어내리는 경우

```
알림 서비스 → 외부 SMS API (장애)
     ↓
주문 서비스 → 알림 서비스 호출 → 타임아웃
     ↓
주문 생성 자체가 실패
```

알림은 주문의 핵심 기능이 아닌데, 동기 호출로 연결되어 있으면 SMS API 장애가 주문 실패로 이어진다. 비핵심 서비스는 비동기로 분리하거나, 실패해도 주문 자체는 성공 처리해야 한다.

## Circuit Breaker

### 개념

전기 회로의 차단기와 같다. 특정 서비스 호출이 계속 실패하면 아예 호출을 차단해버린다. 실패할 게 뻔한 요청을 계속 보내봤자 자원만 낭비하고 상대 서비스 복구도 늦어진다.

### 상태 전이

```
CLOSED (정상) → 실패율 임계치 초과 → OPEN (차단)
                                        ↓
                                   대기 시간 경과
                                        ↓
                                   HALF_OPEN (시험)
                                        ↓
                              성공 → CLOSED / 실패 → OPEN
```

- **CLOSED**: 정상 상태. 모든 호출을 통과시키면서 성공/실패를 기록한다.
- **OPEN**: 차단 상태. 호출을 보내지 않고 바로 fallback을 실행하거나 예외를 던진다.
- **HALF_OPEN**: 일정 시간이 지나면 몇 건만 시험 삼아 보내본다. 성공하면 CLOSED로 복귀하고, 실패하면 다시 OPEN으로 간다.

### Resilience4j 설정

```yaml
resilience4j:
  circuitbreaker:
    instances:
      payment-service:
        sliding-window-type: COUNT_BASED
        sliding-window-size: 10
        failure-rate-threshold: 50
        wait-duration-in-open-state: 10s
        permitted-number-of-calls-in-half-open-state: 3
        slow-call-duration-threshold: 3s
        slow-call-rate-threshold: 80
        record-exceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
        ignore-exceptions:
          - com.example.exception.BusinessException
```

각 항목이 뜻하는 것:

- `sliding-window-size: 10` — 최근 10건의 호출 결과를 기준으로 판단한다.
- `failure-rate-threshold: 50` — 10건 중 5건 이상 실패하면 OPEN으로 전환한다.
- `wait-duration-in-open-state: 10s` — OPEN 상태에서 10초 후에 HALF_OPEN으로 전환한다.
- `permitted-number-of-calls-in-half-open-state: 3` — HALF_OPEN에서 3건을 시험한다.
- `slow-call-duration-threshold: 3s` — 3초 넘게 걸리는 호출은 느린 호출로 간주한다.
- `slow-call-rate-threshold: 80` — 느린 호출이 80%를 넘으면 OPEN으로 전환한다.
- `record-exceptions` — 이 예외만 실패로 기록한다.
- `ignore-exceptions` — 비즈니스 예외(잔액 부족 등)는 circuit breaker 실패에서 제외한다.

`ignore-exceptions` 설정을 빠뜨리면 비즈니스 로직상 정상적인 예외(주문 검증 실패 등)까지 실패로 카운트되어 circuit이 열린다. 실제로 이 설정 하나 빠뜨려서 운영 장애 난 경우가 있다.

### sliding window 타입 선택

```yaml
# 호출 수 기반 — 트래픽이 일정한 경우에 적합
sliding-window-type: COUNT_BASED
sliding-window-size: 10

# 시간 기반 — 트래픽 변동이 큰 경우에 적합
sliding-window-type: TIME_BASED
sliding-window-size: 10  # 최근 10초
```

트래픽이 적은 서비스에 COUNT_BASED를 쓰면 오래된 실패까지 반영되어 오판할 수 있다. 반대로 트래픽이 많은 서비스에 TIME_BASED를 쓰면 짧은 시간에 대량 호출이 들어와 메모리를 많이 쓴다. 서비스 특성에 맞게 골라야 한다.

### fallback 구현

```java
@CircuitBreaker(name = "payment-service", fallbackMethod = "paymentFallback")
public PaymentResponse processPayment(PaymentRequest request) {
    return paymentClient.process(request);
}

// circuit이 열렸을 때 실행된다
private PaymentResponse paymentFallback(PaymentRequest request, Exception e) {
    if (e instanceof CallNotPermittedException) {
        // circuit OPEN 상태 — 빠른 실패
        log.warn("payment circuit open, request={}", request.getOrderId());
        throw new ServiceUnavailableException("결제 서비스 일시 중단");
    }
    // 그 외 예외 — 실제 호출 실패
    log.error("payment call failed, request={}", request.getOrderId(), e);
    throw new PaymentFailedException("결제 처리 실패");
}
```

fallback에서 주의할 점이 있다. 결제처럼 멱등성이 보장되지 않는 호출은 fallback에서 대체 처리를 하면 안 된다. "결제가 실패했으니 일단 무료로 처리"하는 식의 fallback은 사고다. 사용자에게 재시도를 안내하거나 비동기 처리 큐에 넣는 것이 맞다.

## Bulkhead

### 개념

배의 격벽(bulkhead)에서 유래한 이름이다. 배에 구멍이 나도 격벽으로 막혀 있으면 침수가 한 구획에서 멈춘다. 서비스도 마찬가지로, 특정 호출이 자원을 전부 차지하지 못하게 격리한다.

Circuit Breaker가 "실패하는 호출을 차단"하는 거라면, Bulkhead는 "하나의 호출 대상이 전체 자원을 독점하지 못하게 제한"하는 것이다.

### 왜 필요한가

서비스 A가 서비스 B와 서비스 C를 호출한다고 하자. B가 느려져서 스레드 200개가 B를 기다리고 있으면, C를 호출하는 스레드도 부족해진다. B의 장애가 C 호출에까지 영향을 미치는 것이다.

Bulkhead로 "B 호출에는 최대 50개 스레드만 사용"이라고 제한하면, B가 아무리 느려도 나머지 150개 스레드로 C 호출은 정상 처리된다.

### 두 가지 타입

**Semaphore Bulkhead**: 동시 호출 수만 제한한다. 호출 스레드를 그대로 사용한다.

```yaml
resilience4j:
  bulkhead:
    instances:
      payment-service:
        max-concurrent-calls: 20
        max-wait-duration: 500ms
```

`max-concurrent-calls: 20`이면 동시에 20건까지만 결제 서비스를 호출한다. 21번째 요청은 500ms까지 대기하다가 `BulkheadFullException`을 던진다.

**Thread Pool Bulkhead**: 별도 스레드 풀에서 실행한다. 완전한 격리가 가능하지만 비동기로 동작한다.

```yaml
resilience4j:
  thread-pool-bulkhead:
    instances:
      payment-service:
        max-thread-pool-size: 10
        core-thread-pool-size: 5
        queue-capacity: 20
        keep-alive-duration: 100ms
```

Thread Pool Bulkhead를 쓰면 `SecurityContext`, MDC 같은 스레드 로컬 데이터가 전파되지 않는다. `ContextPropagator`를 직접 설정해야 한다.

```java
@Bean
public ContextPropagator mdcContextPropagator() {
    return new ContextPropagator() {
        @Override
        public Supplier<Optional<Object>> retrieve() {
            return () -> Optional.of(MDC.getCopyOfContextMap());
        }

        @Override
        public Consumer<Optional<Object>> copy() {
            return context -> context.ifPresent(c ->
                MDC.setContextMap((Map<String, String>) c));
        }

        @Override
        public Consumer<Optional<Object>> clear() {
            return context -> MDC.clear();
        }
    };
}
```

실무에서는 대부분 Semaphore Bulkhead를 쓴다. Thread Pool Bulkhead는 컨텍스트 전파 문제 때문에 디버깅이 어렵고, 반환 타입이 `CompletableFuture`여야 해서 기존 코드를 바꿔야 한다. 완전한 격리가 꼭 필요한 경우가 아니면 Semaphore로 충분하다.

### Bulkhead + Circuit Breaker 조합

```java
@Bulkhead(name = "payment-service", fallbackMethod = "fallback")
@CircuitBreaker(name = "payment-service", fallbackMethod = "fallback")
public PaymentResponse processPayment(PaymentRequest request) {
    return paymentClient.process(request);
}
```

어노테이션 순서가 아니라 Resilience4j의 내부 실행 순서가 적용된다. 기본 순서는:

```
Retry → CircuitBreaker → RateLimiter → TimeLimiter → Bulkhead
```

바깥(Retry)에서 안쪽(Bulkhead)으로 감싸는 구조다. 실제 실행은 안쪽부터다. Bulkhead에서 동시 호출 수를 확인하고, 통과하면 Circuit Breaker에서 상태를 확인하는 순서로 동작한다.

이 순서를 바꾸고 싶으면 설정으로 제어한다.

```yaml
resilience4j:
  circuitbreaker:
    circuit-breaker-aspect-order: 1
  bulkhead:
    bulkhead-aspect-order: 2
```

숫자가 낮을수록 바깥에 위치한다.

## Rate Limiter

### 개념

일정 시간 동안 허용하는 호출 횟수를 제한한다. Bulkhead가 "동시에 몇 개"를 제한한다면, Rate Limiter는 "시간당 몇 개"를 제한한다.

### 사용하는 경우

- 외부 API에 호출 제한이 걸려 있을 때 (SMS, 결제 등)
- 특정 서비스에 대한 트래픽을 의도적으로 제한할 때
- 복구 중인 서비스에 트래픽을 서서히 풀어줄 때

### Resilience4j 설정

```yaml
resilience4j:
  ratelimiter:
    instances:
      sms-api:
        limit-for-period: 100
        limit-refresh-period: 1s
        timeout-duration: 500ms
```

1초에 100건까지만 허용한다. 100건을 넘으면 500ms까지 대기한 후 `RequestNotPermitted` 예외를 던진다.

```java
@RateLimiter(name = "sms-api", fallbackMethod = "smsFallback")
public void sendSms(String phone, String message) {
    smsClient.send(phone, message);
}

private void smsFallback(String phone, String message, RequestNotPermitted e) {
    log.warn("SMS rate limit exceeded, phone={}", phone);
    // 메시지 큐에 넣어서 나중에 발송
    smsQueue.enqueue(phone, message);
}
```

Rate Limiter의 fallback에서는 큐에 넣어서 나중에 처리하는 패턴이 실용적이다. 사용자에게 "잠시 후 재시도"를 안내하는 것보다 낫다.

### Rate Limiter vs API Gateway Rate Limiting

API Gateway에서도 rate limiting을 건다. 둘의 역할이 다르다.

| 구분 | API Gateway Rate Limiting | Resilience4j Rate Limiter |
|------|---------------------------|---------------------------|
| 위치 | 요청 진입점 | 서비스 내부 |
| 대상 | 클라이언트(IP, API Key 단위) | 호출 대상 서비스 단위 |
| 목적 | 외부 트래픽 제어 | 내부 서비스 간 트래픽 제어 |

Gateway는 "우리 시스템에 들어오는 트래픽을 제한"하는 것이고, Resilience4j Rate Limiter는 "우리가 보내는 트래픽을 제한"하는 것이다. 외부 SMS API에 초당 100건 제한이 있으면 Gateway가 아니라 Rate Limiter로 제어해야 한다.

## Retry

### 개념

실패한 호출을 다시 시도한다. 네트워크 일시 오류, 서비스 순간 부하 같은 일시적 문제에 대응한다.

단순하지만 잘못 쓰면 장애를 키운다. 앞서 설명한 retry storm이 대표적이다.

### Resilience4j 설정

```yaml
resilience4j:
  retry:
    instances:
      payment-service:
        max-attempts: 3
        wait-duration: 500ms
        enable-exponential-backoff: true
        exponential-backoff-multiplier: 2
        retry-exceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
        ignore-exceptions:
          - com.example.exception.BusinessException
          - com.example.exception.ClientException
```

### exponential backoff

재시도 간격을 점점 늘리는 방식이다. 1차 500ms, 2차 1000ms, 3차 2000ms. 서버가 힘든 상황에서 재시도 간격이 일정하면 부하가 줄지 않는다.

여기에 **jitter**를 추가하면 더 좋다. 여러 클라이언트가 동시에 실패하면 exponential backoff만으로는 같은 시점에 재시도가 몰린다. 랜덤 지연을 섞으면 재시도가 분산된다.

```yaml
resilience4j:
  retry:
    instances:
      payment-service:
        max-attempts: 3
        wait-duration: 500ms
        enable-exponential-backoff: true
        exponential-backoff-multiplier: 2
        enable-randomized-wait: true
        randomized-wait-factor: 0.5
```

### 재시도하면 안 되는 경우

모든 실패를 재시도하면 안 된다.

- **4xx 응답**: 클라이언트 잘못이니 다시 보내도 같은 결과다. 400, 404, 409 같은 에러는 재시도 대상이 아니다.
- **비멱등 연산**: POST로 결제를 요청했는데 타임아웃이 났다. 결제가 된 건지 안 된 건지 모르는 상태에서 재시도하면 이중 결제가 발생한다.
- **서비스 전체 장애**: 서비스가 완전히 죽은 상태에서 재시도는 의미 없다. Circuit Breaker가 먼저 차단해야 한다.

```java
@Retry(name = "payment-service", fallbackMethod = "retryFallback")
public PaymentResponse processPayment(PaymentRequest request) {
    return paymentClient.process(request);
}
```

`ignore-exceptions`에 `ClientException`을 넣어서 4xx 응답에는 재시도하지 않게 한다. 서비스 간 통신에서 FeignClient를 쓰면 ErrorDecoder에서 HTTP 상태 코드별로 예외를 나누고, 재시도 대상 여부를 구분한다.

### Retry + Circuit Breaker 조합 주의점

Retry가 Circuit Breaker 바깥에 있다(기본 순서). Circuit Breaker가 OPEN일 때 Retry가 감싸고 있으면 `CallNotPermittedException`에 대해 재시도를 시도할 수 있다. OPEN 상태에서 재시도는 의미 없으니 이 예외를 `ignore-exceptions`에 추가한다.

```yaml
resilience4j:
  retry:
    instances:
      payment-service:
        ignore-exceptions:
          - io.github.resilience4j.circuitbreaker.CallNotPermittedException
```

## Timeout

### 개념

호출에 대한 최대 대기 시간을 설정한다. 장애 격리의 가장 기본이다. 타임아웃이 없으면 느린 서비스 때문에 스레드가 무한정 대기하고, 앞서 설명한 시나리오 1이 발생한다.

### 타임아웃이 걸리는 위치

MSA에서 HTTP 호출 하나에 타임아웃이 걸리는 지점이 여러 곳이다.

```
HTTP Client (Feign/RestClient) → Resilience4j TimeLimiter → Circuit Breaker slow call threshold
```

각각의 타임아웃 관계를 이해해야 한다.

```yaml
# Feign 타임아웃
spring:
  cloud:
    openfeign:
      client:
        config:
          payment-service:
            connect-timeout: 1000    # 연결 타임아웃 1초
            read-timeout: 3000       # 읽기 타임아웃 3초

# Resilience4j TimeLimiter
resilience4j:
  timelimiter:
    instances:
      payment-service:
        timeout-duration: 4s
        cancel-running-future: true

# Circuit Breaker slow call 판단
resilience4j:
  circuitbreaker:
    instances:
      payment-service:
        slow-call-duration-threshold: 3s
```

**Feign 타임아웃 < TimeLimiter 타임아웃**이어야 한다. Feign에서 3초에 끊기는데 TimeLimiter가 2초면, TimeLimiter가 먼저 끊어버려서 Feign 타임아웃이 의미 없다. 반대로 Feign이 5초인데 TimeLimiter가 4초면, TimeLimiter가 4초에 끊고 Feign 커넥션은 백그라운드에서 계속 대기한다. `cancel-running-future: true`를 설정해야 실제로 취소된다.

`slow-call-duration-threshold`는 타임아웃과 별개다. "3초 넘으면 느린 호출로 판단하고, 느린 호출 비율이 높으면 circuit을 연다"는 의미다. 타임아웃에 걸리지 않더라도 느린 호출이 누적되면 circuit이 열린다.

### 타임아웃 설정 기준

타임아웃을 몇 초로 잡을지는 서비스의 정상 응답 시간을 기준으로 한다.

- p99 응답 시간이 500ms인 서비스라면 타임아웃은 1~2초가 적당하다.
- p99가 2초인 외부 API라면 3~5초.
- 30초, 60초 같은 타임아웃은 거의 의미가 없다. 스레드가 30초 동안 묶이면 이미 장애다.

타임아웃은 짧게 잡고, 실패는 빠르게 처리하는 것이 원칙이다. "혹시 성공할 수 있으니 길게 기다리자"는 접근이 장애를 키운다.

## 패턴 조합: 실전 설정

### 결제 서비스 호출에 적용한 설정

```yaml
resilience4j:
  circuitbreaker:
    instances:
      payment-service:
        sliding-window-type: COUNT_BASED
        sliding-window-size: 10
        failure-rate-threshold: 50
        slow-call-duration-threshold: 3s
        slow-call-rate-threshold: 80
        wait-duration-in-open-state: 15s
        permitted-number-of-calls-in-half-open-state: 3
        ignore-exceptions:
          - com.example.exception.BusinessException

  bulkhead:
    instances:
      payment-service:
        max-concurrent-calls: 30
        max-wait-duration: 500ms

  retry:
    instances:
      payment-service:
        max-attempts: 2
        wait-duration: 500ms
        enable-exponential-backoff: true
        exponential-backoff-multiplier: 2
        retry-exceptions:
          - java.io.IOException
        ignore-exceptions:
          - com.example.exception.BusinessException
          - com.example.exception.ClientException
          - io.github.resilience4j.circuitbreaker.CallNotPermittedException

  ratelimiter:
    instances:
      payment-service:
        limit-for-period: 200
        limit-refresh-period: 1s
        timeout-duration: 0
```

결제 서비스는 재시도를 2회로 제한한다. 결제는 멱등 키 없이 무작정 재시도하면 이중 결제가 나기 때문이다. `retry-exceptions`에 IOException만 넣어서 네트워크 오류일 때만 재시도한다. `rate-limiter`의 `timeout-duration: 0`은 한도를 넘기면 바로 실패 처리한다는 뜻이다.

### 외부 알림 API 호출에 적용한 설정

```yaml
resilience4j:
  circuitbreaker:
    instances:
      notification-api:
        sliding-window-type: TIME_BASED
        sliding-window-size: 30
        failure-rate-threshold: 60
        wait-duration-in-open-state: 30s
        permitted-number-of-calls-in-half-open-state: 5

  ratelimiter:
    instances:
      notification-api:
        limit-for-period: 50
        limit-refresh-period: 1s
        timeout-duration: 2s

  retry:
    instances:
      notification-api:
        max-attempts: 3
        wait-duration: 1s
        enable-exponential-backoff: true
        exponential-backoff-multiplier: 2
        enable-randomized-wait: true
```

알림은 핵심 기능이 아니니까 circuit breaker 임계치를 느슨하게 잡는다. Rate limiter로 외부 API 호출 한도를 지키고, 재시도는 3회까지 한다. `TIME_BASED`를 쓴 이유는 알림 트래픽이 이벤트성으로 불규칙하기 때문이다.

## 모니터링

장애 격리 패턴을 적용했으면 상태를 모니터링해야 한다. Circuit이 열렸는지, Bulkhead가 포화 상태인지 모르면 적용 안 한 것과 다를 바 없다.

### Actuator + Prometheus 연동

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health, prometheus, circuitbreakers, ratelimiters
  health:
    circuitbreakers:
      enabled: true
    ratelimiters:
      enabled: true
```

```
# Prometheus 메트릭 예시
resilience4j_circuitbreaker_state{name="payment-service"} 0  # 0=CLOSED, 1=OPEN, 2=HALF_OPEN
resilience4j_circuitbreaker_failure_rate{name="payment-service"} 20.0
resilience4j_bulkhead_available_concurrent_calls{name="payment-service"} 15
resilience4j_ratelimiter_available_permissions{name="sms-api"} 80
```

### 알림 설정

최소한 이 항목들은 알림을 걸어둬야 한다.

- Circuit Breaker 상태가 OPEN으로 변경될 때
- Bulkhead 가용 슬롯이 0에 가까울 때
- Rate Limiter에서 거부된 요청 비율이 높을 때
- Retry 횟수가 급증할 때

Circuit이 열리는 것 자체는 정상 동작이다. 열린 circuit이 CLOSED로 복구되지 않는 것이 문제다.

## 실무에서 겪은 문제와 해결

### 문제 1. Circuit이 너무 쉽게 열린다

배포 직후에 서비스가 워밍업되면서 처음 몇 건이 느리게 응답한다. `sliding-window-size: 5`로 설정했더니 5건 중 3건이 slow call이라 바로 circuit이 열렸다.

해결: window size를 10 이상으로 늘리고, `minimum-number-of-calls` 설정을 추가했다.

```yaml
resilience4j:
  circuitbreaker:
    instances:
      payment-service:
        sliding-window-size: 20
        minimum-number-of-calls: 10
```

`minimum-number-of-calls: 10`이면 최소 10건이 쌓여야 실패율 계산을 시작한다. 배포 직후 소수의 느린 응답으로 circuit이 열리는 것을 방지한다.

### 문제 2. fallback이 원래 로직보다 무거움

Circuit이 열렸을 때 fallback에서 캐시를 조회하도록 했는데, 캐시 서버도 장애 상태였다. fallback에서 다시 타임아웃이 걸렸다.

해결: fallback은 가능한 한 로컬에서 처리한다. 외부 호출이 들어가는 fallback은 그 호출에 대해서도 타임아웃과 circuit breaker를 설정한다. fallback에서 예외가 나면 최종 에러 응답을 내려야 한다.

### 문제 3. 재시도가 이중 결제를 발생시킴

결제 API 호출 후 타임아웃이 걸려서 재시도했는데, 실제로는 첫 번째 요청이 처리된 상태였다. 이중 결제가 발생했다.

해결: 결제 같은 비멱등 연산은 **멱등 키(idempotency key)**를 반드시 사용한다. 같은 키로 재요청하면 기존 결과를 반환하도록 서버 측에서 처리한다.

```java
public PaymentResponse processPayment(PaymentRequest request) {
    String idempotencyKey = request.getOrderId() + "-" + request.getAttemptId();
    return paymentClient.process(request, idempotencyKey);
}
```

### 문제 4. Bulkhead 제한에 걸려 정상 요청이 실패

`max-concurrent-calls: 10`으로 설정했는데, 피크 타임에 동시 요청이 15건이 되면서 5건이 거부됐다.

해결: 실제 트래픽 패턴을 보고 설정해야 한다. 평소 동시 호출이 10건인데 피크 때 20건이면 `max-concurrent-calls: 25` 정도로 여유를 둔다. 너무 낮으면 정상 트래픽이 거부되고, 너무 높으면 격리 효과가 없다.
