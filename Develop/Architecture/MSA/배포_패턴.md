# 배포 패턴

서비스를 운영 중인 상태에서 새 버전을 내보내는 건 단순히 코드를 교체하는 일이 아니다. 트래픽을 어떻게 전환하고, 문제가 생겼을 때 얼마나 빨리 되돌릴 수 있느냐가 배포 패턴을 선택하는 기준이 된다.

---

## Blue-Green 배포

### 동작 방식

운영 환경을 두 벌 준비한다. 현재 트래픽을 받고 있는 쪽이 Blue, 새 버전을 배포하는 쪽이 Green이다.

```
[로드밸런서] ──→ [Blue (v1.0)] ← 현재 트래픽
                [Green (v1.1)] ← 새 버전 배포 완료, 대기 중
```

Green에 새 버전을 배포하고 내부 테스트를 마친 뒤, 로드밸런서가 바라보는 대상을 Green으로 바꾼다. 전환은 한 번에 일어난다.

```yaml
# Kubernetes에서 Service의 selector를 바꾸는 방식
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  selector:
    app: my-app
    version: green   # blue → green으로 변경
  ports:
    - port: 80
      targetPort: 8080
```

### 롤백

로드밸런서를 다시 Blue로 돌리면 된다. Blue 환경이 그대로 살아있기 때문에 롤백 속도가 가장 빠르다. 보통 수 초 이내.

문제는 Green에서 장애를 감지하기까지의 시간이다. 전환 직후 모든 트래픽이 Green으로 가기 때문에, 장애가 발생하면 전체 사용자에게 영향이 간다. 전환 후 5~10분 동안은 에러율, 응답 시간, 비즈니스 메트릭을 집중적으로 모니터링해야 한다.

### 비용 문제

서버를 두 벌 유지해야 하니 비용이 두 배다. 실제 운영에서는 배포 시에만 Green 환경을 띄우고, 안정화 후 Blue를 내리는 방식으로 비용을 줄인다. 하지만 이렇게 하면 빠른 롤백이 불가능해지니까 트레이드오프가 있다.

## Canary 배포

### 동작 방식

새 버전을 전체에 한 번에 내보내지 않고, 일부 트래픽만 먼저 새 버전으로 보낸다. 문제가 없으면 점진적으로 비율을 올린다.

```
[로드밸런서]
   ├── 95% → [v1.0] (기존)
   └──  5% → [v1.1] (canary)
```

Kubernetes 환경에서는 Istio VirtualService로 트래픽 비율을 조절한다.

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app
spec:
  hosts:
    - my-app
  http:
    - route:
        - destination:
            host: my-app
            subset: stable
          weight: 95
        - destination:
            host: my-app
            subset: canary
          weight: 5
```

보통 5% → 25% → 50% → 100% 같은 단계를 거친다. 각 단계에서 에러율, 지연 시간 등을 확인하고 다음 단계로 넘어간다.

### 자동 롤백 조건

수동으로 모니터링하면 놓치는 경우가 많다. Argo Rollouts 같은 도구를 쓰면 메트릭 기반으로 자동 판단한다.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
        - setWeight: 5
        - pause: { duration: 5m }
        - setWeight: 25
        - pause: { duration: 5m }
        - setWeight: 50
        - pause: { duration: 10m }
      analysis:
        templates:
          - templateName: error-rate
        startingStep: 1
        args:
          - name: service-name
            value: my-app
---
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: error-rate
spec:
  metrics:
    - name: error-rate
      interval: 60s
      successCondition: result[0] < 0.05
      provider:
        prometheus:
          address: http://prometheus:9090
          query: |
            sum(rate(http_requests_total{service="{{args.service-name}}", status=~"5.."}[2m]))
            /
            sum(rate(http_requests_total{service="{{args.service-name}}"}[2m]))
```

에러율이 5%를 넘으면 자동으로 canary 트래픽을 0%로 되돌린다.

### 주의할 점

Canary로 보내는 트래픽이 특정 사용자 그룹에 편중되면 의미가 없다. 예를 들어 결제 API의 canary가 내부 테스트 계정에만 걸리면, 실제 사용자 트래픽에서 발생하는 문제를 못 잡는다. 트래픽 분배가 무작위인지 확인해야 한다.

반대로, 특정 사용자 그룹에게만 canary를 적용하고 싶은 경우도 있다. 쿠키나 헤더 기반으로 라우팅하면 내부 QA 팀이 먼저 새 버전을 쓰게 할 수 있다.

```yaml
# 특정 헤더가 있으면 canary로 라우팅
http:
  - match:
      - headers:
          x-canary:
            exact: "true"
    route:
      - destination:
          host: my-app
          subset: canary
  - route:
      - destination:
          host: my-app
          subset: stable
```

## Rolling 배포

### 동작 방식

기존 인스턴스를 하나씩(또는 일정 개수씩) 새 버전으로 교체한다. Kubernetes의 기본 배포 방식이다.

```
시작:  [v1] [v1] [v1] [v1]
1단계: [v2] [v1] [v1] [v1]
2단계: [v2] [v2] [v1] [v1]
3단계: [v2] [v2] [v2] [v1]
완료:  [v2] [v2] [v2] [v2]
```

```yaml
apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 최대 1개 추가 생성 허용
      maxUnavailable: 1  # 최대 1개 중단 허용
```

`maxSurge`와 `maxUnavailable` 설정이 중요하다. `maxSurge: 0, maxUnavailable: 1`이면 기존 pod 하나를 먼저 내리고 새 pod를 올린다. 이 순간 처리 가능한 인스턴스가 3개로 줄어든다. 트래픽이 많은 시간대라면 `maxSurge: 1, maxUnavailable: 0`으로 설정해서 항상 4개 이상 유지되게 해야 한다.

### 롤백의 어려움

Rolling 배포 중간에 문제가 생기면 v1과 v2가 섞여있는 상태다. `kubectl rollout undo`를 실행해도 다시 Rolling 방식으로 되돌아가기 때문에 완전한 롤백까지 시간이 걸린다.

```bash
# 롤백 실행
kubectl rollout undo deployment/my-app

# 롤백 상태 확인
kubectl rollout status deployment/my-app
```

Rolling 중 v1과 v2가 동시에 요청을 처리하는 구간이 생긴다. API 스펙이 바뀌었다면 이 과도기에 호환성 문제가 발생한다. 새 버전에서 추가된 필드를 기존 버전이 처리하지 못하거나, 기존 버전에서 보낸 요청을 새 버전이 다르게 해석하는 경우가 실제로 자주 일어난다.

## 데이터베이스 마이그레이션과 배포가 엮일 때

배포 패턴과 관계없이, DB 스키마 변경이 포함된 배포는 항상 까다롭다.

### 컬럼 추가

NOT NULL 컬럼을 추가하면서 동시에 새 버전을 배포하면, 기존 버전 코드가 해당 컬럼 없이 INSERT를 시도해서 에러가 난다. Rolling이나 Canary처럼 구/신 버전이 공존하는 배포에서 특히 문제가 된다.

해결 방법은 배포를 두 단계로 나누는 것이다.

```
1단계: 컬럼을 nullable로 추가 + 마이그레이션 실행
       (기존 코드는 이 컬럼을 모른다 → 문제 없음)

2단계: 새 코드 배포 (새 컬럼 사용)
       (이제 기존 코드가 없으니 NOT NULL 제약 추가 가능)
```

### 컬럼 삭제

삭제도 마찬가지다. 코드에서 해당 컬럼을 사용하지 않도록 먼저 배포하고, 그 다음에 컬럼을 삭제한다.

```
1단계: 해당 컬럼을 사용하지 않는 코드 배포
2단계: 컬럼 삭제 마이그레이션 실행
```

순서를 뒤집으면 기존 코드가 삭제된 컬럼을 참조해서 장애가 난다.

### 테이블 변경과 락

MySQL에서 대용량 테이블에 `ALTER TABLE`을 실행하면 테이블 락이 걸려 서비스가 멈출 수 있다. pt-online-schema-change나 gh-ost 같은 도구를 써서 온라인 DDL을 수행한다.

```bash
# gh-ost 예시: users 테이블에 email 컬럼 추가
gh-ost \
  --host=db-master \
  --database=mydb \
  --table=users \
  --alter="ADD COLUMN email VARCHAR(255)" \
  --execute
```

이런 도구들은 원본 테이블의 복사본을 만들고, 변경사항을 binlog로 동기화한 뒤, 이름을 바꿔치는 방식으로 동작한다. 배포 타이밍과 마이그레이션 완료 시점을 맞추는 게 관건이다.

## Feature Flag

배포와 릴리스를 분리하는 방법이다. 코드는 배포하되, 기능은 Flag로 끄고 켠다.

### 기본 구조

```java
@RestController
public class OrderController {

    private final FeatureFlagService featureFlags;

    @PostMapping("/orders")
    public ResponseEntity<?> createOrder(@RequestBody OrderRequest request) {
        if (featureFlags.isEnabled("new-payment-flow", request.getUserId())) {
            return newPaymentService.process(request);
        }
        return legacyPaymentService.process(request);
    }
}
```

```java
@Service
public class FeatureFlagService {

    private final FeatureFlagRepository repository;
    // 캐시를 쓰되 TTL을 짧게 잡는다. 긴급 롤백 시 Flag를 끄면 바로 반영돼야 하기 때문.
    private final LoadingCache<String, FeatureFlag> cache;

    public FeatureFlagService(FeatureFlagRepository repository) {
        this.repository = repository;
        this.cache = Caffeine.newBuilder()
            .expireAfterWrite(Duration.ofSeconds(30))
            .build(repository::findByKey);
    }

    public boolean isEnabled(String flagKey, Long userId) {
        FeatureFlag flag = cache.get(flagKey);
        if (flag == null) return false;

        // 전체 ON/OFF
        if (flag.isGloballyEnabled()) return true;
        if (flag.isGloballyDisabled()) return false;

        // 특정 사용자 비율로 점진적 활성화
        return userId % 100 < flag.getRolloutPercentage();
    }
}
```

### Feature Flag로 배포 위험을 줄이는 흐름

```
1. Flag OFF 상태로 새 코드 배포 (아무 영향 없음)
2. 내부 테스트 계정에만 Flag ON
3. 문제 없으면 전체 사용자 1% → 10% → 50% → 100%
4. 안정화 확인 후 Flag 분기 코드 제거 (다음 배포에서)
```

4번을 빼먹는 경우가 많다. Flag 분기가 쌓이면 코드가 점점 읽기 어려워진다. Flag를 만들 때 "제거 예정일"을 같이 기록해두는 게 좋다.

### Flag 저장소

- DB에 넣으면 관리는 편하지만, 서비스가 DB에 의존하게 된다. DB가 죽으면 Flag 조회도 안 된다.
- 설정 파일에 넣으면 변경할 때마다 재배포해야 한다. Flag의 의미가 없어진다.
- LaunchDarkly, Unleash 같은 전용 서비스를 쓰면 SDK가 로컬 캐시를 유지해서 외부 장애에도 동작한다.

## 트래픽 전환 시 세션과 커넥션 처리

배포 패턴이 뭐든, 기존 인스턴스를 내릴 때 진행 중인 요청을 어떻게 처리하느냐가 문제다.

### Graceful Shutdown

SIGTERM을 받으면 새 요청 수락을 중단하고, 처리 중인 요청이 끝날 때까지 기다린 후 종료한다.

```java
@Configuration
public class GracefulShutdownConfig {

    @Bean
    public ServletWebServerFactory webServerFactory() {
        TomcatServletWebServerFactory factory = new TomcatServletWebServerFactory();
        factory.addConnectorCustomizers(connector -> {
            // SIGTERM 시 새 연결을 받지 않고 기존 요청 완료 대기
        });
        return factory;
    }
}
```

Spring Boot 2.3 이상에서는 설정으로 처리한다.

```yaml
# application.yml
server:
  shutdown: graceful

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

Kubernetes에서는 `terminationGracePeriodSeconds`와 맞춰야 한다.

```yaml
spec:
  terminationGracePeriodSeconds: 40  # Spring의 30s보다 넉넉하게
  containers:
    - name: my-app
      lifecycle:
        preStop:
          exec:
            command: ["sh", "-c", "sleep 5"]  # 로드밸런서가 이 pod를 빼는 시간 확보
```

`preStop`의 `sleep 5`가 없으면, SIGTERM과 동시에 로드밸런서에서 빠지기 전에 pod가 종료를 시작해서 요청이 끊길 수 있다. Endpoints 업데이트가 전파되는 데 시간이 걸리기 때문이다.

### 세션 처리

서버 메모리에 세션을 저장하는 구조에서는 배포할 때마다 세션이 날아간다. 사용자가 갑자기 로그아웃되거나, 장바구니가 비거나 하는 문제가 생긴다.

세션을 외부 저장소(Redis 등)로 빼면 해결된다.

```yaml
# Spring Session + Redis
spring:
  session:
    store-type: redis
    redis:
      namespace: myapp:session
    timeout: 1800
```

세션 직렬화 형식이 바뀌는 배포에서는 주의가 필요하다. 새 버전에서 세션 객체에 필드를 추가했는데, 기존 세션을 역직렬화하다가 에러가 나는 경우가 있다. JSON 기반 직렬화를 쓰면 필드 추가에 대해 좀 더 안전하다.

### WebSocket이나 SSE 같은 긴 연결

HTTP 요청은 수 초 내에 끝나지만, WebSocket은 수 시간 유지될 수 있다. Graceful shutdown의 타임아웃(30초 등)이 지나면 강제로 끊긴다.

클라이언트 쪽에서 재연결 로직을 반드시 구현해야 한다.

```javascript
function connectWebSocket() {
    const ws = new WebSocket('wss://api.example.com/ws');

    ws.onclose = function(event) {
        // 서버 배포로 인한 종료일 수 있으니 재연결 시도
        setTimeout(() => connectWebSocket(), 1000 + Math.random() * 2000);
    };
}
```

재연결 시 모든 클라이언트가 동시에 붙으면 thundering herd 문제가 생긴다. `Math.random()`으로 jitter를 줘서 분산시킨다.

### 데이터베이스 커넥션 풀

새 인스턴스가 올라오면서 커넥션 풀을 한꺼번에 생성하면 DB에 순간적으로 부하가 몰린다. Rolling 배포에서 여러 pod가 동시에 재시작되면 이 문제가 커진다.

HikariCP의 경우 `minimumIdle` 설정으로 초기 커넥션 수를 조절한다.

```yaml
spring:
  datasource:
    hikari:
      minimum-idle: 2           # 시작 시 2개만 생성
      maximum-pool-size: 20     # 필요할 때 최대 20개까지
      idle-timeout: 300000
```

## 어떤 패턴을 쓸 것인가

| 기준 | Blue-Green | Canary | Rolling |
|------|-----------|--------|---------|
| 롤백 속도 | 즉시 (로드밸런서 전환) | 빠름 (canary 트래픽 제거) | 느림 (다시 Rolling) |
| 인프라 비용 | 높음 (두 벌) | 중간 | 낮음 |
| 위험 노출 범위 | 전환 시 전체 | 일부 트래픽만 | 점진적으로 확대 |
| 구/신 버전 공존 | 전환 순간만 | 공존 기간 있음 | 공존 기간 있음 |
| 구현 난이도 | 낮음 | 중간~높음 (메트릭 연동) | 낮음 (K8s 기본) |

실무에서는 Canary + Feature Flag 조합을 많이 쓴다. Canary로 인프라 수준의 문제(메모리 릭, CPU 스파이크)를 잡고, Feature Flag로 비즈니스 로직 수준의 문제를 제어한다. 두 레이어를 분리해서 관리하면 배포 사고 확률이 확실히 줄어든다.
