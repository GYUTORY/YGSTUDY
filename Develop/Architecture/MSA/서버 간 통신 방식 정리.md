# 서버 간 통신 방식 실전 정리

## 개요

서버 간 통신 방식은 시스템 구조를 사실상 결정한다. 초기에는 단순해 보이지만, 트래픽 증가나 장애가 발생하면 선택의 결과가 그대로 드러난다.

**통신 방식의 중요성:**
- 시스템의 성능과 확장성을 결정한다
- 장애 전파 범위를 결정한다
- 개발과 운영 복잡도를 결정한다

**실무 현실:**
실무에서는 단일 방식으로 끝나는 경우가 거의 없다. HTTP로 시작하고, 병목이 생긴 구간만 메시지나 RPC로 분리하는 흐름이 일반적이다.

## 통신 방식 개요

### 통신 방식의 분류

서버 간 통신은 크게 4가지로 나뉜다.

**1. 동기 요청-응답:**
- HTTP/REST: 가장 일반적인 방식
- RPC/gRPC: 내부 서비스 간 고성능 통신
- 요청을 보내고 응답을 기다린다

**2. 비동기 메시징:**
- Queue: 점대점 메시지 전달
- Pub-Sub: 발행-구독 패턴
- 요청을 보내고 즉시 응답한다

**3. 실시간 양방향:**
- WebSocket: 양방향 통신
- SSE: 서버에서 클라이언트로 푸시
- 지속적인 연결을 유지한다

**4. 스트리밍:**
- gRPC Streaming: 단방향/양방향 스트림
- Kafka: 이벤트 스트림
- 연속적인 데이터 전송

## HTTP 기반 통신

### HTTP 통신의 특징

HTTP는 가장 널리 사용되는 통신 방식이다. 내부 통신이든 외부 연동이든 기본값에 가깝다.

**HTTP의 특징:**
- 요청-응답 모델: 클라이언트가 요청을 보내고 서버가 응답한다
- 상태 비저장(Stateless): 각 요청은 독립적이다
- 텍스트 기반: 사람이 읽을 수 있는 형식이다

**HTTP 메서드:**
- GET: 조회
- POST: 생성
- PUT: 전체 수정
- PATCH: 부분 수정
- DELETE: 삭제

### REST API

REST는 HTTP를 활용한 아키텍처 스타일이다.

**REST 원칙:**
- 리소스 중심: URL로 리소스를 표현한다
- HTTP 메서드로 동작을 표현한다
- 상태 코드로 결과를 표현한다

**사용되는 상황:**
- 인증 서버 호출
- 주문, 결제, 회원 API
- 외부 파트너 연동
- 브라우저에서 직접 호출 가능한 API

**예시:**
```
POST /internal/orders HTTP/1.1
Content-Type: application/json

{
  "userId": "user-123",
  "items": [
    { "productId": "p1", "quantity": 2 }
  ]
}
```

**응답:**
```
HTTP/1.1 201 Created
Content-Type: application/json

{
  "orderId": "order-456",
  "status": "CREATED"
}
```

### HTTP 통신의 동작 원리

**1. 요청 전송:**
- 클라이언트가 HTTP 요청을 생성한다
- 요청 헤더와 바디를 포함한다
- TCP 연결을 통해 전송한다

**2. 서버 처리:**
- 서버가 요청을 받아 파싱한다
- 비즈니스 로직을 실행한다
- 응답을 생성한다

**3. 응답 수신:**
- 서버가 HTTP 응답을 전송한다
- 클라이언트가 응답을 받아 처리한다
- 연결을 종료한다 (HTTP/1.1 Keep-Alive 제외)

**동기 통신:**
HTTP는 기본적으로 동기 통신이다. 요청을 보내고 응답을 기다린다. 응답이 올 때까지 블로킹된다.

### HTTP 통신의 장점

**범용성:**
- 모든 언어와 플랫폼에서 지원한다
- 브라우저에서 직접 호출 가능하다
- 외부 연동에 적합하다

**가독성:**
- 텍스트 기반이라 사람이 읽을 수 있다
- 로그 분석이 쉽다
- 디버깅이 편하다

**표준화:**
- HTTP 표준을 따르므로 호환성이 좋다
- 다양한 도구와 라이브러리가 있다
- 학습 곡선이 낮다

### HTTP 통신의 단점

**성능:**
- 텍스트 기반이라 직렬화 비용이 있다
- HTTP 헤더 오버헤드가 있다
- gRPC보다 느리다

**장애 전파:**
- 동기 통신이라 한 서비스가 느려지면 전체가 느려진다
- 호출 체인이 길어질수록 지연이 누적된다
- 한 서비스 장애가 다른 서비스 장애로 번진다

**타임아웃 관리:**
- 타임아웃 기준이 제각각이면 장애 원인 추적이 어렵다
- 너무 짧으면 정상 요청도 실패한다
- 너무 길면 장애가 오래 끌어진다

### 실무에서 자주 발생하는 문제

**1. 호출 체인 지연 누적:**

**문제:**
호출 체인이 길어질수록 지연이 누적된다.

**예시:**
- 주문 API → 결제 서비스 (100ms)
- 결제 서비스 → 재고 서비스 (150ms)
- 재고 서비스 → 배송 서비스 (200ms)
- 총 지연: 450ms

**해결:**
- 호출 체인을 줄인다
- 병렬 호출을 사용한다
- 비동기로 전환한다

**2. 장애 전파:**

**문제:**
한 서비스 장애가 다른 서비스 장애로 번진다.

**예시:**
- 결제 서비스가 느려진다
- 주문 서비스가 결제 응답을 기다린다
- 주문 API 전체가 느려진다

**해결:**
- 타임아웃을 짧게 설정한다
- 회로 차단기(Circuit Breaker)를 사용한다
- 비동기로 전환한다

**3. 타임아웃 기준 불일치:**

**문제:**
타임아웃 기준이 제각각이면 장애 원인 추적이 어렵다.

**예시:**
- 주문 서비스 타임아웃: 5초
- 결제 서비스 타임아웃: 10초
- 재고 서비스 타임아웃: 3초
- 어디서 실패했는지 파악하기 어렵다

**해결:**
- 표준 타임아웃을 정한다
- 내부 통신은 짧게 설정한다 (1-3초)
- 타임아웃을 로그에 기록한다

### 운영 시 주의사항

**타임아웃 설정:**
- 내부 통신이라도 타임아웃은 짧게 둔다
- 평균 응답 시간의 3배 정도로 설정한다
- 타임아웃이 지나면 빠르게 실패한다

**재시도:**
- 재시도는 한 곳에서만 한다
- 여러 레이어에서 재시도하면 중복 요청이 발생한다
- 지수 백오프를 사용한다

**로깅:**
- 상태 변경 API는 반드시 로그를 남긴다
- 요청/응답을 로깅한다
- 상관관계 ID를 포함한다

**에러 처리:**
- HTTP 상태 코드를 적절히 사용한다
- 에러 응답에 상세 정보를 포함한다
- 클라이언트가 에러를 처리할 수 있게 한다

## RPC 기반 통신

### RPC의 개념

RPC(Remote Procedure Call)는 원격 프로시저 호출이다. 로컬 함수 호출처럼 원격 서비스를 호출한다.

**RPC의 특징:**
- 함수 호출처럼 사용한다
- 타입 안정성이 있다
- 성능이 좋다

**RPC vs HTTP:**
- RPC: 함수 호출에 가깝다, 타입이 명확하다
- HTTP: 리소스 조작에 가깝다, 유연하다

### gRPC

gRPC는 Google이 개발한 RPC 프레임워크다. 내부 서비스 간 호출 빈도가 높은 경우 선택된다.

**gRPC의 특징:**
- Protocol Buffers를 사용한다
- HTTP/2를 기반으로 한다
- 스트리밍을 지원한다

**사용하는 이유:**

**1. 타입 안정성:**
- 요청/응답 구조가 코드로 고정된다
- 컴파일 타임에 타입 체크가 가능하다
- 런타임 에러를 줄인다

**2. 성능:**
- JSON 직렬화 비용이 없다
- Protocol Buffers는 바이너리 형식이라 빠르다
- 대량 호출 시 HTTP 대비 차이가 체감된다

**3. 스트리밍:**
- 단방향/양방향 스트림을 지원한다
- 실시간 데이터 전송에 적합하다
- HTTP로는 구현하기 어렵다

### Protocol Buffers

Protocol Buffers는 구조화된 데이터를 직렬화하는 방법이다.

**proto 파일 예제:**
```proto
syntax = "proto3";

service OrderService {
  rpc CreateOrder (CreateOrderRequest) returns (CreateOrderResponse);
  rpc GetOrder (GetOrderRequest) returns (GetOrderResponse);
  rpc StreamOrders (StreamOrdersRequest) returns (stream Order);
}

message CreateOrderRequest {
  string userId = 1;
  repeated OrderItem items = 2;
}

message OrderItem {
  string productId = 1;
  int32 quantity = 2;
}

message CreateOrderResponse {
  string orderId = 1;
  string status = 2;
}
```

**proto의 특징:**
- 필드 번호로 식별한다
- 필드 이름이 바뀌어도 호환된다
- 필드 삭제는 위험하다

**버전 관리:**
- 필드 번호는 재사용하지 않는다
- 필드 삭제보다 deprecated 처리로 간다
- 새 필드는 새 번호를 사용한다

### gRPC의 동작 원리

**1. 코드 생성:**
- proto 파일에서 클라이언트/서버 코드를 생성한다
- 타입 안전한 인터페이스가 만들어진다

**2. 요청 전송:**
- 클라이언트가 생성된 코드로 서비스를 호출한다
- Protocol Buffers로 직렬화한다
- HTTP/2로 전송한다

**3. 서버 처리:**
- 서버가 요청을 받아 역직렬화한다
- 비즈니스 로직을 실행한다
- 응답을 직렬화해 전송한다

**4. 응답 수신:**
- 클라이언트가 응답을 받아 역직렬화한다
- 타입 안전한 객체로 반환한다

### gRPC의 장점

**성능:**
- Protocol Buffers는 JSON보다 작고 빠르다
- HTTP/2의 멀티플렉싱을 활용한다
- 헤더 압축으로 오버헤드를 줄인다

**타입 안정성:**
- 컴파일 타임에 타입 체크가 가능하다
- 인터페이스가 코드로 정의된다
- 런타임 에러를 줄인다

**스트리밍:**
- 단방향/양방향 스트림을 지원한다
- 실시간 데이터 전송에 적합하다
- 긴 연결을 유지할 수 있다

### gRPC의 단점

**배포 순서 의존성:**
- proto 변경 시 배포 순서가 꼬이면 바로 장애다
- 서버가 새 proto를 사용하면 클라이언트도 업데이트해야 한다
- 하위 호환성을 신중하게 관리해야 한다

**외부 연동 제한:**
- 외부 연동에는 거의 쓸 수 없다
- 브라우저에서 직접 호출하기 어렵다
- HTTP/REST가 더 적합하다

**가독성:**
- 로그가 사람이 읽기 어렵다
- 바이너리 형식이라 디버깅이 어렵다
- 장애 분석용 로그는 별도로 남긴다

**학습 곡선:**
- Protocol Buffers를 배워야 한다
- 코드 생성 과정을 이해해야 한다
- HTTP보다 복잡하다

### 실무에서 겪는 문제

**1. proto 변경 시 배포 순서:**

**문제:**
proto 변경 시 배포 순서가 꼬이면 바로 장애다.

**시나리오:**
- 서버가 새 proto를 먼저 배포한다
- 클라이언트가 이전 proto를 사용한다
- 필드가 맞지 않아 에러가 발생한다

**해결:**
- 하위 호환성을 유지한다
- 필드 삭제보다 deprecated 처리로 간다
- 점진적으로 마이그레이션한다

**2. 필드 삭제의 위험:**

**문제:**
필드 삭제는 위험하다. 다른 서비스가 사용 중일 수 있다.

**해결:**
- 필드 삭제보다 deprecated 처리로 간다
- 필드 번호를 재사용하지 않는다
- 충분한 기간 후에 삭제한다

**3. 로그 가독성:**

**문제:**
로그가 사람이 읽기 어렵다. 바이너리 형식이라 디버깅이 어렵다.

**해결:**
- 장애 분석용 로그는 별도로 남긴다
- JSON 형식으로 변환해 로깅한다
- 상관관계 ID를 포함한다

### 운영 시 주의사항

**외부 API:**
- 외부 API에는 사용하지 않는다
- 브라우저 호출이 필요한 경우 HTTP/REST를 사용한다
- 내부 서비스 간 통신에만 사용한다

**버전 관리:**
- 필드 삭제보다 deprecated 처리로 간다
- 필드 번호를 재사용하지 않는다
- 충분한 기간 후에 삭제한다

**로깅:**
- 장애 분석용 로그는 별도로 남긴다
- JSON 형식으로 변환해 로깅한다
- 요청/응답을 모두 로깅한다

**모니터링:**
- gRPC 메트릭을 모니터링한다
- 에러율과 지연 시간을 추적한다
- 스트림 연결 상태를 확인한다

## 메시지 기반 비동기 통신

### 메시징의 개념

메시징은 비동기 통신 방식이다. 요청을 보내고 즉시 응답을 받는다. 실제 처리는 나중에 비동기로 이루어진다.

**메시징의 특징:**
- 비동기: 요청을 보내고 기다리지 않는다
- 느슨한 결합: 발행자와 소비자가 독립적이다
- 확장성: 소비자를 늘려 처리량을 확보할 수 있다

**메시징의 목적:**
동기 호출 부담을 줄이기 위해 가장 많이 도입된다. 장애 전파를 막고 처리량을 늘린다.

### Queue (점대점 메시징)

Queue는 하나의 메시지를 하나의 소비자가 처리한다.

**동작 원리:**
1. Producer가 메시지를 큐에 전송한다
2. Consumer가 큐에서 메시지를 가져온다
3. Consumer가 메시지를 처리한다
4. 처리 완료 후 메시지를 삭제한다

**특징:**
- 하나의 메시지는 하나의 Consumer만 처리한다
- 여러 Consumer가 있어도 메시지는 분산된다
- 작업 큐(Job Queue) 패턴에 적합하다

**사용 상황:**
- 주문 후 후처리
- 이미지 처리
- 배치 작업

**예시:**
```
주문 서비스 → 주문 큐에 메시지 전송
이미지 처리 워커 → 큐에서 메시지 수신 및 처리
```

### Pub-Sub (발행-구독)

Pub-Sub은 하나의 메시지를 여러 소비자가 처리한다.

**동작 원리:**
1. Publisher가 토픽에 메시지를 발행한다
2. 토픽을 구독한 모든 Subscriber가 메시지를 받는다
3. 각 Subscriber가 독립적으로 메시지를 처리한다

**특징:**
- 하나의 메시지를 여러 Consumer가 처리한다
- 발행자와 구독자가 느슨하게 결합된다
- 이벤트 알림 패턴에 적합하다

**사용 상황:**
- 알림 발송
- 데이터 복제
- 이벤트 알림

**예시:**
```
주문 서비스 → OrderCreated 이벤트 발행
재고 서비스 → 이벤트 구독 및 재고 차감
알림 서비스 → 이벤트 구독 및 알림 발송
통계 서비스 → 이벤트 구독 및 통계 업데이트
```

### Queue vs Pub-Sub

**Queue:**
- 하나의 메시지를 하나의 Consumer가 처리한다
- 작업 분산에 적합하다
- SQS, RabbitMQ 등

**Pub-Sub:**
- 하나의 메시지를 여러 Consumer가 처리한다
- 이벤트 알림에 적합하다
- SNS, Kafka 등

**선택 기준:**
- 작업을 분산하고 싶으면 Queue
- 여러 서비스에 알림을 보내고 싶으면 Pub-Sub

### 메시징의 장점

**장애 격리:**
- 한 Consumer가 장애가 나도 다른 Consumer는 계속 처리한다
- Producer가 장애가 나도 메시지는 큐에 보관된다
- 장애 전파가 없다

**확장성:**
- Consumer를 늘려 처리량을 확보할 수 있다
- 트래픽 급증 시 자동으로 확장한다
- 처리량이 유연하다

**느슨한 결합:**
- Producer와 Consumer가 독립적이다
- 서비스를 독립적으로 배포하고 확장할 수 있다
- 변경 영향이 적다

### 메시징의 단점

**최종 일관성:**
- 즉시 일관성을 보장하지 않는다
- 메시지 처리 지연이 있다
- 상태 불일치가 잠시 존재할 수 있다

**복잡도:**
- 메시지 중복, 순서 문제 등을 처리해야 한다
- DLQ 모니터링이 필요하다
- 운영 복잡도가 높다

**디버깅:**
- 비동기라 디버깅이 어렵다
- 메시지 흐름을 추적하기 어렵다
- 상태 추적이 필요하다

### 실무에서 자주 발생하는 문제

**1. 메시지 중복 처리:**

**문제:**
메시지가 여러 번 처리될 수 있다. 네트워크 재시도, Consumer 장애 등으로 발생한다.

**해결:**
- 멱등성 키를 사용한다
- 처리 기록을 저장해 중복을 방지한다
- 이벤트 ID를 사용해 중복을 확인한다

**2. 순서 문제:**

**문제:**
메시지 순서가 바뀔 수 있다. 메시지 큐에서 순서가 보장되지 않으면 발생한다.

**해결:**
- 순서가 중요한 경우 파티션 키를 사용한다
- FIFO 큐를 사용한다
- 순서 보장이 필요한 범위를 최소화한다

**3. DLQ 적체:**

**문제:**
DLQ가 쌓여도 아무도 모른다. 처리 실패가 누적된다.

**해결:**
- DLQ 모니터링을 설정한다
- DLQ에 메시지가 쌓이면 알람을 받는다
- 주기적으로 DLQ를 확인한다

**4. 메시지 유실:**

**문제:**
메시지가 유실될 수 있다. Consumer가 처리 전에 죽으면 발생한다.

**해결:**
- 메시지 확인(Ack)을 제대로 처리한다
- 가시성 타임아웃을 적절히 설정한다
- 재시도 메커니즘을 구현한다

### 필수로 넣어야 하는 것

**멱등성 키:**
- 메시지에 고유 ID를 포함한다
- 처리 기록을 저장해 중복을 방지한다
- 같은 메시지를 여러 번 처리해도 결과가 같아야 한다

**이벤트 버전:**
- 이벤트에 버전을 포함한다
- 스키마 변경 시 하위 호환성을 유지한다
- 버전별 처리 로직을 구현한다

**DLQ 모니터링:**
- DLQ에 메시지가 쌓이면 알람을 받는다
- 주기적으로 DLQ를 확인한다
- 실패 원인을 분석하고 해결한다

**상관관계 ID:**
- 메시지에 상관관계 ID를 포함한다
- 같은 요청에서 나온 메시지를 추적한다
- 디버깅에 사용한다

## 실시간 양방향 통신

### WebSocket

WebSocket은 양방향 통신을 위한 프로토콜이다. HTTP 업그레이드를 통해 WebSocket 연결을 만든다.

**WebSocket의 특징:**
- 지속적인 연결을 유지한다
- 양방향 통신이 가능하다
- 서버에서 클라이언트로 푸시할 수 있다

**동작 원리:**
1. 클라이언트가 HTTP 업그레이드 요청을 보낸다
2. 서버가 WebSocket으로 업그레이드한다
3. 양방향 통신이 시작된다
4. 연결이 유지되는 동안 통신한다

**사용하는 상황:**
- 실시간 알림
- 채팅
- 실시간 게임
- 주식 가격 스트림

### SSE (Server-Sent Events)

SSE는 서버에서 클라이언트로 단방향 스트림을 보내는 방식이다.

**SSE의 특징:**
- 서버에서 클라이언트로만 전송한다
- HTTP 연결을 유지한다
- 자동 재연결을 지원한다

**사용하는 상황:**
- 실시간 알림
- 상태 스트림
- 로그 스트림

**WebSocket vs SSE:**
- WebSocket: 양방향 통신이 필요할 때
- SSE: 서버에서 클라이언트로만 전송할 때

### 실시간 통신의 문제점

**1. 서버 재시작 시 연결 끊김:**

**문제:**
서버 재시작 시 연결이 끊긴다. 클라이언트가 연결을 잃는다.

**해결:**
- 클라이언트 재연결 로직이 필수다
- 연결이 끊기면 자동으로 재연결한다
- 재연결 시 상태를 복구한다

**2. 스케일 아웃 시 세션 관리:**

**문제:**
스케일 아웃 시 세션 관리가 복잡해진다. 여러 서버에 연결이 분산된다.

**해결:**
- 스티키 세션을 사용한다
- 세션을 외부 저장소에 저장한다
- 메시지 브로커를 사용한다

**3. 로드밸런서 설정:**

**문제:**
로드밸런서 설정에 따라 장애가 난다. WebSocket 연결이 제대로 유지되지 않을 수 있다.

**해결:**
- WebSocket을 지원하는 로드밸런서를 사용한다
- 스티키 세션을 활성화한다
- 연결 타임아웃을 적절히 설정한다

**4. 메모리 사용:**

**문제:**
서버 메모리에 상태를 두면 메모리 사용량이 증가한다. 연결이 많아지면 문제가 된다.

**해결:**
- 서버 메모리에 상태를 두지 않는다
- 상태는 외부 저장소에 저장한다
- 연결만 유지하고 상태는 별도 관리한다

### 실시간 통신의 주의사항

**재연결 로직:**
- 클라이언트 재연결 로직은 필수다
- 연결이 끊기면 자동으로 재연결한다
- 지수 백오프를 사용한다

**상태 관리:**
- 서버 메모리에 상태를 두지 않는다
- 상태는 외부 저장소에 저장한다
- 연결만 유지하고 상태는 별도 관리한다

**확장성:**
- 트래픽이 늘면 메시지 구조로 바꾸는 경우가 많다
- WebSocket은 확장성이 제한적이다
- 메시지 큐가 더 확장 가능하다

**모니터링:**
- 연결 수를 모니터링한다
- 연결 지속 시간을 추적한다
- 재연결 빈도를 확인한다

## 통신 방식 비교와 선택 기준

### 통신 방식 비교표

| 항목 | HTTP/REST | gRPC | 메시징 | WebSocket |
|------|-----------|------|--------|-----------|
| 통신 방식 | 동기 | 동기/스트리밍 | 비동기 | 양방향 |
| 성능 | 보통 | 빠름 | 빠름 | 빠름 |
| 타입 안정성 | 낮음 | 높음 | 낮음 | 낮음 |
| 외부 연동 | 가능 | 어려움 | 가능 | 가능 |
| 장애 전파 | 있음 | 있음 | 없음 | 있음 |
| 확장성 | 보통 | 보통 | 높음 | 낮음 |
| 복잡도 | 낮음 | 중간 | 높음 | 높음 |
| 학습 곡선 | 낮음 | 중간 | 높음 | 중간 |

### 선택 기준

**HTTP/REST를 선택하는 경우:**
- 외부 API 연동이 필요하다
- 브라우저에서 직접 호출해야 한다
- 빠른 개발이 필요하다
- 표준화된 인터페이스가 필요하다

**gRPC를 선택하는 경우:**
- 내부 서비스 간 고성능 통신이 필요하다
- 타입 안정성이 중요하다
- 스트리밍이 필요하다
- 대량 호출이 발생한다

**메시징을 선택하는 경우:**
- 장애 전파를 막고 싶다
- 비동기 처리가 필요하다
- 처리량을 확보하고 싶다
- 느슨한 결합이 필요하다

**WebSocket을 선택하는 경우:**
- 실시간 양방향 통신이 필요하다
- 서버에서 클라이언트로 푸시해야 한다
- 채팅이나 실시간 게임이 필요하다

### 하이브리드 접근

실무에서는 단일 방식으로 끝나는 경우가 거의 없다. 하이브리드 접근이 일반적이다.

**일반적인 패턴:**
- HTTP로 시작한다
- 병목이 생긴 구간만 메시지나 RPC로 분리한다
- 점진적으로 최적화한다

**예시:**
- 외부 API: HTTP/REST
- 내부 서비스 간 통신: gRPC
- 비동기 처리: 메시징
- 실시간 알림: WebSocket

## 내부 서비스 통신 설계에서 실제로 겪는 문제

### 구조가 과도하게 복잡해짐

**문제:**
초기부터 HTTP, gRPC, 메시지를 동시에 쓰는 경우가 있다. 결과적으로 신규 인력이 구조를 이해하는 데 시간이 많이 든다.

**원인:**
- 각 통신 방식의 장단점을 이해하지 못하고 도입한다
- 일관성 없는 선택으로 복잡도가 증가한다
- 문서화가 부족하다

**해결:**
- 표준을 정한다
- 대부분의 경우 HTTP로 시작한다
- 필요한 경우에만 다른 방식을 도입한다
- 문서화를 철저히 한다

### 동기 호출 장애 전파

**문제:**
A → B → C 구조에서 C가 느려지면 A까지 영향을 받는다.

**예시:**
- 주문 서비스 → 결제 서비스 → 재고 서비스
- 재고 서비스가 느려지면 주문 서비스까지 느려진다
- 전체 시스템이 장애가 난다

**해결:**
- B → C만 메시지로 바꿔도 장애 범위가 줄어든다
- 타임아웃을 짧게 설정한다
- 회로 차단기를 사용한다

**개선 예시:**
- 주문 서비스 → 결제 서비스 (HTTP)
- 결제 서비스 → 재고 서비스 (메시징)
- 재고 서비스 장애가 주문 서비스에 영향을 주지 않는다

### 이벤트의 주인이 사라짐

**문제:**
누가 발행했고 누가 소비하는지 시간이 지나면 모르게 된다.

**원인:**
- 이벤트 스키마가 문서화되지 않는다
- 소비 서비스가 문서화되지 않는다
- 시간이 지나면 맥락이 사라진다

**해결:**
- 이벤트 스키마와 소비 서비스는 문서로 남겨야 한다
- 이벤트 레지스트리를 만든다
- 이벤트 카탈로그를 관리한다

**문서화 항목:**
- 이벤트 타입과 스키마
- 발행 서비스
- 소비 서비스
- 이벤트 용도와 의미

### 타임아웃 기준 불일치

**문제:**
각 서비스의 타임아웃 기준이 제각각이면 장애 원인 추적이 어렵다.

**해결:**
- 표준 타임아웃을 정한다
- 내부 통신: 1-3초
- 외부 통신: 5-10초
- 타임아웃을 로그에 기록한다

### 재시도 폭발

**문제:**
여러 레이어에서 재시도를 하면 재시도가 폭발한다.

**예시:**
- 애플리케이션 레이어: 3회 재시도
- HTTP 클라이언트: 2회 재시도
- 로드밸런서: 재시도
- 총 최대 18회 시도 가능

**해결:**
- 재시도는 한 곳에서만 한다
- 지수 백오프를 사용한다
- 최대 재시도 횟수를 제한한다

## 트러블슈팅 사례

### HTTP 타임아웃으로 주문 장애 발생

**상황:**
- 주문 서비스가 결제 서비스를 HTTP로 호출한다
- 결제 서비스가 지연된다 (외부 결제 API 지연)
- 주문 API 전체가 장애가 난다

**원인:**
- 결제 서비스 타임아웃이 길다 (30초)
- 주문 서비스가 결제 응답을 기다린다
- 주문 API가 블로킹된다

**처리 방식:**
- 결제 요청을 비동기로 전환한다
- 주문 상태를 `PENDING`으로 분리한다
- 결제 완료 후 주문 상태를 업데이트한다

**개선 결과:**
- 주문 API가 즉시 응답한다
- 결제 지연이 주문에 영향을 주지 않는다
- 사용자 경험이 개선된다

### 메시지 중복 처리로 데이터 깨짐

**상황:**
- 주문 생성 이벤트가 여러 번 처리된다
- 재고가 중복 차감된다
- 데이터 정합성이 깨진다

**원인:**
- 메시지 재시도가 발생한다
- 멱등성 처리가 없다
- 같은 이벤트를 여러 번 처리한다

**처리 방식:**
- 처리 이력을 저장한다
- 이미 처리된 이벤트는 무시한다
- 이벤트 ID를 사용해 중복을 확인한다

**개선 결과:**
- 중복 처리가 방지된다
- 데이터 정합성이 유지된다
- 재시도가 안전해진다

### gRPC proto 변경으로 장애 발생

**상황:**
- 서버가 새 proto를 먼저 배포한다
- 클라이언트가 이전 proto를 사용한다
- 필드가 맞지 않아 에러가 발생한다

**원인:**
- 배포 순서가 잘못되었다
- 하위 호환성을 고려하지 않았다
- 필드를 삭제했다

**처리 방식:**
- 필드 삭제 대신 deprecated 처리한다
- 충분한 기간 후에 삭제한다
- 배포 순서를 명확히 한다

**개선 결과:**
- 하위 호환성이 유지된다
- 점진적으로 마이그레이션할 수 있다
- 장애가 발생하지 않는다

### WebSocket 연결 끊김으로 알림 유실

**상황:**
- 서버 재시작 시 WebSocket 연결이 끊긴다
- 클라이언트가 재연결하지 않는다
- 알림이 유실된다

**원인:**
- 클라이언트 재연결 로직이 없다
- 연결 상태를 추적하지 않는다
- 알림을 WebSocket에만 의존한다

**처리 방식:**
- 클라이언트 재연결 로직을 추가한다
- 연결이 끊기면 자동으로 재연결한다
- 알림을 메시지 큐에도 발행한다

**개선 결과:**
- 연결이 끊겨도 재연결된다
- 알림이 유실되지 않는다
- 안정성이 향상된다

## 테스트 방법

### HTTP 테스트

**계약 테스트:**
- API 스펙을 정의한다
- 클라이언트와 서버가 스펙을 준수하는지 확인한다
- 스펙 변경 시 테스트가 실패한다

**통합 테스트:**
- 실제 서비스를 호출한다
- 응답 시간과 에러를 확인한다
- 타임아웃 시나리오를 테스트한다

### gRPC 테스트

**proto 기준 테스트:**
- proto 파일로 클라이언트 코드를 생성한다
- 타입 안정성을 확인한다
- 스키마 변경 시 컴파일 에러로 감지한다

**통합 테스트:**
- 실제 gRPC 서비스를 호출한다
- 스트리밍 시나리오를 테스트한다
- 에러 처리를 확인한다

### 메시지 테스트

**소비 로직 단독 테스트:**
- 메시지 소비 로직을 단독으로 테스트한다
- 이벤트를 직접 생성해 테스트한다
- 멱등성을 확인한다

**통합 테스트:**
- 실제 메시지 큐를 사용한다
- 중복 메시지 시나리오를 테스트한다
- DLQ 동작을 확인한다

### 실패 시나리오 테스트

**통합 테스트에서는 실제 장애 시나리오를 반드시 재현한다:**
- 서비스 장애 시나리오
- 네트워크 지연 시나리오
- 메시지 중복 시나리오
- 타임아웃 시나리오

**카오스 엔지니어링:**
- 의도적으로 장애를 발생시킨다
- 시스템이 장애에 어떻게 반응하는지 확인한다
- 복구 메커니즘을 검증한다
