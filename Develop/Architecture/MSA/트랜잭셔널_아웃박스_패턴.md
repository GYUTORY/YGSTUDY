# Transactional Outbox 패턴

## 문제: DB 저장은 됐는데 메시지가 안 나간다

주문 서비스에서 주문을 생성하고, 결제 서비스에 메시지를 보내야 하는 상황을 생각해보자.

```java
@Transactional
public void createOrder(OrderRequest request) {
    orderRepository.save(order);        // 1. DB 저장
    messageQueue.send(orderCreatedEvent); // 2. 메시지 발행
}
```

이 코드는 두 가지 문제가 있다.

1. DB 저장 후 메시지 발행 전에 애플리케이션이 죽으면, 주문은 저장됐는데 메시지는 안 나간다.
2. 메시지를 먼저 보내고 DB를 저장하면, DB 저장이 실패했을 때 이미 보낸 메시지를 되돌릴 수 없다.

DB 트랜잭션과 메시지 브로커는 서로 다른 시스템이라 하나의 트랜잭션으로 묶을 수 없다. 2PC(Two-Phase Commit)를 쓰면 이론적으로 가능하지만, 성능이 나쁘고 메시지 브로커가 2PC를 지원하지 않는 경우가 대부분이다.

## Transactional Outbox 패턴의 핵심

메시지를 브로커에 직접 보내지 않고, DB의 outbox 테이블에 같이 저장한다. DB 트랜잭션 하나로 비즈니스 데이터와 메시지를 함께 커밋하니까 원자성이 보장된다.

```
[애플리케이션] → BEGIN TX → orders 테이블 INSERT
                         → outbox 테이블 INSERT
                → COMMIT
                
[별도 프로세스] → outbox 테이블 읽기 → 메시지 브로커 발행 → outbox 레코드 처리 완료 표시
```

### Outbox 테이블 설계

```sql
CREATE TABLE outbox (
    id             BIGINT AUTO_INCREMENT PRIMARY KEY,
    event_id       CHAR(36) NOT NULL UNIQUE,    -- UUID, 소비자 dedup 키
    aggregate_type VARCHAR(255) NOT NULL,       -- 'Order', 'Payment' 등
    aggregate_id   VARCHAR(255) NOT NULL,       -- 주문 ID 등
    event_type     VARCHAR(255) NOT NULL,       -- 'OrderCreated', 'OrderCancelled'
    payload        TEXT NOT NULL,               -- JSON 직렬화된 이벤트 데이터
    created_at     TIMESTAMP(6) NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
    sent_at        TIMESTAMP NULL,              -- 발행 완료 시각
    status         VARCHAR(20) NOT NULL DEFAULT 'PENDING'  -- PENDING, SENT, FAILED
);

CREATE INDEX idx_outbox_status ON outbox(status, created_at);
```

`aggregate_type`과 `aggregate_id`는 어떤 엔티티에서 발생한 이벤트인지 추적할 때 쓴다. 나중에 디버깅할 때 "이 주문에 대해 어떤 이벤트가 나갔지?" 하고 찾을 수 있어야 한다. `event_id`는 UUID로 부여하는데, 같은 outbox 레코드가 중복 발행되더라도 소비자가 dedup 할 수 있게 하는 키다. 뒤의 중복 발행 방어 항목에서 자세히 다룬다.

`created_at`을 굳이 마이크로초 정밀도(`TIMESTAMP(6)`)로 잡는 이유는, 동일한 초 안에 수십 건의 outbox가 INSERT되는 환경에서 폴링 워커가 정렬 키로 쓰기 때문이다. 초 단위면 같은 초에 들어온 레코드 순서가 흔들린다.

### 애플리케이션 코드

```java
@Transactional
public void createOrder(OrderRequest request) {
    Order order = Order.create(request);
    orderRepository.save(order);

    OutboxEvent event = OutboxEvent.builder()
        .eventId(UUID.randomUUID().toString())
        .aggregateType("Order")
        .aggregateId(order.getId().toString())
        .eventType("OrderCreated")
        .payload(objectMapper.writeValueAsString(new OrderCreatedEvent(order)))
        .build();
    outboxRepository.save(event);
}
```

비즈니스 로직과 outbox 저장이 같은 트랜잭션 안에 있으니까, 둘 중 하나만 성공하는 일은 없다.

## Outbox에서 메시지 꺼내는 방법: 폴링 vs CDC

Outbox 테이블에 쌓인 메시지를 브로커로 보내는 방법은 크게 두 가지다.

### 폴링(Polling) 방식

주기적으로 outbox 테이블을 조회해서 미발행 메시지를 가져오는 방법이다.

```java
@Scheduled(fixedDelay = 1000) // 1초마다
public void publishOutboxEvents() {
    List<OutboxEvent> events = outboxRepository
        .findByStatusOrderByCreatedAtAsc("PENDING", PageRequest.of(0, 100));
    
    for (OutboxEvent event : events) {
        try {
            kafkaTemplate.send(
                event.getAggregateType() + ".events",
                event.getAggregateId(),
                event.getPayload()
            );
            event.markAsSent();
            outboxRepository.save(event);
        } catch (Exception e) {
            event.markAsFailed();
            outboxRepository.save(event);
            log.error("Outbox event 발행 실패: {}", event.getId(), e);
        }
    }
}
```

**폴링 방식의 현실적 문제들:**

- DB에 부하가 간다. 1초마다 SELECT 쿼리를 날리는 건 트래픽이 적을 때는 괜찮지만, outbox 테이블에 레코드가 수십만 건 쌓이면 인덱스를 타도 부담이 된다.
- 폴링 간격만큼 지연이 생긴다. 1초 간격이면 최대 1초 늦게 메시지가 나간다. 실시간성이 중요한 서비스에는 맞지 않다.
- 인스턴스가 여러 대면 같은 메시지를 중복 처리할 수 있다. 분산 락이나 `SELECT ... FOR UPDATE SKIP LOCKED` 같은 처리가 필요하다.

```sql
-- 여러 인스턴스에서 동시에 폴링할 때
SELECT * FROM outbox 
WHERE status = 'PENDING' 
ORDER BY created_at ASC 
LIMIT 100
FOR UPDATE SKIP LOCKED;
```

`SKIP LOCKED`는 다른 트랜잭션이 이미 잠근 행을 건너뛰니까, 여러 워커가 동시에 돌아도 같은 메시지를 가져가지 않는다.

### 폴링 워커의 분산 락 — 환경별 선택지

`SKIP LOCKED`는 PostgreSQL 9.5+, MySQL 8.0+, Oracle 11g+ 에서 쓸 수 있다. 그 이전 버전이나 MySQL 5.7 환경, 또는 여러 DB에 흩어진 outbox를 단일 워커 풀이 처리해야 하는 경우엔 다른 방법이 필요하다.

**DB 기반 advisory lock — PostgreSQL `pg_try_advisory_lock`**

PostgreSQL은 행 잠금과는 별개로 정수 키 기반의 advisory lock을 제공한다. 행이 아니라 "이 작업 자체"에 락을 거는 방식이다.

```java
@Scheduled(fixedDelay = 1000)
public void publishOutboxEvents() {
    Boolean acquired = jdbcTemplate.queryForObject(
        "SELECT pg_try_advisory_lock(?)", Boolean.class, OUTBOX_LOCK_KEY
    );
    if (Boolean.FALSE.equals(acquired)) {
        return; // 다른 인스턴스가 이미 처리 중
    }
    try {
        // outbox 처리 로직
    } finally {
        jdbcTemplate.queryForObject(
            "SELECT pg_advisory_unlock(?)", Boolean.class, OUTBOX_LOCK_KEY
        );
    }
}
```

이 방식의 장점은 워커가 죽으면 세션이 끊기면서 락도 자동 해제된다는 점이다. Redis 락처럼 TTL을 잡아야 할지 고민할 필요가 없다. 단점은 한 번에 인스턴스 하나만 일한다는 거다. SKIP LOCKED는 여러 워커가 병렬로 다른 행을 가져가지만, advisory lock은 전체 작업을 직렬화한다. 트래픽이 많으면 advisory lock 키를 샤딩(`pg_try_advisory_lock(OUTBOX_LOCK_KEY + shardId)`)해서 워커별로 다른 shardId를 처리하게 해야 한다.

MySQL은 `GET_LOCK(name, timeout)`이 비슷한 역할을 한다. 다만 MySQL 5.7 이전엔 같은 세션에서 두 번째 `GET_LOCK`을 호출하면 첫 번째 락이 풀리는 동작이 있어서 주의해야 한다.

**Redis Redlock 또는 단일 인스턴스 락**

DB 외부에서 락을 관리하고 싶을 때 쓴다. Redisson의 `RLock`이 가장 많이 쓰는 구현이다.

```java
RLock lock = redissonClient.getLock("outbox:publisher");
if (lock.tryLock(0, 30, TimeUnit.SECONDS)) {
    try {
        publishOutboxEvents();
    } finally {
        lock.unlock();
    }
}
```

Redlock 알고리즘은 Redis 인스턴스 N개 중 과반수에서 락을 얻으면 성공으로 보는 방식이다. 단일 Redis 마스터 실패에 대비할 수 있지만, 클럭 드리프트나 GC pause 같은 상황에서 안전하지 않다는 비판(Martin Kleppmann의 "How to do distributed locking" 글)이 유명하다. outbox 처리는 어차피 중복 발행 방어를 소비자 쪽에서 하니까, Redlock 대신 단일 Redis 락으로도 충분한 경우가 많다.

**선택 기준은 단순하다.**

- DB가 PostgreSQL 9.5+, MySQL 8.0+ → `SKIP LOCKED`. 별도 인프라 없이 가장 단순하다.
- DB가 PostgreSQL이지만 SKIP LOCKED 락 경합이 심함 → advisory lock + 샤딩
- Redis 이미 운영 중이고 락을 한 곳에 모으고 싶음 → Redisson `RLock`
- DB가 SKIP LOCKED 미지원이고 Redis도 없음 → DB advisory lock + 단일 워커

### CDC(Change Data Capture) 방식

DB의 트랜잭션 로그(binlog, WAL 등)를 감시해서 outbox 테이블에 변경이 생기면 바로 메시지를 발행하는 방법이다. 애플리케이션이 직접 outbox를 읽지 않는다.

```
[MySQL binlog] → [Debezium Connector] → [Kafka] → [소비자 서비스]
```

이 방식은 폴링의 문제를 대부분 해결한다.

- DB에 추가 쿼리를 날리지 않는다. binlog를 읽는 건 MySQL 레플리케이션과 같은 메커니즘이라 부하가 적다.
- 거의 실시간으로 메시지가 나간다. binlog에 기록되는 즉시 감지한다.
- 폴링 주기에 대한 고민이 없다.

## Debezium CDC 연동

Debezium은 Kafka Connect 기반의 CDC 도구다. MySQL, PostgreSQL, MongoDB 등의 DB 변경 사항을 Kafka 토픽으로 보내준다.

### Debezium 커넥터 설정

```json
{
  "name": "outbox-connector",
  "config": {
    "connector.class": "io.debezium.connector.mysql.MySqlConnector",
    "database.hostname": "order-db",
    "database.port": "3306",
    "database.user": "debezium",
    "database.password": "dbz-password",
    "database.server.id": "184054",
    "database.server.name": "order-service",
    "database.include.list": "order_db",
    "table.include.list": "order_db.outbox",
    "transforms": "outbox",
    "transforms.outbox.type": "io.debezium.transforms.outbox.EventRouter",
    "transforms.outbox.table.field.event.id": "event_id",
    "transforms.outbox.table.field.event.key": "aggregate_id",
    "transforms.outbox.table.field.event.type": "event_type",
    "transforms.outbox.table.field.event.payload": "payload",
    "transforms.outbox.route.topic.replacement": "${routedByValue}.events"
  }
}
```

핵심은 `EventRouter` 트랜스폼이다. outbox 테이블의 row를 읽어서 `aggregate_type` 값을 기반으로 Kafka 토픽을 결정하고, payload를 메시지 본문으로 보낸다. `table.field.event.id`를 `event_id` UUID 컬럼으로 잡아두면 Debezium이 Kafka 메시지 헤더 `id`에 UUID를 채워준다. 소비자는 이 헤더로 dedup 한다.

### Debezium 사용 시 주의할 점

**커넥터가 죽으면 메시지가 밀린다.** Debezium 커넥터가 재시작되면 마지막으로 읽은 binlog 위치부터 다시 읽기 시작한다. 이 사이에 쌓인 메시지가 한꺼번에 밀려오니까, 소비자 측에서 burst 트래픽을 감당할 수 있어야 한다.

**binlog 보존 기간을 확인해야 한다.** 커넥터가 오래 죽어있으면 binlog가 이미 삭제돼서 복구가 안 되는 경우가 있다. MySQL의 `expire_logs_days`나 `binlog_expire_logs_seconds` 설정을 확인하고, 커넥터 다운타임보다 길게 잡아야 한다.

**스키마 변경에 민감하다.** outbox 테이블에 컬럼을 추가하거나 타입을 바꾸면 커넥터가 깨질 수 있다. Debezium이 스키마 변경을 감지하긴 하지만, EventRouter 설정과 맞지 않으면 에러가 난다. 스키마 변경 전에 커넥터를 일시 중지하고, 설정을 업데이트한 후 재시작하는 게 안전하다.

**Debezium 전용 DB 계정을 만들어야 한다.** `REPLICATION SLAVE`, `REPLICATION CLIENT` 권한이 필요하고, 일반 애플리케이션 계정과 분리하는 게 맞다.

```sql
CREATE USER 'debezium'@'%' IDENTIFIED BY 'dbz-password';
GRANT SELECT, REPLICATION SLAVE, REPLICATION CLIENT ON *.* TO 'debezium'@'%';
```

## MySQL binlog vs PostgreSQL WAL — CDC 백엔드 차이

Debezium은 같은 인터페이스로 보이지만, MySQL 커넥터와 PostgreSQL 커넥터는 운영상 신경 써야 하는 지점이 꽤 다르다.

### MySQL binlog 기반

MySQL CDC는 자기 자신을 레플리카로 위장해 마스터의 binlog를 받아간다. 설정은 단순하다.

```ini
# my.cnf
server-id = 1
log_bin = mysql-bin
binlog_format = ROW          # STATEMENT는 CDC에 안 맞다
binlog_row_image = FULL      # 변경 전/후 모든 컬럼 기록
expire_logs_days = 14        # 또는 binlog_expire_logs_seconds
```

`binlog_format=ROW`가 핵심이다. `STATEMENT` 포맷은 SQL 자체를 기록하기 때문에 CDC로 후처리하기 까다롭다(예: `NOW()` 호출이 들어간 INSERT는 row 값이 결정되지 않는다). `binlog_row_image=FULL`을 안 켜두면 변경되지 않은 컬럼이 빠져서 EventRouter가 payload를 못 읽는 경우가 생긴다.

binlog가 디스크에 쌓이는 양은 통상 데이터 변경량의 1.5~3배다. `expire_logs_days`를 너무 짧게 잡으면 Debezium이 잠깐 죽었을 때 복구가 안 되고, 너무 길게 잡으면 디스크가 찬다. 운영에서는 보통 7~14일 정도 잡는다.

### PostgreSQL WAL 기반 — logical replication slot

PostgreSQL CDC는 logical replication slot을 만들어 거기서 WAL을 디코딩해 가져온다. 설정이 조금 더 까다롭다.

```ini
# postgresql.conf
wal_level = logical          # replica가 아니라 logical 이어야 한다
max_replication_slots = 4    # Debezium 커넥터 수만큼
max_wal_senders = 4
```

```sql
-- 권한
CREATE ROLE debezium WITH REPLICATION LOGIN PASSWORD '...';
GRANT CONNECT ON DATABASE order_db TO debezium;
GRANT USAGE ON SCHEMA public TO debezium;
GRANT SELECT ON public.outbox TO debezium;

-- outbox 테이블에 REPLICA IDENTITY 설정
ALTER TABLE outbox REPLICA IDENTITY FULL;
```

`REPLICA IDENTITY`는 UPDATE/DELETE 발생 시 어떤 컬럼을 식별 정보로 WAL에 기록할지 정한다. 기본값은 PK 컬럼만 기록인데, outbox는 INSERT만 일어나니 큰 차이가 없지만, 만약 outbox에 UPDATE를 쓰는 패턴이라면 `FULL`로 잡아야 변경 전 row 전체가 WAL에 남는다.

가장 위험한 함정은 **replication slot이 소비되지 않을 때 WAL이 디스크에 끝없이 쌓인다**는 점이다. Debezium 커넥터가 죽거나 멈춰서 slot이 진행되지 않으면, PostgreSQL은 그 slot이 아직 안 읽은 WAL을 절대 지우지 않는다. 며칠 방치하면 운영 DB 디스크가 가득 차서 쓰기가 막힌다. MySQL의 `expire_logs_days`는 시간 기반으로 무조건 지우지만, PostgreSQL의 logical slot은 소비자가 진행될 때까지 보존하는 차이다.

```sql
-- slot 상태 확인 — confirmed_flush_lsn 이 멈춰있으면 위험 신호
SELECT slot_name, active, restart_lsn, confirmed_flush_lsn,
       pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn)) AS lag_bytes
FROM pg_replication_slots;
```

`lag_bytes`가 수 GB 이상으로 커지거나 시간이 지나도 줄지 않으면 Debezium 쪽 문제다. PostgreSQL 13+ 에는 `max_slot_wal_keep_size` 설정이 있어서, 이 값 이상으로 WAL이 쌓이면 slot을 무효화시켜 디스크를 보호한다. 운영에서는 반드시 이 값을 잡아두는 게 맞다. 다만 무효화되면 Debezium은 처음부터 snapshot을 다시 떠야 한다.

### 정리하면

| 항목 | MySQL binlog | PostgreSQL WAL |
|------|--------------|----------------|
| 활성화 | `log_bin = mysql-bin`, `binlog_format = ROW` | `wal_level = logical`, slot 생성 |
| 보존 정책 | 시간/크기 기반 자동 삭제 | slot 소비 진행 여부에 종속 |
| 미소비 시 위험 | 보존 기간 지나면 데이터 유실 → snapshot 재실행 | 디스크 폭주 → DB 쓰기 중단 |
| 식별자 설정 | 자동 | `REPLICA IDENTITY` 별도 지정 필요 |
| 디스크 보호 장치 | `expire_logs_days` | `max_slot_wal_keep_size` (13+) |

PostgreSQL CDC는 slot 모니터링 알람이 없으면 운영하지 마라. 새벽에 디스크가 차서 깨는 일이 실제로 있다.

## 실패 복구 시나리오

Debezium은 평상시엔 잘 돌지만, 죽었을 때 복구 절차를 모르면 데이터를 통째로 잃을 수 있다.

### 시나리오 1: 커넥터 프로세스 다운 → 자동 복구

가장 흔한 경우다. Kafka Connect 워커가 죽거나 컨테이너가 재기동되면, Connect는 마지막 커밋된 offset부터 다시 읽기 시작한다. offset은 `connect-offsets` 토픽에 저장돼 있다.

```bash
# 커넥터 상태 확인
curl -s http://kafka-connect:8083/connectors/outbox-connector/status | jq

# 결과 예시
# {
#   "name": "outbox-connector",
#   "connector": {"state": "RUNNING", "worker_id": "..."},
#   "tasks": [{"state": "FAILED", "trace": "io.debezium.DebeziumException: ..."}]
# }
```

여기서 중요한 건 `connector.state`와 `tasks[].state`를 따로 확인해야 한다는 점이다. 커넥터는 RUNNING이지만 태스크가 FAILED인 경우가 종종 있다. 운영 헬스체크 스크립트는 `tasks[].state`까지 체크해야 한다.

태스크가 FAILED면 restart API로 살릴 수 있다.

```bash
curl -X POST http://kafka-connect:8083/connectors/outbox-connector/tasks/0/restart
```

자동화하려면 cron이나 Prometheus AlertManager에서 5분마다 status를 확인해 FAILED면 restart를 던지는 식으로 구성한다. 다만 무한 재시도는 위험하니, 3회 연속 실패면 알람만 보내고 사람이 보게 막아둬야 한다.

### 시나리오 2: binlog 만료로 offset 유실

커넥터가 며칠 죽어있는 사이 MySQL binlog가 만료돼 삭제되면, 마지막 offset 위치를 binlog에서 찾을 수 없다. 커넥터는 이런 에러를 띄운다.

```
ERROR: The connector is trying to read binlog starting at GTID ..., 
       but this is no longer available on the server.
```

이 상황에서 선택지는 세 가지다.

1. **snapshot 다시 뜨기 (`snapshot.mode = when_needed`)** — outbox 테이블 전체를 다시 스캔해 메시지를 만든다. 그런데 outbox에 이미 처리된 옛날 레코드가 남아있으면 그게 다 다시 발행돼 폭격이 된다. CDC 환경에서 outbox 즉시 삭제 패턴(아래 "Outbox 테이블 정리" 항목)을 쓴 경우엔 테이블이 비어있으니 안전하지만, 보존하는 경우엔 위험하다.
2. **빈 토픽으로 시작 (`snapshot.mode = schema_only`)** — 스키마만 읽고 데이터는 무시한다. 그 사이 일어난 outbox INSERT는 영구히 유실된다. 폴링 워커로 보강을 동시에 돌리는 게 아니라면 데이터 손실이 확정된다.
3. **수동 복구** — 커넥터를 멈추고, 손실된 기간 동안의 outbox 레코드를 SQL로 직접 뽑아 Kafka에 다시 넣는다. `kafka-console-producer`나 별도 마이그레이션 스크립트로 처리한다. 가장 안전하지만 손이 많이 간다.

운영에서는 보통 `snapshot.mode = when_needed`로 두고, outbox 테이블에 처리 완료된 레코드를 일정 기간(예: 1시간) 남기되 그 이상 지난 건 지우는 식으로 운영한다. 1시간 안에 복구 가능한 다운타임은 snapshot 재실행으로 흡수되고, 그 이상은 사람이 개입한다.

### 시나리오 3: schema_history 토픽 손상 (MySQL)

Debezium MySQL 커넥터는 별도의 `dbhistory.<server-name>` 토픽에 DDL 이력을 저장한다. 이게 손상되거나 의도치 않게 삭제되면 커넥터가 시작 자체를 못 한다.

```
ERROR: The db history topic ... is missing required data
```

복구 절차:

1. 커넥터를 삭제한다 (offset은 `connect-offsets`에 남아있다).
2. schema history 토픽을 재생성한다 (compacted, infinite retention).
3. 같은 이름의 커넥터를 다시 등록하면서 `database.history.kafka.recovery.poll.interval.ms`를 충분히 잡는다.
4. snapshot을 한 번 다시 뜨도록 `snapshot.mode = schema_only_recovery`로 설정한다.

이 과정에서 데이터 메시지는 다시 안 흐른다. 스키마 정보만 복구된다. 운영에서는 schema history 토픽이 절대 지워지지 않게 ACL과 retention 설정을 엄격하게 잡는다 (`cleanup.policy=compact`, `retention.ms=-1`).

### 시나리오 4: Kafka Connect 클러스터 전체 장애

Connect 클러스터 전체가 죽으면 커넥터의 offset이 어디까지 갔는지 알 수 없는 시간이 생긴다. Connect가 다시 살아나면 `connect-offsets`의 마지막 커밋 지점부터 재개하는데, 그 사이 outbox에 이미 데이터가 누적돼 있다. 복구되면 burst가 흘러간다.

소비자 측에서 burst를 감당하려면 컨슈머 그룹의 `max.poll.records`를 적당히 잡고, 처리 시간이 `max.poll.interval.ms` 안에 들어오는지 확인해야 한다. 평상시 0.5초에 처리되던 게 burst 때 30초가 걸리면 rebalance가 일어나면서 같은 메시지를 또 받는다.

## 메시지 순서 보장 — 어디까지 보장되는가

"Kafka에서 순서 보장된다"는 말은 절반만 맞다. 정확히는 **동일 파티션 안에서만** 순서가 보장된다.

### 파티션 키 설계가 순서를 결정한다

같은 주문에 대한 `OrderCreated → OrderPaid → OrderShipped` 이벤트가 순서를 지켜야 한다면, 세 이벤트 모두 같은 파티션으로 가야 한다. 파티션 키를 `aggregate_id`(주문 ID)로 잡으면 해시 결과가 같으니 같은 파티션에 떨어진다.

Debezium의 EventRouter는 `table.field.event.key`로 지정한 컬럼을 Kafka 메시지 키로 쓴다. 위 설정에서는 `aggregate_id`로 잡아뒀으니 자동으로 처리된다.

```
주문 A: OrderCreated → 파티션 3
주문 A: OrderPaid    → 파티션 3 (같은 키 → 같은 파티션 → 순서 보장)
주문 B: OrderCreated → 파티션 7
주문 A: OrderShipped → 파티션 3 (B와는 순서 관계 없음)
```

**서로 다른 aggregate 간에는 순서가 보장되지 않는다.** 주문 A의 이벤트와 주문 B의 이벤트가 시간순으로 처리되리란 보장은 없다. 이게 문제가 되는 도메인이라면(예: 글로벌 시퀀스가 필요한 회계 이벤트) 단일 파티션을 써야 하는데, 그러면 처리량이 그 파티션의 한 컨슈머로 제한된다.

### Single Message Transform이 순서를 흩뜨릴 수 있다

Debezium에서 EventRouter 외에 다른 SMT(Single Message Transform)를 체인으로 걸 때 주의해야 한다. 특히 `RegexRouter`로 토픽을 동적으로 바꾸거나, 필터 SMT로 일부 메시지를 버리는 경우엔 같은 aggregate의 이벤트가 다른 토픽으로 흩어질 수 있다.

```json
"transforms": "outbox,filter",
"transforms.filter.type": "io.debezium.transforms.Filter",
"transforms.filter.language": "jsr223.groovy",
"transforms.filter.condition": "value.eventType != 'OrderShipped'"
```

이 설정은 `OrderShipped` 이벤트를 버린다. 소비자 입장에서는 `OrderPaid` 다음에 그 이벤트가 오지 않는 게 정상인지, 유실된 건지 구분이 안 된다. 필터링은 가능하면 소비자 쪽에서 하고, 커넥터 단에서는 가능한 한 가공을 안 하는 게 디버깅이 편하다.

### 프로듀서 max.in.flight 와 순서

Kafka 프로듀서는 기본적으로 한 connection에 여러 요청을 동시 전송한다(`max.in.flight.requests.per.connection=5`). 일부 요청이 실패해 재시도되면 뒤에 보낸 게 먼저 들어갈 수 있다. 멱등 프로듀서(`enable.idempotence=true`)는 이를 자동 보정해 순서를 보존한다. Debezium은 내부 프로듀서에서 멱등성을 기본으로 켠다.

직접 폴링 워커를 짜는 경우엔 명시적으로 잡아야 한다.

```properties
acks=all
enable.idempotence=true
max.in.flight.requests.per.connection=5  # 멱등 켜져있으면 5까지 안전
retries=2147483647
```

## 중복 발행 방어 심화

Outbox 패턴을 쓰더라도 메시지가 중복될 수 있다. 대표적인 시나리오:

1. 폴링 방식에서 메시지를 브로커에 보냈지만, outbox 레코드를 SENT로 업데이트하기 전에 애플리케이션이 죽는 경우
2. CDC 방식에서 Debezium이 메시지를 Kafka에 보냈지만, offset 커밋 전에 커넥터가 재시작되는 경우
3. 카프카 프로듀서의 재시도로 같은 메시지가 두 번 들어가는 경우

"최소 한 번 전달(at-least-once delivery)"은 보장하지만, "정확히 한 번 전달(exactly-once)"은 보장하지 않는다. 그래서 소비자 측에서 멱등성 처리가 필수다.

### 프로듀서 단 — idempotent + transactional

폴링 워커가 직접 Kafka에 발행할 때는 프로듀서 옵션을 다음과 같이 잡는다.

```properties
enable.idempotence=true
acks=all
retries=Integer.MAX_VALUE
transactional.id=outbox-publisher-${HOSTNAME}
```

`enable.idempotence=true`만 켜도 단일 프로듀서 세션 안에서는 중복이 차단된다. Kafka 브로커가 프로듀서 ID + 시퀀스 번호로 중복을 걸러주기 때문이다. 다만 프로듀서가 죽었다 새로 떠서 같은 메시지를 다시 보내면 새 ID가 부여돼 중복으로 인식되지 않는다.

`transactional.id`를 부여하면 Kafka 트랜잭션을 쓸 수 있고, 프로듀서가 죽었다 다시 떠도 같은 transactional.id를 쓰는 이전 프로듀서의 미완료 트랜잭션을 abort 시킨다. outbox 워커에서 "Kafka send → outbox UPDATE" 사이의 원자성이 필요하다면 Kafka 트랜잭션과 DB 트랜잭션을 같이 묶을 수도 있지만, 운영 복잡도가 크게 오른다. 보통은 소비자 쪽 dedup으로 해결한다.

`transactional.id`는 워커 인스턴스마다 고유해야 한다. 보통 호스트명이나 K8s Pod 이름을 붙인다. 같은 transactional.id를 가진 인스턴스가 둘 동시에 뜨면 fencing 메커니즘이 작동해 늦게 뜬 쪽이 먼저 뜬 쪽을 무효화시킨다.

### 소비자 단 — eventId 기반 dedup

위에서 outbox 테이블에 `event_id` UUID를 두는 이유가 여기서 드러난다. 같은 outbox 레코드에서 비롯된 메시지는 몇 번을 받아도 같은 `event_id`를 가진다.

```java
@KafkaListener(topics = "Order.events")
@Transactional
public void handleOrderCreated(ConsumerRecord<String, String> record) {
    String eventId = new String(record.headers().lastHeader("id").value());

    if (!processedEventRepository.tryMarkProcessed(eventId)) {
        return; // 이미 처리됨
    }

    OrderCreatedEvent event = objectMapper.readValue(record.value(), OrderCreatedEvent.class);
    paymentService.initiatePayment(event.getOrderId(), event.getAmount());
}
```

`tryMarkProcessed`는 다음 SQL로 구현한다.

```sql
INSERT INTO processed_events (event_id, processed_at, consumer_group) 
VALUES (?, NOW(), ?)
ON CONFLICT (event_id, consumer_group) DO NOTHING;
-- 또는 MySQL: INSERT IGNORE ...
```

PK 충돌 시 false를 반환하는 식이다. 비즈니스 로직과 `processed_events` INSERT가 같은 트랜잭션에 있어야 한다. 분리되면 비즈니스 로직만 실행되고 INSERT가 실패해 중복 실행이 일어난다.

`consumer_group` 컬럼을 같이 두는 이유는, 같은 이벤트를 결제 서비스와 알림 서비스가 각자 받는 경우 컨슈머별로 dedup 상태가 달라야 하기 때문이다. 컨슈머 그룹 ID를 그대로 쓴다.

### Dedup window 설계

`processed_events` 테이블은 무한히 커진다. 보존 기간(dedup window)은 메시지가 재발행될 수 있는 최대 기간보다 길게 잡아야 한다. 운영에서 자주 쓰는 기준:

- Debezium 다운타임 최대치 (예: 7일)
- Kafka retention 기간 (예: 7일)
- 둘 중 더 긴 값에 안전 마진 (예: 2배)

7일 retention 환경이면 보통 14~30일 정도 잡는다. 그 이상은 dedup 의미가 없다. Kafka에 메시지가 이미 없는데 중복 처리될 일이 없다.

### processed_events 테이블 파티셔닝

수억 건이 쌓이면 dedup INSERT가 느려진다. PK 인덱스가 커지면서 캐시 미스가 늘어나기 때문이다. 시간 기반 파티셔닝으로 풀어야 한다.

```sql
-- PostgreSQL 선언적 파티셔닝
CREATE TABLE processed_events (
    event_id VARCHAR(64) NOT NULL,
    consumer_group VARCHAR(64) NOT NULL,
    processed_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (processed_at, event_id, consumer_group)
) PARTITION BY RANGE (processed_at);

CREATE TABLE processed_events_2026_06 PARTITION OF processed_events
    FOR VALUES FROM ('2026-06-01') TO ('2026-07-01');
CREATE TABLE processed_events_2026_07 PARTITION OF processed_events
    FOR VALUES FROM ('2026-07-01') TO ('2026-08-01');
```

월별 파티션을 미리 잡아두고, dedup window가 지나면 파티션 자체를 DROP 한다. `DELETE`보다 수십 배 빠르고 테이블 부풀림(bloat)도 없다.

주의할 점: PK에 `processed_at`이 들어가야 파티션 키로 쓸 수 있다. 그러면 PK가 복합키가 되어 INSERT 시 `event_id`만으로 조회하는 게 인덱스를 못 탄다. 보조 인덱스를 따로 만들거나, dedup 조회 시 시간 범위를 같이 주는 식으로 풀어야 한다.

```sql
-- 최근 30일 안에서만 dedup
INSERT INTO processed_events (event_id, consumer_group, processed_at)
VALUES (?, ?, NOW())
ON CONFLICT (processed_at, event_id, consumer_group) DO NOTHING;

-- 조회 시
SELECT 1 FROM processed_events
WHERE event_id = ?
  AND consumer_group = ?
  AND processed_at >= NOW() - INTERVAL '30 days';
```

### 자연 멱등성 활용

모든 경우에 별도 테이블이 필요한 건 아니다. 비즈니스 로직 자체가 멱등한 경우가 있다.

```java
public void cancelOrder(String orderId) {
    Order order = orderRepository.findById(orderId)
        .orElseThrow(() -> new OrderNotFoundException(orderId));

    if (order.getStatus() == OrderStatus.CANCELLED) {
        return; // 이미 취소된 상태면 아무것도 안 함
    }

    order.cancel();
    orderRepository.save(order);
}
```

상태 기계(state machine) 패턴으로 구현하면, 같은 이벤트를 여러 번 받아도 상태 전이가 한 번만 일어난다. 결제 처리처럼 외부 부수효과(돈이 빠져나간다)가 있는 경우엔 자연 멱등성만으로 부족하니 `event_id` dedup이 필수다.

## 운영 모니터링

Outbox + CDC 조합은 평상시엔 조용하지만, 한 번 꼬이면 데이터 유실로 직결된다. 모니터링 지표를 최소한 다음 셋은 잡아야 한다.

### outbox 적체량 (PENDING 카운트)

폴링 방식이든 CDC든 outbox에 처리 안 된 레코드가 비정상적으로 쌓이는지 본다.

```sql
SELECT COUNT(*) AS pending_count,
       MIN(created_at) AS oldest_pending
FROM outbox
WHERE status = 'PENDING';
```

Prometheus exporter로 빼서 알람을 건다.

- `pending_count > 10000` → 워커가 처리량을 못 따라간다
- `oldest_pending < NOW() - INTERVAL '5 minutes'` → 5분 이상 묵은 메시지 존재. 워커가 멈췄을 가능성

CDC 방식에서 outbox를 INSERT 직후 DELETE 하는 패턴이라면 PENDING 카운트가 항상 0에 가까워야 한다. 1000건이 쌓여 있으면 Debezium이 따라오지 못하고 있다는 신호다.

### Debezium lag — source position vs current binlog

Debezium은 JMX 메트릭으로 자신이 어디까지 읽었는지 노출한다.

```
debezium.mysql:type=connector-metrics,context=binlog,server=order-service
  - MilliSecondsBehindSource    # 마스터 binlog 위치와의 시간 차
  - SourceEventPosition          # 마지막으로 읽은 binlog 파일/오프셋
```

`MilliSecondsBehindSource`가 평소엔 100ms 이하인데 30초 이상으로 치솟으면 뭔가 문제다. burst가 들어왔거나, Kafka 발행이 느려졌거나, 커넥터가 일시적으로 멈춘 경우다.

PostgreSQL 커넥터는 slot lag을 직접 조회해야 한다.

```sql
SELECT slot_name,
       pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn)) AS lag_bytes,
       active
FROM pg_replication_slots
WHERE slot_name = 'debezium_outbox';
```

`lag_bytes`가 수 GB 이상이면 디스크 위협으로 연결되니 즉시 알람이 필요하다.

### Kafka Connect REST API 헬스체크

Connect 클러스터의 모든 커넥터/태스크 상태를 5분마다 폴링한다.

```bash
curl -s http://kafka-connect:8083/connectors?expand=status | jq '
  to_entries[] | 
  {
    name: .key,
    connector_state: .value.status.connector.state,
    task_states: [.value.status.tasks[].state]
  }
'
```

여기서 `task_states`에 `FAILED`가 하나라도 있으면 알람. `RUNNING`인 줄 알았는데 task만 죽어있는 케이스가 자주 발생한다.

자동 복구를 걸 거라면 `tasks[0].state = FAILED` 발견 시 1회 restart 시도, 5분 후에도 FAILED면 알람 호출 같은 식으로 구성한다.

### 소비자 lag

마지막으로 컨슈머 그룹 lag도 본다. outbox/CDC가 잘 돌아도 소비자가 처리를 못 하면 도착 보장이 의미가 없다.

```bash
kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
  --group payment-service --describe
```

`LAG`이 평소 대비 10배 이상으로 늘면 소비자 인스턴스가 부족하거나 비즈니스 로직이 느려진 거다.

## Outbox 패턴이 부적합한 경우

Outbox는 만능이 아니다. 다음과 같은 경우엔 다른 패턴이 맞다.

**동기 응답이 필요한 RPC 호출.** 결제 게이트웨이를 호출하고 그 응답으로 주문 상태를 결정해야 한다면 outbox로 풀 수 없다. outbox는 fire-and-forget이라 호출 측에서 결과를 기다리지 못한다. 이런 케이스는 직접 동기 호출하되 멱등 키와 재시도 정책으로 풀어야 한다. 비동기로 풀어도 되는 부분(통계, 알림)만 outbox로 빼라.

**짧은 TTL을 가진 이벤트.** 실시간 입찰 가격 알림, 채팅 typing indicator 같이 1초 안에 의미를 잃는 데이터는 outbox로 보내면 안 된다. outbox 워커가 0.5초만 밀려도 이미 의미가 없는 메시지가 발행된다. WebSocket 직접 push나 Redis Pub/Sub가 더 맞다.

**외부 시스템 직접 호출.** 슬랙 알림이나 Twilio SMS처럼 외부 API에 직접 호출이 필요한 작업을 outbox에 넣으면, 그 API의 일시 장애가 outbox 적체로 이어진다. 외부 호출은 별도 워커 큐(예: AWS SQS + 재시도/DLQ)로 분리하고, outbox에는 "알림 발송 요청" 같은 내부 이벤트만 둔다. 그 이벤트를 받은 외부 호출 서비스가 자기 큐로 옮긴 뒤 처리하는 식이다.

**처리 순서가 글로벌하게 보장돼야 하는 경우.** 위에서 봤듯 Kafka는 파티션 단위 순서만 보장한다. 회계 원장처럼 모든 거래의 글로벌 순서가 필요하면 단일 파티션을 써야 하는데, 그러면 outbox의 장점인 처리량이 사라진다. 이런 도메인은 outbox보다 이벤트 소싱 + append-only 저장소가 맞다.

**트랜잭션 자체가 길어지면 안 되는 경우.** outbox INSERT는 비즈니스 트랜잭션을 살짝 늘린다. payload가 크면(예: 100KB JSON) DB 라이팅이 느려진다. 핵심 OLTP에서 응답시간 P99가 빡빡한 API라면 outbox INSERT 자체가 부담이 될 수 있다. 이런 경우엔 payload를 외부 스토리지에 올리고 outbox엔 참조만 두거나, outbox를 별도 DB로 분리한다.

## Outbox 테이블 정리

outbox 테이블도 처리 완료된 레코드가 계속 쌓인다. 폴링 방식에서는 SENT 상태인 레코드를 주기적으로 지워야 하고, CDC 방식에서는 INSERT만 캡처하면 되니까 바로 지워도 된다.

CDC를 쓰는 경우, 아예 outbox 테이블에 레코드를 INSERT한 직후 같은 트랜잭션에서 DELETE하는 방법도 있다. Debezium은 binlog에서 INSERT를 감지하니까, 테이블에 데이터가 남아있을 필요가 없다.

```java
@Transactional
public void createOrder(OrderRequest request) {
    Order order = Order.create(request);
    orderRepository.save(order);

    OutboxEvent event = OutboxEvent.builder()
        .eventId(UUID.randomUUID().toString())
        .aggregateType("Order")
        .aggregateId(order.getId().toString())
        .eventType("OrderCreated")
        .payload(objectMapper.writeValueAsString(new OrderCreatedEvent(order)))
        .build();
    outboxRepository.save(event);
    outboxRepository.delete(event);  // 바로 삭제 — Debezium은 binlog에서 INSERT를 이미 캡처했다
}
```

이렇게 하면 outbox 테이블은 항상 비어있는 상태를 유지한다. 다만 디버깅할 때 outbox 데이터를 직접 조회할 수 없으니, 로그나 Kafka 토픽에서 확인해야 한다. 또한 PostgreSQL에서는 이 패턴이 dead tuple을 만들어 VACUUM 부담이 커지니, `autovacuum_vacuum_scale_factor`를 outbox 테이블에 한해 작게 잡아두는 게 좋다.

## 실무에서 겪는 문제들

**outbox 테이블에 payload를 어떤 형식으로 저장할지 고민된다.** JSON이 가장 무난하다. Avro나 Protobuf는 스키마 레지스트리까지 고려해야 해서 초기에는 부담스럽다. 다만 payload가 크면 DB 용량을 잡아먹으니까, 큰 데이터는 S3 같은 곳에 올리고 참조만 저장하는 방법도 있다.

**Outbox 패턴을 적용하면 코드가 길어진다.** 비즈니스 로직마다 outbox 저장 코드가 붙으니까 반복이 생긴다. Spring의 `@TransactionalEventListener`와 결합하면 좀 줄일 수 있다.

```java
// 도메인 이벤트 발행
@Transactional
public void createOrder(OrderRequest request) {
    Order order = Order.create(request);
    orderRepository.save(order);
    applicationEventPublisher.publishEvent(new OrderCreatedEvent(order));
}

// 이벤트 리스너에서 outbox 저장
@TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
public void handleOrderCreated(OrderCreatedEvent event) {
    outboxRepository.save(OutboxEvent.from(event));
}
```

`BEFORE_COMMIT` 페이즈를 써야 같은 트랜잭션 안에서 outbox가 저장된다. `AFTER_COMMIT`으로 하면 트랜잭션이 커밋된 후에 실행되니까 원자성이 깨진다.

## 폴링 vs CDC 정리

| 항목 | 폴링 | CDC (Debezium) |
|------|------|----------------|
| 지연 시간 | 폴링 간격만큼 (보통 1~5초) | 거의 실시간 (수백 ms) |
| DB 부하 | 주기적 쿼리 부하 | binlog/WAL 읽기라 부하 적음 |
| 인프라 복잡도 | 낮음 (애플리케이션 코드만) | 높음 (Kafka Connect, Debezium 운영 필요) |
| 순서 보장 | 직접 관리 필요 | binlog/WAL 순서 + 파티션 키 |
| 장애 복구 | 단순 (다시 폴링하면 됨) | 복잡 (offset, schema_history, snapshot 모드) |
| 분산 락 | `SKIP LOCKED` / advisory lock / Redis | 커넥터 fencing이 알아서 처리 |
| 모니터링 지점 | PENDING 카운트, 워커 로그 | Connect status, source lag, slot lag |
| 적합한 상황 | 트래픽 적고 인프라 단순 | 트래픽 많고 실시간성 중요 |

초기에는 폴링으로 시작하고, 트래픽이 늘면 CDC로 전환하는 경우가 많다. 폴링은 구현이 쉽고 디버깅도 편하다. outbox 테이블을 직접 조회하면 어떤 메시지가 안 나갔는지 바로 확인할 수 있다. CDC는 인프라 비용이 들지만, 한 번 정착하면 애플리케이션 코드가 깨끗해진다. 어느 쪽이든 소비자 dedup, 운영 모니터링, 실패 복구 절차는 똑같이 준비해야 한다.
