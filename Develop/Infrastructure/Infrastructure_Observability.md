# 인프라 관찰성 설계

## 1. 시작하기 전에

관찰성(Observability)이라는 단어는 흔히 모니터링과 섞여 쓰인다. 둘은 다르다. 모니터링은 "내가 미리 정해둔 지표가 정상 범위 안에 있는가"를 본다. 관찰성은 "처음 겪는 장애가 났을 때, 외부에서 관측한 신호만 가지고 내부에서 무슨 일이 일어나는지 추론할 수 있는가"를 묻는다. 모니터링은 알려진 실패에 강하고, 관찰성은 모르는 실패에 강하다.

5년차쯤 백엔드를 하면서 관찰성에 대해 배운 건 세 가지다. 첫째, 도구를 깐다고 관찰성이 생기지 않는다. Prometheus와 Grafana, Loki, Tempo를 다 깔아도 새벽 3시에 장애가 났을 때 어디부터 봐야 할지 모르면 그건 관찰성이 없는 것이다. 둘째, 알림은 많을수록 나빠진다. 한 달에 200개씩 페이저가 울리면 정작 진짜 장애 때는 아무도 안 본다. 셋째, 관찰성은 코드와 같이 자라야 한다. 새 기능을 만들 때 메트릭과 로그를 같이 심지 않으면 두 달 뒤에는 아무도 손대지 않는다.

이 문서는 특정 도구를 다루지 않는다. 메트릭/로그/트레이스의 3축을 어떻게 나눠 쓰는지, 골든 시그널과 SLI/SLO를 어떻게 정의하는지, 알림 임계값을 어떻게 잡고 노이즈를 어떻게 줄이는지, on-call이 실제로 디버깅 가능한 대시보드는 어떤 모습인지를 다룬다. 도구는 바뀌어도 이 원칙은 거의 그대로다.

```
        관찰성 (Observability)
                │
   ┌────────────┼────────────┐
   │            │            │
 메트릭        로그         트레이스
(Metrics)    (Logs)       (Traces)
   │            │            │
"얼마나"      "무엇이"      "어디서"
                │
        ┌───────┴───────┐
        │               │
   SLI / SLO         알림 (Alerting)
        │               │
        └───────┬───────┘
                │
            대시보드
        (on-call 디버깅)
```

## 2. 메트릭, 로그, 트레이스 — 세 축의 역할 분담

### 2.1 왜 셋이 필요한가

처음 관찰성을 배울 때 흔히 하는 실수가 "로그만 잘 쌓으면 다 된다"고 생각하는 것이다. 로그는 만능처럼 보인다. 무엇이든 적을 수 있고, 적으면 나중에 검색할 수 있다. 그런데 트래픽이 늘어나면 로그만으로는 두 가지 질문에 답하기 어렵다. "지금 시스템이 얼마나 아픈가?"와 "이 요청 하나는 어디서 막혔는가?"다.

메트릭, 로그, 트레이스는 답하는 질문이 다르다.

- **메트릭**은 "얼마나"를 답한다. 초당 요청 수, p99 지연, 에러율, CPU 사용률 같은 숫자 시계열이다. 집계되어 있어서 저장 비용이 싸고 빠르게 질의된다.
- **로그**는 "무엇이"를 답한다. 특정 시점에 어떤 이벤트가 일어났는지를 문장으로 적는다. 자유도가 높고, 디버깅에 가장 직접적인 단서다. 대신 양이 많아지면 비싸진다.
- **트레이스**는 "어디서"를 답한다. 요청 하나가 여러 서비스를 오가면서 각 구간에 얼마나 머물렀는지를 보여준다. 마이크로서비스에서는 트레이스 없이는 병목을 찾기가 매우 어렵다.

세 가지가 따로 노는 게 아니라 서로를 가리켜야 한다. p99 지연이 튄 메트릭을 보고 → 해당 시간대 느린 트레이스를 골라 → 그 트레이스의 특정 span에 연결된 로그를 본다. 이 흐름이 끊기면 관찰성이 아니라 데이터 더미가 된다.

### 2.2 메트릭의 특성과 함정

메트릭은 시계열 데이터베이스에 저장한다. Prometheus가 대표적이지만, InfluxDB, VictoriaMetrics, Mimir, M3, Datadog의 시계열 백엔드 등 선택지는 많다. 공통점은 "라벨이 붙은 숫자가 시간순으로 쌓인다"는 것이다.

함정은 라벨 카디널리티다. `http_requests_total{method, path, status, user_id}` 같은 식으로 라벨에 사용자 ID를 넣으면 시계열 개수가 사용자 수만큼 폭발한다. 시계열 1개당 메모리와 디스크가 따라붙기 때문에, 카디널리티가 백만을 넘으면 시계열 DB가 OOM으로 죽거나 질의가 수십 초씩 걸린다.

라벨을 정할 때 기준은 단순하다. "이 라벨로 그룹화해서 보고 싶은가?"가 yes이고, "값의 종류가 유한한가?"도 yes일 때만 라벨로 둔다. `path`는 흔히 함정인데, `/users/123`, `/users/456`처럼 ID가 path에 들어가면 카디널리티가 폭발한다. 라우팅 후의 path 템플릿(`/users/:id`)으로 정규화해서 메트릭에 넣어야 한다.

또 하나 자주 놓치는 게 percentile 메트릭이다. p99 지연을 보고 싶다고 클라이언트마다 따로 percentile을 계산해서 보내면, 그건 집계할 수가 없다. p99 두 개를 평균낸다고 전체 p99가 나오는 게 아니기 때문이다. histogram으로 bucket을 쪼개서 보내고, 질의 시점에 `histogram_quantile()`로 계산하는 방식이 정석이다. Prometheus를 쓴다면 `_bucket`, `_sum`, `_count` 세 메트릭이 같이 떨어지는 그 패턴이다.

### 2.3 로그의 특성과 함정

로그는 가장 자유로운 매체다. 자유로워서 가장 망치기 쉽다.

구조화되지 않은 로그(`printf` 스타일)는 grep으로 추적할 때는 빠르지만, 집계나 필터링을 하려면 정규식으로 파싱해야 한다. 트래픽이 늘어나면 정규식 파싱이 병목이 되고, 누가 로그 포맷을 살짝만 바꿔도 모든 대시보드가 깨진다.

JSON 로그처럼 구조화된 로그가 표준이 된 이유다. 키-값 형태로 적어두면 인덱싱과 질의가 빠르다. 다만 모든 로그를 다 인덱싱하면 저장 비용이 폭발하기 때문에, 인덱싱 대상 필드를 정해두는 게 보통이다. `trace_id`, `service`, `level`, `status_code` 같은 검색에 자주 쓰는 필드만 인덱싱하고 나머지는 raw로 둔다.

로그에서 가장 흔한 사고는 "민감 정보가 그대로 찍히는 일"이다. 비밀번호, 토큰, 카드 번호, 주민번호. 인입되는 요청을 통째로 찍는 미들웨어를 별 생각 없이 깔았다가 GDPR 위반으로 조사를 받는 케이스가 실제로 있다. 로그 라이브러리 레벨에서 마스킹 룰을 강제하거나, 인입 직후 정규화 단계에서 민감 필드를 잘라내야 한다.

로그 보관 기간도 비용 문제다. ELK든 Loki든 Datadog이든 30일 보관과 90일 보관은 비용이 두세 배씩 차이난다. 디버깅용 로그는 7~14일, 감사용 로그는 1년 이상 등으로 등급을 나누고 저장 위치를 분리하는 게 일반적이다.

### 2.4 트레이스의 특성과 함정

트레이스는 한 요청이 시스템 안에서 어떻게 흘러갔는지를 span 트리로 보여준다. 사용자 요청이 API Gateway → 인증 서비스 → 주문 서비스 → 결제 서비스 → DB를 거쳤다면, 각 단계가 span이 된다. 부모-자식 관계가 있고, 각 span에 시작/끝 시간과 태그가 붙는다.

트레이스 도구는 OpenTelemetry가 사실상 표준이 됐다. 백엔드는 Jaeger, Tempo, Honeycomb, Datadog APM 등 다양하다.

트레이스의 가장 큰 함정은 두 가지다.

첫째, 샘플링. 모든 요청을 다 트레이스로 남기면 트레이스 백엔드가 터진다. 보통 1~10% 샘플링한다. 그런데 단순 비율 샘플링을 쓰면 "에러난 요청은 거의 항상 못 잡는다." 에러는 전체의 0.1%인데 1% 샘플링이면 99% 확률로 놓친다. tail-based 샘플링(요청이 끝난 후 status가 에러면 무조건 보존)이나, head-based + force-sampling(에러 발생 시 강제로 샘플링) 같은 전략이 필요하다.

둘째, context propagation 누락. 서비스 A가 B를 호출하고 B가 C를 호출할 때, 같은 `trace_id`가 전파돼야 트레이스가 트리로 연결된다. HTTP 호출은 헤더로 자동 전파되는데, 메시지 큐(Kafka, SQS)나 비동기 작업(워커, 배치)에서는 직접 trace context를 페이로드에 실어야 한다. 이걸 놓치면 트레이스가 끊겨서 "DB 쿼리는 느린데 어디서 호출했는지 모름" 상태가 된다.

### 2.5 셋을 연결하는 ID

세 축을 잇는 접착제가 `trace_id`다. 메트릭에 exemplar로 trace_id를 붙이고, 로그에 trace_id를 같이 적고, 트레이스 자체가 trace_id로 식별되면 어디서 시작하든 다른 둘로 점프할 수 있다.

```python
import logging
from opentelemetry import trace

logger = logging.getLogger(__name__)
tracer = trace.get_tracer(__name__)

def process_order(order_id):
    with tracer.start_as_current_span("process_order") as span:
        span.set_attribute("order.id", order_id)

        ctx = span.get_span_context()
        trace_id = format(ctx.trace_id, "032x")
        span_id = format(ctx.span_id, "016x")

        logger.info(
            "주문 처리 시작",
            extra={
                "trace_id": trace_id,
                "span_id": span_id,
                "order_id": order_id,
            },
        )

        try:
            validate(order_id)
            charge(order_id)
            ship(order_id)
        except Exception as e:
            span.record_exception(e)
            span.set_status(trace.Status(trace.StatusCode.ERROR))
            logger.error(
                "주문 처리 실패",
                extra={
                    "trace_id": trace_id,
                    "span_id": span_id,
                    "order_id": order_id,
                    "error": str(e),
                },
            )
            raise
```

이 코드의 핵심은 trace_id를 로그에 항상 같이 적는다는 점이다. 대시보드에서 에러 로그를 클릭하면 trace_id로 트레이스 페이지로 점프할 수 있고, 트레이스의 span ID로 다시 로그를 필터링할 수 있다.

## 3. 골든 시그널 — Latency, Traffic, Errors, Saturation

### 3.1 왜 네 가지인가

Google SRE 책에서 정리된 골든 시그널은 "사용자 관점의 서비스 헬스를 4개 신호로 요약할 수 있다"는 주장이다. 처음 들으면 너무 단순해 보이는데, 실제로 운영을 해보면 이 네 가지가 거의 모든 장애의 첫 신호다.

- **Latency**: 요청을 처리하는 데 걸리는 시간. 성공 요청과 실패 요청의 지연을 분리해서 봐야 한다.
- **Traffic**: 시스템이 받는 요청량. RPS, QPS, 또는 도메인별로는 transaction/min, message/sec 같은 단위.
- **Errors**: 실패한 요청 비율. HTTP 5xx만이 아니라 의미적 실패(예: 결제 실패)도 포함해야 한다.
- **Saturation**: 시스템이 얼마나 가득 차 있는가. CPU, 메모리, 디스크 IO, 큐 깊이 등. 100%에 가까울수록 위험하다.

이 네 가지를 서비스마다, 엔드포인트마다, 의존성마다 측정해두면 장애가 났을 때 "어디가 아픈지"를 빠르게 좁힐 수 있다.

### 3.2 Latency를 잘못 재는 흔한 방식

평균 지연(mean latency)만 보면 거의 항상 속는다. 99%의 요청이 50ms, 1%가 5초여도 평균은 100ms 정도다. 사용자 1%는 5초를 기다리는데 그래프는 평온해 보인다.

지연은 분위수(percentile)로 봐야 한다. p50, p95, p99, p99.9 정도를 같이 띄워두는 게 표준이다. p50은 "보통의 사용자가 어떻게 느끼는가", p99는 "100명 중 1명은 어떻게 느끼는가", p99.9는 "1000명 중 1명"이다. 트래픽이 많은 서비스라면 p99.9에서 1초가 넘어가는 게 매일 수만 명에게 영향을 준다는 뜻이다.

또 흔한 함정이 성공/실패를 합쳐서 percentile을 계산하는 것이다. 타임아웃 30초로 잘린 요청 100개와 정상 100ms 요청 만 개를 같이 넣으면 p99는 30초로 보인다. 그런데 정상 요청만 보면 p99는 200ms일 수 있다. "에러가 늘었는가, 응답이 느려졌는가"는 다른 문제다. 메트릭은 status별로 분리해서 봐야 한다.

```
# Prometheus 메트릭 예시
http_request_duration_seconds_bucket{
  service="order-api",
  endpoint="/orders",
  method="POST",
  status="200",
  le="0.1"
} 12345

http_request_duration_seconds_bucket{
  service="order-api",
  endpoint="/orders",
  method="POST",
  status="500",
  le="0.1"
} 23
```

status 라벨을 분리해두면 질의 시점에 `{status="2.."}` 또는 `{status="5.."}`로 골라서 percentile을 뽑을 수 있다.

### 3.3 Traffic은 단순 RPS만이 아니다

Traffic은 보통 RPS로 잡지만, 도메인에 따라 더 의미 있는 단위가 있다.

- API 서비스: requests/sec
- 메시지 큐 소비자: messages/sec, consumer lag
- 배치 작업: jobs/min
- 결제 시스템: transactions/min, transaction amount/sec
- 검색: queries/sec, average result count

중요한 건 "용량 계획과 비교할 수 있는 단위"여야 한다는 점이다. 시스템이 5000 RPS까지 견딘다는 걸 알고 있는데 실제 트래픽이 4500을 찍고 있으면 "곧 한계"라는 게 한눈에 보인다. 단위가 모호하면 saturation과 연결이 안 된다.

Traffic은 서비스 헬스가 아니라 외부 변수다. 외부 변수가 평소와 다르게 움직이면(트래픽이 갑자기 30% 빠진다든지) 그 자체가 장애 신호다. "갑자기 조용해진" 것도 알려져야 한다.

### 3.4 Errors는 status 코드 너머를 봐야 한다

HTTP 5xx만 에러로 잡으면 빠뜨리는 게 많다. 200 OK로 응답하면서 body에 `{"error": "card_declined"}`를 담아 보내는 결제 응답이 대표적이다. HTTP 레벨에서는 성공이지만 비즈니스 레벨에서는 실패다.

에러는 두 층으로 봐야 한다.

- **인프라 에러**: HTTP 5xx, gRPC UNAVAILABLE, DB connection refused 같은 것. 인프라가 아프다는 신호.
- **비즈니스 에러**: 결제 거절, 재고 부족, 인증 실패 같은 도메인 실패. 사용자 경험이 나쁘다는 신호.

비즈니스 에러는 평소에도 0이 아니다. 결제 거절률이 평소 3%인데 갑자기 15%가 됐다면, 카드사 쪽에 문제가 있거나 우리 호출 방식이 바뀐 것이다. 평균값이 아니라 변화율을 봐야 한다.

### 3.5 Saturation은 가장 늦게 알아챈다

Saturation은 "시스템이 얼마나 가득 찼는가"인데, 이게 가장 늦게 알람이 울리는 경우가 많다. CPU 70%까지는 별 문제 없이 가다가 80%를 넘으면 갑자기 지연이 튀고, 90%면 cascading failure가 시작된다. 선형이 아니라 무릎(knee)이 있는 곡선이다.

Saturation은 자원별로 다르게 잡아야 한다.

- **CPU**: 사용률 + run queue 길이. 사용률 50%여도 run queue가 길면 throttling 직전.
- **메모리**: 사용률 + swap + GC 빈도(JVM, Go, Python).
- **디스크**: 사용률 + IO wait + IOPS 한계 대비 비율.
- **네트워크**: bandwidth + packet drop + retransmission rate.
- **DB**: connection pool 사용률, slow query 수, lock wait 시간.
- **큐**: 대기 메시지 수, consumer lag, oldest message age.

Saturation 알람은 "한계의 70~80%"에 거는 게 보통이다. 100%에서 울리면 이미 늦었다. Connection pool이 50개고 평소 30개를 쓴다면, 40개부터 알람을 울려야 trouble shooting 시간을 번다.

## 4. SLI와 SLO — 알람의 근거 만들기

### 4.1 SLI와 SLO는 왜 필요한가

알람을 임의의 임계값으로 거는 게 일반적이지만, 곧 문제가 생긴다. "왜 p99 500ms에서 알람을 거나? 왜 에러율 1%인가?" 누가 물어보면 답이 궁색하다. 그러다 알람이 자꾸 울리면 "임계값을 좀 올리자"가 되고, 결국 알람이 무의미해진다.

SLI와 SLO는 이 임계값에 근거를 준다.

- **SLI (Service Level Indicator)**: 서비스 품질을 측정하는 지표. "성공 요청 비율", "p99 지연이 500ms 미만인 비율" 같이 비율로 정의되는 경우가 많다.
- **SLO (Service Level Objective)**: SLI가 충족해야 하는 목표. "30일간 가용성 99.9%", "p99 지연 500ms를 30일간 99% 이상" 같은 식.
- **SLA (Service Level Agreement)**: SLO를 외부 고객과 약속한 것. SLA를 깨면 환불이나 크레딧 같은 페널티가 따라온다. SLO보다 보통 한 자리 낮게 잡는다(내부 SLO 99.9%, 외부 SLA 99.5%).

SLI는 사용자가 실제로 느끼는 것이어야 한다. CPU 사용률은 SLI가 아니다. "결제 요청이 3초 안에 성공하는 비율"이 SLI다.

### 4.2 Error Budget — SLO를 운영의 언어로

SLO 99.9%는 한 달에 43분의 다운타임을 허용한다는 뜻이다. 이 43분이 error budget이다. 한 달 동안 이 예산을 다 쓰지 않았다면, 그만큼은 위험한 배포를 해도 된다. 다 썼다면 새 기능 배포를 멈추고 안정화에 집중한다.

이 사고방식은 개발팀과 운영팀의 끝없는 논쟁을 깬다. "왜 배포를 못 하게 하느냐"는 개발팀과 "왜 자꾸 장애를 내느냐"는 운영팀 사이에서, error budget은 객관적인 신호다.

```
SLO: 99.9% 가용성 (30일)
허용 다운타임: 43분/월

이번 달 누적 다운타임:
- Week 1: 12분 (배포 사고)
- Week 2:  5분 (DB failover)
- Week 3: 18분 (트래픽 폭증)
- Week 4:  현재 진행 중

남은 예산: 8분
조치: 비필수 배포 중단, 위험한 변경 다음 달로 연기
```

Error budget이 90% 이상 남았다면 카오스 엔지니어링을 해볼 시점이다. 의도적으로 약점을 찔러서 미리 찾아내는 비용을 예산 안에서 쓸 수 있다.

### 4.3 SLI를 어떻게 정의하나

좋은 SLI에는 세 가지 요소가 있다.

- **분자**: 성공한 이벤트 수
- **분모**: 전체 이벤트 수
- **시간 창**: 측정 윈도우 (30일이 흔하다)

예를 들어 결제 API의 가용성 SLI는 이렇게 정의할 수 있다.

```
SLI = (status가 2xx 또는 의도된 4xx인 결제 요청 수)
    / (모든 결제 요청 수)
    , 측정 윈도우 = 30일
```

여기서 의도된 4xx(`400 Bad Request`, `403 Forbidden` 같이 클라이언트 잘못)는 분자에 포함시킨다. 우리 서비스 잘못이 아니기 때문이다. 반대로 `429 Too Many Requests`는 분모에서 빼야 하는지 분자에 넣어야 하는지 도메인마다 다르다. 우리가 의도적으로 막은 거면 분자, 우리가 못 받아낸 거면 분모에서 빼거나 분자에서 빼야 한다.

지연 SLI는 이렇게 쓴다.

```
SLI = (응답이 500ms 안에 도착한 성공 요청 수)
    / (모든 성공 요청 수)
    , 측정 윈도우 = 30일
SLO = 99% 이상
```

"p99이 500ms 미만"과 "500ms 미만 비율이 99%"는 비슷하지만 알람 거는 방식이 다르다. 후자가 SLO 친화적이다.

### 4.4 Burn Rate 알람

SLO를 정해두면 burn rate 기반 알람이 가능하다. Error budget을 정해진 속도보다 빨리 소진하면 알람을 울리는 방식이다.

```
SLO: 99.9% (30일)
예산: 43.2분 다운타임

1배 burn rate: 30일에 예산을 다 쓰는 속도 = 0.1% 에러율
2배: 15일에 다 씀
10배: 3일에 다 씀
14.4배: 5%만 남기고 마지막 5분이면 24시간 안에 다 씀
```

전형적인 multi-window burn rate 알람은 두 윈도우를 조합한다.

- **빠른 윈도우 (5분)**: 14.4배 burn rate가 5분 연속이면 즉시 페이지
- **느린 윈도우 (1시간)**: 6배 burn rate가 1시간 연속이면 페이지

빠른 윈도우만 보면 트래픽 스파이크에 false positive가 자주 난다. 느린 윈도우만 보면 진짜 장애를 1시간 동안 못 알아챈다. 둘을 AND로 묶으면 노이즈가 줄고 진짜 사고는 빨리 잡힌다.

## 5. 알림 — 새벽에 깨울 가치가 있는가

### 5.1 알림은 행동을 요구해야 한다

알림 설계의 한 줄 원칙은 이거다. "이 알림이 울리면, 누군가가 즉시 행동해야 하는가?" yes면 페이지(전화/SMS), no면 티켓이나 채팅 알림으로 충분하다.

이 원칙을 안 지키면 알림이 너무 많아진다. CPU 80%를 페이지로 보내면 새벽에 일어났더니 그냥 트래픽 스파이크였다. 두세 번 그러면 다음에는 무시한다. 그러다 진짜 장애도 무시한다.

알림은 두 종류로 나눠 생각하는 게 좋다.

- **Symptom alert (증상 알림)**: 사용자가 실제로 영향을 받았거나 곧 받을 신호. SLO burn rate, 에러율 급증, 응답 지연 급증, 큐 적체. 페이지로 보낸다.
- **Cause alert (원인 알림)**: 사용자 영향은 아직 없지만 곧 그렇게 될 가능성. CPU 80%, 디스크 70%, 메모리 누수 추세. 페이지하지 않고 티켓이나 채팅으로.

원인 알림을 줄이고 증상 알림을 늘리는 게 일반적인 방향이다. 원인은 너무 많고(CPU, 메모리, 디스크, 네트워크, 의존성…), 그중 어떤 것이 실제 증상으로 이어질지 모른다. 증상은 적고 사용자 영향과 직결된다.

### 5.2 노이즈 알림 줄이기

알림이 많은 시스템의 공통 패턴이 있다.

**같은 원인으로 알림이 여러 개 울린다.** DB가 죽으면 그 DB를 쓰는 모든 서비스에서 에러 알림이 울린다. 30개 서비스가 있으면 30번 페이지가 온다. 알림 그룹화(Alertmanager의 group_by, PagerDuty의 alert grouping)로 같은 원인의 알림은 묶어서 1번만 보낸다.

**임계값이 너무 민감하다.** "5분간 에러율 1% 이상"이면 트래픽이 적을 때 1건만 실패해도 알람이 울린다. "5분간 에러 건수가 10건 이상이고 에러율이 1% 이상"처럼 절대 수와 비율을 같이 본다.

**Flapping 알림.** 임계값을 살짝 넘었다 안 넘었다 반복하면 알림이 계속 울린다. 임계값에 hysteresis를 주거나(예: 1%에서 울리고 0.5%에서 해제), 일정 시간 이상 지속될 때만 울린다.

**오래된 알림.** 6개월 전에 만든 알림이 한 번도 의미 있게 울린 적이 없으면 지운다. "혹시 모르니 둬두자"가 알림의 가장 큰 적이다.

분기마다 한 번씩 알림 audit을 하는 게 좋다. "지난 3개월간 울린 알림 목록, 각각의 실제 조치 여부"를 뽑아서, 한 번도 조치가 필요하지 않았던 알림은 임계값을 올리거나 페이지 → 채팅으로 강등하거나 삭제한다.

### 5.3 알림에는 컨텍스트가 있어야 한다

새벽 3시에 "Order API error rate high"라는 알림만 오면, 일어나서 노트북을 켜고 대시보드를 찾고 무슨 일이 일어났는지 처음부터 파악해야 한다. 5분 이상 걸린다.

좋은 알림에는 다음이 같이 와야 한다.

- **무엇이**: "Order API 에러율이 5%를 5분간 초과"
- **얼마나 심각한가**: "현재 분당 200건 실패. 평소 5건."
- **언제부터**: "12분 전부터 시작, 지속 중"
- **어디**: "ap-northeast-2 리전, order-api 서비스"
- **연관된 변경**: "10분 전 v1.42 배포"
- **링크**: 대시보드 URL, 트레이스 검색 URL, 런북 URL

이걸 알림 템플릿에 미리 넣어두면, on-call이 일어나기 전에 어디부터 봐야 할지 감을 잡는다.

### 5.4 런북(Runbook)

알림 하나마다 런북이 있어야 한다는 말은 자주 듣지만 실제로 갖춰진 곳은 드물다. 런북이 없으면 매번 새벽 3시에 "이게 뭐였더라"부터 시작한다.

런북에 들어가야 하는 것:

- **이 알림이 의미하는 것** (사용자 영향 포함)
- **확인할 대시보드**
- **흔한 원인 3~5가지**
- **각 원인별 진단 명령어/쿼리**
- **임시 조치 (롤백, 트래픽 차단, capacity 늘리기)**
- **누구에게 에스컬레이션할지**

런북은 위키 어딘가에 묻어두면 안 본다. 알림 자체에 URL이 직접 들어가야 한다.

## 6. On-call이 실제로 디버깅 가능한 대시보드

### 6.1 대시보드는 두 종류로 나눈다

대시보드를 "예쁘게 만드는 데" 시간을 쓰면 안 된다. 목적이 분명한 두 종류로 나눠 만든다.

- **개요 대시보드 (Overview)**: 서비스 헬스를 한눈에. 골든 시그널 4개 + 의존성 상태. 평시에 쳐다보는 용도.
- **디버깅 대시보드 (Drill-down)**: 장애 시 원인을 좁히는 용도. 엔드포인트별, 인스턴스별, 의존성별로 분해 가능해야 함.

개요 대시보드는 모니터에 띄워둬도 부담 없는 정도여야 한다. 차트 8~12개를 넘기지 않는다. 디버깅 대시보드는 정보가 많아도 되지만, 필터로 분해 가능해야 한다.

### 6.2 USE와 RED 패턴

대시보드 구성에 흔히 쓰이는 두 가지 패턴이 있다.

**RED 메서드 (서비스용)**:
- **R**ate: 초당 요청 수
- **E**rrors: 에러율
- **D**uration: 지연 분위수 (p50, p95, p99)

서비스의 외부 동작을 본다. API, RPC 서버, 워커 서비스에 잘 맞는다.

**USE 메서드 (자원용)**:
- **U**tilization: 사용률 (CPU, 메모리, 디스크, 네트워크)
- **S**aturation: 가득 참 정도 (run queue, 큐 길이, swap)
- **E**rrors: 자원 에러 (디스크 IO 에러, 네트워크 drop)

서버, 데이터베이스, 디스크 같은 자원을 본다.

서비스 하나의 개요 대시보드는 보통 이렇게 짠다.

```
┌─────────────────────────────────────────────────────┐
│ Order API - Overview              [region: all ▼]   │
├─────────────────────────────────────────────────────┤
│                                                     │
│  RPS (5분 윈도우)         |  Error Rate (1h)        │
│  현재: 1,240 req/s        |  현재: 0.12%            │
│  [라인 차트]              |  [라인 차트, 임계선 1%] │
│                                                     │
│  Latency p50/p95/p99      |  SLO Burn Rate          │
│  50ms / 180ms / 420ms     |  0.8x (정상)            │
│  [라인 차트]              |  [라인 차트]            │
│                                                     │
├─────────────────────────────────────────────────────┤
│ Dependencies                                        │
│                                                     │
│  PostgreSQL: ✓ (latency 8ms)                        │
│  Redis: ✓ (latency 1ms)                             │
│  Payment Service: ⚠ (error rate 2.1%)               │
│  Notification Queue: ✓ (lag 0)                      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

이 화면을 보면 5초 안에 "지금 시스템이 정상인가 아닌가"를 판단할 수 있어야 한다.

### 6.3 디버깅 대시보드는 필터가 핵심

장애가 났을 때 디버깅 대시보드는 다음을 가능하게 해야 한다.

- **엔드포인트별 분해**: 어느 API가 문제인지. `/orders`가 느린지, `/users`가 느린지.
- **인스턴스별 분해**: 한 인스턴스만 문제인지, 전체인지. 한 인스턴스만 문제면 그 인스턴스만 빼면 된다.
- **status별 분해**: 5xx가 늘었는지, 4xx가 늘었는지.
- **버전별 분해**: 새 배포 버전만 에러가 나는지. 카나리 배포 시 필수.
- **사용자/테넌트별 분해**: 특정 고객만 영향인지.

Grafana의 variable, Datadog의 tag, Honeycomb의 BubbleUp 같은 기능을 잘 쓰면 클릭 몇 번으로 좁혀 들어간다.

### 6.4 자주 보는 안티패턴

**모든 메트릭을 한 화면에.** 차트가 50개씩 있는 대시보드는 아무도 안 본다. 의미 있는 것만 남긴다.

**평균값만 보여주는 지연 차트.** percentile이 없으면 거의 항상 속는다.

**임계값 라인이 없는 차트.** 숫자가 떠 있어도 "이게 정상인지 아닌지" 모른다. SLO와 알람 임계값을 차트에 라인으로 그려둔다.

**현재 값만 보여주는 게이지.** 추세가 안 보인다. 게이지 옆에 작은 trend chart를 같이 띄운다.

**도메인 용어가 없는 차트 제목.** "rate(http_requests_total[5m])"보다 "주문 API 초당 요청 수"가 새벽 3시에 도움이 된다.

### 6.5 한 화면에 시간축 동기화

장애 분석을 할 때 가장 자주 하는 일이 "이 시각에 무슨 일이 동시에 일어났는가"를 보는 것이다. 차트들의 시간축이 동기화되어 있어야 한다. Grafana는 기본으로 그렇게 동작하지만, 별도 도구를 섞으면 시각 비교가 어려워진다.

배포 이벤트, 인프라 변경, 외부 incident도 차트에 annotation으로 같이 찍어두면 인과 분석이 빠르다. "에러율이 튀기 시작한 정확히 그 시각에 배포가 있었다"는 게 한 화면에 보이면, 의심의 출발점이 정해진다.

## 7. 운영하면서 실제로 마주치는 문제들

### 7.1 메트릭이 너무 비싸졌을 때

Prometheus든 Datadog이든, 메트릭 비용이 갑자기 두 배가 됐다는 청구서를 받는 일이 있다. 거의 항상 라벨 카디널리티 폭발이다.

진단은 단순하다. 시계열 수를 라벨별로 세본다.

```promql
topk(10, count by (__name__)({__name__=~".+"}))
```

가장 시계열이 많은 메트릭을 찾고, 그 메트릭의 라벨 중 카디널리티가 높은 것을 본다. 보통 한두 개의 라벨이 범인이다. user_id가 들어갔거나, path가 정규화되지 않았거나, request_id가 라벨로 들어갔거나.

조치는 라벨을 빼거나, allow-list로 값을 제한하거나, 메트릭 자체를 삭제하는 것이다. 카디널리티가 한번 폭발하면 과거 데이터까지 영향을 주기 때문에, retention을 줄여서 빠르게 잘라낸다.

### 7.2 로그가 너무 많아졌을 때

DEBUG 레벨 로그를 프로덕션에 켜둔 채로 잊어버리면 한 달 뒤에 비용이 폭발한다. 흔한 패턴은 이렇다.

- 새 기능 디버깅용으로 INFO에 잔뜩 찍어둠
- 배포 후 잠잠해서 까먹음
- 트래픽이 늘면서 로그가 같이 늘어남
- 청구서 충격

로그 비용 관리는 두 갈래다. 첫째, 무엇을 줄일지 보는 것. 서비스별/레벨별 로그 양 메트릭을 만들어두면 어떤 서비스가 갑자기 많이 찍는지 보인다. 둘째, sampling. 모든 요청 로그를 다 남길 필요는 없다. 성공 요청은 1% 샘플링하고 에러 요청은 100% 남기는 식.

다만 sampling은 디버깅 능력을 떨어뜨린다. "이 사용자의 정확한 요청 흐름을 보고 싶다"가 안 된다. 보통은 trace는 sampling하더라도 로그는 가능한 한 다 남기는 쪽을 택하고, 로그 양은 INFO 레벨 정리로 줄인다.

### 7.3 트레이스가 끊겨 보일 때

트레이스를 봤는데 중간이 뚝 끊겨 있는 경우. 대부분 두 가지 중 하나다.

첫째, context propagation 누락. HTTP 호출은 자동인데 메시지 큐, 비동기 워커, 스케줄러는 직접 trace context를 실어야 한다. Producer에서 message header에 traceparent를 넣고, consumer에서 꺼내 새 span의 부모로 설정한다.

```python
from opentelemetry.propagate import inject, extract

# Producer
def publish(message):
    headers = {}
    inject(headers)
    queue.send(payload=message, headers=headers)

# Consumer
def consume(msg):
    ctx = extract(msg.headers)
    with tracer.start_as_current_span("consume", context=ctx) as span:
        process(msg)
```

둘째, sampling 설정 불일치. 서비스 A는 100% sampling인데 B는 1% sampling이면, B를 호출한 트레이스의 99%가 B에서 끊겨 보인다. tail-based sampling으로 통일하거나, parent-based sampling으로 부모의 sampling 결정을 따르게 한다.

### 7.4 알림이 울려도 일어나지 않게 될 때

특정 사람의 페이저가 한 달에 100번 울리면, 그 사람은 결국 무시하게 된다. 이건 인간의 문제가 아니라 시스템의 문제다.

대응 방법은 두 가지다. 첫째, 알림 audit. 위에서 설명한 대로 조치 필요 없는 알림을 정리한다. 둘째, on-call rotation을 충분한 인원으로 돌린다. 두세 명이 한 달 내내 돌리면 누군가는 항상 피곤하다. 6~8명 정도가 일주일씩 돌리는 게 일반적이다.

페이저 통계를 지표화해서 같이 본다. "주당 평균 페이지 수", "한밤중(00~06시) 페이지 비율", "false positive 비율". 이 숫자가 나빠지면 SRE가 우선순위를 잡고 개선한다. 페이지 통계 자체가 SLI가 된다.

### 7.5 새 서비스를 만들 때 관찰성 체크포인트

새 서비스를 만들 때 관찰성을 처음부터 같이 깔지 않으면, 두 달 뒤에 추가하는 비용이 세 배가 된다. 처음부터 다음이 있어야 한다.

서비스 코드 안에:
- 구조화 로깅 (JSON)
- trace_id가 로그에 자동으로 들어감
- HTTP/gRPC 입출구에 latency/error 메트릭 자동 수집
- 외부 호출(DB, API)에 span 자동 생성
- 헬스체크 엔드포인트 (`/healthz`, `/readyz` 분리)

배포 시:
- 메트릭 scrape 대상 등록
- 로그 수집 파이프라인 연결
- 기본 대시보드 자동 생성 (서비스 이름만 넣으면 RED 대시보드가 생성되는 식)
- 기본 SLO 알람 자동 등록

이걸 platform 팀이 라이브러리/템플릿으로 제공하면, 새 서비스마다 따로 챙길 필요가 없다. 관찰성을 "기본값"으로 만들어야 운영 가능한 시스템이 된다.

## 8. 마무리

관찰성은 도구를 깔고 끝나는 일이 아니다. 메트릭/로그/트레이스를 어떻게 나누고, 골든 시그널을 어떻게 측정하고, SLO를 어떻게 정의하고, 알림을 어떻게 다듬고, 대시보드를 어떻게 구성하는가 — 이 모든 게 합쳐져야 새벽 3시에 일어났을 때 시스템이 무슨 말을 하고 있는지 알 수 있다.

가장 자주 만나는 실패 패턴은 "도구는 다 깔았는데 새벽 3시에는 어디부터 봐야 할지 모름"이다. 이걸 피하려면 평소에 의식적으로 훈련해야 한다. 게임데이를 열어서 의도적으로 장애를 일으키고, on-call 후보들이 대시보드만 보고 원인을 찾을 수 있는지 확인한다. 못 찾으면 대시보드와 알림을 고친다.

관찰성 시스템 자체가 망가질 수도 있다는 점도 잊으면 안 된다. Prometheus가 죽으면 알람이 안 울린다. Loki가 죽으면 로그를 못 본다. 관찰성 시스템은 관찰 대상보다 더 안정적이어야 하고, 관찰성 시스템 자체에 대한 외부 모니터링이 따로 있어야 한다. "내 알람이 살아있는가"를 확인하는 dead man's switch는 흔히 쓰는 패턴이다.

마지막으로, 관찰성은 한 번 만들고 끝나지 않는다. 시스템이 바뀌면 같이 바뀌어야 한다. 새 서비스가 추가되면 RED 대시보드도 자동으로 생기고, 새 의존성이 생기면 SLO도 같이 정의되고, 알림 audit이 분기마다 돌아가는 — 그런 사이클이 있어야 두 해 뒤에도 살아남는 관찰성이 된다.
