# 인프라 설계 원칙

## 1. 시작하기 전에

인프라 설계는 처음 그릴 때보다 운영 중에 부딪히는 결정이 훨씬 많다. AZ가 하나 죽었을 때 어디까지 서비스를 유지할지, 특정 팀의 잘못된 배포가 다른 팀까지 무너뜨리지 않게 어떻게 막을지, 헬스체크 하나 잘못 짜서 멀쩡한 노드가 전부 빠져버리는 사고를 어떻게 피할지. 이 문서는 5년차쯤 백엔드로 일하면서 운영 중에 실제로 부딪힌 결정과 트레이드오프를 정리한 것이다.

원칙이라고 부르지만 사실은 "과거에 한 번씩 깨져봐서 다음번엔 안 그러기로 한 것"에 가깝다. 가용성, 확장성, 복원력, 격리, 단일 장애점 제거 — 다섯 가지를 축으로 잡고 가운데에서 부딪히는 트레이드오프를 본다.

```
              ┌──────────────┐
              │   가용성      │  얼마나 자주 응답하는가
              └──────┬───────┘
                     │
   ┌─────────────────┼─────────────────┐
   │                 │                 │
┌──┴───┐         ┌───┴────┐        ┌───┴───┐
│ 확장성 │         │  복원력  │        │  격리  │
└──┬───┘         └───┬────┘        └───┬───┘
   │                 │                 │
   └─────────────────┴─────────────────┘
                     │
              ┌──────┴───────┐
              │ SPOF 제거    │  무엇 하나가 죽어도 살아남는가
              └──────────────┘
```

다섯 축은 서로 독립이 아니다. 격리를 강하게 가져가면 확장성이 떨어지고, 복원력을 위해 재시도를 넣으면 가용성이 오히려 깎이는 경우가 생긴다. 그래서 원칙을 모두 만족시키려고 하면 시스템이 무거워진다. 어디까지가 비즈니스가 감당할 수 있는 선인지 매번 다시 묻는 게 진짜 일이다.

## 2. 가용성

### 2.1 SLA, SLO, 그리고 현실의 차이

99.9%, 99.95%, 99.99%. 숫자만 보면 비슷한데 운영 부담은 자릿수마다 한 단계씩 올라간다.

| SLO | 월 다운타임 | 연 다운타임 | 실무 체감 |
|---|---|---|---|
| 99% | 7시간 12분 | 3일 15시간 | 사내 도구 수준 |
| 99.9% | 43분 | 8시간 45분 | 일반 서비스 기본선 |
| 99.95% | 21분 | 4시간 22분 | 결제·인증 수준 |
| 99.99% | 4분 | 52분 | 인프라 비용이 2~3배 |
| 99.999% | 26초 | 5분 | 글로벌 백본급 |

99.99%를 약속하면 한 달에 4분 안에 모든 사고를 끝내야 한다. 알람 받고 SSH 붙는 데만 3분이 지난다. 즉 99.99%는 "사람이 대응한다"는 모델로는 불가능하고, 자동 페일오버가 깔려 있어야 가능하다. 99.999%는 글로벌 멀티리전과 자동화된 트래픽 시프트가 모두 갖춰져 있어야 한다.

설계 단계에서 보통 SLO를 부풀리는 경우가 많다. "그래도 99.99%는 가야지" 하는데, 그게 무엇을 의미하는지 모르고 적는다. 99.99%를 적으면 연간 52분, 즉 분기당 13분이다. 분기 한 번 배포 사고가 나도 위반이다. 이걸 알고 약속해야 한다.

### 2.2 가용성을 만들어내는 구조

가용성은 "안 죽게 만든다"가 아니라 "죽어도 다른 게 받는다"로 접근해야 한다. 단일 인스턴스를 아무리 튼튼하게 만들어도 EC2 한 대의 가용성은 결국 EC2 SLA를 못 넘는다. AWS EC2 단일 인스턴스 SLA는 95% 수준이다. 99.9%를 만들려면 이걸 여러 대로 묶고, 앞에 로드밸런서를 둬야 한다.

```
[가용성 = 1 - (전체 다 죽을 확률)]

인스턴스 1대, 가용성 95%:
  서비스 가용성 = 95%

인스턴스 2대 + LB, 각 95% 독립:
  서비스 가용성 = 1 - (0.05 × 0.05) = 99.75%

인스턴스 3대 + LB, 각 95% 독립:
  서비스 가용성 = 1 - (0.05³) = 99.9875%
```

여기서 핵심은 "독립"이다. 같은 AZ에 3대를 띄우면 AZ가 죽을 때 셋이 같이 죽는다. 독립이 아니라 양의 상관관계가 강하다. 그래서 가용성 계산은 늘 "실제로 같이 죽을 확률"을 기준으로 다시 해야 한다.

## 3. 멀티 AZ 구성

### 3.1 AZ란 진짜 무엇인가

AWS 한 리전 안에는 보통 3~6개의 AZ가 있다. AZ는 물리적으로 분리된 데이터센터다. 전력, 냉각, 네트워크 전부 분리되어 있다. 그래서 한 AZ가 죽어도 다른 AZ는 영향을 받지 않는다 — 이론적으로는.

실무에서 AZ 단위 장애는 생각보다 자주 발생한다. 2021년 AWS us-east-1 장애, 2023년 ap-northeast-2 장애 같은 사건이 매년 한두 번씩은 일어난다. 멀티 AZ를 안 해놓았다면 그날은 자동으로 다운타임이다.

### 3.2 멀티 AZ 구성의 비용 구조

```
서비스 ── AZ-a ── [App × 3대]
       │
       ── AZ-b ── [App × 3대]
       │
       ── AZ-c ── [App × 3대]
```

기본형은 위처럼 3개 AZ에 동일 수량을 배치한다. 트래픽이 균등하게 분배된다는 가정 하에, AZ 하나가 죽으면 나머지 둘이 50% 더 받아내야 한다. 그래서 평소 사용률을 66% 이하로 유지해야 한다. 사용률 90%로 빡빡하게 굴리고 있었다면 AZ 하나 나갔을 때 나머지가 폭주한다.

AZ 간 통신은 같은 리전이라도 트래픽 비용이 발생한다. AWS 기준 GB당 0.01달러 정도다. 작아 보이지만 대량 트래픽이 흐르면 월 수백~수천 달러가 된다. 그래서 데이터베이스 읽기는 같은 AZ 안에서 처리하도록 라우팅 힌트를 주는 경우가 많다. RDS는 `read_replica`를 AZ별로 두고 애플리케이션이 자기 AZ의 리플리카를 우선 호출하게 한다.

### 3.3 멀티 AZ에서 흔한 실수

상태를 가진 컴포넌트 — Redis, MySQL, Kafka — 를 단일 AZ에 두는 경우가 의외로 많다. "어차피 백업 있으니까"라고 생각하는데, AZ가 죽으면 백업에서 복구하는 동안 다운타임이 발생한다. 백업 복구는 보통 수 분에서 수십 분이 걸린다. 99.9% SLO를 약속했다면 이 한 번에 분기 예산을 다 쓴다.

Redis Sentinel이나 Cluster, MySQL의 Multi-AZ RDS, Kafka의 `min.insync.replicas=2`와 `replication.factor=3`을 AZ 분산으로 배치 — 이런 설정을 처음부터 들고 가야 한다.

또 하나 자주 실수하는 부분이 NAT Gateway다. 비용을 아끼려고 NAT Gateway를 한 AZ에만 두고 모든 AZ가 그걸 쓰게 만드는 구성을 보는데, 그 NAT가 죽으면 다른 AZ들도 인터넷 통신이 끊긴다. NAT는 AZ마다 하나씩 둬야 한다.

## 4. 확장성

### 4.1 수직 확장과 수평 확장

수직 확장(scale-up)은 인스턴스를 더 큰 사양으로 바꾸는 거다. m5.large → m5.4xlarge 같은 식이다. 빠르고 코드 변경이 없지만 천장이 낮다. AWS에서 가장 큰 인스턴스가 m7i.48xlarge 정도인데, 그걸로도 안 되면 거기서 끝이다.

수평 확장(scale-out)은 인스턴스 수를 늘리는 거다. 천장이 사실상 없지만 애플리케이션이 무상태(stateless)여야 하고, 세션·캐시·잠금 처리를 외부로 빼내야 한다.

실무에서는 처음부터 수평 확장이 가능하게 짜고, 트래픽이 일정 임계점을 넘기 전까지는 수직 확장으로 버티다가 한계가 오면 수평으로 가는 게 보통이다. 처음부터 분산 시스템처럼 짜면 개발 속도가 절반 이하로 떨어진다.

### 4.2 오토스케일링의 함정

CPU 70% 넘으면 인스턴스 추가, 30% 미만이면 제거 — 이런 식의 단순한 정책은 잘 동작하지 않는다. 몇 가지 함정이 있다.

**부팅 시간을 무시한다.** EC2 한 대가 부팅되고 헬스체크 통과해서 트래픽 받는 데까지 보통 2~5분이 걸린다. 컨테이너라도 1~2분은 잡아야 한다. 트래픽이 갑자기 튀면 그 사이에 기존 인스턴스가 다 터진다. 그래서 스케일 아웃 임계점은 CPU 50~60% 정도로 잡고, 부족분을 미리 띄우는 게 안전하다.

**스케일 인을 너무 빨리 한다.** 새벽 트래픽이 적다고 인스턴스를 7대에서 2대로 줄여놨는데, 아침 출근 시간 트래픽이 5분 만에 몰아치면 2대로는 못 받는다. 스케일 인은 보수적으로, 스케일 아웃은 공격적으로 — 비대칭으로 짜야 한다.

**오토스케일링이 죽음의 나선을 만든다.** 한 인스턴스가 부하를 못 받고 죽으면, 그 트래픽이 남은 인스턴스로 몰리면서 거기도 죽는다. 그러면 다시 다음 인스턴스로 몰린다. 이걸 막으려면 회로 차단기(circuit breaker)나 트래픽 셰이핑(rate limit)이 앞단에 깔려 있어야 한다.

### 4.3 데이터베이스 확장의 어려움

애플리케이션은 수평 확장이 쉽다. DB는 어렵다.

- **읽기 확장**: 리드 리플리카를 여러 대 둔다. 비교적 쉽다. 단, 리플리카 지연(replication lag)이 있어서 "방금 쓴 데이터를 바로 읽으면 못 본다"는 문제가 생긴다. 결제·주문 같은 곳은 일관성이 깨지면 사고다.
- **쓰기 확장**: 샤딩이 답인데, 한 번 샤딩 키를 잘못 잡으면 재샤딩이 지옥이다. 사용자 ID로 샤딩했는데 특정 사용자(인플루언서나 봇)가 트래픽의 50%를 차지하면 그 샤드만 폭주한다(핫 샤드).
- **샤딩 키 선정**: 가장 자주 같이 조회되는 데이터가 같은 샤드에 들어가야 한다. 분산 조인은 느리고 복잡하다.

샤딩으로 가기 전에 캐싱·리드 리플리카·쿼리 튜닝·테이블 분리(verticla partition)로 막을 수 있을 만큼 막아야 한다. 샤딩은 마지막 수단이라고 생각하는 게 맞다.

## 5. 복원력

### 5.1 복원력은 가용성이 아니다

가용성은 "지금 응답하는가"이고 복원력은 "고장 났을 때 다시 살아나는가"이다. 둘은 다르다. 가용성을 99.99%로 맞춰도 한 번 무너지면 6시간 동안 못 돌아오는 시스템이 있고, 가용성은 99.9%여도 5분 안에 복구되는 시스템이 있다.

복원력을 만드는 방법은 크게 셋이다.

1. **재시도와 백오프**: 일시적인 장애를 흡수한다.
2. **회로 차단기**: 망가진 의존성에 계속 호출하지 않게 막는다.
3. **그레이스풀 디그레이드**: 일부 기능이 죽어도 핵심은 살린다.

### 5.2 재시도가 시스템을 죽이는 패턴

재시도는 잘못 쓰면 장애를 키운다. 가장 흔한 안티패턴은 "모든 계층이 재시도하는 경우"다.

```
Client ── (3회 재시도) ──▶ API Gateway
                                │
                                │ (3회 재시도)
                                ▼
                          Service A
                                │ (3회 재시도)
                                ▼
                          Service B (장애)
```

Service B가 한 번 실패하면 Service A는 3번 호출한다. API Gateway는 그걸 3번 본다. 클라이언트는 그걸 3번 본다. 결국 Service B에는 27번의 요청이 쏟아진다. 장애가 났을 때 트래픽이 27배가 된다. 이걸 "재시도 폭풍(retry storm)"이라고 부른다.

대응은 두 가지다.

- **계층 중 한 곳만 재시도한다.** 보통 가장 안쪽 계층(직접 호출하는 곳)에서만 재시도하고, 나머지는 그냥 에러를 전파한다.
- **지수 백오프(exponential backoff)에 지터(jitter)를 더한다.** 1초 → 2초 → 4초가 아니라 1±0.5초 → 2±1초 → 4±2초처럼 흩어놓는다. 모두 같은 타이밍에 재시도해서 또 한 번 폭주하는 걸 막는다.

### 5.3 회로 차단기

회로 차단기는 "이 의존성이 망가졌으니 잠시 호출을 끊자"는 메커니즘이다. 보통 세 상태를 가진다.

```
   ┌─ Closed (정상) ──── 실패 임계치 초과 ───▶ Open (차단)
   │                                              │
   │                                              │ 일정 시간 후
   │                                              ▼
   └─ 성공 ─── Half-Open (반열림) ◀────────────────┘
                    │
                    └─ 실패 ───▶ 다시 Open
```

Closed 상태에서는 정상적으로 호출한다. 실패율이 임계치를 넘으면 Open으로 가서 일정 시간 동안 모든 호출을 즉시 실패시킨다. 그 시간이 지나면 Half-Open으로 가서 일부 호출만 흘려보낸다. 잘 되면 Closed로 복귀, 안 되면 다시 Open.

회로 차단기가 없으면 의존성이 죽었을 때 우리 서비스의 스레드 풀이 다 타임아웃을 기다리느라 응답이 막힌다. 한 번 막히면 후속 요청이 쌓이고 메모리가 차고 GC가 돌고 결국 우리도 죽는다. 이게 장애 전파의 가장 흔한 경로다.

### 5.4 그레이스풀 디그레이드

기능이 죽었을 때 전부 죽이지 말고 일부를 죽이는 거다. 예를 들어 이커머스에서 추천 시스템이 죽었다고 메인 페이지가 안 뜨면 안 된다. 추천은 빼고 보여줘야 한다.

설계할 때 "이 기능이 없으면 다른 게 동작할 수 있나"를 의존성 그래프에 표시하고, 의존성 없이도 핵심 경로가 살아남는지 본다.

```
[메인 페이지 요청]
       │
       ├─ 상품 목록 (필수)
       ├─ 가격 정보 (필수)
       ├─ 추천 상품 (선택, 죽으면 빈 영역)
       ├─ 최근 본 상품 (선택, 죽으면 숨김)
       └─ 광고 (선택, 죽으면 기본 광고)
```

선택 의존성은 짧은 타임아웃(200~500ms)을 걸고, 실패하면 캐시된 결과나 빈 결과로 대체한다.

## 6. 격리

### 6.1 격리의 목적

격리는 "한 부분의 문제가 다른 부분으로 번지지 않게 막는 것"이다. 동력의 단위가 인스턴스든 컨테이너든 네임스페이스든 계정이든 — 어떤 단위로 자를지를 정하는 게 격리 설계다.

격리 단위는 보통 다음 순서로 강해진다.

```
약함                                                강함
 ↓                                                  ↓
스레드 < 프로세스 < 컨테이너 < VM < AZ < 리전 < 계정
```

스레드 격리는 같은 프로세스 안에서 작업을 분리하는 것이다. 한 풀이 다 차도 다른 풀은 살아 있다. Hystrix가 했던 일이다. 가볍지만 메모리·OS 자원은 같이 쓴다.

프로세스/컨테이너 격리는 메모리와 파일 디스크립터까지 분리한다. 컨테이너 하나가 OOM으로 죽어도 다른 컨테이너는 살아 있다.

VM 격리는 OS 커널까지 분리한다. 강력하지만 무겁다.

AZ 격리는 데이터센터 단위, 리전 격리는 지리적 단위, 계정 격리는 권한·과금·할당량 단위다.

### 6.2 멀티 테넌시에서의 격리

여러 고객(테넌트)이 한 인프라를 공유할 때 격리는 비즈니스 핵심이다.

- **소프트 격리**: 같은 DB에 `tenant_id` 컬럼으로 구분. 비용은 적지만 한 테넌트의 문제가 다른 테넌트에 번질 위험이 있다.
- **공유 인프라 + 분리 DB**: 애플리케이션은 공유, 데이터베이스는 테넌트별로 따로. 중간 절충안.
- **하드 격리**: 테넌트마다 별도 클러스터·계정. 비용은 크지만 완전히 독립적이다. 엔터프라이즈 SaaS에서 자주 쓰인다.

소프트 격리를 쓰면서 "한 고객이 무거운 쿼리를 날리면 다른 고객의 응답이 느려지는" 문제는 거의 항상 발생한다. 이걸 막으려면 테넌트별 쿼리 제한, 커넥션 풀 분할, CPU·메모리 쿼터 같은 게 들어가야 한다.

### 6.3 셀룰러 아키텍처 (Cell-based architecture)

큰 시스템에서 점점 많이 보이는 격리 패턴이다. 시스템 전체를 여러 개의 독립된 "셀(cell)"로 나눈다. 각 셀은 동일한 코드를 돌리지만 데이터와 부하가 분리되어 있다.

```
[라우터] ──┬──▶ Cell-A ── DB-A
           ├──▶ Cell-B ── DB-B
           └──▶ Cell-C ── DB-C
```

특정 사용자는 항상 같은 셀로 라우팅된다. 한 셀이 죽어도 다른 셀의 사용자는 영향이 없다. AWS Route 53, S3, DynamoDB 같은 서비스가 내부적으로 이렇게 설계되어 있다고 알려져 있다.

작은 조직에서 셀룰러까지 가는 건 과하다. 하지만 사용자 그룹별로 폭주 패턴이 다르거나 SLA가 다르면 셀로 자르는 게 답이 된다.

## 7. 단일 장애점 제거

### 7.1 SPOF 찾는 방법

SPOF(Single Point of Failure)는 "이게 죽으면 전체가 죽는 컴포넌트"다. 다이어그램을 그리고 노드 하나씩 가리면서 "이게 죽으면 다른 경로로 흐를 수 있나"를 본다. 답이 "아니오"인 노드가 전부 SPOF다.

실무에서 흔히 놓치는 SPOF는 다음과 같다.

- **DNS**: 외부 DNS 한 곳을 쓰는데 그게 죽으면 도메인 해석이 안 된다. 보조 DNS를 두는 게 보통.
- **인증서 갱신 스크립트**: 한 서버에 있는 cron job이 갱신하는데 그 서버가 죽으면 인증서가 만료된다.
- **운영 도구**: 배포 시스템, 모니터링이 단일 노드. 사고가 났을 때 정작 대응 도구가 죽어 있다.
- **사람**: "그 시스템은 OO만 알아요"는 가장 위험한 SPOF다. 문서화와 권한 분산이 필요하다.
- **공통 의존성**: 모든 서비스가 같은 외부 API를 동기 호출한다. 그 API가 죽으면 전부 죽는다.

### 7.2 SPOF 제거는 비용이 든다

SPOF를 다 제거하는 건 불가능하고 그럴 필요도 없다. 비용 대비 효과를 보고 결정해야 한다.

| SPOF | 제거 비용 | 장애 시 영향 | 우선순위 |
|---|---|---|---|
| 단일 DB 마스터 | 중 (Multi-AZ RDS) | 전체 다운 | 높음 |
| 단일 AZ | 중 (멀티 AZ 배치) | 전체 다운 | 높음 |
| 단일 리전 | 매우 큼 (DR 사이트) | 지역 단위 다운 | 사업에 따라 |
| 단일 DNS 제공자 | 작음 (보조 DNS) | 도메인 해석 실패 | 중간 |
| 단일 결제 PG | 중 (멀티 PG) | 결제 실패 | 결제 비중에 따라 |

핵심은 "어떤 SPOF가 실제로 죽었을 때 비즈니스에 얼마나 영향을 주는가"를 정량화하는 거다. 그 숫자가 제거 비용을 넘으면 제거하고, 안 넘으면 모니터링과 빠른 복구 절차로 대신한다.

### 7.3 DR (재해 복구) 전략

리전 단위 SPOF를 막는 게 DR이다. 비용 순으로 정리하면:

- **Backup & Restore**: 다른 리전에 백업만 저장. 복구에 수 시간~수 일. 가장 싸다.
- **Pilot Light**: 핵심 컴포넌트만 다른 리전에 최소 사양으로 띄워둠. 복구 시 스케일 업. 수십 분~수 시간.
- **Warm Standby**: 다른 리전에 축소 버전을 항상 띄워두고 트래픽을 받을 수 있게. 수 분~수십 분.
- **Active-Active**: 두 리전이 동시에 트래픽을 받음. 즉시 페일오버. 가장 비싸다.

대부분 조직은 Pilot Light나 Warm Standby로 시작한다. Active-Active는 글로벌 서비스이거나 강한 SLO를 약속한 경우에만 간다. Active-Active는 데이터 동기화가 가장 어려운 부분이고, 잘못 하면 양쪽이 서로 다른 데이터를 쓰는 split-brain이 발생한다.

## 8. 장애 전파 방지와 Blast Radius 축소

### 8.1 장애가 전파되는 경로

장애 전파는 보통 동기 의존성을 따라 일어난다. A가 B를 동기 호출하고 B가 죽으면, A는 B의 응답을 기다리면서 자원이 막힌다. 막힌 A는 자기를 호출하는 C에게도 같은 일을 한다. 이게 전염병처럼 퍼진다.

이걸 막는 원칙은 단순하다.

1. **동기 호출의 깊이를 줄인다.** A → B → C → D처럼 길어지면 D가 죽을 때 전부 영향을 받는다.
2. **타임아웃을 짧게 잡는다.** 외부 호출의 타임아웃이 30초면 그 30초 동안 자원이 묶인다. 보통 2~5초 안으로 잡는다.
3. **비동기 처리를 늘린다.** 즉시 응답이 필요 없는 작업은 큐에 넣고 별도 워커가 처리한다. 큐가 차도 응답 경로는 안 막힌다.
4. **벌크헤드(bulkhead)를 둔다.** 의존성마다 별도 스레드풀·커넥션풀을 둬서 한 의존성이 죽어도 다른 호출이 영향 안 받게 한다.

### 8.2 Blast Radius

Blast radius는 "한 컴포넌트가 죽었을 때 영향을 받는 범위"다. 작을수록 좋다.

큰 모놀리스 하나가 모든 기능을 다루면 그게 죽으면 전부 죽는다. 마이크로서비스로 쪼개면 한 서비스가 죽어도 다른 기능은 살아남는다. 하지만 쪼개기만 하면 안 된다. 서비스 간에 강한 동기 의존성이 있으면 어차피 같이 죽는다.

Blast radius를 줄이는 실질적인 방법:

- **기능별로 자른다.** 결제, 인증, 추천, 검색을 각각의 서비스로. 한 도메인 장애가 다른 도메인을 안 죽이게.
- **사용자 그룹별로 자른다.** 셀룰러 아키텍처. 한 셀이 죽으면 그 셀의 사용자만 영향을 받는다.
- **읽기와 쓰기를 자른다.** CQRS 비슷한 구조. 쓰기 경로가 죽어도 읽기는 살아 있게.
- **공통 의존성을 줄인다.** 모든 서비스가 한 Redis를 보고 있으면 그 Redis가 SPOF다. 도메인별로 분리.

### 8.3 배포 자체의 blast radius

장애의 가장 흔한 원인은 외부 공격이 아니라 자기 배포다. 배포의 blast radius를 줄이는 방법:

- **카나리 배포**: 새 버전을 5%, 25%, 50%, 100% 순으로 트래픽 증가. 문제가 보이면 100% 가기 전에 멈춘다.
- **블루-그린 배포**: 새 버전을 별도 환경에 올리고 트래픽을 한 번에 스위치. 문제 시 즉시 롤백.
- **피처 플래그**: 코드는 배포하되 기능은 끄고 시작. 사용자별·비율별로 단계적으로 켠다.

전사 동시 배포는 가장 위험하다. 문제 있는 배포가 모든 서비스에 한 번에 퍼진다. 시간차를 두고 배포하거나 의존성 그래프 역순으로 배포하는 게 안전하다.

## 9. 헬스체크 설계

### 9.1 헬스체크는 왜 어려운가

헬스체크는 단순해 보이지만 실제로 가장 자주 사고를 부르는 컴포넌트다. 가장 흔한 사고는 두 가지다.

1. **헬스체크가 너무 깐깐해서 멀쩡한 노드가 다 빠진다.** DB 연결을 검사하는 헬스체크가 있는데 DB가 잠깐 느려지면 모든 노드가 unhealthy로 빠지고 트래픽 받을 노드가 없어진다.
2. **헬스체크가 너무 허술해서 죽은 노드가 트래픽을 받는다.** HTTP 200만 보고 살아 있다고 판단하는데 실제로는 백그라운드 워커가 멈춰 있다.

### 9.2 헬스체크 종류

```
[Liveness 체크]    노드가 살아 있나? (실패하면 재시작)
[Readiness 체크]   트래픽 받을 준비됐나? (실패하면 트래픽 제거)
[Startup 체크]     기동 중인가? (기동 중에는 위 둘 보류)
```

Liveness와 Readiness를 헷갈리면 사고가 난다. 예를 들어 DB가 잠깐 끊겼을 때, 그걸 Liveness 실패로 잡으면 컨테이너가 재시작된다. 재시작해도 DB는 여전히 끊겨 있으니 또 재시작. 무한 재시작 루프에 빠진다. DB 문제는 Readiness로 잡고, Liveness는 "프로세스가 응답 자체를 못 하는가"만 본다.

Startup 체크는 부팅이 오래 걸리는 애플리케이션에서 중요하다. JVM 기반 서비스는 워밍업에 30초~수 분이 걸리는데, Liveness가 너무 일찍 실행되면 워밍업 중인 노드를 죽인다. Startup이 완료될 때까지 Liveness를 보류하는 식으로 분리한다.

### 9.3 깊은 헬스체크 vs 얕은 헬스체크

얕은 헬스체크는 "프로세스가 응답하는가"만 본다. `GET /health`가 200을 주는지. 빠르고 안전하지만 실제 기능 동작은 안 본다.

깊은 헬스체크는 DB, 캐시, 외부 의존성까지 다 본다. 정확하지만 의존성이 잠깐만 흔들려도 노드가 다 빠지는 위험이 있다.

실무에서 보통 가는 방향은:

- **로드밸런서 헬스체크**: 얕게. HTTP 200만 본다. 의존성은 안 본다.
- **모니터링용 헬스체크**: 깊게. 의존성 상태까지 본다. 단, 이건 알람 용도지 트래픽 라우팅을 결정하는 데 쓰지 않는다.
- **자가 진단**: 깊은 체크가 실패해도 일정 비율은 트래픽을 계속 받는다. 전부 빠지는 걸 막는 안전장치.

### 9.4 헬스체크 임계값

헬스체크 한 번 실패했다고 바로 노드를 빼면 일시적 네트워크 지터에도 노드가 빠진다. 보통 "연속 3회 실패 → unhealthy", "연속 2회 성공 → healthy 복귀" 같은 임계값을 둔다.

체크 간격은 트레이드오프다. 짧으면(1~2초) 빠른 감지가 되지만 노이즈에 민감하다. 길면(10~30초) 안정적이지만 실제 장애 감지가 늦다. 보통 3~5초 간격에 3회 실패로 시작한다.

### 9.5 헬스체크 자체의 SPOF

헬스체크 엔드포인트가 메인 애플리케이션 스레드풀을 공유하면, 애플리케이션이 막혔을 때 헬스체크도 막혀서 응답을 못 한다. 그러면 로드밸런서가 노드를 빼버린다. 이건 의도한 동작이긴 한데, 헬스체크가 막힌 이유가 일시적인 트래픽 폭주라면 그 노드를 빼는 게 오히려 다른 노드를 더 죽인다.

대응으로 헬스체크 엔드포인트를 별도 포트·별도 스레드풀로 두는 경우가 있다. 헬스체크는 늘 응답하고, 부하 상황은 다른 신호(CPU, 큐 길이)로 판단한다.

## 10. 트레이드오프 표

설계할 때 자주 부딪히는 결정을 한 페이지로.

| 결정 | 한쪽으로 가면 | 반대로 가면 |
|---|---|---|
| 가용성 목표 99.9% vs 99.99% | 운영 부담 적음, 사고 시 여유 있음 | 자동화 필수, 비용 2~3배 |
| 단일 AZ vs 멀티 AZ | 비용 적음, AZ 간 트래픽 비용 없음 | AZ 장애에 살아남음, 비용 2~3배 |
| 멀티 AZ vs 멀티 리전 | 같은 리전 안에서만 분산, 리전 장애 못 막음 | 글로벌 장애 대응, 데이터 동기화 복잡도 폭증 |
| 모놀리스 vs 마이크로서비스 | 단순한 운영, 단일 장애 영향 큼 | 격리 강함, 운영 복잡도·네트워크 복잡도 폭증 |
| 동기 호출 vs 비동기 큐 | 즉시 응답, 강한 결합 | 결합 약함, 결과 확인 어려움 |
| 깊은 헬스체크 vs 얕은 헬스체크 | 정확한 상태, 일시적 의존성 장애에 취약 | 안정적, 죽은 노드 감지 늦음 |
| 공격적 오토스케일링 vs 보수적 | 비용 절감, 트래픽 급증에 취약 | 안정적, 비용 증가 |
| 강한 격리(셀룰러) vs 공유 | 장애 격리 명확, 자원 효율 낮음 | 자원 효율 높음, 장애 영향 범위 큼 |

각 행에서 "정답"은 없다. 비즈니스 단계, 트래픽 규모, SLO 약속, 운영 인력 — 이 네 가지를 보고 결정한다.

---

## 11. 정리하며 — 실무에서 가장 자주 후회하는 것

운영해보고 나서 "그때 이렇게 했어야 했다"고 가장 많이 후회하는 항목들.

- **DB를 처음부터 Multi-AZ로 안 둠.** 트래픽이 적을 때 비용을 아끼다가 첫 AZ 장애에서 몇 시간 다운된다.
- **상태(state)를 애플리케이션에 둠.** 처음에는 편하지만 수평 확장할 때 다 갈아엎어야 한다.
- **재시도를 모든 계층에 넣음.** 장애가 났을 때 트래픽이 수십 배로 튀어서 죽은 의존성을 더 죽인다.
- **타임아웃을 안 정함 또는 너무 길게 잡음.** 30초 타임아웃이 디폴트인 라이브러리들이 많은데, 그게 한번 막히면 스레드풀이 다 차서 우리도 죽는다.
- **헬스체크에 외부 의존성을 넣음.** 의존성이 잠깐 흔들려도 모든 노드가 빠져버린다.
- **단일 NAT, 단일 DNS, 단일 인증서 갱신 cron.** 인프라가 아니라 운영 도구가 SPOF가 되는 경우.
- **Active-Active를 너무 일찍 시도함.** 데이터 동기화가 안 잡혀 있는데 멀티 리전을 가면 split-brain 사고가 난다.

이 일곱 가지를 피하는 것만으로도 운영 사고의 절반은 줄어든다. 나머지는 사고를 겪으면서 조직마다 다른 결을 만들어간다.
