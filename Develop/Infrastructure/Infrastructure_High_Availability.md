# 고가용성 인프라 구성

## 1. 시작하기 전에

고가용성(HA)이라는 단어는 회의실에서 쓰일 때와 새벽 3시 장애 상황실에서 쓰일 때 무게가 다르다. 설계 단계에서 "이중화했다"는 말은 안심처럼 들리지만, 실제로 한쪽이 죽었을 때 정말로 살아남는지는 첫 장애가 나봐야 안다. 그리고 첫 장애에서 살아남지 못한 시스템이 훨씬 많다.

5년차쯤 백엔드를 하면서 HA에 대해 배운 건 두 가지다. 첫째, "이중화"는 구성이 아니라 동작이다. 두 대를 띄우는 게 아니라 한 대가 죽었을 때 다른 한 대가 받아주는 일이 실제로 일어나야 이중화다. 둘째, HA에는 공짜가 없다. 가용성을 한 자릿수 올리는 데 비용은 두세 배가 든다. 그래서 "어느 정도까지 가용해야 하는가"부터 정해야 한다.

이 문서는 Active-Active와 Active-Passive를 어떻게 선택하는지, 멀티 AZ와 멀티 리전을 어디까지 가져가는지, 동기/비동기 복제의 트레이드오프, failover와 RTO/RPO 산정, split-brain 방지, 그리고 헬스체크가 멀쩡한 노드를 죽이는 사고를 어떻게 피할지를 다룬다. 가능하면 실제로 부딪힌 상황 위주로 적는다.

```
        가용성 목표 (SLO)
              │
   ┌──────────┼──────────┐
   │          │          │
이중화      복제 전략   장애 감지
(Active-A/P) (Sync/Async)(Health Check)
   │          │          │
   └──────────┼──────────┘
              │
       RTO / RPO 산정
              │
      ┌───────┴───────┐
      │               │
   Failover      Split-Brain
   메커니즘       방지
```

## 2. Active-Active vs Active-Passive

### 2.1 무엇이 다른가

둘 다 노드를 둘 이상 띄운다는 점은 같다. 차이는 평소에 트래픽을 어떻게 나누느냐다.

Active-Active는 모든 노드가 동시에 요청을 받는다. 로드 밸런서가 라운드 로빈이든 가중치든 어떤 방식으로든 트래픽을 분산한다. 한쪽이 죽으면 남은 쪽이 받아주는 게 자연스럽다.

Active-Passive는 한쪽(Active)만 트래픽을 받고 다른 쪽(Standby)은 대기한다. Active가 죽으면 Standby가 승격해서 받기 시작한다. "Hot Standby"는 즉시 받을 준비가 되어 있는 경우, "Warm Standby"는 시작은 되어 있지만 트래픽을 받으려면 약간의 준비가 필요한 경우, "Cold Standby"는 평소엔 꺼져 있는 경우를 가리킨다.

이론적으로는 Active-Active가 자원을 더 알뜰하게 쓴다. Active-Passive는 Standby가 놀고 있으니까. 그런데 실제로는 그렇게 단순하지 않다.

### 2.2 언제 Active-Active를 쓰는가

상태가 없는 서비스(stateless service)는 거의 무조건 Active-Active가 맞다. API 서버, 웹 서버, 워커 풀 같은 것들이다. 노드마다 동일한 코드가 돌고, 어떤 노드가 요청을 받아도 결과는 같다. 죽은 노드는 헬스체크에서 빼고 나머지가 받으면 그만이다.

상태가 있더라도 분산이 잘 되는 데이터 저장소도 Active-Active로 갈 수 있다. Cassandra, ScyllaDB, DynamoDB 같은 시스템은 처음부터 여러 노드가 동시에 쓰기를 받도록 설계되어 있다. 이 경우 최종 일관성(eventual consistency)을 받아들여야 한다.

### 2.3 언제 Active-Passive를 쓰는가

쓰기가 한 곳에서만 일어나야 하는 데이터베이스가 대표적이다. PostgreSQL, MySQL, MongoDB의 일반적인 구성은 한 노드가 Primary로 쓰기를 받고 나머지는 Replica로 읽기만 받는다. Replica에 쓰기를 보낼 수도 있는 구성을 만들 수는 있지만 충돌 해결이 골치 아파져서 잘 안 쓴다.

상태가 있고 일관성이 강하게 요구되는 시스템은 대체로 Active-Passive 쪽으로 간다. Redis Sentinel 구성도 같은 맥락이고, etcd나 ZooKeeper의 리더 선출도 본질은 Active-Passive다(쓰기는 리더만 받는다).

오래된 레거시 시스템도 Active-Passive로 가는 경우가 많다. 동시에 여러 인스턴스가 돌도록 설계되지 않은 모놀리식 애플리케이션은 Active-Active로 만들려면 코드부터 손봐야 하는데, 그게 안 되니까 우선 Passive로 띄워둔다.

### 2.4 실무에서 부딪히는 함정

Active-Active에서 가장 많이 보는 사고는 "분산은 되는데 상태가 공유되지 않는다"는 것이다. 세션을 메모리에 저장하는 API 서버 두 대를 로드 밸런서 뒤에 둔다. 사용자가 로그인하면 1번 서버에 세션이 생긴다. 다음 요청이 2번 서버로 가면 세션이 없다. 로그아웃된 것처럼 보인다. 이걸 막으려고 sticky session을 켜면 이번엔 한 노드가 죽었을 때 그 노드에 붙어 있던 사용자만 로그아웃된다. 결국 세션을 Redis 같은 외부 저장소로 빼야 진짜 Active-Active가 된다.

Active-Passive에서 가장 많이 보는 사고는 "Passive가 사실은 안 깨어났다"는 것이다. 1년에 한 번 fault injection을 하지 않는 Active-Passive는 거의 작동하지 않는다. 백업 노드의 OS가 업그레이드를 안 받아서 부팅이 안 되거나, 설정 파일이 Active와 다르게 drift되어 있거나, Standby에서 돌던 cron이 Active로 승격한 뒤에도 계속 돌아서 중복 작업이 발생하거나. 실제 장애가 나서야 알게 된다.

```
[Active-Active]                  [Active-Passive]
    LB                                LB
   /  \                                │
  V    V                               V
 N1    N2                            Active ──replication──> Standby
 (둘 다 트래픽)                       (트래픽 받음)         (대기)

 한 쪽 죽으면 LB가 뺀다.            Active 죽으면 Standby가 승격.
 사용자는 모를 수도 있다.            DNS 갱신, 연결 재수립 등이 필요.
```

### 2.5 N+1, N+2, 2N의 선택

Active-Active든 Active-Passive든 몇 대를 띄울지를 정해야 한다.

N+1은 평상시 필요한 N대에 한 대를 더 둔다. 한 대가 죽어도 N대가 살아남는다. 가장 비용이 적게 드는 이중화다. 다만 두 대가 동시에 죽으면 용량이 부족해진다. AZ 단위 장애를 대비하기엔 부족하다.

N+2는 두 대까지 죽어도 N대가 남는다. 배포 중에 한 대가 빠진 상태에서 또 한 대가 죽어도 견딘다. 일반적인 웹 서비스의 최소 구성으로 권장되는 수준이다.

2N은 평상시 필요한 만큼을 한 번 더 둔다. 절반이 죽어도 살아남는다. AZ 단위 장애 대응을 진지하게 고민할 때 자주 본다. 비용은 두 배지만 운영 부담이 줄어든다.

어느 쪽을 선택하든 "평상시 N대로 충분한가"를 다시 점검해야 한다. 트래픽이 두 배가 되었을 때, AZ 하나가 죽었을 때 남은 자원으로 받을 수 있는지 시뮬레이션해보지 않으면 N이 잘못 잡혀 있을 가능성이 높다.

## 3. 멀티 AZ와 멀티 리전

### 3.1 AZ란 무엇인가

Availability Zone은 같은 리전 안에서 물리적으로 분리된 데이터센터 묶음이다. 전원, 냉각, 네트워크가 독립되어 있어서 한 AZ가 죽어도 다른 AZ는 영향받지 않도록 설계되어 있다. AZ 간 거리는 대체로 수십 km 이내이고, 지연은 보통 1~2ms 수준이다. 동기 복제가 실용적으로 가능한 거리다.

리전(Region)은 지리적으로 분리된 단위다. AWS의 서울, 도쿄, 버지니아 같은 것들이다. 리전 간 지연은 수십에서 수백 ms로 늘어난다. 동기 복제는 사실상 불가능하고, 비동기 복제만 현실적이다.

### 3.2 멀티 AZ가 기본선이다

요즘 클라우드에서 단일 AZ로 서비스를 구성하는 건 권장되지 않는다. AZ 장애가 자주 일어나는 건 아니지만, 한 번 나면 단일 AZ로 띄운 서비스는 통째로 죽기 때문이다. 클라우드 사업자가 SLA를 보장하는 단위도 대부분 "멀티 AZ로 구성한 경우"다.

멀티 AZ의 기본 패턴은 이렇다.

- 로드 밸런서를 멀티 AZ로 띄운다(AWS의 ALB는 기본적으로 멀티 AZ).
- API 서버를 AZ별로 N+1대씩 띄운다. 보통 2~3개 AZ를 쓴다.
- RDS는 Multi-AZ 옵션을 켠다. Primary가 한 AZ에 있고 Standby가 다른 AZ에 있다.
- ElastiCache도 멀티 AZ로 띄운다. Redis라면 Primary와 Replica를 AZ별로 분산.

이렇게 구성하면 AZ 하나가 죽어도 다른 AZ가 받아준다. 다만 평상시 RPS의 50%만 한 AZ에서 처리하던 경우, AZ 하나가 죽으면 남은 AZ가 100%를 받아야 한다. 평상시 사용률이 50% 이하여야 한다는 뜻이다. 이걸 잊으면 AZ 장애가 나는 순간 남은 AZ도 과부하로 같이 죽는다.

### 3.3 멀티 리전은 언제 가는가

멀티 리전은 비용도 복잡도도 한 단계 더 올라간다. "리전이 통째로 죽는 일은 거의 없다"는 게 일반론이지만, AWS US-EAST-1은 몇 년에 한 번씩 사고가 나고 그때마다 절반 이상의 인터넷이 멈춘다. 그래서 진지하게 글로벌 서비스를 운영하는 곳은 멀티 리전을 고려한다.

멀티 리전을 가는 이유는 두 가지로 나뉜다. 첫째, 재해 복구(DR). 한 리전이 통째로 죽었을 때 다른 리전에서 서비스를 이어가기 위해. 둘째, 지연 최소화. 글로벌 사용자에게 가까운 리전에서 응답하기 위해.

이 둘은 설계가 다르다. DR 목적이라면 평소엔 한 리전이 Active이고 다른 리전은 대기다(Active-Passive). 지연 최소화 목적이라면 모든 리전이 동시에 트래픽을 받는다(Active-Active). 후자는 데이터 동기화가 훨씬 까다롭다.

### 3.4 리전 간 데이터 동기화의 현실

멀티 리전 Active-Active에서 가장 어려운 게 데이터 동기화다.

서울 리전과 도쿄 리전이 둘 다 쓰기를 받는다고 하자. 같은 사용자 ID로 양쪽에서 동시에 다른 값을 쓰면 어느 쪽이 이길까. 단순한 last-write-wins로 처리하면 한쪽 데이터가 날아간다. CRDT나 vector clock 같은 충돌 해결 방식을 도입할 수 있지만 코드가 복잡해진다.

이 문제를 우회하는 방법이 두 가지 있다.

첫째, 사용자별로 홈 리전을 정해서 그 사용자의 쓰기는 항상 홈 리전에서만 받는다. 한국 사용자는 서울에서, 일본 사용자는 도쿄에서. 다른 리전으로 요청이 오면 라우팅한다. 데이터는 한 곳에서 쓰이고 다른 리전으론 비동기 복제만 한다. 읽기는 어디서든 가능하다.

둘째, 글로벌하게 일관성이 필요한 데이터는 한 리전에 두고, 리전별로 독립적인 데이터만 분산한다. 사용자 인증은 한 곳에서, 콘텐츠는 리전별로.

어느 쪽이든 "글로벌 강한 일관성"은 포기한다. CAP 정리에서 P(분할 내성)를 포기할 수 없으니 C와 A 중 하나를 골라야 한다.

```
멀티 리전의 트레이드오프

서울                   도쿄
  │                     │
  ▼                     ▼
[API + DB]    ◀────▶  [API + DB]
                비동기 복제

쓰기 충돌 가능. last-write-wins 또는 충돌 해결 로직 필요.
지연: 한국 사용자 서울에서 1ms, 도쿄에서 30ms.
장애: 서울 죽어도 도쿄가 살아남음.
```

## 4. 데이터 복제 전략

### 4.1 동기 복제

동기 복제는 Primary가 쓰기를 받았을 때 Replica에도 데이터가 도착(혹은 디스크 반영)할 때까지 응답을 보류한다. 사용자에게 OK를 돌려준 시점에는 양쪽에 데이터가 모두 있다는 보장이 있다.

장점은 명확하다. Primary가 갑자기 죽어도 데이터 손실이 없다. RPO = 0이다.

단점도 명확하다. 매 쓰기마다 네트워크 왕복이 추가된다. 같은 AZ 안의 동기 복제는 1ms 이하지만, AZ 간 동기 복제는 1~2ms가 추가된다. 리전 간 동기 복제는 수십 ms가 추가되어 사실상 불가능하다. 그리고 Replica가 느려지면 Primary의 쓰기 지연이 같이 올라간다. Replica가 죽으면 어떻게 처리할지도 정해야 한다(쓰기를 멈출지, Replica를 빼고 진행할지).

PostgreSQL의 `synchronous_commit = on`, MySQL의 semi-sync replication이 이쪽이다. 정확히는 "Primary가 트랜잭션 로그를 Replica로 보내고 ACK를 받을 때까지 기다린다"는 의미이고, Replica가 그 로그를 실제 데이터에 적용했는지까지는 보장하지 않는 경우가 많다.

### 4.2 비동기 복제

비동기 복제는 Primary가 쓰기를 받으면 즉시 응답하고, Replica로의 전파는 백그라운드에서 한다.

장점은 쓰기가 빠르다는 것이다. Replica가 느리거나 죽어도 Primary의 쓰기에는 영향이 없다.

단점은 Primary가 죽으면 아직 Replica로 전파되지 않은 데이터가 손실된다는 것이다. 이 손실량은 보통 millisecond에서 second 단위지만, replication lag가 크게 벌어진 상황(예: 대량 일괄 작업 중)이면 분 단위로 늘어날 수 있다.

비동기 복제는 RPO > 0을 받아들인다는 뜻이다. "1초 이하의 데이터는 손실되어도 좋다"고 합의해야 쓸 수 있다.

### 4.3 준동기 복제 (semi-sync)

MySQL의 semi-sync처럼 "Replica가 받았다고 ACK는 하지만 적용은 비동기"인 방식이 있다. 동기와 비동기의 중간이다. 동기 복제보다 빠르고, 비동기 복제보다 안전하다.

다만 timeout이 걸리면 자동으로 비동기로 떨어지는 설정이 많다. 이걸 모르고 "우리는 semi-sync니까 안전해"라고 가정하면 사고가 난다. Replica가 죽거나 느려지는 순간 자동으로 비동기 모드로 전환되어 있을 수 있다.

### 4.4 무엇을 선택할 것인가

"데이터 손실 1초 이하 = 회사가 망한다" 수준이라면 동기 복제다. 결제, 인증, 거래 시스템이 보통 여기에 해당한다. 비용으로 지연을 산다.

"몇 초 정도는 손실되어도 복구 가능" 수준이라면 비동기다. 로그, 분석, 캐시, 추천 등이 보통 여기다.

대부분의 RDBMS는 같은 AZ 안에서는 동기, AZ 간에서는 동기 또는 준동기, 리전 간에서는 비동기로 가는 게 일반적이다. 같은 클러스터 안에서도 복제 토폴로지에 따라 모드가 섞일 수 있다.

```
복제 모드별 트레이드오프

         지연 추가    데이터 손실   Replica 장애 영향
동기      높음         없음          Primary가 멈춤(또는 Replica 빼고 진행)
준동기    중간         적음          timeout 후 비동기로 fallback
비동기    없음         있음(lag만큼)  없음
```

### 4.5 복제 지연(replication lag) 문제

비동기/준동기 복제를 쓰면 Replica가 Primary보다 뒤처진다. 이걸 replication lag라고 한다. 평상시 1초 이하지만 가끔 크게 벌어진다.

가장 자주 일어나는 시나리오는 "Primary에서 큰 트랜잭션이 돌 때"다. 100만 행을 한 번에 업데이트하는 쿼리가 Primary에서 30초 걸린다. Replica는 그 트랜잭션 로그를 받고 적용하는 데 또 30초가 걸린다. 그 사이 다른 쓰기들은 큐에 쌓인다. lag가 분 단위로 늘어난다.

이걸 막으려면 큰 트랜잭션을 쪼개야 한다. `LIMIT 1000`씩 잘라서 여러 번 도는 식이다. 운영 중에 한 번 당해보면 그다음부터는 자동으로 쪼개게 된다.

또 한 가지 흔한 시나리오는 "사용자가 쓰기 직후에 읽기를 했는데 Replica로 갔다"는 경우다. 댓글을 달고 새로고침했더니 내 댓글이 안 보인다. 이걸 막으려면 쓰기 직후 일정 시간은 Primary에서 읽거나, 사용자 단위로 sticky를 잡거나, 애플리케이션 레벨에서 캐시한다.

## 5. Failover 메커니즘

### 5.1 무엇을 감지하고 무엇을 시작하는가

Failover는 두 단계로 나뉜다. 첫째, Active 노드가 죽었음을 감지한다. 둘째, Standby를 Active로 승격시키고 트래픽을 그쪽으로 돌린다.

감지에는 헬스체크가 쓰인다. 일정 주기로 ping이든 HTTP 요청이든 보내서 응답이 없으면 죽은 걸로 본다. 이게 생각보다 까다롭다. 한 번 응답이 없다고 죽었다고 판정하면 일시적 네트워크 끊김에도 failover가 일어난다. 그래서 보통 "3번 연속 실패" 같은 규칙을 둔다. 그러면 감지에 수십 초가 걸린다.

승격은 다양한 방식이 있다. RDS Multi-AZ는 클라우드 사업자가 알아서 한다. 자체 구성이라면 etcd나 Consul 같은 분산 합의 시스템에 리더 선출을 맡기거나, 전용 manager 프로세스를 둔다(PostgreSQL의 Patroni, MySQL의 Orchestrator).

### 5.2 자동 failover의 위험

자동 failover는 편리하지만 잘못 동작하면 더 큰 사고를 만든다.

가장 흔한 사고는 "네트워크 일시 단절로 failover가 일어났는데 사실 Primary는 멀쩡했다"는 경우다. Primary와 Standby 사이의 네트워크만 끊긴 상황(파티션)에서 Standby가 Primary가 죽었다고 판단하고 자신을 승격시킨다. 이러면 Primary와 새 Primary가 둘 다 살아 있는 상태가 된다. 이게 split-brain이다.

또 한 가지는 "failover 도중에 다시 살아났다"는 경우다. Primary가 잠깐 GC pause로 멈췄을 뿐인데 Standby가 승격해 버린다. 잠시 후 원래 Primary가 깨어나서 자기는 여전히 Primary라고 생각하고 쓰기를 받는다. 데이터가 갈라진다.

이걸 막으려면 STONITH(Shoot The Other Node In The Head)나 fencing이 필요하다. 새 Primary가 되기 전에 옛 Primary를 확실히 죽여서(전원을 끄거나 네트워크에서 격리해서) 절대 다시 깨어나지 않도록 한다. 이게 분산 시스템에서 가장 골치 아픈 문제 중 하나다.

### 5.3 수동 failover와의 절충

자동 failover의 위험 때문에 수동 failover를 선호하는 팀도 많다. 알람이 울리면 사람이 상황을 보고 결정한다. 다만 새벽 3시에 사람이 깨서 결정하는 데 시간이 걸린다. RTO가 분 단위로 늘어난다.

타협안으로 "감지는 자동, 승격은 수동" 또는 "낮에는 자동, 밤에는 알람만"으로 운영하는 경우도 있다. 정답은 없고 서비스의 RTO 요구와 운영 인력에 달려 있다.

### 5.4 클라이언트 측 failover

서버만 failover된다고 끝이 아니다. 클라이언트가 새 endpoint를 알아야 한다.

DNS로 endpoint를 풀면 DNS TTL만큼 캐시가 남아 있다. TTL 60초면 60초 동안은 옛 IP로 요청이 간다. 그동안 사용자는 에러를 본다. 이걸 줄이려면 TTL을 짧게 잡거나(5~10초), 클라이언트에서 connection 재시도 로직을 잘 만들어야 한다.

DB 클라이언트는 connection pool을 쓰는 경우가 많다. failover가 일어나면 기존 connection은 다 끊긴다. pool이 새 endpoint로 다시 연결하는 동안 짧은 에러가 발생한다. 애플리케이션이 이걸 retry로 흡수해줘야 한다. retry 없이 그대로 사용자에게 노출하면 failover 시점마다 짧은 5xx 폭증이 보인다.

## 6. RTO와 RPO 산정

### 6.1 정의

RTO(Recovery Time Objective)는 "장애 발생부터 서비스 복구까지 허용 가능한 최대 시간"이다. RTO 5분이면 5분 안에 다시 응답을 시작해야 한다.

RPO(Recovery Point Objective)는 "장애 발생 시 허용 가능한 최대 데이터 손실 시간"이다. RPO 1초면 마지막 1초 안의 데이터는 잃어도 좋다.

두 값은 비즈니스 요구에서 나온다. 인프라가 정하는 게 아니라 비즈니스가 정하고, 인프라가 그걸 맞추는 비용을 산정한다.

### 6.2 SLO 자릿수와 RTO의 관계

연간 가용성 99.99%는 다운타임 52분이다. 한 번의 사고가 RTO 30분이면 1년에 한 번만 사고 나도 99.99%를 지킬 수 있다. 그런데 매번 30분 안에 복구한다는 보장은 없다. 큰 사고는 보통 한두 시간이다. 그래서 실제 운영 SLO가 목표 SLO보다 한 자릿수 낮은 경우가 흔하다.

```
SLO       연 다운타임   1회 사고가 RTO 30분이면 허용 사고 수
99%       3일 15시간    매주 한 번 나도 됨
99.9%     8시간 45분    월 1~2회 허용
99.95%    4시간 22분    분기 1~2회
99.99%    52분          연 1회 정도가 한계
99.999%   5분           사실상 자동 복구만으로 가능
```

### 6.3 RPO를 정확히 측정하기

RPO는 복제 방식에서 거의 정해진다.

- 동기 복제: RPO ≈ 0
- 준동기 복제: RPO < 1초(보통)
- 비동기 복제: RPO = replication lag (보통 1초 이내, 가끔 분 단위)
- 백업만 의존: RPO = 백업 주기(보통 시간 단위, 일 단위 백업이면 24시간)

여기서 함정은 비동기 복제의 RPO를 "1초 이하"로 잡았는데 실제 운영 중에는 lag가 30초까지 벌어지는 경우다. 평균값이 아니라 최악값을 잡아야 한다. lag 모니터링을 하지 않으면 RPO 산정 자체가 의미가 없다.

### 6.4 RTO 산정 시 빠뜨리는 것

RTO 산정은 보통 "DB failover에 30초" 같은 식으로 한다. 그런데 실제 사고에선 이게 RTO의 일부일 뿐이다.

- 감지: 헬스체크가 죽음을 인지하기까지 (보통 10~60초)
- 알림: 알람이 울리고 사람이 깨기까지 (수동이라면 5~30분)
- 판단: 상황을 보고 failover를 결정하기까지 (수동이라면 5~15분)
- 승격: 자동이라면 수십 초, 수동이라면 명령어 실행 시간
- DNS 전파: TTL만큼 (60초~5분)
- 클라이언트 재연결: 보통 수십 초

자동 failover라도 "감지 30초 + 승격 30초 + DNS 60초 + 재연결 30초 = 150초"가 나온다. 수동이면 여기에 사람 깨는 시간을 더해야 한다.

### 6.5 RTO/RPO 별 인프라 비용의 점프

| RTO / RPO | 가능한 구성 | 상대 비용 |
|---|---|---|
| RTO 시간 단위, RPO 시간 단위 | 단일 노드 + 백업 | 1x |
| RTO 10분, RPO 분 단위 | 단일 AZ Active-Passive + 비동기 복제 | 1.3x |
| RTO 1분, RPO 1초 | 멀티 AZ Active-Passive + 준동기 | 2x |
| RTO 10초, RPO 0 | 멀티 AZ Active-Active + 동기 | 3x |
| RTO 1초, RPO 0 (리전 단위) | 멀티 리전 Active-Active + 글로벌 일관성 | 5x+ |

마지막 칸이 가장 비싸다. 멀티 리전 강한 일관성은 글로벌 분산 합의를 요구하기 때문에 지연이 누적된다. Spanner나 CockroachDB 같은 시스템을 써야 하고, 자체 구축은 거의 불가능에 가깝다.

## 7. Split-brain 방지

### 7.1 왜 일어나는가

Split-brain은 네트워크 파티션 상황에서 발생한다. 클러스터가 두 그룹으로 갈라졌는데 양쪽이 모두 자기가 살아있는 다수파라고 판단하면, 양쪽이 동시에 쓰기를 받는다. 데이터가 갈라진다. 나중에 네트워크가 복구되어도 어느 쪽 데이터가 맞는지 알 수 없다.

이게 왜 일어나는지 이해하려면 CAP 정리를 떠올리면 된다. 네트워크 파티션(P)이 일어났을 때 일관성(C)과 가용성(A) 둘 다는 가질 수 없다. 일관성을 포기하면 split-brain이고, 가용성을 포기하면 한쪽이 거부하면서 서비스가 부분적으로 멈춘다.

### 7.2 Quorum 기반 방지

가장 흔한 방지법은 quorum이다. 클러스터를 홀수 개로 구성하고, 과반수(majority)가 살아있는 쪽만 쓰기를 받는다. 5대 클러스터라면 3대 이상이 모인 그룹만 권한을 갖는다. 네트워크가 2:3으로 갈라지면 3대 쪽만 살아있고 2대 쪽은 read-only로 떨어지거나 거부한다.

이게 etcd, Consul, ZooKeeper의 기본 동작이다. Raft나 Paxos 같은 분산 합의 알고리즘이 이 quorum 보장을 해준다.

여기서 주의할 점은 "왜 홀수인가"다. 4대 클러스터에서 2:2로 갈라지면 양쪽 모두 quorum이 아니다. 둘 다 멈춘다. 5대로 가면 2:3으로 갈라져도 한쪽은 살아있다. 짝수는 가용성을 깎고 비용만 늘린다.

### 7.3 Witness/Arbiter 노드

DB 클러스터를 두 AZ에 띄우면 자연히 split-brain 위험이 생긴다. AZ A에 2대, AZ B에 2대 두면 AZ 간 네트워크가 끊겼을 때 양쪽 모두 2:2로 동수다.

이걸 해결하는 한 방법이 witness(또는 arbiter) 노드를 제3의 AZ에 두는 것이다. AZ A에 2대, AZ B에 2대, AZ C에 1대(witness). witness는 쓰기를 받지는 않지만 quorum 판정에 참여한다. AZ A와 B 사이 네트워크가 끊기면 witness가 어느 쪽에 붙는지에 따라 quorum이 결정된다. MongoDB의 arbiter, SQL Server의 witness 서버가 같은 개념이다.

### 7.4 STONITH와 Fencing

분산 합의로 split-brain을 막을 수 없는 시스템(예: 대부분의 단순 Primary-Replica DB)에선 fencing이 필요하다. 새 Primary가 되기 직전에 옛 Primary를 확실히 죽인다.

방법은 여러 가지다. 옛 Primary가 돌던 EC2 인스턴스를 강제로 stop하거나, SAN의 볼륨을 분리하거나, 네트워크에서 격리한다(보안 그룹 변경). 클라우드에선 API 호출로 가능하지만, on-premise에선 전원을 끄는 PDU(power distribution unit)를 직접 제어하는 식으로 한다.

Fencing이 실패하면 split-brain이 일어난다. 그래서 fencing은 반드시 성공해야 한다. fencing이 timeout으로 실패하면 failover 자체를 포기하는 게 보통이다. "둘 다 살아있는 것보단 한 쪽만 살아있는 게 낫다"는 원칙이다.

### 7.5 Last-write-wins의 한계

위에서 멀티 리전 Active-Active를 얘기할 때 last-write-wins(LWW) 충돌 해결을 잠깐 언급했다. 이게 split-brain 상황에서 데이터를 복구하는 흔한 방식이다.

LWW는 단순하지만 데이터를 조용히 잃는다. 사용자 A가 한쪽 리전에 쓰고, 사용자 B가 다른 쪽 리전에 같은 키에 쓰고, 나중에 동기화될 때 timestamp가 큰 쪽이 이긴다. 진 쪽 데이터는 없어진 것처럼 보인다. 사용자가 잘못이라고 느낄 수 있다.

CRDT는 이걸 의미적으로 합쳐주는 방식이다. 카운터, 집합, 텍스트 같은 자료구조에 대해 충돌 해결 규칙이 미리 정해져 있다. Riak, Redis CRDT 같은 시스템이 지원한다. 다만 모든 데이터 타입에 적용할 수 있는 건 아니다.

## 8. 헬스체크와 자동 복구

### 8.1 헬스체크의 두 가지 종류

헬스체크는 보통 두 가지를 섞어 쓴다.

Liveness check는 "프로세스가 살아있는가"를 본다. HTTP 응답이 200이면 산다. 죽으면 죽었다. 단순하다.

Readiness check는 "요청을 받을 준비가 되었는가"를 본다. 프로세스는 떠 있지만 DB 연결이 안 됐거나, 캐시가 채워지지 않았거나, 외부 의존성이 끊겨 있으면 not ready다.

쿠버네티스는 둘을 명확히 구분한다. Liveness가 실패하면 컨테이너를 재시작한다. Readiness가 실패하면 서비스 endpoint에서 뺀다(트래픽 차단). 둘을 헷갈리면 무한 재시작 루프에 빠지거나 죽은 노드에 트래픽이 가는 사고가 난다.

### 8.2 헬스체크가 노드를 죽이는 시나리오

헬스체크가 너무 적극적이면 멀쩡한 노드를 빼버린다. 가장 흔한 시나리오 몇 가지다.

첫째, "DB 연결 한 번 실패했더니 readiness가 fail로 떴다". 사용자 트래픽은 정상인데 일시적 DB 연결 끊김으로 헬스체크만 실패. 노드가 빠진다. 남은 노드에 부하가 몰린다. 남은 노드도 헬스체크 실패. 연쇄 장애. 이걸 cascading failure라고 한다.

둘째, "헬스체크 응답이 5초 timeout". 서비스가 평상시 100ms로 응답하다가 일시적 부하로 6초가 됐다. 헬스체크는 timeout. 노드가 죽었다고 판단. 트래픽이 다른 노드로 몰리면서 그쪽도 6초 응답. 연쇄.

셋째, "헬스체크 자체가 DB를 쿼리". 헬스체크가 매번 SELECT 1을 날린다. DB가 잠깐 느려진다. 헬스체크가 다 timeout. 모든 노드가 unhealthy로 판정. 클러스터 전체가 트래픽에서 빠진다. 이게 가장 무섭다.

이걸 막으려면 헬스체크의 의미를 좁혀야 한다. "이 노드가 트래픽을 받을 수 있는가"만 본다. DB가 죽었으면 그건 DB의 문제다. 노드를 빼면 안 된다(전체가 죽을 거다). 노드 자체의 상태만 본다.

### 8.3 헬스체크 주기와 임계값

헬스체크는 주기와 임계값을 둘 다 정해야 한다.

주기는 보통 5~30초다. 너무 짧으면 헬스체크가 시스템에 부하를 준다. 너무 길면 죽은 노드를 늦게 인지한다. 클러스터 크기와 트래픽 양에 맞춰 정한다.

임계값은 "몇 번 연속 실패하면 unhealthy로 판단할지"다. 1번이면 일시적 끊김에 민감하다. 5번이면 25~150초가 걸려서 인지가 늦다. 보통 2~3번이 무난하다.

같은 임계값을 healthy 복귀에도 적용한다. "1번 성공하면 healthy"로 잡으면 flapping(unhealthy/healthy 반복)이 일어난다. healthy 복귀는 더 보수적으로(3~5번 연속 성공) 잡는 게 안전하다.

### 8.4 자동 복구의 함정

자동 복구는 깔끔한 그림이지만 두 가지 사고 패턴이 있다.

첫째, 무한 재시작 루프. 컨테이너가 시작 직후 죽는다. 쿠버네티스가 재시작한다. 또 죽는다. 또 재시작. 로그도 못 남기고 그냥 죽어 나간다. 이걸 방지하려고 쿠버네티스는 backoff(점점 늦게 재시작)를 적용하지만, 그래도 디버깅이 어렵다.

둘째, 자동 복구가 진짜 문제를 가린다. CPU가 100% 찍히면 자동 스케일링이 노드를 추가한다. 추가된 노드도 100% 찍는다. 또 추가. 비용이 폭증하고 진짜 원인(쿼리 비효율, 무한 루프 등)은 발견이 늦어진다. 자동 복구는 디버깅 시간을 벌어주는 것이지 원인 분석을 면제해주는 게 아니다.

### 8.5 Graceful shutdown과의 관계

헬스체크와 자동 복구를 얘기할 때 빠뜨리면 안 되는 게 graceful shutdown이다.

배포 중에 노드를 내리거나, 스케일 다운으로 노드가 빠질 때 진행 중이던 요청은 어떻게 되는가. 그냥 죽이면 진행 중인 요청은 모두 5xx로 끝난다. 평상시에도 배포할 때마다 에러 폭증이 보인다.

이걸 막는 방법은 이렇다. 노드를 내리기로 결정하면 먼저 readiness를 false로 만들어서 로드 밸런서가 새 트래픽을 안 보내게 한다. 잠시(보통 10~30초) 기다려서 진행 중인 요청이 끝나도록 한다. 그다음 프로세스를 종료한다. SIGTERM을 받았을 때 즉시 종료하지 말고 in-flight 요청을 끝마치는 코드가 애플리케이션에 있어야 한다.

쿠버네티스는 이 시나리오를 위해 preStop hook과 terminationGracePeriodSeconds를 제공한다. AWS ALB는 deregistration delay 설정으로 같은 일을 한다. 이걸 설정하지 않으면 배포가 가용성을 깎는 주범이 된다.

## 9. 실제 장애 시나리오와 대응

### 9.1 단일 AZ 장애

가장 흔하다. 1년에 한두 번 정도 클라우드 사업자가 AZ 단위 사고를 낸다.

```
[정상]                     [AZ-A 장애]
   LB                          LB
  / | \                       / | \
 V  V  V                     X  V  V
A1 A2 B1 ─ B2  ...           A1 A2 B1 ─ B2
(AZ-A)  (AZ-B)              (AZ-A)  (AZ-B)
                              ↓
                            트래픽 전부 AZ-B로
```

대응의 핵심은 평소에 멀티 AZ로 띄워두는 것뿐이다. 사고가 나면 LB가 자동으로 죽은 AZ를 뺀다. 사용자는 짧은 에러를 본 뒤 정상으로 돌아온다.

자주 빠뜨리는 점은 "남은 AZ가 모든 트래픽을 받을 수 있는가"다. 평소 두 AZ가 50%씩 받았다면 한 AZ가 100%를 받아야 한다. 평소 사용률 60%였다면 한 AZ가 죽는 순간 남은 AZ는 120%다. 같이 죽는다. 그래서 멀티 AZ를 가져갈 땐 평상시 사용률을 50% 이하로 잡는 게 일반적이다.

또 한 가지는 RDS Multi-AZ failover 시간이다. AWS가 30~120초 정도라고 하지만 실제로는 더 걸리는 경우도 있다. 그 사이 모든 쓰기는 실패한다. 애플리케이션이 retry로 흡수해줘야 한다.

### 9.2 DB Primary 장애

DB Primary가 죽었을 때 시나리오는 자동 failover 구성 여부에 따라 다르다.

자동 failover가 켜져 있으면(RDS Multi-AZ, Patroni 등) 보통 30초~2분 내에 Standby가 승격한다. 그동안 모든 쓰기는 실패한다. 클라이언트는 새 endpoint로 다시 연결한다. DNS TTL이 짧다면 1분 내에 정상화된다.

자동 failover가 없으면 사람이 알람을 받고 수동으로 한다. 새벽 3시면 사람이 깨는 데 10분, 상황 파악에 10분, 실행에 5분으로 총 25분이 보통이다. RPO는 동기 복제라면 0, 비동기라면 lag만큼.

대응 시 가장 흔한 실수는 "옛 Primary가 다시 살아났는데 그대로 두는 것"이다. 새 Primary와 옛 Primary가 둘 다 살아있으면 split-brain이다. 옛 Primary는 격리하거나 read-only로 강등시켜야 한다.

### 9.3 캐시 장애 (cache stampede)

Redis 같은 캐시가 죽으면 모든 요청이 DB로 직접 간다. DB가 평상시 캐시 적중률을 95% 가정하고 산정되어 있으면 트래픽이 20배가 된다. DB도 같이 죽는다.

대응은 여러 단계가 있다. 첫째, 캐시 자체를 이중화한다. Redis Sentinel이나 Redis Cluster를 쓴다. 캐시 노드 하나가 죽어도 다른 노드가 받는다.

둘째, 캐시가 통째로 죽었을 때를 위해 회로 차단기(circuit breaker)를 둔다. 캐시 실패가 일정 비율 이상이면 캐시를 우회하지 말고 차라리 일부 요청을 거부한다. DB를 보호한다.

셋째, "cache stampede 방지"를 한다. 같은 키에 대한 동시 요청이 모두 DB로 가지 않도록 락이나 singleflight 패턴을 쓴다. 한 요청만 DB로 가서 결과를 캐시에 채우고, 다른 요청들은 그걸 기다린다.

### 9.4 외부 의존성 장애

외부 API(결제 게이트웨이, 메시징 서비스 등)가 죽었을 때다. 우리 시스템은 멀쩡한데 의존하는 외부가 죽어서 우리도 같이 죽는 상황을 막아야 한다.

이건 회로 차단기와 타임아웃의 영역이다. 외부 API의 응답이 5초를 넘으면 timeout 처리하고 사용자에겐 "잠시 후 다시 시도해 주세요"를 보낸다. 5초 동안 worker가 잡혀 있으면 다른 요청도 못 받는다. timeout 없이 그냥 기다리는 코드가 가장 위험하다.

외부 API의 실패율이 일정 비율을 넘으면 회로 차단기가 열린다. 그 뒤로는 외부 API를 호출하지 않고 즉시 fallback 응답을 한다(예: 캐시된 옛 데이터, 또는 degraded 응답). 일정 시간이 지나면 시험적으로 다시 호출해보고 성공하면 닫는다.

### 9.5 배포 사고

배포가 가용성을 깎는 가장 흔한 이유 중 하나다.

가장 흔한 패턴은 "새 버전을 배포했더니 시작 시 readiness가 늦게 떴다"는 것이다. 옛 버전을 다 내렸는데 새 버전이 트래픽을 받을 준비가 안 됐다. 그 사이 모든 요청이 실패한다.

이걸 막는 방법은 rolling update를 천천히 하는 것이다. 한 번에 25% 또는 10%만 교체하고, 그게 readiness가 뜰 때까지 기다린다. 다음 묶음을 시작한다. 새 버전에 문제가 있으면 25%만 영향받고, 자동 또는 수동으로 롤백할 수 있다.

더 보수적으로는 blue-green이나 canary 배포를 쓴다. blue-green은 옛 환경을 통째로 두고 새 환경에 배포한 뒤 트래픽을 한 번에 전환한다. canary는 새 버전에 5%만 보내서 일정 시간 관찰하고 문제 없으면 비율을 늘린다. 둘 다 추가 인프라 비용이 든다.

### 9.6 의존성 사슬에서의 장애 전파

마이크로서비스 환경에서 가장 까다로운 패턴이다. 서비스 A → B → C → D 순서로 호출한다. D가 느려진다. C의 요청이 큐에 쌓인다. C의 메모리가 부족해진다. C가 OOM으로 죽는다. B에서 C 호출이 timeout으로 실패. B도 같은 일이 일어난다. 마지막엔 A가 죽고 사용자에게 5xx가 보인다.

이걸 막으려면 모든 서비스 호출에 timeout과 회로 차단기를 두는 게 기본이다. 그리고 backpressure를 흘려보내야 한다. C가 느려지면 B는 큐에 쌓지 말고 빠르게 거부한다. 거부는 거슬리지만 OOM으로 죽는 것보단 낫다. 거부된 요청은 A에서 retry 정책에 따라 처리한다(또는 사용자에게 에러).

bulkhead 패턴도 자주 쓴다. 외부 호출별로 별도의 connection pool과 worker pool을 둔다. C 호출이 느려져도 그 풀만 막히고 다른 호출은 영향이 없다. 한 곳의 장애가 시스템 전체로 번지지 않는다.

## 10. 실무에서 자주 잊는 것들

지금까지의 내용을 운영 중에 체화하는 데 도움이 됐던 몇 가지 정리다.

**HA는 정기적으로 테스트하지 않으면 작동하지 않는다.** Active-Passive를 1년 동안 한 번도 failover해보지 않으면 사고 당일 처음으로 작동시키게 된다. 보통 안 깨어난다. Chaos engineering이 거창해 보이지만, 한 달에 한 번 Standby를 강제로 승격시켜 보는 정도면 많이 잡힌다.

**모니터링이 없으면 HA도 의미가 없다.** Replication lag, 헬스체크 결과, failover 이벤트가 모니터링에 잡혀야 한다. lag가 분 단위로 벌어졌는데 모르고 있으면 RPO 산정이 거짓말이 된다.

**RTO/RPO는 비즈니스와 합의해야 한다.** 인프라가 일방적으로 정하면 안 된다. "결제 시스템 RPO 0"을 인프라가 결정할 수 없다. 비즈니스가 "1초 이하 데이터 손실 시 회사에 얼마의 손실이 나는가"를 정하고, 인프라가 그걸 맞추는 비용을 산정해 협의한다.

**의존하는 컴포넌트의 SLO를 더한다.** 우리 SLO가 99.95%여도 의존하는 DB가 99.9%라면 실제로는 99.85% 정도가 한계다. 직렬로 의존하는 컴포넌트들의 가용성을 곱하면 전체 가용성이 나온다. 의존성이 많을수록 빠르게 깎인다.

**가용성을 깎는 가장 큰 원인은 배포와 운영자다.** 통계적으로 사고의 절반 이상은 사람의 변경에서 나온다. 클라우드 사업자의 인프라 장애보다 자체 배포 사고가 훨씬 잦다. 그래서 자동화된 롤백, canary, runbook이 인프라 이중화만큼 중요하다.

**비용 곡선은 자릿수마다 점프한다.** 99% → 99.9%는 비교적 싸다. 99.9% → 99.99%는 보통 두세 배. 99.99% → 99.999%는 거의 무한대로 보면 된다. 비즈니스가 진짜 그 자릿수를 필요로 하는지 매번 다시 물어야 한다. "최대한 높게"는 답이 아니다.
