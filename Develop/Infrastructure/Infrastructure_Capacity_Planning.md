# 인프라 용량 산정

용량 산정은 "지금 이 서비스가 몇 대의 어떤 스펙 장비를 필요로 하는가"를 숫자로 정하는 작업이다. 감으로 잡으면 두 가지 결과가 나온다. 모자라서 장애가 나거나, 남아돌아 비용을 태운다. 둘 다 흔하다.

5년차쯤 되면 깨닫는 게 있다. 정확한 용량 산정은 불가능하다. 트래픽은 항상 예측을 빗나가고, 새 기능은 항상 예상보다 무겁고, 장애는 항상 새벽 3시에 터진다. 산정의 목표는 "정확하게 맞히기"가 아니라 "틀려도 죽지 않을 마진을 확보하기"다.

## 1. 산정의 출발점: 무엇을 측정할 것인가

용량 산정을 시작할 때 가장 흔한 실수가 곧바로 "CPU 몇 코어, 메모리 몇 GB"를 묻는 것이다. 그 전에 정해야 할 게 있다.

### 1.1 비즈니스 지표를 인프라 지표로 환산

산정의 입력값은 비즈니스 지표다. 출력값이 인프라 지표다. 둘을 잇는 환산표가 없으면 산정 자체가 의미가 없다.

예를 들어 커머스 서비스라면 이런 환산이 필요하다.

```
일 주문 1만 건
→ 일 페이지뷰 약 50만 (주문 1건당 50 PV 가정, 측정값)
→ 일 API 호출 약 250만 (PV 1건당 평균 5 API 콜)
→ 평균 RPS 약 29
→ 피크 RPS 약 290 (피크/평균 비율 10배, 측정값)
```

여기서 "측정값"이라고 적은 부분이 중요하다. PV당 API 콜 수, 피크/평균 비율은 추측하면 안 된다. 운영 중인 서비스라면 APM이나 액세스 로그에서 뽑아내야 한다. 신규 서비스라면 유사 서비스 데이터를 빌리거나, 최소 산정으로 시작해서 부하 테스트로 검증한다.

### 1.2 SLO부터 정한다

용량은 SLO의 함수다. "응답시간 p99 200ms, 가용성 99.95%"를 목표로 하느냐 "p99 1초, 99.9%"를 목표로 하느냐에 따라 필요한 장비 수가 두 배 이상 차이 난다.

특히 p99 같은 꼬리 지연(tail latency)을 SLO로 잡으면 평균 사용률을 60% 이하로 유지해야 한다. 평균 80%로 굴리면 큐잉 효과 때문에 p99가 폭발한다. 이건 대기 행렬 이론에서 나오는 결과다. 사용률 ρ일 때 평균 대기시간은 ρ/(1-ρ)에 비례한다. ρ=0.5면 1, ρ=0.8이면 4, ρ=0.9면 9, ρ=0.95면 19로 늘어난다.

## 2. CPU 산정

### 2.1 기본 공식

CPU 산정은 한 번의 요청이 CPU 코어를 몇 ms 점유하는지에서 출발한다.

```
필요 코어 수 = (RPS × 요청당 CPU 시간(ms) / 1000) / 목표 사용률
```

예를 들어 피크 RPS 300, 요청당 CPU 50ms, 목표 사용률 60%라면:

```
필요 코어 = (300 × 50 / 1000) / 0.6 = 25 코어
```

요청당 CPU 시간은 다음 두 가지를 헷갈리면 안 된다.

- **CPU time**: 실제 코어를 점유한 시간
- **wall time(응답시간)**: 시계상 흐른 시간

DB 쿼리를 기다리는 시간은 wall time에는 들어가지만 CPU time에는 들어가지 않는다. 산정에는 CPU time을 써야 한다. JVM이라면 `ThreadMXBean.getCurrentThreadCpuTime()`, Linux라면 `getrusage(RUSAGE_THREAD)`나 pprof로 측정한다.

### 2.2 vCPU와 물리 코어의 함정

클라우드의 vCPU는 대부분 SMT(하이퍼스레딩)된 논리 코어다. 물리 코어 1개가 vCPU 2개로 노출된다. 같은 코어를 두 스레드가 나눠 쓰니까 vCPU 2개를 풀로 돌려도 물리 코어 1개보다 1.2~1.5배 성능밖에 안 나온다.

산정할 때 vCPU 8개를 물리 코어 8개라고 가정하면 실측에서 30% 부족한 용량이 나온다. CPU 바운드 워크로드일수록 이 차이가 크다. 산정값에 SMT 보정 계수(보통 0.6~0.7)를 곱하거나, 처음부터 물리 코어 기준 벤치마크를 확보해두는 게 안전하다.

### 2.3 컨테이너 CPU limit

쿠버네티스에서 `cpu: 2`로 limit을 걸면 CFS quota가 동작한다. 100ms 주기에서 200ms만 CPU를 쓸 수 있다는 뜻이다. 문제는 이 quota가 throttling을 일으킨다는 점이다.

요청 처리 중에 throttling이 걸리면 응답시간이 튄다. p99 200ms 목표인데 throttling 때문에 갑자기 600ms가 찍힌다. limit을 request의 2~3배로 잡거나, 아예 limit을 빼고 request만 설정하는 운영팀도 많다. 둘 다 일리가 있고 둘 다 함정이 있다.

```yaml
resources:
  requests:
    cpu: "2"
    memory: "4Gi"
  limits:
    cpu: "4"      # request의 2배. 버스트 허용
    memory: "4Gi" # 메모리는 request와 limit을 같게
```

## 3. 메모리 산정

### 3.1 메모리는 평균이 아니라 피크

CPU는 평균 사용률로 산정해도 큰 문제가 없다. 메모리는 다르다. 메모리는 피크 사용량 + 헤드룸으로 산정해야 한다. 평균 60%로 쓰는 서버가 피크에 95%까지 올라가면 OOM Kill로 즉시 죽는다. CPU처럼 throttle되지 않는다.

```
필요 메모리 = (기본 메모리 + 동시 요청 수 × 요청당 메모리) × 1.5
```

여기서 1.5는 헤드룸이자 안전 계수다. JVM이라면 힙 외에 메타스페이스, 코드 캐시, 스레드 스택, 다이렉트 메모리, GC가 쓰는 공간이 추가로 든다. Go라면 GC가 헤드룸 100%까지 잡아둘 수 있다(GOGC=100 기본값).

### 3.2 OS 페이지 캐시 고려

DB나 검색엔진처럼 디스크 I/O가 많은 워크로드는 OS 페이지 캐시가 성능을 좌우한다. PostgreSQL이 shared_buffers를 8GB로 잡았다고 메모리를 8GB만 주면 안 된다. 인덱스와 핫 데이터가 페이지 캐시에 올라가야 한다.

```
PostgreSQL 메모리 = shared_buffers + work_mem × max_connections × 2 + maintenance_work_mem + 페이지 캐시 여유
```

페이지 캐시는 보통 핫 데이터셋의 1.5~2배를 목표로 잡는다. 핫 데이터셋 크기를 모르면 `pg_stat_io`나 `pg_buffercache`를 뜯어봐야 한다.

### 3.3 메모리 누수 보정

장기 운영하면 어디선가 누수가 난다. 코드 버그일 수도 있고, 캐시가 의도와 다르게 자라는 경우도 흔하다. 일주일에 1~2GB씩 늘어나는 게 별로 드문 일이 아니다.

산정할 때 "재시작 주기"를 같이 정해야 한다. 매일 롤링 재시작을 하는 운영이면 누수 보정이 작아도 된다. 한 달에 한 번 배포한다면 한 달 누적치를 헤드룸에 더해야 한다.

## 4. 디스크 산정

### 4.1 용량과 IOPS는 다른 축이다

디스크 산정에서 가장 흔한 실수가 "용량만 보고" 결정하는 것이다. AWS gp3 100GB와 1TB는 가격은 10배 차이가 나지만, 기본 IOPS는 동일하게 3000 IOPS다. 1TB로 키운다고 IOPS가 늘어나는 게 아니다. gp2는 GB당 3 IOPS라 용량이 IOPS를 결정하지만, gp3는 분리되어 있다.

산정할 때 다음 네 가지를 각각 봐야 한다.

- 용량 (GB)
- 읽기 IOPS (피크)
- 쓰기 IOPS (피크)
- 처리량 (MB/s, 시퀀셜 워크로드일 때)

### 4.2 용량 증가율

데이터 증가는 선형이 아닌 경우가 많다. 사용자가 늘면 사용자당 데이터도 늘어난다(컨텐츠가 쌓이니까). 그래서 일일 증가량의 trailing 30일 평균을 보고 30~50% 마진을 더해 6개월치를 잡는다.

```
필요 용량 = (현재 사용량 + 일 증가량 × 180 × 1.4) × 1.3
```

마지막 1.3은 "사용량 80%에서 알람이 와야 늘릴 시간이 있다"는 운영적 헤드룸이다. 디스크 확장은 보통 무중단이지만 EBS는 6시간 쿨다운, GCP PD는 더 짧고, 온프레미스는 더 길다. 사용량 95%에서 알람을 받으면 확장 트리거가 늦는다.

### 4.3 IOPS 산정

요청 1건이 디스크에 몇 번 닿는지를 측정한다. 이건 애플리케이션 레벨이 아니라 DB와 파일시스템 레벨에서 봐야 한다.

```
필요 IOPS = RPS × 요청당 IOPS × 안전계수(2)
```

요청당 IOPS는 `iostat -x 1`로 측정한다. 캐시 히트가 잘 되면 1 미만으로 떨어지고, 분석 쿼리가 섞이면 수십~수백까지 튄다. PostgreSQL이라면 `EXPLAIN (ANALYZE, BUFFERS)`의 "Buffers: shared read" 값이 디스크 IOPS의 출처다.

### 4.4 처리량 산정

대량 ETL, 백업, 로그 출력은 IOPS보다 처리량(throughput)이 병목이다. gp3는 기본 125MB/s, 추가 비용으로 1000MB/s까지 올라간다. NVMe instance store는 GB/s 단위다.

로그 처리량 산정 예시:

```
JSON 한 줄 평균 1KB
RPS 300 × 요청당 로그 5줄 = 1500줄/s
처리량 = 1500 × 1KB = 1.5MB/s
```

로그는 별것 아닌 듯해도 트래픽 폭증 시 응답시간보다 먼저 디스크를 채우는 경우가 흔하다. 특히 에러 로그가 스택트레이스를 통째로 찍으면 한 줄 평균이 갑자기 10~50KB로 늘어난다.

## 5. 네트워크 대역폭 산정

### 5.1 인스턴스 타입별 한계

클라우드 인스턴스는 사양표에 적힌 네트워크 대역폭이 두 가지다.

- **Baseline**: 항상 보장되는 대역폭
- **Burst**: 일시적으로 가능한 대역폭, 크레딧 소진 시 baseline으로 떨어진다

AWS의 t3.large는 burst가 5Gbps라고 적혀 있지만 baseline은 0.7Gbps다. 평소엔 괜찮다가 대용량 응답이 몰리면 burst 크레딧이 소진되어 갑자기 응답시간이 늘어난다. 이 증상은 CPU, 메모리 메트릭으로는 안 보인다. CloudWatch의 `NetworkBandwidthAllowanceExceeded` 같은 메트릭을 따로 봐야 한다.

상시 대역폭이 필요한 서비스는 burst 의존 인스턴스를 쓰면 안 된다. m6i, c6i처럼 baseline이 명시된 인스턴스를 쓴다.

### 5.2 처리량 계산

```
필요 대역폭(Mbps) = RPS × (요청 크기 + 응답 크기) × 8 / 1,000,000
```

요청 1KB, 응답 50KB, RPS 300이라면:

```
필요 대역폭 = 300 × 51KB × 8 / 1000 = 약 122Mbps
```

이게 애플리케이션 트래픽이고, 여기에 다음을 더해야 한다.

- DB 트래픽 (애플리케이션 ↔ DB)
- 캐시 트래픽 (애플리케이션 ↔ Redis)
- 로그/메트릭 송신 트래픽
- 내부 서비스 호출 (마이크로서비스라면 이게 가장 크다)

마이크로서비스 환경에서는 east-west 트래픽이 north-south(외부 트래픽)의 10배 이상인 게 흔하다. 외부 응답 50KB지만 내부에서 5개 서비스를 거치며 각각 100KB씩 주고받으면 그쪽이 진짜 병목이다.

### 5.3 커넥션 수와 PPS

대역폭만큼 중요한 게 PPS(Packets Per Second)와 동시 커넥션 수다.

작은 응답을 많이 주고받는 워크로드(예: Redis, gRPC 헬스체크)는 대역폭은 여유로워도 PPS 한계에 먼저 부딪힌다. AWS EC2의 PPS 한계는 인스턴스 타입에 따라 다르고 공식 문서에 명시되지 않은 경우도 많다. ENA 드라이버의 `ethtool -S`로 `bw_in_allowance_exceeded`, `pps_allowance_exceeded` 카운터를 확인한다.

커넥션 수는 다른 이야기다. L4 로드밸런서(NLB, ALB)는 동시 커넥션 수 제한이 있고, 백엔드 인스턴스의 ephemeral port range는 약 28000개가 한계다. 한 백엔드에 28000개 이상의 outbound 커넥션을 만들 수 없다. 이건 커넥션 풀 설계할 때 반드시 본다.

## 6. 커넥션 풀 산정

### 6.1 DB 커넥션

가장 흔한 사고 패턴 중 하나가 DB 커넥션 산정 실수다.

```
DB 커넥션 = 애플리케이션 인스턴스 수 × 인스턴스당 커넥션 풀 크기
```

애플리케이션 인스턴스 20대, 풀 크기 30이면 600개의 커넥션을 만든다. PostgreSQL `max_connections` 기본값이 100이라 즉시 터진다. 600개로 늘리면 이번엔 PostgreSQL이 커넥션당 수 MB씩 잡아서 메모리가 폭발한다.

PostgreSQL에서 권장하는 동시 활성 커넥션 수는 코어 수의 2~3배 정도다. 16코어면 32~48개. 그 이상이 필요하면 PgBouncer 같은 커넥션 풀러를 앞에 둔다.

### 6.2 풀 크기 산정 공식

```
풀 크기 = (코어 수 × 2) + 디스크 수
```

이건 HikariCP가 권장하는 공식이고 잘 맞는다. 코어 4개에 디스크 1개면 풀 크기 9개. 풀 크기를 무작정 늘려도 처리량은 안 늘어난다. 컨텍스트 스위칭과 락 경합 때문에 오히려 떨어진다.

### 6.3 HTTP 클라이언트 풀

외부 API 호출용 HTTP 풀도 똑같은 산정이 필요하다. Apache HttpClient나 OkHttp의 기본값은 호스트당 2~5개로 매우 작다. 외부 API 호출이 많은 서비스에서 이 값을 늘리지 않으면 동시 요청이 큐에서 대기하다가 타임아웃이 난다.

```
HTTP 풀 = 동시 요청 수 × 외부 API 호출 비율 × 1.5
```

## 7. 피크 트래픽 추정

### 7.1 피크/평균 비율

산정의 기준은 평균이 아니라 피크다. 피크/평균 비율은 서비스 성격에 따라 다르다.

- B2C 커머스: 평균 대비 피크 5~10배 (저녁 시간대)
- B2C 미디어: 평균 대비 피크 3~5배 (방영 시간 등)
- B2B SaaS: 평균 대비 피크 2~3배 (업무 시간)
- 라이브 이벤트: 평균 대비 피크 50~100배

이 비율을 추측하지 말고 측정한다. 과거 30일 RPS의 분 단위 max와 평균을 비교한다. 측정값이 없으면 비슷한 도메인 서비스의 공개 데이터를 참고한다.

### 7.2 특수 이벤트 마진

연간 피크가 평소의 피크보다 또 몇 배 높다. 블랙프라이데이, 명절, 신규 콘텐츠 출시 같은 이벤트다. 이건 평소 산정에 들어가면 비용이 폭발하니까 오토스케일링과 사전 워밍업으로 대응한다.

- 사전 워밍업 가능한 이벤트: 시간을 알면 스케일 아웃 예약
- 사전 워밍업 불가한 이벤트: 헤드룸 50~100%를 평소에 확보

### 7.3 콜드 스타트 시간 = 마진

오토스케일링이 있어도 스케일 아웃에는 시간이 걸린다. EC2는 인스턴스 시작 1~3분 + 부트스트랩 1~5분 + 워밍업 30초~1분이다. ECS Fargate는 30초~1분, Lambda는 100ms~수 초다.

피크가 5분에 걸쳐 천천히 오는 거면 오토스케일링이 따라간다. 5초에 5배로 튀면 따라가지 못한다. "스케일 아웃 완료까지 걸리는 시간 × 평소 트래픽 증가 속도"만큼의 헤드룸이 필요하다.

## 8. 헤드룸 산정

헤드룸은 미리 확보해두는 여유 용량이다. 안전계수와 비슷한 개념인데 운영적 의미가 더 강하다.

### 8.1 헤드룸의 구성

| 항목 | 비율 | 이유 |
|---|---|---|
| 측정 오차 보정 | 10% | 부하 테스트와 실측이 항상 다르다 |
| 트래픽 변동 | 20% | 평균이 평균이 아니다. 분 단위 max는 더 높다 |
| 장애 시 트래픽 흡수 | N+1 만큼 | 인스턴스 1대 죽어도 SLO 유지 |
| 배포/롤링 | 25% | 배포 중에는 일시적으로 일부 인스턴스가 빠진다 |
| 트래픽 성장 | 10~30%/분기 | 다음 산정 주기까지의 자연 증가 |

이걸 다 합치면 평균 사용률 목표가 40~60%가 된다. CPU 평균 60% 넘으면 알람 받고 검토해야 한다는 게 여기서 나온다.

### 8.2 N+1과 N+2

가용성 산정에서 흔히 쓰는 표현이 N+1, N+2다. 정상 운영에 N대가 필요할 때 1대 또는 2대의 여유분을 더 운영한다는 뜻이다.

- N+1: 1대 장애 시에도 SLO 유지
- N+2: 1대 장애 + 1대 배포 진행 중에도 SLO 유지

N이 작을수록 +1, +2의 비용 비중이 크다. N=2면 +1이 50% 추가 비용이지만, N=10이면 10%다. 그래서 소규모 서비스는 N+1만 잡고, 중요 서비스는 N+2로 잡는다.

### 8.3 AZ 헤드룸

가용 영역(AZ) 단위 장애를 고려하면 헤드룸이 더 늘어난다. 3개 AZ에 분산했는데 1개 AZ가 빠지면 33%의 용량이 사라진다. 남은 2개 AZ가 평소 트래픽 + 빠진 AZ의 트래픽을 받으려면 평소 사용률이 (2/3) × 목표 = 약 40% 이하여야 한다.

이건 비용이 많이 든다. 그래서 모든 서비스를 AZ 장애 대응으로 산정하지는 않는다. 코어 서비스만 그렇게 잡고, 나머지는 멀티 AZ 분산은 하되 용량은 N+1 수준으로 가는 운영이 많다.

## 9. 오토스케일링 임계값

### 9.1 스케일 아웃 트리거

가장 흔한 임계값이 CPU 평균 70%다. 이게 늘 옳은 건 아니다.

- CPU 바운드 서비스: CPU 평균이 좋다
- I/O 바운드 서비스: CPU는 30%인데 응답시간이 1초로 늘어난다. 응답시간 기반이 맞다
- 메모리 바운드 서비스: 메모리 사용률 기반. CPU는 한가할 수 있다
- 큐 기반 워커: 큐 적체 길이가 가장 정확하다

쿠버네티스 HPA는 다중 메트릭을 지원한다. CPU, 메모리, 커스텀 메트릭(p95 latency, queue length 등)을 동시에 걸어두고 max를 트리거로 쓴다.

```yaml
metrics:
- type: Resource
  resource:
    name: cpu
    target:
      type: Utilization
      averageUtilization: 60
- type: Pods
  pods:
    metric:
      name: http_request_duration_p95
    target:
      type: AverageValue
      averageValue: 300m  # 300ms
```

### 9.2 스케일 인 임계값

스케일 아웃은 빠르게, 스케일 인은 천천히 한다. 트래픽이 줄어드는 듯하다가 다시 오르는 경우가 흔하기 때문이다. 줄였다 늘렸다를 반복하는 플래핑(flapping)이 일어나면 콜드 스타트가 자주 발생해서 응답시간이 흔들린다.

- 스케일 아웃 임계값: 60%
- 스케일 인 임계값: 40% (10~15분 지속 시)
- Stabilization window: 스케일 아웃 30초, 스케일 인 5~10분

### 9.3 최소/최대 인스턴스 수

```
minReplicas = 평소 트래픽 처리 인스턴스 수 × 0.7
maxReplicas = 피크 트래픽 처리 인스턴스 수 × 1.5
```

minReplicas를 너무 낮게 잡으면 새벽에 줄였다가 출근 시간 트래픽을 못 받는다. maxReplicas를 너무 낮게 잡으면 진짜 피크에서 막힌다. maxReplicas는 비용 폭발 방지 안전장치 정도로 보고 평소 피크의 1.5~2배로 잡는다.

## 10. 부하 테스트 기반 검증

산정값은 부하 테스트로 검증해야 한다. 책상 계산만으로 운영에 들어가면 첫 번째 피크에서 어디서 무엇이 터질지 모른다.

### 10.1 부하 테스트의 4단계

1. **스모크 테스트**: 1 RPS 정도. 시스템이 동작하는지만 본다
2. **램프업 테스트**: 0에서 목표 RPS까지 천천히 올린다. 어디서 응답시간이 무너지는지 본다
3. **정상 부하 테스트**: 피크 RPS로 30분~1시간 유지. 메모리 누수, GC, 커넥션 누수 확인
4. **스트레스 테스트**: 피크의 1.5~2배까지 올린다. 시스템이 어떻게 죽는지 본다

4번이 중요한데 자주 생략된다. 시스템이 어떻게 죽는지 알아야 운영 중 SLO 임계값을 어디에 잡을지 결정할 수 있다.

### 10.2 응답시간 곡선

부하 테스트의 출력은 RPS-응답시간 곡선이다. 보통 이렇게 생겼다.

```
응답시간 (ms)
  │
  │                                    ╱
  │                                  ╱
  │                                ╱
  │                              ╱
  │                          ╱╱
  │                     ╱╱╱
  │_______________╱╱╱╱
  │__________________
  └────────────────────────────────── RPS
  0          knee point         saturation
```

knee point가 산정 한계다. 그 이상으로 가면 응답시간이 비선형으로 튄다. 운영 최대치는 knee point의 60~70%에 잡는다.

### 10.3 부하 테스트의 함정

- **로컬에서 한 결과는 운영과 다르다**: 네트워크 latency, 공유 자원 경합이 다르다
- **단일 엔드포인트 테스트는 의미가 작다**: 실 트래픽 비율로 시나리오를 구성
- **DB는 따로 측정**: 같은 인프라에서 DB가 먼저 죽는 경우가 많다
- **워밍업 없는 측정은 의미가 작다**: JIT, 캐시 적재, 커넥션 풀 준비 후 측정
- **테스트 클라이언트도 한계가 있다**: k6, JMeter 자체가 병목이 되는지 확인

## 11. 잘못된 산정으로 인한 사고 패턴

### 11.1 사례 1: 평균만 보고 산정한 결제 시스템

평균 RPS 50으로 산정하여 8코어 인스턴스 4대(N+1)를 배치. 잘 돌아가다 월말 정산일 오후 2시에 RPS 800이 몰림. 모든 인스턴스 CPU 100%, 응답시간 30초로 늘어남, 결제 실패 폭주.

문제는 산정 자체가 아니라 "피크/평균 비율을 보지 않은 것"이었다. 월말 결제일은 평균의 16배 트래픽이 정기적으로 발생했다. APM 데이터에 분 단위 max가 찍혀 있었지만 산정 회의에서 일 평균만 봤다.

재발 방지: 산정 입력값에 분 단위 max, 시간대별 분포, 요일별 분포, 월별 분포를 모두 포함하도록 절차 변경.

### 11.2 사례 2: 메모리 limit 없는 컨테이너의 노드 점유

쿠버네티스에서 메모리 request만 설정하고 limit을 생략. 의도는 "유연하게 쓰자"였음. 한 파드에서 메모리 누수가 발생, 30GB까지 부풀어 노드 전체 메모리를 점유. 같은 노드의 다른 파드들이 OOM Kill로 연쇄 실패.

문제는 limit 생략의 트레이드오프를 충분히 이해하지 못한 점이다. limit이 있으면 누수 파드만 죽고, 없으면 노드가 죽는다.

재발 방지: 메모리는 request = limit으로 통일. CPU만 limit을 풀어두는 정책으로 변경.

### 11.3 사례 3: 디스크 IOPS는 늘렸는데 처리량을 안 본 케이스

gp3 디스크의 IOPS를 16000으로 늘렸지만 처리량은 기본 125MB/s로 둠. 대량 ETL이 시작되자 IOPS는 여유였는데 처리량이 한계에 부딪혀 작업이 10시간 → 20시간으로 늘어남. ETL이 다음 날 영업 시간을 침범.

재발 방지: 디스크 산정 체크 항목에 처리량을 명시. 시퀀셜 워크로드(백업, ETL, 로그)가 있는 경우 항상 처리량을 별도로 산정.

### 11.4 사례 4: 커넥션 풀 사이즈와 DB max_connections 불일치

애플리케이션 50대 × 풀 30 = 1500 커넥션. PostgreSQL `max_connections=200`. 트래픽 증가로 일시적으로 80대까지 스케일 아웃되자 커넥션 부족으로 신규 요청 거부. 응답 자체가 안 되니 모니터링 알람이 늦게 떴다.

재발 방지: PgBouncer 도입. 애플리케이션 풀과 DB 풀을 분리. 산정 절차에 "애플리케이션 풀 총합 vs DB 한계" 비교 항목 추가.

### 11.5 사례 5: NLB 뒤 ephemeral port 고갈

NLB가 client IP를 보존하는 모드로 동작. 백엔드 인스턴스 1대로 트래픽이 쏠리는 상황에서 동일 클라이언트의 다수 커넥션이 들어옴. 백엔드의 ephemeral port가 고갈되어 신규 커넥션 거부.

이건 산정 단계에서 잘 안 보인다. 부하 테스트도 트래픽 분산이 균일해서 재현이 안 된다. 운영에서 처음 발견되는 케이스다.

재발 방지: client IP 보존이 필요한지 다시 확인. 필요 없으면 source NAT 모드로 변경. 필요하다면 백엔드 인스턴스 수를 더 늘리거나 connection multiplexing 적용.

## 12. 재산정 트리거 기준

산정은 한 번 하고 끝이 아니다. 다음 조건이 발생하면 재산정한다.

### 12.1 정량 트리거

- 평균 사용률이 목표치를 2주 이상 초과 (CPU 60% 목표인데 65% 지속)
- 피크 사용률이 80% 이상 발생 (1회라도 일어나면 검토)
- 응답시간 p95가 SLO의 80%에 도달
- 일 트래픽이 산정 시점 대비 30% 이상 증가
- 인프라 비용이 전월 대비 20% 이상 변동 (비정상 패턴 의심)

### 12.2 정성 트리거

- 신규 기능 출시 (특히 무거운 쿼리, 외부 API 호출 추가 시)
- 트래픽 패턴 변화 (B2C → B2B 비중 변화, 신규 시장 진입)
- 인스턴스 타입 변경 (세대 업그레이드, 아키텍처 변경)
- DB 스키마 큰 변경 (인덱스 추가/삭제, 파티셔닝)
- 장애 발생 후 회고

### 12.3 정기 재산정

위 트리거가 없어도 분기에 1회 정기 재산정을 한다. 트래픽은 천천히 자라고, 인프라 가격은 천천히 떨어진다. 분기마다 다음을 점검한다.

- 평균/피크 사용률의 최근 3개월 추이
- 다음 분기 트래픽 예측
- 신규 인스턴스 타입 검토 (가격/성능 개선)
- 헤드룸 비율 적정성 (남는다면 줄여서 비용 절감)

## 13. 산정 문서화

산정값은 문서로 남긴다. 다음 산정 때 "왜 이렇게 잡았는지" 모르면 또 처음부터 추측한다.

문서에 들어가야 할 것:

- 산정 시점의 측정값 (RPS, 요청당 CPU, 요청당 메모리 등)
- 사용한 가정 (피크/평균 비율, 헤드룸 비율, SLO)
- 산정 결과 (인스턴스 타입, 수량, 스케일링 임계값)
- 검증 방법 (부하 테스트 결과, 운영 모니터링 결과)
- 재산정 트리거 조건

이게 있으면 6개월 뒤 후임자가 같은 산정을 30분에 끝낸다. 없으면 또 일주일이 든다.
