# 함수형 프로그래밍 (Functional Programming)

함수형 프로그래밍은 함수를 값처럼 다루고, 상태를 바꾸는 대신 새 값을 만들어 흐름을 짠다. JavaScript는 함수가 일급 객체라 진입 장벽이 낮다. 이 문서는 Node 서버 코드에서 함수형 기본기를 처음 손에 익힐 때 필요한 만큼만 다룬다 — 순수 함수, 불변성, 고차 함수, 합성, 커링, Maybe/Either, 그리고 Node 특유의 함정인 `process.env`로 인한 순수성 붕괴, require 캐시의 전역 상태, async iterator 지연 처리, 라이브러리 require 비용까지.

깊은 주제는 두 곳으로 나눠 넘긴다. 미들웨어 합성, Stream 위에서의 함수형 처리, 비동기 합성과 unhandledRejection 처리, Worker 분리, 메모이제이션과 GC, fp-ts/Effect 같은 Node 런타임 심화는 [[Functional_Programming]]에 있다. 포인트프리, 트랜스듀서, Lens, Functor/Monad 법칙, 트램폴린 같은 언어 차원의 심화는 [[Functional_Programming_Advanced]]에서 본다. 여기서는 그 경계 직전까지, 처음 손댈 때 실제로 걸려 넘어지는 지점 위주로 정리한다.

## 순수 함수와 부작용

순수 함수는 입력이 같으면 출력이 같고, 외부 상태를 건드리지 않는다. 테스트하기 쉽고, 추론이 쉬워서 디버깅 시간이 줄어든다.

```javascript
// 순수
const add = (a, b) => a + b;

// 비순수: 외부 변수에 의존
let counter = 0;
const tick = () => ++counter;

// 비순수: 호출마다 결과가 다름
const now = () => new Date();
```

Node 서버에서 순수성을 가장 흔하게 깨는 게 `process.env`다. 환경변수나 설정 객체를 함수 안에서 직접 읽으면, 인자가 같아도 실행 환경에 따라 결과가 달라진다.

```javascript
// 비순수: process.env를 함수 내부에서 읽는다
const buildUrl = (path) => `${process.env.API_HOST}/${path}`;

// 테스트에서 API_HOST를 안 넣으면 'undefined/users'가 나온다
buildUrl('users');
```

`buildUrl`은 인자가 `'users'`로 같아도 `API_HOST` 값에 따라 출력이 바뀐다. 테스트할 때마다 `process.env`를 세팅해야 하고, 다른 테스트가 같은 변수를 건드리면 순서에 따라 깨진다. 설정을 인자로 받게 바꾸면 순수해진다.

```javascript
// 순수: 설정을 인자로 받는다
const buildUrl = (host, path) => `${host}/${path}`;

buildUrl('https://api.example.com', 'users'); // 항상 같은 결과
```

`process.env` 읽기는 부트스트랩 한 겹에서 끝내고, 거기서 만든 설정 객체를 아래 함수들에 인자나 클로저로 흘려보낸다. 비즈니스 로직 깊은 곳에서 `process.env`를 직접 읽으면 그 함수는 더 이상 순수 함수가 아니고, 테스트가 환경에 묶인다.

서버 코드는 결국 DB I/O, 로깅, 응답 전송 같은 부작용이 있어야 일을 한다. 모든 함수를 순수하게 만들 수는 없고, 순수 영역과 부작용 영역의 경계를 그어 두는 게 핵심이다. 비즈니스 규칙은 순수 함수로 짜고, I/O는 바깥쪽 한 겹에서 처리하는 식이다.

## 불변성

원본을 바꾸지 않고 새 값을 반환한다. 참조가 그대로 유지되면 어디서 누가 바꿨는지 추적이 어려워지고, 디버깅이 미궁에 빠진다.

```javascript
const user = { name: '김철수', age: 25 };

// 원본 보존
const aged = { ...user, age: 26 };

const nums = [1, 2, 3];
const appended = [...nums, 4];
const removed = nums.filter((_, i) => i !== 1);
```

스프레드 복사는 1단계만 얕게 복사한다는 점을 기억해라. 중첩 객체의 안쪽 참조는 원본과 공유되므로, 깊은 곳을 바꾸면 원본도 같이 바뀐다.

```javascript
const order = { id: 1, items: [{ sku: 'A', qty: 1 }] };
const copy = { ...order };
copy.items[0].qty = 99;
order.items[0].qty; // 99 — items 배열은 같은 참조라 원본도 오염됐다
```

중첩 객체를 안전하게 갱신하려면 Node 17+ 의 `structuredClone`이나 `immer`를 쓴다.

> 참고: `structuredClone`은 Node 17+에서 전역으로 제공된다. 더 낮은 버전을 지원해야 하면 `v8.deserialize(v8.serialize(obj))`나 라이브러리를 쓴다. 그런데 `structuredClone`은 만능 깊은 복사가 아니다. 직렬화 가능한 값만 다루는 구조화 복제 알고리즘을 쓰기 때문에, 함수·심볼·DOM 노드 같은 건 만나는 즉시 `DataCloneError`를 던진다.

```javascript
const state = {
  createdAt: new Date(),     // Date는 복제됨
  cache: new Map(),          // Map도 복제됨
  retry: () => fetch('/'),   // 함수는 복제 불가
};

structuredClone(state);
// DataCloneError: () => fetch('/') could not be cloned.
```

함수만 문제가 아니다. 클래스 인스턴스를 넣으면 에러는 안 나지만 프로토타입이 떨어져 나가서, 데이터는 살아도 메서드가 사라진 평범한 객체가 나온다.

```javascript
class User {
  constructor(name) { this.name = name; }
  greet() { return `hi ${this.name}`; }
}

const u = new User('kim');
const cloned = structuredClone(u);

cloned.name;     // 'kim' — 필드는 복사됨
cloned.greet();  // TypeError: cloned.greet is not a function
cloned instanceof User; // false — 그냥 { name: 'kim' }가 됐다
```

그래서 도메인 상태를 클래스 인스턴스나 함수가 섞인 객체로 들고 다니면 `structuredClone`이 막힌다. 불변 데이터로 굴릴 거면 상태를 순수 데이터(plain object, 배열, 원시값)로만 유지하는 게 전제다. 메서드가 필요하면 상태와 동작을 분리해서, 동작은 상태를 인자로 받는 별도 순수 함수로 빼라. 대량 데이터에서 매번 새 객체를 만들면 GC 압력이 올라가는데, 그 경계와 측정 방법은 [[Functional_Programming]]에서 다룬다. 중첩이 깊어 부분 갱신이 잦으면 Lens가 답인데, 그건 [[Functional_Programming_Advanced]] 쪽이다.

## 모듈 전역 상태와 require 캐시

순수 함수와 불변성을 신경 써서 짜도, 모듈 최상단에 둔 변수 하나가 전부를 망친다. Node는 `require`(또는 `import`)로 같은 모듈을 여러 번 불러도 처음 한 번만 실행하고 그 결과를 캐시한다. 모듈 스코프의 객체는 그 모듈을 가져온 모든 곳이 같은 참조를 공유한다.

```javascript
// config.js
let settings = { retries: 3 };

const get = () => settings;
const update = (patch) => { settings = { ...settings, ...patch }; };

module.exports = { get, update };
```

```javascript
// a.js
const config = require('./config');
config.update({ retries: 5 });

// b.js — a.js와 전혀 무관해 보여도
const config = require('./config');
config.get().retries; // 5 — a.js가 바꾼 값이 그대로 보인다
```

`a.js`와 `b.js`가 받은 `config`는 같은 모듈 인스턴스다. 한쪽이 `update`하면 다른 쪽에도 보인다. 이게 의도라면 싱글톤이지만, 순수 함수처럼 생긴 `get`이 사실은 숨은 전역 상태를 읽고 있다는 게 함정이다. 테스트에서 모듈 상태를 바꿔 두면 다음 테스트로 새어 나간다.

캐시를 우회하려고 `delete require.cache[require.resolve('./config')]`로 모듈을 강제 재로드하는 코드를 가끔 본다. 동작은 하지만 같은 모듈을 두 번 평가해서 클래스·심볼·instanceof가 어긋나는 더 큰 문제를 부른다. ESM(`import`)에는 이런 캐시 무효화 수단조차 공식적으로 없다.

해법은 모듈 스코프에 가변 상태를 두지 않는 것이다. 상태가 필요하면 객체를 모듈에서 export하지 말고, 팩토리 함수로 만들어 호출부가 자기 인스턴스를 들고 있게 한다.

```javascript
// config.js
const createConfig = (initial = {}) => {
  let settings = { retries: 3, ...initial };
  return {
    get: () => settings,
    update: (patch) => { settings = { ...settings, ...patch }; },
  };
};

module.exports = { createConfig };
```

이러면 `createConfig()`를 부른 쪽마다 독립된 상태를 가진다. 테스트는 매번 새 인스턴스를 만들어 서로 간섭하지 않는다. require 캐시가 공유하는 건 팩토리 함수뿐이고, 함수는 호출 전까지 상태가 없다.

## 고차 함수와 합성

함수를 인자로 받거나 함수를 반환하는 함수가 고차 함수다. `map`, `filter`, `reduce`가 가장 흔한 형태다.

```javascript
const numbers = [1, 2, 3, 4, 5];

const doubled = numbers.map((n) => n * 2);
const evens = numbers.filter((n) => n % 2 === 0);
const sum = numbers.reduce((acc, n) => acc + n, 0);
```

여러 함수를 이어 붙이는 `pipe`는 데이터 변환 흐름을 위에서 아래로 읽히게 만든다. 합성이 깊어질수록 중간 변수 이름을 짓느라 머리 쓸 일이 줄어든다.

```javascript
const pipe = (...fns) => (x) => fns.reduce((acc, fn) => fn(acc), x);

const trim = (s) => s.trim();
const lower = (s) => s.toLowerCase();
const slug = (s) => s.replace(/\s+/g, '-');

const toSlug = pipe(trim, lower, slug);
toSlug('  Hello World  '); // "hello-world"
```

## 커링과 부분 적용

다인자 함수를 한 번에 받지 않고 인자를 하나씩 받게 바꾸는 게 커링이다. 일부 인자를 미리 고정해 두는 부분 적용도 같은 결이다.

```javascript
const curry = (fn) => {
  return function curried(...args) {
    return args.length >= fn.length
      ? fn(...args)
      : (...rest) => curried(...args, ...rest);
  };
};

const add3 = curry((a, b, c) => a + b + c);
const addTo10 = add3(10);
addTo10(5)(3); // 18

const log = (level) => (msg) => console.log(`[${level}] ${msg}`);
const warn = log('WARN');
warn('cache miss');
```

설정 객체를 먼저 받고 핸들러를 반환하는 `(config) => (req) => result` 패턴은 미들웨어와 라이브러리에서 자주 보인다.

위 `curry`는 `fn.length`로 인자가 다 모였는지 판단한다. 여기에 함정이 있다. `fn.length`는 기본값이 붙은 매개변수부터, 그리고 rest 매개변수를 세지 않는다. 기본값 하나만 넣어도 arity가 어긋나서, 인자를 덜 받았는데도 커링이 일찍 끝나 버린다.

```javascript
const add = curry((a, b, c = 0) => a + b + c);
add.length; // 무관 — 안쪽 원본 함수의 length가 2다 (c는 기본값이라 제외)

add(1)(2);  // 3 — c를 받을 차례가 오기 전에 length 2를 채워 바로 실행됐다
add(1)(2)(3); // TypeError: add(1)(...) is not a function — 이미 3이 나왔으니 호출 불가
```

rest 매개변수는 더 심하다. `length`가 0으로 잡혀서 첫 호출에 인자를 아무리 넘겨도 모자란 줄 알거나, 반대로 즉시 실행돼 버린다.

```javascript
const sum = curry((...nums) => nums.reduce((a, b) => a + b, 0));
sum.length; // 원본 length 0
sum();      // 0 — 가변 인자인데 빈 호출에 바로 끝났다
```

`fn.length`에 의존하는 자동 커링은 인자 개수가 컴파일 타임에 고정된 함수에만 안전하다. 기본값이나 가변 인자를 쓸 거면 받을 인자 수를 명시적으로 넘기는 형태로 바꿔야 한다.

```javascript
const curryN = (arity, fn) => {
  return function curried(...args) {
    return args.length >= arity
      ? fn(...args)
      : (...rest) => curried(...args, ...rest);
  };
};

const add = curryN(3, (a, b, c = 0) => a + b + c);
add(1)(2)(3); // 6 — arity를 직접 3으로 박아 넣어 정상 동작
```

`lodash`나 `ramda`의 `curry`도 내부적으로 같은 한계를 안고 있어서, 기본값·rest를 섞은 함수에 그대로 물리면 같은 식으로 어긋난다. 자동 커링의 더 깊은 변형인 포인트프리 스타일은 [[Functional_Programming_Advanced]]에서 본다.

## Maybe / Either로 null과 에러 다루기

`null`·`undefined` 체크가 코드 곳곳에 흩어지면 흐름이 끊긴다. Maybe는 값이 있을 수도 없을 수도 있다는 상태를 한 박스에 담는다.

```javascript
class Maybe {
  constructor(value) { this.value = value; }
  static of(v) { return new Maybe(v); }
  map(fn) {
    return this.value == null ? Maybe.of(null) : Maybe.of(fn(this.value));
  }
  getOrElse(fallback) {
    return this.value == null ? fallback : this.value;
  }
}

const user = { profile: { nickname: 'gyu' } };
const nick = Maybe.of(user)
  .map((u) => u.profile)
  .map((p) => p.nickname)
  .getOrElse('anon'); // "gyu"
```

Either는 성공·실패 두 갈래를 한 타입으로 들고 다닌다. throw 없이 실패를 값으로 흘려보내고, 마지막 단계에서 분기한다.

```javascript
class Either {
  constructor(left, right) { this.left = left; this.right = right; }
  static left(v) { return new Either(v, null); }
  static right(v) { return new Either(null, v); }
  map(fn) {
    return this.right !== null ? Either.right(fn(this.right)) : this;
  }
  fold(onLeft, onRight) {
    return this.right !== null ? onRight(this.right) : onLeft(this.left);
  }
}

const divide = (a, b) =>
  b === 0 ? Either.left('divide by zero') : Either.right(a / b);

divide(10, 0)
  .map((n) => n * 2)
  .fold((err) => `에러: ${err}`, (n) => `결과: ${n}`);
```

Either를 처음 쓸 때 가장 많이 새는 곳이 경계다. Either는 "이 함수가 던지는 대신 실패를 값으로 돌려준다"는 약속인데, 정작 안에서 부르는 외부 함수가 throw를 하면 그 약속이 깨진다. throw가 Either 체인을 그대로 뚫고 나간다.

```javascript
const parseConfig = (raw) => Either.right(JSON.parse(raw)).map((c) => c.port);

// 정상 입력은 잘 돈다
parseConfig('{"port":3000}').fold(...); // 3000

// 깨진 입력은 Either.left가 아니라 throw가 튀어나온다
parseConfig('{bad json');
// SyntaxError: Unexpected token b in JSON at position 1  ← fold까지 못 가고 죽는다
```

`JSON.parse`, `fs` 동기 호출, ORM, 외부 SDK처럼 throw로 실패를 알리는 함수는 전부 같은 문제를 일으킨다. `map` 안에 넘긴 변환 함수가 throw해도 마찬가지로 새어 나간다. 해결은 외부 세계와 Either 사이에 try/catch 어댑터를 한 겹 두는 것이다. 던지는 함수를 Either를 돌려주는 함수로 감싼다.

```javascript
const tryCatch = (fn) => (...args) => {
  try {
    return Either.right(fn(...args));
  } catch (e) {
    return Either.left(e);
  }
};

const parseJson = tryCatch(JSON.parse);

const parseConfig = (raw) =>
  parseJson(raw).map((c) => c.port);

parseConfig('{bad json')
  .fold((err) => `설정 오류: ${err.message}`, (port) => `포트 ${port}`);
// "설정 오류: Unexpected token b ..." ← 이제 left로 잡혀 정상 분기
```

규칙은 단순하다. throw할 수 있는 코드는 Either 체인 바깥에서 `tryCatch`로 감싸 left로 변환한 다음에야 체인에 넣는다. 어댑터 한 겹을 빠뜨리면 Either를 써도 try/catch를 안 쓴 것과 똑같이 어딘가에서 프로세스가 죽는다.

async가 끼면 경계가 한 번 더 샌다. 위 `tryCatch`는 동기 함수만 잡는다. 안에 넘긴 함수가 `async`거나 Promise를 반환하면, `try`는 Promise 객체를 정상으로 받아 `Either.right`에 담아 버린다. 진짜 실패는 그 Promise가 나중에 reject될 때 일어나고, 그건 `catch` 바깥이라 잡히지 않는다.

```javascript
const fetchUser = async (id) => {
  const res = await fetch(`/users/${id}`);
  if (!res.ok) throw new Error('not found');
  return res.json();
};

const safe = tryCatch(fetchUser);
const result = safe('123');

result.right; // Promise { <pending> } — 실패가 아니라 "성공한 Promise"로 잡혔다
// 그리고 id가 없으면 reject가 어디에도 안 잡혀 UnhandledPromiseRejection으로 터진다
```

`safe('없는id')`는 `Either.left`를 만들지 않는다. `Either.right`에 pending Promise를 담아 돌려주고, 그 Promise는 뒤늦게 reject된다. 호출부가 `result.right`를 await하지 않으면 Node 15+에서는 unhandledRejection으로 프로세스가 죽는다. 동기 어댑터로 async 함수를 감싼 순간 Either의 약속이 깨진 것이다.

async 경계는 동기용 `tryCatch`로는 막을 수 없고, Promise를 인지하는 별도 어댑터(`async`로 받아 `await fn()`을 try/catch)가 필요하다. `pipeAsync`로 비동기 단계를 합성하고 단계마다 실패를 Result로 잡는 패턴, unhandledRejection을 막는 규칙은 [[Functional_Programming]]에서 다룬다.

진지하게 쓸 거면 직접 짜는 대신 `fp-ts`나 `Effect`처럼 타입까지 받쳐주는 라이브러리 쪽이 정합성이 낫다. 이들은 `tryCatch`와 비슷한 어댑터를 표준으로 제공해서 경계 누수를 처음부터 막는다. Functor·Monad 법칙을 지켜 직접 구현하는 방법은 [[Functional_Programming_Advanced]]에 있다.

## async iterator와 for-await-of로 지연 처리

동기 generator로 `map`/`filter`를 지연 적용하면 한 줄씩만 메모리에 올린다. 그런데 Node 서버에서 다루는 소스는 대부분 비동기다 — 파일 읽기, DB 커서, HTTP 응답 본문. 이때 쓰는 게 async generator와 `for await...of`다. 동기 iterable이 lazy한 동기 컬렉션이라면, async iterable은 lazy한 비동기 컬렉션이고, `for await...of`가 그걸 한 번에 하나씩 당겨 온다.

`map`/`filter`를 async generator로 짜면 동기 버전과 모양이 거의 같다.

```javascript
async function* mapAsync(iter, fn) {
  for await (const x of iter) yield fn(x);
}
async function* filterAsync(iter, pred) {
  for await (const x of iter) if (pred(x)) yield x;
}
```

Node의 `readline`은 그 자체가 async iterable이라 바로 물릴 수 있다. 큰 로그 파일을 한 줄씩 흘려보내면서 변환·필터를 거는데, 어느 시점에도 메모리에는 한 줄과 파이프라인 내부 상태만 있다.

```javascript
import { createReadStream } from 'node:fs';
import { createInterface } from 'node:readline';

const lines = createInterface({
  input: createReadStream('big.log'),
  crlfDelay: Infinity,
});

const parsed = mapAsync(lines, (line) => JSON.parse(line));
const errors = filterAsync(parsed, (log) => log.level === 'error');

for await (const log of errors) {
  await report(log); // 한 줄 끝나야 다음 줄을 당긴다 — 자연스러운 backpressure
}
```

`for await...of`의 핵심은 소비 쪽이 다음 값을 요청하기 전까지 윗단이 계산을 멈춘다는 점이다. `report`가 느리면 파일 읽기도 그만큼 느려진다. 배열로 다 읽어 `arr.map().filter()`로 처리하면 파일 전체가 메모리에 올라오지만, async iterator 체인은 처리 속도에 맞춰 입력을 당긴다.

DB 커서도 같은 모양이다. `pg`의 `cursor`나 `mongodb`의 `find().stream()`은 async iterable을 주므로, 수백만 행을 한 번에 메모리로 가져오지 않고 청크 단위로 흘려보낸다.

```javascript
async function* rowsOf(cursor) {
  for await (const row of cursor) yield row;
}

const cursor = db.query(new Cursor('SELECT * FROM events'));
const mapped = mapAsync(rowsOf(cursor), normalize);

for await (const row of mapped) {
  await sink.write(row);
}
```

주의할 건 에러 전파다. async generator 내부에서 throw가 나면 그건 `for await...of`의 await 지점으로 reject되어 올라온다. 바깥을 try/catch로 감싸지 않으면 그대로 unhandledRejection이 된다.

```javascript
try {
  for await (const log of errors) {
    await report(log);
  }
} catch (err) {
  // JSON.parse 실패, 파일 읽기 에러가 전부 여기로 모인다
  logger.error({ err }, 'log pipeline failed');
}
```

Transform 스트림으로 같은 처리를 짜고 `pipeline()`으로 backpressure·에러·정리를 한 번에 묶는 방식, RxJS로 시간 윈도우 연산을 얹는 방식은 [[Functional_Programming]]에서 다룬다. async iterator는 그 심화로 가기 전 가장 가벼운 lazy 처리 수단이다.

## Node에서 자주 부딪히는 함정

**중간 배열 누적.** `arr.map(...).filter(...).map(...)` 체이닝은 단계마다 새 배열을 만든다. 데이터가 작으면 문제없지만 대용량 로그·CSV를 한 줄씩 처리할 때는 메모리가 튀어 오른다. generator나 Stream으로 흘려보내는 패턴이 안전하다.

```javascript
function* lazyMap(iter, fn) {
  for (const x of iter) yield fn(x);
}
function* lazyFilter(iter, pred) {
  for (const x of iter) if (pred(x)) yield x;
}

// 1억 줄 파일도 한 줄씩만 메모리에 머문다
for (const line of lazyFilter(lazyMap(readLines('big.log'), parse), isError)) {
  report(line);
}
```

**this 바인딩 손실.** 클래스 메서드를 그대로 콜백으로 넘기면 `this`가 끊긴다. 화살표 함수로 감싸거나 생성자에서 `bind`해 두는 식으로 묶어야 한다.

```javascript
class UserService {
  constructor() { this.prefix = 'user:'; }
  decorate(name) { return this.prefix + name; }
}

const svc = new UserService();
['a', 'b'].map(svc.decorate);            // TypeError: cannot read 'prefix' of undefined
['a', 'b'].map((n) => svc.decorate(n));  // 정상
```

함수형 색깔을 강하게 가져갈 거라면 클래스보다 함수 + 클로저로 같은 책임을 짜는 편이 덜 미끄러진다.

## Ramda·Lodash-fp를 백엔드에서 쓸 때

직접 `pipe`·`curry`를 짜는 대신 `ramda`나 `lodash/fp`를 가져다 쓰는 경우가 많다. 이미 커링·불변 헬퍼가 갖춰져 있어 편하다. 다만 Node 백엔드에서는 import 방식이 그대로 require 비용으로 돌아온다.

`const R = require('ramda')`나 `const _ = require('lodash/fp')`처럼 전체를 가져오면, 함수 몇 개만 쓰더라도 모듈 전체가 평가된다. ramda는 200개가 넘는 함수를 등록하고, lodash/fp는 전체 lodash를 함수형 래퍼로 감싼다. 이 평가는 서버 부팅 때 한 번 일어나는 require 비용이고, 서버리스(Lambda)처럼 cold start가 응답 지연으로 직결되는 환경에서는 이 초기화가 눈에 띈다.

```javascript
// 전체 로드 — 쓰지 않는 함수까지 다 평가된다
const R = require('ramda');
const result = R.pipe(R.map(f), R.filter(g))(data);

// 필요한 함수만 — require 비용과 초기 메모리가 준다
const pipe = require('ramda/src/pipe');
const map = require('ramda/src/map');
const filter = require('ramda/src/filter');
```

ESM에 번들러(esbuild·webpack)를 끼우면 트리셰이킹으로 안 쓰는 함수가 떨어져 나가지만, 번들 없이 `node`로 바로 도는 서버 코드는 트리셰이킹이 없다. `import { map } from 'ramda'`라고 써도 런타임에는 ramda 전체가 로드된다. 번들 단계가 없는 백엔드라면 서브경로 import(`ramda/src/map`)로 직접 잘라 가져오거나, 애초에 `pipe`·`curry` 정도는 위에서 본 것처럼 10줄짜리로 직접 두는 편이 의존성도 줄고 디버깅도 쉽다.

lodash 계열은 `require('lodash')` 통짜 로드가 가장 무겁고, `require('lodash/map')`처럼 함수별 경로로 가져오면 훨씬 가볍다. `lodash/fp`는 자동 커링과 인자 순서 재배치(데이터 마지막) 래퍼가 한 겹 더 얹혀서, 같은 함수라도 `lodash/map`보다 초기화가 조금 더 든다. 백엔드에서 서버 부팅 시간이나 Lambda cold start를 재고 있다면 무엇을 통짜로 require하는지부터 본다.

---

실무에서 OOP를 다 버리고 FP로 넘어가는 일은 드물다. 도메인 규칙은 순수 함수로 짜서 테스트를 단순하게 가져가고, 컨트롤러·ORM처럼 프레임워크 경계에 있는 객체 모델은 그대로 두는 절충이 보통이다. Node 런타임 위에서 미들웨어 합성, Stream/RxJS, Worker 분리, 메모이제이션, fp-ts/Effect 같은 라이브러리 선택까지 깊게 들어가려면 [[Functional_Programming]]으로 이어서 보면 된다.
