# Node.js 에러 핸들링

## 왜 어렵나

Node.js는 비동기 채널이 너무 많다. Promise reject, EventEmitter `error`, Stream `error`, child_process `error`/`exit`, worker_threads `messageerror`, IPC, native addon throw. 채널마다 처리 방식이 다르고, 한 군데라도 놓치면 프로세스가 조용히 죽거나, 요청이 응답 없이 타임아웃된다.

5년 동안 다음과 같은 사건을 겪었다.

- Redis 클라이언트의 `error` 이벤트를 안 듣고 있다가 첫 네트워크 글리치에서 프로세스가 그대로 종료
- `pipe()`로 연결한 스트림 중간 단계에서 에러가 났는데 마지막 스트림에만 `on('error')`가 붙어있어서 메모리 누수
- `Promise.all` 안에서 한 작업이 실패하자 나머지 9개 작업이 잠긴 채 트랜잭션이 롤백되지 않음
- `try/catch` 안에서 commit 후 catch로 떨어져서 부분 커밋 발생
- Sentry에 비밀번호와 카드번호가 그대로 올라감

이 문서는 그런 실수를 줄이려고 정리했다.

## 에러를 두 종류로 분리

```typescript
// errors/AppError.ts
export class AppError extends Error {
  constructor(
    message: string,
    public readonly statusCode: number,
    public readonly code: string,
    public readonly isOperational: boolean = true,
    options?: { cause?: unknown; details?: unknown }
  ) {
    super(message, options as ErrorOptions);
    this.name = this.constructor.name;
    this.details = options?.details;
    Error.captureStackTrace(this, this.constructor);
  }

  details?: unknown;
}

export class NotFoundError extends AppError {
  constructor(resource: string, cause?: unknown) {
    super(`${resource}을(를) 찾을 수 없습니다.`, 404, 'NOT_FOUND', true, { cause });
  }
}

export class ValidationError extends AppError {
  constructor(message: string, details?: unknown) {
    super(message, 400, 'VALIDATION_ERROR', true, { details });
  }
}

export class ExternalServiceError extends AppError {
  constructor(service: string, cause: unknown) {
    super(`${service} 호출 실패`, 502, 'EXTERNAL_SERVICE_ERROR', true, { cause });
  }
}
```

`isOperational`이 `true`면 클라이언트에 메시지를 그대로 내려도 안전한 에러다. `false`거나 `AppError`가 아닌 일반 `Error`는 코드 버그로 본다. 코드 버그는 응답을 일반화한 메시지로 내리고 프로세스 재시작을 고민한다. 둘을 섞으면 401이어야 할 응답이 500이 되거나, 내부 스택 트레이스가 외부에 노출된다.

`options.cause`는 Node 16.9+의 표준이다. 외부 라이브러리 에러를 도메인 에러로 감쌀 때 원인을 잃지 않으려고 쓴다. 로거가 `err.cause`를 출력하도록 포맷터를 한 번 손봐두면 디버깅이 한결 편해진다.

## Error.cause 체이닝

원인 추적이 필요한 곳마다 명시적으로 `cause`를 채운다.

```typescript
try {
  await pgPool.query('SELECT * FROM users WHERE id = $1', [id]);
} catch (err) {
  throw new AppError('사용자 조회 실패', 500, 'DB_QUERY_FAILED', false, { cause: err });
}
```

로거는 재귀적으로 cause를 따라가서 출력한다.

```typescript
function formatError(err: unknown, depth = 0): string {
  if (!(err instanceof Error)) return String(err);
  const lines = [`${' '.repeat(depth * 2)}${err.name}: ${err.message}`];
  if (err.stack) lines.push(err.stack.split('\n').slice(1, 3).join('\n'));
  if ('cause' in err && err.cause) {
    lines.push(`${' '.repeat(depth * 2)}Caused by:`);
    lines.push(formatError(err.cause, depth + 1));
  }
  return lines.join('\n');
}
```

cause를 빼먹으면 "DB 조회 실패"만 로그에 찍히고 어떤 SQL이 깨졌는지 추적이 안 된다. 외부 호출 직후의 catch는 거의 항상 cause를 채운다고 보면 된다.

## EventEmitter의 error 이벤트

`EventEmitter`는 `error` 이벤트에 리스너가 하나도 없으면 `process.on('uncaughtException')`로 에러를 던지고, 그것마저 없으면 프로세스를 종료시킨다. 라이브러리 내부에서 발생한 에러로 서버 전체가 내려가는 가장 흔한 원인이다.

```typescript
import Redis from 'ioredis';

const redis = new Redis({ host: 'redis', port: 6379 });

// 이 한 줄이 빠지면 첫 disconnect에서 프로세스가 죽는다
redis.on('error', (err) => {
  logger.error('Redis 에러', { err: formatError(err) });
});

// pg 풀도 마찬가지
pool.on('error', (err, client) => {
  logger.error('Idle PG 클라이언트 에러', { err: formatError(err) });
});
```

직접 만든 EventEmitter도 같다. 사용처에서 listener를 안 붙일 가능성이 있으면 emit 전에 `listenerCount`를 검사해서 자체 처리한다.

```typescript
class JobQueue extends EventEmitter {
  private fail(err: Error) {
    if (this.listenerCount('error') === 0) {
      logger.error('처리되지 않은 큐 에러', { err: formatError(err) });
      return;
    }
    this.emit('error', err);
  }
}
```

`captureRejections: true` 옵션을 켜면 EventEmitter 핸들러 안에서 던진 Promise reject가 자동으로 `error` 이벤트로 변환된다. async 핸들러를 붙이는 EventEmitter라면 이 옵션을 켜두는 게 안전하다.

## Stream 에러 전파 — pipe vs pipeline

`a.pipe(b).pipe(c)`로 연결된 스트림은 에러가 자동으로 전파되지 않는다. `c.on('error')`만 붙이면 `a`나 `b`에서 난 에러를 못 잡는다. 잡지 못한 에러는 EventEmitter 규칙에 따라 프로세스를 죽인다. 그리고 중간 스트림은 `unpipe`되지 않은 채 메모리에 남는다.

```typescript
// 위험한 패턴
fs.createReadStream(input)
  .pipe(zlib.createGzip())
  .pipe(fs.createWriteStream(output))
  .on('error', handler); // 압축 단계 에러는 못 잡는다
```

`stream/promises`의 `pipeline`을 쓴다. 모든 단계의 에러를 모아서 한 번에 던지고, 실패 시 모든 스트림을 자동으로 close한다.

```typescript
import { pipeline } from 'node:stream/promises';
import fs from 'node:fs';
import zlib from 'node:zlib';

async function gzipFile(input: string, output: string) {
  try {
    await pipeline(
      fs.createReadStream(input),
      zlib.createGzip(),
      fs.createWriteStream(output)
    );
  } catch (err) {
    throw new AppError('파일 압축 실패', 500, 'COMPRESSION_FAILED', false, { cause: err });
  }
}
```

HTTP 응답 스트리밍에서 클라이언트가 중간에 끊으면 `EPIPE` / `ECONNRESET`이 뜬다. `pipeline`은 이걸 에러로 던지므로 catch해서 따로 처리한다. 클라이언트 끊김은 Sentry로 보내봐야 노이즈만 만든다.

## child_process와 worker_threads

`child_process.spawn`은 에러 채널이 두 개다.

- `error` 이벤트: 프로세스 자체가 시작되지 못함 (실행 파일 없음, 권한 부족)
- `exit` 이벤트: 시작은 됐지만 비정상 종료

둘 다 듣지 않으면 자식이 죽어도 부모는 모른다.

```typescript
import { spawn } from 'node:child_process';

function runFFmpeg(args: string[]): Promise<void> {
  return new Promise((resolve, reject) => {
    const proc = spawn('ffmpeg', args);
    let stderr = '';

    proc.stderr.on('data', (chunk) => { stderr += chunk; });
    proc.on('error', (err) => {
      reject(new AppError('ffmpeg 실행 실패', 500, 'SPAWN_FAILED', false, { cause: err }));
    });
    proc.on('exit', (code, signal) => {
      if (code === 0) return resolve();
      reject(new AppError(
        `ffmpeg 비정상 종료 (code=${code}, signal=${signal})`,
        500, 'PROCESS_FAILED', false,
        { details: { stderr: stderr.slice(-2000) } }
      ));
    });
  });
}
```

`worker_threads`는 채널이 더 많다.

- `error`: 워커 안에서 잡히지 않은 동기 throw
- `messageerror`: 메인-워커 사이 메시지 직렬화 실패 (BigInt, function 같은 것 던졌을 때)
- `exit`: 정상이면 0, 비정상이면 1

```typescript
import { Worker } from 'node:worker_threads';

const worker = new Worker('./worker.js');

worker.on('error', (err) => logger.error('Worker error', { err }));
worker.on('messageerror', (err) => logger.error('Worker messageerror', { err }));
worker.on('exit', (code) => {
  if (code !== 0) logger.error('Worker 비정상 종료', { code });
});
```

워커 내부의 Promise reject는 `error` 이벤트로 안 올라온다. 워커 진입점에서 `process.on('unhandledRejection')`을 따로 걸어서 메인으로 메시지를 보내야 한다.

IPC(`process.send`)로 통신하는 cluster 환경에서는 부모 프로세스가 자식의 stderr를 캡처하지 않으면 디버깅이 거의 불가능하다. `silent: false`(기본값)로 두고 부모에서 자식 로그를 모아 한곳에 출력하는 편이 낫다.

## Promise.all vs Promise.allSettled, AggregateError

`Promise.all`은 첫 reject에서 즉시 결과를 reject한다. 나머지 작업은 계속 실행되지만 결과는 버려진다. 트랜잭션 안에서 `Promise.all`을 쓰면 한 작업이 실패해도 다른 작업은 commit 직전까지 진행돼서 부작용이 남을 수 있다.

```typescript
// 한 사용자가 실패하면 나머지 결과는 버려진다
const results = await Promise.all(userIds.map(id => fetchUser(id)));

// 각 결과를 독립적으로 보고 싶다면 allSettled
const settled = await Promise.allSettled(userIds.map(id => fetchUser(id)));
const succeeded = settled.filter((r): r is PromiseFulfilledResult<User> => r.status === 'fulfilled').map(r => r.value);
const failed = settled.filter((r): r is PromiseRejectedResult => r.status === 'rejected');

if (failed.length > 0) {
  logger.warn('일부 사용자 조회 실패', { count: failed.length, sample: failed[0].reason });
}
```

여러 에러를 묶어서 던지고 싶으면 `AggregateError`(Node 15+)를 쓴다. `Promise.any`는 모든 promise가 reject되면 자동으로 `AggregateError`를 던진다.

```typescript
const failed = settled.filter((r): r is PromiseRejectedResult => r.status === 'rejected');
if (failed.length > 0) {
  throw new AggregateError(
    failed.map(f => f.reason),
    `${failed.length}개 작업 실패`
  );
}
```

`AggregateError`를 그냥 로깅하면 `errors` 배열이 안 찍힌다. 포매터에서 `err.errors`를 펼쳐서 출력하도록 추가한다.

## 외부 라이브러리 에러를 도메인 에러로 — Boundary 패턴

axios, pg, redis 같은 외부 라이브러리 에러가 그대로 서비스 레이어, 컨트롤러로 올라가면 결합도가 폭발한다. 라이브러리를 바꾸는 순간 수십 군데를 같이 고쳐야 한다. 어댑터 경계에서 한 번에 매핑한다.

```typescript
// repositories/userRepository.ts
import { DatabaseError } from 'pg';

export class UserRepository {
  async findById(id: string): Promise<User | null> {
    try {
      const { rows } = await this.pool.query('SELECT * FROM users WHERE id = $1', [id]);
      return rows[0] ?? null;
    } catch (err) {
      if (err instanceof DatabaseError) {
        if (err.code === '23505') {
          throw new ConflictError('중복된 키', { cause: err });
        }
        if (err.code === '57014') {
          throw new AppError('쿼리 타임아웃', 504, 'QUERY_TIMEOUT', true, { cause: err });
        }
      }
      throw new AppError('DB 조회 실패', 500, 'DB_ERROR', false, { cause: err });
    }
  }
}
```

axios도 마찬가지다. `AxiosError`의 `response.status`로 4xx/5xx를 분기하고, `code`로 `ECONNABORTED`(타임아웃), `ECONNREFUSED`(서버 다운)을 구분한다.

```typescript
import axios, { AxiosError } from 'axios';

async function callPaymentApi(payload: PaymentRequest): Promise<PaymentResult> {
  try {
    const { data } = await axios.post('/payments', payload, { timeout: 3000 });
    return data;
  } catch (err) {
    if (axios.isAxiosError(err)) {
      if (err.code === 'ECONNABORTED') {
        throw new AppError('결제 게이트웨이 타임아웃', 504, 'PAYMENT_TIMEOUT', true, { cause: err });
      }
      if (err.response?.status === 409) {
        throw new ConflictError('이미 처리된 결제', err);
      }
      if (err.response && err.response.status >= 500) {
        throw new ExternalServiceError('PaymentGateway', err);
      }
    }
    throw new AppError('결제 호출 실패', 500, 'PAYMENT_FAILED', false, { cause: err });
  }
}
```

매핑 함수를 한 파일에 모아두면 라이브러리 교체 비용이 크게 줄어든다. 호출하는 쪽 코드는 `ConflictError`인지, `ExternalServiceError`인지만 보면 된다.

## 재시도, 타임아웃, 서킷브레이커

재시도는 idempotent한 작업에만 건다. 결제 API에 무지성 재시도를 걸면 같은 결제가 두 번 일어난다. 멱등키(Idempotency-Key 헤더)를 발급해서 서버가 중복 요청을 한 번만 처리하도록 만들어야 안전하게 재시도할 수 있다.

```typescript
async function withRetry<T>(
  fn: () => Promise<T>,
  opts: { retries: number; baseMs: number; retryOn: (err: unknown) => boolean }
): Promise<T> {
  let lastErr: unknown;
  for (let i = 0; i <= opts.retries; i++) {
    try {
      return await fn();
    } catch (err) {
      lastErr = err;
      if (i === opts.retries || !opts.retryOn(err)) throw err;
      const delay = opts.baseMs * Math.pow(2, i) + Math.random() * 100;
      await new Promise(r => setTimeout(r, delay));
    }
  }
  throw lastErr;
}

// 5xx와 네트워크 에러만 재시도
const result = await withRetry(
  () => callPaymentApi(payload),
  {
    retries: 2,
    baseMs: 200,
    retryOn: (err) => {
      if (axios.isAxiosError(err)) {
        const status = err.response?.status;
        return !status || status >= 500 || err.code === 'ECONNABORTED';
      }
      return false;
    },
  }
);
```

타임아웃은 가장 바깥쪽 호출 단위에 건다. 라이브러리 기본 타임아웃은 보통 너무 길거나 아예 없다. axios도 `timeout`을 명시하지 않으면 무한 대기다.

서킷브레이커는 외부 서비스가 죽었을 때 우리 쪽 자원이 같이 말라죽지 않도록 차단한다. opossum이 Node에서 가장 많이 쓰인다.

```typescript
import CircuitBreaker from 'opossum';

const breaker = new CircuitBreaker(callPaymentApi, {
  timeout: 3000,
  errorThresholdPercentage: 50,
  resetTimeout: 30000,
  volumeThreshold: 10, // 10건 미만 호출에서는 회로 안 열림
});

breaker.on('open', () => logger.warn('PaymentGateway circuit open'));
breaker.on('halfOpen', () => logger.info('PaymentGateway circuit half-open'));
breaker.fallback(() => { throw new AppError('결제 일시 불가', 503, 'PAYMENT_UNAVAILABLE'); });

// 사용
const result = await breaker.fire(payload);
```

도입 시점은 외부 의존이 죽었을 때 우리 응답 시간이 같이 망가지는 게 보일 때다. 너무 일찍 넣으면 운영 복잡도만 올라간다. p99 latency가 외부 서비스 장애와 같이 튀는 패턴이 두세 번 잡히면 그때 넣는다.

## Graceful shutdown

배포 시점에 SIGTERM이 오면 즉시 프로세스를 종료하지 않고 in-flight 요청을 마무리해야 한다. 갑자기 끊으면 트랜잭션이 중간에 죽거나, 클라이언트에 502가 반환된다.

```typescript
import http from 'node:http';

const server = http.createServer(app);
server.listen(3000);

let shuttingDown = false;

async function shutdown(signal: string) {
  if (shuttingDown) return;
  shuttingDown = true;
  logger.info(`${signal} received, draining connections`);

  // 1) Health check가 unhealthy를 응답하도록 플래그 변경
  setHealthCheck(false);

  // 2) 새 연결 차단, 기존 연결 끝나길 대기
  server.close((err) => {
    if (err) logger.error('Server close error', { err });
  });

  // 3) Keep-Alive 연결을 강제 종료시키기 위해 idle socket destroy
  server.closeIdleConnections?.();

  // 4) 일정 시간 후 강제 종료
  const forceTimer = setTimeout(() => {
    logger.warn('Force exit after grace period');
    process.exit(1);
  }, 25_000);
  forceTimer.unref();

  try {
    await Promise.allSettled([
      pgPool.end(),
      redis.quit(),
      kafkaProducer.disconnect(),
    ]);
    logger.info('Cleanup done');
    process.exit(0);
  } catch (err) {
    logger.error('Shutdown cleanup error', { err });
    process.exit(1);
  }
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));
```

쿠버네티스에서 `terminationGracePeriodSeconds`(기본 30초) 안에 끝내야 SIGKILL이 안 온다. 강제 종료 타이머는 그보다 짧게 잡는다. Health check를 먼저 unhealthy로 바꿔서 LB가 트래픽을 빼주는 시간을 확보하는 게 핵심이다. 보통 readiness 빠진 뒤 5초 정도 sleep 후 server.close를 호출한다.

draining 중에 들어오는 요청은 정상 처리하되, 응답에 `Connection: close`를 붙여서 클라이언트가 keep-alive를 닫게 만든다.

## AsyncLocalStorage로 요청 컨텍스트 자동 전파

에러 로그에 매번 `userId`, `requestId`를 손으로 끼워 넣는 건 누락이 너무 많다. `AsyncLocalStorage`로 한 번 박아두면 어디서든 꺼낼 수 있다.

```typescript
import { AsyncLocalStorage } from 'node:async_hooks';

interface RequestContext {
  requestId: string;
  userId?: string;
  ip?: string;
}

export const requestContext = new AsyncLocalStorage<RequestContext>();

// Express 미들웨어
app.use((req, res, next) => {
  const ctx: RequestContext = {
    requestId: req.header('x-request-id') ?? crypto.randomUUID(),
    userId: req.user?.id,
    ip: req.ip,
  };
  requestContext.run(ctx, () => next());
});

// 로거에서 자동으로 컨텍스트를 가져온다
export const logger = {
  error(message: string, meta: Record<string, unknown> = {}) {
    const ctx = requestContext.getStore() ?? {};
    pino.error({ ...ctx, ...meta }, message);
  },
};
```

이러면 service, repository 어디서 에러를 로깅하든 `requestId`가 자동으로 붙는다. Sentry breadcrumb이나 APM trace에도 같은 방식으로 컨텍스트를 주입할 수 있다.

worker_threads로 작업을 넘길 때 컨텍스트가 끊긴다. 메시지 페이로드에 `requestId`를 같이 실어 보내고, 워커 진입점에서 다시 `requestContext.run`으로 감싸야 연속된다.

## APM 연동 — Sentry / Datadog 컨텍스트와 PII

Sentry로 에러를 보낼 때 사용자 식별과 추가 컨텍스트는 미들웨어에서 주입한다.

```typescript
import * as Sentry from '@sentry/node';

app.use((req, res, next) => {
  Sentry.withScope((scope) => {
    scope.setUser({ id: req.user?.id, email: req.user?.email });
    scope.setTag('route', req.route?.path);
    scope.setExtra('requestId', requestContext.getStore()?.requestId);
    next();
  });
});
```

그리고 PII 필터링이 가장 중요하다. 비밀번호, 토큰, 카드번호, 주민번호 같은 건 절대 외부 SaaS로 나가면 안 된다. `beforeSend` 훅으로 한 번 거른다.

```typescript
const SENSITIVE_KEYS = /password|token|authorization|cookie|cardNumber|cvv|ssn/i;

function scrub(obj: any): any {
  if (!obj || typeof obj !== 'object') return obj;
  if (Array.isArray(obj)) return obj.map(scrub);
  const out: Record<string, unknown> = {};
  for (const [k, v] of Object.entries(obj)) {
    out[k] = SENSITIVE_KEYS.test(k) ? '[REDACTED]' : scrub(v);
  }
  return out;
}

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  beforeSend(event) {
    if (event.request) {
      event.request.data = scrub(event.request.data);
      event.request.headers = scrub(event.request.headers);
      event.request.cookies = undefined;
    }
    if (event.extra) event.extra = scrub(event.extra);
    return event;
  },
});
```

`Authorization` 헤더는 기본 필터에 있긴 한데 커스텀 헤더(`x-internal-token` 같은 것)는 따로 잡아야 한다. 한 번이라도 카드번호가 Sentry에 올라가면 PCI 감사에서 깨진다.

## 트랜잭션 롤백과 흔한 안티패턴

트랜잭션 안에서 에러가 나면 catch한 뒤 commit으로 넘어가는 코드는 거의 항상 버그다. 부분 커밋이 발생한다.

```typescript
// 안티패턴
async function transferBad(from: string, to: string, amount: number) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    await debit(client, from, amount);
    try {
      await credit(client, to, amount);
    } catch (err) {
      logger.error('credit 실패', { err });
      // catch했으니 정상 흐름이라 판단해서 commit으로 넘어감 → 잔액이 사라짐
    }
    await client.query('COMMIT');
  } finally {
    client.release();
  }
}
```

트랜잭션 안에서는 catch로 잡았어도 다시 throw하거나 ROLLBACK을 호출해야 한다. 한 군데서 헬퍼로 묶는 게 안전하다.

```typescript
async function withTransaction<T>(fn: (client: PoolClient) => Promise<T>): Promise<T> {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    const result = await fn(client);
    await client.query('COMMIT');
    return result;
  } catch (err) {
    await client.query('ROLLBACK').catch(() => {}); // ROLLBACK 자체가 실패해도 원본 에러 우선
    throw err;
  } finally {
    client.release();
  }
}

// 사용
await withTransaction(async (client) => {
  await debit(client, from, amount);
  await credit(client, to, amount);
});
```

Prisma, TypeORM도 같은 패턴이다. `prisma.$transaction(async (tx) => { ... })` 콜백 안에서 throw하면 자동 롤백된다. 콜백 안에서 catch만 하고 throw를 안 하면 commit이 일어나니 주의한다.

## Fastify의 setErrorHandler

Fastify는 Express와 미들웨어 모델이 다르다. 모든 에러는 `setErrorHandler`로 들어온다.

```typescript
import Fastify from 'fastify';

const app = Fastify({ logger: true });

app.setErrorHandler((error, request, reply) => {
  if (error instanceof AppError && error.isOperational) {
    reply.status(error.statusCode).send({
      success: false,
      code: error.code,
      message: error.message,
      details: error.details,
    });
    return;
  }

  request.log.error({ err: error }, 'Unexpected error');
  reply.status(500).send({
    success: false,
    code: 'INTERNAL_SERVER_ERROR',
    message: '서버 내부 오류',
  });
});

// 404는 별도
app.setNotFoundHandler((req, reply) => {
  reply.status(404).send({ success: false, code: 'NOT_FOUND', message: '경로 없음' });
});
```

스키마 검증 실패는 `error.validation` 배열이 채워져서 들어온다. 별도 분기로 400 응답을 만든다. 플러그인별로 `setErrorHandler`를 따로 둘 수도 있어서 도메인 단위로 응답 포맷을 다르게 가져갈 수 있다.

## Koa 미들웨어 에러 캐스케이드

Koa는 에러를 가장 바깥 미들웨어에서 잡는다. 모든 라우터를 감싸는 try/catch 미들웨어를 최상단에 둔다.

```typescript
import Koa from 'koa';

const app = new Koa();

app.use(async (ctx, next) => {
  try {
    await next();
  } catch (err) {
    if (err instanceof AppError && err.isOperational) {
      ctx.status = err.statusCode;
      ctx.body = { success: false, code: err.code, message: err.message };
      return;
    }
    ctx.app.emit('error', err, ctx);
    ctx.status = 500;
    ctx.body = { success: false, code: 'INTERNAL_SERVER_ERROR', message: '서버 내부 오류' };
  }
});

app.on('error', (err, ctx) => {
  logger.error('Koa unhandled error', {
    err: formatError(err),
    method: ctx?.method,
    url: ctx?.url,
  });
});
```

`app.on('error')`는 응답을 클라이언트에 못 보내는 단계의 에러(네트워크 끊김 등)도 들어온다. APM 보고는 여기서 한다.

## NestJS Exception Filter 적용 범위

NestJS는 Filter를 어디에 붙이느냐로 적용 범위가 결정된다.

| 적용 위치 | 코드 | 범위 |
|---|---|---|
| 전역 (main.ts) | `app.useGlobalFilters(new GlobalFilter())` | 모든 라우트. DI 못 받음 |
| 전역 (모듈) | `provide: APP_FILTER, useClass: GlobalFilter` | 모든 라우트. DI 받음 |
| 컨트롤러 | `@UseFilters(MyFilter)` 클래스 데코레이터 | 해당 컨트롤러 모든 핸들러 |
| 핸들러 | `@UseFilters(MyFilter)` 메서드 데코레이터 | 해당 핸들러만 |

`APP_FILTER`로 등록하면 `Logger`, `ConfigService` 같은 의존성을 주입받을 수 있다. `useGlobalFilters`로 등록하면 인스턴스를 직접 만들기 때문에 DI 컨테이너 밖이다. 99%는 `APP_FILTER` 방식이 맞다.

```typescript
// app.module.ts
@Module({
  providers: [
    {
      provide: APP_FILTER,
      useClass: GlobalExceptionFilter,
    },
  ],
})
export class AppModule {}
```

특정 도메인만 다른 응답 포맷이 필요하면 컨트롤러 단위 필터를 추가한다. NestJS는 더 좁은 범위의 필터를 우선 적용한다.

WebSocket이나 RPC는 `BaseExceptionFilter`가 아니라 `BaseWsExceptionFilter` / `BaseRpcExceptionFilter`를 상속해야 한다. HTTP용 필터를 그대로 쓰면 응답이 안 나간다.

## 프로덕션에서 자주 보는 실수

**catch에서 throw 누락**

```typescript
try {
  await chargeCard(payload);
} catch (err) {
  logger.error('결제 실패', { err });
  // throw가 없어서 호출자는 성공으로 판단 → 주문이 결제 없이 생성됨
}
```

catch는 "에러를 처리할 능력이 있는 곳"에서만 한다. 로그만 찍고 throw 안 하는 패턴은 거의 항상 버그다. 진짜로 무시해도 되는 경우(통계 수집 실패 등)면 주석으로 명시한다.

**Error 객체가 아닌 값 throw**

```typescript
// 안 됨
throw 'Invalid input';
throw { code: 400, message: 'bad' };

// 스택 트레이스가 없어서 디버깅이 안 된다. catch 쪽에서 instanceof Error도 안 통한다
```

항상 `Error` 또는 그 서브클래스만 던진다. 라이브러리가 문자열을 던지는 경우가 있어서 catch 쪽에서도 방어한다.

```typescript
catch (err) {
  const error = err instanceof Error ? err : new Error(String(err));
  // ...
}
```

**Promise 안에서 동기 throw**

```typescript
// new Promise 콜백 안에서 throw하면 reject로 변환되지만,
// then 안에서 동기 throw는 다음 catch로 가지만 chain이 끊기면 unhandled
new Promise((resolve, reject) => {
  if (!input) throw new Error('bad'); // 다행히 reject로 변환됨
  resolve(process(input));
});

// EventEmitter 핸들러에서 throw는 Promise가 아니라 sync flow → uncaughtException
emitter.on('data', (chunk) => {
  if (!chunk.valid) throw new Error('bad'); // 프로세스가 죽는다
});
```

**async 콜백을 동기 API에 넘기기**

```typescript
[1, 2, 3].forEach(async (id) => {
  await processItem(id); // forEach는 Promise를 안 기다림
});

// for...of 또는 Promise.all 쓴다
for (const id of [1, 2, 3]) {
  await processItem(id);
}
```

**setTimeout/setInterval 안에서 throw**

이것도 EventEmitter와 같다. 콜백 안에서 던진 동기 에러는 어디서도 잡지 못하고 `uncaughtException`까지 올라간다. setTimeout 콜백을 async로 만들고 안쪽에 try/catch를 둔다.

```typescript
setInterval(async () => {
  try {
    await pollExternalApi();
  } catch (err) {
    logger.error('Poll 실패', { err });
  }
}, 60_000);
```

**unref된 타이머와 graceful shutdown**

`setInterval`의 결과를 변수로 잡아두지 않으면 shutdown 시점에 clearInterval을 못 한다. 프로세스 종료가 지연된다.

## 마지막 방어선 — uncaughtException / unhandledRejection

```typescript
process.on('uncaughtException', (err: Error) => {
  logger.fatal('uncaughtException — 프로세스 종료', { err: formatError(err) });
  // 이 시점부터는 메모리 상태를 신뢰할 수 없다
  process.exit(1);
});

process.on('unhandledRejection', (reason: unknown) => {
  logger.fatal('unhandledRejection', { reason });
  process.exit(1);
});
```

여기 도달하면 이미 늦은 상태다. 코드 어딘가에 누수가 있다는 신호이므로 로그를 꼭 남기고 재시작한다. 살려두려고 하면 메모리 손상, half-open 트랜잭션 같은 잠재 폭탄이 쌓인다. PM2, systemd, 쿠버네티스가 자동 재시작해주니 종료가 안전하다.

Node 15부터 `unhandledRejection` 기본 동작이 `throw`로 바뀌어서 결국 `uncaughtException`으로 올라간다. `--unhandled-rejections=strict` 플래그로 확실히 잡고 싶으면 명시한다.
