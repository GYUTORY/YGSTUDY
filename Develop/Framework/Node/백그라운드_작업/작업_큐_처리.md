# Node.js 백그라운드 작업 큐 처리

## 큐가 필요한 이유

HTTP 요청 응답 사이클 안에서 처리하기엔 부담스러운 작업이 있다. 외부 SMTP 서버가 느리면 사용자가 회원가입 화면에서 5초씩 기다리고, 결제 후 영수증 PDF를 합성하느라 응답이 늦어지면 타임아웃이 난다. 게다가 이런 작업들은 실패해도 사용자에게 즉시 보여줄 필요가 없다. 메일 발송이 실패하면 5분 뒤 재시도하면 그만이다.

큐로 분리하기 좋은 작업:

- 이메일·SMS·푸시 알림 발송 (외부 서비스 의존)
- 이미지 리사이징, 영상 트랜스코딩, PDF 합성 (CPU 바운드)
- 대용량 집계, 리포트 빌드, 엑셀 export
- 결제 사후 정산, 환불 처리 (외부 PG 호출)
- 검색 인덱스 갱신, 캐시 워밍업
- 웹훅 수신 후 재처리 (다른 시스템으로 fan-out)

큐로 분리하지 말아야 할 작업도 있다. 응답에 결과가 즉시 필요한 경우(주문번호 발급, 결제 승인), 트랜잭션과 강하게 묶여 있어야 하는 경우(잔액 차감과 동시에 일어나야 하는 작업)는 큐로 빼면 정합성이 깨진다.

## 라이브러리 선택

Node.js 생태계에서 자주 쓰는 큐 라이브러리는 네 가지다. 무엇을 고를지는 메시지 백엔드, 분산 처리 규모, 기능 요구사항에 따라 갈린다.

### BullMQ

Redis 기반. 사실상 표준이라고 봐도 무방하다. TypeScript로 새로 쓰여 타입 지원이 좋고, 우선순위, 지연 작업, 반복 작업, 진행률, 자식 작업(Flow), QueueEvents 등 기능이 풍부하다. lockDuration 기반 stalled 감지가 견고하다. 단점은 Redis ≥ 5.0 (Streams 의존), `maxRetriesPerRequest: null` 같은 ioredis 설정 강제, 학습 곡선이 조금 있다.

새 프로젝트라면 BullMQ를 고른다. 레퍼런스가 가장 많고 유지보수도 활발하다.

### Bull (v3)

BullMQ의 전신. JavaScript로 작성됐고 API가 더 단순하다. 신규 기능 추가는 멈췄고 유지보수 모드다. 이미 Bull로 돌아가는 시스템이라면 굳이 BullMQ로 마이그레이션하지 않아도 된다. 다만 BullMQ와 데이터 포맷이 호환되지 않으므로 두 라이브러리를 동시에 못 쓴다.

### Agenda

MongoDB 기반. Redis를 안 쓰는 환경에서 고를 만하다. cron 표현식 스케줄링이 강점이고, MongoDB에 작업이 도큐먼트로 저장되니 직접 쿼리해서 디버깅하기 편하다. 단점은 처리량이 BullMQ 대비 낮다(MongoDB가 큐 워크로드에 최적화되어 있지 않다). 초당 수백 건 이상이면 부적합하다.

리포트 스케줄링이나 일 단위 배치처럼 처리량이 낮고 MongoDB가 이미 있는 환경에 적합하다.

### Bee-Queue

Redis 기반. BullMQ보다 가볍고 빠르다. 단일 큐에서 초당 만 건 단위 처리량이 필요할 때 고려한다. 단점은 기능이 적다 — 우선순위는 있지만 반복 작업, 자식 작업 같은 고급 기능이 없다. 멱등성과 재시도 정도만 필요한 단순한 워크로드에 맞는다.

### 선택 기준 요약

| 상황 | 선택 |
|------|------|
| 신규 프로젝트, Redis 사용 가능 | BullMQ |
| 기존 Bull 시스템 유지 | Bull v3 그대로 |
| MongoDB만 있고 처리량 낮음 | Agenda |
| 초당 수천 건+, 기능은 최소 | Bee-Queue |
| 메시지 안정성·트랜잭션·라우팅 필요 | RabbitMQ + amqplib (큐 라이브러리가 아닌 메시지 브로커 사용) |

이 문서는 BullMQ 기준으로 설명한다.

## Redis 운영

큐를 운영하면서 가장 많이 사고가 나는 지점이 Redis다. 큐 전용 Redis와 캐시용 Redis를 같은 인스턴스에 두면 거의 반드시 문제가 생긴다.

### maxmemory-policy

캐시용 Redis는 보통 `allkeys-lru` 또는 `volatile-lru`로 설정한다. 메모리가 가득 차면 오래 안 쓴 키를 자동으로 evict한다. 그런데 같은 인스턴스에 큐를 같이 두면 처리 대기 중인 작업이 LRU로 날아가버린다. 이메일이 사라지고, 결제 후속 처리가 사라지고, 데이터 정합성이 깨진다.

큐용 Redis는 `noeviction`으로 설정한다. 메모리가 차면 새 작업 등록이 실패하지만, 기존 작업이 사라지는 것보다 훨씬 낫다.

```bash
# /etc/redis/redis-queue.conf
maxmemory 4gb
maxmemory-policy noeviction
appendonly yes
appendfsync everysec
```

`appendonly yes`는 AOF를 켜는 설정이다. RDB만 켜놓으면 마지막 스냅샷 이후 들어온 작업이 재시작 시 사라진다. 큐는 데이터 손실에 민감하므로 AOF는 켜둬야 한다. `appendfsync everysec`이 성능·내구성 균형점이다.

### 캐시와 큐 분리

이상적으로는 두 개의 Redis 인스턴스를 두지만, 비용이 부담되면 같은 인스턴스 안에서 DB를 분리한다 (`db: 0` 캐시, `db: 1` 큐). 단, 같은 프로세스이므로 메모리 한도는 공유된다. 캐시가 메모리를 다 먹어버리면 큐 작업이 등록 실패한다. DB 분리는 임시방편이고 트래픽이 커지면 분리해야 한다.

```typescript
import IORedis from 'ioredis';

const queueRedis = new IORedis({
  host: 'redis-queue.internal',
  port: 6379,
  db: 0,
  maxRetriesPerRequest: null,    // BullMQ 필수
  enableReadyCheck: false,        // 클러스터 환경에서 권장
});

const cacheRedis = new IORedis({
  host: 'redis-cache.internal',
  port: 6379,
  db: 0,
});
```

`maxRetriesPerRequest: null`을 빼먹으면 BullMQ가 시작 시 에러를 던진다. blocking 명령(`BRPOPLPUSH`)을 무한 대기로 호출해야 하기 때문이다. 캐시용 Redis 연결에는 이 옵션을 넣지 않는다 — 일반 명령이 무한 대기로 매달리면 곤란하다.

### Redis 메모리 모니터링

큐가 쌓이면 메모리도 함께 쌓인다. `INFO memory`로 `used_memory_human`을 주기적으로 확인하고, `maxmemory`의 70% 도달 시 알람을 띄운다. 90% 넘어가면 신규 작업 등록 실패가 시작되므로 그 전에 워커 수를 늘리거나 큐를 비워야 한다.

## Producer 페이로드 직렬화

작업을 큐에 넣을 때 페이로드는 JSON으로 직렬화되어 Redis에 저장된다. 이 단계에서 함정이 많다.

### Date 객체

`JSON.stringify(new Date())` 결과는 ISO 문자열이다. Worker에서 받을 때는 다시 `Date` 객체로 변환되지 않는다. 그대로 `.getTime()`을 호출하면 `TypeError: job.data.scheduledAt.getTime is not a function`이 난다.

```typescript
// 잘못된 예
await emailQueue.add('send', { scheduledAt: new Date() });

// Worker
const worker = new Worker('email', async (job) => {
  job.data.scheduledAt.getTime();  // TypeError
});

// 올바른 예 - 명시적으로 직렬화/역직렬화
await emailQueue.add('send', { scheduledAt: new Date().toISOString() });

const worker = new Worker('email', async (job) => {
  const scheduledAt = new Date(job.data.scheduledAt);
  scheduledAt.getTime();
});
```

### Buffer 객체

이미지 바이너리를 그대로 페이로드에 넣으면 큰일 난다. `JSON.stringify(buffer)`는 `{"type":"Buffer","data":[...]}` 형태로 직렬화되어 원본보다 3~4배 커진다. 1MB 이미지가 4MB 문자열이 되어 Redis에 박힌다.

```typescript
// 절대 하지 말 것
await imageQueue.add('resize', { image: imageBuffer });
```

바이너리는 S3나 로컬 스토리지에 먼저 업로드하고 URL이나 키만 페이로드에 넣는다.

### 큰 객체 — ID만 저장 패턴

리포트를 생성할 때 사용자 정보, 주문 100건, 상품 정보를 통째로 페이로드에 넣고 싶어진다. 이러면 Redis 키 하나가 수십 MB가 된다. Redis는 단일 키 6MB 이상이면 클러스터 마이그레이션 시 문제가 생기고, 메모리 단편화도 빨라진다.

```typescript
// 잘못된 예 - 페이로드 비대화
await reportQueue.add('monthly', {
  user: userObject,
  orders: orderList,  // 1000개일 수도
  products: productMap,
});

// 올바른 예 - ID만 저장, Worker가 다시 조회
await reportQueue.add('monthly', {
  userId: user.id,
  reportMonth: '2026-05',
});

const worker = new Worker('report', async (job) => {
  const user = await userRepo.findById(job.data.userId);
  const orders = await orderRepo.findByMonth(user.id, job.data.reportMonth);
  // ...
});
```

ID만 저장하면 작업 처리 시점에 DB에서 최신 상태를 다시 읽으니 데이터 일관성도 좋아진다. 예를 들어 작업 등록 후 처리 전에 사용자가 주문을 취소했다면, 페이로드에 박혀 있던 옛날 주문 목록이 아닌 최신 상태를 본다.

### 페이로드 크기 제한

운영 룰로 페이로드 1KB 미만을 권장한다. ID 몇 개와 작업 메타데이터로 충분하다. 페이로드가 커지는 순간 의심해야 한다.

## Worker 동시성

동시성 설정은 큐 운영에서 가장 자주 잘못 잡는 부분이다. concurrency와 limiter, 워커 프로세스 수가 어떻게 곱해지는지 정확히 이해해야 한다.

### concurrency

한 워커 프로세스 안에서 동시에 처리할 작업 수다. BullMQ는 단일 프로세스에서 `concurrency: 10`이면 한 번에 10개 작업을 promise.all 비슷하게 처리한다.

```typescript
const worker = new Worker('email', processor, {
  connection,
  concurrency: 10,
});
```

I/O 바운드 작업(외부 API 호출, DB 쿼리)은 concurrency를 높게 잡아도 된다. CPU 바운드 작업(이미지 처리, 암호화)에 concurrency를 높이면 이벤트 루프가 막혀 오히려 처리량이 떨어진다.

### limiter

큐 단위 rate limit이다. 같은 큐에 연결된 모든 워커가 합해서 분당 N개 이상 처리하지 못하게 막는다.

```typescript
const worker = new Worker('sms', processor, {
  connection,
  concurrency: 50,
  limiter: {
    max: 100,
    duration: 60_000,  // 1분에 100개
  },
});
```

외부 API rate limit이 분당 100건이라면 limiter를 100/60000으로 잡는다. 워커 프로세스를 몇 개 띄우든 큐 차원에서 분당 100건을 초과하지 않는다.

### 실제 동시 처리량 계산

3대 서버에 워커 프로세스 2개씩, concurrency 5라면 동시 처리량은 3 × 2 × 5 = 30이다. 외부 API가 동시 요청 10개까지만 받는다면 30이 넘는 순간 429 응답이 떨어진다. 이때 limiter를 워커마다 max=10으로 잡으면 전체 30이 된다 — 큐 전체로 봐야 max=10이다.

```typescript
// 외부 API가 분당 600건, 동시 요청 10건 제한이라면
const worker = new Worker('external-api', processor, {
  connection,
  concurrency: 10,           // 단일 워커 동시 처리
  limiter: {
    max: 600,
    duration: 60_000,
  },
});
// 단일 워커 프로세스만 띄울 것. 여러 프로세스 띄우면 limiter가 큐 단위가 아니라
// 워커 단위로 동작하므로 합산 처리량이 초과될 수 있다.
```

BullMQ v4 이상에서는 limiter가 큐 전체에 걸리므로 워커 여러 개를 띄워도 합산 한도를 지킨다. v3 이하는 워커별이므로 주의해야 한다. 버전 확인하고 쓰는 게 안전하다.

### 프로세스 단위 vs 큐 단위

워커 프로세스를 한 코드에서 여러 큐 처리하게 묶을 수 있지만 권장하지 않는다. 큐별로 트래픽 패턴이 달라서 한 큐가 막히면 다른 큐도 영향을 받는다. PM2나 쿠버 deployment로 큐별로 분리해서 독립 스케일링하는 게 운영하기 쉽다.

```typescript
// emailWorker.ts - 이메일 큐 전용 프로세스
new Worker('email', emailProcessor, { connection, concurrency: 20 });

// imageWorker.ts - 이미지 큐 전용 프로세스
new Worker('image', imageProcessor, { connection, concurrency: 4 });
```

## At-least-once 전달과 멱등성

BullMQ를 포함한 대부분 큐 라이브러리는 at-least-once 전달을 보장한다. exactly-once는 분산 시스템에서 사실상 불가능하다고 보면 된다. 같은 작업이 두 번 실행될 가능성을 항상 염두에 둬야 한다.

중복 실행이 일어나는 시나리오:

1. Worker가 작업을 받아서 처리 완료했는데 ACK 직전에 죽음 → 재시작 후 같은 작업 다시 받음
2. 네트워크 끊김으로 Redis 연결 잃음 → lock 만료 후 다른 워커가 가져감
3. CPU 바운드로 이벤트 루프 막혀 lock 갱신 실패 → stalled 처리되어 재실행

### jobId 멱등화의 한계

BullMQ는 같은 `jobId`로 작업을 등록하면 중복 등록을 막아준다.

```typescript
await emailQueue.add('send', payload, {
  jobId: `welcome-email-${userId}`,
});
```

같은 사용자에게 환영 메일이 두 번 등록되지 않는다. 그런데 이건 등록 시점 중복만 막는다. 작업이 한 번 완료된 후 같은 jobId로 다시 등록하면 새 작업으로 처리된다 (`removeOnComplete` 정책으로 이전 작업이 삭제됐다면). 그리고 처리 중 중복 실행은 막지 못한다.

### DB 유니크 제약

가장 견고한 방법은 DB에 멱등성 키로 유니크 인덱스를 거는 것이다.

```sql
CREATE TABLE email_logs (
  id BIGSERIAL PRIMARY KEY,
  idempotency_key VARCHAR(100) NOT NULL UNIQUE,
  user_id BIGINT NOT NULL,
  sent_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

```typescript
const worker = new Worker('email', async (job) => {
  const idempotencyKey = `welcome-${job.data.userId}`;

  try {
    await db.query(
      'INSERT INTO email_logs (idempotency_key, user_id) VALUES ($1, $2)',
      [idempotencyKey, job.data.userId]
    );
  } catch (err) {
    if (err.code === '23505') {  // PostgreSQL unique violation
      return { status: 'already_sent', skipped: true };
    }
    throw err;
  }

  await sendEmail(job.data);
  return { status: 'sent' };
});
```

INSERT가 먼저 성공한 워커만 실제 발송한다. 중복으로 들어온 두 번째 실행은 unique violation을 받고 조용히 종료된다. 이 패턴의 핵심은 INSERT를 발송 전에 한다는 점이다. 발송 후 INSERT하면 발송은 두 번, INSERT는 한 번 일어나서 의미가 없다.

### Redis 키 멱등성 패턴

DB INSERT 부담이 큰 경우 Redis `SET NX EX`를 쓴다.

```typescript
const worker = new Worker('notification', async (job) => {
  const key = `idempotent:notify:${job.data.notificationId}`;
  const acquired = await redis.set(key, '1', 'EX', 3600, 'NX');

  if (!acquired) {
    return { status: 'duplicate', skipped: true };
  }

  await sendPushNotification(job.data);
  return { status: 'sent' };
});
```

`NX`는 키가 없을 때만 set, `EX 3600`은 1시간 TTL이다. TTL은 잡 처리 가능 최대 시간보다 길어야 한다. 너무 길면 같은 키 재사용 못 함, 너무 짧으면 중복 차단 못 함. 보통 24시간 정도가 무난하다.

이 패턴의 약점은 Redis가 날아가면 멱등성도 같이 날아간다는 점이다. 결제처럼 절대 중복되면 안 되는 작업은 DB 유니크 제약을 쓴다.

## Stalled 작업과 lockDuration

BullMQ는 워커가 작업을 가져갈 때 lock을 건다. 기본 lockDuration은 30초다. 워커는 처리하면서 주기적으로 lock을 갱신한다 (기본 lockDuration의 절반마다). 만약 갱신이 안 되면 다른 워커가 그 작업을 가져갈 수 있다.

```typescript
const worker = new Worker('processing', async (job) => {
  // ... 작업 처리
}, {
  connection,
  lockDuration: 30_000,
  lockRenewTime: 15_000,  // lockDuration의 절반
});
```

### CPU 바운드 작업의 함정

이미지 리사이징처럼 동기 CPU 작업을 하면 이벤트 루프가 막힌다. lock 갱신 타이머도 이벤트 루프 위에서 돌기 때문에 갱신이 실패한다.

```typescript
// 위험한 코드
const worker = new Worker('image', async (job) => {
  const image = await fs.readFile(job.data.path);
  const resized = sharp(image).resize(1000).toBuffer();  // 동기적 처리
  // resize에 40초 걸리면 lock(30초) 만료
  // 다른 워커가 같은 작업 가져가서 또 resize
  await fs.writeFile(job.data.outputPath, await resized);
});
```

증상: 로그에 같은 jobId가 두 번 처리됐다고 찍힘, 처리 결과 파일이 손상되거나 두 번 쓰임, `worker.on('stalled')` 이벤트 발생.

해결 방법은 세 가지:

1. lockDuration을 충분히 길게 (가장 긴 작업의 2배 정도)
2. 작업을 더 작은 단위로 쪼개기
3. Sandboxed processor로 별도 프로세스에서 처리 (아래 섹션 참고)

```typescript
const worker = new Worker('image', processor, {
  connection,
  lockDuration: 120_000,    // 2분
  lockRenewTime: 60_000,
});
```

lockDuration을 무작정 늘리면 진짜로 워커가 죽었을 때 재실행이 늦어진다. 100초 정도가 보통 무난한 상한이다.

### Stalled 감지

```typescript
worker.on('stalled', (jobId) => {
  logger.warn('Stalled job detected', { jobId });
  metrics.increment('queue.stalled', { queue: 'image' });
});
```

stalled가 자주 뜨면 CPU 바운드 작업이 너무 길거나 워커 프로세스가 OOM으로 죽고 있을 가능성이 높다. 알람을 걸어두고 빈도를 모니터링한다.

## UnrecoverableError로 영구 실패 처리

기본적으로 작업이 실패하면 `attempts`까지 재시도한다. 그런데 어떤 실패는 재시도해도 의미가 없다. 사용자가 존재하지 않거나, 이메일 주소가 형식적으로 잘못됐거나, 권한이 없는 경우.

```typescript
import { UnrecoverableError } from 'bullmq';

const worker = new Worker('email', async (job) => {
  const user = await userRepo.findById(job.data.userId);

  if (!user) {
    throw new UnrecoverableError('User not found');
  }

  if (!isValidEmail(user.email)) {
    throw new UnrecoverableError(`Invalid email format: ${user.email}`);
  }

  try {
    await smtpSend(user.email, job.data.body);
  } catch (err) {
    if (err.code === 'EAUTH') {
      throw new UnrecoverableError('SMTP auth failed');
    }
    throw err;  // 일시적 실패는 재시도
  }
});
```

`UnrecoverableError`를 던지면 BullMQ가 즉시 failed 상태로 옮기고 재시도하지 않는다. attempts를 5로 잡았어도 한 번에 실패 처리한다. 큐가 영구 실패 작업으로 막히는 것을 막아준다.

일시적 에러와 영구 에러를 구분하는 게 핵심이다. SMTP 일시 장애(421, 450)는 재시도, 인증 실패(535)는 즉시 포기. HTTP 4xx 중 408·429는 재시도, 400·404는 영구 실패.

## 반복 작업 중복 등록 함정

반복 작업(repeatable job)은 cron 표현식이나 interval로 등록한다. 그런데 코드 배포할 때마다 등록 코드를 실행하면 같은 반복 작업이 여러 개 쌓인다.

```typescript
// 매 배포마다 실행되는 부트스트랩 코드
await reportQueue.add(
  'daily-report',
  {},
  { repeat: { pattern: '0 9 * * *' } }
);
// 배포 10번 하면 매일 오전 9시에 리포트가 10번 만들어짐
```

BullMQ는 반복 작업의 repeat key를 패턴+이름+endDate 등의 조합으로 만든다. opts가 같으면 같은 키가 되어 중복 등록을 막아주지만, opts가 조금이라도 다르면 새 반복 작업이 추가된다.

### 멱등 재등록

배포 시 기존 반복 작업을 모두 삭제하고 다시 등록한다.

```typescript
async function setupRecurringJobs() {
  const repeatable = await reportQueue.getRepeatableJobs();
  for (const job of repeatable) {
    await reportQueue.removeRepeatableByKey(job.key);
  }

  await reportQueue.add(
    'daily-report',
    {},
    {
      repeat: { pattern: '0 9 * * *', tz: 'Asia/Seoul' },
      jobId: 'daily-report',
    }
  );

  await reportQueue.add(
    'hourly-sync',
    {},
    {
      repeat: { every: 60 * 60 * 1000 },
      jobId: 'hourly-sync',
    }
  );
}
```

`getRepeatableJobs()`는 작업 정의를 반환한다. 이미 실행 중이거나 실행 예정인 인스턴스는 영향받지 않으니 안전하게 정리할 수 있다.

타임존도 함정이다. `pattern: '0 9 * * *'`만 쓰면 서버 로컬 타임존 기준이다. 컨테이너 환경에서 UTC로 돌면 한국 시간 오후 6시에 실행된다. `tz: 'Asia/Seoul'`을 명시한다.

## Graceful shutdown

쿠버네티스가 파드를 종료할 때 SIGTERM을 보낸다. 워커는 진행 중인 작업을 마치고 종료해야 한다. 그렇지 않으면 stalled로 처리되어 다른 워커가 다시 가져가고, 멱등성이 없으면 중복 실행이 일어난다.

```typescript
import { Worker } from 'bullmq';

const worker = new Worker('email', processor, {
  connection,
  concurrency: 10,
});

let shuttingDown = false;

async function shutdown(signal: string) {
  if (shuttingDown) return;
  shuttingDown = true;

  logger.info(`Received ${signal}, starting graceful shutdown`);

  // worker.close()는 진행 중인 작업을 기다림
  // 새 작업은 받지 않음
  await worker.close();

  // 연결 정리
  await connection.quit();

  logger.info('Shutdown complete');
  process.exit(0);
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));
```

`worker.close()`는 진행 중인 작업이 끝날 때까지 기다린다. 작업이 5분 걸리는데 쿠버 `terminationGracePeriodSeconds`가 기본값 30초면 강제 SIGKILL이 날아온다.

### terminationGracePeriodSeconds

가장 긴 작업 시간보다 살짝 길게 잡는다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: email-worker
spec:
  template:
    spec:
      terminationGracePeriodSeconds: 120
      containers:
        - name: worker
          image: my-worker:latest
          lifecycle:
            preStop:
              exec:
                command: ["sh", "-c", "sleep 5"]  # readiness 빠지는 시간 확보
```

preStop의 sleep은 서비스에서 readiness가 빠지는 동안 새 요청이 도착하지 않도록 하는 패턴인데, 워커는 HTTP 요청을 받지 않으므로 이 부분은 생략해도 된다. 다만 워커 컨테이너가 HTTP 헬스체크도 함께 노출하는 경우엔 의미가 있다.

타임아웃이 부족해서 SIGKILL이 자주 일어나면 stalled 작업이 늘어나고 운영이 어려워진다. 작업 평균/최대 시간을 메트릭으로 보고 적절히 조정한다.

## Sandboxed processor

CPU 바운드 작업은 메인 프로세스의 이벤트 루프를 막는다. lock 갱신이 실패해서 stalled 발생, 다른 큐의 I/O 작업도 함께 느려진다. BullMQ는 워커를 별도 프로세스에서 실행하는 sandboxed 모드를 제공한다.

```typescript
// processors/imageProcessor.ts
import { Job } from 'bullmq';
import sharp from 'sharp';

export default async function imageProcessor(job: Job): Promise<{ size: number }> {
  const image = await fs.readFile(job.data.path);
  const resized = await sharp(image).resize(1000).toBuffer();
  await fs.writeFile(job.data.outputPath, resized);
  return { size: resized.length };
}
```

```typescript
// workers/imageWorker.ts
import { Worker } from 'bullmq';
import path from 'path';

const processorFile = path.join(__dirname, '../processors/imageProcessor.js');

const worker = new Worker('image', processorFile, {
  connection,
  concurrency: 4,  // 별도 프로세스 4개 spawn
});
```

processor 경로를 문자열로 넘기면 BullMQ가 child_process로 띄운다. 메인 프로세스는 작업 분배만 하고 실제 처리는 자식 프로세스에서 일어난다. 자식 프로세스가 죽어도 메인 프로세스는 살아 있다.

주의할 점:

- processor 파일은 컴파일된 .js여야 한다 (TypeScript는 ts-node 등으로 처리 필요)
- 부모 프로세스의 변수, 함수, DB 커넥션을 공유할 수 없다 — processor는 격리된 환경에서 실행됨
- 각 처리마다 새 프로세스를 띄우는 게 아니라 worker 시작 시 pool을 만들고 재사용
- 자식 프로세스의 메모리는 부모와 별도 — concurrency를 너무 높이면 OOM

이미지 트랜스코딩, PDF 합성, 큰 데이터 압축 같은 CPU 작업에 적합하다. I/O 작업에는 오버헤드만 추가될 뿐 이점이 없다.

## DLQ 패턴

Dead Letter Queue는 최대 재시도 후 실패한 작업을 별도 큐에 옮겨 보관하는 패턴이다. BullMQ에 DLQ가 기본 내장돼 있지는 않고 직접 구현해야 한다.

```typescript
// queues/index.ts
import { Queue, Worker, QueueEvents } from 'bullmq';

const connection = createConnection();

export const emailQueue = new Queue('email', { connection });
export const emailDLQ = new Queue('email-dlq', { connection });

// DLQ로 옮기는 이벤트 리스너
const events = new QueueEvents('email', { connection });

events.on('failed', async ({ jobId, failedReason }) => {
  const job = await emailQueue.getJob(jobId);
  if (!job) return;

  const maxAttempts = job.opts.attempts ?? 1;
  if (job.attemptsMade < maxAttempts) return;  // 아직 재시도 가능

  await emailDLQ.add('dead-letter', {
    originalQueue: 'email',
    originalJobId: jobId,
    originalName: job.name,
    originalData: job.data,
    failedReason,
    failedAt: new Date().toISOString(),
    attemptsMade: job.attemptsMade,
    stacktrace: job.stacktrace,
  }, {
    removeOnComplete: false,  // DLQ는 명시적으로 처리할 때까지 보관
    removeOnFail: false,
  });

  metrics.increment('queue.dlq.added', { queue: 'email' });
});
```

DLQ는 자동 처리하지 않는다. 사람이 보고 원인 분석 후 수동으로 재처리하거나 폐기한다.

```typescript
// DLQ에서 원래 큐로 재시도
async function retryDLQJob(dlqJobId: string) {
  const job = await emailDLQ.getJob(dlqJobId);
  if (!job) throw new Error('DLQ job not found');

  await emailQueue.add(job.data.originalName, job.data.originalData, {
    attempts: 3,
  });

  await job.remove();
}
```

운영 화면에서 DLQ 목록 조회 → 원인 확인 → 코드 수정 또는 데이터 수정 → 재시도 또는 삭제. 결제·정산처럼 누락되면 안 되는 작업은 반드시 DLQ를 둔다.

## 큐 메트릭

큐 상태는 5가지 핵심 상태로 본다.

| 상태 | 의미 | 알람 기준 |
|------|------|---------|
| waiting | 대기 중, 워커가 가져가지 않음 | 평소 대비 10배, 또는 절대값 임계치 초과 |
| active | 처리 중 | concurrency × 워커 수 근처에서 안정. 0이 지속되면 워커 다운 의심 |
| completed | 처리 완료 | 일정 시간당 처리량 감소 시 알람 |
| failed | 재시도 끝까지 실패 | failed 비율(failed / completed) 1% 초과 시 알람 |
| delayed | 지연 작업 (delay, repeat 대기 중) | 갑작스러운 증가 시 의심 |

```typescript
// 메트릭 수집
async function collectQueueMetrics(queue: Queue): Promise<void> {
  const counts = await queue.getJobCounts(
    'waiting', 'active', 'completed', 'failed', 'delayed'
  );

  metrics.gauge('queue.waiting', counts.waiting, { queue: queue.name });
  metrics.gauge('queue.active', counts.active, { queue: queue.name });
  metrics.gauge('queue.failed', counts.failed, { queue: queue.name });
  metrics.gauge('queue.delayed', counts.delayed, { queue: queue.name });
}

setInterval(() => {
  collectQueueMetrics(emailQueue).catch(console.error);
}, 10_000);
```

알람 기준 예시:

- `waiting > 1000` 5분 지속 → 워커 부족 또는 처리 시간 증가
- `active == 0` 1분 지속 → 워커 전원 다운 또는 Redis 연결 끊김
- `failed/completed > 0.05` 5분 윈도우 → 외부 의존성 장애 가능성
- `stalled` 카운터 분당 5건 이상 → CPU 바운드 작업이 lock 갱신 못 함

처리 지연 시간도 중요한 지표다. 작업 등록 시점과 active 진입 시점 차이로 wait time을 측정한다.

```typescript
const events = new QueueEvents('email', { connection });

events.on('active', async ({ jobId, prev }) => {
  const job = await emailQueue.getJob(jobId);
  if (!job?.timestamp) return;

  const waitMs = Date.now() - job.timestamp;
  metrics.histogram('queue.wait_time', waitMs, { queue: 'email' });
});

events.on('completed', async ({ jobId, returnvalue }) => {
  const job = await emailQueue.getJob(jobId);
  if (!job?.processedOn || !job?.finishedOn) return;

  const processMs = job.finishedOn - job.processedOn;
  metrics.histogram('queue.process_time', processMs, { queue: 'email' });
});
```

## 흔한 함정 트러블슈팅

### 워커가 작업을 가져가지 않는다

가장 흔한 원인은 큐 이름 오타다. Producer는 `'email'`로 add하는데 Worker는 `'Email'`로 listen 중. Redis는 케이스 센서티브하므로 다른 큐로 본다.

```bash
# Redis CLI로 확인
redis-cli
> KEYS bull:*
1) "bull:email:waiting"
2) "bull:Email:waiting"  # 두 개가 따로 보이면 오타
```

다음으로 흔한 원인은 connection 설정 실수. Producer는 `db: 0`, Worker는 `db: 1`로 연결하면 다른 Redis DB라 작업을 못 본다.

### 같은 작업이 두 번 처리됨

원인은 보통 셋 중 하나:

1. lockDuration 짧음 + CPU 바운드 작업 → Sandboxed processor 또는 lockDuration 증가
2. Graceful shutdown 누락 → 워커 SIGKILL로 죽으면 lock 만료 후 재실행
3. 멱등성 누락 → DB unique 제약 또는 Redis NX 패턴 적용

### Redis 메모리 폭증

대부분 `removeOnComplete` 누락 때문이다. 기본값은 모든 완료 작업을 영구 보관이다.

```typescript
await queue.add('job', data, {
  removeOnComplete: { count: 100, age: 24 * 3600 },  // 최근 100개 또는 24시간
  removeOnFail: { count: 500, age: 7 * 24 * 3600 },  // 실패는 더 길게 보관
});
```

전역 기본값을 큐 옵션에 박아두면 매번 안 적어도 된다.

```typescript
export const emailQueue = new Queue('email', {
  connection,
  defaultJobOptions: {
    attempts: 3,
    backoff: { type: 'exponential', delay: 2000 },
    removeOnComplete: { count: 100 },
    removeOnFail: { count: 500 },
  },
});
```

### 처리량이 안 나옴

체크 순서:

1. `active` 카운트가 `concurrency × 워커 수`에 도달하는가 — 도달 안 하면 작업 자체가 부족하거나 limiter에 걸림
2. CPU 사용률 확인 — 100%면 sandboxed processor 또는 워커 프로세스 증가
3. DB·외부 API 응답 시간 확인 — 워커는 빠른데 의존성이 느린 경우

### Job lost (작업이 사라짐)

거의 항상 Redis 운영 문제다.

- maxmemory-policy가 `allkeys-lru`였다 → noeviction으로 변경
- AOF 미적용 상태에서 Redis 재시작 → AOF 켜기
- Redis 클러스터 페일오버 중 미동기화 데이터 손실 → 재시도 가능한 작업으로 설계

### "Missing lock for job" 에러

Worker가 작업 처리 도중 lock을 잃은 경우다. 보통 lockDuration보다 처리가 오래 걸렸을 때 발생. lockDuration을 늘리거나, `job.updateProgress()`를 주기적으로 호출해서 lock을 자동 갱신하게 한다. (BullMQ는 progress 업데이트 시 lock도 함께 갱신한다.)

```typescript
const worker = new Worker('long-job', async (job) => {
  const items = await fetchItems();

  for (let i = 0; i < items.length; i++) {
    await processItem(items[i]);
    if (i % 10 === 0) {
      await job.updateProgress((i / items.length) * 100);
    }
  }
});
```

### Producer 측 add() 응답이 느림

Redis 응답 자체가 느린 경우다. `addBulk()`로 묶어서 등록하면 라운드트립이 줄어든다. 같은 트랜잭션 안에서 100개 작업을 등록한다면 add()를 100번 호출하지 말고 한 번에 addBulk() 호출한다.

```typescript
await queue.addBulk(
  users.map(user => ({
    name: 'send-welcome',
    data: { userId: user.id },
    opts: { attempts: 3, jobId: `welcome-${user.id}` },
  }))
);
```

## 마무리 노트

큐는 한 번 잘못 설계하면 운영 중에 데이터 정합성 문제로 돌아온다. 처음부터 멱등성, graceful shutdown, DLQ를 갖춰서 시작하는 게 나중에 사고 치고 후회하는 것보다 비용이 적다. 모니터링은 waiting/active/failed 3개만 보고도 대부분 문제를 잡을 수 있으니 알람만 제대로 걸어두자.
