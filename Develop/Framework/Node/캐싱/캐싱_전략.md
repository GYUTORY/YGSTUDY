# Node.js 캐싱 기본

## 문서 범위

이 문서는 캐싱을 처음 도입하는 사람이 무엇을 알고 시작해야 하는지에 집중한다. hit/miss가 뭔지, TTL을 얼마로 잡아야 하는지, Cache-Aside와 Write-Through가 어떻게 다른지, node-cache와 Redis 중 뭘 골라야 하는지 같은 1차 의사결정을 다룬다.

LRU/LFU 같은 교체 알고리즘 내부 구현, Cache Stampede 방어, L1+L2 다층 캐시는 이 문서 범위 밖이다. 기본 패턴이 손에 익은 다음에 [Node.js 캐싱 심화](Node_Cache_Advanced.md)와 [다층 캐시](Multi_Level_Cache.md) 문서로 넘어가면 된다.

---

## 캐시가 왜 필요한가

같은 데이터를 같은 방식으로 반복해서 조회한다면 매번 DB까지 갈 필요가 없다. 사용자 프로필, 카테고리 목록, 환율 같은 것들은 1초 안에 같은 키로 수십 번 들어온다. 한 번만 DB에서 읽고 나머지 N-1번은 메모리에서 응답하면 그 N-1번 만큼의 부하가 사라진다.

실제로 측정해보면 PostgreSQL에서 단일 row를 인덱스로 조회하는 데 LAN 환경이라도 평균 1~3ms가 든다. 같은 데이터를 인메모리 캐시에서 꺼내면 1µs도 안 걸린다. 1000배 차이가 나는데, 이게 왜 중요하냐면 동일 요청을 동시에 100개 받았을 때 DB는 100개 커넥션이 필요하고 캐시는 그냥 100번 읽기만 하면 끝이다. 커넥션 풀이 마르면 그 다음 들어오는 요청은 다 대기열에 쌓인다.

캐싱을 검토할 만한 신호는 대략 이런 것들이다.

- 같은 SQL이 슬로우 쿼리 로그에 안 잡혀도 초당 수백 번 실행되고 있다
- DB CPU의 대부분이 같은 read에 쓰이고 있다
- 외부 API 응답이 느리거나 호출당 과금된다
- JSON 파싱·변환·집계 같은 동기 연산이 반복된다

반대로 매 요청마다 다른 데이터를 보거나, 데이터가 초당 변하거나, 정합성이 절대적인 영역(잔액, 재고)에서는 일단 캐시 도입을 미루는 게 낫다.

## hit과 miss, 그리고 hit ratio

캐시를 조회했을 때 데이터가 있으면 hit, 없으면 miss다. miss가 나면 원본 소스(DB, API)에 가서 가져온 다음 캐시에 채워두고 응답한다. 다음 같은 요청은 hit이 된다.

hit ratio는 전체 조회 중 hit의 비율이다.

```
hit ratio = hits / (hits + misses)
```

100번 조회해서 80번은 캐시에서 답이 나오고 20번은 DB까지 갔다면 hit ratio는 0.8(80%)이다. 이 숫자가 캐시 효율을 가늠하는 가장 직관적인 지표다.

hit ratio가 얼마면 괜찮은지에 정답은 없지만 경험적으로는 이렇다. 자주 바뀌지 않는 마스터성 데이터(카테고리, 설정)는 95% 이상 나와야 정상이다. 사용자별 프로필 같이 사용자마다 다른 데이터는 70~90% 정도. 처음 들어오는 신규 사용자가 많거나 매번 다른 검색 쿼리를 받는 영역은 30~50%로 떨어지기도 한다. 50% 미만이면 캐시 도입 자체가 의미 있는지 다시 봐야 한다. miss할 때마다 원본 조회 + 캐시 쓰기 두 번 일이 늘어나기 때문이다.

측정하는 코드는 이 정도면 충분하다.

```typescript
class HitRateTracker {
  private hits = 0;
  private misses = 0;

  record(isHit: boolean) {
    if (isHit) this.hits++;
    else this.misses++;
  }

  snapshot() {
    const total = this.hits + this.misses;
    const ratio = total === 0 ? 0 : this.hits / total;
    return { hits: this.hits, misses: this.misses, ratio };
  }

  reset() {
    this.hits = 0;
    this.misses = 0;
  }
}

const tracker = new HitRateTracker();

setInterval(() => {
  const { hits, misses, ratio } = tracker.snapshot();
  logger.info('cache stats', { hits, misses, ratio: ratio.toFixed(3) });
  tracker.reset();
}, 60_000);
```

처음 도입했을 때 hit ratio가 0.3 근처에서 안 올라간다면 TTL이 너무 짧거나, 키가 너무 다양해서 같은 키가 재사용되지 않는 경우가 대부분이다. 그 다음에 의심해볼 곳은 캐시 용량이다. 용량이 부족해서 채우자마자 evict되고 있으면 hit이 안 난다.

## 4가지 읽기/쓰기 패턴

캐시와 원본 저장소(DB)의 관계를 어떻게 묶을지 결정하는 4가지 표준 패턴이 있다. 어떤 걸 쓰느냐에 따라 코드 위치, 일관성, 장애 시 동작이 전부 달라진다.

| 패턴 | 읽기 | 쓰기 | 장점 | 단점 | 적합한 상황 |
|------|------|------|------|------|-------------|
| Cache-Aside | 앱이 캐시 먼저 보고 miss면 DB | 앱이 DB 쓰고 캐시 갱신 또는 무효화 | 단순, 캐시 장애에 강함 | 첫 조회는 무조건 miss, 정합성 책임이 앱에 있음 | 대부분의 일반 서비스 |
| Read-Through | 앱은 캐시만 호출, 캐시가 내부적으로 DB 조회 | 앱이 DB에 직접 씀 | 앱 코드가 깔끔 | 캐시 라이브러리/프록시 의존 | 캐시 추상 라이브러리를 쓸 때 |
| Write-Through | Cache-Aside와 동일 | 앱이 캐시와 DB를 동시에 씀 | 캐시가 항상 최신 | 쓰기 지연 2배, DB 장애 시 쓰기 실패 | 읽기 비율이 압도적이고 stale을 못 견딤 |
| Write-Back | Cache-Aside와 동일 | 앱이 캐시에만 쓰고 DB는 비동기 반영 | 쓰기 처리량 최고 | 캐시 장애 시 데이터 유실 | 메트릭 카운터, 조회수 등 유실 허용 |

### Cache-Aside

가장 흔하고 가장 무난하다. 처음 캐시를 도입한다면 거의 항상 이 패턴이다.

```typescript
async function getProduct(id: string): Promise<Product> {
  const cached = await cache.get<Product>(`product:${id}`);
  if (cached) return cached;

  const product = await productRepository.findById(id);
  if (!product) throw new NotFoundError('상품');

  await cache.set(`product:${id}`, product, 300);
  return product;
}

async function updateProduct(id: string, data: UpdateProductDto): Promise<Product> {
  const updated = await productRepository.update(id, data);
  await cache.del(`product:${id}`);
  return updated;
}
```

쓰기 시 캐시를 무효화(`del`)하느냐 갱신(`set`)하느냐는 선택의 문제다. 무효화는 다음 조회에서 다시 채우는 lazy 방식이라 단순하고, 동시 쓰기 시 잘못된 값이 들어갈 위험이 적다. 갱신은 캐시 hit이 끊기지 않지만 동시에 두 트랜잭션이 다른 값을 쓰면 마지막 쓰기가 이긴다. 헷갈릴 때는 무효화가 안전하다.

장점은 캐시가 죽어도 서비스가 계속 돈다는 점이다. 캐시 미스가 100%인 상태로 DB만 보면 된다. 단점은 처음 조회는 항상 miss라서 cold start 시 DB가 한 번 두들겨 맞는다.

### Read-Through

앱은 캐시에만 요청하고, 캐시 라이브러리가 miss를 감지하면 직접 DB에서 가져온다. 코드만 보면 Cache-Aside와 비슷하지만 책임 위치가 다르다.

```typescript
const cache = new ReadThroughCache({
  loader: async (key: string) => {
    const id = key.replace('product:', '');
    return productRepository.findById(id);
  },
  ttl: 300,
});

const product = await cache.get('product:123');
```

NestJS의 `@Cacheable` 데코레이터, Spring의 `@Cacheable`, AWS DAX 같은 게 이 모델이다. Node.js 생태계에서는 직접 구현하거나 `cacheable` 같은 패키지를 쓴다. Cache-Aside의 보일러플레이트가 반복되면 그때 도입하는 게 맞다. 처음부터 끌고 들어오면 디버깅이 어려워진다.

### Write-Through

쓰기 시점에 캐시와 DB를 동시에 갱신한다. Cache-Aside의 갱신 방식과 비슷해 보이지만 차이는 의도다. Write-Through는 캐시가 항상 DB와 일치한다고 가정하고 읽기 시 캐시를 무조건 신뢰한다.

```typescript
async function updateUser(id: string, data: UpdateUserDto): Promise<User> {
  const user = await userRepository.update(id, data);
  await cache.set(`user:${id}`, user, 600);
  return user;
}
```

문제는 두 가지 쓰기를 어떻게 원자적으로 묶느냐다. DB 쓰기는 성공했는데 캐시 쓰기가 실패하면 캐시에 stale이 남는다. 반대로 캐시는 됐는데 DB가 실패하면 캐시에 미래 값이 들어간다. 트랜잭션으로 묶을 수 없는 이상 어느 쪽으로든 깨지는데, 보통 "DB 먼저 쓰고 캐시 갱신, 캐시 실패 시 무효화"로 처리한다. 정합성에 정말 민감하면 Cache-Aside의 무효화 방식이 더 안전하다.

### Write-Back (Write-Behind)

쓰기를 캐시에만 하고 DB 반영은 큐를 통해 비동기로 일괄 처리한다. 처리량은 가장 높지만 캐시가 죽으면 미반영 쓰기가 통째로 사라진다.

```typescript
async function increaseViewCount(postId: string): Promise<void> {
  await cache.incr(`post:view:${postId}`);
}

setInterval(async () => {
  const keys = await cache.scanKeys('post:view:*');
  for (const key of keys) {
    const count = await cache.get<number>(key);
    const postId = key.replace('post:view:', '');
    await postRepository.addViewCount(postId, count ?? 0);
    await cache.del(key);
  }
}, 30_000);
```

조회수, 좋아요 수, 메트릭 같이 1~2개 누락돼도 큰 문제 없는 영역에만 쓴다. 결제 정보나 주문 상태에 쓰면 사고난다. 실제로 본 사례 중에 Write-Back으로 게시판 조회수를 처리하다가 Redis가 OOM으로 재시작되면서 30분치 조회수가 날아간 적이 있다. 그 정도는 허용 가능하다 판단했으니까 이 패턴을 골랐던 것이다.

## TTL을 얼마로 잡을 것인가

TTL(Time To Live)은 캐시 항목이 살아있는 시간이다. 1시간으로 잡으면 1시간 동안은 같은 키를 묻는 모든 요청이 hit이 되고, 1시간 1초가 되면 다음 요청에서 다시 DB를 친다.

TTL을 결정하는 두 축은 데이터 변경 주기와 허용 가능한 stale 시간이다.

데이터 변경 주기는 그 데이터가 평균 얼마마다 바뀌는지다. 카테고리 목록은 한 달에 한두 번 추가되고, 사용자 프로필은 가끔 수정되고, 상품 가격은 분 단위로 바뀔 수도 있다. 변경이 거의 없으면 TTL을 길게 가져갈 수 있다.

허용 가능한 stale 시간은 비즈니스 요구다. "수정 후 5분까지는 옛날 값을 보여줘도 괜찮다"는 합의가 있다면 TTL을 5분으로 잡으면 된다. "수정하자마자 다른 사용자에게 보여야 한다"면 TTL과 별개로 무효화 로직이 반드시 필요하다.

대략적인 가이드라인은 이렇다.

| 데이터 종류 | 권장 TTL | 이유 |
|-------------|----------|------|
| 환율, 날씨 같은 외부 API | 5~30분 | 외부 호출 비용을 줄이는 게 목적, 실시간성 낮음 |
| 사용자 프로필 | 5~10분 | 수정 시 무효화 로직 같이 둠 |
| 카테고리, 공통 설정 | 1~24시간 | 거의 안 바뀜 |
| 상품 상세, 게시글 | 1~5분 | 자주 보고 가끔 수정됨 |
| 검색 결과 | 30초~2분 | 길면 stale, 짧으면 hit ratio 안 나옴 |
| 인증 토큰 검증 결과 | 30초~1분 | 길면 권한 변경이 반영 안 됨 |

기본값을 정해놓고 시작하는 게 낫다. 보통 5분(300초)을 기본으로 쓰고, 데이터 특성에 따라 늘리거나 줄인다. 처음부터 키마다 다른 TTL을 정밀하게 설계하려 하면 시간만 쓰고 결과는 비슷하다. 일단 도입하고 hit ratio와 stale 신고 빈도를 보면서 조정한다.

TTL을 짧게 잡으면 안전하지만 hit ratio가 떨어지고, 길게 잡으면 hit ratio는 올라가지만 stale 위험이 커진다. 두 가지를 동시에 잡고 싶으면 TTL은 길게 두고 무효화를 명시적으로 하는 방식이 표준이다.

## 캐시 키 설계와 네임스페이스 충돌

캐시 키는 한 번 잘못 잡으면 디버깅이 정말 까다롭다. 다른 객체가 같은 키에 덮어쓰여서 사용자 A의 데이터가 사용자 B에게 보이는 사고가 실제로 일어난다.

기본 규칙은 이렇다.

```
{도메인}:{식별자}:{서브키}
```

예: `user:123`, `user:123:posts`, `product:list:category:5:page:1`. 콜론으로 구분하면 사람이 읽기 쉽고 Redis에서 패턴 삭제(`SCAN MATCH user:*`)가 자연스럽다.

자주 보는 실수 사례를 몇 개 든다.

**환경 구분 누락.** 개발/스테이징/프로덕션이 같은 Redis 인스턴스를 공유하는 경우가 의외로 많다. 키를 `user:123`으로만 쓰면 개발에서 넣은 데이터를 프로덕션이 읽는다. `prod:user:123`, `dev:user:123` 식으로 환경 prefix를 붙이거나, Redis DB 번호를 분리한다(0번은 prod, 1번은 staging 등).

**인스턴스/테넌트 구분 누락.** 멀티테넌트 SaaS에서 `user:123`만 쓰면 테넌트 A의 user 123과 테넌트 B의 user 123이 충돌한다. `tenant:abc:user:123`로 잡아야 한다.

**버전이 안 들어가서 직렬화 포맷 변경 시 깨짐.** User 객체에 새 필드를 추가했는데 캐시에 옛날 포맷이 살아있으면 코드가 `undefined`를 만나서 터진다. 키에 `v1:user:123`처럼 버전을 박아두면 코드 배포 시 prefix만 v2로 바꿔서 통째로 무효화할 수 있다.

**파라미터를 그대로 키에 넣어서 키가 무한 증식.** 검색 결과를 `search:{query}`로 캐싱하면 사용자가 입력한 모든 검색어가 키가 된다. 1주일이면 수십만 키가 쌓이고 대부분 다시 안 쓰여서 메모리만 먹는다. 짧은 TTL(30~60초)을 강제하거나, 인기 검색어만 캐시하도록 조건을 둔다.

키 생성을 한곳에 모아두면 사고가 줄어든다.

```typescript
const ENV = process.env.NODE_ENV ?? 'development';
const CACHE_VERSION = 'v1';

class CacheKeys {
  private static prefix(domain: string) {
    return `${ENV}:${CACHE_VERSION}:${domain}`;
  }

  static user(id: string) {
    return `${this.prefix('user')}:${id}`;
  }

  static userPosts(userId: string) {
    return `${this.prefix('user')}:${userId}:posts`;
  }

  static product(id: string) {
    return `${this.prefix('product')}:${id}`;
  }
}
```

문자열 리터럴을 코드 곳곳에 흩뿌리면 오타 한 번에 hit ratio가 0이 된다. `user:${id}`와 `users:${id}`처럼 단수/복수만 다른 키가 섞이면 같은 데이터를 두 키로 캐싱해서 무효화도 두 번 해야 한다.

## JSON 직렬화 비용

Redis는 바이트 시퀀스만 저장하므로 객체를 보내려면 직렬화가 필요하다. 대부분 `JSON.stringify`/`JSON.parse`를 쓰는데, 이게 생각보다 비싸다.

100KB 정도 되는 객체를 `JSON.parse`로 파싱하면 약 1~2ms가 든다. Redis 호출 자체가 0.5~1ms인데 직렬화에서 비슷한 시간을 또 쓰는 셈이다. 작은 객체(1KB 이하)는 무시할 수 있지만, 큰 배열·중첩 객체를 캐시한다면 직렬화 비용이 캐시 효과를 깎아먹는다.

대안은 두 가지 정도다.

**Buffer 그대로 저장.** 이미 직렬화된 형태(예: 외부 API 응답을 그대로 받은 경우)라면 객체로 파싱하지 말고 Buffer/문자열 그대로 캐시했다가 그대로 응답에 흘려보낸다. 파싱·재직렬화 두 번을 0번으로 줄인다.

**MessagePack 같은 바이너리 포맷.** `msgpackr`이나 `notepack.io` 같은 라이브러리는 JSON보다 30~50% 작고 파싱이 2~3배 빠르다. 단 디버깅 시 Redis에서 값을 직접 보면 사람이 읽을 수 없다는 단점이 있다. 큰 객체를 자주 캐시하는 경우에만 검토한다.

처음부터 MessagePack부터 쓰지는 말자. JSON으로 시작하고 프로파일링에서 직렬화가 병목으로 잡힐 때 바꾸는 게 맞다. 대부분의 서비스는 JSON으로도 충분하다.

## node-cache vs lru-cache vs Redis

뭘 골라야 하는지에 대한 답은 트래픽 규모와 인스턴스 수다.

| 라이브러리 | 위치 | 적합한 상황 | 인스턴스 간 공유 | 메모리 한계 |
|------------|------|-------------|------------------|-------------|
| `node-cache` | 인메모리 | 단일 프로세스, 개발용, 단순 키-값 | 안 됨 | 프로세스 RAM |
| `lru-cache` | 인메모리 | 단일 프로세스, 용량 제한 필요, 핫 데이터 | 안 됨 | 프로세스 RAM |
| Redis (`ioredis`, `redis`) | 외부 서버 | 다중 인스턴스, 영속성 필요, 큰 용량 | 됨 | Redis 인스턴스 RAM |

판단 기준은 이렇다.

**인스턴스가 1개고 트래픽이 작다(수십~수백 RPS).** node-cache로 시작한다. 셋업이 가장 단순하고 외부 의존이 없다. 인스턴스가 죽으면 캐시도 같이 사라지는데, 어차피 트래픽이 작으니 cold start 영향이 미미하다.

**인스턴스가 1개지만 캐시 항목이 많거나 메모리 제어가 필요하다.** lru-cache로 간다. node-cache는 용량이 무한정 자라다 OOM이 날 수 있는데 lru-cache는 max 항목 수나 max 바이트로 자동 evict된다. 핫 데이터만 메모리에 유지하는 게 목적이라면 lru-cache가 정답이다.

**인스턴스가 2개 이상이다.** 무조건 Redis다. 인메모리만 쓰면 인스턴스마다 캐시가 다르게 채워져서 어떤 사용자는 stale을 보고 어떤 사용자는 fresh를 본다. 한 인스턴스에서 무효화해도 다른 인스턴스는 모른다. 디버깅이 정말 어려워진다. 부하분산 뒤에 Node가 2대 이상 있다면 첫날부터 Redis로 시작한다.

**캐시 데이터가 인스턴스 재시작에도 살아있어야 한다.** Redis에 persistence(RDB/AOF)를 켜거나, 어차피 Redis는 외부 프로세스니까 Node 재시작과 무관하게 유지된다.

**최고 성능이 필요하다.** Redis만으로 부족하면 그때 인메모리(L1) + Redis(L2) 다층 구조를 검토한다. 자세한 내용은 [다층 캐시](Multi_Level_Cache.md) 문서로 넘긴다. 다층은 일관성 문제가 새로 생기므로 단순 Redis만으로 처리량이 부족할 때 마지막 수단으로 고민한다.

처음 시작할 때는 node-cache나 lru-cache 중 하나로 단일 인스턴스에서 검증하고, 스케일아웃 시점에 Redis로 마이그레이션하는 흐름이 자연스럽다. 단, 마이그레이션을 염두에 두고 처음부터 캐시 호출을 추상화 레이어 뒤에 두면 교체가 쉽다.

```typescript
interface CacheStore {
  get<T>(key: string): Promise<T | null>;
  set(key: string, value: unknown, ttl: number): Promise<void>;
  del(key: string): Promise<void>;
}

class NodeCacheStore implements CacheStore { /* node-cache 래핑 */ }
class RedisCacheStore implements CacheStore { /* ioredis 래핑 */ }

const cache: CacheStore = process.env.REDIS_URL
  ? new RedisCacheStore()
  : new NodeCacheStore();
```

## 로컬 개발 시 캐시 끄기

캐시는 운영에서 빛나지만 개발할 때는 발목을 잡는다. 코드를 고쳐서 응답이 바뀌어야 하는데 캐시 때문에 옛날 값이 계속 나온다. 매번 Redis FLUSHDB를 치는 것도 짜증나는 일이다.

환경변수로 끄는 패턴이 가장 단순하다.

```typescript
const CACHE_DISABLED = process.env.CACHE_DISABLED === 'true';

class NoOpCache implements CacheStore {
  async get<T>(): Promise<T | null> { return null; }
  async set(): Promise<void> {}
  async del(): Promise<void> {}
}

const cache: CacheStore = CACHE_DISABLED
  ? new NoOpCache()
  : new RedisCacheStore();
```

`get`이 항상 `null`을 반환하면 모든 호출이 miss로 처리되고 매번 DB로 간다. 코드 흐름은 똑같이 타지만 캐시 효과는 0이다. 디버깅이 끝나면 환경변수만 풀면 된다.

`.env.development`에 `CACHE_DISABLED=true`를 기본으로 깔아두는 팀도 있고, 의심될 때만 켜는 팀도 있다. 개발자 사이에 합의만 되어 있으면 어느 쪽이든 괜찮다.

테스트 환경은 보통 캐시를 꺼두는 게 안전하다. 통합 테스트에서 캐시가 끼어들면 테스트 간 격리가 깨진다. 테스트 시작마다 캐시를 비우거나, NoOpCache로 주입한다.

## 캐시 워밍업

워밍업(warmup)은 서비스 시작 시점에 캐시를 미리 채워두는 작업이다. cold start 시 첫 요청들이 miss를 다 먹는 걸 막는 게 목적이다.

워밍업이 필요한 상황과 안 해도 되는 상황을 구분해야 한다.

워밍업이 필요한 경우는 대략 이렇다.

- 인기 데이터가 명확히 정해져 있다(상위 100개 카테고리, 공통 설정 등)
- cold start 시 들어오는 첫 트래픽이 한 번에 수백~수천 RPS 수준이다(블랙 프라이데이 오픈 같은 상황)
- DB 쿼리가 무거워서 동시에 100개가 들어오면 DB가 못 버틴다
- 배포·재시작이 잦아서 cold start가 일상적이다

워밍업이 필요 없는 경우는 이렇다.

- 트래픽이 점진적으로 차오른다(평일 아침 유입 곡선이 부드러움)
- 캐시 대상 키가 너무 다양해서 어떤 걸 미리 채워야 할지 명확하지 않다(사용자별 데이터)
- DB가 cold start 트래픽을 충분히 받아낼 수 있다
- 사용 안 하는 데이터를 미리 채우면 메모리만 낭비된다

워밍업 코드는 간단하다.

```typescript
async function warmupCache() {
  logger.info('cache warmup started');

  const categories = await categoryRepository.findAll();
  await cache.set('category:all', categories, 3600);

  const topProducts = await productRepository.findTopByViewCount(100);
  await Promise.all(
    topProducts.map(p => cache.set(`product:${p.id}`, p, 600))
  );

  const config = await configRepository.findActive();
  await cache.set('config:active', config, 1800);

  logger.info('cache warmup completed');
}

app.listen(PORT, async () => {
  await warmupCache();
});
```

워밍업이 길어지면 헬스체크 통과가 늦어져서 배포 도중 트래픽이 끊긴다. 1초 안에 끝나야 하는 워밍업과 1분 걸려도 괜찮은 워밍업은 처리 방식이 다르다. 길게 걸리는 워밍업은 백그라운드 비동기로 돌리고 헬스체크는 먼저 통과시킨다. 단, 그 사이에 들어오는 요청은 miss를 받아들여야 한다. 어느 쪽이든 트레이드오프가 있다.

워밍업이 모든 인스턴스에서 동시에 실행되면 시작 시점에 DB가 한 번 두들겨 맞는다. 인스턴스가 10대인데 다 같이 cold start 워밍업을 돌리면 DB로 같은 쿼리가 10배로 몰린다. 인스턴스마다 시작 시점을 약간 어긋나게 하거나, 한 인스턴스가 워밍업하고 Redis에 채우면 다른 인스턴스는 Redis만 보게 한다.
