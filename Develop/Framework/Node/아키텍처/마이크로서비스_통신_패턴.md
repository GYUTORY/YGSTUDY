# 마이크로서비스 통신 패턴

## 개요

마이크로서비스로 쪼개고 나면 가장 먼저 부딪히는 문제가 통신이다. 모놀리식에서는 함수 호출 한 번이면 끝나던 게 네트워크를 타기 시작하는 순간 타임아웃, 재시도, 부분 실패, 순서 뒤바뀜, 중복 메시지 같은 골치 아픈 것들이 쏟아진다.

통신 방식은 크게 두 축으로 나눈다. 호출자가 응답을 기다리느냐(동기), 아니면 던져놓고 자기 갈 길 가느냐(비동기). 그리고 호출 시점에 상대를 직접 지목하느냐(point-to-point), 아니면 누군가 듣고 있을 거라는 약속만 하느냐(pub/sub)도 또 다른 축이다.

```
동기(blocking)         비동기(non-blocking)
────────────────       ─────────────────────
REST                   메시지 큐 (RabbitMQ)
gRPC unary             이벤트 스트림 (Kafka, NATS)
GraphQL                Pub/Sub (Redis, NATS Core)

point-to-point         pub/sub
────────────────       ─────────────────────
REST 호출              Kafka topic 구독
gRPC 호출              NATS JetStream consumer
RabbitMQ direct queue  RabbitMQ topic exchange
```

이 분류를 머리에 박아두지 않으면 "이거 RabbitMQ로 해야 하나 Kafka로 해야 하나" 같은 질문에 매번 막힌다. 일을 하다 보면 한 시스템 안에 세 가지가 다 섞여 있게 된다. 결제는 동기 gRPC, 주문 이벤트는 Kafka, 사용자 알림은 RabbitMQ 같은 식으로.

이 문서에서는 각 패턴을 실제 사용할 때 부딪히는 구체적인 문제와 그걸 해결하는 코드를 정리한다. 단순히 "Kafka는 빠르다, RabbitMQ는 라우팅이 강하다" 같은 카탈로그식 비교가 아니라, 새벽 3시에 알림 받고 디버깅했던 종류의 이야기들이다.

## 동기 통신 패턴

### REST를 쓸 때 흔히 놓치는 것들

REST는 익숙하기 때문에 별생각 없이 쓰게 된다. 그런데 마이크로서비스 환경에서 REST가 망가지는 패턴은 거의 정해져 있다. 타임아웃을 너무 길게 잡거나, 너무 짧게 잡거나, 재시도를 무한정 하거나, idempotency를 고려하지 않고 POST를 재시도하거나.

기본 클라이언트 구성부터 보자.

```typescript
// http/client.ts
import axios, { AxiosInstance, AxiosError } from 'axios';
import axiosRetry from 'axios-retry';
import { trace, context, propagation } from '@opentelemetry/api';

interface ClientOptions {
  baseURL: string;
  timeout?: number;
  retries?: number;
  retryableStatuses?: number[];
}

export function createHttpClient(opts: ClientOptions): AxiosInstance {
  const client = axios.create({
    baseURL: opts.baseURL,
    timeout: opts.timeout ?? 3000,
    headers: { 'Content-Type': 'application/json' },
    // keep-alive 를 켜야 TCP handshake 비용이 사라진다.
    httpAgent: new (require('http').Agent)({ keepAlive: true, maxSockets: 64 }),
    httpsAgent: new (require('https').Agent)({ keepAlive: true, maxSockets: 64 }),
  });

  axiosRetry(client, {
    retries: opts.retries ?? 2,
    retryDelay: (retryCount) => {
      // 지수 백오프 + 지터. 같은 시점에 모든 인스턴스가 동시에 재시도하면 부하가 한꺼번에 몰린다.
      const exp = Math.min(1000 * 2 ** retryCount, 8000);
      const jitter = Math.random() * 200;
      return exp + jitter;
    },
    retryCondition: (error: AxiosError) => {
      const status = error.response?.status;
      // 5xx, 408, 429 만 재시도. 4xx 는 재시도해봐야 그대로 실패한다.
      if (!status) return axiosRetry.isNetworkError(error);
      return [408, 429, 500, 502, 503, 504].includes(status);
    },
  });

  // 요청 ID, traceparent 전파
  client.interceptors.request.use((config) => {
    const span = trace.getActiveSpan();
    if (span) {
      const headers: Record<string, string> = {};
      propagation.inject(context.active(), headers);
      Object.assign(config.headers, headers);
    }
    return config;
  });

  return client;
}
```

여기서 짚을 부분이 몇 개 있다.

첫째, `keepAlive`. Node.js axios의 기본 agent는 keep-alive가 꺼져 있다. 마이크로서비스 환경에서는 같은 서비스로 초당 수백 건씩 호출하는데, 매번 TCP 핸드셰이크와 TLS 협상을 다시 한다고 생각해보면 답이 없다. p99 latency가 30ms에서 8ms로 떨어지는 걸 직접 봤다.

둘째, 재시도 조건. 4xx를 재시도하면 안 된다. 400은 클라이언트가 잘못 보낸 거니까 다시 보내도 똑같이 실패한다. 401, 403도 마찬가지. 408(Request Timeout), 429(Too Many Requests), 5xx 만 재시도한다.

셋째, 지터(jitter). 한 인스턴스가 재시도할 때 다른 인스턴스도 같은 타이밍에 재시도하면 thundering herd가 발생한다. 하위 서비스가 잠깐 흔들리던 걸 완전히 죽일 수도 있다.

### 타임아웃 계층화

타임아웃은 서비스가 깊어질수록 까다로워진다. A → B → C → D 호출 체인에서 A가 5초 타임아웃을 걸어도, B가 6초 걸리고 있으면 의미가 없다. 또 D에서 4초 걸려서 응답을 줘도 B 입장에서는 이미 A가 끊은 후라 응답 폐기다.

원칙은 단순하다. **하위 호출의 타임아웃 합계가 상위 타임아웃보다 작아야 한다.** 그리고 가능하면 deadline을 헤더로 전파해서 하위가 자기 작업이 무의미해질 것을 알면 미리 포기하게 한다.

```typescript
// http/deadline.ts
export interface DeadlineContext {
  deadlineMs: number; // epoch ms
}

export function withDeadline<T>(
  ctx: DeadlineContext,
  fn: (remainingMs: number) => Promise<T>,
): Promise<T> {
  const remaining = ctx.deadlineMs - Date.now();
  if (remaining <= 0) {
    return Promise.reject(new Error('deadline_exceeded'));
  }
  return Promise.race([
    fn(remaining),
    new Promise<T>((_, reject) =>
      setTimeout(() => reject(new Error('deadline_exceeded')), remaining),
    ),
  ]);
}

// 사용
async function fetchOrderDetails(orderId: string, ctx: DeadlineContext) {
  const order = await withDeadline(ctx, (ms) =>
    orderClient.get(`/orders/${orderId}`, { timeout: ms }),
  );

  // 남은 시간을 다음 호출에도 전파
  const items = await withDeadline(ctx, (ms) =>
    catalogClient.post('/items/batch', { ids: order.itemIds }, { timeout: ms }),
  );

  return { order, items };
}

// 컨트롤러에서 진입점에 deadline 설정
app.get('/orders/:id/details', async (req, res) => {
  const ctx: DeadlineContext = { deadlineMs: Date.now() + 3000 }; // 3초
  const result = await fetchOrderDetails(req.params.id, ctx);
  res.json(result);
});
```

이 패턴을 모든 호출에 일관되게 적용하면, 한 호출이 폭주해도 전체 cascade가 멈추지 않는다. 4초 일이 1초 안에 끝나야 하는 진입점에서 호출되면, 1초 후 deadline에 걸려서 일찍 포기한다. gRPC에서는 이게 빌트인이고(`call.deadline`), HTTP에서는 직접 만들어 써야 한다.

### 재시도 폭발(retry storm) 방지

재시도는 잘못 쓰면 트래픽을 몇 배로 부풀린다. A → B → C 체인에서 각 단계가 3번 재시도한다고 하면, C가 한 번 흔들렸을 때 받는 호출량은 1 × 3 × 3 = 9배다. 정작 C는 부하 때문에 더 망가지고, B와 A도 같이 죽는다.

방어책은 세 가지다.

```typescript
// http/budget.ts — 재시도 예산(budget)
class RetryBudget {
  private successes = 0;
  private retries = 0;
  private windowStart = Date.now();
  private readonly windowMs = 10_000;
  private readonly ratio: number;

  constructor(ratio = 0.1) {
    // 성공 호출의 10% 만큼만 재시도 허용
    this.ratio = ratio;
  }

  canRetry(): boolean {
    this.rotate();
    const allowed = Math.floor(this.successes * this.ratio) + 5; // 최소 5회
    return this.retries < allowed;
  }

  recordSuccess() {
    this.rotate();
    this.successes++;
  }

  recordRetry() {
    this.rotate();
    this.retries++;
  }

  private rotate() {
    if (Date.now() - this.windowStart > this.windowMs) {
      this.successes = 0;
      this.retries = 0;
      this.windowStart = Date.now();
    }
  }
}

const budget = new RetryBudget(0.1);

// axios-retry retryCondition 안에서 budget 체크
axiosRetry(client, {
  retries: 3,
  retryCondition: (error) => {
    if (!isRetryableError(error)) return false;
    if (!budget.canRetry()) {
      logger.warn('retry budget exhausted, giving up');
      return false;
    }
    budget.recordRetry();
    return true;
  },
});
```

`RetryBudget`은 Envoy/Istio가 쓰는 방식이다. 평소 호출 성공률이 정상이면 재시도가 거의 안 일어나니까 budget이 차 있고, 다운스트림이 흔들리기 시작하면 재시도가 급증해서 금세 budget이 소진된다. 그러면 더 이상 재시도하지 않고 원본 에러를 그대로 올린다. 결과적으로 다운스트림이 받는 호출량은 평소의 1.1배 정도로 묶인다.

두 번째 방어책은 **재시도 상한 깊이 제한**이다. 호출 헤더에 `x-retry-depth` 같은 걸 넣어서, 이 호출이 이미 N번 재시도된 결과물이면 더 이상 재시도하지 말라고 표시한다. 그러지 않으면 A가 재시도 → B도 재시도 → C도 재시도 식으로 곱셈이 발생한다.

세 번째는 **non-idempotent 호출은 재시도하지 않는 것**이다. POST를 재시도하면 결제가 두 번 일어날 수 있다. idempotency key를 헤더로 보내서 서버 쪽에서 중복 처리하게 만들거나(뒤에 다룬다), 아니면 아예 재시도를 끈다.

### gRPC의 4가지 호출 방식

gRPC는 단순히 "REST보다 빠른 RPC" 정도로 보면 절반만 쓰는 거다. 진가는 4가지 호출 방식에 있다.

먼저 proto 정의부터.

```protobuf
// proto/file.proto
syntax = "proto3";

package file;

service FileService {
  // 1. unary: 단일 요청, 단일 응답
  rpc GetMetadata(FileRequest) returns (FileMetadata);

  // 2. server streaming: 단일 요청, 여러 응답
  rpc DownloadChunks(FileRequest) returns (stream FileChunk);

  // 3. client streaming: 여러 요청, 단일 응답
  rpc UploadChunks(stream FileChunk) returns (UploadResult);

  // 4. bidirectional streaming: 여러 요청, 여러 응답
  rpc Sync(stream SyncEvent) returns (stream SyncEvent);
}

message FileRequest {
  string file_id = 1;
}

message FileMetadata {
  string file_id = 1;
  int64 size = 2;
  string content_type = 3;
}

message FileChunk {
  string file_id = 1;
  int32 sequence = 2;
  bytes data = 3;
}

message UploadResult {
  string file_id = 1;
  string sha256 = 2;
}

message SyncEvent {
  string id = 1;
  string type = 2;
  bytes payload = 3;
}
```

#### Unary

가장 많이 쓰는 형태. 평범한 RPC 호출이다.

```typescript
// grpc/server-unary.ts
import * as grpc from '@grpc/grpc-js';
import * as protoLoader from '@grpc/proto-loader';

const packageDef = protoLoader.loadSync('proto/file.proto', {
  keepCase: true,
  longs: String,
  enums: String,
  defaults: true,
  oneofs: true,
});
const proto = grpc.loadPackageDefinition(packageDef) as any;

const server = new grpc.Server();

server.addService(proto.file.FileService.service, {
  GetMetadata: async (
    call: grpc.ServerUnaryCall<{ file_id: string }, FileMetadata>,
    callback: grpc.sendUnaryData<FileMetadata>,
  ) => {
    try {
      const meta = await fileRepository.findById(call.request.file_id);
      if (!meta) {
        callback({
          code: grpc.status.NOT_FOUND,
          message: `file ${call.request.file_id} not found`,
        });
        return;
      }
      callback(null, meta);
    } catch (err: any) {
      callback({ code: grpc.status.INTERNAL, message: err.message });
    }
  },
});
```

```typescript
// grpc/client-unary.ts
function getMetadata(fileId: string, deadlineMs: number): Promise<FileMetadata> {
  return new Promise((resolve, reject) => {
    const deadline = new Date(Date.now() + deadlineMs);
    client.GetMetadata(
      { file_id: fileId },
      { deadline },
      (err: grpc.ServiceError | null, response: FileMetadata) => {
        if (err) reject(err);
        else resolve(response);
      },
    );
  });
}
```

deadline은 ms가 아니라 `Date` 객체로 넘긴다. gRPC가 알아서 hop-by-hop으로 전파한다(자세한 건 뒤에서).

#### Server streaming

서버가 응답을 여러 번 나눠서 보낸다. 큰 파일을 chunk 단위로 다운로드하거나, 검색 결과를 점진적으로 스트림할 때 쓴다.

```typescript
// grpc/server-server-streaming.ts
DownloadChunks: async (
  call: grpc.ServerWritableStream<{ file_id: string }, FileChunk>,
) => {
  const stream = await fileStorage.openReadStream(call.request.file_id);
  let sequence = 0;

  for await (const chunk of stream) {
    // back-pressure: write가 false 를 반환하면 drain 까지 기다린다.
    const ok = call.write({
      file_id: call.request.file_id,
      sequence: sequence++,
      data: chunk,
    });
    if (!ok) {
      await new Promise<void>((resolve) => call.once('drain', resolve));
    }

    // 클라이언트가 끊겼는지 확인
    if (call.cancelled) {
      stream.destroy();
      return;
    }
  }

  call.end();
},
```

```typescript
// grpc/client-server-streaming.ts
function downloadFile(fileId: string): Promise<Buffer> {
  return new Promise((resolve, reject) => {
    const call = client.DownloadChunks({ file_id: fileId });
    const chunks: Buffer[] = [];

    call.on('data', (chunk: FileChunk) => {
      chunks[chunk.sequence] = chunk.data;
    });
    call.on('end', () => resolve(Buffer.concat(chunks)));
    call.on('error', reject);
  });
}
```

`call.cancelled` 체크를 잊으면 클라이언트가 끊긴 뒤에도 서버가 계속 데이터를 읽어서 메모리만 잡아먹는다. 큰 파일에서 이거 누락하면 클라이언트 disconnect가 잦은 환경에서 메모리 누수처럼 보인다.

#### Client streaming

클라이언트가 여러 메시지를 보내고, 서버는 다 받은 뒤 한 번 응답한다. 청크 업로드, 배치 ingestion에 쓴다.

```typescript
// grpc/server-client-streaming.ts
UploadChunks: async (
  call: grpc.ServerReadableStream<FileChunk, UploadResult>,
  callback: grpc.sendUnaryData<UploadResult>,
) => {
  const hasher = crypto.createHash('sha256');
  let fileId: string | null = null;
  let totalBytes = 0;

  call.on('data', (chunk: FileChunk) => {
    fileId ??= chunk.file_id;
    hasher.update(chunk.data);
    totalBytes += chunk.data.length;
    fileStorage.append(fileId, chunk.data);
  });

  call.on('end', () => {
    if (!fileId) {
      callback({ code: grpc.status.INVALID_ARGUMENT, message: 'no chunks received' });
      return;
    }
    callback(null, { file_id: fileId, sha256: hasher.digest('hex') });
  });

  call.on('error', (err) => {
    if (fileId) fileStorage.cleanup(fileId);
    callback({ code: grpc.status.INTERNAL, message: err.message });
  });
},
```

```typescript
// grpc/client-client-streaming.ts
async function uploadFile(fileId: string, source: AsyncIterable<Buffer>): Promise<UploadResult> {
  return new Promise((resolve, reject) => {
    const call = client.UploadChunks((err, result) => {
      if (err) reject(err);
      else resolve(result);
    });

    let sequence = 0;
    (async () => {
      for await (const chunk of source) {
        const ok = call.write({ file_id: fileId, sequence: sequence++, data: chunk });
        if (!ok) {
          await new Promise<void>((resolve) => call.once('drain', resolve));
        }
      }
      call.end();
    })().catch((err) => call.cancel());
  });
}
```

#### Bidirectional streaming

양쪽이 동시에 메시지를 주고받는다. 채팅, 실시간 동기화, 협업 편집 같은 곳에 적합하다.

```typescript
// grpc/server-bidi.ts
Sync: (call: grpc.ServerDuplexStream<SyncEvent, SyncEvent>) => {
  const userId = call.metadata.get('user-id')[0]?.toString();

  // 클라이언트가 보내는 이벤트 처리
  call.on('data', async (event: SyncEvent) => {
    await eventBus.publish(`user.${userId}.event`, event);
  });

  call.on('end', () => call.end());
  call.on('error', (err) => logger.error('sync stream error', err));

  // 서버가 푸시하는 이벤트 구독
  const subscription = eventBus.subscribe(`user.${userId}.broadcast`, (event) => {
    const ok = call.write(event);
    if (!ok) call.once('drain', () => {});
  });

  call.on('cancelled', () => subscription.unsubscribe());
},
```

bidirectional은 코드가 복잡해진다. 서버 push와 클라이언트 send가 비동기로 섞이니까. 종료 처리(end, error, cancelled)를 빠뜨리면 dangling subscription이 쌓이기 좋다.

### gRPC interceptor

interceptor는 모든 호출을 가로채서 공통 처리(인증, 로깅, 메트릭, 트레이싱)를 하는 메커니즘이다. Express의 미들웨어와 비슷하다.

```typescript
// grpc/server-interceptor.ts
import * as grpc from '@grpc/grpc-js';

const authInterceptor: grpc.ServerInterceptor = (methodDescriptor, call) => {
  return new grpc.ServerInterceptingCall(call, {
    start: (next) => {
      const token = call.metadata.get('authorization')[0]?.toString();
      if (!token) {
        call.sendStatus({ code: grpc.status.UNAUTHENTICATED, details: 'missing token' });
        return;
      }
      try {
        const user = verifyJwt(token.replace('Bearer ', ''));
        // 다음 interceptor 와 핸들러로 user 전달
        (call as any).user = user;
        next(call.metadata);
      } catch {
        call.sendStatus({ code: grpc.status.UNAUTHENTICATED, details: 'invalid token' });
      }
    },
  });
};

const loggingInterceptor: grpc.ServerInterceptor = (methodDescriptor, call) => {
  const start = process.hrtime.bigint();
  const method = methodDescriptor.path;

  return new grpc.ServerInterceptingCall(call, {
    sendStatus: (status, next) => {
      const durationMs = Number(process.hrtime.bigint() - start) / 1e6;
      logger.info('grpc call', {
        method,
        code: status.code,
        durationMs,
        userId: (call as any).user?.id,
      });
      grpcRequestDuration.observe({ method, code: String(status.code) }, durationMs);
      next(status);
    },
  });
};

const server = new grpc.Server({
  interceptors: [authInterceptor, loggingInterceptor],
});
```

클라이언트 쪽 interceptor도 비슷하다.

```typescript
// grpc/client-interceptor.ts
const authClientInterceptor: grpc.Interceptor = (options, nextCall) => {
```

### gRPC deadline propagation

gRPC의 가장 큰 장점 중 하나가 deadline 자동 전파다. 클라이언트가 `deadline: now + 5s` 로 호출하면, 서버 핸들러 안의 `call.getDeadline()` 으로 그 시각을 알 수 있다. 서버가 다른 gRPC를 호출할 때 그 deadline을 그대로 넘기면, 시간이 흐를수록 남은 시간이 줄어든 deadline이 전파된다.

```typescript
// grpc/deadline-propagation.ts
GetUserOrderSummary: async (call, callback) => {
  const deadline = call.getDeadline();

  // 동일한 deadline 으로 다른 서비스 호출
  const ordersPromise = new Promise<Order[]>((resolve, reject) => {
    orderClient.ListByUser(
      { user_id: call.request.user_id },
      { deadline },
      (err, response) => err ? reject(err) : resolve(response.orders),
    );
  });

  const userPromise = new Promise<User>((resolve, reject) => {
    userClient.GetUser(
      { id: call.request.user_id },
      { deadline },
      (err, response) => err ? reject(err) : resolve(response),
    );
  });

  try {
    const [orders, user] = await Promise.all([ordersPromise, userPromise]);
    callback(null, { user, orders });
  } catch (err: any) {
    if (err.code === grpc.status.DEADLINE_EXCEEDED) {
      // 이 호출은 이미 시간 초과. 더 진행해도 의미 없음.
      callback(err);
    } else {
      callback({ code: grpc.status.INTERNAL, message: err.message });
    }
  }
},
```

이게 왜 중요하냐. 진입점에서 5초 deadline을 설정했을 때, 4.5초가 경과한 시점에 하위 서비스를 호출하면 0.5초만 기다린다. 그 안에 응답이 안 오면 빠르게 포기한다. 진입점이 어차피 응답을 못 줄 거라면 하위 서비스에게 일을 시키지 않는 게 맞다. 하위 서비스도 자기 호출이 무의미하다는 걸 알면 일찍 종료할 수 있다.

REST에서 이걸 하려면 직접 헤더(`X-Deadline-Ms` 같은)를 만들고 모든 서비스가 합의해서 처리해야 한다. gRPC는 빌트인이다.

### Circuit Breaker

동기 호출에서 하위 서비스가 죽으면 호출자도 같이 죽기 시작한다. 5초 타임아웃을 걸어놨는데 하위가 계속 5초 걸려서 응답하면, 호출자의 모든 worker가 그 호출에 묶여서 다른 요청을 못 받는다. cascade failure다.

```typescript
// http/circuit-breaker.ts
import CircuitBreaker from 'opossum';

interface BreakerOptions {
  timeout: number;
  errorThresholdPercentage: number;
  resetTimeout: number;
  volumeThreshold: number;
  rollingCountTimeout: number;
  rollingCountBuckets: number;
}

const breaker = new CircuitBreaker(getUserById, {
  timeout: 3000,
  errorThresholdPercentage: 50,
  resetTimeout: 10000,
  // 최소 호출 수: 너무 적은 호출로 통계 잡으면 false-open 이 잦다.
  volumeThreshold: 20,
  rollingCountTimeout: 10000,
  rollingCountBuckets: 10,
});

breaker.fallback((id: string) => {
  // 캐시된 마지막 데이터 반환
  return cache.get(`user:${id}`) ?? { id, name: 'Unknown', stale: true };
});

breaker.on('open', () => {
  logger.warn('breaker opened: user-service');
  metrics.increment('breaker.opened', { service: 'user-service' });
});
breaker.on('halfOpen', () => logger.info('breaker half-open: user-service'));
breaker.on('close', () => logger.info('breaker closed: user-service'));

// 사용
async function getUser(id: string) {
  return breaker.fire(id);
}
```

`volumeThreshold`를 빼먹으면 평소 호출량이 적은 시점에 한두 번 실패한 걸로 breaker가 열려버린다. 그러면 정상 시점에 가끔 fallback이 발동해서 사용자가 stale 데이터를 보는 일이 생긴다. 보통 10~50 정도로 잡는다.

`fallback`은 빠르게 끝나는 함수여야 한다. 거기서 또 다른 서비스를 호출하면 그 서비스가 죽었을 때 breaker가 의미가 없어진다.

## 비동기 통신 패턴

비동기 통신은 호출자가 응답을 기다리지 않는다. 메시지를 보내고 자기 갈 길을 가고, 메시지는 broker에 쌓이고, consumer가 자기 페이스로 처리한다. 가용성이 높고 결합도가 낮은 대신, 처리 결과가 언제 끝났는지 알기 어렵고 디버깅이 까다롭다.

선택지가 너무 많아서 헷갈리는데, 큰 그림은 이렇다.

```
RabbitMQ          → 라우팅이 복잡한 작업 큐, RPC 응답 매칭
Kafka             → 대량 이벤트 스트림, 재처리, 로그성 데이터
NATS Core         → 매우 가벼운 pub/sub (at-most-once)
NATS JetStream    → Kafka 와 RabbitMQ 사이의 절충
Redis Streams     → 간단한 작업 큐 정도
SQS / Pub/Sub     → 클라우드 매니지드, 운영 부담 회피
```

### RabbitMQ: exchange, binding, DLQ

RabbitMQ의 강점은 라우팅이다. exchange에 메시지를 보내면 binding 규칙에 따라 여러 queue로 fan-out 되거나 특정 queue로 매칭된다. exchange 종류가 4개 있다.

- `direct`: routing key가 정확히 일치하는 queue로
- `topic`: routing key를 패턴 매칭(`order.*.created`)
- `fanout`: 모든 binding 된 queue로 broadcast
- `headers`: 헤더 매칭

```typescript
// rabbitmq/setup.ts
import amqp from 'amqplib';

const conn = await amqp.connect('amqp://rabbitmq:5672');
const channel = await conn.createChannel();

// 1. 메인 exchange 와 queue, DLQ 까지 한 번에 설정
await channel.assertExchange('orders', 'topic', { durable: true });
await channel.assertExchange('orders.dlx', 'topic', { durable: true });

await channel.assertQueue('orders.payment-service', {
  durable: true,
  // 메시지 거부/만료 시 DLX 로 보냄
  arguments: {
    'x-dead-letter-exchange': 'orders.dlx',
    'x-dead-letter-routing-key': 'orders.payment-service.dead',
    'x-message-ttl': 60_000, // 60초 동안 처리 안 되면 dead
    'x-max-length': 100_000,
  },
});

// 2. binding: orders 에서 orders.created.* 패턴을 payment-service queue 로
await channel.bindQueue('orders.payment-service', 'orders', 'orders.created.*');

// 3. DLQ
await channel.assertQueue('orders.payment-service.dead', { durable: true });
await channel.bindQueue(
  'orders.payment-service.dead',
  'orders.dlx',
  'orders.payment-service.dead',
);
```

핵심은 DLX(dead letter exchange) 패턴이다. 메시지가 nack 되거나 TTL 초과로 죽으면 DLX로 라우팅된다. DLQ에 쌓인 메시지는 사람이 들여다보고 수동으로 재처리하거나, 자동 재시도 worker가 일정 간격으로 다시 메인 큐로 넣는다.

```typescript
// rabbitmq/consumer.ts
await channel.prefetch(10); // 한 번에 처리할 메시지 수 제한

await channel.consume('orders.payment-service', async (msg) => {
  if (!msg) return;
  const event = JSON.parse(msg.content.toString());

  try {
    await processPayment(event);
    channel.ack(msg);
  } catch (err: any) {
    const retryCount = (msg.properties.headers?.['x-retry-count'] as number) ?? 0;
    if (retryCount >= 3) {
      // 3번 재시도 후에도 실패하면 DLQ 로
      channel.nack(msg, false, false);
      logger.error('moved to DLQ', { event, error: err.message });
    } else {
      // 헤더 갱신 후 재발행
      channel.publish('orders', msg.fields.routingKey, msg.content, {
        ...msg.properties,
        headers: { ...msg.properties.headers, 'x-retry-count': retryCount + 1 },
      });
      channel.ack(msg);
    }
  }
});
```

`prefetch`를 잊으면 모든 메시지가 한 consumer로 쏠린다. consumer가 여러 인스턴스 떠 있어도 첫 인스턴스가 큐의 모든 메시지를 가져가서 한 명만 일하고 나머지는 놀게 된다. 보통 5~50 사이로 잡는다.

`nack(msg, false, false)`의 마지막 인자가 `requeue`다. `false`로 하면 큐에 다시 넣지 않고 DLX로 보낸다. `true`로 하면 다시 큐에 들어가는데, 이러면 같은 메시지가 같은 에러로 무한 루프 돌 수 있다. 그래서 위 코드처럼 retry count를 헤더로 추적해서 일정 횟수 후에는 DLQ로 보낸다.

### Kafka vs NATS JetStream 선택 기준

Kafka는 일단 무거운 시스템이다. ZooKeeper(또는 KRaft), 다중 broker, partition 관리, consumer group lag 모니터링까지 운영이 만만치 않다. 대신 처리량과 보존 정책에서는 압도적이다. 1MB짜리 메시지를 초당 수십만 건 처리하고, 7일짜리 retention으로 모든 이벤트를 다시 재생할 수 있다.

NATS JetStream은 그 중간이다. Kafka의 persistence/replay 기능을 갖되, RabbitMQ 수준의 운영 단순함을 노린다. 단일 binary로 클러스터를 띄우고, 메시지 보존, consumer 재시작, exactly-once delivery 같은 걸 지원한다. 내 경험으로는 트래픽이 초당 수천 건 미만이고 24시간 정도의 retention이면 충분한 경우 JetStream이 운영 부담이 훨씬 낮다.

```
                Kafka              JetStream         RabbitMQ
─────────────────────────────────────────────────────────────
처리량          매우 높음          중간              중간
보존            최대 무제한        최대 ~수일        없음(큐 휘발)
운영 복잡도     높음               낮음              낮음
라우팅          partition key      subject           exchange
재처리          offset reset       seq replay        불가
순서 보장       partition 단위     stream 단위       단일 consumer 단위
exactly-once    transactional     ack-based         수동
```

선택 기준은 단순화하면 이렇다.

- 이벤트 소싱, 로그성 데이터, 재처리가 필요한 경우 → Kafka
- 작업 큐, 라우팅이 중요한 경우 → RabbitMQ
- 트래픽이 적당하고 둘 다 필요한 경우 → JetStream
- 단순 fire-and-forget pub/sub → NATS Core

신규 서비스에서 무조건 Kafka를 까는 건 추천 안 한다. 운영 부담을 감당할 팀이 있는지 확인하고, 트래픽 추정치를 먼저 보고 결정한다. 초당 수백 건 정도는 RabbitMQ나 JetStream으로도 차고 넘친다.

### Kafka: 파티션 키와 순서 보장

Kafka의 순서 보장은 **파티션 단위**다. 한 파티션 안의 메시지는 producer가 보낸 순서대로 consumer가 읽는다. 파티션이 다르면 순서가 뒤섞일 수 있다.

따라서 같은 entity에 대한 이벤트를 순서대로 처리하려면 같은 키를 줘서 같은 파티션에 들어가게 해야 한다.

```typescript
// kafka/producer-with-partition-key.ts
import { Kafka, Partitioners } from 'kafkajs';

const kafka = new Kafka({
  clientId: 'order-service',
  brokers: process.env.KAFKA_BROKERS!.split(','),
});

const producer = kafka.producer({
  // legacy partitioner 는 hash 가 일관되지 않다. JavaCompatible 권장.
  createPartitioner: Partitioners.JavaCompatiblePartitioner,
  // 멱등성 보장: 같은 메시지가 broker 단위에서 중복되지 않는다.
  idempotent: true,
  // 트랜잭션을 쓰지 않을 때도 max in-flight=5 에서 idempotent 가능.
  maxInFlightRequests: 5,
});

await producer.connect();

// 같은 orderId 면 같은 파티션 → 순서 보장
async function publishOrderEvent(orderId: string, event: any) {
  await producer.send({
    topic: 'orders',
    messages: [{
      key: orderId, // 핵심: key 로 partition 결정
      value: JSON.stringify(event),
      headers: {
        'event-type': event.type,
        'event-id': event.id,
      },
    }],
  });
}
```

`key`를 빈 문자열이나 `null`로 두면 round-robin 으로 분산된다. 그러면 같은 주문에 대한 `order.created` 와 `order.updated` 가 다른 파티션에 들어갈 수 있고, consumer는 update를 create 보다 먼저 받게 될 수 있다. 그러면 update가 "그런 주문 없음"으로 실패한다.

idempotent producer를 켜면 broker 입장에서 중복이 제거된다. producer가 메시지를 보냈는데 ack가 안 와서 재시도했을 때, 같은 메시지가 두 번 저장되는 걸 막는다. 추가 비용은 거의 없으니 항상 켜두는 편이다.

### Kafka: consumer group 리밸런싱과 중복

consumer group은 partition을 나눠서 처리한다. 6개 partition에 consumer가 3개면 각자 2개씩, 6개면 1:1, 12개면 한 명이 놀게 된다. 그런데 consumer가 죽거나 새로 추가되면 **리밸런싱**이 일어나서 partition 할당이 바뀐다.

문제는 리밸런싱 도중에 중복 처리가 잦다는 것이다. consumer A가 partition 0의 offset 100까지 처리하고 commit 못 하고 죽으면, consumer B가 partition 0을 받았을 때 offset 100부터 다시 읽는다. 100~사망 시점까지의 메시지가 두 번 처리된다.

이걸 막을 수 없다. Kafka는 at-least-once 가 기본이다. 그래서 **consumer 쪽에서 멱등성을 보장**해야 한다. 그게 멱등성 키 패턴이다.

```typescript
// kafka/consumer-idempotent.ts
const consumer = kafka.consumer({
  groupId: 'payment-service',
  // 리밸런싱 전 cleanup 시간 확보
  rebalanceTimeout: 60_000,
  sessionTimeout: 30_000,
  heartbeatInterval: 3_000,
});

await consumer.connect();
await consumer.subscribe({ topics: ['orders'], fromBeginning: false });

await consumer.run({
  // partition 별로 동시에 처리. 한 partition 안에서는 순차.
  partitionsConsumedConcurrently: 4,
  eachMessage: async ({ topic, partition, message, heartbeat }) => {
    const event = JSON.parse(message.value!.toString());
    const eventId = message.headers?.['event-id']?.toString();

    if (!eventId) {
      logger.warn('event without id, skipping', { topic, partition });
      return;
    }

    // 멱등성 체크: 이미 처리한 이벤트면 skip
    const alreadyProcessed = await processedEventsRepo.exists(eventId);
    if (alreadyProcessed) {
      logger.debug('event already processed', { eventId });
      return;
    }

    await db.transaction(async (tx) => {
      await processPayment(event, tx);
      // 같은 트랜잭션에서 처리 마크
      await processedEventsRepo.insert({ eventId, processedAt: new Date() }, tx);
    });

    // 긴 처리 중 heartbeat 보내서 session timeout 방지
    await heartbeat();
  },
});
```

`processed_events` 테이블은 (event_id PRIMARY KEY, processed_at) 정도로 만들고, 일정 기간 후 cleanup job 으로 정리한다. event_id가 PK라서 같은 이벤트를 두 번 insert 하면 실패하고, 트랜잭션이 롤백되면서 결과적으로 멱등 처리가 된다.

`partitionsConsumedConcurrently`는 한 consumer 인스턴스가 동시에 처리할 partition 수다. 너무 높이면 메모리 과부하, 너무 낮으면 처리 속도가 partition 수만큼만 나온다. partition 4개에 1로 두면 4개를 순차로 도는데, partition 간 이벤트는 독립이니 동시에 돌리는 게 낫다.

heartbeat 호출도 중요하다. 한 메시지 처리가 길어지면 broker가 consumer를 dead로 판단해서 리밸런싱을 트리거한다. session_timeout 보다 짧은 주기로 heartbeat를 호출해야 한다. kafkajs는 자동으로 보내주지만, 진짜 긴 작업(수십 초)에서는 명시적으로 호출하는 게 안전하다.

### eachBatch와 backpressure

`eachMessage`는 한 번에 하나씩 처리한다. 메시지가 작고 빠르게 처리되면 충분한데, DB write가 들어가는 경우 batch insert가 훨씬 빠르다.

```typescript
// kafka/consumer-batch.ts
await consumer.run({
  eachBatchAutoResolve: false,
  eachBatch: async ({
    batch,
    resolveOffset,
    heartbeat,
    isRunning,
    isStale,
    commitOffsetsIfNecessary,
  }) => {
    const events: Event[] = [];

    for (const message of batch.messages) {
      if (!isRunning() || isStale()) break;
      events.push(JSON.parse(message.value!.toString()));
    }

    // 100개씩 묶어서 batch insert
    for (let i = 0; i < events.length; i += 100) {
      const chunk = events.slice(i, i + 100);
      await db.batchInsert('payment_events', chunk);

      // 처리한 만큼 offset 마크
      const lastOffset = batch.messages[i + chunk.length - 1].offset;
      resolveOffset(lastOffset);

      // 주기적으로 commit, heartbeat
      await commitOffsetsIfNecessary();
      await heartbeat();
    }
  },
});
```

`eachBatchAutoResolve: false` 가 핵심이다. 기본값은 true 라서 batch 전체가 처리된 걸로 간주하고 commit 해버린다. batch 도중에 실패하면 이미 처리한 부분을 다시 읽어야 하는데, auto resolve가 켜져 있으면 안 읽는다. false 로 두면 `resolveOffset`을 명시적으로 호출한 만큼만 commit 한다.

`isStale()`은 리밸런싱이 일어나서 이 partition이 다른 consumer에게 넘어갔을 때 true를 반환한다. 이때 처리를 멈추지 않으면 같은 메시지가 양쪽에서 처리된다.

### Backpressure 일반론

backpressure 처리를 안 하면 메시지가 쌓이는 속도가 처리 속도보다 빠를 때 메모리가 폭발한다. consumer가 100ms 마다 1000개 메시지를 받는데 한 메시지 처리에 200ms 걸리면, 큐가 무한히 커진다.

방어 방법은 두 가지다.

**1. 동시 처리 한도 제한 (semaphore)**

```typescript
// async/semaphore.ts
class Semaphore {
  private queue: Array<() => void> = [];
  private active = 0;

  constructor(private max: number) {}

  async acquire(): Promise<() => void> {
    if (this.active < this.max) {
      this.active++;
      return () => this.release();
    }
    return new Promise<() => void>((resolve) => {
      this.queue.push(() => {
        this.active++;
        resolve(() => this.release());
      });
    });
  }

  private release() {
    this.active--;
    const next = this.queue.shift();
    if (next) next();
  }
}

const sem = new Semaphore(50);

// 사용
await consumer.run({
  eachMessage: async ({ message }) => {
    const release = await sem.acquire();
    try {
      await processMessage(message);
    } finally {
      release();
    }
  },
});
```

**2. RxJS 기반 스트림 처리**

```typescript
// rxjs/backpressure.ts
import { Subject } from 'rxjs';
import { mergeMap, bufferTime, filter } from 'rxjs/operators';

const source$ = new Subject<KafkaMessage>();

source$
  .pipe(
    bufferTime(1000, null, 100), // 1초 또는 100개 차면 emit
    filter((batch) => batch.length > 0),
    mergeMap(
      async (batch) => {
        await db.batchInsert('events', batch);
      },
      5, // 동시 5개 batch 까지 처리
    ),
  )
  .subscribe({
    error: (err) => logger.error('stream error', err),
  });

await consumer.run({
  eachMessage: async ({ message }) => {
    source$.next(message);
  },
});
```

`mergeMap`의 두 번째 인자가 동시 처리 한도다. 이걸 빼면 모든 batch가 동시 시작되어서 DB connection을 폭주시킨다.

### Request-Response 비동기 패턴

비동기 broker로 동기처럼 응답을 받아야 할 때가 있다. RPC 호출인데 트래픽 분산이나 재시도, queue 기반 처리가 필요한 경우. RabbitMQ의 `reply-to` + `correlation-id` 패턴이 정석이다.

```typescript
// rabbitmq/rpc-server.ts
await channel.consume('rpc.compute', async (msg) => {
  if (!msg) return;
  const request = JSON.parse(msg.content.toString());

  try {
    const result = await compute(request);

    channel.publish(
      '', // default exchange
      msg.properties.replyTo, // 클라이언트가 지정한 응답 큐
      Buffer.from(JSON.stringify(result)),
      {
        correlationId: msg.properties.correlationId,
      },
    );
    channel.ack(msg);
  } catch (err: any) {
    channel.publish(
      '',
      msg.properties.replyTo,
      Buffer.from(JSON.stringify({ error: err.message })),
      { correlationId: msg.properties.correlationId },
    );
    channel.ack(msg);
  }
});
```

```typescript
// rabbitmq/rpc-client.ts
import { randomUUID } from 'crypto';

class RpcClient {
  private replyQueue!: string;
  private pending = new Map<string, {
    resolve: (v: any) => void;
    reject: (e: Error) => void;
    timer: NodeJS.Timeout;
  }>();

  async init(channel: amqp.Channel) {
    const q = await channel.assertQueue('', { exclusive: true, autoDelete: true });
    this.replyQueue = q.queue;

    await channel.consume(this.replyQueue, (msg) => {
      if (!msg) return;
      const corrId = msg.properties.correlationId;
      const pending = this.pending.get(corrId);
      if (!pending) return;

      this.pending.delete(corrId);
      clearTimeout(pending.timer);

      const body = JSON.parse(msg.content.toString());
      if (body.error) pending.reject(new Error(body.error));
      else pending.resolve(body);
    }, { noAck: true });
  }

  async call(channel: amqp.Channel, queue: string, payload: any, timeoutMs = 5000): Promise<any> {
    const correlationId = randomUUID();

    return new Promise((resolve, reject) => {
      const timer = setTimeout(() => {
        this.pending.delete(correlationId);
        reject(new Error(`rpc timeout: ${queue}`));
      }, timeoutMs);

      this.pending.set(correlationId, { resolve, reject, timer });

      channel.publish('', queue, Buffer.from(JSON.stringify(payload)), {
        correlationId,
        replyTo: this.replyQueue,
        expiration: String(timeoutMs), // 메시지 자체에 TTL
      });
    });
  }
}

const rpc = new RpcClient();
await rpc.init(channel);

const result = await rpc.call(channel, 'rpc.compute', { input: 42 });
```

이 패턴의 장점은 producer와 consumer가 분리되어 있어서 consumer를 자유롭게 스케일하고, 한 인스턴스가 죽어도 다른 인스턴스가 처리한다는 점이다. 단점은 latency가 broker hop 두 번 더해져서 동기 HTTP/gRPC 보다 느리다.

언제 쓰냐. 처리 시간이 긴(수백ms~수 초) 작업, 갑작스러운 burst를 흡수해야 하는 작업, 다양한 우선순위로 라우팅이 필요한 작업에 적합하다. p99 가 100ms 이하여야 하는 경우는 이걸로 못 한다.

correlation id 매핑 테이블을 메모리에 두면, 인스턴스가 죽었을 때 진행 중이던 RPC가 모두 timeout 으로 죽는다. 단순한 작업이면 호출자가 재시도하면 되는데, idempotent 하지 않은 작업이면 별도 추적이 필요하다.

## 이벤트 vs 명령 (Event vs Command)

이 둘을 명확하게 구분 안 하고 시작하면 시스템이 꼬인다. 메시지 종류는 둘인데, 사람들이 같은 broker에 같은 식으로 publish 한다.

**Command** (명령): "X 해라" 라는 지시. 한 명의 receiver를 향한다. 처리 결과가 호출자에게 의미가 있다.
- `payment.charge`, `email.send`, `inventory.reserve`
- 보통 RabbitMQ direct queue, gRPC, REST 같은 point-to-point
- 멱등성 키, 응답 매칭 필요

**Event** (이벤트): "X 가 일어났다" 라는 사실. 여러 명이 들을 수 있다. 발행자는 누가 듣는지 모르거나 알 필요 없다.
- `order.created`, `payment.completed`, `user.signed-up`
- 보통 Kafka, NATS JetStream, RabbitMQ topic exchange
- 과거형(past tense) 으로 명명

```
나쁜 예: order.create  ← 명령처럼 보인다
좋은 예: order.created ← 이미 일어난 사실
```

같은 Kafka 토픽에 command 와 event 를 섞어서 보내는 사람이 종종 있다. `order.process` 같은 거. 이건 안 좋다. 누가 듣고 있는지, 왜 듣는지가 명확하지 않다. event는 broadcast 가 자연스럽지만, command는 한 명만 처리해야 한다. 같은 채널에서 둘 다 일어나면 디버깅이 지옥이다.

이름 짓기 규칙은 단순하다. `<entity>.<verb-past-tense>` 면 event, `<entity>.<verb-imperative>` 면 command. 그리고 둘을 다른 토픽 prefix 또는 다른 broker로 분리한다.

```
events.order.created
events.payment.completed
commands.email.send
commands.payment.charge
```

### 이벤트 스키마 진화

서비스가 분리되면 producer 와 consumer 가 다른 팀이다. producer 가 이벤트 스키마를 바꾸면 consumer 들이 깨진다. 처음 한두 번은 실수하고 슬랙에 사과하지만, 시스템이 커지면 그 비용이 어마어마하다.

해법은 schema registry 다. Avro, Protobuf, JSON Schema 중 하나를 쓰고, 각 메시지가 자기 스키마 ID 를 헤더에 달고 다닌다. consumer 는 ID 로 스키마를 조회해서 메시지를 파싱한다.

```typescript
// schema-registry/avro-producer.ts
import { SchemaRegistry, SchemaType } from '@kafkajs/confluent-schema-registry';

const registry = new SchemaRegistry({
  host: 'http://schema-registry:8081',
});

const orderCreatedSchema = `{
  "type": "record",
  "name": "OrderCreated",
  "namespace": "com.acme.orders",
  "fields": [
    { "name": "orderId", "type": "string" },
    { "name": "userId", "type": "string" },
    { "name": "totalAmount", "type": "long" },
    { "name": "currency", "type": "string", "default": "USD" }
  ]
}`;

const { id } = await registry.register({
  type: SchemaType.AVRO,
  schema: orderCreatedSchema,
});

async function publishOrderCreated(event: any) {
  const encoded = await registry.encode(id, event);
  await producer.send({
    topic: 'events.order',
    messages: [{ key: event.orderId, value: encoded }],
  });
}
```

```typescript
// schema-registry/avro-consumer.ts
await consumer.run({
  eachMessage: async ({ message }) => {
    const decoded = await registry.decode(message.value!);
    await handleOrderCreated(decoded);
  },
});
```

스키마 진화 규칙은 호환성 모드에 따라 다르다.

- **BACKWARD** (기본): 새 스키마로 옛 데이터를 읽을 수 있어야 함. consumer 먼저 업데이트.
- **FORWARD**: 옛 스키마로 새 데이터를 읽을 수 있어야 함. producer 먼저 업데이트.
- **FULL**: 둘 다.

내 경험상 BACKWARD 로 시작하는 게 안전하다. consumer 를 먼저 업데이트해서 신/구 양쪽을 처리할 수 있게 한 뒤, producer 를 업데이트한다.

호환성을 깨는 변경(필드 삭제, 타입 변경)은 신/구 토픽을 같이 운영하는 식으로 해결한다. `events.order.v1` 과 `events.order.v2` 를 동시에 발행하다가, 모든 consumer 가 v2 로 옮긴 뒤 v1 발행을 중단한다.

기본 규칙:
- 필드 추가는 항상 default 값 지정
- 필드 이름 변경 금지(추가 → 옛 필드 제거 절차)
- 타입을 좁히는 변경 금지 (string → enum, long → int)
- enum 값 추가는 가능하지만 producer 가 새 값을 보내기 전에 모든 consumer 가 먼저 업데이트되어야 함

JSON Schema 와 비교하면 Avro/Protobuf 는 binary 인코딩이라서 메시지 크기가 작다. 트래픽이 많으면 차이가 크다. 대신 사람이 메시지를 직접 들여다보기 어렵다. 디버깅용 tool 을 같이 갖추는 게 필수.

### 트랜잭셔널 아웃박스(Transactional Outbox)

DB 변경과 이벤트 발행을 atomically 처리하는 패턴이다. "주문 저장 → 이벤트 발행" 을 두 단계로 나눠서 실행하면, 둘 사이에서 죽으면 inconsistency 가 발생한다. DB 만 저장되고 이벤트는 못 보내거나, 그 반대.

해결은 outbox 테이블에 이벤트를 같은 트랜잭션으로 저장하고, 별도 worker 가 outbox 를 polling 해서 broker 에 발행하는 것이다. Debezium 같은 CDC 도구로 DB 의 commit log 를 읽어서 자동 발행하기도 한다.

자세한 패턴은 별도 문서를 참고: [트랜잭셔널 아웃박스 패턴](./Transactional_Outbox.md) (해당 문서가 없다면 작성 예정 표시).

이 문서에서는 간단한 구조만 보인다.

```typescript
// outbox/save-with-event.ts
async function createOrder(input: CreateOrderInput) {
  await db.transaction(async (tx) => {
    const order = await orderRepo.insert(input, tx);

    await tx.insert('outbox', {
      id: randomUUID(),
      aggregate_type: 'order',
      aggregate_id: order.id,
      event_type: 'order.created',
      payload: JSON.stringify(order),
      created_at: new Date(),
      published_at: null,
    });
  });
}

// 별도 worker 가 outbox 를 polling
async function outboxRelay() {
  while (true) {
    const rows = await db.query(
      `SELECT * FROM outbox WHERE published_at IS NULL ORDER BY created_at LIMIT 100 FOR UPDATE SKIP LOCKED`,
    );
    if (rows.length === 0) {
      await sleep(500);
      continue;
    }

    for (const row of rows) {
      await producer.send({
        topic: `events.${row.aggregate_type}`,
        messages: [{ key: row.aggregate_id, value: row.payload, headers: { 'event-id': row.id } }],
      });
      await db.query('UPDATE outbox SET published_at = NOW() WHERE id = $1', [row.id]);
    }
  }
}
```

`FOR UPDATE SKIP LOCKED`가 핵심이다. worker 인스턴스가 여러 개 떠 있어도 같은 row 를 두 번 처리하지 않는다. 한 인스턴스가 잡고 있는 row 는 다른 인스턴스가 skip 한다.

## Saga 패턴 (분산 트랜잭션)

여러 서비스에 걸친 작업을 atomic 하게 처리해야 할 때 쓰는 패턴이다. 핵심은 **분산 락이나 2PC 를 쓰지 않고**, 각 서비스가 로컬 트랜잭션으로 자기 작업을 하고, 실패 시 보상 트랜잭션으로 되돌리는 것이다.

### Choreography Saga

각 서비스가 이벤트를 듣고 자기 일을 한 뒤 다음 이벤트를 발행한다. 중앙 조율자가 없다.

```
OrderService:    order.created
                       ↓
PaymentService:  payment.completed  /  payment.failed
                       ↓                       ↓
InventoryService: inventory.reserved    OrderService: order.cancelled
                       ↓                       
ShippingService: shipping.scheduled
                       ↓
OrderService:    order.confirmed
```

```typescript
// payment-service: order.created 듣기
consumer.subscribe({ topics: ['events.order'] });

await consumer.run({
  eachMessage: async ({ message }) => {
    const event = JSON.parse(message.value!.toString());
    if (event.type !== 'order.created') return;

    try {
      await db.transaction(async (tx) => {
        await paymentRepo.charge(event.orderId, event.totalAmount, tx);
        await tx.insert('outbox', {
          event_type: 'payment.completed',
          payload: JSON.stringify({ orderId: event.orderId, amount: event.totalAmount }),
        });
      });
    } catch (err) {
      await db.transaction(async (tx) => {
        await tx.insert('outbox', {
          event_type: 'payment.failed',
          payload: JSON.stringify({ orderId: event.orderId, reason: err.message }),
        });
      });
    }
  },
});
```

```typescript
// order-service: 보상 처리
consumer.subscribe({ topics: ['events.payment', 'events.inventory'] });

await consumer.run({
  eachMessage: async ({ message }) => {
    const event = JSON.parse(message.value!.toString());

    if (event.type === 'payment.failed') {
      await orderRepo.updateStatus(event.orderId, 'CANCELLED');
    } else if (event.type === 'inventory.reservation-failed') {
      // 결제는 성공했는데 재고 예약 실패 → 환불
      await db.transaction(async (tx) => {
        await orderRepo.updateStatus(event.orderId, 'CANCELLED', tx);
        await tx.insert('outbox', {
          event_type: 'payment.refund-requested',
          payload: JSON.stringify({ orderId: event.orderId }),
        });
      });
    }
  },
});
```

choreography 의 단점은 흐름이 한눈에 안 보인다는 거다. order 가 confirmed 까지 가는 데 어떤 이벤트들이 거치는지 알려면 여러 서비스 코드를 다 읽어야 한다. 또 새로운 보상 흐름이 필요할 때 수정해야 할 서비스가 많다.

### Orchestration Saga

중앙 조율자(saga orchestrator)가 모든 단계를 관리한다.

```typescript
// orchestrator/order-saga.ts
type SagaStep = 'payment' | 'inventory' | 'shipping' | 'completed';

class OrderSaga {
  async execute(orderId: string): Promise<void> {
    const state = await sagaRepo.create({ orderId, currentStep: 'payment' });

    try {
      await this.runStep('payment', () => paymentClient.charge(orderId));
      await this.runStep('inventory', () => inventoryClient.reserve(orderId));
      await this.runStep('shipping', () => shippingClient.schedule(orderId));
      await sagaRepo.complete(state.id);
      await orderRepo.updateStatus(orderId, 'CONFIRMED');
    } catch (err) {
      await this.compensate(state.id);
      throw err;
    }
  }

  private async runStep(name: SagaStep, fn: () => Promise<void>) {
    await fn();
    await sagaRepo.markStepCompleted(name);
  }

  private async compensate(sagaId: string) {
    const completed = await sagaRepo.getCompletedSteps(sagaId);

    for (const step of completed.reverse()) {
      try {
        switch (step) {
          case 'shipping':
            await shippingClient.cancel(sagaId);
            break;
          case 'inventory':
            await inventoryClient.release(sagaId);
            break;
          case 'payment':
            await paymentClient.refund(sagaId);
            break;
        }
      } catch (err) {
        // 보상 실패는 인간 개입 필요. alert 발행.
        await alertingService.notify('saga compensation failed', { sagaId, step });
      }
    }
  }
}
```

orchestration 은 흐름이 한곳에 모여 있다. 디버깅도 쉽고, 변경도 한 파일로 끝난다. 단점은 orchestrator 가 모든 도메인을 알아야 해서 결합도가 높아진다는 점이다. orchestrator 자체의 가용성도 챙겨야 한다.

내 경험으로는 단순한 흐름은 choreography, 5단계 이상이거나 보상이 복잡하면 orchestration 으로 가는 게 좋았다. orchestration 은 별도 워크플로 엔진(Temporal, Camunda)을 쓰는 게 직접 만드는 것보다 안전하다.

## 분산 트레이싱: 컨텍스트 전파

서비스가 5개를 넘어가면 한 요청이 어떤 경로로 흐르는지 로그만으로는 안 보인다. trace ID 를 모든 호출에 전파하고, 각 단계의 시간을 기록해서 한눈에 보는 게 분산 트레이싱이다.

W3C Trace Context 표준이 생기면서 헤더 포맷이 통일됐다.

```
traceparent: 00-{trace-id}-{span-id}-{flags}
tracestate: vendor1=value1,vendor2=value2

예:
traceparent: 00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01
```

OpenTelemetry SDK 를 쓰면 대부분 자동으로 처리된다.

```typescript
// telemetry/setup.ts
import { NodeSDK } from '@opentelemetry/sdk-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
```

`auto-instrumentations-node` 를 쓰면 HTTP, gRPC, kafkajs, ioredis, pg, mysql2 등이 자동으로 계측된다. 메시지 broker 에 들어가는 메시지에도 traceparent 가 헤더로 자동 주입된다.

직접 span 을 만들고 싶을 때는 이렇게.

```typescript
// telemetry/manual-span.ts
import { trace, SpanStatusCode } from '@opentelemetry/api';

const tracer = trace.getTracer('order-service');

async function processOrder(orderId: string) {
  return tracer.startActiveSpan('process_order', async (span) => {
    try {
      span.setAttribute('order.id', orderId);

      const order = await tracer.startActiveSpan('fetch_order', async (s) => {
        const result = await orderRepo.findById(orderId);
        s.end();
        return result;
      });

      const result = await tracer.startActiveSpan('charge_payment', async (s) => {
        s.setAttribute('payment.amount', order.totalAmount);
        try {
          return await paymentClient.charge(order);
        } catch (err: any) {
          s.recordException(err);
          s.setStatus({ code: SpanStatusCode.ERROR, message: err.message });
          throw err;
        } finally {
          s.end();
        }
      });

      span.setStatus({ code: SpanStatusCode.OK });
      return result;
    } catch (err: any) {
      span.recordException(err);
      span.setStatus({ code: SpanStatusCode.ERROR, message: err.message });
      throw err;
    } finally {
      span.end();
    }
  });
}
```

### Kafka 메시지에 trace 전파

producer 가 보낸 trace 를 consumer 에서 이어 받으려면 `traceparent` 를 메시지 헤더에 직접 주입/추출해야 한다. auto-instrumentation 이 어느 정도 해주지만, 직접 다루면 이렇다.

```typescript
// telemetry/kafka-propagation.ts
import { context, propagation, trace, SpanKind } from '@opentelemetry/api';

const tracer = trace.getTracer('payment-service');

// producer
async function publishWithTrace(topic: string, key: string, payload: any) {
  return tracer.startActiveSpan(
    `kafka.send ${topic}`,
    { kind: SpanKind.PRODUCER, attributes: { 'messaging.system': 'kafka', 'messaging.destination': topic } },
    async (span) => {
      const headers: Record<string, string> = {};
      propagation.inject(context.active(), headers);

      await producer.send({
        topic,
        messages: [{ key, value: JSON.stringify(payload), headers }],
      });
      span.end();
    },
  );
}

// consumer
await consumer.run({
  eachMessage: async ({ topic, message }) => {
    const headers: Record<string, string> = {};
    for (const [k, v] of Object.entries(message.headers ?? {})) {
      if (v) headers[k] = v.toString();
    }

    const parentCtx = propagation.extract(context.active(), headers);

    await context.with(parentCtx, async () => {
      await tracer.startActiveSpan(
        `kafka.consume ${topic}`,
        { kind: SpanKind.CONSUMER },
        async (span) => {
          try {
            await processMessage(JSON.parse(message.value!.toString()));
            span.setStatus({ code: SpanStatusCode.OK });
          } catch (err: any) {
            span.recordException(err);
            span.setStatus({ code: SpanStatusCode.ERROR });
            throw err;
          } finally {
            span.end();
          }
        },
      );
    });
  },
});
```

`propagation.extract` 와 `propagation.inject` 가 표준 인터페이스다. 나중에 traceparent 외 다른 포맷(B3, Jaeger)을 추가해도 코드 변경 없이 헤더 포맷이 바뀐다.

trace 를 다 깔고 나면 한 요청에 대해 service A -> kafka -> service B -> grpc -> service C 같은 흐름이 한 화면에 다 보인다. p99 가 800ms 인데 어디서 시간을 쓰는지 안 보일 때 이게 없으면 디버깅 못 한다.

## 통신 패턴별 트러블슈팅

여기부터는 실제로 새벽에 겪은 사례들이다. 케이스마다 증상, 원인, 해결, 예방을 정리한다.

### 사례 1: Kafka consumer lag 폭증

**증상.** Grafana 의 consumer lag 그래프가 평소 1000건 미만이던 게 갑자기 30만 건으로 뛰어오름. 처리 속도는 정상인데 producer 가 갑자기 많이 보내는 건지, consumer 가 느려진 건지 즉시는 안 보임. 한 시간 뒤 lag 가 180만 건. 모니터링 알람이 PagerDuty 로 6번 째 와 있음.

**원인.** consumer 한 인스턴스가 메시지 처리 중 외부 API 호출에서 멈춰 있었음. timeout 미설정이라 8분짜리 호출 1번이 그 partition 의 다음 메시지들을 모두 막고 있었다. consumer group 의 다른 partition 들은 정상이었지만, 막힌 partition 1개에서 lag 가 누적됐다.

**확인 방법.**

```bash
# consumer group 상태
kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
  --describe --group payment-service

# 출력에서 LAG 컬럼이 비정상적으로 높은 partition 확인
# CURRENT-OFFSET, LOG-END-OFFSET, LAG, CONSUMER-ID, HOST 도 같이 나옴
```

특정 partition 의 LAG 만 폭증해 있고 CONSUMER-ID 가 동일하면 그 instance 가 stuck 상태다. 보통 정상 인스턴스는 LAG 가 0~100 정도.

**해결.** stuck 인스턴스를 죽이면 리밸런싱이 일어나고 다른 인스턴스가 그 partition 을 받아서 처리 시작. 단, 같은 메시지가 다시 외부 API 를 호출하면 또 막힐 수 있으므로, 그 메시지는 DLQ 로 보내거나 skip 해야 한다.

**예방.**

```typescript
// 모든 외부 호출에 타임아웃 강제
const result = await Promise.race([
  externalApi.call(payload),
  new Promise<never>((_, reject) =>
    setTimeout(() => reject(new Error('external_api_timeout')), 5000),
  ),
]);
```

```typescript
// 그리고 consumer group 의 max.poll.interval.ms 를 처리 시간보다 짧게.
// kafkajs 에는 직접 옵션이 없지만 sessionTimeout 으로 비슷하게.
const consumer = kafka.consumer({
  groupId: 'payment-service',
  sessionTimeout: 30_000,
  heartbeatInterval: 3_000,
  // 내부적으로 maxWaitTimeInMs 와 결합
  maxWaitTimeInMs: 5000,
});
```

장기적으로는 lag 알람을 partition 단위로 걸어야 한다. 전체 lag 평균은 정상으로 보여도 한 partition 만 폭증하는 경우가 자주 있다.

### 사례 2: gRPC HTTP/2 connection draining

**증상.** Kubernetes 에서 gRPC 서버 pod 를 rolling update 하는 도중에 client 쪽에서 `UNAVAILABLE: Endpoint read failed` 에러가 5초 정도 쏟아짐. healthcheck 는 다 통과하고 readiness 도 정상.

**원인.** gRPC 는 HTTP/2 long-lived connection 을 쓴다. client 가 서버 IP 를 한 번 resolve 하고 그 connection 을 계속 재사용한다. pod 가 종료될 때 SIGTERM 을 받으면 새 connection 을 거부해야 하는데, 기존 connection 의 in-flight 호출은 처리해야 한다. 이걸 graceful 하게 처리하지 않으면 connection 이 갑자기 끊기면서 client 에서 에러.

**해결.**

```typescript
// grpc/graceful-shutdown.ts
const server = new grpc.Server();
// ... 서비스 등록 ...

server.bindAsync('0.0.0.0:50051', grpc.ServerCredentials.createInsecure(), (err, port) => {
  if (err) throw err;
  server.start();
  logger.info(`grpc server listening on ${port}`);
});

// SIGTERM 처리
let shuttingDown = false;
process.on('SIGTERM', async () => {
  if (shuttingDown) return;
  shuttingDown = true;
  logger.info('SIGTERM received, draining grpc server');

  // 1. readiness 를 false 로 만들어서 service mesh / k8s 가 새 트래픽을 안 보내게.
  global.isReady = false;

  // 2. 기존 connection 에 GOAWAY 를 보내고 in-flight 호출을 끝까지 처리.
  server.tryShutdown((err) => {
    if (err) logger.error('grpc shutdown error', err);
    process.exit(0);
  });

  // 3. 30초 안에 끝나지 않으면 강제 종료
  setTimeout(() => {
    logger.warn('forcing shutdown after timeout');
    server.forceShutdown();
    process.exit(1);
  }, 30_000);
});
```

readiness probe 응답 코드.

```typescript
app.get('/healthz', (req, res) => {
  if (!global.isReady) return res.status(503).send('not ready');
  res.send('ok');
});
```

**예방.** Kubernetes deployment 의 `terminationGracePeriodSeconds` 를 SIGTERM 이후 충분히 길게(60s 정도) 주고, preStop hook 으로 sleep 을 넣어서 readiness 가 false 로 바뀐 뒤 service mesh 가 endpoint 를 빼는 시간을 확보한다.

```yaml
spec:
  terminationGracePeriodSeconds: 60
  containers:
    - name: app
      lifecycle:
        preStop:
          exec:
            command: ["/bin/sh", "-c", "sleep 10"]
```

client 쪽에서도 `MAX_CONNECTION_AGE`, `MAX_CONNECTION_AGE_GRACE` 같은 keepalive 설정을 잡아서 connection 이 너무 오래 유지되지 않게 한다.

```typescript
const client = new proto.UserService(
  'user-service:50051',
  grpc.credentials.createInsecure(),
  {
    'grpc.keepalive_time_ms': 30_000,
    'grpc.keepalive_timeout_ms': 5_000,
    'grpc.max_connection_age_ms': 5 * 60 * 1000,
    'grpc.max_connection_age_grace_ms': 30_000,
  },
);
```

### 사례 3: 동기 호출 cascade failure

**증상.** API gateway 의 응답 시간이 평균 100ms 에서 5초로 급증. 5분 뒤 서비스 전체 다운. CPU/메모리는 정상인데 인스턴스가 새 요청을 받지 못하는 상태.

**원인.** A → B → C 호출 체인에서 C 가 DB 락 때문에 응답이 5초씩 걸리기 시작. B 의 worker pool 이 모두 C 응답 대기로 점유됨. A 가 B 를 호출하면 B 가 응답 못 함. A 의 worker pool 도 점유. gateway 가 A 호출 못 함. 전체 stall.

이게 cascade failure 다. 한 곳의 latency 증가가 호출 체인 전체를 동시에 죽인다.

**해결 (사후).**

1. C 의 DB 락 해소 (longest running query 죽이기, lock 보유 transaction 찾기)
2. A, B 인스턴스 재시작으로 worker pool 비우기

**예방 (사전).**

1. **타임아웃 계층화.** gateway 가 5초로 두고 끝까지 기다리면 안 된다. gateway 1초, A 700ms, B 400ms 같이 깊이가 깊어질수록 짧게. C 가 5초 걸려도 B 가 400ms 안에 포기해야 worker pool 이 풀린다.

2. **Circuit Breaker.** B 가 C 호출 실패율이 50% 넘으면 회로 차단. fallback 으로 빈 응답이라도 빠르게 줘야 한다.

3. **Bulkhead.** B 의 worker pool 을 endpoint 별로 분리. C 호출이 필요한 endpoint 의 worker pool 이 점유되어도 다른 endpoint 는 정상 동작.

```typescript
// http/bulkhead.ts
import { Sema } from 'async-sema';

const criticalSema = new Sema(50); // 중요한 호출용
const normalSema = new Sema(20);   // 일반 호출용
const slowSema = new Sema(5);      // 느린 외부 API 호출용 (격리)

async function callSlowExternalApi(payload: any) {
  await slowSema.acquire();
  try {
    return await externalClient.post('/slow', payload);
  } finally {
    slowSema.release();
  }
}
```

slow 호출이 5개로 제한되니, 외부 API 가 늦어져도 worker 5개만 점유되고 나머지 worker 는 다른 endpoint 에 응답 가능.

4. **비동기로 전환.** 진짜 응답이 즉시 필요한가? 아니면 "주문 접수했다"만 응답하고 나머지는 background 로 처리할 수 있나? 결제 같은 건 비동기로 못 가지만, 알림 발송, 통계 업데이트, 외부 시스템 동기화 같은 건 비동기로 빼는 게 답이다.

### 사례 4: RabbitMQ 메시지 무한 루프

**증상.** RabbitMQ management UI 에서 `orders.processing` 큐의 message rate 가 갑자기 초당 5만 건. consumer CPU 가 100%. 같은 메시지가 ack 도 안 되고 nack 만 되면서 큐를 계속 도는 상태.

**원인.** consumer 가 메시지 처리 중 unhandled exception 으로 nack(requeue=true) 를 호출. 같은 메시지가 다시 큐로 돌아왔는데, 같은 에러가 반복. 무한 루프.

**해결 (사후).**

```bash
# 큐 비우기 (메시지 손실을 감수해야 함)
rabbitmqctl purge_queue orders.processing
```

또는 management UI 의 "Get messages" 로 일부를 떠내서 분석한 뒤 purge.

**예방 (사전).**

```typescript
// requeue=false 로 바꾸고 DLQ 로 보내기
channel.consume('orders.processing', async (msg) => {
  if (!msg) return;
  try {
    await processMessage(JSON.parse(msg.content.toString()));
    channel.ack(msg);
  } catch (err) {
    logger.error('processing failed', { err });
    channel.nack(msg, false, false); // requeue=false → DLX 로
  }
});
```

그리고 DLQ 에 message rate alert 를 걸어서 비정상 누적 시 즉시 알림이 가게.

### 사례 5: Kafka producer 와 consumer 시간 동기화 문제

**증상.** consumer 에서 메시지의 timestamp 와 처리 시각의 차이가 갑자기 벌어짐. 평소 100ms 이내였는데 30초씩 차이남. lag 는 정상.

**원인.** producer 인스턴스의 시계가 NTP 동기화 실패로 30초 빠르게 가고 있었음. 메시지의 timestamp 는 정상보다 미래값. consumer 는 현재 시간이 정확해서 30초 차이가 나는 것처럼 보였음.

**해결.** producer pod 의 NTP 동기화 복구. Kubernetes 에서는 보통 host 의 NTP 를 따라가지만, 컨테이너 안에서 시간이 설정되는 경우 잘못 설정될 수 있다.

**예방.** broker timestamp(`LogAppendTime`) 를 쓰는 옵션도 있다.

```
# Kafka topic 설정
message.timestamp.type=LogAppendTime
```

이러면 broker 가 메시지를 받은 시점의 시간을 timestamp 로 쓴다. producer 시계가 틀어져도 영향 없다. 단점은 producer 가 보낸 실제 시각을 잃어버린다는 것. 비즈니스 로직에서 producer 시점이 중요한지, broker 시점이 중요한지에 따라 결정한다.

### 사례 6: gRPC stream 에서 메모리 누수

**증상.** server-streaming RPC 를 사용하는 서비스의 메모리가 시간에 따라 선형 증가. 24시간마다 OOM kill. 처리량이 늘었나 싶었지만 호출 수는 비슷.

**원인.** server 코드에서 stream 종료 처리 누락. `call.cancelled`, `call.on('error')`, `call.on('close')` 처리가 없어서 client 가 disconnect 한 뒤에도 server 가 데이터를 계속 읽어서 send 하려고 시도. 내부 buffer 가 무한 증가.

**해결.**

```typescript
// 모든 streaming RPC 에 종료 처리 추가
DownloadChunks: async (call) => {
  const stream = await fileStorage.openReadStream(call.request.file_id);

  const cleanup = () => {
    stream.destroy();
  };

  call.on('cancelled', cleanup);
  call.on('error', cleanup);
  call.on('close', cleanup);

  try {
    for await (const chunk of stream) {
      if (call.cancelled || call.destroyed) break;
      const ok = call.write({ /* ... */ });
      if (!ok) await new Promise((r) => call.once('drain', r));
    }
    call.end();
  } finally {
    cleanup();
  }
},
```

**예방.** streaming RPC 는 unary 보다 훨씬 까다롭다. 가능하면 unary 로 충분한 케이스에서는 streaming 을 쓰지 않는다. 페이지네이션으로 unary 호출을 여러 번 하는 게 streaming 보다 운영하기 쉬운 경우가 많다.

### 사례 7: 분산 트랜잭션 Saga 의 compensation 실패

**증상.** order saga 가 실행되다가 inventory.reserve 실패. 보상으로 payment.refund 를 실행했는데 그것마저 실패. order 는 PENDING 상태로 남아 있고, payment 는 환불 안 됨. 사용자는 결제가 빠졌는데 주문이 없는 상태.

**원인.** payment 서비스가 일시적인 네트워크 이슈로 응답 안 함. saga orchestrator 는 한 번 시도하고 실패 처리. 그게 끝.

**해결.**

1. 보상 트랜잭션은 무한 재시도해야 한다 (단, exponential backoff). 그래도 안 되면 alert 띄우고 인간이 개입.

2. 보상 단계도 별도 큐에 publish 해서 worker 가 재시도. saga orchestrator 가 직접 호출하지 말고.

```typescript
// orchestrator/compensate.ts
async function compensate(sagaId: string, completedSteps: SagaStep[]) {
  for (const step of completedSteps.reverse()) {
    await compensationQueue.publish({
      sagaId,
      step,
      attempt: 0,
      scheduledAt: new Date(),
    });
  }
}

// compensation worker
await consumer.consume('saga.compensation', async (msg) => {
  const { sagaId, step, attempt } = JSON.parse(msg.content);

  try {
    await runCompensation(sagaId, step);
    await sagaRepo.markCompensated(sagaId, step);
    channel.ack(msg);
  } catch (err) {
    if (attempt >= 10) {
      await alertingService.notify('compensation failed permanently', { sagaId, step });
      await sagaRepo.markCompensationFailed(sagaId, step);
      channel.ack(msg); // 더 이상 재시도 안 함
    } else {
      // exponential backoff 후 재시도
      const delay = Math.min(2 ** attempt * 1000, 5 * 60 * 1000);
      setTimeout(() => {
        compensationQueue.publish({ sagaId, step, attempt: attempt + 1 });
      }, delay);
      channel.ack(msg);
    }
  }
});
```

**예방.** 보상 트랜잭션은 항상 idempotent 해야 한다. 같은 환불 요청이 여러 번 처리되어도 한 번만 환불되도록. payment.refund(orderId) 같은 식으로 idempotency key 가 자연스러운 식별자(orderId) 가 되게 설계.

### 사례 8: Kafka rebalance 폭풍

**증상.** consumer group 에서 리밸런싱이 1분에 5번씩 일어남. 그 동안 처리는 거의 멈춤. lag 가 빠르게 누적.

**원인.** consumer 인스턴스 중 하나가 GC pause 로 heartbeat 를 못 보냄. broker 가 dead 로 판단하고 리밸런싱 트리거. 잠시 후 그 인스턴스가 깨어나서 다시 join. 또 리밸런싱. 반복.

**확인 방법.**

```typescript
consumer.on(consumer.events.GROUP_JOIN, ({ payload }) => {
  logger.info('group join', payload);
});
consumer.on(consumer.events.HEARTBEAT, ({ payload }) => {
  logger.debug('heartbeat', payload);
});
```

쌓이는 로그를 보면 join 이 너무 자주 일어나는 걸 발견할 수 있다.

**해결.**

1. `sessionTimeout` 을 늘림 (기본 30s → 60s 이상). GC pause 보다 길게.
2. heap 크기 늘리기, GC 알고리즘 변경
3. cooperative rebalancing 켜기. 기본 eager 는 모든 partition 을 unassign 한 뒤 다시 분배라 stop-the-world. cooperative 는 점진적이다.

```typescript
const consumer = kafka.consumer({
  groupId: 'payment-service',
  partitionAssigners: [
    require('kafkajs/src/consumer/assigners/cooperativeStickyAssigner'),
  ],
  sessionTimeout: 60_000,
});
```

**예방.** Kafka consumer 는 단순한 작업만 시키고, 무거운 작업은 별도 worker pool 에 비동기로 넘긴다. consumer 가 직접 30초 짜리 작업을 하면 heartbeat 도 못 보내고 GC 도 길어진다.

### 사례 9: 이벤트 순서가 뒤바뀌는 문제

**증상.** order.created 이벤트를 처리한 직후 order.updated 가 와야 하는데, 가끔 updated 가 먼저 와서 "order not found" 에러. consumer 재시작해도 같은 에러.

**원인.** producer 가 partition key 를 잘못 설정했음. created 와 updated 가 다른 partition 에 들어가서 순서 보장이 깨졌다. consumer 가 multi-partition 으로 동시 처리하면서 updated partition 이 먼저 진행됨.

**해결.** key 를 orderId 로 통일.

```typescript
// 같은 orderId 면 같은 partition 보장
await producer.send({
  topic: 'events.order',
  messages: [
    { key: orderId, value: JSON.stringify(createdEvent) },
  ],
});

await producer.send({
  topic: 'events.order',
  messages: [
    { key: orderId, value: JSON.stringify(updatedEvent) },
  ],
});
```

**예방.** topic 별로 partition key 규약을 문서화. 새 producer 가 들어올 때 review 단계에서 확인. 그리고 consumer 쪽에서 순서 가정을 최소화. 정말 순서가 필요한 경우만 같은 key 로 묶고, 나머지는 멱등성/재처리 가능하게 설계.

또 단순히 "create 직후 update" 패턴이 자주 생긴다면 둘을 한 이벤트(`order.placed-with-details`)로 합치는 것도 방법이다. 두 이벤트로 분리되면 비동기 환경에서 항상 race condition 위험이 있다.

### 사례 10: 비동기 처리 후에도 응답이 필요한 케이스

**증상.** 사용자가 "리포트 생성" 버튼을 누르면 5~10분 걸리는 작업. HTTP 동기 호출로 처리했더니 timeout 잦음. 비동기로 바꿨더니 사용자가 결과를 어떻게 받는지 문제.

**해결.** 두 가지 접근.

1. **Polling.** 사용자에게 jobId 를 즉시 반환하고, 사용자는 주기적으로 `/jobs/{id}` 를 조회.

```typescript
// POST /reports
app.post('/reports', async (req, res) => {
  const jobId = randomUUID();
  await jobRepo.insert({ id: jobId, status: 'PENDING', createdAt: new Date() });
  await jobQueue.publish({ jobId, params: req.body });
  res.status(202).json({ jobId, statusUrl: `/jobs/${jobId}` });
});

// GET /jobs/:id
app.get('/jobs/:id', async (req, res) => {
  const job = await jobRepo.findById(req.params.id);
  if (!job) return res.status(404).end();
  res.json(job);
});
```

2. **Server-Sent Events / WebSocket.** 더 나은 UX. 작업이 끝나면 서버가 push.

```typescript
// SSE
app.get('/jobs/:id/stream', async (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.flushHeaders();

  const subscriber = redis.duplicate();
  await subscriber.subscribe(`job:${req.params.id}`);

  subscriber.on('message', (_, message) => {
    res.write(`data: ${message}\n\n`);
    const event = JSON.parse(message);
    if (event.status === 'COMPLETED' || event.status === 'FAILED') {
      subscriber.unsubscribe();
      res.end();
    }
  });

  req.on('close', () => subscriber.unsubscribe());
});
```

worker 는 진행 상황을 redis pub/sub 으로 publish.

```typescript
// worker
async function processJob(job: Job) {
  await redis.publish(`job:${job.id}`, JSON.stringify({ status: 'PROCESSING', progress: 0 }));

  for (let i = 0; i < 100; i++) {
    await doWork(i);
    await redis.publish(`job:${job.id}`, JSON.stringify({ status: 'PROCESSING', progress: i }));
  }

  await jobRepo.complete(job.id);
  await redis.publish(`job:${job.id}`, JSON.stringify({ status: 'COMPLETED', result: '...' }));
}
```

**주의사항.** SSE/WebSocket 은 long-lived connection 이라서 load balancer 의 timeout, sticky session, connection 한도 같은 것들을 신경 써야 한다. 특히 idle timeout 이 보통 60초 정도라서 keep-alive ping 을 보내야 한다.

## 통신 패턴 선택 기준

표는 빠르게 보기 위한 것이다. 실제로는 트래픽, 팀 역량, 기존 인프라를 같이 고려해야 한다.

| 상황 | 권장 패턴 | 이유 |
|------|----------|------|
| 즉각 응답 필요한 단순 조회 | REST + keep-alive | 간단, 어디서나 동작 |
| 내부 서비스 간 고성능/저지연 호출 | gRPC unary | binary, deadline propagation |
| 큰 파일 업/다운로드 | gRPC streaming | 메모리 효율 |
| 양방향 실시간 동기화 | gRPC bidi 또는 WebSocket | 단방향만 필요하면 SSE |
| 비즈니스 이벤트 전파 (다수 consumer) | Kafka 또는 JetStream | 재처리 가능, 보존 |
| 작업 큐 (단일 consumer 처리) | RabbitMQ direct queue | 라우팅, DLQ |
| 비동기 RPC (응답 매칭) | RabbitMQ reply-to + correlationId | broker 분산 |
| 분산 트랜잭션 (2~3단계) | Choreography Saga | 단순, 결합 낮음 |
| 분산 트랜잭션 (4단계 이상) | Orchestration Saga (Temporal) | 가시성, 디버깅 |
| 하위 서비스 장애 격리 | Circuit Breaker + Bulkhead | cascade 차단 |
| 긴 작업 후 결과 반환 | 비동기 + polling/SSE | timeout 회피 |
| 트래픽 buffering/throttling | 메시지 큐 + worker pool | spike 흡수 |

여러 패턴이 섞이는 게 정상이다. 한 시스템에서 결제는 gRPC, 주문 이벤트는 Kafka, 알림은 RabbitMQ, 리포트는 SSE 식으로. 일관성을 위해 모든 호출을 한 패턴으로 통일하려고 하면 어딘가는 무리가 간다.

## 마무리

마이크로서비스 통신은 단순히 "어떤 broker 를 쓸까" 의 문제가 아니다. 타임아웃, 재시도, 멱등성, 순서, backpressure, observability, 장애 격리 — 각각이 독립적으로 챙겨야 하는 주제고, 하나만 빠져도 실제 운영에서 사고가 난다.

핵심은 이렇다.

- **동기 호출은 항상 타임아웃과 deadline propagation 을 명시적으로.** 무한 대기가 cascade failure 의 시작.
- **재시도는 exponential backoff + jitter + budget.** 그렇지 않으면 다운스트림을 죽인다.
- **비동기는 항상 at-least-once 라고 가정하고 멱등성으로 방어.** exactly-once 는 환상.
- **순서가 필요한 곳에는 partition key.** 그 외에는 순서를 가정하지 않게 설계.
- **backpressure 는 명시적 제어.** 처리 속도가 도달 속도보다 빠르도록.
- **observability 가 없으면 디버깅 불가.** trace 를 항상 전파.
- **단순한 패턴부터.** 처음부터 Kafka + Saga + CDC 다 까는 건 운영 부담만 키운다.

여기 정리한 패턴들과 트러블슈팅 사례는 결국 한 번씩 실수하면서 배우는 것들이다. 이 문서를 보고 비슷한 상황에서 한 시간이라도 빨리 원인을 짚을 수 있으면 충분하다.
