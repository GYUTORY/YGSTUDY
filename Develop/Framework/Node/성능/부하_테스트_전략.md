# Node.js 부하 테스트 전략

## 개요

부하 테스트는 실제 트래픽 상황에서 시스템이 어떻게 동작하는지 검증한다. 프로덕션 배포 전에 병목을 찾고, 서버 스펙 결정의 근거 데이터를 확보한다.

```
부하 테스트 없이 배포하면:
  - "트래픽이 몰리면 얼마나 버티나?" → 모름
  - "DB 커넥션 풀이 충분한가?" → 모름
  - "메모리 누수가 있는가?" → 모름
  - "서버 몇 대가 필요한가?" → 감으로 결정
```

---

## 테스트 종류

| 종류 | 목적 | 특징 |
|------|------|------|
| **Load Test** | 예상 트래픽에서 성능 검증 | 목표 RPS로 일정 시간 유지 |
| **Stress Test** | 한계점 파악 | 점점 트래픽을 높여 실패 시점 확인 |
| **Spike Test** | 급격한 트래픽 증가 처리 | 짧은 시간에 트래픽 폭증 시뮬레이션 |
| **Soak Test** | 장시간 안정성 | 보통 부하로 수 시간~하루 유지 (메모리 누수 탐지) |

## k6 기본 사용

### 설치

```bash
# macOS
brew install k6

# Docker
docker run --rm -i grafana/k6 run - <script.js
```

### 기본 스크립트

```javascript
// load-test.js
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate, Trend } from 'k6/metrics';

// 커스텀 메트릭
const errorRate = new Rate('errors');
const loginDuration = new Trend('login_duration');

export const options = {
  stages: [
    { duration: '1m', target: 50 },   // 1분 동안 50 VU로 램프업
    { duration: '3m', target: 50 },   // 3분 유지
    { duration: '1m', target: 0 },    // 1분 동안 0으로 램프다운
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],  // P95 응답 시간 500ms 이하
    http_req_failed: ['rate<0.01'],    // 에러율 1% 이하
    errors: ['rate<0.01'],
  },
};

const BASE_URL = __ENV.BASE_URL ?? 'http://localhost:3000';

export default function () {
  // 1. 로그인
  const loginRes = http.post(`${BASE_URL}/api/auth/login`, JSON.stringify({
    email: 'test@example.com',
    password: 'password123',
  }), {
    headers: { 'Content-Type': 'application/json' },
  });

  check(loginRes, {
    'login: status 200': (r) => r.status === 200,
    'login: has token': (r) => !!r.json('token'),
  });

  errorRate.add(loginRes.status !== 200);
  loginDuration.add(loginRes.timings.duration);

  if (loginRes.status !== 200) return;

  const token = loginRes.json('token');

  // 2. 사용자 목록 조회
  const usersRes = http.get(`${BASE_URL}/api/users`, {
    headers: { Authorization: `Bearer ${token}` },
  });

  check(usersRes, {
    'users: status 200': (r) => r.status === 200,
    'users: response time < 300ms': (r) => r.timings.duration < 300,
  });

  errorRate.add(usersRes.status !== 200);

  sleep(1); // 1초 대기 (실제 사용자 행동 모방)
}
```

### 실행

```bash
# 기본 실행
k6 run load-test.js

# 환경변수 전달
k6 run -e BASE_URL=https://api.example.com load-test.js

# VU, 시간 CLI 지정
k6 run --vus 100 --duration 30s load-test.js

# 결과를 파일로 저장
k6 run --out json=results.json load-test.js
```

## 시나리오별 테스트

### Stress Test (한계점 탐색)

```javascript
export const options = {
  stages: [
    { duration: '2m', target: 100 },
    { duration: '5m', target: 100 },
    { duration: '2m', target: 200 },
    { duration: '5m', target: 200 },
    { duration: '2m', target: 300 },
    { duration: '5m', target: 300 },  // 여기서 시스템이 무너지면 한계점 = 300 VU
    { duration: '2m', target: 0 },
  ],
};
```

### Spike Test (급격한 트래픽 증가)

```javascript
export const options = {
  stages: [
    { duration: '30s', target: 10 },   // 정상 상태
    { duration: '10s', target: 500 },  // 갑자기 500명으로 폭증
    { duration: '1m', target: 500 },   // 유지
    { duration: '10s', target: 10 },   // 다시 정상
    { duration: '3m', target: 10 },    // 회복 확인
  ],
};
```

### Soak Test (장시간 메모리 누수 탐지)

```javascript
export const options = {
  stages: [
    { duration: '5m', target: 100 },
    { duration: '4h', target: 100 },  // 4시간 유지 → 메모리 점진적 증가 여부 확인
    { duration: '5m', target: 0 },
  ],
};
```

## 주요 지표 해석

```
k6 결과 화면:
  http_req_duration......: avg=143ms  min=12ms  med=98ms  max=2.1s   p(90)=312ms  p(95)=489ms
  http_req_failed........: 0.02%     ✓ 1998 ✗ 0
  http_reqs..............: 2000      (33.3/s)
  vus....................: 50
```

| 지표 | 의미 |
|------|------|
| `avg` | 평균 응답 시간 — 이상치에 민감하므로 단독으로 보지 않음 |
| `p(95)` | 상위 5% 요청의 응답 시간 — SLA 기준으로 주로 사용 |
| `p(99)` | 상위 1% — 최악의 케이스 |
| `http_req_failed` | 에러율 — 1% 넘으면 문제 |
| `vus` | 현재 가상 사용자 수 |

## Node.js 성능 병목 체크리스트

```bash
# 1. 이벤트 루프 지연 확인
node --prof app.js          # V8 프로파일링
node --prof-process isolate-*.log > processed.txt

# 2. 메모리 스냅샷
node --inspect app.js       # Chrome DevTools로 힙 스냅샷

# 3. 클러스터 모드 확인 (CPU 코어 활용)
# pm2 start app.js --instances max
```

### 흔한 병목 원인과 해결책

| 원인 | 증상 | 해결책 |
|------|------|------|
| N+1 쿼리 | DB 쿼리 수가 요청 수에 비례 | Eager Loading, DataLoader |
| DB 인덱스 누락 | 쿼리 시간이 데이터 증가에 비례 | EXPLAIN 분석 후 인덱스 추가 |
| DB 커넥션 부족 | 동시 요청 시 타임아웃 | 커넥션 풀 크기 조정 |
| 동기 블로킹 코드 | CPU 100%, 이벤트 루프 지연 | Worker Thread, 비동기 전환 |
| 메모리 누수 | 시간이 지날수록 메모리 증가 | 전역 변수, 이벤트 리스너 정리 |
| 캐싱 없음 | 동일 쿼리 반복 실행 | Redis 캐싱 적용 |
