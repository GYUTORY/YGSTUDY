# 라우터 — 패킷이 다음 홉을 찾는 방법

## 들어가며

같은 서브넷 안에서는 통신이 잘 되는데 다른 대역으로만 가면 안 되는 경우가 있다. 핑을 때려보면 게이트웨이까지는 가는데 그 너머는 죽는다. 이럴 때 십중팔구 라우팅 문제다. 서버에 라우트가 빠졌거나, 중간 라우터에 돌아오는 경로가 없거나, 양쪽이 서로 다른 길을 보고 있거나 셋 중 하나다.

라우터는 IP 패킷을 받아서 목적지를 보고 다음으로 어디로 보낼지 결정하는 장치다. L3에서 동작한다. 하는 일 자체는 단순한데, 그 단순한 결정이 어긋나면 증상이 묘하게 나타난다. 한 방향만 트래픽이 흐른다거나, TTL이 다 닳아서 패킷이 사라진다거나, 어떤 대역으로 보낸 패킷이 응답 없이 그냥 증발한다거나. 이 문서는 라우터가 라우팅 테이블을 어떻게 읽고 패킷을 넘기는지, 그리고 그 과정에서 운영 중에 부딪히는 문제들을 정리한 글이다.

리눅스 서버 자체도 라우터다. `net.ipv4.ip_forward`를 켜면 인터페이스 사이에서 패킷을 넘긴다. 그래서 라우터 동작을 이해하면 전용 장비뿐 아니라 서버에서 일어나는 라우팅 이슈도 같이 잡힌다.

## 라우팅과 포워딩은 다른 일이다

이 둘을 뭉뚱그려 쓰는 경우가 많은데 실제로는 분리된 작업이다.

포워딩은 패킷 하나가 들어왔을 때 라우팅 테이블을 찾아 다음 홉으로 내보내는 동작이다. 패킷마다 매번 일어나고, 빨라야 한다. 라우터에서 패킷이 초당 수백만 개씩 들어오면 이걸 일일이 CPU가 처리할 수 없어서 하드웨어(ASIC)가 담당한다. 이 영역을 데이터 플레인이라고 부른다.

라우팅은 그 테이블 자체를 만드는 작업이다. 어떤 목적지로 가려면 어느 홉을 거쳐야 하는지를 계산해서 테이블에 채워 넣는다. 정적 라우팅이면 사람이 손으로 넣고, 동적 라우팅이면 OSPF나 BGP 같은 프로토콜이 이웃 라우터와 정보를 주고받아 채운다. 이쪽은 컨트롤 플레인이다.

분리해서 봐야 하는 이유는 장애 양상이 다르기 때문이다. 컨트롤 플레인이 멈춰도(라우팅 프로세스가 죽어도) 이미 만들어진 포워딩 테이블이 살아 있으면 한동안은 패킷이 잘 흐른다. 그래서 "라우팅 프로토콜이 죽었는데 통신은 된다"는 상황이 생긴다. 반대로 테이블은 멀쩡한데 데이터 플레인 하드웨어가 망가지면 라우팅 정보는 정상으로 보이는데 패킷만 안 나간다.

## 라우팅 테이블의 구조

라우팅 테이블 한 줄은 결국 "이 목적지로 가려면 어디로 보내라"는 규칙이다. 핵심 필드는 넷이다.

- **목적지 네트워크**: 프리픽스 형태(`10.0.2.0/24`)로 표현한다. 이 라우트가 커버하는 IP 범위.
- **넥스트홉**: 다음으로 패킷을 넘길 라우터의 IP. 목적지가 직접 연결된 대역이면 넥스트홉 없이 인터페이스로 바로 내보낸다.
- **인터페이스**: 패킷이 나갈 물리/논리 포트(`eth0`, `ens5` 등).
- **메트릭**: 같은 목적지로 가는 길이 여러 개일 때 우선순위를 가르는 값. 작을수록 우선한다.

리눅스에서 실제로 보면 이렇게 나온다.

```bash
$ ip route
default via 10.0.0.1 dev eth0 proto dhcp metric 100
10.0.0.0/24 dev eth0 proto kernel scope link src 10.0.0.37 metric 100
10.0.2.0/24 via 10.0.0.1 dev eth0 metric 100
169.254.169.254 via 10.0.0.1 dev eth0 proto dhcp metric 100
```

한 줄씩 읽으면 이렇다. 첫 줄 `default`는 테이블에서 매칭되는 다른 라우트가 없을 때 쓰는 기본 게이트웨이다. `0.0.0.0/0`과 같은 뜻이다. 둘째 줄은 `10.0.0.0/24` 대역이 `eth0`에 직접 붙어 있다는 뜻이라 넥스트홉이 없다(`scope link`). `src 10.0.0.37`은 이 대역으로 나갈 때 출발지 IP로 무엇을 쓸지를 정한다. 셋째 줄은 `10.0.2.0/24`로 가려면 `10.0.0.1`을 거치라는 명시적 라우트다. 마지막 줄은 AWS 메타데이터 주소(`169.254.169.254`)로 가는 별도 경로다.

`proto`는 이 라우트를 누가 넣었는지 표시한다. `kernel`은 인터페이스에 IP를 붙이면 커널이 자동으로 만든 것, `dhcp`는 DHCP 클라이언트가, `static`은 사람이 손으로, `bgp`/`ospf`는 라우팅 데몬이 넣은 것이다. 라우트가 의도대로 들어왔는지 확인할 때 이 필드를 본다.

## Longest Prefix Match — 가장 구체적인 라우트가 이긴다

테이블에 목적지를 커버하는 라우트가 여러 개 겹칠 수 있다. 이때 라우터는 프리픽스가 가장 긴(가장 구체적인) 라우트를 고른다. 이게 Longest Prefix Match다.

예를 들어 목적지가 `10.0.2.15`인 패킷이 들어왔고 테이블에 이런 라우트가 있다고 하자.

```
0.0.0.0/0      via 192.168.1.1   (default)
10.0.0.0/8     via 192.168.1.2
10.0.2.0/24    via 192.168.1.3
```

`10.0.2.15`는 세 라우트 모두에 매칭된다. `/0`은 모든 주소를 포함하고, `/8`은 `10.x.x.x` 전부를, `/24`는 `10.0.2.x`를 커버한다. 하지만 라우터는 프리픽스 길이가 가장 긴 `10.0.2.0/24`를 선택해서 `192.168.1.3`으로 보낸다. 메트릭이나 순서는 보지 않는다. 프리픽스 길이가 먼저다.

메트릭은 프리픽스 길이가 같은 라우트끼리 경쟁할 때만 의미가 있다. `10.0.2.0/24`짜리 라우트가 두 개 있고 넥스트홉만 다르다면, 그때 메트릭이 낮은 쪽을 고른다.

이 규칙을 알면 운영에서 자주 쓰는 트릭이 보인다. 전체 트래픽은 default로 흘려보내되 특정 대역만 다른 길로 빼고 싶을 때, 그 대역을 더 긴 프리픽스로 명시하면 default를 덮어쓴다. 예를 들어 사내 백본 대역(`172.16.0.0/12`)만 전용선으로 보내고 나머지는 인터넷 게이트웨이로 보내는 식이다. default는 그대로 두고 `172.16.0.0/12` 라우트 한 줄만 추가하면 된다.

```bash
# 사내 대역만 전용선 게이트웨이로
$ sudo ip route add 172.16.0.0/12 via 10.0.0.254 dev eth0
```

## 기본 게이트웨이

기본 게이트웨이는 매칭되는 구체적 라우트가 없을 때 패킷을 떠넘기는 곳이다. 대부분의 호스트는 자기가 직접 붙은 대역 라우트 몇 개와 default 하나만 가지고 있다. 인터넷 어디로 가든 일단 게이트웨이로 던지고, 그 다음은 게이트웨이가 알아서 한다.

서버에 default가 없으면 같은 서브넷 안에서는 통신되는데 밖으로는 한 발자국도 못 나간다. 이게 신규 인스턴스 셋업이나 네트워크 재구성 직후에 흔히 보는 증상이다. `ip route`에 `default` 줄이 있는지부터 확인한다.

```bash
$ ip route show default
default via 10.0.0.1 dev eth0 proto dhcp metric 100
```

게이트웨이가 두 개인 경우(메인 회선 + 백업 회선)는 메트릭으로 우선순위를 준다. 메인은 낮은 메트릭, 백업은 높은 메트릭으로 두면 평소엔 메인만 쓰다가 메인 라우트가 사라지면 백업으로 넘어간다. 다만 리눅스 기본 라우팅은 라우트가 테이블에서 빠져야 전환되는 거라, 메인 게이트웨이가 ping은 안 받는데 라우트는 남아 있으면 자동 전환이 안 된다. 그래서 회선 상태를 감시해서 라우트를 넣고 빼는 별도 장치가 필요하다.

## 정적 라우팅 vs 동적 라우팅

정적 라우팅은 사람이 라우트를 직접 박는 방식이다. 작은 망, 경로가 거의 안 바뀌는 환경에서는 이게 제일 깔끔하다. 동작이 예측 가능하고, 라우팅 프로토콜이 CPU를 먹지도 않고, 누가 봐도 트래픽이 어디로 가는지 명확하다. 클라우드 VPC 라우팅 테이블도 본질적으로 정적 라우팅이다(자세한 건 [AWS 라우팅 테이블](../../../AWS/Network/Route_Table.md) 문서에 정리했다).

단점은 경로가 바뀌면 사람이 손으로 고쳐야 한다는 점이다. 링크가 끊겼을 때 우회로가 자동으로 잡히지 않는다. 라우터가 수십 대 되고 경로가 수시로 바뀌는 망에서 정적으로 관리하면 한 군데 빠뜨려서 블랙홀이 생기는 사고가 반드시 난다.

동적 라우팅은 라우터끼리 프로토콜로 경로 정보를 주고받아 테이블을 자동으로 채우고 갱신한다. 링크가 죽으면 이웃 라우터들이 그 사실을 전파하고, 각자 우회로를 다시 계산한다. 사람이 개입하지 않아도 망이 스스로 복구된다. 대신 프로토콜 설정과 동작을 이해해야 하고, 수렴(convergence)이 끝나기 전까지는 일시적으로 경로가 불안정할 수 있다.

실무에서는 보통 섞어 쓴다. 백본은 동적 라우팅으로 돌리고, 변하지 않는 특정 목적지나 default는 정적으로 박는 식이다.

## 동적 라우팅 프로토콜 — RIP, OSPF, BGP

세 개를 다 외울 필요는 없고, 어디에 쓰는 물건인지만 구분하면 된다.

**RIP**는 홉 카운트로 경로를 고른다. 거쳐 가는 라우터 수가 적은 길을 택한다. 최대 15홉까지만 지원하고, 30초마다 테이블 전체를 이웃에게 뿌리는 방식이라 망이 커지면 수렴이 느리고 트래픽 낭비가 심하다. 요즘 새로 설계하는 망에서 RIP를 쓰는 경우는 거의 없다. 레거시 장비나 아주 작은 망에서만 가끔 본다.

**OSPF**는 한 조직 내부(AS 내부)를 굴리는 표준이다. 링크 상태 방식이라 각 라우터가 망 전체의 토폴로지 지도를 가지고 있고, 그 지도 위에서 최단 경로를 직접 계산한다(다익스트라). 링크의 대역폭을 cost로 환산해서 빠른 길을 고른다. 수렴이 RIP보다 빠르고, Area로 망을 나눠서 큰 조직도 감당한다. 데이터센터 내부, 캠퍼스 망, 기업 백본은 대부분 OSPF로 돈다.

**BGP**는 서로 다른 조직(AS) 사이를 잇는 프로토콜이다. 인터넷 자체가 BGP로 굴러간다. ISP끼리, 또는 기업과 ISP 사이에서 "나는 이 대역들을 가지고 있다"를 광고하고, 받은 광고들 중에서 정책에 따라 경로를 고른다. OSPF가 "가장 빠른 길"을 찾는다면 BGP는 "정책상 허용되고 적절한 길"을 찾는다. 경로 선택이 순수하게 거리 기반이 아니라 AS 경로 길이, 로컬 우선순위 같은 정책 속성으로 결정된다.

클라우드를 쓰면 BGP를 직접 만질 일이 생긴다. 온프레미스와 AWS를 Direct Connect나 VPN으로 잇고 라우트를 자동 교환할 때 BGP를 쓴다. 이쪽 대역을 광고하고 저쪽 대역을 받아서 라우팅 테이블에 반영하는 구조다.

정리하면 이렇게 고른다. 한 조직 내부의 라우팅이면 OSPF, 다른 조직과 경로를 교환해야 하면 BGP, 둘 다 과한 아주 작은 망이면 정적 라우팅. RIP는 새로 선택할 이유가 거의 없다.

## L2 스위치와 뭐가 다른가

스위치와 라우터를 헷갈리는 경우가 있는데 동작 계층과 판단 기준이 다르다.

L2 스위치는 MAC 주소로 프레임을 넘긴다. 같은 브로드캐스트 도메인(같은 VLAN/서브넷) 안에서 동작하고, MAC 주소 테이블을 보고 어느 포트로 내보낼지 정한다. IP는 보지 않는다. 같은 서브넷 안의 호스트끼리 통신은 스위치가 처리한다. 이 단계는 라우터를 거치지 않는다.

라우터는 IP 주소로 패킷을 넘긴다. 서로 다른 서브넷 사이를 잇는다. 호스트가 다른 대역으로 패킷을 보낼 때, 먼저 자기 게이트웨이의 MAC을 ARP로 알아내서 그쪽으로 프레임을 보내고, 라우터가 그 패킷의 목적지 IP를 보고 다음 홉을 정한다. 이 ARP 부분은 [ARP 문서](ARP.md)에 정리해 두었다.

핵심 차이는 패킷이 라우터를 지날 때마다 TTL이 1씩 깎인다는 점이다. 스위치는 TTL을 건드리지 않는다. 그래서 traceroute로 보이는 홉은 전부 라우터(L3 장비)지, 중간의 스위치는 안 보인다. TTL이 0이 되면 라우터가 패킷을 버리고 ICMP Time Exceeded를 돌려주는데, traceroute가 이 성질을 이용한다(ICMP 동작은 [ICMP 문서](ICMP.md) 참고).

요즘은 L3 스위치라는 물건도 흔하다. 스위칭 하드웨어에 라우팅 기능을 얹어서 양쪽을 다 한다. 데이터센터에서 서브넷 간 라우팅을 빠르게 처리하려고 쓴다. 개념상으로는 여전히 L2 부분과 L3 부분이 나뉘어 동작한다.

## 리눅스에서 라우팅 테이블 다루기

`ip route`(iproute2)가 표준이다. 예전 `route -n`은 IPv6를 제대로 못 다루고 멀티 테이블도 안 보여줘서 지금은 쓰지 않는다.

테이블 확인:

```bash
$ ip route show
default via 10.0.0.1 dev eth0 proto dhcp metric 100
10.0.0.0/24 dev eth0 proto kernel scope link src 10.0.0.37 metric 100
```

특정 목적지로 패킷이 어느 경로를 탈지 커널에게 직접 물어볼 수도 있다. 이게 디버깅할 때 제일 유용하다. 테이블 전체를 눈으로 훑으면서 LPM을 머릿속으로 계산하지 않아도, 커널이 실제로 고를 경로를 알려준다.

```bash
$ ip route get 10.0.2.15
10.0.2.15 via 10.0.0.1 dev eth0 src 10.0.0.37 uid 1000
    cache

$ ip route get 8.8.8.8
8.8.8.8 via 10.0.0.1 dev eth0 src 10.0.0.37 uid 1000
    cache
```

라우트 추가/삭제:

```bash
# 특정 대역 라우트 추가
$ sudo ip route add 10.0.5.0/24 via 10.0.0.254 dev eth0

# default 게이트웨이 교체
$ sudo ip route replace default via 10.0.0.1 dev eth0

# 라우트 삭제
$ sudo ip route del 10.0.5.0/24
```

여기서 추가한 라우트는 메모리에만 있어서 재부팅하면 사라진다. 영구로 남기려면 배포판별 네트워크 설정(netplan, NetworkManager, `/etc/network/interfaces` 등)에 적어야 한다. 운영 서버에서 `ip route add`로 급히 라우트를 넣어 장애를 막은 다음, 설정 파일에 반영하는 걸 깜빡해서 다음 재부팅 때 같은 장애가 재발하는 경우가 흔하다.

리눅스 서버를 라우터로 쓰려면 포워딩을 켜야 한다. 안 켜면 인터페이스 사이에서 패킷을 넘기지 않고 버린다.

```bash
# 현재 상태 확인 (0이면 꺼짐)
$ cat /proc/sys/net/ipv4/ip_forward
0

# 켜기 (재부팅 시 사라짐)
$ sudo sysctl -w net.ipv4.ip_forward=1

# 영구 적용
$ echo 'net.ipv4.ip_forward=1' | sudo tee /etc/sysctl.d/99-forward.conf
$ sudo sysctl --system
```

여러 라우팅 테이블을 쓰는 정책 라우팅(policy routing)도 있다. 출발지 IP나 들어온 인터페이스에 따라 다른 테이블을 보게 하는 방식인데, 멀티 홈(회선 두 개) 환경에서 "eth1로 들어온 트래픽은 eth1로 돌려보낸다" 같은 규칙을 만들 때 쓴다. 비대칭 라우팅 문제를 푸는 도구이기도 하다.

```bash
$ ip rule show
0:      from all lookup local
32766:  from all lookup main
32767:  from all lookup default
```

## traceroute로 경로 추적하기

패킷이 어디까지 가다가 막히는지 보려면 traceroute를 쓴다. TTL을 1부터 하나씩 올려가며 패킷을 보내고, 각 홉의 라우터가 TTL 소진으로 돌려주는 ICMP Time Exceeded의 출발지를 모아 경로를 그린다.

```bash
$ traceroute 8.8.8.8
traceroute to 8.8.8.8 (8.8.8.8), 30 hops max, 60 byte packets
 1  10.0.0.1 (10.0.0.1)  0.412 ms  0.389 ms  0.371 ms
 2  100.64.0.1 (100.64.0.1)  1.21 ms  1.18 ms  1.15 ms
 3  * * *
 4  72.14.215.85 (72.14.215.85)  3.45 ms  3.42 ms  3.51 ms
 5  8.8.8.8 (8.8.8.8)  3.88 ms  3.79 ms  3.81 ms
```

읽는 법. 각 줄이 한 홉이고, 한 줄에 세 번 보낸 결과의 왕복 시간이 나온다. 3번 홉의 `* * *`는 응답이 안 온 것이다. 이걸 보고 "여기서 막혔다"고 단정하면 안 된다. 중간 라우터가 ICMP Time Exceeded를 일부러 안 돌려주도록 설정된 경우가 많고(보안 정책), 그래도 그 너머 홉은 멀쩡히 응답한다. 4번, 5번이 정상으로 나오니 3번은 그냥 ICMP를 안 주는 라우터일 뿐 경로는 살아 있다. 진짜로 막힌 경우는 그 지점 이후가 끝까지 `* * *`로 나온다.

목적지까지 다 갔는데도 마지막 줄이 목적지 IP로 안 끝나고 `* * *`만 반복하면, 목적지가 ICMP/UDP를 막고 있는 거다. 이때는 프로토콜을 바꿔서 다시 본다. 기본 traceroute는 UDP를 쓰는데, 방화벽이 UDP를 막으면 TCP(`-T`, 보통 80/443 포트)나 ICMP(`-I`)로 바꾸면 응답이 나오기도 한다.

```bash
# TCP 443으로 추적 (UDP가 막힌 경로)
$ sudo traceroute -T -p 443 example.com
```

홉마다 RTT가 갑자기 튀는 구간이 보이면 그 사이 링크에 문제가 있을 가능성이 있다. 다만 라우터가 자기 앞으로 온 ICMP 응답을 후순위로 처리해서 RTT가 높게 찍히는 경우도 있으니, 특정 홉의 RTT 하나만 보고 판단하면 안 되고 그 홉 이후로 계속 높아지는지를 본다.

## 운영에서 부딪히는 라우팅 문제들

### 비대칭 라우팅

나가는 길과 돌아오는 길이 다른 상황이다. 그 자체로는 문제가 아니다. 인터넷에서는 흔한 일이다. 문제는 경로 중간에 상태를 추적하는 장비(스테이트풀 방화벽, NAT)가 있을 때 생긴다.

상황을 그려보면 이렇다. 클라이언트가 보낸 요청 패킷은 방화벽 A를 거쳐 서버에 도착하는데, 서버의 응답은 라우팅 테이블상 다른 경로를 타서 방화벽 B로 나간다. 방화벽 B는 이 연결의 시작(SYN)을 본 적이 없으니 "처음 보는 연결의 응답"이라며 패킷을 버린다. 결과적으로 한 방향만 트래픽이 흐르고 연결이 안 맺어진다.

회선이 두 개인 서버에서 자주 터진다. 두 인터페이스에 각각 IP가 붙어 있는데 default 게이트웨이는 하나뿐이라, eth1로 들어온 요청의 응답이 default가 가리키는 eth0로 나가버린다. 진단은 양쪽 끝에서 동시에 패킷 캡처를 떠서 한 방향만 보이는지 확인하는 식으로 한다.

```bash
# 서버에서: 요청은 들어오는데(SYN) 응답이 안 나가는지 확인
$ sudo tcpdump -i eth1 -nn 'tcp port 8080'
# SYN은 보이는데 SYN-ACK가 안 보이거나, SYN-ACK가 다른 인터페이스로 나가면 비대칭
```

해결은 정책 라우팅으로 "들어온 인터페이스로 응답도 내보내라"는 규칙을 거는 것이다. 들어온 인터페이스별로 별도 라우팅 테이블을 만들고 `ip rule`로 출발지 IP에 따라 그 테이블을 보게 한다.

### 라우팅 루프

A는 목적지로 가려면 B로 보내고, B는 같은 목적지를 A로 돌려보내는 상황이다. 패킷이 두 라우터 사이를 왕복하다가 TTL이 0이 되면 버려진다. 정적 라우팅을 잘못 박았을 때, 또는 동적 라우팅이 수렴하는 도중에 일시적으로 생긴다.

증상은 특정 목적지로만 통신이 안 되고, traceroute를 떠보면 같은 두 IP가 번갈아 반복해서 나타나다가 끝난다.

```bash
$ traceroute 10.0.99.10
 1  10.0.0.1   0.5 ms
 2  10.0.0.5   1.2 ms
 3  10.0.0.6   1.8 ms
 4  10.0.0.5   2.1 ms   # 다시 .5
 5  10.0.0.6   2.5 ms   # 다시 .6
 6  10.0.0.5   3.0 ms   # 또 .5 ... 반복
 ...
30  10.0.0.6   ...      # TTL 소진까지 왕복
```

`.5`와 `.6`이 핑퐁하는 게 보이면 그 두 라우터의 라우팅 테이블에서 해당 목적지 라우트를 확인한다. 양쪽이 서로를 넥스트홉으로 가리키고 있을 것이다. 동적 라우팅 환경이라면 수렴이 끝나면 사라지는 일시적 루프인지, 설정 충돌로 고착된 루프인지를 구분해야 한다.

### 블랙홀

패킷이 어떤 라우터에 도착했는데 그 목적지로 가는 라우트가 없으면, 라우터는 패킷을 버린다. 운이 좋으면 ICMP Destination Unreachable을 출발지로 돌려주지만, 그걸 안 주도록 설정된 경우도 많아서 패킷이 그냥 소리 없이 사라진다. 이게 블랙홀이다.

루프와 증상이 비슷한데(목적지로 통신 안 됨) 원인은 반대다. 루프는 라우트가 잘못 가리켜서 왕복하는 거고, 블랙홀은 라우트가 아예 없어서 떨어뜨리는 거다. traceroute로 구분된다. 루프는 같은 IP가 반복되고, 블랙홀은 특정 홉 이후로 그냥 `* * *`만 끝까지 나온다.

흔한 원인 몇 가지. 정적 라우트를 넣다가 돌아오는 경로(return route)를 깜빡한 경우. 한쪽 라우터에만 라우트를 넣어서 갈 수는 있는데 응답이 못 돌아오는 상황도 블랙홀처럼 보인다. 클라우드에서는 라우팅 테이블에 라우트가 없거나, 보안 그룹/방화벽이 조용히 드롭하는 경우가 블랙홀로 나타난다. PMTUD 블랙홀은 별개의 메커니즘인데 증상이 비슷해서 헷갈린다 — 큰 패킷만 사라지면 라우팅 블랙홀이 아니라 MTU 문제일 수 있다([MTU·MSS와 PMTUD](MTU_MSS_PMTUD.md) 참고).

진단은 경로를 끊어서 좁힌다. 출발지에서 목적지로 가는 길의 라우터들에 차례로 `ip route get 목적지`를 때려보면, 어느 라우터에서 라우트가 끊기는지 나온다. 라우트가 멀쩡한데도 패킷이 사라지면 그 라우터의 방화벽 룰이나 데이터 플레인을 의심한다.
