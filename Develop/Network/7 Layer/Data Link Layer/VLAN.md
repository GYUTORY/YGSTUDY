# VLAN — 스위치 한 대를 논리적으로 쪼개기

## 들어가며

VLAN은 평소에 신경 쓸 일이 없다가, 어느 날 갑자기 발목을 잡는 기술이다. 온프레미스 데이터센터를 써본 적이 있다면 한 번쯤 겪는다. 새 서버를 랙에 꽂고 IP를 줬는데 게이트웨이로 핑이 안 나간다. 케이블도 멀쩡하고 IP도 맞는데 통신이 안 된다. 결국 스위치에 콘솔로 붙어서 `show interfaces switchport`를 쳐보면 그 포트가 엉뚱한 VLAN에 들어가 있다. 포트 하나 VLAN만 바꿔주면 끝나는 일인데, VLAN이 뭔지 모르면 한나절을 날린다.

VLAN(Virtual LAN)은 물리적으로 한 대인 스위치를 논리적으로 여러 개의 독립된 스위치처럼 나누는 기술이다. 802.1Q 표준으로 정의돼 있고, 거의 모든 관리형 스위치가 지원한다. 클라우드로 넘어오면서 직접 만질 일은 줄었지만, 클라우드의 VPC와 서브넷이 왜 그렇게 생겼는지 이해하려면 VLAN을 알아야 한다.

이 문서는 VLAN이 풀려고 하는 문제, 802.1Q 태그 구조, Access/Trunk 포트, VLAN 간 라우팅, 그리고 클라우드에서 VLAN 대신 VPC를 쓰는 이유와 보안 이슈를 정리한 글이다.

## VLAN이 없으면 생기는 문제

스위치는 같은 세그먼트에 붙은 호스트들을 하나의 브로드캐스트 도메인으로 묶는다. 호스트 A가 ARP Request처럼 브로드캐스트 프레임을 쏘면 그 프레임은 스위치에 연결된 모든 포트로 퍼진다. 호스트가 10대면 별 문제가 안 되지만, 한 세그먼트에 수백 대가 붙으면 브로드캐스트만으로도 대역폭과 CPU를 갉아먹는다. ARP 폭풍이라도 한번 일면 네트워크 전체가 느려진다.

부서를 나누고 싶을 때도 곤란하다. 개발팀 서버와 회계팀 PC가 같은 스위치에 붙어 있으면, 물리적으로는 같은 세그먼트라 서로 L2로 다 보인다. 격리하려면 스위치를 따로 사야 한다. 부서가 늘 때마다 스위치를 사는 건 말이 안 된다.

VLAN은 이 두 문제를 동시에 푼다. 스위치 한 대 안에서 포트들을 그룹으로 묶고, 그룹마다 별도의 브로드캐스트 도메인을 준다. VLAN 10에 속한 포트의 브로드캐스트는 VLAN 10 포트들에만 퍼지고 VLAN 20으로는 넘어가지 않는다. 물리적으로 같은 스위치에 꽂혀 있어도 다른 VLAN끼리는 L2로 서로를 볼 수 없다. 스위치 한 대를 여러 대처럼 쪼갠 셈이다.

```
물리 스위치 1대
┌─────────────────────────────────────┐
│ VLAN 10 (개발)   VLAN 20 (회계)       │
│  port1 port2      port5 port6        │
│   │     │          │     │           │
│  [브로드캐스트     [브로드캐스트       │
│   도메인 A]         도메인 B]          │
│   서로 격리됨, L2로 통신 불가          │
└─────────────────────────────────────┘
```

## 802.1Q 태그 — 프레임에 VLAN 번호를 새긴다

VLAN이 한 스위치 안에서만 끝나면 단순하다. 포트마다 어느 VLAN인지 스위치가 기억하면 그만이다. 문제는 스위치가 여러 대로 늘어날 때다. 스위치 A의 VLAN 10과 스위치 B의 VLAN 10을 연결하려면, 두 스위치를 잇는 케이블 하나에 여러 VLAN의 트래픽이 섞여서 흐른다. 이때 프레임마다 "나는 VLAN 10이다"라고 표시가 붙어야 받는 쪽에서 구분한다. 그 표시가 802.1Q 태그다.

802.1Q는 이더넷 프레임의 출발지 MAC 주소 다음, EtherType 앞에 4바이트를 끼워 넣는다. 원래 이더넷 프레임을 건드려서 중간에 태그를 삽입하는 방식이다.

```
일반 이더넷 프레임
[목적지 MAC 6][출발지 MAC 6][EtherType 2][페이로드 ...][FCS 4]

802.1Q 태그가 붙은 프레임
[목적지 MAC 6][출발지 MAC 6][TPID 2][TCI 2][EtherType 2][페이로드 ...][FCS 4]
                            └─── 4바이트 태그 ───┘
```

4바이트 태그는 둘로 나뉜다.

- **TPID(Tag Protocol Identifier, 2바이트)**: 값이 `0x8100`이면 이 프레임에 802.1Q 태그가 붙어 있다는 표시다. 스위치는 EtherType 자리에서 `0x8100`을 보면 "아, 다음 2바이트는 VLAN 정보구나" 하고 해석한다.
- **TCI(Tag Control Information, 2바이트)**: 실제 정보가 든 부분이다. 안에서 또 셋으로 쪼개진다.
  - **PCP(Priority Code Point, 3비트)**: QoS 우선순위. 0~7. 음성 트래픽 같은 걸 우선 처리할 때 쓴다.
  - **DEI(Drop Eligible Indicator, 1비트)**: 혼잡할 때 이 프레임을 먼저 버려도 되는지 표시.
  - **VID(VLAN Identifier, 12비트)**: VLAN 번호. 핵심이다.

VID가 12비트라서 표현 가능한 값은 0~4095, 총 4096개다. 그중 0과 4095는 예약돼 있어서 실제로 쓰는 건 1~4094, 4094개다. 온프레미스에서는 차고 넘치는 숫자지만, 대규모 클라우드 사업자가 수십만 테넌트를 격리하려면 4094개로는 어림도 없다. 이게 나중에 VXLAN이 나온 이유 중 하나다.

태그가 붙으면 프레임 크기가 4바이트 늘어난다. 표준 이더넷 MTU 1500을 기준으로 하면 태그된 프레임은 최대 1522바이트가 된다. 오래된 장비 중에는 이 4바이트를 베이비 자이언트 프레임으로 취급해 떨궈버리는 경우가 있어서, 트렁크 구간에서 MTU 문제로 헤매는 일이 가끔 있다.

## Access 포트와 Trunk 포트

VLAN을 다룰 때 포트는 크게 두 종류로 동작한다.

**Access 포트**는 VLAN 하나에만 속한다. 서버나 PC처럼 끝단 장치가 꽂히는 포트다. 이 포트로 들어오는 프레임에는 태그가 없다. 일반 장치는 VLAN을 모르니까 평범한 이더넷 프레임만 보낸다. 스위치가 그 프레임을 받으면 "이 포트는 VLAN 10이다"라는 설정을 보고 내부적으로 VLAN 10 태그를 붙여서 처리한다. 반대로 VLAN 10 프레임을 이 포트로 내보낼 때는 태그를 떼고 일반 프레임으로 만들어 보낸다. 그래서 서버 입장에서는 VLAN의 존재를 전혀 모른 채 통신한다.

**Trunk 포트**는 여러 VLAN의 트래픽을 한꺼번에 실어 나른다. 스위치와 스위치를 잇거나, 스위치와 라우터를 잇거나, 가상화 호스트의 NIC처럼 VLAN을 직접 다루는 장치에 연결할 때 쓴다. 트렁크로 나가는 프레임에는 802.1Q 태그가 붙어 있어서, 받는 쪽이 어느 VLAN인지 구분한다.

Cisco 장비 기준 설정은 이런 식이다.

```
! Access 포트 — 서버가 꽂히는 포트
interface GigabitEthernet0/1
 switchport mode access
 switchport access vlan 10

! Trunk 포트 — 다른 스위치로 가는 포트
interface GigabitEthernet0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
```

`allowed vlan`을 빼먹으면 기본으로 모든 VLAN이 트렁크를 통과한다. 보안상으로도 트래픽 관리상으로도 필요한 VLAN만 명시적으로 허용하는 게 맞다. 실수로 모든 VLAN을 흘려보내면 한쪽 세그먼트의 브로드캐스트 폭풍이 트렁크를 타고 다른 데이터센터까지 넘어가기도 한다.

## Native VLAN — 태그 없는 프레임의 처리

트렁크 포트는 기본적으로 태그된 프레임을 주고받는다. 그런데 태그가 없는 프레임이 트렁크로 들어오면 어떻게 처리할까. 802.1Q는 트렁크마다 Native VLAN을 하나 지정해서, 태그 없는 프레임은 그 VLAN으로 간주한다. 기본값은 VLAN 1이다.

Native VLAN 프레임은 트렁크를 지날 때 태그를 붙이지 않고 그대로 나간다. 이게 호환성 때문에 만들어진 동작인데, 보안 문제의 출발점이 되기도 한다. 양쪽 스위치의 Native VLAN 설정이 다르면 트래픽이 의도치 않게 VLAN을 넘나든다. 스위치 A의 Native가 VLAN 1이고 스위치 B의 Native가 VLAN 99면, A가 태그 없이 보낸 VLAN 1 프레임을 B는 VLAN 99로 받아들인다. 이런 불일치는 디버깅하기 까다롭다.

실무에서는 Native VLAN을 평소에 안 쓰는 번호로 따로 빼두는 경우가 많다. VLAN 1을 그대로 Native로 두지 않고, 아무 호스트도 붙지 않은 VLAN 999 같은 걸 Native로 지정한다. 이렇게 하면 뒤에 나올 VLAN hopping 공격의 한쪽 길을 막는다.

```
interface GigabitEthernet0/24
 switchport mode trunk
 switchport trunk native vlan 999
 switchport trunk allowed vlan 10,20,30
```

## VLAN 간 라우팅 — 격리한 걸 다시 잇기

VLAN으로 브로드캐스트 도메인을 갈라놓으면 같은 VLAN 안에서는 L2로 자유롭게 통신하지만, 다른 VLAN끼리는 서로 닿지 못한다. 그런데 개발팀 VLAN의 서버가 회계팀 VLAN의 DB에 붙어야 하는 일은 항상 생긴다. VLAN을 넘으려면 L3 라우팅이 필요하다. 다른 네트워크끼리 통신하는 거니까 결국 라우터의 일이다.

방법은 두 가지다.

### Router-on-a-stick

라우터의 물리 인터페이스 하나를 트렁크로 연결하고, 그 안에 VLAN마다 서브인터페이스를 만든다. 서브인터페이스가 각 VLAN의 게이트웨이 역할을 한다. 프레임이 VLAN 10에서 VLAN 20으로 가야 하면, VLAN 10 프레임이 트렁크를 타고 라우터로 올라가서 VLAN 10 서브인터페이스로 들어오고, 라우터가 라우팅한 뒤 VLAN 20 서브인터페이스로 내보낸다. 케이블 하나(stick)로 모든 VLAN 간 라우팅을 처리한다고 해서 이런 이름이 붙었다.

```
interface GigabitEthernet0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface GigabitEthernet0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
```

이 방식은 장비가 라우터 하나면 되니까 싸지만, 모든 VLAN 간 트래픽이 케이블 하나를 거쳐가서 병목이 생긴다. VLAN 10과 20 사이 트래픽이 많으면 그 한 가닥 케이블이 한계가 된다. 소규모나 지점 사무실에서 주로 쓴다.

### L3 스위치

요즘 데이터센터에서는 router-on-a-stick보다 L3 스위치(멀티레이어 스위치)를 쓴다. L3 스위치는 스위칭과 라우팅을 한 박스에서 한다. VLAN마다 SVI(Switched Virtual Interface)라는 가상 인터페이스를 만들고 거기에 게이트웨이 IP를 준다. VLAN 간 라우팅이 스위치 ASIC에서 와이어 스피드로 처리돼서 router-on-a-stick 같은 병목이 없다.

```
interface Vlan10
 ip address 192.168.10.1 255.255.255.0

interface Vlan20
 ip address 192.168.20.1 255.255.255.0

ip routing
```

서버 입장에서는 두 방식이 똑같이 보인다. 자기 VLAN의 게이트웨이 IP로 패킷을 보내면, 그 게이트웨이가 라우터의 서브인터페이스든 L3 스위치의 SVI든 알 바 아니다. 어느 쪽이든 게이트웨이가 VLAN을 넘겨주는 역할을 한다.

## 클라우드에서는 VLAN을 안 쓴다 — VPC와 서브넷

AWS, GCP를 쓰다 보면 VLAN을 설정할 일이 없다. 콘솔 어디에도 802.1Q나 트렁크 같은 게 없다. 대신 VPC와 서브넷으로 네트워크를 나눈다. 왜 클라우드는 VLAN을 버렸을까.

VLAN의 가장 큰 제약은 VID 12비트, 즉 4094개라는 한계다. 한 데이터센터 안에서 수만 개의 테넌트를 격리해야 하는 퍼블릭 클라우드 입장에서 4094개는 턱없이 부족하다. 그래서 클라우드 사업자는 VLAN 대신 오버레이 네트워크를 깐다. VXLAN이 대표적인데, VLAN ID를 24비트(VNI)로 늘려서 약 1600만 개의 세그먼트를 표현한다. VXLAN은 L2 프레임을 UDP로 캡슐화해서 L3 네트워크 위로 터널링하기 때문에, 물리 토폴로지에 묶이지 않고 데이터센터 어디에나 같은 세그먼트를 펼칠 수 있다.

사용자가 보는 VPC와 서브넷은 이 오버레이 위에 올라간 추상화다. 둘의 성격이 다르다.

- **VLAN**: L2 브로드캐스트 도메인을 나눈다. 같은 VLAN이면 L2로 직접 통신하고, 다른 VLAN은 라우터가 이어준다.
- **서브넷**: L3 주소 블록이다. AWS VPC 안에서 서브넷을 나누는 건 IP 대역을 가르는 일이지, 물리적인 브로드캐스트 도메인을 만드는 게 아니다. AWS 내부는 사실상 라우팅된 네트워크라서 전통적인 의미의 브로드캐스트가 없다. ARP도 흉내만 낼 뿐 실제로 브로드캐스트가 퍼지지 않는다.

그래서 클라우드에서 보안 격리를 하고 싶으면 VLAN을 나누는 게 아니라 Security Group과 NACL로 트래픽을 통제한다. 온프레미스에서 "VLAN을 나눠서 부서를 격리한다"는 발상은, 클라우드에서는 "서브넷을 나누고 Security Group으로 막는다"로 바뀐다. 다른 서브넷이라도 같은 VPC 안이면 라우팅 테이블상 기본으로 통신이 되기 때문에, 서브넷을 나눴다고 격리됐다고 생각하면 사고가 난다. 격리는 어디까지나 Security Group과 NACL의 몫이다.

정리하면 클라우드가 VLAN을 안 쓰는 이유는 세 가지다. 첫째, 4094개 한계로는 테넌트를 다 못 담는다. 둘째, VLAN은 물리 토폴로지에 묶여서 대규모 데이터센터에 펼치기 어렵다. 셋째, 오버레이(VXLAN) 위에 VPC라는 추상화를 얹으면 사용자가 물리 네트워크를 몰라도 자기만의 격리된 네트워크를 가질 수 있다.

## VLAN hopping — VLAN을 넘어가는 공격

VLAN은 격리 수단이라 보안 경계로 쓰이는데, 그 경계를 뚫는 공격이 VLAN hopping이다. 공격자가 자기가 속한 VLAN을 벗어나 다른 VLAN의 트래픽에 접근한다. 방식이 두 가지 있다.

### Switch Spoofing

많은 Cisco 스위치는 기본 포트 모드가 `dynamic auto` 또는 `dynamic desirable`이다. 이 모드의 포트는 상대가 트렁크를 맺자고 협상(DTP, Dynamic Trunking Protocol)을 걸어오면 트렁크로 전환된다. 공격자가 자기 노트북을 스위치인 척 꾸며서 DTP 협상을 걸면, Access 포트가 Trunk 포트로 바뀐다. 트렁크가 되는 순간 모든 VLAN의 트래픽이 공격자에게 흘러간다.

막는 법은 단순하다. 끝단 장치가 붙는 포트는 명시적으로 Access로 고정하고 DTP를 끈다.

```
interface GigabitEthernet0/1
 switchport mode access
 switchport access vlan 10
 switchport nonegotiate
```

`switchport nonegotiate`가 DTP 협상을 막는다. 끝단 포트가 협상으로 트렁크가 될 일이 없어진다.

### Double Tagging

Native VLAN의 동작을 악용하는 공격이다. 공격자가 프레임에 802.1Q 태그를 두 개 겹쳐 붙인다. 바깥 태그는 자기 포트의 Native VLAN(보통 1), 안쪽 태그는 공격 대상 VLAN(예: 20)으로 단다.

첫 번째 스위치가 이 프레임을 트렁크로 내보낼 때, 바깥 태그가 Native VLAN과 같으면 그 태그를 떼고 내보낸다. 그러면 안쪽 VLAN 20 태그만 남은 프레임이 트렁크를 타고 다음 스위치로 간다. 두 번째 스위치는 남은 VLAN 20 태그를 보고 그 프레임을 VLAN 20으로 흘려보낸다. 공격자는 VLAN 1에 있으면서 VLAN 20으로 프레임을 밀어 넣은 셈이다. 단방향이라 응답은 못 받지만, 공격용 패킷을 다른 VLAN에 주입하는 데는 충분하다.

이 공격이 성립하는 조건은 공격자의 포트와 트렁크의 Native VLAN이 같다는 점이다. 그래서 앞에서 말한 대로 Native VLAN을 아무도 안 쓰는 번호로 빼두면 바깥 태그를 떼는 조건이 안 맞아서 공격이 막힌다. 모든 트렁크 프레임에 태그를 강제하는 옵션(`vlan dot1q tag native`)을 켜는 방법도 있다.

VLAN을 보안 경계로 신뢰하려면 이런 설정을 기본으로 깔고 가야 한다. "VLAN으로 나눴으니 안전하다"는 가정은, DTP를 안 끄고 Native를 VLAN 1로 둔 상태에서는 깨진다. 실제로 격리됐다고 믿었던 관리 네트워크가 VLAN hopping으로 뚫린 사고가 적지 않다.

## 정리

VLAN은 물리 스위치 한 대를 여러 브로드캐스트 도메인으로 쪼개는 L2 기술이다. 802.1Q 태그 4바이트로 프레임에 VLAN 번호를 새기고, Access 포트는 태그 없이 끝단 장치를 받고 Trunk 포트는 태그를 붙여 여러 VLAN을 실어 나른다. 갈라놓은 VLAN을 다시 잇는 건 L3의 일이라 router-on-a-stick이나 L3 스위치의 SVI가 게이트웨이가 된다.

온프레미스에서는 여전히 핵심 기술이지만, 클라우드에서는 4094개 한계와 물리 토폴로지 종속 때문에 VXLAN 오버레이 위의 VPC/서브넷으로 대체됐다. 클라우드를 주로 쓰더라도 VLAN을 알아야 VPC가 왜 그렇게 생겼는지, 서브넷을 나눈 게 왜 격리가 아닌지 이해할 수 있다. 그리고 VLAN을 보안 경계로 쓸 거면 DTP를 끄고 Native VLAN을 따로 빼두는 것까지 해야 hopping에 안 뚫린다.
