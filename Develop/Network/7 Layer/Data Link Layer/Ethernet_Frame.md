# 이더넷 프레임 — L2가 실제로 주고받는 바이트

## 들어가며

서버 운영을 하다 보면 패킷 캡처를 떠야 할 때가 온다. 애플리케이션 로그상으로는 요청을 보냈다는데 상대는 못 받았다고 하고, 방화벽도 다 열려 있고, 라우팅 테이블도 멀쩡하다. 이쯤 되면 결국 `tcpdump`를 켜고 와이어 위에 실제로 뭐가 흘러다니는지 봐야 한다. 그런데 캡처 결과를 열어보면 IP 헤더 위에 또 뭔가가 붙어 있다. 출발지·목적지 MAC 주소, EtherType 같은 것들. 이게 이더넷 프레임이다.

IP가 동네 주소(어느 건물)라면 MAC은 그 건물 안에서 실제로 물건을 받는 사람이다. 같은 네트워크 세그먼트 안에서 호스트끼리 데이터를 주고받을 때 실제로 와이어를 타고 가는 단위가 이더넷 프레임이고, IP 패킷은 그 프레임의 페이로드로 실려 간다. L3 이상만 보다가 통신이 안 되는 문제를 만나면 결국 이 L2 레벨까지 내려와야 풀리는 경우가 많다.

이 문서는 이더넷 프레임의 구조, CSMA/CD와 충돌 도메인이 왜 옛날 이야기가 됐는지, 스위치가 MAC을 어떻게 학습하고 모를 때 어떻게 행동하는지, 그리고 점보 프레임과 MTU 문제를 정리한 글이다.

## 이더넷 프레임 구조

이더넷 프레임은 정해진 순서로 필드가 나열된 바이트 덩어리다. 흔히 쓰는 Ethernet II 프레임 기준으로 보면 이렇게 생겼다.

```
+----------+-----+--------+--------+-----------+------------------+-----+
| Preamble | SFD | 목적지  | 출발지  | EtherType |     Payload      | FCS |
|   7B     | 1B  |  MAC   |  MAC   |   2B      |   46 ~ 1500B     | 4B  |
|          |     |   6B   |   6B   |           |                  |     |
+----------+-----+--------+--------+-----------+------------------+-----+
            <------------ 여기부터 FCS까지가 캡처에 잡히는 프레임 ------->
```

### 프리앰블과 SFD

프리앰블(Preamble)은 `10101010`이 7바이트 반복되는 비트열이다. 받는 쪽 NIC가 들어오는 신호에 클록을 맞추라고 보내는 신호다. 그 뒤에 SFD(Start Frame Delimiter) 1바이트(`10101011`)가 붙어서 "여기부터 진짜 프레임 시작"이라고 알린다.

프리앰블과 SFD는 하드웨어가 처리하고 버린다. `tcpdump`로 캡처해도 이 부분은 안 보인다. 캡처에서 첫 바이트로 잡히는 건 목적지 MAC부터다. 처음 캡처를 뜨고 "프리앰블이 왜 안 보이지" 하는 경우가 있는데, 원래 안 보이는 게 정상이다.

### MAC 주소

목적지 MAC 6바이트, 출발지 MAC 6바이트가 이어진다. 목적지가 먼저 온다는 점이 중요하다. 스위치가 프레임을 받자마자 어디로 보낼지 결정하려면 목적지를 먼저 읽는 게 빠르기 때문이다.

MAC 주소는 48비트(6바이트)고 `00:1a:2b:3c:4d:5e` 같은 16진수로 쓴다. 앞 3바이트(OUI)는 제조사를 나타내고 뒤 3바이트는 제조사가 장비마다 부여한다. 그래서 캡처에서 MAC 앞부분만 봐도 어느 회사 NIC인지 대충 알 수 있다.

특수한 MAC도 알아둬야 한다. `ff:ff:ff:ff:ff:ff`는 브로드캐스트로, 세그먼트의 모든 호스트가 받는다. ARP Request가 이 주소로 나간다. 첫 바이트의 최하위 비트가 1이면 멀티캐스트다(`01:00:5e:...` 같은 IPv4 멀티캐스트).

### EtherType

2바이트짜리 EtherType은 페이로드가 어떤 상위 프로토콜인지 알려준다. 받는 쪽이 페이로드를 IP 스택으로 올릴지 ARP로 올릴지 판단하는 근거다.

| 값 (16진수) | 프로토콜 |
|---|---|
| 0x0800 | IPv4 |
| 0x0806 | ARP |
| 0x86DD | IPv6 |
| 0x8100 | 802.1Q VLAN 태그 |

여기서 헷갈리는 지점이 하나 있다. 이 자리의 2바이트 값이 1500(0x05DC) 이하면 EtherType이 아니라 페이로드 길이(Length)를 뜻한다. 이게 옛날 IEEE 802.3 프레임 방식이다. 1536(0x0600) 이상이면 EtherType으로 해석한다. 현대 네트워크는 거의 다 Ethernet II라 EtherType으로 보면 되지만, 캡처에서 가끔 Length로 잡히는 프레임을 만나면 이 차이를 떠올려야 한다.

802.1Q VLAN 태그가 붙으면 출발지 MAC 뒤에 4바이트 태그가 끼어들고 그 뒤에 실제 EtherType이 온다. VLAN 트렁크 포트를 캡처하면 프레임 길이가 4바이트씩 늘어나는 게 이 때문이다. VLAN 자체는 별도 문서에서 다룬다.

### 페이로드와 최소 길이

페이로드는 46~1500바이트다. 위쪽 1500이 우리가 흔히 말하는 MTU다. IP 패킷이 이 자리에 통째로 들어간다.

아래쪽 46바이트 최소 제한이 실무에서 가끔 발목을 잡는다. 페이로드가 46바이트보다 작으면 0으로 패딩을 채워 46바이트를 맞춘다. 프레임 전체가 최소 64바이트는 돼야 한다는 규칙 때문인데, 이건 뒤에 나오는 CSMA/CD와 관련이 있다. ARP 응답처럼 작은 프레임을 캡처하면 뒤에 의미 없는 0이 잔뜩 붙어 있는 걸 보게 된다. 그게 패딩이다.

### FCS

맨 뒤 4바이트는 FCS(Frame Check Sequence)다. CRC-32로 계산한 체크섬이고, 받는 쪽 NIC가 프레임이 전송 중 깨졌는지 검사한다. CRC가 안 맞으면 NIC가 프레임을 조용히 버린다. 상위 계층은 그런 프레임이 있었는지도 모른다.

FCS도 보통 NIC와 드라이버가 처리하기 때문에 `tcpdump` 출력에는 안 나온다. 깨진 프레임은 아예 캡처에 안 올라오는 경우가 많다. "분명히 패킷을 보냈는데 캡처에 안 잡힌다"면 중간에서 FCS 에러로 드롭됐을 가능성을 의심해야 한다. 이럴 땐 스위치 포트의 에러 카운터(`show interface`의 CRC errors)를 봐야 원인이 잡힌다.

## CSMA/CD와 충돌 도메인

CSMA/CD는 이더넷이 허브와 동축 케이블을 쓰던 시절의 핵심 동작이다. 지금은 거의 죽은 기술이지만, 왜 프레임 최소 길이가 64바이트인지, 충돌 도메인이라는 말이 왜 나왔는지 이해하려면 알아야 한다.

옛날 이더넷은 하나의 와이어를 여러 호스트가 공유했다. 동시에 두 호스트가 신호를 쏘면 신호가 겹쳐서 망가진다. 이게 충돌(Collision)이다. CSMA/CD는 이걸 다루는 규칙이다.

- **CS (Carrier Sense)**: 보내기 전에 와이어가 비었는지 먼저 듣는다.
- **MA (Multiple Access)**: 여러 호스트가 같은 매체를 공유한다.
- **CD (Collision Detection)**: 보내는 도중에도 계속 들으면서 충돌이 났는지 감지한다. 충돌을 감지하면 즉시 멈추고, 무작위 시간만큼 기다렸다가(백오프) 다시 시도한다.

충돌이 일어날 수 있는 호스트들의 범위를 충돌 도메인이라고 한다. 허브는 받은 신호를 모든 포트로 그대로 뿌리기 때문에 허브에 붙은 모든 호스트가 하나의 충돌 도메인이다. 호스트가 많아질수록 충돌이 잦아지고 실효 대역폭이 떨어진다.

프레임 최소 64바이트 규칙이 여기서 나온다. 충돌을 제대로 감지하려면 프레임이 와이어 끝까지 갔다가 충돌 신호가 돌아올 때까지 송신이 끝나면 안 된다. 그 최소 시간을 바이트로 환산한 게 64바이트다. 프레임이 이보다 짧으면 충돌이 났는데도 송신이 이미 끝나버려서 감지를 못 한다. 그래서 짧은 프레임은 패딩으로 64바이트를 채운다.

지금은 이게 거의 무의미하다. 스위치가 허브를 대체하면서 각 포트가 독립된 충돌 도메인이 됐고, 풀 듀플렉스에서는 송신과 수신 회선이 분리돼서 충돌 자체가 발생하지 않는다. 풀 듀플렉스 링크에서는 CSMA/CD가 아예 꺼진다. 그래도 캡처에서 64바이트짜리 패딩된 프레임을 보면 이 옛날 규칙의 흔적이라는 걸 알 수 있다.

## 스위치의 MAC 학습과 플러딩

스위치가 허브와 다른 점이 바로 이거다. 허브는 받은 걸 무조건 다 뿌리지만, 스위치는 목적지 MAC을 보고 필요한 포트로만 보낸다. 이걸 하려면 어떤 MAC이 어느 포트에 붙어 있는지 알아야 하고, 스위치는 이걸 스스로 학습한다.

학습 과정은 단순하다. 스위치는 MAC 주소 테이블(CAM 테이블)을 들고 있다. 프레임이 들어오면 출발지 MAC과 그 프레임이 들어온 포트를 짝지어 테이블에 적는다. "MAC X는 포트 3에 있다." 이렇게 들어오는 프레임마다 출발지를 보고 테이블을 채워간다.

```
초기 상태 (테이블 비어 있음)
호스트 A(포트1) → 호스트 B로 프레임 전송

1) 스위치: 출발지 A를 포트1로 학습
   [A → 포트1]
2) 목적지 B는 테이블에 없음 → 들어온 포트 빼고 전부로 플러딩
3) B가 응답하면 출발지 B를 포트2로 학습
   [A → 포트1, B → 포트2]
4) 이후 A↔B 통신은 해당 포트로만 전달 (플러딩 안 함)
```

목적지 MAC이 테이블에 있으면 그 포트로만 보낸다(포워딩). 테이블에 없으면(Unknown Unicast) 들어온 포트를 제외한 모든 포트로 내보낸다. 이게 플러딩이다. 브로드캐스트(`ff:ff:ff:ff:ff:ff`)도 마찬가지로 모든 포트로 플러딩한다.

CAM 테이블의 각 엔트리에는 수명(보통 5분)이 있다. 일정 시간 그 MAC에서 프레임이 안 오면 엔트리를 지운다. 호스트를 다른 포트로 옮겼는데 잠깐 통신이 이상하다가 정상으로 돌아오는 경우가 있는데, 오래된 엔트리가 만료되고 새 포트로 다시 학습될 때까지 걸리는 시간 때문이다.

실무에서 알아둘 함정이 몇 가지 있다.

플러딩이 많아지면 의심해야 한다. CAM 테이블이 꽉 차거나(MAC flooding 공격이 대표적) 엔트리가 자꾸 만료되면 스위치가 유니캐스트를 계속 플러딩한다. 그러면 원래 한 포트로만 갔어야 할 트래픽이 전 포트로 퍼져서 대역폭을 갉아먹고, 다른 포트에서 캡처를 떠도 남의 트래픽이 보이게 된다. 캡처할 때 "왜 나랑 상관없는 트래픽이 잡히지" 싶으면 플러딩 상황을 의심한다.

같은 MAC이 두 포트에서 번갈아 학습되면(MAC flapping) 루프가 의심된다. 스위치가 같은 출발지 MAC을 포트3에서 봤다가 포트5에서 봤다가 하면 어딘가 케이블이 루프로 연결됐을 가능성이 크다. 스위치 로그에 MAC flapping 경고가 뜨면 STP 설정과 물리 연결을 먼저 확인해야 한다.

## 점보 프레임과 MTU

표준 이더넷 페이로드 최대치는 1500바이트, 즉 MTU 1500이다. 점보 프레임은 이 페이로드를 9000바이트까지 키운 프레임이다. 표준은 아니지만 데이터센터 장비 대부분이 지원한다.

왜 키우냐면, 큰 데이터를 보낼 때 프레임 개수가 줄기 때문이다. 1500바이트씩 쪼개 보내는 것보다 9000바이트씩 보내면 헤더 오버헤드가 줄고, 프레임당 인터럽트 처리 횟수가 줄어 CPU 부담이 내려간다. iSCSI나 NFS처럼 큰 블록을 많이 옮기는 스토리지 트래픽, 백업 트래픽에서 체감 차이가 난다.

문제는 경로상의 모든 장비가 같은 MTU를 써야 한다는 점이다. 한 호스트만 MTU 9000으로 올리고 중간 스위치나 상대 호스트가 1500이면 통신이 깨진다. 이게 실무에서 정말 자주 겪는 함정이다.

증상이 고약하다. 작은 패킷(핑 기본 크기)은 잘 나가는데 큰 데이터 전송만 멈춘다. SSH 접속은 되는데 `scp`로 큰 파일을 보내면 중간에 멈추는 식이다. ping은 되니까 "네트워크는 멀쩡한데?" 하고 엉뚱한 데를 뒤지게 된다.

MTU 불일치는 패킷 크기를 키워서 ping으로 확인한다. DF(Don't Fragment) 비트를 세우고 큰 패킷을 보내본다.

```bash
# 리눅스: 페이로드 8972바이트 + ICMP 8B + IP 20B = 9000
# -M do 로 단편화 금지, 못 가면 에러
ping -M do -s 8972 192.168.1.10

# 정상이면:
# 8980 bytes from 192.168.1.10: icmp_seq=1 ttl=64 time=0.3 ms

# MTU가 모자라면:
# ping: local error: message too long, mtu=1500
```

`message too long, mtu=1500` 같은 에러가 나오면 경로 어딘가 MTU 1500 구간이 있다는 뜻이다. 어느 인터페이스의 MTU가 얼마인지는 이렇게 본다.

```bash
ip link show eth0
# eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 ...
#                                          ^^^^^^^^

# 점보 프레임으로 올리려면
sudo ip link set eth0 mtu 9000
```

TCP에는 PMTUD(Path MTU Discovery)라는 자동 조정 장치가 있어서, 경로 최소 MTU를 찾아 세그먼트 크기를 맞춘다. 문제는 PMTUD가 ICMP "Fragmentation Needed" 메시지에 의존하는데, 방화벽이 ICMP를 막아버리면 이 신호가 안 오고 PMTUD가 먹통이 된다(ICMP black hole). 그러면 위에서 말한 "작은 건 되고 큰 건 멈추는" 증상이 그대로 재현된다. 방화벽에서 ICMP를 무작정 다 막으면 안 되는 이유가 이거다.

## tcpdump로 프레임 헤더 확인하기

이론은 이쯤 하고, 실제로 프레임을 어떻게 들여다보는지 본다. `tcpdump`에 `-e` 옵션을 주면 이더넷 헤더(MAC, EtherType)까지 출력한다. 평소엔 안 보이는 L2 정보를 봐야 할 때 이 옵션이 필요하다.

```bash
# -e: 이더넷 헤더(MAC) 출력
# -n: 이름 해석 끄기 (IP/포트를 숫자로)
# -i eth0: 캡처할 인터페이스
sudo tcpdump -e -n -i eth0
```

출력은 이렇게 나온다.

```
14:02:11.123456 00:1a:2b:3c:4d:5e > 00:50:56:aa:bb:cc, ethertype IPv4 (0x0800),
  length 98: 192.168.1.20 > 192.168.1.10: ICMP echo request, id 1, seq 1, length 64
```

읽는 법은 이렇다. `00:1a:2b:3c:4d:5e > 00:50:56:aa:bb:cc`가 출발지 MAC → 목적지 MAC이다. `ethertype IPv4 (0x0800)`이 EtherType 필드 값으로, 페이로드가 IPv4라는 뜻이다. 그 뒤 `length 98`이 프레임 길이고, 이어서 IP 레벨 정보가 나온다.

ARP를 캡처하면 EtherType이 달라지는 걸 볼 수 있다.

```bash
sudo tcpdump -e -n -i eth0 arp
```

```
14:05:33.654321 00:1a:2b:3c:4d:5e > ff:ff:ff:ff:ff:ff, ethertype ARP (0x0806),
  length 42: Request who-has 192.168.1.10 tell 192.168.1.20
```

목적지 MAC이 `ff:ff:ff:ff:ff:ff` 브로드캐스트고 EtherType이 `ARP (0x0806)`인 게 보인다. ARP Request가 브로드캐스트로 나간다는 게 캡처상으로 확인되는 셈이다. length 42는 14바이트 이더넷 헤더 + 28바이트 ARP인데, 앞서 말한 최소 프레임 64바이트(헤더 14 + 페이로드 46)를 맞추려면 NIC가 패딩을 붙인다. 그 패딩은 보통 캡처에 안 나오거나 0으로 채워져 보인다.

특정 MAC만 골라 보고 싶으면 `ether host` 필터를 쓴다.

```bash
# 특정 MAC이 오가는 프레임만
sudo tcpdump -e -n -i eth0 ether host 00:1a:2b:3c:4d:5e

# 브로드캐스트 프레임만 (브로드캐스트 폭주 의심될 때)
sudo tcpdump -e -n -i eth0 ether broadcast

# 나중에 분석하려면 파일로 저장 (Wireshark로 열어 보면 편하다)
sudo tcpdump -e -n -i eth0 -w capture.pcap
```

캡처 파일을 떠두면 Wireshark에서 프레임을 바이트 단위로 펼쳐 볼 수 있다. 프리앰블·SFD·FCS는 NIC가 처리해서 안 보이지만, 목적지 MAC부터 페이로드까지는 그대로 펼쳐진다. L2 문제를 잡을 땐 이 캡처가 가장 확실한 증거가 된다.

## 정리

이더넷 프레임은 L2에서 실제로 와이어를 타는 단위다. 목적지·출발지 MAC, EtherType, 페이로드, FCS로 구성되고, 프리앰블·SFD·FCS는 하드웨어가 처리해서 캡처에는 안 잡힌다. CSMA/CD와 충돌 도메인은 허브 시절의 유산이고 스위치·풀 듀플렉스 환경에서는 거의 의미가 없지만, 64바이트 최소 프레임 규칙의 배경이 된다. 스위치는 출발지 MAC을 보고 CAM 테이블을 채우며, 모르는 목적지는 플러딩한다. 플러딩이 잦거나 MAC flapping이 보이면 테이블 포화나 루프를 의심한다. 점보 프레임은 처리량을 높이지만 경로 전체가 같은 MTU를 써야 하고, MTU 불일치는 "작은 건 되고 큰 건 멈추는" 증상으로 나타난다. 문제를 만나면 `tcpdump -e`로 프레임 헤더부터 확인하는 게 빠른 길이다.
