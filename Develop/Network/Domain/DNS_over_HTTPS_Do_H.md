# DNS over HTTPS (DoH) / DNS over TLS (DoT)

## 왜 평문 DNS가 문제였나

기존 DNS는 UDP 53번 포트로 평문을 그대로 주고받는다. `dig example.com` 한 번 날리면 중간 경로의 누구든 그 패킷을 읽는다. 카페 와이파이, ISP, 회사 게이트웨이 어디서든 "이 사람이 지금 어떤 도메인을 풀고 있는지"가 다 보인다.

문제는 두 가지다. 첫째는 도청이다. HTTPS로 본문은 암호화돼도 어떤 사이트에 접속하려는지는 DNS 질의에서 노출된다. 둘째는 변조다. 평문이니까 중간에서 응답을 바꿔치기할 수 있다. 공공 와이파이에서 가짜 응답을 끼워넣어 피싱 사이트로 보내거나, ISP가 특정 도메인 응답을 NXDOMAIN으로 바꿔 차단하는 식이다.

DoT와 DoH는 이 DNS 질의/응답 구간을 TLS로 감싸서 중간자가 못 읽고 못 바꾸게 만들었다. 이름은 비슷하지만 운영 입장에서 둘은 성격이 꽤 다르다.

## DoT와 DoH의 차이

### DNS over TLS (DoT) — 853 포트

DoT는 RFC 7858로 정의됐다. 동작은 단순하다. 평문 DNS를 TLS 세션 안에 그대로 넣는다. DNS 와이어 포맷(평소 53번에서 쓰던 그 바이너리)을 TLS 위에 얹어서 보낸다. 포트만 853번으로 바뀐다.

```
[클라이언트] --- TLS handshake (853) ---> [리졸버]
[클라이언트] --- DNS query (TLS 안) ---> [리졸버]
[클라이언트] <-- DNS response (TLS 안) -- [리졸버]
```

핵심은 전용 포트 853을 쓴다는 점이다. 네트워크 관리자 입장에서 853번 트래픽은 "이건 DoT다"라고 바로 식별된다. 막고 싶으면 853 포트를 차단하면 끝이다. 그러면 클라이언트는 보통 평문 53번으로 fallback 한다.

### DNS over HTTPS (DoH) — 443 포트

DoH는 RFC 8484로 정의됐다. DNS 질의를 HTTPS 요청 본문에 실어 보낸다. 일반 웹 트래픽과 똑같은 443번 포트를 쓴다. 이게 DoT와 갈리는 결정적 지점이다.

HTTP 메서드는 GET이나 POST 둘 다 가능하다. GET이면 base64url로 인코딩한 DNS 메시지를 쿼리 파라미터로 붙이고, POST면 본문에 바이너리로 넣는다.

```
GET /dns-query?dns=AAABAAABAAAAAAAAA3d3dwdleGFtcGxlA2NvbQAAAQAB HTTP/2
Host: dns.google
Accept: application/dns-message
```

```
POST /dns-query HTTP/2
Host: cloudflare-dns.com
Content-Type: application/dns-message
Content-Length: 33

(바이너리 DNS 메시지)
```

응답도 `application/dns-message` 콘텐츠 타입으로 바이너리 DNS 응답이 그대로 온다.

### 853 vs 443 — 이 포트 차이가 전부다

기술적으로 둘 다 TLS로 DNS를 감싸는 건 같다. 차이는 식별 가능성이다.

| | DoT | DoH |
|---|---|---|
| 포트 | 853 (전용) | 443 (HTTPS와 공유) |
| 식별 | 포트만 봐도 DNS인 줄 안다 | 일반 HTTPS와 구분 안 됨 |
| 차단 | 853 막으면 끝 | 막으려면 SNI/IP 단위로 막아야 함 |
| 표준 | RFC 7858 | RFC 8484 |
| 와이어 포맷 | DNS 그대로 | HTTP 안에 DNS |

DoT는 853이라는 깃발을 들고 다닌다. 그래서 기업/통신사가 통제하기 쉽다. DoH는 443에 섞여 들어가서 HTTPS 트래픽과 구분이 안 된다. 패킷만 봐서는 누가 구글에 접속하는 건지 구글 DNS에 질의하는 건지 알 수 없다. 이 "구분 안 됨"이 DoH의 프라이버시 장점이자, 동시에 망 관리자에게는 골칫거리가 된다.

## 동작 방식 — 첫 질의의 닭-달걀 문제

DoH/DoT를 쓸 때 한 가지 짚고 넘어갈 게 있다. 리졸버 주소를 도메인으로 지정하면 그 도메인은 어떻게 푸느냐다.

`https://dns.google/dns-query`로 DoH를 쓴다고 하자. `dns.google`을 IP로 풀어야 접속할 텐데, 그걸 풀려면 DNS가 필요하다. 그래서 DoH 클라이언트는 보통 부트스트랩 IP를 하드코딩하거나 설정에 IP를 직접 받는다. 구글은 `8.8.8.8`, 클라우드플레어는 `1.1.1.1`을 부트스트랩으로 쓴다.

```
1. 부트스트랩: 8.8.8.8 (하드코딩된 IP)로 먼저 TLS 연결
2. dns.google 인증서 검증 (SAN에 dns.google 포함 확인)
3. 이후 모든 DNS 질의를 이 연결로 처리
```

인증서 검증이 중요하다. 평문 DNS는 응답 출처를 검증할 방법이 없었지만, DoH/DoT는 TLS 인증서로 "내가 지금 진짜 클라우드플레어와 말하는 게 맞는지"를 확인한다. 중간자가 가짜 리졸버로 가로채면 인증서 검증에서 막힌다.

## 사내망에서 터지는 진짜 문제

여기가 운영하는 사람 입장에서 제일 골치 아픈 부분이다. DoH는 내부 DNS 인프라를 통째로 우회한다.

### 내부 도메인이 안 풀린다

회사 내부망에는 보통 `intra.company.local`, `git.internal` 같은 내부 전용 도메인이 있다. 이건 사내 DNS 서버만 알고, 외부 공용 리졸버(8.8.8.8, 1.1.1.1)는 모른다.

직원이 브라우저에서 DoH를 켜면 브라우저가 OS DNS 설정을 무시하고 직접 클라우드플레어로 질의를 보낸다. 그러면 `git.internal` 같은 내부 도메인은 외부 리졸버가 NXDOMAIN을 돌려주고, 사내 서비스 접속이 안 된다. 헬프데스크에 "VPN 켰는데 깃랩이 안 들어가져요" 문의가 들어오면 십중팔구 브라우저 DoH가 범인이다.

### 보안 필터링이 무력화된다

많은 회사가 DNS 단에서 악성 도메인, 피싱, C2 서버 도메인을 차단한다. 사내 DNS가 블랙리스트에 걸린 도메인을 NXDOMAIN으로 막는 방식이다. DNS 기반 차단은 구축이 싸고 전사 적용이 쉬워서 널리 쓰인다.

DoH가 켜지면 이 차단이 그냥 무시된다. 질의가 사내 DNS를 안 거치고 443번 HTTPS에 섞여 외부로 나가버리니까 방화벽에서 "DNS 질의"로 잡아낼 수도 없다. 보안팀 입장에서는 가시성이 사라진다. 어떤 직원 PC가 어떤 악성 도메인을 풀고 있는지 로그조차 안 남는다.

### 망 관리자가 대응하는 방법

업무망에서 DoH를 막는 방법이 몇 가지 있다.

Canary 도메인 차단. 파이어폭스는 `use-application-dns.net`이라는 특수 도메인을 먼저 조회한다. 이 도메인이 NXDOMAIN을 돌려주면 "이 네트워크는 DoH를 원하지 않는다"로 해석하고 DoH를 끈다. 사내 DNS에서 이 도메인을 NXDOMAIN 처리하면 파이어폭스 DoH가 자동으로 비활성화된다. 단, 이건 파이어폭스에만 통하고 크롬/엣지에는 안 먹힌다.

```
; 사내 DNS zone에 추가
use-application-dns.net. IN  CNAME  .   ; 또는 NXDOMAIN 응답
```

알려진 DoH 서버 IP/도메인 차단. 방화벽에서 `1.1.1.1`, `8.8.8.8`, `dns.google`, `cloudflare-dns.com` 같은 공개 DoH 제공자의 IP와 SNI를 막는다. 문제는 목록이 계속 늘어난다는 점이다. 새 DoH 제공자가 끝없이 생기고, 일부는 일반 CDN IP를 공유해서 막으면 멀쩡한 서비스까지 같이 죽는다.

그룹 정책으로 강제. 가장 확실한 건 관리하는 단말에 정책을 박는 것이다. 크롬/엣지는 `DnsOverHttpsMode`를 `off`로 강제하면 사용자가 못 켠다. 윈도우 도메인 환경이면 GPO로 일괄 배포한다.

```
; Chrome/Edge 정책 (관리 단말)
DnsOverHttpsMode = "off"
```

관리 안 되는 BYOD 단말은 이 정책이 안 먹히니까 결국 망 분리나 NAC로 가는 수밖에 없다.

## 브라우저 / OS 설정

### 크롬 / 엣지

`설정 → 개인정보 및 보안 → 보안 → 보안 DNS 사용`에서 켠다. 기본값은 "현재 서비스 제공업체"인데, 이건 OS에 설정된 DNS가 DoH를 지원하면 자동으로 업그레이드하는 모드다. 직접 제공자를 고르면(`Google`, `Cloudflare` 등) OS 설정을 무시하고 그 제공자로 직행한다.

크롬의 "자동 업그레이드" 모드는 그나마 덜 위험하다. OS에 설정된 리졸버가 DoH도 지원하는 경우에만 암호화로 올리기 때문에, 사내 DNS를 쓰면 사내 DNS의 DoH 엔드포인트로 ��다(있다면). 사내 DNS가 DoH를 지원 안 하면 평문 그대로 둔다. 문제는 사용자가 수동으로 제공자를 박아버릴 때다.

### 파이어폭스

파이어폭스가 제일 공격적이다. 일부 지역에서는 DoH가 기본 켜져 있고, 클라우드플레어가 기본 제공자다. `설정 → 개인정보 및 보안 → DNS over HTTPS`에서 본다. 위에 말한 canary 도메인을 존중하는 게 그나마 다행이다.

보호 수준이 세 단계 있다. 기본(Default), 강화(Increased), 최대(Max). Max로 두면 DoH 실패 시 평문 fallback 없이 그냥 실패시킨다. 보안은 세지만 내부망에서 쓰면 내부 도메인이 다 깨진다.

### OS 레벨

윈도우 11은 OS 차원에서 DoH를 지원한다. `설정 → 네트워크 → 어댑터 속성 → DNS 서버 할당`에서 암호화를 켤 수 있다. OS 레벨로 켜면 브라우저뿐 아니라 모든 앱의 DNS가 암호화된다.

리눅스는 `systemd-resolved`가 DoT를 지원한다. DoH는 기본 지원이 약해서 보통 별도 데몬을 쓴다.

```ini
# /etc/systemd/resolved.conf
[Resolve]
DNS=1.1.1.1#cloudflare-dns.com
DNSOverTLS=yes
```

`#cloudflare-dns.com`은 인증서 검증에 쓸 호스트명이다. 이걸 빼면 IP만 맞으면 통과시켜서 중간자 검증이 약해진다.

안드로이드는 9 이상에서 "비공개 DNS"(Private DNS)로 DoT를 지원한다. 호스트명만 넣으면 된다. 안드로이드는 DoH가 아니라 DoT를 쓴다는 점을 헷갈리지 말아야 한다.

## 캐시 동작 차이

평문 DNS에서는 OS 리졸버 캐시(`nscd`, 윈도우 DNS Client 서비스)가 시스템 전체 질의를 한곳에 모아 캐싱했다. `ipconfig /flushdns` 한 방이면 전부 비워졌다.

브라우저 DoH를 켜면 이 그림이 깨진다. 브라우저가 자체 DNS 캐시를 따로 들고 돈다. OS 캐시와 브라우저 캐시가 따로 노는 상태가 된다.

이게 트러블슈팅할 때 사람을 헷갈리게 한다. 레코드를 바꿨는데 `nslookup`(OS 리졸버 사용)에서는 새 값이 나오고 브라우저에서는 옛날 값이 잡히는 경우가 있다. OS 캐시 플러시해도 안 바뀐다. 브라우저가 DoH로 독자 캐시를 굴리니까 그렇다.

크롬은 `chrome://net-internals/#dns`에서 자체 DNS 캐시를 보고 비울 수 있다. 브라우저 DNS 문제를 의심하면 OS 캐시가 아니라 여기를 먼저 봐야 한다.

```
chrome://net-internals/#dns  → "Clear host cache"
```

캐시 키 단위도 다르다. OS 캐시는 시스템 전역이라 모든 프로세스가 공유하지만, 브라우저 DoH 캐시는 브라우저 프로세스 안에 갇혀 있다. 같은 PC에서 `curl`로 풀면 OS 리졸버를 타고, 크롬으로 풀면 DoH 캐시를 타서, 둘이 다른 IP를 잡는 상황도 생긴다. 마이그레이션 직후 "어떤 앱은 신규 서버 보고 어떤 앱은 구 서버 본다"는 현상의 흔한 원인이다.

TTL 해석도 미묘하게 다르다. DoH 응답에도 DNS와 똑같이 TTL이 실려 오지만, 브라우저가 자체 정책으로 최소/최대 TTL을 클램핑하는 경우가 있다. 권한 서버에서 TTL 30초로 줘도 브라우저가 60초 밑으로 안 내려가게 묶어버리면, 빠른 페일오버를 기대하고 짧은 TTL을 박아둔 게 안 먹힌다.

## 언제 쓰고 언제 막나

개인 단말, 카페 와이파이, 검열 회피 같은 환경에서 DoH는 분명히 쓸 값어치가 있다. 도청과 변조를 막아주고, 망 사업자의 DNS 조작에서 벗어난다.

반대로 회사 업무망 관리자 입장에서 DoH는 통제 밖으로 빠져나가는 트래픽이다. 내부 도메인 해석이 깨지고, DNS 기반 보안 필터가 무력화되고, 로그 가시성이 사라진다. 관리 단말은 그룹 정책으로 DoH를 끄고, 사내 DNS 자체를 DoT/DoH 지원으로 올려서 "암호화는 하되 경로는 사내를 거치게" 만드는 쪽이 현실적인 절충이다.

기억할 핵심은 853과 443의 차이다. DoT는 막기 쉽고 DoH는 막기 어렵다. 그 한 줄이 이 기술을 둘러싼 거의 모든 운영 논쟁의 출발점이다.
