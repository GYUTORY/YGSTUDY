웹 통신은 사용자가 브라우저에서 웹사이트에 접속할 때부터 페이지가 완전히 로드될 때까지의 모든 과정을 의미합니다. 이 과정은 여러 단계로 나뉘며, 각 단계마다 다양한 기술과 프로토콜이 사용됩니다.

### 🔑 핵심 용어 정리

| 용어 | 설명 |
|------|------|
| **프로토콜** | 컴퓨터 간 통신 규칙 (HTTP, HTTPS, TCP 등) |
| **도메인** | 웹사이트 주소 (예: www.google.com) |
| **IP 주소** | 인터넷상 컴퓨터의 고유 주소 (예: 192.168.1.1) |
| **포트** | 서비스 구분 번호 (HTTP: 80, HTTPS: 443) |
| **DNS** | 도메인 이름을 IP 주소로 변환하는 시스템 |
| **TCP** | 데이터 전송을 보장하는 통신 프로토콜 |
| **TLS/SSL** | 데이터 암호화를 위한 보안 프로토콜 |

---

사용자가 브라우저 주소창에 `https://www.example.com/login`을 입력하거나 링크를 클릭합니다.


**1. 요청 라인**
- `GET`: HTTP 메서드 (데이터 요청)
- `/login?redirect=home`: 요청 경로
- `HTTP/1.1`: HTTP 버전

**2. 요청 헤더**
- `Host`: 서버 도메인
- `User-Agent`: 브라우저 정보
- `Accept`: 클라이언트가 받을 수 있는 콘텐츠 타입
- `Cookie`: 이전 방문 정보

**3. 요청 본문**
- GET 요청의 경우 보통 비어있음
- POST 요청의 경우 데이터 포함

---

도메인 이름(`www.example.com`)을 실제 IP 주소(`192.168.1.100`)로 변환

클라이언트와 서버 간 안정적인 데이터 전송 채널 확립


```
인터넷 → 로드 밸런서 → 웹 서버 → 애플리케이션 서버 → 데이터베이스
```


#### 1단계: 미들웨어 체인 실행

**인증 미들웨어**
```
서버 → "이 사용자가 로그인했나?"
→ 세션 토큰 확인
→ JWT 토큰 검증
→ 권한 확인
```

**로깅 미들웨어**
```
서버 → 요청 정보 기록
- 요청 시간
- IP 주소
- 요청 경로
- 응답 시간
```

**CORS 미들웨어**
```
서버 → "다른 도메인에서 온 요청인가?"
→ 허용된 도메인인지 확인
→ 적절한 헤더 추가
```

#### 2단계: 라우터 처리
```
서버 → URL 패턴 매칭
요청: GET /api/users/123
매칭: /api/users/:id → UserController.getUser(id)
```

#### 3단계: 컨트롤러 실행
```javascript
// 예시: Node.js Express
app.get('/api/users/:id', async (req, res) => {
  const userId = req.params.id;
  const user = await UserService.findById(userId);
  res.json(user);
});
```

#### 4단계: 비즈니스 로직
```
컨트롤러 → 서비스 레이어 호출
서비스 → 데이터베이스 쿼리 실행
서비스 → 외부 API 호출 (필요시)
서비스 → 데이터 가공 및 반환
```

#### 5단계: 데이터베이스 연동
```sql
-- 예시 쿼리
SELECT * FROM users WHERE id = 123;
```


**Content-Type**
- `text/html`: HTML 문서
- `application/json`: JSON 데이터
- `image/png`: PNG 이미지
- `text/css`: CSS 스타일시트

**Cache-Control**
- `no-cache`: 캐시 사용 금지
- `max-age=3600`: 1시간 동안 캐시
- `private`: 브라우저만 캐시
- `public`: 모든 중간 서버가 캐시

**Set-Cookie**
- `sessionId=xyz789`: 세션 식별자
- `Path=/`: 쿠키 적용 경로
- `HttpOnly`: JavaScript 접근 금지
- `Secure`: HTTPS에서만 전송


#### 1단계: 데이터 준비
```
서비스 → 데이터베이스 결과 처리
→ 외부 API 결과 통합
→ 데이터 정제 및 변환
```

#### 2단계: 직렬화
```
서버 → JSON 직렬화
{
  "id": 123,
  "name": "홍길동"
}
```

#### 3단계: 헤더 설정
```
서버 → 응답 헤더 구성
→ 상태 코드 설정
→ 캐시 정책 적용
→ 보안 헤더 추가
```

#### 4단계: 압축 (선택적)
```
서버 → gzip 압축
→ Content-Encoding: gzip 헤더 추가
```

---

```
서버 → TCP 연결을 통해 응답 전송
→ 패킷 단위로 분할 전송
→ 전송 완료 확인
```


#### 1단계: 응답 수신
```
브라우저 → TCP 패킷 수신
→ 패킷 재조립
→ HTTP 응답 파싱
```

#### 2단계: 응답 헤더 분석
```
브라우저 → 상태 코드 확인
→ Content-Type 확인
→ 캐시 정책 적용
→ 쿠키 저장
```

#### 3단계: 콘텐츠 처리

**HTML 응답의 경우**
```
브라우저 → HTML 파싱
→ DOM 트리 생성
→ CSS 파싱 및 스타일 계산
→ JavaScript 실행
→ 렌더링 트리 생성
→ 레이아웃 계산
→ 화면에 픽셀 렌더링
```

**JSON 응답의 경우**
```
브라우저 → JSON 파싱
→ JavaScript 객체로 변환
→ 콜백 함수 실행
→ DOM 업데이트
```

#### 4단계: 추가 리소스 요청
```
브라우저 → 이미지, CSS, JS 파일 요청
→ 비동기 API 호출
→ 웹소켓 연결 (필요시)
```


### 🏗️ 서버 아키텍처 레벨

#### 1. 로드 밸런서 레벨
```
로드 밸런서 → 여러 웹 서버로 요청 분산
- 라운드 로빈: 순차적 분배
- 가중치 기반: 서버 성능에 따른 분배
- 헬스 체크: 서버 상태 확인
```

#### 2. 웹 서버 레벨 (Nginx/Apache)
```
웹 서버 → 정적 파일 서빙 또는 애플리케이션 서버 전달
- 정적 파일: 직접 서빙
- 동적 요청: 애플리케이션 서버로 전달
- SSL/TLS 종료
- 압축 및 캐싱
```

#### 3. 애플리케이션 서버 레벨
```
애플리케이션 서버 → 비즈니스 로직 처리
- 미들웨어 체인 실행
- 라우터에서 컨트롤러 선택
- 데이터베이스 연동
- 외부 API 호출
```

#### 4. 데이터베이스 레벨
```
데이터베이스 → 데이터 저장 및 조회
- SQL 쿼리 실행
- 트랜잭션 처리
- 인덱스 활용
- 결과 반환
```

### 🔄 라우팅 흐름 예시

#### 정적 파일 요청
```
요청: GET /images/logo.png
로드 밸런서 → 웹 서버 A
웹 서버 A → /var/www/html/images/logo.png 파일 전송
응답: 200 OK + 이미지 데이터
```

#### 동적 API 요청
```
요청: GET /api/users/123
로드 밸런서 → 웹 서버 B
웹 서버 B → 애플리케이션 서버 (포트 3000)
애플리케이션 서버 → UserController.getUser(123)
UserController → UserService.findById(123)
UserService → 데이터베이스 쿼리 실행
데이터베이스 → 사용자 데이터 반환
UserService → 데이터 가공
UserController → JSON 응답 생성
애플리케이션 서버 → HTTP 응답 전송
웹 서버 B → 클라이언트로 응답 전달
```

### 🛠️ 라우팅 설정 예시

#### Nginx 설정
```nginx
server {
    listen 80;
    server_name example.com;
    
    # 정적 파일 서빙
    location /images/ {
        root /var/www/html;
        expires 1y;
    }
    
    # API 요청을 애플리케이션 서버로 전달
    location /api/ {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

#### Express.js 라우팅
```javascript
// 미들웨어
app.use(express.json());
app.use(cors());
app.use(authMiddleware);

// 라우트 정의
app.get('/api/users/:id', async (req, res) => {
    try {
        const user = await UserService.findById(req.params.id);
        res.json(user);
    } catch (error) {
        res.status(500).json({ error: error.message });
    }
});
```

### 🌐 전체 통신 과정 시나리오

#### 시나리오: 사용자가 로그인 페이지에 접속

**1. 사용자 행동**
```
사용자가 브라우저에 "https://myapp.com/login" 입력
```

**2. 브라우저 요청 생성**
```
브라우저 → URL 파싱
- 프로토콜: https
- 도메인: myapp.com
- 경로: /login
브라우저 → HTTP 요청 생성
GET /login HTTP/1.1
Host: myapp.com
```

**3. DNS 조회**
```
브라우저 → DNS 캐시 확인 (실패)
브라우저 → ISP DNS 서버 조회
ISP DNS → 루트 DNS → TLD DNS → 권한 DNS
권한 DNS → "myapp.com = 203.0.113.10"
```

**4. TCP 연결**
```
브라우저 → 203.0.113.10:443으로 TCP 연결
TCP 3-way handshake 실행
TLS handshake 실행 (HTTPS)
```

**5. 서버 요청 처리**
```
로드 밸런서 → 웹 서버 선택
웹 서버 → 정적 파일 확인 (실패)
웹 서버 → 애플리케이션 서버로 전달
애플리케이션 서버 → 라우터에서 LoginController 선택
LoginController → 로그인 페이지 HTML 생성
```

**6. 응답 생성**
```
서버 → HTML 응답 생성
HTTP/1.1 200 OK
Content-Type: text/html
<html>
  <head><title>로그인</title></head>
  <body>...</body>
</html>
```

**7. 클라이언트 처리**
```
브라우저 → HTML 파싱
브라우저 → CSS, JS 파일 추가 요청
브라우저 → 페이지 렌더링 완료
```

### ⏱️ 전체 소요 시간

| 단계 | 시간 | 누적 시간 |
|------|------|-----------|
| DNS 조회 | 150ms | 150ms |
| TCP 연결 | 200ms | 350ms |
| 서버 처리 | 100ms | 450ms |
| 응답 전송 | 50ms | 500ms |
| 클라이언트 처리 | 200ms | 700ms |
| **총 시간** | **700ms** | **700ms** |

### 🔄 HTTP 메서드

| 메서드 | 용도 | 예시 |
|--------|------|------|
| **GET** | 데이터 조회 | 사용자 정보 가져오기 |
| **POST** | 데이터 생성 | 새 사용자 등록 |
| **PUT** | 데이터 전체 수정 | 사용자 정보 전체 변경 |
| **PATCH** | 데이터 부분 수정 | 사용자 이름만 변경 |
| **DELETE** | 데이터 삭제 | 사용자 삭제 |

### 🌐 웹 서버 vs 애플리케이션 서버

| 구분 | 웹 서버 | 애플리케이션 서버 |
|------|---------|-------------------|
| **역할** | 정적 파일 서빙 | 비즈니스 로직 처리 |
| **예시** | Nginx, Apache | Node.js, Django, Spring |
| **처리 속도** | 빠름 | 상대적으로 느림 |
| **메모리 사용** | 적음 | 많음 |

### 🔒 HTTP vs HTTPS

| 구분 | HTTP | HTTPS |
|------|------|-------|
| **암호화** | 없음 | TLS/SSL 암호화 |
| **포트** | 80 | 443 |
| **보안** | 취약 | 안전 |
| **속도** | 빠름 | 상대적으로 느림 |

---


1. **DNS 조회는 도메인을 IP로 변환하는 과정**
2. **TCP 연결은 안정적인 데이터 전송을 위한 핸드셰이크**
3. **웹 서버는 정적 파일, 애플리케이션 서버는 동적 로직 처리**
4. **미들웨어는 요청/응답을 가로채서 추가 처리**
5. **캐싱은 성능 향상을 위한 핵심 기술**
6. **HTTPS는 HTTP에 보안 레이어를 추가한 프로토콜**

- GET 요청의 경우 보통 비어있음
- POST 요청의 경우 데이터 포함

```
브라우저 → 203.0.113.10:443으로 TCP 연결
TCP 3-way handshake 실행
TLS handshake 실행 (HTTPS)
```

**5. 서버 요청 처리**
```
로드 밸런서 → 웹 서버 선택
웹 서버 → 정적 파일 확인 (실패)

LoginController → 로그인 페이지 HTML 생성
```

**6. 응답 생성**
```
서버 → HTML 응답 생성
HTTP/1.1 200 OK
Content-Type: text/html
<html>
  <head><title>로그인</title></head>
  <body>...</body>
</html>
```

**7. 클라이언트 처리**
```
브라우저 → HTML 파싱
브라우저 → CSS, JS 파일 추가 요청
브라우저 → 페이지 렌더링 완료
```

### 🌐 웹 서버 vs 애플리케이션 서버

|------|---------|-------------------|

### 🔒 HTTP vs HTTPS

| 구분 | HTTP | HTTPS |
|------|------|-------|
| **암호화** | 없음 | TLS/SSL 암호화 |
| **포트** | 80 | 443 |
| **보안** | 취약 | 안전 |
| **속도** | 빠름 | 상대적으로 느림 |

1. **DNS 조회는 도메인을 IP로 변환하는 과정**
2. **TCP 연결은 안정적인 데이터 전송을 위한 핸드셰이크**
3. **웹 서버는 정적 파일, 애플리케이션 서버는 동적 로직 처리**
4. **미들웨어는 요청/응답을 가로채서 추가 처리**
5. **캐싱은 성능 향상을 위한 핵심 기술**
6. **HTTPS는 HTTP에 보안 레이어를 추가한 프로토콜**

## 1단계: 브라우저 요청 생성

### 🔍 URL 파싱 과정

브라우저는 입력된 URL을 다음과 같이 분석합니다:

```
https://www.example.com:443/login?redirect=home#section1
```

| 구성 요소 | 값 | 설명 |
|-----------|-----|------|
| **프로토콜** | `https://` | 보안 통신 방식 |
| **도메인** | `www.example.com` | 웹사이트 주소 |
| **포트** | `:443` | HTTPS 기본 포트 (생략 가능) |
| **경로** | `/login` | 서버 내 특정 페이지 |
| **쿼리** | `?redirect=home` | 추가 정보 전달 |
| **프래그먼트** | `#section1` | 페이지 내 특정 위치 |

### 📝 HTTP 요청 메시지 생성

## 2단계: DNS 조회

### 🔄 DNS 조회 과정

#### 1단계: 브라우저 캐시 확인
```
브라우저 → "www.example.com의 IP를 알고 있나?"
캐시 → "네, 192.168.1.100입니다!" (캐시 히트)
```

#### 2단계: 운영체제 캐시 확인
```
브라우저 → 운영체제: "www.example.com의 IP가 있나?"
운영체제 → hosts 파일 확인
운영체제 → DNS 캐시 확인
```

#### 3단계: 라우터 캐시 확인
```
브라우저 → 라우터: "www.example.com의 IP가 있나?"
라우터 → DNS 캐시 확인
```

#### 4단계: ISP DNS 서버 조회
```
브라우저 → ISP DNS 서버: "www.example.com의 IP가 뭐야?"
ISP DNS → "모르겠어, 루트 DNS 서버에 물어볼게"
```

#### 5단계: 계층적 DNS 조회

**5-1. 루트 DNS 서버 조회**
```
ISP DNS → 루트 DNS: ".com 도메인은 어디서 관리해?"
루트 DNS → ISP DNS: ".com은 TLD 서버에서 관리해"
```

**5-2. TLD DNS 서버 조회**
```
ISP DNS → TLD DNS: "example.com은 어디서 관리해?"
TLD DNS → ISP DNS: "example.com은 권한 있는 DNS 서버에서 관리해"
```

**5-3. 권한 있는 DNS 서버 조회**
```
ISP DNS → 권한 DNS: "www.example.com의 IP가 뭐야?"
권한 DNS → ISP DNS: "192.168.1.100이야"
```

### ⏱️ DNS 조회 시간
- 캐시 히트: 0-1ms
- ISP DNS 조회: 10-50ms
- 전체 DNS 조회: 100-300ms

## 3단계: TCP/IP 연결

### 🤝 TCP 3-Way Handshake

#### 1단계: SYN (Synchronize)
```
클라이언트 → 서버: "연결하고 싶어요" (SYN 플래그 설정)
```

#### 2단계: SYN-ACK (Synchronize-Acknowledgment)
```
서버 → 클라이언트: "연결해도 돼요, 확인했어요" (SYN+ACK 플래그 설정)
```

#### 3단계: ACK (Acknowledgment)
```
클라이언트 → 서버: "알겠어요, 연결 시작해요" (ACK 플래그 설정)
```

### 🔒 HTTPS의 경우: TLS Handshake 추가

#### 1단계: Client Hello
```
클라이언트 → 서버: "저는 이런 암호화 방식을 지원해요"
- 지원하는 암호화 알고리즘 목록
- 클라이언트 랜덤 값
- 세션 ID (재연결용)
```

#### 2단계: Server Hello
```
서버 → 클라이언트: "이 암호화 방식을 사용할게요"
- 선택한 암호화 알고리즘
- 서버 인증서
- 서버 랜덤 값
```

#### 3단계: 인증서 검증
```
클라이언트 → 인증서 검증
- 인증서 유효성 확인
- 인증서 체인 검증
- 도메인 일치 확인
```

#### 4단계: 키 교환
```
클라이언트 → 서버: "이 대칭키로 암호화할게요"
- Pre-Master Secret 생성
- 서버 공개키로 암호화하여 전송
```

#### 5단계: 암호화 통신 시작
```
양쪽 → Master Secret 생성
양쪽 → 암호화 통신 시작
```

### ⏱️ 연결 시간
- TCP 연결: 10-50ms
- TLS Handshake: 100-300ms
- 총 연결 시간: 110-350ms

```
- 동적 요청인가? → 애플리케이션 서버로 전달
```

#### 3단계: 정적 파일 서빙 (예시)
```
요청: GET /images/logo.png
처리: /var/www/html/images/logo.png 파일 전송
응답: 200 OK + 이미지 데이터
```

#### 4단계: 동적 요청 전달 (예시)
```
요청: GET /api/users/123
처리: 애플리케이션 서버(포트 3000)로 전달
```

### ⏱️ 서버 처리 시간
- 웹 서버 처리: 1-10ms
- 애플리케이션 서버: 10-100ms
- 데이터베이스 쿼리: 5-50ms
- 총 서버 처리: 16-160ms
