## 무한 시퀀스라는 게 왜 가능한가

배열에 자연수를 끝없이 담을 수는 없다. 메모리가 유한하기 때문이다. 그런데 제너레이터를 쓰면 "끝이 없는 수열"을 코드로 표현할 수 있다. 비결은 값을 미리 다 만들어 두지 않고, `next()`가 호출될 때마다 그 순간 필요한 값 하나만 계산해서 넘기는 데 있다. 이걸 지연 평가(lazy evaluation)라고 부른다.

핵심은 제너레이터 함수가 `yield`에서 멈춘다는 점이다. 멈춘 상태로 함수의 지역 변수와 실행 위치를 그대로 들고 있다가, 다음 `next()`가 오면 멈췄던 자리에서 이어서 실행한다. `while (true)` 안에 `yield`가 있어도 무한 루프가 한 번에 다 돌지 않는 이유가 이거다. `yield`까지만 실행하고 제어권을 호출자에게 돌려준다.

```javascript
function* naturalNumbers() {
    let n = 1;
    while (true) {
        yield n++;
    }
}

const nums = naturalNumbers();
console.log(nums.next().value); // 1
console.log(nums.next().value); // 2
console.log(nums.next().value); // 3
```

`while (true)`가 있지만 브라우저가 멈추지 않는다. `next()`를 부른 만큼만 루프가 돈다. 값을 더 안 꺼내면 함수는 두 번째 `yield`에서 멈춘 채로 가만히 있는다.

## 가장 흔한 사고: 무한 제너레이터를 통째로 펼치기

여기서 실무자가 제일 많이 당하는 사고가 있다. 무한 제너레이터를 `for...of`, 스프레드(`...`), `Array.from`에 그냥 넣는 것이다. 이 셋은 이터레이터가 `done: true`를 줄 때까지 끝까지 값을 빨아들인다. 무한 시퀀스는 절대 `done`이 되지 않으므로 영원히 멈추지 않는다.

```javascript
// 셋 다 절대 끝나지 않는다. 탭이 멈추거나 Node 프로세스가 죽는다.
const arr = [...naturalNumbers()];        // 메모리 소진까지 무한 루프
const arr2 = Array.from(naturalNumbers()); // 동일
for (const n of naturalNumbers()) {
    console.log(n); // break가 없으면 영원히 돈다
}
```

`for...of`는 그나마 `break`로 빠져나올 수 있다. 하지만 스프레드와 `Array.from`은 중간에 멈출 방법이 없다. 한 번 호출하면 끝이다. 로컬에서 무심코 `console.log([...primes()])` 같은 걸 찍었다가 노트북 팬이 돌고 Node가 `JavaScript heap out of memory`로 죽는 경험을 한 번쯤 하게 된다.

해결은 간단하다. 무한 시퀀스는 항상 "앞에서 n개만" 잘라서 쓴다. `take(n)` 헬퍼를 만들어 둔다.

```javascript
function* take(n, sequence) {
    let count = 0;
    for (const item of sequence) {
        if (count >= n) return;
        yield item;
        count++;
    }
}

console.log([...take(5, naturalNumbers())]); // [1, 2, 3, 4, 5]
```

`take` 자체도 제너레이터라서 `n`개를 다 꺼낼 때까지만 원본 시퀀스에서 값을 당겨온다. `return`이 실행되면 더 이상 원본 `next()`를 부르지 않는다. 무한 시퀀스를 다룰 때는 스프레드나 `Array.from`을 쓰기 전에 항상 `take`로 한 번 자르는 습관을 들여야 한다.

## 이터레이터를 직접 만들면 무슨 일이 일어나는가

제너레이터가 편하긴 한데, 그 밑에서 실제로 동작하는 이터레이터 프로토콜을 알아두면 동작이 명확해진다. 객체에 `[Symbol.iterator]` 메서드가 있고, 그게 `next()`를 가진 객체를 돌려주면 그 객체는 이터러블이다. `for...of`와 스프레드는 내부적으로 `[Symbol.iterator]()`를 호출해서 이터레이터를 얻은 뒤 `next()`를 반복한다.

```javascript
const naturals = {
    [Symbol.iterator]() {
        let n = 1;
        return {
            next() {
                return { value: n++, done: false };
            }
        };
    }
};

console.log([...take(3, naturals)]); // [1, 2, 3]
```

같은 걸 제너레이터로 쓰면 이렇게 줄어든다.

```javascript
function* naturalsGen() {
    let n = 1;
    while (true) yield n++;
}
```

둘이 하는 일은 같지만 차이가 있다. 커스텀 이터레이터는 `value`와 `done`을 직접 만들어 반환해야 하고, 상태(`n`)를 클로저로 직접 관리한다. 제너레이터는 `function*` 키워드만 붙이면 `[Symbol.iterator]`, `next`, `return`, `throw`를 엔진이 다 만들어 준다. `yield`에서 멈췄다 재개하는 상태 저장도 엔진이 알아서 한다. 실무에서 무한 시퀀스를 손으로 짤 일이 있으면 거의 항상 제너레이터가 맞다. 커스텀 이터레이터를 직접 구현하는 경우는 이미 다른 자료구조(트리, 연결 리스트 등)가 있고 거기에 이터러블 인터페이스만 얹고 싶을 때 정도다.

## next()에 인자를 넘겨 양방향으로 주고받기

`yield`는 값을 밖으로 내보내기만 하는 게 아니다. `yield` 표현식 자체가 값을 반환하는데, 그 값은 다음 `next(arg)`에 넘긴 `arg`가 된다. 즉 제너레이터 바깥에서 안으로 값을 밀어넣을 수 있다.

```javascript
function* counter() {
    let step = 1;
    while (true) {
        const reset = yield step;
        if (reset !== undefined) {
            step = reset; // 바깥에서 넘긴 값으로 카운터를 다시 맞춘다
        } else {
            step++;
        }
    }
}

const g = counter();
console.log(g.next().value);   // 1
console.log(g.next().value);   // 2
console.log(g.next(10).value); // 10  — yield가 10을 반환해서 step이 10이 됨
console.log(g.next().value);   // 11
```

여기서 자주 헷갈리는 지점이 하나 있다. 첫 번째 `next()`에 넘긴 인자는 버려진다. 첫 `next()`는 함수를 처음부터 첫 `yield`까지 실행시키는 역할이라, 받아갈 `yield` 표현식이 아직 없기 때문이다. 그래서 `g.next(100)`을 맨 처음에 호출해도 `100`은 어디에도 안 들어간다. 양방향 통신을 쓸 거면 첫 `next()`는 항상 인자 없이 부르는 걸 전제로 코드를 짜야 한다.

## 무한 루프를 끊을 때 자원 정리하기

무한 제너레이터 안에서 파일 핸들이나 커넥션 같은 자원을 잡았다면, 루프가 끊길 때 그걸 닫아야 한다. 제너레이터는 `return()`과 `throw()` 메서드로 바깥에서 중단시킬 수 있고, 이때 `try...finally`의 `finally` 블록이 실행된다.

```javascript
function* readChunks(source) {
    source.open();
    try {
        while (true) {
            yield source.read();
        }
    } finally {
        source.close(); // return()/throw()/break 어느 경로로 끝나도 실행된다
    }
}

const g = readChunks(source);
g.next();   // 첫 청크
g.next();   // 다음 청크
g.return(); // finally 실행 → source.close(), { value: undefined, done: true }
```

`for...of`로 돌리다가 `break`나 예외로 빠져나가면 엔진이 자동으로 이터레이터의 `return()`을 호출해 준다. 그래서 `finally`만 제대로 걸어두면 `take(3, readChunks(source))`처럼 잘라 써도 3개를 다 읽은 뒤 `close()`가 호출된다. 무한 시퀀스에서 외부 자원을 다룰 때는 `yield`를 반드시 `try...finally`로 감싸는 걸 기본으로 잡아야 한다. 안 그러면 시퀀스를 중간에 버렸을 때 커넥션이 새는 식으로 문제가 생긴다.

## take · filter · map 파이프라인

무한 시퀀스의 진짜 쓸모는 작은 변환 제너레이터를 엮어 파이프라인을 만들 때 나온다. `filter`, `map`, `take`를 각각 제너레이터로 만들어 두면, 무한 시퀀스를 원본으로 두고 필요한 부분만 잘라서 가공할 수 있다.

```javascript
function* filter(predicate, sequence) {
    for (const item of sequence) {
        if (predicate(item)) yield item;
    }
}

function* map(transform, sequence) {
    for (const item of sequence) {
        yield transform(item);
    }
}

// 자연수 → 짝수만 → 제곱 → 앞 5개
const result = take(5,
    map(n => n * n,
        filter(n => n % 2 === 0, naturalNumbers())
    )
);

console.log([...result]); // [4, 16, 36, 64, 100]
```

이 파이프라인의 동작 방식이 지연 평가의 핵심을 보여준다. 배열 메서드 체인(`.filter().map().slice()`)과 달리, 이 코드는 중간 배열을 하나도 안 만든다. `take`가 값 하나를 요청하면 그 요청이 `map`으로, `map`이 다시 `filter`로, `filter`가 `naturalNumbers`로 거슬러 올라간다. 자연수 하나가 필터를 통과해 제곱된 다음 `take`까지 도착하면 그게 결과 하나가 된다. 즉 값이 파이프라인을 한 칸씩 통과하는 구조다. `take`가 5개를 다 받으면 거기서 멈추고, 원본 자연수 시퀀스도 정확히 필요한 만큼만 돌아간다. 짝수 5개를 얻으려고 자연수를 10번까지만 당겨오고 그 이상은 건드리지 않는다.

이 방식이 메모리 면에서 유리한 이유가 여기 있다. 전체를 메모리에 올린 뒤 거르는 게 아니라, 한 번에 값 하나씩만 흘려보내기 때문이다. 입력이 무한이어도 동시에 메모리에 떠 있는 값은 파이프라인 한 칸당 하나뿐이다.

## 동작하는 소수 시퀀스 (incremental sieve)

소수를 무한히 뽑는 제너레이터는 무한 시퀀스 예제로 자주 나오는데, 에라토스테네스의 체를 그대로 무한 버전으로 옮기려다 로직이 틀리는 경우가 많다. 동작하는 방식은 이렇다. 각 소수가 "자기 배수 중 다음에 지워야 할 위치"를 Map에 적어두고, 합성수에 도달하면 그 위치를 다음 배수로 한 칸 옮긴다.

```javascript
function* primes() {
    // composite(합성수) -> 그 수를 배수로 갖는 소수들의 목록
    const sieve = new Map();
    let n = 2;

    while (true) {
        const factors = sieve.get(n);
        if (factors === undefined) {
            // sieve에 없으면 n은 소수다
            yield n;
            // n의 배수는 n*n부터 표시하기 시작한다 (n*2 ~ n*(n-1)은 더 작은 소수가 이미 거른다)
            sieve.set(n * n, [n]);
        } else {
            // n은 합성수. n을 만든 각 소수를 다음 배수 위치로 옮긴다
            for (const p of factors) {
                const next = n + p;
                if (sieve.has(next)) {
                    sieve.get(next).push(p);
                } else {
                    sieve.set(next, [p]);
                }
            }
            sieve.delete(n); // 현재 위치 정보는 옮겼으니 지운다
        }
        n++;
    }
}

console.log([...take(6, primes())]); // [2, 3, 5, 7, 11, 13]
```

손으로 따라가 보면 동작이 보인다. `n=2`는 비어 있으니 소수, `sieve[4]=[2]`를 적는다. `n=3`도 소수, `sieve[9]=[3]`. `n=4`는 `[2]`가 적혀 있으니 합성수, `2`를 `4+2=6`으로 옮겨 `sieve[6]=[2]`, `sieve[4]`는 삭제. `n=5`는 소수. `n=6`은 `[2]`라 합성수, `2`를 `8`로 옮김. 이렇게 각 소수가 자기 배수 위에서 한 칸씩 걸어다니며 합성수를 표시한다.

원래 문서에 있던 구현은 `sieve.set(num * num, [num])`까지는 맞지만 합성수를 옮기는 부분의 인덱싱이 어긋나서 일부 합성수를 소수로 잘못 뱉는 문제가 있었다. 소수 생성기는 손으로 처음 10개 정도를 직접 따라가 검증하지 않으면 미묘하게 틀린 채로 넘어가기 쉽다.

## 메모리 누수: sieve와 캐시는 무한히 자란다

위 `primes`에는 짚고 넘어갈 함정이 있다. `sieve` Map은 계속 자란다. 소수를 하나 뽑을 때마다 `sieve`에 새 항목이 들어가고, 합성수에서 위치를 옮길 때도 항목이 늘었다 줄었다 한다. 소수를 충분히 많이 뽑으면 `sieve`가 차지하는 메모리도 그만큼 커진다. 무한 시퀀스라고 메모리가 상수인 게 아니라, 내부 상태를 들고 있는 시퀀스는 그 상태가 누적된다.

이건 "값을 미리 만들지 않으니 메모리를 안 쓴다"는 흔한 오해의 반대 사례다. 자연수나 피보나치처럼 변수 몇 개만 들고 있는 시퀀스는 상수 메모리다. 하지만 소수 체나 캐시를 들고 있는 시퀀스는 진행할수록 메모리를 더 먹는다. 장시간 도는 서비스에서 이런 시퀀스를 계속 진행시키면 천천히 메모리가 차오른다. 무한히 뽑을 게 아니라면 주기적으로 시퀀스를 새로 만들어 상태를 비우거나, 애초에 상한을 두고 `take`로 잘라 쓰는 게 안전하다.

## 캐시로 값을 다시 꺼내 쓰기

피보나치 같은 시퀀스는 제너레이터 자체가 이미 각 값을 한 번씩만 계산한다. `[prev, curr]`만 들고 다음 값을 만들기 때문에 재계산이 없다. 그래서 피보나치 제너레이터에 메모이제이션을 얹는 건 의미가 없다. 원래 문서의 `memoizedFibonacci`는 `memo`에 값을 넣기만 하고 어디서도 읽지 않아서 동작상 아무 역할을 못 했다.

캐시가 실제로 쓸모 있는 경우는 따로 있다. 이미 한 번 흘려보낸 값을 인덱스로 다시 꺼내 쓰고 싶을 때다. 제너레이터는 한 방향으로만 흐르고 뒤로 못 가니, 지나간 값을 배열에 적어두면 같은 인덱스를 재계산 없이 돌려줄 수 있다.

```javascript
function cached(generatorFn) {
    const cache = [];
    let gen = null;

    return function at(index) {
        if (gen === null) gen = generatorFn();
        while (cache.length <= index) {
            const { value, done } = gen.next();
            if (done) return undefined;
            cache.push(value);
        }
        return cache[index];
    };
}

const fibAt = cached(fibonacci);
console.log(fibAt(10)); // 89  — 여기까지 한 번 계산하며 cache 채움
console.log(fibAt(3));  // 3   — cache에서 즉시 반환, 재계산 없음
console.log(fibAt(20)); // 10946 — 11~20만 추가로 계산
```

`fibonacci`는 앞에서 본 표준 제너레이터다.

```javascript
function* fibonacci() {
    let [prev, curr] = [0, 1];
    while (true) {
        yield curr;
        [prev, curr] = [curr, prev + curr];
    }
}
```

`cached`는 요청한 인덱스까지만 제너레이터를 진행시키고, 그 사이 값들을 `cache`에 쌓는다. 이미 본 인덱스는 배열에서 바로 꺼낸다. 다만 여기서도 앞 절의 함정이 그대로 적용된다. `cache`는 가장 멀리 요청한 인덱스만큼 자란다. `fibAt(1_000_000)`을 부르면 백만 개를 배열에 들고 있게 된다. 캐시는 공짜가 아니다.

## 애니메이션 프레임과 requestAnimationFrame

제너레이터로 프레임 정보를 뽑는 예제를 보면 `Date.now()`를 `yield`하는 코드가 자주 보이는데, 이것만으로는 애니메이션이 되지 않는다. 제너레이터는 스스로 시간에 맞춰 멈췄다 깨어나지 않는다. `next()`를 누가 불러줘야 한 칸 진행한다. 화면 갱신 주기에 맞춰 `next()`를 불러주는 건 `requestAnimationFrame`의 몫이다.

```javascript
function* frameTimings() {
    let last = performance.now();
    let frame = 0;
    while (true) {
        const now = performance.now();
        yield { frame: frame++, delta: now - last };
        last = now;
    }
}

const timings = frameTimings();

function loop() {
    const { value } = timings.next();
    // value.delta(이전 프레임과의 시간 간격)로 이동량을 계산한다
    element.style.left = `${parseFloat(element.style.left || 0) + value.delta * 0.1}px`;
    requestAnimationFrame(loop);
}

requestAnimationFrame(loop);
```

`requestAnimationFrame`이 매 프레임마다 `loop`을 부르고, `loop`이 `timings.next()`로 제너레이터를 한 칸 진행시킨다. 제너레이터는 프레임 번호와 직전 프레임과의 시간 차이(`delta`)를 계산해서 넘긴다. 시간 흐름을 만드는 건 `requestAnimationFrame`이고, 제너레이터는 그 시점마다 필요한 상태를 계산해 주는 역할만 한다. 둘을 분리해서 보면 왜 `yield Date.now()`만으로는 아무 일도 안 일어나는지 이해된다. 진행시켜 주는 주체가 없으면 제너레이터는 첫 `yield`에서 멈춰 있을 뿐이다.

`delta`를 쓰는 이유도 짚어둘 만하다. 모니터 주사율이 60Hz든 144Hz든 프레임 간격이 다르므로, 프레임 수가 아니라 실제 경과 시간으로 이동량을 계산해야 기기와 무관하게 같은 속도로 움직인다.

## 정리할 때 기억할 것

무한 시퀀스를 다룰 때 실수가 나오는 지점은 거의 정해져 있다. 무한 제너레이터를 스프레드나 `Array.from`에 통째로 넣으면 멈추니 항상 `take`로 자른다. 자원을 잡았으면 `try...finally`로 감싸 중단 시 정리되게 한다. 내부 상태(체, 캐시)를 들고 있는 시퀀스는 진행할수록 메모리가 자란다는 걸 잊지 않는다. 그리고 소수 생성기처럼 로직이 복잡한 시퀀스는 앞 몇 개를 손으로 검증하고 넘어간다.
