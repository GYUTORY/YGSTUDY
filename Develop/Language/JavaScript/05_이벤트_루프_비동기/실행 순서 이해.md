# JavaScript 실행 순서 이해하기

JavaScript는 단일 스레드로 동작한다. 한 번에 하나의 작업만 콜 스택에서 실행한다. 그런데도 타이머, 네트워크 요청, 이벤트 핸들러가 동시에 도는 것처럼 보이는 이유는 콜 스택 바깥에 큐와 이벤트 루프가 따로 있기 때문이다. 실행 순서를 예측하려면 콜 스택, 매크로태스크 큐, 마이크로태스크 큐, 이벤트 루프 네 가지가 어떻게 맞물리는지를 알아야 한다.

순서를 외우려 하지 말고 다음 한 문장으로 기억한다. **콜 스택이 비면 이벤트 루프가 마이크로태스크 큐를 전부 비우고, 그다음 매크로태스크를 하나만 꺼낸다.** 아래 모든 예제의 출력은 이 규칙 하나로 설명된다.

Promise 자체의 내부 상태 전이(pending → fulfilled/rejected)와 then 체인이 어떻게 콜백을 등록하는지는 [Promise 내부 동작 과정](../04_심화_JavaScript/Promise%20내부%20동작%20과정.md)에서 다룬다. 여기서는 "어떤 콜백이 언제 큐에 들어가고 언제 실행되는가"에 집중한다.

## 콜 스택(Call Stack)

콜 스택은 현재 실행 중인 함수의 위치를 추적하는 LIFO 자료구조다. 함수를 호출하면 스택 프레임이 쌓이고, 함수가 반환되면 프레임이 빠진다. 스택이 완전히 비어야 이벤트 루프가 큐를 들여다본다.

```javascript
function first() {
    console.log('첫 번째 함수');
    second();
}

function second() {
    console.log('두 번째 함수');
    third();
}

function third() {
    console.log('세 번째 함수');
}

first();
```

실행 과정은 이렇게 흐른다.

1. `first()`가 스택에 쌓이고 `'첫 번째 함수'`를 출력한다.
2. `first` 안에서 `second()`를 호출해 그 위에 쌓이고 `'두 번째 함수'`를 출력한다.
3. `second` 안에서 `third()`를 호출해 그 위에 쌓이고 `'세 번째 함수'`를 출력한다.
4. `third` → `second` → `first` 순으로 반환되며 스택에서 빠진다.

이 코드에는 비동기가 없어 스택이 한 번 차올랐다가 그대로 비워진다. 동기 코드는 항상 큐보다 먼저, 작성된 순서대로 끝난다.

콜 스택이 너무 깊어지면 `RangeError: Maximum call stack size exceeded`가 난다. 재귀를 비동기로 풀어야 하는 경우 `setTimeout`이나 `queueMicrotask`로 호출을 다음 틱으로 미뤄 스택을 끊는다.

## 매크로태스크 큐(Macrotask Queue)

매크로태스크 큐(태스크 큐라고도 부른다)에는 한 번의 이벤트 루프 사이클마다 하나씩 꺼내 실행하는 콜백이 쌓인다. 여기 들어가는 대표적인 작업이다.

- `setTimeout`, `setInterval`
- DOM 이벤트 핸들러(클릭, 입력 등)
- I/O 콜백(파일 읽기, 네트워크 응답)
- 브라우저의 `MessageChannel`, Node.js의 `setImmediate`

```javascript
console.log('1. 시작');

setTimeout(() => {
    console.log('2. 첫 번째 타이머');
}, 0);

setTimeout(() => {
    console.log('3. 두 번째 타이머');
}, 0);

console.log('4. 종료');
```

출력:

```
1. 시작
4. 종료
2. 첫 번째 타이머
3. 두 번째 타이머
```

동기 코드 `1`, `4`가 먼저 끝난다. 두 `setTimeout` 콜백은 등록된 순서대로 매크로태스크 큐에 들어가고, 콜 스택이 빈 뒤 한 사이클에 하나씩 FIFO로 꺼낸다. `setTimeout(fn, 0)`의 `0`은 "즉시"가 아니라 "최소 지연 0ms"라는 의미다. 브라우저는 중첩 깊이가 5단계를 넘으면 최소 지연을 4ms로 강제하므로, 타이머를 빽빽하게 중첩하면 한 콜백당 4ms씩 밀린다.

## 마이크로태스크 큐(Microtask Queue)

마이크로태스크 큐는 매크로태스크보다 우선순위가 높다. 핵심 차이는 처리량이다. 매크로태스크는 사이클당 하나만 꺼내지만, **마이크로태스크 큐는 비어질 때까지 전부 처리한다.** 여기 들어가는 작업이다.

- `Promise.then` / `catch` / `finally` 콜백
- `queueMicrotask`
- `MutationObserver`
- Node.js의 `process.nextTick`(아래 별도 설명)

```javascript
console.log('1. 시작');

setTimeout(() => {
    console.log('2. 타이머');
}, 0);

Promise.resolve().then(() => {
    console.log('3. 프로미스 1');
}).then(() => {
    console.log('4. 프로미스 2');
});

console.log('5. 종료');
```

출력:

```
1. 시작
5. 종료
3. 프로미스 1
4. 프로미스 2
2. 타이머
```

동기 코드 `1`, `5`가 끝나면 콜 스택이 빈다. 이벤트 루프는 매크로태스크(`타이머`)로 넘어가기 전에 마이크로태스크 큐부터 비운다. `3. 프로미스 1`이 실행되면서 두 번째 `then`이 새 마이크로태스크를 큐에 추가하는데, 큐가 빌 때까지 처리하는 규칙 때문에 `4. 프로미스 2`까지 같은 사이클에서 연달아 실행한다. 그래서 `2. 타이머`는 모든 프로미스 콜백이 끝난 뒤에야 나온다.

이 "마이크로태스크는 끝까지 비운다" 규칙이 실수의 원인이 되기도 한다. `then` 안에서 또 다른 마이크로태스크를 무한히 만들면 매크로태스크가 영원히 실행되지 못해 타이머나 렌더링이 멈춘다.

## 이벤트 루프(Event Loop)

이벤트 루프는 위 세 요소를 묶는다. 한 사이클(틱)은 이렇게 돈다.

1. 콜 스택에 남은 동기 코드를 전부 실행한다.
2. 콜 스택이 비면 마이크로태스크 큐를 **빌 때까지** 전부 실행한다. 이 과정에서 새로 추가된 마이크로태스크도 같은 단계에서 처리한다.
3. 매크로태스크 큐에서 콜백을 **하나만** 꺼내 콜 스택에 올려 실행한다.
4. 다시 2번으로 돌아가 마이크로태스크 큐를 비운다. (렌더링이 필요하면 이 지점에서 한다.)

```javascript
console.log('시작');

setTimeout(() => {
    console.log('타이머 완료');
}, 0);

Promise.resolve().then(() => {
    console.log('프로미스 완료');
});

console.log('종료');
```

출력:

```
시작
종료
프로미스 완료
타이머 완료
```

`시작`, `종료`는 동기. 콜 스택이 빈 뒤 마이크로태스크(`프로미스 완료`)를 먼저 비우고, 그다음 매크로태스크(`타이머 완료`)를 하나 꺼낸다. 매크로태스크 하나를 실행할 때마다 마이크로태스크 큐를 다시 확인한다는 점이 중요하다. 타이머 콜백 안에서 새 Promise를 만들면 그 콜백이 끝난 직후, 다음 타이머로 넘어가기 전에 처리된다.

## 실행 순서 예제

### 예제 1: 기본 비동기 실행

```javascript
console.log('1. 스크립트 시작');

setTimeout(() => {
    console.log('2. 타이머 1');
}, 0);

Promise.resolve().then(() => {
    console.log('3. 프로미스 1');
});

setTimeout(() => {
    console.log('4. 타이머 2');
}, 0);

console.log('5. 스크립트 종료');
```

출력:

```
1. 스크립트 시작
5. 스크립트 종료
3. 프로미스 1
2. 타이머 1
4. 타이머 2
```

틱 단위로 따라간다.

- 동기 실행 구간에서 `1`을 출력한다. 첫 `setTimeout` 콜백이 매크로태스크 큐에 들어간다(큐: `[타이머1]`). `Promise.resolve().then`이 마이크로태스크 큐에 들어간다(큐: `[프로미스1]`). 두 번째 `setTimeout`이 매크로태스크 큐에 들어간다(큐: `[타이머1, 타이머2]`). 마지막으로 `5`를 출력한다.
- 콜 스택이 빈다. 마이크로태스크 큐를 비운다 → `3. 프로미스 1` 출력.
- 매크로태스크 하나를 꺼낸다 → `2. 타이머 1` 출력. 끝난 뒤 마이크로태스크 큐를 확인하지만 비어 있다.
- 다음 사이클에서 매크로태스크 하나를 또 꺼낸다 → `4. 타이머 2` 출력.

마이크로태스크 하나가 두 타이머 사이가 아니라 두 타이머 **앞**에 통째로 처리되는 것이 핵심이다.

### 예제 2: 중첩된 비동기 작업

```javascript
console.log('1. 시작');

setTimeout(() => {
    console.log('2. 외부 타이머');

    Promise.resolve().then(() => {
        console.log('3. 내부 프로미스');
    });

    setTimeout(() => {
        console.log('4. 내부 타이머');
    }, 0);
}, 0);

Promise.resolve().then(() => {
    console.log('5. 외부 프로미스');
});

console.log('6. 종료');
```

출력:

```
1. 시작
6. 종료
5. 외부 프로미스
2. 외부 타이머
3. 내부 프로미스
4. 내부 타이머
```

틱 단위로 보면 이렇다.

- 동기 구간: `1` 출력 → 외부 `setTimeout` 매크로태스크 등록(큐: `[외부타이머]`) → 외부 Promise 마이크로태스크 등록(큐: `[외부프로미스]`) → `6` 출력.
- 콜 스택이 빈다. 마이크로태스크 비우기 → `5. 외부 프로미스`.
- 매크로태스크 하나 꺼냄 → 외부 타이머 콜백 실행. `2. 외부 타이머` 출력. 이 콜백 안에서 내부 Promise가 마이크로태스크 큐에, 내부 `setTimeout`이 매크로태스크 큐에 들어간다.
- 외부 타이머 콜백이 끝나 콜 스택이 빈다. **다음 매크로태스크로 넘어가기 전에** 마이크로태스크 큐를 비운다 → `3. 내부 프로미스`.
- 다음 사이클에서 매크로태스크(내부 타이머)를 꺼낸다 → `4. 내부 타이머`.

`3. 내부 프로미스`가 `4. 내부 타이머`보다 먼저 나오는 이유는, 매크로태스크 하나를 실행할 때마다 마이크로태스크 큐를 다시 비우기 때문이다.

### 예제 3: 매크로·마이크로태스크가 서로를 만드는 경우

```javascript
console.log('1. 시작');

setTimeout(() => {
    console.log('2. 타이머 1');
    Promise.resolve().then(() => {
        console.log('3. 프로미스 1');
    });
}, 0);

Promise.resolve().then(() => {
    console.log('4. 프로미스 2');
    setTimeout(() => {
        console.log('5. 타이머 2');
    }, 0);
});

setTimeout(() => {
    console.log('6. 타이머 3');
}, 0);

console.log('7. 종료');
```

출력:

```
1. 시작
7. 종료
4. 프로미스 2
2. 타이머 1
3. 프로미스 1
6. 타이머 3
5. 타이머 2
```

틱 단위 추적이다. 매크로 큐 등록 순서를 끝까지 따라가는 것이 관건이다.

- 동기 구간: `1` 출력 → 타이머 1 등록(매크로 큐: `[T1]`) → 프로미스 2 등록(마이크로 큐: `[P2]`) → 타이머 3 등록(매크로 큐: `[T1, T3]`) → `7` 출력.
- 콜 스택이 빈다. 마이크로태스크 비우기 → `4. 프로미스 2` 출력. 이 콜백 안에서 타이머 2를 등록한다(매크로 큐: `[T1, T3, T2]`).
- 매크로태스크 하나 꺼냄(T1) → `2. 타이머 1` 출력. 이 안에서 프로미스 1을 등록한다(마이크로 큐: `[P1]`).
- T1 콜백이 끝나 콜 스택이 빈다. 다음 매크로로 넘어가기 전 마이크로태스크 비우기 → `3. 프로미스 1`.
- 매크로태스크 하나 꺼냄(T3) → `6. 타이머 3`.
- 매크로태스크 하나 꺼냄(T2) → `5. 타이머 2`.

`6. 타이머 3`이 `5. 타이머 2`보다 먼저 나오는 이유가 핵심이다. 큐에 들어간 순서가 기준이다. T3은 동기 구간에서, T2는 그보다 늦은 마이크로태스크 단계(P2 콜백 안)에서 등록되므로 매크로 큐는 `[T1, T3, T2]`다. 콜백 코드의 작성 위치가 아니라 실제로 큐에 진입한 순서를 따라야 출력 순서를 맞게 추적한다.

## Node.js의 process.nextTick과 마이크로태스크 순서

Node.js에는 일반 마이크로태스크 큐와 별개로 `process.nextTick` 큐가 있다. 이벤트 루프는 각 단계 사이에서 `nextTick` 큐를 **Promise 마이크로태스크보다 먼저** 비운다.

```javascript
Promise.resolve().then(() => console.log('promise'));
process.nextTick(() => console.log('nextTick'));

// 출력:
// nextTick
// promise
```

`nextTick`이 먼저 나온다. 콜 스택이 비면 Node는 `nextTick` 큐를 전부 비우고, 그다음 Promise 마이크로태스크 큐를 비운다. `nextTick`을 재귀적으로 호출하면 Promise 콜백과 I/O가 영영 실행되지 못하므로, 라이브러리 코드에서 `nextTick`을 반복 호출하는 패턴은 피한다.

## setTimeout(0)과 setImmediate의 차이

둘 다 "다음 기회에 실행"이지만 이벤트 루프 안에서 들어가는 단계가 다르다. `setTimeout(fn, 0)`은 timers 단계, `setImmediate(fn)`은 check 단계에 콜백을 넣는다.

```javascript
setTimeout(() => console.log('setTimeout'), 0);
setImmediate(() => console.log('setImmediate'));
```

이 코드를 메인 모듈에서 그냥 실행하면 출력 순서가 실행할 때마다 뒤바뀐다. 타이머의 최소 지연 처리와 이벤트 루프 시작 시점이 미세하게 겹쳐 어느 단계가 먼저 도는지 보장되지 않기 때문이다. 반면 I/O 콜백 안에서 호출하면 순서가 고정된다.

```javascript
const fs = require('fs');

fs.readFile(__filename, () => {
    setTimeout(() => console.log('setTimeout'), 0);
    setImmediate(() => console.log('setImmediate'));
});

// 항상:
// setImmediate
// setTimeout
```

I/O 콜백은 poll 단계에서 실행되고, 그 직후가 check 단계(`setImmediate`)다. timers 단계는 다음 루프를 한 바퀴 돌아야 오므로 `setImmediate`가 항상 먼저다. "다음 I/O 직후에 무언가를 확실히 실행하고 싶다"면 `setTimeout(0)`이 아니라 `setImmediate`를 쓴다.

## requestAnimationFrame의 타이밍

브라우저의 `requestAnimationFrame` 콜백은 매크로·마이크로태스크 큐 어디에도 속하지 않는다. 렌더링 직전, 즉 한 프레임을 화면에 그리기 바로 전에 실행된다. 화면 갱신 주기(보통 60Hz면 약 16.6ms)에 맞춰 호출되므로, 레이아웃이나 스타일을 바꾸는 코드를 여기에 넣으면 한 프레임에 한 번만 반영되어 깜빡임 없이 그려진다.

```javascript
console.log('1. 동기');

requestAnimationFrame(() => {
    console.log('3. rAF (렌더 직전)');
});

Promise.resolve().then(() => {
    console.log('2. 마이크로태스크');
});
```

출력은 `1` → `2` → `3` 순이다. 동기 코드와 마이크로태스크가 먼저 끝나고, 그 사이클의 렌더링 직전에 `rAF` 콜백이 실행된다. 스크롤 위치 계산이나 애니메이션 프레임 업데이트를 `setTimeout`으로 처리하면 렌더 타이밍과 어긋나 프레임이 끊기므로, 시각적 변화는 `requestAnimationFrame`에 넣는다.

## 실무에서 자주 마주치는 상황

### API 호출과 상태 업데이트

```javascript
console.log('1. 컴포넌트 마운트');

fetch('https://api.example.com/data')
    .then(response => response.json())
    .then(data => {
        console.log('2. 데이터 수신');
        setState(data);
    });

setTimeout(() => {
    console.log('3. UI 업데이트');
}, 0);

console.log('4. 렌더링 완료');
```

`fetch`는 네트워크 응답이 와야 콜백이 큐에 들어가므로 `2. 데이터 수신`은 응답 시점에 달려 있다. 동기 코드 `1`, `4`는 즉시 끝나고, 응답이 빠르면 마이크로태스크인 fetch 콜백이 매크로태스크인 타이머보다 먼저 처리된다. 단, 응답이 늦으면 `3. UI 업데이트`가 먼저 나올 수 있다. 네트워크가 끼면 순서를 코드 위치로 단정하면 안 된다.

### 이벤트 핸들러 안의 비동기

```javascript
button.addEventListener('click', () => {
    console.log('1. 클릭 이벤트 발생');

    Promise.resolve().then(() => {
        console.log('2. 상태 업데이트');
        updateState();
    });

    setTimeout(() => {
        console.log('3. UI 갱신');
        renderUI();
    }, 0);
});
```

클릭 핸들러 자체가 하나의 매크로태스크다. 핸들러 안의 동기 코드(`1`)가 끝나면 콜 스택이 비고, 다음 매크로태스크(타이머)로 넘어가기 전에 마이크로태스크(`2`)를 비운다. 그래서 상태 업데이트가 항상 UI 갱신보다 먼저 끝난다. 클릭 한 번에 상태를 바꾸고 그 결과로 다시 그려야 할 때, 상태 변경은 마이크로태스크에 두면 같은 사이클 안에서 확정된다.

### 애니메이션과 타이밍

```javascript
console.log('1. 애니메이션 시작');

requestAnimationFrame(() => {
    console.log('2. 프레임 렌더링');

    Promise.resolve().then(() => {
        console.log('3. 상태 업데이트');
    });
});

setTimeout(() => {
    console.log('4. 타이머 완료');
}, 0);

console.log('5. 초기화 완료');
```

동기 코드 `1`, `5`가 끝난다. `setTimeout` 콜백과 `requestAnimationFrame` 콜백이 각각 다른 경로로 대기한다. 타이머는 매크로태스크라 콜 스택이 비는 즉시 후보가 되고, `rAF`는 다음 렌더 직전에 호출된다. 화면 갱신 주기보다 타이머 0ms가 빨리 돌면 `4. 타이머 완료`가 `2. 프레임 렌더링`보다 먼저 나오기도 한다. rAF 콜백 안의 `Promise.then`(`3`)은 그 프레임 렌더링이 끝난 직후 마이크로태스크로 처리된다.

## 정리

실행 순서를 가르는 규칙은 네 가지다.

1. 동기 코드가 항상 먼저, 작성된 순서대로 끝난다.
2. 콜 스택이 비면 마이크로태스크 큐를 빌 때까지 전부 비운다.
3. 매크로태스크는 한 사이클에 하나만 꺼내고, 그 사이마다 마이크로태스크 큐를 다시 비운다.
4. 같은 큐 안에서는 들어온 순서(FIFO)대로 실행한다.

Node.js는 여기에 `process.nextTick` 큐가 Promise 마이크로태스크보다 앞선다는 규칙이 더 붙는다. 브라우저의 `requestAnimationFrame`은 큐가 아니라 렌더 직전 타이밍에 묶인다. 이 규칙들을 알면 비동기 코드의 출력 순서를 코드만 보고도 추적할 수 있다. Promise가 콜백을 큐에 등록하기까지의 내부 상태 전이는 [Promise 내부 동작 과정](../04_심화_JavaScript/Promise%20내부%20동작%20과정.md)에서 이어 본다.
