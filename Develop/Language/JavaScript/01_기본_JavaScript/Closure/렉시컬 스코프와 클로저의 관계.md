# JavaScript 렉시컬 스코프와 클로저의 관계

## 1. 렉시컬 스코프란 무엇인가

학습 자료를 보면 "렉시컬"이라는 단어를 어원부터 풀어쓰는 경우가 많은데, 실무에서 디버깅을 하다 보면 어원보다는 ECMAScript 명세가 어떻게 동작을 정의하는지가 훨씬 중요하다. 렉시컬 스코프는 결국 명세상 `LexicalEnvironment` 레코드와 `OuterEnv` 포인터 체인이라는 두 개의 구조로 환원된다.

ECMAScript에서 함수가 호출되면 실행 컨텍스트(Execution Context)가 만들어지고, 그 안에 두 개의 환경 레코드 슬롯이 들어간다. 하나는 `VariableEnvironment`, 다른 하나는 `LexicalEnvironment`다. 둘 다 `Environment Record`라는 동일한 자료구조를 가리키지만, `let`/`const`의 블록 스코프 처리 등 일부 동작에서 분리되어 쓰인다. 그리고 각 `Environment Record`에는 `OuterEnv`라는 포인터가 있다. 이 포인터는 함수가 정의된 시점의 외부 환경을 가리킨다. 호출된 시점이 아니라 정의된 시점이라는 점이 핵심이다.

다음 코드를 보자.

```javascript
const x = 'global';

function outer() {
    const x = 'outer';
    return function inner() {
        console.log(x);
    };
}

const fn = outer();
function caller() {
    const x = 'caller';
    fn();
}
caller();
```

여기서 `fn()`을 `caller` 안에서 호출했지만 출력은 `'outer'`다. `inner`의 환경 레코드가 가지는 `OuterEnv`는 `caller`의 환경이 아니라 `outer`가 호출되어 만들어진 환경을 가리키기 때문이다. 이 포인터는 함수 객체가 만들어질 때 내부 슬롯 `[[Environment]]`에 박혀버린다. 이후 어디서 호출되든 변하지 않는다.

스코프 체인이라는 표현은 이 `OuterEnv` 포인터를 따라 올라가는 단방향 연결 리스트에 붙은 별명일 뿐이다. 식별자 해석(Identifier Resolution)은 명세의 `ResolveBinding` 알고리즘이 현재 환경 레코드부터 시작해 `OuterEnv`를 따라 글로벌 환경까지 거슬러 올라가며 첫 번째로 매칭되는 바인딩을 반환하는 과정이다.

## 2. 동적 스코프와의 비교

렉시컬 스코프의 동작이 와닿지 않는다면, 정반대 모델인 동적 스코프(Dynamic Scope)와 비교해보면 분명해진다. bash나 옛날 Perl(Perl 4 시절, 그리고 Perl 5에서도 `local` 키워드)이 동적 스코프를 채택한다.

```bash
#!/bin/bash
x="global"

print_x() {
    echo "$x"
}

caller() {
    local x="caller"
    print_x
}

caller
```

이 스크립트는 `caller`를 출력한다. bash에서 변수 해석은 호출 스택(call stack)을 거슬러 올라가며 일어나기 때문이다. `print_x`는 자기 자신에 `x`가 없으니 호출자인 `caller`의 지역 변수 `x`를 발견하고 그걸 쓴다. 함수가 어디에 정의되었는지는 의미가 없다. 누가 어떤 상태에서 불렀는지가 의미가 있다.

같은 모양을 JavaScript로 옮기면 정반대 결과가 나온다.

```javascript
let x = 'global';

function printX() {
    console.log(x);
}

function caller() {
    let x = 'caller';
    printX();
}

caller();
```

출력은 `'global'`이다. `printX`가 정의된 시점의 외부 환경은 모듈/스크립트 전역이고, `caller`의 지역 환경과는 아무 관계가 없다. 호출 스택은 식별자 해석에 관여하지 않는다.

## 3. 왜 렉시컬이 디버깅에 유리한가

동적 스코프 언어를 다뤄본 사람은 알겠지만, 디버깅이 굉장히 힘들다. 함수의 동작이 호출 경로에 따라 바뀌기 때문에, 동일한 함수가 어떤 호출자에서는 잘 돌고 어떤 호출자에서는 이상하게 동작하는 일이 흔하다. 그리고 그 차이가 호출자가 어떤 지역 변수를 우연히 같은 이름으로 가지고 있느냐에서 발생한다.

렉시컬 스코프에서는 함수의 자유 변수(free variable)가 어디에서 해석되는지가 소스 코드를 보는 순간 확정된다. 텍스트만 따라가면 답이 나온다. 그래서 스택 트레이스도 의미가 있다.

다음과 같이 일부러 에러를 발생시켜 보자.

```javascript
function makeFormatter(prefix) {
    return function format(value) {
        return prefix.toUpperCase() + ': ' + value.toFixed(2);
    };
}

const fmt = makeFormatter(null);
fmt(3.14);
```

크롬 V8에서 던지는 스택 트레이스는 다음과 같다.

```
TypeError: Cannot read properties of null (reading 'toUpperCase')
    at format (file:///app/index.js:3:23)
    at file:///app/index.js:8:1
```

`format` 함수의 정의 위치가 그대로 찍힌다. 이 정보만으로 어떤 외부 변수가 잘못된 상태인지 추적할 수 있다. `prefix`는 `format`이 정의된 시점의 외부 환경 즉 `makeFormatter`의 활성화 레코드에서 해석된다. 그래서 `format`이 어디에서 호출되었는지를 다 무시하고 곧장 `makeFormatter` 호출 시점으로 이동해 인자를 검사하면 된다.

같은 버그를 동적 스코프 모델로 옮기면 트레이스는 호출 경로 전체를 더듬어야 의미가 생긴다. 호출자 A는 `prefix`를 `'log'`로 두고 호출자 B는 `null`로 두는데 둘 다 같은 `format` 함수를 부른다면, 스택 트레이스가 `format`을 가리켜도 그 한 줄만으로는 어느 호출자가 잘못된 상태를 만들었는지 알 수가 없다. 호출 체인을 거슬러 올라가며 각 프레임의 환경을 일일이 확인해야 한다.

다른 예시로 비동기 콜백을 보자.

```javascript
function scheduleReport(report) {
    setTimeout(function emit() {
        console.log(report.name);
    }, 1000);
}

scheduleReport(null);
```

```
TypeError: Cannot read properties of null (reading 'name')
    at emit (file:///app/index.js:3:28)
```

`emit`은 `setTimeout`의 내부 큐에서 꺼내져 실행된다. 호출 경로상으로는 `scheduleReport`와 단절되어 있다. 동적 스코프였다면 `setTimeout` 워커의 임의의 환경에서 `report`를 찾으려 했을 것이고, 십중팔구 엉뚱한 값이 잡히거나 `undefined`가 잡혔을 것이다. 렉시컬 스코프이기 때문에 `emit`의 `OuterEnv`는 여전히 `scheduleReport` 호출 시 만들어진 환경을 가리키고, 트레이스의 한 줄과 정의 위치만으로 디버깅이 끝난다.

## 4. 클로저는 렉시컬 환경의 부산물이다

클로저를 따로 떼어놓고 "함수와 그 함수가 선언된 환경의 조합"이라고 외우는 식의 설명이 많은데, 사실 위에서 본 `[[Environment]]` 슬롯과 `OuterEnv` 포인터 체인을 받아들이면 클로저는 별도 개념이 아니다. 함수가 만들어질 때 외부 환경을 슬롯에 박아두는 동작 그 자체가 클로저다. 모든 JavaScript 함수는 클로저다. 다만 외부 환경을 실제로 참조하는 자유 변수가 있을 때만 그 동작이 눈에 띌 뿐이다.

```javascript
function createCounter() {
    let count = 0;

    return {
        increment() {
            count++;
            return count;
        },
        decrement() {
            count--;
            return count;
        },
        getCount() {
            return count;
        }
    };
}

const counter = createCounter();
counter.increment(); // 1
counter.increment(); // 2
counter.decrement(); // 1
counter.getCount();  // 1
```

`increment`, `decrement`, `getCount` 세 함수는 모두 같은 `OuterEnv`를 가리킨다. 그 환경 안에 `count`라는 단 하나의 바인딩이 있다. 세 함수의 호출이 같은 카운터를 공유하는 이유가 여기서 나온다. 만약 `createCounter`를 두 번 호출하면 호출마다 새로운 환경 레코드가 만들어지고, 각 호출에서 반환된 객체의 메서드들은 서로 다른 `count`를 본다.

```javascript
function createMultiplier(x) {
    return function (y) {
        return x * y;
    };
}

const multiplyByTwo = createMultiplier(2);
const multiplyByThree = createMultiplier(3);

multiplyByTwo(4);   // 8
multiplyByThree(4); // 12
```

`multiplyByTwo`와 `multiplyByThree`는 함수 본문이 동일하지만 `[[Environment]]` 슬롯이 다른 환경 레코드를 가리킨다. 각 환경에 박힌 `x`의 값이 다르기 때문에 결과가 갈린다. 이게 클로저로 자유 변수에 값을 "고정"시키는 패턴의 정체다.

## 5. 데이터 캡슐화

ES2022에 `#privateField` 문법이 들어오기 전까지, 클로저는 사실상 유일한 캡슐화 수단이었다. 지금도 클래스가 부담스러운 코드에서는 흔히 쓰인다.

```javascript
function createPerson(name) {
    let privateName = name;

    return {
        getName() {
            return privateName;
        },
        setName(newName) {
            if (typeof newName !== 'string' || newName.length === 0) {
                throw new TypeError('name must be a non-empty string');
            }
            privateName = newName;
        }
    };
}

const person = createPerson('John');
person.getName();         // 'John'
person.setName('Jane');
person.getName();         // 'Jane'
person.privateName;       // undefined
```

`privateName`은 반환된 객체의 프로퍼티가 아니다. 반환된 객체에는 `getName`과 `setName` 두 함수만 있다. 외부에서 `person.privateName`으로 접근해도 `undefined`가 나온다. `privateName`은 두 함수의 `OuterEnv`가 가리키는 환경 레코드 안의 바인딩일 뿐이고, 그 환경 레코드는 객체의 프로퍼티 테이블과는 별도의 자료구조다.

## 6. 반복문에서 흔히 마주치는 함정

```javascript
function createFunctionsVar() {
    const functions = [];
    for (var i = 0; i < 3; i++) {
        functions.push(function () {
            return i;
        });
    }
    return functions;
}

const funcsVar = createFunctionsVar();
funcsVar[0](); // 3
funcsVar[1](); // 3
funcsVar[2](); // 3
```

`var`로 선언한 `i`는 함수 스코프이기 때문에 세 번의 반복이 모두 같은 환경 레코드의 같은 바인딩을 공유한다. 반복이 끝난 시점에 `i`는 3이고, 세 함수가 모두 그 3을 본다.

```javascript
function createFunctionsLet() {
    const functions = [];
    for (let i = 0; i < 3; i++) {
        functions.push(function () {
            return i;
        });
    }
    return functions;
}

const funcsLet = createFunctionsLet();
funcsLet[0](); // 0
funcsLet[1](); // 1
funcsLet[2](); // 2
```

`let`을 쓰면 명세상 매 반복마다 새 `LexicalEnvironment`가 만들어진다. 정확히는 `for` 루프의 `PerIterationEnvironment` 동작이 작동해서 반복 진입 시점마다 새로운 `i` 바인딩이 새 환경에 생성된다. 푸시되는 함수들은 각자 다른 환경을 `OuterEnv`로 가지게 되고, 그래서 0, 1, 2가 그대로 캡처된다. 이 차이를 모르고 `var`로 짠 코드를 콜백 기반으로 옮기다가 버그가 생기는 경우가 정말 많다.

## 7. 메모리 누수의 메커니즘

클로저가 메모리에 외부 환경을 붙들어둔다는 사실은 양날의 검이다.

```javascript
function attachHandler(node) {
    const heavyData = new Array(1_000_000).fill('payload');

    node.addEventListener('click', function onClick() {
        console.log(heavyData.length);
    });
}
```

`onClick`의 `[[Environment]]`는 `attachHandler`의 활성화 레코드를 가리킨다. 그 레코드 안에는 `heavyData`가 있다. `node`가 DOM에서 제거되어도 어딘가에서 `node`에 대한 참조가 살아 있고 그 `node`가 `onClick` 리스너를 잡고 있다면, `onClick`의 환경 레코드가 살아 있고, 그래서 `heavyData`도 살아 있다. 100만 칸짜리 배열이 계속 떠 있게 된다.

해결은 단순하다. 핸들러가 실제로 참조하는 변수만 클로저 안에 두든가, 리스너를 제거해 참조 사슬을 끊든가다. V8의 경우 컴파일러가 함수 본문에서 사용되지 않는 외부 변수는 환경에서 제외하는 최적화를 하기는 하지만, 동적 평가(`eval`, `with`, `new Function`)가 끼어 있으면 이 최적화가 꺼진다. 운영 코드에서 `eval`을 안 쓰는 게 일반적인 권고지만, 이런 메모리 측면의 이유도 있다.

## 8. 이벤트 핸들러에서의 상태 보존

```javascript
function createButtonHandler(buttonId) {
    let clickCount = 0;

    return function () {
        clickCount++;
        console.log(`Button ${buttonId} clicked ${clickCount} times`);
    };
}

const button1Handler = createButtonHandler('button1');
const button2Handler = createButtonHandler('button2');

button1Handler(); // Button button1 clicked 1 times
button1Handler(); // Button button1 clicked 2 times
button2Handler(); // Button button2 clicked 1 times
```

`createButtonHandler` 호출마다 별개의 환경 레코드가 만들어지고, 그 안에 `clickCount`가 0으로 초기화된다. 반환된 함수는 자기 환경의 `clickCount`만 본다. 그래서 두 핸들러의 카운터가 섞이지 않는다. 이 패턴은 React 훅이 등장하기 전 함수 컴포넌트나 jQuery 시절 코드에서 상태를 가지는 대표적인 방법이었다.
