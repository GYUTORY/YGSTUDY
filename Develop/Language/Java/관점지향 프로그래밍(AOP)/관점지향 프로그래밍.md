# 관점지향 프로그래밍(AOP) 입문

이 문서는 AOP를 처음 만난 사람을 위한 짧은 진입점이다. 동작 원리, 프록시 종류, 포인트컷 표현식, 트러블슈팅 같은 실제 내용은 [AOP Deep Dive](./AOP_Deep_Dive.md)에서 다룬다.

## AOP가 뭘 해결하는가

서비스 메서드마다 로깅, 트랜잭션 시작·종료, 권한 검사, 실행 시간 측정 같은 코드를 한두 줄씩 박아넣다 보면 비즈니스 로직보다 부가 로직이 더 길어진다. 여러 모듈에 횡단으로 흩어지는 이런 관심사를 본래 로직에서 떼어내는 게 AOP의 출발점이다. 객체 책임은 그대로 두고, 메서드 호출 직전·직후·예외 시점에 끼어들어 부가 동작을 실행하는 메커니즘이라고 보면 된다.

OOP를 대체하는 게 아니라 보완한다. 책임이 명확한 객체 설계가 먼저고, 그 위에 횡단 관심사를 얹는 도구다.

## 알아둬야 할 다섯 단어

- **Aspect**: 관심사 자체. `@Aspect`가 붙은 클래스 하나가 하나의 Aspect다.
- **JoinPoint**: 부가 동작이 끼어들 수 있는 지점. Spring AOP에서는 메서드 실행 시점만 해당한다.
- **Pointcut**: JoinPoint 중에서 어디에 끼어들지 골라내는 표현식.
- **Advice**: 끼어들었을 때 실제로 할 일. `@Before`, `@After`, `@Around` 등이 종류를 정한다.
- **Weaving**: Pointcut으로 고른 JoinPoint에 Advice를 결합하는 과정.

JoinPoint는 후보 지점, Pointcut은 그 후보 중에서 내가 고른 것이라고 외워두면 헷갈리지 않는다.

## 가장 단순한 예제

`@Around`로 메서드 실행 시간을 찍는 Aspect 한 개.

```java
@Aspect
@Component
public class TimingAspect {

    private static final Logger log = LoggerFactory.getLogger(TimingAspect.class);

    @Around("execution(* com.example.service..*(..))")
    public Object measure(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.nanoTime();
        try {
            return pjp.proceed();
        } finally {
            long elapsedMs = (System.nanoTime() - start) / 1_000_000;
            log.info("{} took {} ms", pjp.getSignature().toShortString(), elapsedMs);
        }
    }
}
```

`pjp.proceed()`를 호출하지 않으면 원본 메서드가 실행되지 않는다. `@Around`에서 가장 흔한 사고다. 객체 반환 메서드에서 누락하면 호출자에게 조용히 null이 흐르고, 운영에 올라간 다음에야 발견된다.

## 더 깊이 들어가기

이 정도면 AOP가 뭐 하는 물건인지는 감이 잡힌다. 실제 운영 코드에서 부딪히는 문제는 따로 있다.

- 같은 클래스 안에서 메서드를 호출하면 `@Transactional`이 안 먹는 self-invocation 문제
- JDK Dynamic Proxy와 CGLIB의 차이, 그리고 Spring Boot가 둘 중 뭘 기본값으로 쓰는지
- `target` vs `this`, `@within` vs `@target` 같은 헷갈리는 포인트컷 지정자
- Aspect를 분명히 만들었는데 동작하지 않을 때 의심할 항목들
