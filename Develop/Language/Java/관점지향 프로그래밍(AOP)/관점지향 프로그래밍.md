# 관점지향 프로그래밍(AOP) 입문

이 문서는 AOP를 처음 만난 사람이 첫 Aspect를 직접 띄워볼 수 있을 정도까지만 다룬다. 프록시 종류, 포인트컷 표현식 전체 문법, 운영에서 부딪히는 깊은 트러블슈팅은 [AOP Deep Dive](./AOP_Deep_Dive.md)로 넘긴다.

## AOP가 해결하는 문제

서비스 메서드마다 로깅, 트랜잭션 시작·종료, 권한 검사, 실행 시간 측정 코드를 한두 줄씩 박아넣다 보면 본래 비즈니스 로직보다 부가 로직이 더 길어진다. 같은 패턴이 50개 메서드에 흩어져 있으면 한 줄 바꾸려고 50번 손대야 한다. 누락된 한두 곳이 운영 사고가 된다. 이렇게 여러 모듈에 횡단으로 흩어지는 관심사(cross-cutting concern)를 본래 로직에서 떼어내는 게 AOP의 출발점이다.

객체 책임은 그대로 두고, 메서드 호출 직전·직후·예외 시점에 끼어들어 부가 동작을 실행하는 메커니즘이라고 보면 된다. OOP를 대체하는 게 아니라 보완한다. 책임이 명확한 객체 설계가 먼저고, 그 위에 횡단 관심사를 얹는 도구다.

## AOP 없는 코드와 있는 코드 비교

말로는 잘 와닿지 않으니 같은 요구사항을 두 방식으로 써 본다. "주문 서비스의 모든 public 메서드 실행 시간을 로깅한다."

AOP 없이 작성한 모습.

```java
@Service
public class OrderService {

    private static final Logger log = LoggerFactory.getLogger(OrderService.class);

    public Order place(OrderRequest req) {
        long start = System.nanoTime();
        try {
            // 진짜 비즈니스 로직
            Order order = doPlace(req);
            return order;
        } finally {
            log.info("place took {} ms", (System.nanoTime() - start) / 1_000_000);
        }
    }

    public Order cancel(long orderId) {
        long start = System.nanoTime();
        try {
            Order order = doCancel(orderId);
            return order;
        } finally {
            log.info("cancel took {} ms", (System.nanoTime() - start) / 1_000_000);
        }
    }
    // ... 메서드마다 같은 try/finally가 반복된다
}
```

같은 try/finally가 메서드마다 들러붙는다. 메서드 5개면 5번, 50개면 50번이다. 로그 포맷을 한 번 바꾸려면 전부 손대야 한다.

AOP를 적용한 모습.

```java
@Service
public class OrderService {

    public Order place(OrderRequest req) {
        return doPlace(req);  // 비즈니스 로직만 남는다
    }

    public Order cancel(long orderId) {
        return doCancel(orderId);
    }
}

@Aspect
@Component
public class TimingAspect {

    private static final Logger log = LoggerFactory.getLogger(TimingAspect.class);

    @Around("execution(* com.example.order.OrderService.*(..))")
    public Object measure(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.nanoTime();
        try {
            return pjp.proceed();
        } finally {
            long elapsedMs = (System.nanoTime() - start) / 1_000_000;
            log.info("{} took {} ms", pjp.getSignature().toShortString(), elapsedMs);
        }
    }
}
```

비즈니스 코드에서 부가 관심사가 사라졌다. 로그 포맷을 바꿔도 `TimingAspect` 한 곳만 고치면 된다. 새 메서드를 추가해도 자동으로 적용된다. 이 차이가 AOP가 주는 실질적 이점이다.

## 알아둬야 할 다섯 단어

용어가 많아 보이지만 실제로 손에 익는 건 다섯 개다. 위 `TimingAspect` 예제에 다 들어 있다.

- **Aspect**: 관심사 자체. `@Aspect`가 붙은 클래스 하나가 하나의 Aspect다. 위 예제에서는 `TimingAspect`.
- **JoinPoint**: 부가 동작이 끼어들 수 있는 후보 지점. Spring AOP에서는 메서드 실행 시점만 해당한다. `OrderService.place()` 호출 순간이 JoinPoint다.
- **Pointcut**: JoinPoint 중에서 어디에 끼어들지 골라내는 표현식. `execution(* com.example.order.OrderService.*(..))`가 그것이다.
- **Advice**: 끼어들었을 때 실제로 할 일. `@Before`, `@After`, `@Around` 같은 어노테이션이 종류를 정한다. 위에서는 `@Around`.
- **Weaving**: Pointcut으로 고른 JoinPoint에 Advice를 결합하는 과정. Spring AOP는 런타임에 프록시를 만들어 위빙한다.

JoinPoint는 후보 지점, Pointcut은 그 후보 중에서 내가 고른 것이라고 외워두면 헷갈리지 않는다.

## 시작 준비 - 의존성과 활성화

첫 Aspect를 만들었는데 동작하지 않는다고 한 시간을 헤매는 경우, 거의 다 두 가지 중 하나다.

첫째, `spring-boot-starter-aop` 의존성이 빠져 있다. Spring Boot에서는 이 스타터를 추가해야 AspectJ 런타임과 위빙 인프라가 함께 들어온다.

```gradle
// build.gradle
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-aop'
}
```

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
```

둘째, `@EnableAspectJAutoProxy`가 빠져 있는 경우다. Spring Boot는 `spring-boot-starter-aop`만 있으면 자동 설정으로 활성화하지만, 일반 Spring 프로젝트에서는 직접 켜야 한다.

```java
@Configuration
@EnableAspectJAutoProxy
public class AopConfig { }
```

Spring Boot에서도 `proxyTargetClass` 같은 옵션을 명시하고 싶으면 직접 붙인다. 어떤 프로젝트는 명시적으로 두는 걸 컨벤션으로 삼기도 한다.

## 다섯 종류 Advice와 실행 순서

Advice는 다섯 개를 알면 된다. 같은 메서드에 모두 붙였을 때 실행 순서가 정해져 있다.

```java
@Aspect
@Component
public class LoggingAspect {

    @Pointcut("execution(* com.example.order.OrderService.place(..))")
    public void placeMethod() {}

    @Before("placeMethod()")
    public void before(JoinPoint jp) {
        // 메서드 실행 직전. 인자는 볼 수 있지만 반환값은 모른다.
    }

    @AfterReturning(pointcut = "placeMethod()", returning = "result")
    public void afterReturning(JoinPoint jp, Object result) {
        // 메서드가 정상 반환했을 때만 실행. 반환값을 받을 수 있다.
    }

    @AfterThrowing(pointcut = "placeMethod()", throwing = "ex")
    public void afterThrowing(JoinPoint jp, Throwable ex) {
        // 메서드가 예외를 던졌을 때만 실행.
    }

    @After("placeMethod()")
    public void after(JoinPoint jp) {
        // try/finally의 finally처럼 정상/예외 모두 실행.
    }

    @Around("placeMethod()")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        // 가장 강한 Advice. proceed() 호출로 원본을 직접 실행한다.
        return pjp.proceed();
    }
}
```

실행 순서는 `@Around` → `@Before` → 원본 메서드 → `@AfterReturning`(또는 `@AfterThrowing`) → `@After` → `@Around` 후반부다. `@Around`가 가장 바깥을 감싸기 때문에 시작과 끝 양쪽에서 실행된다.

세 가지를 기억해두면 실수가 줄어든다. `@Before`는 인자만 보고, 반환값을 보려면 `@AfterReturning`이 필요하다. `@After`는 finally와 같아서 예외가 나도 무조건 실행된다. `@Around`에서 `proceed()`를 호출하지 않으면 원본 메서드가 실행되지 않는다. Object 반환 메서드라면 호출자에게 조용히 null이 흐른다. `@Around`에서 가장 흔한 사고다.

웬만한 일은 `@Before`나 `@AfterReturning`로 해결된다. `@Around`는 반환값을 가공하거나, 호출 자체를 막거나, 재시도 같은 흐름 제어가 필요할 때만 쓴다.

## Spring AOP가 못 잡는 JoinPoint

Spring AOP가 가진 가장 큰 제약을 모르고 시작하면 디버깅에 시간을 버린다. Spring AOP는 프록시 기반이고, **메서드 실행 시점**만 JoinPoint로 잡는다. 다음 셋은 절대 잡지 못한다.

- **생성자 호출**: `new OrderService()` 시점에는 프록시가 끼어들 수 없다. `@PostConstruct`로 우회하거나 팩토리에 Aspect를 거는 식으로 풀어야 한다.
- **필드 접근**: `order.status = "PAID"` 같은 직접 필드 읽기·쓰기는 잡히지 않는다. setter 메서드를 거치게 만들어야 한다.
- **private 메서드**: 프록시는 외부에서 들어오는 호출만 가로챈다. 클래스 내부에서 `this.calculate()`로 호출되는 private 메서드는 프록시를 거치지 않으므로 Aspect가 동작하지 않는다.

이 셋이 정말 필요하면 AspectJ를 컴파일·로드 타임 위빙으로 써야 한다. 다만 빌드와 운영 모두 복잡해지므로, 대부분의 경우는 설계를 바꿔서 메서드 호출로 만들어 해결한다.

## 입문자가 가장 자주 빠지는 함정 셋

같은 클래스 안에서 자기 메서드를 호출하면 Aspect가 동작하지 않는 self-invocation 문제. 가장 악명 높다.

```java
@Service
public class OrderService {

    public void place(OrderRequest req) {
        validate(req);  // 같은 클래스 내부 호출 → 프록시 우회 → Aspect 미동작
    }

    @Transactional
    public void validate(OrderRequest req) {
        // 트랜잭션이 시작되지 않는다. @Transactional이 안 먹는 흔한 케이스.
    }
}
```

프록시는 외부에서 들어오는 호출만 가로챈다. `this.validate()`는 원본 객체의 메서드를 직접 부르는 것이라 프록시를 거치지 않는다. 해결 방법은 자기 자신을 빈으로 주입받아 호출하거나, 두 책임을 다른 클래스로 분리하는 것이다.

인터페이스 캐스팅 실패. JDK Dynamic Proxy를 쓸 때 발생한다.

```java
public interface OrderService { }

@Service
public class OrderServiceImpl implements OrderService { }

@Autowired
private OrderService orderService;  // 실제 타입은 프록시

OrderServiceImpl impl = (OrderServiceImpl) orderService;  // ClassCastException
```

JDK 프록시는 인터페이스 타입으로만 참조 가능하다. 구현체로 캐스팅하면 `ClassCastException`이 터진다. 대부분의 경우 인터페이스 타입으로만 쓰면 되지만, 라이브러리가 구현체 캐스팅을 시도하면 사고가 난다. Spring Boot가 CGLIB을 기본값으로 쓰는 이유 중 하나가 이것이다.

`final` 메서드 누락. CGLIB은 대상 클래스를 상속해서 프록시를 만든다.

```java
@Service
public class ReportService {

    @Transactional
    public final Report generate() {  // final이라 오버라이드 불가
        // CGLIB이 이 메서드만 조용히 프록시 우회 → @Transactional 동작 안 함
    }
}
```

상속이 막힌 메서드는 오버라이드할 수 없으니 CGLIB이 그 메서드만 우회한다. 에러도 안 나고 조용히 동작만 안 한다. Aspect가 걸린 클래스에는 `final`을 쓰지 않는 게 안전하다.

## 더 깊이 들어가기

이 정도면 첫 Aspect를 띄우고 실행 순서까지는 잡힌다. 운영 코드에서는 더 까다로운 문제들이 기다린다.

- JDK Dynamic Proxy와 CGLIB의 차이, Spring Boot가 둘 중 뭘 기본값으로 쓰는지
- `target` vs `this`, `@within` vs `@target` 같은 헷갈리는 포인트컷 지정자
- 여러 Aspect의 우선순위 제어와 `@Order`
- Aspect를 분명히 만들었는데 동작하지 않을 때 의심할 항목들

이런 내용은 [AOP Deep Dive](./AOP_Deep_Dive.md)에 정리해뒀다.
