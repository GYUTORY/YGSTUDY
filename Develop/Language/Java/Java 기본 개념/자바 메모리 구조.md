# 자바 메모리 구조

## JVM 메모리 영역 개요

JVM이 프로그램을 실행하면 OS로부터 메모리를 할당받고, 이를 용도에 따라 여러 영역으로 나눠서 관리한다. 크게 **메서드 영역**, **스택 영역**, **힙 영역** 세 가지로 구분한다.

## 메서드 영역 (Method Area)

프로그램 실행에 필요한 **클래스 수준의 데이터**를 저장하는 공간이다. JVM이 시작될 때 생성되고, 모든 스레드가 공유한다.

<div align="center">
    <img src="../../../../etc/image/Language/Java/메서드_영역.png" alt="메서드 영역" width="50%">
</div>

메서드 영역에 저장되는 것들:

- **클래스 정보**: 클래스의 바이트코드, 필드 정보, 메서드 코드, 생성자 코드
- **static 변수**: `static`으로 선언된 변수들이 여기에 올라간다
- **런타임 상수 풀(Runtime Constant Pool)**: 리터럴 상수와 심볼릭 레퍼런스를 저장한다

인스턴스 메서드를 호출하면 실제로는 메서드 영역에 있는 코드를 실행한다. 객체마다 메서드 코드를 복사해서 갖고 있는 게 아니다. 필드 데이터만 각 인스턴스별로 힙에 따로 저장된다.

### Metaspace (Java 8 이후)

Java 7까지는 메서드 영역이 힙 안의 **Permanent Generation(PermGen)**에 있었다. PermGen은 고정 크기라서 클래스를 많이 로드하면 `java.lang.OutOfMemoryError: PermGen space`가 발생했다. Spring처럼 프록시 클래스를 동적으로 생성하는 프레임워크를 쓰면 이 에러를 자주 만났다.

Java 8부터 PermGen이 사라지고 **Metaspace**로 대체됐다. Metaspace는 힙이 아니라 네이티브 메모리를 사용하고, 기본적으로 크기 제한이 없다. 그래도 운영 환경에서는 `MaxMetaspaceSize`를 설정해두는 게 좋다. 메모리 누수가 있을 때 서버 전체 메모리를 잡아먹는 걸 방지할 수 있다.

```bash
# Metaspace 크기 제한 설정
java -XX:MetaspaceSize=128m -XX:MaxMetaspaceSize=256m -jar app.jar
```

## 스택 영역 (Stack Area)

메서드 호출 정보를 저장하는 공간이다. **스레드마다 하나씩** 독립적으로 생성된다. 다른 스레드의 스택에는 접근할 수 없다.

메서드를 호출하면 **스택 프레임(Stack Frame)**이 하나 생기고, 메서드가 끝나면 해당 프레임이 제거된다. 스택 프레임에는 다음이 들어간다:

- **지역 변수(Local Variables)**: 메서드 내 선언된 변수와 매개변수
- **오퍼랜드 스택(Operand Stack)**: 연산 중간 결과를 임시 저장
- **프레임 데이터**: 메서드의 반환 주소, 예외 처리 정보 등

### 스택 프레임의 동작

<div align="center">
    <img src="../../../../etc/image/Language/Java/스택_구조.png" alt="스택 구조" width="50%">
</div>

스택은 **후입선출(LIFO)** 구조다. 가장 나중에 호출된 메서드가 가장 먼저 종료된다.

```java
public class StackExample {
    public static void main(String[] args) {  // 스택 프레임 1 생성
        int result = add(3, 5);                // 스택 프레임 2 생성
        System.out.println(result);
    }  // 스택 프레임 1 제거

    static int add(int a, int b) {
        int sum = a + b;  // 지역 변수 a, b, sum 모두 이 프레임에 저장
        return sum;
    }  // 스택 프레임 2 제거 → main으로 돌아감
}
```

`main()`이 먼저 스택에 쌓이고, `add()`가 그 위에 쌓인다. `add()`가 반환되면 해당 프레임이 제거되고 `main()`으로 복귀한다.

### StackOverflowError

재귀 호출이 너무 깊어지면 스택 공간이 부족해져서 `StackOverflowError`가 발생한다. 실무에서는 재귀 깊이가 예측 불가능한 경우 반복문으로 변환하는 게 안전하다.

```java
// StackOverflowError 발생 예시
public void recursive() {
    recursive();  // 종료 조건 없는 재귀 → 스택 프레임이 무한히 쌓임
}
```

스택 크기는 `-Xss` 옵션으로 조절할 수 있다. 기본값은 보통 512KB~1MB다.

```bash
java -Xss1m -jar app.jar
```

## 힙 영역 (Heap Area)

`new` 키워드로 생성한 **객체와 배열**이 저장되는 공간이다. 모든 스레드가 공유한다. GC(Garbage Collection)가 더 이상 참조되지 않는 객체를 정리한다.

### 힙의 내부 구조

힙은 GC 성능을 위해 세대별로 나뉜다.

```
┌──────────────────────────────────────────────┐
│                  Heap                         │
│                                               │
│  ┌────────────────────────────────────┐      │
│  │         Young Generation           │      │
│  │  ┌────────┬─────────┬─────────┐   │      │
│  │  │  Eden  │   S0    │   S1    │   │      │
│  │  └────────┴─────────┴─────────┘   │      │
│  └────────────────────────────────────┘      │
│                                               │
│  ┌────────────────────────────────────┐      │
│  │         Old Generation             │      │
│  │    (오래 살아남은 객체가 이동)       │      │
│  └────────────────────────────────────┘      │
└──────────────────────────────────────────────┘
```

- **Eden**: 새로 생성된 객체가 처음 할당되는 곳
- **Survivor (S0, S1)**: Minor GC에서 살아남은 객체가 이동하는 곳. 두 영역을 번갈아 사용한다
- **Old Generation**: Survivor에서 여러 번 살아남은 객체가 승격(Promotion)되는 곳

객체가 Eden에 생성되고, Minor GC가 발생하면 살아있는 객체가 Survivor로 이동한다. Survivor에서 일정 횟수(기본 15번) 이상 GC를 견디면 Old Generation으로 넘어간다. Old Generation이 가득 차면 Major GC(Full GC)가 발생하는데, 이때 애플리케이션이 멈추는 Stop-The-World 시간이 길어질 수 있다.

### 힙 크기 설정

```bash
# 초기 힙 512MB, 최대 힙 2GB
java -Xms512m -Xmx2g -jar app.jar
```

운영 환경에서는 `-Xms`와 `-Xmx`를 같은 값으로 설정하는 경우가 많다. 힙 크기가 동적으로 변할 때 OS에 메모리를 요청하고 반환하는 오버헤드가 생기기 때문이다.

## 객체 참조 흐름

자바에서 변수에 객체를 대입하면, 변수에는 객체 자체가 아니라 **힙에 있는 객체의 참조값(메모리 주소)**이 저장된다.

```java
public class ReferenceExample {
    public static void main(String[] args) {
        // 1. Person 객체가 힙에 생성된다
        // 2. p1에는 힙 주소(참조값)가 저장된다 (스택에 위치)
        Person p1 = new Person("김철수", 30);

        // 3. p2에 p1의 참조값을 복사한다
        //    p1과 p2는 같은 객체를 가리킨다
        Person p2 = p1;

        p2.setName("이영희");

        // "이영희" 출력 — p1과 p2가 같은 객체를 참조하고 있기 때문
        System.out.println(p1.getName());
    }
}
```

이걸 그림으로 보면:

```
스택                          힙
┌──────────────┐         ┌──────────────────┐
│ p1 [0x100]───┼────────▶│ Person           │
├──────────────┤    ┌───▶│ name = "이영희"  │
│ p2 [0x100]───┼────┘    │ age  = 30        │
└──────────────┘         └──────────────────┘
```

기본형(primitive) 변수는 스택에 값 자체가 저장되지만, 참조형(reference) 변수는 스택에 주소만 저장되고 실제 데이터는 힙에 있다. 메서드 매개변수로 객체를 넘기면 참조값이 복사되는 것이지 객체가 복사되는 게 아니다. 이 차이를 모르면 의도치 않게 원본 객체가 변경되는 버그를 만들 수 있다.

## String Pool

자바에서 문자열 리터럴은 힙 내의 **String Pool**이라는 특별한 영역에 저장된다.

```java
public class StringPoolExample {
    public static void main(String[] args) {
        // 리터럴로 생성 → String Pool에 저장
        String s1 = "hello";
        String s2 = "hello";

        // 같은 리터럴이므로 Pool에서 같은 객체를 참조
        System.out.println(s1 == s2);  // true

        // new로 생성 → 힙에 새 객체 생성 (Pool과 별개)
        String s3 = new String("hello");
        System.out.println(s1 == s3);  // false

        // intern()으로 Pool의 문자열과 매칭
        String s4 = s3.intern();
        System.out.println(s1 == s4);  // true
    }
}
```

```
힙 메모리
┌───────────────────────────────────┐
│  String Pool                      │
│  ┌───────────┐                    │
│  │  "hello"  │ ◀── s1, s2, s4    │
│  └───────────┘                    │
│                                    │
│  ┌───────────┐                    │
│  │  "hello"  │ ◀── s3 (별도 객체) │
│  └───────────┘                    │
└───────────────────────────────────┘
```

Java 6까지는 String Pool이 PermGen에 있어서 크기가 제한적이었다. Java 7부터 힙으로 이동했기 때문에 GC 대상이 되고, 메모리 관리가 나아졌다.

실무에서 주의할 점: `==`로 문자열을 비교하면 참조값을 비교하기 때문에 의도와 다른 결과가 나올 수 있다. 문자열 비교는 항상 `equals()`를 써야 한다. `==`가 `true`를 반환하는 건 String Pool 덕분이지, 보장되는 동작이 아니다.

## 메모리 할당 과정

`new Person("김철수", 30)` 을 실행하면 내부적으로 다음 순서로 메모리가 할당된다:

1. **클래스 로딩 확인**: `Person` 클래스가 메서드 영역에 로딩됐는지 확인한다. 안 됐으면 클래스 로더가 바이트코드를 읽어서 메서드 영역에 올린다
2. **힙에 메모리 할당**: Eden 영역에서 객체 크기만큼 메모리를 확보한다. 이때 두 가지 방식이 있다:
    - **Bump-the-pointer**: 힙이 연속적으로 정리되어 있으면, 마지막 할당 위치에서 포인터만 이동
    - **Free List**: 메모리가 단편화되어 있으면, 빈 공간 목록에서 적절한 크기를 찾음
3. **메모리 초기화**: 할당된 메모리를 0/null로 초기화한다 (int는 0, boolean은 false, 참조는 null)
4. **객체 헤더 설정**: 어떤 클래스의 인스턴스인지, 해시코드, GC 나이 등의 메타 정보를 기록한다
5. **생성자 실행**: `<init>` 메서드가 실행되어 필드에 실제 값을 대입한다

멀티스레드 환경에서는 여러 스레드가 동시에 Eden에 객체를 할당하면 충돌이 발생할 수 있다. 이를 피하기 위해 각 스레드에 **TLAB(Thread Local Allocation Buffer)**라는 작은 영역을 미리 할당해두고, 각 스레드는 자기 TLAB 안에서만 객체를 생성한다. TLAB이 가득 차면 새 TLAB을 할당받는다.

## 메모리 누수 패턴과 진단

자바는 GC가 메모리를 관리하지만, 참조를 놓치지 않으면 GC가 수거하지 못한다. 아래는 실무에서 자주 만나는 누수 패턴이다.

### static 컬렉션에 계속 추가하는 경우

```java
public class CacheService {
    // 애플리케이션이 살아있는 동안 이 Map은 절대 GC되지 않는다
    private static final Map<String, Object> cache = new HashMap<>();

    public void put(String key, Object value) {
        cache.put(key, value);  // 넣기만 하고 빼지 않으면 메모리가 계속 증가
    }
}
```

크기 제한이 없는 캐시를 직접 구현하면 이런 문제가 생긴다. `LinkedHashMap`을 상속해서 `removeEldestEntry()`를 오버라이드하거나, Caffeine 같은 라이브러리를 쓰는 게 낫다.

### 진단 도구

```bash
# 힙 사용량 요약
jmap -heap <PID>

# 힙 덤프 생성
jmap -dump:format=b,file=heap.hprof <PID>

# OOM 발생 시 자동으로 덤프 생성하도록 설정
java -XX:+HeapDumpOnOutOfMemoryError \
     -XX:HeapDumpPath=/tmp/oom-dump.hprof \
     -jar app.jar

# 실시간 GC 모니터링 (1초 간격)
jstat -gcutil <PID> 1000
```

생성된 `.hprof` 파일은 Eclipse MAT이나 VisualVM으로 분석한다. MAT의 Leak Suspects 리포트가 누수 원인을 자동으로 찾아주니까, 덤프 파일을 열면 이걸 먼저 확인하면 된다.

## GC 알고리즘 선택

운영 환경에서 어떤 GC를 쓸지는 애플리케이션 특성에 따라 다르다.

| GC | Java 버전 | 특징 | 적합한 상황 |
|---|---|---|---|
| G1GC | 9+ 기본값 | 힙을 리전 단위로 관리, STW 시간 예측 가능 | 대부분의 서버 애플리케이션 |
| ZGC | 15+ | STW 10ms 이하, 대용량 힙 지원 | 지연 시간에 민감한 서비스 |
| Parallel GC | | 처리량(throughput) 우선 | 배치 처리, 대량 데이터 연산 |
| Shenandoah | | ZGC와 유사, Red Hat 계열에서 지원 | OpenJDK 기반 환경 |

```bash
# G1GC (별도 설정 없으면 Java 9+에서 기본)
java -XX:+UseG1GC -jar app.jar

# ZGC
java -XX:+UseZGC -jar app.jar

# GC 로그 남기기 (Java 9+)
java -Xlog:gc*:file=gc.log:time,level,tags -jar app.jar
```

GC 로그에서 Full GC가 자주 발생하면 힙 크기가 부족하거나 Old Generation으로 넘어가는 객체가 많다는 뜻이다. `-Xmx`를 늘리거나, 객체 생존 시간을 줄이는 방향으로 코드를 수정해야 한다.
