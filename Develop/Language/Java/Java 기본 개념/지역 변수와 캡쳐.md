## 지역 변수 캡처란

메서드 안에서 선언한 지역 변수를 익명 클래스나 람다 표현식 내부에서 참조하는 것을 **캡처(capture)**라고 한다. 자바 컴파일러는 캡처 시점에 지역 변수의 **값을 복사**해서 내부 클래스나 람다 객체에 전달한다.

```java
public void example() {
    int count = 10; // 스택에 저장

    Runnable r = () -> {
        // count의 복사본을 사용한다
        System.out.println(count);
    };

    r.run();
}
```

`count`는 `example()` 메서드가 끝나면 스택에서 사라진다. 하지만 `Runnable` 객체는 힙에 살아 있을 수 있다. 이 시간차 문제를 해결하기 위해 컴파일러가 값을 복사한다.

---

## effectively final 제약

캡처되는 지역 변수는 반드시 `final`이거나 **effectively final**(선언 후 값이 변경되지 않는 변수)이어야 한다.

```java
public void invalid() {
    int value = 5;

    Runnable r = () -> System.out.println(value);

    value = 10; // 컴파일 오류: Variable used in lambda should be final or effectively final
}
```

**왜 이런 제약이 있는가?**

캡처는 값 복사다. 원본 변수가 바뀌면 복사본과 값이 달라진다. 개발자는 같은 변수라고 생각하지만 실제로는 서로 다른 메모리에 저장된 별개의 값이다. 이 불일치를 원천 차단하기 위해 컴파일러가 변경을 금지한다.

원시 타입은 값 자체가 복사되고, 참조 타입은 참조(주소)가 복사된다. 참조 타입의 경우 참조 자체를 바꾸는 건 안 되지만 객체 내부 상태를 바꾸는 건 가능하다.

```java
public void referenceCapture() {
    final StringBuilder sb = new StringBuilder("hello");

    Runnable r = () -> sb.append(" world"); // 객체 상태 변경은 허용

    r.run();
    System.out.println(sb); // "hello world"
}
```

## 스택-힙 메모리 흐름

캡처가 발생할 때 메모리에서 일어나는 일을 단계별로 보자.

```
[메서드 호출 시점]

  스택 (Stack)              힙 (Heap)
  ┌──────────────┐
  │ example()    │
  │  count = 10  │
  │              │
  └──────────────┘

[람다 객체 생성 시점 — 캡처 발생]

  스택 (Stack)              힙 (Heap)
  ┌──────────────┐         ┌─────────────────────┐
  │ example()    │         │ Lambda 객체          │
  │  count = 10 ─┼─복사──→ │  captured_count = 10 │
  │  r ──────────┼────────→│                     │
  └──────────────┘         └─────────────────────┘

[메서드 종료 후]

  스택 (Stack)              힙 (Heap)
  ┌──────────────┐         ┌─────────────────────┐
  │ (스택 프레임  │         │ Lambda 객체          │
  │  소멸됨)     │         │  captured_count = 10 │
  └──────────────┘         │  (여전히 유효)       │
                           └─────────────────────┘
```

핵심: 스택의 지역 변수는 메서드 종료와 함께 사라지지만, 힙의 복사본은 람다/익명 클래스 객체가 GC될 때까지 유지된다.

## 바이트코드로 보는 캡처 동작

실제로 컴파일러가 캡처를 어떻게 처리하는지 바이트코드 수준에서 확인할 수 있다.

### 익명 클래스의 캡처

```java
public class AnonymousCapture {
    public void run() {
        int value = 42;

        Runnable r = new Runnable() {
            @Override
            public void run() {
                System.out.println(value);
            }
        };
    }
}
```

컴파일하면 `AnonymousCapture$1.class` 파일이 생성된다. `javap -c`로 보면:

```
// AnonymousCapture$1 (익명 클래스)
class AnonymousCapture$1 implements java.lang.Runnable {
    final int val$value;                      // 캡처된 값을 저장하는 필드
    final AnonymousCapture this$0;            // 외부 클래스 참조

    AnonymousCapture$1(AnonymousCapture, int); // 생성자에서 값을 받음
        aload_0
        aload_1
        putfield AnonymousCapture$1.this$0    // 외부 클래스 참조 저장
        aload_0
        iload_2
        putfield AnonymousCapture$1.val$value // 캡처한 값 저장
}
```

컴파일러가 하는 일:

1. 익명 클래스에 `val$value`라는 **synthetic 필드**를 만든다
2. 생성자 파라미터로 캡처할 값을 받는다
3. 생성자에서 파라미터 값을 필드에 저장한다
4. 외부 클래스 참조(`this$0`)도 함께 전달한다

### 람다의 캡처

```java
public class LambdaCapture {
    public void run() {
        int value = 42;

        Runnable r = () -> System.out.println(value);
    }
}
```

`javap -c -p`로 보면:

```
public void run();
    Code:
        bipush 42
        istore_1                              // value = 42
        iload_1
        invokedynamic #2, 0                   // InvokeDynamic #0:run:(I)Ljava/lang/Runnable;
                                              // value를 인자로 전달
        astore_2
        return

private static void lambda$run$0(int);        // 캡처된 값을 파라미터로 받는 static 메서드
    Code:
        getstatic java/lang/System.out
        iload_0
        invokevirtual java/io/PrintStream.println:(I)V
        return
```

람다는 익명 클래스와 다르게 처리된다:

1. 별도의 `.class` 파일을 생성하지 않는다
2. `invokedynamic` 명령어로 런타임에 구현체를 생성한다
3. 캡처된 값은 생성된 **private static 메서드의 파라미터**로 전달된다
4. `LambdaMetafactory`가 실제 구현 클래스를 런타임에 만든다

## 익명 클래스 vs 람다: 컴파일러 관점의 차이

겉보기에는 비슷하지만 컴파일러가 처리하는 방식이 근본적으로 다르다.

### 클래스 생성 방식

| 구분 | 익명 클래스 | 람다 |
|------|-----------|------|
| 클래스 파일 | 컴파일 타임에 `Outer$1.class` 생성 | 런타임에 `LambdaMetafactory`가 생성 |
| 생성 명령어 | `new` + `invokespecial` | `invokedynamic` |
| 캡처 저장 위치 | `val$변수명` synthetic 필드 | static 메서드의 파라미터 |
| 외부 클래스 참조 | `this$0` 필드로 항상 보유 | 사용하지 않으면 참조하지 않음 |

### this 참조 차이

이 차이가 실무에서 문제가 되는 경우가 있다.

```java
public class ThisExample {
    private String name = "outer";

    public void test() {
        // 익명 클래스: this는 익명 클래스 자신
        Runnable anonymous = new Runnable() {
            @Override
            public void run() {
                // System.out.println(this.name); // 컴파일 오류 - Runnable에 name 없음
                System.out.println(ThisExample.this.name); // 외부 클래스 접근
            }
        };

        // 람다: this는 외부 클래스(ThisExample)
        Runnable lambda = () -> {
            System.out.println(this.name); // 바로 외부 클래스의 name에 접근
        };
    }
}
```

바이트코드 수준에서 보면:

- 익명 클래스: `aload_0`이 익명 클래스 인스턴스(`this`)를 가리킨다. 외부 클래스에 접근하려면 `this$0` 필드를 거쳐야 한다.
- 람다: `aload_0`이 외부 클래스 인스턴스를 직접 가리킨다. 람다는 자신만의 `this`가 없다.

### 메모리 영향

```
[익명 클래스 사용 시]

  힙 (Heap)
  ┌────────────────────────┐
  │ AnonymousCapture$1     │
  │  this$0 ───────────────┼──→ 외부 클래스 인스턴스 (항상 참조)
  │  val$value = 42        │
  └────────────────────────┘

[람다 사용 시 — value만 캡처, this 미사용]

  힙 (Heap)
  ┌────────────────────────┐
  │ Lambda 구현체           │
  │  arg$1 = 42            │    외부 클래스 참조 없음
  └────────────────────────┘
```

익명 클래스는 외부 클래스를 사용하지 않아도 `this$0` 참조를 가진다. 이 때문에 외부 클래스가 GC되지 못하는 메모리 누수가 발생할 수 있다. 람다는 외부 클래스의 인스턴스 멤버를 사용하지 않으면 참조를 잡지 않는다.

## 실무에서 주의할 점

### 반복문 안에서의 캡처

```java
// 잘못된 코드 — 컴파일 오류
for (int i = 0; i < 10; i++) {
    new Thread(() -> System.out.println(i)).start(); // i는 effectively final이 아님
}

// 해결: 루프 변수를 지역 변수에 복사
for (int i = 0; i < 10; i++) {
    int captured = i; // 매 반복마다 새로운 변수
    new Thread(() -> System.out.println(captured)).start();
}
```

### 캡처 우회: 배열이나 AtomicInteger 사용

effectively final 제약을 피해야 하는 상황이 있다. 스트림 안에서 카운터를 증가시키는 경우가 대표적이다.

```java
// 원시 타입은 캡처 후 변경 불가
// int count = 0;
// list.forEach(item -> count++); // 컴파일 오류

// 방법 1: AtomicInteger
AtomicInteger count = new AtomicInteger(0);
list.forEach(item -> count.incrementAndGet());

// 방법 2: 단일 요소 배열 (참조는 final, 내용은 변경 가능)
int[] count2 = {0};
list.forEach(item -> count2[0]++);
```

배열 트릭은 동작하지만 의도가 명확하지 않다. 멀티스레드 환경이면 `AtomicInteger`를 쓰고, 단일 스레드면 스트림 대신 for 루프를 쓰는 게 낫다.

### 익명 클래스의 메모리 누수

내부 클래스가 외부 클래스를 `this$0`로 참조하기 때문에, 콜백이나 리스너를 익명 클래스로 등록하면 외부 클래스가 GC되지 않는 문제가 생길 수 있다.

```java
public class Outer {
    private byte[] largeData = new byte[10_000_000]; // 10MB

    public Runnable createCallback() {
        // 익명 클래스: this$0로 Outer 인스턴스를 참조 → largeData도 GC 불가
        return new Runnable() {
            @Override
            public void run() {
                System.out.println("callback");
            }
        };
    }

    public Runnable createLambdaCallback() {
        // 람다: 외부 인스턴스 멤버 미사용 → Outer 참조 없음
        return () -> System.out.println("callback");
    }
}
```

오래 유지되는 콜백에 익명 클래스를 사용하면 의도치 않게 큰 객체를 잡고 있게 된다. 특별한 이유가 없으면 람다를 쓰는 게 안전하다.

## 중첩 클래스 유형별 캡처 가능 범위

자바의 중첩 클래스 네 가지 유형은 각각 접근 가능한 범위가 다르다.

| 유형 | 외부 인스턴스 접근 | 지역 변수 캡처 | 정적 멤버 접근 |
|------|:---:|:---:|:---:|
| 정적 중첩 클래스 (`static class Inner`) | X | X | O |
| 인스턴스 내부 클래스 (`class Inner`) | O | X | O |
| 지역 클래스 (메서드 안의 `class Local`) | O | O (effectively final) | O |
| 익명 클래스 (`new Interface() {}`) | O | O (effectively final) | O |

```java
public class NestedClassAccess {
    private int instanceField = 1;
    private static int staticField = 2;

    public void method() {
        int localVar = 3;

        // 지역 클래스: instanceField, staticField, localVar 모두 접근 가능
        class Local {
            void print() {
                System.out.println(instanceField);  // O
                System.out.println(staticField);     // O
                System.out.println(localVar);        // O (effectively final)
            }
        }
    }

    // 정적 중첩 클래스: staticField만 접근 가능
    static class StaticNested {
        void print() {
            // System.out.println(instanceField); // 컴파일 오류
            System.out.println(staticField);       // O
        }
    }
}
```

## 정리

- 지역 변수 캡처는 **값 복사**다. 스택의 값을 힙 객체에 복사하는 것이다.
- effectively final 제약은 복사본과 원본의 불일치를 막기 위해 존재한다.
- 익명 클래스는 컴파일 타임에 별도 클래스 파일을 만들고, 캡처된 값을 `val$` synthetic 필드에 저장한다.
- 람다는 `invokedynamic`으로 런타임에 처리되며, 캡처된 값은 static 메서드 파라미터로 전달된다.
- 익명 클래스는 외부 클래스를 항상 참조(`this$0`)하지만, 람다는 필요할 때만 참조한다. 이 차이가 메모리 누수 여부를 결정한다.
