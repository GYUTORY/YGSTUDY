# 단일 책임 원칙 (Single Responsibility Principle)

## 정의를 다시 짚는 이유

흔히 SRP를 "클래스는 한 가지 일만 해야 한다"고 외운다. 그런데 이 정의가 실무에서 가장 헷갈린다. "한 가지 일"의 단위가 무엇인지 사람마다 다르게 해석하기 때문이다. `UserService`가 회원가입과 로그인을 같이 하면 SRP 위반인가? `OrderService`에 결제 로직이 들어가면 위반인가? 이 질문에 일관되게 답하지 못하면 코드 리뷰 때마다 결론이 달라진다.

Robert C. Martin이 나중에 정의를 바꿔 적은 표현이 더 정확하다.

> A module should be responsible to one, and only one, actor.

여기서 actor는 변경을 요구하는 사람(또는 부서, 시스템)이다. 결제 정책을 바꾸는 사람과 회원 약관을 바꾸는 사람이 다르다면, 이 둘은 다른 클래스에 있어야 한다. SRP는 코드의 모양이 아니라 **변경 압력의 출처가 같은가**를 묻는 원칙이다.

5년차쯤 되면 이 관점이 왜 중요한지 체감한다. 신입 때는 "기능이 다르니까 분리"라고 생각했다면, 이제는 "누가 이걸 바꾸자고 들어오는가"를 먼저 묻게 된다.

## SRP 위반을 식별하는 실무 패턴

### God Class: 진입 흐름이 한 클래스로 빨려 들어가는 경우

가장 흔한 SRP 위반은 God Class다. 이름은 `OrderManager`, `MemberService`, `BillingHelper` 같은 식으로 무난해 보이지만 열어보면 아래 같은 모양이 나온다.

```java
public class OrderService {

    public Order placeOrder(OrderRequest req) {
        // 1. 입력 검증
        if (req.getItems().isEmpty()) throw new IllegalArgumentException();

        // 2. 재고 차감
        for (Item item : req.getItems()) {
            int stock = jdbc.queryForObject("SELECT stock FROM ...", Integer.class);
            if (stock < item.getQty()) throw new OutOfStockException();
            jdbc.update("UPDATE inventory SET stock = stock - ? ...");
        }

        // 3. 결제 호출
        HttpResponse pgRes = httpClient.post(pgUrl, req.toPgPayload());
        if (pgRes.statusCode() != 200) throw new PaymentFailedException();

        // 4. 주문 저장
        Order order = new Order(...);
        jdbc.update("INSERT INTO orders ...");

        // 5. 알림 발송
        smtpClient.send(req.getEmail(), "주문 완료", buildHtml(order));
        slackClient.notify("#sales", "신규 주문: " + order.getId());

        // 6. 통계 적재
        kafkaProducer.send("order-events", order.toJson());

        return order;
    }
}
```

이 클래스는 변경 사유가 최소 6개다. 재고 정책이 바뀌면 수정, 결제 게이트웨이가 바뀌어도 수정, 알림 채널이 추가돼도 수정, 분석팀이 이벤트 스키마를 바꿔도 수정. 한 메서드에 운영팀, 결제팀, 마케팅팀, 데이터팀이 모두 들어와 있다. SRP를 가장 정확히 위반한 형태다.

식별 신호는 단순하다.

- 클래스 길이가 500줄을 넘기 시작한다
- import 문에 SMTP, HTTP 클라이언트, Kafka, JDBC가 동시에 등장한다
- 이 클래스를 수정한 git log를 뽑으면 변경 이유가 매번 다르다 (`feat: 결제`, `fix: 메일`, `chore: 재고 SQL 튜닝`)

특히 마지막 신호가 강력하다. **git log의 변경 사유가 한 클래스에서 여러 부서로 갈리면, 그 클래스는 SRP를 위반하고 있다.** 코드만 봐서는 안 보일 때 git이 알려준다.

### 변경 이유 다중성: 같은 메서드를 여러 이유로 고치는 패턴

God Class보다 더 교묘한 위반이 있다. 클래스는 작은데 한 메서드가 여러 정책을 동시에 들고 있는 경우다.

```java
public BigDecimal calculatePrice(Order order, Member member) {
    BigDecimal price = order.getOriginalPrice();

    // 등급별 할인
    if (member.getGrade() == Grade.VIP) price = price.multiply(new BigDecimal("0.9"));

    // 쿠폰 할인
    if (order.getCoupon() != null) price = price.subtract(order.getCoupon().getAmount());

    // 시즌 프로모션
    if (LocalDate.now().getMonth() == Month.DECEMBER) {
        price = price.multiply(new BigDecimal("0.95"));
    }

    // 배송비
    price = price.add(order.getRegion().getShippingFee());

    return price;
}
```

마케팅팀이 시즌 프로모션 규칙을 바꾸면 이 메서드를 건드린다. CS팀이 쿠폰 정책을 바꿔도 건드린다. 물류팀이 배송비를 바꿔도 건드린다. 메서드 한 개에 actor가 셋이다. 이런 건 별도 정책 객체로 빼야 한다 (`DiscountPolicy`, `ShippingFeePolicy` 등).

### 데이터 클래스인데 행위가 흩어진 경우

반대 패턴도 있다. `User`라는 도메인 객체에 회원가입 검증, 비밀번호 암호화, 권한 판정, 활동 로그 기록이 모두 들어 있는 경우다. 도메인 객체는 자기 상태에 대한 불변식만 책임지면 된다. 인증, 인가, 감사 로그는 다 다른 actor의 영역이다.

## Service 레이어를 어떻게 쪼개는가

실무에서 SRP를 적용하는 가장 큰 결정은 Service를 어떻게 자르느냐다. 정답은 없지만 자주 쓰는 기준이 있다.

### Use Case 단위로 쪼개기

`OrderService` 하나에 `placeOrder`, `cancelOrder`, `refundOrder`, `searchOrders`를 다 넣지 말고, 각 use case를 별도 클래스로 분리한다.

```java
public class PlaceOrderService {
    private final InventoryReducer inventoryReducer;
    private final PaymentGateway paymentGateway;
    private final OrderRepository orderRepository;
    private final OrderEventPublisher eventPublisher;

    public Order place(OrderRequest req) {
        inventoryReducer.reduce(req.getItems());
        PaymentResult payment = paymentGateway.charge(req.toPaymentCommand());
        Order order = orderRepository.save(Order.from(req, payment));
        eventPublisher.publish(new OrderPlacedEvent(order));
        return order;
    }
}
```

`PlaceOrderService`는 "주문을 접수한다"는 흐름의 오케스트레이션만 담당한다. 재고 차감 정책이 바뀌면 `InventoryReducer`만 수정한다. 결제 게이트웨이가 PG사에서 자체 결제로 바뀌면 `PaymentGateway` 구현체만 갈아끼운다. 알림 정책이 바뀌면 `OrderEventPublisher`를 구독하는 핸들러를 추가한다. `PlaceOrderService` 자체는 손대지 않는다.

이렇게 자르면 클래스 수는 늘어난다. 하지만 한 클래스를 수정할 때 영향 범위가 줄어든다. 5년차쯤 되면 클래스 수가 늘어나는 비용보다 영향 범위가 좁아지는 이득이 훨씬 크다는 걸 알게 된다.

### Command-Query 분리

읽기와 쓰기를 같은 Service에 두는 경우가 많은데, 어느 정도 규모가 커지면 분리가 답이다.

```java
public class OrderCommandService {
    public Order place(OrderRequest req) { ... }
    public void cancel(Long orderId) { ... }
}

public class OrderQueryService {
    public OrderDetailView findById(Long orderId) { ... }
    public Page<OrderListView> search(OrderSearchCondition cond) { ... }
}
```

쓰기는 트랜잭션, 도메인 규칙, 이벤트 발행이 중요하다. 읽기는 조회 성능, DTO 변환, 캐싱이 중요하다. 둘은 actor가 다르다. 운영팀이 "주문 검색이 너무 느려요"라고 들어오는 사람과, 기획팀이 "취소 정책 바꿉시다" 들어오는 사람이 다르다.

## SRP가 테스트 작성을 어떻게 바꾸는가

SRP의 진짜 효과는 테스트를 작성할 때 드러난다. God Class를 테스트하려면 모든 의존성을 mock해야 한다.

```java
@Test
void placeOrder_test() {
    // 위에 있던 OrderService를 테스트하려면
    JdbcTemplate jdbc = mock(JdbcTemplate.class);
    HttpClient httpClient = mock(HttpClient.class);
    SmtpClient smtpClient = mock(SmtpClient.class);
    SlackClient slackClient = mock(SlackClient.class);
    KafkaProducer kafkaProducer = mock(KafkaProducer.class);

    when(jdbc.queryForObject(...)).thenReturn(10);
    when(httpClient.post(any(), any())).thenReturn(...);
    // ... mock 설정 30줄
}
```

mock 설정만 30줄이고, 한 테스트가 6가지 책임을 모두 알아야 한다. 결제 로직 하나 바꿨는데 알림 mock 설정도 깨지는 일이 빈번하다.

쪼개진 후의 테스트는 다르다.

```java
@Test
void place_재고차감과_결제와_저장이_순서대로_호출된다() {
    InventoryReducer inventory = mock(InventoryReducer.class);
    PaymentGateway payment = mock(PaymentGateway.class);
    when(payment.charge(any())).thenReturn(PaymentResult.success("tx-1"));
    OrderRepository repo = mock(OrderRepository.class);
    when(repo.save(any())).thenAnswer(inv -> inv.getArgument(0));
    OrderEventPublisher publisher = mock(OrderEventPublisher.class);

    PlaceOrderService service = new PlaceOrderService(inventory, payment, repo, publisher);
    service.place(sampleRequest());

    InOrder order = inOrder(inventory, payment, repo, publisher);
    order.verify(inventory).reduce(any());
    order.verify(payment).charge(any());
    order.verify(repo).save(any());
    order.verify(publisher).publish(any());
}
```

이 테스트가 검증하는 건 오케스트레이션이다. 재고 차감의 세부 로직은 `InventoryReducer` 자체의 테스트에서 본다. 각 클래스의 테스트가 그 클래스의 책임에 정확히 1:1로 대응된다.

여기서 한 가지 더. SRP를 잘 따른 코드는 단위 테스트가 빨라진다. 데이터베이스나 외부 API를 직접 부르는 코드가 한 클래스에 응집되어 있으면, 도메인 로직만 테스트하고 싶어도 통합 테스트가 되어 버린다. 책임이 분리되면 도메인 클래스는 순수 자바로 테스트할 수 있다.

## SRP와 OCP, DIP의 관계

SOLID는 따로 외우지만 실제로는 SRP가 OCP와 DIP의 전제조건처럼 작동한다.

OCP(개방-폐쇄 원칙)는 "확장에는 열려 있고 변경에는 닫혀 있어야 한다"이다. 한 클래스에 여러 책임이 섞여 있으면 OCP를 적용할 수가 없다. `OrderService`에 결제 로직이 섞여 있는 상태에서 PG사를 추가한다면, 클래스 자체를 수정해야 한다. 그런데 `PaymentGateway` 인터페이스를 따로 두고 `OrderService`가 그것에만 의존하면, PG사 추가는 새 구현체를 만들어 끼우는 일로 끝난다. OCP가 가능해진 이유는 SRP로 결제 책임을 따로 빼냈기 때문이다.

DIP(의존성 역전 원칙)도 마찬가지다. "고수준 모듈이 저수준 모듈에 의존하지 않는다"는 원칙은, 책임이 잘 쪼개져 있어야 적용 가능하다. 책임이 한 클래스에 뭉쳐 있으면 무엇이 고수준이고 무엇이 저수준인지 구분조차 안 된다. SRP가 책임을 쪼개 놓아야 그 사이에 인터페이스를 끼울 자리가 생긴다.

순서로 보면 이렇다.

```
SRP로 책임을 분리한다
  → 분리된 책임 사이에 인터페이스가 자연스럽게 생긴다
    → DIP로 의존 방향을 뒤집을 수 있다
      → OCP로 확장 가능한 구조가 된다
```

SRP를 무시하고 OCP나 DIP만 들고 와서 인터페이스를 만들면, 인터페이스 뒤에 여전히 God Class가 숨어 있는 형태가 된다. 모양만 SOLID인 코드다.

## 과한 분리가 만드는 부작용

5년차쯤 되면 SRP를 너무 신봉한 코드의 부작용도 만나게 된다.

### 객체 폭발

`UserService`를 잘게 쪼갠다고 `UserCreator`, `UserValidator`, `UserPasswordEncoder`, `UserEmailVerifier`, `UserRoleAssigner`, `UserActivityLogger`로 6개로 만들면, 회원가입 한 번에 6개의 빈을 주입받아야 한다. 생성자 파라미터가 8개씩 늘어나기 시작한다.

```java
public class UserCreator {
    private final UserValidator validator;
    private final PasswordEncoder encoder;
    private final EmailVerifier emailVerifier;
    private final RoleAssigner roleAssigner;
    private final ActivityLogger logger;
    private final UserRepository repository;
    private final EventPublisher eventPublisher;
    private final WelcomeMailSender mailSender;

    public UserCreator(...) { /* 8개 파라미터 */ }
}
```

이 정도면 SRP를 지키려다 클래스 자체가 뭐 하는 건지 안 보이는 단계다. 흐름을 따라가려면 8개 클래스를 모두 열어봐야 한다.

### 추적 비용 증가

요청 하나가 처리되는 흐름을 디버깅할 때, 클래스를 5번 6번 점프해야 끝까지 따라갈 수 있다면 그 자체가 비용이다. 특히 신규 입사자가 코드를 읽을 때 이 비용이 크다.

### 단일 actor가 분리된 경우

actor가 같은데 클래스만 쪼개 놓으면 항상 같이 변경된다. `UserNameValidator`, `UserEmailValidator`, `UserPhoneValidator`를 각각 클래스로 만들었는데 회원 정보 정책 변경은 한 번에 들어온다면, 이 셋은 한 클래스의 메서드 셋이었어야 했다. 분리의 기준이 actor가 아니라 단어였던 것이다.

### 적정선 잡는 기준

경험적으로 다음 신호가 보일 때 분리를 멈추거나 합치는 걸 고려한다.

- 항상 같이 수정되는 클래스 두 개가 있다 (variance가 같다)
- 한 클래스가 다른 클래스를 1:1로만 사용하고 다른 곳에서 쓰이지 않는다
- 클래스를 열었는데 메서드가 한 개고 그 한 메서드는 두세 줄짜리다

이런 경우 합쳤을 때 더 읽기 쉬워진다면 합치는 게 SRP의 정신에 맞다. SRP는 클래스를 잘게 쪼개라는 원칙이 아니라, 변경 압력의 단위로 정렬하라는 원칙이다.

## 5년차가 SRP를 보는 관점

신입 때 SRP를 코드 모양으로 외웠다면, 5년차쯤 되면 git log와 조직도로 본다. 어떤 클래스가 자주 충돌나는지, 어느 팀이 어느 코드를 자주 건드리는지를 보면 자연스럽게 분리선이 그어진다.

리뷰할 때 던지는 질문도 바뀐다.

- "이 클래스를 누가 바꾸자고 들어오겠는가?"
- "이 변경이 들어왔을 때 다른 actor의 코드가 같이 수정되어야 하는가?"
- "이 두 메서드는 항상 같이 변경되는가, 아니면 따로 변경되는가?"

코드의 외형보다 변경의 패턴을 본다. SRP가 가르쳐 주는 진짜 관점은 그것이다.
