# 프로세스의 자원 구조

프로세스의 메모리 영역 기본 구조(Text, Data, BSS, Heap, Stack)와 각 영역의 역할은 [프로세스의 구조](프로세스의%20구조.md#프로세스-메모리-구조) 문서에 정리되어 있다. 이 문서에서는 각 영역의 세부 특성과, 실제 리눅스 환경에서 메모리 레이아웃을 확인하고 문제를 추적하는 방법을 다룬다.

---

## 각 영역의 세부 특성

### 코드 영역 (Text Segment)

컴파일된 기계어 코드가 올라가는 영역이다. 읽기 전용이고 실행 권한이 있다.

같은 프로그램을 여러 개 실행하면 코드 영역은 물리 메모리에서 공유된다. nginx 워커 프로세스 10개를 띄워도 코드 영역은 하나만 물리 메모리에 올라간다. `/proc/[pid]/maps`에서 같은 inode를 가리키는 `r-xp` 영역이 이것이다.

### 데이터 영역 (Data Segment)

세 개의 섹션으로 나뉜다:

**`.data` — 초기화된 전역/정적 변수**

```c
int count = 100;           // .data에 저장
static char name[] = "app"; // .data에 저장
```

바이너리 파일에 초기값이 그대로 들어가기 때문에 초기화된 전역 변수가 많으면 바이너리 크기가 커진다.

**`.bss` — 초기화되지 않은 전역/정적 변수**

```c
int buffer[10000];         // .bss에 저장
static int counter;        // .bss에 저장 (0으로 초기화)
```

바이너리에는 크기 정보만 기록된다. 프로그램 로드 시 커널이 0으로 채운다. 그래서 `int arr[1000000] = {0};`으로 선언하면 `.data`에 4MB가 들어가지만, `int arr[1000000];`으로 선언하면 `.bss`에 크기만 기록되어 바이너리 크기에 영향이 없다.

```bash
# .data와 .bss 크기 확인
$ size /usr/bin/bash
   text    data     bss     dec     hex filename
 925433   36344   23264  985041   f0bd1 /usr/bin/bash
```

**`.rodata` — 읽기 전용 데이터**

```c
const int MAX = 1024;              // .rodata
printf("hello world\n");          // "hello world\n" 문자열이 .rodata에 저장
```

이 영역에 쓰기를 시도하면 Segmentation fault가 발생한다.

### 힙 영역 (Heap Segment)

`malloc()`, `new`, `mmap()` 등으로 동적 할당하는 영역이다. 낮은 주소에서 높은 주소로 성장한다.

glibc의 `malloc()`은 내부적으로 두 가지 방식을 쓴다:
- 작은 할당(기본 128KB 미만): `brk()`/`sbrk()`로 힙 영역을 확장
- 큰 할당(기본 128KB 이상): `mmap()`으로 별도 매핑 생성

```bash
# 힙 영역 확인
$ cat /proc/<pid>/maps | grep heap
01a3e000-01a80000 rw-p 00000000 00:00 0  [heap]
```

`mmap()`으로 할당된 메모리는 `[heap]`에 안 잡힌다. 큰 객체를 많이 할당하는 프로그램에서 힙 크기만 보고 메모리 사용량을 판단하면 빠진 부분이 생긴다.

### 스택 영역 (Stack Segment)

함수 호출마다 스택 프레임이 쌓인다. 높은 주소에서 낮은 주소로 성장한다.

스택 프레임 하나에 들어가는 것:
- 함수 매개변수
- 반환 주소 (호출한 곳으로 돌아갈 주소)
- 이전 프레임 포인터 (rbp)
- 지역 변수

```bash
# 현재 스택 크기 제한 확인
$ ulimit -s
8192  # 8MB

# 특정 프로세스의 스택 영역 확인
$ cat /proc/<pid>/maps | grep stack
7ffd4b200000-7ffd4b221000 rw-p 00000000 00:00 0  [stack]
```

스레드별로 별도의 스택이 생긴다. 기본 스레드 스택 크기는 pthread에서 8MB다. 스레드 수백 개를 만들면 스택만으로 수 GB를 먹는 경우가 있다.

```bash
# Java에서 스레드 스택 크기 조절
$ java -Xss512k -jar app.jar
```

## 실제 메모리 레이아웃 확인하기

### /proc/[pid]/maps

프로세스의 가상 메모리 매핑을 전부 보여준다.

```bash
$ cat /proc/$(pidof nginx)/maps
# 주소범위              퍼미션 오프셋  디바이스 inode  경로
55d3a4400000-55d3a44c8000 r-xp 00000000 08:01 131073 /usr/sbin/nginx  # Text
55d3a46c7000-55d3a46cb000 r--p 000c7000 08:01 131073 /usr/sbin/nginx  # .rodata
55d3a46cb000-55d3a46e8000 rw-p 000cb000 08:01 131073 /usr/sbin/nginx  # .data
55d3a5b30000-55d3a5d80000 rw-p 00000000 00:00 0      [heap]
7f2a8c000000-7f2a8c021000 rw-p 00000000 00:00 0                       # mmap 영역
7f2a8d100000-7f2a8d2c0000 r-xp 00000000 08:01 262145 /lib/x86_64-linux-gnu/libc-2.31.so
7ffd6bc9e000-7ffd6bcbf000 rw-p 00000000 00:00 0      [stack]
7ffd6bdf1000-7ffd6bdf4000 r--p 00000000 00:00 0      [vvar]
7ffd6bdf4000-7ffd6bdf5000 r-xp 00000000 00:00 0      [vdso]
```

퍼미션 필드 읽는 법:
- `r-xp`: 읽기 + 실행, private → 코드 영역
- `rw-p`: 읽기 + 쓰기, private → 데이터/힙/스택
- `r--p`: 읽기만, private → .rodata
- `p`: private (COW), `s`: shared

공유 라이브러리(`.so`)도 여기에 매핑된다. 프로세스가 어떤 라이브러리를 로드하고 있는지 확인할 때 유용하다.

### pmap

`/proc/[pid]/maps`를 읽기 좋게 정리해서 보여준다.

```bash
$ pmap -x $(pidof nginx)
Address           Kbytes     RSS   Dirty Mode  Mapping
000055d3a4400000     800     640       0 r-x-- nginx
000055d3a46c7000      16      16      16 r---- nginx
000055d3a46cb000     116      80      80 rw--- nginx
000055d3a5b30000    2368    1200    1200 rw---   [ anon ]   # 힙
00007f2a8d100000    1792    1024       0 r-x-- libc-2.31.so
00007ffd6bc9e000     132      32      32 rw---   [ stack ]
---------------- ------- ------- -------
total kB           12848    4800    1520
```

- **Kbytes**: 가상 메모리 크기
- **RSS** (Resident Set Size): 실제 물리 메모리에 올라간 크기
- **Dirty**: 수정되어서 디스크에 쓰기 대기 중인 페이지 크기

RSS와 Kbytes의 차이가 크면 할당만 해놓고 실제로는 접근하지 않은 페이지가 많다는 뜻이다. Linux는 lazy allocation을 하기 때문에 `malloc()`으로 1GB를 요청해도 실제로 접근하기 전까지 물리 메모리를 안 잡는다.

```bash
# RSS 기준으로 메모리 많이 먹는 매핑 확인
$ pmap -x <pid> | sort -k3 -rn | head -10
```

### /proc/[pid]/smaps — 더 상세한 메모리 정보

```bash
$ cat /proc/<pid>/smaps | head -20
55d3a5b30000-55d3a5d80000 rw-p 00000000 00:00 0  [heap]
Size:               2368 kB
KernelPageSize:        4 kB
Rss:                1200 kB
Pss:                1200 kB      # Proportional Set Size (공유 페이지를 비율로 나눔)
Shared_Clean:          0 kB
Shared_Dirty:          0 kB
Private_Clean:         0 kB
Private_Dirty:      1200 kB
Referenced:         1100 kB
Anonymous:          1200 kB
Swap:                  0 kB
```

- **Pss**: 공유 메모리를 프로세스 수로 나눈 값. 여러 프로세스의 Pss를 합치면 실제 물리 메모리 사용량에 가깝다.
- **Swap**: 스왑 아웃된 크기. 이 값이 크면 메모리가 부족한 상황이다.

## 메모리 단편화와 실무 문제

### 힙 단편화

장시간 실행되는 서버 프로세스에서 자주 발생한다. 다양한 크기의 객체를 반복적으로 할당/해제하면 빈 공간은 있는데 연속된 큰 블록을 할당할 수 없는 상태가 된다.

```c
// 단편화를 유발하는 패턴
for (int i = 0; i < 100000; i++) {
    void *small = malloc(32);
    void *large = malloc(4096);
    free(small);  // 32바이트짜리 구멍이 생김
    // large는 해제하지 않음
}
// 32바이트 구멍이 10만 개 생기지만
// 연속된 1KB 블록을 할당하려면 새 메모리를 확장해야 함
```

glibc `malloc`이 `brk()`로 확장한 힙 영역은 맨 끝 부분이 해제되기 전까지 OS에 반환되지 않는다. 중간중간 해제해도 RSS가 줄지 않는 이유가 이것이다.

```bash
# glibc malloc 통계 확인 (프로그램 안에서)
$ MALLOC_STATS_=1 ./app
# 또는 코드에서
# malloc_stats();  # 호출하면 stderr로 출력

Arena 0:
system bytes     =     135168
in use bytes     =      50000
# system bytes - in use bytes = 85168 (단편화로 낭비되는 공간)
```

`jemalloc`이나 `tcmalloc`은 이 문제를 완화한다. Redis가 jemalloc을 쓰는 이유 중 하나다.

### OOM Killer

리눅스는 메모리 overcommit을 허용한다. 물리 메모리 + 스왑보다 더 많은 메모리를 `malloc()`으로 할당할 수 있다. 실제로 물리 메모리가 부족해지면 커널의 OOM Killer가 프로세스를 강제로 죽인다.

```bash
# OOM Killer에 의해 죽은 프로세스 확인
$ dmesg | grep -i "out of memory"
[32145.123456] Out of memory: Killed process 12345 (java) total-vm:8192000kB, anon-rss:4096000kB

# 또는 journalctl로 확인
$ journalctl -k | grep -i oom
```

어떤 프로세스가 죽을지는 oom_score로 결정된다. 메모리를 많이 쓰는 프로세스일수록 점수가 높고, 먼저 죽는다.

```bash
# 프로세스별 OOM 점수 확인
$ cat /proc/<pid>/oom_score
150

# OOM 점수 조정 (-1000 ~ 1000, -1000이면 OOM Killer 대상에서 제외)
$ echo -1000 > /proc/<pid>/oom_score_adj   # 이 프로세스는 절대 죽이지 마라
$ echo 1000 > /proc/<pid>/oom_score_adj    # 이 프로세스를 먼저 죽여라
```

DB 서버처럼 죽으면 안 되는 프로세스는 `oom_score_adj`를 낮게 설정한다. systemd 유닛 파일에서 `OOMScoreAdjust=-900`으로 지정할 수 있다.

```bash
# overcommit 정책 확인
$ cat /proc/sys/vm/overcommit_memory
0   # 0: 휴리스틱 (기본값), 1: 항상 허용, 2: 물리 메모리 + 스왑 이내만 허용

# overcommit 비율 확인 (overcommit_memory=2일 때)
$ cat /proc/sys/vm/overcommit_ratio
50  # 물리 메모리의 50% + 스왑까지 허용
```

운영 환경에서 `overcommit_memory=2`로 설정하면 OOM Killer가 발동하는 대신 `malloc()`이 NULL을 반환한다. 예측 가능한 동작을 원하면 이쪽이 낫다.

### 메모리 릭 추적

**1단계: 메모리가 계속 증가하는지 확인**

```bash
# RSS 추이 모니터링 (1초 간격)
$ while true; do
    ps -o pid,rss,vsz,comm -p <pid>
    sleep 1
done

# 또는 pidstat으로
$ pidstat -r -p <pid> 1
```

RSS가 계속 올라가면 메모리 릭이 의심된다.

**2단계: 어떤 영역이 늘어나는지 확인**

```bash
# 힙이 늘어나는지 확인
$ pmap -x <pid> | grep heap
# 시간 간격을 두고 여러 번 실행해서 비교

# anonymous 매핑이 늘어나는지 확인
$ awk '/^Rss/ {sum += $2} END {print sum " kB"}' /proc/<pid>/smaps
```

**3단계: 릭 원인 추적**

```bash
# Valgrind로 메모리 릭 탐지 (C/C++)
$ valgrind --leak-check=full --show-leak-kinds=all ./app
==12345== 1,024 bytes in 1 blocks are definitely lost in loss record 1 of 1
==12345==    at 0x4C2FB0F: malloc (in /usr/lib/valgrind/vgpreload_memcheck-amd64-linux.so)
==12345==    by 0x401156: process_request (server.c:42)
==12345==    by 0x4012A3: main (server.c:78)
```

Valgrind는 프로세스를 10~50배 느리게 만들기 때문에 운영 환경에서는 쓸 수 없다. 운영 중인 프로세스에는 다른 방법을 쓴다.

```bash
# gdb로 실행 중인 프로세스에 붙어서 malloc 통계 확인
$ gdb -p <pid>
(gdb) call malloc_info(0, stdout)
(gdb) call malloc_stats()

# 또는 LD_PRELOAD로 malloc 추적 라이브러리를 끼우기
$ LD_PRELOAD=/usr/lib/libtcmalloc.so HEAPPROFILE=/tmp/heap ./app
$ pprof --text ./app /tmp/heap.0001.heap
```

**Java 프로세스의 경우:**

```bash
# 힙 덤프
$ jmap -dump:live,format=b,file=/tmp/heap.hprof <pid>

# 네이티브 메모리 추적 (JVM 외부 메모리 릭 의심 시)
$ java -XX:NativeMemoryTracking=summary -jar app.jar
$ jcmd <pid> VM.native_memory summary
```

Java에서 RSS가 -Xmx보다 훨씬 큰 경우는 네이티브 메모리 릭(JNI, DirectByteBuffer, 스레드 스택 등)이 원인인 경우가 많다.

## /proc/[pid]/status로 전체 그림 보기

```bash
$ cat /proc/<pid>/status
Name:   java
VmPeak:  8388608 kB    # 가상 메모리 최대치
VmSize:  8200000 kB    # 현재 가상 메모리 크기
VmLck:         0 kB    # mlock으로 잠긴 메모리
VmHWM:   4096000 kB    # RSS 최대치 (High Water Mark)
VmRSS:   3800000 kB    # 현재 RSS
VmData:  7000000 kB    # 데이터 + 힙 + mmap anonymous
VmStk:       136 kB    # 메인 스레드 스택
VmExe:        12 kB    # 코드 영역
VmLib:     52000 kB    # 공유 라이브러리
VmSwap:    50000 kB    # 스왑 아웃된 크기
Threads:      200
```

- **VmHWM**: RSS의 최대값. 메모리 릭이 있으면 이 값이 계속 올라간다.
- **VmSwap**: 0이 아니면 메모리 압박 상태다. 값이 크면 성능이 심하게 떨어진다.
- **VmRSS vs VmSize**: VmSize는 가상 메모리라 실제 물리 메모리와 무관한 경우가 많다. RSS를 봐야 한다.
