# 프로세스/스레드 관점의 가상 메모리

가상 메모리의 일반적인 동작 원리(페이징, TLB, 페이지 폴트 등)는 [가상 메모리](../Memory/가상 메모리.md)에서 다룬다. 이 문서는 **프로세스와 스레드가 가상 메모리를 어떻게 사용하는지**에 집중한다.

---

## 프로세스의 가상 주소 공간 레이아웃

프로세스가 생성되면 커널이 독립적인 가상 주소 공간을 할당한다. x86-64 리눅스 기준으로 유저 공간은 하위 128TB, 커널 공간은 상위 128TB를 사용한다.

```
높은 주소 (0x7FFF_FFFF_FFFF)
┌────────────────────────────────────────┐
│             커널 공간                    │ ← 모든 프로세스가 동일한 매핑 공유
│         (유저 모드 접근 불가)             │
├────────────────────────────────────────┤ 0x7FFF_FFFF_F000
│             스택 (Stack)                │ ← 아래쪽으로 성장
│         지역 변수, 함수 호출 프레임       │
│                  ↓                      │
│           (빈 공간 - Guard Page)         │
│                  ↑                      │
│          mmap 영역 (Memory Mapping)     │ ← 공유 라이브러리, mmap 파일
│         libc.so, libpthread.so 등       │
│                  ↑                      │
│           (빈 공간)                      │
│                  ↑                      │
│              힙 (Heap)                  │ ← 위쪽으로 성장
│         malloc/new로 할당한 메모리       │
├────────────────────────────────────────┤
│              BSS                        │ ← 초기화되지 않은 전역/정적 변수
├────────────────────────────────────────┤
│           데이터 (Data)                 │ ← 초기화된 전역/정적 변수
├────────────────────────────────────────┤
│           텍스트 (Text/Code)            │ ← 실행 코드, 읽기 전용
└────────────────────────────────────────┘
낮은 주소 (0x0000_0000_0000 근처)
```

각 영역은 커널의 `vm_area_struct`(VMA)로 관리된다. VMA에는 시작/끝 주소, 접근 권한(읽기/쓰기/실행), 매핑된 파일 정보가 들어있다.

```bash
# 프로세스의 VMA 목록 확인
cat /proc/<PID>/maps

# 출력 예시:
# 00400000-00452000 r-xp /usr/bin/nginx    ← 텍스트 (실행 코드)
# 00651000-00652000 r--p /usr/bin/nginx    ← 읽기 전용 데이터
# 00652000-00653000 rw-p /usr/bin/nginx    ← 읽기/쓰기 데이터
# 01a3e000-01a5f000 rw-p [heap]            ← 힙
# 7f1234000-7f1256000 r-xp /lib/libc.so.6 ← 공유 라이브러리
# 7ffc89000-7ffc8a000 rw-p [stack]         ← 스택
```

### 스택과 힙이 충돌하면?

스택은 아래로, 힙은 위로 성장한다. 예전에는 두 영역이 겹치면서 데이터가 망가지는 사고가 있었다. 현재 리눅스 커널은 Guard Page를 둬서 스택이 다른 영역을 침범하면 SIGSEGV를 보낸다.

스택 크기 제한은 `ulimit -s`로 확인한다. 기본값은 보통 8MB다. 재귀가 깊어지거나 스택 변수가 큰 경우 이 제한에 걸린다.

```bash
# 스택 크기 제한 확인
ulimit -s
# 8192 (KB, 즉 8MB)

# 특정 프로세스에서 변경
ulimit -s 16384
```

## fork()와 메모리: Copy-on-Write 동작 흐름

`fork()`로 자식 프로세스를 만들 때, 부모의 전체 메모리를 복사하면 낭비가 심하다. 리눅스는 Copy-on-Write(COW)로 실제 수정이 일어날 때까지 복사를 미룬다.

### COW 동작 단계

```
[1단계: fork() 호출]

  부모 프로세스                       자식 프로세스
  ┌─────────────┐                   ┌─────────────┐
  │ 페이지 테이블 │                   │ 페이지 테이블 │ ← 부모 것을 복사
  │ PTE: Page A │───┐           ┌───│ PTE: Page A │
  │ PTE: Page B │──┐│           │┌──│ PTE: Page B │
  │ PTE: Page C │─┐││           ││┌─│ PTE: Page C │
  └─────────────┘ │││           │││ └─────────────┘
                  │││  물리 메모리│││
                  ││└→┌─────────┐│││
                  ││  │ Frame X │←┘││  ← 모든 페이지를 읽기 전용으로 표시
                  │└→ │ Frame Y │←─┘│
                  └→  │ Frame Z │←──┘
                      └─────────┘

  * 페이지 테이블만 복사한다. 물리 프레임은 공유.
  * 공유 중인 페이지의 PTE에 쓰기 권한을 제거한다.
  * 각 물리 프레임의 참조 카운트를 2로 올린다.


[2단계: 자식이 Page B에 쓰기 시도]

  자식 프로세스: Page B에 write → 쓰기 권한 없음 → 페이지 폴트 발생
                                                        │
                                                        v
                                            ┌──────────────────────┐
                                            │ 커널 COW 폴트 핸들러  │
                                            │                      │
                                            │ 1. 새 물리 프레임 할당 │
                                            │ 2. Frame Y 내용 복사  │
                                            │ 3. 자식 PTE 갱신      │
                                            │    → 새 프레임 + 쓰기 │
                                            │ 4. Frame Y 참조 카운트│
                                            │    2 → 1로 감소      │
                                            │ 5. 참조 카운트 1이면  │
                                            │    부모 PTE 쓰기 복원 │
                                            └──────────────────────┘

[3단계: COW 완료 후]

  부모 프로세스                       자식 프로세스
  ┌─────────────┐                   ┌─────────────┐
  │ PTE: Page A │────→ Frame X ←────│ PTE: Page A │  (여전히 공유)
  │ PTE: Page B │────→ Frame Y      │ PTE: Page B │────→ Frame W (새 프레임)
  │ PTE: Page C │────→ Frame Z ←────│ PTE: Page C │  (여전히 공유)
  └─────────────┘                   └─────────────┘

  * Page B만 복사됐다. A와 C는 양쪽 다 읽기만 하므로 계속 공유.
  * Frame Y의 참조 카운트가 1이 됐으므로 부모의 PTE에 쓰기 권한이 복원된다.
```

### COW가 문제되는 실무 사례

**Redis의 BGSAVE/BGREWRITEAOF**

Redis는 RDB 스냅샷이나 AOF 리라이트 시 `fork()`를 호출한다. 자식 프로세스가 데이터를 읽어서 디스크에 쓰는 동안, 부모 프로세스에 쓰기 요청이 들어오면 COW가 발생한다.

쓰기가 활발한 Redis 인스턴스에서 `fork()` 직후 메모리 사용량이 거의 2배로 뛰는 경우가 있다. 부모가 수정하는 모든 페이지마다 복사본이 생기기 때문이다.

```bash
# Redis fork 시 COW로 인한 추가 메모리 확인
# INFO 명령에서 확인 가능
redis-cli INFO memory | grep cow
# latest_fork_usec:1234       ← fork 소요 시간 (마이크로초)

# /proc에서 직접 확인
cat /proc/<redis-child-pid>/smaps_rollup | grep Private
# Private_Dirty: 이 값이 COW로 복사된 메모리 양
```

Transparent Huge Pages가 켜져 있으면 COW가 2MB 단위로 발생해서 메모리 증가가 훨씬 심해진다. Redis는 THP를 끄라고 권장한다.

**대용량 프로세스의 fork()**

수 GB 메모리를 쓰는 프로세스에서 `fork()`를 하면, 페이지 테이블 복사와 PTE 권한 변경에 시간이 걸린다. `fork()` 자체는 메모리를 복사하지 않지만, 페이지 테이블 크기가 크면 수십~수백 밀리초가 걸리는 경우가 있다.

외부 명령 실행이 목적이라면 `posix_spawn()`이나 `vfork()` + `exec()`을 쓰는 게 낫다. `vfork()`는 부모의 주소 공간을 그대로 빌려쓰고, `exec()` 호출 시 새 주소 공간으로 교체된다. 페이지 테이블 복사가 없다.

## 스레드의 메모리 공유 구조

같은 프로세스의 스레드들은 가상 주소 공간을 공유한다. 이게 프로세스와 스레드의 근본적인 차이다.

```
프로세스 A (스레드 3개)
┌──────────────────────────────────────────────────────────┐
│                                                          │
│   공유 영역 (모든 스레드가 같은 주소로 접근)               │
│  ┌──────────────────────────────────────────────────┐    │
│  │ 텍스트 (코드)    ← 같은 코드를 실행               │    │
│  │ 데이터/BSS       ← 전역 변수 공유                 │    │
│  │ 힙               ← malloc 결과를 모든 스레드가 접근│    │
│  │ mmap 영역        ← 파일 매핑, 공유 라이브러리     │    │
│  │ 페이지 테이블    ← 하나의 테이블을 공유            │    │
│  │ 파일 디스크립터  ← 열린 파일도 공유                │    │
│  └──────────────────────────────────────────────────┘    │
│                                                          │
│   스레드별 독립 영역                                      │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐     │
│  │   스레드 1    │ │   스레드 2    │ │   스레드 3    │     │
│  │              │ │              │ │              │     │
│  │ 스택 (8MB)   │ │ 스택 (8MB)   │ │ 스택 (8MB)   │     │
│  │ TLS          │ │ TLS          │ │ TLS          │     │
│  │ 레지스터 상태 │ │ 레지스터 상태 │ │ 레지스터 상태 │     │
│  │ 시그널 마스크 │ │ 시그널 마스크 │ │ 시그널 마스크 │     │
│  └──────────────┘ └──────────────┘ └──────────────┘     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 페이지 테이블을 공유한다는 것의 의미

리눅스에서 스레드는 `clone()` 시스템 콜에 `CLONE_VM` 플래그를 넘겨서 생성한다. 이 플래그가 있으면 자식이 부모의 `mm_struct`(메모리 디스크립터)를 그대로 쓴다. 페이지 테이블이 같으므로 한 스레드가 `malloc()`으로 할당한 메모리 주소를 다른 스레드에 그냥 넘기면 된다.

반면 `fork()`는 `CLONE_VM` 없이 호출되므로 `mm_struct`를 복제한다. 부모와 자식이 같은 주소를 가리켜도 물리 메모리가 다를 수 있다(COW 이후).

```c
// 스레드 생성 - CLONE_VM으로 주소 공간 공유
clone(child_fn, stack_top,
      CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD,
      arg);

// 프로세스 생성 - CLONE_VM 없음, 주소 공간 복제
fork();  // 내부적으로 clone(SIGCHLD)
```

### 스레드 스택의 위치

메인 스레드의 스택은 주소 공간 최상단에 위치하고, 커널이 자동으로 확장한다. 추가 스레드의 스택은 `mmap`으로 할당한 별도 메모리 영역이다.

```
주소 공간 상단
┌──────────────────────┐
│ 메인 스레드 스택       │ ← 커널이 자동 확장, ulimit -s 제한
│ (기본 8MB)            │
├──────────────────────┤
│ Guard Page (접근 불가) │ ← 스택 오버플로 감지용
├──────────────────────┤
│ 스레드 3 스택          │ ← pthread_create 시 mmap으로 할당
│ (기본 8MB)            │    크기 고정, 자동 확장 안 됨
├──────────────────────┤
│ Guard Page            │
├──────────────────────┤
│ 스레드 2 스택          │
│ (기본 8MB)            │
├──────────────────────┤
│ Guard Page            │
├──────────────────────┤
│ ... (mmap 영역)       │
└──────────────────────┘
```

메인 스레드 스택은 자동 확장되지만, `pthread_create`로 만든 스레드 스택은 고정 크기다. 재귀가 깊거나 스택 변수가 큰 스레드는 `pthread_attr_setstacksize`로 크기를 늘려야 한다.

```c
pthread_attr_t attr;
pthread_attr_init(&attr);
pthread_attr_setstacksize(&attr, 16 * 1024 * 1024);  // 16MB

pthread_t thread;
pthread_create(&thread, &attr, worker_fn, NULL);
pthread_attr_destroy(&attr);
```

### Thread-Local Storage (TLS)

전역 변수처럼 선언하지만 스레드마다 독립적인 복사본을 갖는 저장소다. `errno`가 대표적인 TLS 변수다. 멀티스레드 환경에서 `errno`가 깨지지 않는 이유가 이것이다.

```c
// C11
_Thread_local int counter = 0;

// GCC/Clang 확장
__thread int counter = 0;

// POSIX API
pthread_key_t key;
pthread_key_create(&key, NULL);
pthread_setspecific(key, value);
void *val = pthread_getspecific(key);
```

TLS는 각 스레드의 스택 근처에 별도 영역으로 매핑된다. `__thread` 변수가 많으면 스레드 생성 비용이 올라간다.

## 프로세스 간 메모리 공유

프로세스끼리는 주소 공간이 분리되어 있어서 직접 메모리를 공유할 수 없다. 공유하려면 커널의 도움이 필요하다.

### 공유 라이브러리 (Shared Library)

여러 프로세스가 같은 `.so` 파일을 로드하면, 코드 영역(텍스트 세그먼트)은 물리 메모리에 한 번만 올라간다. 각 프로세스의 페이지 테이블이 같은 물리 프레임을 가리킨다.

```
프로세스 A                    프로세스 B
┌──────────────┐            ┌──────────────┐
│ VMA:         │            │ VMA:         │
│ 0x7f10...    │            │ 0x7f20...    │ ← 가상 주소는 다를 수 있다
│ r-xp libc.so │            │ r-xp libc.so │
└──────┬───────┘            └──────┬───────┘
       │                           │
       └──────────┬────────────────┘
                  v
          ┌──────────────┐
          │ 물리 프레임    │ ← libc.so 코드가 한 번만 로드됨
          │ (읽기 전용)    │
          └──────────────┘
```

가상 주소가 프로세스마다 다를 수 있으므로, 공유 라이브러리는 Position-Independent Code(PIC)로 컴파일된다. 절대 주소 대신 상대 주소를 사용해서 어느 가상 주소에 로드해도 동작한다.

데이터 세그먼트(전역 변수 등)는 각 프로세스가 COW로 독립 복사본을 갖는다.

### mmap을 이용한 공유 메모리

`MAP_SHARED`로 같은 파일을 매핑하면 프로세스 간 메모리 공유가 가능하다.

```
프로세스 A                           프로세스 B
┌─────────────────┐               ┌─────────────────┐
│ mmap(            │               │ mmap(            │
│   fd,            │               │   fd,            │
│   MAP_SHARED)    │               │   MAP_SHARED)    │
│                  │               │                  │
│ 가상 주소:       │               │ 가상 주소:       │
│ 0x7f4a_1000_0000│               │ 0x7f3b_2000_0000│
└────────┬────────┘               └────────┬────────┘
         │                                  │
         │     ┌─────────────────┐         │
         └────→│ 페이지 캐시      │←────────┘
               │ (물리 메모리)    │
               └────────┬────────┘
                        │
                        v
               ┌─────────────────┐
               │ 디스크 파일      │
               └─────────────────┘

 * 프로세스 A가 쓴 내용을 프로세스 B가 바로 읽을 수 있다.
 * 동기화는 별도로 처리해야 한다 (세마포어, 뮤텍스 등).
```

파일 없이 익명 공유 메모리를 쓸 수도 있다.

```c
// POSIX 공유 메모리
int fd = shm_open("/my_shm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, 4096);
void *ptr = mmap(NULL, 4096, PROT_READ | PROT_WRITE,
                 MAP_SHARED, fd, 0);
```

## 페이지 매핑 과정: 프로세스 생성부터 메모리 접근까지

프로세스가 `exec()`으로 새 프로그램을 실행할 때 가상 주소 공간이 어떻게 설정되는지 전체 흐름이다.

```
exec("./my_program") 호출
         │
         v
┌────────────────────────────────────────────────┐
│ 1. 기존 주소 공간 해제                           │
│    - 이전 프로그램의 VMA, 페이지 테이블 제거      │
│    - 공유 라이브러리 매핑 해제                    │
└──────────────────────┬─────────────────────────┘
                       v
┌────────────────────────────────────────────────┐
│ 2. ELF 바이너리 파싱                             │
│    - 프로그램 헤더에서 세그먼트 정보 읽기         │
│    - .text, .data, .bss 영역의 가상 주소와       │
│      크기, 권한 확인                              │
└──────────────────────┬─────────────────────────┘
                       v
┌────────────────────────────────────────────────┐
│ 3. VMA 생성 (아직 물리 메모리 할당 없음)         │
│                                                  │
│    VMA: 0x400000-0x452000  r-xp  (텍스트)       │
│    VMA: 0x651000-0x652000  rw-p  (데이터)       │
│    VMA: 0x652000-0x653000  rw-p  (BSS)          │
│    VMA: [heap 시작 주소]                          │
│    VMA: [stack 시작 주소]                         │
│                                                  │
│    * 페이지 테이블 엔트리는 모두 Invalid 상태     │
└──────────────────────┬─────────────────────────┘
                       v
┌────────────────────────────────────────────────┐
│ 4. 동적 링커(ld-linux.so) 로드                   │
│    - 공유 라이브러리 mmap (MAP_PRIVATE)          │
│    - 역시 Demand Paging으로 실제 로드는 나중에   │
└──────────────────────┬─────────────────────────┘
                       v
┌────────────────────────────────────────────────┐
│ 5. 프로그램 시작 → 첫 명령어 접근                │
│                                                  │
│    CPU: 0x400000 접근 → PTE Invalid              │
│         → Minor Page Fault 발생                  │
│         → 커널이 ELF 파일에서 해당 페이지 읽기    │
│         → 물리 프레임 할당, PTE 갱신              │
│         → 명령어 재실행                           │
│                                                  │
│    이후 접근하는 페이지마다 같은 과정 반복         │
└────────────────────────────────────────────────┘
```

Demand Paging 덕분에 프로그램이 빠르게 시작된다. 바이너리 전체를 메모리에 올리지 않고, 실행하면서 필요한 페이지만 로드한다. 대형 바이너리(수백 MB)도 시작 시간이 짧은 이유다.

## Active/Inactive 리스트 기반 페이지 교체

리눅스 커널은 물리 메모리가 부족할 때 어떤 페이지를 회수할지 결정해야 한다. 순수 LRU는 매 접근마다 리스트를 갱신해야 해서 비용이 크다. 리눅스는 Active/Inactive 2개 리스트로 LRU를 근사한다.

### 리스트 구조와 페이지 흐름

```
                    페이지 접근 패턴에 따른 흐름
                    ──────────────────────────

  [새로 할당된 페이지]
         │
         v
  ┌──────────────────────────────────────────────┐
  │              Inactive 리스트                   │
  │                                                │
  │  Head ← 최근 추가                              │
  │  ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐     │
  │  │Page G │→│Page F │→│Page E │→│Page D │     │
  │  │ref=0  │ │ref=1  │ │ref=0  │ │ref=0  │     │
  │  └───────┘ └───────┘ └───────┘ └───────┘     │
  │                                    Tail →      │
  │                                    회수 대상    │
  └──────────┬─────────────────────────┬──────────┘
             │                         │
        재접근 시                   메모리 부족 시
        (ref=1인 페이지)            (ref=0인 페이지)
             │                         │
             v                         v
  ┌──────────────────────┐    ┌────────────────┐
  │   Active 리스트       │    │  페이지 회수    │
  │                      │    │                │
  │  ┌───────┐ ┌───────┐│    │ - 파일 캐시:   │
  │  │Page A │→│Page B ││    │   그냥 버림    │
  │  │ref=1  │ │ref=1  ││    │ - Dirty 페이지:│
  │  └───────┘ └───────┘│    │   디스크에 쓴  │
  │                      │    │   후 버림      │
  │  메모리 압박 시       │    │ - Anonymous:   │
  │  ref=0인 페이지를    │    │   스왑으로 이동│
  │  Inactive로 강등     │    └────────────────┘
  └──────────────────────┘

  동작 규칙:
  ─────────
  1. 새 페이지는 Inactive 리스트 Head에 추가
  2. Inactive 상태에서 재접근되면 ref 비트를 1로 설정
  3. kswapd가 Inactive 리스트를 스캔할 때:
     - ref=1 → Active 리스트로 승격, ref=0으로 리셋
     - ref=0 → 회수 대상
  4. Active 리스트가 너무 커지면:
     - Tail 쪽 페이지의 ref 확인
     - ref=1 → ref=0으로 리셋, 다음 기회
     - ref=0 → Inactive 리스트로 강등
```

### 파일 캐시 페이지 vs 익명 페이지

리눅스는 Active/Inactive 리스트를 파일 캐시용과 익명 페이지용으로 별도 관리한다. 총 4개 리스트다.

```
┌─────────────────────────────────────────────┐
│ 파일 캐시 (File-backed Pages)                │
│                                              │
│  Active File    ← 자주 읽는 파일의 페이지    │
│  Inactive File  ← 한동안 안 읽은 파일 페이지 │
│                                              │
│  회수 시: 디스크에 원본이 있으므로 바로 버림  │
│  (Dirty면 먼저 Write-back)                   │
├─────────────────────────────────────────────┤
│ 익명 페이지 (Anonymous Pages)                │
│                                              │
│  Active Anon    ← 자주 접근하는 힙/스택     │
│  Inactive Anon  ← 한동안 안 쓰는 힙/스택    │
│                                              │
│  회수 시: 디스크에 원본 없음 → 스왑에 써야 함│
│  스왑 I/O가 필요하므로 비용이 더 크다        │
└─────────────────────────────────────────────┘
```

`vm.swappiness`가 이 두 종류 사이의 회수 비율을 조절한다. 값이 낮으면 파일 캐시를 먼저 회수하고, 높으면 익명 페이지도 적극적으로 스왑한다.

```bash
# 각 리스트의 크기 확인
grep -E "Active|Inactive" /proc/meminfo
# Active:          3145728 kB
# Inactive:        1572864 kB
# Active(anon):    2097152 kB
# Inactive(anon):   524288 kB
# Active(file):    1048576 kB
# Inactive(file):  1048576 kB
```

## exec(), fork(), pthread_create() 시 메모리 처리 비교

| 항목 | exec() | fork() | pthread_create() |
|------|--------|--------|-----------------|
| 주소 공간 | 기존 것 폐기, 새로 생성 | 부모 것 복제 (COW) | 부모와 동일한 것 공유 |
| 페이지 테이블 | 새로 생성 | 복제 후 읽기 전용 표시 | 같은 테이블 사용 |
| 힙 | 새로 시작 | 부모와 공유 (COW) | 같은 힙 공유 |
| 스택 | 새 스택 할당 | 부모 스택 복제 (COW) | mmap으로 별도 할당 |
| 코드 | 새 바이너리 매핑 | 부모 코드 공유 (읽기 전용) | 같은 코드 공유 |
| 파일 디스크립터 | 상속 (close-on-exec 제외) | 복제 | 공유 |
| 메모리 비용 | 바이너리 크기에 비례 | 페이지 테이블 크기에 비례 | 스택 크기 (기본 8MB) |
| TLB | 전체 플러시 | 전체 플러시 | 플러시 없음 |

스레드가 프로세스보다 생성 비용이 낮은 핵심 이유는 페이지 테이블 복제와 TLB 플러시가 없기 때문이다. 반면 메모리 격리가 없어서 한 스레드의 버그가 전체 프로세스에 영향을 준다.

## 실무 트러블슈팅

### 프로세스 메모리 레이아웃 분석

```bash
# 프로세스의 전체 메모리 맵 확인
pmap -x <PID>

# 출력 예시:
# Address           Kbytes     RSS   Dirty Mode  Mapping
# 0000000000400000     328     328       0 r-x-- nginx
# 00007f1234000000   65536   32768   32768 rw---   [ anon ]  ← 힙 또는 mmap
# 00007f1234100000    1808    1808       0 r-x-- libc-2.31.so
# ...
# total            524288  131072   65536

# RSS가 큰 anon 영역이 있으면 힙 메모리 사용량이 많다는 뜻
```

### fork() 후 COW 페이지 수 확인

```bash
# 부모/자식 프로세스의 공유/비공유 메모리 비교
cat /proc/<PID>/smaps_rollup

# Shared_Clean:  공유 중이고 수정 안 된 페이지 (COW 전)
# Shared_Dirty:  공유 중이고 수정된 페이지
# Private_Clean: 이 프로세스만 쓰는 깨끗한 페이지
# Private_Dirty: 이 프로세스만 쓰는 수정된 페이지 (COW 후)
```

### 스레드별 스택 사용량 확인

```bash
# /proc/<PID>/task/<TID>/maps에서 스레드별 스택 확인
ls /proc/<PID>/task/

# 각 스레드의 스택 영역
cat /proc/<PID>/task/<TID>/maps | grep stack

# 스택 사용량이 제한에 가까운지 확인 (스택 오버플로 위험)
cat /proc/<PID>/task/<TID>/status | grep -E "VmStk|StackUsage"
```
