## 스레드가 공유하는 것과 독립적으로 갖는 것

스레드의 개념은 [스레드의 개념](스레드의 개념.md)에서 다룬다. 이 문서는 스레드가 실제로 어떤 자원을 공유하고, 독립 자원은 메모리에 어떻게 배치되며, 디버깅할 때 어떻게 확인하는지에 집중한다.

---

## 공유 자원 vs 독립 자원 한눈에 보기

스레드를 다룰 때 가장 자주 헷갈리는 게 "이건 같이 쓰는 거였나, 따로 갖는 거였나"다. 문제 분석에 들어가기 전에 한 번 정리해 두면 디버깅 시간이 줄어든다.

| 자원 | 공유/독립 | 커널 구조체 | 메모 |
|---|---|---|---|
| 코드 세그먼트(.text) | 공유 | `mm_struct.exe_file` 매핑 | 같은 함수 주소를 모든 스레드가 호출 |
| 전역/정적 변수(.data, .bss) | 공유 | `mm_struct` | race condition의 주범 |
| 힙(malloc) | 공유 | `mm_struct.brk`, mmap arena | glibc arena는 스레드별로 잡힐 수 있음 |
| 파일 디스크립터 테이블 | 공유 | `files_struct` | 한쪽이 close하면 다른 쪽도 영향 |
| 시그널 핸들러 | 공유 | `sighand_struct` | 프로세스 단위로 등록됨 |
| 현재 작업 디렉토리/umask | 공유 | `fs_struct` | `chdir`는 전 스레드에 영향 |
| 스택 | **독립** | mmap된 별도 영역 | 가드 페이지 포함, 기본 8MB |
| 레지스터 셋트(PC/SP/BP/범용/FPU) | **독립** | `task_struct.thread`, `pt_regs` | 컨텍스트 스위치의 핵심 대상 |
| TLS 블록(`__thread` 변수) | **독립** | fs/gs 베이스가 가리킴 | 스레드 생성 시 `.tdata` 복사 |
| 커널 스택 | **독립** | `task_struct.stack` (보통 16KB) | 시스템 콜 처리용 |
| 시그널 마스크/sigaltstack | **독립** | `task_struct.blocked` | `pthread_sigmask`로 스레드별 설정 |
| PID(Tgid) | 공유 | `task_struct.tgid` | `getpid()`는 동일 |
| TID | **독립** | `task_struct.pid` | `gettid()`는 다름 |
| nice / 스케줄링 정책 | **독립** | `task_struct` | 스레드별로 우선순위 다르게 가능 |

표의 "독립" 항목이 컨텍스트 스위치 비용을 만드는 주체다. 스레드를 바꿔 끼울 때 커널이 저장/복원해야 하는 것이 곧 이 목록이다.

## clone() 플래그별 공유 범위

Linux에서 스레드와 프로세스의 차이는 `clone()` 호출 시 넘기는 플래그 조합으로 결정된다. 각 플래그가 정확히 뭘 공유하는지 알아야 문제가 생겼을 때 원인을 좁힐 수 있다.

| 플래그 | 공유하는 커널 구조체 | 공유 대상 | 빠지면 어떻게 되나 |
|---|---|---|---|
| `CLONE_VM` | `mm_struct` | 가상 메모리 공간 전체 (코드, 데이터, 힙, mmap 영역) | 별도 주소 공간 복사 (fork 동작) |
| `CLONE_FILES` | `files_struct` | 파일 디스크립터 테이블 | fd 테이블 복사. 한쪽에서 close해도 다른 쪽 영향 없음 |
| `CLONE_FS` | `fs_struct` | 루트 디렉토리, cwd, umask | 각자 독립적인 cwd를 가짐 |
| `CLONE_SIGHAND` | `sighand_struct` | 시그널 핸들러 테이블 | 시그널 핸들러 복사. 한쪽에서 변경해도 반영 안 됨 |
| `CLONE_THREAD` | thread group | PID(Tgid), exit signal 처리 | 별도 프로세스로 취급 (다른 Tgid) |
| `CLONE_IO` | `io_context` | I/O 스케줄링 컨텍스트 | 독립적인 I/O 우선순위 |

`pthread_create`는 내부에서 `CLONE_VM | CLONE_FILES | CLONE_FS | CLONE_SIGHAND | CLONE_THREAD | CLONE_SYSVSEM | CLONE_SETTLS` 등을 한꺼번에 넘긴다. 이게 "스레드"의 정의다.

```bash
# 실제로 어떤 플래그가 넘어가는지 확인
$ strace -f -e clone3 ./my_threaded_app 2>&1 | grep clone3
clone3({flags=CLONE_VM|CLONE_FS|CLONE_FILES|CLONE_SIGHAND|CLONE_THREAD|
        CLONE_SYSVSEM|CLONE_SETTLS|CLONE_PARENT_SETTID|CLONE_CHILD_CLEARTID,
        ...}, 88) = 12345
```

### CLONE_VM만 빠지면?

`CLONE_VM` 없이 `CLONE_FILES`를 주면 메모리는 복사하되 fd 테이블은 공유하는 형태가 된다. 실제로 이런 조합을 쓸 일은 거의 없지만, 컨테이너 런타임이나 샌드박스에서 특수한 격리 구성을 만들 때 간혹 나온다.

```c
// 주소 공간은 분리하되 fd 테이블은 공유하는 특수한 clone
// 일반 애플리케이션에서는 쓸 일이 없다
clone(child_fn, stack_top,
      CLONE_FILES | CLONE_FS | SIGCHLD,
      arg);
```

## 스레드별 독립 레지스터 셋트

스레드를 "실행의 단위"라고 부를 때 그 실행 상태를 구성하는 게 레지스터다. 메모리는 공유해도 "지금 어디까지 실행했나"를 표현하는 레지스터는 절대 공유할 수 없다. 스레드 N개가 있다면 레지스터 셋트도 N개가 필요하다.

### 어떤 레지스터들이 스레드별로 따로 잡히나 (x86-64 기준)

| 레지스터 | 역할 | 컨텍스트로서의 의미 |
|---|---|---|
| RIP (PC) | 다음 실행할 명령어 주소 | "지금 코드 어디까지 진행됐나" |
| RSP (SP) | 현재 스택 포인터 | "내 스택 어디까지 썼나" |
| RBP (BP) | 프레임 포인터 | 호출 체인 추적의 기준점 |
| RAX, RBX, RCX, RDX, RSI, RDI | 범용 레지스터 | 함수 인자/리턴값/임시 저장 |
| R8 ~ R15 | 추가 범용 레지스터 | 인자 5~6번, 임시 저장 |
| RFLAGS | 비교/연산 결과 플래그 | 분기 판단에 필수 |
| XMM0 ~ XMM15 | SSE 128비트 | float 연산, memcpy 가속 |
| YMM0 ~ YMM15 | AVX 256비트 | 벡터 연산 |
| ZMM0 ~ ZMM31 | AVX-512 512비트 | 행렬 연산, 일부 string 함수 |
| FS_BASE | TLS 베이스 주소 | `__thread` 변수 접근의 시작점 |
| GS_BASE | 사용자 모드에서 보조 TLS | 일부 런타임이 사용 |

이 모두가 스레드별로 별도 값이다. 컨텍스트 스위치가 무겁다는 말의 실체가 여기 있다. 범용 레지스터만 저장한다고 끝나는 게 아니라 AVX-512까지 쓰는 워크로드는 ZMM 32개(각 64바이트) = 2KB만 저장/복원에 더 필요하다. lazy FPU 저장 같은 최적화가 들어가는 이유다.

### task_struct / thread_struct / pt_regs

커널 안에서 스레드의 레지스터는 두 군데에 산다. 하나는 시스템 콜이나 인터럽트로 진입했을 때 커널 스택 맨 아래에 쌓이는 `pt_regs`, 다른 하나는 컨텍스트 스위치 시 보존되는 `task_struct.thread`(아키텍처별 `thread_struct`)다.

```c
// arch/x86/include/asm/processor.h (대략적인 구조)
struct thread_struct {
    unsigned long       sp;       // 커널 모드 SP (스위치 시점의)
    unsigned short      es, ds, fsindex, gsindex;
    unsigned long       fsbase, gsbase;   // TLS 베이스
    struct fpu          fpu;      // XMM/YMM/ZMM 포함된 FPU 상태
    unsigned long       cr2, trap_nr, error_code;
    // ...
};

// arch/x86/include/asm/ptrace.h
struct pt_regs {
    unsigned long r15, r14, r13, r12, bp, bx;
    unsigned long r11, r10, r9, r8, ax, cx, dx, si, di;
    unsigned long orig_ax;
    unsigned long ip;      // RIP
    unsigned long cs;
    unsigned long flags;   // RFLAGS
    unsigned long sp;      // RSP
    unsigned long ss;
};
```

유저 모드에서 실행 중이던 RIP, RSP, RFLAGS, 범용 레지스터들은 시스템 콜 진입 시 커널 스택 위에 `pt_regs` 형태로 저장된다. 이게 `/proc/<pid>/task/<tid>/syscall`이나 `stat`에서 노출되는 값의 원천이다.

### __switch_to가 실제로 하는 일

스케줄러가 다음 스레드를 고른 뒤 `__switch_to()`(arch/x86/kernel/process_64.c)에서 레지스터 셋트가 교체된다. 핵심만 추리면 이렇다.

```c
// arch/x86/kernel/process_64.c, 단순화한 의사 코드
__visible __notrace_funcgraph struct task_struct *
__switch_to(struct task_struct *prev_p, struct task_struct *next_p)
{
    struct thread_struct *prev = &prev_p->thread;
    struct thread_struct *next = &next_p->thread;

    // 1) FPU/SSE/AVX 상태: 이전 스레드 저장 + 다음 스레드 복원
    //    XSAVE/XRSTOR 명령어로 한 번에 처리
    switch_fpu_prepare(prev_p, cpu);
    // ...
    switch_fpu_finish(next_p);

    // 2) TLS 베이스: fs/gs 세그먼트 베이스 교체
    x86_fsgsbase_load(prev, next);

    // 3) 커널 스택 포인터 갱신 (TSS의 sp0)
    //    다음 인터럽트/시스템 콜이 어느 커널 스택을 쓸지 결정
    update_task_stack(next_p);

    // 4) RSP/RIP 교체는 호출 직전의 어셈블리 헬퍼 __switch_to_asm에서
    //    inline 처리됨. 함수 리턴 시점에 next의 스택으로 이미 점프
    // ...
    return prev_p;
}
```

범용 레지스터(RAX, RBX 등)는 함수 호출 규약에 따라 호출자가 알아서 스택에 보존하므로 `__switch_to`가 명시적으로 다루지 않는다. 진짜 비용이 큰 건 XSAVE로 통째로 저장하는 FPU/SIMD 영역과 fs/gs 베이스 교체다.

## /proc과 gdb로 스레드 레지스터 상태 들여다보기

데드락이나 무한 루프가 의심될 때, 프로세스를 중단시키지 않고 "지금 이 스레드가 정확히 어느 명령어에서 멈춰 있나"를 확인하는 절차다.

### /proc/[pid]/task/[tid]/syscall

스레드가 시스템 콜에 들어가 있는 동안의 상태를 보여준다.

```bash
# tid=12347이 어떤 시스템 콜에 묶여 있는지
$ cat /proc/12345/task/12347/syscall
202 0x7f1c8c7fe9c0 0x80 0x0 0x0 0x0 0x0 0x7f1c8c7fea30 0x7f1c80123456
# │   │             │     │   │   │   │   │              │
# │   인자 6개                                              SP            PC
# syscall 번호 (202 = futex)
```

가장 앞이 시스템 콜 번호다. 202면 `futex` — 락 대기 중이라는 뜻이다. 마지막 두 값이 사용자 모드 SP(RSP)와 PC(RIP)다. 시스템 콜에 들어가지 않은 실행 중인 스레드는 `running`만 나온다.

```bash
$ cat /proc/12345/task/12346/syscall
running
```

### /proc/[pid]/task/[tid]/stat의 kstkeip / kstkesp

`stat`의 28번째 필드가 kstkesp(스택 포인터), 29번째 필드가 kstkeip(인스트럭션 포인터)다. ([procfs man page](https://man7.org/linux/man-pages/man5/proc.5.html) 기준)

```bash
$ awk '{printf "RSP=0x%x RIP=0x%x\n", $28, $29}' /proc/12345/task/12347/stat
RSP=0x7f1c8c7fe9c0 RIP=0x7f1c80123456
```

이 RIP를 `/proc/<pid>/maps`와 대조하면 어떤 .so 안에 있는지 알 수 있다.

```bash
$ grep "7f1c80" /proc/12345/maps
7f1c80100000-7f1c80200000 r-xp 00000000 08:01 1234567  /lib/x86_64-linux-gnu/libc.so.6
```

libc 영역이다. 정확한 함수는 `addr2line` 또는 `objdump`로 추적한다.

```bash
$ addr2line -e /lib/x86_64-linux-gnu/libc.so.6 -f 0x123456
__lll_lock_wait
futex_wait_cancelable
```

futex_wait에서 멈춰 있다 — 락을 기다리는 중이라는 결론이 나온다.

### gdb로 멀티스레드 상태 확인

`gdb -p <pid>`로 attach하면 모든 스레드를 멈춘 채 들여다볼 수 있다.

```bash
$ gdb -p 12345

# 스레드 목록과 현재 위치
(gdb) info threads
  Id   Target Id                  Frame
* 1    Thread 0x... (LWP 12345)  __select () at ...
  2    Thread 0x... (LWP 12346)  futex_wait () at ...
  3    Thread 0x... (LWP 12347)  futex_wait () at ...

# 특정 스레드의 레지스터 전체 덤프
(gdb) thread 3
(gdb) info registers
rax            0x0
rbx            0x7f1c8c7fea30
rcx            0xffffffffffffffff
...
rip            0x7f1c80123456   <__lll_lock_wait+24>
rsp            0x7f1c8c7fe9c0
rbp            0x7f1c8c7fea00
fs_base        0x7f1c8c7ff700      # TLS 베이스
gs_base        0x0

# 전체 스레드의 백트레이스를 한 번에
(gdb) thread apply all bt

# AVX 레지스터까지 보고 싶으면
(gdb) info registers ymm
(gdb) info registers zmm
```

`fs_base`가 스레드마다 다른 게 보일 거다. TLS 변수 접근이 어떻게 스레드별로 분리되는지의 실체다.

운영 환경에서 코어 덤프만 있을 때도 동일하게 동작한다.

```bash
$ gdb /path/to/binary /var/crash/core.12345
(gdb) thread apply all bt full
```

## TLS(Thread Local Storage) 세그먼트

전역 변수처럼 보이지만 스레드마다 독립적인 복사본을 가지는 변수가 TLS다. `errno`가 대표적인 예시다.

### TLS 변수 선언

```c
// C11
_Thread_local int request_count = 0;

// GCC 확장 (C11 이전부터 사용 가능)
__thread int request_count = 0;

// C++11
thread_local int request_count = 0;
```

### TLS가 메모리에 배치되는 구조

ELF 바이너리에는 `.tdata`(초기화된 TLS)와 `.tbss`(초기화되지 않은 TLS) 섹션이 있다. 스레드가 생성되면 이 섹션의 내용을 복사해서 스레드별 TLS 블록을 만든다.

```bash
# 바이너리의 TLS 섹션 확인
$ readelf -S ./my_app | grep tls
  [22] .tdata            PROGBITS  0000000000604000  004000  000010  00  WAT  0   0   8
  [23] .tbss             NOBITS    0000000000604010  004010  000008  00  WAT  0   0   8

# TLS 세그먼트 크기
$ readelf -l ./my_app | grep TLS
  TLS            0x0000000000004000 0x0000000000604000 0x0000000000604000
                 0x0000000000000010 0x0000000000000018  R      0x8
```

### fs/gs 세그먼트 레지스터와 arch_prctl

x86-64 Linux에서 `fs` 세그먼트 레지스터의 베이스 주소가 현재 스레드의 TLS 블록을 가리킨다. 이 베이스는 일반 레지스터처럼 `mov`로 못 바꾸고 전용 시스템 콜 `arch_prctl(ARCH_SET_FS, addr)`로 설정한다. glibc의 `pthread_create` 내부에서 새 TCB(Thread Control Block) 영역을 잡은 뒤 이 주소를 fs 베이스로 박는다.

```c
// glibc nptl/pthread_create.c의 흐름을 단순화하면
void *tcb = allocate_stack_and_tls(...);
// clone3 호출 시 CLONE_SETTLS + tls 인자로 fs 베이스를 직접 설정
clone3(&(struct clone_args){
    .flags = CLONE_VM | CLONE_FS | CLONE_FILES |
             CLONE_SIGHAND | CLONE_THREAD | CLONE_SETTLS,
    .tls = (uintptr_t)tcb,
    // ...
});

// 수동으로 바꾸려면 (거의 안 씀)
#include <asm/prctl.h>
#include <sys/prctl.h>
syscall(SYS_arch_prctl, ARCH_SET_FS, my_tls_base);
```

새 커널(5.9+)과 새 CPU에서는 `wrfsbase`/`rdfsbase` 명령어로 시스템 콜 없이 fs 베이스를 직접 다룰 수 있다. 일부 사용자 공간 그린 스레드 라이브러리(예: 일부 코루틴 런타임)가 이를 활용해 fs를 빠르게 갈아끼운다.

TLS 변수 접근은 결국 `fs:offset` 한 줄로 컴파일된다.

```bash
$ objdump -d ./my_app | grep -A2 request_count
  4011a0:  64 8b 04 25 fc ff ff ff    mov    %fs:0xfffffffc,%eax
```

`%fs:offset` — 세그먼트 레지스터를 통한 간접 접근이다. mutex 같은 동기화 없이 스레드별 독립 저장소에 접근하는 구조다. fs 베이스가 스레드마다 다르기 때문에 같은 offset이 결과적으로 서로 다른 물리 주소로 풀린다.

`gs`는 x86-64 사용자 공간에서는 거의 비어 있고, 커널이 per-CPU 데이터를 가리키는 용도로 쓴다. 일부 런타임(Go 런타임의 일부 빌드 등)이 사용자 모드 gs를 활용하기도 한다.

### 동적 라이브러리의 TLS

공유 라이브러리(.so)에 TLS 변수가 있으면 `dlopen`으로 런타임에 로드할 때 TLS 블록을 확장해야 한다. glibc는 이를 위해 `__tls_get_addr` 함수를 사용하는데, 정적 TLS보다 접근 비용이 높다.

```c
// 공유 라이브러리 안에서 선언된 TLS
// 접근할 때마다 __tls_get_addr 호출이 필요할 수 있다
__thread int counter = 0;

// initial-exec 모델을 강제하면 fs:offset 접근으로 최적화되지만,
// dlopen으로 로드하면 크래시한다
__attribute__((tls_model("initial-exec")))
__thread int fast_counter = 0;
```

TLS 모델은 4가지가 있다: `local-exec`, `initial-exec`, `global-dynamic`, `local-dynamic`. 실행 파일 본체에서는 `local-exec`(가장 빠름), 공유 라이브러리에서는 `global-dynamic`(가장 느림)이 기본이다. `-ftls-model` 컴파일러 옵션으로 변경할 수 있지만, 잘못 설정하면 dlopen 시 segfault가 나니 주의해야 한다.

## 시그널 핸들러와 레지스터 스냅샷 (ucontext_t)

시그널이 스레드에 전달되면 커널이 현재 레지스터 상태를 사용자 스택(또는 sigaltstack)에 떠놓고 핸들러로 점프한다. 떠놓는 자료구조가 `ucontext_t`다. 즉 시그널 핸들러는 인터럽트당한 시점의 모든 레지스터를 통째로 받는다.

```c
#include <signal.h>
#include <ucontext.h>
#include <stdio.h>

static void crash_handler(int sig, siginfo_t *info, void *ucontext)
{
    ucontext_t *uc = (ucontext_t *)ucontext;
    mcontext_t *mc = &uc->uc_mcontext;

    // x86-64 Linux: gregs 배열로 접근
    fprintf(stderr, "SIGSEGV at RIP=0x%llx\n",
            (unsigned long long)mc->gregs[REG_RIP]);
    fprintf(stderr, "  RSP=0x%llx  RBP=0x%llx\n",
            (unsigned long long)mc->gregs[REG_RSP],
            (unsigned long long)mc->gregs[REG_RBP]);
    fprintf(stderr, "  fault addr=%p\n", info->si_addr);

    // 핸들러에서 그냥 리턴하면 커널이 ucontext를 복원해 원래 자리로 돌아간다.
    // mc->gregs[REG_RIP]을 수정하면 다른 명령어로 점프시킬 수도 있다.
    _exit(1);
}

int main(void)
{
    struct sigaction sa = {
        .sa_sigaction = crash_handler,
        .sa_flags = SA_SIGINFO,
    };
    sigaction(SIGSEGV, &sa, NULL);
    // ...
}
```

크래시 리포터(예: Google Breakpad, Sentry native)가 이 메커니즘으로 동작한다. SIGSEGV 시점의 RIP를 받아 심볼로 풀면 어디서 죽었는지 알 수 있다.

주의할 점이 두 가지 있다.

- 시그널 핸들러는 인터럽트당한 그 스레드에서 실행된다. 락을 잡고 있던 스레드가 시그널을 받아 핸들러로 들어가면 그 안에서 같은 락을 다시 잡으면 데드락이다. async-signal-safe 함수만 써야 한다는 제약이 여기서 나온다.
- `sigaltstack`을 설정해 두면 핸들러가 별도 스택에서 실행된다. 스택 오버플로우로 인한 SIGSEGV를 처리해야 한다면 필수다(원래 스택이 망가져 있으니까).

## /proc/[pid]/maps로 스레드 메모리 배치 확인하기

같은 프로세스의 스레드들은 하나의 `mm_struct`를 공유하므로 `/proc/<pid>/maps`가 모든 스레드에 동일하다. 하지만 maps 안에서 스레드별 스택 영역이 따로 잡혀 있는 걸 확인할 수 있다.

### 스레드 스택 영역 찾기

```bash
# 3개 스레드를 가진 프로세스의 메모리 맵
$ cat /proc/12345/maps | grep -i stack
7f1c8c000000-7f1c8c800000 rw-p 00000000 00:00 0
7f1c8d000000-7f1c8d800000 rw-p 00000000 00:00 0
7f1c8e000000-7f1c8e800000 rw-p 00000000 00:00 0
7ffc4b2e0000-7ffc4b301000 rw-p 00000000 00:00 0          [stack]
```

`[stack]`이라고 표시된 영역이 메인 스레드의 스택이다. 나머지 `rw-p` 영역 중 8MB(0x800000) 크기로 잡힌 것들이 pthread로 생성한 스레드의 스택이다.

### 스레드별 스택 영역 매핑하기

어떤 스택이 어떤 스레드의 것인지 확인하는 방법이다.

```bash
# 각 스레드의 스택 포인터 확인
$ for tid in $(ls /proc/12345/task/); do
    sp=$(awk '{print $29}' /proc/12345/task/$tid/stat)
    printf "tid=%s sp=0x%x\n" "$tid" "$sp"
done
tid=12345 sp=0x7ffc4b2ffa80    # 메인 스레드 — [stack] 범위 안에 있음
tid=12346 sp=0x7f1c8c7fe9c0    # 첫 번째 자식 스레드
tid=12347 sp=0x7f1c8d7fe9c0    # 두 번째 자식 스레드
```

`/proc/<pid>/task/<tid>/stat`의 29번째 필드가 스택 포인터(kstkesp)다. 이 값이 maps의 어떤 영역에 속하는지 비교하면 된다.

### 스택 가드 페이지

pthread로 만든 스레드 스택 앞에는 보호(guard) 페이지가 있다. 스택 오버플로우를 감지하기 위한 것이다.

```bash
# 스레드 스택 영역을 자세히 보면
$ cat /proc/12345/smaps | grep -A 5 "7f1c8c000000"
7f1c8c000000-7f1c8c001000 ---p 00000000 00:00 0   # 가드 페이지 (접근 불가)
Size:                  4 kB
...
7f1c8c001000-7f1c8c800000 rw-p 00000000 00:00 0   # 실제 스택 영역
Size:               8188 kB
```

`---p`로 표시된 영역이 가드 페이지다. 읽기/쓰기/실행 모두 불가하므로 스택 오버플로우가 발생하면 SIGSEGV로 잡힌다. 기본 가드 크기는 1페이지(4KB)이고, `pthread_attr_setguardsize`로 변경할 수 있다.

```c
pthread_attr_t attr;
pthread_attr_init(&attr);
pthread_attr_setguardsize(&attr, 64 * 1024);  // 가드 페이지를 64KB로 늘리기
```

## 실무에서 자원 구조 확인이 필요한 상황

### fd 누수가 어떤 스레드에서 발생하는지 찾기

스레드는 fd 테이블을 공유하므로, 어떤 스레드가 fd를 열었는지 직접 확인하기 어렵다. 하지만 스레드별로 strace를 걸면 추적할 수 있다.

```bash
# 특정 스레드만 추적
$ strace -p <tid> -e openat,close 2>&1 | head
[pid 12346] openat(AT_FDCWD, "/tmp/data_001.csv", O_RDONLY) = 47
[pid 12346] openat(AT_FDCWD, "/tmp/data_002.csv", O_RDONLY) = 48
# close 없이 계속 열고만 있다면 이 스레드가 범인

# 전체 스레드 대상으로 fd 관련 시스템 콜 추적
$ strace -f -p <pid> -e openat,close,dup 2>&1 | grep -v "= -1"
```

### 스레드 간 메모리 사용량 확인

스레드끼리 힙을 공유하므로 "스레드별 메모리 사용량"이라는 개념은 정확히는 없다. 확인할 수 있는 것은 스레드별 스택 사용량 정도다.

```bash
# 스레드별 실제 스택 사용량 (RSS 기준)
$ for tid in $(ls /proc/12345/task/); do
    rss=$(awk '/^VmStk/ {print $2}' /proc/12345/task/$tid/status 2>/dev/null)
    echo "tid=$tid stack_rss=${rss}kB"
done
```

pmap으로 좀 더 상세하게 볼 수 있다.

```bash
$ pmap -x 12345 | grep -E "stack|anon" | head
# anon 영역 중 8MB 근처 크기의 것들이 스레드 스택
```

### mmap 영역이 어떤 스레드에서 생겼는지 추적

Arena 기반 할당자(glibc malloc)는 스레드별로 arena를 나눠 쓴다. 스레드가 많을수록 arena 수가 늘어나고, 각 arena가 mmap으로 메모리를 잡으면서 RSS가 예상보다 커지는 경우가 있다.

```bash
# glibc malloc arena 정보 확인 (프로그램 내부에서)
$ gdb -p 12345 -batch -ex 'call malloc_stats()'
Arena 0:
system bytes     =     135168
in use bytes     =      98304
Arena 1:
system bytes     =     135168
in use bytes     =      12288
...
Total (incl. mmap):
system bytes     =    2097152
in use bytes     =     413696
```

arena 수를 제한하려면 `MALLOC_ARENA_MAX` 환경 변수를 설정한다.

```bash
# arena 수를 CPU 코어 수 이하로 제한
$ MALLOC_ARENA_MAX=4 ./my_app
```

### 컨텍스트 스위치 빈도와 비용 측정

레지스터 셋트가 통째로 교체되는 게 컨텍스트 스위치다. 빈도가 비정상적으로 높으면 락 경합 또는 잘못된 스레드 풀 사이즈가 원인인 경우가 많다.

```bash
# 프로세스의 자발적/비자발적 스위치 횟수
$ cat /proc/12345/status | grep ctxt
voluntary_ctxt_switches:        2103984
nonvoluntary_ctxt_switches:        47218

# 스레드별로 보고 싶다면
$ for tid in $(ls /proc/12345/task/); do
    grep ctxt /proc/12345/task/$tid/status |
        paste -sd' ' | sed "s/^/tid=$tid /"
done

# perf로 실시간 측정
$ perf stat -e context-switches,cpu-migrations -p 12345 sleep 10
```

`voluntary`가 많으면 락/IO 대기, `nonvoluntary`가 많으면 CPU 시간 소진이다. 후자가 많다면 워크로드가 CPU 코어 수보다 많은 active 스레드를 만들고 있다는 뜻이라 스레드 풀 크기 점검이 우선이다.
