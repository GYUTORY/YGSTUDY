# 스레드풀 (Thread Pool)

## 스레드풀이 필요한 이유

스레드를 매번 새로 만들면 OS 커널에 시스템 콜을 날려야 한다. 스택 메모리 할당, TCB 생성, 스케줄러 등록까지 합치면 하나당 약 0.1ms가 걸린다. 요청 하나에 0.1ms면 별거 아닌 것 같지만, 초당 수천 건이 들어오는 서버에서는 스레드 생성/소멸 비용만으로 CPU를 상당량 잡아먹는다.

더 큰 문제는 스레드 수 제한이 없다는 점이다. 요청이 폭증하면 스레드도 폭증하고, 스레드마다 기본 1MB 스택을 잡으니 1000개면 1GB다. 거기에 컨텍스트 스위칭 비용까지 올라가면서 오히려 처리량이 떨어진다.

스레드풀은 이 문제를 해결한다. 미리 스레드를 만들어두고, 작업이 오면 놀고 있는 스레드에 할당하고, 끝나면 다시 풀로 돌려보낸다.

```
작업 요청 → 작업 큐에 추가 → 유휴 스레드가 큐에서 꺼내 실행 → 완료 후 다시 대기
```

## Java ExecutorService 종류별 차이

Java에서 스레드풀은 `ExecutorService` 인터페이스로 추상화되어 있고, 실제 구현은 대부분 `ThreadPoolExecutor`다. `Executors` 팩토리 메서드로 만드는 것들도 내부적으로 `ThreadPoolExecutor` 파라미터만 다르게 넣은 것이다.

### ThreadPoolExecutor 핵심 파라미터

```java
new ThreadPoolExecutor(
    corePoolSize,      // 기본 유지 스레드 수
    maximumPoolSize,    // 최대 스레드 수
    keepAliveTime,      // 초과 스레드 유휴 대기 시간
    timeUnit,           // keepAliveTime 단위
    workQueue,          // 작업 대기 큐
    threadFactory,      // 스레드 생성 방식
    rejectedHandler     // 작업 거부 정책
);
```

작업이 들어오면 이런 순서로 처리한다:

1. 현재 스레드 수 < `corePoolSize` → 새 스레드를 만들어서 실행
2. 현재 스레드 수 >= `corePoolSize` → `workQueue`에 넣음
3. `workQueue`가 꽉 참 + 현재 스레드 수 < `maximumPoolSize` → 새 스레드를 만들어서 실행
4. `workQueue`가 꽉 참 + 현재 스레드 수 >= `maximumPoolSize` → `rejectedHandler` 실행

이 순서가 중요하다. 큐가 먼저 차고, 그 다음에 스레드가 늘어난다. `LinkedBlockingQueue`처럼 무제한 큐를 쓰면 `maximumPoolSize`는 의미가 없어진다.

### FixedThreadPool

```java
ExecutorService pool = Executors.newFixedThreadPool(10);
// 내부 구현:
// new ThreadPoolExecutor(10, 10, 0L, MILLISECONDS, new LinkedBlockingQueue<>())
```

`corePoolSize`와 `maximumPoolSize`가 같다. 스레드가 10개 고정이고 줄어들지 않는다. 큐는 `LinkedBlockingQueue`라서 사실상 무제한이다.

**문제점**: 요청이 처리 속도보다 빨리 들어오면 큐에 작업이 계속 쌓인다. 큐가 무제한이니까 `OutOfMemoryError`가 날 때까지 쌓인다. 프로덕션에서 이걸 그대로 쓰면 안 되는 이유다.

**적합한 경우**: 동시 처리량을 정확히 제한해야 하는 작업. 외부 API 호출처럼 동시 연결 수를 맞춰야 할 때.

### CachedThreadPool

```java
ExecutorService pool = Executors.newCachedThreadPool();
// 내부 구현:
// new ThreadPoolExecutor(0, Integer.MAX_VALUE, 60L, SECONDS, new SynchronousQueue<>())
```

`corePoolSize`가 0이고 `maximumPoolSize`가 `Integer.MAX_VALUE`다. `SynchronousQueue`를 써서 큐에 작업을 보관하지 않고 바로 스레드에 넘긴다. 유휴 스레드가 없으면 새로 만들고, 60초 동안 놀면 제거한다.

**문제점**: 스레드 수에 상한이 없다. 요청이 급증하면 스레드가 수천 개 생기면서 시스템이 멈출 수 있다.

**적합한 경우**: 짧은 비동기 작업이 간헐적으로 들어오는 경우. 작업 시간이 짧고 동시에 몰리지 않는다는 보장이 있을 때만 쓴다.

### ScheduledThreadPool

```java
ScheduledExecutorService pool = Executors.newScheduledThreadPool(5);

// 3초 후 1회 실행
pool.schedule(() -> doSomething(), 3, TimeUnit.SECONDS);

// 1초 후 시작, 이후 5초 간격으로 반복
pool.scheduleAtFixedRate(() -> collectMetrics(), 1, 5, TimeUnit.SECONDS);

// 이전 작업 완료 후 5초 뒤에 다음 실행
pool.scheduleWithFixedDelay(() -> syncData(), 1, 5, TimeUnit.SECONDS);
```

`scheduleAtFixedRate`와 `scheduleWithFixedDelay`의 차이를 혼동하는 경우가 많다. `FixedRate`는 이전 작업 시작 시점 기준으로 다음 실행을 잡고, `FixedDelay`는 이전 작업 완료 시점 기준이다. 작업 시간이 간격보다 길면 `FixedRate`에서 작업이 밀린다.

**적합한 경우**: 메트릭 수집, 캐시 갱신, 주기적 정리 작업.

### WorkStealingPool (ForkJoinPool)

```java
ExecutorService pool = Executors.newWorkStealingPool();
// CPU 코어 수만큼 스레드 생성
// 내부적으로 ForkJoinPool 사용
```

일반 스레드풀과 구조가 다르다. 스레드마다 자기 작업 큐(deque)를 갖는다. 자기 큐가 비면 다른 스레드의 큐 끝에서 작업을 가져온다(stealing). 이렇게 하면 특정 스레드만 바쁘고 나머지는 노는 상황을 줄일 수 있다.

```
[Thread-1 큐] ← task1, task2, task3
[Thread-2 큐] ← (비어있음) → Thread-1 큐에서 task3을 steal
[Thread-3 큐] ← task4
```

**적합한 경우**: 작업을 작은 단위로 분할할 수 있는 재귀적 작업. `parallelStream()`이 내부적으로 `ForkJoinPool.commonPool()`을 쓴다.

**주의**: `ForkJoinPool.commonPool()`은 JVM 전체에서 공유한다. 여기서 블로킹 작업을 하면 `parallelStream()` 등 다른 곳에도 영향이 간다. 블로킹 작업이 있으면 별도 `ForkJoinPool`을 만들어 사용한다.

### 직접 ThreadPoolExecutor 만들기 (권장)

프로덕션에서는 `Executors` 팩토리 메서드를 쓰지 않고 직접 `ThreadPoolExecutor`를 만드는 게 낫다. 큐 크기와 스레드 수를 명시적으로 제어할 수 있다.

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,                          // corePoolSize
    20,                          // maximumPoolSize
    60L, TimeUnit.SECONDS,       // keepAliveTime
    new ArrayBlockingQueue<>(100), // 크기 제한된 큐
    new ThreadPoolExecutor.CallerRunsPolicy() // 거부 정책
);
```

## 작업 거부 정책 (Rejection Policy)

큐가 꽉 차고 스레드도 전부 바쁠 때 새 작업을 어떻게 할 것인가.

| 정책 | 동작 | 실무 선택 기준 |
|------|------|--------------|
| `AbortPolicy` (기본) | `RejectedExecutionException` 발생 | 작업 유실을 감지해야 할 때 |
| `CallerRunsPolicy` | 제출한 스레드가 직접 실행 | 작업 유실이 안 되고, 호출자가 느려져도 괜찮을 때 |
| `DiscardPolicy` | 조용히 버림 | 최신 데이터만 의미 있는 경우 (로그 등) |
| `DiscardOldestPolicy` | 가장 오래된 작업을 버리고 새 작업 넣음 | 오래된 작업이 의미 없는 실시간 처리 |

`CallerRunsPolicy`가 실무에서 자주 쓰인다. 풀이 바쁘면 요청을 보낸 스레드(보통 톰캣 워커 스레드)가 직접 실행하니까 자연스럽게 백프레셔가 걸린다. 다만 호출자가 블로킹되는 점은 알고 있어야 한다.

커스텀 정책이 필요하면 `RejectedExecutionHandler`를 구현한다:

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10, 20, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100),
    (runnable, pool) -> {
        log.warn("Task rejected. Queue size: {}, Active: {}",
            pool.getQueue().size(), pool.getActiveCount());
        // 메트릭 카운터 증가, 알림 발송 등
    }
);
```

## 풀 크기 산정

### CPU bound 작업

계산 위주 작업이라 스레드가 CPU를 계속 쓴다. 스레드를 코어 수보다 많이 만들어봤자 컨텍스트 스위칭만 늘어난다.

```
풀 크기 = CPU 코어 수 + 1
```

+1은 페이지 폴트 등으로 한 스레드가 잠깐 멈출 때를 대비한 여유분이다.

```java
int poolSize = Runtime.getRuntime().availableProcessors() + 1;
```

### I/O bound 작업

DB 쿼리, HTTP 호출, 파일 읽기 등 대기 시간이 긴 작업이다. 스레드가 I/O를 기다리는 동안 CPU는 놀고 있으니 스레드를 더 만들어야 CPU를 활용할 수 있다.

```
풀 크기 = CPU 코어 수 × (1 + 대기시간 / 처리시간)
```

예를 들어 8코어 서버에서 외부 API 호출이 평균 200ms 걸리고 그 외 처리가 20ms라면:

```
8 × (1 + 200/20) = 8 × 11 = 88
```

실제로 88개를 그대로 쓰는 건 아니다. 이 공식은 출발점이고, 부하 테스트로 검증해야 한다. DB 커넥션 풀 크기, 외부 시스템 동시 연결 제한 같은 하류 자원 제약도 고려해야 한다.

### 실무에서 자주 하는 실수

**FixedThreadPool을 큐 크기 제한 없이 사용**: 앞에서 말한 대로 `OutOfMemoryError` 원인이 된다.

**CachedThreadPool을 I/O 작업에 사용**: I/O 대기 중인 스레드가 쌓이면서 스레드가 수천 개 생길 수 있다.

**parallelStream()에서 블로킹 호출**: `ForkJoinPool.commonPool()`을 공유하기 때문에 다른 `parallelStream()`까지 느려진다. 별도 풀을 만들거나 `CompletableFuture`와 커스텀 `Executor`를 조합한다.

```java
// parallelStream()에서 블로킹 작업을 피하는 방법
ExecutorService ioPool = Executors.newFixedThreadPool(50);
List<CompletableFuture<Result>> futures = items.stream()
    .map(item -> CompletableFuture.supplyAsync(() -> callExternalApi(item), ioPool))
    .collect(Collectors.toList());

List<Result> results = futures.stream()
    .map(CompletableFuture::join)
    .collect(Collectors.toList());
```

## Tomcat 스레드풀 설정과 모니터링

### 주요 설정값

`application.yml` (Spring Boot):

```yaml
server:
  tomcat:
    threads:
      max: 200       # 최대 워커 스레드 수 (기본 200)
      min-spare: 10  # 최소 유지 스레드 수 (기본 10)
    max-connections: 8192  # 최대 TCP 연결 수 (기본 8192)
    accept-count: 100      # 큐 대기 가능 요청 수 (기본 100)
```

이 값들의 관계를 알아야 한다:

1. 요청이 들어오면 TCP 연결이 맺어진다 (`max-connections` 범위 내)
2. 워커 스레드가 할당되어 요청을 처리한다 (`threads.max` 범위 내)
3. 워커 스레드가 전부 바쁘면 `accept-count` 큐에서 대기한다
4. 큐도 꽉 차면 OS 레벨에서 연결을 거부한다

`max-connections`와 `threads.max`는 다르다. NIO 커넥터(기본값)에서는 하나의 스레드가 여러 연결을 다룰 수 있다. `max-connections`이 8192이고 `threads.max`가 200이라는 건, 최대 8192개 연결을 유지하되 동시에 처리하는 건 200개라는 뜻이다.

### threads.max 튜닝

기본값 200은 대부분의 경우 적당하지만, 애플리케이션 특성에 따라 조정이 필요하다.

**API 서버 (I/O 바운드)**: 외부 API 호출, DB 쿼리가 많으면 스레드가 대기하는 시간이 길다. 200~500 범위에서 조정한다. 단, DB 커넥션 풀보다 크게 잡으면 커넥션 대기로 병목이 이동할 뿐이다.

**연산 위주 서버**: CPU 바운드 작업이 많으면 스레드를 늘려도 의미가 없다. 코어 수의 2배 정도로 줄이는 게 나을 수 있다.

### Actuator로 모니터링

Spring Boot Actuator + Micrometer를 쓰면 톰캣 스레드풀 메트릭을 Prometheus로 내보낼 수 있다.

```yaml
management:
  endpoints:
    web:
      exposure:
        include: metrics,health
  metrics:
    tags:
      application: my-app
```

주요 메트릭:

```
tomcat_threads_current_threads     # 현재 스레드 수
tomcat_threads_busy_threads        # 처리 중인 스레드 수
tomcat_threads_config_max_threads  # 설정된 최대값
```

Grafana 대시보드에서 `tomcat_threads_busy_threads / tomcat_threads_config_max_threads` 비율을 보면 풀 사용률을 알 수 있다. 이 값이 지속적으로 80%를 넘으면 풀 크기를 늘리거나 처리 시간을 줄여야 한다.

## 풀 고갈 시 증상과 대응

### 증상

스레드풀이 고갈되면 다음 현상이 나타난다:

**응답 시간 급증**: 평소 100ms이던 API가 갑자기 5초, 10초가 걸린다. 작업이 큐에서 대기하는 시간이 응답 시간에 그대로 더해진다.

**타임아웃 에러 폭증**: 클라이언트 쪽에서 `SocketTimeoutException`이나 504 Gateway Timeout이 쏟아진다.

**CPU 사용률은 낮은데 처리가 안 됨**: 스레드가 전부 I/O 대기 중이면 CPU는 놀고 있는데 요청은 처리가 안 된다. 이 상태를 처음 보면 원인을 찾기 어렵다.

**OOM 발생**: 무제한 큐를 쓰는 경우, 큐에 작업 객체가 계속 쌓이면서 힙이 터진다.

### 원인 파악

대부분 하류 시스템(DB, 외부 API, 레디스 등)의 응답 지연이 원인이다. 하나의 외부 시스템이 느려지면 그쪽을 호출하는 스레드가 전부 블로킹되고, 결국 풀 전체가 고갈된다.

### 대응

**서킷 브레이커 적용**: 하류 시스템이 느려지면 빠르게 실패시켜서 스레드를 점유하지 않게 한다. Resilience4j나 Spring Cloud Circuit Breaker를 쓴다.

**타임아웃 설정**: 외부 호출에 반드시 타임아웃을 건다. 연결 타임아웃(connection timeout)과 읽기 타임아웃(read timeout)을 분리해서 설정한다.

```java
// RestTemplate 타임아웃 예시
RestTemplate restTemplate = new RestTemplate();
HttpComponentsClientHttpRequestFactory factory = new HttpComponentsClientHttpRequestFactory();
factory.setConnectTimeout(3000);    // 연결 타임아웃 3초
factory.setReadTimeout(5000);       // 읽기 타임아웃 5초
restTemplate.setRequestFactory(factory);
```

**벌크헤드 패턴**: 서로 다른 외부 시스템 호출에 별도 스레드풀을 할당한다. A 시스템이 느려져도 B 시스템 호출용 풀은 영향 없다.

```java
// 외부 시스템별 별도 풀
ExecutorService paymentPool = new ThreadPoolExecutor(10, 20, ...);
ExecutorService inventoryPool = new ThreadPoolExecutor(5, 10, ...);
```

## jstack으로 풀 상태 확인

서버가 느려졌을 때 가장 먼저 할 수 있는 건 스레드 덤프를 뜨는 것이다.

### 덤프 뜨기

```bash
# PID 확인
jps -l

# 스레드 덤프 출력
jstack <PID> > thread_dump.txt

# 여러 번 뜨기 (3초 간격으로 3번)
for i in 1 2 3; do jstack <PID> > dump_$i.txt; sleep 3; done
```

여러 번 떠서 비교하는 게 중요하다. 한 번만 보면 그 순간의 스냅샷이라 블로킹인지 일시적인 상태인지 구분이 안 된다. 3초 간격으로 3번 떠서 같은 스레드가 같은 위치에 계속 있으면 문제가 있는 것이다.

### 덤프 분석

```
"pool-1-thread-3" #15 prio=5 os_prio=0 tid=0x00007f... nid=0x1a03 waiting on condition
   java.lang.Thread.State: WAITING (parking)
        at sun.misc.Unsafe.park(Native Method)
        - parking to wait for <0x00000000d6a08e88>
        at java.util.concurrent.locks.LockSupport.park(LockSupport.java:175)
        at java.util.concurrent.LinkedBlockingQueue.take(LinkedBlockingQueue.java:442)
        at java.util.concurrent.ThreadPoolExecutor.getTask(ThreadPoolExecutor.java:1074)
        at java.util.concurrent.ThreadPoolExecutor.runWorker(ThreadPoolExecutor.java:1134)
```

이 스레드는 `LinkedBlockingQueue.take()`에서 대기 중이다. 큐에서 작업을 꺼내려고 기다리는 상태니까 유휴 상태다. 정상이다.

문제가 되는 패턴:

```
"http-nio-8080-exec-5" #25 prio=5 os_prio=0 tid=0x00007f... nid=0x1b07 runnable
   java.lang.Thread.State: RUNNABLE
        at java.net.SocketInputStream.socketRead0(Native Method)
        at java.net.SocketInputStream.socketRead(SocketInputStream.java:116)
        at java.net.SocketInputStream.read(SocketInputStream.java:171)
        at org.apache.http.impl.io.SessionInputBufferImpl.streamRead(...)
        at org.apache.http.impl.conn.DefaultHttpResponseParser.parseHead(...)
```

톰캣 워커 스레드가 `SocketInputStream.socketRead0`에 걸려 있다. 외부 HTTP 호출의 응답을 기다리고 있는 것이다. 이런 스레드가 대량으로 보이면 외부 시스템 응답 지연이 원인이다.

### 자주 보이는 패턴과 의미

| 스택 트레이스 키워드 | 상태 | 의미 |
|---------------------|------|------|
| `LinkedBlockingQueue.take` | WAITING | 유휴 상태 (정상) |
| `SocketInputStream.socketRead0` | RUNNABLE | 외부 호출 대기 |
| `Object.wait` | WAITING | 락 대기 또는 조건 대기 |
| `LockSupport.parkNanos` | TIMED_WAITING | 타임아웃 대기 |
| `AbstractQueuedSynchronizer` | WAITING | ReentrantLock 대기 |
| `mysql-cj` 또는 JDBC 드라이버 | RUNNABLE | DB 쿼리 응답 대기 |

### 정리용 원라이너

```bash
# BLOCKED 상태 스레드 개수
grep -c "Thread.State: BLOCKED" thread_dump.txt

# 특정 스레드풀의 스레드 상태 요약
grep "pool-1-thread" thread_dump.txt | wc -l

# 가장 많이 나타나는 스택 트레이스 상위 10개
grep -A 1 "Thread.State" thread_dump.txt | grep -v "^--$" | sort | uniq -c | sort -rn | head -20
```

## ExecutorService 종료 처리

스레드풀을 안 닫으면 JVM이 종료되지 않는다. non-daemon 스레드가 살아있기 때문이다.

```java
executor.shutdown(); // 새 작업 거부, 기존 작업은 완료까지 대기
try {
    if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
        executor.shutdownNow(); // 30초 안에 안 끝나면 강제 종료
        if (!executor.awaitTermination(10, TimeUnit.SECONDS)) {
            log.error("Pool did not terminate");
        }
    }
} catch (InterruptedException e) {
    executor.shutdownNow();
    Thread.currentThread().interrupt();
}
```

Spring에서는 `@Bean` 메서드로 `ExecutorService`를 등록하면 `@PreDestroy`에서 자동으로 `shutdown()`을 호출하지 않는다. `ThreadPoolTaskExecutor`를 쓰거나, `DisposableBean`을 구현하거나, `@Bean(destroyMethod = "shutdown")`을 명시해야 한다.

```java
@Bean(destroyMethod = "shutdown")
public ExecutorService taskExecutor() {
    return new ThreadPoolExecutor(10, 20, 60L, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(100));
}
```

## 스레드풀 모니터링 지표

`ThreadPoolExecutor`에서 확인할 수 있는 값들:

```java
ThreadPoolExecutor pool = ...;

pool.getPoolSize();          // 현재 스레드 수
pool.getActiveCount();       // 작업 실행 중인 스레드 수
pool.getQueue().size();      // 큐 대기 작업 수
pool.getCompletedTaskCount(); // 완료된 작업 수
pool.getLargestPoolSize();   // 역대 최대 스레드 수
pool.getTaskCount();         // 총 제출된 작업 수
```

이 값들을 주기적으로 Prometheus 등에 내보내면 문제가 생기기 전에 감지할 수 있다.

**경고 기준 예시**:
- `activeCount / maximumPoolSize > 0.8` → 풀 포화 임박
- `queue.size()` 가 지속적으로 증가 → 처리 속도가 유입 속도를 못 따라감
- `completedTaskCount` 증가율이 떨어짐 → 개별 작업 처리 시간이 길어지고 있음
