# CPU 스케줄링

## 개요

CPU 스케줄링은 **어떤 프로세스에 CPU를 할당할지** 결정하는 운영체제의 핵심 기능이다. 한정된 CPU 자원을 여러 프로세스가 공유해야 하므로, 효율적인 스케줄링이 시스템 성능을 좌우한다.

### 스케줄링이 필요한 이유

```
CPU-bound 프로세스:  ██████████████████████  (계산 위주, CPU 오래 사용)
I/O-bound 프로세스:  ██░░░░██░░░░██░░░░██  (I/O 대기가 많음)

→ I/O 대기 중인 시간에 다른 프로세스를 실행하면 CPU 활용도가 올라간다
```

### 스케줄링 성능 지표

| 지표 | 설명 | 최적화 방향 |
|------|------|------------|
| **CPU 이용률** | CPU가 일하는 시간 비율 | 높을수록 좋음 |
| **처리량 (Throughput)** | 단위 시간당 완료 프로세스 수 | 높을수록 좋음 |
| **대기 시간 (Waiting Time)** | Ready Queue에서 대기한 시간 | 낮을수록 좋음 |
| **응답 시간 (Response Time)** | 요청 후 첫 응답까지 시간 | 낮을수록 좋음 |
| **반환 시간 (Turnaround Time)** | 제출~완료까지 총 시간 | 낮을수록 좋음 |

### 선점 vs 비선점

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **비선점 (Non-preemptive)** | 프로세스가 CPU를 자발적으로 반납할 때까지 대기 | 구현 단순, 오버헤드 적음 | 응답 시간 보장 불가 |
| **선점 (Preemptive)** | OS가 강제로 CPU를 회수 가능 | 응답 시간 보장 | 컨텍스트 스위칭 오버헤드 |

## 핵심

### 1. FCFS (First Come, First Served)

가장 단순한 알고리즘. **먼저 도착한 프로세스**가 먼저 실행된다. 비선점.

```
프로세스 도착 순서: P1(24ms) → P2(3ms) → P3(3ms)

간트 차트:
|-------- P1 (24ms) --------|-- P2 --|- P3 -|
0                           24      27     30

대기 시간: P1=0, P2=24, P3=27  →  평균 = 17ms
```

**Convoy Effect**: CPU-bound 프로세스 뒤에 I/O-bound 프로세스가 오래 대기하는 현상.

```
❌ P1(24ms) → P2(3ms) → P3(3ms)   평균 대기: 17ms
✅ P2(3ms) → P3(3ms) → P1(24ms)   평균 대기: 2ms  (순서만 바꿔도 큰 차이)
```

### 2. SJF (Shortest Job First)

**실행 시간이 가장 짧은 프로세스**를 먼저 실행한다. 평균 대기 시간이 이론적으로 최소.

```
프로세스: P1(6ms), P2(8ms), P3(7ms), P4(3ms)

SJF 순서:
|- P4 -|--- P1 ---|---- P3 ----|------- P2 --------|
0      3         9           16                   24

대기 시간: P4=0, P1=3, P3=9, P2=16  →  평균 = 7ms
```

| 장점 | 단점 |
|------|------|
| 최적의 평균 대기 시간 | 실행 시간을 미리 알 수 없음 (예측 필요) |
| 이론적 하한선 | **기아 현상**: 긴 프로세스가 계속 밀림 |

#### SRTF (Shortest Remaining Time First)

SJF의 **선점 버전**. 남은 시간이 더 짧은 프로세스가 도착하면 현재 프로세스를 선점한다.

```
시간 0: P1(7ms) 도착 → P1 실행
시간 2: P2(4ms) 도착 → P2가 P1 선점 (P2 남은 4ms < P1 남은 5ms)
시간 4: P3(1ms) 도착 → P3가 P2 선점
시간 5: P3 완료 → P2 재개 (남은 2ms)
시간 7: P2 완료 → P1 재개 (남은 5ms)
시간 12: P1 완료
```

### 3. Priority Scheduling

각 프로세스에 **우선순위**를 부여하고, 높은 우선순위부터 실행한다.

```
프로세스: P1(우선순위 3), P2(우선순위 1), P3(우선순위 4), P4(우선순위 2)
(숫자가 낮을수록 높은 우선순위)

실행 순서: P2(1) → P4(2) → P1(3) → P3(4)
```

**기아 문제 해결 — Aging**:

```
대기 시간이 길어질수록 우선순위를 점진적으로 높여준다.

시간 0: P1(우선순위 40)  ← 낮은 우선순위
시간 1: P1(우선순위 39)
시간 2: P1(우선순위 38)
...
시간 30: P1(우선순위 10)  ← 충분히 높아져서 실행됨
```

### 4. Round Robin (RR)

**타임 퀀텀**(Time Quantum)만큼 돌아가며 실행하는 선점 알고리즘. 현대 OS의 기본.

```
타임 퀀텀: 4ms
프로세스: P1(24ms), P2(3ms), P3(3ms)

|--P1--|P2-|P3-|--P1--|--P1--|--P1--|--P1--|--P1--|
0     4   7  10    14    18    22    26    30

P2, P3는 빨리 완료. P1은 나눠서 실행.
```

#### 타임 퀀텀 크기의 영향

| 타임 퀀텀 | 특징 |
|----------|------|
| **너무 크면** (∞) | FCFS와 동일, 응답 시간 길어짐 |
| **너무 작으면** (1ms) | 컨텍스트 스위칭 오버헤드 급증 |
| **적절한 크기** | 보통 10~100ms, 전체 CPU burst의 80%가 퀀텀 이내 |

```
타임 퀀텀 = 1ms (너무 작음):
|P1|P2|P3|P1|P2|P3|P1|P2|P3|...  → 컨텍스트 스위칭이 실행 시간보다 길어짐

타임 퀀텀 = 100ms (너무 큼):
|-------P1 (24ms)-------|P2 (3ms)|P3 (3ms)|  → FCFS와 다를 바 없음

타임 퀀텀 = 4ms (적절):
|--P1--|P2-|P3-|--P1--|...  → 균형 잡힌 응답 시간
```

### 5. MLFQ (Multi-Level Feedback Queue)

**현대 OS가 실제로 사용하는** 스케줄링. 여러 우선순위 큐를 두고 프로세스의 행동에 따라 큐를 이동시킨다.

```
Queue 0 (최고 우선순위, 퀀텀 8ms):   [P1] [P4]
    │ 타임 퀀텀 내 완료 못하면 ↓
Queue 1 (중간 우선순위, 퀀텀 16ms):  [P2] [P5]
    │ 타임 퀀텀 내 완료 못하면 ↓
Queue 2 (최저 우선순위, FCFS):       [P3]
```

#### MLFQ 규칙

| 규칙 | 설명 |
|------|------|
| 1 | 높은 큐의 프로세스를 먼저 실행 |
| 2 | 같은 큐면 Round Robin |
| 3 | 새 프로세스는 최상위 큐에 배치 |
| 4a | 타임 퀀텀을 다 쓰면 한 단계 아래로 |
| 4b | 타임 퀀텀 전에 CPU를 반납하면 같은 큐 유지 |
| 5 | 주기적으로 모든 프로세스를 최상위 큐로 이동 (**Priority Boost**) |

```
프로세스 동작 예시:

I/O-bound (짧은 CPU burst):
  Queue 0 → 실행 2ms → I/O 대기 → Queue 0 (유지)
  → 항상 높은 우선순위, 빠른 응답

CPU-bound (긴 CPU burst):
  Queue 0 → 8ms 실행 → Queue 1
  Queue 1 → 16ms 실행 → Queue 2
  → 점점 낮은 우선순위, 배치 처리

Priority Boost (S초마다):
  Queue 2의 P3 → Queue 0으로 이동 (기아 방지)
```

### 6. 알고리즘 비교

| 알고리즘 | 선점 | 기아 | 평균 대기 | 응답 시간 | 사용처 |
|---------|------|------|----------|----------|--------|
| **FCFS** | 비선점 | 없음 | 길 수 있음 | 길 수 있음 | 배치 시스템 |
| **SJF** | 비선점 | 있음 | 최소 (이론) | 보통 | 이론적 |
| **SRTF** | 선점 | 있음 | 최소 | 좋음 | 이론적 |
| **Priority** | 둘 다 | 있음 (Aging 해결) | 보통 | 보통 | 실시간 시스템 |
| **Round Robin** | 선점 | 없음 | 보통 | 좋음 | 타임셰어링 |
| **MLFQ** | 선점 | 없음 (Boost) | 좋음 | 좋음 | **현대 범용 OS** |

### 7. 컨텍스트 스위칭

프로세스 전환 시 현재 상태를 저장하고 다음 프로세스의 상태를 복원하는 과정이다.

```
1. 현재 프로세스(P1)의 PCB에 상태 저장
   - 레지스터, 프로그램 카운터, 스택 포인터 저장

2. 다음 프로세스(P2)의 PCB에서 상태 복원
   - 저장된 레지스터, PC 등 복원

3. P2 실행 재개
```

| 항목 | 프로세스 전환 | 스레드 전환 |
|------|-------------|-----------|
| **저장/복원 대상** | 전체 메모리 맵, 레지스터, PCB | 레지스터, 스택 포인터만 |
| **비용** | 높음 (TLB 플러시 포함) | 낮음 (메모리 공유) |
| **소요 시간** | 수~수십 μs | 수 μs 이하 |

### 8. 실시간 스케줄링

실시간 시스템에서는 **데드라인**을 반드시 지켜야 한다.

| 알고리즘 | 기준 | 특징 |
|---------|------|------|
| **RM (Rate Monotonic)** | 주기가 짧을수록 높은 우선순위 | 정적 우선순위, 분석 용이 |
| **EDF (Earliest Deadline First)** | 데드라인이 가까울수록 높은 우선순위 | 동적 우선순위, CPU 이용률 높음 |

```
RM 예시:
  Task A: 주기 10ms, 실행 3ms → 우선순위 1 (높음)
  Task B: 주기 20ms, 실행 5ms → 우선순위 2

EDF 예시:
  시간 0: A 데드라인 10ms, B 데드라인 20ms → A 먼저
  시간 3: A 완료, B 실행
  시간 8: B 실행 중, A 새 데드라인 20ms → B 계속 (B 데드라인이 더 가까움)
```

### 9. Linux 스케줄러

#### CFS (Completely Fair Scheduler)

Linux 기본 스케줄러. **vruntime**(가상 실행 시간)이 가장 작은 프로세스를 선택한다.

```
Red-Black Tree (vruntime 기준 정렬):

       P3 (vruntime: 15)
      /                \
P1 (vruntime: 10)   P4 (vruntime: 20)
                    /
               P2 (vruntime: 18)

→ P1이 vruntime 가장 작으므로 다음 실행
→ P1 실행 후 vruntime 증가 → 트리 재배치
```

```bash
# 프로세스 우선순위 확인
ps -eo pid,ni,pri,comm --sort=-pri

# nice 값 변경 (-20 ~ 19, 낮을수록 높은 우선순위)
nice -n -5 ./my_process        # 높은 우선순위로 실행
renice -n 10 -p 1234           # 실행 중인 프로세스 우선순위 변경

# 실시간 스케줄링 정책 설정
chrt -f 50 ./my_process        # SCHED_FIFO, 우선순위 50
chrt -r 30 ./my_process        # SCHED_RR, 우선순위 30
```

| 스케줄링 클래스 | 용도 | 정책 |
|---------------|------|------|
| `SCHED_NORMAL` | 일반 프로세스 | CFS |
| `SCHED_FIFO` | 실시간 (비선점) | 우선순위 기반 FCFS |
| `SCHED_RR` | 실시간 (선점) | 우선순위 기반 RR |
| `SCHED_DEADLINE` | 데드라인 기반 | EDF |
