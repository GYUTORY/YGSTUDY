# 메모리 관리 (Memory Management)

## 메모리 관리란

운영체제가 물리 메모리를 관리하는 기능이다. 여러 프로세스가 동시에 실행되려면 각 프로세스에 메모리를 할당하고, 프로세스끼리 메모리를 침범하지 못하게 격리해야 한다.

핵심 역할은 네 가지다:

- **다중 프로그래밍 지원**: 여러 프로세스가 동시에 메모리에 올라와 실행된다
- **메모리 보호**: 프로세스가 자기 영역 밖의 메모리에 접근하면 커널이 차단한다
- **메모리 활용**: 제한된 물리 메모리를 낭비 없이 쓴다
- **투명성**: 프로세스는 메모리 관리 방식을 몰라도 동작한다

## 메모리 주소 체계

### 논리적 주소 vs 물리적 주소

**논리적 주소 (Logical Address)**
- 프로세스가 사용하는 주소 공간이다
- 각 프로세스마다 독립적으로 0부터 시작한다
- CPU가 생성하는 주소로, 가상 주소라고도 한다

**물리적 주소 (Physical Address)**
- 실제 메모리 하드웨어의 주소다
- MMU(Memory Management Unit)가 논리적 주소를 물리적 주소로 변환한다
- 시스템 전체에서 유일하다

### 주소 변환 과정

1. CPU가 논리적 주소를 생성한다
2. MMU가 논리적 주소를 물리적 주소로 변환한다
3. 변환된 물리적 주소로 실제 메모리에 접근한다

MMU는 하드웨어 수준에서 동작하므로, 프로세스 입장에서는 변환 과정이 보이지 않는다. 다만 TLB(Translation Lookaside Buffer) 미스가 잦아지면 성능이 눈에 띄게 떨어지는 경우가 있다. `perf stat`으로 TLB miss 비율을 확인할 수 있다.

## 메모리 할당

### 연속 메모리 할당 (Contiguous Memory Allocation)

프로세스를 연속된 메모리 공간에 배치하는 방식이다. 구현이 단순하고 주소 변환이 간단하지만, 프로세스가 생성/종료를 반복하면 빈 공간이 파편화되어 외부 단편화가 발생한다. 현대 운영체제는 이 문제를 페이징으로 해결한다.

### 할당 알고리즘 — 교과서 vs 실제 커널

교과서에서 다루는 세 가지 알고리즘부터 보자.

#### 최초 적합 (First Fit)
빈 공간 리스트를 순서대로 탐색해서, 요청 크기를 수용할 수 있는 첫 번째 공간에 할당한다. 탐색이 빠르고 구현이 간단하다. 다만 메모리 앞쪽에 작은 조각이 쌓이는 경향이 있다.

#### 최적 적합 (Best Fit)
요청 크기를 수용할 수 있는 공간 중 가장 작은 공간에 할당한다. 낭비를 줄이려는 의도지만, 매번 전체 리스트를 탐색해야 하고 너무 작은 잔여 조각이 많이 생긴다.

#### 최악 적합 (Worst Fit)
가장 큰 빈 공간에 할당한다. 남는 공간이 크므로 재활용 가능성이 높다는 아이디어지만, 실제로는 큰 블록이 빠르게 소모되어 대형 할당 요청에 대응하기 어려워진다.

실험적으로 First Fit이 속도와 메모리 활용 면에서 가장 균형 잡힌 결과를 보여준다.

#### 실제 Linux 커널의 메모리 할당

Linux 커널은 위의 교과서적 알고리즘을 직접 쓰지 않는다. 대신 용도에 따라 두 가지 할당기를 조합한다.

**Buddy System (버디 시스템)**

물리 페이지 프레임 할당에 사용한다. 메모리를 2^n 페이지 크기 블록으로 관리하며, 할당 요청이 오면 적절한 크기의 블록을 찾아 반환한다. 블록이 부족하면 큰 블록을 절반으로 쪼개고, 해제 시에는 인접한 "버디" 블록끼리 다시 합친다.

```bash
# 현재 buddy system 상태 확인
$ cat /proc/buddyinfo
Node 0, zone   Normal   1024   512   256   128   64   32   16   8   4   2   1
#                        2^0   2^1   2^2  ...  각 order별 사용 가능한 블록 수
```

order별 블록 수가 고르게 분포되어 있으면 정상이다. 높은 order 블록이 0인 상태가 계속되면 메모리 단편화가 심한 것이므로, 해당 서버의 메모리 증설이나 워크로드 분산을 검토해야 한다.

**Slab Allocator (슬랩 할당기)**

커널 내부 객체(task_struct, inode, dentry 등) 할당에 사용한다. Buddy System은 페이지(4KB) 단위로 할당하므로, 수십~수백 바이트짜리 커널 객체를 매번 페이지 단위로 할당하면 낭비가 심하다. Slab Allocator는 같은 크기의 객체를 미리 잘라둔 캐시(slab)에서 꺼내 쓰고, 다 쓰면 캐시에 반환한다.

```bash
# slab 캐시 사용 현황
$ slabtop -o | head -15
# OBJS     ACTIVE  USE   OBJ SIZE  SLABS  OBJ/SLAB  CACHE SIZE  NAME
# 42840    42840  100%    0.19K   2142       20     8568K  dentry
# 31200    28953   92%    1.06K    975       32    31200K  ext4_inode_cache

# /proc/slabinfo로도 확인 가능
$ cat /proc/slabinfo | head -5
```

dentry나 inode 캐시가 메모리를 많이 차지하는 건 정상이다. 파일 시스템 메타데이터를 캐싱하고 있는 것이므로, 메모리 압박이 생기면 커널이 자동으로 회수한다.

**glibc malloc (유저 스페이스)**

애플리케이션의 `malloc()` 호출은 glibc의 malloc 구현이 처리한다. 내부적으로 작은 할당은 `brk()`로 힙을 확장하고, 큰 할당(기본 128KB 이상)은 `mmap()`으로 별도 영역을 매핑한다. 멀티스레드 환경에서는 스레드별 arena를 만들어 lock 경합을 줄인다.

```bash
# glibc malloc 디버깅 — 환경 변수로 통계 출력
$ MALLOC_STATS_=1 ./my_app
# 종료 시 arena/heap 사용 통계가 stderr로 출력된다

# jemalloc, tcmalloc 같은 대안 할당기를 쓰는 경우도 많다
# Redis는 jemalloc, Go runtime은 자체 할당기를 사용한다
```

## 메모리 단편화 (Memory Fragmentation)

### 외부 단편화 (External Fragmentation)

빈 메모리 총량은 충분한데, 연속된 공간이 없어서 할당이 안 되는 상황이다.

해결 방법:
- **압축 (Compaction)**: 사용 중인 블록을 한쪽으로 몰아서 연속 공간을 확보한다. 비용이 크다.
- **페이징 (Paging)**: 고정 크기 페이지로 나눠 할당한다. 연속 공간이 필요 없으므로 외부 단편화가 원천 차단된다.
- **세그멘테이션 (Segmentation)**: 논리적 단위로 분할한다. 외부 단편화가 여전히 발생할 수 있다.

### 내부 단편화 (Internal Fragmentation)

할당된 블록 안에서 실제로 쓰이지 않는 공간이 남는 현상이다. 고정 크기 블록으로 할당할 때 프로세스 크기가 블록보다 작으면 발생한다. 메모리 정렬(alignment) 요구 때문에 패딩이 붙는 경우도 있다.

페이징을 쓰면 외부 단편화는 없어지지만, 마지막 페이지에서 내부 단편화가 생긴다. 4KB 페이지 기준으로 프로세스당 평균 2KB 정도 낭비된다. 프로세스 수가 많아지면 무시할 수 없는 양이 된다.

## 스와핑 (Swapping)

메모리가 부족할 때 프로세스의 일부를 디스크로 내보내고(swap out), 필요할 때 다시 불러오는(swap in) 기법이다. 디스크 I/O가 발생하므로 성능이 떨어진다. 스왑 사용률이 지속적으로 높다면 메모리 부족 신호다.

스와핑의 동작 원리, 페이지 교체 알고리즘, OS별 설정 방법은 [Swap Memory](Swap Memory.md) 문서에서 다룬다.

## 메모리 보호 (Memory Protection)

프로세스 간 메모리를 격리하는 메커니즘이다. 한 프로세스가 다른 프로세스의 메모리를 건드리면 시스템이 불안정해지고, 보안 문제도 생긴다.

**경계 레지스터 (Boundary Register)**
- 프로세스의 메모리 영역 시작과 끝을 정의한다
- 범위를 벗어난 접근이 발생하면 하드웨어가 트랩을 발생시킨다
- 초기 시스템에서 사용한 방식이다

**페이지 테이블 접근 권한 비트**
- 현대 시스템은 페이지 테이블에 읽기(R), 쓰기(W), 실행(X) 권한 비트를 둔다
- 권한 없는 접근 시 segmentation fault가 발생한다
- NX 비트(No-eXecute)는 데이터 영역에서 코드가 실행되는 것을 차단하여 버퍼 오버플로 공격을 방어한다

## 현대 메모리 관리

현대 운영체제는 가상 메모리를 기본으로 사용한다. 각 프로세스가 독립적인 가상 주소 공간을 가지며, 물리 메모리보다 큰 프로그램도 실행할 수 있다.

주요 기법:
- **다단계 페이지 테이블**: x86-64는 4단계(PGD → PUD → PMD → PTE) 페이지 테이블을 사용한다. 전체 주소 공간에 대한 페이지 테이블을 한 번에 만들면 메모리 낭비가 심하므로, 실제 사용하는 영역만 테이블을 생성한다.
- **TLB (Translation Lookaside Buffer)**: 최근 주소 변환 결과를 캐싱하는 하드웨어다. TLB 히트율이 높아야 성능이 나온다.
- **Huge Pages**: 기본 4KB 대신 2MB나 1GB 페이지를 사용한다. TLB 엔트리 하나로 더 넓은 영역을 커버하므로 TLB 미스가 줄어든다. 데이터베이스처럼 대용량 메모리를 쓰는 애플리케이션에서 성능 차이가 크다.

가상 메모리의 상세 내용은 [가상 메모리](가상 메모리.md) 문서에서 다룬다.

## 코드 예제

### free / top — 메모리 사용 현황 확인

```bash
# ── free — 메모리 요약 ────────────────────────────────────
$ free -h
              total   used    free   shared  buff/cache  available
Mem:           15Gi   6.0Gi  2.0Gi   512Mi    7.0Gi       8.5Gi
Swap:         4.0Gi   200Mi  3.8Gi

# ── top — 프로세스별 메모리 (대화형) ─────────────────────
$ top
# 실행 후 키:
#   M  → 메모리 사용량 순 정렬
#   P  → CPU 사용량 순 정렬
#   k  → PID 입력 후 프로세스 종료
#   q  → 종료

# 메모리 컬럼 의미:
#   VIRT: 예약된 가상 메모리 총량 (실제 사용 아님)
#   RES:  실제 물리 메모리 사용량 (RSS)
#   SHR:  다른 프로세스와 공유되는 메모리

# ── ps — 메모리 사용 상위 프로세스 ──────────────────────
$ ps aux --sort=-%mem | head -10

# 특정 프로세스 메모리만 확인
$ ps -o pid,vsz,rss,comm -p $(pgrep nginx)
#   PID    VSZ   RSS COMMAND
#  1234 512000 45000 nginx
```

### /proc/\<pid\>/smaps — 프로세스 메모리 상세 분석

```bash
# 프로세스 PID 확인
$ PID=$(pgrep -x nginx | head -1)

# 메모리 맵 상세 조회
$ cat /proc/$PID/smaps | head -30
# 7f3a2c000000-7f3a2c021000 r--p ... libpthread.so.0
# Size:                132 kB   ← 예약된 가상 메모리
# Rss:                  44 kB   ← 실제 물리 메모리 사용
# Pss:                  11 kB   ← 공유 비례 계산값
# Private_Dirty:        44 kB   ← 프로세스만 사용, 수정된 메모리

# 전체 물리 메모리 사용량 합산
$ awk '/^Rss:/{sum += $2} END {printf "Total RSS: %.1f MB\n", sum/1024}' \
  /proc/$PID/smaps

# 단편화 확인 — 메모리 영역 수
$ wc -l /proc/$PID/maps
```

### C 언어 malloc/free — 메모리 할당과 해제

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int main() {
    // ── malloc — 동적 메모리 할당 ────────────────────────
    int *arr = (int *)malloc(10 * sizeof(int));
    if (arr == NULL) {
        perror("malloc 실패");
        return 1;
    }
    for (int i = 0; i < 10; i++) arr[i] = i * 2;
    printf("arr[5] = %d\n", arr[5]);  // 10
    free(arr);    // 반드시 해제
    arr = NULL;   // dangling pointer 방지

    // ── calloc — 0으로 초기화된 메모리 할당 ──────────────
    int *zeros = (int *)calloc(10, sizeof(int));
    printf("zeros[3] = %d\n", zeros[3]);  // 0
    free(zeros);

    // ── realloc — 메모리 크기 변경 ───────────────────────
    char *buf = (char *)malloc(64);
    strncpy(buf, "Hello", 64);
    buf = (char *)realloc(buf, 256);  // 256 바이트로 확장
    strncat(buf, ", World!", 256 - strlen(buf) - 1);
    printf("%s\n", buf);  // Hello, World!
    free(buf);

    return 0;
}
```

```bash
# 컴파일
$ gcc -g -o mem_example mem_example.c

# Valgrind로 메모리 누수 검사 (Linux)
$ valgrind --leak-check=full ./mem_example
# ==1234== LEAK SUMMARY:
# ==1234==    definitely lost: 0 bytes in 0 blocks  ← 누수 없음

# AddressSanitizer로 빠른 검사 (GCC/Clang)
$ gcc -g -fsanitize=address -o mem_asan mem_example.c
$ ./mem_asan
```
