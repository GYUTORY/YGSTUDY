# 가상 메모리(Virtual Memory)

## 개요

가상 메모리는 프로그램이 실제 물리 메모리보다 큰 메모리 공간을 사용할 수 있게 해주는 기술이다. 운영체제와 하드웨어(MMU)가 협력해서 구현한다.

각 프로세스는 독립적인 가상 주소 공간을 가진다. 프로세스 A의 0x1000 주소와 프로세스 B의 0x1000 주소는 서로 다른 물리 메모리를 가리킨다. 프로세스 간 메모리 격리가 자동으로 보장되는 이유다.

### 왜 가상 메모리가 필요한가?

초기 컴퓨터는 프로그램이 물리 메모리 크기에 직접 제한을 받았다. 4GB RAM이면 실행 중인 모든 프로그램의 메모리 합이 4GB를 넘을 수 없었다. 현실에서는 브라우저 탭 수십 개, IDE, DB 클라이언트 등이 동시에 돌아가야 하니 물리 메모리만으로는 한계가 있다.

가상 메모리가 해결하는 핵심 문제 세 가지:

- **메모리 부족**: 물리 메모리보다 많은 메모리를 사용할 수 있다
- **프로세스 격리**: 다른 프로세스의 메모리를 건드릴 수 없다
- **메모리 공유**: 여러 프로세스가 같은 라이브러리 코드를 물리 메모리 한 곳에서 공유한다 (Copy-on-Write)

## 주요 구성 요소

### 1. 가상 주소 공간 (Virtual Address Space)

모든 프로세스는 자신만의 가상 주소 공간을 가진다. 32비트 시스템에서는 4GB, 64비트 시스템에서는 이론상 16EB(엑사바이트)까지 가능하지만 실제로는 CPU 아키텍처에 따라 제한이 있다. x86-64 기준 현재 48비트(256TB)를 사용한다.

이 주소 공간은 논리적이다. 프로그램이 0x1000 주소에 데이터를 저장해도, 물리 메모리의 0x1000에 저장되는 게 아니다.

### 2. 페이지 (Page)와 페이지 프레임 (Page Frame)

가상 메모리는 **페이지**라는 고정 크기 블록으로 나뉜다. 일반적인 페이지 크기는 4KB다.

- **페이지**: 가상 메모리의 고정 크기 블록
- **페이지 프레임**: 물리 메모리의 고정 크기 블록
- **Huge Page**: 2MB 또는 1GB 크기의 페이지. 대량 메모리를 사용하는 프로세스에서 TLB 미스를 줄이기 위해 쓴다

### 3. 페이지 테이블 (Page Table)

페이지 테이블은 가상 주소를 물리 주소로 변환하는 매핑 정보를 저장하는 자료구조다. 각 프로세스마다 독립적인 페이지 테이블을 가진다.

페이지 테이블 엔트리(PTE)의 구성:

- **유효 비트 (Valid Bit)**: 해당 페이지가 물리 메모리에 있는지 여부
- **물리 주소**: 매핑된 물리 페이지 프레임의 주소
- **접근 권한**: 읽기, 쓰기, 실행 권한
- **더티 비트 (Dirty Bit)**: 페이지가 수정되었는지 여부. 스왑 아웃할 때 디스크에 다시 써야 하는지 판단하는 데 쓴다
- **참조 비트 (Reference Bit)**: 최근에 접근되었는지 여부. 페이지 교체 알고리즘이 참고한다

페이지 테이블 자체도 메모리를 차지한다. 64비트 시스템에서 단일 레벨 페이지 테이블을 쓰면 엔트리가 너무 많아지므로, 실제로는 **다단계 페이지 테이블**을 사용해서 실제로 매핑이 필요한 영역만 테이블을 생성한다.

### 다단계 페이지 테이블 구조 (x86-64 4단계)

x86-64에서 48비트 가상 주소는 4단계 페이지 테이블로 변환된다. 각 단계의 테이블은 512개의 엔트리를 가진다(9비트씩 사용).

```
  63    48 47    39 38    30 29    21 20    12 11     0
 ┌────────┬────────┬────────┬────────┬────────┬────────┐
 │ Sign   │ PML4   │ PDPT   │  PD    │  PT    │ Offset │
 │ Extend │ Index  │ Index  │ Index  │ Index  │        │
 │(16bit) │(9bit)  │(9bit)  │(9bit)  │(9bit)  │(12bit) │
 └────────┴───┬────┴───┬────┴───┬────┴───┬────┴────────┘
              │        │        │        │
              v        v        v        v
           ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
     CR3──>│PML4  │→│PDPT  │→│ PD   │→│ PT   │→ 물리 페이지 프레임
           │(512  │ │(512  │ │(512  │ │(512  │   + Offset
           │엔트리)│ │엔트리)│ │엔트리)│ │엔트리)│   = 물리 주소
           └──────┘ └──────┘ └──────┘ └──────┘
```

이 구조의 핵심은 **사용하지 않는 주소 범위에 대한 하위 테이블을 만들지 않는다**는 점이다. 프로세스가 주소 공간의 일부만 사용하면 대부분의 PML4/PDPT 엔트리가 비어있고, 해당 하위 테이블은 생성되지 않는다. 단일 레벨로 48비트 전체를 커버하려면 2^36개(약 690억)의 엔트리가 필요하지만, 다단계 구조에서는 실제 사용 영역만큼만 메모리를 쓴다.

최신 x86-64 CPU(Ice Lake 이후)는 5단계 페이지 테이블(LA57)을 지원한다. 57비트 가상 주소를 사용하며 주소 공간이 128PB로 늘어난다. 현재는 대부분의 환경에서 4단계를 사용하고, 서버용 대용량 메모리 환경에서 5단계를 사용하는 경우가 있다.

## 동작 원리

### 1. 주소 변환 과정

프로그램이 메모리에 접근하면 다음 순서로 물리 주소를 찾는다:

```
프로그램: mov rax, [0x00007f4a_12345678]
                          │
                          v
              ┌──────────────────────┐
              │  1. TLB 조회         │
              │  가상 페이지 번호     │
              │  → 물리 프레임 번호  │
              └─────────┬────────────┘
                   ┌────┴────┐
                   │         │
              TLB Hit    TLB Miss
                   │         │
                   v         v
          ┌────────────┐  ┌──────────────────┐
          │물리 주소    │  │ 2. 페이지 테이블  │
          │바로 사용    │  │    워킹 (4단계)   │
          │(~1 cycle)  │  │    (~100 cycles)  │
          └────────────┘  └────────┬─────────┘
                              ┌────┴────┐
                              │         │
                         PTE 유효    PTE 무효
                              │         │
                              v         v
                     ┌────────────┐  ┌──────────────┐
                     │TLB에 캐싱  │  │ 3. 페이지    │
                     │+ 물리 주소 │  │    폴트 처리  │
                     │   접근     │  │ (~ms 단위)   │
                     └────────────┘  └──────────────┘
```

4단계 페이지 테이블 워킹이 발생하면 메모리를 4번 읽어야 한다. 각 단계에서 한 번씩 메모리 접근이 일어나므로, TLB 미스 한 번에 메모리 접근 4번이 추가된다. 메인 메모리 접근 시간이 약 100ns라면, 페이지 테이블 워킹에만 400ns가 걸리는 셈이다.

### 2. TLB (Translation Lookaside Buffer)

페이지 테이블 조회는 메모리 접근이 필요하다. 다단계 페이지 테이블이면 한 번의 주소 변환에 여러 번 메모리를 읽어야 한다. 이 오버헤드를 줄이기 위해 MMU 안에 **TLB**라는 하드웨어 캐시가 있다.

TLB는 최근 사용된 가상 주소→물리 주소 매핑을 저장한다. TLB에 히트하면 페이지 테이블 조회 없이 바로 물리 주소를 얻는다. 일반적인 프로그램은 지역성(locality)이 높아서 TLB 히트율이 99% 이상이다.

```
                 TLB 동작 흐름
                 ─────────────

  CPU 메모리 접근 요청
          │
          v
  ┌───────────────┐     ┌────────────────────────────────┐
  │  TLB 조회     │     │  TLB 내부 구조                 │
  │  (병렬 비교)  │     │                                │
  └───────┬───────┘     │  VPN(가상)  │ PFN(물리) │ 플래그│
     ┌────┴────┐        │  ──────────┼──────────┼──────  │
     │         │        │  0x7f4a12  │ 0x3a21   │ RW-   │
  TLB Hit  TLB Miss     │  0x7f4a13  │ 0x1b05   │ R-X   │
     │         │        │  0x401     │ 0x5c32   │ R-X   │
     v         v        │  ...       │ ...      │ ...   │
 물리 주소  Page Table   └────────────────────────────────┘
  즉시 반환  Walk 시작
             │
             v
         TLB에 결과 캐싱
         (기존 엔트리 퇴거)
```

TLB 미스가 자주 발생하는 경우:

- 대량의 메모리를 무작위로 접근하는 패턴 (예: 해시 테이블의 랜덤 룩업)
- 프로세스가 매우 넓은 주소 범위를 사용하는 경우
- 컨텍스트 스위칭이 잦은 경우 (TLB가 플러시될 수 있다)

이런 상황에서는 Huge Page를 쓰면 하나의 TLB 엔트리가 커버하는 메모리 범위가 넓어져서 미스가 줄어든다. 4KB 페이지일 때 TLB 엔트리 하나가 4KB를 커버하지만, 2MB Huge Page면 같은 엔트리 하나로 2MB를 커버한다. TLB 엔트리 수가 제한되어 있으니(보통 L1 dTLB가 64개 정도) 차이가 크다.

**PCID (Process Context Identifier)**

컨텍스트 스위칭 시 TLB를 전부 플러시하면 새 프로세스가 시작할 때 TLB 미스가 쏟아진다. PCID는 TLB 엔트리에 프로세스 ID를 태깅해서, 컨텍스트 스위칭 시 TLB를 플러시하지 않고도 프로세스별 매핑을 구분할 수 있게 해준다. Linux 4.14부터 PCID를 활용하며, Spectre/Meltdown 패치(KPTI) 이후 성능 저하를 줄이는 데 중요한 역할을 한다.

### 3. 페이지 폴트 (Page Fault)

요청된 페이지가 물리 메모리에 없을 때 발생하는 인터럽트다. 두 종류가 있다:

**Minor Page Fault**
물리 메모리에 해당 데이터가 있지만 페이지 테이블에 매핑이 안 된 경우. 디스크 I/O 없이 매핑만 설정하면 된다. `malloc`으로 메모리를 할당하면 실제로 접근할 때까지 물리 메모리를 할당하지 않는데(Demand Paging), 첫 접근 시 minor fault가 발생한다.

**Major Page Fault**
데이터가 물리 메모리에 없어서 디스크에서 읽어와야 하는 경우. 디스크 I/O가 발생하므로 수백 배 느리다. major fault가 빈번하면 메모리가 부족하다는 신호다.

페이지 폴트 처리 과정:

```
  CPU: 가상 주소 접근
         │
         v
  ┌──────────────┐
  │ MMU: PTE 확인 │
  └──────┬───────┘
         │ Valid bit = 0
         v
  ┌──────────────────┐
  │ 페이지 폴트      │
  │ 인터럽트 발생    │
  └──────┬───────────┘
         v
  ┌──────────────────┐     ┌─────────────────────┐
  │ 폴트 유형 판별   │────>│ 잘못된 접근?        │
  └──────┬───────────┘     │ → SIGSEGV 전달      │
         │ 정상 접근       └─────────────────────┘
         v
    ┌────┴────┐
    │         │
  Minor     Major
    │         │
    v         v
 매핑만    ┌──────────────────┐
 설정     │ 빈 프레임 확보   │
    │     │ (없으면 페이지   │
    │     │  교체 알고리즘)  │
    │     └──────┬───────────┘
    │            v
    │     ┌──────────────────┐
    │     │ 디스크에서 페이지 │
    │     │ 읽기 (I/O 대기)  │
    │     └──────┬───────────┘
    │            │
    v            v
  ┌──────────────────┐
  │ PTE 업데이트     │
  │ (Valid=1, 물리   │
  │  프레임 주소)    │
  └──────┬───────────┘
         v
  ┌──────────────────┐
  │ 명령어 재실행    │
  └──────────────────┘
```

### 4. Demand Paging과 prefault

가상 메모리 시스템은 메모리를 요청 즉시 물리 페이지로 할당하지 않는다. `malloc(1GB)`나 `mmap(1GB)`를 호출해도 그 시점에는 가상 주소 영역만 예약된다. 실제 물리 페이지는 해당 주소에 처음 쓰기 접근이 일어날 때 minor fault와 함께 할당된다. 이 방식을 **demand paging**이라고 한다.

지연 할당은 메모리 효율을 높이지만, 응답 시간이 중요한 서비스에서는 골치 아픈 패턴을 만든다. 워밍업이 끝났다고 생각한 직후 첫 요청이 들어왔을 때 폴트가 몰려서 지연이 튀는 경우가 있다. 트레이딩 시스템, 실시간 게임 서버, 저지연 캐시에서 자주 겪는다.

미리 페이지를 채워두려면 mmap에 `MAP_POPULATE` 플래그를 준다. 매핑 시점에 모든 페이지가 prefault되어 첫 접근의 폴트 비용이 사라진다.

```c
// 일반 mmap: demand paging
char *lazy = mmap(NULL, size, PROT_READ | PROT_WRITE,
                  MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
// 이 시점 RSS 거의 변화 없음
// lazy[0] 첫 접근 시 minor fault 발생

// MAP_POPULATE: 매핑 시 prefault
char *eager = mmap(NULL, size, PROT_READ | PROT_WRITE,
                   MAP_PRIVATE | MAP_ANONYMOUS | MAP_POPULATE, -1, 0);
// 이 시점 RSS가 size만큼 증가
// 매핑 호출이 오래 걸리지만 이후 접근은 폴트 없음
```

이미 매핑된 영역을 prefault하려면 `madvise(addr, len, MADV_WILLNEED)` 또는 `mlock(addr, len)`을 사용한다. `mlock`은 메모리를 락해서 스왑 아웃까지 막는다. 락 없이 단순히 미리 채우기만 하려면 `madvise` 또는 영역을 한 번 순회하며 더미 쓰기를 하는 방식을 쓴다.

`MAP_POPULATE`의 비용도 무시할 수 없다. 100GB 영역을 prefault하면 매핑 호출에서 모든 페이지가 폴트되며 수 초가 걸린다. 시작 시간을 희생해서 런타임 지연을 안정화하는 트레이드오프다.

## mmap을 이용한 파일 매핑

`mmap`은 파일을 프로세스의 가상 주소 공간에 직접 매핑하는 시스템 콜이다. 매핑된 영역에 메모리 접근하면 커널이 해당 파일의 내용을 페이지 단위로 물리 메모리에 로드한다. `read`/`write` 시스템 콜과 달리 커널 버퍼에서 유저 버퍼로 복사하는 과정이 없다.

```
  ┌─────────────────────────────────┐
  │   프로세스 가상 주소 공간        │
  │                                 │
  │  ┌──────────┐                   │
  │  │ 코드     │                   │
  │  ├──────────┤                   │
  │  │ 힙       │                   │
  │  ├──────────┤                   │
  │  │ mmap     │◄── 파일 매핑 영역 │
  │  │ 영역     │                   │
  │  ├──────────┤                   │
  │  │ 스택     │                   │
  │  └──────────┘                   │
  └──────────┬──────────────────────┘
             │ 페이지 폴트 시
             v
  ┌──────────────────┐
  │  페이지 캐시     │ ← 물리 메모리
  │  (Page Cache)    │
  └────────┬─────────┘
           │ 캐시 미스 시
           v
  ┌──────────────────┐
  │  디스크 파일     │
  └──────────────────┘
```

### mmap 사용법과 동작

```c
#include <sys/mman.h>
#include <fcntl.h>
#include <unistd.h>
#include <string.h>
#include <stdio.h>

int main() {
    int fd = open("data.bin", O_RDWR);
    off_t size = lseek(fd, 0, SEEK_END);

    // 파일을 가상 주소 공간에 매핑
    char *mapped = mmap(NULL, size, PROT_READ | PROT_WRITE,
                        MAP_SHARED, fd, 0);
    close(fd);  // 매핑 후 fd는 닫아도 된다

    // 메모리처럼 직접 읽기/쓰기
    printf("첫 4바이트: %.4s\n", mapped);
    memcpy(mapped + 100, "hello", 5);  // 파일에 직접 쓰기

    // 변경 내용을 디스크에 동기화
    msync(mapped, size, MS_SYNC);

    munmap(mapped, size);
    return 0;
}
```

`mmap`의 플래그에 따라 동작이 달라진다:

| 플래그 | 의미 |
|--------|------|
| `MAP_SHARED` | 변경이 파일에 반영되고, 다른 프로세스와 공유된다 |
| `MAP_PRIVATE` | Copy-on-Write. 수정하면 프로세스 전용 복사본이 생긴다 |
| `MAP_ANONYMOUS` | 파일 없이 익명 메모리 매핑. `malloc`의 대량 할당이 내부적으로 이걸 쓴다 |
| `MAP_HUGETLB` | Huge Page로 매핑한다 |

### mmap을 쓰는 실제 사례

**DB 엔진의 데이터 파일 접근**: SQLite의 WAL 모드, MongoDB의 MMAPv1(현재 deprecated), LMDB 같은 DB가 mmap으로 데이터 파일을 접근한다. 인덱스 파일처럼 랜덤 접근이 잦은 데이터에 적합하다.

**로그 파일 분석**: 수 GB짜리 로그 파일을 mmap으로 매핑하면 전체를 메모리에 올리지 않고도 문자열 검색이 가능하다. 접근하는 부분만 페이지 단위로 로드된다.

**프로세스 간 공유 메모리**: `MAP_SHARED`와 `MAP_ANONYMOUS`를 조합하면 파일 없이 프로세스 간 공유 메모리를 만들 수 있다. `shm_open` + `mmap` 패턴을 많이 쓴다.

### mmap 사용 시 주의할 점

**`msync` 호출 시점**: `MAP_SHARED`로 매핑한 파일에 쓴 내용은 `msync`를 호출하거나 `munmap`할 때 디스크에 반영된다. 그 사이에 프로세스가 죽으면 데이터가 날아갈 수 있다. 데이터 무결성이 중요하면 쓰기 후 바로 `msync(addr, len, MS_SYNC)`를 호출해야 한다.

**SIGBUS 시그널**: 매핑된 파일의 크기가 줄어들면(다른 프로세스가 truncate한 경우), 매핑 범위를 벗어난 접근에서 SIGBUS가 발생한다. 파일 크기 변경이 있을 수 있는 환경에서는 SIGBUS 핸들러를 설치하거나, 파일 락을 걸어야 한다.

**32비트 시스템 제약**: 32비트에서는 가상 주소 공간이 4GB(유저 영역 3GB)로 제한되므로, 큰 파일을 mmap으로 매핑하기 어렵다. 64비트에서는 이 제약이 사실상 없다.

**페이지 캐시와의 관계**: mmap으로 읽은 데이터는 페이지 캐시에 올라간다. 같은 파일을 `read` 시스템 콜로 읽어도 같은 페이지 캐시를 사용한다. 다만 mmap은 커널→유저 버퍼 복사가 없으므로 대용량 파일의 랜덤 읽기에서 유리하다. 순차 읽기는 `read` + `readahead`가 나은 경우가 있다.

### madvise로 커널에 접근 패턴 힌트 제공

`madvise`는 특정 가상 메모리 영역의 사용 패턴을 커널에 알려주는 시스템 콜이다. 커널은 이 힌트를 기반으로 readahead, 페이지 회수, Huge Page 승격 등을 조정한다. 힌트일 뿐 강제는 아니지만, 워크로드 특성을 알릴 수 있는 거의 유일한 채널이라 실무에서 쓰임새가 많다.

자주 쓰는 advice 값:

| advice | 동작 | 사용 사례 |
|--------|------|-----------|
| `MADV_DONTNEED` | 즉시 페이지 회수. 익명 매핑은 다음 접근 시 0으로 채워짐 | GC가 빈 region을 OS에 반납, 메모리 풀의 큰 청크 해제 |
| `MADV_WILLNEED` | 곧 접근할 예정. 백그라운드로 prefault | 응답 직전 인덱스 영역을 미리 채울 때 |
| `MADV_HUGEPAGE` | 이 영역을 Transparent Huge Page로 승격 요청 | 대량 버퍼를 쓰는 영역에만 선택적으로 THP 적용 |
| `MADV_NOHUGEPAGE` | THP 적용 금지 | fork 기반 백업이 있는 Redis 등에서 쓰기 증폭 방지 |
| `MADV_SEQUENTIAL` | 순차 접근 예정. readahead 강화, 읽은 페이지 빨리 회수 | 로그 파일 순차 스캔 |
| `MADV_RANDOM` | 랜덤 접근. readahead 비활성화 | DB 인덱스 페이지 랜덤 룩업 |
| `MADV_DONTDUMP` | 코어 덤프에서 제외 | 비밀키, 토큰이 든 메모리 보호 |

**MADV_DONTNEED 실전 사용**:

```c
// 큰 버퍼를 다 쓴 뒤, 메모리는 유지하지 않고 가상 영역만 남기고 싶을 때
char *buf = mmap(NULL, 1L << 30, PROT_READ | PROT_WRITE,
                 MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);

// ... buf 사용 ...

// 물리 페이지 즉시 반납 (RSS 감소). 가상 매핑은 유지됨
madvise(buf, 1L << 30, MADV_DONTNEED);

// 다음 접근 시 새 페이지가 0으로 채워져 다시 할당됨
buf[0] = 'x';  // minor fault, 새 zero page 할당
```

JVM의 G1 GC가 빈 region을 OS에 돌려줄 때 내부적으로 이 패턴을 쓴다. `free()`는 보통 메모리를 OS에 반환하지 않지만, `MADV_DONTNEED`는 즉시 반환한다.

**MADV_HUGEPAGE 선택 적용**:

전역 THP를 `always`로 두면 의도치 않은 영역까지 2MB 페이지로 묶여서 Copy-on-Write 비용이나 메모리 단편화 문제가 생긴다. `madvise` 모드로 두고, 큰 버퍼를 쓰는 영역만 명시적으로 요청하는 패턴이 안전하다.

```bash
# 시스템 전역: madvise 모드로 설정
echo madvise | sudo tee /sys/kernel/mm/transparent_hugepage/enabled
```

```c
// 큰 버퍼만 THP 후보로 등록
char *big = mmap(NULL, 4L << 30, PROT_READ | PROT_WRITE,
                 MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
madvise(big, 4L << 30, MADV_HUGEPAGE);
```

`MADV_DONTDUMP`는 코어 덤프 분석 도중 비밀 정보가 노출되는 사고를 막는다. JWT 서명 키, DB 비밀번호, AES 키 같은 민감한 메모리에 적용해두면 코어 덤프에서 해당 영역이 빠진다.

## Copy-on-Write 동작 검증

`fork()`는 자식 프로세스를 만들 때 부모의 가상 주소 공간을 그대로 복사하지만, 실제 물리 페이지는 즉시 복사하지 않는다. 부모와 자식이 같은 물리 페이지를 읽기 전용으로 공유하다가, 어느 한쪽이 쓰기를 하려는 순간 폴트가 발생하면서 그 페이지만 복사된다. **Copy-on-Write(CoW)** 라고 한다.

CoW 덕분에 100MB짜리 부모 프로세스를 fork해도 자식의 RSS가 즉시 100MB 추가로 잡히지 않는다. 자식이 실제로 수정한 페이지만큼만 메모리가 늘어난다. `getrusage`의 `ru_maxrss`로 RSS 추이를 추적해보면 동작을 확인할 수 있다.

```c
#include <sys/mman.h>
#include <sys/wait.h>
#include <sys/resource.h>
#include <stdio.h>
#include <string.h>
#include <unistd.h>
#include <stdlib.h>

#define MB (1024UL * 1024UL)
#define PAGE 4096UL

static void print_rss(const char *tag) {
    struct rusage ru;
    getrusage(RUSAGE_SELF, &ru);
    // Linux의 ru_maxrss는 KB 단위. macOS는 byte 단위
    printf("[%-25s] RSS: %ld KB\n", tag, ru.ru_maxrss);
}

int main(void) {
    size_t size = 100 * MB;
    char *buf = mmap(NULL, size, PROT_READ | PROT_WRITE,
                     MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
    if (buf == MAP_FAILED) { perror("mmap"); exit(1); }

    print_rss("매핑 직후");

    // 부모가 모든 페이지를 채워 물리 메모리 확보
    memset(buf, 0xAA, size);
    print_rss("부모 - 100MB 채움");

    pid_t pid = fork();
    if (pid == 0) {
        // 자식: fork 직후엔 부모와 페이지 공유 (CoW)
        print_rss("자식 - fork 직후");

        // 자식이 1MB만 수정 → 해당 페이지만 자식 전용으로 복사됨
        for (size_t i = 0; i < 1 * MB; i += PAGE) {
            buf[i] = 0xBB;
        }
        print_rss("자식 - 1MB 수정 후");

        // 자식이 전체를 수정 → 모든 페이지가 자식 전용 사본으로 복사됨
        memset(buf, 0xCC, size);
        print_rss("자식 - 전체 수정 후");
        _exit(0);
    }
    waitpid(pid, NULL, 0);
    return 0;
}
```

실행 결과의 일반적인 패턴:

```
[매핑 직후              ] RSS:   2048 KB    ← 매핑만, 페이지 미할당
[부모 - 100MB 채움      ] RSS: 104400 KB    ← 100MB 폴트로 채워짐
[자식 - fork 직후       ] RSS: 104400 KB    ← 부모와 공유, 새 메모리 없음
[자식 - 1MB 수정 후     ] RSS: 105424 KB    ← 1MB만 자식 전용으로 복제
[자식 - 전체 수정 후    ] RSS: 207000 KB    ← 100MB 전체가 자식 전용으로 복제
```

`ru_maxrss`는 최대값을 추적하므로 감소를 보지 못한다는 점에 주의해야 한다. 시점별 RSS 변화를 정확히 보려면 `/proc/self/statm` 또는 `/proc/self/status`의 `VmRSS`를 직접 읽는 게 낫다.

CoW가 실무에서 문제를 일으키는 대표적인 사례는 Redis의 `bgsave`다. Redis는 fork로 자식을 만들어 자식이 RDB 파일을 쓰는 동안 부모는 계속 클라이언트 요청을 처리한다. 부모가 키를 활발히 갱신하면 갱신된 페이지가 부모 쪽에서 복제되어, 메모리 사용량이 부모 데이터셋의 1배에서 최악의 경우 2배까지 늘어난다. Transparent Huge Page가 켜져 있으면 페이지 단위가 4KB가 아닌 2MB라 단 한 바이트 수정에도 2MB가 복제된다. Redis가 THP를 끄라고 권하는 이유가 이거다.

## NUMA 환경에서의 메모리 접근

### NUMA (Non-Uniform Memory Access)란

멀티소켓 서버에서 각 CPU 소켓이 자신의 로컬 메모리를 가지는 구조다. CPU가 자기 소켓에 붙은 메모리(로컬)에 접근하는 것이 다른 소켓의 메모리(리모트)에 접근하는 것보다 빠르다.

```
  ┌─────────────────────┐     QPI/UPI     ┌─────────────────────┐
  │     NUMA Node 0     │◄──────────────►│     NUMA Node 1     │
  │                     │   인터커넥트    │                     │
  │  ┌───────────────┐  │                 │  ┌───────────────┐  │
  │  │   CPU 소켓 0  │  │                 │  │   CPU 소켓 1  │  │
  │  │  Core 0..7    │  │                 │  │  Core 8..15   │  │
  │  └───────┬───────┘  │                 │  └───────┬───────┘  │
  │          │           │                 │          │           │
  │  ┌───────┴───────┐  │                 │  ┌───────┴───────┐  │
  │  │ 로컬 메모리   │  │                 │  │ 로컬 메모리   │  │
  │  │ 64GB DDR4     │  │                 │  │ 64GB DDR4     │  │
  │  │ ~80ns 접근    │  │                 │  │ ~80ns 접근    │  │
  │  └───────────────┘  │                 │  └───────────────┘  │
  │                     │                 │                     │
  │  Node 0→Node 1     │                 │  Node 1→Node 0     │
  │  리모트: ~140ns    │                 │  리모트: ~140ns    │
  └─────────────────────┘                 └─────────────────────┘
```

로컬 메모리 접근 대비 리모트 메모리 접근은 보통 1.5~2배 느리다. 단일 스레드에서는 체감하기 어렵지만, 멀티스레드 워크로드에서 NUMA를 고려하지 않으면 성능이 크게 떨어지는 경우가 있다.

### NUMA와 가상 메모리의 관계

가상 메모리의 페이지 할당 시 어느 NUMA 노드의 물리 메모리를 사용할지가 성능에 영향을 준다. Linux 커널은 기본적으로 "first-touch" 정책을 사용한다. 페이지에 처음 접근하는 CPU가 속한 NUMA 노드에서 메모리를 할당한다.

```bash
# NUMA 노드 구성 확인
numactl --hardware
# available: 2 nodes (0-1)
# node 0 cpus: 0 1 2 3 4 5 6 7
# node 0 size: 65536 MB
# node 1 cpus: 8 9 10 11 12 13 14 15
# node 1 size: 65536 MB
# node distances:
# node   0   1
#   0:  10  21    ← 리모트 접근이 2.1배 느리다
#   1:  21  10

# 프로세스를 특정 NUMA 노드에 바인딩
numactl --cpunodebind=0 --membind=0 ./my_server

# 프로세스별 NUMA 메모리 사용 현황
numastat -p <pid>
# Node 0: 2048 MB
# Node 1: 15 MB     ← 대부분 로컬 노드에서 할당됨 (정상)

# NUMA 밸런싱 상태 확인
cat /proc/sys/kernel/numa_balancing
# 1 (활성화)
```

### 실무에서 NUMA 문제가 나타나는 경우

**JVM 기반 서비스**: JVM이 힙을 초기화할 때 메인 스레드가 모든 메모리를 할당하면, 전체 힙이 하나의 NUMA 노드에 몰린다. GC 스레드가 다른 노드에서 돌면 리모트 접근이 발생한다. `-XX:+UseNUMA` 옵션으로 NUMA 인지 GC를 활성화할 수 있다.

**Redis**: 싱글 스레드 기반이므로 하나의 NUMA 노드에 바인딩하는 게 유리하다. `numactl --cpunodebind=0 --membind=0 redis-server`로 실행한다.

**MySQL/PostgreSQL**: 커넥션별로 스레드/프로세스를 사용하므로, 버퍼 풀이 여러 NUMA 노드에 걸쳐 할당될 수 있다. `innodb_numa_interleave=1`(MySQL) 옵션으로 인터리브 할당을 하면 최악의 경우를 피할 수 있다. 모든 접근이 균등하게 약간 느려지는 대신, 특정 노드에 쏠려서 급격히 느려지는 상황을 막는다.

```bash
# 인터리브 모드: 모든 NUMA 노드에 라운드 로빈으로 할당
numactl --interleave=all ./mysql_server

# NUMA 메모리 할당 통계 확인
cat /proc/vmstat | grep numa
# numa_hit 12345678       ← 로컬 노드에서 할당 성공
# numa_miss 234            ← 로컬 노드에 메모리 부족으로 리모트 할당
# numa_foreign 234         ← 다른 노드의 요청이 이 노드에서 할당됨
```

`numa_miss`가 `numa_hit` 대비 높은 비율이면 NUMA 바인딩을 검토해야 한다.

## 스왑과의 관계

물리 메모리가 부족하면 운영체제는 사용하지 않는 페이지를 디스크의 스왑 영역으로 옮긴다. 스왑은 가상 메모리 시스템이 물리 메모리보다 큰 주소 공간을 제공하기 위한 구체적인 수단이다.

스왑의 동작 방식, 페이지 교체 알고리즘, OS별 설정 방법은 [Swap Memory](Swap Memory.md)에서 다룬다.

## 메모리 오버커밋(Overcommit) 정책

Linux는 기본적으로 가상 메모리를 **물리 메모리 + 스왑**보다 많이 약속할 수 있다. `malloc(10GB)`이 8GB RAM 서버에서도 성공한다는 뜻이다. 실제로 그 가상 영역을 다 만지면 OOM이 나지만, 만지지 않는 한 문제가 없다. 대부분의 프로그램이 받은 메모리를 다 쓰지 않는다는 통계적 가정 위에서 동작한다.

이 동작은 `vm.overcommit_memory` 커널 파라미터로 제어한다.

| 값 | 모드 | 동작 |
|---|------|------|
| 0 | heuristic (기본값) | 휴리스틱하게 판단. 명백히 무리한 단일 요청만 거부 |
| 1 | always overcommit | 무조건 허용. 어떤 요청도 NULL을 돌려주지 않음 |
| 2 | never overcommit | `CommitLimit`까지만 허용. 초과 요청은 즉시 실패 |

`vm.overcommit_memory=2`일 때 적용되는 한도는 다음과 같다.

```
CommitLimit = (RAM × vm.overcommit_ratio / 100) + Swap
```

`vm.overcommit_ratio`는 기본 50이다. RAM 64GB, 스왑 8GB, ratio 50 환경이면 `CommitLimit = 64 × 0.5 + 8 = 40GB`. 이 한도까지의 가상 메모리만 약속할 수 있다.

```bash
# 현재 정책과 한도 확인
cat /proc/sys/vm/overcommit_memory       # 0 (기본)
cat /proc/sys/vm/overcommit_ratio        # 50
grep -E "Commit(Limit|ted)" /proc/meminfo
# CommitLimit:   41943040 kB    ← 약속 가능한 한도
# Committed_AS:  18234567 kB    ← 현재 약속된 양

# 정책을 영구 적용하려면 sysctl
echo "vm.overcommit_memory = 1" | sudo tee -a /etc/sysctl.conf
echo "vm.overcommit_ratio = 80" | sudo tee -a /etc/sysctl.conf
sudo sysctl -p
```

**언제 어느 정책을 쓰나**

`vm.overcommit_memory=1` (always)이 권장되는 워크로드는 fork로 큰 프로세스를 복제하는 경우다. Redis가 대표적이다. 32GB짜리 Redis가 `bgsave`를 위해 fork하면 휴리스틱 모드에서는 잠재적으로 32GB가 추가로 약속되는 것처럼 보여 fork가 실패할 수 있다. 실제로는 CoW로 거의 메모리가 복제되지 않는데도 그렇다. Redis 공식 문서가 `vm.overcommit_memory=1`을 강력히 권하는 이유다.

`vm.overcommit_memory=2` (never)는 OOM이 절대 일어나면 안 되는 환경에서 쓴다. 임베디드 시스템, 결제/거래 처리 서버 등이다. malloc이 명시적으로 실패해서 애플리케이션이 처리할 수 있게 하는 것이 목표다. 단점은 sparse 매핑(예: JVM이 큰 힙을 reserve만 해두는 경우)을 그대로 약속으로 잡아서 한도를 빨리 소진한다는 점이다. 64bit JVM의 `-Xmx32g`가 실제 사용량과 상관없이 32GB를 commit으로 카운트하는 식이다. ratio를 80~100까지 올려 한도를 넓히는 게 일반적이다.

기본값 0은 데스크톱과 일반 서버에 무난하다. 일부러 바꿀 이유가 없으면 그대로 둔다.

```bash
# 현재 약속된 메모리가 한도 대비 얼마인지 빠르게 확인
awk '/CommitLimit:/{l=$2} /Committed_AS:/{c=$2}
     END{printf "%.1f%% of CommitLimit\n", c/l*100}' /proc/meminfo
# 65.3% of CommitLimit    ← 80% 넘으면 모드 2에서 위험 신호
```

## 장점과 단점

### 장점

**프로세스 격리**: 각 프로세스가 독립된 주소 공간을 가지므로 다른 프로세스의 메모리를 직접 건드릴 수 없다. 버그가 있는 프로세스가 다른 프로세스를 망가뜨리지 않는다.

**물리 메모리 이상 사용 가능**: 프로그램이 실제 사용하는 부분만 물리 메모리에 올리고, 나머지는 디스크에 둔다.

**메모리 공유**: 여러 프로세스가 같은 라이브러리를 로드하면 물리 메모리에는 한 카피만 올라간다. fork 시에도 Copy-on-Write로 실제 수정이 발생할 때까지 메모리를 복사하지 않는다.

### 단점

**주소 변환 오버헤드**: 매번 가상→물리 주소 변환이 필요하다. TLB로 대부분 커버하지만, TLB 미스가 나면 다단계 페이지 테이블을 타고 들어가야 한다.

**페이지 폴트 비용**: Major page fault가 발생하면 디스크 I/O를 기다려야 한다. 메모리 접근이 나노초 단위인데, 디스크 I/O는 밀리초 단위다.

**페이지 테이블 메모리 사용**: 페이지 테이블 자체가 메모리를 차지한다. 프로세스가 많으면 무시할 수 없는 양이 된다.

### Linux

```bash
# 프로세스별 가상 메모리 매핑 확인
cat /proc/<pid>/maps

# 출력 예시:
# 00400000-0040b000 r-xp  /usr/bin/cat     ← 코드 영역 (실행 권한)
# 0060a000-0060b000 r--p  /usr/bin/cat     ← 읽기 전용 데이터
# 0060b000-0060c000 rw-p  /usr/bin/cat     ← 읽기/쓰기 데이터
# 7f8a1000-7f8c2000 r-xp  /lib/libc.so.6  ← 공유 라이브러리
# 7ffd5000-7ffd6000 rw-p  [stack]          ← 스택

# 더 상세한 정보 (RSS, PSS, Shared 등)
cat /proc/<pid>/smaps_rollup

# 페이지 테이블이 차지하는 메모리 확인
grep PageTables /proc/meminfo
```

### VSZ / RSS / PSS / USS 차이

프로세스의 메모리 사용량은 보는 지표에 따라 값이 크게 다르다. 어떤 지표를 봐야 하는지가 디버깅의 출발점이다.

| 지표 | 의미 | 공유 페이지 처리 |
|------|------|------------------|
| VSZ (Virtual Size) | 가상 주소 공간 전체 크기 | 무관. 매핑된 영역의 합 |
| RSS (Resident Set Size) | 실제 물리 메모리에 올라간 페이지의 합 | 공유 페이지를 **중복 카운트** |
| PSS (Proportional Set Size) | 공유 페이지를 공유한 프로세스 수로 나눠 분배 | 비례 분배 |
| USS (Unique Set Size) | 해당 프로세스만 쓰는 페이지 | 공유 페이지 제외 |

**예시**: `libc.so` 2MB가 100개 프로세스에 매핑되어 있다고 하자.

- 각 프로세스의 RSS에는 2MB 전부가 잡힌다 → 100개 합치면 200MB로 보이지만 실제 물리 메모리는 2MB
- PSS에는 2MB / 100 = 20KB만 잡힌다 → 100개 합치면 2MB로 정확
- USS에는 0KB (공유 페이지라서 아무도 단독으로 쓰지 않음)

운영 서버에서 흔한 착각이 RSS의 합으로 메모리 사용량을 추산하는 것이다. 공유 라이브러리, fork된 워커 프로세스의 CoW 페이지 등이 모두 중복 카운트되어 실제보다 부풀려 보인다. 정확한 측정은 PSS의 합을 봐야 한다.

```bash
# 한 프로세스의 RSS/PSS/USS 한 번에 보기
cat /proc/<pid>/smaps_rollup
# Rss:              524288 kB
# Pss:              318204 kB    ← 실제 분담분
# Pss_Anon:         298100 kB    ← 익명 페이지 (단독)
# Pss_File:          18204 kB    ← 파일 매핑 (공유 라이브러리 등)
# Pss_Shmem:          1900 kB
# Shared_Clean:      45000 kB
# Shared_Dirty:          0 kB
# Private_Clean:      4000 kB
# Private_Dirty:    475288 kB    ← USS 본체
```

USS는 직접 필드가 없고 `Private_Clean + Private_Dirty`로 계산한다. 한 줄로 보려면 `smem` 도구가 편하다.

```bash
# smem으로 모든 프로세스의 USS/PSS/RSS 비교
sudo smem -k
# PID  User  Command         Swap    USS    PSS    RSS
# 1234 root  /usr/bin/redis    0   320M   325M   332M
# 5678 mysql /usr/sbin/mysqld  0   1.2G   1.4G   1.8G
# 9012 www   nginx: worker     0    24M    52M   178M    ← 공유가 많음
```

nginx 워커처럼 부모 fork로 만들어진 프로세스는 RSS와 USS의 차이가 크다. 이 차이가 곧 부모/형제와 공유 중인 메모리다.

**fork로 검증해보기**:

```bash
# 1. Redis가 32GB 사용 중일 때
ps -o pid,vsz,rss,cmd -p <redis_pid>
# PID    VSZ      RSS    CMD
# 1234  33000000  32500000  redis-server

# 2. bgsave 도중 자식과 부모 둘 다 측정
ps -o pid,vsz,rss,cmd -C redis-server
# 1234  33000000  32500000  redis-server      (parent)
# 5678  33000000  32400000  redis-server (bgsave)  (child) ← RSS 합치면 64GB?

# 실제 물리 메모리는 PSS로 봐야 정확
sudo cat /proc/1234/smaps_rollup | grep ^Pss:
# Pss:   16500000 kB    ← 부모 분담
sudo cat /proc/5678/smaps_rollup | grep ^Pss:
# Pss:   16400000 kB    ← 자식 분담
# 합: ~32GB. 거의 모든 페이지를 CoW로 공유 중인 상태
```

부모와 자식이 같은 페이지를 공유하면 PSS가 절반씩 분배되어 합산이 실제 물리 사용량과 맞아떨어진다.

### Huge Page 설정

대량의 메모리를 쓰는 애플리케이션(JVM, DB 등)에서 TLB 미스를 줄이기 위해 Huge Page를 쓸 수 있다.

```bash
# Transparent Huge Pages(THP) 상태 확인
cat /sys/kernel/mm/transparent_hugepage/enabled
# [always] madvise never

# 명시적 Huge Page 할당 (2MB * 1024 = 2GB)
echo 1024 | sudo tee /proc/sys/vm/nr_hugepages

# 할당된 Huge Page 확인
grep Huge /proc/meminfo
# HugePages_Total:    1024
# HugePages_Free:     1024
# Hugepagesize:       2048 kB
```

주의할 점: Redis나 MongoDB 같은 DB를 쓸 때 THP가 성능 문제를 일으키는 경우가 있다. fork 기반 백그라운드 저장 시 Copy-on-Write가 2MB 단위로 발생해서 메모리 사용량이 급증한다. 이런 DB들은 대부분 THP를 끄라고 권장한다.

```bash
# THP 비활성화 (Redis, MongoDB 등에서 권장)
echo never | sudo tee /sys/kernel/mm/transparent_hugepage/enabled
```

### 페이지 폴트 측정과 분석

페이지 폴트는 두 종류의 비용 차이가 극명하다. minor fault는 마이크로초 단위, major fault는 디스크 I/O가 끼어 밀리초 단위다. 어느 쪽이 얼마나 발생하는지 정확히 구분해서 봐야 진단이 가능하다.

**프로세스 시점 누적 폴트 카운트**:

```bash
# /proc/<pid>/stat의 10번째 필드가 minflt, 12번째가 majflt (누적)
cat /proc/<pid>/stat | awk '{
    print "Minor faults (no I/O):", $10
    print "Major faults (I/O 발생):", $12
}'

# ps로도 동일 값 확인 가능
ps -o pid,minflt,majflt,cmd -p <pid>
#   PID  MINFLT MAJFLT CMD
#  1234  152300     12 redis-server
```

**프로그램 단위 측정**:

```bash
# /usr/bin/time -v는 종료 시점에 폴트 합계를 출력
/usr/bin/time -v python3 my_script.py 2>&1 | grep -E "page fault|resident"
#   Major (requiring I/O) page faults: 3
#   Minor (reclaiming a frame) page faults: 1523
#   Maximum resident set size (kbytes): 65432
```

major fault가 수십~수백 단위면 정상이다. 수천 이상이면 워킹셋이 메모리에 안 들어가서 스왑/디스크 I/O가 일어나고 있다는 신호다.

**perf로 실시간 측정**:

`perf stat`은 minor/major fault를 별도 이벤트로 카운트한다.

```bash
# 워크로드 실행 중 폴트 발생률 확인
sudo perf stat -e minor-faults,major-faults -p <pid> -- sleep 10
#  Performance counter stats for process id '1234':
#       234,567      minor-faults
#            3       major-faults
#      10.001 seconds time elapsed

# 시스템 전역 측정
sudo perf stat -e minor-faults,major-faults -a sleep 5
```

**major fault의 호출 스택 추적**:

major fault가 어디서 발생하는지 알면 원인을 좁힐 수 있다. 라이브러리 로딩 단계인지, 특정 데이터 구조 접근인지 등을 구분 가능하다.

```bash
# major fault 이벤트 발생 시 스택 샘플링
sudo perf record -e major-faults -g -p <pid> -- sleep 30
sudo perf report

# bcc-tools의 trace나 bpftrace로도 추적 가능
sudo bpftrace -e 'tracepoint:exceptions:page_fault_user
                   /pid == 1234/ {
                     @[ustack] = count();
                   }'
```

**TLB 미스 측정**:

페이지 폴트와 별개로 TLB 미스도 성능에 큰 영향을 준다. dTLB(데이터)와 iTLB(명령어)를 따로 본다.

```bash
sudo perf stat -e dTLB-load-misses,dTLB-loads,iTLB-load-misses,iTLB-loads ./my_program
# dTLB-load-misses: 1,234       (0.01% of all dTLB loads)  ← 정상
# dTLB-load-misses: 500,000     (5.2% of all dTLB loads)   ← 문제 있음
```

미스율이 1%를 넘으면 의심해볼 만하다. 5%를 넘으면 Huge Page 적용이나 메모리 접근 패턴 개선을 검토해야 한다.

**vmstat로 시스템 전역 폴트 흐름**:

```bash
# 1초 간격으로 마이너/메이저 폴트 비율 관찰
vmstat 1
# r b swpd  free  buff  cache  si  so  bi   bo   in   cs  us sy id wa
# 0 0    0 12345  234  56789   0   0   0    0  234  456   2  1 97  0
#                                ↑   ↑
#                                스왑 인/아웃이 0이면 major fault 거의 없음
```

`si`(swap-in)와 `so`(swap-out)가 0이 아니면 스왑이 동작 중이고, major fault가 발생할 가능성이 높다.

### OOM Killer

메모리가 완전히 부족하면 Linux 커널이 OOM Killer를 발동해서 프로세스를 강제 종료한다.

```bash
# OOM Killer 발동 로그 확인
dmesg | grep -i "out of memory"
journalctl -k | grep -i oom

# 프로세스의 OOM 점수 확인 (높을수록 종료 대상)
cat /proc/<pid>/oom_score       # 0~1000
cat /proc/<pid>/oom_score_adj   # 조정값 (-1000 ~ 1000)

# 중요 프로세스를 OOM Killer로부터 보호
echo -1000 | sudo tee /proc/<pid>/oom_score_adj
```

## 컨테이너 환경에서의 가상 메모리

컨테이너(Docker, Kubernetes)는 호스트 OS의 커널을 공유하므로 가상 메모리 시스템도 호스트와 동일하다. VM과 달리 별도의 가상 메모리 계층이 없다.

다만 cgroup으로 메모리 사용량을 제한할 수 있다. 컨테이너가 설정된 메모리 한도를 넘으면 cgroup의 OOM Killer가 동작한다. 호스트의 OOM Killer와는 별개다.

```bash
# 컨테이너 메모리 제한 설정
docker run -m 512m --memory-swap 1g my-app

# -m 512m: 물리 메모리 제한 512MB
# --memory-swap 1g: 메모리+스왑 합계 제한 1GB (스왑 512MB)

# Kubernetes에서 메모리 제한
# resources:
#   requests:
#     memory: "256Mi"
#   limits:
#     memory: "512Mi"    ← 이 값을 넘으면 OOM Kill
```

실무에서 자주 겪는 문제: 컨테이너 안에서 `free`나 `/proc/meminfo`를 보면 호스트의 전체 메모리가 보인다. 컨테이너의 실제 메모리 한도를 확인하려면 cgroup 파일을 봐야 한다.

```bash
# 컨테이너 안에서 실제 메모리 한도 확인
cat /sys/fs/cgroup/memory/memory.limit_in_bytes     # cgroup v1
cat /sys/fs/cgroup/memory.max                        # cgroup v2
```

JVM처럼 시작 시 사용 가능 메모리를 기반으로 힙 크기를 정하는 런타임은, 컨테이너 메모리 한도를 인식하지 못하면 호스트의 전체 메모리를 보고 힙을 크게 잡아서 OOM Kill을 당한다. Java 10 이후에는 `-XX:+UseContainerSupport`(기본 활성화)로 cgroup 한도를 인식한다.
