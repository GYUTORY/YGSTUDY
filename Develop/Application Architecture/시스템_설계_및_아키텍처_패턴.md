# 시스템 설계 및 아키텍처 패턴 – 실무 기준 정리

> 프로덕션 환경을 전제로 한 시스템 설계 의사결정 기준을 정리한 문서다.
> *왜 이 선택을 하는가*, *언제 문제가 되는가*, *운영에서 무엇을 본다*를 중심으로 작성했다.

---

## 1. 시스템 설계의 본질

시스템 설계의 목적은 단순히 “동작하는 시스템”이 아니라 다음을 만족하는 구조를 만드는 것입니다.

- 예측 가능한 성능
- 장애 시 통제 가능한 영향 범위
- 트래픽 증가 시 구조 변경 없이 대응 가능
- 인력 교체·확장에도 유지 가능한 복잡도

> **좋은 설계는 코드보다 오래 남고, 나쁜 설계는 사람을 태운다.**

---

## 2. 핵심 설계 원칙 (Production Grade)

### 2.1 단순성 우선 원칙

- 복잡성은 *필요해질 때* 추가합니다.
- 초기부터 마이크로서비스, 서비스 메시를 도입하는 것은 대부분 과잉 설계입니다.

**권장 진화 경로**
```
Monolith
 → Modular Monolith
 → Service Separation
 → Microservices (필요 시)
```

---

### 2.2 확장성(Scalability)의 실체

확장성은 “서버를 늘릴 수 있다”가 아닙니다.

| 항목 | 진짜 의미 |
|---|---|
| Scale Out | 상태를 외부로 밀어냈는가 |
| Scale Up | 장애 시 복구 전략이 있는가 |
| Auto Scaling | Scale-In 시 데이터 손실이 없는가 |

**실무 결론**
- App Server: 수평 확장
- Cache / MQ: 수평 확장
- DB Write Node: 제한적 수직 확장 + 샤딩 준비

---

## 3. CAP 정리의 실무적 해석

> **현실의 분산 시스템은 항상 P를 가진다.**

따라서 선택지는 두 가지뿐입니다.

| 선택 | 의미 |
|---|---|
| CP | 정확하지만 멈출 수 있음 |
| AP | 계속 동작하지만 틀릴 수 있음 |

### 3.1 CP를 선택해야 하는 영역
- 결제
- 재고
- 예약
- 권한/인증

### 3.2 AP를 선택해야 하는 영역
- 로그
- 알림
- 피드
- 통계

> 하나의 시스템 안에서도 **도메인별로 CAP 선택은 달라져야 합니다.**

## 4. 로드 밸런싱 설계 기준

### 4.1 L4 vs L7 선택 기준

| 기준 | L4 | L7 |
|---|---|---|
| 성능 | 매우 우수 | 상대적으로 느림 |
| 라우팅 | IP/Port | Header/Path |
| 운영 난이도 | 낮음 | 높음 |
| 마이크로서비스 | 부적합 | 적합 |

**현실적인 구조**
```
[ L4 (NLB) ]
      ↓
[ L7 (ALB / Nginx) ]
      ↓
[ App ]
```

L4를 앞에 두는 이유는 TLS 종단이나 경로 기반 라우팅 전에 TCP 레벨에서 트래픽을 분산하기 위해서다. L7만 단독으로 두면 TLS handshake 부하가 LB 한 대에 집중되는 경우가 있다.

### 4.2 Health Check 설정에서 자주 틀리는 부분

Health Check 주기를 너무 짧게 잡으면 LB가 정상 노드를 비정상으로 판단하는 일이 생긴다. GC가 긴 JVM 애플리케이션에서 500ms interval로 설정했다가 Full GC 구간에서 노드가 빠지고 트래픽이 쏠리면서 연쇄 장애가 난 사례가 있다.

Health Check endpoint가 DB 연결까지 확인하는 deep check인 경우, DB 지연이 곧 LB 탈락으로 이어진다. `/health`는 프로세스 생존 여부만 확인하는 shallow check로 두고, `/health/ready`를 별도로 만들어서 트래픽 수신 가능 여부를 판단하는 방식이 안전하다.

### 4.3 운영에서 겪는 문제

**Connection Draining 미설정**: 배포 시 기존 연결을 끊어버리면 진행 중인 요청이 실패한다. ALB 기준으로 deregistration delay를 최소 30초 이상 설정해야 한다. 긴 요청(파일 업로드 등)이 있는 서비스는 더 늘려야 한다.

**Sticky Session 남용**: 세션을 서버 메모리에 저장하고 Sticky Session으로 해결하려는 경우가 많다. 해당 노드가 죽으면 세션이 날아가고, 특정 노드에 트래픽이 편중되면서 스케일 아웃 효과가 사라진다. 세션은 Redis 같은 외부 저장소로 빼는 게 맞다.

**알고리즘 선택**: Round Robin은 요청 처리 시간이 균일할 때만 의미가 있다. API 응답 시간 편차가 큰 서비스에서는 Least Connection이 낫고, 캐시 히트율을 높여야 하면 Consistent Hashing을 쓴다. 기본값을 그대로 두지 말고 서비스 특성에 맞춰 선택한다.

## 6. 캐싱 설계의 실제 기준

### 6.1 캐시 계층 구조

```
L1: In-Memory (Process) — 가장 빠르지만 프로세스 재시작 시 소멸
L2: Redis / Memcached — 네트워크 비용 있지만 공유 가능
L3: CDN — 정적 리소스, 변경 빈도 낮은 API 응답
L4: DB Buffer Pool — DB 엔진이 자체 관리
```

L1과 L2를 같이 쓰는 경우, L1에 있는 데이터가 L2에서 무효화됐는데 L1에서는 계속 오래된 값을 내려주는 문제가 생긴다. L2 무효화 시 Pub/Sub으로 L1도 같이 날리는 구조를 만들어야 하고, 이게 없으면 차라리 L1을 안 쓰는 게 낫다.

### 6.2 캐시 장애 사례

**Cache Stampede (Thundering Herd)**: 인기 상품 페이지의 캐시 TTL이 동시에 만료되면, 수백 개 요청이 동시에 DB를 때린다. 실제로 이커머스 서비스에서 타임세일 시작 시점에 상품 캐시가 만료되면서 DB CPU가 100%로 치솟고 전체 서비스가 멈춘 경우가 있다. 대응법은 두 가지다.

```java
// 1. 뮤텍스 패턴: 한 요청만 DB 조회, 나머지는 대기
String value = redis.get(key);
if (value == null) {
    if (redis.setnx(lockKey, "1", Duration.ofSeconds(5))) {
        value = db.query(id);
        redis.set(key, value, Duration.ofMinutes(10));
        redis.del(lockKey);
    } else {
        // 짧게 대기 후 재시도
        Thread.sleep(50);
        return getFromCache(key);
    }
}
```

```java
// 2. TTL 분산: 기본 TTL에 랜덤 오프셋 추가
int baseTtl = 600; // 10분
int jitter = ThreadLocalRandom.current().nextInt(0, 120); // 0~2분
redis.set(key, value, Duration.ofSeconds(baseTtl + jitter));
```

**Cache Penetration**: 존재하지 않는 키를 반복 조회하면 매번 DB까지 간다. 악의적인 요청이 아니더라도, 삭제된 데이터를 조회하는 패턴에서 발생한다. Bloom Filter를 앞에 두거나, null 결과도 짧은 TTL로 캐싱한다.

**Write-Behind 유실**: 캐시에만 쓰고 비동기로 DB에 반영하는 패턴에서, 캐시 노드가 죽으면 아직 DB에 반영 안 된 데이터가 날아간다. 조회수 같은 손실 허용 데이터에만 써야 하고, 금액이나 재고에는 절대 쓰면 안 된다.

### 6.3 무효화 설계

캐시에서 가장 어려운 부분은 "언제 지울 것인가"다.

**이벤트 기반 무효화**: 데이터가 변경되는 시점에 캐시를 삭제한다. 변경 경로가 하나일 때는 문제없지만, 어드민 페이지, 배치, API 등 변경 경로가 여러 개면 하나라도 빠뜨리면 오래된 데이터가 남는다. 모든 변경이 하나의 이벤트 버스를 통하도록 구조를 잡아야 한다.

**TTL 기반 무효화**: 설정이 간단하지만, TTL 동안은 오래된 데이터가 노출될 수 있다. 상품 가격처럼 즉시 반영돼야 하는 데이터에는 TTL만으로는 부족하다.

실무에서는 이벤트 기반 삭제를 기본으로 하고, TTL은 이벤트 누락 시 안전망으로 건다. TTL을 30분이나 1시간 정도로 넉넉하게 잡으면 이벤트가 정상 동작하는 한 TTL 만료가 실질적으로 발생하지 않는다.

## 7. 데이터베이스 샤딩 판단 기준

### 7.1 샤딩이 필요한 시점

Read 부하는 Read Replica로 분산할 수 있다. 샤딩이 필요해지는 건 Write 병목이 생겼을 때다.

- 단일 테이블 Row 수가 수천만을 넘기면서 인덱스 크기가 RAM을 초과한다. 이 시점부터 쿼리 성능이 급격히 떨어진다.
- Write Lock 경합이 늘면서 INSERT 지연이 발생한다. 특히 AUTO_INCREMENT PK를 쓰는 테이블에서 동시 INSERT가 많으면 경합이 심해진다.
- ALTER TABLE이 수 시간 걸려서 스키마 변경 자체가 불가능해진다.

그런데 이 증상이 나타났다고 바로 샤딩하면 안 된다. 파티셔닝, 아카이빙(오래된 데이터 분리), 쿼리 최적화를 먼저 시도한다. 이걸로 해결되면 샤딩의 운영 복잡도를 떠안을 이유가 없다.

### 7.2 샤딩 설계 시 판단 기준

**샤드 키 선택이 전부다.** 잘못된 샤드 키를 고르면 특정 샤드에 데이터가 몰리고(핫 샤드), 나중에 키를 바꾸려면 전체 데이터를 마이그레이션해야 한다.

이커머스에서 `seller_id`를 샤드 키로 잡았는데, 대형 셀러 한 명이 전체 주문의 30%를 차지하면 그 샤드만 터진다. 이런 경우 `order_id`의 해시를 샤드 키로 쓰는 게 분포가 균일하다. 대신 특정 셀러의 주문 목록 조회 시 모든 샤드를 뒤져야 하는 트레이드오프가 생긴다.

**Cross-Shard 쿼리는 비싸다.** Join이 필요한 테이블은 같은 샤드에 있어야 한다. 사용자와 주문이 다른 샤드에 있으면, “이 사용자의 주문 목록”이라는 단순한 쿼리조차 scatter-gather 패턴으로 풀어야 한다. 응답 시간은 가장 느린 샤드에 맞춰진다.

**리밸런싱 계획 없이 시작하지 않는다.** 샤드 수를 고정하면 데이터가 늘어났을 때 다시 쪼개야 한다. Consistent Hashing을 쓰면 샤드 추가 시 전체 데이터가 아니라 인접 샤드의 일부만 이동하므로 영향을 줄일 수 있다. Vitess나 ProxySQL 같은 미들웨어를 쓰면 애플리케이션 레벨에서 샤딩 로직을 분리할 수 있지만, 미들웨어 자체의 운영 부담이 생긴다.

> **샤딩은 성능 문제가 아니라 조직 문제다.** 샤드별 백업, 복구, 스키마 변경, 모니터링을 각각 관리할 인력이 있는지 먼저 확인한다.

## 8. 데이터 복제(Replication)

DB 쓰기 노드가 하나면 그 노드가 죽는 순간 서비스가 멈춘다. 복제는 이 단일 장애점을 없애고, 읽기 부하를 분산하기 위해 쓴다.

### 8.1 Master-Slave (Primary-Replica)

가장 흔한 구조다. 쓰기는 Master 한 대가 받고, Slave는 Master의 변경 로그(binlog, WAL 등)를 받아서 자기 데이터에 반영한다. 읽기 요청은 Slave에서 처리한다.

```
Client (Write) → Master
Client (Read)  → Slave 1 / Slave 2 / Slave 3
                     ↑       ↑       ↑
              Master binlog 복제
```

**복제 지연(Replication Lag)이 문제의 핵심이다.** Master에 쓴 직후 Slave에서 읽으면 아직 반영이 안 된 데이터를 읽는다. 사용자가 글을 쓰고 새로고침했는데 자기 글이 안 보이는 현상이 대표적이다.

대응 방법은 세 가지다.

- **Read-your-writes 패턴**: 자기가 쓴 데이터를 읽을 때는 Master에서 읽는다. 세션에 마지막 쓰기 타임스탬프를 기록하고, 해당 시점 이전의 데이터만 Slave에서 읽는다.
- **동기 복제**: Slave가 반영을 확인할 때까지 Master가 커밋 응답을 보류한다. 정합성은 보장되지만 Slave 장애가 Master 쓰기 성능을 직접 때린다. MySQL의 semi-synchronous replication은 Slave 중 하나라도 ack하면 커밋하는 절충안이다.
- **쿼리 라우팅**: 정합성이 필요한 쿼리(결제 직후 잔액 조회 등)는 Master로, 나머지는 Slave로 보낸다. ProxySQL이나 애플리케이션 레벨에서 분기한다.

**Slave 승격(Failover) 시 주의할 점**: Master가 죽으면 Slave 중 하나를 Master로 올려야 한다. 이때 가장 최신 상태의 Slave를 골라야 하는데, 복제 지연 때문에 각 Slave의 반영 시점이 다르다. 가장 최신 binlog 위치를 가진 Slave를 찾고, 나머지 Slave가 새 Master를 바라보도록 재설정한다. 이 과정이 자동화되어 있지 않으면 복구에 수십 분이 걸린다. MHA(MySQL), Patroni(PostgreSQL) 같은 자동 페일오버 도구를 쓴다.

### 8.2 Multi-Master

여러 노드가 동시에 쓰기를 받는다. 지리적으로 분산된 서비스에서 각 리전에 쓰기 가능한 노드를 두고 싶을 때 검토한다.

```
Region A: Master A ←→ Master B :Region B
          (쌍방향 복제)
```

**쓰기 충돌이 가장 큰 문제다.** 같은 row를 A와 B에서 동시에 수정하면 둘 중 어느 값이 맞는지 시스템이 판단해야 한다. 충돌 해소 방법은 몇 가지가 있다.

| 방법 | 동작 | 한계 |
|---|---|---|
| Last Write Wins (LWW) | 타임스탬프가 늦은 쪽이 이긴다 | 시계 동기화에 의존, 데이터 유실 가능 |
| 애플리케이션 레벨 해소 | 충돌을 기록해두고 앱에서 병합 | 구현이 복잡하고, 모든 충돌 케이스를 커버하기 어려움 |
| CRDT | 데이터 타입 자체가 충돌 없이 병합 가능 | 적용 가능한 데이터 구조가 제한적 |

경험상 Multi-Master를 써야 하는 상황은 드물다. 대부분은 리전별 Master-Slave + 글로벌 라우팅으로 해결된다. Multi-Master는 "두 리전에서 같은 데이터를 동시에 써야 하고, 리전 간 지연이 커서 단일 Master로는 쓰기 지연을 감당할 수 없는" 경우에만 검토한다. 그게 아니면 운영 복잡도만 올라간다.

### 8.3 복제 토폴로지 선택 기준

| 요구사항 | 권장 구조 |
|---|---|
| 읽기 분산, 단순한 운영 | Master-Slave |
| 쓰기 가용성, 리전 분산 | Multi-Master (충돌 해소 필수) |
| Master 장애 시 자동 복구 | Master-Slave + 자동 페일오버 |
| 강한 정합성 | 동기 복제 or 합의 기반(다음 섹션) |

## 9. 합의 알고리즘(Consensus)

분산 시스템에서 "이 값이 맞다"를 여러 노드가 동의하는 문제를 합의라고 한다. Leader Election, 설정값 관리, 분산 로그 복제의 기반이 되는 알고리즘이다.

### 9.1 왜 합의가 필요한가

Master-Slave에서 Master가 죽었을 때, "누가 새 Master인가"를 남은 노드들이 합의해야 한다. 합의 없이 각자 판단하면 두 노드가 동시에 자신이 Master라고 주장하는 Split Brain이 발생한다. 두 Master에 서로 다른 쓰기가 들어오면 데이터가 갈라진다.

### 9.2 Paxos

Leslie Lamport가 1989년에 제안한 합의 알고리즘이다. 정확성은 수학적으로 증명되어 있지만, 이해하고 구현하기가 어렵기로 유명하다. Google의 Chubby, Spanner가 Paxos 기반이다.

**동작 구조 (Single-Decree Paxos)**

하나의 값에 대해 합의하는 과정이다. 역할은 Proposer, Acceptor, Learner 세 가지다.

```
Phase 1: Prepare
  Proposer → Acceptor: "제안 번호 N으로 값을 제안하겠다"
  Acceptor → Proposer: "OK. 이전에 수락한 값이 있으면 알려준다"

Phase 2: Accept
  Proposer → Acceptor: "제안 번호 N으로 값 V를 수락해달라"
  Acceptor: 더 높은 번호의 제안을 받은 적 없으면 수락

합의 성립: 과반수(Quorum) Acceptor가 같은 값을 수락
```

**실무에서 Paxos를 직접 구현하는 경우는 거의 없다.** Multi-Paxos(연속적인 값에 대한 합의)로 확장하면 복잡도가 급격히 올라가고, 논문마다 구현 세부사항이 다르다. 대부분은 Paxos를 직접 쓰지 않고 Raft를 쓰거나, Paxos 기반으로 만들어진 ZooKeeper 같은 제품을 쓴다.

### 9.3 Raft

Paxos의 난해함을 해결하기 위해 2014년에 나온 알고리즘이다. "이해하기 쉬운 합의 알고리즘"을 목표로 설계됐고, 실제로 구현체가 많다. etcd, Consul, CockroachDB, TiKV가 Raft 기반이다.

**핵심 개념**

노드는 Leader, Follower, Candidate 세 가지 상태를 가진다. 정상 상태에서는 Leader 하나가 모든 쓰기를 받고, Follower에 로그를 복제한다.

```
정상 상태:
Client → Leader → Follower 1 (로그 복제)
                → Follower 2 (로그 복제)
                → Follower 3 (로그 복제)

과반수 복제 확인 → Client에 커밋 응답
```

**Leader Election 과정**

1. Follower가 Leader의 heartbeat를 일정 시간(election timeout) 동안 못 받으면 Leader가 죽었다고 판단한다.
2. 해당 Follower가 Candidate로 전환되고, 자기 자신에게 투표한 뒤 다른 노드에 투표를 요청한다.
3. 과반수 투표를 받으면 새 Leader가 된다.
4. election timeout에 랜덤 값을 넣어서 여러 노드가 동시에 Candidate가 되는 상황(split vote)을 줄인다.

```
시간 흐름:

T=0   Leader heartbeat 정상
T=150 Leader 장애 발생
T=300 Follower 2의 election timeout 만료 (timeout: 150~300ms 랜덤)
T=300 Follower 2 → Candidate, 투표 요청
T=310 Follower 1, 3 투표 응답
T=310 Follower 2 → 새 Leader (3/5 과반)
T=320 새 Leader가 heartbeat 시작
```

**Log Replication**

Leader가 받은 쓰기 요청은 로그 엔트리로 만들어져 Follower에 복제된다. 각 엔트리에는 term(선거 기간 번호)과 index가 있다. Leader는 과반수 Follower가 해당 엔트리를 저장했음을 확인한 뒤에야 커밋으로 처리한다. 커밋된 엔트리는 절대 덮어쓰이지 않는다.

Follower의 로그가 Leader와 다르면(이전 Leader의 미완성 엔트리가 남아있는 경우), Leader가 Follower의 로그를 자기 것과 일치하도록 덮어쓴다. 이 과정이 자동이라서 노드 복구 후 별도 작업이 필요 없다.

**실무에서 Raft를 쓸 때 주의할 점**

- **노드 수는 홀수로**: 3대면 1대 장애 허용, 5대면 2대 장애 허용이다. 짝수로 하면 비용 대비 장애 허용 수가 같다. 3대와 4대 모두 1대 장애 허용이므로 4대는 낭비다.
- **네트워크 파티션**: Leader가 소수 파티션에 갇히면 다수 파티션에서 새 Leader가 선출된다. 소수 파티션의 이전 Leader는 쓰기를 받아도 커밋할 수 없다(과반수를 채울 수 없으므로). 파티션이 복구되면 이전 Leader는 자동으로 Follower로 내려간다.
- **디스크 성능이 중요하다**: 모든 로그 엔트리는 디스크에 fsync해야 한다. 디스크가 느리면 복제 지연이 생기고, election timeout 안에 heartbeat를 처리 못 하면 불필요한 Leader 교체가 반복된다. etcd 운영 시 SSD는 필수다.

### 9.4 Paxos vs Raft 비교

| 항목 | Paxos | Raft |
|---|---|---|
| 정확성 | 수학적 증명 완료 | 수학적 증명 완료 |
| 이해 난이도 | 매우 높음 | 상대적으로 낮음 |
| 구현체 | Chubby, Spanner (비공개) | etcd, Consul, CockroachDB |
| Leader 필수 여부 | 아님 (Leaderless 가능) | 필수 |
| 실무 채택 | 대형 인프라 내부 | 오픈소스 생태계에서 주류 |

새로 분산 시스템을 설계한다면 Raft 기반 구현체를 쓰는 게 현실적이다. Paxos를 직접 구현할 이유가 있는 경우는 기존 Paxos 기반 시스템을 유지보수하거나, Leaderless 합의가 필요한 매우 특수한 경우뿐이다.

## 10. Leader Election

분산 시스템에서 하나의 노드가 특정 역할(쓰기 담당, 작업 조율 등)을 맡아야 할 때, 어떤 노드가 그 역할을 할지 결정하는 과정이다. Raft의 Leader Election은 합의 알고리즘 내부의 메커니즘이고, 여기서는 애플리케이션 레벨에서 Leader를 뽑는 방법을 다룬다.

### 10.1 왜 필요한가

- 배치 작업을 여러 인스턴스 중 한 대에서만 실행해야 할 때
- 분산 스케줄러에서 작업 분배를 한 노드가 담당해야 할 때
- Master-Slave 구조에서 Master 장애 시 새 Master를 선출할 때

Leader 없이 모든 노드가 독립적으로 동작하면 좋겠지만, "정확히 한 번만 실행"이 필요한 작업에서는 조율자가 있어야 한다.

### 10.2 구현 방법

**ZooKeeper Ephemeral Node 방식**

가장 고전적인 방법이다. ZooKeeper에 ephemeral(임시) 노드를 생성하고, 생성에 성공한 노드가 Leader가 된다. Leader가 죽으면 ZooKeeper 세션이 끊기면서 ephemeral 노드가 삭제되고, 나머지 노드가 이를 감지해서 다시 경쟁한다.

```
Node 1: create /election/leader (성공 → Leader)
Node 2: create /election/leader (실패 → watch 설정)
Node 3: create /election/leader (실패 → watch 설정)

Node 1 장애:
  → ephemeral 노드 자동 삭제
  → Node 2, 3에 watch 이벤트 발생
  → Node 2, 3 다시 create 경쟁
```

주의할 점은 ZooKeeper 세션 타임아웃이다. Leader가 GC pause나 네트워크 지연으로 잠시 ZooKeeper와 연결이 끊기면, 아직 살아있는데 Leader 자격을 잃는다. 이때 새 Leader가 선출되면서 두 노드가 동시에 Leader처럼 동작하는 순간이 생긴다. 이를 방지하려면 Leader가 작업을 시작하기 전에 자기가 아직 Leader인지 다시 확인하는 fencing 로직이 필요하다.

**etcd Lease 방식**

etcd의 Lease(임대)를 이용한다. 일정 시간 동안 유효한 Lease를 발급받고, 해당 Lease에 키를 연결한다. Lease를 주기적으로 갱신(keepalive)하지 않으면 키가 삭제된다.

```go
// etcd Leader Election (Go 예제)
lease, _ := client.Grant(ctx, 10) // 10초 Lease
// Lease에 연결된 키를 생성 (Leader 선출)
txn := client.Txn(ctx).
    If(clientv3.Compare(clientv3.CreateRevision("/leader"), "=", 0)).
    Then(clientv3.OpPut("/leader", nodeID, clientv3.WithLease(lease.ID))).
    Else(clientv3.OpGet("/leader"))
resp, _ := txn.Commit()

if resp.Succeeded {
    // Leader가 됨, keepalive 시작
    ch, _ := client.KeepAlive(ctx, lease.ID)
    // ch에서 주기적으로 응답을 받으며 Lease 유지
}
```

**Redis SETNX 방식**

간단하지만 정합성 보장이 약하다. `SETNX`로 키를 잡고, TTL을 설정해서 Leader가 죽으면 자동 만료되게 한다.

```python
# Redis Leader Election
acquired = redis.set("leader", node_id, nx=True, ex=30)
if acquired:
    # Leader로 동작, 주기적으로 TTL 갱신
    while is_running:
        redis.expire("leader", 30)
        do_leader_work()
        time.sleep(10)
```

Redis는 AP 시스템이라서 네트워크 파티션 시 두 노드가 동시에 Leader가 될 수 있다. 배치 중복 실행 정도는 감수할 수 있는 경우에만 쓴다. 결제나 재고처럼 정확히 한 번 실행이 필요하면 etcd나 ZooKeeper를 쓴다.

### 10.3 Leader Election에서 자주 겪는 문제

**Thundering Herd**: Leader가 죽었을 때 모든 노드가 동시에 선출을 시도하면 부하가 생긴다. 각 노드에 랜덤 지연을 넣어서 경합을 줄인다.

**Split Brain**: 네트워크 파티션으로 양쪽에서 각각 Leader를 뽑으면 두 Leader가 공존한다. Quorum 기반 선출(과반수 동의 필요)로 방지한다. 3대 중 2대가 동의해야 Leader가 되므로, 파티션된 소수 쪽에서는 Leader가 나올 수 없다.

**Fencing Token**: Leader가 바뀌는 순간 이전 Leader의 진행 중인 작업이 완료되면서 새 Leader의 작업과 충돌할 수 있다. Leader가 바뀔 때마다 단조 증가하는 토큰(fencing token)을 발급하고, 외부 자원(DB 등)에 접근할 때 이 토큰을 같이 보낸다. 외부 자원은 이전 토큰의 요청을 거부한다.

## 11. Back-of-Envelope Estimation (트래픽 산정)

시스템을 설계하기 전에 대략적인 규모를 추산하는 과정이다. 정확한 수치가 아니라 자릿수(order of magnitude)를 맞추는 게 목적이다. "초당 100건인지 10만 건인지"에 따라 아키텍처가 완전히 달라진다.

### 11.1 기본 수치 (외워두면 좋다)

```
지연 시간:
  L1 cache reference              0.5 ns
  L2 cache reference                7 ns
  RAM reference                   100 ns
  SSD random read             150,000 ns (150 μs)
  HDD random read          10,000,000 ns (10 ms)
  같은 데이터센터 네트워크 왕복   500,000 ns (0.5 ms)
  다른 리전 네트워크 왕복     150,000,000 ns (150 ms)

저장 용량 (2의 거듭제곱):
  1 KB = 1,024 bytes
  1 MB = 1,024 KB ≈ 10^6 bytes
  1 GB = 1,024 MB ≈ 10^9 bytes
  1 TB = 1,024 GB ≈ 10^12 bytes

시간 변환:
  1일 ≈ 86,400초 ≈ 10^5초 (계산 편의상)
  1달 ≈ 2.6 × 10^6초 ≈ 2.5 × 10^6초
  1년 ≈ 3.15 × 10^7초 ≈ 3 × 10^7초
```

### 11.2 트래픽 산정 방법

DAU(일간 활성 사용자) → 초당 요청 수(QPS) → 피크 QPS 순서로 계산한다.

**예시: SNS 타임라인 서비스**

```
가정:
  DAU = 1,000만 명
  1인당 하루 평균 타임라인 조회 = 20회

일간 요청 수:
  10,000,000 × 20 = 200,000,000 (2억)

평균 QPS:
  200,000,000 / 86,400 ≈ 2,300 QPS

피크 QPS (평균의 3~5배로 잡는다):
  2,300 × 3 ≈ 7,000 QPS
```

7,000 QPS면 단일 서버로는 무리고, 로드밸런서 뒤에 여러 대를 붙여야 한다. 하지만 수십만 QPS는 아니므로 샤딩까지는 필요 없을 수 있다.

### 11.3 저장 용량 산정

```
예시: 채팅 서비스

가정:
  DAU = 500만 명
  1인당 하루 평균 메시지 = 40건
  메시지 평균 크기 = 200 bytes (텍스트 기준)

일간 데이터:
  5,000,000 × 40 × 200 = 40,000,000,000 bytes = 40 GB/일

연간 데이터:
  40 GB × 365 ≈ 14.6 TB/년

5년 보관:
  14.6 TB × 5 ≈ 73 TB
```

73TB면 단일 DB로는 안 된다. 샤딩이 필요하고, 오래된 데이터는 cold storage로 옮기는 정책이 필요하다.

### 11.4 대역폭 산정

```
예시: 이미지 공유 서비스

가정:
  평균 이미지 크기 = 500 KB
  초당 업로드 = 100건
  초당 다운로드 = 5,000건

업로드 대역폭:
  100 × 500 KB = 50 MB/s ≈ 400 Mbps

다운로드 대역폭:
  5,000 × 500 KB = 2,500 MB/s ≈ 20 Gbps
```

20 Gbps는 CDN 없이는 감당할 수 없다. 이 수치 하나로 "CDN은 선택이 아니라 필수"라는 결론이 나온다.

### 11.5 산정할 때 자주 하는 실수

**피크를 빼먹는다**: 평균 QPS로 서버를 산정하면 피크에 터진다. 서비스 특성에 따라 피크 배수가 다르다. 이커머스 타임세일은 평균의 10배 이상 나올 수 있고, 일반 SNS는 3~5배 정도다.

**메타데이터를 빼먹는다**: 메시지 본문 200 bytes라고 해도, 인덱스·타임스탬프·발신자 정보·복제본까지 합치면 실제 저장 공간은 3~5배다. 계산 후에 여유 계수를 곱한다.

**읽기/쓰기 비율을 무시한다**: 대부분의 서비스는 읽기가 쓰기보다 훨씬 많다. SNS는 보통 100:1이고, 채팅은 1:1에 가깝다. 이 비율에 따라 캐시 필요 여부, Read Replica 수가 결정된다.

**자릿수만 맞추면 된다**: 2,314 QPS와 2,500 QPS의 차이는 의미 없다. 2,000인지 20,000인지가 중요하다. 계산 중간에 반올림을 적극적으로 해서 빠르게 결론에 도달한다.

## 12. 실전 아키텍처 사례

### 12.1 SNS 타임라인

AP를 선택한다. 타임라인에 팔로워의 최신 글이 1~2초 늦게 보여도 사용자는 인지하지 못하지만, 서비스가 멈추면 바로 이탈한다.

핵심 결정은 **Fan-out 시점**이다. Fan-out on Write는 글 작성 시 모든 팔로워의 타임라인에 미리 넣어두는 방식이다. 읽기가 빠르지만, 팔로워가 100만 명인 유명인이 글을 쓰면 100만 건의 쓰기가 발생한다. Fan-out on Read는 타임라인 조회 시점에 팔로잉 목록을 보고 실시간으로 합치는 방식이다. 쓰기 부하는 없지만 읽기가 느리다.

실무에서는 혼합 방식을 쓴다. 팔로워가 일정 수(예: 1만 명) 이하인 일반 사용자는 Fan-out on Write, 그 이상인 셀럽 계정은 Fan-out on Read로 처리한다. Twitter(현 X)가 이 방식을 쓰고, 이 경계값을 어디로 잡느냐가 인프라 비용에 직접 영향을 준다.

타임라인 데이터는 Redis Sorted Set에 넣는 경우가 많다. score를 타임스탬프로 잡으면 범위 조회가 간단하다. 오래된 타임라인 데이터는 주기적으로 잘라내고, 스크롤이 깊어지면 DB에서 가져오는 구조로 비용을 줄인다.

### 12.2 결제 시스템

CP를 선택한다. 결제 금액이 틀리거나 중복 결제가 발생하면 실제 금전 손해로 이어진다.

**멱등성 보장이 핵심이다.** 네트워크 타임아웃으로 클라이언트가 재시도하면 같은 결제가 두 번 발생할 수 있다. 모든 결제 요청에 클라이언트가 생성한 idempotency key를 붙이고, 서버에서 이 키의 중복 여부를 확인한다. PG사 API 호출 전에 이 키를 DB에 INSERT하고, UNIQUE 제약으로 중복을 막는다.

**부분 실패 처리**: 결제 승인은 됐는데 주문 상태 업데이트가 실패하는 경우가 있다. 이때 자동 취소를 시도하되, 취소마저 실패하면 "결제 승인됨 + 주문 미확정" 상태로 남긴다. 이 불일치 상태를 별도 테이블에 기록하고, 배치로 주기적으로 PG사 상태와 대조해서 정리한다. 실시간으로 모든 걸 맞추려고 하면 복잡도만 올라가고 장애 범위가 넓어진다.

**장애 시 Fail Fast**: 결제 응답이 3초 이상 걸리면 타임아웃 처리하고 사용자에게 재시도를 안내한다. 느린 응답을 기다리면서 커넥션을 물고 있으면 전체 시스템이 함께 멈춘다. Circuit Breaker를 걸어서 PG사 장애 시 빠르게 차단한다.

### 12.3 IoT / 실시간 데이터 수집

Event Driven 구조를 쓴다. 수만 대 디바이스가 동시에 데이터를 보내므로 동기 API로는 감당이 안 된다.

**MQ 선택 기준**: Kafka는 순서 보장과 재처리가 필요한 경우에 쓴다. 센서 데이터를 시계열로 저장해야 하면 파티션 키를 디바이스 ID로 잡아서 같은 디바이스의 데이터가 순서대로 처리되게 한다. 단순 이벤트 알림이면 SQS나 RabbitMQ가 운영 부담이 적다.

**백프레셔 처리**: 디바이스가 갑자기 몰려서 MQ consumer가 처리 속도를 못 따라가면, 큐가 쌓이면서 메모리가 터지거나 디스크가 찬다. consumer lag을 모니터링하고, 임계치를 넘으면 consumer 수를 늘리는 오토스케일링을 걸어둔다. Kafka는 파티션 수가 consumer 수의 상한이므로, 파티션을 충분히 확보해둬야 한다.

**재처리 가능 구조**: consumer에서 처리 실패한 이벤트를 Dead Letter Queue에 넣고, 원인 파악 후 재투입한다. 이때 consumer 로직이 멱등해야 같은 이벤트를 두 번 처리해도 결과가 달라지지 않는다. 디바이스 데이터에 시퀀스 번호를 붙이면 중복 판별이 쉬워진다.
