# 서비스 분해 및 도메인 설계

## 왜 서비스 경계가 중요한가

MSA로 전환할 때 첫 번째로 해야 할 일은 서비스 경계를 정하는 것이다. 코드를 분리하고, 배포 파이프라인을 나누고, 데이터베이스를 쪼개는 작업 전부가 이 경계 설정에 의존한다.

경계를 잘못 잡으면 어떻게 되는가. 서비스 A의 변경이 서비스 B, C, D를 연쇄적으로 수정해야 하는 상황이 반복된다. 배포를 독립적으로 하겠다고 MSA를 도입했는데, 결국 서비스 3~4개를 동시에 배포해야만 기능이 돌아가는 구조가 된다. 이게 분산 모놀리스다.

실제로 경계를 잘못 잡아서 서비스를 다시 합친 경험이 있다. 주문(Order)과 결제(Payment)를 처음부터 별도 서비스로 분리했는데, 실무에서 주문 생성과 결제 승인이 거의 항상 하나의 트랜잭션처럼 동작해야 했다. 결제 실패 시 주문 상태 롤백, 부분 결제 처리, 환불 시 주문 상태 변경 등 두 서비스가 서로의 상태에 깊이 의존하는 로직이 계속 늘어났다. API 호출 한 번에 서비스 간 통신이 4~5번 오가고, 네트워크 오류 한 번이면 데이터 정합성이 깨졌다. 결국 주문-결제를 하나의 서비스로 다시 합쳤다.

## Bounded Context 기반 서비스 분해

### Bounded Context란

DDD(Domain-Driven Design)에서 Bounded Context는 특정 도메인 모델이 일관성을 유지하는 범위를 뜻한다. 같은 용어라도 Context가 다르면 의미가 다르다.

예를 들어 "상품(Product)"이라는 용어가 있다.

- **상품 카탈로그 Context**: 상품명, 설명, 카테고리, 이미지 등 전시 정보
- **재고 관리 Context**: SKU, 수량, 입출고 이력
- **주문 Context**: 주문 시점의 상품명, 가격 스냅샷

세 곳에서 "상품"이라는 단어를 쓰지만 각각 다른 속성과 행위를 갖는다. 이걸 하나의 Product 엔티티로 만들면 필드가 40~50개짜리 거대 모델이 되고, 한 쪽의 변경이 나머지를 전부 건드린다.

### 서비스 분해 과정

실무에서 Bounded Context를 찾는 과정은 다음과 같다.

**1단계: 도메인 이벤트 나열**

비즈니스에서 발생하는 이벤트를 먼저 뽑아낸다. 이벤트 스토밍(Event Storming) 워크숍이 여기에 해당한다.

```
주문이 생성되었다
결제가 승인되었다
재고가 차감되었다
배송이 시작되었다
리뷰가 작성되었다
포인트가 적립되었다
```

**2단계: 이벤트를 도메인 그룹으로 묶기**

관련된 이벤트들을 묶으면 자연스럽게 도메인 영역이 드러난다.

```
[주문] 주문 생성, 주문 취소, 주문 상태 변경
[결제] 결제 승인, 결제 취소, 환불 처리
[재고] 재고 차감, 재고 복원, 입고 처리
[배송] 배송 시작, 배송 완료, 반품 수거
[회원] 가입, 탈퇴, 등급 변경, 포인트 적립
```

**3단계: 그룹 간 의존 관계 파악**

이벤트 그룹 사이에서 데이터가 얼마나 자주 오가는지 확인한다. 여기서 중요한 건 동기적으로 데이터를 주고받아야 하는지, 비동기 이벤트만으로 충분한지를 구분하는 것이다.

```
주문 → 결제: 주문 생성 시 결제 요청 (동기)
결제 → 주문: 결제 완료 이벤트 (비동기 가능)
주문 → 재고: 재고 확인 (동기), 재고 차감 (비동기 가능)
주문 → 배송: 배송 요청 (비동기)
```

동기 의존성이 높은 그룹은 하나의 서비스로 묶는 걸 고려해야 한다. 비동기로 처리 가능한 관계는 서비스를 분리해도 문제가 적다.

**4단계: 서비스 경계 확정**

```java
// 서비스 분리 결과 예시
// 주문-결제 서비스 (동기 의존성이 높아 하나로 묶음)
@Service
public class OrderPaymentService {
    
    public OrderResult createOrder(OrderRequest request) {
        // 재고 확인 - 동기 호출이 필요한 부분
        StockInfo stock = stockClient.checkStock(request.getItems());
        if (!stock.isAvailable()) {
            throw new OutOfStockException(stock.getUnavailableItems());
        }
        
        // 주문 생성과 결제 처리가 하나의 트랜잭션
        Order order = orderRepository.save(Order.create(request));
        PaymentResult payment = paymentProcessor.process(order);
        
        if (payment.isFailed()) {
            order.cancel("결제 실패");
            orderRepository.save(order);
            throw new PaymentFailedException(payment.getReason());
        }
        
        order.confirmPayment(payment.getTransactionId());
        orderRepository.save(order);
        
        // 재고 차감, 배송 요청은 비동기 이벤트로 처리
        eventPublisher.publish(new OrderConfirmedEvent(order));
        
        return OrderResult.from(order);
    }
}
```

```java
// 재고 서비스 - 비동기 이벤트 수신
@Service
public class StockEventHandler {
    
    @EventListener
    public void onOrderConfirmed(OrderConfirmedEvent event) {
        for (OrderItem item : event.getItems()) {
            stockRepository.deduct(item.getSkuId(), item.getQuantity());
        }
    }
    
    @EventListener
    public void onOrderCancelled(OrderCancelledEvent event) {
        for (OrderItem item : event.getItems()) {
            stockRepository.restore(item.getSkuId(), item.getQuantity());
        }
    }
}
```

## 분산 모놀리스: 경계를 잘못 나눈 결과

### 분산 모놀리스의 특징

서비스를 나누긴 했는데, 실질적으로 모놀리스와 다를 바 없는 상태를 분산 모놀리스라고 한다. 오히려 모놀리스보다 나쁜데, 네트워크 오버헤드와 운영 복잡도만 추가된 꼴이다.

분산 모놀리스에 빠졌다는 신호들:

- 기능 하나 배포하려면 서비스 2~3개를 동시에 배포해야 한다
- 서비스 A의 코드를 고치면 서비스 B의 API 스펙도 바꿔야 한다
- 서비스 간 공유 라이브러리가 있고, 버전을 맞추지 않으면 장애가 난다
- 로컬에서 개발할 때 다른 서비스 3~4개를 같이 띄워야 테스트가 된다
- 서비스 간 순환 참조가 있다 (A → B → C → A)

### 실무에서 흔한 잘못된 분리 패턴

**1. 기술 레이어 기반 분리**

```
# 이렇게 나누면 안 된다
API Gateway 서비스
비즈니스 로직 서비스
데이터 접근 서비스
```

기술 레이어로 나누면 기능 하나를 변경할 때 세 서비스를 전부 수정해야 한다. MSA는 비즈니스 기능 단위로 나눠야 의미가 있다.

**2. 데이터베이스를 공유하는 분리**

```
# 서비스는 나눴는데 DB는 하나
주문 서비스 ──┐
결제 서비스 ──┼── 공유 DB
배송 서비스 ──┘
```

DB를 공유하면 테이블 스키마를 변경할 때 모든 서비스에 영향이 간다. 서비스별 독립 배포가 불가능해진다. 각 서비스는 자기 데이터를 소유해야 한다.

**3. 너무 잘게 나눈 분리**

상품의 기본 정보, 가격, 이미지를 각각 별도 서비스로 나눈 경우가 있었다. 상품 상세 페이지 하나를 그리려면 세 서비스를 전부 호출해야 하고, 하나라도 응답이 느리면 전체가 느려진다. 함께 조회되고 함께 변경되는 데이터는 같은 서비스에 있어야 한다.

## 서비스 간 의존성 관리

### 의존 방향 규칙

서비스 간 의존성은 한 방향으로만 흘러야 한다. 순환 의존이 생기면 분리한 의미가 없다.

```
# 의존 방향 예시
상위 서비스 (BFF, Orchestrator)
    ↓
핵심 도메인 서비스 (주문, 결제)
    ↓
지원 도메인 서비스 (알림, 로그)
    ↓
공통 인프라 서비스 (인증, 설정)
```

상위 서비스가 하위 서비스를 호출하는 건 괜찮지만, 하위 서비스가 상위 서비스를 직접 호출하면 안 된다. 하위에서 상위로 정보를 전달해야 할 때는 이벤트를 쓴다.

### 의존성을 줄이는 방법

**1. 이벤트 기반 통신**

서비스 간 직접 호출 대신 이벤트를 발행하고 구독하는 방식이다.

```java
// 주문 서비스 - 이벤트 발행만 한다
// 누가 이 이벤트를 구독하는지 주문 서비스는 모른다
@Service
public class OrderService {
    
    private final EventPublisher eventPublisher;
    
    public void completeOrder(Long orderId) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));
        
        order.complete();
        orderRepository.save(order);
        
        // 주문 완료 이벤트 발행
        eventPublisher.publish(new OrderCompletedEvent(
            order.getId(),
            order.getUserId(),
            order.getTotalAmount(),
            order.getItems()
        ));
    }
}
```

```java
// 알림 서비스 - 이벤트를 구독해서 처리
@Component
public class NotificationEventListener {
    
    @KafkaListener(topics = "order-events")
    public void handleOrderCompleted(OrderCompletedEvent event) {
        notificationService.sendOrderConfirmation(
            event.getUserId(), 
            event.getOrderId()
        );
    }
}

// 포인트 서비스 - 같은 이벤트를 별도로 구독
@Component
public class PointEventListener {
    
    @KafkaListener(topics = "order-events")
    public void handleOrderCompleted(OrderCompletedEvent event) {
        pointService.accumulate(
            event.getUserId(), 
            event.getTotalAmount()
        );
    }
}
```

주문 서비스는 알림이나 포인트 서비스의 존재를 모른다. 나중에 쿠폰 서비스가 추가되어도 주문 서비스 코드를 건드릴 필요가 없다.

**2. 데이터 복제 (CQRS 읽기 모델)**

다른 서비스의 데이터가 자주 필요하면 이벤트를 통해 필요한 데이터를 복제해두는 방법이 있다.

```java
// 배송 서비스에서 회원 정보가 필요한 경우
// 매번 회원 서비스를 호출하는 대신, 필요한 정보만 복제해둔다

@Component
public class MemberInfoProjection {
    
    @KafkaListener(topics = "member-events")
    public void onMemberUpdated(MemberUpdatedEvent event) {
        // 배송에 필요한 정보만 저장
        DeliveryMemberInfo info = new DeliveryMemberInfo(
            event.getMemberId(),
            event.getName(),
            event.getPhone(),
            event.getDefaultAddress()
        );
        deliveryMemberRepository.save(info);
    }
}
```

이 방식은 데이터 일관성이 약간 느슨해지지만(Eventual Consistency), 서비스 간 런타임 의존성을 제거한다. 회원 서비스가 잠시 다운되어도 배송 서비스는 이미 복제된 데이터로 정상 동작한다.

**3. Anti-Corruption Layer**

외부 서비스나 레거시 시스템과 통신할 때, 상대방의 모델이 내 도메인 모델을 오염시키지 않도록 변환 계층을 둔다.

```java
// 외부 결제 PG사 응답을 내부 모델로 변환
@Component
public class PaymentGatewayAdapter {
    
    private final ExternalPGClient pgClient;
    
    public PaymentResult requestPayment(PaymentCommand command) {
        // 내부 모델 → 외부 모델 변환
        PGPaymentRequest pgRequest = PGPaymentRequest.builder()
            .merchantId(pgConfig.getMerchantId())
            .amt(command.getAmount().toString())  // PG사는 문자열로 받음
            .goodsNm(command.getProductName())     // PG사 필드명이 다름
            .build();
        
        PGPaymentResponse pgResponse = pgClient.pay(pgRequest);
        
        // 외부 모델 → 내부 모델 변환
        return PaymentResult.builder()
            .transactionId(pgResponse.getTid())
            .amount(new BigDecimal(pgResponse.getAmt()))
            .status(mapStatus(pgResponse.getResultCd()))
            .paidAt(parseDateTime(pgResponse.getAuthDt()))
            .build();
    }
    
    private PaymentStatus mapStatus(String pgStatusCode) {
        return switch (pgStatusCode) {
            case "0000" -> PaymentStatus.SUCCESS;
            case "3001" -> PaymentStatus.CARD_DECLINED;
            case "3002" -> PaymentStatus.INSUFFICIENT_BALANCE;
            default -> PaymentStatus.FAILED;
        };
    }
}
```

PG사의 필드명이나 코드 체계가 바뀌어도 Adapter만 수정하면 된다. 내부 도메인 모델은 영향을 받지 않는다.

## 서비스 경계를 다시 잡아야 할 때

### 합쳐야 하는 신호

- 두 서비스가 거의 항상 같이 배포된다
- 서비스 간 API 호출이 한 요청에 3회 이상 발생한다
- 하나의 비즈니스 로직이 두 서비스에 걸쳐 있다
- 데이터 정합성 문제가 반복적으로 발생한다

### 나눠야 하는 신호

- 하나의 서비스에 팀 3개 이상이 동시에 코드를 수정한다
- 서비스 내 특정 기능만 스케일 아웃이 필요한데 전체를 늘려야 한다
- 배포 주기가 기능별로 크게 다르다 (인증은 월 1회, 상품은 일 3회)
- 장애 영향 범위를 격리하고 싶은 부분이 명확하다

### 합치는 과정에서 주의할 점

서비스를 합칠 때 가장 어려운 부분은 데이터베이스 통합이다.

```sql
-- 주문 서비스 DB의 orders 테이블
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    status VARCHAR(20),
    total_amount DECIMAL(10, 2),
    created_at TIMESTAMP
);

-- 결제 서비스 DB의 payments 테이블
CREATE TABLE payments (
    id BIGINT PRIMARY KEY,
    order_id BIGINT,  -- 주문 서비스의 order id를 참조
    amount DECIMAL(10, 2),
    pg_transaction_id VARCHAR(100),
    status VARCHAR(20),
    paid_at TIMESTAMP
);

-- 합칠 때: 한쪽 DB로 마이그레이션
-- 1. 결제 테이블을 주문 DB로 복사
-- 2. order_id에 FK 제약 추가
-- 3. 양쪽 서비스가 새 DB를 바라보도록 전환
-- 4. 이전 DB 제거
```

한 번에 전환하면 위험하다. 듀얼 라이트(두 DB에 동시에 쓰기) 기간을 두고, 데이터가 일치하는지 검증한 후에 이전 DB를 제거하는 게 안전하다.

## 실무에서의 판단 기준 정리

서비스를 나눌지 말지 고민될 때 확인하는 것들:

**데이터 소유권이 명확한가**: 이 데이터를 누가 생성하고, 누가 수정하고, 누가 삭제하는지 확인한다. 소유권이 하나의 서비스에 명확히 있으면 분리해도 된다. 여러 서비스가 같은 데이터를 직접 수정해야 하면 경계가 잘못된 것이다.

**팀 구조와 일치하는가**: 콘웨이 법칙에 따라 서비스 경계는 팀 경계와 일치해야 한다. 한 팀이 여러 서비스를 관리하거나, 한 서비스를 여러 팀이 관리하면 문제가 생긴다.

**독립 배포가 가능한가**: 서비스를 나눈 후에 각 서비스를 독립적으로 배포할 수 있는지 확인한다. 배포할 때마다 다른 서비스와 조율이 필요하면 분리의 의미가 없다.

**장애 격리가 되는가**: 서비스 A가 죽었을 때 서비스 B가 정상 동작하는지 확인한다. A가 죽으면 B도 같이 죽는 구조면 분리할 이유가 약하다.

처음부터 서비스를 잘게 나누는 것보다, 모놀리스로 시작해서 경계가 명확해지는 시점에 분리하는 게 실수를 줄인다. 경계가 불확실한 상태에서 나누면 높은 확률로 분산 모놀리스가 된다.
