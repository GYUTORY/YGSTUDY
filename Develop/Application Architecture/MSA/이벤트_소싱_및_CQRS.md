# 이벤트 소싱과 CQRS 개념 정리

## 개요

이벤트 소싱과 CQRS는 분산 시스템에서 상태 관리를 다르게 접근하는 패턴이다. 기존 CRUD 방식의 한계를 해결하기 위해 등장했다.

이 문서는 이벤트 소싱과 CQRS의 동작 원리, 사용 이유, 실무에서 겪는 문제를 다룬다.

## 전통적인 CRUD 방식의 문제점

### CRUD 방식의 동작

전통적인 CRUD 방식은 현재 상태만 저장한다.

**주문 처리 예시:**

**1. 주문 생성:**
- `orders` 테이블에 주문 레코드를 INSERT한다
- `status = 'CREATED'`, `amount = 50000`으로 저장한다

**2. 결제 처리:**
- 주문 레코드를 UPDATE한다
- `status = 'PAID'`로 변경한다

**3. 배송 준비:**
- 주문 레코드를 UPDATE한다
- `status = 'SHIPPING'`로 변경한다

**문제점:**
- 과거 상태가 사라진다. `status = 'PAID'`가 되면 이전 상태인 `'CREATED'`는 알 수 없다
- 변경 이력이 없다. 언제, 왜 상태가 바뀌었는지 알 수 없다
- 감사 추적이 어렵다. 누가 변경했는지 추적하려면 별도 로그 테이블이 필요하다

### 분산 시스템에서의 문제

서비스가 분리되면 상태 변경이 여러 저장소에 흩어진다.

**문제 상황:**

**1. 하나의 요청이 여러 서비스 상태를 바꾼다:**
- 주문 서비스: 주문 생성
- 결제 서비스: 결제 처리
- 재고 서비스: 재고 차감
- 각 서비스의 상태가 따로 관리된다

**2. 일부 단계만 성공할 수 있다:**
- 주문은 생성되었지만 결제는 실패했다
- 결제는 성공했지만 재고 차감은 실패했다
- 전체 상태를 파악하기 어렵다

**3. 같은 요청이 중복으로 들어올 수 있다:**
- 네트워크 재시도로 같은 주문이 여러 번 생성된다
- 멱등성이 보장되지 않으면 중복 데이터가 생긴다

**4. 순서가 바뀐 이벤트가 도착할 수 있다:**
- 메시지 큐에서 순서가 보장되지 않는다
- 결제 완료 이벤트가 주문 생성 이벤트보다 먼저 도착할 수 있다

**결과:**
이 상태에서 "현재 상태가 무엇인지"를 단일 테이블로 표현하기가 어려워진다. 이벤트 소싱과 CQRS는 이 문제를 다른 방향에서 해결한다.

## 이벤트 소싱(Event Sourcing)

### 이벤트 소싱의 핵심 개념

이벤트 소싱은 **상태를 저장하지 않고, 상태 변경 내역만 저장**한다.

**기본 원리:**
- 현재 상태 = 과거 이벤트들을 순서대로 재생한 결과
- 데이터의 원본은 항상 이벤트다
- 테이블에 있는 상태 값은 파생 데이터다

**이벤트의 특징:**
- 이벤트는 불변(Immutable)이다. 한 번 저장되면 변경되지 않는다
- 이벤트는 시간 순서대로 저장된다
- 이벤트는 "무엇이 일어났는지"를 기록한다

### 기존 CRUD 방식과의 차이

**CRUD 방식:**

**주문 처리 과정:**
1. 주문 생성: `orders` 테이블에 `status = 'CREATED'`, `amount = 50000` INSERT
2. 결제 처리: `orders` 테이블에서 `status = 'PAID'`로 UPDATE
3. 배송 준비: `orders` 테이블에서 `status = 'SHIPPING'`로 UPDATE

**최종 상태:**
- `status = 'SHIPPING'`
- `amount = 50000`
- 과거 상태는 사라진다

**이벤트 소싱 방식:**

**주문 처리 과정:**
1. 주문 생성: `OrderCreated` 이벤트 저장
   - `order_id = 'order-123'`
   - `user_id = 'user-456'`
   - `amount = 50000`
   - `created_at = '2026-01-14 10:00:00'`

2. 결제 처리: `PaymentCompleted` 이벤트 저장
   - `order_id = 'order-123'`
   - `payment_id = 'payment-789'`
   - `amount = 50000`
   - `completed_at = '2026-01-14 10:01:00'`

3. 배송 준비: `ShippingStarted` 이벤트 저장
   - `order_id = 'order-123'`
   - `shipping_id = 'shipping-012'`
   - `started_at = '2026-01-14 10:02:00'`

**이벤트 스토어:**
모든 이벤트가 시간 순서대로 저장된다. 현재 상태는 이벤트를 재생해서 계산한다.

**차이점:**
- CRUD: 현재 값만 저장, 과거는 사라짐
- 이벤트 소싱: 모든 변경 내역 저장, 과거 상태 재현 가능

### 이벤트 소싱의 동작 원리

**1. 이벤트 저장:**

**이벤트 구조:**
- `event_id`: 이벤트 고유 ID
- `aggregate_id`: 집계 ID (order_id 등)
- `event_type`: 이벤트 타입 (OrderCreated, PaymentCompleted 등)
- `event_data`: 이벤트 데이터 (JSON)
- `version`: 집계 버전 (동시성 제어용)
- `timestamp`: 발생 시간

**저장 과정:**
1. 비즈니스 로직이 실행된다
2. 상태 변경이 발생한다
3. 이벤트가 생성된다
4. 이벤트가 이벤트 스토어에 저장된다
5. 이벤트가 발행된다 (선택사항)

**2. 상태 복원:**

**재생 과정:**
1. 집계 ID로 모든 이벤트를 조회한다
2. 이벤트를 시간 순서대로 정렬한다
3. 각 이벤트를 순서대로 적용한다
4. 최종 상태를 계산한다

**예시:**
- `OrderCreated` 이벤트: 주문 객체 생성, `status = 'CREATED'`
- `PaymentCompleted` 이벤트: `status = 'PAID'`로 변경
- `ShippingStarted` 이벤트: `status = 'SHIPPING'`로 변경
- 최종 상태: `status = 'SHIPPING'`

**3. 이벤트 재생:**

**재생의 의미:**
- 과거 시점의 상태를 재현할 수 있다
- 특정 이벤트까지 재생하면 그 시점의 상태를 얻을 수 있다
- 새로운 뷰를 만들기 위해 이벤트를 다시 처리할 수 있다

**예시:**
- 2026-01-14 10:01:00 시점의 상태를 보고 싶다
- `OrderCreated`와 `PaymentCompleted` 이벤트만 재생한다
- `status = 'PAID'` 상태를 얻는다

### 이벤트 스토어

이벤트 스토어는 이벤트를 저장하는 저장소다.

**이벤트 스토어의 특징:**
- 이벤트는 추가만 가능하다 (Append-only)
- 이벤트는 삭제되지 않는다
- 이벤트는 수정되지 않는다
- 이벤트는 시간 순서대로 저장된다

**구현 방식:**
- 파일 기반: 이벤트를 파일에 순차적으로 저장
- 데이터베이스: 이벤트 테이블에 저장
- 전용 이벤트 스토어: EventStore, Kafka 등

**이벤트 스토어 스키마:**
- `event_id`: 이벤트 고유 ID
- `stream_id`: 스트림 ID (집계 ID)
- `event_type`: 이벤트 타입
- `event_data`: 이벤트 데이터
- `version`: 스트림 버전
- `timestamp`: 발생 시간

**스트림:**
- 같은 집계의 이벤트들을 묶은 개념
- `order-123` 스트림에는 주문 `order-123`의 모든 이벤트가 포함된다
- 스트림 단위로 이벤트를 조회한다

### 스냅샷(Snapshot)

이벤트가 많아지면 상태 복원이 느려진다. 스냅샷은 특정 시점의 상태를 저장해 복원 시간을 단축한다.

**스냅샷의 필요성:**
- 이벤트가 1만 개면 모두 재생하는 데 시간이 걸린다
- 스냅샷을 저장하면 마지막 스냅샷 이후의 이벤트만 재생하면 된다
- 복원 시간이 크게 단축된다

**스냅샷 저장 시점:**
- N개 이벤트마다 저장 (예: 100개마다)
- 특정 시간마다 저장 (예: 1시간마다)
- 특정 이벤트 타입마다 저장

**스냅샷 사용:**
1. 마지막 스냅샷을 로드한다
2. 스냅샷 이후의 이벤트를 재생한다
3. 최종 상태를 계산한다

**예시:**
- 스냅샷: 100번째 이벤트까지 재생한 상태 저장
- 현재: 150번째 이벤트까지 있음
- 복원: 스냅샷 로드 + 101~150번 이벤트 재생
- 전체 재생 대비 시간 단축

### 프로젝션(Projection)

프로젝션은 이벤트를 읽기 모델로 변환하는 과정이다.

**프로젝션의 목적:**
- 이벤트를 조회에 최적화된 형태로 변환한다
- 여러 읽기 모델을 만들 수 있다
- 각 읽기 모델은 특정 용도에 최적화된다

**프로젝션 동작:**
1. 이벤트를 읽는다
2. 읽기 모델을 업데이트한다
3. 조회에 최적화된 형태로 저장한다

**예시:**
- `OrderCreated` 이벤트 → 주문 목록 읽기 모델에 추가
- `PaymentCompleted` 이벤트 → 주문 상세 읽기 모델 업데이트
- `ShippingStarted` 이벤트 → 배송 상태 읽기 모델 업데이트

**읽기 모델의 종류:**
- 주문 목록: 리스트 조회용
- 주문 상세: 상세 조회용
- 통계: 집계 데이터용
- 검색: 검색 엔진용

## 이벤트 소싱을 사용하는 이유

### 상태 변경 이력을 정확히 남겨야 할 때

**금융 도메인:**
- 계좌 잔액 변경 이력
- 거래 내역 추적
- 정산 데이터 검증

**주문 도메인:**
- 주문 상태 변경 이력
- 결제 과정 추적
- 취소/환불 이력

**장애 재현:**
- 장애 발생 시 과거 상태를 재현해야 하는 경우
- 이벤트를 재생해 문제 상황을 재현할 수 있다
- 디버깅이 쉬워진다

**이벤트 소싱의 장점:**
이벤트 소싱은 "왜 이 상태가 되었는지"를 항상 설명할 수 있다. 모든 변경 내역이 이벤트로 저장되어 있다.

### 비즈니스 규칙이 자주 바뀌는 경우

**과거 데이터 재처리:**
- 새로운 통계 기준이 생긴다
- 과거 이벤트를 다시 처리해 새로운 통계를 만든다
- 이벤트 자체는 바뀌지 않는다

**새로운 뷰 생성:**
- 새로운 상태 분류가 필요하다
- 이벤트를 다시 해석해 새로운 뷰를 만든다
- 기존 이벤트를 재사용할 수 있다

**예시:**
- 기존: 주문 상태를 `CREATED`, `PAID`, `SHIPPING`으로 분류
- 변경: `PROCESSING`, `COMPLETED`, `CANCELLED`로 재분류 필요
- 해결: 이벤트를 다시 처리해 새로운 분류 적용

### 감사 추적(Audit Trail)

**완전한 감사 추적:**
- 모든 변경 내역이 이벤트로 저장된다
- 누가, 언제, 무엇을 변경했는지 추적 가능하다
- 별도 로그 시스템 없이도 감사가 가능하다

**규정 준수:**
- 금융, 의료 등 규제가 엄격한 도메인에서 필요하다
- 변경 이력을 완전히 보존해야 한다
- 이벤트 소싱이 자연스럽게 해결한다

## 이벤트 소싱의 문제점과 해결 방법

### 읽기 성능 문제

**문제:**
이벤트가 많아질수록 상태 복원이 느려진다. 이벤트가 1만 개면 모두 재생하는 데 시간이 걸린다.

**해결 방법:**

**1. 스냅샷:**
- 특정 시점의 상태를 저장한다
- 마지막 스냅샷 이후의 이벤트만 재생한다
- 복원 시간을 크게 단축한다

**2. 읽기 모델:**
- 이벤트를 조회에 최적화된 형태로 변환한다
- 조회는 읽기 모델에서 한다
- 이벤트 재생은 쓰기 시에만 한다

**3. 이벤트 아카이빙:**
- 오래된 이벤트는 별도 저장소로 이동한다
- 자주 조회하는 이벤트만 메인 스토어에 둔다
- 성능과 저장 비용의 균형을 맞춘다

### 설계 난이도

**이벤트 스키마 변경:**
- 이벤트 스키마가 변경되면 하위 호환성을 고려해야 한다
- 이전 버전 이벤트를 처리할 수 있어야 한다
- 버전 관리가 필요하다

**이벤트 버전 관리:**
- 이벤트에 버전을 포함한다
- 새 버전 이벤트를 발행해도 이전 버전 소비자가 처리할 수 있어야 한다
- 변환 로직이 필요하다

**재처리 시 멱등성:**
- 이벤트를 재처리할 때 중복 실행을 방지해야 한다
- 멱등성을 보장해야 한다
- 처리 기록을 저장해 중복을 방지한다

**초기 설계 부담:**
초기에 단순 CRUD보다 설계 부담이 크다. 이벤트 구조, 버전 관리, 재처리 로직 등을 고려해야 한다.

### 운영 복잡도

**이벤트 중복:**
- 같은 이벤트가 여러 번 저장될 수 있다
- 네트워크 재시도, 메시지 큐 중복 등으로 발생한다
- 멱등성을 보장해야 한다

**순서 뒤집힘:**
- 이벤트 순서가 바뀔 수 있다
- 메시지 큐에서 순서가 보장되지 않으면 발생한다
- 순서 보장이 필요한 경우 파티션 키를 사용한다

**일부 이벤트만 처리된 상태:**
- 프로젝션이 일부 이벤트만 처리하고 실패할 수 있다
- 읽기 모델이 불일치 상태가 될 수 있다
- 재처리 메커니즘이 필요하다

**이벤트 스토어 관리:**
- 이벤트가 계속 쌓인다
- 저장 공간 관리가 필요하다
- 아카이빙 전략이 필요하다

**이걸 감당할 준비가 되어 있어야 한다:**
이벤트 소싱은 강력하지만 복잡도가 높다. 운영 복잡도를 감당할 수 있을 때만 사용해야 한다.

## CQRS(Command Query Responsibility Segregation)

### CQRS의 핵심 개념

CQRS는 **쓰기(Command)** 와 **읽기(Query)** 를 분리한다.

**기본 원리:**
- 쓰기 모델: 도메인 규칙 중심, 비즈니스 로직 처리
- 읽기 모델: 조회 성능 중심, 최적화된 구조
- 하나의 데이터 모델로 둘 다 해결하지 않는다

**전통적인 방식:**
- 같은 테이블에서 쓰기와 읽기를 모두 처리한다
- 쓰기 최적화와 읽기 최적화가 충돌한다
- 인덱스 추가가 쓰기 성능에 영향을 준다

**CQRS 방식:**
- 쓰기 모델과 읽기 모델을 분리한다
- 각각 독립적으로 최적화한다
- 쓰기와 읽기가 서로 영향을 주지 않는다

### 쓰기 모델과 읽기 모델의 차이

**쓰기 모델 (Command Side):**

**특징:**
- 도메인 규칙 중심으로 설계된다
- 비즈니스 로직을 처리한다
- 정규화된 구조를 사용한다
- 트랜잭션 일관성이 중요하다

**예시:**
- 주문 생성 시 복잡한 검증 로직
- 재고 차감 시 동시성 제어
- 결제 처리 시 비즈니스 규칙 적용

**읽기 모델 (Query Side):**

**특징:**
- 조회 성능 중심으로 설계된다
- 비정규화된 구조를 사용한다
- 조인을 최소화한다
- 최종 일관성을 허용한다

**예시:**
- 주문 목록: 조인 없이 한 번의 쿼리로 조회
- 주문 상세: 필요한 모든 정보를 한 테이블에 저장
- 통계: 집계 데이터를 미리 계산해 저장

### CQRS의 동작 원리

**1. 쓰기 처리:**

**Command 수신:**
- 클라이언트가 Command를 보낸다
- Command Handler가 Command를 처리한다
- 비즈니스 로직을 실행한다
- 쓰기 모델을 업데이트한다

**예시:**
- `CreateOrderCommand` 수신
- 주문 생성 로직 실행
- `orders` 테이블에 INSERT
- 이벤트 발행 (선택사항)

**2. 읽기 모델 동기화:**

**동기화 방식:**
- 쓰기 모델이 변경되면 읽기 모델도 업데이트한다
- 이벤트를 구독해 읽기 모델을 업데이트한다
- 비동기로 처리한다

**예시:**
- 주문이 생성되면 `OrderCreated` 이벤트 발행
- 읽기 모델 프로젝션이 이벤트를 구독
- 주문 목록 읽기 모델에 추가
- 주문 상세 읽기 모델 생성

**3. 읽기 처리:**

**Query 처리:**
- 클라이언트가 Query를 보낸다
- Query Handler가 읽기 모델에서 조회한다
- 최적화된 구조로 빠르게 응답한다

**예시:**
- `GetOrderListQuery` 수신
- 주문 목록 읽기 모델에서 조회
- 조인 없이 한 번의 쿼리로 응답

### CQRS를 사용하는 이유

**1. 읽기와 쓰기의 성격이 다를 때**

**쓰기의 특징:**
- 복잡한 검증과 규칙이 많다
- 트랜잭션 일관성이 중요하다
- 정규화된 구조가 적합하다

**읽기의 특징:**
- 단순 조회, 리스트, 통계가 많다
- 성능이 중요하다
- 비정규화된 구조가 적합하다

**충돌:**
같은 테이블로 처리하면 어느 쪽도 만족시키기 어렵다. 쓰기 최적화를 하면 읽기가 느려지고, 읽기 최적화를 하면 쓰기가 느려진다.

**해결:**
CQRS로 분리하면 각각 독립적으로 최적화할 수 있다.

**2. 읽기 트래픽이 압도적으로 많을 때**

**읽기 트래픽이 많은 경우:**
- 읽기 요청이 쓰기 요청보다 훨씬 많다
- 읽기 전용 DB를 사용할 수 있다
- 읽기 모델을 여러 개 복제할 수 있다

**최적화:**
- 읽기 전용 DB: 쓰기 부하 없이 읽기만 처리
- 캐시 친화적 구조: 자주 조회하는 데이터 구조화
- 조인 최소화: 한 번의 쿼리로 필요한 데이터 조회

**확장성:**
쓰기 모델과 분리하면 조회 구조를 자유롭게 바꿀 수 있다. 읽기 모델을 수평 확장할 수 있다.

**3. 읽기 모델을 독립적으로 변경하고 싶을 때**

**변경의 자유:**
- 읽기 모델 구조를 변경해도 쓰기 모델에 영향이 없다
- 새로운 읽기 모델을 추가할 수 있다
- 기존 읽기 모델을 유지하면서 새 모델을 만들 수 있다

**예시:**
- 기존: 주문 목록 읽기 모델
- 추가: 주문 검색 읽기 모델 (Elasticsearch)
- 추가: 주문 통계 읽기 모델 (시계열 DB)

### CQRS의 구현 방식

**1. 단일 데이터베이스에서 분리:**

**구조:**
- 같은 DB에 쓰기 테이블과 읽기 테이블을 분리한다
- 쓰기는 정규화된 테이블에 한다
- 읽기는 비정규화된 테이블에서 한다
- 트리거나 애플리케이션 로직으로 동기화한다

**장점:**
- 구현이 단순하다
- 트랜잭션 일관성을 보장할 수 있다

**단점:**
- 읽기 모델 업데이트가 쓰기 성능에 영향을 줄 수 있다
- 확장성이 제한적이다

**2. 별도 데이터베이스로 분리:**

**구조:**
- 쓰기 DB와 읽기 DB를 분리한다
- 이벤트를 통해 읽기 모델을 동기화한다
- 비동기로 처리한다

**장점:**
- 읽기와 쓰기가 완전히 분리된다
- 각각 독립적으로 확장할 수 있다
- 읽기 DB를 여러 개 복제할 수 있다

**단점:**
- 최종 일관성을 허용해야 한다
- 동기화 로직이 필요하다

**3. 이벤트 기반 동기화:**

**구조:**
- 쓰기 모델 변경 시 이벤트를 발행한다
- 읽기 모델 프로젝션이 이벤트를 구독한다
- 이벤트를 처리해 읽기 모델을 업데이트한다

**장점:**
- 느슨한 결합이다
- 여러 읽기 모델을 독립적으로 만들 수 있다
- 확장성이 좋다

**단점:**
- 이벤트 처리 지연이 있다
- 최종 일관성을 허용해야 한다

### CQRS에서 흔히 생기는 오해

**1. "무조건 이벤트 소싱이 필요하다"**

아니다. CQRS는 이벤트 소싱 없이도 가능하다.

**이벤트 소싱 없이 CQRS:**
- 쓰기는 RDB 트랜잭션으로 처리한다
- 읽기는 조회 전용 테이블에서 한다
- 트리거나 애플리케이션 로직으로 동기화한다

**이벤트 소싱과 함께 CQRS:**
- 쓰기는 이벤트 소싱으로 처리한다
- 읽기는 이벤트를 프로젝션해 만든다
- 더 강력하지만 복잡도가 높다

**이벤트 소싱은 선택 사항이다:**
CQRS는 읽기와 쓰기를 분리하는 패턴이다. 이벤트 소싱은 쓰기 모델을 구현하는 방법 중 하나다.

**2. "CQRS = 마이크로서비스"**

아니다. 모놀리식에서도 CQRS는 쓸 수 있다.

**모놀리식에서 CQRS:**
- 같은 애플리케이션 내에서 읽기와 쓰기를 분리한다
- 같은 DB에 다른 테이블을 사용한다
- 서비스 분리 없이도 적용 가능하다

**마이크로서비스에서 CQRS:**
- 서비스 간 읽기와 쓰기를 분리한다
- 각 서비스가 독립적인 데이터 모델을 가진다
- 자연스럽게 CQRS가 적용된다

**서비스 분리 여부와 직접적인 연관은 없다:**
CQRS는 아키텍처 패턴이고, 마이크로서비스는 배포 패턴이다. 둘은 독립적이다.

## 이벤트 소싱 + CQRS 조합

### 조합의 형태

이벤트 소싱과 CQRS를 함께 사용하면 강력한 아키텍처가 된다.

**구조:**

**1. 이벤트 소싱 (쓰기 모델):**
- 이벤트 스토어가 쓰기 모델의 진실이다
- 모든 상태 변경이 이벤트로 저장된다
- 이벤트는 불변이다

**2. 프로젝션 (읽기 모델 생성):**
- 이벤트를 읽기 모델로 변환한다
- 여러 읽기 모델을 만들 수 있다
- 각 읽기 모델은 특정 용도에 최적화된다

**3. CQRS (경로 분리):**
- 쓰기 경로: Command → 이벤트 저장
- 읽기 경로: Query → 읽기 모델 조회
- 완전히 분리된 경로

**동작 흐름:**

**쓰기:**
1. Command 수신
2. 비즈니스 로직 실행
3. 이벤트 생성 및 저장
4. 이벤트 발행

**읽기 모델 업데이트:**
1. 이벤트 구독
2. 프로젝션 처리
3. 읽기 모델 업데이트

**읽기:**
1. Query 수신
2. 읽기 모델에서 조회
3. 응답 반환

### 조합의 전제 조건

이 구조는 다음을 전제로 한다.

**1. 읽기 지연을 허용한다:**
- 쓰기 후 읽기가 즉시 반영되지 않을 수 있다
- 이벤트 처리 지연이 있다
- 사용자가 최신 데이터를 즉시 볼 수 없을 수 있다

**2. 최종 일관성을 받아들인다:**
- 쓰기와 읽기가 일시적으로 불일치할 수 있다
- 시간이 지나면 일관성이 유지된다
- 강한 일관성이 필요하면 부적합하다

**3. 상태 불일치가 잠시 존재할 수 있다:**
- 쓰기는 완료되었지만 읽기 모델은 아직 업데이트되지 않았다
- 프로젝션이 실패하면 읽기 모델이 오래된 상태로 남는다
- 재처리 메커니즘이 필요하다

### 조합의 장점

**1. 완전한 감사 추적:**
- 모든 변경 내역이 이벤트로 저장된다
- 언제든지 과거 상태를 재현할 수 있다
- 감사 추적이 자연스럽게 해결된다

**2. 유연한 읽기 모델:**
- 여러 읽기 모델을 독립적으로 만들 수 있다
- 각 읽기 모델은 특정 용도에 최적화된다
- 읽기 모델 변경이 쓰기에 영향을 주지 않는다

**3. 확장성:**
- 쓰기와 읽기를 독립적으로 확장할 수 있다
- 읽기 모델을 여러 개 복제할 수 있다
- 각 모델을 독립적으로 최적화할 수 있다

### 조합의 단점

**1. 복잡도 증가:**
- 이벤트 소싱과 CQRS의 복잡도가 합쳐진다
- 프로젝션 로직이 추가된다
- 운영 부담이 크다

**2. 일관성 문제:**
- 최종 일관성을 허용해야 한다
- 읽기 지연이 발생할 수 있다
- 상태 불일치를 감당해야 한다

**3. 개발 비용:**
- 초기 개발 비용이 크다
- 이벤트 스키마 설계가 필요하다
- 프로젝션 로직 구현이 필요하다

## 실무에서 겪는 문제

### 이벤트 스키마 변경

**문제:**
이벤트 스키마가 변경되면 기존 이벤트와 호환성 문제가 생긴다.

**해결 방법:**
- 이벤트 버전을 관리한다
- 이전 버전 이벤트를 처리할 수 있는 변환 로직을 만든다
- 점진적으로 마이그레이션한다

**예시:**
- 기존: `OrderCreated` 이벤트에 `amount` 필드만 있음
- 변경: `amount`, `discount`, `total_amount` 필드 추가
- 해결: 이전 버전 이벤트는 `discount = 0`, `total_amount = amount`로 변환

### 프로젝션 실패

**문제:**
프로젝션이 실패하면 읽기 모델이 오래된 상태로 남는다.

**해결 방법:**
- 프로젝션 실패를 재시도한다
- 실패한 이벤트를 별도 큐에 저장한다
- 주기적으로 프로젝션 상태를 확인한다

**예시:**
- `OrderCreated` 이벤트 처리 중 DB 연결 실패
- 이벤트를 DLQ에 저장
- 재시도 프로세스가 나중에 처리

### 읽기 모델 동기화 지연

**문제:**
이벤트 처리 지연으로 읽기 모델이 오래된 상태로 남는다.

**해결 방법:**
- 이벤트 처리 지연을 모니터링한다
- 지연이 발생하면 프로세스를 확장한다
- 중요한 읽기는 쓰기 모델에서 직접 조회한다

**예시:**
- 주문 생성 후 즉시 주문 상세를 조회해야 하는 경우
- 읽기 모델 대신 이벤트 스토어에서 직접 조회
- 또는 쓰기 모델에서 조회

### 이벤트 재처리

**문제:**
이벤트를 재처리할 때 중복 실행을 방지해야 한다.

**해결 방법:**
- 멱등성을 보장한다
- 처리 기록을 저장해 중복을 방지한다
- 이벤트 ID를 사용해 중복을 확인한다

**예시:**
- 같은 `OrderCreated` 이벤트를 여러 번 처리해도 주문이 한 번만 생성되어야 한다
- 이벤트 ID를 키로 처리 기록을 저장한다
- 이미 처리된 이벤트는 건너뛴다

## 도입 판단 기준

### 이벤트 소싱을 쓰는 게 맞는 경우

**상태 변경 이력이 중요하다:**
- 금융, 의료 등 규제가 엄격한 도메인
- 모든 변경 내역을 추적해야 하는 경우
- 감사 추적이 필수인 경우

**장애 재현이 잦다:**
- 복잡한 비즈니스 로직으로 인해 장애가 자주 발생한다
- 과거 상태를 재현해 문제를 분석해야 한다
- 디버깅이 어려운 경우

**비즈니스 규칙이 자주 바뀐다:**
- 새로운 통계 기준이 자주 생긴다
- 과거 데이터를 다시 처리해야 한다
- 새로운 뷰를 만들어야 한다

**단순 CRUD로 설명이 안 된다:**
- 복잡한 상태 머신이 있다
- 여러 단계를 거쳐 상태가 변경된다
- 상태 변경 이력이 비즈니스 로직의 일부다

### 이벤트 소싱을 쓰지 말아야 하는 경우

**단순 CRUD 작업:**
- 단순 게시판, 설정 관리
- 상태 변경 이력이 중요하지 않다
- CRUD로 충분한 경우

**트랜잭션 범위가 작다:**
- 단일 서비스 내에서만 처리된다
- 분산 트랜잭션이 필요 없다
- 단순한 작업만 있다

**팀에 운영 여력이 없다:**
- 이벤트 소싱 운영 경험이 없다
- 복잡도를 감당할 수 없다
- 학습 곡선이 높다

**빠른 개발이 최우선이다:**
- 프로토타입이나 MVP 단계
- 빠르게 기능을 만들어야 한다
- 복잡한 패턴이 오히려 방해가 된다

### CQRS를 쓰는 게 맞는 경우

**읽기와 쓰기의 성격이 다르다:**
- 쓰기는 복잡한 비즈니스 로직이 있다
- 읽기는 단순 조회가 많다
- 같은 모델로 처리하기 어렵다

**읽기 트래픽이 압도적으로 많다:**
- 읽기 요청이 쓰기 요청보다 훨씬 많다
- 읽기 성능 최적화가 필요하다
- 읽기를 수평 확장해야 한다

**읽기 모델을 독립적으로 변경하고 싶다:**
- 새로운 읽기 모델을 추가하고 싶다
- 읽기 구조를 자주 변경한다
- 쓰기에 영향을 주지 않고 읽기를 최적화하고 싶다

### CQRS를 쓰지 말아야 하는 경우

**읽기와 쓰기가 단순하다:**
- 단순 CRUD 작업만 있다
- 복잡한 비즈니스 로직이 없다
- 분리할 필요가 없다

**일관성이 중요하다:**
- 강한 일관성이 필요하다
- 읽기 지연을 허용할 수 없다
- 최종 일관성으로는 부족하다

**팀에 운영 여력이 없다:**
- CQRS 운영 경험이 없다
- 동기화 로직을 관리할 수 없다
- 복잡도를 감당할 수 없다

**패턴은 문제를 해결할 때 쓰는 도구다:**
도입 자체가 목표가 되면 실패한다. 실제 문제가 있을 때만 사용해야 한다.

## 정리

**이벤트 소싱:**
- 상태가 아니라 **변화**를 저장한다
- 모든 변경 내역이 이벤트로 저장된다
- 과거 상태를 재현할 수 있다

**CQRS:**
- 읽기와 쓰기를 **역할 기준으로 분리**한다
- 각각 독립적으로 최적화한다
- 확장성이 좋다

**관계:**
- 둘은 독립적이지만 같이 쓰이는 경우가 많다
- 이벤트 소싱 + CQRS 조합이 강력하다
- 하지만 복잡도도 높아진다

**선택 기준:**
- 운영 복잡도를 감당할 수 있을 때만 선택해야 한다
- 실제 문제가 있을 때만 사용해야 한다
- 패턴 도입 자체가 목표가 되면 안 된다

**결론:**
이 패턴들은 시스템을 "단순하게" 만들지 않는다. 대신 **설명 가능한 시스템**을 만든다. 모든 변경 내역을 추적하고, 읽기와 쓰기를 독립적으로 최적화할 수 있다.
