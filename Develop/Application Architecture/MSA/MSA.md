# 마이크로 서비스 아키텍처(MSA)

## 1. 모노리틱 아키텍처의 한계

### 1.1 모노리틱 아키텍처란?
- 모든 기능이 하나의 애플리케이션에 통합되어 있는 전통적인 소프트웨어 개발 방식
- 예: 온라인 쇼핑몰의 경우 사용자 관리, 상품 관리, 주문 관리, 결제 시스템 등이 하나의 애플리케이션에 통합

### 1.2 모노리틱 아키텍처의 장점
- 단일 코드베이스로 개발되어 빌드, 테스트, 배포 과정이 단순함
- 개발 초기에는 빠른 개발과 배포가 가능
- 서비스 간 통신이 내부 메서드 호출로 이루어져 성능상 이점이 있음
- 단일 데이터베이스를 사용하여 데이터 일관성 유지가 용이

### 1.3 모노리틱 아키텍처의 한계점
- 애플리케이션 규모가 커질수록 개발 속도가 저하
- 새로운 기능 추가 시 기존 코드와의 의존성으로 인해 개발 시간 증가
- 일부 기능의 버그가 전체 시스템에 영향을 미칠 수 있음
- 특정 기능의 확장이 필요할 때 전체 애플리케이션을 확장해야 함
- 기술 스택 변경이 어려움 (예: Java에서 Node.js로의 전환)

## 2. 마이크로 서비스 아키텍처의 등장

### 2.1 마이크로 서비스 아키텍처란?
- 하나의 애플리케이션을 독립적인 기능 단위의 작은 서비스들로 분리하여 개발하는 방식
- 각 서비스는 독립적으로 개발, 배포, 확장이 가능
- 서비스 간 통신은 HTTP/REST API 또는 메시지 큐를 통해 이루어짐

### 2.2 마이크로 서비스 아키텍처의 주요 특징
- 서비스별 독립적인 데이터베이스 운영
- 서비스별 독립적인 기술 스택 사용 가능
- 서비스별 독립적인 배포와 확장
- 서비스별 독립적인 팀 운영 가능

## 3. 마이크로 서비스 아키텍처의 장점

### 3.1 확장성과 유연성
- 서비스별 독립적인 확장 가능
  - 예: 주문 서비스의 트래픽이 증가할 경우 주문 서비스만 확장
- 서비스별 독립적인 기술 스택 선택 가능
  - 예: 실시간 알림 서비스는 Node.js, 데이터 분석 서비스는 Python 사용 가능

### 3.2 개발 생산성
- 작은 단위의 서비스로 인해 코드 이해도 향상
- 새로운 개발자의 온보딩 시간 단축
- 서비스별 독립적인 개발과 배포로 인한 개발 속도 향상

### 3.3 장애 격리
- 특정 서비스의 장애가 다른 서비스에 미치는 영향 최소화
- 서비스별 독립적인 장애 복구 가능

### 3.4 컨테이너 기술과의 시너지
- Docker, Kubernetes 등의 컨테이너 기술과의 높은 적합성
- 서비스별 독립적인 배포와 관리 용이
- 클라우드 환경에서의 효율적인 자원 활용

## 4. 마이크로 서비스 아키텍처의 단점

### 4.1 시스템 복잡성
- 분산 시스템의 복잡성 증가
- 서비스 간 통신 관리의 어려움
- 트랜잭션 관리의 복잡성
  - 예: 주문 생성 시 재고 확인, 결제 처리 등의 분산 트랜잭션 관리

### 4.2 운영 복잡성
- 서비스 모니터링과 로깅의 어려움
- 장애 추적과 디버깅의 복잡성
- 서비스 간 의존성 관리의 어려움

### 4.3 데이터 관리
- 분산 데이터베이스 관리의 어려움
- 데이터 일관성 유지의 어려움
- 데이터 중복 가능성

### 4.4 네트워크 오버헤드
- 서비스 간 통신으로 인한 네트워크 지연
- API 호출 증가로 인한 성능 저하 가능성

## 5. 마이크로 서비스 아키텍처 구현 시 고려사항

### 5.1 서비스 경계 설정
- 비즈니스 도메인 기반의 서비스 분리
- 서비스 간 결합도 최소화
- 서비스 크기 적절성 유지

### 5.2 통신 방식
- 동기 통신 (REST API, gRPC)
- 비동기 통신 (메시지 큐, 이벤트 스트림)
- API 게이트웨이 활용

### 5.3 데이터 관리
- 서비스별 독립적인 데이터베이스
- 이벤트 소싱 패턴 활용
- CQRS 패턴 활용

### 5.4 모니터링과 관찰성
- 분산 추적 시스템 (예: Jaeger, Zipkin)
- 중앙화된 로깅 시스템
- 서비스 메시 활용

## 6. 마이크로 서비스 아키텍처 도입 시 고려사항

### 6.1 조직적 측면
- 팀 구조의 재조정 필요
- DevOps 문화의 도입
- 서비스 소유권과 책임의 명확화

### 6.2 기술적 측면
- 적절한 기술 스택 선택
- 인프라 자동화 도구 활용
- CI/CD 파이프라인 구축

### 6.3 운영적 측면
- 모니터링 시스템 구축
- 장애 대응 프로세스 수립
- 보안 정책 수립

## 7. 결론

마이크로 서비스 아키텍처는 모노리틱 아키텍처의 한계를 극복하기 위한 효과적인 대안이지만, 그 자체로 만능 해결책은 아닙니다. 조직의 규모, 개발 문화, 기술 역량 등을 고려하여 신중하게 도입 여부를 결정해야 합니다. 또한 마이크로 서비스 아키텍처의 성공적인 운영을 위해서는 DevOps 문화, 자동화 도구, 모니터링 시스템 등의 지원이 필수적입니다.



# 마이크로 서비스 아키텍처의 장점
- 가장 큰 장점은 확장성
- 다양한 기능이 독립적인 서비스로 나뉘어 있어서 기능별로 배포할 수 있음
- 단일 서비스에 장애가 발생해도 해당 서비스 외의 다른 기능에 큰 영향을 주지 않음
- 개별 서비스에 새로운 기술을 적용하는 것도 상대적으로 쉬움
- 서비스 단위가 모노리틱 아키텍처에 비해 작기 때문에 새로 합류한 개발자가 기능을 이해하
기도 쉬움
- 마이크로 서비스 아키텍처의 확장성을 더욱 빛나게 해주는 것이 컨테이너 가상화 기술
- 클라우드 IaaS 서비스의 가상머신을 사용해 애플리케이션을 통째로 배포할 수도 있지만 컨
테이너 기술을 사용해 개별 서비스를 독립적으로 배포하면 자원을 보다 효율적으로 활용할 수 있음

# 마이크로 서비스 아키텍처의 단점
- 하나의 애플리케이션을 너무 작은 서비스 단위로 잘게 쪼개면 네트워크에 메시지를 보내는
트래픽이 많아지면서 지연
- 통합 테스트에서 문제가 발생하면 여러 서비스 중 어떤 서비스에서 발생한 문제인지 찾아내
는 과정이 추가로 필요
- 마이크로 서비스 아키텍처는 개별 서비스가 독립적인 기능을 담당하기 때문에 서비스별로
독립적인 데이터베이스를 활용
- 데이터베이스가 분산되어 있으므로 여러 데이터베이스에 흩어진 데이터를 관리하기 어려움

# 마이크로 서비스 아키텍처
- 마이크로 서비스 아키텍처가 모노리틱 아키텍처에 비해 장점과 단점이 뚜렷하지만 최근 추
  세는 여러 자동화 도구와 컨테이너 기술의 발전에 힘입어 각각의 단점을 보완하는 방향으로
  진화하고 있음
- 하지만 좋은 아키텍처도 결국 사용하는 개발자와 문화에 따라 효과가 다르기 때문에 데브옵
  스, 지속적인 통합/배포, 컨테이너 기술을 적극적으로 활용하는 노력이 병행되어야 함
