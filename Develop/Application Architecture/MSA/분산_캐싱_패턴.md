# 분산 캐싱 패턴

## MSA에서 캐싱이 복잡해지는 이유

모놀리스에서는 애플리케이션 레벨 캐시 하나로 충분했다. HashMap이나 Ehcache 같은 로컬 캐시를 쓰면 됐고, 캐시 무효화도 같은 프로세스 안에서 일어나니까 단순했다.

MSA로 넘어가면 상황이 달라진다. 주문 서비스가 상품 정보를 캐싱하고 있는데, 상품 서비스에서 가격을 바꿨다면? 주문 서비스의 캐시에는 여전히 옛날 가격이 남아 있다. 서비스가 독립적으로 배포되고 각자의 메모리 공간을 갖기 때문에, 캐시 정합성 문제가 모놀리스와는 비교할 수 없이 까다로워진다.


## 로컬 캐시 vs 공유 캐시

### 로컬 캐시

각 서비스 인스턴스의 메모리에 데이터를 저장하는 방식이다. Caffeine, Guava Cache 같은 라이브러리를 쓴다.

```java
@Bean
public CacheManager cacheManager() {
    CaffeineCacheManager manager = new CaffeineCacheManager("products");
    manager.setCaffeine(Caffeine.newBuilder()
        .maximumSize(10_000)
        .expireAfterWrite(5, TimeUnit.MINUTES));
    return manager;
}
```

장점은 네트워크 호출이 없어서 빠르다는 것이다. 레이턴시가 수십 나노초 수준이다.

문제는 인스턴스가 여러 개일 때 발생한다. 주문 서비스가 3개 인스턴스로 떠 있으면, 각 인스턴스의 로컬 캐시가 서로 다른 데이터를 갖고 있을 수 있다. A 인스턴스는 캐시가 갱신됐는데 B, C는 아직 옛날 데이터를 보여주는 상황이 생긴다. 로드밸런서가 어디로 요청을 보내느냐에 따라 사용자가 보는 데이터가 달라진다.

로컬 캐시가 적합한 경우:

- 변경 빈도가 낮은 데이터 (국가 코드, 카테고리 목록)
- 짧은 TTL로 일시적 불일치를 허용할 수 있는 데이터
- 서비스 인스턴스가 1~2개뿐인 경우

### 공유 캐시 (Redis Cluster)

Redis를 중앙 캐시로 두고 모든 인스턴스가 같은 캐시를 바라보는 구조다.

```java
@Bean
public LettuceConnectionFactory redisConnectionFactory() {
    RedisClusterConfiguration config = new RedisClusterConfiguration();
    config.clusterNode("redis-1", 6379);
    config.clusterNode("redis-2", 6379);
    config.clusterNode("redis-3", 6379);
    return new LettuceConnectionFactory(config);
}
```

모든 인스턴스가 같은 데이터를 보기 때문에 정합성 문제가 줄어든다. 하지만 네트워크 호출이 발생하므로 로컬 캐시보다 느리다. 보통 1ms 내외인데, 로컬 캐시의 나노초와 비교하면 수만 배 차이다.

Redis Cluster를 운영하면서 자주 겪는 문제:

- **핫키 문제**: 특정 키에 요청이 몰리면 해당 슬롯을 담당하는 노드에 부하가 집중된다. 인기 상품 정보 같은 게 대표적이다. 키를 분산시키거나 로컬 캐시와 조합해서 해결한다.
- **대용량 키**: 하나의 키에 수MB 이상의 데이터를 넣으면 네트워크 지연과 메모리 단편화가 발생한다. 키를 쪼개야 한다.
- **노드 장애 시 페일오버**: 마스터 노드가 죽으면 슬레이브가 승격되는데, 이 과정에서 몇 초간 해당 슬롯의 키에 접근이 안 될 수 있다.

### 로컬 + 공유 캐시 조합 (L1/L2 캐시)

실무에서 가장 많이 쓰는 구조다. 로컬 캐시를 L1, Redis를 L2로 둔다.

```
요청 → L1(로컬 캐시) → 미스 → L2(Redis) → 미스 → DB
```

```java
public Product getProduct(String productId) {
    // L1: 로컬 캐시 확인
    Product cached = localCache.getIfPresent(productId);
    if (cached != null) {
        return cached;
    }

    // L2: Redis 확인
    cached = redisTemplate.opsForValue().get("product:" + productId);
    if (cached != null) {
        localCache.put(productId, cached);
        return cached;
    }

    // DB 조회
    Product product = productRepository.findById(productId)
        .orElseThrow(() -> new NotFoundException("상품 없음: " + productId));

    redisTemplate.opsForValue().set("product:" + productId, product, 30, TimeUnit.MINUTES);
    localCache.put(productId, product);
    return product;
}
```

L1의 TTL은 짧게(30초~2분), L2의 TTL은 길게(10분~1시간) 설정한다. L1은 빈번한 조회의 응답 속도를 높이는 용도고, L2는 DB 부하를 줄이는 용도다.


## 캐시 무효화 — 가장 어려운 문제

> "컴퓨터 과학에서 어려운 것 두 가지: 캐시 무효화와 이름 짓기" — Phil Karlton

MSA에서 캐시 무효화가 특히 어려운 이유는, 데이터를 변경하는 서비스와 그 데이터를 캐싱하는 서비스가 다르기 때문이다.

### 이벤트 기반 무효화

데이터가 변경되면 이벤트를 발행하고, 해당 데이터를 캐싱하고 있는 서비스가 이벤트를 소비해서 캐시를 갱신하거나 삭제한다.

```java
// 상품 서비스: 가격 변경 시 이벤트 발행
@Transactional
public void updatePrice(String productId, BigDecimal newPrice) {
    Product product = productRepository.findById(productId).orElseThrow();
    product.updatePrice(newPrice);
    productRepository.save(product);

    eventPublisher.publish(new ProductPriceChangedEvent(productId, newPrice));
}
```

```java
// 주문 서비스: 이벤트 수신 후 캐시 무효화
@KafkaListener(topics = "product-events")
public void handleProductEvent(ProductPriceChangedEvent event) {
    String cacheKey = "product:" + event.getProductId();
    redisTemplate.delete(cacheKey);
    localCache.invalidate(event.getProductId());
}
```

이벤트 기반 무효화에서 조심할 점:

- **이벤트 유실**: Kafka 컨슈머가 일시적으로 죽어있는 동안 이벤트가 지나갈 수 있다. TTL을 반드시 설정해서 최악의 경우에도 TTL 만큼만 불일치가 발생하도록 해야 한다.
- **순서 보장**: 같은 상품의 가격이 1000원 → 2000원 → 1500원으로 바뀌었는데, 이벤트가 순서대로 처리되지 않으면 캐시에 2000원이 남을 수 있다. 파티션 키를 상품 ID로 설정해서 같은 상품의 이벤트는 같은 파티션으로 보내야 한다.

### TTL 기반 무효화

가장 단순한 방법이다. 캐시에 만료 시간을 설정하고, 시간이 지나면 자동으로 삭제되게 한다. 이벤트 기반 무효화와 함께 안전장치로 쓴다.

TTL 설정 시 고려사항:

- 데이터 성격에 따라 다르게 설정한다. 상품 목록은 5분, 사용자 프로필은 30분, 환율 정보는 1분 같은 식이다.
- TTL을 모든 키에 똑같이 설정하면 만료 시점에 캐시 미스가 동시에 발생한다. 기본 TTL에 랜덤 값을 더해서 만료 시점을 분산시켜야 한다.

```java
private Duration randomizedTtl(Duration baseTtl) {
    long jitter = ThreadLocalRandom.current().nextLong(baseTtl.toSeconds() / 10);
    return baseTtl.plusSeconds(jitter);
}
```


## 캐시 읽기/쓰기 패턴

### Cache-Aside (Lazy Loading)

가장 흔한 패턴이다. 애플리케이션이 직접 캐시와 DB를 관리한다.

```
읽기: 캐시 확인 → 미스면 DB 조회 → 결과를 캐시에 저장 → 반환
쓰기: DB 업데이트 → 캐시 삭제
```

```java
public Product getProduct(String id) {
    Product cached = cache.get(id);
    if (cached != null) return cached;

    Product product = db.findById(id);
    cache.put(id, product, TTL);
    return product;
}

public void updateProduct(String id, ProductUpdateRequest request) {
    db.update(id, request);
    cache.delete(id);  // 갱신이 아니라 삭제한다
}
```

쓰기에서 캐시를 "갱신"하지 않고 "삭제"하는 이유가 있다. DB 업데이트와 캐시 갱신 사이에 다른 스레드가 끼어들면, DB에는 새 값이 있는데 캐시에는 옛날 값이 덮어씌워질 수 있다. 삭제하면 다음 읽기에서 DB의 최신 값을 가져오니까 이 문제를 피할 수 있다.

Cache-Aside의 약점:

- 캐시 미스 시 응답이 느리다. DB 조회 후 캐시 저장까지 해야 하니까.
- 첫 요청은 무조건 캐시 미스다. 자주 조회되는 데이터는 서비스 시작 시 미리 워밍업하는 게 좋다.

### Write-Through

쓰기 시 캐시와 DB에 동시에 쓴다. 캐시가 항상 최신 상태를 유지한다.

```
쓰기: 캐시에 쓰기 → DB에 쓰기 (동기)
읽기: 캐시에서 바로 읽기
```

```java
public void saveProduct(Product product) {
    // 캐시와 DB에 동시에 쓴다
    cache.put(product.getId(), product, TTL);
    db.save(product);
}

public Product getProduct(String id) {
    // 캐시에 항상 있다는 전제
    Product cached = cache.get(id);
    if (cached != null) return cached;

    // 캐시 미스는 TTL 만료나 장애 시에만 발생
    Product product = db.findById(id);
    cache.put(id, product, TTL);
    return product;
}
```

읽기가 빠르고 캐시 미스가 거의 없다는 장점이 있지만, 쓰기가 느려진다. DB 쓰기뿐 아니라 캐시 쓰기도 동기로 일어나기 때문이다. 쓰기가 적고 읽기가 많은 데이터에 적합하다. 상품 카탈로그 같은 경우다.

### Write-Behind (Write-Back)

쓰기를 캐시에만 먼저 하고, DB 반영은 비동기로 나중에 한다.

```
쓰기: 캐시에 쓰기 → (비동기) DB에 쓰기
읽기: 캐시에서 바로 읽기
```

```java
public void saveProduct(Product product) {
    cache.put(product.getId(), product);
    // DB 쓰기는 별도 스레드에서 배치로 처리
    writeQueue.add(new WriteOperation("product", product.getId(), product));
}

// 별도 스케줄러
@Scheduled(fixedDelay = 1000)
public void flushWriteQueue() {
    List<WriteOperation> batch = writeQueue.drain(100);
    if (batch.isEmpty()) return;

    try {
        db.batchSave(batch);
    } catch (Exception e) {
        // 실패한 항목은 큐에 다시 넣는다
        writeQueue.addAll(batch);
        log.error("DB 배치 쓰기 실패, 재시도 예정", e);
    }
}
```

쓰기 성능이 좋고, 여러 건의 쓰기를 배치로 모아서 DB에 한 번에 반영할 수 있다.

위험한 점은 캐시에만 있고 DB에 아직 반영되지 않은 데이터가 존재한다는 것이다. 이 상태에서 캐시 노드가 죽으면 데이터가 유실된다. Redis의 AOF 영속화를 켜거나, 별도의 WAL(Write-Ahead Log)을 운영해서 유실 가능성을 줄여야 한다.

### 패턴 선택 기준

| 상황 | 권장 패턴 |
|------|-----------|
| 읽기 많고 쓰기 적음 (상품 정보) | Cache-Aside 또는 Write-Through |
| 쓰기가 빈번함 (조회수, 좋아요) | Write-Behind |
| 데이터 정합성이 중요함 (결제) | Cache-Aside (캐시 삭제 방식) |
| 읽기 응답 속도가 최우선 | Write-Through + L1/L2 |


## 캐시 스탬피드(Stampede) 방지

캐시 스탬피드는 캐시가 만료되는 순간 동일한 키에 대해 수백~수천 개의 요청이 동시에 DB로 몰리는 현상이다. 인기 상품 페이지의 캐시가 만료되면 그 순간 몰려있던 요청이 전부 DB를 때린다.

### 뮤텍스 락 (Lock-based Recomputation)

캐시 미스 시 한 스레드만 DB를 조회하고, 나머지는 대기시킨다.

```java
public Product getProductWithLock(String productId) {
    String cacheKey = "product:" + productId;
    Product cached = redisTemplate.opsForValue().get(cacheKey);
    if (cached != null) return cached;

    String lockKey = "lock:" + cacheKey;
    boolean acquired = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);

    if (acquired) {
        try {
            // 락 획득: DB 조회 후 캐시 갱신
            Product product = productRepository.findById(productId).orElseThrow();
            redisTemplate.opsForValue().set(cacheKey, product, 30, TimeUnit.MINUTES);
            return product;
        } finally {
            redisTemplate.delete(lockKey);
        }
    } else {
        // 락 미획득: 잠시 대기 후 캐시 재확인
        try { Thread.sleep(50); } catch (InterruptedException ignored) {}
        return getProductWithLock(productId);  // 재귀 호출
    }
}
```

동작은 하지만, 대기 중인 스레드가 많으면 응답 시간이 길어진다. 재귀 호출 깊이 제한도 넣어야 한다.

### 사전 갱신 (Probabilistic Early Expiration)

캐시가 실제로 만료되기 전에 확률적으로 미리 갱신하는 방법이다. TTL이 임박하면 일정 확률로 DB를 조회해서 캐시를 미리 갱신한다.

```java
public Product getProductWithEarlyRefresh(String productId) {
    String cacheKey = "product:" + productId;
    CacheEntry<Product> entry = getWithExpiry(cacheKey);

    if (entry == null) {
        // 완전 미스: DB 조회
        return loadAndCache(productId);
    }

    long remainingTtl = entry.getRemainingTtlSeconds();
    long totalTtl = 1800; // 30분

    // 남은 TTL이 전체의 10% 이하면 확률적으로 갱신
    if (remainingTtl < totalTtl * 0.1) {
        double refreshProbability = 1.0 - ((double) remainingTtl / (totalTtl * 0.1));
        if (ThreadLocalRandom.current().nextDouble() < refreshProbability) {
            CompletableFuture.runAsync(() -> loadAndCache(productId));
        }
    }

    return entry.getValue();
}
```

이 방식은 캐시가 만료되기 전에 자연스럽게 갱신이 일어나므로, 만료 시점에 요청이 DB로 몰리는 현상을 줄일 수 있다.

### 서비스 시작 시 캐시 워밍업

서비스가 재배포되거나 새 인스턴스가 뜰 때, 로컬 캐시가 비어있는 상태에서 트래픽을 받으면 DB에 부하가 몰린다. 자주 조회되는 데이터를 미리 캐시에 올려두는 게 좋다.

```java
@Component
public class CacheWarmer implements ApplicationRunner {

    @Override
    public void run(ApplicationArguments args) {
        log.info("캐시 워밍업 시작");

        List<String> hotProductIds = productRepository.findTopByOrderCount(1000);
        for (String id : hotProductIds) {
            Product product = productRepository.findById(id).orElse(null);
            if (product != null) {
                cache.put(id, product);
            }
        }

        log.info("캐시 워밍업 완료: {}건", hotProductIds.size());
    }
}
```

워밍업 중에 트래픽을 받지 않도록 쿠버네티스의 Readiness Probe와 연동하는 것이 좋다. 워밍업이 끝난 후에 Ready 상태로 전환시킨다.


## 실무에서 자주 겪는 문제

### 캐시 침투 (Cache Penetration)

존재하지 않는 데이터를 반복 조회하는 경우다. DB에도 없는 데이터이므로 매번 캐시 미스가 발생하고 DB까지 조회가 간다. 악의적인 요청이나 잘못된 파라미터로 자주 발생한다.

해결 방법은 null 결과도 캐싱하는 것이다. 단, TTL을 짧게 설정한다.

```java
public Product getProduct(String id) {
    Product cached = cache.get(id);
    if (cached != null) {
        return cached == NULL_PRODUCT ? null : cached;
    }

    Product product = db.findById(id);
    if (product == null) {
        // null도 캐싱한다. TTL은 짧게.
        cache.put(id, NULL_PRODUCT, Duration.ofMinutes(2));
        return null;
    }

    cache.put(id, product, Duration.ofMinutes(30));
    return product;
}
```

조회 가능한 ID 범위가 예측 가능하면 블룸 필터를 앞에 두는 방법도 있다. 블룸 필터에 없는 ID는 DB 조회 자체를 하지 않는다.

### 캐시 쇄도 (Cache Avalanche)

대량의 캐시가 동시에 만료되는 현상이다. 서비스 재시작 후 모든 캐시가 같은 시점에 생성되면, TTL이 동일할 경우 같은 시점에 만료된다.

앞서 설명한 TTL 랜덤화로 해결한다. 캐시 워밍업 시에도 각 키의 TTL을 다르게 설정해야 한다.

### Redis 장애 시 대응

Redis가 다운되면 모든 요청이 DB로 간다. DB가 버틸 수 없으면 서비스 전체가 죽는다.

```java
public Product getProductSafe(String id) {
    try {
        Product cached = redisTemplate.opsForValue().get("product:" + id);
        if (cached != null) return cached;
    } catch (RedisConnectionException e) {
        log.warn("Redis 연결 실패, DB 직접 조회: {}", id);
        // Redis 장애 시 로컬 캐시라도 사용
        Product localCached = localCache.getIfPresent(id);
        if (localCached != null) return localCached;
    }

    Product product = db.findById(id);
    localCache.put(id, product);
    return product;
}
```

Redis 장애 시에는 로컬 캐시로 폴백하되, 서킷 브레이커를 걸어서 Redis 호출 자체를 일시적으로 차단하는 것이 좋다. 매 요청마다 Redis 연결을 시도하고 타임아웃을 기다리는 것보다, 일정 시간 동안 로컬 캐시만 사용하는 것이 응답 시간 측면에서 낫다.


## 직렬화 — 간과하기 쉬운 부분

Redis에 객체를 저장하려면 직렬화가 필요하다. 기본 JdkSerializationRedisSerializer는 클래스 구조가 바뀌면 역직렬화에 실패한다. 서비스를 배포하면서 필드를 추가하거나 삭제하면, 이전 버전으로 직렬화된 캐시 데이터를 읽지 못한다.

```java
@Bean
public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
    RedisTemplate<String, Object> template = new RedisTemplate<>();
    template.setConnectionFactory(factory);
    template.setKeySerializer(new StringRedisSerializer());

    ObjectMapper mapper = new ObjectMapper();
    mapper.activateDefaultTyping(
        mapper.getPolymorphicTypeValidator(),
        ObjectMapper.DefaultTyping.NON_FINAL
    );
    // 알 수 없는 필드 무시 — 배포 시 호환성 확보
    mapper.configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);

    template.setValueSerializer(new GenericJackson2JsonRedisSerializer(mapper));
    return template;
}
```

`FAIL_ON_UNKNOWN_PROPERTIES`를 false로 설정하면, 새 버전에서 필드를 추가해도 이전 캐시 데이터를 읽을 수 있다. 반대로 필드가 삭제된 경우에도 에러 없이 무시된다. 배포 시 캐시를 전부 날리지 않아도 되니까 운영이 편해진다.
