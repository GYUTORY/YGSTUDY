# 시간복잡도와 공간복잡도

## 들어가며

알고리즘을 공부하다 보면 Big-O라는 표기법을 가장 먼저 배우게 되는데, 정작 실무에 들어가면 이걸 제대로 이해하지 못해서 고생하는 경우가 많다. N이 100만일 때 O(N^2)이 왜 1조 연산인지, 왜 C++에서는 되는데 파이썬에서는 시간초과가 나는지, 왜 이론상 O(N)인 해시맵이 실제로는 O(N^2)처럼 동작하는지 같은 것들이다.

복잡도 분석은 "이 코드가 얼마나 빠른가"를 재는 도구가 아니라 "입력이 커지면 얼마나 느려지는가"를 예측하는 도구다. 이 관점 차이가 중요하다. 실제 서비스에서 트래픽이 10배가 되면 레이턴시가 10배가 될지 100배가 될지를 미리 계산할 수 있어야 장애를 피할 수 있다. 이 문서는 Big-O/Big-Θ/Big-Ω의 엄밀한 정의부터 시작해서, 재귀 복잡도 계산(Master Theorem), amortized analysis, 그리고 언어별로 자료구조 연산이 실제로 어떻게 동작하는지까지 다룬다.

## 왜 복잡도 분석인가

입력이 작을 때는 어떤 알고리즘을 써도 체감 차이가 없다. N=100이면 O(N^3)도 100만 연산이라 0.01초면 끝난다. 문제는 N이 커질 때다. N=10만이면 O(N^3)은 10^15 연산이라 하루가 지나도 못 끝난다. 같은 입력에 O(N log N)은 170만 연산으로 0.02초면 끝난다. 이 차이를 사전에 계산하는 게 복잡도 분석이다.

실제 머신에서 초당 처리할 수 있는 연산 수를 대략 알고 있어야 한다. C++ 기준으로 단순 산술 연산은 초당 10억(10^9) 정도 되고, 메모리 접근이 많으면 1억(10^8) 정도로 떨어진다. Java는 C++의 1/2~1/3, Python은 1/10~1/30 수준이다. 이 감각이 있어야 "N=10만일 때 O(N^2)은 안 된다"는 판단을 즉시 내릴 수 있다.

### 실무에서의 복잡도 감각

서비스 코드에서는 경쟁 프로그래밍과 다른 감각이 필요하다. 경쟁에서는 "1초 안에 끝나면 통과"지만, 서비스에서는 "P99 레이턴시가 100ms 이내여야 한다" 같은 SLO가 있다. 100ms 안에 O(N^2) 알고리즘을 돌리려면 N이 얼마여야 하는지 역산하는 감각이 필요하다.

또 서비스 코드는 입력이 예측하기 어렵다. 평균 N이 100이라도 최악의 경우 N이 10만이 될 수 있다면, 평균 기준으로 O(N^2)을 쓰면 안 된다. 최악의 경우 100배 느려지는 API는 장애의 원인이 된다.

## Big-O, Big-Θ, Big-Ω 정의

많은 개발자가 Big-O만 알고 Big-Θ, Big-Ω는 구분 없이 쓰는데, 정확한 정의를 알아두면 면접이나 논문을 읽을 때 도움이 된다.

### Big-O (상한)

$f(n) = O(g(n))$이란, 충분히 큰 $n$에 대해 $f(n) \leq c \cdot g(n)$을 만족하는 상수 $c > 0$과 $n_0$이 존재한다는 뜻이다. 즉 $g(n)$은 $f(n)$의 상한이다. "이 알고리즘이 최악의 경우 이 정도 시간 안에는 끝난다"를 말할 때 쓴다.

중요한 건 Big-O는 tight할 필요가 없다는 것이다. 예를 들어 정확히 $3n$만큼 걸리는 알고리즘은 $O(n)$이기도 하고 $O(n^2)$이기도 하고 $O(n!)$이기도 하다. 다 상한이기 때문이다. 일반적으로 가장 tight한 상한을 쓰는 게 관례지만, 엄밀히는 아무 상한이나 써도 틀린 게 아니다.

### Big-Ω (하한)

$f(n) = \Omega(g(n))$이란, 충분히 큰 $n$에 대해 $f(n) \geq c \cdot g(n)$을 만족하는 $c > 0$이 존재한다는 뜻이다. "이 알고리즘은 아무리 빨라도 이 정도 시간은 걸린다"를 말할 때 쓴다.

비교 기반 정렬이 $\Omega(n \log n)$이라는 증명이 대표적인 예다. 결정 트리의 높이가 최소 $\log_2(n!) = \Theta(n \log n)$이라는 정보이론적 하한을 말한다. 이런 하한이 있어야 "이 문제는 이보다 빠른 알고리즘이 존재할 수 없다"는 주장이 가능하다.

### Big-Θ (정확한 증가율)

$f(n) = \Theta(g(n))$이란 Big-O와 Big-Ω가 동시에 성립할 때다. 즉 상한과 하한이 같은 증가율일 때 쓴다. "정확히 이 정도의 증가율을 가진다"를 말할 때 쓴다.

실무에서 "이 알고리즘은 O(N log N)"이라고 말할 때 대부분 사실은 $\Theta(N \log N)$을 의미한다. 최선과 최악이 모두 $N \log N$에 비례한다는 뜻이다. 병합 정렬이 대표적이다.

### 언제 어느 표기를 써야 하는가

실무 대화에서는 Big-O만 써도 대부분 통한다. "퀵소트는 평균 O(N log N), 최악 O(N^2)"처럼 최악의 경우를 분리해서 말하면 충분하다. 하지만 논문을 읽거나 알고리즘 책을 볼 때는 Θ/Ω가 자주 나온다. 특히 lower bound 증명에서는 Ω가 핵심이다.

한 가지 혼동 포인트가 있다. "이 알고리즘은 O(N^2)"라고 말하면 대부분의 사람은 "N^2에 비례한다"로 이해하지만, 정확히는 "N^2보다 빠르지 않다"는 뜻이다. 실제로 O(N)인 알고리즘도 O(N^2)라고 말하는 게 틀린 건 아니다. 다만 너무 loose한 상한은 정보량이 떨어져서 실무에서는 가장 tight한 상한을 쓴다.

## 숨겨진 상수의 함정

Big-O는 상수 계수를 무시한다. $10N$과 $10000N$은 둘 다 $O(N)$이다. 이론적으로는 맞지만 실무에서는 이게 성능을 결정한다. 특히 같은 복잡도를 가진 두 알고리즘을 비교할 때 상수가 관건이 된다.

### 캐시 친화성

배열 순회와 연결리스트 순회는 둘 다 O(N)이지만, 실제 성능은 10배 이상 차이 난다. CPU 캐시 때문이다. 현대 CPU는 메모리에서 64바이트(캐시 라인) 단위로 데이터를 가져오기 때문에, 배열처럼 연속된 메모리를 순회하면 한 번 읽은 캐시 라인에서 여러 원소를 연속으로 처리할 수 있다. 연결리스트는 노드가 메모리 곳곳에 흩어져 있어서 매번 캐시 미스가 발생한다.

```cpp
// 배열 순회: 캐시 친화적
int sum_array(vector<int>& arr) {
    int sum = 0;
    for (int x : arr) sum += x;
    return sum;
}

// 연결리스트 순회: 캐시 미스 다발
int sum_list(Node* head) {
    int sum = 0;
    while (head) {
        sum += head->val;
        head = head->next;
    }
    return sum;
}
```

두 함수 모두 O(N)이지만 N=10^7에서 실측하면 배열이 10~30배 빠르다. 복잡도로는 설명 안 되는 현상이다.

### 상수가 큰 알고리즘

이론상 더 좋은 복잡도를 가진 알고리즘이 실제로는 더 느린 경우가 많다. 대표적인 게 행렬 곱셈이다. 스트라센 알고리즘은 $O(N^{2.807})$로 단순 구현의 $O(N^3)$보다 좋지만, 상수가 크고 재귀 오버헤드 때문에 실제로는 N이 수천 이상일 때만 이득이 난다.

해시 테이블도 비슷하다. 이론상 평균 O(1)이지만 해시 함수 계산, 충돌 처리, 리해싱 때문에 상수가 크다. 배열 기반 자료구조로 풀 수 있는 문제라면 해시맵보다 배열이 훨씬 빠르다. C++에서 `unordered_map<int, int>`보다 `vector<int>`가 10배 빠른 경우가 흔하다.

### 로그의 밑

$O(\log N)$이라고 할 때 밑을 명시하지 않는데, 이는 로그의 밑이 상수이기 때문이다. $\log_2 N = \frac{\log_{10} N}{\log_{10} 2}$이고 $\frac{1}{\log_{10} 2}$은 상수이므로 Big-O에서는 무시된다. 하지만 실제 성능에서는 밑이 중요하다. 이진 탐색 트리의 높이와 B-Tree의 높이는 다르다. 밑이 1000인 B-Tree는 높이가 훨씬 낮아서 디스크 I/O가 적다.

## Amortized Analysis (분할 상환 분석)

어떤 연산은 대부분의 경우 빠른데 가끔 느리다. 이런 경우 최악의 경우만 보고 복잡도를 잡으면 과대평가된다. Amortized analysis는 여러 연산에 걸친 평균 비용을 계산하는 방법이다.

### 동적 배열의 push_back

C++의 `vector`나 Java의 `ArrayList`는 용량이 꽉 차면 크기를 2배로 늘리는 방식으로 동작한다. 이 재할당은 O(N) 연산이지만, push_back 연산의 amortized 복잡도는 O(1)이다.

증명은 간단하다. N개를 push_back하는 동안 재할당은 $\log_2 N$번 일어나고, 각 재할당의 총 비용은 $1 + 2 + 4 + \ldots + N/2 + N = 2N - 1$이다. 즉 N번 연산에 총 $O(N)$이 들었으니, 연산당 평균은 O(1)이다.

이걸 개별 연산으로 보면 O(N)짜리 연산이 포함되어 있지만, 시퀀스 전체로 보면 O(1)이다. 이런 경우 "amortized O(1)"이라고 부른다.

### 해시맵의 리해싱

해시맵도 비슷하다. load factor가 임계값(보통 0.75)을 넘으면 테이블 크기를 2배로 늘리고 모든 원소를 재삽입한다. 이 리해싱은 O(N)이지만, 삽입 연산의 amortized 복잡도는 O(1)이다.

실무에서 이걸 모르고 있으면 "평균 O(1)"이라는 표현을 잘못 이해한다. 평균이 아니라 amortized다. 개별 연산은 O(N)이 될 수 있다. 실시간성이 중요한 시스템(게임 서버, 금융 거래)에서는 이 spike가 문제가 되기도 한다. 그래서 일부 시스템은 점진적 리해싱(incremental rehashing)을 쓴다. Redis가 대표적이다.

### 두 포인터의 합

두 포인터 알고리즘이 O(N)인 이유도 amortized로 설명된다. 매 스텝에서 i 또는 j가 증가하고, 둘 다 최대 N까지만 증가하므로 전체 연산은 2N 이하다. 개별 스텝을 보면 while 루프 안에 또 while 루프가 있어서 O(N^2)로 보이지만, amortized로 O(N)이다.

```python
def two_sum_sorted(arr, target):
    i, j = 0, len(arr) - 1
    while i < j:
        s = arr[i] + arr[j]
        if s == target:
            return (i, j)
        elif s < target:
            i += 1
        else:
            j -= 1
    return None
```

내부 루프 구조가 없으니 이 예제는 당연해 보이지만, 슬라이딩 윈도우에서는 더 복잡해진다.

## 재귀 복잡도 계산

재귀 알고리즘의 복잡도는 점화식으로 표현된다. $T(N) = a \cdot T(N/b) + f(N)$ 형태다. a는 부분 문제 개수, b는 입력 축소 비율, f(N)은 현재 레벨의 작업량이다. 이걸 푸는 방법은 세 가지다.

### 반복 대입법

점화식을 몇 번 전개해서 패턴을 찾는 방식이다. 예를 들어 병합 정렬은 $T(N) = 2T(N/2) + N$이다.

$T(N) = 2T(N/2) + N$  
$= 2(2T(N/4) + N/2) + N = 4T(N/4) + 2N$  
$= 4(2T(N/8) + N/4) + 2N = 8T(N/8) + 3N$  
$= \ldots = 2^k T(N/2^k) + kN$

$N/2^k = 1$에서 $k = \log_2 N$이 되고, $T(N) = N T(1) + N \log N = O(N \log N)$이다.

### 재귀 트리

점화식을 트리로 그려서 각 레벨의 작업량을 합산하는 방식이다. 병합 정렬의 경우 각 레벨에서 총 N의 작업을 하고, 레벨은 $\log N$개이므로 $O(N \log N)$이다.

### Master Theorem (마스터 정리)

$T(N) = a T(N/b) + f(N)$ 형태 점화식을 푸는 공식이다. $a \geq 1$, $b > 1$, $f(N)$이 점근적으로 양수일 때 세 가지 경우로 나뉜다.

$\log_b a$를 임계 지수라고 한다. 이 값과 $f(N)$의 증가율을 비교해서 판단한다.

**Case 1**: $f(N) = O(N^{\log_b a - \epsilon})$ (어떤 $\epsilon > 0$에 대해)  
→ $T(N) = \Theta(N^{\log_b a})$  
부분 문제들의 비용이 지배적인 경우

**Case 2**: $f(N) = \Theta(N^{\log_b a})$  
→ $T(N) = \Theta(N^{\log_b a} \log N)$  
부분 문제 비용과 합치는 비용이 같은 수준인 경우

**Case 3**: $f(N) = \Omega(N^{\log_b a + \epsilon})$ (어떤 $\epsilon > 0$에 대해) 그리고 regularity condition ($a f(N/b) \leq c f(N)$ for some $c < 1$)  
→ $T(N) = \Theta(f(N))$  
합치는 비용이 지배적인 경우

### Master Theorem 적용 예시

**병합 정렬**: $T(N) = 2T(N/2) + N$  
$a=2$, $b=2$, $f(N) = N$, $\log_b a = \log_2 2 = 1$, $N^1 = N$  
$f(N) = \Theta(N^1)$이므로 Case 2, $T(N) = \Theta(N \log N)$

**이진 탐색**: $T(N) = T(N/2) + 1$  
$a=1$, $b=2$, $f(N) = 1$, $\log_b a = 0$, $N^0 = 1$  
$f(N) = \Theta(N^0) = \Theta(1)$이므로 Case 2, $T(N) = \Theta(\log N)$

**단순 행렬 곱셈**: $T(N) = 8T(N/2) + N^2$ (N×N 행렬을 절반 크기로 쪼개 재귀)  
$a=8$, $b=2$, $\log_b a = 3$, $N^3$  
$f(N) = N^2 = O(N^{3-1})$이므로 Case 1, $T(N) = \Theta(N^3)$

**스트라센**: $T(N) = 7T(N/2) + N^2$  
$a=7$, $b=2$, $\log_b a = \log_2 7 \approx 2.807$  
$f(N) = N^2 = O(N^{2.807 - \epsilon})$이므로 Case 1, $T(N) = \Theta(N^{2.807})$

**선형 시간 재귀**: $T(N) = T(N/2) + N$  
$a=1$, $b=2$, $\log_b a = 0$  
$f(N) = N = \Omega(N^{0+1})$이고 regularity 만족 → Case 3, $T(N) = \Theta(N)$  
퀵소트의 partition 같은 연산이 이 형태다.

### Master Theorem이 안 되는 경우

점화식이 $T(N) = T(N-1) + N$ 같은 형태일 때는 Master Theorem을 쓸 수 없다. 이 점화식은 $T(N) = 1 + 2 + \ldots + N = O(N^2)$이다. 재귀 트리나 반복 대입으로 풀어야 한다.

또 $T(N) = 2T(N/2) + N \log N$ 같은 경우는 Case 2와 Case 3 사이의 틈에 떨어져서 기본 정리로는 안 된다. 확장된 Master Theorem(Akra-Bazzi)으로 $T(N) = \Theta(N \log^2 N)$을 얻는다.

## 재귀 복잡도 분석 실전 예시

### 피보나치 (naive)

```python
def fib(n):
    if n <= 1:
        return n
    return fib(n - 1) + fib(n - 2)
```

$T(N) = T(N-1) + T(N-2) + 1$  
이건 피보나치 수 자체와 비슷한 증가율이라 $T(N) = O(\phi^N) \approx O(1.618^N)$이다. 지수적으로 느리다. N=40만 되어도 10초 이상 걸린다.

### 피보나치 (메모이제이션)

```python
from functools import lru_cache

@lru_cache(maxsize=None)
def fib(n):
    if n <= 1:
        return n
    return fib(n - 1) + fib(n - 2)
```

동일한 부분 문제를 두 번 계산하지 않으므로 $T(N) = O(N)$이다. 상태 공간이 N개이고 각 상태 계산이 O(1)이다.

### 퀵소트

평균 $T(N) = 2T(N/2) + N = O(N \log N)$ (Master Theorem Case 2)  
최악 $T(N) = T(N-1) + N = O(N^2)$  
이미 정렬된 배열에 첫 원소를 pivot으로 잡으면 최악이 나온다. 그래서 randomized pivot이나 median-of-three를 쓴다.

### 재귀 DFS

트리/그래프에서 DFS의 복잡도는 방문하는 노드 수와 간선 수의 합이다. $O(V + E)$. 재귀로 구현하더라도 각 노드는 정확히 한 번씩만 방문되므로 Master Theorem을 쓸 필요가 없다.

## 공간복잡도

시간복잡도에 비해 덜 주목받지만, 공간복잡도는 메모리 제한이 있는 환경(임베디드, 모바일, 대용량 데이터 처리)에서 결정적이다.

### 알고리즘의 추가 공간

공간복잡도는 보통 입력을 제외한 추가 메모리를 기준으로 계산한다. 정렬 알고리즘을 예로 들면, 병합 정렬은 O(N) 추가 공간이 필요하지만 퀵소트는 O(log N)이다(재귀 스택). 둘 다 시간은 O(N log N)이지만 공간이 다르다.

in-place 알고리즘은 O(1) 추가 공간만 쓰는 알고리즘이다. 퀵소트, 힙소트, 버블소트가 여기 해당한다(재귀 스택은 예외로 치기도 하고 아니기도 하다).

### 재귀 호출 스택

재귀 함수는 호출 스택에 프레임을 쌓으므로 재귀 깊이만큼 공간이 든다. 파이썬은 기본적으로 1000 깊이에서 RecursionError가 난다. `sys.setrecursionlimit`으로 늘릴 수 있지만, OS의 스택 크기 제한(기본 8MB)에 걸린다.

```python
# 재귀 깊이 O(N)
def sum_recursive(n):
    if n == 0:
        return 0
    return n + sum_recursive(n - 1)

# 꼬리 재귀처럼 보이지만 파이썬은 최적화 안 함
```

DFS를 재귀로 구현하면 트리/그래프가 편향됐을 때 스택 오버플로우가 난다. 선형 리스트를 트리로 보면 N 깊이 재귀가 생긴다. 실무에서 DFS를 반복 버전(스택 자료구조)으로 바꿔야 하는 경우가 있다.

### 메모리 계층과 공간복잡도

공간복잡도 O(N)이라도 실제 메모리 사용량은 데이터 타입에 따라 다르다. Python의 int는 객체라서 28바이트씩 먹는다. C++의 `int`는 4바이트다. 같은 N=10^7 리스트라도 파이썬은 280MB, C++은 40MB다. 공간복잡도 이론은 동일하지만 실제 메모리는 7배 차이다.

## 언어별 자료구조 연산 복잡도

이론적인 자료구조 복잡도는 언어마다 다르다. 구현이 다르기 때문이다. 실무에서 자주 헷갈리는 것들을 정리한다.

### 배열 (Array/List)

| 연산 | Python list | C++ vector | Java ArrayList |
|------|-------------|------------|----------------|
| 인덱스 접근 | O(1) | O(1) | O(1) |
| 끝에 삽입 | amortized O(1) | amortized O(1) | amortized O(1) |
| 앞에 삽입 | O(N) | O(N) | O(N) |
| 중간 삽입 | O(N) | O(N) | O(N) |
| 끝에서 삭제 | O(1) | O(1) | O(1) |
| 길이 조회 | O(1) | O(1) | O(1) |

모든 언어에서 동적 배열의 기본 성능은 같다. 파이썬 list도 내부적으로 C의 동적 배열이다.

### 해시맵

| 연산 | Python dict | C++ unordered_map | Java HashMap |
|------|-------------|-------------------|--------------|
| 삽입 | amortized O(1) | amortized O(1) | amortized O(1) |
| 조회 | 평균 O(1), 최악 O(N) | 평균 O(1), 최악 O(N) | 평균 O(1), 최악 O(log N) Java 8+ |

Java 8부터 HashMap은 한 버킷에 8개 이상 들어가면 해당 버킷을 red-black tree로 바꿔서 최악도 O(log N)을 보장한다. Python과 C++은 여전히 최악 O(N)이다. 악의적인 해시 충돌 공격에 취약하다.

Python dict는 3.7부터 삽입 순서를 보장한다. 이건 복잡도는 아니지만 알고리즘 문제 풀 때 의외로 중요하다.

### 트리 기반 Map/Set

| 연산 | C++ map | Java TreeMap |
|------|---------|--------------|
| 삽입/삭제/조회 | O(log N) | O(log N) |
| 최솟값/최댓값 | O(log N) | O(log N) |
| 범위 순회 | O(log N + K) | O(log N + K) |

C++ `map`은 red-black tree다. `unordered_map`과 달리 순서 있는 순회가 가능하다. 파이썬은 표준 라이브러리에 트리 기반 자료구조가 없어서 `SortedList`나 `SortedDict`(sortedcontainers) 같은 외부 라이브러리를 쓴다.

### 연결리스트

Python은 기본 `list`가 동적 배열이라 연결리스트가 기본 자료구조에 없다. `collections.deque`가 양방향 연결리스트 역할을 한다.

| 연산 | Python deque | Java LinkedList | C++ list |
|------|--------------|-----------------|----------|
| 앞/뒤 삽입 | O(1) | O(1) | O(1) |
| 인덱스 접근 | O(N) | O(N) | O(N) |

Java `LinkedList`는 실무에서 잘 안 쓴다. 캐시 친화적이지 않아서 `ArrayList`보다 거의 항상 느리다. `LinkedList`를 쓰는 건 큐/덱 역할일 때 정도다.

### 우선순위 큐 (Heap)

| 연산 | Python heapq | C++ priority_queue | Java PriorityQueue |
|------|--------------|---------------------|---------------------|
| 삽입 | O(log N) | O(log N) | O(log N) |
| 최댓값/최솟값 조회 | O(1) | O(1) | O(1) |
| 최댓값/최솟값 삭제 | O(log N) | O(log N) | O(log N) |
| 힙 생성 | O(N) | O(N) | O(N) |

Python `heapq`는 최소 힙이다. 최대 힙을 쓰려면 음수로 넣어야 한다. 함정 포인트 중 하나다.

힙 생성이 O(N)인 건 의외인데, heapify 연산은 bottom-up으로 하면 O(N)이 된다. N번 삽입하면 O(N log N)이지만, 한 번에 heapify하면 O(N)이다. 이건 amortized가 아니라 진짜 O(N)이다.

### 문자열

| 연산 | Python | Java | C++ string |
|------|--------|------|------------|
| 접근 | O(1) | O(1) | O(1) |
| 길이 | O(1) | O(1) | O(1) |
| 연결(+) | O(N+M) | O(N+M) | O(N+M) |
| 부분 문자열 | O(K) | O(K) Java 6-, O(1) Java 7+ | O(K) |

Python과 Java의 문자열은 불변(immutable)이다. 문자열을 += 연산자로 반복해서 붙이면 매번 새 문자열을 만들어서 O(N^2)이 된다. 대신 `list`에 모아뒀다가 `''.join()`을 쓰는 게 O(N)이다.

```python
# O(N^2)
result = ""
for s in strings:
    result += s

# O(N)
result = "".join(strings)
```

Java도 `String` 연결은 O(N^2)이 되므로 `StringBuilder`를 써야 한다.

### Python 특화 함정

`in` 연산자의 복잡도가 자료구조에 따라 다르다. `list`에서 `in`은 O(N), `set`/`dict`에서 `in`은 평균 O(1)이다. 코드에 `if x in items`만 보고는 복잡도를 알 수 없다. items가 무엇인지가 중요하다.

```python
# O(N^2)
items = [1, 2, 3, ...]  # list
for x in queries:
    if x in items:  # O(N)
        ...

# O(N)
items = {1, 2, 3, ...}  # set
for x in queries:
    if x in items:  # O(1)
        ...
```

`list.pop(0)`은 O(N)이다. 앞쪽 원소를 빼면 뒤 원소들이 다 앞으로 밀린다. BFS 구현할 때 이걸 실수하면 O(V + E)여야 할 알고리즘이 O(V^2 + VE)가 된다. `collections.deque`의 `popleft()`를 써야 O(1)이다.

## 복잡도 분석 실전 팁

### 루프 안의 연산 놓치지 않기

이중 루프가 보이면 O(N^2)이라고 반사적으로 말하는 사람이 있는데, 내부 연산을 봐야 한다. 안쪽 루프 안에서 문자열 concat을 N번 한다면 O(N^3)이다. 해시맵 조회라면 O(N^2 · 1) = O(N^2)이다.

```python
# O(N^2 · M): substring이 O(M)
for i in range(n):
    for j in range(n):
        if s[i:j] == pattern:  # O(M)
            ...
```

Python 슬라이싱은 O(K)다. `s[i:j]`는 j-i 길이의 새 문자열을 만든다. 이게 루프 안에 있으면 전체 복잡도가 달라진다.

### 로그가 숨어 있는 경우

`set`에 넣고 빼는 연산은 평균 O(1)이지만, `TreeSet` 같은 정렬된 집합은 O(log N)이다. 자료구조 선택만으로 N log N과 N 차이가 난다.

파이썬 `bisect.insort`는 O(N)이다. 이진 탐색은 O(log N)이지만 삽입 자체가 배열 이동이라 O(N)이 든다. 이걸 O(log N)으로 착각하면 전체 복잡도가 크게 틀어진다.

### 입력 크기와 복잡도의 괴리

문자열 처리에서 입력 크기는 문자 수 M인데 문자열 개수를 N으로 잡으면 복잡도가 이상해진다. 입력 크기의 정의를 명확히 해야 한다.

그래프 문제에서 V(정점 수)와 E(간선 수)를 구분해야 한다. 밀집 그래프는 E = O(V^2)이고 희소 그래프는 E = O(V)다. O(V + E)와 O(V^2)는 경우에 따라 같고 경우에 따라 다르다.

### 실무 복잡도 프로파일링

이론적 복잡도 분석은 코드 작성 전에 하는 것이고, 실제 돌려보고 나서는 프로파일링을 해야 한다. Python은 `cProfile`, Java는 JProfiler나 async-profiler, C++은 gprof나 perf를 쓴다.

프로파일링으로만 알 수 있는 것들이 있다. GC 비용, JIT 워밍업, 캐시 미스, 잘못된 가설(내가 짠 O(N log N)이 사실 O(N^2)였던 경우). 복잡도 분석은 상한을 예측하는 것이고, 프로파일링은 실제 동작을 관찰하는 것이다. 둘 다 필요하다.

## 마무리

복잡도 분석은 알고리즘 문제 풀 때만 쓰는 게 아니라, 서비스 코드를 짤 때도 일상적으로 필요한 감각이다. 핵심은 입력 크기가 늘어났을 때 성능이 어떻게 변할지 예측하는 능력이다. 이론을 알아도 언어별 자료구조 구현 차이, 캐시, GC, JIT 같은 실제 환경 요인을 모르면 예측이 빗나간다.

Big-O는 상한이고 실제 성능은 상수에 좌우된다는 점, amortized analysis가 개별 연산의 최악과 다르다는 점, Master Theorem으로 재귀 복잡도를 기계적으로 풀 수 있다는 점, 언어별로 같은 자료구조 연산이라도 구현이 다르다는 점. 이것들을 실제 코드 예제와 함께 익혀두면 대부분의 실무 상황에서 복잡도 판단을 정확히 할 수 있다.
