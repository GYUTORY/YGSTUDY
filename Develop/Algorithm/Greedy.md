# Greedy Algorithm (그리디 알고리즘)

## 들어가며

그리디는 알고리즘 중에 가장 무섭다. 코드 자체는 10줄도 안 되고, 정답처럼 보이는 답이 너무 쉽게 나와서 "이게 맞겠지" 하고 넘겼다가 반례 하나에 무너진다. DP는 점화식이 안 나오면 아예 풀 수가 없어서 막힌 게 명확한데, 그리디는 틀린 코드가 정답처럼 굴러간다. 반례가 나올 때까지는 내 코드가 틀렸다는 사실조차 모른다.

5년 동안 실무와 사이드 프로젝트를 하면서 가장 많이 당한 알고리즘이 그리디다. "매 시점마다 가장 좋아 보이는 선택을 하면 결과도 최선이겠지"라는 직관이 자연스러워서, 검증 없이 구현부터 시작하는 경우가 많다. 그리고 보통 50~70% 정도의 테스트 케이스는 통과한다. 나머지를 반례로 놓치는 게 그리디의 전형적인 실패 패턴이다.

그리디 알고리즘은 매 단계에서 그 순간의 최적해(local optimum)를 선택해서, 전체의 최적해(global optimum)에 도달하려는 접근법이다. 모든 경우를 탐색하지 않기 때문에 브루트포스보다 훨씬 빠르고, DP처럼 상태 공간을 전부 채우지 않아도 된다. 하지만 이 "빠르다"가 성립하려면 "지금의 최선이 나중의 최선을 망치지 않는다"는 강한 가정이 필요하다. 그 가정이 성립하는 문제가 생각보다 많지 않다.

## 그리디가 성립하려면 필요한 두 가지 성질

### 그리디 선택 속성 (Greedy Choice Property)

현재 시점에서 가장 좋아 보이는 선택이, 전체 최적해의 일부가 될 수 있어야 한다. 다르게 말하면, 지금의 선택 때문에 나중에 더 좋은 선택이 막히는 일이 없어야 한다. 이게 그리디의 핵심이자, 대부분의 사람들이 그리디에 실패하는 이유다. "좋아 보이는" 기준이 뭐냐에 따라 이 속성이 성립할 수도, 성립하지 않을 수도 있는데, 이 기준을 잘못 잡으면 반례가 나온다.

예를 들어 회의실 배정 문제에서 "가장 짧은 회의부터 배정한다"는 기준은 틀렸다. 긴 회의 두 개를 방해하는 중간 길이 회의가 있으면 최적해를 놓친다. 반면 "가장 일찍 끝나는 회의부터 배정한다"는 기준은 성립한다. 같은 문제에서도 기준을 어떻게 잡느냐에 따라 그리디가 맞기도 하고 틀리기도 한다.

### 최적 부분 구조 (Optimal Substructure)

큰 문제의 최적해가 작은 부분 문제의 최적해로 만들어져야 한다. 이건 DP와 공유하는 속성이다. 그리디는 여기에 추가로 그리디 선택 속성까지 요구한다. 즉 최적 부분 구조만으로는 DP는 되지만 그리디는 안 된다. 최적 부분 구조가 있고, 그 부분 문제를 푸는 시작점이 "매번 같은 기준으로 뽑을 수 있다"면 그리디가 가능한 경우가 많다.

실무에서 "이 문제 그리디로 풀릴까?"를 판단할 때 나는 이 순서로 생각한다. 먼저 부분 문제로 쪼갤 수 있는지 본다. 그다음 매 단계에서 한 가지 기준으로 선택을 고정했을 때, 그 선택이 남은 부분 문제의 최적해를 막지 않는지 반례를 찾아본다. 반례가 금방 떠오르면 그리디는 포기하고 DP나 다른 접근을 쓴다. 반례가 안 떠오르면 그때부터 증명을 시도한다.

## 그리디가 통하는 문제와 안 통하는 문제의 경계

가장 흔한 함정은 거스름돈 문제다. "큰 단위 동전부터 먼저 준다"는 직관은 한국 원화나 미국 달러처럼 동전 체계가 잘 설계된 경우에만 맞다. 500/100/50/10원은 그리디가 통한다. 하지만 1/7/10원짜리만 있다고 하면, 14원을 거슬러줄 때 그리디는 10+1+1+1+1 = 5개를 내놓지만 최적해는 7+7 = 2개다. 동전 체계(coin system)가 canonical이냐에 따라 그리디 성립 여부가 갈린다. 실무에서 이걸 모르고 "동전 거스름돈 = 그리디"라고 외워뒀다가 면접에서 당한 사람을 여럿 봤다.

반대로 회의실 배정, 작업 스케줄링, 최소 신장 트리(Kruskal, Prim), 허프만 코딩, 활동 선택 문제는 그리디가 통한다. 공통점은 매 단계의 선택 기준이 명확하고, 그 선택이 나머지 문제의 구조를 훼손하지 않는다는 점이다.

그리디가 안 통하는 대표 예는 0/1 배낭 문제다. 분수 배낭(fractional knapsack)은 "가치/무게 비율이 높은 것부터 넣는다"로 그리디가 성립하지만, 쪼갤 수 없는 0/1 배낭은 이 기준이 반례를 만든다. 가치/무게 비율이 가장 높은 물건을 먼저 넣으면 남은 공간에 더 큰 가치의 물건이 못 들어가는 경우가 생긴다. 이건 DP로 풀어야 한다.

판별 요령은 이렇다. "지금 고른 것을 나중에 되돌릴 필요가 있나?"를 생각한다. 되돌릴 필요가 없으면 그리디 후보다. 되돌릴 가능성이 있으면 DP나 백트래킹을 쓴다. 또 하나는 "부분 문제의 수가 매우 많거나 상태 의존적인가?"다. 그렇다면 DP, 아니라면 그리디가 유리하다.

## 교환 논증 (Exchange Argument)

그리디 알고리즘의 정당성을 증명할 때 가장 자주 쓰는 방법이 교환 논증이다. 아이디어는 단순하다. 그리디가 고른 해와, 최적해가 있다고 가정했을 때 그 두 해의 차이를 만드는 선택을 하나씩 교환해도 해의 품질이 떨어지지 않는다는 것을 보인다. 그렇다면 그리디 해는 최적해와 같거나 더 나쁘지 않다. 이게 그리디 정답의 구조다.

회의실 배정을 예로 들면, 그리디는 "가장 일찍 끝나는 회의 A"를 먼저 선택한다. 최적해 OPT가 A가 아닌 다른 회의 B로 시작한다고 가정하자. A의 종료 시간은 B의 종료 시간보다 빠르거나 같다(그리디가 그렇게 골랐으니까). 그러면 OPT의 B를 A로 바꿔도 남은 회의들과 충돌하지 않는다. A가 더 일찍 끝나므로, B 다음에 올 수 있던 회의들은 A 다음에도 전부 올 수 있기 때문이다. 결국 OPT와 같은 크기의 해를 만들 수 있고, 이걸 반복하면 그리디 해가 OPT와 크기가 같다는 결론이 나온다.

교환 논증은 코드를 짜기 전에 머릿속에서 한 번 돌려보는 습관을 들이면 좋다. "내가 고른 걸 다른 걸로 바꿨을 때 더 나빠지지 않음을 보일 수 있나?"를 스스로 물어본다. 이게 막히면 그리디가 안 되는 문제일 가능성이 높다. 면접에서 그리디 문제를 풀고 나면 "왜 그게 최적인가?"를 꼭 물어보는데, 여기서 "그냥 직관적으로 맞는 것 같아요"라고 하면 감점이다. 교환 논증이 정석 답변이다.

## DP와의 경계

DP와 그리디는 둘 다 최적 부분 구조를 요구한다. 차이는 그리디 선택 속성의 유무다. DP는 모든 선택지를 다 고려해서 그중 최선을 뽑는다. 그리디는 고려 없이 한 가지만 고른다. 그래서 그리디가 통하면 DP보다 훨씬 빠르다. 그리디는 보통 O(n log n)이면 끝나는데, 같은 문제를 DP로 풀면 O(n²)이나 O(n·W) 같은 복잡도가 된다.

반대로 그리디가 틀리는 문제를 그리디로 풀려고 하면 반례에 당한다. 그래서 실무에서 쓰는 규칙은 "그리디는 증명 없이 쓰지 않는다"이다. 코딩테스트에서 시간이 없을 때는 어쩔 수 없이 직관에 의존하지만, 실무 코드로 들어가면 다르다. 잘못된 그리디가 프로덕션에 올라가면, 전체 테스트는 통과해도 특정 입력에서 조용히 틀린 결과를 내놓는다. 이게 제일 무서운 버그다.

내가 쓰는 판단 순서는 이렇다. 먼저 그리디 후보가 보이면 반례를 만들어보려고 시도한다. 5분 안에 반례가 안 나오면 교환 논증을 머릿속으로 돌린다. 교환 논증이 막히면 DP로 넘어간다. 교환 논증이 성립하면 그때 코드를 짠다. 이 순서를 지키면 틀린 그리디를 쓰는 경우가 확실히 줄어든다.

## 거스름돈 문제

```java
public int minCoins(int amount, int[] coins) {
    Arrays.sort(coins);
    int count = 0;
    for (int i = coins.length - 1; i >= 0; i--) {
        while (amount >= coins[i]) {
            amount -= coins[i];
            count++;
        }
    }
    return amount == 0 ? count : -1;
}
```

동전 체계가 `{500, 100, 50, 10}` 같은 canonical system이면 이 코드가 최적이다. 하지만 `{1, 7, 10}` 같은 임의 체계에서는 틀린 답을 낸다. 14원을 거슬러주는 경우 그리디는 `10 + 1 + 1 + 1 + 1 = 5개`를 반환하지만, 최적은 `7 + 7 = 2개`다. 실무에서 새로운 화폐 체계(예: 쿠폰, 포인트 차감 단위)를 다룰 때 "당연히 그리디로 되겠지" 하고 넘어가면 이런 버그가 생긴다. 임의 체계라면 DP로 풀어야 한다.

## 회의실 배정 문제

하루에 여러 회의가 있고, 서로 시간이 겹치지 않게 최대한 많은 회의를 배정하는 문제다. 기준은 "종료 시간이 빠른 회의부터 고른다"이다.

```java
public int maxMeetings(int[][] meetings) {
    Arrays.sort(meetings, (a, b) -> a[1] - b[1]);
    int count = 0;
    int lastEnd = Integer.MIN_VALUE;
    for (int[] meeting : meetings) {
        if (meeting[0] >= lastEnd) {
            count++;
            lastEnd = meeting[1];
        }
    }
    return count;
}
```

시작 시간 기준, 회의 길이 기준은 전부 반례가 있다. 종료 시간 기준만 교환 논증으로 증명된다. 이 문제를 처음 본다면 세 가지 기준을 다 시도해보고 각각 반례를 만들어보길 추천한다. 왜 종료 시간 기준만 맞는지 체감하는 게 그리디를 이해하는 데 도움이 된다.

## 작업 스케줄링 (Deadline Scheduling)

각 작업에 처리 시간과 마감 기한이 있고, 마감을 넘기면 페널티가 발생한다. 페널티를 최소화하거나, 마감 내 끝낼 수 있는 작업 수를 최대화하는 문제다. 변형이 많지만 기본 아이디어는 "마감이 빠른 것부터" 혹은 "보상이 큰 것부터 우선순위 큐로 관리"다.

```java
public int scheduleJobs(int[][] jobs) {
    Arrays.sort(jobs, (a, b) -> a[1] - b[1]);
    PriorityQueue<Integer> pq = new PriorityQueue<>();
    int time = 0;
    for (int[] job : jobs) {
        int duration = job[0];
        int deadline = job[1];
        pq.offer(duration);
        time += duration;
        if (time > deadline) {
            time -= pq.poll();
        }
    }
    return pq.size();
}
```

마감이 빠른 순으로 정렬하고, 일단 작업을 큐에 넣는다. 현재 시각이 마감을 넘어가면 가장 오래 걸리는 작업 하나를 버린다. 이 "오래 걸리는 걸 버린다"가 교환 논증 포인트다. 버릴 작업을 바꿔도 결과가 더 나빠지지 않는다는 걸 증명할 수 있기 때문에 그리디가 성립한다.

## 허프만 코딩 (Huffman Coding)

문자 빈도를 기반으로 가변 길이 접두어 코드를 만들어서 전체 길이를 최소화하는 알고리즘이다. gzip, JPEG, MP3 내부에서 쓰인다. 기준은 "빈도가 가장 낮은 두 개를 합친다"이다.

```java
public Map<Character, String> huffmanCodes(Map<Character, Integer> freq) {
    PriorityQueue<Node> pq = new PriorityQueue<>((a, b) -> a.freq - b.freq);
    for (Map.Entry<Character, Integer> e : freq.entrySet()) {
        pq.offer(new Node(e.getKey(), e.getValue()));
    }
    while (pq.size() > 1) {
        Node left = pq.poll();
        Node right = pq.poll();
        Node merged = new Node('\0', left.freq + right.freq);
        merged.left = left;
        merged.right = right;
        pq.offer(merged);
    }
    Map<Character, String> codes = new HashMap<>();
    buildCodes(pq.poll(), "", codes);
    return codes;
}
```

허프만은 그리디의 교과서적 예제다. 빈도가 가장 낮은 두 노드가 가장 깊은 위치에 있어야 전체 길이가 최소가 된다는 게 핵심 보조정리다. 이것도 교환 논증으로 증명된다. 실제로 이 증명을 한 번 따라가 보면, 그리디 증명이 왜 "선택을 바꿔도 나빠지지 않는다"는 구조를 반복하는지 체감할 수 있다.

## 실무에서 그리디에 당한 경험담

실무에서 그리디를 섣불리 쓰다가 당했던 사례 몇 가지를 적는다.

첫 번째는 트래픽 분배 로직이다. 서버 여러 대에 요청을 분배하는데, 초기에는 "가장 한가한 서버로 보낸다"는 그리디로 구현했다. 평상시에는 문제가 없었는데, 특정 서버가 짧게 멈췄다가 복구되는 순간에 문제가 생겼다. 그 서버가 "가장 한가해" 보여서 트래픽이 한꺼번에 몰렸고, 곧바로 다시 쓰러졌다. 이 현상을 thundering herd라고 부르는데, 매 순간의 최적 선택이 전체 시점에서는 최악이 된 예다. 해결은 라운드로빈과 가중치 기반으로 바꿨다. 지금 시점의 "가장 좋아 보이는 선택"이 항상 정답은 아니다.

두 번째는 빌링 할인 계산이다. 한 주문에 여러 쿠폰을 적용할 수 있는데, "할인율이 큰 것부터 먼저 적용"하는 그리디를 썼다. 실제로는 쿠폰마다 최소 주문 금액 조건이 달라서, 큰 할인율을 먼저 적용하면 주문 금액이 줄어들어 그다음 쿠폰의 최소 금액 조건에 미달하는 경우가 있었다. 고객이 "할인이 덜 됐다"고 문의를 넣어서야 알게 됐다. 이건 DP나 조합 탐색으로 풀어야 했다. 내 직관이 반례를 못 만들었을 뿐, 반례가 없는 게 아니었다.

세 번째는 캐시 퇴출 정책이다. LRU 구현 중에 "최근 접근 시각이 가장 오래된 것을 퇴출"하는 그리디를 기본으로 쓴다. 이건 일반적으로 잘 동작하는데, 스캔 패턴(한 번 읽고 말 데이터가 대량으로 들어오는 경우)에서 캐시가 전부 날아간다. 이 문제를 피하려고 LRU-K나 ARC 같은 알고리즘이 나왔다. 그리디 자체가 틀렸다기보다는, "접근 시각"이라는 단일 지표가 모든 상황에서 최선의 선택 기준은 아니었다는 얘기다.

세 사례의 공통점은 "반례를 찾기 전에 구현부터 했다"는 것이다. 5분만이라도 반례를 의식적으로 만들어보는 습관이 있었다면 프로덕션 전에 걸렀을 문제들이다. 지금은 그리디 코드를 짤 때 주석으로 "이 기준이 성립하는 이유"를 한 줄이라도 적어두려고 한다. 적을 수 없으면 그리디가 아닌 것이고, 다른 접근을 찾아야 한다는 신호다.

## 정리

그리디는 빠르고 간결하지만, 틀린 답을 정답처럼 돌려주는 치명적 성질이 있다. 그래서 쓰기 전에 반드시 두 가지를 확인해야 한다. 그리디 선택 속성이 성립하는지, 그리고 교환 논증으로 그걸 증명할 수 있는지. 증명이 안 되면 DP로 넘어간다. 코딩테스트에서는 시간 제약상 직관을 믿고 제출하는 경우도 있지만, 실무에서는 그리디야말로 가장 의심해야 할 알고리즘이다. "이렇게 하면 되겠지"라는 생각이 가장 먼저 드는 알고리즘일수록, 그리디인지 아닌지부터 의심하는 습관이 반례에 당하지 않는 유일한 방법이다.
