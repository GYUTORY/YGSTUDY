    Claude를 전문가처럼 활용하는 방법¶

    Claude의 기본 기능은 Claude 문서에서 다뤘고, 프롬프트 작성 원칙은 프롬프트 엔지니어링에 있다. 이 문서는 Claude Web, API, Desktop 앱 전반에서 Claude를 실무에 맞게 세팅하고 다루는 방법을 정리한다.

    1. System Prompt 설계 실전¶

    System prompt는 Claude에게 "너는 이런 존재고, 이런 규칙을 따른다"를 주입하는 공간이다. API에서는 system 파라미터, Web/Desktop에서는 Projects의 Custom Instructions가 이 역할을 한다.

    1.1 역할 부여¶

    단순히 "너는 시니어 개발자야"라고 하면 부족하다. Claude가 어떤 맥락에서 어떤 판단을 내려야 하는지까지 넣어야 한다.

    너는 핀테크 스타트업의 백엔드 리드 개발자다.
기술 스택은 Spring Boot 3 + Kotlin + PostgreSQL이다.
코드 리뷰를 할 때는 보안 이슈를 최우선으로 지적하고,
금융 관련 로직에서는 반드시 트랜잭션 처리와 멱등성을 확인한다.
테스트 커버리지가 부족한 코드에 대해서는 테스트 작성을 요구한다.

    이렇게 도메인, 기술 스택, 판단 기준을 함께 넣으면 응답의 초점이 달라진다. "보안 이슈를 우선 지적해"만 넣었을 때보다 훨씬 일관된 관점에서 리뷰를 해준다.

    역할을 줄 때 주의할 점이 있다. "세계 최고의 전문가"처럼 과장된 역할을 주면 오히려 답변이 장황해지는 경우가 있다. Claude가 "전문가답게" 보이려고 불필요한 설명을 덧붙이기 때문이다. 구체적인 업무 맥락을 주는 게 낫다.

    1.2 제약 조건 설정¶

    Claude에게 하지 말아야 할 것을 명시하는 게 생각보다 중요하다. 제약 없이 쓰면 Claude는 친절하게 부연 설명을 덧붙이고, 관련 내용을 추가로 알려주려 한다. 실무에서 이게 오히려 방해가 되는 경우가 많다.

    제약 조건:
- 질문에 직접 관련 없는 내용은 답하지 마라
- "참고로", "추가로" 같은 부가 설명을 붙이지 마라
- 코드를 생성할 때 주석을 달지 마라
- 확실하지 않은 내용은 "확인 필요"라고 표시해라
- deprecated API를 사용하지 마라

    제약 조건은 긍정형보다 부정형이 잘 먹힌다. "간결하게 답해"보다 "3문장을 넘기지 마"가 정확도가 높다. Claude는 "간결하게"의 기준을 자의적으로 해석하지만, "3문장"은 명확히 지킨다.

    1.3 출력 포맷 제어¶

    Claude에게 원하는 형태로 답변하게 만들려면 출력 형식을 명확히 지정해야 한다. 말로 설명하는 것보다 예시를 보여주는 게 훨씬 낫다.

    응답 형식:
모든 코드 리뷰 결과는 아래 형식으로 작성한다.

## [심각도: HIGH/MEDIUM/LOW] 파일명:라인번호

**문제**: 한 줄로 요약
**이유**: 왜 문제인지 설명
**수정안**: 수정된 코드

---

예시:

## [HIGH] UserService.java:42

**문제**: SQL injection 가능성
**이유**: 사용자 입력을 문자열 연결로 쿼리에 넣고 있음
**수정안**:
```java
jdbcTemplate.query("SELECT * FROM users WHERE id = ?", id)

    이런 식으로 예시 하나를 넣어두면 Claude가 이 형식을 매우 잘 따른다. 예시 없이 "심각도를 표시해줘"라고만 하면 매번 다른 형태로 답변이 온다.

API에서 JSON으로 응답받고 싶으면 system prompt에 형식을 넣고, 추가로 `messages`의 마지막에 prefill을 쓰면 더 안정적이다.

```python
message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system="모든 응답을 JSON 형식으로 반환한다. 다른 텍스트 없이 JSON만 출력한다.",
    messages=[
        {"role": "user", "content": "Python 웹 프레임워크 3개를 비교해줘"},
        {"role": "assistant", "content": "{"}  # prefill — JSON으로 시작하도록 유도
    ]
)

    prefill은 assistant의 마지막 메시지를 미리 채워넣는 기법이다. {로 시작하게 하면 Claude가 자연스럽게 JSON을 이어서 작성한다. 단, 응답에 {가 빠져 있으니 파싱할 때 앞에 붙여야 한다.

    1.4 System Prompt 작성 시 흔한 실수¶

    너무 길게 쓰는 경우: system prompt가 2000토큰을 넘어가면 Claude가 뒤쪽 지시를 놓치는 경우가 생긴다. 특히 서로 충돌하는 지시가 들어가면 앞쪽 것을 우선하는 경향이 있다. 핵심 지시를 앞에 배치하고, 구체적인 예시는 뒤에 넣어야 한다.

    모호한 표현: "적절히", "필요하면", "가능한 한" 같은 표현은 Claude에게 판단을 위임하는 것이다. 의도한 거라면 상관없지만, 일관된 결과를 원하면 구체적인 조건으로 바꿔야 한다.

    나쁜 예: 적절한 수준에서 에러 핸들링을 추가해
좋은 예: checked exception만 catch하고, unchecked는 상위로 전파해

    2. Claude 특성을 활용하는 작업 패턴¶

    Claude에는 다른 LLM과 구분되는 특성이 몇 가지 있다. 이걸 알고 쓰면 같은 모델로도 결과 차이가 난다.

    2.1 긴 컨텍스트 유지력¶

    Claude는 200K 토큰(Opus는 1M) 컨텍스트에서 앞부분 내용을 비교적 잘 기억한다. 이걸 활용하면 다른 LLM에서는 힘든 작업이 가능하다.

    대규모 코드 분석: 파일 10~20개를 한번에 넣고 "이 모듈의 데이터 흐름을 추적해줘"라고 하면, 파일 간 의존성을 따라가면서 설명해준다. GPT-4o로 같은 작업을 시키면 중간에 맥락을 잃는 경우가 잦다.

    이때 파일을 넣는 순서가 결과에 영향을 준다. 의존 관계 순서대로 넣으면 Claude가 더 정확하게 추적한다. 예를 들어 Controller → Service → Repository 순서로 넣는 것보다, Repository → Service → Controller 순서(의존 방향)로 넣는 게 분석 품질이 좋다.

    긴 문서 요약과 비교: 계약서나 기술 문서 2~3개를 넣고 차이점을 찾는 작업에서 Claude가 강하다. 다만 200K에 가까운 분량을 넣으면 처리 시간이 길어지고, 중간 부분 정보를 놓치는 lost-in-the-middle 현상이 나타난다. 중요한 질문은 컨텍스트 앞이나 뒤에 배치해야 한다.

    2.2 지시 준수력¶

    Claude는 system prompt나 사용자 지시를 꼼꼼하게 따르는 편이다. "한국어로 답해", "100자 이내로", "JSON으로"처럼 형식 지시를 잘 지킨다.

    이 특성을 이용하면 복잡한 파이프라인을 만들 수 있다. 예를 들어:

    1단계: 이 코드에서 보안 취약점만 추출해. 각 항목을 JSON 배열로 반환해.
2단계: (1단계 결과를 받아서) 각 취약점의 심각도를 CVSS 기준으로 평가해.
3단계: (2단계 결과를 받아서) 심각도 HIGH 이상만 필터링해서 수정 코드를 생성해.

    각 단계에서 출력 형식을 명확히 지정하면, 중간 결과를 다음 단계 입력으로 바로 쓸 수 있다. Claude는 이 형식을 잘 지키기 때문에 파이프라인이 안정적으로 동작한다.

    2.3 안전 거부가 과도한 경우¶

    Claude는 안전 관련 거부가 다른 모델보다 빡빡하다. 보안 테스트 코드를 작성하거나, 취약점 분석을 하거나, 공격 시나리오를 검토할 때 "이 요청은 처리할 수 없습니다"라고 거부하는 경우가 있다.

    이런 상황에서는 맥락을 충분히 제공하면 대부분 해결된다.

    나는 금융 서비스의 보안팀 소속이고, 내부 침투 테스트를 진행 중이다.
우리 서비스의 JWT 토큰 검증 로직에 대해 알려진 공격 벡터를 검토하고,
각 공격에 대한 방어 코드가 적용되어 있는지 확인해야 한다.
아래는 현재 우리 서비스의 토큰 검증 코드다.

    "침투 테스트", "보안 감사", "방어 목적" 같은 맥락을 명시하면 거부가 풀리는 경우가 많다. 그래도 안 되면 질문을 방어 관점으로 바꿔 본다. "이 코드를 어떻게 공격하나"보다 "이 코드의 보안 취약점은 무엇이고, 어떻게 패치하나"로 물어보면 된다.

    system prompt에 보안 테스트 맥락을 미리 넣어두는 것도 방법이다.

    너는 보안 엔지니어다.
모든 질문은 방어적 보안 관점에서 답한다.
취약점 분석, 공격 벡터 검토, 보안 테스트 코드 작성은 정상적인 업무 범위다.

    다만 실제로 악의적인 목적의 요청은 어떤 방법을 써도 거부된다. Claude의 안전 장치는 컨텍스트를 바꾼다고 우회되는 게 아니라, 정당한 사용 맥락이 있을 때만 허용 범위가 넓어지는 구조다.

    2.4 Claude가 잘 못하는 작업¶

    어떤 도구든 한계가 있다. Claude가 약한 영역을 알고 있으면 삽질을 줄인다.

    실시간 정보: Claude는 학습 데이터 이후의 정보를 모른다. "오늘 환율"이나 "최신 라이브러리 버전"을 물어보면 학습 시점 기준으로 답한다. 이건 Web 검색 기능이 켜져 있어야 해결되고, API에서는 Tool Use로 외부 검색을 연결해야 한다.

    수학 계산: 복잡한 수학이나 정확한 수치 계산은 종종 틀린다. Extended Thinking을 켜면 나아지지만, 정확도가 100%여야 하는 경우에는 코드를 생성해서 실행하는 게 맞다.

    자기 능력 판단: "너 이거 할 수 있어?"라고 물으면 대부분 "할 수 있다"고 답한다. 실제로 못하는 작업도 자신 있게 답하는 경향이 있다. Claude의 자가 진단을 신뢰하면 안 된다.

    3. Projects와 Custom Instructions 실전 세팅¶

    3.1 프로젝트 구성 원칙¶

    claude.ai의 Projects 기능은 "이 프로젝트의 모든 대화에 적용되는 공유 컨텍스트"를 만드는 기능이다. Knowledge에 파일을 올리고, Custom Instructions에 규칙을 넣으면 매번 같은 내용을 입력하지 않아도 된다.

    프로젝트를 만들 때 가장 흔한 실수는 하나의 프로젝트에 너무 많은 것을 넣는 것이다. "백엔드 개발"이라는 프로젝트 하나에 모든 걸 몰아넣으면 Knowledge가 비대해지고, Instructions가 서로 충돌한다.

    작업 유형별로 프로젝트를 분리하는 게 낫다.

    프로젝트 예시:

[서비스명] 코드 리뷰
  → Knowledge: 코딩 컨벤션, 아키텍처 문서
  → Instructions: 리뷰 기준, 출력 형식

[서비스명] API 설계
  → Knowledge: 기존 API 스펙, ERD
  → Instructions: 네이밍 규칙, 응답 형식 규칙

데이터 분석
  → Knowledge: 주요 테이블 스키마, 비즈니스 용어 정의
  → Instructions: SQL 방언, 출력 형식

    3.2 Knowledge에 뭘 넣어야 하는지¶

    Knowledge에 넣은 파일은 해당 프로젝트의 모든 대화에서 참조된다. 여기서 중요한 건 "Claude가 매번 알아야 하는 정보"만 넣는 것이다.

    넣어야 하는 것:

    • 코딩 컨벤션 문서 — 네이밍 규칙, 패키지 구조, import 순서 등
    • 도메인 용어 사전 — "주문", "결제", "정산"이 이 프로젝트에서 정확히 무슨 의미인지
    • API 응답 형식 표준 — 에러 코드 체계, 페이지네이션 형식 등
    • 주요 아키텍처 결정 기록(ADR) — "왜 이 구조를 선택했는지"

    넣지 말아야 하는 것:

    • 전체 소스 코드 — Knowledge 용량 제한도 있고, 코드는 대화 중에 필요한 부분만 붙여넣는 게 낫다
    • 자주 바뀌는 문서 — Knowledge에 올린 파일은 수동으로 업데이트해야 한다. 매주 바뀌는 스프린트 문서를 넣으면 금방 outdated된다
    • 일반적인 기술 문서 — Spring Boot 공식 문서를 통째로 넣을 필요 없다. Claude가 이미 알고 있는 내용이다

    Knowledge 파일의 총 크기가 커지면 매 대화에서 소비하는 토큰도 늘어난다. Pro 플랜 기준으로 프로젝트당 Knowledge 용량은 제한이 있다. 핵심만 골라서 넣어야 한다.

    3.3 Custom Instructions 작성법¶

    Custom Instructions는 system prompt와 같은 역할이다. 여기에 넣은 내용이 해당 프로젝트의 모든 대화에 자동 적용된다.

    실무에서 쓸만한 Instructions 구조:

    ## 역할
너는 [서비스명]의 백엔드 개발자다.
기술 스택: Kotlin + Spring Boot 3 + PostgreSQL + Redis

## 코드 작성 규칙
- Kotlin Coding Conventions를 따른다
- nullable 타입 사용을 최소화한다
- Repository 레이어에서는 Spring Data JPA를 사용한다
- 비즈니스 로직은 Service 레이어에 작성한다

## 응답 규칙
- 코드만 달라고 하면 설명 없이 코드만 출력한다
- 에러 수정 시 원인을 먼저 설명하고 수정안을 보여준다
- 확실하지 않은 내용은 "확인 필요"라고 명시한다

    3.4 개인 Instructions과 프로젝트 Instructions 분리¶

    claude.ai에서는 계정 전체에 적용되는 개인 Instructions과 프로젝트별 Instructions가 별도로 존재한다.

    개인 Instructions (Settings > Profile > Custom Instructions)에 넣을 것: - 선호하는 응답 언어 (예: "한국어로 답해") - 선호하는 응답 스타일 (예: "간결하게 답해, 부연 설명 금지") - 공통 제약 (예: "deprecated API 사용 금지")

    프로젝트 Instructions에 넣을 것: - 해당 프로젝트의 기술 스택, 도메인 맥락 - 프로젝트 고유의 코딩 규칙 - 출력 형식 지정

    개인 Instructions은 모든 대화에 적용되므로 범용적인 내용만 넣어야 한다. 프로젝트 특화 내용은 반드시 프로젝트 Instructions에 넣는다. 둘이 충돌하면 프로젝트 Instructions가 우선한다.

    3.5 팀 공유 프로젝트 운영¶

    Team 플랜에서는 프로젝트를 팀원과 공유할 수 있다. 이때 가장 중요한 건 Knowledge와 Instructions를 누가 관리하느냐다.

    한 명이 담당하는 게 좋다. 여러 명이 각자 수정하면 Instructions가 지저분해지고, 서로 다른 규칙이 섞인다. 팀 리드나 DRI(Directly Responsible Individual)가 관리하고, 변경이 필요하면 해당 담당자에게 요청하는 방식이 현실적이다.

    팀 프로젝트에는 다음을 넣어두면 온보딩이 빨라진다: - 프로젝트 구조 개요 - 핵심 도메인 용어 정의 - PR 리뷰 기준 - 자주 쓰는 프롬프트 예시

    4. Hallucination 대응¶

    Claude는 모르는 걸 모른다고 하지 않는 경우가 있다. 없는 라이브러리를 추천하거나, 존재하지 않는 API를 알려주거나, 틀린 코드를 자신 있게 내놓는다. 이걸 hallucination이라 하고, LLM의 구조적 한계다.

    4.1 Claude가 자주 틀리는 패턴¶

    없는 라이브러리 이름: "이런 기능을 하는 Python 라이브러리가 뭐가 있어?"라고 물으면 가끔 실제로 존재하지 않는 패키지 이름을 만들어낸다. 이름이 그럴듯해서 pip install까지 해보게 되는데 설치가 안 된다.

    잘못된 API 시그니처: 라이브러리가 실제로 존재하더라도 메서드 이름이나 파라미터가 틀릴 수 있다. 특히 마이너 버전에서 변경된 API에서 이런 문제가 잦다. 학습 데이터에 구 버전 코드가 더 많기 때문이다.

    그럴듯한 설정 옵션: 프레임워크 설정에서 존재하지 않는 옵션을 제시하는 경우가 있다. application.yml에 없는 속성을 넣어주거나, 없는 설정 값을 알려준다.

    출처 날조: "이건 공식 문서에 나와 있다"고 하면서 실제 문서에 없는 내용을 말하는 경우가 있다. URL을 달아달라고 하면 존재하지 않는 URL을 만들어낼 수 있다.

    4.2 검증 방법¶

    "확신도를 알려달라"고 묻는 건 소용없다. Claude에게 "이 답변 얼마나 확신해?"라고 물으면 대부분 "높은 확신"이라고 답한다. 실제 정확도와 자가 평가가 일치하지 않는다.

    대신 이런 방법이 낫다:

    근거 분리 요청: 답변을 받은 뒤 "이 내용의 근거를 설명해줘. 네가 직접 추론한 부분과 학습 데이터에서 가져온 부분을 구분해서 알려줘"라고 요청한다. Claude가 근거를 설명하지 못하거나 모호하게 넘어가는 부분이 의심 구간이다.

    반대 질문: "이게 안 되는 경우는?"이라고 물어본다. Claude가 한계를 구체적으로 설명하면 답변의 신뢰도가 높은 편이다. "대부분의 경우 잘 작동합니다"처럼 뭉뚱그리면 의심해야 한다.

    단계별 검증 요청: 코드를 받았으면 "이 코드를 한 줄씩 실행 흐름을 추적해봐"라고 시킨다. 추적 과정에서 앞뒤가 안 맞는 부분이 드러나는 경우가 많다.

    4.3 Hallucination을 줄이는 프롬프트 구조¶

    다음 규칙을 반드시 따라라:
- 확실하지 않은 내용은 "확인 필요"라고 표시해라
- 라이브러리나 API를 언급할 때 정확한 패키지명과 버전을 명시해라
- 내가 제공한 코드/문서에 근거한 답변과, 네가 추론한 답변을 구분해서 표시해라
- 공식 문서 URL을 제시하지 마라. 대신 "공식 문서의 [섹션명]을 참고"라는 형태로 안내해라

    "공식 문서 URL을 제시하지 마라"가 중요하다. Claude가 만든 URL은 상당수가 404다. URL 대신 문서 섹션명이나 검색 키워드를 안내하게 하면 사용자가 직접 찾기 편하고, 잘못된 링크를 따라가는 일을 막는다.

    4.4 검증이 필수인 경우¶

    다음 상황에서는 Claude의 답변을 반드시 검증해야 한다:

    • 특정 라이브러리 버전의 API — pip show나 공식 문서로 확인
    • 보안 관련 코드 — SQL injection, XSS 방어 코드가 실제로 작동하는지 테스트
    • 인프라 설정 — Kubernetes, Terraform 등의 설정은 잘못되면 장애로 이어진다
    • 수치 계산 — 비용 계산, 성능 추정치 등은 직접 계산해서 대조

    "Claude가 말했으니까 맞겠지"는 실무에서 사고로 이어진다. 코드 리뷰를 받더라도 최종 판단은 사람이 해야 한다.

    5. 작업 유형별 프롬프트 구조¶

    5.1 코드 리뷰¶

    코드 리뷰를 시킬 때는 리뷰 관점을 명시해야 한다. "이 코드 리뷰해줘"라고만 하면 일반적인 피드백만 온다.

    아래 코드를 리뷰해줘.

리뷰 관점:
1. 보안 취약점 (SQL injection, XSS, 인증/인가 누락)
2. 동시성 문제 (race condition, deadlock 가능성)
3. 에러 처리 누락
4. 성능 문제 (N+1 쿼리, 불필요한 DB 호출)

리뷰하지 않아도 되는 것:
- 코딩 스타일, 네이밍
- 테스트 코드 유무

출력 형식:
발견한 이슈를 심각도(HIGH/MEDIUM/LOW) 순으로 정렬해서 보여줘.
각 이슈마다 해당 라인, 문제 설명, 수정 코드를 포함해.
이슈가 없으면 "발견된 이슈 없음"이라고만 답해.

    "이미 확인한 것"을 넣는 게 중요하다. 이걸 안 넣으면 Claude가 "DB 커넥션을 확인해보세요" 같은 이미 해본 걸 다시 제안한다. 헛수고를 줄이려면 이미 시도한 것, 이미 확인한 것을 꼼꼼히 적어야 한다.

    간헐적 문제를 디버깅할 때는 Claude에게 가능한 원인을 여러 개 나열하게 하고, 각 원인을 확인하는 방법까지 받는 게 좋다. "원인 하나만 알려줘"보다 "가능한 원인을 가능성 높은 순으로 3가지 알려줘"가 실무에서 유용하다.

    5.4 데이터 분석¶

    SQL 쿼리 작성이나 데이터 분석을 시킬 때는 스키마를 반드시 제공해야 한다.

    아래 테이블 스키마를 기반으로 쿼리를 작성해줘.

테이블:
- orders (id, user_id, status, total_amount, created_at)
- order_items (id, order_id, product_id, quantity, unit_price)
- products (id, name, category, price)

DB: PostgreSQL 15
인덱스: orders(user_id), orders(created_at), order_items(order_id)

요구사항:
2026년 1월 한 달간, 카테고리별 매출 합계를 구해줘.
취소된 주문(status = 'CANCELLED')은 제외.
매출이 높은 카테고리 순으로 정렬.

쿼리 작성 시 주의:
- 서브쿼리보다 JOIN 선호
- 인덱스를 탈 수 있는 형태로 작성
- 실행 계획을 고려해서 작성

    인덱스 정보를 넣으면 Claude가 인덱스를 활용하는 쿼리를 작성한다. 이걸 빼면 WHERE 절에서 인덱스를 못 타는 쿼리가 나오는 경우가 있다.

    복잡한 분석은 한번에 시키지 말고 단계를 나눠야 한다. "월별 코호트 분석 쿼리 작성해줘"보다 "먼저 사용자별 첫 주문일을 구하는 CTE를 작성해줘 → 이걸 기반으로 월별 리텐션을 계산해줘" 식으로 나누면 각 단계의 결과를 검증할 수 있고, 실수가 줄어든다.

    6.1 백엔드 코드 리뷰 프로젝트 세팅¶

    Custom Instructions:

    너는 [서비스명] 백엔드 코드 리뷰어다.
기술 스택: Java 17 + Spring Boot 3.2 + JPA + PostgreSQL

리뷰 우선순위:
1. 보안 취약점
2. 데이터 정합성 (트랜잭션, 동시성)
3. 성능 (N+1, 불필요한 쿼리)

하지 않는 것:

출력 형식:

  설명: ...
  수정안: ...

    Knowledge에 올린 파일: - coding-conventions.md — 팀 코딩 컨벤션 - architecture-overview.md — 레이어 구조, 패키지 규칙 - error-codes.md — 에러 코드 체계

    6.2 API에서 반복 작업 자동화¶

    매일 PR 코드 리뷰를 돌리는 파이프라인을 만든다면, system prompt를 고정하고 코드만 바꿔 넣는 구조가 된다.

    import anthropic
import json

client = anthropic.Anthropic()

REVIEW_SYSTEM_PROMPT = """
너는 자동 코드 리뷰 봇이다.
아래 규칙을 따라 리뷰 결과를 JSON으로 반환한다.

출력 형식:
{
  "issues": [
    {
      "severity": "HIGH | MEDIUM | LOW",
      "file": "파일 경로",
      "line": 라인 번호,
      "message": "문제 설명",
      "suggestion": "수정 제안"
    }
  ],
  "summary": "전체 요약 (1~2문장)"
}

이슈가 없으면 issues를 빈 배열로 반환한다.
JSON 외의 텍스트를 출력하지 마라.
"""

def review_code(diff: str) -> dict:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=4096,
        system=REVIEW_SYSTEM_PROMPT,
        messages=[
            {"role": "user", "content": f"아래 diff를 리뷰해줘:\n\n{diff}"},
            {"role": "assistant", "content": "{"}
        ]
    )
    result_text = "{" + response.content[0].text
    return json.loads(result_text)

    이 구조에서 system prompt에 Prompt Caching을 적용하면 매 요청마다 system prompt 토큰 비용을 90% 줄일 수 있다. 하루에 수십 건씩 돌리는 자동화에서는 비용 차이가 크다.

    7. 정리¶

    Claude를 잘 쓰려면 결국 두 가지다.

    첫째, Claude에게 명확한 맥락을 준다. 역할, 제약, 출력 형식, 도메인 지식을 구체적으로 넣을수록 결과가 좋아진다. "알아서 잘 해줘"는 가장 나쁜 프롬프트다.

    둘째, Claude의 출력을 맹신하지 않는다. 코드는 돌려봐야 하고, API 이름은 문서에서 확인해야 하고, 수치는 직접 검산해야 한다. Claude는 초안을 빠르게 만들어주는 도구지, 최종 검수를 대신해주는 도구가 아니다.

    이 두 가지만 지키면 Claude로 실무 생산성을 확실히 올릴 수 있다.
