    Claude Code 모델별 차이¶

    Claude 문서에서 API 관점의 모델 비교를, Claude Code 실전 팁에서 모델 선택을 간단히 다뤘다. 이 문서는 Claude Code 에이전틱 실행 환경에서 모델이 실제로 어떻게 다르게 동작하는지에 집중한다.

    1. 에이전틱 실행에서 모델별 도구 사용 패턴¶

    API에서 Claude를 호출하면 한 번의 요청-응답이다. Claude Code는 다르다. 모델이 스스로 도구를 골라서 호출하고, 결과를 보고, 다음 도구를 고르는 과정을 반복한다. 이 자율적인 판단 과정에서 모델 간 차이가 확연하게 드러난다.

    Opus — 넓게 탐색하고, 확인 후 작업한다¶

    Opus는 코드를 수정하기 전에 주변 맥락을 먼저 파악하려 한다. "이 함수를 고쳐줘"라고 시키면, 해당 함수만 보는 게 아니라 호출하는 쪽, 테스트 코드, 관련 타입 정의까지 Grep과 Glob으로 먼저 훑는다.

    사용자: "UserService.updateProfile에서 이메일 중복 체크 빠져있어. 추가해줘."

Opus의 도구 호출 순서:
1. Grep: "updateProfile" → 호출부 확인
2. Read: UserService.ts → 메서드 전체 코드 확인
3. Grep: "email" + "duplicate" → 기존 중복 체크 패턴 탐색
4. Read: UserRepository.ts → DB 접근 계층 확인
5. Read: user.service.spec.ts → 기존 테스트 패턴 확인
6. Edit: UserService.ts → 중복 체크 로직 추가
7. Edit: user.service.spec.ts → 테스트 추가
8. Bash: npm test → 검증

    도구 호출이 많아서 비용과 시간이 든다. 하지만 "기존 패턴을 따라서" 수정하기 때문에, 코드 스타일이 튀지 않고 엣지 케이스를 놓치는 일이 적다.

    Sonnet — 직선적으로 해결한다¶

    Sonnet은 목표 지점으로 바로 간다. 같은 요청을 받으면 해당 파일을 읽고, 수정하고, 테스트를 돌린다. 중간에 주변 코드를 넓게 탐색하는 단계가 짧다.

    Sonnet의 도구 호출 순서:
1. Read: UserService.ts
2. Edit: UserService.ts → 중복 체크 로직 추가
3. Bash: npm test → 검증

    대부분의 일반적인 작업에서는 이 정도면 충분하다. 문제가 생기는 건 수정 범위가 여러 파일에 걸치거나, 기존 패턴과 다른 방식으로 구현해버릴 때다. Sonnet이 Repository에서 findByEmail이 이미 있는데도 직접 쿼리를 작성하는 식이다.

    Haiku — 단순 작업에서만 쓴다¶

    Haiku는 변수명 변경, 주석 추가, import 정리 같은 단순 작업에 적합하다. 멀티스텝 도구 호출이 필요한 작업에서는 중간에 맥락을 잃는 경우가 잦다. 앞에서 읽은 파일 내용을 다음 도구 호출에서 제대로 반영하지 못하거나, 도구 호출 자체를 잘못 구성하는 경우가 있다.

    "파일 3개를 읽고, 공통 인터페이스를 추출해줘" 같은 작업을 Haiku에게 시키면, 첫 번째 파일의 구조를 세 번째 파일을 처리할 때 이미 잊어버리는 식이다.

    2. Fast 모드의 정체¶

    Claude Code에서 /fast로 토글하거나 설정에서 Fast 모드를 켤 수 있다. 이걸 켜면 "Sonnet으로 바뀌는 건가?" 하고 착각하기 쉽다. 모델은 바뀌지 않는다. Opus를 쓰고 있었으면 여전히 Opus다.

    Fast 모드는 출력 생성 속도를 높이는 대신 일부 추론 깊이를 줄이는 설정이다. 체감상 응답이 빨라지고, 간단한 작업에서는 결과 차이가 거의 없다.

    켜는 게 나을 때¶

    • 단순한 파일 수정, 포맷팅, import 정리
    • 명확한 지시가 있는 단일 파일 작업
    • 대화형으로 빠르게 주고받아야 할 때

    끄는 게 나을 때¶

    • 여러 파일에 걸친 리팩토링
    • 버그 원인 추적 (탐색 범위가 넓은 작업)
    • 아키텍처 설계나 의사결정이 필요한 작업

    기본 Opus에서 Fast 모드를 켜면 Sonnet 정도의 속도로 Opus 수준의 결과를 내는 느낌이다. 다만 복잡한 멀티스텝 작업에서는 Fast 모드가 탐색 깊이를 줄여서 Opus의 장점인 "넓게 보는 능력"이 약해진다.

    3. 컨텍스트 윈도우와 세션 한계¶

    Claude Code는 대화가 이어질수록 도구 호출 결과, 파일 내용, 명령 출력이 컨텍스트에 쌓인다. 모델별 컨텍스트 윈도우 크기가 세션의 수명을 결정한다.

    모델 컨텍스트 윈도우 Claude Code 세션에서의 의미
    Opus 4.6 200K (1M 확장) 대규모 코드베이스 탐색 가능. 파일 수십 개를 읽어도 여유 있음
    Sonnet 4.6 200K 일반적인 작업에는 충분. 파일 20~30개 넘게 읽으면 압축 시작
    Haiku 4.5 200K 동일한 200K지만, 긴 컨텍스트에서 정보 활용 능력이 떨어짐

    Opus 1M 확장의 실제 효과¶

    Opus는 1M 토큰까지 컨텍스트를 확장할 수 있다. Claude Code 세션에서 이 차이가 크게 느껴지는 건 대규모 레거시 코드베이스를 다룰 때다.

    수백 개 파일로 구성된 모노레포에서 "이 API의 요청 흐름을 추적해줘"라고 시키면, Opus 1M은 컨트롤러 → 서비스 → 리포지토리 → 미들웨어 → 설정 파일까지 읽으면서도 앞에서 읽은 내용을 유지한다. Sonnet 200K에서 같은 작업을 시키면 중간에 자동 압축이 발생하면서 앞부분 맥락이 날아간다.

    하지만 1M 컨텍스트를 가득 채우는 건 곧 비용이다. Opus의 입력 토큰 단가($15/1M)로 1M 토큰을 채우면 한 번의 요청에 $15다. 컨텍스트가 커질수록 매 도구 호출마다 이 비용이 반복된다는 점을 기억해야 한다.

    컨텍스트 절약 습관¶

    모델과 상관없이 세션을 오래 쓰면 컨텍스트가 차는 건 같다.

    • 작업이 바뀌면 /clear로 초기화
    • 탐색이 길어졌으면 /compact로 압축 후 계속
    • "전체 프로젝트 분석" 같은 넓은 범위 요청은 피하고, 디렉토리나 모듈 단위로 쪼개서 요청

    4. 세션 기준 비용 비교¶

    API 토큰 단가표만 보면 "Opus가 5배 비싸네"로 끝나는데, Claude Code 세션에서의 실제 비용은 그보다 복잡하다. 같은 작업을 시켜도 모델마다 도구 호출 횟수와 탐색 범위가 다르기 때문이다.

    30분 작업 세션 비용 비교 (체감 기준)¶

    아래는 "NestJS 프로젝트에서 인증 모듈에 refresh token rotation 기능 추가"라는 중간 규모 작업을 각 모델로 수행했을 때의 대략적인 패턴이다.

    항목 Opus Sonnet Haiku
    도구 호출 횟수 25~40회 10~20회 8~15회
    읽는 파일 수 10~15개 4~8개 2~5개
    입력 토큰 (누적) 300K~500K 100K~200K 50K~100K
    출력 토큰 30K~50K 15K~30K 10K~20K
    대략 비용 $7~12 $0.5~1.5 $0.1~0.3

    Opus는 넓게 탐색하는 만큼 도구 호출이 많고, 매 호출마다 누적 컨텍스트가 재전송되니까 비용이 급격히 늘어난다. Sonnet 대비 단가가 5배인데 도구 호출도 2배 이상이라, 총비용은 5~10배 차이가 나는 경우가 흔하다.

    Max 플랜 vs API 직접 과금¶

    Claude Code의 과금 방식은 두 가지다.

    API 키 방식 (종량제) - Anthropic Console에서 API 키 발급 후 사용 - 토큰 단위 정확한 과금 - Opus를 많이 쓰면 비용이 빠르게 올라감 - /cost로 실시간 비용 확인 가능

    Max 플랜 (월 구독) - Pro ($20/월), Max ($100/월, $200/월) 등 구독 플랜 - 사용량 제한이 있지만 구간 내에서는 추가 비용 없음 - 무거운 작업을 하다 보면 하루 만에 사용량 제한에 걸리는 경우가 있음

    선택 기준:

    API 직접 과금이 맞는 경우: - CI/CD 파이프라인에서 Claude Code를 돌릴 때 - 사용량 예측이 가능하고, 월 비용이 $100 이하일 때 - Haiku나 Sonnet 위주로 쓸 때 (단가가 낮아서 월 구독 대비 저렴)

    Max 플랜이 맞는 경우: - Opus를 주력으로 쓸 때 (API 과금이면 하루에 $50 넘길 수 있음) - 비용 예측 없이 자유롭게 쓰고 싶을 때 - 팀원이 여럿이고 각자 API 키 관리가 번거로울 때

    Max $200 플랜에서 Opus를 하루 종일 쓰면 보통 2~3일 안에 사용량 제한에 걸린다. 그 시점에서 Sonnet으로 전환하거나 다음 리셋까지 기다려야 한다. 작업 패턴에 따라 API 과금이 더 자유로울 수 있다.

    5. 오케스트레이션에서 모델 조합¶

    Claude Code 오케스트레이션 문서에서 다단계 워크플로우를 다뤘다. 여기서는 각 단계에 어떤 모델을 배치하는지, 그 근거가 뭔지를 다룬다.

    기본 조합: 설계 Opus, 구현 Sonnet, 검증 Haiku¶

    // 1단계: Opus가 설계한다
const planner = new ClaudeCode({ model: "claude-opus-4-6", maxTurns: 5 });
const plan = await planner.run({
  prompt: "auth 모듈을 리팩토링할 계획을 JSON으로 출력해.",
});

// 2단계: Sonnet이 구현한다
const builder = new ClaudeCode({ model: "claude-sonnet-4-6", maxTurns: 30 });
await builder.run({
  prompt: `다음 계획대로 코드를 수정해:\n${plan.output}`,
});

// 3단계: Haiku가 검증한다
const checker = new ClaudeCode({ model: "claude-haiku-4-5", maxTurns: 10 });
const review = await checker.run({
  prompt: "npm test를 실행하고, 실패한 테스트가 있으면 원인을 보고해.",
});

    왜 이렇게 나누는가:

    • 설계에 Opus를 쓰는 이유: 리팩토링 범위를 잡을 때 "이 파일만 고치면 되나, 아니면 호출하는 쪽도 바꿔야 하나"를 판단하는 능력이 Sonnet보다 확실히 낫다. 넓게 탐색하는 Opus의 특성이 설계 단계에서 빛난다.
    • 구현에 Sonnet을 쓰는 이유: 계획이 구체적으로 나와 있으면 직선적으로 처리하는 Sonnet이 더 빠르고 저렴하다. 설계를 Opus가 해줬으니 Sonnet이 주변을 넓게 탐색할 필요가 없다.
    • 검증에 Haiku를 쓰는 이유: "테스트 돌리고 결과 보고"는 판단이 필요한 작업이 아니다. 명령 실행하고 출력 정리만 하면 되니까 가장 저렴한 모델이면 된다.

    이 조합이 안 맞는 경우¶

    작업이 작을 때: "함수 하나 수정하고 테스트 돌려줘" 같은 작업을 3단계로 나누면 오버헤드만 늘어난다. Sonnet 하나로 한 번에 처리하는 게 낫다.

    Haiku 검증이 실패하는 경우: 테스트가 실패했을 때 "왜 실패했고 어떻게 고쳐야 하는지"까지 판단하게 하면 Haiku가 엉뚱한 수정을 시도하는 경우가 있다. 검증 단계에서 수정까지 해야 한다면 Sonnet 이상을 써야 한다.

    설계가 불필요한 경우: 명확한 지시가 있는 작업("이 API에 페이지네이션 파라미터 추가해줘")은 설계 단계를 건너뛰고 바로 Sonnet으로 구현해도 된다.

    비용 관점에서의 조합 효과¶

    위 3단계 조합을 전부 Opus로 돌렸을 때와 비교하면:

    방식 예상 비용 결과 품질
    전부 Opus $10~15 높음
    Opus + Sonnet + Haiku 조합 $3~5 거의 동일
    전부 Sonnet $1~2 설계 단계에서 누락 가능

    조합을 쓰면 Opus만 쓸 때 대비 비용이 1/3 정도로 줄면서, 결과 품질은 큰 차이가 없다. 비용이 아깝다면 Opus를 설계에만 쓰고 나머지는 Sonnet으로 통일하는 것도 괜찮다.

    6. 모델 전환 실무¶

    세션 중 전환¶

    # 방법 1: 슬래시 명령
/model claude-opus-4-6

# 방법 2: Shift+Tab 단축키

# 방법 3: 시작 시 지정
claude --model claude-opus-4-6

    세션 중에 모델을 바꾸면 컨텍스트는 유지된다. Sonnet으로 작업하다가 "이 부분은 복잡하니까 Opus로 바꿔야겠다" 싶으면 바로 전환하면 된다.

    실전에서 자주 쓰는 패턴¶

    1. Sonnet으로 시작 → 막히면 Opus로 전환: 가장 흔한 패턴이다. 대부분의 작업은 Sonnet으로 처리하고, "왜 이 테스트가 실패하는지 모르겠다"는 상황에서 Opus로 바꾸면 원인을 찾아내는 경우가 많다.

    2. Opus로 설계 → Sonnet으로 구현: Tips 문서에서도 언급한 패턴이다. /clear 없이 같은 세션에서 모델만 바꿔도 되지만, 설계 결과를 CLAUDE.md나 별도 파일에 저장한 뒤 새 세션을 여는 게 더 깔끔하다.

    3. 복잡한 디버깅은 Opus + Extended Thinking: Opus에 Extended Thinking까지 켜면 비용이 상당하지만, "3일째 못 찾는 버그"를 한 번에 잡는 경우가 있다. 이 조합은 마지막 수단으로 남겨두는 게 좋다.
