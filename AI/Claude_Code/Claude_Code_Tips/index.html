    Claude Code 실전 팁¶

    Claude Code 기본 기능(MCP, 훅, 슬래시 명령어 등)은 Claude Code 문서를 참고한다. 이 문서는 실제로 쓰면서 쌓인 노하우만 다룬다.

    1. CLAUDE.md — 매번 설명하지 말고 한 번 써두기¶

    Claude Code는 세션 시작 시 CLAUDE.md를 자동으로 읽는다. 기술 스택을 넣는 것보다 팀 코드리뷰에서 자주 걸리는 규칙을 넣는 게 훨씬 효과가 좋다.

    # 프로젝트 컨텍스트

## 기술 스택
- Runtime: Node.js 20, TypeScript 5
- Framework: NestJS
- Test: Jest (unit), Supertest (e2e)
- DB: PostgreSQL + TypeORM

## 반드시 지킬 것
- any 타입 금지
- console.log 커밋 금지 (logger 사용)
- 커밋 메시지: Conventional Commits 형식
- TypeORM에서 save() 대신 update() 사용
  (save()를 쓰면 의도치 않게 새 레코드가 생성됨)

## 테스트
- 단위 테스트: `npm run test`
- e2e: `npm run test:e2e`
- 커버리지: `npm run test:cov`

    /init으로 초안을 만들고 실제로 겪은 삽질을 하나씩 추가하면 된다. "왜 이걸 안 하지?"라고 느낀 적이 있다면 그게 CLAUDE.md에 들어가야 할 내용이다.

    2. 모델 선택 — 작업 단계별로 나눠 쓰기¶

    무조건 비싼 모델을 쓴다고 좋은 결과가 나오지 않는다.

    모델 쓰는 상황
    Opus 아키텍처 설계, 복잡한 알고리즘, 대규모 리팩토링 방향 잡을 때
    Sonnet 일반 구현, 버그 수정, 코드 리뷰
    Haiku 변수명 변경, 주석 추가, 포맷팅

    가장 실용적인 패턴은 설계와 구현을 분리하는 거다. Opus로 방향을 잡고, 구현은 Sonnet에게 넘긴다. Sonnet으로 설계를 시키면 중간에 "이게 맞나?" 싶은 구간이 생긴다.

    세션 중에 /model이나 Shift+Tab으로 전환할 수 있어서, 한 세션 안에서도 단계에 따라 모델을 바꿀 수 있다.

    3. Extended Thinking — 복잡한 문제에서 정확도 올리기¶

    Extended Thinking은 Claude가 답변 전에 내부적으로 사고 과정을 거치는 기능이다. 기본 모드보다 시간과 토큰을 더 쓰지만, 복잡한 문제에서 정확도가 눈에 띄게 올라간다.

    언제 쓰나¶

    • 여러 파일에 걸친 버그 추적
    • 기존 코드와 충돌 없이 새 기능을 끼워넣어야 할 때
    • 리팩토링 범위와 영향도를 판단해야 할 때
    • 테스트 실패 원인이 명확하지 않을 때

    언제 안 써도 되나¶

    • 단순한 파일 수정이나 변수명 변경
    • 이미 방향이 명확한 구현 작업
    • 타입 추가, import 정리 같은 기계적인 작업

    실전 활용법¶

    Extended Thinking 모드에서는 프롬프트를 구체적으로 줄수록 사고 과정이 집중된다.

    # 단순 요청 — thinking이 분산됨
"이 코드 고쳐줘"

# 제약을 걸어서 사고를 집중시키는 요청
"src/order/order.service.ts의 calculateTotal 메서드에서
할인 쿠폰이 두 개 이상 적용될 때 금액이 음수가 되는 버그가 있어.
기존 테스트는 깨지면 안 되고, 쿠폰 적용 순서에 따라 결과가 달라지지 않게 해줘."

    Thinking 결과가 길어지면 그만큼 컨텍스트를 먹는다. 간단한 작업에 Extended Thinking을 켜놓으면 비용만 올라간다. 복잡한 문제 → thinking 켜기, 단순 작업 → 끄기. 이 구분만 해도 충분하다.

    4. 멀티턴 컨텍스트 관리 — 긴 작업에서 품질 유지하기¶

    Claude Code로 30분 이상 작업하다 보면 응답 품질이 떨어지는 구간이 온다. 컨텍스트 윈도우에 이전 대화, 파일 읽기 결과, 명령 실행 출력이 계속 쌓이기 때문이다.

    작업 전환 패턴¶

    기능 구현 끝남 → /clear → 테스트 작성
테스트 끝남 → /clear → 코드 리뷰

    작업 성격이 바뀌면 /clear를 치는 게 거의 항상 맞다. 이전 작업의 잔해가 남아 있으면 Claude가 엉뚱한 맥락으로 빠진다.

    같은 작업이 길어질 때¶

    대화 20턴 넘어감 → /compact "지금까지 한 작업 요약해줘"

    /compact에 커스텀 요약 지시를 같이 넣을 수 있다. 그냥 /compact만 치면 Claude가 알아서 요약하는데, 중요한 결정 사항이 빠지는 경우가 있다. "어떤 파일을 왜 수정했는지 포함해서 요약해줘" 같은 지시를 추가하면 압축 후에도 작업을 이어갈 수 있다.

    대규모 작업 분할 패턴¶

    파일 10개 이상 수정하는 작업을 한 세션에서 끝내려 하면 높은 확률로 후반부에 실수가 생긴다.

    1세션: 설계 → /plan으로 수정 대상 파일 목록과 방향 확정
  ↓ /clear
2세션: 핵심 모듈 수정 (3~4개 파일)
  ↓ /clear
3세션: 나머지 파일 수정
  ↓ /clear
4세션: 테스트 실행 + 리뷰

    세션 사이에 컨텍스트가 끊기는 게 걱정될 수 있다. CLAUDE.md에 현재 진행 상황을 메모해두거나, Auto Memory가 핵심 결정 사항을 기억하고 있어서 다음 세션에서 "아까 하던 작업 이어서 해줘"가 먹힌다.

    컨텍스트를 아끼는 요청 방식¶

    # 컨텍스트를 많이 먹는 요청
"이 프로젝트의 인증 관련 코드 전부 리뷰해줘"

# 컨텍스트를 적게 먹는 요청
"src/auth/auth.service.ts의 refreshToken 메서드만 봐줘.
JWT 만료 처리 부분에 버그가 있는 것 같아."

    범위를 좁힐수록 Claude가 읽는 파일 수가 줄어들고, 그만큼 컨텍스트가 남아서 실제 작업에 쓸 수 있다.

    5. 대규모 코드베이스 탐색 노하우¶

    파일이 수백 개인 프로젝트에서 Claude Code를 쓸 때, "전체 분석해줘"는 최악의 요청이다. 컨텍스트만 날리고 유용한 답은 안 나온다.

    진입점부터 추적하기¶

    코드베이스를 처음 보는 상황에서 가장 잘 먹히는 패턴은 진입점(엔드포인트, main 함수)에서 시작해서 안쪽으로 파고드는 거다.

    "POST /api/orders 엔드포인트의 요청 흐름을 추적해줘.
라우터 → 컨트롤러 → 서비스 → 레포지토리 순서로."

    이렇게 하면 Claude가 라우터 파일부터 찾아서 호출 체인을 따라간다. 전체를 훑는 것보다 관련 파일만 정확하게 읽는다.

    서브에이전트로 병렬 탐색¶

    큰 코드베이스에서 관련 파일을 찾을 때 서브에이전트가 빠르다. "동시에", "병렬로" 같은 키워드를 쓰면 Claude Code가 Explore 서브에이전트를 투입한다.

    "두 가지를 동시에 찾아줘:
1. 결제 금액 계산하는 서비스 파일
2. 결제 관련 에러 타입 정의 파일"

    서브에이전트가 탐색을 마치고 결과만 메인 컨텍스트로 가져오기 때문에, 탐색 과정에서 생기는 노이즈(잘못 읽은 파일, 중간 검색 결과)가 쌓이지 않는다. 메인 컨텍스트를 깨끗하게 유지하는 핵심이다.

    Grep/Glob을 직접 지시하기¶

    Claude Code가 파일을 찾을 때 내부적으로 Glob이나 Grep을 쓰는데, 이걸 직접 지시하면 불필요한 탐색을 줄일 수 있다.

    "먼저 grep으로 'PaymentService'를 import하는 파일만 찾아줘.
그 다음에 해당 파일들만 읽어서 결제 로직의 흐름을 설명해줘."

    Claude Code에게 탐색 순서를 알려주는 거다. 안 그러면 프로젝트 구조를 이해하겠다고 관련 없는 파일까지 읽는 경우가 있다.

    모노레포에서의 범위 지정¶

    모노레포에서 "이 버그 고쳐줘"만 하면 Claude가 여러 패키지를 오가면서 컨텍스트를 소진한다.

    "packages/payment 디렉토리 안에서만 작업해줘.
다른 패키지의 파일은 읽지 마."

    작업 범위를 디렉토리 수준에서 명시적으로 제한하면 탐색이 빨라지고 컨텍스트도 아낀다. CLAUDE.md에 모노레포 구조와 패키지별 역할을 적어두면 매번 설명할 필요가 없다.

    6. 프롬프트 — 구체적일수록 좋다¶

    막연하게 물어보면 막연하게 답한다. 이건 Claude Code만의 문제가 아니다.

    목적 + 제약 + 기대 결과¶

    # 이렇게 말고
"이 코드 고쳐줘"

# 이렇게
"이 함수는 동시 요청이 100개 이상일 때 race condition이 생겨.
Mutex나 queue로 순차 처리하도록 수정해줘.
기존 인터페이스는 건드리지 말고, 테스트 코드도 같이 써줘."

    한 번에 다 시키지 말고 단계별로¶

    # 이렇게 말고
"로그인 구현하고, 테스트 쓰고, 문서도 업데이트해줘"

# 이렇게
1단계: "로그인 API 구현해줘" → 확인
2단계: "방금 구현한 로그인 API 유닛 테스트 써줘" → 확인
3단계: "API 스펙 README에 추가해줘"

    한 번에 다 시키면 중간에 실수가 생겨도 어디서 잘못됐는지 파악하기 힘들다.

    파일 경로는 항상 명시¶

    "서비스 파일에서 에러 처리 추가해줘"           # Claude가 추측함
"src/user/user.service.ts의 createUser 메서드"  # 정확하게 찾아감

    파일 경로를 안 쓰면 Claude가 추측하는데, 비슷한 이름의 파일이 여러 개면 틀리는 경우가 꽤 있다.

    7. 서브에이전트 — 메인 컨텍스트를 보호하는 방법¶

    서브에이전트의 가장 큰 장점은 메인 컨텍스트를 깨끗하게 유지하는 거다. 탐색 과정에서 읽은 파일 수십 개가 메인에 안 쌓인다.

    타입별 사용 기준¶

    타입 언제 쓰나
    Explore 파일 찾기, 코드 구조 파악. 읽기만 하고 수정 안 함
    Plan 구현 방향 수립, 영향도 분석
    general-purpose 탐색 + 수정까지 한꺼번에 위임

    병렬 처리가 빠른 경우¶

    "두 가지를 동시에 분석해줘:
- 현재 에러 처리 패턴
- 테스트 커버리지 현황"

    독립적인 작업 2~3개를 순차로 하면 60초, 병렬이면 30초 수준으로 줄어든다. 의존관계가 없는 작업이면 "동시에"를 쓰는 게 거의 항상 낫다.

    서브에이전트가 오히려 손해인 경우¶

    파일 1~2개만 확인하면 되는 단순한 탐색에 서브에이전트를 쓰면 오버헤드만 생긴다. 서브에이전트를 생성하고 결과를 받아오는 과정 자체가 비용이니까, 간단한 건 그냥 직접 시키는 게 맞다.

    8. 스킬(Skills) — 반복 작업을 명령어 하나로¶

    스킬은 여러 단계로 이뤄진 복잡한 루틴을 하나의 명령으로 감싼 것이다. ~/.claude/skills/ (전역) 또는 .claude/skills/ (프로젝트)에 마크다운 파일로 만든다.

    팀에서 쓸만한 커스텀 스킬 예시¶

    <!-- .claude/skills/pr-ready.md -->
# pr-ready

PR 올리기 전 점검을 자동으로 실행한다.

## 수행 단계

1. 현재 브랜치의 커밋 메시지가 Conventional Commits 형식인지 확인
2. /review 실행 — 코드 품질, 보안, 성능 이슈 점검
3. 테스트 전체 실행 후 실패 여부 확인
4. 결과 요약 출력

    <!-- .claude/skills/security-scan.md -->
# security-scan

OWASP Top 10 기준으로 현재 변경사항의 취약점을 스캔한다.

## 점검 항목

- SQL Injection: 사용자 입력이 쿼리에 직접 들어가는지
- XSS: 출력 시 이스케이핑 처리 여부
- 인증 로직: 토큰 검증 누락, 권한 체크 누락
- 민감 정보 노출: API 키, 비밀번호 하드코딩 여부
- 결과를 심각도(Critical/High/Medium/Low)별로 분류

    팀에서 반복하는 작업이 있다면 스킬로 만들어서 .claude/skills/에 넣고 git으로 관리한다. 누구나 같은 방식으로 처리하게 되고, 실수도 줄어든다.

    9. 비용 관리 — API 과금이니까 습관이 중요하다¶

    토큰 기반 과금이다. 쓰는 방식에 따라 같은 작업이라도 비용이 2~3배 차이난다.

    비용을 올리는 습관¶

    • "전체 프로젝트 읽어줘" 같은 넓은 범위 요청
    • 세션을 길게 유지하면서 같은 파일을 반복 참조
    • 요청이 불명확해서 수정을 여러 번 반복
    • 간단한 작업에 Opus나 Extended Thinking 사용

    비용을 낮추는 습관¶

    • /plan으로 방향 확인 후 구현 → 수정 횟수 감소
    • 파일/함수 단위로 범위 지정
    • /compact로 주기적 압축
    • 단순 작업은 Haiku, 일반 구현은 Sonnet, 설계만 Opus
    • CLAUDE.md로 매 세션 반복되는 설명 제거

    /cost로 현재 세션 비용을 수시로 확인하는 습관을 들이면 감각이 생긴다.

    10. 자주 하는 실수들¶

    Claude가 엉뚱한 파일을 수정한다¶

    파일 경로를 안 쓰면 Claude가 추측한다. 비슷한 이름이 여러 개면 틀린다.

    "auth guard 파일 수정해줘"          # 추측
"src/auth/auth.guard.ts 수정해줘"   # 확실

    세션을 넘기면 이전 작업을 잊는다¶

    Auto Memory가 핵심 결정 사항은 기억하지만, 세부 작업 내역까지 다 남지는 않는다. 한 번 해결한 문제는 CLAUDE.md에 주의사항으로 남기는 게 확실하다.

    대용량 파일 편집 중에 멈춘다¶

    컨텍스트 초과다. 파일을 모듈 단위로 쪼개거나, 한 번에 하나의 함수만 요청하거나, /compact로 공간을 확보하고 재시도한다.

    훅이 걸려서 작업이 중단된다¶

    훅에서 에러가 발생하면 Claude Code가 멈출 수 있다. 훅 명령어 끝에 2>&1 | head -20 같은 출력 제한을 걸어두면 훅 출력이 컨텍스트를 먹는 것도 방지된다.

    11. Oh My Claude Code (OMC)¶

    Oh My Zsh가 Zsh에 플러그인과 테마를 추가하듯, OMC는 Claude Code에 전문 에이전트와 스킬을 한꺼번에 추가하는 프레임워크다.

    npm install -g oh-my-claude-sisyphus

    5가지 실행 모드¶

    모드 언제 쓰나 특징
    Autopilot 평소 개발 요청을 분석해서 적합한 에이전트 자동 선택
    Ultrapilot 퀄리티 최우선 여러 에이전트를 동시 동원, 느리고 비쌈
    Swarm 대규모 리팩토링 에이전트 다수를 병렬 투입
    Pipeline 순서가 정해진 작업 단계별 파이프라인 구성
    Ecomode 비용 절감 토큰 최소화

    요청을 분석해서 알맞은 에이전트를 자동으로 고르기 때문에, 직접 "아키텍트 에이전트 써줘"라고 지정할 필요가 없다.

    Claude Code 외에 Cursor, Aider 같은 오픈소스 도구를 쓴다면 Oh My Open Code (OMO)를 찾아보면 된다. 특정 벤더에 종속되지 않는 유사 에코시스템이다.

    12. 팀 도입 — 혼자 쓸 때와 다른 점¶

    혼자 쓸 때는 CLAUDE.md 하나 만들고 끝이다. 팀 전체가 쓰기 시작하면 "누구 설정이 맞는 거야?"라는 문제가 바로 생긴다.

    CLAUDE.md — 팀 규칙과 개인 설정 분리¶

    CLAUDE.md를 git에 올리면 팀 전원이 같은 규칙을 공유한다. 문제는 개인마다 다른 부분이 있다는 거다. 한 명은 커밋 전에 항상 lint를 돌리고 싶고, 다른 한 명은 그거 없이 빠르게 돌리고 싶다.

    Claude Code는 CLAUDE.md를 여러 경로에서 읽는다. 이걸 이용해서 분리한다.

    프로젝트 루트/
├── CLAUDE.md              ← git 추적. 팀 공통 규칙
├── CLAUDE.local.md        ← .gitignore. 개인 설정
└── src/
    └── payments/
        └── CLAUDE.md      ← 해당 디렉토리 전용 규칙

    팀 공통 CLAUDE.md에는 코딩 컨벤션, 테스트 실행 방법, 커밋 메시지 형식 같은 걸 넣는다. 개인적으로 다른 모델을 쓰고 싶거나, 특정 파일을 항상 무시하고 싶으면 CLAUDE.local.md에 넣고 .gitignore에 추가한다.

    디렉토리별 CLAUDE.md는 특정 모듈에만 적용되는 규칙이 있을 때 쓴다. 결제 모듈에서는 "금액 계산 시 반드시 BigDecimal 사용"처럼 해당 디렉토리 작업할 때만 자동으로 로드된다.

    실제 팀에서 겪는 문제는 CLAUDE.md가 점점 커지는 거다. 규칙이 50줄 넘어가면 Claude가 뒤쪽 내용을 놓치는 경우가 생긴다. 핵심 규칙은 위에 두고, 참고사항은 아래로 밀어야 한다.

    settings.json과 settings.local.json¶

    CLAUDE.md가 "Claude에게 주는 지시"라면, settings.json은 "Claude Code 자체의 동작 설정"이다. 두 가지 레벨이 있다.

    프로젝트 루트/
└── .claude/
    ├── settings.json        ← git 추적. 팀 공통 설정
    └── settings.local.json  ← .gitignore. 개인 설정

    팀 공통 settings.json 예시:

    {
  "permissions": {
    "allow": [
      "Bash(npm run test)",
      "Bash(npm run lint)",
      "Bash(npm run build)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force)"
    ]
  },
  "hooks": {
    "PreCommit": [
      {
        "command": "npm run lint-staged"
      }
    ]
  }
}

    팀원 전원이 같은 허용/차단 명령어를 쓰게 된다. rm -rf *이나 git push --force를 팀 레벨에서 막아둘 수 있다.

    개인 settings.local.json은 본인만 쓰는 MCP 서버 연결이나, 특정 디렉토리에 대한 추가 권한 같은 걸 넣는다. 이 파일은 .gitignore에 자동으로 들어가 있다.

    주의할 점이 하나 있다. settings.json과 settings.local.json이 충돌하면 local이 이긴다. 팀에서 deny로 막아둔 명령을 개인 설정에서 allow로 풀 수 있다. 이건 의도된 동작이지만, 보안이 중요한 환경이라면 팀원들한테 local에서 deny를 풀지 말라고 공유해야 한다.

    팀 공통 스킬 관리¶

    스킬 파일을 .claude/skills/ 디렉토리에 넣고 git으로 관리하면 팀 전원이 같은 스킬을 쓴다.

    프로젝트 루트/
└── .claude/
    └── skills/
        ├── pr-ready.md        ← PR 전 점검
        ├── security-scan.md   ← 보안 스캔
        └── migration-check.md ← DB 마이그레이션 검증

    팀에서 스킬을 만들 때 주의할 점:

    • 스킬 이름은 동사형으로 통일한다. pr-ready, security-scan, migration-check처럼. 명사형으로 만들면 "이게 뭘 하는 건지" 매번 열어봐야 한다.
    • 스킬 안에서 외부 도구를 호출하면 팀원 환경에도 그 도구가 있어야 한다. 스킬에서 npx sonarqube-scanner를 쓰는데 다른 팀원한테 SonarQube가 없으면 에러가 난다. 스킬 파일 상단에 필요한 도구를 명시하거나, package.json의 devDependencies에 포함시킨다.
    • 스킬이 너무 많아지면 오히려 안 쓴다. 5개 이상이면 팀원들이 뭐가 있는지 파악하지 못한다. 자주 쓰는 3~4개만 유지하고, 나머지는 개인 스킬(~/.claude/skills/)로 빼는 게 낫다.

    비용 분배 패턴¶

    팀에서 Claude Code를 쓸 때 비용 관리 방식은 보통 세 가지다.

    1. API 키 공유 방식

    팀 공용 API 키 하나를 쓴다. 설정이 간단하지만, 누가 얼마를 썼는지 구분이 안 된다. 초반에 사용 감각을 익히는 단계에서는 괜찮은데, 한 명이 Opus로 대규모 리팩토링을 돌리면 예산이 한 번에 날아간다.

    2. 개인 키 + 팀 예산 방식

    각자 API 키를 발급받되, Anthropic 콘솔에서 키별 사용량을 추적한다. 월별 사용량을 공유하고, 팀 예산 한도를 정해놓는다. 대부분의 팀이 이 방식으로 안착한다.

    3. Max Plan 활용

    Anthropic의 Max Plan을 팀원별로 구독하면 API 키 없이 사용량 제한이 자동으로 걸린다. 관리 부담이 가장 적다. 단, Plan별 사용량 상한이 있어서 헤비 유저에게는 부족할 수 있다.

    어떤 방식이든 팀 내에서 공유해야 하는 건 비용 감각이다. Sonnet으로 30분 작업하면 대략 얼마, Opus로 설계를 돌리면 대략 얼마. 이런 감각이 없으면 "Claude Code 비싸다"만 남는다. /cost 결과를 주간 회의에서 한 번씩 공유하면 자연스럽게 팀 전체의 사용 패턴이 개선된다.
