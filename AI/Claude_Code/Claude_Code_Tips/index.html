    Claude Code를 효율적으로 쓰는 법¶

    솔직히 말하면, Claude Code를 처음 켰을 때 그냥 ChatGPT에 코드 붙여넣는 것과 별 차이를 못 느낄 수 있다. 설정을 제대로 안 하면 실제로도 별 차이가 없다.

    이 글은 "설치했는데 뭔가 아쉽다"는 사람들을 위한 가이드다.

    1. CLAUDE.md — 매번 설명하지 말고 한 번 써두기¶

    Claude Code는 세션을 시작할 때 CLAUDE.md 파일을 자동으로 읽는다. 여기다 프로젝트 규칙을 써두면, "이 프로젝트는 NestJS고 TypeScript 쓰고..." 같은 설명을 매번 안 해도 된다.

    어디에 두냐에 따라 범위가 달라진다¶

    위치 적용 범위
    ~/.claude/CLAUDE.md 내 모든 프로젝트
    {프로젝트}/CLAUDE.md 해당 프로젝트 (팀 공유 가능)
    {프로젝트}/.claude/CLAUDE.md 해당 프로젝트 (개인 전용)

    실제로 도움이 됐던 내용은 기술 스택보다 팀 코드리뷰에서 자주 걸리는 규칙이었다. "any 타입 쓰지 마", "console.log 커밋하지 마" 같은 것들. 이걸 CLAUDE.md에 넣으니까 Claude가 처음부터 알아서 피했다.

    # 프로젝트 컨텍스트

## 기술 스택
- Runtime: Node.js 20, TypeScript 5
- Framework: NestJS
- Test: Jest (unit), Supertest (e2e)
- DB: PostgreSQL + TypeORM

## 반드시 지킬 것
- any 타입 금지
- console.log 커밋 금지 (logger 사용)
- 커밋 메시지: Conventional Commits 형식

## 테스트
- 단위 테스트: `npm run test`
- e2e: `npm run test:e2e`

    /init 명령으로 현재 프로젝트를 분석해서 CLAUDE.md 초안을 자동 생성해주기도 한다. 처음엔 이걸로 시작하고 조금씩 다듬는 게 낫다.

    2. 모델 선택 — 무조건 Opus가 좋은 건 아니다¶

    비싸다고 좋은 게 아니라, 작업에 맞는 모델을 골라야 한다.

    모델 언제 써야 하나
    Opus 아키텍처 설계, 복잡한 알고리즘, 전체 시스템 리팩토링
    Sonnet 일반적인 기능 구현, 버그 수정, 코드 리뷰
    Haiku 변수명 바꾸기, 주석 추가, 간단한 포맷팅 
    claude --model claude-opus-4-6    # 설계할 때
claude --model claude-sonnet-4-6  # 평소에 (기본값)
claude --model claude-haiku-4-5   # 빠르게 확인할 때

    실제로 가장 효과적인 패턴은, 새 기능 방향을 Opus로 잡고 나서 구현은 Sonnet으로 넘기는 거다. 설계 단계에서 Sonnet을 쓰다 보면 "이게 맞나?" 싶은 순간이 생긴다.

    3. 슬래시 명령어 — 다 외울 필요 없다, 이것만¶

    전부 알 필요는 없다. 자주 쓰는 것만 몸에 익히면 충분하다.

    /clear — 새 작업 시작 전에 습관처럼¶

    이전 대화 내용이 쌓이면 Claude가 엉뚱한 맥락으로 답하는 경우가 생긴다. 새 작업 시작할 때 /clear를 치는 걸 습관으로 만들면 이런 문제가 줄어든다.

    /compact — 대화가 길어졌는데 버리기 아까울 때¶

    작업을 이어가야 하는데 대화가 너무 길어졌을 때 쓴다. 현재 맥락을 요약본으로 압축해서 토큰을 확보한다.

    /commit — 커밋 메시지 고민하기 싫을 때¶

    변경된 파일을 분석해서 Conventional Commits 형식으로 메시지를 만들어준다. 직접 쓰는 것보다 오히려 더 일관성 있게 나올 때가 많다.

    /review — PR 올리기 전에¶

    현재 변경사항을 보안, 성능, 유지보수성 기준으로 리뷰해준다. 코드리뷰 전에 한 번 돌리면 사소한 것들을 미리 잡을 수 있다.

    /plan — 복잡한 작업 전에 먼저 방향 확인¶

    바로 구현하지 않고 계획을 먼저 세워서 승인을 받는다. 큰 작업일수록 이 단계를 거치는 게 결국 더 빠르다. "일단 해봐" 하다가 방향이 잘못되면 되돌리는 게 더 오래 걸린다.

    4. 컨텍스트 관리 — 아끼는 게 곧 돈이다¶

    컨텍스트 윈도우가 가득 차면 Claude의 품질이 떨어진다. 그리고 API 과금이니까 낭비하면 실제 돈이 나간다.

    새 작업 시작 → /clear
같은 작업인데 대화가 너무 길어짐 → /compact
완전히 다른 프로젝트로 전환 → 새 세션

    가장 흔한 실수: "우리 프로젝트 전체 분석해줘" 같은 요청. 이러면 Claude가 필요 없는 파일까지 다 읽고 컨텍스트를 채워버린다.

    범위를 좁힐수록 정확도가 올라간다.

    # 이렇게 말고
"인증 관련 코드 전부 리뷰해줘"

# 이렇게
"src/auth/auth.service.ts의 refreshToken 메서드만 봐줘.
JWT 만료 처리 부분에 버그가 있는 것 같아."

    5. 프롬프트 — 구체적일수록 좋다¶

    막연하게 물어보면 막연하게 답한다. 이건 Claude Code뿐 아니라 AI 도구 전반에 해당하는 이야기다.

    목적 + 제약 + 기대 결과를 같이 넣어라¶

    # 이렇게 말고
"이 코드 고쳐줘"

# 이렇게
"이 함수는 동시 요청이 100개 이상일 때 race condition이 생겨.
Mutex나 queue로 순차 처리하도록 수정해줘.
기존 인터페이스는 건드리지 말고, 테스트 코드도 같이 써줘."

    한 번에 다 시키지 말고 단계별로¶

    # 이렇게 말고
"로그인 구현하고, 테스트 쓰고, 문서도 업데이트해줘"

# 이렇게
1단계: "로그인 API 구현해줘" → 확인
2단계: "방금 구현한 로그인 API 유닛 테스트 써줘" → 확인
3단계: "API 스펙 README에 추가해줘"

    한 번에 다 시키면 중간에 실수가 생겨도 어디서 잘못됐는지 파악하기 힘들다.

    파일 경로는 항상 명시하라¶

    # 모호한 요청
"서비스 파일에서 에러 처리 추가해줘"

# 명확한 요청
"src/user/user.service.ts의 createUser 메서드에
중복 이메일 검사와 적절한 에러 응답 추가해줘"

    6. MCP 서버 — Claude Code의 손발을 늘리는 방법¶

    MCP(Model Context Protocol)는 Claude Code에 외부 도구를 연결하는 방식이다. 설치하면 Claude Code가 직접 DB를 조회하거나 GitHub PR을 열거나 Slack 메시지를 보낼 수 있게 된다.

    자주 쓰이는 MCP 서버¶

    MCP 서버 뭘 할 수 있나
    @modelcontextprotocol/server-github PR/Issue 조회·생성·코멘트
    @modelcontextprotocol/server-postgres DB 직접 쿼리
    @modelcontextprotocol/server-brave-search 실시간 웹 검색
    @modelcontextprotocol/server-slack 채널 메시지

    설정은 settings.json에¶

    // ~/.claude/settings.json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_..."
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "POSTGRES_CONNECTION_STRING": "postgresql://..."
      }
    }
  }
}

    연결하고 나면 이런 게 된다:

    "이번 주에 머지된 PR들 요약해줘"
"users 테이블에서 최근 7일 가입자 수 뽑아줘"

    7. 서브에이전트 — 병렬로 일 시키기¶

    서브에이전트는 Claude Code가 복잡한 탐색이나 분석을 독립된 하위 에이전트에 위임하는 기능이다. 쓰는 이유는 두 가지다. 메인 컨텍스트를 보호하고 싶을 때, 그리고 여러 작업을 동시에 처리하고 싶을 때.

    서브에이전트 타입¶

    타입 언제 쓰나
    Explore 코드베이스 탐색, 파일 검색. 빠르고 읽기 전용
    Plan 구현 전략 수립, 아키텍처 설계
    general-purpose 탐색 + 수정까지 전부 자율 처리

    병렬 처리가 진짜 강점¶

    같은 작업을 순차적으로 하면 60초 걸리는 걸 병렬로 돌리면 30초로 줄어든다. "동시에", "같이", "병렬로" 같은 키워드를 쓰면 Claude Code가 서브에이전트를 투입한다.

    "두 가지를 동시에 찾아줘:
1. 인증 관련 미들웨어 파일들
2. JWT 토큰 처리하는 서비스 파일들"

    "아래 두 가지를 병렬로 분석해줘:
- 현재 에러 처리 패턴
- 테스트 커버리지 현황"

    서브에이전트가 탐색을 마치고 결과만 가져오기 때문에, 탐색 과정에서 생기는 노이즈가 메인 컨텍스트에 쌓이지 않는다.

    8. 스킬(Skills) — 반복 작업을 명령어 하나로¶

    스킬은 /skill-name 형식으로 호출하는 재사용 가능한 워크플로우다. 슬래시 명령어(/clear, /compact)가 단순 동작이라면, 스킬은 여러 단계로 이뤄진 복잡한 루틴을 하나의 명령으로 감싼 것이다.

    기본 제공 스킬¶

    스킬 하는 일
    /commit 변경 파일 분석 → 커밋 메시지 자동 생성
    /review 변경사항 리뷰 → 이슈·개선점 리포트
    /simplify 변경된 코드 품질 검토 후 리팩토링

    커스텀 스킬 만들기¶

    ~/.claude/skills/ (전역) 또는 .claude/skills/ (프로젝트)에 마크다운 파일을 만들면 된다. 팀 공유가 필요한 스킬은 .claude/skills/에 두고 git으로 관리한다.

    <!-- .claude/skills/pr-ready.md -->
# pr-ready

PR 올리기 전 체크리스트를 자동으로 실행한다.

## 수행 단계

1. 현재 브랜치의 커밋 메시지가 Conventional Commits 형식인지 확인
2. /review 실행 — 코드 품질·보안·성능 이슈 점검
3. 테스트 전체 실행 후 실패 여부 확인
4. 결과 요약 출력 (통과/실패 항목 목록)

    <!-- .claude/skills/security-scan.md -->
# security-scan

OWASP Top 10 기준으로 현재 변경사항의 취약점을 스캔한다.

## 점검 항목

- SQL Injection: 사용자 입력이 쿼리에 직접 들어가는지
- XSS: 출력 시 이스케이핑 처리 여부
- 인증 로직: 토큰 검증 누락, 권한 체크 누락
- 민감 정보 노출: API 키, 비밀번호 하드코딩 여부
- 결과를 심각도(Critical/High/Medium/Low)별로 분류

    <!-- .claude/skills/api-doc.md -->
# api-doc

현재 코드베이스의 API 엔드포인트를 분석해서 OpenAPI 3.0 스펙으로 출력한다.

## 수행 단계

1. 라우터/컨트롤러 파일에서 엔드포인트 목록 수집
2. 요청/응답 타입 분석
3. OpenAPI 3.0 YAML 형식으로 변환
4. docs/openapi.yaml 파일로 저장

    팀에서 반복적으로 하는 작업이 있다면 스킬로 만드는 게 낫다. 누구나 같은 방식으로 처리하게 되고, 실수도 줄어든다.

    9. 훅(Hook) — 파일 수정할 때마다 자동으로 실행¶

    훅은 Claude Code가 파일을 편집하거나 명령을 실행할 때 앞뒤로 자동 실행되는 쉘 명령이다. 린트, 포맷팅, 테스트 같은 걸 자동화하기 딱 좋다.

    훅 종류¶

    훅 실행 시점
    PreToolUse 도구 실행 직전
    PostToolUse 도구 실행 직후
    Stop Claude 응답 완료 후

    실전 예시¶

    // ~/.claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "cd $PROJECT_ROOT && npm run lint:fix -- $FILE_PATH 2>&1 | head -20"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "cd $PROJECT_ROOT && npm test -- --passWithNoTests 2>&1 | tail -5"
          }
        ]
      }
    ]
  }
}

    위 설정을 쓰면 파일을 수정할 때마다 ESLint가 자동으로 돌고, Claude 응답이 끝날 때마다 테스트가 실행된다. 처음엔 번거롭게 느껴질 수 있는데, 익숙해지면 이게 없는 게 더 불편하다.

    10. 비용 관리 — API 과금이니까 습관이 중요하다¶

    월정액이 아니라 토큰 기반 과금이다. 쓰는 방식에 따라 비용 차이가 꽤 난다.

    비용을 올리는 습관¶

    • "전체 프로젝트 읽어줘" 같은 넓은 범위 요청
    • 세션을 길게 유지하면서 같은 파일을 계속 재참조
    • 요청이 불명확해서 수정을 여러 번 반복

    비용을 낮추는 습관¶

    • 작업 시작 전 /plan으로 방향 확인 → 수정 횟수 감소
    • 관련 파일만 명시적으로 지목
    • /compact로 주기적인 압축
    • 단순한 작업은 Haiku로 처리
    • CLAUDE.md로 반복 설명 제거

    현재 비용 확인¶

    > /cost

# 출력 예시
Session cost: $0.42
Input tokens: 125,000
Output tokens: 8,500

    11. 자주 하는 실수들¶

    Claude가 엉뚱한 파일을 수정한다¶

    파일 경로를 안 알려주면 Claude가 추측한다. 추측이 틀리는 경우가 생각보다 많다.

    "auth guard 파일 수정해줘"     # 이렇게 말고
"src/auth/auth.guard.ts 수정해줘"  # 이렇게

    코드는 맞는데 테스트가 계속 실패한다¶

    테스트 실행 명령어가 CLAUDE.md에 없어서 Claude가 틀린 명령을 실행하는 경우다.

    ## 테스트
- 단위 테스트: `npm run test`
- e2e: `npm run test:e2e`
- 커버리지: `npm run test:cov`

    CLAUDE.md에 이걸 추가하면 해결된다.

    같은 실수를 반복한다¶

    세션이 끊기면 이전 대화를 기억하지 못한다. 한 번 해결한 문제는 CLAUDE.md에 기록해두는 게 낫다.

    ## 주의사항
- TypeORM에서 save() 대신 반드시 update() 사용
  (save()를 쓰면 의도치 않게 새 레코드가 생성됨)

    대용량 파일 편집 중에 멈춘다¶

    컨텍스트 윈도우 초과다. 해결책은 단순하다. 파일을 모듈 단위로 쪼개거나, 한 번에 하나의 함수만 요청하거나, /compact로 공간을 확보하고 재시도하면 된다.

    12. Oh My Claude Code (OMC) — Claude Code의 Oh My Zsh¶

    터미널 사용자라면 Oh My Zsh를 알 거다. Zsh에 플러그인·테마·단축키를 한 방에 추가해주는 프레임워크. OMC는 그걸 Claude Code에 적용한 개념이다.

    설치 한 번으로 32개 전문 에이전트와 40개 이상의 스킬을 바로 쓸 수 있게 된다. 일일이 커스텀 스킬을 만들고 서브에이전트 사용법을 익힐 필요 없이, 이미 잘 만들어진 워크플로우를 가져다 쓰는 방식이다.

    설치¶

    npm install -g oh-my-claude-sisyphus

    또는 Claude Code 안에서:

    /oh-my-claudecode:omc-setup

    설치하면 프로젝트 루트에 .omc/ 디렉토리가 생기고, 여기서 세션 상태와 HUD 정보를 관리한다.

    5가지 실행 모드¶

    OMC의 핵심은 상황에 따라 다른 모드를 선택할 수 있다는 거다.

    모드 언제 쓰나 특징
    Autopilot 평소 개발 입력을 분석해서 적합한 에이전트 자동 선택
    Ultrapilot 퀄리티가 최우선일 때 여러 에이전트를 동시 동원, 가장 느리고 비쌈
    Swarm 대규모 리팩토링 에이전트 다수를 병렬 투입, 빠르게 처리
    Pipeline 정해진 순서가 있는 작업 단계별 파이프라인 구성, 중간 결과 확인 가능
    Ecomode 비용 절감이 목표일 때 토큰 최소화, 가장 저렴하게

    자동 라우팅이 편한 이유¶

    일일이 "디자이너 에이전트 써줘", "아키텍트 에이전트로 분석해줘" 하지 않아도 된다. OMC가 요청을 파악해서 알아서 보낸다.

    "Fix the UI button colors" → 디자이너 에이전트
"Debug this crash" → 아키텍트 에이전트
"Don't stop until done" → 퍼시스턴스 모드 자동 활성화

    Oh My Open Code (OMO)¶

    OMC가 Claude Code 전용이라면, OMO는 오픈소스 AI 코딩 도구들을 위한 유사 에코시스템이다. Cursor, Continue, Aider 같은 도구들을 쓴다면 OMO 쪽을 찾아보는 게 맞다.

    특정 벤더에 종속되지 않는다는 게 가장 큰 차이다. 여러 AI 백엔드를 섞어 쓰거나, Claude Code 없이 오픈소스 환경만으로 구성하고 싶다면 OMO가 더 맞는 선택이다.

    마치며¶

    결국 핵심은 단순하다.

    Claude Code를 쓸 때 가장 흔한 실수는 AI한테 너무 많은 걸 한 번에 기대하는 거다. 범위를 좁히고, 맥락을 줘야 할 때 주고, 반복적인 작업은 자동화해두면 처음과는 다른 도구가 된다.

    원칙 실천
    맥락을 줘라 CLAUDE.md 작성
    범위를 좁혀라 파일·함수 단위로 요청
    방향을 먼저 잡아라 /plan 활용
    반복을 없애라 훅 + 스킬 + MCP
    한 단계 더 나아가고 싶다면 OMC 도입 검토
