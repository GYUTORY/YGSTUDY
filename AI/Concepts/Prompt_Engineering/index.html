    코딩을 위한 프롬프트 엔지니어링¶

    1. 프롬프트 엔지니어링이란¶

    프롬프트 엔지니어링은 AI 코딩 도구에게 원하는 결과를 정확히 뽑아내기 위해 입력을 설계하는 기술이다. 단순히 "잘 쓰는 법"이 아니라, 모델의 동작 방식을 이해하고 그에 맞게 입력을 구성하는 작업이다.

    1.1 왜 중요한가¶

    같은 AI 도구를 사용해도 프롬프트에 따라 결과 품질이 크게 달라진다.

    나쁜 예: "로그인 고쳐줘"

    좋은 예: "src/auth/login.ts의 handleLogin 함수에서 bcrypt 비교 로직이 항상 false를 반환하는 버그를 수정해줘. 테스트도 업데이트해줘."

    차이는 명확하다. 나쁜 예는 모델이 프로젝트 어디를 봐야 하는지, 무엇이 문제인지 추측해야 한다. 좋은 예는 파일, 함수, 증상, 추가 작업까지 한 번에 전달한다.

    2. 메시지 구조: System / User / Assistant¶

    API로 LLM을 호출할 때 프롬프트는 세 가지 역할의 메시지로 구성된다. 각 역할이 모델의 동작에 미치는 영향이 다르다.

    flowchart LR
    subgraph 입력["요청 (Request)"]
        direction TB
        S["<b>System</b><br/>모델의 행동 규칙 설정<br/>─────────────<br/>• 역할 부여<br/>• 응답 형식 지정<br/>• 금지 사항 명시"]
        U["<b>User</b><br/>실제 작업 요청<br/>─────────────<br/>• 질문, 코드 생성 요청<br/>• 컨텍스트 전달<br/>• 예시 포함"]
        S --> U
    end

    subgraph 출력["응답 (Response)"]
        A["<b>Assistant</b><br/>모델의 응답<br/>─────────────<br/>• System 규칙을 따름<br/>• User 요청에 대한 답변<br/>• 이전 Assistant 응답 참조"]
    end

    U --> A

    style S fill:#1a1a2e,stroke:#e94560,color:#fff
    style U fill:#1a1a2e,stroke:#0f3460,color:#fff
    style A fill:#1a1a2e,stroke:#16c79a,color:#fff
    style 입력 fill:#0d1117,stroke:#30363d,color:#c9d1d9
    style 출력 fill:#0d1117,stroke:#30363d,color:#c9d1d9

    실제 API 호출에서 이 구조가 어떻게 쓰이는지 보면:

    response = client.messages.create(
    model="claude-sonnet-4-20250514",
    system="너는 Spring Boot 전문 코드 리뷰어야. 보안 이슈만 지적해.",  # System
    messages=[
        {"role": "user", "content": "이 코드 리뷰해줘: ..."},         # User
        {"role": "assistant", "content": "SQL injection 위험: ..."},   # Assistant (이전 응답)
        {"role": "user", "content": "수정 방법도 알려줘"},             # User (후속 요청)
    ]
)

    System 메시지는 매 턴마다 유지되므로, 반복 지시를 넣기에 적합하다. User 메시지에 넣은 규칙은 대화가 길어지면 모델이 놓칠 수 있지만, System 메시지는 상대적으로 잘 유지된다.

    3. 4블록 프롬프트 패턴¶

    프롬프트의 기본 구조다. 모든 프롬프트가 이 구조를 따를 필요는 없지만, 복잡한 요청일수록 이 틀을 갖추면 결과가 좋아진다.

    3.1 구조¶

    1. INSTRUCTIONS (지시)     — 무엇을 해야 하는지
2. CONTEXT (컨텍스트)       — 배경 정보, 제약 조건
3. TASK (작업)             — 구체적인 작업 내용
4. OUTPUT FORMAT (출력 형식) — 원하는 결과 형태

    3.2 예시¶

    [INSTRUCTIONS]
Spring Boot REST API 엔드포인트를 작성해줘.

[CONTEXT]
- 프로젝트: Spring Boot 3.x + Java 17
- 패키지 구조: com.example.api.{domain}
- 인증: Spring Security + JWT
- DB: PostgreSQL + JPA

[TASK]
사용자 프로필 조회 API를 만들어줘.
- GET /api/v1/users/{id}/profile
- 인증 필요 (JWT)
- 응답에 이름, 이메일, 가입일 포함
- 사용자가 없으면 404 반환

[OUTPUT FORMAT]
Controller, Service, Repository, DTO 클래스를 각각 작성해줘.

    4. 핵심 원칙¶

    4.1 구체적으로 작성¶

    원칙 나쁜 예 좋은 예
    파일 지정 "코드 수정해줘" "src/auth/login.ts 수정해줘"
    함수 지정 "버그 고쳐줘" "handleLogin 함수의 null 체크 추가해줘"
    범위 명시 "테스트 추가해줘" "UserService의 createUser 메서드에 대한 단위 테스트 작성해줘"
    제약 조건 "API 만들어줘" "REST API + JWT 인증 + 페이지네이션 포함해서 만들어줘"

    4.2 컨텍스트 제공¶

    좋은 예:
"이 프로젝트는 NestJS + TypeORM을 사용해.
기존 UserModule의 패턴을 따라서
OrderModule을 만들어줘."

나쁜 예:
"주문 모듈 만들어줘"

    모델은 프로젝트 구조를 모른다. 기술 스택, 기존 패턴, 네이밍 규칙 같은 정보를 직접 넣어야 프로젝트에 맞는 코드가 나온다.

    4.3 단계별 분리¶

    복잡한 작업은 한번에 요청하지 말고 단계별로 나눈다.

    Step 1: "먼저 현재 인증 구조를 분석해줘"
Step 2: "OAuth2 적용 계획을 세워줘"
Step 3: "계획대로 구현해줘"
Step 4: "테스트를 작성하고 실행해줘"

    한 번에 "OAuth2 인증 전체 구현해줘"라고 하면 모델이 중간에 맥락을 놓치거나, 기존 구조와 맞지 않는 코드를 생성하는 경우가 많다. 각 단계의 결과를 확인하고 다음으로 넘어가야 한다.

    4.4 출력 형식 지정¶

    "결과를 다음 형식으로 보여줘:
- 변경된 파일 목록
- 각 파일별 변경 사항 요약
- 테스트 실행 결과"

    형식을 지정하지 않으면 모델이 자유롭게 출력하는데, 이때 필요한 정보가 빠지거나 불필요한 설명이 붙는다.

    5. 주요 프롬프팅 기법¶

    Zero-shot vs Few-shot 비교¶

    프롬프팅 기법의 핵심 분기점은 "예시를 주느냐 안 주느냐"다. 아래 흐름도로 어떤 상황에서 어떤 기법을 쓸지 판단할 수 있다.

    flowchart TD
    Start(["작업 요청"]) --> Q1{"모델이 작업의<br/>형식/패턴을<br/>알고 있는가?"}

    Q1 -->|"예"| ZS["<b>Zero-shot</b><br/>예시 없이 지시만 전달"]
    Q1 -->|"아니오 / 불확실"| Q2{"커스텀 형식이나<br/>특정 패턴이<br/>필요한가?"}

    Q2 -->|"아니오"| ZS
    Q2 -->|"예"| FS["<b>Few-shot</b><br/>입출력 예시 2~3개 포함"]

    ZS --> ZR["지시: 이 에러 메시지를<br/>사용자 친화적으로 바꿔줘"]
    FS --> FR["예시:<br/>입력: Connection refused<br/>출력: DB 연결 실패 (ERR_DB_CONN)<br/>───<br/>입력: Token expired<br/>출력: 로그인 만료 (ERR_AUTH)"]

    ZR --> ZO["모델이 자체 판단으로<br/>형식을 결정"]
    FR --> FO["예시 패턴을 학습해서<br/>일관된 형식으로 출력"]

    Q1 -->|"복잡한 추론 필요"| COT["<b>Chain-of-Thought</b><br/>단계별 분석 순서 지정"]
    COT --> CR["1단계: 리소스 나열<br/>2단계: 해제 지점 확인<br/>3단계: 누락 분석<br/>4단계: 수정안 제시"]

    style Start fill:#1a1a2e,stroke:#e94560,color:#fff
    style ZS fill:#0d2137,stroke:#0f3460,color:#fff
    style FS fill:#0d2137,stroke:#16c79a,color:#fff
    style COT fill:#0d2137,stroke:#e94560,color:#fff
    style Q1 fill:#1a1a2e,stroke:#f5a623,color:#fff
    style Q2 fill:#1a1a2e,stroke:#f5a623,color:#fff
    style ZR fill:#161b22,stroke:#30363d,color:#c9d1d9
    style FR fill:#161b22,stroke:#30363d,color:#c9d1d9
    style ZO fill:#161b22,stroke:#30363d,color:#c9d1d9
    style FO fill:#161b22,stroke:#30363d,color:#c9d1d9
    style CR fill:#161b22,stroke:#30363d,color:#c9d1d9

    정리하면:

    • Zero-shot: 일반적인 작업. 코드 생성, 버그 수정, 리팩토링처럼 모델이 이미 형식을 아는 작업에 쓴다.
    • Few-shot: 커스텀 포맷이 필요한 작업. 에러 메시지 변환, 코드 스타일 전환, 네이밍 규칙 적용처럼 "내가 원하는 형식"이 있을 때 예시로 보여준다.
    • Chain-of-Thought: 복잡한 분석. 버그 추적, 성능 병목 파악처럼 여러 단계를 거쳐야 하는 작업에 쓴다.

    5.1 Few-shot Prompting¶

    예시를 프롬프트에 포함해서 원하는 패턴을 모델에게 보여주는 기법이다. "이런 식으로 해줘"라고 말로 설명하는 것보다, 입출력 예시를 직접 보여주는 게 훨씬 정확하다.

    다음 형식으로 에러 메시지를 한국어 사용자용 메시지로 변환해줘.

예시 1:
입력: "Connection refused: localhost:5432"
출력: "데이터베이스 연결에 실패했습니다. 관리자에게 문의하세요. (ERR_DB_CONN)"

예시 2:
입력: "Token expired"
출력: "로그인이 만료되었습니다. 다시 로그인해 주세요. (ERR_AUTH_EXPIRED)"

변환 대상:
입력: "File not found: /uploads/avatar.png"

    예시가 2~3개면 대부분 패턴을 잡는다. 복잡한 변환이면 5개까지 늘린다. 그 이상은 토큰 낭비다.

    Few-shot이 잘 먹히는 상황:

    • 코드 변환 (한 스타일에서 다른 스타일로)
    • 에러 메시지 포맷팅
    • API 응답 구조 변환
    • 네이밍 규칙 적용

    주의할 점: 예시에 일관성이 없으면 모델도 일관성 없는 결과를 낸다. 예시 간에 패턴이 충돌하지 않는지 확인해야 한다.

    5.2 Chain-of-Thought (단계별 추론 유도)¶

    모델에게 "바로 답하지 말고, 단계별로 생각해봐"라고 유도하는 기법이다. 복잡한 버그 분석이나 아키텍처 결정에서 점프 없이 추론하게 만든다.

    이 함수에서 메모리 누수가 발생하는 원인을 찾아줘.
바로 답하지 말고, 다음 순서로 분석해줘:

1. 함수 내에서 할당되는 리소스를 모두 나열해
2. 각 리소스가 해제되는 지점을 확인해
3. 해제되지 않는 리소스가 있다면 어떤 조건에서 누락되는지 설명해
4. 수정 방법을 제시해

    그냥 "메모리 누수 찾아줘"라고 하면 모델이 한두 줄로 답하고 끝내거나, 실제 원인이 아닌 일반적인 패턴만 나열하는 경우가 있다. 분석 순서를 지정하면 각 단계를 거치면서 실제 문제를 짚어낼 확률이 높아진다.

    Chain-of-Thought가 필요한 상황:

    • 여러 파일에 걸친 버그 추적
    • 성능 병목 분석
    • 데이터 흐름이 복잡한 코드에서 원인 파악
    • 설계 대안 비교 (A vs B를 각각 분석 후 결론)

    간단한 코드 생성이나 단순 질문에는 필요 없다. 오히려 불필요한 추론 과정만 늘어난다.

    5.3 역할 부여¶

    System prompt나 프롬프트 앞부분에서 모델에게 역할을 부여한다. 단, "세계 최고의 전문가"처럼 과장된 역할을 주면 오히려 답변이 장황해진다. 구체적인 업무 맥락을 주는 게 낫다.

    너는 핀테크 스타트업의 백엔드 리드 개발자야.
기술 스택은 Spring Boot 3 + Kotlin + PostgreSQL이야.
코드 리뷰 시 보안 이슈를 최우선으로 지적하고,
금융 관련 로직에서는 트랜잭션 처리와 멱등성을 확인해.

    5.4 제약 조건 설정¶

    모델에게 하지 말아야 할 것을 명시하는 게 생각보다 중요하다. 제약 없이 쓰면 모델은 친절하게 부연 설명을 덧붙이고, 관련 내용을 추가로 알려주려 한다.

    제약 조건:
- 질문에 직접 관련 없는 내용은 답하지 마라
- 코드를 생성할 때 주석을 달지 마라
- 확실하지 않은 내용은 "확인 필요"라고 표시해라
- deprecated API를 사용하지 마라

    제약 조건은 긍정형보다 부정형이 잘 먹힌다. "간결하게 답해"보다 "3문장을 넘기지 마"가 정확도가 높다. 모델은 "간결하게"의 기준을 자의적으로 해석하지만, "3문장"은 명확히 지킨다.

    6. 컨텍스트 관리¶

    6.1 토큰 한계 인식¶

    LLM의 추론 품질은 컨텍스트가 길어질수록 저하된다. 프롬프트는 150~300 단어 범위가 집중도가 높다.

    상황 대응
    프롬프트가 길어질 때 핵심만 추려서 전달
    큰 파일을 다룰 때 관련 함수/섹션만 지정
    이전 대화를 참조할 때 대화 정리 후 핵심 정보 재전달

    6.2 프롬프트 체이닝¶

    복잡한 작업을 순차적 프롬프트로 분해한다. 응답 시간은 늘지만 정확도가 올라간다.

    Prompt 1: "현재 DB 스키마를 분석해줘"
  → (결과 확인)
Prompt 2: "분석 결과를 바탕으로 마이그레이션 계획을 세워줘"
  → (계획 확인)
Prompt 3: "계획대로 마이그레이션 스크립트를 작성해줘"
  → (코드 확인)
Prompt 4: "작성된 스크립트를 테스트해줘"

    6.3 멀티턴 대화에서 컨텍스트 흘러가는 문제¶

    대화가 10턴 이상 넘어가면 모델이 초반 지시를 잊거나, 앞에서 합의한 내용과 다른 코드를 생성하는 일이 생긴다. 이건 LLM의 구조적 한계다. 토큰이 많아질수록 초반 정보의 영향력이 줄어든다.

    컨텍스트가 흘러가는 증상:

    • 앞에서 정한 네이밍 규칙을 안 지킨다
    • 이미 논의한 내용을 다시 물어본다
    • 초반에 지정한 기술 스택과 다른 라이브러리를 갑자기 쓴다
    • 이전 턴에서 생성한 코드와 호환되지 않는 코드를 낸다

    대응 방법:

    1. 핵심 제약을 반복 삽입: 중요한 규칙은 대화 중간중간 다시 언급한다. "앞에서 말한 것처럼 TypeORM을 사용해"라고 한 줄 넣는 것만으로 모델이 맥락을 유지하는 데 도움이 된다.

    2. 대화 끊고 새로 시작: 15~20턴을 넘기면 새 대화를 여는 게 낫다. 이전 대화의 결론만 요약해서 새 대화의 첫 프롬프트에 넣는다.

    이전 대화에서 결정된 사항:
- OrderService는 CQRS 패턴 적용
- Command와 Query를 별도 클래스로 분리
- 이벤트는 Spring ApplicationEvent 사용

이어서 OrderCommandHandler를 구현해줘.
    1. 설정 파일 활용: AI 코딩 도구에는 프로젝트 설정 파일(CLAUDE.md, .cursor/rules 등)이 있다. 여기에 넣은 내용은 매 턴마다 자동으로 주입되므로 컨텍스트가 흘러가지 않는다. 반복적으로 지켜야 할 규칙은 프롬프트가 아닌 설정 파일에 넣어야 한다.

    7. 프롬프트가 안 먹힐 때 디버깅¶

    프롬프트를 작성했는데 원하는 결과가 안 나오는 경우가 자주 있다. 이때 프롬프트를 통째로 다시 쓰기 전에, 문제가 어디에 있는지 좁혀야 한다.

    7.1 모델이 지시를 무시하는 경우¶

    증상: "주석 달지 마"라고 했는데 주석을 단다. "JSON으로 답해"라고 했는데 마크다운으로 답한다.

    원인과 대응:

    • 지시가 프롬프트 중간에 묻혀 있다: 중요한 지시는 프롬프트 앞부분이나 맨 뒤에 배치한다. 긴 컨텍스트 중간에 넣으면 모델이 놓치는 경우가 있다.
    • 긍정형으로 작성했다: "간결하게 답해" 대신 "3문장 이내로 답해"처럼 수치를 넣는다.
    • 상충하는 지시가 있다: "자세히 설명해"와 "짧게 답해"가 같은 프롬프트에 있으면 모델이 하나를 무시한다. 프롬프트 내에 모순이 없는지 확인한다.

    7.2 모델이 엉뚱한 코드를 생성하는 경우¶

    증상: 요청한 것과 다른 라이브러리를 쓰거나, 프로젝트 구조와 맞지 않는 코드를 낸다.

    원인과 대응:

    • 컨텍스트 부족: 기술 스택이나 프로젝트 구조를 명시하지 않았다. 모델은 기본값으로 가장 일반적인 패턴을 선택한다. 프로젝트 정보를 추가한다.
    • 예시 부족: 원하는 패턴을 말로 설명하는 대신, 기존 코드를 예시로 보여준다. "UserService와 같은 패턴으로 OrderService를 만들어줘"처럼 참조점을 준다.
    • 한 번에 너무 많이 요청했다: 여러 파일을 동시에 생성하거나 복잡한 로직을 한 번에 요청하면 품질이 떨어진다. 단계를 나눈다.

    7.3 결과가 불안정한 경우 (같은 프롬프트인데 결과가 다름)¶

    원인과 대응:

    • temperature가 높다: API 사용 시 temperature를 0에 가깝게 설정한다. 코드 생성 작업에서는 temperature 0~0.2가 적절하다.
    • 프롬프트가 모호하다: "적절한 에러 핸들링 추가해줘"보다 "try-catch로 감싸고, catch에서 로그를 남기고 커스텀 에러를 throw해"처럼 구체적으로 쓴다.
    • 출력 형식을 지정하지 않았다: 형식을 명시하면 결과의 편차가 줄어든다.

    7.4 디버깅 절차¶

    프롬프트가 안 먹힐 때 순서대로 시도한다:

    1. 프롬프트를 줄여본다: 불필요한 내용을 빼고 핵심 지시만 남겨서 테스트한다. 이걸로 작동하면, 제거한 부분 중 무언가가 간섭하고 있다는 뜻이다.
    2. 지시를 분리해본다: 하나의 프롬프트에 여러 지시가 있다면, 각 지시를 별도 프롬프트로 테스트해서 어떤 지시가 문제인지 찾는다.
    3. 예시를 추가한다: 원하는 입출력 예시를 1~2개 넣어본다. 말로 설명하는 것보다 예시가 확실하다.
    4. 모델을 바꿔본다: 같은 프롬프트라도 모델에 따라 결과가 다르다. 현재 모델이 작업에 적합한지 확인한다.

    8. 프로덕션 환경에서 프롬프트 관리¶

    프롬프트를 개인 작업용으로 쓸 때는 자유롭게 수정하면 되지만, API를 통해 서비스에 프롬프트를 반복 사용할 때는 코드처럼 관리해야 한다.

    8.1 프롬프트 버전 관리¶

    프롬프트도 코드와 같이 Git으로 관리한다. 프롬프트 변경이 출력에 미치는 영향이 크기 때문에, 언제 무엇을 바꿨는지 추적할 수 있어야 한다.

    prompts/
├── review/
│   ├── code_review_v1.txt
│   ├── code_review_v2.txt      # v1에서 출력 형식 변경
│   └── code_review_v2.1.txt    # v2에서 한국어 품질 개선
├── generation/
│   └── api_endpoint_v1.txt
└── changelog.md                # 각 버전의 변경 사유

    파일명에 버전을 넣는 단순한 방식이 가장 관리하기 쉽다. 프롬프트 내용, 변경 사유, 적용 모델을 함께 기록한다.

    8.2 프롬프트 테스트¶

    프롬프트를 변경할 때마다 기존 결과가 깨지지 않는지 확인해야 한다. 프롬프트는 코드와 달리 컴파일 에러가 없으니, 결과물을 직접 확인하는 수밖에 없다.

    테스트 방법:

    1. 골든 테스트 세트 구축: 자주 쓰는 입력과 기대 결과를 모아둔다. 프롬프트를 바꿀 때마다 이 세트로 돌려본다.
    test_cases = [
    {
        "input": "NullPointerException at UserService.java:42",
        "expected_contains": ["null 체크", "UserService"],
        "expected_not_contains": ["확실하지 않"]
    },
    {
        "input": "SELECT * FROM users WHERE id = " + user_input,
        "expected_contains": ["SQL injection", "파라미터 바인딩"]
    }
]

for case in test_cases:
    result = call_llm(prompt_template, case["input"])
    for keyword in case["expected_contains"]:
        assert keyword in result, f"'{keyword}' not found in result"

    1. A/B 비교: 프롬프트 v1과 v2의 결과를 같은 입력으로 비교한다. 자동화하기 어렵다면, 결과를 나란히 놓고 사람이 판단하는 방식이라도 해야 한다.

    2. 회귀 방지: 프롬프트를 개선하면서 기존에 잘 되던 케이스가 깨지는 일이 잦다. 개선 대상 케이스만 보지 말고, 기존 케이스도 같이 확인한다.

    8.3 프롬프트 템플릿화¶

    반복 사용하는 프롬프트는 변수 부분과 고정 부분을 분리해서 템플릿으로 만든다.

    REVIEW_PROMPT = """
다음 코드를 리뷰해줘.

언어: {language}
파일: {file_path}
맥락: {context}

코드:
```{language}
{code}

    다음 항목을 확인해줘: - 버그 가능성 - 보안 취약점 - 네이밍 규칙 위반 """

    result = call_llm( REVIEW_PROMPT.format( language="java", file_path="src/main/java/com/example/UserService.java", context="사용자 등록 로직, Spring Boot 3 + JPA", code=code_content ) ) ```

    템플릿의 고정 부분을 바꾸면 전체 서비스에 영향이 가니, 반드시 테스트 후 배포해야 한다.

    9. 안티패턴 (피해야 할 것)¶

    안티패턴 문제 대안
    과도하게 긴 프롬프트 핵심이 묻힘, 토큰 낭비 150~300 단어로 압축
    모호한 지시 모델이 잘못된 방향으로 진행 구체적 파일/함수/결과 명시
    한번에 모든 것 요청 품질 저하 단계별 분리
    컨텍스트 없는 요청 프로젝트와 맞지 않는 코드 기술 스택/규칙 제공
    결과 확인 없이 연속 요청 오류 누적 각 단계 결과 확인 후 진행
    프롬프트를 통째로 재작성 잘 되던 부분까지 깨짐 한 번에 하나만 변경

    10. 프로젝트 설정 파일 활용¶

    각 AI 도구에는 프로젝트 설정 파일이 있다. 매 대화에 자동으로 적용되는 "영구적 프롬프트" 역할이다. 반복 지시를 프롬프트에 쓰는 대신 여기에 넣어야 한다.

    도구 파일 역할
    Claude Code CLAUDE.md 프로젝트 규칙, 컨벤션, 빌드 명령
    Codex AGENTS.md 프로젝트 개요, 테스트 명령, 스타일 규칙
    Cursor .cursor/rules/*.mdc glob 패턴별 규칙
    Gemini CLI GEMINI.md 프로젝트 규칙
    Copilot .github/copilot-instructions.md 커스텀 지시

    이 설정 파일들을 잘 작성하면 매번 프롬프트에 컨텍스트를 반복할 필요가 없다. 도구별 상세한 활용법은 각 도구의 문서를 참고한다.
